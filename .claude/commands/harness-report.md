ハーネス実行結果を分析し、ハーネス自体の改善点レポートを出力してください。

## 評価フレームワーク

改善提案は以下の基準で妥当性を判定する。基準を満たさない提案は却下する。

### Mission適合性
ハーネスの目的: 10Mステップでもユーザーの指示通りの修正を一度の実行で完遂する。
提案がこの目的に貢献するか。「便利だが意図精度に無関係」な提案は優先度を下げる。

### 6本柱チェック（P1-P6）
各提案を6本柱に照合し、侵害がないか検証する:
- P1違反: ゲートをバイパスする提案 → 却下
- P2違反: LLM出力を信頼する提案 → 却下
- P3違反: リトライを前提とする提案 → 再設計
- P4違反: LLM判断をゲートに使う提案 → 却下
- P5違反: コンテキストを増やす提案 → 代替案を検討
- P6違反: プロジェクト固有知識に依存する提案 → 汎用化

### 問題の分類
提案を以下に分類し、対策を変える:
- ハーネス設計の問題 → コード修正
- 運用の問題（使い方） → ドキュメント/テンプレート改善
- LLMの能力限界 → プロンプト改善 or 決定的チェックで補完
- 誤認（既に対応済み） → 却下 + 既存機能の周知

### コード検証
提案の前提が正しいか、実際のコードを確認する:
- 「Xが存在しない」→ 本当に存在しないか Grep/Read で検証
- 「Yが動かない」→ テストが通っているか確認
- 既存の代替手段がないか調査

## データ収集

1. 現在のタスク状態を取得: `harness_status` で taskId を確認
2. ワークフロー状態ファイルを読む: `.claude/state/workflows/` 配下の `workflow-state.toon`
3. proofLog を分析（retries, force transitions, DoD failures）
4. docs/workflows/ 配下の成果物を確認
5. git log でセッション中のコミット履歴を確認

## 分析観点

1. forceTransition の使用状況 — 回数、使用フェーズ、本来パスすべきだったか
2. DoD ゲート失敗パターン — false positive の有無、基準の妥当性
3. フェーズの実効性 — 追認フェーズの有無、サイズに対するフェーズ数の過剰
4. テンプレート・artifact問題 — パースエラー、ファイル名不一致
5. ツールチェーン摩擦 — hookブロック、Agent起動オーバーヘッド
6. セッション・コンテキスト問題 — コンテキスト切れ、復帰非効率
7. 意図精度チェーン — userIntentとscopeの乖離、AC/RTMカバレッジ

## 出力フォーマット

各改善提案に以下を含める:

1. 問題の具体的な記述
2. 影響（ユーザー体験への影響）
3. 分類（設計/運用/LLM限界/誤認）
4. 6本柱チェック結果（P1-P6のどれに影響するか）
5. Mission適合性（意図精度への貢献度）
6. 提案（具体的な改善案）
7. 判定（採用/却下/要再設計）

重要度の高い順に並べ、最後に「総評」を1段落で記述する。
