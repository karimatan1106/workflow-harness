# ADR-006: 言語前提を3点に閉じ込める

Status: Accepted
Date: 2026-05-01

## Context
ハーネスは TypeScript / Python 中心の言語前提でフェーズゲート(PHASE_EXT)、Bash 許可カテゴリ(BASH_COMMANDS)、Worker 拡張子許可、skill テンプレを設計している。Rust(および将来追加される Go / Java / C++ 等)を sidecar として扱うとき、言語特化コードが多数のファイルに散らばると、次の言語追加時に網羅探索が必要になり保守不能化する。

## Decision
言語前提を以下の3点に閉じ込める:

1. `hooks/phase-config.js` — `BASH_COMMANDS` および `PHASE_EXT`(SoT)
2. `mcp-server/src/phases/registry.ts` — 各フェーズの `allowedExtensions`
3. `.claude/skills/workflow-harness/workflow-*.md` — 教示用 reference(SoT は hook 側)

これら以外の場所で言語特化条件分岐(例: `if (lang === 'rust')`)を書かない。新言語追加は上記3点の追記のみで完結する。

## Consequences

### Positive
- 新言語追加コストの定量化(3ファイル追記で完結)
- BASH_COMMANDS と skill 表記の二重管理を SoT を hook に決めることで解消(skill は説明用 reference)
- Worker allowed-extensions の運用が簡潔化(既存 `.agent/.worker-allowed-extensions` は言語非依存)

### Negative / Trade-offs
- skill md と hook の文言乖離(rotted)を許容する代わりに、テスト assert は hook 側のみで判定する
- 言語特化のビルドコマンド(例: build_check ランタイム検知)は別 ADR で扱う

## Out of Scope
- Rust workspace(workspace members 多 crate 構成)サポート
- rust-toolchain.toml 自動生成
- cargo bench / cargo doc / cargo publish の追加対応
