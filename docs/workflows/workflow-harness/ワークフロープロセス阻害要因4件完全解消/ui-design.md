# ui-design.md - ワークフロープロセス阻害要因4件完全解消

## サマリー

本ドキュメントはワークフロープロセス阻害要因4件の修正タスクにおけるUI設計結果を記述したものである。重要な点として、本タスクが対象とする4件の阻害要因（B-1〜B-4）は全てバックエンドのロジック修正およびドキュメント更新に限定されており、フロントエンドのUIコンポーネント、画面設計、インタラクション設計といった視覚的なユーザーインターフェース要素には一切の変更が及ばない構成になっている。

修正対象はワークフロープラグインの内部実装（hooks/discover-tasks.js、phase-edit-guard.js）とMCP サーバー実装（test-tracking.ts）、並びに開発者向けドキュメント（CLAUDE.md、MEMORY.md）に限定されている。これらの修正は全て標準出力やログメッセージレベルでの影響に留まり、UIコンポーネントの描画、レイアウト、スタイルシート、アクセシビリティ属性といった要素には影響を与えない設計になっている。

ただし、ユーザーが実際に認識する可能性のあるCLIレベルのメッセージについて、B-2およびB-3で新規追加されるエラーメッセージおよび警告ログについて本セクションで記載する。次フェーズの実装でこれらメッセージの内容と表示形式を確定することが重要である。

## 変更対象の分析

### B-1: discover-tasks.jsのタスクソート

**UI/UX への直接的な影響**: なし

ソート処理はバックエンド内部で完結する。discover-tasks.js は複数のアクティブなワークフロータスクが存在する場合に、タスク ID の降順でソートして配列を返却するロジックを追加する。この変更により、phase-edit-guard.js の `findActiveWorkflowTask()` が常に最新のタスクを選択できるようになる結果、ユーザーの操作フローに予測可能性がもたらされる。

ただしこの改善は内部動作の決定性向上であり、画面表示、メニュー、ダイアログ、アラートといったUIコンポーネントには影響しない。

### B-2: commitフェーズおよびpushフェーズでのgit操作許可

**UI/UX への直接的な影響**: なし（CLIメッセージのみ新規追加）

phase-edit-guard.js の analyzeBashCommand() 関数に commitフェーズと pushフェーズでの git操作ホワイトリストロジックを追加する修正である。許可される操作（git add、git commit、git tag、git push）が実行可能になることで、ワークフローの commitフェーズと pushフェーズが正常に進行可能になる。

修正の結果として、以下の２つのシナリオにおいてエラーメッセージが表示されなくなる：
- commitフェーズで `git add .` や `git commit -m "message"` を実行した場合に発生していたブロックメッセージが消滅する
- pushフェーズで `git push origin main` を実行した場合に発生していたブロックメッセージが消滅する

一方で、破壊的オプション（--force、--amend等）を使用した場合には新規追加の拒否メッセージが表示される：

```
【commit/pushフェーズでのgit操作制限】
- git add/commit/tag（commitフェーズ）
- git push（pushフェーズ）
のみ許可されています。

実行しようとしたコマンド: ${command}
現在のフェーズ: ${currentPhase}

破壊的オプション（--force, --amend等）は禁止されています。
```

### B-3: testingフェーズでのベースライン記録許可

**UI/UX への直接的な影響**: なし（ログメッセージのみ新規追加）

test-tracking.ts の workflowCaptureBaseline() 関数のフェーズ検証条件を研究フェーズと テスト フェーズの2つに拡張する修正である。現状では researchフェーズでのみベースライン記録が許可されており、testingフェーズでの記録を忘れた場合には regression_test フェーズへ進む際に記録不可能状態となる問題が解消される。

修正の結果として、testingフェーズでも captureBaseline() ツールが利用可能になり、柔軟な運用が可能になる。この際、testingフェーズでの記録を「遅延ベースライン」として扱う旨の警告ログが標準出力に表示される：

```
[警告] testingフェーズでのベースライン記録（遅延ベースライン）
タスク: ${taskId}
推奨: 今後はresearchフェーズでの記録を推奨します
```

このログメッセージにより開発者は自身が推奨されていない経路を採用していることを認識でき、今後の改善につながる情報を得られる設計になっている。

### B-4: MCPサーバー再起動に関するドキュメント化

**UI/UX への直接的な影響**: なし（ドキュメント追記のみ）

CLAUDE.md および MEMORY.md に Node.js のモジュールキャッシュ機構とそれに起因する MCPサーバー再起動の必要性を記載する修正である。これは開発者向けのベストプラクティス情報の拡充であり、ユーザーが実際に操作するUI層には影響しない。

ただしこの情報の周知が不十分な場合、開発者がコード修正を加えても変更が反映されないという混乱が生じる可能性があるため、ドキュメントの記載品質が間接的に開発生産性に影響する。

## CLIインターフェースへの影響

### B-2: git操作ブロック時の新規エラーメッセージ

commitフェーズにて破壊的 git操作（`git commit --amend`等）を実行した場合、以下の形式でエラーメッセージが表示される：

```
commitフェーズでのgit commit --amendおよび--no-verifyは禁止されています。
```

pushフェーズにて強制プッシュ（`git push --force`等）を実行した場合、以下の形式でエラーメッセージが表示される：

```
pushフェーズでのgit push --forceは禁止されています。
```

これらのメッセージは既存のブロック メッセージスタイルと一貫性を保つ形式で表示される。

### B-3: テストフェーズベースライン記録時の警告ログ

testingフェーズで captureBaseline() ツールを使用してテストベースラインを記録する際、以下の警告ログが console.warn() により標準エラー出力に表示される：

```
[警告] testingフェーズでのベースライン記録（遅延ベースライン）
タスク: ${taskId}
推奨: 今後はresearchフェーズでの記録を推奨します
```

このログは記録されたことの確認と同時に、推奨されるベストプラクティスを開発者に通知する役割を担う。

### ログ出力フォーマットの統一性

本修正で新規追加されるエラーメッセージおよび警告ログについて、既存のログ出力スタイルとの統一性を維持する必要がある。具体的には以下の点に留意する：

- エラーメッセージはブロックされた理由を明確に説明する文言を含めること
- ログレベルの区別（console.error vs console.warn）を適切に使い分けること
- 日本語表記とコマンド例の混在について見読性を損なわないこと
- 複数行メッセージの場合、行頭の記号や括弧の使用で見出し性を確保すること

## まとめ

本修正タスクはワークフロープロセスの内部ロジック修正およびドキュメント更新に限定されており、HTMLコンポーネント、CSS スタイル、JavaScriptフレームワーク（React等）の UI実装、アクセシビリティ要件（ARIA属性、キーボード操作等）、レスポンシブデザイン、アニメーション、ダークモード対応といった従来の UI設計対象項目は全く関与しない構成になっている。

唯一のユーザーインターフェース的な変更点は、B-2のgit操作ブロック時のエラーメッセージおよびB-3のテストフェーズベースライン記録時の警告ログという、CLIレベルのテキストメッセージ出力である。これらメッセージの具体的な文言、フォーマット、表示タイミング、ログレベルについては、実装フェーズで確定することとする。
