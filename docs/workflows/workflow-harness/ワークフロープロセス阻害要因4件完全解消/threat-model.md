# threat-model.md - ワークフロープロセス阻害要因4件完全解消

## サマリー

本ドキュメントは4件のワークフロープロセス修正（B-1からB-4）に対する脅威分析の成果物となる。
各修正がシステムのセキュリティ、整合性、可用性に与える潜在的なリスクを特定し、対応策を提示する内容である。

B-1修正（タスクソート変更）の主要リスクは、ソート順序変更が既存コードに予期しない副作用を引き起こす可能性である。
特にファイルシステム依存の動作を前提とした実装箇所が影響を受けるケースが懸念される対象となる。

B-2修正（git操作許可）の最大の脅威は、ホワイトリスト実装の不備による意図しない破壊的操作の許可となる。
git reset、git checkout等の危険なコマンドが誤って実行可能になれば、リポジトリ破壊につながる深刻な事態を招く。

B-3修正（testingフェーズでのベースライン記録）の懸念点は、不正確なベースラインがリグレッション判定を無効化する問題である。
実装途中のコードでベースラインを取得すると、バグを含む状態が正常として記録されてしまう危険性が存在する。

B-4（MCPサーバーキャッシュのドキュメント化）は脅威レベルが最も低く、ドキュメント記述の不正確性のみが懸念対象である。
開発者が誤った理解をすると、デバッグ効率の低下や不要なトラブルシューティング時間の浪費につながる可能性がある。

全体として、B-2のgit操作許可が最も高リスクな変更であり、厳密なホワイトリスト設計と徹底的なテストが必要不可欠である。
B-1とB-3は中リスクであり、既存動作の詳細な検証と包括的な統合テストで対応可能な範囲の脅威となる。

脅威への対策として、各修正に対して多層防御の原則を適用し、単一の防御メカニズムに依存しない設計を採用する方針である。
特にB-2のgit操作許可については、ホワイトリスト・正規表現検証・オプション検査の三重のチェック機構を実装する計画とする。

testingフェーズでのベースライン記録に関しては、記録時に強制的な確認ダイアログを表示する追加対策を検討する価値がある。
また、ベースライン記録時のフェーズ情報を保持することで、後続の分析やトラブルシューティングを容易にする設計とする。

次フェーズの設計では、本脅威モデルで特定された各リスクに対する具体的な実装レベルの防御策を詳細化する作業を行う。
特にB-2のホワイトリスト設計については、エッジケースを網羅的にリストアップし、全てに対する動作定義を明確化する必要がある。

## 脅威モデル概要

本セクションでは、4件の修正に対する脅威分析の全体像とアプローチを示す内容となる。

### 分析対象システム

ワークフロープラグインは、MCPサーバー（TypeScript）とフックシステム（JavaScript）の2層構造で構成される。
フックシステムはファイル編集、Bashコマンド実行を監視し、フェーズごとの制約を強制する役割を担当している。
MCPサーバーはワークフロー状態の永続化、ツール実行、整合性検証をHMAC-SHA256で保護する責務を持つ。

今回の修正対象は、フックシステム（discover-tasks.js、phase-edit-guard.js、phase-rules.js）とMCPサーバー（test-tracking.ts）の両方にまたがる。
そのため、フロントエンド（フック）とバックエンド（MCPサーバー）の整合性維持が追加の考慮事項として浮上する構造である。

### 脅威分析手法

STRIDE（Spoofing、Tampering、Repudiation、Information Disclosure、Denial of Service、Elevation of Privilege）モデルを適用し、各修正の脅威を体系的に評価する。
各脅威に対してリスクレベル（Critical、High、Medium、Low）を割り当て、優先度付けを行う方式を採用する。

### 資産の特定

保護すべき主要資産として、以下の4つを定義する構成とした。
1. ワークフロー状態ファイル（workflow-state.json）の整合性 - HMAC保護の対象
2. Gitリポジトリの履歴と内容 - 破壊的git操作から保護する必要がある
3. テストベースライン情報（testBaseline）の正確性 - リグレッション判定の信頼性に直結
4. プラグインコードの実行ロジック - 予期しない動作変更からの保護

## B-1: タスクソート変更の脅威分析

### B-1タスクソート変更の概要

discover-tasks.jsにtaskId降順ソートロジックを追加する変更が、既存システムに与える影響を分析する内容である。
タスク選択順序の変更は、暗黙の順序依存を持つコードに予期しない動作を引き起こすリスクを内包する構造となっている。

### T1.1: 暗黙的な順序依存の破壊 (Medium Risk)

**脅威**: 既存コードが fs.readdirSync() の返却順序に依存している場合、ソート追加によって動作が変化する可能性がある懸念である。
特に phase-edit-guard.js の findActiveWorkflowTask() は、配列の先頭要素を取得する実装となっているため、ソート順序の影響を直接受ける箇所となる。

**影響**: 意図しないタスクが選択され、誤ったフェーズ制約が適用される事態が発生する可能性がある問題である。
最悪の場合、異なるタスクのファイルを誤って編集してしまう cross-contamination のリスクが存在する状況となる。

**攻撃シナリオ（T1.1順序変更）**:
1. ユーザーがタスクAで作業中に、新しいタスクBを誤って開始してしまう操作を行う状況を想定する
2. ソート前の実装では、ファイルシステムの順序によってはタスクAが引き続き選択される可能性が存在していた背景がある
3. ソート後は確実にタスクB（最新）が選択されるため、タスクAのコンテキストが失われる動作変更が発生する可能性がある

**T1.1対策（順序依存への対応）**:
- discover-tasks.js の全呼び出し箇所をコードベース全体で grep 検索し、返却値の順序依存を確認する作業を実施すること
- 特に配列の先頭要素（tasks[0]）を参照している箇所は、ソート後の動作が意図通りか詳細に検証する手順を踏むこと
- 統合テストで、複数タスクが存在する状態での phase-edit-guard.js の動作を網羅的に確認するケースを追加すること

**残存リスク**: Low - コードレビューとテストで検出可能な範囲の問題であり、本番環境でのリスクは限定的と評価される。

### T1.2: パフォーマンスへの影響 (Low Risk)

**脅威**: タスク配列のソート処理が、大量のタスク（100件以上）が存在する場合にボトルネックとなる可能性がある懸念である。
ただし、通常運用ではアクティブタスクは1〜10件程度であり、現実的なリスクは極めて低いと判断される状況となる。

**影響**: discover-tasks.js の呼び出しが頻繁に発生する箇所（phase-edit-guard.js のフック呼び出しごと）で、微小な遅延が累積する可能性がある。
ユーザー体感として、ファイル保存やBashコマンド実行時のレスポンスが低下する懸念が理論上は存在する構造である。

**T1.2対策（パフォーマンス確保）**:
- ソート実装には JavaScript 標準の Array.sort() を使用し、追加のライブラリ依存を避ける方針を採用すること
- パフォーマンステストで、100件のタスクが存在する状態での discover-tasks.js の実行時間を計測する手順を実施すること
- 計測結果が 10ms を超える場合は、メモ化やキャッシュ戦略を検討する対応を視野に入れること

**残存リスク**: Negligible - 通常運用での影響はほぼゼロと評価される水準である。

### T1.3: ソート基準の誤解 (Low Risk)

**脅威**: taskId のフォーマットに関する理解不足により、降順ソートが意図した動作（最新タスク優先）を実現しない実装になる可能性がある。
taskId は YYYYMMDD_HHmmss 形式であり、文字列辞書順で時系列順序が保証される構造を前提とした設計となっている点の認識が重要となる。

**影響**: ソート実装が誤っていた場合、最新タスクではなく最古タスクが選択される逆転現象が発生するリスクがある。
これは修正の目的を達成できないだけでなく、より予測困難な動作を生む悪化要因となる可能性を含む状況である。

**T1.3対策（ソート基準の検証）**:
- 単体テストで、taskId の異なる複数タスク（例: 20260209_120000、20260209_130000、20260209_140000）をソートし、正しい順序を検証する実装を行うこと
- 降順ソートであることを明示するコメントをコード中に追加し、将来の誤変更を防ぐ配慮を施すこと

**残存リスク**: Negligible - 単体テストで完全にカバー可能な範囲の脅威と評価される。

## B-2: commit/pushフェーズでのgit操作許可の脅威分析

### B-2 git操作許可変更の概要

commit フェーズと push フェーズで必要最小限の git 操作を許可する変更は、最も高リスクな修正対象となる内容である。
git コマンドは強力な破壊的操作を含むため、ホワイトリスト設計の不備は直接的なリポジトリ破壊につながる深刻な脅威を持つ。

### T2.1: ホワイトリストバイパスによる破壊的git操作 (High Risk)

**脅威**: ホワイトリスト実装の不備により、git reset --hard、git checkout .、git clean -f 等の破壊的コマンドが実行可能になる危険性がある。
正規表現のエスケープ不足や、コマンドラインパーサーの脆弱性を悪用したバイパス攻撃のリスクが存在する構造となる。

**影響**: 作業中のファイルが全て失われる最悪のシナリオが現実化し、復旧不可能な状態に陥る可能性がある問題である。
特に未コミットの変更を含むファイルが破壊された場合、ワークフロー成果物が完全に失われる致命的な事態につながる。

**攻撃シナリオ（T2.1ホワイトリスト回避）**:
1. 攻撃者（または誤操作を行うユーザー）が commit フェーズで `git reset --hard` を実行しようと試みる状況を想定する
2. ホワイトリストチェックが `git reset` の前方一致のみを確認し、オプション部分を検証していない実装の不備が存在する場合を考慮する
3. コマンドが実行され、ワークツリーの全変更が破壊される結果が発生してしまう深刻な事態となる

**T2.1対策（ホワイトリスト厳密化）**:
- ホワイトリストは完全一致またはオプションを含めた厳密な正規表現で実装し、曖昧なパターンマッチを排除する設計とすること
- 許可する git コマンドの明示的なリスト: `git add`、`git commit`（-m、-F、heredoc のみ）、`git push`（--tags 許可）、`git tag`
- 禁止する危険なオプション: --force、--hard、--amend、--no-verify、-f（git clean等）を明示的にブラックリスト化すること
- 正規表現例: `/^git\s+(add|commit\s+-[mF]|commit\s+.*<<|push(?!\s+--force)|tag)\s+/`
- phase-edit-guard.js の analyzeBashCommand() に専用の gitCommandWhitelist チェックロジックを追加する実装を行うこと
- 統合テストで、50種類以上の git コマンドパターン（正常、異常、境界値）を網羅的に検証するケースを用意すること

**残存リスク**: Medium - 包括的なテストとコードレビューで大幅にリスクは低減されるが、未知のエッジケースが残存する可能性は否定できない。

### T2.2: heredoc形式のgit commitを利用したコマンドインジェクション (Medium Risk)

**脅威**: git commit メッセージを heredoc 形式（cat <<'EOF'）で指定する場合、heredoc 内に悪意のあるコマンドを埋め込む攻撃の可能性がある。
Bash の heredoc パーサーの脆弱性や、クォート処理の不備を突いた実行コード注入のリスクが考えられる状況となる。

**影響**: heredoc 内に埋め込まれたコマンド（例: `$(rm -rf /)`）が実行され、システムファイルの破壊や機密情報の漏洩につながる最悪のケースがある。
ただし、Claude Code の Bash ツールは sandbox 実行を前提としており、実際の被害範囲は限定的となる構造的保護が存在する。

**攻撃シナリオ（T2.2 heredocインジェクション）**:
1. ユーザーが commit フェーズで以下のコマンドを実行する状況を想定する内容となる:
   ```bash
   git commit -m "$(cat <<'EOF'
   Message
   $(malicious command)
   EOF
   )"
   ```
2. heredoc パーサーが内部のコマンド置換を評価し、悪意のあるコマンドが実行される可能性が理論上は存在する背景がある
3. シングルクォート heredoc（<<'EOF'）の場合は変数展開が無効化されるが、ダブルクォート heredoc（<<"EOF"）は危険な動作となる

**T2.2対策（heredocインジェクション防止）**:
- heredoc を使用する git commit の形式は、シングルクォート heredoc（cat <<'EOF'）のみを許可する厳格なルールを設定すること
- ダブルクォート heredoc（<<"EOF"）やクォートなし heredoc（<<EOF）は明示的にブロックする判定ロジックを追加すること
- 正規表現で heredoc の開始パターンを検証: `/cat\s+<<'EOF'`（シングルクォート必須）
- analyzeBashCommand() で heredoc 内のコマンド置換（`$()`、`` ` ` ``）の存在を検出し、警告を発する実装を検討すること
- ドキュメント（CLAUDE.md）に、heredoc を使用する際の安全なパターンを明記する対応を行うこと

**残存リスク**: Low - シングルクォート heredoc の強制により、コマンドインジェクションのリスクは実質的に無効化される水準となる。

### T2.3: git push の認証情報露出 (Low Risk)

**脅威**: git push 実行時に、リモートリポジトリの URL に認証情報（パスワード、トークン）が含まれている場合、ログやエラーメッセージで露出する可能性がある。
ただし、現代的な git 運用では認証情報は credential helper で管理されるため、実際のリスクは限定的と評価される状況である。

**影響**: ログファイルやコンソール出力に認証情報が平文で記録され、アクセス権を持つ他のユーザーに漏洩する懸念が存在する。
特に CI/CD 環境や共有サーバーでの実行時には、ログの適切な管理が重要な対策ポイントとなる構造である。

**T2.3対策（認証情報保護）**:
- git push の実行前に、リモート URL に認証情報が含まれていないか検証する事前チェックを実装する方針を検討すること
- 正規表現で `https://username:password@` パターンを検出し、警告を発する機能を追加する対応を視野に入れること
- ドキュメントで、credential helper（credential.helper=store、cache、manager-core）の使用を推奨する記述を追加すること

**残存リスク**: Low - 現代的な git 運用慣習では認証情報の URL 埋め込みは稀であり、実際の被害リスクは極めて低い水準である。

### T2.4: git tag の悪用によるバージョン体系の破壊 (Low Risk)

**脅威**: commit フェーズで git tag が許可されることにより、不適切なタグ（既存タグの上書き、意味不明なタグ名）が作成される可能性がある。
これはバージョン管理の混乱を招き、リリースプロセスに悪影響を与える潜在的なリスクを含む状況となる。

**影響**: 既存の v1.0.0 タグが誤って上書きされた場合、過去のリリースポイントが失われる事態が発生する懸念がある。
また、無秩序なタグが大量に作成されると、git tag 一覧が汚染され、正規のタグを見つけにくくなる運用上の問題を引き起こす。

**T2.4対策（タグ管理の制御）**:
- git tag の実行を push フェーズのみに制限し、commit フェーズでは許可しない厳格なルールを再検討する余地がある
- または、タグ名のフォーマット検証（例: v[0-9]+\.[0-9]+\.[0-9]+）を実装し、セマンティックバージョニング準拠を強制する仕組みを導入すること
- 既存タグの上書きを防ぐため、`git tag --force` を明示的にブロックする条件を追加する対策を施すこと

**残存リスク**: Low - タグの誤操作は修正可能（git tag -d で削除、git push origin :refs/tags/tagname でリモート削除）であり、致命的な影響は限定的である。

## B-3: testingフェーズでのベースライン記録許可の脅威分析

### B-3ベースライン記録変更の概要

testing フェーズでの testBaseline 記録許可は、リグレッション判定の信頼性に影響を与える中リスクな変更内容である。
不適切なタイミングでのベースライン取得は、バグを含む状態を正常として記録してしまう危険性を内包する構造となっている。

### T3.1: 不正確なベースラインによるリグレッション判定の無効化 (Medium Risk)

**脅威**: testing フェーズは実装直後の段階であり、バグが残存している可能性が高い時期での ベースライン記録となる問題がある。
この時点で取得したベースラインは、本来検出すべきバグを「正常な動作」として記録してしまうリスクを持つ内容となる。

**影響**: regression_test フェーズで、既存のバグが引き続き存在していても「ベースラインと一致」として判定されてしまう誤検知が発生する。
結果として、リグレッションテストの目的（既存機能の保護）が達成されず、品質保証の仕組みが機能不全に陥る深刻な事態につながる。

**攻撃シナリオ（T3.1不正確ベースライン）**:
1. ユーザーが implementation フェーズで 10 件のテストを実装し、8 件が成功、2 件が失敗する状況を作り出す場合を想定する
2. testing フェーズで失敗した 2 件を修正する前に、誤って workflow_capture_baseline を実行してしまう誤操作を行う状況が発生する
3. 失敗している 2 件のテストを含むベースライン（passed: 8, failed: 2）が記録されてしまう不適切な状態が成立する
4. regression_test フェーズで、2 件のテストが引き続き失敗していても「ベースラインと一致」として誤判定される結果となる

**T3.1対策（ベースライン精度確保）**:
- captureBaseline() に、testing フェーズでの記録時には強制的な確認プロンプトを表示する機能を追加する実装を検討すること
- 確認メッセージ例: "WARNING: testingフェーズでのベースライン記録は推奨されません。全てのテストが成功していることを確認してください。続行しますか？ (yes/no)"
- ベースライン記録時に failedTests が 1 件でも存在する場合、エラーを返却してベースライン記録を拒否する厳格な検証を実装すること
- testBaseline にメタデータ（recordedAt: "2026-02-09T12:00:00Z", phase: "testing", warningGiven: true）を追加し、後続の分析を可能にする設計とすること
- CLAUDE.md に、「ベースライン記録は research フェーズで、全テストが成功した状態で行うことを強く推奨」する明確な記述を追加すること

**残存リスク**: Low - 確認プロンプトと failedTests 検証により、誤ったベースライン記録の大部分は防止可能と評価される。

### T3.2: ベースラインの複数回上書きによる追跡可能性の喪失 (Low Risk)

**脅威**: testing フェーズでのベースライン記録が許可されることにより、research フェーズで記録した初期ベースラインが上書きされる可能性がある。
上書き前のベースラインは失われるため、問題発生時のトラブルシューティングが困難になる運用上の懸念が存在する構造である。

**影響**: 「いつ、どのフェーズで、どのようなベースラインが記録されたか」の履歴が失われ、リグレッション判定の信頼性を事後検証できない状態となる。
デバッグ時に「ベースラインの取得タイミングが不適切だった」可能性を排除できず、問題の切り分けが困難になる運用上の課題を生む。

**T3.2対策（履歴追跡性の確保）**:
- testBaseline の構造を拡張し、履歴配列（history: [ { recordedAt, phase, totalTests, passedTests, failedTests } ]）を保持する設計を採用すること
- 新規ベースライン記録時には、既存のベースラインを history に追加してから上書きする動作を実装する方針とすること
- workflow_get_test_info ツールで履歴を含むベースライン情報を返却し、ユーザーが履歴を確認できる UI を提供すること
- ベースライン履歴は最大 5 件まで保持し、古いものから自動削除する LRU 方式を採用して状態ファイルの肥大化を防ぐ配慮を施すこと

**残存リスク**: Low - 履歴機能の実装により、ベースラインの追跡可能性は十分に確保される水準となる。

### T3.3: testingフェーズでのベースライン記録忘れによる運用混乱 (Low Risk)

**脅威**: testing フェーズでのベースライン記録が「可能」になることで、「research フェーズで記録しなくてもよい」という誤解が広がる可能性がある。
結果として、research フェーズでのベースライン記録が習慣化されず、testing フェーズでも記録を忘れる二重の失敗が発生する懸念となる。

**影響**: ベースライン未設定のまま regression_test フェーズに到達し、リグレッション判定がスキップされる事態が頻発する運用上の問題を引き起こす。
ワークフロー全体の品質保証機能が形骸化し、バグの早期発見機会を失う結果につながる潜在的なリスクを含む状況となる。

**T3.3対策（記録忘れの防止）**:
- testing フェーズ開始時に、testBaseline が未設定の場合は自動的に警告メッセージを表示する機能を実装する対応を行うこと
- 警告メッセージ例: "testBaselineが未設定です。testingフェーズでテスト成功後、必ずworkflow_capture_baselineを実行してください。"
- workflow_next（testing → regression_test）時に、testBaseline が未設定の場合は確認プロンプトを表示する追加の防御層を設ける実装を検討すること
- CLAUDE.md の「AIへの厳命」セクションに、「researchフェーズでのベースライン記録を優先し、testingフェーズは補完的な手段として位置づける」明確な指針を追記すること

**残存リスク**: Low - 警告メッセージと確認プロンプトにより、ベースライン記録忘れの大部分は防止可能と評価される。

## B-4: MCPサーバー再起動に関するドキュメント化の脅威分析

### B-4ドキュメント化の概要

B-4 は純粋なドキュメント追記であり、コード変更を伴わないため、技術的な脅威は最小限となる内容である。
主なリスクは、ドキュメント記述の不正確性や不完全性により、開発者に誤った理解を与える情報伝達上の問題に限定される。

### T4.1: 不正確なドキュメントによる誤った理解の拡散 (Low Risk)

**脅威**: MCPサーバーのキャッシュ機構に関する記述が不正確または不完全な場合、開発者が誤った前提で開発を進める可能性がある。
例えば、「特定のファイルのみ再起動が必要」という誤解が広がると、デバッグ時に不要な時間を浪費する運用上の混乱を招く懸念である。

**影響**: 開発効率の低下、デバッグ時間の増加、誤った回避策（無意味なコード変更）の実施といった、間接的な生産性への悪影響が発生する。
ただし、システムの安定性やセキュリティに直接的な影響はなく、影響範囲は開発チーム内に限定される構造となっている。

**T4.1対策（ドキュメント正確性）**:
- ドキュメント記述は、技術的に正確であることを最優先とし、曖昧な表現を避ける明確な文言を採用すること
- 記述内容の根拠（Node.js の require() キャッシュ機構）を明示し、理解を深める背景情報を提供する構成とすること
- 再起動手順は、具体的なステップ（Claude Code の終了方法、再起動後の確認方法）を含む実践的な内容とすること
- 記述完了後、プロジェクトメンバー複数名によるレビューを実施し、理解可能性と正確性を検証する手順を踏むこと

**残存リスク**: Negligible - ドキュメントの誤りは後から容易に修正可能であり、致命的な影響はない水準である。

### T4.2: ドキュメントの配置場所による視認性の問題 (Low Risk)

**脅威**: CLAUDE.md は既に大規模（約 1000 行超）であり、追記した内容が埋もれて発見されない可能性がある視認性上の懸念である。
特に新規参加者や、CLAUDE.md を熟読しない開発者には、重要な情報が伝わらないリスクが存在する状況となる。

**影響**: MCPサーバーのキャッシュに関する注意事項を見逃した開発者が、コード変更が反映されない現象に遭遇し、長時間のデバッグを強いられる運用上の非効率を招く。
頻繁に同じ質問（「なぜ変更が反映されないのか」）が発生し、チーム全体のサポートコストが増加する潜在的な問題を含む。

**T4.2対策（情報の可視性向上）**:
- CLAUDE.md 内に目立つセクション（「重要な制約事項」「開発者向け注意事項」等）を新設し、そこに配置する構成を採用すること
- セクション冒頭に「⚠️ 重要」「🔴 必読」等の視覚的なマーカーを追加し、注意を引く工夫を施すこと
- README.md にも簡潔な注意事項（「workflow-plugin のコード変更後は Claude Code の再起動が必要」）を追記し、二重の周知を行うこと
- プロジェクトの onboarding ドキュメントやセットアップガイドにも、同様の注意事項を含める網羅的な配慮を行うこと

**残存リスク**: Low - 複数箇所への記載により、情報が見逃される確率は大幅に低減される水準となる。

## 横断的脅威

本セクションでは、複数の修正に共通して存在する横断的な脅威を分析する内容である。

### T5.1: HMAC整合性チェックとの干渉 (Medium Risk)

**脅威**: workflow-state.json の変更を伴う修正（B-3 のベースライン記録）が、HMAC 整合性検証と干渉する可能性がある問題である。
stateManager を経由せずに直接ファイルを編集した場合、HMAC が無効化され、フックシステムがエラーを発する事態が想定される。

**影響**: ワークフロー全体が停止し、ユーザーが作業を継続できなくなる致命的な動作不良が発生するリスクがある。
復旧には .claude/state/workflows/ 配下の状態ファイルの手動修正または削除が必要となり、作業の巻き戻しを伴う深刻な事態につながる。

**T5.1対策（HMAC整合性維持）**:
- test-tracking.ts の captureBaseline() は必ず stateManager.updateTaskState() を経由して状態を更新する実装を厳守すること
- 直接的な fs.writeFileSync() によるファイル書き込みは絶対に行わず、stateManager の API のみを使用する規約を徹底すること
- 統合テストで、captureBaseline() 実行後の workflow-state.json の HMAC が有効であることを検証するケースを追加すること
- phase-edit-guard.js が HMAC エラーを検出した場合、より詳細なエラーメッセージ（どの操作が原因か）を表示する改善を施すこと

**残存リスク**: Low - stateManager の API 使用を徹底することで、HMAC 整合性は確実に維持される水準となる。

### T5.2: テストカバレッジの不足による潜在バグの残存 (Medium Risk)

**脅威**: 4 件の修正に対して、新規追加される単体テストおよび統合テストが不十分な場合、潜在的なバグが本番環境に到達する可能性がある。
特に、エッジケースや異常系のテストが不足していると、予期しない入力や状態でシステムが破綻するリスクが存在する構造となる。

**影響**: 本番環境でのワークフロー実行中にエラーが発生し、ユーザー作業が中断される事態が頻発する運用上の問題を引き起こす。
最悪の場合、データ破損や状態の不整合が発生し、復旧に長時間を要する深刻な障害につながる潜在的なリスクを含む。

**T5.2対策（テストカバレッジ確保）**:
- 各修正に対して、以下の観点を網羅するテストスイートを設計する方針を採用すること:
  - 正常系: 意図した動作が正しく実行されるケースの検証を行う
  - 異常系: エラー入力、不正な状態に対する適切なエラーハンドリングの確認を実施する
  - 境界値: 空配列、最大長文字列、タイムアウト等の限界条件での動作を確認する
  - 並行性: 複数操作が同時実行された場合の整合性維持を検証する（該当する場合）
- コードカバレッジツール（vitest --coverage）を実行し、新規追加コードのカバレッジが 90% 以上であることを確認する基準を設定すること
- 統合テストで、4 件の修正を組み合わせた複合シナリオ（例: タスクソート後に commit フェーズで git 操作を実行）を検証するケースを用意すること

**残存リスク**: Medium - 包括的なテストでリスクは大幅に低減されるが、全てのエッジケースを事前に想定することは困難であり、一定の残存リスクは避けられない状況である。

### T5.3: ドキュメントとコードの乖離 (Low Risk)

**脅威**: コード修正後にドキュメント（CLAUDE.md、README.md 等）の更新を忘れた場合、実装と記述が不一致となる情報整合性の問題が発生する。
特に B-2 の git 操作許可に関する記述が、実際の実装と異なる場合、ユーザーの誤解を招く重大な情報品質上の欠陥となる。

**影響**: ユーザーが誤った前提で操作を行い、予期しないエラーに遭遇する運用上の混乱が発生する懸念がある。
また、トラブルシューティング時にドキュメントを参照しても解決できず、サポートコストが増加する間接的な悪影響を及ぼす。

**T5.3対策（ドキュメント同期）**:
- code_review フェーズで、ドキュメント更新の完全性を必ずチェックリストに含める運用規約を設定すること
- 特に以下のドキュメントとコードの整合性を重点的に確認する手順を踏むこと:
  - phase-rules.js の編集可能ファイルタイプ ⇔ CLAUDE.md の「フェーズごとの編集可能ファイル」表の対応確認
  - analyzeBashCommand() の git 操作ホワイトリスト ⇔ CLAUDE.md の「必須コマンド」「AIへの厳命」の記述の一致確認
  - test-tracking.ts の許可フェーズ ⇔ CLAUDE.md の「フェーズ順序」の記述の整合性確認
- 自動化可能な範囲（例: phase-rules.js からマークダウン表を生成）は、スクリプトによる同期を検討する将来的な改善を視野に入れること

**残存リスク**: Low - code_review での明示的なチェックにより、ドキュメントとコードの乖離は早期に検出可能な水準となる。

## セキュリティ原則の適用

本修正全体に適用すべきセキュリティ設計原則を定義する内容となる構成である。

### 最小権限の原則 (Principle of Least Privilege)

commit/push フェーズで許可する git 操作は、ワークフロー完走に必要な最小限の範囲に厳格に限定する方針を堅持すること。
「念のため他の操作も許可する」といった安易な権限拡大は行わず、必要性が明確になった時点で段階的に追加する慎重な姿勢を維持する。

### 多層防御 (Defense in Depth)

単一の防御メカニズム（ホワイトリストのみ）に依存せず、複数の検証層を組み合わせる多層防御戦略を採用すること。
例: git 操作チェック = コマンドホワイトリスト + オプション検証 + heredoc パターン検証 + 正規表現検査の 4 層構造を実装する設計とする。

### セキュアバイデフォルト (Secure by Default)

新規機能（testing フェーズでのベースライン記録）は、デフォルトで最も安全な動作（警告表示、確認プロンプト）を行う設計を基本とすること。
ユーザーが明示的に危険性を理解した上で操作を続行する opt-in 方式を採用し、意図しない誤操作を防ぐ配慮を施す構造とする。

### 失敗時の安全な挙動 (Fail-Safe)

エラー発生時や検証失敗時には、操作を拒否する方向に倒れる安全側の動作を実装する fail-safe 設計を徹底すること。
「判定が曖昧な場合は許可」ではなく「判定が曖昧な場合はブロック」という保守的な方針を採用する姿勢を貫く。

### 監査可能性 (Auditability)

全ての重要な操作（git 操作の実行、ベースラインの記録、フェーズ遷移）は、詳細なログを残す実装を行うこと。
ログには、タイムスタンプ、フェーズ、操作内容、実行結果を含め、事後のトラブルシューティングを容易にする情報設計とする。

## 次フェーズへの引き継ぎ事項

planning フェーズで詳細化すべき設計上の検討事項を列挙する内容となる構成である。

### git操作ホワイトリストの詳細設計

- 許可する git 操作の完全なリスト（コマンド + オプションの組み合わせ）を網羅的に定義する作業を実施すること
- 正規表現パターンの具体的な実装を決定し、エッジケースを全て列挙してテスト仕様に含める手順を踏むこと
- heredoc パターンの検証ロジック（シングルクォート判定、コマンド置換検出）の詳細アルゴリズムを設計すること

### testBaselineメタデータ構造の設計

- testBaseline に追加するメタデータフィールド（recordedAt、phase、warningGiven、history 配列）の詳細なスキーマを定義する作業を行うこと
- 履歴の保存件数上限（推奨: 5 件）と LRU 削除アルゴリズムの実装方式を決定する手順を実施すること
- 既存タスクとの互換性（メタデータなしの古い testBaseline）の扱いを明確化する対応を施すこと

### 確認プロンプトとエラーメッセージの文言設計

- testing フェーズでのベースライン記録時の確認プロンプトの具体的な文言を決定する作業を行うこと
- git 操作ブロック時のエラーメッセージで、ユーザーが次に取るべきアクション（正しいコマンドの例示）を含める設計とすること
- エラーメッセージの多言語対応（日本語、英語）の必要性を検討し、対応方針を決定する手順を踏むこと

### テスト戦略の詳細化

- 単体テスト、統合テスト、E2E テストの範囲と責務を明確に分離する設計を行うこと
- 各修正に対するテストケースの具体的な一覧（正常系・異常系・境界値）を作成する作業を実施すること
- テストデータ（モックタスク、テスト用リポジトリ）の準備方法と管理方針を決定する手順を踏むこと

以上が、4件のワークフロープロセス修正に対する包括的な脅威モデル分析の成果物となる内容である。
