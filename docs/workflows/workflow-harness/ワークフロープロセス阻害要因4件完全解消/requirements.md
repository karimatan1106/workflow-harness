# requirements.md - ワークフロープロセス阻害要因4件完全解消

## サマリー

本タスクの目的は、ワークフロー全19フェーズの完走を阻害する4件の構造的問題を完全に解消することである。
調査の結果、全て再現可能な問題であり、コード修正またはドキュメント化によって解決可能であることが判明した。
4件の阻害要因のうちB-1からB-3はコード修正を必要とし、B-4のみドキュメント対応となる見込みである。

B-1「複数タスク存在時の非決定的タスク選択」は、タスク一覧の返却順序がファイルシステム依存になっている問題を解消する。
discover-tasks.jsに明示的なソートロジック（taskId降順）を追加し、常に最新タスクが優先されることを保証する設計とする。

B-2「commit/pushフェーズでgit操作が全てブロック」は、phase-rules.jsのフェーズ定義とBashコマンド解析の両方を修正対象とする。
commitフェーズとpushフェーズに新しいファイルタイプ'git'を追加し、必要最小限のgit操作のみを許可する仕組みを導入する計画である。

B-3「testBaselineの記録がresearchフェーズに限定」は、フェーズ制限を緩和してtestingフェーズでもベースライン記録を可能にする。
regression_testへの遷移時にベースライン未設定の場合でも、testingフェーズで取得できるようにすることで運用性を改善する。

B-4「MCPサーバーのモジュールキャッシュ」は、Node.jsのrequire()機構に起因する既知の制約である。
コード修正では根本解決できないため、開発者向けドキュメントに再起動の必要性を明記する方針を取る。

次フェーズでは、各ブロッカーの正確なコード修正範囲と具体的な実装方法を仕様書として詳細化する必要がある。
テスト戦略としては、各修正に対して単体テストと統合テストの両方を用意し、リグレッションを防ぐ体制を整える計画である。

## 機能要件

### BR-1: 決定的タスク選択の実装

**目的**: 複数のアクティブタスクが存在する場合でも、常に一貫した基準でタスクが選択されることを保証する仕組みを提供する。

**要件**:
1. discover-tasks.jsのdiscoverTasks()関数がタスク一覧をtaskId降順で明示的にソートして返却すること
2. taskIdは生成時刻のタイムスタンプ部分を含むため、降順ソートによって最新タスクが先頭に配置される設計となること
3. ソートロジックはfs.readdirSync()の呼び出し直後に適用され、他の処理への影響を最小限に抑えること
4. phase-edit-guard.jsのfindActiveWorkflowTask()は、ソート済み配列の先頭要素を取得することで最新タスクを確実に選択できること

**副作用**: 既存の動作に依存しているコードがある場合、タスク選択の順序変更によって影響を受ける可能性がある点に注意する。

### BR-2: commit/pushフェーズでのgit操作許可

**目的**: commitフェーズとpushフェーズで必要最小限のgit操作を可能にし、ワークフロー完走を妨げている制約を解除する。

**要件**:
1. phase-rules.jsにおいて、commitフェーズの編集可能ファイルタイプとして新規タイプ'git'を追加定義すること
2. phase-rules.jsにおいて、pushフェーズの編集可能ファイルタイプとしても新規タイプ'git'を追加定義すること
3. phase-edit-guard.jsのanalyzeBashCommand()に、commit/pushフェーズ専用のホワイトリストチェックロジックを導入すること
4. ホワイトリストには以下のgit操作のみを含めること: git add、git commit（heredoc形式含む）、git push、git tag
5. git commit実行時の--amendや--no-verify等のオプションは、明示的に許可しない限りブロックされる設計とすること
6. git pushの--forceや--force-with-lease等の破壊的オプションは引き続きブロック対象として維持すること
7. ホワイトリスト外のgit操作（git reset、git checkout等）は従来通りブロックされる挙動を保持すること

**セキュリティ考慮**: 許可するgit操作は最小限に限定し、破壊的コマンドは引き続きブロックする設計方針を堅持する。

### BR-3: testingフェーズでのベースライン記録許可

**目的**: testBaselineの記録タイミングを柔軟化し、researchフェーズでの記録忘れによるブロック状況を回避可能にする。

**要件**:
1. test-tracking.tsのcaptureBaseline()において、許可フェーズをresearchとtestingの2つに拡張すること
2. researchフェーズでの記録は「初期ベースライン」として扱い、既存の動作を変更しないこと
3. testingフェーズでの記録は「遅延ベースライン」として扱い、ログに明示的な警告メッセージを出力すること
4. testBaselineが未設定の場合でも、testingフェーズからregression_testへの遷移を妨げない設計とすること
5. regression_testフェーズ開始時にtestBaselineが未設定の場合、リグレッション判定をスキップして警告を表示すること
6. captureBaseline()の戻り値に、記録されたフェーズ情報（research/testing）を含めること

**推奨運用**: 可能な限りresearchフェーズでのベースライン記録を推奨する旨を、ユーザー向けガイドに明記する方針とする。

### BR-4: MCPサーバー再起動の必要性に関するドキュメント化

**目的**: Node.jsのモジュールキャッシュによってコード変更が即座に反映されない動作を開発者に周知する。

**要件**:
1. CLAUDE.mdの適切なセクション（「AIへの厳命」や新規セクション）に、MCPサーバーのモジュールキャッシュに関する注意事項を追記すること
2. workflow-plugin配下のコード（dist/*.js）を変更した場合、MCPサーバーの再起動が必須であることを明記すること
3. MCPサーバー再起動の具体的な手順（Claude Codeの再起動方法）をドキュメントに含めること
4. 開発時のベストプラクティスとして、コード変更後の動作確認前には必ず再起動を行うべきことを強調すること
5. Memory.mdの「Key Learnings」セクションにも、同様の注意事項を追記すること（既存の記述がある場合は補完）

**対象読者**: プラグイン開発者およびワークフロープロセスをカスタマイズする高度なユーザーを想定した記述とする。

## 非機能要件

### NFR-1: パフォーマンスへの影響最小化

**要件**: タスクソートロジックの追加によるオーバーヘッドは、通常のタスク数（1〜10件程度）において無視できる範囲（1ms未満）に抑えること。

### NFR-2: 後方互換性の維持

**要件**: 既存のワークフローや成果物ファイルに影響を与えず、アップデート後も継続して利用可能であることを保証すること。

### NFR-3: コードの可読性と保守性

**要件**: 追加するロジックはコメントで明確に意図を説明し、将来のメンテナンス担当者が理解しやすい形で実装すること。

### NFR-4: テストカバレッジの維持

**要件**: 既存のテスト（732件）を全て維持し、新規追加ロジックに対しても適切な単体テストを追加することで全体のテストカバレッジを低下させないこと。

### NFR-5: ログとエラーメッセージの充実

**要件**: testingフェーズでのベースライン記録時やgit操作許可時には、明確なログメッセージを出力し、トラブルシューティングを容易にすること。

## 受け入れ基準

### AC-1: タスク選択の決定性検証

**基準**: discover-tasks.jsの単体テストにおいて、複数タスクが存在する状態でdiscoverTasks()を呼び出した場合、常にtaskIdが最大のタスクが配列の先頭に配置されることを確認できること。

**テスト方法**:
- taskIdの異なる3つ以上のテストタスクを用意する実装を行う
- discoverTasks()を10回連続で呼び出し、全ての呼び出しで同一の順序が返却されることを検証する仕組みを作る
- 返却された配列の先頭要素が、taskIdの降順ソート結果の先頭と一致することをアサートする

### AC-2: commitフェーズでのgit操作実行可能性検証

**基準**: commitフェーズでgit add、git commit（heredoc形式含む）、git tagが実行可能であり、それ以外のgit操作（git resetなど）は引き続きブロックされることを確認できること。

**テスト方法**:
- モックワークフロー環境でタスクをcommitフェーズに設定する手順を準備する
- 許可対象のgit操作（git add .、git commit -m "test"、git tag v1.0.0）を実行し、エラーが発生しないことを検証する
- ブロック対象のgit操作（git reset --hard、git checkout -b new-branch）を実行し、適切なエラーメッセージが返却されることを確認する

### AC-3: pushフェーズでのgit操作実行可能性検証

**基準**: pushフェーズでgit pushが実行可能であり、破壊的オプション（--force等）は引き続きブロックされることを確認できること。

**テスト方法**:
- モックワークフロー環境でタスクをpushフェーズに設定する環境を用意する
- git pushを実行し、エラーが発生しないことを検証する手順を実施する
- git push --forceを実行し、適切なブロックメッセージが表示されることを確認する試験を行う

### AC-4: testingフェーズでのベースライン記録可能性検証

**基準**: testingフェーズでworkflow_capture_baselineツールを呼び出した場合、成功レスポンスが返却され、testBaselineが正しく記録されることを確認できること。

**テスト方法**:
- モックワークフロー環境でタスクをtestingフェーズに設定する準備を行う
- workflow_capture_baselineを呼び出し、success: trueが返却されることを検証する仕組みを実装する
- タスク状態ファイルを読み込み、testBaselineフィールドにデータが記録されていることをアサートする

### AC-5: regression_testフェーズへの遷移可能性検証

**基準**: testingフェーズでtestBaselineを記録した後、regression_testフェーズへの遷移がブロックされないことを確認できること。

**テスト方法**:
- testingフェーズでベースライン記録を行ったタスクを作成する手順を準備する
- workflow_nextを呼び出してregression_testへ遷移し、エラーが発生しないことを確認する検証を実施する

### AC-6: 既存テストスイートの全件パス

**基準**: コード修正後も、既存の732件のテストが全て成功し、リグレッションが発生していないことを確認できること。

**テスト方法**:
- vitestを実行し、全テストケースが成功することを検証する試験を行う
- テスト結果レポートで失敗件数が0件であることを確認する手順を実施する

### AC-7: ドキュメントの完全性確認

**基準**: CLAUDE.mdおよびMEMORY.mdに、MCPサーバーのモジュールキャッシュと再起動の必要性に関する記述が追加され、内容が明確であることを確認できること。

**テスト方法**:
- 更新されたCLAUDE.mdを第三者がレビューし、記述内容が理解可能であることを確認する手順を行う
- 記述されている再起動手順に従って実際にMCPサーバーを再起動し、手順の正確性を検証する試験を実施する

## 制約事項

### 技術的制約
- Node.jsのrequire()キャッシュ機構は言語仕様であり、MCPサーバー側のコード修正では根本的に回避できない構造になっている点を受け入れる
- phase-edit-guard.jsは約2000行の複雑なフックシステムであり、新規ロジック追加時には既存の条件分岐への影響を慎重に検証する必要がある
- Bashコマンド解析はセキュリティ上重要な役割を持つため、ホワイトリストの設計は厳格に行い、予期しない操作の許可を防ぐ設計とする

### 運用的制約
- commit/pushフェーズでのgit操作許可は、ユーザーが誤った操作（破壊的オプション等）を試みる可能性を生じさせる点を考慮する必要がある
- testingフェーズでのベースライン記録は「遅延対応」であり、理想的にはresearchフェーズでの記録を推奨する運用方針を維持すべきである
- 複数タスクが同時にアクティブな状態は、ワークフロー設計上想定されていない異常状態である可能性があり、根本原因の調査も並行して行うべきである

### ドキュメント制約
- CLAUDE.mdは現時点で既に大規模（約1000行超）であり、新規追記する場合は適切なセクション構成を検討する必要がある
- Memory.mdはユーザー個人の設定ファイルであるため、プロジェクトのコミット対象とせず、READMEなど別の手段での周知も検討すべきである

## 実装上の注意点

### ソート実装時の注意
taskIdのフォーマットはYYYYMMDD_HHmmss形式であり、文字列の辞書順ソートで時系列順序が保証される構造になっている点を活用する。
数値変換やDate型への変換は不要であり、シンプルな文字列比較でソート可能である特性を利用すべきである。

### git操作ホワイトリストの設計
正規表現や文字列パターンマッチングを用いて、意図しない操作の許可を防ぐ厳格なチェックを実装する必要がある。
特にgit commitの--amendオプションやgit pushの--forceオプションは、明示的にブロック条件を追加することを検討すべきである。

### testBaselineのデータ構造
既存のtestBaselineフィールドに加えて、記録されたフェーズ情報（phase: 'research' | 'testing'）を保持する拡張を検討する価値がある。
これにより、リグレッション判定時にベースラインの信頼性を評価できる仕組みを将来的に導入可能になる。

### エラーメッセージの一貫性
新規追加するエラーメッセージや警告メッセージは、既存のメッセージスタイルと一貫性を保つ形で記述する必要がある。
ユーザーが理解しやすく、トラブルシューティングに役立つ情報を含めることを優先する方針で文言を決定すべきである。
