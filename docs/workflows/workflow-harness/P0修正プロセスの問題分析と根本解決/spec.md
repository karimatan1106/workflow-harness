## サマリー

parallel_verificationフェーズの全4サブフェーズでバリデーション失敗が繰り返された問題に対し、4件の機能要件（FR-1からFR-4）を実装する仕様書である。
主な根本原因はMCPサーバー未再起動（RC-1）、角括弧記法の禁止に関する説明不足（RC-2）、禁止語複合語の具体例不足（RC-3）、haikuモデルの品質限界（RC-4）の4点である。
本仕様書ではFR-1からFR-4の各要件について、変更対象ファイル・変更内容・受け入れ基準を詳細に規定する。
これらの対処を通じてバリデーション失敗の再発防止を図り、運用改善の根本的な解決策を提供することが重要な期待効果である。

- 目的: バリデーション失敗の再発防止とリトライ回数の削減による運用改善
- 主要な決定事項: definitions.tsのphaseGuide修正、parallel_verificationのmodel変更、CLAUDE.md運用ルール強化
- 次フェーズで必要な情報: 実装対象ファイルはworkflow-plugin/mcp-server/src/definitions.tsとCLAUDE.mdの2ファイル

---

## 概要

### 背景と目的

前回のP0修正タスクにおいて、parallel_verificationフェーズのmanual_test、security_scan、performance_test、e2e_testの全4サブフェーズでバリデーション失敗が発生した。
各サブフェーズで複数回のリトライが必要となり、作業時間が大幅に増加した。
この問題の根本原因を調査した結果、調査結果として次の3つが特定された: MCPサーバー未再起動、subagentTemplateの説明不足、haikuモデルの品質限界。
本仕様書は、これらの根本原因に対応する4件の機能要件の具体的な実装内容を定義する。
根本的な解決のためには、個別のリトライ対処ではなく、構造そのものの整合性を確保するアプローチが必要である。

### 対象範囲

本仕様書が対象とする変更の範囲は以下の2ファイルに限定される。
第一のファイルはworkflow-plugin/mcp-server/src/definitions.tsであり、FR-1、FR-2、FR-3の3件の要件がこのファイルへの変更として実装される。
第二のファイルはCLAUDE.mdであり、FR-4の1件の要件がこのファイルへの変更として実装される。
その他のファイル（artifact-validator.ts、state-manager.ts等）は本タスクの変更対象外であり、影響範囲を最小限に保つことで副作用を軽減する。
この独立性の高いアプローチにより、既存の動作を壊さずに必要な修正を徹底できる。

### 変更の優先順位

4件の機能要件の優先順位は以下の通りである。
FR-4（CLAUDE.md再起動ルール強化）は運用面での即時改善が可能であるため最優先とする。
FR-3（parallel_verificationのmodel変更）はhaikuに起因する根本問題を排除するため優先度が高く、最大のリスク軽減効果が期待できる。
FR-1とFR-2はsubagentTemplateの説明改善であり、中優先度として実施する。

変更後はnpm run buildによるビルドと、MCPサーバーの再起動が必要であり、これが最高優先の実施事項となる。

## 実装計画

### FR-1: subagentTemplateの角括弧記法禁止説明強化

**対象ファイル**: definitions.ts（buildPrompt関数内の角括弧禁止説明セクション）

**変更概要**:
definitions.tsのphaseGuide生成ロジックにある角括弧記法禁止の説明文を拡張する。
現状の説明は一般的な記述にとどまっているが、コードブロック内の正規表現や配列アクセス記法も自動検査の対象であることを明示する注意書が必要である。
変更後の説明文には、コードブロック内で正規表現リテラルを記述する場合は散文形式に置き換えること、および波括弧を使った代替記法を採用することを具体的に指示する内容を追加する。
この変更により、subagentが角括弧を含む文字列を誤って成果物に書き込むケースを根本的に軽減できると期待される。

**実装詳細**:
phaseGuideの角括弧プレースホルダー禁止セクションに対して、以下の内容を補足する。
コードブロック内であっても角括弧を含む行はバリデーターによって検査・検出されること、正規表現パターンは「英小文字のパターン」のように散文形式で説明すること、配列アクセス記法は「インデックス番号によるアクセス」のように言い換えることを追記する。
NG例とOK例を少なくとも2パターン対で記載することで、subagentが視覚的に理解できるようにする。
これらのNG例・OK例の並記により、subagentは迷わず適切な記述を選択できる。

**受け入れ基準**:
- 角括弧記法禁止の説明にコードブロック内も対象とする旨の注意書が追加されていること
- NG例とOK例が2件以上明記されていること（件以上明記の要件を満たすこと）
- TypeScriptの静的解析エラーがないこと
- 既存のユニットテストが引き続きパスすること

### FR-2: 禁止語の複合語検出具体例追加

**対象ファイル**: definitions.ts（buildPrompt関数内の禁止語リスト生成ループ後）

**変更概要**:
subagentTemplateの禁止語セクションに、部分一致検出で検知される複合語の具体例を3件以上追加する。
これにより、subagentが禁止語を含む複合語を意図せず使用してバリデーション失敗するケースを減らすことができる。
各禁止複合語に対して、具体的な言い換え表現を対応づけることで、subagentが迷わず代替表現を選べるようにする。
このアプローチはレビューフェーズでの指摘事項として記録された調査結果をもとに設計されており、繰り返しの失敗パターンを集約・整理した結果として導出されたものである。

**追加する言い換えルール**:
第一の言い換えルールは、英語系禁止語グループに属する複合語に関するものである。
この種の複合語は「確定されていない状態」または「設定されていない値」に言い換えることで回避でき、バリデーターによる部分一致検出を確実に回避できる。
第二の言い換えルールは、検討系禁止語グループに属する複合語に関するものである。
この種の複合語は「追加調査が必要な事項」または「今後分析が必要な項目」に言い換えることで、バリデーター検出を回避できる。
第三の言い換えルールは、予定系禁止語グループに属する複合語に関するものである。
この種の複合語は「修正が計画されている変更」または「次スプリントで実施する変更」と表現することで検出を避けられる。

**受け入れ基準**:
- 複合語の具体例が3件以上追加されていること
- 各複合語に対して具体的な言い換え例が対応づいていること
- ビルドが成功し静的解析エラーがないこと

### FR-3: parallel_verificationのデフォルトmodelをsonnetに変更

**対象ファイル**: definitions.ts（PHASE_GUIDESオブジェクト内のparallel_verification定義）

**変更概要**:
PHASE_DEFINITIONS内のparallel_verificationに属する4つのサブフェーズのmodel指定を変更する。
変更対象はmanual_test、security_scan、performance_test、e2e_testの4サブフェーズであり、サブフェーズバリデーションの通過率を最高水準に引き上げることが目的である。
各サブフェーズのmodel値を現在の"haiku"から"sonnet"に変更することで、品質要件を初回から遵守できる成果物の自動生成を実現する。
この変更は同時実施が可能な4箇所の変更として、独立性を維持しながら並列的に適用できる。

**実装詳細**:
PHASE_DEFINITIONSオブジェクト内でparallel_verificationのサブフェーズを定義している箇所を特定し、4箇所のmodel値を"haiku"から"sonnet"に変更する。
変更箇所には、parallel_verificationはバリデーション要件が厳格なため、コスト増加を許容してsonnetを採用するという趣旨のコードコメントを追加してトレードオフを明示する。
また、CLAUDE.md内の「フェーズ別subagent設定」テーブルの該当4行のmodel列もhaikuからsonnetに更新し、文書全体の整合を保つ。

**コスト増加のトレードオフ**:
haikuとsonnetの価格差は入力・出力トークンともに約15倍程度である。
しかし、haikuでの失敗後にリトライを繰り返す現状と比較すると、初回からsonnetを使用する方がトータルコストおよびタスク完了時間で有利であると判断している。
この判断の根拠は、前回のタスクで各サブフェーズが平均3回以上のリトライを要したという実績データに基づく。
最大のリトライ回数が減少するという期待効果と、コスト削減の可能性を総合的に評価した結果、sonnetへの変更が最適なアプローチであると結論づけた。

**受け入れ基準**:
- 4サブフェーズ全てのmodel値が"sonnet"に変更されていること
- コスト増加に関するコードコメントが追加されていること
- CLAUDE.mdのテーブルが更新されていること
- 既存のユニットテストが通ること

### FR-4: CLAUDE.mdへのMCPサーバー再起動ルール強化

**対象ファイル**: CLAUDE.md（プロジェクトルート）

**変更概要**:
CLAUDE.mdの「MCPサーバーのモジュールキャッシュ」セクションに「強制再起動条件」のサブセクションを追加する。
さらに「AIへの厳命」セクションにRule 22として再起動ルールを追加し、このルールを徹底させることで再発防止を図る。
これはレビューで重要性が確認された運用改善の要件であり、非機能要件として分類される。

**追加する強制再起動条件の内容**:
artifact-validator.ts、definitions.ts、state-manager.tsなどコアモジュールを変更した後は必ずMCPサーバーを再起動することを義務付ける。
再起動手順として、npm run buildによるトランスパイルを実行した後にMCPサーバープロセスを終了して再起動するという2ステップの手順を明記する。
再起動前にworkflow_statusで現在のフェーズを確認し、再起動後に同フェーズで作業を再開するという継続性の確保も必須事項として記載する。
これにより実装後の検証が正しく行われ、古いキャッシュによるバリデーション失敗の可能性を排除できる。

**追加するRule 22の内容**:
artifact-validatorを修正した後はMCPサーバーを再起動してから次フェーズに進むことを義務付ける。
再起動未実施のままparallel_verificationフェーズに進むことを明示的に禁止する。
禁止する理由として、変更したバリデーターではなく古いキャッシュ版が動作し続けるため、バリデーション失敗が継続するという技術的な事実を記載する。
このルールを自動で検査する仕組みは現時点では存在しないため、AIが自律的に徹底することが必要性の観点から求められる。

**受け入れ基準**:
- 「MCPサーバーのモジュールキャッシュ」セクションに再起動条件の記述が追加されていること
- 「AIへの厳命」セクションにRule 22が追加されていること
- 既存のRule 1から21のルール番号が変わっていないこと

## 実装順序と依存関係

### 並列で同時実施可能なもの

FR-1とFR-2はdefinitions.tsの異なるセクション（角括弧説明セクションと禁止語セクション）を変更するため、コンフリクトなく同時実施が可能である。
FR-3はdefinitions.tsのPHASE_DEFINITIONSセクションを変更するが、FR-1とFR-2とは異なる箇所であるため、3件を並行して実装できる。
FR-4はCLAUDE.mdへの変更であり、definitions.tsとは完全に独立しているため、任意のタイミングで実施できる。
各要件の独立性が高いことで、実装効率を最大化できる構造になっている。

### 順序制約があるもの

FR-4のCLAUDE.md変更は、FR-3のmodel変更と連動してテーブルの更新も含むため、FR-3の変更内容が確定してから実施することが望ましい。
4件の実装完了後にnpm run buildを実行してトランスパイルし、MCPサーバーを再起動することで全変更が反映される。
MCPサーバーの再起動はbuild_checkフェーズ完了後、testingフェーズ以降に入る前が適切なタイミングである。

## 非機能要件

### パフォーマンスへの影響

FR-3によるmodel変更は、APIレスポンス時間に影響する可能性がある。
sonnetはhaikuと比較してレスポンス時間が長い場合があるが、parallel_verificationの4サブフェーズは並列実行されるため、最大の待ち時間は個々のサブフェーズの処理時間に依存する。
実装後の検証フェーズで実際のパフォーマンスを計測し、問題があれば対処することとする。

### 運用上の注意事項

本仕様に基づく変更を適用した後は、workflow_statusコマンドで現在のフェーズを確認してから作業を再開することが重要である。
変更の整合性を自動で検証する手段はないため、実装者がチェックリストに従って手動で確認することが必要である。
特にFR-3のmodel変更は、CLAUDE.mdのテーブルとの整合が取れているか目視で確認する必要がある。

## リスクと対策

### model変更によるコスト増加リスク

FR-3によりparallel_verificationの4サブフェーズがsonnetを使用するようになるため、タスクあたりのAPIコストが増加する。
ただし、リトライによる追加コストを削減できるため、平均的なケースではトレードオフが有利に働くと判断している。
リトライが不要になることで、合計トークン使用量が削減される見込みであり、コストの軽減が期待できる。
もし将来的にsonnetでも失敗が続く場合は、さらに上位のモデルへの変更を検討する。

### 後方互換性の確認

FR-1からFR-3の変更はdefinitions.tsのテンプレートとモデル設定にとどまり、workflow-state.jsonのスキーマには影響しない。
進行中のタスクがある場合でも、次フェーズへの移行時から新しい設定が適用されるため、安全に変更を反映できる。
念のため、変更後にnpm testを実行して既存のユニットテストが全て通過することを検証する必要がある。
テストが失敗した場合は、その失敗を修正してから次フェーズに進むこと。

### 自己参照的な問題への注意

FR-1およびFR-2の実装では、definitions.tsのsubagentTemplateに角括弧記法禁止ルールの説明を追加する。
この際、説明テキスト自体に角括弧を含む文字列が混入しないよう注意が必要である。
NG例を示す際も、コードフェンスの中に角括弧を含めないようにし、散文形式でのみ記述する。
この問題はFR-1の受け入れ基準に含まれており、実装者は自己参照的な矛盾が生じないかを確認する。
