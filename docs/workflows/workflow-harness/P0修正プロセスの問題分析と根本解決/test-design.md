## サマリー

本仕様書はタスク「P0修正プロセスの問題分析と根本解決」のテスト設計書である。
テスト対象はworkflow-plugin/mcp-server/src/phases/definitions.tsへの3件の変更（FR-1, FR-2, FR-3）と、CLAUDE.mdへの1件の変更（FR-4）である。
自動化テストはFR-1、FR-2、FR-3を対象とし、TypeScriptで記述したvitestユニットテストとして実装する。
FR-4はドキュメント変更であるため、自動テストではなく手動確認チェックリストで検証する。

- 目的: FR-1〜FR-4の受け入れ基準をテストケースとして具体化し、実装フェーズの開発者とレビュアーが検証可能な形式に整理すること
- 主要な決定事項:
  - 自動テストはvitest形式で記述し、既存のdefinitions.test.tsと同一ディレクトリ（phases/__tests__/）に配置する
  - FR-3のmodel変更テストはPHASE_GUIDESオブジェクトの構造を直接検査する方式とする
  - FR-1とFR-2のbuildPrompt出力テストは関数の返り値を文字列検索で検証する方式とする
  - templateとして生成される出力テキストにコードフェンスやリテラル表記を含める際の自己参照的問題に留意する
- 次フェーズで必要な情報: テストファイルの配置先（ファイルパス）はworkflow-plugin/mcp-server/src/phases/__tests__/definitions-fr1-fr2-fr3.test.tsとする

---

## 背景と根本原因の整理

### なぜこのテスト設計が必要か

前回のP0修正タスクにおいて、parallel_verificationフェーズの全4サブフェーズでバリデーション失敗が発生し、複数回のリトライが必要となった。
各サブフェーズで平均3回以上のリトライ回数が発生したため、完了時間が大幅に増加した。
この問題の根本原因として調査結果から特定されたのは、MCPサーバー未再起動（RC-1）、角括弧記法禁止に関する説明不足（RC-2）、禁止語複合語の具体例不足（RC-3）、そしてhaikuモデルの品質限界（RC-4）の4点である。
本テスト設計書はワークフロープラグインのFR-1からFR-4の各要件が根本問題を解消することを検証するために作成した。
これにより根本的な解決策を提供し、再発防止の徹底を確保することが目的である。

### テスト戦略の背景

spec.mdで定義された実装計画は、変更対象ファイルをworkflow-plugin/mcp-server/src/definitions.tsとCLAUDE.mdの2件に限定している。
プロジェクトルールドキュメントであるCLAUDE.mdへの変更（FR-4）と、ワークフロープラグインのコア設定ファイルへの変更（FR-1〜FR-3）は、それぞれ独立して検証可能である。
この変更対象外のファイルを増やさないアプローチにより、artifact-validator.tsやstate-manager.ts等のコアモジュール、あるいはworkflow-state.jsonのスキーマに影響を与えずに変更できる。
テスト設計においても、この副作用を最小限に保つ方針を反映し、変更された2ファイルの挙動のみを検証の対象範囲とする。
definitions.tsはTypeScriptで記述されており、ビルド後のトランスパイル結果がMCPサーバープロセスにロードされるため、テスト実行前にnpm run buildが完了していることが前提となる。

この方式は実装の内部詳細に依存せず、振る舞い（出力される文字列の内容）とデータ（モデル設定値）を直接検証できる。
FR-4は文書のみの変更であり、CLIツールの動作には直接影響しないため、自動テストではなく手動の目視確認で受け入れ基準を満たすかを検証する。

## テストケース詳細

### FR-1: 角括弧記法禁止説明強化のテスト

FR-1の根本原因はRC-2（角括弧記法禁止に関する説明不足）であり、subagentがコードフェンス内でリテラル記法を使用してバリデーションエラーを引き起こしていた。
受け入れ基準は、buildPrompt関数が生成するプロンプト文字列に「コードブロック内も検出対象である」旨の警告文とNG例・OK例が含まれることである。
角括弧記法禁止の適用範囲を説明文として拡張し、subagentが視覚的に理解できるようにすることが重要である。
本仕様書の各要件に基づいてテストを設計する際、角括弧説明追加の内容がtemplate出力に正しく反映されているかを検証する。

#### TC-FR1-01: コードブロック内検出警告の存在確認

**目的**: buildPrompt関数の返り値にコードブロック内の角括弧も検出対象である旨のテキストが含まれることを確認する。

**前提条件**: PHASE_GUIDESのmanual_testエントリを入力として使用する。

**検証内容**:
buildPromptの返り値が「コードブロック（コードフェンス記号区切り）内も検出対象」または同等の意味を持つ文字列を含むこと。
この警告文は既存のqualitySectionセクション（セクション5）の「角括弧プレースホルダー禁止」のサブセクションに含まれる形で生成される。
FR-1の受け入れ基準は「コードブロック内であっても検出される」という点を明示的に表現することを求めており、templateの文字列に追記する。

**テストコード（参照用）**:
テストでは buildPrompt(manualTestGuide, 'テストタスク', 'テスト意図', '/docs/workflows/test') を呼び出し、返り値をresultとする。
result が「コードブロック」という文字列を含むことを expect(result).toContain で確認する。

#### TC-FR1-02: NG例の存在確認

**目的**: buildPrompt関数の返り値にNG例が少なくとも1件含まれることを確認する。

**検証内容**:
返り値に「NG」または「ng」または「ng:」を含む行が存在することを確認する。
これにより、subagentがどの記法が禁止かを視覚的に理解できる形式でtemplateが出力されていることを保証する。
NG例は散文形式で記述され、角括弧を直接含まない形式で説明されること（自己参照的な矛盾を防ぐための制約）。
このNG例の存在確認はFR-1の説明文拡張が成功したことの証拠となり、テストが成功することでFR-1の実装完了と判断できる。

#### TC-FR1-03: OK例の存在確認

**目的**: buildPrompt関数の返り値にOK例が少なくとも1件含まれることを確認する。

**検証内容**:
返り値に「OK」または「ok:」を含む行が存在することを確認する。
OK例として散文形式への置き換え方法が記述されていることで、subagentが代替表現を選択できる。
波括弧を使った代替記法がOK例として示されているか確認することで、英小文字パターンの説明が適切であるかも検証できる。
OK例の数はNG例と同数以上であることが望ましく、1対1の対応関係により理解を促進する。

#### TC-FR1-04: 配列アクセス記法の言い換え指示確認

**目的**: buildPrompt関数の返り値に「配列アクセス」または「インデックス」に関する説明が含まれることを確認する。

**検証内容**:
返り値に「配列アクセス」または「N番目の要素」または「最初の要素」のような散文形式への言い換え指示が含まれること。
この説明はFR-1の受け入れ基準で必須とされており、subagentがコード例における配列アクセスを散文形式で説明するよう誘導する効果がある。
正規表現のリテラル表記が禁止される背景としてバリデーターの動作原理を理解させる記述があることも確認する。
自動検査の仕組みがtemplate内に説明されていることで、subagentがバリデーターの検知ロジックを事前に把握できる。

受け入れ基準は、PHASE_GUIDESのparallel_verification配下4サブフェーズのmodel値が全て「sonnet」であることである。
haikuとsonnetの価格差は入力・出力トークンともに約15倍程度であるが、リトライによるトータルコストを考慮するとsonnetが有利と判断されている。
前回のタスクで各サブフェーズが平均3回以上のリトライを要したという実績データから、コストと品質のトレードオフを評価した結果の変更である。

このフェーズのAPIレスポンス時間への影響は非機能要件として記録されており、並列実行により最大待ち時間への影響を軽減できる。

## テストファイルの配置と実行方法

### テストファイルの配置先

テストファイルは既存のdefinitions.test.tsと同一ディレクトリに配置する。

- 配置先のファイルパス: `workflow-plugin/mcp-server/src/phases/__tests__/definitions-fr1-fr2-fr3.test.ts`
- 既存テストファイル: `workflow-plugin/mcp-server/src/phases/__tests__/definitions.test.ts`

テストファイルの命名に「fr1-fr2-fr3」を含めることで、本タスクの変更要件に対応するテストであることを明確にする。
機能要件番号を含むファイル名により、将来のレビューフェーズでの確認時に変更とテストの対応関係を容易に把握できる。
テストファイルはTypeScriptで記述し、ビルドプロセス（トランスパイル）を経ずにvitestが直接実行できる形式とする。
プロジェクトルートのpackage.jsonではなく、workflow-plugin/mcp-serverのpackage.jsonに定義されたテスト設定を参照する。

### テストのimport構成

テストファイルはvitest形式を使用し、以下のimportを含む。

- describe, it, expect を vitest からインポートする
- PHASE_GUIDES, buildPrompt を ../definitions.js からインポートする（ESMモジュールとしてインポート）
- 型情報として PhaseGuide 型が必要な場合は型インポートとして追加する

definitions.tsはESMモジュールとして実装されており、import文を使用してインポートする必要がある。
CommonJSのrequireは使用できないため、インポート文の記述方式に注意すること。

### テスト実行コマンド

テストを単独で実行する場合は以下のコマンドを使用する。

```
cd workflow-plugin/mcp-server && npx vitest run src/phases/__tests__/definitions-fr1-fr2-fr3.test.ts
```

全テストを実行してリグレッションを確認する場合は以下のコマンドを使用する。

```
cd workflow-plugin/mcp-server && npx vitest run
```

### ビルドとサーバープロセスの関係

テスト実行前にnpm run buildでトランスパイルが完了していることを確認すること。
サーバープロセスが起動中の場合、モジュールキャッシュにより古いコードが使われる場合がある。
MCPサーバーの再起動は、テスト実行の観点ではなく統合動作確認の観点で必要であり、ユニットテストはビルド後のファイルを直接参照して実行される。
再起動後に作業を再開する際は、workflow_statusで現在のフェーズを確認してから進めること。

## テスト設計の補足事項

### FR-1・FR-2テストにおけるbuildPrompt引数の設定

buildPrompt関数を呼び出す際の引数として、PHASE_GUIDESからparallel_verification.subPhases.manual_testのPhaseGuideオブジェクトを使用する。
このオブジェクトはrequiredSectionsとallowedBashCategoriesを持つため、buildPromptが全9セクションを生成する完全なプロンプトを返す。
docsDirの引数には「/docs/workflows/test-task」のような実在しないパスを使用しても構わない（buildPromptはI/O操作を行わないため）。
テストコード内でbuildPrompt出力の期待値を文字列として記述する際も、角括弧を使用しない表現で記述すること（自己参照的な問題への対処）。
波括弧を代替記法として使用する場合も、その説明は散文形式で行い、templateの自己参照的な矛盾が生じないよう注意する。

### 既存テストとの相互関係

本テストファイルのテストケースは、既存のdefinitions.test.tsのテストケースと独立している。
既存のテストはフェーズ配列の構造やgetNextPhase関数の動作を検証しており、本テストはFR-1〜FR-3の実装変更を検証する。
両テストファイルはCIでの全テスト実行時に共存して動作する設計であり、既存テストのパスが維持されることがFR-1〜FR-3の受け入れ基準の一部（後方互換性）でもある。
また、task-index.jsonのスキーマやworkflow-state.jsonの整合性には影響を与えないことも確認する。
静的解析のエラーがないことも確認対象に含まれ、これはFR-1とFR-2の受け入れ基準に規定された要件である。

### 優先順位と実装計画との整合

spec.mdで示した優先順位（FR-4が最優先、次にFR-3）に対応するテスト実施順序を以下に示す。
FR-4の手動チェックリストは実装完了後すぐに確認できるため、最初に確認することを推奨する。
FR-3のTC-FR3-01〜05は実装後のビルドが完了した段階で実行し、model値が正しく変更されていることを確認する。
FR-1とFR-2はbuildPrompt出力を検証するため、ビルド後にTC-FR1-01〜04とTC-FR2-01〜04を順次実行する。
実装順序の依存関係として、FR-1とFR-2は同時実施可能であり、並列的に実装できる。FR-3は独立した変更箇所であり、任意の順序で実施可能である。

### コスト影響と継続性確認

FR-3によりparallel_verificationの4サブフェーズがsonnetを使用するため、タスクあたりのAPIコストが増加する。
haikuとsonnetの価格差は約15倍程度だが、リトライを繰り返す現状と比較するとトータルコストで有利と判断している。
リトライが不要になることで、合計トークン使用量が削減される見込みであり、最大のリトライ回数が減少するという期待効果がある。
将来的にsonnetでも失敗が続く場合は、さらに上位のモデルへの変更を検討する必要があり、そのトレードオフ分析は別タスクとして実施する。
本テスト設計は「ビルドと再起動を経た後の状態」を前提とし、サーバープロセス再起動前後での動作の継続性を保証する観点でも設計されている。

### 実装者への注意事項

変更概要として整理された4件の機能要件（FR-1〜FR-4）は、それぞれ独立した変更箇所であり、同時実施可能なものとして設計されている。
全変更を適用した後のビルドと再起動が、本テストを有効な状態で実行するための前提条件となる。
再起動後に進行中のタスクがあれば移行時の状態を確認し、混入した不整合がないかをworkflow_statusで検証すること。
本仕様の実施事項として、テスト実行前に静的解析（TypeScript型チェック）が通過していることを確認することが必須事項である。
これらの手段と対策を総合的に実施することで、バリデーション失敗の再発防止という結論に向けた根本的な改善が実現される。
