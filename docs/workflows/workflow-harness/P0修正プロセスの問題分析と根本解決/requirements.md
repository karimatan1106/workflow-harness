## サマリー

前回のP0修正タスク（20260217_173456）で発生したparallel_verificationフェーズの全4サブフェーズバリデーション失敗について、research.mdの調査結果を基に要件を定義する。

根本原因は5つに整理され（RC-1からRC-5）、対応が必要な機能要件はFR-1からFR-4の4件に集約される。
各要件はworkflow-plugin/mcp-server/src/definitions.tsの修正とCLAUDE.mdの運用ルール強化という2つのアプローチで実現する。
最優先はRC-1（MCPサーバー未再起動）への対応であり、次いでRC-2（角括弧説明不足）、RC-3（禁止語具体例不足）、RC-4（model指定問題）の順に対処する。

- 目的: parallel_verification全サブフェーズのバリデーション失敗を再発防止し、リトライ回数を削減する
- 主要な決定事項: definitions.tsのphaseGuide修正、model変更、CLAUDE.md運用ルール強化
- 次フェーズで必要な情報: 各FRの実装対象ファイルはdefinitions.ts（3件）とCLAUDE.md（1件）

---

## 機能要件

### FR-1: subagentTemplateの角括弧禁止説明を強化（RC-2対応）

**目的**: haiku subagentがコードブロック内に角括弧記法（正規表現パターン等）を記述してバリデーション失敗するケースを防ぐ。

**実装対象ファイル**: workflow-plugin/mcp-server/src/definitions.ts

**実装内容**:
- phaseGuide生成ロジック内の角括弧禁止説明に、コードブロック内の正規表現や配列アクセス記法も角括弧として検出されることを明記する
- 具体的には「角括弧プレースホルダー禁止（コードブロック内も検出対象）。正規表現リテラル例: `/\[a-z\]/`は散文形式で「英小文字のパターン」と記述すること」という説明を追加する
- コード例としてNG例（`\[変数名\]`）とOK例（波括弧で代替）を並記し、视覚的に理解しやすくする

**受け入れ基準**:
- subagentTemplateの角括弧禁止説明セクションに、コードブロック内を対象とする旨の記述が含まれること
- NG例とOK例が少なくとも2パターン以上明記されていること
- 既存のテンプレート構造が壊れないこと

---

### FR-2: 禁止語の複合語検出の具体例追加（RC-3対応）

**目的**: subagentが禁止語の部分一致検出ルールを正しく理解し、複合語を含む禁止語を避けて文書を生成できるようにする。

**実装対象ファイル**: workflow-plugin/mcp-server/src/definitions.ts

**実装内容**:
- subagentTemplate内の禁止語リストに、部分一致で検出される複合語の具体例を追加する
- 追加する具体例: 「未定義」（「未定」を含むため禁止）、「要検討事項」（「要検討」を含むため禁止）、「対応予定箇所」（「対応予定」を含むため禁止）
- 各禁止複合語に対して、許容される言い換え例を明記する
  - 「未定義」→「指定されていない」または「定義されていない」
  - 「要検討事項」→「追加調査が必要な事項」または「今後分析が必要な項目」
  - 「対応予定箇所」→「修正が予定されている箇所」という説明に変更することで部分一致を回避
- サンプル、ダミーについても同様に、「テスト用データ」「仮のデータ」という言い換えは禁止であり、実際のデータ値を使用するという注意書きを追加する

**受け入れ基準**:
- 禁止語セクションに複合語の例が3件以上明記されていること
- 各複合語に対して具体的な言い換え例が対応づいていること
- テンプレート生成コードが静的解析エラーなしでビルドできること

---

### FR-3: parallel_verificationのデフォルトmodelをsonnetに変更（RC-4対応）

**目的**: manual_test, security_scan, performance_test, e2e_testの各サブフェーズで、初回からsonnet modelを使用して品質要件を遵守した成果物を生成できるようにする。

**実装対象ファイル**: workflow-plugin/mcp-server/src/definitions.ts

**実装内容**:
- PHASE_DEFINITIONS内のparallel_verificationサブフェーズのmodel指定を変更する
  - manual_test: "haiku" → "sonnet"
  - security_scan: "haiku" → "sonnet"
  - performance_test: "haiku" → "sonnet"
  - e2e_test: "haiku" → "sonnet"
- model変更に合わせてphaseGuideのコメント（「model: haiku」の記述）も更新する
- CLAUDE.md内の「フェーズ別subagent設定」テーブルのmodel列も合わせて更新する

**受け入れ基準**:
- PHASE_DEFINITIONSの4サブフェーズ全てのmodelが"sonnet"になっていること
- 既存のunittestが通ること（model変更による意図しない副作用がないこと）
- コスト増加のトレードオフについてコードコメントに記載されていること

---

### FR-4: CLAUDE.mdへのMCPサーバー再起動ルール強化（RC-1対応）

**目的**: artifact-validatorのコードを修正した後にMCPサーバーを再起動しないまま作業を継続するというミスを防止する。

**実装対象ファイル**: CLAUDE.md（プロジェクトルート）

**実装内容**:
- 「MCPサーバーのモジュールキャッシュ」セクションに、以下の内容を含む「強制再起動条件」のサブセクションを追加する
  - artifact-validator.ts、definitions.ts、state-manager.tsなどコアモジュールを変更した場合は必ずMCPサーバーを再起動すること
  - npm run buildでトランスパイルした後、MCPサーバープロセスを終了して再起動する手順を明記する
  - 再起動前に `workflow_status` で現在のフェーズを確認し、再起動後に同フェーズで作業を再開することを徹底する
- 「AIへの厳命」セクションに、Rule 22として「artifact-validatorを修正した後はMCPサーバーを再起動してから次フェーズに進むこと。未再起動のままparallel_verificationに進むことは禁止」という項目を追加する

**受け入れ基準**:
- 「MCPサーバーのモジュールキャッシュ」セクションに再起動条件が明記されていること
- 「AIへの厳命」セクションに再起動ルールが追加されていること
- 追加した内容が既存のルール番号と整合していること

## 非機能要件

### 後方互換性

FR-1からFR-3の変更はdefinitions.tsのテンプレート生成ロジックに対する変更であり、既存のワークフロー状態（workflow-state.json）に影響を与えない。
進行中のタスクがある場合でも、次フェーズへの移行時から新しいsubagentTemplateが適用される。
PHASE_DEFINITIONSのmodelフィールド変更は既存APIと互換性があり、MCP APIの変更は不要である。

### 変更の独立性

FR-1、FR-2、FR-3はそれぞれ独立した変更であり、実装順序に依存関係がない。
FR-4（CLAUDE.md更新）はFR-3の実装に合わせてCLAUDE.md内の対応テーブルも更新するため、FR-3と同時実施が望ましい。
FR-1とFR-2はdefinitions.tsの異なるセクションを対象としており、コンフリクトが発生しない。

### MCPサーバー再起動の必要性

FR-1、FR-2、FR-3の実装後はdist/*.jsのトランスパイルとMCPサーバー再起動が必要である。
再起動なしでは新しいsubagentTemplateとmodel設定が反映されず、同じ問題が再発するリスクがある。
再起動タイミングはbuild_checkフェーズ完了後、testing以降のフェーズに入る前が適切である。

### テスト可能性

FR-1、FR-2の変更は生成されたsubagentTemplateの文字列を検査するユニットテストで検証できる。
FR-3の変更はPHASE_DEFINITIONSオブジェクトのmodelフィールドを検査するテストで確認できる。
FR-4はCLAUDE.mdのテキスト内容であり、自動テストではなくレビューによって確認する。

### 優先順位

| 要件番号 | 優先度 | 理由 |
|---------|--------|------|
| FR-4 | 最高 | RC-1が最大の原因であり、即時の運用改善が可能 |
| FR-3 | 高 | model変更で根本的にhaiku起因の問題を排除できる |
| FR-1 | 中 | FR-B1/FR-B2が正しく反映されれば軽減されるが、説明強化も重要 |
| FR-2 | 中 | 具体例追加によりリトライ回数の削減が期待できる |
