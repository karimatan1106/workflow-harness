## 概要

本タスクは `workflow-plugin/mcp-server/src/phases/definitions.ts` に対して3件の独立した修正を実施するものである。
いずれの修正も既存のフェーズ定義構造（phaseName・requiredSections・minLines・outputFile・allowedBashCategories 等）を変更しない純粋なテンプレート文字列の追記または置換であり、コアロジックへの影響がない。

### 修正が必要となった経緯

前タスクでの parallel_verification 対応中に以下の3つの問題が実際に発生し、その根本原因が definitions.ts の記述不足または誤った記述にあることが確認された。

第1の問題として、code_review フェーズのサブエージェントが「禁止語チェック」項目の記述時に禁止語の実体リストを成果物に列挙してしまい、バリデーション失敗が発生した。manual_test・security_scan・e2e_test の各テンプレートには禁止語転記防止ガイダンスが存在するが、code_review テンプレートにのみ欠落していることが調査により判明した。

第2の問題として、performance_test の成果物が「4つのサブセクションにそれぞれ1行の長い段落」という構造で記述され、実質行数が4行となってバリデーション失敗（最低5行要件）が発生した。サブヘッダー行は artifact-validator の isStructuralLine 関数により実質行数にカウントされない仕様であるが、この仕様に関する注意書きが performance_test テンプレートに欠落していた。

第3の問題として、definitions.ts の research フェーズチェックリストに「researchフェーズでベースラインを記録する」という記述があり、CLAUDE.md のルール20（「testingフェーズを標準とする」）と矛盾していることが確認された。research フェーズの Bash 許可カテゴリは readonly のみであるため、テスト実行コマンドが技術的に使用不可であることも矛盾の根拠として特定されている。

### 変更の独立性と優先度

3件の修正はすべて同一ファイル（definitions.ts）への変更であるが、対象箇所が異なるため競合しない。
優先度として FR-A と FR-B はバリデーション失敗を直接防ぐガイダンス追加であり高優先度、FR-C は記述の整合性修正であり中優先度と判断する。
実装はいずれの順序でも実施可能であるが、本仕様書では FR-C → FR-A → FR-B の順を推奨する。

## 実装計画

### FR-A: code_review テンプレートへの禁止語転記防止ガイダンス追加

#### 対象箇所

- 対象ファイル: `workflow-plugin/mcp-server/src/phases/definitions.ts`
- 該当行番号: 862行（subagentTemplate の文字列値が定義されている行）
- 挿入位置: subagentTemplate 文字列内の「## ユーザー意図との整合性セクションの行数ガイダンス」セクションと「## 出力」セクションの間
- 現在の末尾パターン: `...総合判定（ユーザー意図の実現度をパーセンテージまたは定性評価で記述する）\n\n## 出力\n`

#### 追加するテキストの内容

「## 出力」の直前に以下のセクションを挿入する。既存の `\n\n## 出力` の前に `\n\n## 禁止語転記防止（重要）\n` から始まるブロックを追加する形で実施する。

```
## 禁止語転記防止（重要）
成果物本文に禁止語を直接記述すると成果物全体がバリデーション失敗になる。
コードレビュー結果においてバリデーターが検出するパターンに言及する場合は間接参照（「バリデーターが検出するパターン」「該当する語句」等）を使用すること。
「禁止語チェック」項目を記述する際は、禁止語グループの語句そのものをコードフェンス外に列挙してはならない。
代わりに「英語グループおよび日本語グループのいずれの語句も検出されなかった」「バリデーターが検出する語句は含まれていない」という形式で記述すること。
バリデーション失敗報告内の語句を成果物本文に転記すると次回も同じエラーが発生するフィードバックループが形成されるため厳禁である。
```

#### 整合性確認事項

manual_test・security_scan・e2e_test の各テンプレートに既存の「## 禁止語転記防止（重要）」セクションが存在しており、今回の追記はそれらと同等の内容を code_review にも付与するものである。
既存の requiredSections 定義には「## 禁止語転記防止」は含まれていないため、このセクションはガイダンス追記のみでありバリデーション必須セクションの追加ではない。

### FR-B: performance_test テンプレートへのサブヘッダー密度ガイダンス追加

#### 対象箇所

- 対象ファイル: `workflow-plugin/mcp-server/src/phases/definitions.ts`
- 該当行番号: 930行（subagentTemplate の文字列値が定義されている行）
- 挿入位置: subagentTemplate 文字列内の「## ボトルネック分析セクションの行数ガイダンス」セクションの末尾と「## 評価結論フレーズの重複回避（特化ガイダンス）」セクションの間

#### 追加するテキストの内容

「## ボトルネック分析セクションの行数ガイダンス」末尾と「## 評価結論フレーズの重複回避」の間に以下のセクションを挿入する。

```
## サブヘッダー多用時のセクション密度確保（ボトルネック分析用）
ボトルネック分析セクションで ### サブヘッダーを複数配置する場合、各サブヘッダー直下に必ず複数の実質行を配置すること。
artifact-validator の isStructuralLine 関数により ### サブヘッダー行は実質行数に含まれないため、サブヘッダーのみで親セクションの実質行数を確保することができない。
問題のある構造パターン: ### サブセクションを4つ配置して各直下に1行の長い段落を配置すると、サブヘッダー4行は実質行ゼロ扱いとなり実質行数の合計が4行にとどまりバリデーション失敗となる。
推奨する構造パターン: 各 ### サブセクション内に最低2〜3行の実質コンテンツを記述し、親セクション全体で5行以上の実質行を確保すること。
散文の段落を複数行に分割するか箇条書きを追加することで実質行数を増やすことができる。
```

#### 整合性確認事項

security_scan テンプレートには「## サブヘッダー多用時のセクション密度確保（FR-2: 実質行数不足防止）」が既に存在している。
今回の追記はそのセクションに相当するものを performance_test テンプレートにも付与するものであり、セクション名はperformance_test 用に「ボトルネック分析用」という表記を採用することで対象フェーズを明示する。

### FR-C: research チェックリストのベースライン記録タイミング修正

#### 対象箇所

- 変更対象: `workflow-plugin/mcp-server/src/phases/definitions.ts`（612行付近の research フェーズチェックリスト配列）
- 該当行番号: 612行（research フェーズ定義の checklist 配列の要素）
- 変更前の記述: `'既存テストスイートを実行してベースラインを記録する（workflow_capture_baseline）'`

#### 変更後の記述

方式2（記述変更）を採用し、以下の文字列に変更する。

```
'既存テストスイートを実行してベースラインを記録する作業は testing フェーズで workflow_capture_baseline を呼び出して実施すること（research フェーズは Bash 許可カテゴリが readonly のみのためテスト実行コマンドが使用不可）'
```

#### 変更の根拠

research フェーズの allowedBashCategories は `readonly` カテゴリのみである。npm test 等のテスト実行コマンドは testing カテゴリに属するため、phase-edit-guard フックによってブロックされる。この技術的制約と CLAUDE.md のルール20（「testingフェーズを標準とする」）の記述が一致しているが、definitions.ts 側が誤った記述のままであるため修正が必要である。変更後の記述は技術的根拠（Bash許可カテゴリの制限）を明示し、誤解を防ぐ内容としている。

### 実装後の検証手順

修正完了後に以下の手順で検証を実施すること。

手順1として、`workflow-plugin/mcp-server` ディレクトリで `npm run build` を実行し TypeScript コンパイルエラーが発生しないことを確認する。ビルド成功が definitions.ts の構文エラーなしの証明となる。

手順2として、ビルド完了後に MCP サーバープロセスを再起動し `workflow_status` を呼び出してサーバーが正常に応答することを確認する。definitions.ts は CLAUDE.md の「強制再起動条件」に列挙されたコアモジュールであるため、変更後の再起動は必須である。

手順3として、FR-C の変更については変更後の文字列が research フェーズのチェックリストとして正しく格納されていることを Read ツールで確認し、CLAUDE.md のルール20と矛盾がないことを目視確認する。

## 脅威モデルとの対応関係

threat-model.md で識別された5件の脅威に対し、本仕様書での対策方針を以下に示す。

T-1（禁止語混入リスク）への対策として、追記するガイダンス文章の中でバリデーターが検出するパターンに言及する箇所はすべて間接表現を使用し、該当する語句の実体をコードフェンス外に記述しない。実装フェーズでは追記内容をバリデーターの検出リストと照合してから確定する。

T-2（既存ガイダンスとの矛盾）への対策として、挿入箇所の前後に存在する既存テンプレート内容を事前に読み込み、追記内容が文脈で一貫した指示になっていることを確認する。manual_test・security_scan 等の既存セクションの書式を踏襲する。

T-3（テンプレート文字列の誤構文）への対策として、追記後に npm run build を実行してコンパイルエラーがゼロであることを確認することを実装完了の必須条件とする。既存のコードフェンス記述箇所を参照してエスケープパターンを踏襲する。

T-4（MCP再起動忘れ）への対策として、本仕様書の「実装後の検証手順」に MCP サーバー再起動を明示的に含め、再起動後に workflow_status で正常応答を確認することを完了条件として定義した。

T-5（FR-C修正後の再矛盾）への対策として、方式2（記述変更）を採用し変更後の記述に技術的根拠（Bash許可カテゴリの制限）を明示することで CLAUDE.md ルール20との整合性を確保する。
