## サマリー

- 目的: `workflow-plugin/mcp-server/src/phases/definitions.ts` に対する3件の修正（FR-A・FR-B・FR-C）のUIインターフェース設計を定義する。本ドキュメントでは MCP サーバープラグインとしての「UI」すなわち CLI インターフェース・エラーメッセージ・API レスポンス・設定ファイル構造の設計方針を明確にする。
- 評価スコープ: `workflow-plugin/mcp-server/src/phases/definitions.ts` の3箇所（行番号 612・862・930）、および変更後のテンプレートを利用する MCP API（`workflow_get_subphase_template`・`workflow_complete_sub`・`workflow_next`）のレスポンス形式が対象である。
- 主要な決定事項: FR-A は code_review サブエージェントテンプレートへの禁止語転記防止セクション挿入、FR-B は performance_test テンプレートへのサブヘッダー密度確保ガイダンス挿入、FR-C は research チェックリスト記述の修正を行う。変更後も既存の MCP API レスポンス構造は維持される。
- 変更後の `workflow_get_subphase_template` 呼び出しで返されるテンプレート文字列のセクション構成が変化するため、呼び出し元 Orchestrator がテンプレートを利用する際の動作は既存の手順と変わらない。
- 次フェーズで必要な情報: 各 FR の挿入・変更対象テキストの詳細は spec.md「実装計画」セクションに記載されている。ビルド成功と MCP サーバー再起動が全変更完了後に必要であること。definitions.ts はコアモジュールのため再起動後に `workflow_status` で正常応答を確認することを完了条件とする。

## CLIインターフェース設計

本タスクが変更する `definitions.ts` は MCP サーバーが提供するサブフェーズテンプレートの定義ファイルである。Orchestrator が利用する主要な MCP API は以下の3種類であり、各 API の呼び出し形式は変更後も変わらない。

### workflow_get_subphase_template の呼び出し形式

このツールはサブフェーズ名を指定してテンプレート文字列を取得する。FR-A・FR-B の変更により、`code_review` および `performance_test` を対象とする呼び出しで返されるテンプレート文字列の本文が拡張される。

引数の仕様を以下に定義する。
- `subPhaseName`（必須）: 対象サブフェーズの識別子文字列。有効値は `code_review`・`performance_test` 等の定義済みサブフェーズ名である。
- `taskId`（省略可）: タスク ID 文字列。省略時はアクティブなタスクが自動選択される。

変更後の呼び出し例として、`code_review` を指定した場合は FR-A で追加された「## 禁止語転記防止（重要）」セクションを含むテンプレートが返される。`performance_test` を指定した場合は FR-B で追加された「## サブヘッダー多用時のセクション密度確保」セクションを含むテンプレートが返される。

### workflow_next の呼び出し形式

`workflow_next` はフェーズ遷移を実施するツールであり、成果物バリデーションを伴う。FR-A・FR-B の変更によりテンプレートの品質ガイダンスが充実するため、code_review および performance_test フェーズで生成される成果物のバリデーション通過率の向上が期待される。

呼び出し時の引数形式は変更なく、`taskId`（省略可）および `sessionToken`（省略可）のみである。

### workflow_complete_sub の呼び出し形式

並列フェーズのサブフェーズ完了通知を行うツールである。`subPhase` 引数に `code_review`・`performance_test` 等を指定して呼び出す。定義ファイルの変更はこのツールの引数形式に影響しない。

## エラーメッセージ設計

本タスクの変更後に予想されるエラーメッセージ形式と、変更前後での差異を定義する。

### バリデーション失敗エラーの形式

`workflow_next` および `workflow_complete_sub` がバリデーション失敗を検出した場合のエラーメッセージ形式は以下の通りである。

エラーメッセージの構成要素として、エラー種別識別子・失敗箇所の説明・修正に必要な情報の3要素が含まれる。形式の例として「必須セクション不足: `## ボトルネック分析` セクションの実質行数が基準に達していない（現在: 3行、必要: 5行以上）」という形式で出力される。

禁止語検出エラーの形式として「禁止パターン検出: コードフェンス外に禁止語が含まれている。該当箇所を間接表現に修正すること」という形式が使用される。このエラーは FR-A の変更により code_review サブエージェントがテンプレートのガイダンスを参照することで発生頻度が低下する見込みである。

セクション密度不足エラーの形式として「セクション密度不足: `## ボトルネック分析` セクションの密度が基準を下回っている（現在: 20%、必要: 30%以上）」という形式が使用される。このエラーは FR-B の変更により performance_test サブエージェントがサブヘッダー多用時の構造ガイダンスを参照することで発生頻度が低下する見込みである。

### FR-C 変更後の research フェーズエラー

FR-C の変更後、research フェーズのチェックリスト表示でテスト実行コマンド使用を促す記述が除去される。サブエージェントが誤って testing カテゴリのコマンドを実行しようとした場合のエラーは phase-edit-guard フックが返す標準的なブロックメッセージ形式（「このフェーズでは許可されていないコマンドです: npm test」）で表示される。

### MCP サーバー再起動前後のエラー

definitions.ts 変更後かつ MCP サーバー再起動前の状態では、古いキャッシュ版の definitions.ts が動作し続けるため変更が反映されない。この状態では `workflow_get_subphase_template` が変更前のテンプレートを返す。再起動完了後に `workflow_status` を呼び出して正常応答が返ることを確認してから次フェーズに進む必要がある。

## APIレスポンス設計

MCP API のレスポンス形式を定義する。定義ファイル変更後も JSON 構造の変更はなく、テンプレート文字列の内容が拡張される点のみが変化する。

### workflow_get_subphase_template の成功レスポンス

成功時のレスポンス構造として以下のフィールドが含まれる。`subagentTemplate` フィールドにはテンプレートの完全な文字列が格納される。`minLines` フィールドにはそのフェーズの成果物最低行数が整数値で格納される。`requiredSections` フィールドには必須セクション名の配列が格納される。`outputFile` フィールドには成果物の出力ファイル名が格納される。`docsDir` フィールドには成果物の保存先ディレクトリパスが格納される。

FR-A の変更後は `code_review` を指定した場合の `subagentTemplate` フィールドに「## 禁止語転記防止（重要）」セクションのテキストが含まれるようになる。

FR-B の変更後は `performance_test` を指定した場合の `subagentTemplate` フィールドに「## サブヘッダー多用時のセクション密度確保」セクションのテキストが含まれるようになる。

### workflow_next の成功レスポンス

フェーズ遷移成功時のレスポンスには `currentPhase`（遷移後のフェーズ名）・`taskId`（タスク識別子）・`phaseGuide`（次フェーズのガイダンス情報）が含まれる。`phaseGuide` 内の `subagentTemplate` は Fix-2 以降スリムガイドのみを返す設計であり、テンプレートの完全取得には `workflow_get_subphase_template` を使用する。

### workflow_next のバリデーション失敗レスポンス

バリデーション失敗時のレスポンスには `error`（エラー種別）・`message`（エラーの説明文）・`details`（具体的な失敗箇所）が含まれる。`details` フィールドには失敗したバリデーション規則の識別子と該当箇所の情報が配列形式で格納される。Orchestrator はこの `details` フィールドを解析してリトライプロンプトを生成する。

## 設定ファイル設計

`definitions.ts` のテンプレート文字列の構造規約と、変更を加える際の設計ルールを定義する。

### テンプレート文字列の構造規約

`subagentTemplate` フィールドには Markdown 形式のテキストが格納される。テンプレート内のセクション構造として、最上位ヘッダー（`##` レベル）によるセクション分割が基本単位である。各セクションは役割に応じて以下の種類に分類される。

ガイダンスセクションとして「## 禁止語転記防止（重要）」「## サブヘッダー多用時のセクション密度確保」等が存在し、サブエージェントが成果物を作成する際の品質確保指針を提供する。作業指示セクションとして「## 入力」「## 出力」「## 作業内容」等が存在し、フェーズ固有の実施内容を明示する。

### テンプレート追記時のエスケープ規約

TypeScript のテンプレートリテラル文字列内でバッククォートを含む内容を記述する際はバックスラッシュエスケープ（`\``）が必要である。改行はバックスラッシュエスケープ（`\n`）または実際の改行文字を使用できるが、既存テンプレートの書式に従って統一する。

`\n` による改行は見通しの悪いテンプレート文字列を生む可能性があるため、実際の改行文字を使用している既存の定義スタイルに合わせることが望ましい。

### FR-A・FR-B の挿入位置規約

挿入位置は各テンプレートの「## 出力」セクション直前（FR-A）または既存ガイダンスセクションの末尾（FR-B）と定義されている。挿入後もテンプレート全体の論理的な順序（作業背景 → 入力 → 作業内容 → ガイダンス → 出力）が維持されることを確認する必要がある。

### FR-C の置換規約

research フェーズの `checklist` 配列要素の文字列を新しい内容に置換する。配列内の他の要素には影響を与えない単純な文字列置換であり、配列構造の変更は行わない。

変更後の文字列には技術的根拠として Bash 許可カテゴリ（`readonly`）の制限を明示することで、フェーズルールと実際の制約が一致した記述となる。CLAUDE.md のルール20と矛盾がない内容であることを変更後に目視確認することが検証手順として定められている。
