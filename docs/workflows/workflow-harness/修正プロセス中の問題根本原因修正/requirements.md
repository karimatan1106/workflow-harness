## サマリー

- 目的: 前タスク（parallel_verification-subagentTemplate品質問題修正）の実施中に発生した3つの問題について、根本原因を解消するための修正要件を定義する
- 主要な決定事項: FR-A（code_reviewテンプレートへの禁止語転記防止ガイダンス追加）、FR-B（performance_testテンプレートへのサブヘッダー多用時の密度ガイダンス追加）、FR-C（researchフェーズチェックリストのベースライン記録タイミング記述を修正）の3件を実施する
- 修正対象ファイル: `workflow-plugin/mcp-server/src/phases/definitions.ts` の2箇所と `CLAUDE.md` の1箇所（合計3修正）
- 次フェーズで必要な情報: FR-Aの挿入箇所（code_review subagentTemplateのユーザー意図セクション直後）、FR-Bの挿入箇所（performance_test subagentTemplateのボトルネック分析ガイダンスセクション直後）、FR-Cの修正箇所（definitions.ts行612のチェックリスト項目）

## 機能要件

### FR-A: code_review テンプレートへの禁止語転記防止ガイダンス追加

code_review の subagentTemplate（definitions.ts 行862付近）の末尾、「## ユーザー意図との整合性セクションの行数ガイダンス」セクションと「## 出力」セクションの間に、禁止語転記防止に関するガイダンスセクションを新たに追加する必要がある。

追加するガイダンスの内容は manual_test・security_scan・e2e_test の各テンプレートに存在する「禁止語転記防止（重要）」セクションに相当するものとし、以下の要点を含めること。
「成果物本文に禁止語を直接記述すると成果物全体がバリデーション失敗になる」という警告文を記載すること。
「コードレビュー結果でバリデーターが検出するパターンに言及する場合は間接参照（バリデーターが検出するパターン・該当する語句等）を使用すること」という指示を記載すること。
「バリデーション失敗報告内の語句を成果物本文に転記するとフィードバックループが形成されるため厳禁である」という注意書きを記載すること。

この修正が必要な理由は、code_review フェーズのサブエージェントが「禁止語チェック」項目として禁止語の実体リストを成果物に列挙してしまい、バリデーション失敗が発生したことが確認されているためである。
同等の防止ガイダンスが parallel_verification の他のサブフェーズ（manual_test・security_scan・e2e_test）には存在しているが、code_review テンプレートにのみ欠落していることが調査により特定されている。

### FR-B: performance_test テンプレートへのサブヘッダー多用時の密度ガイダンス追加

performance_test の subagentTemplate（definitions.ts 行930付近）の「## ボトルネック分析セクションの行数ガイダンス」セクションの直後に、サブヘッダー多用時のセクション密度確保に関するガイダンスを追加する必要がある。

追加するガイダンスの内容は security_scan テンプレートの「## サブヘッダー多用時のセクション密度確保（FR-2: 実質行数不足防止）」セクションに相当するものとし、以下の要点を含めること。
「セクション内に複数のサブヘッダーを配置する場合は、親セクション全体で5行以上の実質行が確保されるように各サブヘッダー直下に十分なコンテンツを記述すること」という指示を記載すること。
問題のある構造パターン（サブヘッダーを3つ使用して各直下1行のみ記述すると合計3行となりバリデーション失敗する例）を具体的に記載すること。
推奨する構造パターン（各サブヘッダー直下に最低2〜3行の実質行を記述して5行以上を確保する方法）を具体的に記載すること。

この修正が必要な理由は、前タスクでの performance_test 成果物が「4つのサブセクションにそれぞれ1行の長い段落」という構造で記述され、実質行数が4行となってバリデーション失敗（最低5行要件を満たさない）が発生したためである。
「ボトルネック分析セクションの行数ガイダンス」として5項目の必須記述要件は既に存在しているが、サブヘッダーを多用した場合に実質行がカウントされないというartifact-validatorの仕様に関する注意書きが欠落していることが問題の根本原因として特定されている。

### FR-C: definitions.ts の research チェックリストにおけるベースライン記録タイミングの記述修正

definitions.ts 行612の research フェーズチェックリストに含まれる「既存テストスイートを実行してベースラインを記録する（workflow_capture_baseline）」という項目を、testing フェーズで実施することを示す記述に修正する必要がある。

具体的な修正内容は以下のいずれかの方式で実施すること。
方式1として、該当チェックリスト項目を削除し、testing フェーズのチェックリストに同等の項目を追加する方式。
方式2として、該当チェックリスト項目の記述を「既存テストスイートを実行してベースラインを記録する作業は testing フェーズで workflow_capture_baseline を呼び出して実施する（research フェーズの Bash 許可カテゴリが readonly のみのためテスト実行が不可）」のように、フェーズを明示した記述に変更する方式。

この修正が必要な理由は、CLAUDE.md のルール20が「testingフェーズを標準とする」と明記しているのに対し、definitions.ts の research チェックリストに「researchフェーズでベースラインを記録する」と記載されており、両者の記述が矛盾していることが調査により確認されているためである。
research フェーズの Bash 許可カテゴリが `readonly` のみであるため、npm test 等のテスト実行コマンドが使用できず、ベースライン数値の取得手段がない点も技術的な根拠として整合性修正の必要性を裏付けている。

## 非機能要件

### 後方互換性の維持

FR-A および FR-B の修正は definitions.ts のサブエージェントテンプレート文字列への追記であり、既存のフェーズ定義の構造（phaseName・requiredSections・minLines・outputFile 等）を変更してはならない。
追加するガイダンスセクションは既存のガイダンスセクションと同じ書式（`## セクション名\nコンテンツ\n` 形式）で記述すること。
変更対象は definitions.ts のテンプレート文字列のみであり、artifact-validator.ts・state-manager.ts・test-tracking.ts 等の他のコアモジュールへの変更は行わないこと。

### MCPサーバー再起動の要否判定

FR-A と FR-B は definitions.ts のテンプレート文字列の追記変更であり、コアロジック（バリデーションルール・状態管理・HMAC計算）には影響しない。
しかし definitions.ts は CLAUDE.md の「強制再起動条件」に列挙されているコアモジュールのひとつであるため、修正後に MCP サーバーのビルドと再起動を実施すること。
FR-C は CLAUDE.md 本体への追記変更であり、MCPサーバーは参照しないため再起動は不要である。

### 修正の独立性

3件の修正（FR-A・FR-B・FR-C）は相互に依存関係を持たない独立した変更であり、任意の順序で実施可能である。
各修正は definitions.ts の異なる箇所または異なるファイルを対象としているため、競合が発生しない。
修正実施後に既存のバリデーションテストが引き続きパスすることを確認すること（ビルドエラーが発生しないことの確認も含む）。

### テストおよびリグレッション要件

FR-A・FR-B の修正完了後、MCP サーバーの npm run build が正常完了することを確認すること。
既存の TypeScript テストスイート（mcp-server/src 配下）のビルドエラーが発生しないことを確認すること。
FR-C の修正は CLAUDE.md の更新のみのため、ビルドへの影響はないが、記述変更後の内容がルール20と整合していることを確認すること。
