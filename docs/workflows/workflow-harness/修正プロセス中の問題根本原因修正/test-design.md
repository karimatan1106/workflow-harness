## サマリー

- 目的: definitions.ts への3件の修正（FR-A・FR-B・FR-C）が正しく実装されているかを検証するテスト設計書であり、実装フェーズで作成するテストコードの指針を提供する。
- 主要な決定事項: 本タスクはソースコードの変更ではなくテンプレート文字列の追記・置換であるため、単体テストは TypeScript コンパイル成功とテンプレート文字列の内容検証に集中する。各 FR について正常系（変更が適用されている）・異常系（変更前の状態が残っている）・境界値（挿入位置の前後が壊れていない）の3観点でテストケースを設計する。
- 次フェーズで必要な情報: 実装フェーズでは spec.md「実装計画」セクションに記載された挿入テキストをそのまま使用すること。ビルド成功がすべてのテストの前提条件であり、MCP サーバー再起動なしではテンプレート検証テストは意味をなさない点を考慮すること。

---

## テスト方針

### 検証の基本方針

本タスクの変更対象は `workflow-plugin/mcp-server/src/phases/definitions.ts` の3箇所であり、いずれも既存の関数シグネチャや型定義を変更しないテンプレート文字列の追記・置換である。そのため、テストは以下の2つの軸で実施する。

第1の軸は静的検証であり、TypeScript コンパイルが成功することで definitions.ts の構文的正確性を確認する。文字列リテラルの閉じ忘れやエスケープ不足は npm run build で即座に検出できる。

第2の軸は内容検証であり、変更後のファイルを Read ツールまたは grep で読み込み、追記・置換されたテキストが期待通りの位置と内容で存在することを確認する。MCP サーバー再起動後に workflow_get_subphase_template ツールでテンプレートを取得して実際の動作を確認することも検証手段として採用する。

### テスト実施の前提条件

テストを実施するためには以下の前提が満たされている必要がある。

FR-A・FR-B・FR-C の3件の修正がすべて definitions.ts に反映されていること、`npm run build` が完了して `dist/phases/definitions.js` に最新のコンパイル結果が生成されていること、MCP サーバープロセスが再起動して新しい definitions.js を読み込んでいること、の3点が前提となる。いずれかの前提が欠けている場合、動的テストの結果は修正前の状態を示す可能性があるため注意が必要である。

### テスト優先度の考え方

FR-A と FR-B はバリデーション失敗を直接防ぐための高優先度修正であるため、これらのテストを最初に実施する。FR-C は記述整合性の修正であり中優先度であるが、CLAUDE.md ルール20との整合確認は必須である。

## テストケース

### TC-1: TypeScript コンパイル成功の確認（ビルドテスト）

このテストは FR-A・FR-B・FR-C の実装後に最初に実施する全体確認テストである。

**テスト目的**: 3件の修正がすべて適用された状態で TypeScript コンパイルエラーが発生しないことを確認する。テンプレート文字列内のバックティック、バックスラッシュ、改行文字 `\n` 等のエスケープが正しく記述されていることを間接的に検証する。

**実施手順**: `workflow-plugin/mcp-server` ディレクトリで `npm run build` コマンドを実行し、終了コードが0であることを確認する。

**合格条件**: コマンドが終了コード0で完了し、stderr にコンパイルエラーが出力されないこと。`dist/phases/definitions.js` ファイルの更新日時が修正作業後の時刻であることをあわせて確認する。

**失敗時の対応**: エラーメッセージに示された行番号を確認し、文字列リテラルの閉じ忘れやエスケープの誤りを修正してから再実施する。

---

### TC-2: FR-A の追記内容の存在確認

このテストは FR-A（code_review テンプレートへの禁止語転記防止ガイダンス追加）が正しく適用されていることを確認するテストである。

**テスト目的**: `definitions.ts` の code_review subagentTemplate 文字列内に「## 禁止語転記防止（重要）」セクションのテキストが含まれており、かつ「## 出力」セクションより前に配置されていることを検証する。

**実施手順**: TC-1 のビルドが成功していることを前提に、Read ツールで definitions.ts の 850 〜 900 行付近を読み込む。grep ツールで `禁止語転記防止` という文字列が definitions.ts 内に存在することを確認する。

**合格条件その1**: 文字列 `禁止語転記防止（重要）` が definitions.ts 内に1箇所以上存在すること。
**合格条件その2**: 挿入されたセクションの末尾と「## 出力」セクション開始の間の順序が前後していないこと（挿入ブロックが「## 出力」より先に出現していること）。
**合格条件その3**: 追記されたガイダンス内のテキストに、バリデーターが検出するパターンの実体語句がコードフェンス外に直接記述されていないこと（間接表現が使用されていること）。

**失敗の原因として想定されるケース**: 挿入位置が「## 出力」セクションの後になっている場合、または文字列が存在しない場合は FR-A の実装が正しく反映されていない。

---

### TC-3: FR-A の挿入位置の境界確認

このテストは FR-A の挿入によって既存のテンプレート構造が壊れていないことを確認する境界値テストである。

**テスト目的**: 「## 禁止語転記防止（重要）」セクションの挿入前後にある既存テキスト（「## ユーザー意図との整合性セクションの行数ガイダンス」の末尾部分と「## 出力」セクションの開始部分）が欠損または変形していないことを確認する。

**実施手順**: Read ツールで definitions.ts の code_review テンプレート定義箇所を読み込み、挿入前後のテキストが維持されていることを目視で確認する。

**合格条件**: `総合判定` に関連するテキスト（spec.md 記載の挿入前パターン）が挿入セクションの直前に存在すること、かつ「## 出力」セクションが挿入セクションの直後に存在すること。2つの隣接セクション間に意図しない改行の過剰追加や文字列の欠損がないこと。

---

### TC-4: FR-B の追記内容の存在確認

このテストは FR-B（performance_test テンプレートへのサブヘッダー密度ガイダンス追加）が正しく適用されていることを確認するテストである。

**テスト目的**: definitions.ts の performance_test subagentTemplate 文字列内に「## サブヘッダー多用時のセクション密度確保（ボトルネック分析用）」セクションが含まれていることを検証する。

**実施手順**: Read ツールで definitions.ts の 920 〜 960 行付近を読み込む。grep ツールで `サブヘッダー多用時のセクション密度確保` という文字列が definitions.ts 内に存在することを確認する。

**合格条件その1**: 文字列 `サブヘッダー多用時のセクション密度確保（ボトルネック分析用）` が definitions.ts 内に1箇所以上存在すること。
**合格条件その2**: 追記セクションが「## ボトルネック分析セクションの行数ガイダンス」セクションの末尾より後に出現し、「## 評価結論フレーズの重複回避」セクションより前に出現すること。
**合格条件その3**: 追記セクション内に `isStructuralLine` または「実質行数にカウントされない」に関する説明文が含まれていること（バリデーター仕様の周知が目的であるため）。

---

### TC-5: FR-B の挿入位置の境界確認

このテストは FR-B の挿入によって performance_test テンプレートの既存セクション構造が壊れていないことを確認する境界値テストである。

**テスト目的**: 「## ボトルネック分析セクションの行数ガイダンス」の末尾テキストと「## 評価結論フレーズの重複回避」の開始テキストが、FR-B の追記後も欠損なく維持されていることを確認する。

**実施手順**: Read ツールで定義箇所を読み込み、挿入前後のセクションが正しい順序で存在することを確認する。

**合格条件**: 「## ボトルネック分析セクションの行数ガイダンス」セクションのヘッダーと「## サブヘッダー多用時のセクション密度確保（ボトルネック分析用）」セクションのヘッダーが連続して存在し、その後に「## 評価結論フレーズの重複回避」セクションが続く順序が維持されていること。各セクション間の `\n\n` による区切りが適切であること。

### TC-6: FR-C の文字列置換の確認

このテストは FR-C（research フェーズチェックリストのベースライン記録タイミング修正）が正しく適用されていることを確認するテストである。

**テスト目的**: definitions.ts の research フェーズ定義の checklist 配列において、変更前の誤った記述が削除され、変更後の正しい記述（testing フェーズでの実施を明示した文字列）が配置されていることを検証する。

**実施手順**: Read ツールで definitions.ts の 605 〜 625 行付近を読み込む。grep ツールで `workflow_capture_baseline` という文字列が definitions.ts 内に存在することを確認し、その行の内容を確認する。

**合格条件その1**: `workflow_capture_baseline` を含む行に `testing フェーズ` または `testing` に関する表現が含まれており、testing フェーズでの実施を明示していること。
**合格条件その2**: 変更前の誤った記述（spec.md に記載の変更前テキスト）が definitions.ts 内に存在しないこと。grep で変更前テキストを検索して0件であることを確認する。
**合格条件その3**: 変更後の文字列に `readonly` カテゴリのみのため testing カテゴリコマンドが使用不可であるという技術的根拠が明示されていること。

### TC-7: MCP サーバー再起動後の正常応答確認

このテストは TC-1 のビルド成功後に MCP サーバーを再起動し、新しい definitions.js が正しく読み込まれていることを確認するテストである。

**テスト目的**: MCP サーバーが再起動後にクラッシュせず正常に動作し、`workflow_status` の呼び出しに対して正常なレスポンスを返すことを確認する。definitions.ts の変更が新しい Node.js モジュールとして読み込まれていることを間接的に検証する。

**実施手順**: MCP サーバープロセスを再起動する。その後 `workflow_status` MCP ツールを呼び出し、レスポンスが正常に返ることを確認する。

**合格条件**: `workflow_status` の呼び出しが正常レスポンスを返し、エラーや例外が発生しないこと。サーバーログにモジュール読み込みエラーや起動時例外が出力されていないこと。

### TC-8: workflow_get_subphase_template での code_review テンプレート確認

このテストは TC-7 の MCP サーバー再起動後に、実際に返されるテンプレート内容で FR-A の変更を動的に確認するテストである。

**テスト目的**: `workflow_get_subphase_template` ツールで code_review サブフェーズのテンプレートを取得し、FR-A で追記されたガイダンスセクションが実際のテンプレート応答に含まれていることを確認する。

**実施手順**: `workflow_get_subphase_template` MCP ツールを `subPhaseName: "code_review"` で呼び出し、返された `subagentTemplate` に `禁止語転記防止` に関するテキストが含まれていることを確認する。

**合格条件**: 返された subagentTemplate 文字列内に「禁止語転記防止」を含むセクションが存在すること。当該セクションの内容が間接表現（バリデーターが検出するパターン、という表現等）を使用しており、バリデーター検出語句の実体を直接記述していないこと。

**テスト目的**: `workflow_get_subphase_template` ツールで performance_test サブフェーズのテンプレートを取得し、FR-B で追記されたガイダンスセクションが実際のテンプレート応答に含まれていることを確認する。

**実施手順**: `workflow_get_subphase_template` MCP ツールを `subPhaseName: "performance_test"` で呼び出し、返された `subagentTemplate` に `サブヘッダー多用時のセクション密度確保` に関するテキストが含まれていることを確認する。

### TC-10: CLAUDE.md ルール20との整合性目視確認

このテストは FR-C の変更後に definitions.ts の記述が CLAUDE.md の既定ルールと矛盾していないことを確認する整合性テストである。

**テスト目的**: definitions.ts の research フェーズチェックリストに記載されたベースライン記録の説明が、CLAUDE.md ルール20「既存テストが存在する場合は testing フェーズを標準とする」の記述と一致していることを確認する。

**実施手順**: CLAUDE.md のルール20（「既存テストのベースライン記録義務」セクション）を Read ツールで読み込み、記述内容を確認する。definitions.ts の変更後テキストを Read ツールで読み込み、両者の内容が矛盾していないことを目視で確認する。

**合格条件**: CLAUDE.md ルール20の「testingフェーズを標準とする」という方針と、definitions.ts の変更後チェックリスト項目の記述が内容的に一致していること。definitions.ts の変更後テキストが `readonly` カテゴリのみである技術的根拠を含んでおり、research フェーズでテスト実行が不可能であることを説明していること。

## テスト実施順序

各テストケースは以下の順序で実施することを推奨する。

最初に TC-1（ビルドテスト）を実施し、コンパイルが成功することを確認する。コンパイルが失敗した場合は他のテストを実施せずに実装を修正する。

TC-1 成功後に TC-2・TC-3・TC-4・TC-5・TC-6 を静的検証として実施する。これらはすべて Read ツールと grep による definitions.ts の内容確認であり、MCP サーバーの再起動前でも実施可能である。

次に MCP サーバーを再起動し、TC-7（正常応答確認）を実施する。TC-7 が成功したことを確認してから TC-8 と TC-9 の動的テストを実施する。

最後に TC-10（整合性確認）を実施してすべての検証を完了する。

---

## テスト対象外の範囲

以下の項目は本テスト設計のスコープ外である。

`artifact-validator.ts` の動作検証は本テストの対象外である。バリデーターの仕様変更は本タスクに含まれないため検証不要。`state-manager.ts` の動作検証も対象外であり、HMAC 整合性の確認は不要。既存の他フェーズ（manual_test・security_scan・e2e_test）のテンプレートへの影響確認も対象外である（本タスクはこれらのテンプレートを変更しない）。E2E テストや実際のサブエージェント起動による統合テストは parallel_verification フェーズで実施するものであり本テスト設計の対象外とする。
