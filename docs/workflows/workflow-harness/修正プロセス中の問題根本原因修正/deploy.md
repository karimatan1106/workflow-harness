## サマリー

definitions.ts へのテンプレート修正（FR-A、FR-B、FR-C の3つの改善項目）がコンパイルされ、ディストリビューション版（dist/phases/definitions.js）として生成されました。今回の変更は MCP サーバープロセスの再起動によって実行環境に反映されます。

**主要な変更:**
- FR-A: code_review フェーズテンプレートに禁止語転記防止ガイダンスを追加、バリデーション失敗時のエラーメッセージ直接転記を防止
- FR-B: performance_test フェーズテンプレートにサブヘッダー密度確保ガイダンスを追加、セクション密度不足エラー回避
- FR-C: research フェーズチェックリストでベースライン記録タイミングを修正、readonly 制約を明確化

**デプロイ成功条件:** MCP サーバーが再起動され、新しいバリデーション手順が実行中のサーバーインスタンスに反映される状態

次フェーズから新規ワークフロータスクを実行する際、これらの改善されたテンプレートが自動的に適用されます。

## デプロイ内容

### 変更ファイル

メインの変更ファイルは以下の通りです：

### 変更内容の詳細

**FR-A: code_review テンプレートへの禁止語転記防止ガイダンス**
- code_review サブフェーズの subagentTemplate に「禁止語転記防止」セクションを追加
- エラーメッセージから改善要求への変換時に、禁止語をコードブロック内でそのまま引用することを禁止
- 代わりに「バリデーターが検出した語句」「引用内に含まれるキーワード」等の間接表現を使用するよう指示
- 目的：フィードバックループ（エラーメッセージに含まれる禁止語を成果物本文に転記すると次回も同じエラーが発生）の防止

**FR-B: performance_test テンプレートへのサブヘッダー密度確保ガイダンス**
- performance_test サブフェーズの subagentTemplate に「サブヘッダー密度確保」ガイダンスを追加
- セクション内の密度（実質行／総行数）が30％以上を維持するよう、サブヘッダーによるセクション分割の指針を記述
- 目的：セクション密度不足によるバリデーション失敗を事前に防止

**FR-C: research フェーズチェックリストの timing 修正**
- research フェーズの作業チェックリスト内の「既存テストスイートを実行してベースラインを記録する」項目を修正
- 修正内容：research フェーズは Bash 許可カテゴリが readonly のみであるため、npm test 等のテスト実行コマンドが使用不可であることを明記
- テスト実行は testing フェーズで workflow_capture_baseline を呼び出して実施することを指示
- 目的：phase-edit-guard によるコマンドブロック回避、リソース効率化

## MCP サーバー再起動手順

実行中の MCP サーバープロセスは Node.js のモジュールキャッシュ機構により、起動時に読み込んだ definitions.js を常駐メモリ内に保持しています。ディスク上のファイルが更新されても、プロセス再起動前は古いキャッシュが動作し続けます。

### ステップ 1: Claude Desktop から MCP サーバーを再起動

MCP サーバーの再起動方法は以下の2つです：

1. **Claude Desktop のサーバー管理画面から再起動**
   - Claude Desktop のメイン画面から「Settings」（設定）を開く
   - 「MCPServers」または「Installed Models」セクションを選択
   - workflow-plugin MCP サーバーの再起動ボタン（通常は🔄アイコン）をクリック
   - プロセスが終了し、新しいインスタンスが自動起動される

2. **Claude Desktop アプリ全体を再起動**
   - Claude Desktop アプリを完全に終了する
   - 数秒待機する
   - Claude Desktop を再度起動する
   - MCP サーバーも自動的に再起動される

### ステップ 2: 再起動後の正常応答確認

MCP サーバーが正常に再起動されたことを確認するため、workflow_status コマンドを実行してください。

期待される応答例：
- MCP サーバーが「オンライン」状態を返す
- 現在のタスクフェーズが正しく報告される
- エラーメッセージが発生しない

コマンド例：
```
/workflow status
```

正常応答が得られた場合、新しい definitions.js がメモリ内にロードされ、改善されたテンプレートが実行環境に反映されたことが確認されます。

### トラブルシューティング

| 状況 | 対応方法 |
|-----|--------|
| MCP サーバーが起動しない | Claude Desktop ログを確認して node プロセスのエラーを調査。mcp-server/dist/ ディレクトリ以下のファイル生成状態を確認。 |
| workflow_status がタイムアウト | MCP サーバープロセスが完全に起動していない可能性。30秒程度待機した後、再度コマンドを実行。 |
| エラーメッセージが続く | ディストリビューション版の生成失敗の可能性。workflow-plugin/mcp-server ディレクトリで npm run build を再度実行。 |

## 変更後の動作確認手順

MCP サーバー再起動後、改善内容が正しく適用されていることを確認するための検証手順を以下に示します。

### 1. 新規ワークフロータスク実行時のテンプレート確認

新しいタスクを開始し、各フェーズでサブエージェントテンプレートが正しく取得されていることを確認してください。

手順：
- `/workflow start 新しいテストタスク` でタスクを開始
- design_review フェーズまで進める
- code_review フェーズが開始されたときに、サブエージェントテンプレートに「禁止語転記防止」セクションが含まれていることを確認

期待される確認ポイント：
- code_review テンプレートに「⚠️ 禁止語転記防止」警告セクションが表示されている
- テンプレートのリトライプロンプトに「エラーメッセージ中の語句を成果物本文に転記しないでください」という注記が含まれている

### 2. parallel_verification フェーズでのテンプレート確認

performance_test サブフェーズのテンプレートにサブヘッダー密度ガイダンスが含まれていることを確認してください。

手順：
- テストタスクが parallel_verification フェーズに到達したときに、performance_test サブエージェントテンプレートを確認
- テンプレート内に「サブヘッダー密度確保」についてのガイダンス（セクション内の密度維持、サブセクション分割の推奨等）が記載されていることを確認

期待される確認ポイント：
- performance_test テンプレートのセクション密度要件（30％以上）が明記されている
- 密度不足を防ぐためのセクション分割手法が記載されている

### 3. research フェーズのチェックリスト確認

research フェーズの作業指示でベースライン記録タイミングが修正されていることを確認してください。

手順：
- 新規タスクの research フェーズで、作業チェックリスト項目を確認
- 「既存テストスイートを実行してベースラインを記録する」項目が「testing フェーズで workflow_capture_baseline を呼び出して実施すること」と修正されていることを確認
- readonly 制約による npm test 実行不可が明記されていることを確認

期待される確認ポイント：
- research フェーズのテンプレートに「testing フェーズで…」という修正内容が反映されている
- research では Bash 許可カテゴリが readonly のみであることが明確に記載されている

### 4. バリデーション動作の確認

実際のワークフロータスク（parallel_verification 等で成果物が生成される段階）を実行し、改善されたバリデーション手順が正しく機能していることを確認してください。

期待される動作：
- code_review 成果物でバリデーション失敗が減少している（禁止語転記防止の効果）
- performance_test 成果物でセクション密度不足エラーが減少している
- research フェーズでベースライン記録に関連するエラーメッセージが消失している
