## サマリー

本タスク「FR修正プロセスで発生した問題の根本原因解決」は、`workflow-plugin/mcp-server/src/phases/definitions.ts` 単一ファイルへの文字列リテラル変更・配列要素追加・ループ処理置き換えを対象とした内部コード修正である。
修正の対象はsubagentプロンプト生成ロジックと `code_review` サブフェーズの必須セクション定義に限定されており、エンドユーザーが操作するCLIインターフェース・APIレスポンス構造・設定ファイルのスキーマには一切変更がない。
CLIコマンド群（`/workflow start`、`/workflow next`、`/workflow status` 等）の引数・オプション・出力フォーマットは今回の修正で変わらない。
MCPサーバーが返すAPIレスポンスのJSONスキーマ・フィールド構造・HTTPステータスコードも変更対象外である。
subagentプロンプトの文言変更はユーザーに直接表示されず、Orchestratorが内部的に参照するためUI影響はゼロとなる。
本UI設計書は「UIに変更がない」という事実を明示的に記録し、今後のレビュー者が変更範囲を正確に把握できるようにすることを目的としている。

## CLIインターフェース設計

本タスクではCLIインターフェースに変更は加えない。現行のコマンド体系はそのまま維持される。

ユーザーが日常的に利用する `/workflow start <タスク名>`、`/workflow next`、`/workflow status`、`/workflow approve design`、`/workflow list` の各コマンドは、引数仕様・出力フォーマット・エラー表示方式を含めて今回の修正対象外である。
CLIコマンドのヘルプテキストや説明文も変更しない。
`/workflow complete-sub <サブフェーズ>` コマンドについても引数の種類・受け入れ値・応答メッセージに変更はない。
`/workflow reset [理由]` および `/workflow back <targetPhase>` コマンドの動作仕様も変更されない。
ユーザーが入力するコマンド文字列のパース処理は `definitions.ts` のsubagentTemplate変更によって影響を受けない構造である。
FR-A〜FR-Eの修正はいずれも `definitions.ts` の内部定義変更であり、CLIレイヤーのコードには手を触れない。
コマンド実行時に表示されるプロンプト・ステータス表示・進捗メッセージの書式も維持される。
MCPサーバー再起動後もCLIの応答速度やコマンド処理のタイムアウト値に変更はなく、ユーザー体験への影響はない。

## エラーメッセージ設計

subagentプロンプト変更に伴うバリデーションメッセージへの影響について以下に記述する。

FR-Aの修正（`code_review` の `requiredSections` 配列に `'## パフォーマンス'` を追加）により、`artifact-validator.ts` が `code-review.md` を検証する際に従来は検出していなかった「## パフォーマンスセクション欠落」エラーがsubagentに事前通知されるようになる。これはバリデーションエラーメッセージの種類が増えるのではなく、subagentが事前に正しいセクションを作成するようになることでエラー発生頻度が低下する方向への変化である。
MCPサーバーの `workflow_next` ツールが返すバリデーション失敗メッセージの形式（JSONオブジェクト構造）は変更しない。
`workflow_record_test_result` ツールがFR-Eの修正後に返すブロックエラーメッセージの構造も変わらない。ただし、subagentがメッセージ内容を事前に把握するためエラー発生頻度が低下する。
FR-Bの修正（禁止語ループ処理のグループ参照への置き換え）により、subagentが禁止語を成果物に転記するリスクが低下し、結果として禁止パターン検出エラーの出現頻度が減少すると見込まれる。
エラーメッセージのテキスト内容・エラーコード・フォーマットの仕様変更は今回の修正範囲に含まれない。
FR-Cの修正（manual_testテンプレートへの重複行回避ガイダンス追加）は、subagentが重複行エラーを受け取る頻度を低下させるが、エラーメッセージの構造自体には変更を加えない。
FR-Dの修正（commitテンプレートへのサブモジュール確認手順追加）は、コミットフェーズでの見落としを減らすための事前指示であり、コミット失敗時のエラーメッセージ形式には影響しない。

## APIレスポンス設計

MCPサーバーが提供するAPIのレスポンス仕様に変更はない。

`workflow_start`・`workflow_next`・`workflow_status`・`workflow_approve`・`workflow_list`・`workflow_complete_sub` の各MCPツールが返すレスポンスのJSONスキーマは今回の修正対象外である。
`workflow_record_test_result` ツールのレスポンス構造（成功時の応答オブジェクトおよび失敗時のエラーオブジェクト形式）も変更しない。
`workflow_capture_baseline`・`workflow_get_test_info`・`workflow_set_scope` 等のユーティリティツールについても、APIインターフェースの変更は一切含まれない。
`definitions.ts` はビルド時にトランスパイルされてMCPサーバー起動時に読み込まれる内部モジュールであり、外部クライアントから直接アクセスされるAPIエンドポイントではない。
修正後に `npm run build` と MCPサーバー再起動を実施しても、Orchestratorが呼び出すAPIの型シグネチャ・パラメータ名・戻り値のフィールド名は変わらない。
FR-A〜FR-Eが影響するのは各フェーズで生成されるプロンプト文字列の内容のみであり、プロンプトを生成する `buildPrompt()` 関数の呼び出しインターフェース（引数・戻り値の型）は変更されない。
MCPツールのRPCメッセージフォーマットおよびトランスポート層の仕様も今回の修正範囲外である。
レスポンスに含まれるタイムスタンプ・タスクID・フェーズ名等のメタデータフィールドの形式や値域にも変更はない。

## 設定ファイル設計

`definitions.ts` 内の設定変更がプロンプト生成にのみ影響する理由と範囲を以下に説明する。

`definitions.ts` はフェーズ定義・モデル設定・subagentプロンプトテンプレートのコアファイルであり、MCPサーバーが起動時に一度だけ読み込む内部設定モジュールである。今回変更する設定項目は `subagentTemplate` フィールドの文字列値と `requiredSections` 配列の要素、および `buildPrompt()` 関数内のループ処理の3種類に限定される。
これらの変更はOrchestratorが各フェーズでTask toolを呼び出す際に生成するプロンプト文字列にのみ影響する。プロンプト文字列はsubagentへの指示内容として使われるものであり、MCPサーバーの状態管理・HMAC検証・フェーズ遷移ロジックには影響を与えない。
`.claude/settings.json`・`.mcp.json`・`package.json` 等の外部設定ファイルへの変更は今回のタスクに含まれない。
`artifact-validator.ts` の `PHASE_ARTIFACT_REQUIREMENTS` 定義も今回は変更しない。FR-Aの修正は `definitions.ts` 側の配列を `artifact-validator.ts` 側の定義に合わせる対症療法であり、バリデーターのコードには手を触れない。
設定変更を有効化するためには `npm run build` によるトランスパイルとMCPサーバープロセスの再起動が必須であり、ファイル編集だけでは変更が実行中のサーバーに反映されない点に注意が必要である。
FR-B〜FR-Eの修正で変更する `subagentTemplate` フィールドはビルド成果物である `dist/` ディレクトリ以下の `.js` ファイルに反映されて初めて効力を持つ。修正後に `dist/` ファイルの更新日時を確認してトランスパイルが成功したことを検証することが推奨される。
外部の設定ファイル（`.env` 等）が存在する場合も今回の修正とは独立しており、本タスクの実施によって環境変数や接続設定が変更されることはない。
将来的に `definitions.ts` と `artifact-validator.ts` の二重管理を統合する際は、設定ファイルスキーマの変更が発生するため別途UI設計が必要になるが、今回の修正では対象外である。
