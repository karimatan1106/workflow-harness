# 脅威モデル: FR修正プロセスで発生した問題の根本原因解決

## サマリー

本脅威モデルは、`definitions.ts` の文字列リテラル変更5件に対するセキュリティリスク評価を行う。

- 目的: `definitions.ts` の修正5件（requiredSections追加、禁止語間接参照化、manual_testテンプレート追記、commitテンプレート追記、testing/regression_testテンプレート追記）が引き起こす潜在的な問題を事前に特定し、対策を定義する
- 主要な決定事項: 全5件の修正はロジック非変更の文字列・配列操作のみであり、認証・暗号化・外部通信などのセキュリティ機構には一切触れない。MCPサーバー再起動を必須とする。
- 次フェーズで必要な情報: 各脅威の残存リスクレベルと対応策。ビルド確認とMCPサーバー再起動の手順確認。
- 識別された脅威数: 5件（Medium 2件、Low 3件）。全てのリスクは実装フェーズの完了条件として定義された対策により許容可能なレベルに低減できると判断する
- 最重要リスク: MCPサーバー再起動忘れ（脅威1）とTypeScriptコンパイルエラー（脅威2）であり、実装フェーズの完了条件として再起動とビルド確認を義務付けることで対応する

---

## 変更概要と対象範囲

今回の変更対象は `workflow-plugin/mcp-server/src/phases/definitions.ts` 単一ファイルである。
変更の性質は以下の5件に集約される。

- 変更1: `code_review` サブフェーズの `requiredSections` 配列に `'## パフォーマンス'` を1件追加する
- 変更2: `buildPrompt()` 関数内の禁止語列挙ループを、グループ名称による間接参照に書き換える
- 変更3: `manual_test` フェーズの `subagentTemplate` 文字列に重複行回避のガイダンスを追記する
- 変更4: `commit` フェーズの `subagentTemplate` 文字列にサブモジュール対応手順を追記する
- 変更5: `testing` および `regression_test` フェーズの `subagentTemplate` にAPI制約の説明を追記する

実行環境はローカルの開発ツールとして動作するMCPサーバーであり、インターネット公開されるサービスではない。
全件がプロンプト文字列の内容変更であり、認証・認可・暗号化・外部通信などのセキュリティ機構には一切触れない。

---

## 保護対象の資産

このタスクで変更するファイルと、それが守るべき価値を整理する。

- `definitions.ts` の `requiredSections` 配列: code_reviewフェーズでsubagentが作成する成果物の必須セクション定義であり、バリデーターとの整合性が品質保証の基盤となる
- `definitions.ts` の `buildPrompt()` 関数: 全フェーズのsubagentに渡すプロンプトを生成する中枢であり、禁止語がプロンプトに含まれると連鎖的なバリデーション失敗を引き起こす
- `definitions.ts` の各フェーズ `subagentTemplate`: subagentの動作指針を定義する文字列であり、誤った指示はsubagentの成果物品質と操作安全性に影響する
- subagentの成果物品質: バリデーション通過率という形で現れるシステム全体の出力品質
- Gitサブモジュールの操作安全性: commitフェーズでのサブモジュール未ステージングによる変更損失リスク

## STRIDE評価

STRIDEフレームワークを採用し、6カテゴリの脅威について評価する。
本変更の影響範囲は `definitions.ts` の文字列定義に限定されるため、各カテゴリの評価は簡潔にまとめる。

評価対象の信頼境界は以下の通りである。
- MCPサーバープロセスとClaude Desktopプロセス間の通信路
- `definitions.ts` が生成するプロンプトとsubagentの受信データ間の境界
- ローカルファイルシステム上の `workflow-state.json` に対するMCPサーバーの読み書き操作

### Spoofing（なりすまし）評価

本変更で追加・変更される文字列はプロンプトテンプレートの内容であり、認証トークンやアイデンティティ情報を含まない。
`requiredSections` への要素追加はバリデーション仕様のプロンプト側への反映であり、なりすまし脅威の攻撃対象には該当しない。
評価: Spoofingリスクなし（リスクレベル 極めて低、認証トークンや識別子を変更しないため）

### Tampering（改ざん）評価

`buildPrompt()` の禁止語ループ削除は生成プロンプトの内容変化を引き起こすが、この変化はsubagentへの指示の改善であり不正な改ざんではない。
`subagentTemplate` への文字列追加は既存テンプレートを上書きせず連結する形式であるため、既存指示内容が意図せず削除されるリスクは低い。
評価: 影響なし（リスクレベル 低）

### Repudiation（否認）評価

本変更はプロンプトテンプレートの修正であり、監査ログや操作履歴の仕組みには関与しない。
Git履歴によって変更内容と変更者は追跡可能であるため、否認リスクは軽減されている。
評価: Repudiationリスクなし（リスクレベル 極めて低、Git追跡により否認リスクは軽減済みのため）

### Information Disclosure（情報漏洩）評価

変更2（禁止語列挙からグループ名称への変換）は、むしろプロンプト内の機密情報露出を低減する方向の変更である。
禁止語を直接列挙するより、グループ名称で間接参照することでプロンプトの内容が成果物に転記されにくくなる。
他の変更（変更3〜5のテンプレート追記）はサブモジュール操作手順やAPIの動作説明であり、機密情報ではない。
評価: 情報漏洩リスクが若干低減する（正の効果）

### Denial of Service（サービス拒否）評価

文字列の追加によりプロンプトの全体長が増加するが、その量はサービス拒否を引き起こすほどではない。
LLMのコンテキストウィンドウ制限に対して追記内容は数百文字以内であり、実用上の問題は生じない。
評価: DoSリスクなし（リスクレベル 極めて低、追記量は数百文字以内でコンテキスト上限に影響しないため）

### Elevation of Privilege（権限昇格）評価

本変更はプロンプト文字列の修正のみであり、システム権限やファイルアクセス権限の変更は含まれない。
`definitions.ts` 自体への書き込み権限はすでにimplementationフェーズで付与されており、権限の昇格や拡大は発生しない。
評価: EoPリスクなし（リスクレベル 極めて低、権限変更を伴わない文字列定義の修正に過ぎないため）

## 脅威1: MCPサーバー再起動忘れによる変更未反映状態

### 脅威の説明

`definitions.ts` を修正してもMCPサーバーを再起動しなければ、Node.jsのモジュールキャッシュにより
古いプロンプト生成ロジックがメモリ上で動作し続ける。
実装者が再起動を忘れた場合、修正が反映されていないにもかかわらず「修正済み」と誤認する可能性がある。
過去のタスクでも再起動忘れによる問題が発生した経緯があり、このリスクは既知の課題として認識されている。

### 影響範囲

実装フェーズとテストフェーズの間で再起動が行われなかった場合、全フェーズで旧プロンプトが使用され続け、修正効果を確認できない。
特に変更1（requiredSections追加）は再起動なしではcode_reviewプロンプトに「## パフォーマンス」が追加されず、バリデーション失敗が継続する。
また変更2（禁止語間接参照化）は再起動なしでは禁止語が引き続きプロンプトに直接展開され、改善効果が現れない。

### リスクレベルと発生確率

発生確率: 中（過去の類似タスクで再起動忘れが発生した実績あり）
影響度: 高（修正効果を確認できず、後続フェーズで問題が発覚する）
総合リスク: Medium（モジュールキャッシュにより修正未反映のまま後続フェーズが進行するリスクがある）

### 対策

要件定義で定められた3ステップ手順（`npm run build` → サーバー再起動 → `workflow_status` で確認）を
実装フェーズの完了条件として明確に定義する。ビルド出力のタイムスタンプを確認してから再起動することで、ビルド成功を証拠として残す。

## 脅威2: TypeScriptコンパイルエラーによるMCPサーバー起動失敗

### 脅威の説明

`definitions.ts` の変更後にTypeScriptのコンパイルエラーが発生しないことが要求されている。
修正は文字列リテラルの変更のみのため型定義や関数シグネチャには影響しないが、
バックティック文字列内でバックスラッシュエスケープを誤るとテンプレートリテラルが壊れてコンパイルエラーになる可能性がある。
特に変更4（サブモジュール手順の追記）や変更5（API制約の文書化）はシェルコマンドの例示を含むため、
バックスラッシュ文字の扱いに注意が必要である。

### 影響範囲

ビルド失敗が発生するとMCPサーバーが起動できず、ワークフロー全体が停止する。
さらに `enforce-workflow` フックが全てのファイル編集をブロックする状態と組み合わさると、
「MCPサーバーダウン + フック全ブロック」のデッドロック状態が発生しうる。
過去のタスクでこの状態が実際に発生した経緯があり、特に注意が必要な脅威である。

### リスクレベルと発生確率

発生確率: 低（文字列リテラルのみの変更であるため型エラーは発生しない）
影響度: 高（MCPサーバー停止によりワークフロー全体が停止する）
総合リスク: Medium（発生確率は低いが影響範囲が大きいため、`npm run build` による事前確認が必須となる）

### 対策

修正後に必ず `npm run build` を実行してコンパイルエラーがないことを確認する。
バックティック文字列内のシェルコマンド例示ではバックスラッシュを使わない書き方を選択するか、
エスケープが必要な場合は前後の文字列全体を確認してから修正する。

## 脅威3: 変更2の間接参照化が不十分でリトライ成功率が低下する

### 脅威の説明

変更2（禁止語列挙ループの間接参照化）の目的は、subagentがプロンプトの内容を転記して
成果物に禁止語を混入させるリスクを低減することである。
しかし間接参照への書き換えが不十分だと、subagentがどのカテゴリの語句を避けるべきかを
具体的に理解できず、禁止パターン検出エラー時のリトライ成功率が低下する可能性がある。

### 影響範囲

禁止パターン検出エラーが発生した場合のリトライ成功率に直接影響する。
間接参照の指示が不明確だと、リトライ後の成果物でも同じ語句が残り、無限リトライに陥る可能性がある。
これはリトライ回数削減という本変更の目標に反する。

### リスクレベルと発生確率

発生確率: 低（要件定義に具体的なグループ名称と語数の記載あり）
影響度: 中（禁止パターンエラー時のリトライに限定）
総合リスク: Low（3グループ名称と語数の説明により、subagentが対象カテゴリを特定できるため軽減可能）

### 対策

修正後のプロンプトには英語系略語グループ（4語）・検討系グループ（4語）・予定仮値系グループ（4語）という
3グループの名称と語数を含め、言い換えパターンの説明も保持する。
実装後にビルドして生成プロンプトを確認し、グループ名称が正しく含まれていることを検証する。

## 脅威4: 変更4（commit手順）の誤記によるサブモジュール操作ミス誘発

### 脅威の説明

変更4は `commit` フェーズのsubagentにGitサブモジュールの操作手順を追記するものである。
追記内容にサブモジュール操作の手順が誤って記載された場合、subagentが誤ったGit操作を実行して
変更の一部をステージング漏れまたは誤コミットする可能性がある。
特に `git reset --hard` や `git push --force` などの危険なGit操作への誘導が
意図せず含まれてしまうリスクがある。

### 影響範囲

commitフェーズのsubagentのGit操作に影響する。
サブモジュール内変更のステージング手順が誤っていると、意図したコミットが作成されない。
危険なGit操作への誘導が含まれた場合は、コミット履歴の破損や変更の喪失につながりうる。

### リスクレベルと発生確率

発生確率: 低（要件定義に追記すべき内容が明確に記載されており、危険操作への誘導は含まれない）
影響度: 中（誤操作が発生した場合のGitリポジトリへの影響）
総合リスク: Low（追記内容は `git status --short` と `git add` の標準操作に限定されており、危険操作への誘導は設計上含まれない）

### 対策

追記するサブモジュール操作手順は `git status --short` の確認と `git add` によるステージングのみに限定する。
CLAUDE.mdで禁止されているGit操作（`--force`、`reset --hard` 等）への言及は追記しない。
実装後に追記内容を読み返し、危険なGit操作への誘導が含まれていないことを確認する。

## 脅威5: 変更5（API制約説明）の不正確な記述によるsubagentの試行錯誤継続

### 脅威の説明

変更5は `workflow_record_test_result` のバリデーション動作をsubagentTemplate内で説明するものである。
制約（a）の「失敗を示す語句を含む出力のブロック」と制約（b）の「同一出力の重複チェック」の説明が
不正確な場合、subagentが制約の存在を認識しながらも適切な回避策を取れず試行錯誤が継続する。
特に `regression_test` フェーズの再記録例外規定が誤って記載されると、
subagentが不要なエラー回避策を実行して操作が複雑化する可能性がある。

### 影響範囲

`testing` および `regression_test` フェーズのsubagentが `workflow_record_test_result` を呼び出す際の
成功率に影響する。説明が不正確だと改善前と同様の試行錯誤が継続し、リトライ回数が増加する。
`regression_test` フェーズでの再記録例外の誤記は、正当な操作がブロックされたと誤認させる恐れがある。

### リスクレベルと発生確率

発生確率: 低（要件定義に追記すべき内容が具体的に記載されている）
影響度: 低（testing・regression_testフェーズのみに限定）
総合リスク: Low（既存のAPIバリデーション動作を正確に文書化するだけであり、新しい挙動を導入しないため影響は限定的）

### 対策

追記内容は既存の `record-test-result.ts` に実装されているバリデーション動作を正確に記述することに限定する。
制約の説明は要件定義に記載された内容に沿って作成し、実装フェーズで追記内容の論理的整合性を確認する。
`regression_test` フェーズの例外規定については、既存コードの実際の動作と一致していることを確認してから追記する。

## 残存リスクの整理

本タスクで識別した脅威を優先度順に整理する。

- 優先度1（Medium）: 脅威1（MCPサーバー再起動忘れ）— 実装フェーズの完了条件として再起動手順を義務化することで対応
- 優先度2（Medium）: 脅威2（TypeScriptコンパイルエラー）— `npm run build` の必須実行で対応し、デッドロック状態を予防する
- 優先度3（Low）: 脅威3（間接参照の不明確さ）— 3グループ名称と語数の明記、言い換えパターン保持で対応
- 優先度4（Low）: 脅威4（commitテンプレートの誤記）— 危険なGit操作への誘導を含まない標準操作に限定することで対応
- 優先度5（Low）: 脅威5（API制約説明の不正確さ）— 既存コードの動作と照合した正確な記述で対応

全ての残存リスクはLowまたはMediumの範囲内であり、実装フェーズの完了条件として定義された対策を実施することで許容可能なレベルまで低減できる。
本変更はセキュリティ機構への影響がなく、ロジック変更も含まない文字列定義の修正であるため、追加のセキュリティ対策は不要と判断する。
