## 検証戦略

### 静的照合テストの採用根拠

本仕様書（spec.md）の「変更対象ファイル」セクションが示す通り、今回の修正は `workflow-plugin/mcp-server/src/phases/definitions.ts` の単一ファイルに集約される。変更種別はすべて文字列リテラル・配列・ループ構文に対する局所的な変更であり、外部 API 呼び出しや複雑な制御フローを持たない。このため、実行時テストよりも Readツールによる静的文字列照合の方が修正仕様との照合コストが低く、修正後の影響範囲（subagentプロンプト生成ロジックと `code_review` 必須セクション定義）を直接確認できる点で優れている。

### テスト分類とカバレッジ

修正対象の影響範囲は以下の2領域である。どちらの領域も静的照合テスト（TC-1〜TC-6）でカバーする。

ビルド成功検証（TC-7）とMCPサーバー応答確認（TC-8）はシステムレベルの検証として別途実施する。Node.js のモジュールキャッシュを無効化するためのトランスパイルと再起動が実際に機能していることを確認する目的を持つ。

## TC-1: code_review requiredSections パフォーマンス追加の検証

### 検証シナリオ概要

FR-Aの修正は `code_review` サブフェーズの `requiredSections` 配列に `'## パフォーマンス'` を末尾追加する変更である。spec.md の「変更内容の説明」セクションが示す根本原因は、`artifact-validator.ts` の `PHASE_ARTIFACT_REQUIREMENTS` が `code-review.md` に対してパフォーマンスセクションを必須と定義しているにもかかわらず、`definitions.ts` 側の配列からこの要素が欠落していた点にある。欠落によって `buildPrompt()` がsubagentプロンプトに生成する「必須セクション一覧」にパフォーマンスが表示されず、subagentが当該セクションを作成しないままバリデーション失敗が発生し続けていた。本テストではこの不一致が解消されたことを確認する。

### 検証手順（TC-1）

- 手順1: Readツールで `definitions.ts` の815〜830行目付近を読み込む
- 手順2: `code_review` サブフェーズ定義のブロック内を特定し、`requiredSections` フィールドを探す。このフィールドは phaseGuide オブジェクトの定義部に含まれている
- 手順3: `requiredSections` 配列に5要素が存在することを確認する（修正前は4要素であった）
- 手順4: 配列の末尾要素が `'## パフォーマンス'` であることを目視で照合する
- 手順5: 修正前から存在する4要素（`## サマリー`・`## 設計-実装整合性`・`## コード品質`・`## セキュリティ`）が引き続き存在することを確認し、要素の削除・順序変更がないことを検証する

### 期待する照合結果（TC-1）

対象行に以下の条件がすべて成立していること。第1条件は `requiredSections` フィールドが存在し、その配列末尾に `パフォーマンス` が含まれること。第2条件は配列要素数が5であること（修正前の4から1増加）。第3条件は `artifact-validator.ts` の `code-review.md` 定義との整合が取れること。4要素のままであれば修正未実施と判定し、TC-1 を不合格とする。

## TC-4: commit subagentTemplate サブモジュール確認手順の検証

### 検証手順（TC-4）

- 手順1: Readツールで `definitions.ts` の940〜960行目付近を読み込む

- 手順3: `git status --short` の実行指示が含まれることを確認する。全変更ファイルをリストアップする操作内容がガイダンスとして記載されているかを照合する

### 期待する照合結果（TC-4）

`commit` フェーズの `subagentTemplate` に `git status --short` の実行指示とサブモジュール確認ガイダンスが含まれていること。修正前の「変更をコミットしてください」のみの記述から、3ステップの確認手順が追記された状態になっていることが合否の判定基準となる。スコープ設定で指定された全変更のステージングを確認する旨の記述が存在するかも確認する。

## TC-8: MCPサーバー再起動後の応答確認

### 検証シナリオ概要

## 不合格時の対処方針

各テストが不合格となった場合の対処方針を以下に示す。

- TC-1不合格の場合: `definitions.ts` の821行目付近を再確認し、`requiredSections` 配列の末尾に `'## パフォーマンス'` が追加されていることを確認して再修正する。5要素への変更が確定するまで実施する
- TC-2不合格の場合: `definitions.ts` の1093〜1098行目を再確認し、ループ構文が残存している場合は削除して3行のグループガイダンス（英語系略語・検討系・予定仮値系の各行）に置き換える
- TC-3不合格の場合: `definitions.ts` の890行目付近の `manual_test` の `subagentTemplate` フィールドを再確認し、重複行回避ガイダンスと一意化の具体例を追記する。`${docsDir}/manual-test.md` の出力先パスが保持されていることも確認する
- TC-4不合格の場合: `definitions.ts` の945行目付近の `commit` の `subagentTemplate` フィールドを再確認し、`git status --short` の実行指示とサブモジュールディレクトリへの移動・個別ステージングを含む3ステップ確認手順を追記する
- TC-5不合格の場合: `definitions.ts` の862行目付近の `testing` の `subagentTemplate` フィールドを再確認し、`workflow_record_test_result` のAPI制約注意事項（exitCode=0ブロックと重複送信ブロック）を追記する
- TC-6不合格の場合: `definitions.ts` の871行目付近の `regression_test` の `subagentTemplate` フィールドを再確認し、制約説明と「このフェーズは例外として扱われる」旨の例外規定を追記する
- TC-7不合格の場合: `npm run build` のエラーメッセージを確認し、構文エラーが発生している行番号を特定して修正する
- TC-8不合格の場合: サーバーのログを確認し、起動エラーの原因（インポートエラー等）を特定して対処する
