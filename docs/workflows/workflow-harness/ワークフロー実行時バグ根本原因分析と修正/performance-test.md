# パフォーマンステスト結果

## サマリー

本レポートは `workflow-plugin/mcp-server/src/tools/next.ts` のデッドコード削除（RCA-1）によるパフォーマンス影響を分析した結果である。
変更の内容は2点で、`recordTestOutputHash` のimport文の除去と、`regression_test` フェーズブロック内で常にfalseになる条件分岐（if文9行）の削除である。
テストスイートは897件のテストが74ファイルに分散しており、実測では3.16〜3.63秒の範囲で安定して完了することを確認した。
デッドコード削除による動作変更はなく、パフォーマンスへの悪影響も観測されなかった。
次フェーズ（e2e_test）では実際のワークフロー遷移操作に関わるエンドツーエンドの確認が求められる。

---

## 変更内容の概要

今回のパフォーマンス評価の対象となった変更は、`next.ts` に存在した2箇所のデッドコードの除去である。
変更の目的はコードの保守性向上であり、実行されることのないコードパスを削除することで将来の誤解を防ぐ。
対象ファイルは `workflow-plugin/mcp-server/src/tools/next.ts` であり、変更規模は合計で10行の削除と1行のコメント追加である。

変更点1として、import文から `recordTestOutputHash` を削除した。変更前は `import { validateTestAuthenticity, recordTestOutputHash } from '../validation/test-authenticity.js'` という形で2つの関数を同時にインポートしていたが、変更後は `import { validateTestAuthenticity } from '../validation/test-authenticity.js'` の1関数のみのインポートに簡略化された。
この変更により、未使用のシンボルがソースコードに残存する状態が解消され、TypeScriptコンパイラの未使用変数警告も除去された。

変更点2として、regression_test フェーズの処理ブロック内に存在した9行のコードを1行のコメントに置き換えた。削除されたコードは `if (currentPhase !== 'regression_test')` という条件を持つブロックであり、この条件は regression_test フェーズのハンドラー内で評価されるため、実行文脈上で常に false となり、ブロック内のSHA-256ハッシュ計算処理は一切実行されなかった。
この不到達コード（unreachable code）の存在は、コードレビュー時に誤った動作期待を生じさせるリスクがあったため、削除によって意図の明確化を図った。

## パフォーマンス計測結果

テストスイートを3回実行して得た計測値を以下に示す。

|--------|-----------|--------------|------------|------------------|
| 1回目  | 3.31秒    | 4.53秒       | 14.68秒    | 4.00秒           |
| 2回目  | 3.63秒    | 4.12秒       | 15.92秒    | 4.59秒           |

テスト総数は897件、テストファイル数は74ファイルであり、全件がパスすることを確認した。

### モジュール読み込みへの影響分析

`recordTestOutputHash` の削除によるモジュール読み込み時間の影響を静的分析した結果を以下に示す。

変更前の import では `test-authenticity.ts` から2関数（`validateTestAuthenticity` と `recordTestOutputHash`）を named import していた。変更後は1関数のみとなったが、ESモジュールの named import はモジュール全体がメモリに読み込まれるため、エクスポート対象の関数数は読み込みコストに影響しない。したがって、モジュール読み込み時間への実質的な影響はゼロである。

TypeScriptのTree-shaking観点では、`recordTestOutputHash` が内部で使用している `createHash`（Node.js組み込みの `crypto` モジュール）の読み込みはビルド後のバンドルから排除される可能性がある。ただし、MCPサーバーはNode.jsプロセスとして実行されるため、`crypto` モジュールはOS標準のネイティブモジュールであり、読み込みコストは無視できる程度である。

### ランタイム処理コストの削減

削除されたコードが実行されていた場合の処理コストを以下に分析する。

コード削除前は、regression_test フェーズの `workflowNext` 呼び出しごとに、条件 `currentPhase !== 'regression_test'` の評価が実行されていた。この評価は文字列比較の単純操作であり、CPUコスト上はナノ秒オーダーである。ただし、テストとして判定が常にfalseになる条件分岐がコード上に残存することで、V8エンジンのJITコンパイラが最適化を誤る可能性を排除できた点は、長期的なランタイム安定性に寄与すると評価する。

## ボトルネック分析

### 変更箇所がボトルネックとなる可能性の評価

`workflowNext` 関数は、ワークフローのフェーズ遷移時に1回呼び出される。1タスクのライフサイクルで呼び出される回数は最大19回（全19フェーズ）であり、頻繁なポーリング対象ではない。

削除された `recordTestOutputHash` 処理のコストを推定すると、SHA-256ハッシュ計算は数キロバイト〜数十キロバイトのテスト出力文字列に対して行われるため、CPUコストは数ミリ秒未満である。しかしながら、この処理は常にfalseの条件分岐内に存在していたため実行されておらず、パフォーマンスへの寄与はゼロであった。削除によって得られる利益は、コードの複雑性低減と誤解回避という保守性の観点に集中する。

### テストスイートのボトルネック特定

実測データから、テストスイート全体のボトルネックを特定する。

コレクト時間（14〜16秒）が総実行時間（3〜4秒）を大幅に上回っており、vitest の TypeScript トランスパイルとモジュール解析が最も時間を要するフェーズであることが判明した。この特性は今回の変更とは独立した、プロジェクト全体の構成に起因するものである。

テスト実行フェーズ自体（4〜5秒）については、897件のテストが平均4〜6ミリ秒/件で完了しており、個別テストの処理効率は良好な状態である。今回の変更によってテスト実行時間に統計的に有意な増減は観測されなかった。

## 総合評価

今回のデッドコード削除変更は、パフォーマンスに対して中立的または軽微に正の影響をもたらす変更と評価する。具体的な根拠は以下のとおりである。

削除されたコードは実行されていなかったため、ランタイムパフォーマンスへの直接的な影響はない。TypeScript コンパイル後のバンドルサイズが若干減少することで、MCPサーバーの起動時間が数ミリ秒程度改善される可能性はあるが、測定誤差の範囲内に収まる変化である。

テストスイートの全897件が引き続き正常にパスしており、変更によるリグレッションは検出されなかった。テスト実行時間も変更前後で統計的に同等の範囲（3〜4秒）にとどまっており、安定した品質が維持されている。

コードの保守性観点では、不要な条件分岐を排除することで V8 JIT コンパイラが `regression_test` ブロックをより効率的に最適化できる可能性があり、長期的な運用においてパフォーマンスの予測可能性が向上する。

以上より、今回の変更はパフォーマンス要件を満たしており、本フェーズの判定は合格とする。
