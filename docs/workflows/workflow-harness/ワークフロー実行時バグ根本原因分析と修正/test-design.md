# ワークフロー実行時バグ根本原因分析と修正 - テスト設計書

## サマリー

本ドキュメントは、仕様書（spec.md）の概要セクションおよび実装計画セクションの決定事項に基づき、`workflow-plugin/mcp-server/src/tools/next.ts` に対するデッドコード削除（RCA-1）のテスト設計を定義する。
調査結果から判明した 3 つの根本原因のうち、当該テスト設計が対象とするのは高優先度と分類された RCA-1 のみである。
変更後の `next.ts` が仕様書（FR-1, FR-2, FR-3）の受け入れ基準を満たしていることを、コード静的確認・ビルド確認・動作一貫性確認の 3 観点から検証する。

**主要な決定事項:**
削除対象は `next.ts` 内の 2 箇所（regression_test ブロック内のデッドコードと import 文）に限定される。
変更はコードの動作に影響しないため、テストの焦点は「削除が正しく行われたか」と「ビルドが成功するか」の 2 点に絞る。
動作不変性は静的コードレビューおよびビルド成功をもって確認し、実行時ランタイムテストは不要と判断した。

**次フェーズで必要な情報:**
test_impl フェーズでは、本設計書に記載したテストケース TC-01 から TC-07 を静的確認スクリプトまたはシェルコマンドとして実装する。
ビルド確認（TC-01）は `npm run build` の実行結果を評価し、その他のケース（TC-02 から TC-07）はファイル内容の grep 検査で確認する。

---

## テスト背景と根拠

### spec.md の概要セクションとの対応

spec.md の概要セクションでは、当該デッドコードが `currentPhase === 'regression_test'` が確定している文脈に配置されており、条件式の評価時に必ず false となることが示されている。
この設計意図と実装の乖離を解消することが本タスクの目的であり、テスト設計はその解消が正確に行われたことを検証する観点で構成している。
変更範囲は限定的であり、波及的な影響は発生しないと spec.md の概要セクションが明確に述べている。
コードの読者（開発者）が設計意図を把握・理解しやすい状態へ改善することが、品質向上の主な便益である。

### spec.md の実装計画セクションとの対応

spec.md の実装計画セクションでは、削除対象となる行数の合計が約 10 行（コメント行 1 行 + if 文ブロック 9 行）であり、中身（ブロックの処理内容）の全体が除去対象となることが明示されている。
この行数（10 行以内）という制約は変更の最小化を保証するものであり、本テスト設計の合否判定はその制約への適合を前提とする。
実装計画の第 1 ステップから第 4 ステップまでが完了条件を構成しており、本テスト設計の TC-01 から TC-07 はその各ステップの成果物に対応している。
高優先度に分類された RCA-1 の修正完了を確認することが、本テスト設計の最終的な目標である。

### デッドコードの構造的問題

spec.md の「問題の根拠」セクションによれば、削除対象のブロックはコメントに「スキップ」と記載しているにもかかわらず、ブロックの処理内容がハッシュチェックの実行を試みる構造であった。
この問題は可読性の低下を招き、開発者がコードを読んだ際に処理内容を誤解するリスクがあった。
安全にデッドコードを除去するために、削除後の箇所には設計意図を示すコメント行追加が必要とされており、TC-04 がその確認を担当する。
`outputhashes` などの変数名（taskState.testOutputHashes 等）や testStrict 変数が当該コード内で参照されており、これらが削除後に残存していないことも合否判定の対象となる。

## テスト対象と方針

### テスト対象ファイル

本設計が対象とするファイルは以下の 1 件のみである。

- `workflow-plugin/mcp-server/src/tools/next.ts` - デッドコード削除および import 文整理の対象ファイル

仕様書の制約に従い、本タスクのスコープ配下には含まれない非テスト対象ファイルとして以下が挙げられる。
`record-test-result.ts`、`test-authenticity.ts`、`bash-whitelist.js`、`definitions.ts`、`discover-tasks.js` はいずれも今回の変更対象外である。
なお、record-test-result.ts の対応箇所（空配列渡し設計）は正常に機能していることを前提とし、next.ts 側ではその処理に関与しない設計であることが spec.md に明示されている。
これらのファイルは回帰的な影響確認の範囲外とする。

### テスト方針

本タスクの変更内容は、実行文脈上で一切実行されないデッドコードを削除する純粋なクリーンアップ作業である。
設計意図と実装の乖離を解消することが主目的であり、動作を変えるコード修正ではないため、ランタイムレベルのテストよりも静的コード確認を主体とする方針を採用する。
具体的には、削除後のファイル内容を grep 検索によって確認し、期待するコードパターンの存在・非存在を確認する手法をとる。
ビルド成功確認は `npm run build` の終了コード（0 = 成功）を評価し、構文エラーを含むトランスパイルエラーが発生しないことを検証する。
非機能要件（可読性向上、保守性、コードスタイルの一貫性）への適合もテスト観点として取り上げ、テストカバレッジの範囲に含める。
変更は限定的な内容（行数は合計 10 行以内の削除と 1 行追加）であり、MCPサーバー再起動は必須ではないが、ビルド後の動作確認として実施することが望ましい。

### テスト対象外の確認（スコープ外定義）

本テスト設計は、調査結果と要件定義書の判断に従い、RCA-2 および RCA-3 に関連する問題の検証を明示的にスコープ外とする。

RCA-2（修正対象外、中優先度）は `bash-whitelist.js` の `hasRedirection` 関数とブラックリスト regex パターンの間のスペース要求の差異という不整合に関する問題である。
この差異はリダイレクト検出の用途とコマンド単位のマッチ用途の違いに由来し、変更による副作用リスクが便益を上回ると判断されたため今回のスコープには含まれない。
したがって、リダイレクト判定のロジックや正規表現パターンのマッチ単位の妥当性を検証するテストケースは作成しない。

RCA-3（修正対象外、低優先度）は `task-index.json` と `workflow-state.json` の同期欠如という問題に関するものであり、解決済みのフェーズブロック問題として分類されている。
この同期欠如は TTL への短縮（FIX-2 対策）と commit/push フェーズの早期リターン処理によって実用上の影響が解消済みである。
TTL キャッシュ管理のロジックは `discover-tasks.js` が担い、根本修正は `state-manager.ts` および MCP サーバー全体への大規模変更を要することから、今回は検証スコープから除外する。

## テストケース詳細

### TC-01: コンパイル確認テスト

**テスト目的:**
本仕様書の FR-3 に基づき、`next.ts` のデッドコード削除および import 文整理後に `npm run build` が成功することを確認する。
構文エラーを含むトランスパイルエラーが発生しないことが合格条件であり、変更内容の全体像が TypeScript のコンパイラによって正しく受け入れられることを保証する。
リンターによる警告（未使用 import など）も合わせて確認し、コードスタイルの観点でも問題がないことを検証する。
ビルドエラーが発生した場合は、import 文の記述内容を再確認してから再実行することを推奨する。

**前提条件:**
`next.ts` の変更（FR-1 デッドコード削除 + FR-2 import 整理）が完了している状態であること。
`workflow-plugin/mcp-server/` ディレクトリに `package.json` が存在し、`build` スクリプトが定義されていること。
`next.ts` が変更対象のコアモジュール（`artifact-validator.ts`、`definitions.ts`、`state-manager.ts`）ではなく tools ディレクトリ配下のファイルであることを確認済みであること。
Node.js および npm が実行環境にインストール済みであること。
変更範囲が限定的（合計 10 行以内）であることを事前に再確認し、削除範囲の開始行・終了行が仕様書（spec.md 実装計画セクション）と一致していること。

**手順:**

第 1 ステップとして、作業ディレクトリを `workflow-plugin/mcp-server/` に移動する。

第 2 ステップとして、以下のコマンドを実行する。

```bash
cd workflow-plugin/mcp-server && npm run build
```

第 3 ステップとして、コマンドの終了コードを確認する。終了コードが 0 であればビルド成功と判断する。

第 4 ステップとして、標準エラー出力に TypeScript のコンパイルエラーメッセージが含まれないことを確認する。

**期待結果:**
`npm run build` が終了コード 0 で完了し、`dist/` ディレクトリ以下に生成ファイルが出力されること。
標準エラー出力に `error TS` で始まる行が存在しないこと。
未使用 import に関するリンターの警告（`'recordTestOutputHash' is declared but its value is never read` 等）が出力されないこと。
トランスパイルエラーが一件も発生しないことがビルド成功の最低条件である。

### TC-02: import 文整理の確認テスト

**テスト目的:**
本仕様書の FR-2 に基づき、`recordTestOutputHash` が import リストから除去（未使用参照の一括削除）されており、かつ `validateTestAuthenticity` の import が維持されていることを確認する。
両条件が同時に成立することが合格条件である。
`recordTestOutputHash` は条件分岐内に存在したデッドコードブロック内でのみ参照されており、その削除によって未使用参照となるため import 文から除去が必要である。
spec.md の概要セクションに示される通り、当該関数の呼び出しは次.ts 内のハッシュチェックスキップ処理（ハッシュチェックスキップを意図するがデッドコードになっていた部分）に集約されていたことが除去の根拠である。

**前提条件:**
`next.ts` の変更（FR-1 デッドコード削除 + FR-2 import 整理）が完了している状態であること。
ファイルが UTF-8 エンコーディングで保存されていること。

**手順:**

第 1 ステップとして、`next.ts` 内で `recordTestOutputHash` という文字列を grep 検索し、出現件数を確認する。

```bash
grep -n "recordTestOutputHash" workflow-plugin/mcp-server/src/tools/next.ts
```

第 2 ステップとして、`next.ts` 内で `validateTestAuthenticity` という文字列を grep 検索し、出現件数を確認する。

```bash
grep -n "validateTestAuthenticity" workflow-plugin/mcp-server/src/tools/next.ts
```

第 3 ステップとして、両コマンドの出力内容を照合し、期待結果と一致するか判定する。

**期待結果:**
第 1 のコマンド（`recordTestOutputHash` の検索）が 0 件のマッチを返すこと。
第 2 のコマンド（`validateTestAuthenticity` の検索）が 1 件以上のマッチを返すこと。
`test-authenticity.js` の import 行に `validateTestAuthenticity` のみが含まれ、`recordTestOutputHash` が含まれていないこと。

削除によって設計意図と実装の乖離が解消されることが本テストの検証目標である。
spec.md の実装計画セクションで示された削除範囲（開始行・終了行）が正確に実施されたことを再確認する観点も含む。

**前提条件:**

### TC-04: スキップコメントの確認テスト

**テスト目的:**
本仕様書の FR-1（削除後に残すコメントの仕様）に基づき、デッドコード削除後の位置に設計意図を示すコメントが追加されていることを確認する。
このコメントは、ハッシュ重複チェックの管理責任が `record-test-result.ts` の対応箇所にあることを将来のメンテナーに伝え、同様のデッドコードを再導入するリスクを低減するためのものである。
コードスタイルの観点では、testing フェーズのハッシュチェックスキップ処理と同様の形式で記述されることを要求する。
コードの読者（開発者）への配慮として、「next.ts 側では関与しない」という設計意図を明示することが本コメントの目的であり、その文言が適切に表現されていることを検証する。

**前提条件:**
`next.ts` の変更（FR-1 デッドコード削除 + FR-2 import 整理）が完了している状態であること。
スキップコメント（行追加によるコメント）が regression_test フェーズの処理ブロック内に追加されていること。

**手順:**

第 1 ステップとして、`next.ts` 内で `record-test-result.ts` という文字列をコメント行として含む行を検索する。

```bash
grep -n "record-test-result.ts" workflow-plugin/mcp-server/src/tools/next.ts
```

第 2 ステップとして、コメント行の内容が「ハッシュ重複チェックは record-test-result.ts 側で対処済みのためスキップ」という趣旨を含むことを確認する。

第 3 ステップとして、コメント行の前後のコードが regression_test フェーズのハンドラー内にあることを、ファイル全体のコンテキストで確認する。

**期待結果:**
コメント行が 1 行以上存在し、`// ハッシュ重複チェックは record-test-result.ts 側で対処済みのためスキップ` という内容（または同等の意味内容）が含まれること。
コメント行が regression_test フェーズの処理ブロック内（`parallel_verification` への遷移処理より前）に配置されていること。
testing フェーズのスキップコメント形式（`//` による単行コメント）と統一された形式で記述されており、コードスタイルからの逸脱がないこと。
コメントの保守性が高く、設計意図が明確に読み取れる内容であること。

設計審査を通じてコードの構造的な等価性を検証することが合格条件である。
デッドコードブロックは到達不能コードであり、削除しても動作の変化は生じない点を設計レビューとして確認する。
spec.md の NFR-2 セクションに記された「失敗条件および解決済みのフェーズブロック問題も影響を受けない」という記述が実装レベルで守られていることを、grep による再実行で検証する。

**前提条件:**
`next.ts` の変更（FR-1 デッドコード削除 + FR-2 import 整理）が完了している状態であること。
変更前の `next.ts` の内容が git history または別途記録として参照可能であること。

**手順:**

第 1 ステップとして、`next.ts` の regression_test フェーズ終了処理（`parallel_verification` への遷移を指示する部分）を確認する。

### TC-06: 未使用参照の排除確認テスト

**テスト目的:**
本仕様書の FR-2 および NFR-1（可読性向上）に基づき、デッドコード削除後に `next.ts` 内に未使用参照が残存していないことを確認する。
`recordTestOutputHash` は regression_test ブロック内の条件分岐内（常に false となる評価時に到達しない箇所）でのみ参照されており、デッドコード削除後は一切参照されない未使用参照となる。
この未使用参照をそのまま残すと、コードの品質と保守性を損ない、将来的にリンターの警告が発生する原因となるため、import 文からの除去が必要である。
仕様書の変更範囲として集約された 2 つの変更点（if ブロック削除 + import 整理）が両方とも実施されたことを確認する観点である。

**前提条件:**
TC-01 のビルド確認が完了し、成功していること。
`next.ts` の変更（FR-1 デッドコード削除 + FR-2 import 整理）が完了している状態であること。

**手順:**

第 1 ステップとして、`next.ts` 内の import ブロック全体を確認し、`test-authenticity.js` からの import 行を特定する。

```bash
grep -n "test-authenticity" workflow-plugin/mcp-server/src/tools/next.ts
```

第 2 ステップとして、import 行の内容が `validateTestAuthenticity` のみを名前付き import で取得し、`recordTestOutputHash` を含んでいないことを目視確認する。

第 3 ステップとして、ファイル全体を対象に `recordTestOutputHash` の出現箇所を全文検索し、import 文・関数呼び出し・コメント内を含め 0 件であることを確認する。

## リスクと対応策

### リスク 1: import 文の部分削除ミス

削除対象が `recordTestOutputHash` のみであるにもかかわらず、`validateTestAuthenticity` も誤って削除されるリスクがある。
この場合、TC-02 の第 2 ステップと TC-07 が同時に失敗し、ビルド時にビルドエラーとして `validateTestAuthenticity is not defined` が発生する。
対応策として TC-02 と TC-07 の実施を TC-01（ビルド）より先に行い、import 文を目視確認してから build を実行することを推奨する。
spec.md の実装計画セクションでは、この変更が「正確に」実施されることが求められており、import 行の削除前に除去対象を再確認することが安全な実施方法として示されている。

### リスク 2: デッドコードブロックの削除範囲の過誤

削除対象が if ブロック全体（コメント行を含む合計約 10 行）であるにもかかわらず、一部の行が削除されず残存するリスクがある。
残存した行が不完全なコードとなる場合はトランスパイルエラーが発生し TC-01 が失敗する。
残存した行がコメント行のみの場合は TC-03 の検索で検出されない可能性があるため、TC-03 では `existingHashes` や `hashResult` の残存確認も補足的に実施する。
削除範囲の開始行・終了行を spec.md 実装計画セクションで事前に把握しておくことで、この過誤リスクを最小化できる。

### リスク 3: スキップコメントの配置誤り

コメントが regression_test フェーズのブロック外（たとえば testing フェーズのブロック内）に誤って配置されるリスクがある。
TC-04 の第 3 ステップでコメントのコンテキスト（前後のコード）を確認することで、この誤りを検出できる。
コードスタイルの一貫性も TC-04 の確認観点として含まれるため、testing フェーズとの形式差異も合わせて検出できる。
コードスタイルからの逸脱がないかどうかを、開発者の視点から丁寧に確認することが望ましい。

### リスク 4: ビルドキャッシュによる変更未反映

`npm run build` がキャッシュを参照し、変更前の状態でビルドされる可能性がある。
この場合、TC-01 が合格しても実際の変更が反映されていない状態となり、正常動作の確認が不完全となる。
対応策として、ビルド前にキャッシュクリアが必要か確認し、`dist/` 以下の生成ファイルの更新日時が変更後であることを TC-01 の補足確認として実施する。
MCPサーバー再起動は今回の変更では必須ではないが、動作確認として再起動後に動作確認を行うことで、ビルドキャッシュ起因の問題を完全に排除できる。
