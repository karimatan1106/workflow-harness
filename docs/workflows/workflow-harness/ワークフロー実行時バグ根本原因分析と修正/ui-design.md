# UI設計書: ワークフロー実行時バグ根本原因分析と修正

## サマリー

本タスクはMCPサーバーの内部コード修正（`workflow-plugin/mcp-server/src/tools/next.ts`のデッドコード削除）を実施する。
ユーザー向けのUIやAPIレスポンス形式に影響する変更は含まれないため、エンドユーザーが使用するワークフローコマンドと動作には変化がない。
修正対象はregression_testフェーズハンドラーの未実行コードブロック（10行以内）の削除と、対応するimport文の整理のみである。
次フェーズではこの分析結果を踏まえて実装を進め、変更後にビルド確認を実施することで品質を保証する。
CLIインターフェース・エラーメッセージ・APIレスポンス・設定ファイルのいずれについても、本修正に起因する変更は発生しないことが以下の各セクションで確認されている。

---

## UIへの影響分析

### ユーザー操作フローの不変性

本修正は内部ロジックの整理であり、ユーザーが操作するワークフローコマンドおよびレスポンス形式に変更は一切ない。
`/workflow next` コマンドのレスポンスは修正前後で同一の構造を維持し、ユーザーが受け取るメッセージやステータス情報に変化は発生しない。
regression_testフェーズから並列検証フェーズへの遷移動作は変更前後で完全に同一であることが仕様書で確認されている。
ユーザーがコマンドラインで実行するワークフロー制御コマンド（status、next、complete-subなど）の動作に影響しない修正である。

### エラーメッセージの不変性

削除されるコードブロックは条件式が常にfalseであるため、ブロック内で定義されているエラーメッセージは実行環境で一度も送出されていない。
したがって本修正によって、ユーザーが受け取るエラーメッセージの種類や内容に変更は発生しない。
testingフェーズおよびregression_testフェーズで実装されている正規の検証ロジックは変更されず、それらが送出するエラーメッセージも修正前後で同一である。
削除後も、ユーザーが受け取るすべてのエラーメッセージはvalidateTestAuthenticityおよびrecord-test-result.ts側で制御される。

### regression_testフェーズのユーザー操作フロー

regression_testフェーズでユーザーが実行する操作は変更されない。
`workflow_next`コマンドの入力パラメータ（既存テストのベースライン結果など）は修正前後で同一の形式を維持し、ユーザーが入力値を変更する必要は生じない。
フェーズから次のフェーズへの遷移条件が修正前後で同一であることが仕様書で確認されているため、ユーザーが経験するワークフロー進行に違いはない。
削除されるコードは「常に実行されない」デッドコードであるため、その削除によってユーザーの操作が新たに拒否されたり、予期しない動作が発生する可能性はない。

## エラーメッセージ設計

### 削除対象コード内のエラーメッセージ分析

削除されるデッドコードブロック内には「リグレッションテスト出力が以前と同一です（コピペの可能性）。実際にテストを実行してください。」というメッセージが定義されている。
このメッセージは`currentPhase !== 'regression_test'`という条件式が常にfalseになる文脈に配置されているため、ユーザーが実際のワークフロー実行中にこのメッセージを受け取ったことは一度もない。
削除後もメッセージ体系に空白が生じることはなく、同等の目的（重複テスト出力の検出）はrecord-test-result.tsの空配列渡し設計によって別の経路で実現されている。
エラーメッセージカタログ（ドキュメント・コードコメント・ユーザーガイドに記載されたメッセージ一覧）に本メッセージが登録されていないことを確認済みであり、ドキュメント更新は不要である。
本削除によってユーザーが受け取るエラーメッセージの集合は変化せず、新規メッセージの追加も既存メッセージの削除も発生しない。

### 既存エラーメッセージの保護

testingフェーズで定義されているエラーメッセージ（validateTestAuthenticity関数が返すメッセージ群）は本修正の対象外であり、内容・形式・送出条件のいずれも変化しない。
regression_testフェーズ処理で送出される可能性のあるエラーメッセージ（バリデーション失敗・リトライ上限超過など）はrecord-test-result.ts側で制御されており、next.tsのデッドコード削除による影響を受けない。
ユーザーが実際に体験するエラーメッセージの内容・日本語文言・送出タイミングはすべて修正前後で同一である。
エラーハンドリングの改善が必要と判断された場合は、本修正完了後に別タスクとして対応すること。

レスポンスに含まれるフェーズ情報（現在フェーズ・次フェーズ・残りフェーズ数・サブフェーズ一覧など）の形式はすべて保持される。

## import文削除による外部影響の分析

### validateTestAuthenticityの継続使用

import文から削除されるのは`recordTestOutputHash`関数のみであり、`validateTestAuthenticity`関数はtestingフェーズおよびregression_testフェーズで引き続き使用される。
修正後のimport文は`import { validateTestAuthenticity } from '../validation/test-authenticity.js';`という形式になり、既存の検証ロジックに影響しない。
testingフェーズのハッシュチェックスキップ処理（line 267-270）でvalidateTestAuthenticityが呼び出されており、そのコード経路は変更されない。

### test-authenticity.js側への波及

test-authenticity.js内の`recordTestOutputHash`関数定義は、他のモジュールから呼び出されていないため、next.tsでのimport削除は外部への影響を生じない。
修正によってtest-authenticity.jsファイル自体の変更は発生せず、エクスポート対象の関数群も不変である。
将来的にrecordTestOutputHashが別のモジュールから参照される場合でも、test-authenticity.jsの定義は維持されているため機能性に問題はない。
