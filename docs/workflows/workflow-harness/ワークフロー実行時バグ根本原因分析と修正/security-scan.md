# セキュリティスキャン結果 - ワークフロー実行時バグ根本原因分析と修正

## サマリー

本スキャンは、`next.ts` のデッドコード削除（RCA-1）と `verify-sync.test.ts` の期待値修正を対象とした変更内容について、セキュリティ観点での検証を実施した。

変更対象ファイルは以下の3つである。
- `workflow-plugin/mcp-server/src/tools/next.ts`（`recordTestOutputHash` import 削除 + デッドコード if ブロック削除）
- `workflow-plugin/mcp-server/src/verify-sync.test.ts`（TC-1-3 の期待値 26 → 25 に修正、TC-6-4 の閾値 26 → 25 に修正）
- `workflow-plugin/mcp-server/src/__tests__/verify-sync.test.ts`（同上の修正が __tests__ ディレクトリ版にも適用）

スキャン結論として、今回の変更はセキュリティ機能を損なわず、認証・認可フローへの影響もない。テスト期待値の変更はバリデーション要件の緩和ではなく、モックデータの正確な反映に起因する修正であることを確認した。

主要な決定事項は「変更によるセキュリティリスクなし」である。HMAC 整合性チェックを担う `state-manager.ts` は今回のスコープに含まれておらず、テスト真正性を検証する `validateTestAuthenticity` 関数も変更されていない。

次フェーズで必要な情報として、`recordTestOutputHash` の import 削除後も `test-authenticity.ts` 内の同関数の実装は維持されており、将来の再利用が可能な状態であることを記録しておく。

## 脆弱性スキャン結果

### 変更 1: `next.ts` の `recordTestOutputHash` import 削除

対象コードは `next.ts` の冒頭 import 文から `recordTestOutputHash` を除去するものである。
この関数は `test-authenticity.ts` に定義された SHA-256 ハッシュ計算関数であり、テスト出力の重複提出を防止するセキュリティ機能を担う。

スキャン観点として、import 削除が当該セキュリティ機能の無効化を意味するかどうかを検証した。
`next.ts` の regression_test フェーズブロックを精査した結果、削除されたコードは以下の構造であることが判明した。

```typescript
// (削除前)
if (currentPhase !== 'regression_test') {
  // ハッシュチェックロジック
}
```

この条件式は regression_test フェーズハンドラーの内側で評価されるため、`currentPhase` は必ず `'regression_test'` であり、条件が真になることは一切ない。よって、ハッシュチェックロジックは実行不可能なデッドコードであり、削除してもセキュリティ機能の変化は発生しない。

スキャン結果: 脆弱性なし、認証・認可フローへの影響なし。

### 変更 2: `next.ts` の regression_test フェーズ内デッドコード if ブロック削除

削除対象は `currentPhase !== 'regression_test'` という条件が常に false になる if ブロック全体（コメント行を含む）である。
セキュリティ観点での懸念点は「ハッシュ重複チェックが意図せず有効化されていたのではないか」という点であった。

実際のコードを参照すると、削除後の regression_test ブロックには `validateTestAuthenticity` の呼び出しが明示的に維持されており（line 319-335）、テスト出力のフレームワーク構造確認・タイムスタンプ検証・テスト数チェックは引き続き実行される。

`recordTestOutputHash` に相当するハッシュ重複チェックは `record-test-result.ts` 側で対処済みとコメントに記載されており（line 336）、今回の削除はその設計方針を明文化したに過ぎない。よって、テスト捏造防止の観点でセキュリティが低下する変更ではないと判断する。

スキャン結果: 脆弱性なし、テスト真正性検証は維持されている。

### 変更 3: `verify-sync.test.ts` の TC-1-3 期待値修正（26 → 25）

`src/verify-sync.test.ts` の TC-1-3 では「全フェーズ展開後の件数」を検証するテストケースが修正された。
旧テストは「非並列フェーズ 15 件 + サブフェーズ 11 件 = 26 件」を期待していたが、修正後は「非並列フェーズ 14 件 + サブフェーズ 11 件 = 25 件」を期待する。
この差分は `completed` フェーズをモックデータに含めるかどうかの違いに起因する。

セキュリティ観点での懸念点は「フェーズ件数の削減がワークフロー制御のバイパスに繋がるか否か」である。
`completed` フェーズの扱いを変更したのではなく、テストのモックデータ構成を正確なフェーズ定義に合わせたものであり、実際のワークフロー制御ロジックは変更されていない。

スキャン結果: 脆弱性なし、フェーズスキップや不正遷移のリスクはない。

### 変更 4: `verify-sync.test.ts` の TC-6-4 閾値修正（26 → 25）

`src/verify-sync.test.ts` および `src/__tests__/verify-sync.test.ts` の TC-6-4 は「フェーズ数不足時のバリデーションエラー検証」である。
旧テストは `toBeLessThan(26)` を期待し、修正後は `toBeLessThan(25)` を期待する。

この修正は TC-1-3 の変更と整合性を保つためのものであり、バリデーション閾値の実質的な引き下げではない。
「25 件未満で終了コード 2」という要件は変わっておらず、テストのモックデータが正確に 24 件を生成することを検証するように修正されただけである。

セキュリティ観点での懸念点は「バリデーション閾値の緩和によってフェーズ定義の欠落を見逃すリスクがあるか否か」である。
モックデータの構成を `deploy` を省略した 13 件の非並列フェーズ + 11 件のサブフェーズ = 24 件に変更しており、`toBeLessThan(25)` という閾値チェックは 24 < 25 として依然として有効である。
より少ない閾値への変更が実際のバリデーション要件を弱体化させることはなく、テストロジックの整合性を確保するための調整に留まる。

スキャン結果: バリデーション要件の弱体化なし、脆弱性なし。

## 検出された問題

### 重大度: 高（Critical）

今回の変更スコープにおいて重大度が高い問題は検出されなかった。

### 重大度: 中（Medium）

今回の変更スコープにおいて重大度が中程度の問題は検出されなかった。

### 重大度: 低（Low）

#### 観察事項 L-1: `recordTestOutputHash` は `test-authenticity.ts` にのみ残存

`next.ts` の import から `recordTestOutputHash` が削除されたが、この関数は `test-authenticity.ts` 内の実装として残存している（line 177-193）。現時点では `record-test-result.ts` が呼び出す仕組みとなっており、機能は維持されている。将来の開発者が `next.ts` 内での呼び出し復活を試みる際に、設計意図が伝わりやすいよう `record-test-result.ts` 側のコメントを充実させることが望ましい。ただし、セキュリティ上の緊急性はなく対応は任意である。

#### 観察事項 L-2: `TEST_AUTHENTICITY_STRICT` 環境変数によるバイパス

`next.ts` の line 254 に `process.env.TEST_AUTHENTICITY_STRICT !== 'false'` という環境変数制御が存在する。
今回の変更はこの部分に触れていないが、スキャン対象コードを読む中で発見した観察事項として記録する。
本番環境でこの変数を `false` に設定すると `validateTestAuthenticity` の失敗を無視して進行できる点は既知のリスクである。今回の変更によってこの動作に変化はない。

### 問題なし（該当なし）

import の削除がセキュリティ機能の実質的な無効化につながることはなく、テスト期待値の変更がバリデーション要件を弱体化させることも確認されなかった。今回の変更は仕様書 spec.md の記述と整合しており、コードクリーンアップとテスト修正の範囲に収まっている。

## HMAC 整合性チェックへの影響分析

HMAC 整合性チェックは `state-manager.ts` で実装されており、ワークフロー状態ファイル（`workflow-state.json`）の改ざん防止に使用される。
具体的には、状態遷移のたびに `stateIntegrity` フィールドが SHA-256 ベースの HMAC-SHA256 ハッシュで更新され、次回のフェーズ遷移時に検証される仕組みである。

今回の変更対象は `next.ts` のデッドコード削除および `verify-sync.test.ts` のテスト期待値修正であり、`state-manager.ts` には一切触れていない。したがって、HMAC 整合性チェックのロジックは変更前と完全に同一であることを確認した。
HMAC 生成に使用される秘密鍵（`STATE_SECRET` 環境変数）も今回の変更スコープには含まれず、鍵管理に関する問題は発生しない。

`next.ts` は `stateManager.updateTaskPhase()` などのメソッドを呼び出すことで間接的に HMAC 更新をトリガーする構造になっているが、今回削除されたデッドコードはこれらのメソッド呼び出しを含まないため、HMAC の生成・検証フローに変化はない。
デッドコードブロック内には状態書き込み処理が存在しないことをコードレビューで確認しており、削除による HMAC 更新タイミングのズレも生じない。

`verify-sync.test.ts` のテスト期待値変更（26 → 25）は、HMAC 検証ロジックとは独立したフェーズカウント検証に関するものであり、`stateIntegrity` フィールドの計算・照合処理に影響を与えないことを確認した。
また、`stateManager` のモックを使用するテストが期待値を調整した場合でも、本番の HMAC チェックは実際の `state-manager.ts` 実装が担うため、テストの期待値変更が本番セキュリティ機能の変化を意味することはない。

## 結論

今回スキャンした3ファイルの変更について、セキュリティ上の問題は検出されなかった。
デッドコード削除は認証・認可フローに影響せず、テスト期待値の修正はセキュリティバリデーションの弱体化を伴わない。
HMAC 整合性チェックを担う `state-manager.ts` は今回のスコープに含まれておらず、ワークフロー状態の改ざん防止機能は変更後も完全に維持されている。
テスト真正性を検証する `validateTestAuthenticity` 関数の呼び出しも `next.ts` の regression_test フェーズハンドラー内に明示的に残存しており、テスト出力の信頼性確保の仕組みは損なわれていない。
観察事項として記録した `TEST_AUTHENTICITY_STRICT` 環境変数によるバイパスリスクは既知の設計であり、今回の変更によってそのリスクが新たに増大することはない。
以上の分析から、今回の修正はコードクリーンアップとテスト修正の範囲に収まっており、変更全体として安全であると最終判断する。
