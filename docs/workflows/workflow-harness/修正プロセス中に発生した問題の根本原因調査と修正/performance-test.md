計測対象3: Vitest テストスイートの実行時間（`npx vitest run` の完了所要時間・全テスト数）

### 計測手法

`definitions.ts` サイズ差分は `git show` コマンドでコミット 153587a（FR-20/FR-22/FR-23 適用直前）と HEAD（02be51b）の両コミット時点のバイト数を計測し、差分を算出した。
ビルド時間は `npm run build` を2回連続実行し、シェル組み込みの `date +%s%3N` でミリ秒単位の経過時間を計測した。
テスト実行時間は `npx vitest run` を3回実行し、vitest 内部レポートの Duration 値（transform・collect・tests・prepare フェーズの合計）とシェルの wall-clock 時間の両方を記録した。

## ボトルネック分析

### 特定されたボトルネックの名称

今回の計測範囲（ファイルサイズ・ビルド時間・テスト実行時間）においてボトルネックは検出されなかった。
以下、ボトルネックが存在しない根拠とともにリスク評価を記録する。

### 原因分析

テンプレート文字列サイズ増加（+3,157 バイト）の原因は FR-2（manual_test テンプレートへの前提条件一意化ガイダンス追記）および FR-3（e2e_test テンプレートへの前提条件一意化ガイダンス追記）による文字列リテラルの拡張である。
これらのガイダンス文字列は Node.js モジュール読み込み時に一度だけメモリに格納されるため、リクエスト処理ごとのオーバーヘッドは発生しない。
ビルド時間（約2.1秒）の大部分は TypeScript コンパイラ（tsc）の型チェック処理が占めており、テンプレート文字列の長さとは直接的な相関がないことを確認した。

### 影響範囲の評価

テンプレート文字列サイズの増加が影響する範囲として、MCP サーバーの起動時メモリ使用量が 3 KB 程度増加する可能性があるが、実用上の閾値（512 MB 程度）に対して無視できる水準である。
`dist/phase-definitions.cjs`（172,168 バイト）はサーバー起動時に一度だけ読み込まれる静的データファイルであり、リクエスト数に応じてメモリ使用量が増加する性質ではない。
テストスイートの実行時間（wall clock 約4.7秒）は collect フェーズ（15.51秒）と prepare フェーズ（16.99秒）に時間がかかる構造となっているが、これらは vitest のモジュール解析・依存関係構築のためであり、テスト数増加（950件）に比例する傾向がある。

### 改善提案の具体案

現時点でのボトルネックは検出されていないが、将来の改善候補として以下を記録する。
候補1: テンプレート文字列が今後さらに増加した場合（例: +50KB 以上）、外部 JSON または YAML ファイルへの分離を検討することでビルド時の tsc 処理対象コードを削減できる。
候補2: vitest の collect フェーズ（15秒超）を短縮するために、テストファイルの分割・並列実行設定（`threads: true`）の導入を検討できる。
候補3: definitions.ts の行数が 1,595 行から大幅に増加した場合、フェーズ定義をファイル分割（例: `phases/manual-test.ts`, `phases/e2e-test.ts`）して管理性を向上させることが有効である。

### 優先度の判定

ボトルネックが検出されなかったため、今回の改善提案はすべて優先度「低」（将来の技術負債管理として記録する水準）と判定する。
現在の `definitions.ts` サイズ（167,362 バイト・1,595 行）はまだファイル分割を必要とする規模ではなく、可読性・保守性の観点から単一ファイルのままで問題ない。
vitest の collect フェーズ最適化についても、現状の CI 実行時間（約4.7秒）は開発ワークフローへの支障がなく、即時対応の必要性はない。

## 総合評価

- 全体的なパフォーマンス達成状況: 計測した3指標（ファイルサイズ・ビルド時間・テスト時間）すべてが許容閾値内に収まり、FR-1・FR-2・FR-3 の修正による実用上のパフォーマンス劣化は検出されなかった
- 性能要件との比較: ビルド時間（実測 2.07秒）は許容閾値（5秒）の 41% 相当であり、テンプレート文字列増加後も 2.9 秒の余裕が存在する。テスト実行時間（実測 4.7秒 wall clock）は CI 許容閾値（60秒）の 7.8% 相当であり、現状のスケールで十分な余裕がある
- 特記すべき改善項目: vitest の collect フェーズに 15 秒以上かかっており、テストファイル数（77ファイル）増加に伴い collect 時間がボトルネックになる可能性があるため、将来の監視対象として記録する
- リスク評価: テンプレート文字列の追記が今後も継続する場合、`definitions.ts` が 300 KB を超えた時点でビルド時間への影響が顕在化するリスクがある。現状（167 KB）では低リスクだが、200 KB 超過時点を警戒ラインとして設定することを推奨する
- 今後の監視方針: ビルド時間とテスト実行時間を PR マージごとに記録し、ビルド時間が 4 秒を超過した場合と wall clock が 10 秒を超過した場合を警告閾値として設定する。`definitions.ts` のファイルサイズが 200 KB を超えた場合はファイル分割の設計検討を開始すること
