  - テンプレート文字列へのCLAUDE.md禁止パターン完全リストに定義された語句の混入リスクは軽微だが、バリデーターが成果物全体をブロックするため事前確認が必要
- 次フェーズ（planning）で必要な情報: 各脅威シナリオに対する具体的な対策実装箇所（CLAUDE.mdの該当行・definitions.tsの該当テンプレートの行番号）

## 脅威シナリオ

### T-01: sessionToken制限緩和による意図しない権限昇格（CLAUDE.md修正）

CLAUDE.md厳命23番を「二層構造」に書き換える際、「Orchestratorが直接呼び出すMCPツールにsessionTokenを渡す」という記述が、subagentへの無制限なsessionToken引き渡しを正当化するように読める文面になった場合、Orchestratorが任意のsubagentにsessionTokenを渡す判断をし得る。これにより、docs_update・implementation・commit などのフェーズのsubagentが `workflow_next` を呼び出して自律的にフェーズを進めてしまうリスクが生じる。過去（2026-02-24）にdocs_update subagentがワークフロー制御を奪取した事例が実際に発生しており、この脅威は現実的かつ高影響度と評価される。

攻撃パス: 誤記の二層構造文言 → OrchestratorがsubagentへsessionTokenを広く引き渡す → subagentが workflow_next などを呼び出してフェーズを自律進行 → 承認フェーズが飛ばされ、ユーザーによる設計・コードレビューが機能しない

### T-02: CLAUDE.md修正後の既存記述との矛盾（ドキュメント整合性リスク）

厳命23番を書き換えた後、CLAUDE.md内の他箇所（「sessionTokenの取得先はworkflow_statusのみ、使用先はworkflow_record_test_resultのみ」という旧記述、またはMEMORY.mdに記録されている旧ルール）との矛盾が残った場合、OrchestratorがどのルールMを優先するか不明確になる。矛盾した記述が同一ドキュメントに存在すると、Orchestratorが誤った解釈を行う確率が上がる。

攻撃パス: 旧記述の残存 → OrchestratorがCLAUDE.mdの前後の記述を参照して誤解 → sessionTokenをsubagentに渡さない（不足）またはsubagentへ広く渡す（過剰）という誤動作が発生

### T-03: definitions.tsテンプレートリテラルの構文破壊（ビルド失敗リスク）

definitions.tsのmanual_testまたはe2e_testのsubagentTemplateはバックティックで囲まれたテンプレートリテラルである。ガイダンス追加テキストにバックティックが含まれると、テンプレートリテラルが途中で閉じられてTypeScriptコンパイルエラーが発生する。また、ドル記号+中括弧（`${`）形式の文字列がテンプレートリテラル内に素のまま記述されると、変数展開の意図しない動作またはコンパイルエラーが発生する。

攻撃パス: バックティック/ドル記号の混入 → `npm run build` 失敗 → MCPサーバーが更新できず古いテンプレートが継続稼働 → 追加ガイダンスが一切反映されない

### T-04: artifact-validator検出対象語句の混入によるバリデーター品質違反（バリデーター検出リスク）

定義テンプレートに追加するガイダンステキストに、artifact-validatorのCLAUDE.md禁止パターン完全リストに定義された語句群（英語グループ4語・日本語グループ8語）が含まれた場合、そのテンプレートを使って生成された成果物がバリデーション失敗し、subagentが延々とリトライを繰り返す可能性がある。ガイダンスの例示テキスト（「こう書いてはダメ」という悪い例の記述）にバリデーター検出対象の語句が含まれる場合が特に危険である。

攻撃パス: ガイダンス例示にバリデーター検出対象語句が混入 → subagentが例示テキストをそのまま成果物に記述 → バリデーション失敗 → リトライループが発生してフェーズ進行が停滞

### T-05: 前提条件行一意化ガイダンスの不完全性（バリデーション再発リスク）

追加するガイダンスが「TC-1など番号を付与せよ」という指示のみで、実際の記述例が不足していた場合、subagentがガイダンスを正しく解釈できずに従前と同様の重複行を生成する可能性がある。また、ガイダンスを追加しても操作手順の各ステップが共通する場合の対処まで言及されていない場合、前提条件は解決されても操作手順での重複エラーが新たに発生する。

攻撃パス: ガイダンスの不完全性 → subagentが重複行を生成 → バリデーション失敗が再発 → 問題の根本解決に至らず追加の修正コストが発生

## リスク評価

| シナリオ番号 | 脅威名 | 影響度 | 発生可能性 | 総合リスク | 対策優先度 |
|-------------|--------|-------|-----------|-----------|-----------|
| T-01 | sessionToken制限緩和による権限昇格 | 高（承認フェーズの無効化） | 中（文言の曖昧さ次第） | 高 | 最優先 |
| T-02 | CLAUDE.md修正後の既存記述との矛盾 | 中（Orchestratorの誤解釈） | 高（他箇所に旧記述が残存し得る） | 高 | 優先 |
| T-03 | テンプレートリテラルの構文破壊 | 高（MCPサーバービルド失敗） | 低（注意すれば防げる） | 中 | 優先 |
| T-04 | バリデーター検出対象語句混入による品質違反 | 低（リトライで解決可能） | 低（事前確認で防げる） | 低 | 通常 |

T-01（シナリオ番号T-01）は発生した場合の影響がシステム全体の承認フェーズ無効化に及ぶため、最高優先度で対策する。
T-02（シナリオ番号T-02）は修正箇所と既存記述の照合を怠ると発生確率が高まり、二次的な誤解釈を誘発するため優先対策が必要である。
T-03（シナリオ番号T-03）はTypeScriptの構文チェック段階で検出されるが、検出後のMCPサーバー再起動が必須であり手戻りが大きい。
T-04（シナリオ番号T-04）はsubagentのリトライで解消可能だが、リトライコストを発生させないための事前確認が推奨される。
T-05（シナリオ番号T-05）はガイダンスの具体的な例示の質に依存し、機能要件FR-2・FR-3の受け入れ基準達成に直結する。

## セキュリティ要件

### SR-01: sessionToken二層構造記述の明確な分離

CLAUDE.md厳命23番の改訂後の記述では、以下の2層を明確に分離した文章構造を採用すること。

第1層（Orchestratorが直接呼び出すMCPツールへの渡し方）と第2層（subagentへの引き渡し条件）は、別の段落または番号付きリストとして物理的に分離して記述し、読み手が混同できない構造にすること。
「subagentへの引き渡しは workflow_record_test_result 呼び出し目的に限定」という制限が、Orchestratorの直接呼び出しルールとは別のルールであることを、接続詞（「一方、」「これに対し、」等）や見出し区分を使って明示すること。
旧記述「sessionTokenの使用先は workflow_record_test_result のみ」というフレーズが、Orchestratorの直接呼び出しにも適用されると読める場合はただちに削除または脚注化すること。

### SR-02: 修正前後のCLAUDE.md整合性検証

厳命23番を書き換えた後、CLAUDE.md全文を検索して「sessionToken」が出現する全ての箇所を列挙し、改訂後の記述と矛盾がないことを確認すること。
MEMORY.mdに記録された旧ルール（「使用先はworkflow_record_test_resultのみ」という記述）が残存する場合、MEMORY.mdの当該箇所も合わせて更新すること。
修正後のレビューでは、「任意のsubagentへsessionTokenを渡してよい」という解釈が生まれないことを第三者的な視点で確認すること。

### SR-03: definitions.tsビルド安全性確保

テンプレートリテラルへの追記作業では、追加テキストにバックティック（`` ` ``）・ドル記号+中括弧（`${`）が含まれていないことを、書き込みの前に目視確認すること。
追記完了後は `npm run build` を即座に実行し、コンパイルエラーがないことを確認してから次の作業に進むこと。
コンパイルエラーが検出された場合はただちに修正し、MCPサーバーの再起動が必要かどうかを判断すること（CLAUDE.md のMCPサーバーモジュールキャッシュのルールに従う）。

### SR-04: 追加ガイダンステキストのバリデーター検出対象語句事前スクリーニング

definitions.tsへのガイダンス追加テキストを確定する前に、CLAUDE.mdの禁止パターン完全リストに定義された英語グループ4語および日本語グループ8語がテキスト内に含まれていないことを確認すること。
ガイダンスに「悪い例」を例示する場合、悪い例のテキスト自体にバリデーター検出対象の語句が含まれないように言い換えること（「- NG例: 悪い前提条件行」という形式で、行自体に該当語句を含めないようにする）。
角括弧プレースホルダー（`[#xxx#]`形式）をガイダンステキストに含めないこと。

### SR-05: 前提条件行ガイダンスの例示充実度要件

FR-2・FR-3で追加するガイダンスには、具体的な書き換え前・書き換え後の例示を少なくとも2パターン以上含めること。
操作手順ステップの重複についても、前提条件行と同様の一意化指示を明示的に記述すること。
ガイダンスの文量が不足している場合はバリデーション再発を防ぎ切れないため、最低でも5行以上の実質的な記述を確保すること。
