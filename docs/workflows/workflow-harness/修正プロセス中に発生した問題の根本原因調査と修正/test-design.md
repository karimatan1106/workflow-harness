## サマリー

- 目的: FR-1（CLAUDE.md厳命23番の二層構造修正）・FR-2（manual_testテンプレートへの前提条件行一意化ガイダンス追記）・FR-3（e2e_testテンプレートへの前提条件行一意化ガイダンス追記）の実装が仕様通りであることを検証する。
- 主要な決定事項: テスト方法は主にファイル内容の手動検証（grep検索・内容確認）とnpm run buildによるビルド確認とする。自動テストとしてはnpx vistestによる既存テストスイートのリグレッション検証を用いる。
- 次フェーズで必要な情報: implementationフェーズ完了後に parallel_verification の manual_test サブフェーズでテストケースを実施する。各テストケースの合否は手動で目視確認する。

---

## テスト方針

本タスクはコードの新規作成ではなく、既存ファイルへのテキスト修正のみを行うタスクである。そのため、テスト戦略は以下の3層で構成する。

### 層1: ファイル内容の手動検証（最重要）

CLAUDE.md と definitions.ts の修正内容を grep 検索および Read ツールで確認する。修正後のテキストが仕様書（spec.md）に記載された変更後テキストと完全一致しているかを確認することが主目的である。目視検証は parallel_verification の manual_test サブフェーズで実施する。

### 層2: ビルド確認（TypeScriptコンパイル）

definitions.ts はTypeScriptファイルであり、テンプレート文字列への追記後にTypeScriptコンパイルエラーが発生しないことを確認する必要がある。workflow-plugin/mcp-server ディレクトリで npm run build を実行し、コンパイルエラーがゼロであることを確認する。ビルド確認は implementation フェーズで実施する。

### 層3: 既存テストスイートのリグレッション検証

npx vitest を実行して既存テストスイートが全合格することを確認する。追記内容はテンプレート文字列のみであり、ロジック変更がないため既存テストへの悪影響は想定されないが、念のため全スイートを実行する。テスト実行は testing フェーズで実施し、結果を workflow_record_test_result で記録する。

### test_implフェーズのスキップについて

本タスクのスコープにはテストファイルが含まれないため、MCPサーバーの自動判定により test_impl フェーズはスキップされる。テストケース実施は implementation フェーズ後の parallel_verification で行う。

## テスト実施順序

テストケースの推奨実施順序は以下の通りである。

1. TC-4（ビルド確認）を実施する。TypeScriptコンパイルが成功しないと後続のファイル内容確認が無意味になるため最初に実施する。
2. TC-1（FR-1内容確認）・TC-2（FR-2内容確認）・TC-3（FR-3内容確認）を並列に実施する。これらは互いに独立した確認であり、順不同で実施できる。
3. TC-5（リグレッション検証）を実施する。ビルド確認後にテストを実行する。
4. TC-6（統合確認）を最後に実施する。全個別確認が完了した後に意図との整合性を確認する。

## 合否判定基準

### 全体合格の条件

以下の条件を全て満たした場合に「合格」と判定する。

- TC-1a から TC-1e の全テストケースが合格していること。修正前テキストが存在せず、修正後の二層構造テキストが全て確認できる状態であること。
- TC-2a から TC-2e の全テストケースが合格していること。FR-22 ガイダンスが正しい位置に追記されていること。
- TC-3a から TC-3e の全テストケースが合格していること。FR-23 ガイダンスが正しい位置に追記されていること。
- TC-4a と TC-4b が合格していること。ビルドが成功し dist ファイルが更新されていること。
- TC-5a が合格していること。既存テストスイートが全合格であること。
- TC-6a から TC-6d の全テストケースが合格していること。修正の意図が正確に実現されていること。

### 失敗時の対応

TC-4（ビルド確認）が失敗した場合は implementation フェーズで修正を継続する。TC-5（リグレッション検証）に今回の修正に起因する失敗が発生した場合は実装を修正して再テストを実施する。TC-1 から TC-3・TC-6 の目視確認で内容の相違が発見された場合は implementation フェーズに差し戻して修正を実施する。

## テスト環境と前提条件

- 実施環境: Windows 11、Node.js v20.11.0 以上がインストールされた開発環境
- 必要なコマンド: npm run build（workflow-plugin/mcp-server ディレクトリで実行）、npx vitest（同ディレクトリで実行）
- 実施前提: implementation フェーズでの修正が全て完了していること
- 実施フェーズ: parallel_verification の manual_test サブフェーズで TC-1 から TC-3・TC-6 を実施する。TC-4 と TC-5 は implementation フェーズおよび testing フェーズで実施する。
