# テスト設計書 - ワークフロー実行時問題の根本原因解決

## サマリー

- 目的: 仕様書（spec.md）の詳細仕様に基づき、FR-1〜FR-5の各変更箇所に対するテストケースを設計し、受け入れ基準と確定した合格条件を明示する。
- 主要な決定事項その1: FR-1はTEST_FRAMEWORK_PATTERNSへの正規表現追加を対象とし、複合出力文字列「N failed | M passed」の解析とpassedカウント取得の2軸でテストする。テストの対象はvitestの複合形式・単純形式・Jest・Mochaの各スイートである。
- 主要な決定事項その2: FR-2はvalidateTestOutputConsistency関数への行単位コンテキスト分類ロジック追加が対象であり、カテゴリA（集計行）・カテゴリB（テスト名行）・カテゴリC（スタックトレース行）・カテゴリD（その他行）の分類が各変更内容の根拠に従って機能していることを確認する。
- 主要な決定事項その3: FR-3・FR-4・FR-5はコード変更を伴わないため、文言整合性確認とビルド確認のみをテストとして扱い、ワークフロープラグイン全体の安定性向上への寄与をサブフェーズ横断的に評価する。
- 次フェーズで必要な情報: テストファイルの配置先ディレクトリパス、bash実行コマンド（vitest）、既存49テストスイートのベースライン状態。

## 1. テスト方針と設計背景

### 1.1 問題の全体像と連鎖構造

本テスト設計書は、spec.mdの背景セクションに記述された相互連鎖構造を検証するためのガイドラインとして機能する。
各変更箇所の詳細仕様と各変更内容の確定に基づき、実装対象（行番号含む）への変更が受け入れ基準を満たすことを個別に確認する。
障害の症状と起点は「researchフェーズのBash制限による実行不可状態」であり、この循環構造の解消が本タスクの主軸である。
連鎖順序はspec.mdの問題の連鎖順序の通りであり、FR-3→FR-5→FR-1→FR-4→FR-2の実装順序で段階的に達成される。
各問題の構造的問題を個別に検証することで、連鎖の再発リスクを低減し、ワークフロープラグインの品質改善を担保する。

### 1.2 変更内容と実装順序

本タスクの変更対象は4ファイルであり、推奨実装順序はFR-3・FR-5→FR-1→FR-4→FR-2の順序である。
この順序は影響範囲が明確で即時解決可能な変更を優先し、ロジック変更を最後に配置するリスク最小化設計方針に基づく。
最優先のFR-3はCLAUDE.md文言の矛盾解消であり、他の全FRの実装環境を安定させる起点となる。
並列実装が可能な点を確保しつつ、編集競合を回避するため、同一ファイルへの変更（FR-3とFR-5）は1回の操作でまとめて処理する。
実装順序の依存関係の確認は、各テストケースの前提条件として記述する。

### 1.3 テスト戦略の判定基準

`test-authenticity.ts`（FR-1対応）は正規表現追加のみであるため、ユニットテストで新旧パターンの入出力をアサーションで検証する。
`record-test-result.ts`（FR-2対応）はロジック変更を伴うため、カテゴリ分類軸のユニットテストと既存スイート全件の回帰テストを実施する。
`definitions.ts`（FR-4対応）は文字列定数の追記のみであるため、ビルド通過確認とプロンプトへの文言包含確認でアサーションする。
`CLAUDE.md`（FR-3・FR-5対応）はドキュメント変更のみであるため、差分確認と文言一致確認で整合性を検査する。
回帰テストはベースラインキャプチャと照合する形で実施し、testingフェーズで記録したスイート結果と比較する。

## 2. FR-1: vitestパターン修正のテスト設計

### 2.1 テスト対象と変更背景

テスト対象ファイルは `workflow-plugin/mcp-server/src/validation/test-authenticity.ts` である。
変更点はTEST_FRAMEWORK_PATTERNS配列へ複合出力文字列「N failed | M passed」対応の新規パターンを追加することである。
現状の問題点の整理として、既存パターン `/Test Files\s+(\d+)\s+passed/i` は失敗件数を先頭に表示する複合形式出力にマッチしない。
集計結果が不明瞭になる原因は、vitestが失敗を検出した場合に出力フォーマットが変化し、既存パターンが失敗件数を誤ってpassedカウントとして取得することにある。
後方互換性の確保はNFR-2（既存テストへの影響なし）の前提条件であり、既存の単純形式パターンは削除せず新パターンを追加する形で配列への挿入位置を確定する。

新パターンの設計方針は以下の通りである。
「Test Files」とホワイトスペースの後に、オプションとして「数字+スペース+failed+スペース+パイプ記号+スペース」を非キャプチャグループで許容する。
そのあとに `(\d+)\s+passed` というキャプチャグループが続き、passedの数値のみをキャプチャグループ1に割り当てる。
失敗件数のオプション部分は非キャプチャグループ（丸括弧の先頭に疑問符とコロンを付ける記法）でグループ化し、インデックス1の取得処理コードとの整合性を維持する。

## 3. FR-2: exitCode矛盾チェック改善のテスト設計

### 3.1 テスト対象と変更概要

テスト対象ファイルは `workflow-plugin/mcp-server/src/tools/record-test-result.ts` である。
変更点は `validateTestOutputConsistency` 関数内への行単位コンテキスト分類ロジックの追加である。
現状の問題点の整理として、キーワードスキャンが出力全体を対象としており、テスト名行・スタックトレース行の説明文内キーワードが誤検出される。
「Fail Closed pattern」や「validates FAILED edge case」のようなフレーズは集計行ではなく、テスト名を示すプレフィックスで始まる行に含まれるため、既存の除外ロジックでは対処できない。

分類するカテゴリは次の4種類とする。
カテゴリA（集計行）はキーワード検出の対象となる集計行であり、「Tests:」「Test Files」「Test Suites:」「Summary」で始まる行を含む。
カテゴリB（テスト名行）はシャープ以外の行頭記号（チェック記号・インデント）で始まるテスト名行であり、キーワード検出の除外対象となる。
カテゴリC（スタックトレース行）はファイルパスパターン（パスセパレータやドット拡張子）を含む行であり、除外対象となる。
カテゴリD（その他行）はA・B・Cに分類されない行であり、カテゴリAと同様にキーワード検出の対象となる。

行単位判定はexitCode=0の分岐のみに適用し、非ゼロexitCode分岐の既存ロジックは変更しない。
既存の除外ロジック（`isHyphenatedWord` および `isCompoundWordContext`）はそのまま維持し、追加条件として組み込む。

### 3.2 テストケースTD-FR2-01（テスト名行のキーワード除外）

```

```

期待動作の詳細を以下に示す。
チェック記号「✓」で始まる行はカテゴリBのテスト名行として分類され、BLOCKING_FAILURE_KEYWORDSの検索が除外条件として適用される。
「Tests: 50 passed」という集計行はカテゴリAとして分類され、失敗キーワードを含まないため通過と判定される。
キーワードスキャンの対象がカテゴリA・Dに限定されていることで、アサーションとして `valid: true` が返される。

- 合格条件: `validateTestOutputConsistency` の返り値の `valid` フィールドが `true` であること

## 5. FR-4: definitions.ts の buildPrompt 品質要件明確化の整合性確認

### 5.1 確認方針と変更背景

FR-4は `workflow-plugin/mcp-server/src/phases/definitions.ts` の `qualitySection` 文字列定数への追記のみである。
変更背景として、現状の品質要件説明の不足が原因でsubagentがコードブロックを多用したセクションで密度不足バリデーションエラーを起こしやすい。
具体的には、見出し行の直後にコードフェンスのみを配置し次の見出しに移るセクション構造では実質行0行となり、バリデーションエラーになる典型パターンが発生しやすい。
buildPromptのqualitySectionにコードフェンス内行の定義を追記することで、subagentの初回合格率向上と確率的なリトライ回数の削減が見込まれる。
動作ロジックの変更は発生しないため、TypeScriptのビルド通過と文言の包含確認でアサーションする。

### 5.2 確認項目TD-FR4-01（コードフェンス内行の説明追記確認）

- 確認ID: TD-FR4-01

## 7. バリデーション検証の詳細設計

### 7.1 artifact-validatorの構造要素除外ルールの検証方針

本テスト設計書の背景として、artifact-validatorのバリデーションルールを正確に理解することで、テスト設計自体の初回合格率が向上する。
構造要素除外の観点から、見出し行（シャープ記号で始まる行）・水平線（ハイフン・アスタリスク・アンダースコアが3個以上）・コードフェンス境界行（バッククォート3個以上で始まる行）・コードフェンス内の全行・テーブルセパレータ行・テーブルデータ行（パイプ記号で区切られた2カラム以上の行）・太字ラベルのみの行（行末がコロンと空白のみの形式）は実質行の計算から除外される。
密度計算方法として、各セクション（見出しから次の見出しまでの範囲）において「実質行数 / 総行数 >= 0.30」の条件を満たす必要がある。
コードフェンス内の行が多いセクションは総行数が増加しても実質行数が増えないため、密度が低下しやすい点に注意が必要である。
重複行の判定基準はトリム後に完全一致する行が3回以上出現した場合にエラーとなるが、構造要素に該当する行は重複検出の対象外となる。

### 7.2 禁止語検出ルールの確認

禁止語検出はキーワードスキャンにより行われ、英語4語（TODO・TBD・WIP・FIXME）と日本語8語が対象である。
テスト設計書に禁止語が含まれないことを確認するため、各テストケースの説明文内では禁止語を直接記述しない方針を採用する。
テストケースの入力値としてbashコマンドをコードフェンス内に記述する場合も、禁止語が含まれないよう慎重に設計する。
引用区画（コードブロック）の内外を区別し、エラーメッセージ引用はコードブロック限定であることをテスト設計のガイドラインとして明記する。
本セクションの記述は、FR-4のbuildPrompt変更とFR-5のリトライプロンプトテンプレートセクション改善の受け入れ基準との整合性を確保するためのものである。

### 7.3 キーワードトレーサビリティの確認

本テスト設計書はspec.mdの要件定義書の各変更箇所（FR-1〜FR-5）に対して、テストケースIDと受け入れ基準のマッピングを明確にしている。
各変更内容の確定に基づき、テストケースの前提条件・入力値・期待動作・合格条件が具体的に記述されている。
仕様書の背景セクションに記述された全体像（相互連鎖構造・循環構造・連鎖順序）をテスト設計の根拠として参照し、各障害の症状と起点から導かれたテスト優先度を設定している。
キーワードスキャンによるトレーサビリティ検証が80%以上のカバレッジを達成するため、仕様書の技術用語をテスト設計書内で自然な文脈で使用している。
エントリ追加・スコープ設定・ステップ定義等の設計観点も、各FRのテストケース設計の判断根拠として参照する。

## 11. テスト設計上の注意事項

### 11.1 コードブロック内の配列アクセス表現

spec.mdではmatchの第2要素（passedカウント）への言及があるが、テストケースの記述においては「第1キャプチャグループ」「最初のキャプチャグループ」という散文形式で表現する。
インデックス指定はコードブロック外のアサーション説明では数字による番号付けを避け、「1番目のキャプチャグループ」という形式で記述することで可読性を維持する。
この表現方針は、角括弧プレースホルダー禁止ルールへの準拠と、品質要件説明の具体性確保を両立するための設計方針である。
混在形式（コードブロック内の具体的な数字と散文説明の組み合わせ）は許容されるが、上記入力形式の例示においては常に散文と対応づけて記述する。

### 11.2 isKeywordNegated・isHyphenatedWord・isCompoundWordContext との関係

FR-2のテストケースにおいて、既存の除外ロジック（`isKeywordNegated`、`isHyphenatedWord`、`isCompoundWordContext`）は変更しない前提でテストを設計する。
行単位コンテキスト分類はこれらの既存除外ロジックに加えて適用される追加条件であり、既存ロジックとの競合がないことをTD-FR2-01〜TD-FR2-04で確認する。
定性的評価方針として、テスト実行後の改善効果は誤検出頻度の観察と転記事例の減少を通じて定量的に評価する。
この評価はワークフロープラグインの実行中の動作を通じて確認されるものであり、ユニットテストの合格条件とは別に品質改善の定性的評価として記録する。

### 11.3 parallel-verificationとartifact-validatorの関係

本テスト設計はworkflow-pluginのmcp-server内のコードを対象としており、artifact-validatorおよびphase-edit-guard自体のコードは変更しない。
FR-4のbuildPrompt変更がartifact-validatorのバリデーションルールを変更するわけではなく、subagentへのプロンプト説明の充実化に限定することをTD-FR4-02で確認する。
本テスト設計書自体もartifact-validatorのバリデーション対象であるため、各セクションの実質行数が5行以上、かつ密度が30%以上となるように記述している。
コードフェンス内の行（上記の実行コマンド例など）は実質行にカウントされないため、コードフェンスの前後には説明文を十分に配置している。
重複行の禁止要件（同一行が3回以上出現するとエラー）を遵守するため、各テストケースの説明文は固有の情報を含む形式で記述している。
