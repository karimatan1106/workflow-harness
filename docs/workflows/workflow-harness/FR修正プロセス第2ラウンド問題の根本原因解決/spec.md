## サマリー

本仕様書は、FR修正プロセス第2ラウンドで発生した4件の問題（regression_testフェーズの繰り返し拒否、manual_testサマリー密度不足、security_scanでの重複行検出、performance_testサマリー密度不足）を根本的に解決するための実装計画を定義する。
全修正は単一ファイル `workflow-plugin/mcp-server/src/phases/definitions.ts` のみを対象とし、各フェーズの `subagentTemplate` 文字列に必要なガイダンス文言を追記することで問題を解消する。
対象フェーズは testing、regression_test、manual_test、security_scan、performance_test の5フェーズであり、バリデーター本体（artifact-validator.ts）や状態管理（state-manager.ts）は変更しない方針を採用している。
次フェーズ（test_design）では本仕様書の実装計画に基づいて各テストケースを設計し、implementationフェーズではテンプレート文字列の追記箇所と追記内容を精密に指定して作業を進める必要がある。
修正完了後は必ず npm run build によるトランスパイルと MCP サーバーの再起動を実施し、変更が実際に反映されることを確認してから次フェーズへ進む。

---

## 概要

本タスクが解決しようとする根本的な問題は、definitions.ts 内の subagentTemplate が各フェーズで必要な品質ガイダンスを省略しているという構造的な欠陥にある。
この欠陥により、フェーズを担当する subagent がバリデーション要件を正しく理解できずに不合格の成果物を生成し、何度もリトライが発生するという非効率な状況が繰り返されてきた。
具体的には、テストフレームワークの出力形式に関する知識の欠如（FR-R1）、ラベル行が実質行としてカウントされないというバリデーション挙動の無通知（FR-R2・FR-R4）、複数評価対象を扱う際の重複行禁止ルールに関する指示の不在（FR-R3）という3種類の知識ギャップが存在する。
修正方針として、各フェーズのテンプレートに対してピンポイントの追記を行い、subagent が成果物を作成する前から品質要件を認識できるようにする。この方針は最小変更原則に基づいており、既存テンプレートの構造や他フェーズへの影響を与えずに必要な情報を補完するものである。
実装後は5フェーズそれぞれで subagent が1回のフェーズ実行でバリデーションを通過できるようになり、parallel_verificationフェーズ全体のリトライ削減に寄与する。

## 実装計画

### FR-R1: testing フェーズおよび regression_test フェーズのテスト出力形式ガイダンス変更

definitions.ts の 862 行目（testing フェーズの subagentTemplate フィールド）と 871 行目（regression_test フェーズの subagentTemplate フィールド）に対して変更を加える。現在のテンプレートには「テスト完了。失敗件数0、成功件数N」のようなサマリー形式に整形してから渡すという誤った指示が存在しており、これが validateTestAuthenticity 関数の真正性チェック失敗を引き起こす原因となっている。

追記・変更すべき内容の方向性として、まず「output 引数にはテストコマンドの標準出力をそのまま貼り付けること。日本語に翻訳したり、人間向けに整形したりしてはいけない」という明確な禁止指示を加える必要がある。次に、vitest または jest が出力する集計行の具体的な形式例として「Test Files 3 passed (3)」「Tests: 12 passed (12)」などの文字列パターンを示し、これらの文字列が output に含まれていれば真正性チェックに合格できることを説明する。カスタムランナーを使用している場合でも「passed: N」「failed: N」「total: N」のいずれかの形式を含めることを必須要件として明記する。

regression_test フェーズについては「同一テキストの再送信は記録が許可されている」という既存の例外注記を維持しつつ、testing フェーズと同様の出力形式指示に統一する方針で両フェーズを対称的に修正する。

この変更により、testing フェーズと regression_test フェーズの両方において subagent が実際のテストコマンド出力をそのまま `workflow_record_test_result` の `output` 引数に渡すようになり、純粋な日本語文のみを渡すケースが発生しなくなる。結果として regression_test フェーズの `workflow_next` が1回で通過できるようになることを期待する。

### FR-R2: manual_test フェーズのサマリーセクション行数ガイダンス追加

definitions.ts の 889 行目（manual_test サブフェーズの subagentTemplate フィールド）に対して追記を行う。artifact-validator.ts の checkSectionDensity 関数は各セクションに実質行 5 行以上を要求しているが、subagent がサマリーセクションに「目的:」「主要な決定事項:」「次フェーズで必要な情報:」という 3 つのラベル行のみを記述すると、これらはすべて 50 文字以内のコロン終端ラベルとして isStructuralLine 判定によりゼロカウントされる。

追記すべき内容として、最初に「目的: 手動テストにより XXX 機能の動作を検証した」のように各ラベルのコロン直後に必ず実際のコンテンツを続けて 1 行に収めることを明示する。次に、コロン後にコンテンツがないラベル行は実質行としてカウントされないため、必ず 5 行以上の実質行（コンテンツを含む行）を書くよう数値で指示する。

この追記は manual_test テンプレートの既存の「重複行回避の注意事項」セクションの後に配置し、同一スタイルで記述することで subagent が両方の注意事項を同時に参照できるようにする。追記内容は散文形式で簡潔に記述し、NG/OK の対比例を 2 行程度含めることで理解しやすい形にする。

追記後は manual-test.md の `## サマリー` セクションが artifact-validator に合格し、実質行数が 5 行以上となることを確認する。subagent が「目的: 内容」形式で各項目を 1 行に収めるよう誘導され、空のラベル行のみを列挙するケースが発生しなくなることを受け入れ基準とする。

### FR-R3: security_scan フェーズの重複行回避ガイダンス追加

definitions.ts の 902 行目（security_scan サブフェーズの subagentTemplate フィールド）に対して新しいセクション「重複行回避の注意事項」を追加する。現在のテンプレートは「セキュリティスキャンを実施してください。」という 1 文のみを含んでおり、重複行回避の指示が一切ない。

security_scan フェーズは複数の FR（FR-A から FR-E など）を同一フォーマットで評価する性質を持つため、各 FR に対して「セキュリティリスク: 低」「外部入力の関与: なし」のような同一フレーズを繰り返しやすい。これらの行が 3 件以上完全一致すると artifact-validator が重複行エラーを返す。

追記すべき内容として、各 FR や評価対象のファイル名・関数名を行の先頭に含めて行を一意にすることを第一の指示とする。次に「問題なし」「リスクなし」のような短い評価結論を単独で繰り返す代わりに、評価対象の具体名と判断根拠を同じ行に含める方式を説明する。manual_test に追記されている重複行回避ガイダンスの文体を参考にし、「NG: 同一評価フレーズを 3 項目で繰り返す」「OK: 各 FR の識別子と判断根拠を含めた一意な行を記述する」という具体的な対比例を示す。

最後に、完全一致する行が 3 回以上出現すると artifact-validator がエラーを返す旨を注記として明記し、security_scan の `workflow_complete_sub` が 1 回で通過できるようになることを期待する。

### FR-R4: performance_test フェーズのサマリーセクション行数ガイダンス追加

definitions.ts の 914 行目（performance_test サブフェーズの subagentTemplate フィールド）に対して、FR-R2（manual_test）と同内容のサマリー行数ガイダンスを追記する。performance_test テンプレートは現在「パフォーマンステストを実施してください。」という最小限の指示のみを含んでおり、サマリーセクションの充実に関する指示が完全に欠如している。

追記すべき内容として、FR-R2 と共通する「ラベルのコロン直後にコンテンツを続けること」「5 行以上の実質行を確保すること」という基本指示に加えて、performance_test に特有の要素として計測対象処理・計測条件・計測結果の数値・評価（合否判定）の 4 項目をそれぞれ 1 行以上で記述することを求める。

「対象処理: workflow_next の呼び出し応答時間を 10 回計測した」「計測結果: 平均 45ms、最大 120ms、最小 30ms を記録した」のような数値を含む行が実質行としてカウントされるという説明を加え、subagent が適切な行を生成できるよう誘導する形とする。追記後は performance-test.md の `## サマリー` セクションが artifact-validator に合格し、performance_test の `workflow_complete_sub` が 1 回で通過できることを受け入れ基準とする。

## 変更対象ファイル

変更対象は `workflow-plugin/mcp-server/src/phases/definitions.ts` の単一ファイルのみである。このファイルはワークフローの全フェーズ定義と各フェーズの subagentTemplate を保持しており、変更後は npm run build によるトランスパイルと MCP サーバーの再起動が必須となる。

FR-R1 の変更対象箇所は 862 行目の testing フェーズ subagentTemplate と 871 行目の regression_test フェーズ subagentTemplate の 2 箇所である。現在の「テスト完了。失敗件数0、成功件数N」という誤った指示を削除または置換し、実際のフレームワーク出力をそのまま渡すという正しい指示に差し替える。この変更は既存の `workflow_record_test_result 呼び出し時の注意` セクションの内容を改訂する形で実施する。

FR-R2 の変更対象は 889 行目の manual_test subagentTemplate であり、既存の重複行回避セクションの後に実質行ガイダンスを追記する位置とする。追記内容は 3 行程度の散文形式で、数値による明示（5 行以上）を含む形で記述する。

FR-R3 の変更対象は 902 行目の security_scan subagentTemplate であり、現在は「セキュリティスキャンを実施してください。」という 1 文のみが存在する。ここに重複行回避の注意事項セクションを新設し、FR 識別子を行に含める手法を具体的に説明する形とする。

FR-R4 の変更対象は 914 行目の performance_test subagentTemplate であり、現在は「パフォーマンステストを実施してください。」という 1 文のみが存在する。ここにサマリー密度ガイダンスを追記し、計測値を含む具体的な行の記述例を示すことで subagent を正しい方向に誘導する形とする。変更後のファイルサイズは各テンプレートに数十文字から数百文字の追記が入るため、元のファイルよりも若干増加するが、TypeScript のコンパイル対象として問題ない範囲に収まる見込みである。

## 非機能要件との整合性確認

FR-R1〜FR-R4 の全変更は definitions.ts の subagentTemplate 文字列の追記・変更のみで構成される。artifact-validator.ts、state-manager.ts、next.ts は一切変更しないため、NFR-1（既存フェーズへの副作用なし）を完全に満たす。変更箇所は 5 行（testing、regression_test、manual_test、security_scan、performance_test の各 subagentTemplate）に限定される。

NFR-2（MCP サーバー再起動）との整合として、definitions.ts を変更した後は implementation フェーズ完了後に `npm run build` によるトランスパイルと MCP サーバーの再起動が必須である。再起動未実施のまま次フェーズに進んだ場合、旧バイナリのキャッシュにより変更が反映されないため、このオペレーション手順を忘れずに実施する必要がある。

NFR-3（テンプレート追記の形式統一）との整合として、各追記ガイダンスは manual_test の既存ガイダンス（重複行回避の注意事項セクション）と同一の記述スタイル（箇条書き・NG/OK 例示形式）で記述し、subagent が受け取る指示の一貫性を確保する。

NFR-4（regression_test と testing の対称性維持）との整合として、FR-R1 では testing と regression_test の両フェーズに同等のガイダンスを適用する。両フェーズはともに `workflow_record_test_result` を呼び出す点で共通しており、一方のみを修正した場合に将来 testing フェーズで同様の問題が再発するリスクを排除するために対称的な修正が必要である。

NFR-5（変更の最小性）との整合として、各変更は問題解決に必要な最小限の文言追加・変更に留め、既存テンプレートの構造（フェーズ名・出力ファイルパス・必須セクション指示）は削除・置換しない方針を貫く。

## 実装順序と検証手順

実装は FR-R1 から FR-R4 の順で実施し、各 FR の追記後にファイルを保存して構文エラーがないことを確認する。
全 FR の追記が完了した後に `npm run build` でトランスパイルを実行し、dist ディレクトリ内の definitions.js の更新タイムスタンプを確認する。
ビルド成功後に MCP サーバーを再起動し、workflow_status で現在のフェーズが正しく表示されることを検証する。
再起動後の動作確認として、各フェーズの subagentTemplate 内容が期待通りに変更されているかを workflow_status のレスポンスで間接的に確認できる。
全検証が完了した段階で refactoring フェーズへ進み、追記内容の文体統一と可読性の最終確認を行う。

## 前回修正との差分と根本原因の根絶方針

前回の FR 修正プロセスでは manual_test テンプレートへの重複行回避ガイダンス追記のみを実施したが、security_scan や performance_test には同等の記述指示を提示していなかった。この不完全な対処が今回の問題の根本原因であり、各評価対象に固有識別子（FR 番号やファイル名称）を含める指示が不要と誤認されていた。
テストシナリオの観点では、前回の testing フェーズと regression_test フェーズで node コマンドの標準出力フォーマットをそのまま渡す必要性が事前に提示されておらず、subagent が日本語に翻訳した結果行を生成する事態を根絶できていなかった。
各行の実質行要件（最低 5 行以上）は artifact-validator のロジックとして実行中のモジュールキャッシュに保持されており、即座に有効となる。コードフェンス内の行はトリム後も構造要素として除外される点を各要件の詳細として記載し、subagent への記述方法の指示を完結させる。
見出し配下の各評価行に同一内容が 3 件以上含まれる場合のエラー遷移を保証するため、各行に固有の理由と番号を付記する方式を採用する。この方式により security-scan.md の評価行が独立した内容を持ち、レビューフェーズでの正常完了を最小化されたリトライ回数で達成できる。
前回修正で既存構造を維持する必要最小限の追記に留めた方針は有効であり、本タスクでも同一方針を継続する。各要件は definitions.ts 内で完結し、バリデーターや状態管理の変更は不要である。
