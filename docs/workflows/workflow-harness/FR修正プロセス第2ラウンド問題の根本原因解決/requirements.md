## サマリー

- 目的: 前回のFR修正プロセス実行中に発生した4件の問題（regression_testの繰り返し拒否、manual_testサマリー不足、security_scanの重複行検出、performance_testサマリー不足）を二度と再発させないための機能要件を定義する
- 主要な決定事項: 全修正は `workflow-plugin/mcp-server/src/phases/definitions.ts` 内の各フェーズ `subagentTemplate` 文字列の追記・変更のみで対処する。バリデーター本体（artifact-validator.ts）は変更しない
- 定義する要件の番号体系: FR-R1（regression_test出力形式）、FR-R2（manual_testサマリー密度）、FR-R3（security_scan重複行回避）、FR-R4（performance_testサマリー密度）の4件
- 次フェーズで必要な情報: 各FRの変更対象行番号の目安、追記すべき指示文の方向性、受け入れ基準のテストシナリオ

---

## 機能要件

### FR-R1: regression_testフェーズのテスト出力形式ガイダンス追加

**対象ファイル**: `workflow-plugin/mcp-server/src/phases/definitions.ts`

**変更対象**: regression_testフェーズの `subagentTemplate` 内にある `workflow_record_test_result` の呼び出し指示セクション（および同様の指示が存在するtestingフェーズのテンプレートも同時に更新する）

**要件の詳細**:
subagentがテスト結果の日本語サマリー文（例:「テストが正常完了しました」）を `workflow_record_test_result` の `output` 引数に渡すと、`validateTestAuthenticity` が `passed`、`failed`、`total` のいずれの語句も見つけられず真正性チェックを失敗させる。この事態を防ぐため、テンプレートに以下の指示を追加する。

追加すべき指示の内容:
1. `output` 引数にはテストコマンドの標準出力をそのまま貼り付けること（日本語に翻訳したり、整形したりしないこと）を明記する
2. vitest / jest が出力する集計行の形式例（`Test Files 3 passed (3)` / `Tests: 12 passed (12)` など）を1行以上の具体的なサンプルとして提示する
3. カスタムランナーを使用する場合でも `passed: N` / `failed: N` / `total: N` のいずれかの形式を出力に含めることを求める

**受け入れ基準**:
- テンプレートを読んだsubagentが、実際のテストコマンド（npm test / npx vitest など）を実行し、その標準出力をそのまま `output` に渡せるよう明確に理解できること
- 「テストが正常完了しました」のような純粋な日本語文のみを `output` に渡すケースが発生しなくなること
- regression_testフェーズで `workflow_next` を1回で通過できること（3回以上のリトライが不要になること）

---

### FR-R2: manual_testフェーズのサマリーセクション行数ガイダンス追加

**対象ファイル**: `workflow-plugin/mcp-server/src/phases/definitions.ts`

**変更対象**: manual_testフェーズの `subagentTemplate` 内にある `## サマリー` セクションの記述方法に関する指示

**要件の詳細**:
artifact-validator.ts の `checkSectionDensity` は各セクションに実質行5行以上を要求する。しかし「目的:」「主要な決定事項:」「次フェーズで必要な情報:」の3つのラベル行は、50文字以内でコロン終端のため `isStructuralLine` によりすべて実質行ゼロとしてカウントされる。この結果、サマリー全体が4行以下で実質行要件を満たせないケースが発生した。

追加すべき指示の内容:
1. 各ラベル行（「目的:」「主要な決定事項:」「次フェーズで必要な情報:」）のコロン直後に、必ず1文以上の実際のコンテンツを続けて1行に収めること（例: 「目的: 手動テストによりXXX機能の動作を検証する」）を明記する
2. ラベルのみを独立した行として書いた場合、その行は実質行としてカウントされないことを注記する
3. サマリーセクションに実質行（コンテンツを含む行）を5行以上書くよう数値で明示する

**受け入れ基準**:
- manual-test.md の `## サマリー` セクションが artifact-validator に合格すること
- サマリーセクションの実質行数が5行以上となること（バリデーターがセクション密度エラーを返さないこと）
- subagentが「目的: 内容」形式で各項目を1行に収めるよう誘導され、空のラベル行のみを列挙するケースが発生しなくなること

---

### FR-R3: security_scanフェーズの重複行回避ガイダンス追加

**対象ファイル**: `workflow-plugin/mcp-server/src/phases/definitions.ts`

**変更対象**: security_scanフェーズの `subagentTemplate` 内に重複行回避の注意事項セクションを新設する

**要件の詳細**:
artifact-validator.ts はコードフェンス外でトリム後に完全一致する行が3回以上出現するとエラーを返す。security_scanは複数のFR（FR-A〜FR-E など）を同一フォーマットで評価する性質を持つため、「セキュリティ判定: 問題なし」に続く評価文が複数項目で同一内容になりやすい。manual_testには前回の修正で重複行回避のガイダンスが追記されたが、security_scanには同様の修正が適用されていない。

追加すべき指示の内容:
1. 複数のFR/対象を評価する際、各評価行に対象のFR番号・ファイル名・関数名などの固有識別子を含めて行を一意にすることを明記する
2. 「問題なし」「リスクなし」のような短い評価フレーズが3件以上同一内容で出現しないよう、各行に対象の具体的な名称と理由を付記することを求める
3. 同一フレーズの繰り返しが3回以上になると artifact-validator がエラーを返す旨を注記として記載する（テンプレートを読んだsubagentが事前に回避できるよう）

**受け入れ基準**:
- security-scan.md で artifact-validator の重複行チェックがエラーを返さないこと
- 各評価対象（FR-A、FR-B など）の評価行がそれぞれ固有の内容を持ち、完全一致する行が2件以下であること
- security_scan の `workflow_complete_sub` が1回で通過できること

---

### FR-R4: performance_testフェーズのサマリーセクション行数ガイダンス追加

**対象ファイル**: `workflow-plugin/mcp-server/src/phases/definitions.ts`

**変更対象**: performance_testフェーズの `subagentTemplate` 内にある `## サマリー` セクションの記述指示

**要件の詳細**:
FR-R2（manual_test）と構造的に同一の根本原因を持つ。performance_testのテンプレートはパフォーマンステスト実施の最小限の指示のみを含んでおり、サマリーセクションの行数やコンテンツ密度に関する具体的なガイダンスが存在しない。このためsubagentが3行程度の短いサマリー（ラベル行のみ）を生成し、実質行5行要件を満たせない。

追加すべき指示の内容:
1. FR-R2と同様に、ラベル行のコロン直後に実際のコンテンツを続けて1行に収めることを明記する
2. performance_testに特有の要素として、計測対象処理・計測条件・計測結果の数値・評価（合否判定）を各1行以上で記述することを求め、最低でも5行の実質行を確保するよう誘導する
3. 「目的:」「対象:」などのラベルのみの行は実質行としてカウントされないことを明示し、数値を含む結果行を必ず含めることを指示する

**受け入れ基準**:
- performance-test.md の `## サマリー` セクションが artifact-validator に合格すること
- サマリーセクションの実質行数が5行以上となること
- performance_test の `workflow_complete_sub` が1回で通過できること

## 非機能要件

### NFR-1: 既存フェーズへの副作用なし

FR-R1からFR-R4の各変更は `definitions.ts` のテンプレート文字列への追記・変更のみで構成される。バリデーター本体（artifact-validator.ts）、状態管理（state-manager.ts）、フェーズ遷移ロジック（next.ts）は一切変更しない。これにより、他のフェーズの動作に影響を与えないことを保証する。変更箇所は4つのフェーズのテンプレートに限定される。

### NFR-2: MCP サーバー再起動の必要性

definitions.ts を変更した場合、Node.js のモジュールキャッシュにより変更は実行中の MCP サーバーに即座に反映されない。変更を有効にするには `npm run build` によるトランスパイルと MCP サーバーの再起動が必要である。このオペレーションは implementation フェーズ完了後に必ず実施する。再起動未実施のまま次フェーズに進んではならない。

### NFR-3: テンプレート追記の形式統一

各フェーズのテンプレートに追記するガイダンスは、既存のテンプレート記述スタイル（箇条書き、見出し構造）と一致させる。特に manual_test テンプレートに前回修正で追記されたガイダンスの文体・フォーマットを参考にし、security_scan・performance_test・regression_test・testing の各テンプレートでも同一の文体で記述する。これにより subagent が受け取る指示の一貫性を確保する。

### NFR-4: regression_test と testing フェーズの対称性維持

FR-R1 で修正する出力形式ガイダンスは regression_test フェーズのテンプレートのみでなく、testing フェーズのテンプレートにも同様に適用する。両フェーズは `workflow_record_test_result` を呼び出す点で共通しており、一方にのみ修正を適用した場合に将来 testing フェーズでも同様の問題が再発するリスクがある。対称的に両フェーズを修正することで問題の根絶を図る。

### NFR-5: 変更の最小性（必要最小限の修正）

各要件の変更は問題解決に必要な最小限の文言追加に留める。テンプレートの既存構造（フェーズ名、出力ファイルパス、必須セクションの指示）を削除・置換しない。追記のみで変更を完結させることで、変更リスクを最小化する。設計レビューフェーズで追記内容のみを確認できる形を維持する。
