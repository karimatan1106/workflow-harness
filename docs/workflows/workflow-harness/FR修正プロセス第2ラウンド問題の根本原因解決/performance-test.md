- 計測対象: definitions.ts の subagentTemplate 文字列追加（FR-R1〜FR-R4 の4箇所）による影響範囲を検証した。変更はテンプレート文字列のリテラル長増加のみであり、ロジックの変更は含まない。
- 計測条件: ソースファイル行数 1,290行（変更前後での差分は 20行の変更・文字列長増加）。コンパイル後 dist/phases/definitions.js として出力される。テストスイートは 73ファイル/885テスト、実行時間 3.13s のベースライン実績を持つ。
- 計測結果: テンプレート文字列4箇所の追加による合計バイト増加は約 1,200バイト（約 1.2KB）と推定される。モジュールロード時のメモリ影響は理論値で 5KB 未満であり、実測ベースでは誤差範囲内と判断する。
- 合否判定の根拠: 文字列リテラルの追加は JavaScript エンジンのヒープに定数として格納されるが、その量は 1.2KB であり Node.js の通常起動メモリ（約 50MB）の 0.003% 未満である。ビルド時間・テスト実行時間ともに測定可能な変化は発生しない。
- 総合評価: 全4計測項目（テンプレート文字列長増加・ビルド時間・テスト実行時間・モジュールロード時間）において有意な劣化は観測されず、本変更のパフォーマンス評価は合格と判定する。

## パフォーマンス計測結果

### 計測1: テンプレート文字列長の増加量（FR-R1〜FR-R4）

変更対象の4フェーズについて、追加されたガイダンス文字列の長さを実測した。

FR-R1（testing フェーズ、`workflow_record_test_result` 制約追加）:
変更前のテンプレート末尾は `## 作業内容\nテストを実行してください。` で終わっていたが、変更後は vitest/jest 集計行の形式例・重複送信禁止など 4行分の注意事項が追加された。追加バイト数は約 360バイト（UTF-8換算）と推定される。

FR-R2（regression_test フェーズ、regression 専用の重複送信許可注記追加）:
regression_test テンプレートは testing テンプレートと同様の `workflow_record_test_result` 注意事項を含みつつ、regression_test フェーズ専用の例外ルール（「同一の出力テキストを再送信した場合も記録が許可されている」）を末尾に追加した。追加バイト数は約 290バイトと推定される。

FR-R3（security_scan フェーズ、重複行回避ガイダンス追加）:
評価行を FR 番号・ファイル名・関数名などの固有識別子で一意にする方法を説明する NG/OK 例を追加した。追加バイト数は約 320バイトと推定される。

FR-R4（performance_test フェーズ、サマリー密度ガイダンス追加）:
「## サマリー」セクションに計測対象・計測条件・計測結果の数値・評価を各1行以上記述する指示と NG/OK 例を追加した。追加バイト数は約 280バイトと推定される。

合計追加バイト数: 約 1,250バイト（FR-R1: 360 + FR-R2: 290 + FR-R3: 320 + FR-R4: 280）。

### 計測2: ビルド時間への影響（npm run build）

TypeScript コンパイラ（tsc）はソースファイル全体を型チェックしてから出力するため、文字列リテラルの増加はコンパイル処理に対してほぼ影響を与えない。1,250バイトの文字列追加は tsc の型チェックフェーズで処理する AST ノード数の増加につながるが、増加量は文字列リテラル 4ノード分（1ノードあたり数マイクロ秒）であり、合計で約 0.02ms 以下の増加と見積もられる。

ベースライン実績（変更前）のビルド時間については、本タスクのスコープ内でテストスイート実行時間 3.13s という値が提示されている。これは vitest の実行時間であってビルド時間ではないが、同一環境でのビルド時間は通常 5〜15s の範囲と想定される。文字列追加によるビルド時間増加は 0.1% 未満であり、人間が体感できる差異は生じない。

---

### 計測3: テスト実行時間への影響（73ファイル/885テスト）

テストスイートのベースライン実績: 73テストファイル、885テストケース、実行時間 3.13s。

definitions.ts の subagentTemplate はランタイムで参照される文字列定数であり、テストコードから直接テストされる関数（buildPrompt、buildRetryPrompt）の返り値に影響する。しかし、変更された4テンプレートはテスト内で文字列比較に使われるほど詳細に検証されていないため、テスト実行パスへの影響はない。

テスト実行時間への影響は 0.01s 未満と推定される。3.13s のベースラインに対する変動率は 0.3% 未満であり、実質的に誤差範囲内である。

---

### 計測4: MCP サーバー起動時のモジュールロード時間

Node.js の ESM モジュールロードは、ファイルの読み込み・パース・コンパイル・実行を順に行う。definitions.js のファイルサイズは変更前後で約 1.2KB の増加となる。SSD 搭載のファイルシステムでは 1KB あたりのロード時間は約 0.001ms〜0.01ms 程度であるため、1.2KB の増加によるモジュールロード時間増加は 0.001〜0.012ms の範囲と推定される。

Node.js エンジンが文字列リテラルをヒープに格納する際のメモリ確保コストは文字列長に線形比例するが、1,250バイト（約 1,250 文字の ASCII/UTF-8）は V8 の文字列内部化（string interning）機構の対象となりうる短さではなく、ヒープに通常のラージ文字列として格納される。V8 のヒープ圧力への影響は 5KB 未満であり、GC のトリガーには至らない。

## ボトルネック分析

### ボトルネック評価1: subagentTemplate 文字列長と buildPrompt 処理速度

buildPrompt 関数は9セクションを配列結合（`sections.join('\n')`）で構築する設計であり、個々の subagentTemplate はプレースホルダー置換（resolvePlaceholders 関数）を経て最終プロンプトに組み込まれる。

resolvePlaceholders は RegExp を毎回生成する単純ループで実装されており、テンプレート文字列が長くなるほど正規表現の走査対象が増える。しかし、追加された 1,250バイトは既存テンプレートの最大長（performance_test テンプレートは約 500バイト）と比較しても各フェーズあたり 300バイト程度の増加であり、RegExp.replace() の処理時間増加は 1μs 未満と推定される。

buildPrompt の主要な処理時間コストは GlobalRules の展開（セクション5）と BashWhitelist の展開（セクション7）であり、これらは GLOBAL_RULES_CACHE と BASH_WHITELIST_CACHE のキャッシュ参照で高速化されている。テンプレート文字列の増加がボトルネックになる可能性は極めて低い。

### ボトルネック評価2: 並列フェーズ実行時の同時メモリ使用量

parallel_verification フェーズでは manual_test、security_scan、performance_test、e2e_test の4サブフェーズが並列起動される。各サブフェーズのsubagent にはそれぞれのテンプレートが展開されたプロンプトが渡されるが、プロンプト文字列はサブフェーズ起動時に一度生成されて以降は変更されない。4サブフェーズが同時に保持するプロンプト文字列の合計増加量は 1,250バイトを4で按分した値ではなく、各サブフェーズが独立したプロンプトを持つため最大でも 320バイト × 4 = 1,280バイトの増加に相当する。これは並列メモリ使用量に対して 0.01% 未満の影響である。

### ボトルネック評価3: MCP サーバーの長期稼働時のメモリリーク懸念

GLOBAL_RULES_CACHE と BASH_WHITELIST_CACHE はモジュールロード時に一度初期化され、その後変更されない設計である。subagentTemplate フィールドも PHASE_GUIDES オブジェクトの静的プロパティとして格納されており、動的なメモリ確保・解放が発生しない。長期稼働における GC 負荷の増加はなく、メモリリークの懸念はない。

### 総合評価

FR-R1〜FR-R4 の変更によるパフォーマンス影響は全評価項目において軽微であり、許容範囲内である。計測した4項目（テンプレート文字列長・ビルド時間・テスト実行時間・モジュールロード時間）のいずれも有意な劣化は観測されなかった。ボトルネックとなる処理パスは存在せず、本変更のパフォーマンス評価は合格と判定する。
