## サマリー

本テスト設計書は、FR修正プロセス第2ラウンドで発生した4件の問題を根本的に解決するために定義された機能要件FR-R1〜FR-R4が、実装フェーズ完了後に正しく機能することを保証するテストシナリオの概要を記述する。本仕様書が提示するテストシナリオは、全修正が `workflow-plugin/mcp-server/src/phases/definitions.ts` の単一ファイルに収束しているという方向性を前提とする。

テスト対象の変更対象ファイルは `workflow-plugin/mcp-server/src/phases/definitions.ts` のみであり、各フェーズの `subagentTemplate` 文字列への追記箇所と追記内容が仕様通りであることを検証する。テストフレームワークとしてvitestを採用し、`PHASE_SETTINGS` オブジェクトをインポートして文字列内容を確認するユニットテストと、トランスパイル後のビルド成功を確認する統合テストの2種類を組み合わせる。

主要な決定事項として、テストシナリオの数値目標は合計14件（ユニットテスト12件、ビルド統合テスト2件）とし、各テストケースがそれぞれ独立した内容を持つことで重複行禁止のバリデーションロジックに適合した評価行の独立性を維持する方針を採用した。テストの実施順序はFR-R1からFR-R4まで順次検証し、最後に全体ビルドとリグレッションスイートを実行する設計とした。

次フェーズで必要な情報として、テストファイルのパス（`workflow-plugin/mcp-server/src/phases/__tests__/subagent-template-guidance.test.ts`）、ビルド検証コマンド（`cd workflow-plugin/mcp-server && npm run build`）、vitest実行コマンド（`npx vitest run src/phases/__tests__/subagent-template-guidance.test.ts`）を引き継ぐ。

## テスト前提条件と概要

### 本テスト設計書が対象とする根絶方針

本テスト設計書が扱う根本原因は3種類の知識ギャップに集約され、これらのギャップを根絶することが各テストシナリオの目的となる。第一のギャップはテストフレームワーク標準出力の渡し方に関する知識欠如（FR-R1）であり、第二のギャップはラベル行が実質行としてカウントされないというバリデーション挙動の無通知（FR-R2・FR-R4）であり、第三のギャップは複数評価対象を同時に扱う際の重複行禁止ルールに関する指示の不在（FR-R3）である。各テストケースはこの根絶方針に沿ってピンポイントで追記箇所を特定し、仕様通りの内容が保存されていることを確認する。definitions.ts内でガイダンスを省略していた箇所を補完することでリトライ削減を図り、テスト件数も必要最小限に抑制する方針である。

### ファイルサイズと変更の影響範囲

definitions.ts への追記後は、各テンプレートに数十文字から数百文字の追記が加わる。変更後のファイルサイズは元のファイルより若干増加する見込みであるが、TypeScript のコンパイル対象として問題ない範囲に収まることを TC-BUILD-1 で確認する。一切変更しないファイル（artifact-validator.ts、state-manager.ts、next.ts）が正常な状態を維持していることは TC-BUILD-2 および NFR-1 検証テストが保証する。

### 手動確認項目と自動テストの境界

自動ユニットテストでは文字列の存在確認のみを行い、文体統一と可読性の最終確認は refactoring フェーズでの手動確認として実施する。各追記ガイダンスの文体や例示形式の統一性は NFR-3 検証として code-review.md の必須記載事項とする。実装順序（FR-R1からFR-R4の順）に従った手動テスト手順は「テスト実施順序とワークフロー上の手順」セクションで詳述する。

## FR-R1テスト設計（testing/regression_testのテスト出力形式ガイダンス）

### 検証の背景と根本原因との関連

FR-R1は「testingおよびregression_testフェーズのsubagentTemplateに誤った整形指示が存在し、subagentが純粋な日本語サマリー文を `workflow_record_test_result` の `output` 引数に渡すことでvalidateTestAuthenticity関数の真正性チェックが失敗する」という根本的な問題を解決することを目的とする。

この問題の具体的な挙動として、現行テンプレートには「テスト完了。失敗件数0、成功件数N」のような整形済み日本語サマリーを渡す指示が存在する。このロジックに従ったsubagentが生成する日本語文の出力文字列には `passed`・`failed`・`total` のいずれの語句も含まれないため、`TEST_OUTPUT_INDICATORS` パターンとの照合に失敗し、フェーズ遷移が拒否される。結果として regression_test フェーズで3回以上のリトライが発生するという非効率な状況が生まれていた。なお `xxx` フィールドなど definitions.ts 内の正規表現パターン名は変更せず、テンプレート文字列の追記のみで問題を解消する。

解決方針として、subagentが vitest または jest が出力する集計行の形式例（`Test Files 3 passed (3)` や `Tests: 12 passed (12)` など）を理解し、標準出力をそのまま転送するよう誘導するガイダンスを追記・変更することが定義されている。カスタムランナーを使用する場合でも `passed: N`・`failed: N`・`total: N` のいずれかの形式を含める必要があることも明示が求められる。

### TC-R1-1: testingフェーズのテンプレートに標準出力そのまま転送の指示が含まれること

検証対象は `PHASE_SETTINGS.testing.subagentTemplate` フィールドである。検証方法として `toContain('標準出力をそのまま')` または同等の文字列（「そのまま渡す」「整形しない」など）の部分一致で確認する。期待結果として、subagentがテストコマンドの実際のターミナル出力を加工せずに `output` 引数として渡すことを促す文字列が含まれていること。失敗条件は、変更前の誤った整形指示がそのまま残存していてガイダンスの追記が一切確認できない場合。この検証は FR-R1 の最高優先度の受け入れ基準を直接検査するものであり、サブフェーズを通じてワークフロー全体の信頼性に寄与する。

### TC-R1-2: testingフェーズのテンプレートにvitest/jest集計行の形式例が含まれること

検証対象は `PHASE_SETTINGS.testing.subagentTemplate` フィールドである。検証方法として `toContain('Test Files')` または `toContain('passed')` で集計行形式例の存在を確認する。期待結果として、vitestが出力する `Test Files 3 passed (3)` のような具体的な文字列パターンが少なくとも1行以上含まれていること。このテストにより subagent が指示を読んだ直後にどのような形式の出力が合格となるかを理解できることを保証する。失敗条件は形式例が全く記載されておらず、「標準出力を渡す」という抽象的な説明しか存在しない場合。

### TC-R1-3: testingフェーズのテンプレートに日本語翻訳・整形禁止の明示が含まれること

検証対象は `PHASE_SETTINGS.testing.subagentTemplate` フィールドである。検証方法として `toContain('翻訳')` または `toContain('整形')` の禁止指示を確認する。期待結果として、日本語に翻訳したり人間向けに整形したりしてはいけないという趣旨の文字列が存在すること。この検証は根本的な問題の再発を防ぐための最重要チェックであり、subagentが日本語サマリーを生成するパターンを排除するために必要な文言が追記されていることを確認する。失敗条件は変更後も翻訳禁止・整形禁止に関する記述が一切存在しない場合。

### TC-R1-4: regression_testフェーズのテンプレートに同等の出力形式指示が含まれること

検証対象は `PHASE_SETTINGS.regression_test.subagentTemplate` フィールドである。検証方法として TC-R1-1 と対称的な `toContain` 検証を適用する。期待結果として、regression_test のテンプレートにも testing フェーズと同等の出力形式ガイダンスが含まれていること。この検証は NFR-4（regression_testとtestingフェーズの対称性維持）の技術要件を直接保証するものであり、将来 testing フェーズで同様の問題が再発するリスクを根絶するために不可欠なテストシナリオ。失敗条件は regression_test のテンプレートだけが変更されず、testing と regression_test の間に指示内容の非対称が生じている場合。

### TC-R1-5: regression_testフェーズの既存の例外注記が維持されていること

検証対象は `PHASE_SETTINGS.regression_test.subagentTemplate` フィールドである。検証方法として `toContain('例外')` または `toContain('重複')` で既存の再送信例外注記の残存を確認する。期待結果として「regression_testフェーズでは同一テキストの再送信が許可される」旨の注記が FR-R1 の変更後も削除されずに残っていること。最小変更原則（NFR-5）に基づき、既存の有効な指示を誤って削除していないことを保証するための検証。失敗条件は FR-R1 の変更時にこの例外注記が意図せず除去されている場合。

## FR-R2テスト設計（manual_testサマリーセクション行数ガイダンス）

### 検証の背景と根本原因との関連

FR-R2は「manual_testフェーズのsubagentTemplateにサマリーセクションの実質行数ガイダンスが存在せず、subagentがラベル行のみを列挙した3行程度の短いサマリーを生成してバリデーションに不合格となる」という根本的な問題を解決する。手動テスト実施後に作成される manual-test.md の `## サマリー` セクションの充実が不十分であった点がタスクの本質的な欠陥である。

artifact-validator.tsの `checkSectionDensity` 関数は各 `##` セクション内に実質行5行以上を要求するが、`isStructuralLine` 関数の判定により「目的:」「主要な決定事項:」「次フェーズで必要な情報:」という3つのラベル行はすべて50文字以内のコロン終端ラベルとして実質行ゼロとカウントされる。このロジックへの知識欠如がサマリー密度不足の直接的な原因であった。解決策として各ラベル行のコロン直後に実際のコンテンツを続けることと、5行以上の実質行を確保することを数値で明示するガイダンスを追記する。

追記箇所は889行目のmanual_test subagentTemplate内の既存「重複行回避の注意事項」セクションの後であり、追記内容は散文形式で2〜3行の簡潔な指示とNG/OK対比例で構成される。追記後に保存し構文エラーがないことを確認してから次のFRへ進む手順を守る。サマリーセクションへの各追記は5行程度含む形で記述することが推奨され、空のラベル行のみを列挙する作成パターンを改訂して実質行を確保する。何度もリトライが発生した根本原因の担当箇所として、manual_test は各項目の詳細を充実させる義務を持つ。

### TC-R2-1: manual_testのテンプレートに5行以上の実質行要件が数値で明示されていること

検証対象は `PHASE_SETTINGS.parallel_verification.subPhases.manual_test.subagentTemplate` フィールドである。検証方法として `toContain('5行以上')` で具体的な数値ガイダンスの存在を確認する。期待結果として「5行以上」という数値を含む指示文字列が追記されていること。この数値による明記はsubagentが「何行書けばよいか」を明確に理解するための必須要件であり、密度不足を根本から防ぐ手法。失敗条件は行数要件が暗示的にしか記述されておらず「十分に記述する」のような曖昧な表現にとどまっている場合。各項目の詳細な説明はTC-R2-2とTC-R2-3が補完する。

### TC-R2-2: manual_testのテンプレートにコロン直後にコンテンツを続ける指示が含まれること

検証対象は `PHASE_SETTINGS.parallel_verification.subPhases.manual_test.subagentTemplate` フィールドである。検証方法として `toContain('コロン')` または `toContain('コンテンツ')` でラベル行の記述方法に関する指示を確認する。期待結果としてコロン直後に実際の情報を続けることを促す文字列が含まれていること。この検証は isStructuralLine 関数の挙動（コロン後にコンテンツがない行はゼロカウント）に関する知識をsubagentに伝えるための指示が存在することを保証する。失敗条件はラベル行がカウント外となる問題について何も追記されていない場合。

### TC-R2-3: manual_testのテンプレートに既存の重複行回避ガイダンスが維持されていること

検証対象は `PHASE_SETTINGS.parallel_verification.subPhases.manual_test.subagentTemplate` フィールドである。検証方法として `toContain('重複行回避')` で前回修正のガイダンスが残存していることを確認する。期待結果として前回のFRラウンドで追記された重複行回避の注意事項セクションが削除されずに維持されていること。最小変更原則（NFR-5）の遵守確認であり、今回のFR-R2追記作業が既存の有効なガイダンスを上書き・削除していないことを保証する。失敗条件はFR-R2の追記時に既存の重複行回避ガイダンスが意図せず消失している場合。

## FR-R3テスト設計（security_scan重複行回避ガイダンス）

### 検証の背景と根本原因との関連

FR-R3は「security_scanフェーズのsubagentTemplateに重複行回避の指示が皆無であり、複数のFR（FR-AからFR-Eなど）を同一フォーマットで評価する際に同一評価フレーズが3回以上出現してartifact-validatorの重複行検出エラーを引き起こす」という根本的な問題を解決する。複数評価対象を同時に記述する性質が重複行を生みやすくしており、指示の不在が根本原因となっている。

重複行検出のロジックはコードフェンス外の行においてトリム後に完全一致する行が3件以上完全一致するとエラーを返す閾値設計となっており、構造要素（見出し行・水平線・テーブルセパレータ・コードフェンス内行など）として除外される行には当てはまらない通常の評価文が検出対象となる。manual_testには前回のFRラウンドで重複行回避ガイダンスが追記済みだが、security_scanには同様の修正が採用されていなかったという不完全な対処がこのラウンドの問題の起点となった。

追記箇所は902行目のsecurity_scan subagentTemplate内に「重複行回避の注意事項」セクションを新設する位置であり、FR識別子や固有識別子を行に含める手法、NG/OK対比例、3件以上の完全一致でエラーとなる旨の注記を含む内容を追記する。セキュリティリスクの評価行においても各FR番号や関数名称を含めることが各行の一意性を確保する方式の中心となる。security_scan が扱う外部入力の関与の有無といった評価観点は、各FRごとに固有の文脈で記述することで重複を回避する。適切な改訂により共通フォーマットの繰り返しを防ぎ、各行を一意に保つことが求められる。

### TC-R3-1: security_scanのテンプレートに重複行回避ガイダンスのセクションが追加されていること

検証対象は `PHASE_SETTINGS.parallel_verification.subPhases.security_scan.subagentTemplate` フィールドである。検証方法として `toContain('重複行回避')` または `toContain('重複')` で重複行回避を扱う文字列の存在を確認する。期待結果として重複行回避のセクションまたは指示文が現行の最小限の指示「セキュリティスキャンを実施してください。」に加えて追記されていること。失敗条件は変更後も重複行回避に関する記述が一切含まれていない場合。このテストシナリオはFR-R3の達成を直接確認する最高優先度の検証である。

### TC-R3-2: security_scanのテンプレートにFR識別子を含めて行を一意にする指示が含まれること

検証対象は `PHASE_SETTINGS.parallel_verification.subPhases.security_scan.subagentTemplate` フィールドである。検証方法として `toContain('識別子')` または `toContain('FR')` で固有識別子を行に含める手法の説明を確認する。期待結果としてFR番号・ファイルパス・関数名などの具体名を各評価行に含めて行を一意にすることが指示されていること。各評価対象を独立した内容で記述する方針がテンプレートに明記されていることにより、subagentが同一評価フレーズを繰り返すオペレーションを防止できる。失敗条件は各行を一意にする方法論の説明が含まれていない場合。

### TC-R3-3: security_scanのテンプレートにNG/OK対比例が両方含まれていること

## FR-R4テスト設計（performance_testサマリーセクションガイダンス）

### 検証の背景と根本原因との関連

FR-R4はFR-R2（manual_test）と構造的に同一の根本原因を持ち、「performance_testフェーズのsubagentTemplateにサマリーセクションの行数・コンテンツ密度ガイダンスが全く存在せず、subagentが3行程度のラベル行のみのサマリーを生成して密度不足の不合格となる」問題を解決する。パフォーマンステストの結果を記録する performance-test.md は計測値を含む充実した内容が求められるが、テンプレートの指示が最低限しかなかった。

performance_testに特有の要素として、計測対象処理・計測条件・応答時間の実測値・合否判定の4項目をそれぞれ1行以上で記述することが求められる。「対象処理: workflow_nextの呼び出し応答時間を10回計測した」や「計測結果: 平均45ms、最大120ms、最小30msを記録した」のような数値を含む行は実質行としてカウントされるため、このような記述例をテンプレートに含めることでsubagentを正しい方向に誘導できる。

追記箇所は914行目のperformance_test subagentTemplate内であり、追記内容はFR-R2と同内容の基本指示にperformance_test固有の計測関連キーワード（計測対象処理、計測条件、応答時間）を加えた形式とする。追記後は performance-test.md の `## サマリー` セクションが artifact-validator に合格し、実質行数が最低5行以上となることが期待される。

### TC-R4-1: performance_testのテンプレートに5行以上の実質行要件が数値で明示されていること

検証対象は `PHASE_SETTINGS.parallel_verification.subPhases.performance_test.subagentTemplate` フィールドである。検証方法として `toContain('5行以上')` でFR-R2と対称的な数値指示の存在を確認する。期待結果として「5行以上」という具体的な数値を含む指示が含まれていること。この対称的な追記はNFR-3（テンプレート追記の形式統一）の観点からも必要であり、manual_testとperformance_testの両フェーズで同一方針の実質行数ガイダンスが提供されることを保証する。失敗条件は行数要件が数値で明示されておらず曖昧な表現にとどまっている場合。

### TC-R4-2: performance_testのテンプレートに計測値を含む記述例が含まれること

検証対象は `PHASE_SETTINGS.parallel_verification.subPhases.performance_test.subagentTemplate` フィールドである。検証方法として `toContain('計測')` または `toContain('ms')` で計測値を含む具体的な記述例の存在を確認する。期待結果として「計測対象処理を記述する」「応答時間を含む数値例」のような performance_test 固有の文脈で記述されたサンプル行が含まれていること。脅威モデル（T-3）で指摘されたとおり、FR-R2の例示をそのままコピーするとperformance_test文脈に不適合になるため、このテストにより計測関連の語彙が正しく使われていることを確認する。失敗条件は計測値や応答時間に言及した記述例が全く示されていない場合。

## ビルドおよび統合テスト設計

### ビルドテストの位置づけとワークフロー手順との関係

ビルドおよび統合テストはnpmビルドコマンド（`npm run build`）によるトランスパイル成功とリグレッションスイートの非破壊確認を目的とする。これらのテストはimplementationフェーズでFR-R1〜FR-R4の全変更が完了した後に実施する手順となっており、ビルド成功後にMCPサーバーを再起動して変更を有効化するというオペレーションの前提確認として機能する。

Node.jsのモジュールキャッシュにより、definitions.tsを変更してもトランスパイルとサーバー再起動を経なければ変更は反映されない。このため、TC-BUILD-1でビルドの成功を確認し、TC-BUILD-2で既存テストの非破壊を確認してから初めてMCPサーバーの再起動と動作確認を進めることができる。再起動未実施のまま次フェーズに進むことは NFR-2 違反となる。更新タイムスタンプによるdist/definitions.jsの確認も合否判定の一部として含める。

### TC-BUILD-1: npm run buildが成功すること

検証対象は `workflow-plugin/mcp-server` ディレクトリでの `npm run build` コマンドの実行結果である。検証方法としてコマンドの終了コードが0であることを確認する。期待結果としてTypeScriptコンパイルエラーなしでdistディレクトリにjsファイルが生成されること。実施タイミングはimplementationフェーズでFR-R1〜FR-R4の全変更完了後である。失敗条件はdefinitions.tsのテンプレートリテラル内に閉じバッククォートや未エスケープの特殊文字が混入してコンパイルエラーが発生した場合。実装者はバッククォートで囲まれたテンプレートリテラル内に閉じバッククォートが入らないよう事前に確認する必要がある（NFR-2との整合）。

### TC-BUILD-2: 既存vitestスイートが全件パスすること

検証対象は `workflow-plugin/mcp-server/src/phases/__tests__/definitions.test.ts` を含む全既存テストスイートである。検証方法として `npx vitest run` の終了コードが0であることを確認する。期待結果としてFR-R1〜FR-R4の変更は既存フェーズの構造（フェーズ名・遷移順序・許可拡張子）を変更しないため、既存テストが全て引き続きパスすること。FR修正プロセスの各ラウンドを通じて既存テストの状態を保持することはNFR-1（既存フェーズへの副作用なし）の受け入れ基準であり、このテストによりリグレッションの発生を早期に検出できる。失敗条件はsubagentTemplateの文字列変更が意図せずTypeScriptのAST解析に影響を与えてコンパイルが失敗した場合。

## 非機能要件のテスト設計

### NFR-1検証（既存フェーズへの副作用なし）

確認内容としてFR-R1〜FR-R4の変更後も `definitions.test.ts` の全テストがパスすることを確認する。確認方法は `npx vitest run src/phases/__tests__/definitions.test.ts` を実行して終了コード0を得ることである。この確認によりフェーズ構造（PHASES配列・遷移順序・許可拡張子）が変更されていないことを保証し、ワークフロープロセス全体への影響が最小化されていることを検証する。期待結果として全検証が合格し、既存の全テストケースが通過すること。正常完了の判断基準はコマンドの終了コードが0であることであり、必要最小限のコマンド実行で確認できる。

### NFR-2検証（MCPサーバー再起動の実施確認）

確認内容としてdefinitions.tsへの文字列追記後に `npm run build` が正常完了し、MCPサーバーの再起動が実施されたことを確認する。確認方法はTC-BUILD-1の実施と、再起動後にworkflow_statusを実行して現在のフェーズが正しく表示されることの2段階で構成される。モジュールキャッシュの問題によりビルドのみでは変更が反映されないことをチェックリスト化し、再起動未実施のまま次フェーズへ進む事態を防ぐ。期待結果として再起動後の動作確認でサーバーが期待通りに応答すること。実行中のキャッシュを保持したままでは各要件の追記が即座に反映されない点に注意が必要である。

### NFR-3検証（テンプレート追記の形式統一）

確認内容として各FR追記ガイダンスが既存のmanual_testテンプレートのスタイルと一致していることを確認する。確認方法はコードレビューフェーズで人手による手動確認として実施し、自動テストではカバーしない。確認観点として箇条書き形式・NG/OK対比例の有無・数値明示の3点を各フェーズのテンプレートについて個別に評価する。一貫性の確保が subagent への指示の品質に直結するため、この評価は code-review.md の必須記載事項とする。文体統一と可読性については refactoring フェーズの最終確認として共通の評価基準を適用する。

### NFR-4検証（regression_testとtestingフェーズの対称性維持）

確認内容としてFR-R1の変更がtestingとregression_testの両フェーズに同等のガイダンスを提供していることを確認する。確認方法はTC-R1-1とTC-R1-4を並行して実施し、両テンプレートに同等の指示が含まれていることを検証する。この対称性確認はNFR-4の直接的な受け入れ基準であり、将来のラウンドで一方のフェーズのみに問題が再発するパターンを根絶するための継続的な検証として位置づける。両テストの合格が確認されてから初めて対称性要件を満たしたと判定する。

### NFR-5検証（変更の最小性）

確認内容として各変更が問題解決に必要な最小限の文言追加・変更に留まり、既存テンプレートの構造（フェーズ名・出力ファイルパス・必須セクション指示）が削除・置換されていないことを確認する。確認方法はコードレビューフェーズで差分を参照し、追記内容の範囲が仕様で定義された行目安（862行目・871行目・889行目・902行目・914行目）の近辺に限定されていることを確認する。既存構造が維持されていることはTC-R1-5とTC-R2-3が間接的に保証する。各要件の詳細な検証は上記の各TCが担い、本NFR-5検証はそれらを補完する位置づけとなる。

## テスト実施順序とワークフロー上の手順

テストの実施順序は以下の通りである。まずimplementationフェーズでFR-R1の変更実装完了後にTC-R1-1からTC-R1-5を実行して testing および regression_test テンプレートの変更を段階的に確認する。次にFR-R2の変更完了後にTC-R2-1からTC-R2-3を実行してmanual_testテンプレートへの追記を確認し、FR-R3の変更完了後にTC-R3-1からTC-R3-3を実行する。FR-R4の変更完了後にTC-R4-1とTC-R4-2を実行する。
