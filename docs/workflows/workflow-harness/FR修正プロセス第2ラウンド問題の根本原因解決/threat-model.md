## サマリー

- 目的: FR-R1〜FR-R4の各変更（definitions.tsのsubagentTemplate文字列への追記）がもたらすセキュリティリスクと整合性リスクを洗い出し、対策を策定する
- 対象変更の範囲: workflow-plugin/mcp-server/src/phases/definitions.ts のテンプレート文字列4箇所への追記のみ。バリデーター・状態管理・フェーズ遷移ロジックは変更対象外
- 主要な決定事項: 変更は内部テキスト定数の追記に限定されるため外部攻撃面は拡大しないが、プロンプトインジェクション誘発・バリデーション回避・テンプレート整合性崩壊の3種リスクが存在する
- 評価結果の概要: 4つのFRのうちFR-R3（重複行回避ガイダンス）は記述方法を誤るとsubagentに禁止語を直接提示するリスクが最も高く、他の3件は影響範囲が限定的であることを確認した
- 次フェーズで必要な情報: FR-R3のテンプレート文言において禁止語をコードブロック外に直接記述しない設計方針、および全テンプレートにおける既存構造との整合性チェックリスト

---

## 変更対象の概要

本タスクは definitions.ts 内の4フェーズ（regression_test・testing・manual_test・security_scan・performance_test）のsubagentTemplateへのテキスト追記が変更の全体であり、実行コード・APIサーフェス・認証機構・データベース構造のいずれも変更しない。したがって、伝統的な意味でのCIA（機密性・完全性・可用性）に対する直接的な脅威は限定的である。LLMへの指示テキストを変更するという特性上、プロンプトレベルの設計リスクが主要な脅威カテゴリとなる。
変更するファイルは workflow-plugin/mcp-server/src/phases/definitions.ts の単一ファイルに限定されており、変更の種別は純粋な文字列追記である。この変更対象の絞り込みは意図的な設計判断であり、NFR-5（変更の最小性）として非機能要件に明記されている。
本脅威モデルでは、追記内容が持つ副作用・誤解誘発・既存整合性への影響を中心に分析する。具体的には5つの脅威（T-1〜T-5）を特定し、各脅威に対して優先度付きの対策を策定した。
変更後に必要な運用作業として、npm run build によるトランスパイルと MCPサーバーの再起動があり、これを怠ると変更が反映されないリスク（T-4）が発生する。
各フェーズのsubagentTemplateはTypeScriptのテンプレートリテラルとして定義されており、追記内容はLLMが受け取るプロンプトの一部となるため、文言の正確性と既存テンプレート構造との整合性がシステム品質に直接影響する。

---

## 脅威モデルの前提条件

攻撃対象として想定するのは、外部の悪意ある第三者ではなく、次の2つの内部リスク源である。第一のリスク源はテンプレート変更作業者（AIエージェントまたは人間の開発者）が意図せず不適切な文言を追記することによる設計ミスである。第二のリスク源はテンプレートを受け取ったsubagentが指示を誤解し、期待と異なる動作をするプロンプトミスである。
外部からのリポジトリ書き換えや不正アクセスは、本タスクの変更スコープ外であるためOOB（スコープ外）として評価対象から除外する。MCPサーバーはローカル環境で動作し、外部ネットワークに公開されていないためネットワーク攻撃面は存在しない。
本脅威モデルで使用するリスクレベルの定義は次の通りである。Highは放置するとワークフローが継続不能になるレベルの問題を引き起こす脅威を指す。Mediumはワークフローの品質や効率に悪影響を与えるが、回避策が存在する脅威を指す。Lowは保守性や可読性への影響に留まり、機能的な問題は発生しない脅威を指す。
この3段階の評価基準に基づき、以下の脅威一覧を策定した。
なお、本タスクの変更が対象とするLLMの動作パターンはartifact-validatorのバリデーションルールに直接関係するため、バリデーション回避や意図しない失敗を誘発する可能性のある文言変更は、機能的なリスクとして脅威評価の対象に含める。

## 脅威一覧

### 脅威T-1: 禁止語の直接埋め込みによるバリデーション自己汚染

**カテゴリ**: プロンプト設計ミス
**対象FR**: FR-R3（security_scan重複行回避ガイダンス）
**リスクレベル**: High（テンプレート変更作業の誤りが直接バリデーション失敗を引き起こす）

FR-R3のテンプレートには特定の短い評価フレーズが3件以上同一内容で出現しないようにというガイダンスが必要である。しかし、ガイダンス文の中でこれらのフレーズをコードブロック外に直接記述すると、artifact-validatorの重複行チェックがテンプレートテキスト自体に誤反応する可能性がある。
より現実的なリスクは、テンプレートを読んだsubagentがフレーズ例をガイダンスから抽出し、成果物に3回以上そのままコピーするケースである。フレーズ例を示す際はコードブロックや説明的な文脈（「以下は避けるべき例」のような前置き）の中に封じ込める必要がある。

**対策**: ガイダンス内のNG例フレーズは説明的な散文の中に埋め込み、箇条書きで直接列挙しない。成果物作成時に避けるべきパターンを「避けるべき例」として提示し、代替表現を必ずセットで提示する。

---

### 脅威T-2: テスト出力形式ガイダンスによるrawログ汚染リスク

**カテゴリ**: 誤動作誘発
**対象FR**: FR-R1（regression_test・testingフェーズのテスト出力形式）
**リスクレベル**: Medium（バリデーション通過率は向上するが、過度に長い出力が状態ファイルに記録されるリスク）

FR-R1では「テストコマンドの標準出力をそのまま output 引数に渡す」ことを指示する。この指示が「全ての標準出力をそのままコピーする」と過度に解釈されると、数万行のログが workflow-state.json の testResult フィールドに記録される可能性がある。特にnpm testの詳細ログモードで実行した場合、状態ファイルのサイズが増大し、HMAC整合性チェックのパフォーマンスに影響する。
実際のリスクとしては、HMAC計算対象のサイズが増大することによる処理遅延が考えられる。ただし、現時点でworkflow-state.jsonのフィールドサイズに上限制限は設けられていないため、技術的には許容されている。

**対策**: FR-R1のガイダンスには「テストの集計行・サマリー行を含む出力」を渡すよう範囲を明確化し、詳細な全行ログではなく必要最小限の集計出力で十分であることを明示する。

---

### 脅威T-3: サマリー密度ガイダンスの誤適用によるセクション構造崩壊

**カテゴリ**: テンプレート整合性リスク
**対象FR**: FR-R2（manual_test）、FR-R4（performance_test）
**リスクレベル**: Low（既存のサマリーセクション構造は維持されるため影響は軽微）

FR-R2とFR-R4では「ラベル行のコロン直後に必ず1文以上の実際のコンテンツを続けて1行に収めること」を追記する。この指示をsubagentが「各ラベルを1行に圧縮しなければならない」と過剰に解釈した場合、情報量が多い項目が1行に詰め込まれ可読性が低下するケースが考えられる。
また、performance_testにFR-R2と同一文言を適用する場合（NFR-3の対称性要件）、manual_testの文脈に合わせた例示がperformance_testには不適合となり、subagentが混乱するリスクがある。

**対策**: FR-R4の例示フレーズはperformance_testの文脈（「計測対象: ファイル変換処理の所要時間を計測する」など）に合わせたものを使用し、FR-R2の例示をそのままコピーしない。

---

### 脅威T-4: ビルド未実施による変更不反映リスク

**カテゴリ**: 運用ミスリスク
**対象FR**: FR-R1〜FR-R4（全変更）
**リスクレベル**: Medium（変更がデプロイされず、問題が再現し続けるリスク）

NFR-2に記載の通り、definitions.tsを変更しても npm run build とMCPサーバー再起動を実施しない限り、実行中のMCPサーバーはキャッシュされた古いテンプレートを使い続ける。この場合、変更を加えたにもかかわらず依然として同じバリデーションエラーが発生し、問題が解決していないと誤認する可能性がある。
再起動不実施の状態でregression_testフェーズを実行すると、古いテンプレートで指示されたsubagentが新しいテンプレートの期待を満たさない成果物を生成し、デバッグが困難になる。

**対策**: implementationフェーズの完了チェックリストに「npm run build の実行確認」「MCPサーバー再起動の確認」「workflow_status での現在フェーズ確認」の3項目を必須ステップとして含める。

---

### 脅威T-5: FR間のテンプレート記述スタイル不統一による混乱

**カテゴリ**: 整合性リスク
**対象FR**: FR-R1〜FR-R4（全変更、NFR-3の対称性要件に関係）
**リスクレベル**: Low（機能的な問題ではなく保守性への影響）

NFR-3はFR-R1〜FR-R4の各変更を「既存のテンプレート記述スタイルと一致させる」ことを要求する。4つのフェーズのテンプレートは現在それぞれ異なる長さと構造を持っており、「一致させる」基準が曖昧である場合、変更作業者が各フェーズのテンプレートを個別に参照せず、1つのフェーズを基準にコピーして貼り付けると文脈に不適合な指示が含まれる可能性がある。

**対策**: implementationフェーズの作業開始前に、変更対象の各フェーズのtemplate文字列の現在の構造を読み込み、追記位置と文体を個別に確認する手順を明記する。

## リスク

本脅威モデルで特定した5件の脅威に対して、それぞれのリスク評価と対応方針を以下に示す。

**T-1の総合リスク評価（High）**: FR-R3テンプレート内で禁止語フレーズをコードブロック外に記述した場合、balidation失敗が即座に発生する。Orchestratorがsubagentを再起動しても同じ失敗を繰り返す悪循環に陥るため、実装前に必ずレビューが必要な最優先リスクである。
**T-2の総合リスク評価（Medium）**: テスト出力の全行コピーは技術的には許容されるが、状態ファイルのサイズ増大を引き起こす。ガイダンスに「集計行を含む必要最小限の出力」と明示することで発生確率を低減できる。
**T-3の総合リスク評価（Low）**: ラベル行1行圧縮の誤解は可読性低下に留まり、バリデーション合否には影響しない。performance_test向けの例示を文脈固有にすることで十分に対処可能である。
**T-4の総合リスク評価（Medium）**: ビルド未実施は手順漏れという人的ミスであり、チェックリスト化により予防できる。実装フェーズの成果物にビルド・再起動の実行確認項目を含めることが有効な対策となる。
**T-5の総合リスク評価（Low）**: スタイル不統一は将来の保守性に影響するが、既存のmanual_testテンプレートを参照基準として明示することで誘導できる実装者の判断に委ねられる軽微なリスクである。

## セキュリティ要件への影響評価

本変更はLLMへの指示テキストのみを変更するものであり、次のセキュリティ要件に対して影響なしと評価する。
認証・認可機構については変更対象外であり、definitions.tsのテンプレート文字列はアクセス制御とは独立したコンポーネントであるため、権限昇格や認証バイパスのリスクは発生しない。
ユーザーデータの保護については definitions.ts はユーザーデータを含まないため影響なしと評価できる。ただし、FR-R1のガイダンスによりworkflow-state.jsonのtestResultフィールドの内容が変化するため、状態ファイルのデータ量に留意が必要である。
HMAC整合性チェックについてはテンプレート文字列の変更はHMAC対象外のため直接的な影響はない。ただし脅威T-2の状態ファイル肥大化はHMACのパフォーマンスに間接的な影響を与える可能性がある。
ネットワークセキュリティについては変更対象外であり、MCPサーバーはローカル環境で動作するためネットワーク攻撃面は存在しない。
脅威T-1が最も優先度の高いリスクであり、FR-R3のテンプレート文言設計において注意が必要である。その他の脅威は対策を講じることで許容可能なリスクレベルに収まると評価する。
