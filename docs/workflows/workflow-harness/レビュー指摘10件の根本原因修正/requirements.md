# 要件定義: レビュー指摘10件の根本原因修正

## サマリー

本要件定義では、ワークフロープラグインに対するレビュー指摘事項10件と大規模プロジェクト対応欠陥5件について、明確な修正要件を策定します。

調査フェーズにおいて実施された詳細な分析により、これらの問題が「ルール定義は存在するが技術的強制が不十分」という共通パターンに帰結することが判明しました。

修正対象は大きく二つのカテゴリに分類されます。第一はMCPサーバーのソースコードを修正する必要がある問題群であり、第二はCLAUDE.mdへのガイダンス追加で対応可能な問題群です。

コード修正が必要な問題としては、承認ゲート管理の一元化、テスト品質検証機能の追加、AST解析活用の強化、タスクサイズ選択肢の復活、依存関係解析の改善が挙げられます。

ドキュメント修正で対応可能な問題としては、subagent判断ログの追加、完了表現制御の技術的制約明記、リグレッションテストフィルタリング指針、並列フェーズ依存関係コメント修正、スコープ設定ガイダンス、コンテキスト予算管理推奨値、research探索戦略、subagent分割基準が含まれます。

最も効果的な改善策は、small/mediumタスクサイズの復活です。現在は全タスクが19フェーズ強制となっていますが、typo修正や1行バグ修正のような軽微な変更に対して19フェーズを実行することは開発者の生産性を著しく低下させます。

幸いにも、definitions.tsには既にPHASES_SMALLとPHASES_MEDIUMの定義が残存しているため、実装コストは最小限に抑えられます。

問題8の並列フェーズ依存関係については、技術的には既に強制されていることが判明しました。complete-sub.tsのロジックにより依存関係チェックが実装されているものの、definitions.tsのコメントが「推奨」と記述されているため誤解を招いています。

問題3の完了表現制御については、Claude Codeのhookシステムの技術的制約により完全な解決は不可能です。hookはツール呼び出しの監視のみが可能であり、AIが生成する応答テキストそのものは検証範囲外となります。

欠陥1から欠陥5までの大規模プロジェクト対応については、段階的なアプローチを採用します。短期的にはCLAUDE.mdへのガイダンス追加によりAIエージェントの振る舞いを改善し、長期的にはTypeScript Compiler APIの活用やストリーミング処理の導入を検討します。

全修正完了後には、mcp-server/ディレクトリ内で実行されるテストスイート732件が全て合格することが受け入れ基準となります。

修正の優先順位は、影響範囲と実装コストを考慮して決定されます。問題6のタスクサイズ復活が最優先事項であり、続いて問題1の承認管理一元化、問題4のテスト品質検証追加、問題5のAST解析強化、各種ドキュメント修正の順で実施します。

本プロジェクトの成功基準は、技術的改善とドキュメント整備の両面から定義されます。技術面では既存テストの全合格と新規テストの追加、ドキュメント面ではCLAUDE.mdの明確化とAIエージェントの振る舞い改善が求められます。

これらの修正により、ワークフロープラグインは小規模から大規模まで幅広いプロジェクトに対応可能となり、品質保証と開発生産性のバランスを実現します。

## 背景

ワークフロープラグインは、Claude Codeにおける開発タスクを19フェーズの構造化されたワークフローで管理するシステムです。research、requirements、planning、design、test、implementation、quality check、verificationの各段階を経て、品質の高いコード変更を実現することを目的としています。

プロジェクトの成熟に伴い、外部レビュアーからの指摘事項が蓄積されました。これらの指摘は、設計の一貫性、品質保証の完全性、大規模プロジェクトへの対応力といった多岐にわたる観点からの改善提案を含んでいます。

当初のレビュー指摘は10件でしたが、調査を進める中で大規模プロジェクトへの対応に関する5つの構造的欠陥が追加で発見されました。これらの欠陥は相互に関連しており、research探索戦略の欠如がスコープ設定の不正確性を招き、それが依存関係解析の信頼性低下につながり、最終的にsubagentのコンテキスト管理問題を引き起こすという連鎖関係が明らかになりました。

問題の根本には、「ルールは定義されているが技術的に強制されていない」という共通パターンが存在します。例えば、CLAUDE.mdには詳細な完了宣言ルールが記述されていますが、これを検証するhookは存在しません。同様に、テスト品質基準は文書化されているものの、test_implフェーズでの自動検証機構は未実装です。

一方で、問題8のように「既に技術的に強制されているにもかかわらず、ドキュメントが不正確なために誤解を招いている」という逆のパターンも発見されました。並列フェーズの依存関係はcomplete-sub.tsで厳格にチェックされていますが、definitions.tsのコメントは「推奨」と記述されており、実装とドキュメントの不一致が生じています。

最も深刻な問題は、全タスクに19フェーズを強制している現状です。小規模な変更に対しても、脅威モデリング、ステートマシン設計、フローチャート作成、UI設計、リグレッションテスト等の全工程を要求することは、開発効率を著しく損ないます。

幸いなことに、過去のバージョンではsmall(8フェーズ)とmedium(14フェーズ)のタスクサイズが存在しており、そのコードはdefinitions.tsに残存しています。「品質一貫性のために廃止」という方針が採られましたが、実際には柔軟性の損失による弊害の方が大きいことが運用上明らかになっています。

大規模プロジェクト対応の欠陥については、技術的な根本解決には時間を要します。dependency-graph.tsを正規表現ベースからTypeScript Compiler APIベースに書き換える作業や、スコープ設定のストリーミング処理化は、影響範囲が広く慎重な設計が必要です。

したがって、短期的対応としてCLAUDE.mdへのガイダンス追加を優先し、AIエージェントの振る舞い改善を図ります。具体的には、research探索戦略の段階的手順、スコープ設定時のファイル数制限、subagent起動時のmax_turns設定、分割基準の明確化といった実践的な指針を提供します。

問題3の完了表現制御については、技術的制約として受け入れるしかありません。Claude Codeのhookシステムは、Bash、Edit、Write等のツール呼び出しを監視することはできますが、AIが生成するテキスト応答を事前に検証する機能は持ちません。

この制約は、hookの実行タイミングに起因します。AIの応答は、ツール呼び出しが行われる前に生成されるため、hookによる事前チェックが構造上不可能です。したがって、完了表現ルールはAIの自己規律に依存せざるを得ません。

本プロジェクトでは、これらの背景を踏まえた上で、技術的に実現可能な範囲での最大限の改善を目指します。コード修正とドキュメント整備を並行して進め、短期的な効果と長期的な品質向上の両立を図ります。

## 機能要件

### FR-1: 承認ゲート管理の一元化

現在、requirements、design_review、test_designの3フェーズはREVIEW_PHASESに定義されているが、code_reviewのみがnext.ts内で個別に承認チェックされている。この設計の不統一を解消し、全ての承認ゲートをREVIEW_PHASESで一元管理する。

definitions.ts L311のREVIEW_PHASES配列に'code_review'を追加する。これにより、4つの承認ゲート全てが単一の定義箇所で管理されるようになる。

next.ts L419-428のcode_review個別チェックロジックを削除する。一元化後は、既存のREVIEW_PHASESチェックロジックがcode_reviewにも適用されるため、個別実装は冗長となる。

承認管理の一元化により、将来的に新たな承認ゲートを追加する際の実装コストが削減される。REVIEW_PHASESに追加するだけで、承認待ち状態の管理、workflow_approve APIによる承認処理、next.ts内の遷移チェックが自動的に適用される。

### FR-2: subagent判断ログの記録

CLAUDE.md L153-194のsubagent起動テンプレートに「## 選択理由」セクションを追加する。このセクションでは、特定のsubagent_typeとmodelを選択した根拠を明示的に記録する。

記録すべき情報には、タスクの性質に基づく判断(探索型か実装型か等)、入力ファイルのサイズと複雑度、期待される出力の種類、モデル選択の理由(精度重視かコスト重視か)が含まれる。

判断ログは成果物ファイルの冒頭に記載し、後続フェーズのsubagentが前フェーズの意思決定プロセスを理解できるようにする。これにより、ワークフロー全体の透明性と監査可能性が向上する。

### FR-3: 完了表現制御の技術的制約明記

CLAUDE.md L497-522の完了宣言ルールセクションに、技術的制約に関する注記を追加する。具体的には、「このルールはhookによる技術的強制はできず、AIの自己規律に依存する」という制約事項を明示する。

制約の理由として、Claude Codeのhookシステムがツール呼び出しのみを監視対象とし、AI応答テキストは検証範囲外であることを説明する。hookの実行タイミングが、AI応答生成後のツール呼び出し時点であることも補足情報として記載する。

代替策として、フェーズ完了報告テンプレートを明確化する。「【{フェーズ名}フェーズ完了】- 完了した作業: {作業内容}」という形式を提示し、AIが従いやすい構造化されたフォーマットを提供する。

### FR-4: テスト品質検証機能の追加

PHASE_TO_ARTIFACTマッピングに'test_impl'フェーズを追加する。成果物種別として'test-files'を指定し、複数のテストファイルを検証対象とする。

test-file-validator.tsを新規作成し、基本的なテスト品質チェックを実装する。検証項目には、テストファイルが.test.ts/.spec.ts拡張子を持つこと、describe/it/test等のテスト関数が存在すること、expect/assert等のアサーション関数が含まれることが含まれる。

より高度な検証として、各テストケースが少なくとも1つ以上のアサーションを含むことをAST解析で確認する。テストファイル内のテストケース数をカウントし、最低限の網羅性基準(例: 3テストケース以上)を満たすかチェックする。

artifact-validator.ts内のvalidateArtifact()関数を拡張し、test_implフェーズの場合にtest-file-validator.tsを呼び出すロジックを追加する。検証失敗時には、具体的な改善提案を含むエラーメッセージを返す。

next.ts内のtest_impl→implementation遷移処理に、テスト品質検証のチェックを追加する。検証が失敗した場合は、フェーズ遷移をブロックし、テストの改善を促すエラーを返す。

### FR-5: AST解析活用の強化

ast-analyzer.tsに、メソッドbodyの実質的な内容を評価する関数群を追加する。第一の関数はgetFunctionStatementCount()であり、関数bodyのstatement数をカウントする。第二の関数はhasReturnStatement()であり、関数がreturnステートメントを含むかチェックする。

第三の関数はisStubImplementation()であり、メソッドが実質的なロジックを持たないスタブであるかを判定する。判定基準は、statement数が2以下かつreturnステートメントがnullまたは未実装マーカーを返す場合とする。

design-validator.ts L78-103のスタブ検出ロジックを、正規表現ベースからAST解析ベースに書き換える。ast-analyzer.tsの新規関数を呼び出し、各メソッドの実装状況を精密に評価する。

AST解析の強化により、偽陰性(スタブなのに検出漏れ)と偽陽性(正常な実装をスタブ判定)の両方を削減する。特に、コメントのみのメソッドや、エラースローのみのメソッドを確実に検出できるようになる。

### FR-6: タスクサイズ選択肢の復活

CLAUDE.md L85の「small/mediumサイズは廃止されました」という記述を撤回し、タスクサイズ選択肢を正式に復活させる。small、medium、largeの3つのサイズを再び利用可能とする。

各サイズの適用基準を明確に定義する。smallタスク(8フェーズ)は、typo修正、1行バグ修正、設定ファイル微調整等、変更行数が10行未満かつ影響範囲が単一ファイルのみの場合に適用する。

mediumタスク(14フェーズ)は、既存機能の改善、中規模リファクタリング、API仕様変更等、変更行数が100行未満かつ影響範囲が単一モジュール内の場合に適用する。

largeタスク(19フェーズ)は、新機能追加、アーキテクチャ変更、セキュリティ対策等、変更行数が100行以上または影響範囲が複数モジュールに及ぶ場合に適用する。

タスクサイズの判断指針をCLAUDE.mdに追加し、AIエージェントが適切なサイズを選択できるようガイダンスを提供する。判断に迷った場合は、より大きいサイズを選択することを推奨する。

definitions.ts内のPHASES_SMALLとPHASES_MEDIUMの定義は既に存在するため、コード変更は最小限となる。start.ts内のtaskSizeパラメータ処理ロジックも既に実装されており、デフォルト値をlargeに維持しつつ、small/mediumの明示的指定を許可する。

### FR-7: リグレッションテストフィルタリング指針の追加

CLAUDE.md L651付近のリグレッションテストセクションに、スコープベースのテストフィルタリングに関するガイダンスを追加する。影響範囲内のテストのみを実行する方針を明記する。

具体的には、「regression_testフェーズでは、workflow-get-test-info APIで取得したテスト一覧から、スコープ内のファイルに関連するテストのみを選択して実行する」という指針を提示する。

テストとソースファイルの関連付けは、ファイル名ベースとディレクトリベースの両方で行う。例えば、src/backend/domain/user.tsが変更された場合、src/backend/tests/unit/user.test.tsとsrc/backend/tests/integration/user-service.test.tsの両方を実行対象とする。

影響範囲が広範囲に及ぶ場合(例: 共通ユーティリティの変更)は、全テストを実行することを推奨する。判断基準として、影響ファイル数が20ファイル以上、または複数のドメイン境界をまたぐ変更の場合は全テスト実行とする。

将来的な実装として、workflow-record-test APIを拡張し、テストファイルと依存ソースファイルのマッピング情報を記録する機能を追加することを計画に含める。これにより、より精密なテスト選択が可能になる。

### FR-8: 並列フェーズ依存関係コメントの修正

definitions.ts L165のコメントを「planning: threat_modelingの完了を待機(技術的に強制)」に修正する。「推奨」という誤解を招く表現を削除し、実際の動作である「強制」を明記する。

CLAUDE.md L649-661の並列フェーズセクションに、サブフェーズ依存関係が技術的に強制されることを明記する。「SUB_PHASE_DEPENDENCIESで定義された依存関係は、complete-sub.tsにより厳格にチェックされ、依存先が未完了の場合はエラーが返される」という説明を追加する。

各並列フェーズグループの依存関係を表形式で整理する。parallel_analysis: planning→threat_modeling、parallel_design: flowchart→state_machine、ui_design→flowchart、parallel_quality: code_review→build_checkという関係性を明示する。

parallel_verificationには依存関係が定義されていないことも明記し、manual_test、security_scan、performance_test、e2e_testは任意の順序で実行可能であることを説明する。

### FR-9: スコープ設定ガイダンスの追加

CLAUDE.md L959-987の「workflow set-scope」セクションに、大規模プロジェクト向けのガイダンスを追加する。スコープ設定時のパフォーマンスとメモリ効率に関する注意事項を提示する。

globパターン使用時の推奨事項として、「必要最小限に絞り込む」「src/**/*.tsではなくsrc/backend/domain/**/*.tsのように具体化する」「tests/ディレクトリ等の巨大なディレクトリは避ける」といった指針を提供する。

ファイル数制限として、「1000ファイル以上のスコープ設定は避け、複数のタスクに分割することを推奨」という基準を明記する。メモリ使用量の観点から、1タスクあたりのスコープは500ファイル以下が理想的であることを説明する。

dirsパラメータ使用時の注意点として、「ディレクトリ全体を指定する場合は、事前にls -Rでファイル数を確認する」「node_modules、.git、dist等の大規模ディレクトリは除外する」といった実践的アドバイスを追加する。

段階的なスコープ設定戦略として、「まず影響が確実な数ファイルのみをスコープに含める」「依存関係解析の結果を見て追加が必要なファイルを判断」「段階的にスコープを拡大」というアプローチを推奨する。

### FR-10: コンテキスト予算管理推奨値の追加

CLAUDE.md L153-194のsubagent起動テンプレートに、max_turnsパラメータの推奨値を追加する。通常のタスクでは「max_turns: 10」、大規模プロジェクトでは「max_turns: 5」を推奨する。

max_turnsの意味と効果を説明するセクションを追加する。「max_turnsはsubagentの最大ターン数を制限し、コンテキスト消費を抑制する」「1ターンあたり平均5K-10Kトークンを消費すると想定」「max_turns: 10の場合、最大50K-100Kトークンの消費に制限」といった情報を提供する。

フェーズごとの推奨max_turns値を表形式で整理する。researchとplanningは探索的作業のため高めの値(15)、test_implとimplementationは集中的作業のため中程度(10)、refactoringとcode_reviewは短時間で完了すべきため低めの値(5)を推奨する。

コンテキスト超過時の対処法として、「max_turnsに達した場合は、subagentが途中で停止する」「その場合は、タスクを複数のsubagentに分割する」「各subagentに担当範囲を明確に指定する」というガイダンスを提示する。

### FR-11: research探索戦略ガイダンスの追加

CLAUDE.md L153-194のresearchフェーズ用subagent起動テンプレートに、「## 探索戦略」セクションを追加する。段階的な探索手順を明示的に指示する。

ステップ1として、プロジェクト概要把握を位置づける。「README.md、package.json、tsconfig.json等のルート設定ファイルを読み、プロジェクトの技術スタックと構成を理解する」という指示を含める。

ステップ2として、ディレクトリ構成把握を設定する。「ls -Rまたはglobツールでディレクトリ構成を把握し、src/、tests/、docs/等の主要ディレクトリの役割を特定する」という手順を提示する。

ステップ3として、関連モジュール特定を配置する。「grepツールでタスクに関連するキーワードを検索し、変更が必要なファイルの候補を絞り込む」という戦略を提案する。

ステップ4として、詳細調査を位置づける。「readツールで特定されたファイルを読み込み、実装の詳細、依存関係、テストの状況を分析する」という最終ステップを定義する。

探索の優先順位として、「広く浅く→狭く深く」という方針を推奨する。最初は全体像把握に集中し、詳細な実装内容は必要最小限のファイルのみを読むことで、コンテキスト消費を抑制する。

### FR-12: subagent分割基準ガイダンスの追加

CLAUDE.md L153-194のsubagent起動テンプレートに、「## 分割基準」セクションを追加する。入力ファイルサイズとコンテキスト量に基づく分割指針を提供する。

基本的な分割基準として、「入力ファイルが5万行を超える場合は、章ごと、セクションごと、またはモジュールごとにsubagentを分割する」という閾値を設定する。

### FR-13: 依存関係解析の改善

dependency-graph.tsの依存関係解析ロジックを、正規表現ベースからTypeScript Compiler APIベースに書き換える。これは長期的な改善項目として位置づけ、段階的な実装を計画する。

第二段階として、TypeScript Compiler APIを導入する。ts.createProgram()でプロジェクト全体のプログラムを構築し、ts.preProcessFile()で各ファイルの依存関係を抽出する。

第三段階として、path mapping(tsconfig.jsonのpaths設定)に対応する。ts.getResolvedModuleName()を使用して、エイリアスパスを実際のファイルパスに解決する。

第四段階として、型レベルの依存関係も検出対象に含める。型定義ファイル(.d.ts)のimportや、type-only import(`import type { } from`)も依存関係グラフに追加する。

各段階の完了条件として、既存のテストスイートが全て合格することに加え、新規のテストケースを追加して新機能の動作を検証する。特に、実際のTypeScriptプロジェクト(このワークフロープラグイン自体)で依存関係を正確に抽出できることを確認する。
