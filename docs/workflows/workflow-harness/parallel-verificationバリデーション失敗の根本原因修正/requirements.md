## サマリー

- 目的: parallel_verificationの4サブフェーズ（manual_test・security_scan・performance_test・e2e_test）で繰り返されるバリデーション失敗を根本解決するため、CLAUDE.mdの説明文誤りとdefinitions.tsのテンプレートガイダンス誤りを修正する要件を定義する。
- 調査フェーズで特定された根本原因は3点: (1) artifact-validator.tsの実際の角括弧検出パターン（`[#xxx#]`形式のみ）とCLAUDE.md記述の乖離、(2) サマリーセクション実質行数不足を引き起こすラベルのみ行の誤認識、(3) 上記2点がsubagentTemplateを通じてsubagentに伝播し誤った回避策を取らせるフィードバックループ。
- 主要な決定事項: CLAUDE.mdの角括弧禁止説明を正確な仕様に更新し、definitions.tsのFR-3・FR-4テンプレートガイダンスを修正することで、subagentが正確な制約情報に基づいて成果物を作成できる環境を整備する。
- 次フェーズで必要な情報: planning（spec.md作成）では修正対象ファイル・修正方針・期待されるテスト結果の3点を具体的に記述すること。設計フェーズではCLAUDE.mdとdefinitions.tsの両方の修正箇所を明記したステートマシン図・フローチャートを作成すること。

---

## 機能要件

### FR-A: CLAUDE.mdの角括弧プレースホルダー説明文修正

CLAUDE.mdの「成果物品質要件（artifact-validator準拠）」セクションに含まれる角括弧プレースホルダーの説明文を、artifact-validator.tsの実際の検出パターンと整合した内容に更新しなければならない。

現在の記述は「コードフェンス外のMarkdown本文（散文テキスト・箇条書き等）に記述された角括弧が禁止」と過剰な制約を示しているが、実際の検出パターンは`[#xxx#]`形式のみである。この乖離が「通常の角括弧も禁止」という誤解をsubagentに与えている。

修正後の説明文は「禁止されるのは`[#任意テキスト#]`形式の角括弧プレースホルダーのみであり、通常の角括弧・配列アクセス記法・正規表現文字クラスはコードフェンス外でも許可される」という内容にする必要がある。

また、インラインコード（バックティック囲み）も除外対象であることを説明文に明記することで、subagentが安全な記述方法を正確に理解できるようにする。

### FR-B: definitions.tsのFR-3・FR-4ガイダンス修正

definitions.tsのe2e_testサブフェーズテンプレート（FR-3）とperformance_testサブフェーズテンプレート（FR-4）に含まれる「コードフェンス外の角括弧禁止」という記述を削除または修正しなければならない。

FR-3では「ファイル名列挙時の配列記法禁止と散文形式への誘導」という不要な制約が記述されており、subagentがファイル名を自然な形式で記述できない状況を作っている。正確な内容は「禁止されるのは`[#xxx#]`形式のプレースホルダーのみ」に修正する。

FR-4では「複数シナリオの計測日時行の一意化要件と角括弧禁止」という記述があるが、角括弧禁止の部分が過剰制約となっている。一意化要件（有効な対策）は維持しつつ、角括弧禁止ガイダンスのみを正確な説明に置き換える。

### FR-C: subagentTemplateの成果物品質要件ガイダンス整合性確保

subagentTemplateに転写されている「成果物品質要件」セクションの角括弧説明が、FR-AでCLAUDE.mdを修正した後も一貫した内容になっていることを確認しなければならない。

subagentTemplateはCLAUDE.mdのテキストから転写されているため、CLAUDE.md修正後にテンプレート内の記述も同じ内容になっているか検証する必要がある。特に以下のサブフェーズのテンプレートが対象となる: manual_test、security_scan、performance_test、e2e_test。

テンプレート内に残存する「コードフェンス外の角括弧禁止」という誤記述を特定して全て修正する。

### FR-D: サマリーセクション実質行数ガイダンスの精度向上

parallel_verificationの各サブフェーズテンプレートで提供しているサマリー行数ガイダンスが、実質行カウントの仕様（コロン後コンテンツなし行は非カウント）を正確に反映していることを確認しなければならない。

FR-11（manual_test総合評価ガイダンス）とFR-12（security_scan行数確保計算ロジック）は既に正確な情報を提供しているため、これらは維持する。ただし、他のサブフェーズ（performance_test・e2e_test）のサマリーガイダンスに同様の実質行カウント注意事項が含まれていない場合は追加する。

## 非機能要件

### 変更の後方互換性

CLAUDE.mdの説明文修正はdocumentationの修正であり、実行時の動作には影響しない。artifact-validator.tsのコード変更は含まれないため、既存のバリデーション済み成果物は引き続き有効であり続けなければならない。

### テスト通過率の維持

definitions.tsの修正後、既存のテストスイート（workflow-plugin/mcp-server配下のテスト）が全て通過しなければならない。修正前の通過率（945/945）を下回ってはならない。

### MCPサーバー再起動の要否

definitions.tsを修正した場合、MCPサーバーの再起動が必要である。コア定義ファイルを変更した後は`npm run build`でトランスパイルを実行し、サーバーを再起動してから次フェーズに進まなければならない。

### 修正範囲の最小化

本タスクで修正するファイルはCLAUDE.mdとdefinitions.tsの2ファイルに限定する。artifact-validator.tsの実装変更は含まない。スコープ外のファイル（state-manager.ts・artifact-validator.ts等）への変更は禁止する。

## 受入条件

### AC-1: CLAUDE.md角括弧説明の正確性

CLAUDE.mdの「成果物品質要件」セクションを読んだ際に、「禁止されるのは`[#xxx#]`形式のプレースホルダーのみであり、通常の角括弧はコードフェンス外でも許可される」という内容が明確に理解できること。現在の「コードフェンス外の全角括弧禁止」という誤解を招く記述が削除されていること。

### AC-2: definitions.tsテンプレートの整合性

e2e_testとperformance_testのsubagentTemplateを読んだsubagentが、通常の角括弧をコードフェンス外で使用することを不必要に回避しようとしないこと。FR-3・FR-4の「配列記法禁止」という記述が削除または正確な説明に置き換えられていること。

### AC-3: テストスイートの無劣化

`cd workflow-plugin/mcp-server && npm test`を実行した際に、修正前と同じかそれ以上のテスト通過数を示すこと（基準: 945件以上）。

### AC-4: サマリーガイダンスの有効性

parallel_verificationの全4サブフェーズのsubagentTemplateに、コロン後コンテンツなし行が実質行としてカウントされない旨のガイダンスが明示的に含まれていること。FR-11・FR-12が既存の対応を担っているため、performance_testとe2e_testの対応状況を確認して不足があれば追加すること。

### AC-5: 修正の一貫性

CLAUDE.mdとdefinitions.tsの修正内容が矛盾しないこと。両ファイルで角括弧プレースホルダーの禁止形式として`[#xxx#]`が統一して記述されていること。
