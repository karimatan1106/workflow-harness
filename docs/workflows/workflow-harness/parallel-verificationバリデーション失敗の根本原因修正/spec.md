## サマリー

- 目的: parallel_verificationの4サブフェーズで繰り返されるバリデーション失敗を根本解決するため、CLAUDE.mdとdefinitions.tsの説明文誤りを修正する仕様を定義する。
- 根本原因の確認: artifact-validator.tsの実際の角括弧検出パターンは `/\[#[^\]]{0,50}#\]/g` であり、ハッシュ記号付き形式のみが対象となる。通常の角括弧・配列アクセス記法・正規表現文字クラスは検出されない。
- 主要な決定事項: CLAUDE.mdのsubagentTemplate内の「角括弧プレースホルダー禁止」説明を正確な仕様に更新し、definitions.tsのFR-3（e2e_test）とFR-4（performance_test）テンプレートの誤ったガイダンスを削除または修正する。
- スコープ制限: 変更対象はCLAUDE.mdとdefinitions.tsの2ファイルに限定し、artifact-validator.tsのコード変更は含まない。
- 次フェーズで必要な情報: design（設計レビュー）では修正内容の整合性確認と受入条件への適合確認が必要となる。state-machine.mmdとflowchart.mmdでは変更対象の状態遷移とデータフローを記述すること。

## 概要

### 問題の背景

parallel_verificationの各サブフェーズ（manual_test・security_scan・performance_test・e2e_test）のsubagentが、バリデーション失敗を繰り返す現象が発生している。調査の結果、以下の3つの説明文誤りが根本原因として特定された。

1. CLAUDE.mdのsubagentTemplate内の「角括弧プレースホルダー禁止」説明が過剰な制約を記述しており、「コードフェンス外の全角括弧が禁止」という誤解をsubagentに与えている。実際の検出パターンは `[#xxx#]` 形式のみである。

2. definitions.tsのFR-3（e2e_testテンプレート）に「コードフェンス外での配列記法は禁止」という記述があり、subagentが通常の角括弧を回避しようとして不自然な散文を生成し、かえってセクション密度不足を引き起こすケースがある。

3. definitions.tsのFR-4（performance_testテンプレート）にも同様の「コードフェンス外での配列記法は禁止」という記述があり、ファイル名列挙時の不要な制限となっている。

### 検出パターンの正確な仕様

artifact-validator.tsのソースコードを確認した結果、実装されている角括弧検出パターンは次のとおりである。

```
/\[#[^\]]{0,50}#\]/g
```

この正規表現が示す検出対象は「`[#`で始まり`#]`で終まる文字列で、内部が50文字以内のもの」のみである。通常の角括弧（配列アクセス・正規表現文字クラス・Markdownリンク等）はこのパターンに一致しないため、検出対象外となる。また、extractNonCodeLines関数によりコードフェンス内の行とインラインコード部分はすでに検出対象から除外されている。

## 実装計画

### FR-A: CLAUDE.mdの角括弧プレースホルダー説明文修正

#### 修正対象箇所

CLAUDE.mdのsubagentTemplate（「## ★重要★ 成果物品質要件（artifact-validator準拠）」セクション内）の「禁止パターン」項目における角括弧説明文を修正する。

現行の記述（誤）:
```
角括弧プレースホルダーは禁止（波括弧を使用）。検出の対象はコードフェンス外のMarkdown本文（散文テキスト・箇条書き等）に記述された角括弧であり、コードフェンス内の行は検出から除外される。したがってコードフェンスを使用すれば配列アクセス記法や正規表現の文字クラス表記を安全に記述できる。ただしMarkdown本文の箇条書きや散文テキストに角括弧を直接記述することは引き続き禁止されている。
```

修正後の記述（正）:
```
角括弧プレースホルダー禁止。禁止されるのは `[#任意テキスト#]` 形式のハッシュ記号付きプレースホルダーのみである。通常の角括弧・配列アクセス記法・正規表現の文字クラス表記はコードフェンス外でも検出対象ではない。コードフェンス内の行はextractNonCodeLinesにより検出対象から除外される。シングルバックティックで囲んだインラインコードもremoveInlineCode関数により除去されるため検出対象外となる。
```

また、CLAUDE.mdの「禁止パターン（完全リスト）」セクション末尾の角括弧説明を同様の内容に更新する。

#### 修正後の期待効果

CLAUDE.mdを読んだsubagentが「ハッシュ記号付き形式のみが禁止」と正確に理解し、通常の角括弧を使った自然な記述が可能になる。これによりセクション内の実質行数が確保しやすくなる。

### FR-B: definitions.tsのFR-3・FR-4ガイダンス修正

#### FR-3（e2e_testテンプレート）の修正

e2e_testテンプレートの末尾近くにある「テスト文書固有の角括弧禁止パターン（FR-3: e2e_test特化ガイダンス）」セクションを修正する。

現行の誤記述部分:
```
- 禁止パターン（コードフェンス外での配列記法）: 散文テキストや箇条書きで複数ファイルを列挙する際に配列記法で記述すること
```

修正後の正確な記述:
```
- 禁止パターン（ハッシュ記号付きプレースホルダー）: `[#変数名#]` や `[#パス#]` のような `[#...#]` 形式の角括弧プレースホルダーのみが検出対象である。通常の角括弧・配列アクセス記法・Markdownリンク記法はコードフェンス外でも使用可能である。
```

推奨パターン1（散文形式）の記述は有用なため維持するが、「禁止」という表現を「代替手段のひとつ」という説明に変更する。

#### FR-4（performance_testテンプレート）の修正

performance_testテンプレートの「計測日時行の一意化と角括弧禁止パターン（FR-4: performance_test横断対策）」セクションの後半部分を修正する。

現行の誤記述部分:
```
パフォーマンスレポートでファイル名やディレクトリ名を列挙する場合もコードフェンス外での配列記法は禁止されている。複数ファイルの列挙は散文形式（「definitions.tsファイルおよびstate-manager.tsファイルを計測対象とした」）またはインラインコードを使用すること。コードフェンス内であれば配列記法を安全に使用できる。
```

修正後の正確な記述:
```
パフォーマンスレポートでファイル名やディレクトリ名を列挙する場合、禁止されるのは `[#変数名#]` 形式のハッシュ記号付きプレースホルダーのみである。通常の角括弧や配列アクセス記法はコードフェンス外でも使用可能であるが、散文形式またはインラインコードを使用することで可読性が向上する。
```

### FR-C: buildSubagentTemplate関数の角括弧説明修正

definitions.tsのbuildSubagentTemplate関数内（行1150〜1162付近）の「角括弧プレースホルダー禁止」セクションを修正する。

現行の誤記述:
- 行1151: `[変数名]、[パス]等の角括弧プレースホルダーは使用禁止です。`
- 行1153〜1154: コードフェンス外の箇条書きに文字クラス表記や配列アクセス記法を直接書くことが禁止対象という記述

修正後の正確な記述:
- 行1151: `` `[#変数名#]`、`[#パス#]` 等のハッシュ記号付き角括弧プレースホルダーは使用禁止です。通常の角括弧はコードフェンス外でも使用可能です。``
- 行1153〜1154: コードフェンス外でも通常の角括弧は使用可能であるが、ハッシュ記号付きプレースホルダー形式は禁止という記述に変更する

また、buildRetryPrompt関数内のエラーパターン「プレースホルダー括弧」に対応するメッセージ（行1307〜1308）も同様に修正する。現行の「コードフェンス外のMarkdown本文に角括弧が使われている箇所を特定し、散文形式の説明または波かっこ記法に変更してください」という過剰な修正指示を「`[#xxx#]`形式のハッシュ記号付きプレースホルダーを検出した箇所を特定し、波かっこ形式または具体的な値に変更してください」という正確な修正指示に変更する。

### FR-D: サマリーセクション実質行数ガイダンスの精度確認と補完

#### performance_testのガイダンス確認

performance_testテンプレートの「サマリーセクションの行数ガイダンス」では既に5項目の記述が求められているが、コロン後にコンテンツがない行は実質行としてカウントされないという注意事項の記述を確認する。現行テンプレートには具体的なNG例が含まれているため、この部分は維持する。

#### e2e_testのガイダンス確認

e2e_testテンプレートの「サマリーセクションの行数ガイダンス」も同様に確認する。FR-16で追加された5項目のOK例が含まれており、実質行数不足を防ぐ記述が存在する。この部分も維持する。

## 受入条件の対応方針

### AC-1への対応

CLAUDE.mdの修正後に「禁止されるのは `[#xxx#]` 形式のプレースホルダーのみ」という内容が明確に記述されていることを確認する。「コードフェンス外の全角括弧禁止」という記述が削除されていることを検証する。

### AC-2への対応

e2e_testとperformance_testのsubagentTemplateを読んだsubagentが、通常の角括弧使用を不必要に回避しないように、FR-3・FR-4の「配列記法禁止」という表現が削除または正確な説明に置き換えられていることを確認する。

### AC-3への対応

definitions.ts修正後にテストスイートを実行し、945件以上の通過を確認する。変更はテンプレート文字列の内容修正のみであり、テストロジックへの影響はないことを確認する。

### AC-4への対応

performance_testとe2e_testのテンプレートにコロン後コンテンツなし行が実質行としてカウントされないという注意事項が明示されていることを確認する。現行テンプレートにはNG例（コロン後コンテンツなしの行）が既に含まれており、FR-Dでの追加は最小限となる見込みである。

### AC-5への対応

CLAUDE.mdとdefinitions.tsの両ファイルで角括弧プレースホルダーの禁止形式として `[#xxx#]` が統一して記述されていることを確認し、両ファイル間の矛盾がないことを検証する。

## MCPサーバー再起動要件

definitions.tsはMCPサーバーのコアファイルに該当するため、修正後には以下の手順でMCPサーバーを再起動する必要がある:

1. `cd workflow-plugin/mcp-server && npm run build` を実行してTypeScriptをトランスパイルする。
2. dist/*.jsファイルの更新日時を確認し、修正が反映されたことを検証する。
3. MCPサーバープロセスを再起動する（Claude Desktopのサーバー再起動機能を使用）。
4. 再起動後に `workflow_status` を実行して現在のフェーズを確認し、同フェーズから作業を再開する。

CLAUDE.mdの変更はドキュメント修正のみであり、MCPサーバーの再起動は不要である。ただし実装フェーズではdefinitions.tsとCLAUDE.mdの両方を修正するため、definitions.tsの修正完了後に一度再起動してからCLAUDE.mdの修正に進む順序が望ましい。
