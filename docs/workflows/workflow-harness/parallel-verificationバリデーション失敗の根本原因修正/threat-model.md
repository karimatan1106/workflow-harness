## サマリー

- **脅威の本質**: CLAUDE.md と definitions.ts に記述されている角括弧プレースホルダーの禁止規則が、artifact-validator.ts の実装と不一致しており、subagent が不正確な制約情報に基づいて成果物を作成する状況が発生している。実装が禁止するのは `[#任意テキスト#]` 形式のみだが、ドキュメントは「コードフェンス外の全角括弧禁止」と過度に記述されている。
- **影響範囲**: parallel_verification の4つのサブフェーズ（manual_test・security_scan・performance_test・e2e_test）のすべてが該当。特に e2e_test と performance_test のサブエージェントテンプレートに FR-3・FR-4 として実装されたガイダンスが過剰な制約を加えている。
- **発生メカニズム**: 要件定義（requirements フェーズ）で特定された通り、CLAUDE.md の誤った説明がテンプレート文字列を通じてサブエージェントプロンプトに伝播し、サブエージェントが不必要な回避策（散文形式への強制、配列記法の避退等）を取らせる悪循環が形成されている。
- **根本原因の3層構造**: 第1層は CLAUDE.md 298 行目の「コードフェンス外の全角括弧禁止」という過度な記述。第2層は definitions.ts の FR-3（e2e_test）と FR-4（performance_test）テンプレートが CLAUDE.md からの誤った説明を転写し増幅している。第3層は artifact-validator.ts の実装が `[#xxx#]` 形式のみを禁止しているにもかかわらず、ドキュメント側が「全角括弧禁止」と解釈させることで、制約が過度に実装される状況である。
- **修正戦略**: CLAUDE.md の角括弧説明を artifact-validator.ts の実装仕様に正確に合わせ、definitions.ts の FR-3・FR-4 テンプレートを修正することで、subagent が正確な制約情報に基づいて自然な成果物を作成できる環境を整備する。修正は「正確な仕様情報の提供」であり、バリデーター実装の変更は含まない。

---

## 脅威シナリオ

### シナリオ1: ドキュメント-実装の仕様乖離による過度な制約

**状況**: artifact-validator.ts の括弧検出パターンは `/\[#[^\]]{0,50}#\]/g` という特定の形式（開き括弧・シャープ・任意テキスト・シャープ・閉じ括弧）のみを禁止しているが、CLAUDE.md 298行目では「コードフェンス外のMarkdown本文（散文テキスト・箇条書き等）に記述された角括弧は禁止」と過度に記述されている。

**実例**: e2e_test のサブエージェントが「ファイル名をコードフェンス外で列挙すること禁止」というガイダンスを受け、自然な文体での記述を回避し、不自然な散文形式に強制される。具体的には、本来なら「code-review.md および security-scan.md を対象とした」と自然に記述できるところを、過度な制約への対応として「当該マークダウンファイルを対象とした」といった迂遠な表現になり、成果物の可読性が低下する。

**因果連鎖**: CLAUDE.md の誤った説明 → subagentTemplate への転写 → subagent への伝播 → 過度な回避策の実装 → 成果物品質の低下

---

### シナリオ2: テンプレートガイダンスの矛盾による subagent の混乱

**状況**: definitions.ts の FR-3（e2e_test テンプレート 954行目付近）と FR-4（performance_test テンプレート 930行目付近）に記述されているガイダンスが、CLAUDE.md と矛盾した内容を含んでいる。特に「複数ファイルの列挙は散文形式またはインラインコードを使用すること」という指示は、実装の禁止パターン（`[#xxx#]` 形式）とは無関係な制約を追加している。

**具体例**:
- FR-3 の e2e_test テンプレート: 「ファイル名・ディレクトリ名を列挙する際は、コードフェンス外での配列記法の使用が禁止されている」と記述
- 実装の実際: `/\[#[^\]]{0,50}#\]/g` 検出パターンは `array[index]` 形式を禁止していない（括弧内が「#xxx#」形式でないため）
- subagent への悪影響: 「配列記法一般が禁止」と誤解し、必要以上に複雑な言い換え表現を使用

**結果**: テスト結果レポートの可読性低下、技術的に正確な表現が避けられる

---

### シナリオ3: フィードバックループによる継続的なバリデーション失敗

**状況**: 過度な制約に基づいて作成された成果物でも artifact-validator がバリデーション成功を返す（禁止パターンが実装どおりに機能するため）。しかし制約が過度であるため、subagent が自然な表現を使用した場合にのみバリデーション失敗が発生する可能性が生じる。このため、subagent は「ドキュメントの指示に従った結果、バリデーション成功」と誤認識し、不自然な表現を継続する悪循環が形成される。

**ループの構造**:
1. CLAUDE.md 298行目: 「コードフェンス外の全角括弧禁止」と記述
2. subagent がこれを信じて「配列記法は禁止」と解釈
3. 配列記法を避けた不自然な成果物を作成
4. バリデーションで失敗しない（実装が `[#xxx#]` 形式のみ検出）
5. subagent が「この指示が正しい」と確信を深める
6. 後続タスクでも同じ誤った理解が伝播

---

### シナリオ4: ドキュメント修正の遅延による技術的負債

**状況**: artifact-validator.ts の実装は正確だが、CLAUDE.md と definitions.ts のドキュメントが誤った説明を含み続けた場合、以下の問題が蓄積される：
- 新規プロジェクト開始時に不正確な制約情報が伝播
- subagent の不自然な表現癖が引き継がれる
- マニュアル確認の負荷が増加（不自然な表現を何度も指摘する必要）
- ドキュメント修正の優先度が低下し、修正期限が延伸

**長期的な影響**: システム全体のドキュメント信頼性低下、複数プロジェクトへの波及

## セキュリティ要件

### 要件1: ドキュメント修正の正確性

**内容**: CLAUDE.md を修正する際に、実装の禁止パターンを正確に反映することを必須とする。単なる「誤った説明を削除」ではなく、「artifact-validator.ts の実装仕様に基づいた正確な説明に置き換える」こと。

**具体的には**:
- 禁止されるのは `[#任意テキスト#]` 形式のプレースホルダーのみであること
- 通常の角括弧（`array[index]` など）はコードフェンス外でも許可されること
- インラインコード（バックティック囲み）は角括弧検出の対象外であること

**検証方法**: 修正後の CLAUDE.md を読んだ新規ユーザーが、正確な禁止パターンを理解できるかを確認する。

---

### 要件2: テンプレート間の一貫性

**内容**: parallel_verification の4つのサブフェーズテンプレート（manual_test・security_scan・performance_test・e2e_test）で、角括弧に関する記述が矛盾しないこと。

**具体的には**:
- FR-3（e2e_test）と FR-4（performance_test）の「配列記法禁止」ガイダンスを削除または修正
- 代わりに「禁止されるのは `[#xxx#]` 形式のプレースホルダーのみ」という統一説明を追加
- manual_test と security_scan のテンプレートも同一の説明になっていることを確認

**検証方法**: 4つのテンプレート内の「角括弧」「プレースホルダー」に関する記述を比較し、矛盾がないか確認する。

---

### 要件3: MCPサーバー再起動の忘却防止

**内容**: definitions.ts を修正した場合、MCPサーバーの再起動が必須であることをドキュメント化し、修正者が再起動なしに進めることを防ぐ。

**具体的には**:
- 修正後に `cd workflow-plugin/mcp-server && npm run build` でトランスパイル
- MCPサーバープロセスを再起動
- `workflow_status` で現在のフェーズを確認してから次フェーズに進む

**技術的根拠**: Node.js のモジュールキャッシュにより、サーバープロセス起動時のバイナリのみが実行され、ディスク上の変更は反映されない。

---

### 要件4: 修正漏れ検出

**内容**: 修正後にテストスイートを実行し、以下の基準を満たすことを確認すること：
- テスト通過数: 945件以上（修正前と同じ）
- リグレッション: なし
- テンプレート内容の一貫性: 角括弧説明が矛盾しないこと

## 修正の最小化原則

本脅威モデルで指摘する修正は、以下の3つのファイルに限定される。影響範囲を最小化することで、既存テストへのリグレッションリスクを低減する。

1. **CLAUDE.md**: 角括弧プレースホルダーの説明文を artifact-validator.ts の実装仕様に合わせて修正
2. **definitions.ts**: FR-3（e2e_test）と FR-4（performance_test）のテンプレート内で、過度な角括弧禁止ガイダンスを削除・修正

3. **修正対象外**: artifact-validator.ts、state-manager.ts、その他のコアモジュール。実装が既に正確であり、ドキュメント側の説明不足が主原因であるため、バリデーター実装の変更は不要である。

修正はドキュメント内容の正確性を向上させるものであり、実装ロジックの変更は含まない。これにより、既存のバリデーション済み成果物への影響は最小化される。

## まとめ

parallel_verification フェーズで繰り返されるバリデーション失敗の根本原因は、CLAUDE.md と definitions.ts のドキュメントが artifact-validator.ts の実装仕様と乖離しており、subagent が不正確な制約情報に基づいて成果物を作成する状況である。

修正戦略は単純であり、ドキュメントを実装に合わせた正確な説明に更新することで、サブエージェントが正確な制約情報に基づいて自然で高品質な成果物を作成できる環境を整備する。

修正後の検証方法としては、修正内容が全てのサブフェーズテンプレート（manual_test・security_scan・performance_test・e2e_test）に一貫して反映されていることを確認する必要がある。

修正実施後の期待される効果は、サブエージェントが過度な制約に基づかず自然な表現で成果物を作成し、parallel_verification の全サブフェーズ報告書の可読性が向上することである。

修正後のテスト実行により、既存の 945/945 テスト通過率が維持されることを確認することで、修正の信頼性を検証できる。
