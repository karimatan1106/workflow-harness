## サマリー

- 目的: parallel_verificationバリデーション失敗の根本原因修正において、CLIインターフェース（MCPサーバーAPI）・エラーメッセージ・APIレスポンス・設定ファイルの設計を定義する。
- このタスクは画面UIを持たないCLI/MCPサーバー内部修正であるため、UI設計の代わりにインターフェース設計（API・エラーメッセージ・ドキュメント構造）を定義する。
- 主要な決定事項: エラーメッセージは `[#xxx#]` 形式を正確に示すこと、APIレスポンスはバリデーション詳細をsubagentが理解できる形で返すこと、CLAUDE.mdの角括弧説明は最小限かつ正確な記述とすること。
- 次フェーズ（design_review）では、このUI設計で定義したメッセージ仕様・API仕様が実装計画（spec.md）と整合しているかを確認すること。
- 評価スコープ: CLAUDE.md・definitions.ts・artifact-validator.tsの3ファイルが主な修正対象であり、state-manager.tsおよびテストスイートは変更対象外とする。

## CLIインターフェース設計

### MCPサーバーAPIの概要

このタスクで改善対象となるAPIは `workflow_next` と `workflow_complete_sub` の2つである。いずれもsubagentがフェーズ完了を通知するために呼び出すものであり、バリデーション結果をレスポンスとして返す。

### workflow_next のインターフェース定義

入力パラメータは現行仕様と変更なし。変更点はレスポンス内のエラーメッセージ文字列のみである。

呼び出し形式（変更なし）:
```
workflow_next({ taskId: "タスクID" })
```

バリデーション成功時レスポンス（変更なし）:
```
{ success: true, phase: "次のフェーズ名", phaseGuide: { ... } }
```

バリデーション成功時には phaseGuide に subagentTemplate・requiredSections・minLines 等が含まれ、次フェーズの作業情報がOrchestratorに提供される。
Orchestratorはこのレスポンスから subagentTemplate を取得し、そのままサブエージェントのプロンプト本体として使用することが設計上の前提である。

バリデーション失敗時レスポンス（エラーメッセージを改善）:
```
{ success: false, error: "バリデーション失敗の詳細メッセージ（改善後）" }
```

### workflow_complete_sub のインターフェース定義

入力パラメータは現行仕様と変更なし。レスポンス内のエラーメッセージ文字列を改善する。

呼び出し形式（変更なし）:
```
workflow_complete_sub({ taskId: "タスクID", subPhase: "サブフェーズ名" })
```

バリデーション失敗時レスポンス（エラーメッセージを改善）:
```
{ success: false, error: "改善されたバリデーション失敗の詳細メッセージ" }
```

## エラーメッセージ設計

### 角括弧プレースホルダーエラーのメッセージ改善

現行のエラーメッセージ（誤解を招く表現）:
```
コードフェンス外のMarkdown本文に角括弧が使われている箇所を特定し、
散文形式の説明または波かっこ記法に変更してください
```

改善後のエラーメッセージ（正確な仕様を反映）:
```
[#xxx#] 形式のハッシュ記号付きプレースホルダーを検出しました。
該当箇所を特定し、波かっこ形式または具体的な値に変更してください。
通常の角括弧・配列アクセス記法・正規表現の文字クラスはこのエラーの対象ではありません。
```

### エラーメッセージが持つべき情報要件

改善後のエラーメッセージは以下の4項目を含むこと:
- エラーの種別を明示する（角括弧プレースホルダー検出であることを示す）
- 検出パターンの正確な形式を示す（`[#変数名#]` 形式の具体例を含む）
- 対象外であることを明示する（通常の角括弧・配列記法は非対象と明記）
- 修正方法を具体的に示す（波かっこ形式または具体的な値への変更を指示）

### buildRetryPromptの改善後メッセージ構造

generateImprovementsFromError関数内の「プレースホルダー括弧」パターンに対応するメッセージを以下の構造に改善する。

改善後の修正指示文字列:
```
`[#xxx#]`形式のハッシュ記号付きプレースホルダーを検出した箇所を特定し、
波かっこ形式（{変数名}）または具体的な値に変更してください。
通常の角括弧・配列アクセス記法（array[0] など）・Markdownリンク記法は
このエラーの対象ではないため、それらを変更する必要はありません。
```

## APIレスポンス設計

### バリデーション結果レスポンスの構造

workflow_next および workflow_complete_sub のレスポンスにおいて、バリデーション失敗時の情報構造を明確化する。

バリデーション失敗レスポンスの情報構造（概念定義）:
- エラー種別フィールド: どのバリデーションルールに違反したかを示す文字列
- エラー詳細フィールド: 違反箇所と修正方法を説明するメッセージ文字列
- 検出内容フィールド: 実際に検出されたパターンや行番号の情報（利用可能な場合）
- 修正方針フィールド: Orchestratorがsubagentをリトライする際に伝えるべき具体的指示

### 角括弧プレースホルダーエラーのレスポンス設計

`bracket_placeholder` エラー種別に対応するレスポンスの設計方針:
- エラー種別: `"bracket_placeholder_detected"` として識別可能にする
- エラー詳細: 検出された `[#xxx#]` 形式のプレースホルダーを具体的に示す
- 検出除外の案内: 通常の角括弧が検出対象外であることをレスポンスに含める
- リトライ指示: 修正が必要なのはハッシュ記号付き形式のみであることを伝える

### APIレスポンスで使用するメッセージテンプレート変数

buildSubagentTemplate関数が生成するテンプレート内の角括弧禁止説明セクションの変数設計:
- 禁止形式の例示変数: `[#変数名#]` および `[#パス#]` の2種類の例を提示する
- 検出除外リスト変数: 「通常の角括弧・配列アクセス記法・Markdownリンク記法」の列挙
- コードフェンス除外説明変数: extractNonCodeLines関数による除外の説明文

### phaseGuideのerrorPatterns設計

workflow_nextレスポンスに含まれるphaseGuideの errorPatterns フィールドで角括弧プレースホルダーエラーの説明を改善する。改善後の設計:
- エラーパターン名: `"bracket_placeholder"` のまま維持（互換性のため）
- 説明文字列: 現行の過剰な制約説明から、`[#xxx#]` 形式特定の説明に変更する
- 改善指示文字列: 「コードフェンス外の全角括弧を回避してください」から「ハッシュ記号付き形式のみを修正してください」に変更する

## 設定ファイル設計

### CLAUDE.mdの角括弧説明セクション構造設計

CLAUDE.mdは subagentTemplate をサブエージェントに伝達するためのガイドドキュメントであり、角括弧プレースホルダーの説明が正確であることが重要である。

現行の問題のある構造（概要）:
```
禁止パターン:
- 角括弧プレースホルダーは禁止（波括弧を使用）
- 検出の対象はコードフェンス外のMarkdown本文の角括弧
- コードフェンス内は安全
- ただしMarkdown本文の角括弧は禁止されている
```

改善後の正確な構造設計:
```
禁止パターン:
- [#任意テキスト#] 形式のハッシュ記号付きプレースホルダーのみが禁止
- 通常の角括弧はコードフェンス外でも使用可能
- extractNonCodeLinesによりコードフェンス内は除外
- removeInlineCode関数によりインラインコードも除外
```

### 「禁止パターン（完全リスト）」セクションの末尾追記設計

CLAUDE.mdの「禁止パターン（完全リスト）」セクション末尾に、角括弧プレースホルダーの正確な仕様を追記する。追記内容の構造:
- 第1項: 検出対象パターンの正規表現を示す（コードフェンス内に記載）
- 第2項: 検出対象の具体例として `[#変数名#]` と `[#パス#]` を示す
- 第3項: 検出対象外の列挙（通常の角括弧・配列アクセス・Markdownリンク・正規表現文字クラス）
- 第4項: extractNonCodeLinesとremoveInlineCodeによる除外範囲の説明

### definitions.tsのテンプレート文字列セクション構造設計

buildSubagentTemplate関数内の角括弧プレースホルダー禁止セクション（行1150〜1162付近）の構造を以下のように再設計する。

改善後のセクション構造:
```
## 角括弧プレースホルダー禁止
- 禁止されるのは [#変数名#] 形式（ハッシュ記号付き）のプレースホルダーのみ
- 通常の角括弧・配列アクセス記法はコードフェンス外でも使用可能
- コードフェンス内の全行は検出対象外
- インラインコード内も検出対象外
```

### definitions.tsのFR-3・FR-4テンプレートセクション構造設計

e2e_testテンプレート（FR-3）の角括弧禁止パターンセクション:
```
テスト文書固有の角括弧禁止パターン（FR-3: e2e_test特化ガイダンス）:
- 禁止対象: [#変数名#] や [#パス#] のような [#...#] 形式のプレースホルダー
- 使用可能: 通常の角括弧・配列アクセス記法・Markdownリンク記法
- 推奨: 散文形式またはインラインコードを使うと可読性が向上する
```

performance_testテンプレート（FR-4）の計測日時行と角括弧ガイダンスセクション:
```
計測日時行の一意化と角括弧ガイダンス（FR-4: performance_test横断対策）:
- 禁止対象: [#変数名#] 形式のハッシュ記号付きプレースホルダーのみ
- 通常の角括弧・配列アクセス・ファイル名列挙はコードフェンス外でも使用可能
- 散文形式またはインラインコードを使うと可読性が向上する（推奨）
```
