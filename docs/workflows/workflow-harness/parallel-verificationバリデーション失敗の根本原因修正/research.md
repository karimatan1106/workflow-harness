## サマリー

- 調査対象: parallel_verificationの4サブフェーズ（manual_test・security_scan・performance_test・e2e_test）で発生するバリデーション失敗の根本原因を特定するため、definitions.tsのsubagentTemplateとartifact-validator.tsの検出ロジックを精査した。
- 主要な発見1（角括弧禁止パターンの誤記載）: CLAUDE.mdの成果物品質要件セクションに「コードフェンス外の角括弧が禁止」と記載されているが、artifact-validator.tsの実際の検出パターンは`\[#[^\]]{0,50}#\]`（`[#xxx#]`形式のみ）であり、通常の角括弧は現在の実装では検出されない。subagentTemplateに含まれるCLAUDE.md転写テキストがこの誤情報をsubagentに与えており、不必要な回避策を増やしている可能性がある。
- 主要な発見2（サマリーセクションの行数カウント仕様）: コロン後にコンテンツがない行（例: 「- 目的:」）は実質行としてカウントされないため、subagentが箇条書きのラベルだけを書いてコンテンツを省略すると即座に行数不足となる。この仕様はFR-11・FR-12ガイダンスで対策済みだが、リトライが発生した場合の精度に差がある。
- 主要な発見3（subagentTemplateの品質ガイダンス記述量）: 各サブフェーズのsubagentTemplateには既にFR-1〜FR-12の詳細ガイダンスが埋め込まれており、内容は十分に充実している。初回通過率が低い根本原因は、テンプレートの量的不足ではなく、「CLAUDE.md転写部分の誤情報」と「コンテンツなしラベル行の誤認識」にある。
- 次フェーズで必要な情報: requirements/planning フェーズでは3点の問題（角括弧パターン説明の修正・CLAUDE.md整合性の確保・誤案内によるリトライ発生メカニズムの断絶）に対して具体的な修正要件を定義すること。

## 調査結果

### 問題1: 角括弧プレースホルダー検出パターンとCLAUDE.md記述の乖離

artifact-validator.tsのソースコード（行368）を確認すると、角括弧プレースホルダーの検出パターンは以下のとおりである。

```typescript
const bracketPlaceholderPattern = /\[#[^\]]{0,50}#\]/g;
```

このパターンは `[#任意のテキスト#]` という形式にのみ一致し、通常の角括弧 `[xxx]` や正規表現の文字クラス表記 `\s+`・`[^}]`・`[a-z]` などはマッチしない。コードフェンス外に記述された一般的な角括弧は現在の実装では検出対象とならない。

一方、CLAUDE.mdの「成果物品質要件」セクションには以下の記述が含まれている。

```
角括弧プレースホルダーは禁止（波括弧を使用）。検出の対象はコードフェンス外の
Markdown本文（散文テキスト・箇条書き等）に記述された角括弧であり、コードフェンス
内の行は検出から除外される。
```

この記述は実際のコードと乖離しており、subagentに「通常の角括弧も禁止」という誤解を与える原因となっている。subagentTemplateのFR-3（e2e_test）・FR-4（performance_test）各ガイダンスにも「コードフェンス外での配列記法は禁止」という記述が含まれており、これも同じ誤情報から派生している。

問題タスク「角括弧プレースホルダーパターンを[#xxx#]形式に修正」（2026-02-25）で検出パターンが絞り込まれたが、その際にCLAUDE.mdの説明文が更新されなかったことが乖離の原因である。

### 問題2: サマリーセクション実質行数不足の再発メカニズム

artifact-validator.tsは`isSubstantialLine()`関数でコロン後のコンテンツ有無を判定する。以下のような行はコロン後にコンテンツがないため実質行として計上されない。

- 「- 目的:」（コロン後が空白のみ）
- 「- 使用したスキャンツール:」（コロン後が空白のみ）
- 「- 検出件数:」（コロン後が空白のみ）

subagentTemplateには既にこの注意事項を示すガイダンスが含まれているが、問題は「なぜリトライで同じ失敗が繰り返されるか」という点にある。Orchestratorがリトライプロンプトに誤情報を含むCLAUDE.mdの説明を引用し続けると、subagentが問題の本質を誤認したまま修正を試みるフィードバックループが形成される。

各サブフェーズの必要行数はminLines 20行であり、必須セクションが3つ存在する場合（security_scan: 脆弱性スキャン結果・検出された問題・サマリー）は1セクションあたり平均6.7行の実質行が必要となる計算だが、コロン後コンテンツなしのラベル行が多くなると実質行数が大幅に下回る。

FR-12（security_scan）のガイダンスには「3セクション×5行+追加5行で合計20行」という計算ロジックが明記されており、この対策は正しい方向性である。

### 問題3: subagentTemplateとCLAUDE.md転写内容の整合性問題

Orchestratorは `workflow_next` のレスポンスから `subagentTemplate` を取得してsubagentに渡す。このとき、CLAUDE.mdの一部（成果物品質要件セクション）が `subagentTemplate` 内にテキストとして転写されている。

転写されている記述例：
```
角括弧プレースホルダーは禁止（波括弧を使用）。検出の対象はコードフェンス外の
Markdownの本文（散文テキスト・箇条書き等）に記述された角括弧である。
```

この記述が実際のバリデーター実装と乖離していると、subagentは誤った制約下で成果物を作成することになる。特にe2e_testサブフェーズでは、ファイル名の列挙や辞書キー参照で通常の角括弧を使いたい場面が多く、誤情報によって不必要にコードフェンス回避を強制されている可能性がある。

### artifact-validator.tsの検出ロジック確認

`workflow-plugin/mcp-server/src/validation/artifact-validator.ts`のブラケット検出ロジックを確認した結果：
- 行368: `const bracketPlaceholderPattern = /\[#[^\]]{0,50}#\]/g;`
- 行155〜184: `extractNonCodeLines()`関数でコードフェンス外の行を抽出してから検索を実施
- 行136〜141: `removeInlineCode()`関数でインラインコード（バックティック囲み）を除去してから検索

つまり、現在の実装では以下の3種類の記述が許可されている：
- コードフェンス内の全テキスト（配列記法・正規表現・文字クラスを含む）
- インラインコード内のテキスト（バックティック囲み）
- コードフェンス外のMarkdown本文で、`[#xxx#]`パターンに合致しない通常の角括弧

CLAUDE.mdの「角括弧プレースホルダー禁止」の説明はこれらの実際の動作を正確に反映していない状態にあり、subagentTemplateに転写された制約説明も同様の問題を抱えている。

## 修正対象の特定

### 修正が必要な箇所（優先度順）

優先度1として修正が必要な箇所はCLAUDE.mdの角括弧プレースホルダー説明文である。CLAUDE.mdの「成果物品質要件」セクションに記載されている角括弧禁止の説明が実際のartifact-validator.ts実装と乖離しており、subagentに誤情報を与え続けている。修正方針は「通常の角括弧は許可されており、禁止されるのは`[#xxx#]`形式のみ」という正確な説明に更新することである。

優先度2として修正が必要な箇所はdefinitions.tsのsubagentTemplateに含まれる角括弧禁止ガイダンスである。FR-3（e2e_test）とFR-4（performance_test）に含まれる「コードフェンス外の角括弧禁止」という記述が、現在の検出パターン（`[#xxx#]`形式のみ）と乖離している。散文テキスト内での通常の配列アクセス記法（`array[0]`）や正規表現文字クラスをコードフェンス外で記述することは、現在のバリデーターでは問題とならないため、不必要に制約する記述を削除または修正する必要がある。

優先度3として確認が必要な箇所はCLAUDE.mdの「成果物品質要件（artifact-validator準拠）」セクションとartifact-validator.tsの実装との整合性全体である。FR系のテンプレート変更が定期的に行われているため、CLAUDE.mdの説明テキストが古いバリデーター仕様に基づいている可能性がある。

### 修正不要と判断した箇所

subagentTemplateのサマリー行数ガイダンス（FR-11・FR-12）は正確な情報を提供しており、修正不要である。これらのガイダンスがあっても失敗が繰り返される場合は、リトライプロンプトの構成に問題がある可能性が高い。

各サブフェーズでsonnetモデルを採用する判断（行893〜894のコメント参照）は適切であり、モデル変更による修正は不要と判断した。
