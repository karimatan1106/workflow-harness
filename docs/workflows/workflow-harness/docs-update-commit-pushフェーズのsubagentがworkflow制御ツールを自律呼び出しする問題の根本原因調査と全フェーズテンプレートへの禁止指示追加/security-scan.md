## サマリー

- 目的: FR-19として実装された全フェーズへのワークフロー制御ツール禁止指示追加に対して、セキュリティスキャンを実施し、依存パッケージの脆弱性と実装内容の安全性を評価する
- 評価スコープ: `workflow-plugin/mcp-server/src/phases/definitions.ts`（FR-19変更対象）、依存パッケージ（pnpm-lock.yaml記録の全パッケージ）
- 主要な決定事項: 依存パッケージに高深刻度の脆弱性が2件（`@modelcontextprotocol/sdk@1.25.2` のクロスクライアントデータ漏洩、`minimatch` の ReDoS）検出された。FR-19の実装内容自体は静的文字列テンプレートの追記であり、新たな攻撃面を生み出す変更ではないと評価する
- 検証状況: `pnpm audit` による自動スキャン実施済み（10件の脆弱性を検出）、FR-19追記内容の静的コードレビューを実施済み
- 次フェーズで必要な情報: `@modelcontextprotocol/sdk` を `^1.26.0` 以上に更新するアップグレード対応が推奨される

---

## 脆弱性スキャン結果

pnpm auditを実行した結果、合計10件の脆弱性が検出された。
内訳は、深刻度「高（high）」が2件、「中（moderate）」が6件、「低（low）」が2件であった。

### 高深刻度（2件）

**第1件目 — `@modelcontextprotocol/sdk@1.25.2` クロスクライアントデータ漏洩**
現在インストールされているバージョンは `1.25.2` であり、脆弱なバージョン範囲 `>=1.10.0 <=1.25.3` に該当する。
修正バージョンは `>=1.26.0` であり、アップグレードにより解消できる。
脆弱性の内容はサーバー・トランスポートインスタンスの再利用によるクロスクライアントデータ漏洩であり、
MCP サーバーが複数クライアントのセッションを誤って混在させる可能性がある。
アドバイザリ参照: GHSA-345p-7cg4-v4c7

**第2件目 — `minimatch` ReDoS（正規表現サービス拒否）**
`@vitest/coverage-v8 > test-exclude > minimatch` の依存パス経由で使用されるパッケージが脆弱バージョンに該当する。
修正バージョンは `>=10.2.1` であり、vitest/coverage-v8 の更新を通じて解消できる。
ReDoS は特定のパターン入力で正規表現エンジンが指数的な時間を消費する問題であり、テスト実行時に影響を受ける可能性がある。
アドバイザリ参照: GHSA-3ppc-4f35-3m26

### 中深刻度（6件）

`esbuild@0.24.2` の開発サーバーへのクロスオリジンリクエスト許可（修正: `>=0.25.0`）が検出された。
この問題は `vitest > vite > esbuild` の経路で影響し、テスト実行の開発環境にのみ関連する。
他5件は `@modelcontextprotocol/sdk` 経由の推移的依存関係（`qs`、`body-parser` 等）に起因する既知の問題であった。
いずれも MCP サーバーの本番利用環境（Claude Desktop 経由のローカルプロセス）では直接の攻撃経路が存在しないと評価する。

### 低深刻度（2件）

`hono < 4.11.10` のタイミング比較強化不足（basicAuth/bearerAuth）が `@modelcontextprotocol/sdk > @hono/node-server > hono` 経路で検出された。
本プロジェクトでは hono を認証目的で使用していないため、実際のリスクは最小限と評価する。

## 検出された問題

### 問題1: `@modelcontextprotocol/sdk` バージョンが脆弱範囲内

`package.json` の specifier が `^1.0.0` であり、lockfile のバージョンが `1.25.2` であるため、
GHSA-345p-7cg4-v4c7 の修正バージョン `>=1.26.0` が未適用である。
クロスクライアントデータ漏洩はローカル開発環境ではリスクが限定的であるが、将来的に MCP サーバーをリモートで提供する場合には深刻な問題になり得る。
推奨対応: `pnpm update @modelcontextprotocol/sdk --latest` を実行してバージョンを `1.26.0` 以上に更新する。

### 問題2: `minimatch` の ReDoS 脆弱性（テスト依存経路）

テスト実行時にのみ影響するが、悪意ある glob パターンが渡された場合に CPU 資源を過剰消費する。
テストファイルのパス解決や除外パターン処理で使用されるため、CI/CD 環境でのリスクが考えられる。
推奨対応: `vitest` および `@vitest/coverage-v8` を最新バージョンに更新することで解消できる。

### 問題3: `esbuild` の開発サーバー CORS 問題（軽微）

`esbuild@0.21.5` が明示的に devDependencies に固定されており、脆弱バージョン `<=0.24.2` の範囲に含まれる。
この問題は `esbuild --serve` コマンドを使用する場合のみ関連し、通常のビルドや vitest 実行では影響しない。
推奨対応: `esbuild` の固定バージョン `0.21.5` を削除し、vitest が要求する最新バージョンを使用させる。

## FR-19 実装内容のセキュリティレビュー

### 評価対象

FR-19 は `definitions.ts` の各フェーズ `subagentTemplate` 文字列に、ワークフロー制御ツールの呼び出し禁止を指示するテキストを追記した変更である。
変更対象のフェーズは `docs_update`、`commit`、`push` の3フェーズであり、既存フェーズ（`research`、`requirements` など）に加えて合計25フェーズ・サブフェーズのテンプレートに禁止指示が存在することを `grep -c` で確認した。

### 安全性評価

追記内容は JavaScript テンプレートリテラル内の静的文字列であり、実行時に評価される動的コードや外部入力を含まない。
テキストは自然言語の指示文であり、MCP ツールの識別子（`workflow_next` 等）を列挙することで AI モデルへの禁止範囲を明確化している。
この種の静的テキスト追記は、コードインジェクションやプロトタイプ汚染等の攻撃面を生み出さない。
テンプレート文字列への外部ユーザー入力の混入は `${userIntent}` 等の既存プレースホルダーに限定されており、FR-19 の追記部分には新たなプレースホルダーは導入されていない。

### SR-1（禁止指示テキスト安全性）適合評価

脅威モデルの SR-1 要件「禁止指示テキストが外部入力や危険なコードを含まないこと」への適合を評価した。
FR-19 で追記されたすべてのテキストは事前定義の固定文字列であり、外部からの制御は受け付けない。
禁止対象の MCP ツール名一覧（`workflow_next`、`workflow_approve`、`workflow_complete_sub`、`workflow_start`、`workflow_reset`）は設計仕様に従い正確に列挙されている。
以上から、FR-19 の実装内容は SR-1 要件を満たすと判断する。

## 総合評価

FR-19 の実装内容に対するセキュリティ観点からの評価を以下の通りまとめる。

### 実装変更の安全性: 問題なし

テンプレート文字列への静的テキスト追記のみであり、攻撃面の増加はない。
動的コード実行、外部入力参照、プロトタイプ汚染等の脆弱性パターンに該当しない。
既存の `${userIntent}` 等の入力点は変更前から存在しており、FR-19 によって新たに追加されたものではない。

### 依存パッケージの脆弱性: アップグレード推奨

`@modelcontextprotocol/sdk@1.25.2` の高深刻度脆弱性については、`^1.26.0` への更新で解消可能であるため、次スプリントでの対応を推奨する。
テスト専用依存の `minimatch` ReDoS、開発用 `esbuild` の CORS 問題については、本番環境への直接リスクは低く、vitest/esbuild のバージョン更新により対処できる。
これら3件の問題はいずれも FR-19 の変更とは独立した既存の問題であり、今回の実装タスク起因ではない。

### 次フェーズへの引き継ぎ事項

`@modelcontextprotocol/sdk` を `>=1.26.0` にアップグレードするタスクを別途作成することを推奨する。
アップグレード後は MCP サーバーを再起動し、全フェーズのワークフロー遷移が正常に機能することを確認する必要がある。
