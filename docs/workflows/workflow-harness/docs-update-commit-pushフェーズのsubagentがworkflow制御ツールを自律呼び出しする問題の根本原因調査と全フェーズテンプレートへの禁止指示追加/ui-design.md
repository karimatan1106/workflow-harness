# ui-design.md: 実装者インターフェース設計

## サマリー

- 目的: definitions.tsの21フェーズのsubagentTemplateに「★ワークフロー制御ツール禁止★」セクションを追加するための実装者向けインターフェース設計を定義する
- 主要な決定事項: このプロジェクトはMCPサーバーであり従来のUIは存在しない。実装者インターフェースとして、definitions.tsの各フェーズのsubagentTemplate文字列末尾に禁止指示テキストを追記する位置と形式を規定する
- 追加対象は4種別（直線フェーズ・並列サブフェーズ・承認フェーズ・git操作フェーズ）に分類され、種別ごとに異なる禁止指示テキストが割り当てられる
- 次フェーズで必要な情報: 各フェーズのsubagentTemplate末尾の識別方法、TypeScript文字列リテラルの終端文字の特定ルール、npm run buildによるビルド成功確認手順
- 注意事項: TypeScript文字列リテラルのエスケープ規則、MCPサーバーの再起動義務、ESM環境でのモジュールキャッシュについて設定ファイル設計セクションで詳述する

---

## CLIインターフェース設計

このプロジェクトのMCPサーバーはNode.jsプロセスとして動作しており、従来の意味でのCLIインターフェースは存在しない。このセクションでは「実装者インターフェース設計」として、definitions.tsファイルをどのように編集するかを規定する。

### 追加対象ファイルの特定

変更対象ファイルは1つのみである。絶対パスは `C:\ツール\Workflow\workflow-plugin\mcp-server\src\phases\definitions.ts` であり、現在の行数は1595行である。subagentTemplateプロパティはシングルクォートまたはバッククォートで区切られた文字列リテラルとして定義されている。

### 追加位置の特定ルール

各フェーズのsubagentTemplateプロパティの文字列末尾を特定する手順は以下の通りである。対象フェーズ名でGrepを実行してフェーズ定義のブロック開始行を見つける。次にsubagentTemplateプロパティの文字列リテラルを読み取り、終端のシングルクォートまたはバッククォートの直前の文字位置を特定する。禁止指示テキストをその位置に挿入する。既存のテキストを削除せず追記のみを行うことが鉄則である。

### 追加するテキストの形式

禁止指示テキストは改行シーケンスで始まる。具体的には `\n\n## ★ワークフロー制御ツール禁止★\n\n` というパターンで始まり、禁止ツールのリストと終了指示が続く。文字列末尾のクォート終端文字の直前に配置することで、成果物として出力されたMarkdownドキュメントの末尾セクションとして表示される。

### フェーズ種別と追加位置の対応

直線フェーズ（research・requirements・implementation・refactoring・ci_verification・deploy）はそれぞれ独立したトップレベルのフェーズ定義ブロックに存在する。各ブロックのsubagentTemplateプロパティを見つけ、その文字列末尾に直線フェーズ用禁止指示を追加する。並列サブフェーズ（threat_modeling・planning・state_machine・flowchart・ui_design・build_check・code_review・manual_test・security_scan・performance_test・e2e_test）はparallelフェーズのsubPhasesオブジェクト内のネストしたオブジェクトに定義されている。承認フェーズ（design_review・test_design）は独立したフェーズ定義ブロックに存在する。git操作フェーズ（commit・push）は独立したフェーズ定義ブロックに存在し、現時点では禁止指示が存在しないことを確認済みである。

## エラーメッセージ設計

実装作業中または実装後に発生し得るエラーと対処フローを記述する。

### TypeScriptビルドエラーが発生した場合の対処

文字列リテラル内の特殊文字（シングルクォートやバックスラッシュ）を適切にエスケープしなかった場合にビルドエラーが発生する。エラーメッセージはTypeScriptコンパイラが出力する構文エラーとして表示される。対処方法は、問題の発生した文字列リテラルを読み取り、エスケープが必要な文字を修正してから再ビルドする。ビルドコマンドは `npm run build` であり、実行ディレクトリは `workflow-plugin/mcp-server/` である。

### subagentTemplateが文字列として認識されない場合のエラー

TypeScriptの型定義でsubagentTemplateプロパティが存在しないフェーズに追記しようとした場合、コンパイルエラーが発生する。エラーメッセージには「Property 'subagentTemplate' does not exist on type」のような内容が含まれる。対処方法は、対象フェーズの型定義を確認し、subagentTemplateプロパティが定義されているかどうかを検証する。存在しない場合はフェーズ定義にsubagentTemplateプロパティを追加する。

### バリデーション失敗時のエラーパターン

禁止指示テキスト自体にバリデーションで検出される語句が含まれている場合、subagentがそのテンプレートを使用して生成した成果物がバリデーションを通過しない可能性がある。現在の禁止指示テキストにはバリデーター検出対象の語句（禁止語12語・角括弧プレースホルダー等）は含まれておらず、この問題は発生しない設計になっている。ただし将来的に禁止語リストが変更された場合は、禁止指示テキスト自体の再検証が必要になる点に注意すること。

### テスト失敗時のエラーメッセージ対処

実装後のテストスイートで既存テストが失敗した場合、テストフレームワーク（vitest）が失敗したテストケース名と期待値・実際値を出力する。subagentTemplateの変更はプロパティの文字列追記のみのため、テスト失敗が発生した場合は追記位置または追記内容に問題がある可能性が高い。テスト名で対象のフェーズを特定し、definitions.tsの対応する箇所を再確認する。

## APIレスポンス設計

MCPサーバーのworkflow_nextが返すレスポンスにおける、本タスクの実装後の変化を記述する。

### subagentTemplateを含むレスポンス形式

workflow_nextはフェーズ遷移時にphaseGuideオブジェクトを返す。このオブジェクトのsubagentTemplateプロパティには、今回の実装によって禁止指示セクションが末尾に追加されたテンプレートが格納される。OrchestratorはこのsubagentTemplateをそのままsubagentに渡すことで、subagentが「ワークフロー制御ツール禁止」の指示を自動的に受け取る設計になっている。

### workflow_statusのレスポンス形式

MEMORY.mdに記載の通り、workflow_statusはslimSubPhaseGuide設計によりsubagentTemplateを含まない。subagentTemplateが必要な場合はworkflow_nextのレスポンスまたはworkflow_get_subphase_templateMCPツールから取得する必要がある。本タスクの実装によってworkflow_get_subphase_templateが返すテンプレートにも禁止指示が追加される点を確認すること。

### エラー時のレスポンス形式

definitions.tsのビルドが失敗した場合、MCPサーバーが起動できずにすべてのMCPツール呼び出しがエラーを返す。MCPツールのレスポンスとして「MCPサーバーに接続できない」または「ツールが見つからない」旨のエラーが返ってくる場合は、まずMCPサーバーのビルドとプロセス起動状態を確認すること。

### 実装後の変化まとめ

本タスクの実装完了後、workflow_nextが返すphaseGuideのsubagentTemplateは全25フェーズで禁止指示セクションを含む状態になる。
Orchestratorがsubagentを起動する際にsubagentTemplateをそのままプロンプトに使用することで、追加の手動対応なしに禁止指示が全フェーズのsubagentへ自動的に伝達される。

## 設定ファイル設計

実装に影響する設定ファイルの内容を記述する。

### MCP server package.jsonのビルド設定

ビルド対象ファイルのパスは `workflow-plugin/mcp-server/package.json` に定義されている。TypeScriptのトランスパイルには `npm run build` コマンドを使用する。トランスパイルの出力先は `dist/` ディレクトリであり、各TypeScriptファイルに対応するJavaScriptファイルが生成される。definitions.tsを変更した場合は `workflow-plugin/mcp-server/dist/phases/definitions.js` が更新されることを日付・サイズで確認する。

### TypeScriptの文字列リテラルの扱いに関する注意事項

definitions.tsで使用されているsubagentTemplateの文字列リテラルはシングルクォート区切りが主体である。シングルクォート文字列内でバックスラッシュnは改行シーケンスとして解釈される。禁止指示テキストを追記する際は、既存のエスケープシーケンスのパターンに合わせて記述する必要がある。具体的には既存テキストが `'...既存内容...'` の形式であれば、追記後は `'...既存内容...\n\n## ★ワークフロー制御ツール禁止★\n\n...'` の形式になる。

### ESM環境でのモジュール読み込み設定

package.jsonに `"type": "module"` が設定されているため、definitions.tsはESMモジュールとして扱われる。createRequireを使用してCommonJSモジュールを読み込む部分はすでに実装済みであり、本タスクの変更はこの設定に影響を与えない。Node.jsのモジュールキャッシュにより、ビルド後にMCPサーバープロセスを再起動しないと変更が反映されないため、実装・ビルド完了後は必ずMCPサーバーの再起動を実施すること。

### テストスイートの確認基準

実装後のテスト実行コマンドは `cd workflow-plugin/mcp-server && npm test` である。現在のテスト件数は945件であり、実装後も945件以上のテストが全件パスすることが受入条件となっている。テスト数が減少した場合や失敗が増えた場合は、追記したテンプレートテキストが既存のテストロジックと競合していないかを確認する。特にsubagentTemplateのパターンマッチングを行うテストが存在する場合は、追記によってそのマッチングが変化していないかを重点的に確認する。
