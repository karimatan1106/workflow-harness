# UI設計フェーズ成果物

## サマリー

本タスク「FR修正プロセス第4ラウンド問題の根本原因解決」は、`definitions.ts`のperformance_testサブフェーズ向けsubagentTemplateへのガイダンス追記（FR-R4A）と`bash-whitelist.js`のverificationPhases配列への要素追加（FR-R4B）という2点のみを修正対象とする内部変更タスクである。
修正対象となる2ファイルは、いずれもシステム内部でのプロンプト文字列生成およびBashコマンド許可判定に使用されるものであり、ユーザーが操作するUI・CLIコマンド体系・APIエンドポイント・レスポンス形式・設定ファイル書式のいずれにも影響を与えない。
FR-R4Aはsubagentに渡すプロンプトテキストの末尾付近に必須セクション向けガイダンスを追記するテキスト変更であり、CLIコマンド、APIレスポンス、設定ファイルのスキーマ変更は一切発生しない設計方針で確定している。
FR-R4BはJavaScriptの配列リテラルに文字列1要素を追加する変更であり、ワークフローの内部ホワイトリスト判定ロジックを修正するのみで、ユーザーへの通知文やエラーメッセージのフォーマットも変更しない設計方針を採用している。
以上の理由から、本タスクにおいてUI設計として新たに設計・変更すべきインターフェース要素は存在しない。

---

## CLIインターフェース設計

本タスクの修正対象はMCPサーバーの内部プロンプトテンプレートとBashホワイトリスト判定ロジックに限定されており、ユーザーが入力するCLIコマンド体系は変更対象に含まれない。
ユーザーが利用するCLIコマンド（`/workflow start`、`/workflow next`、`/workflow status`、`/workflow approve`等）のコマンド名・引数・オプションは現行のまま維持され、新しいコマンドの追加も廃止もない。
FR-R4Bは`parallel_verification`フェーズ中にtestingカテゴリコマンド（npm test、npx vitestなど）が誤ってブロックされていた問題を修正するものであるが、この修正によってユーザーが新たなコマンドを覚える必要はない。
FR-R4A・FR-R4Bの修正はシステムの内部動作を改善するものであり、CLIの出力フォーマット（進捗報告テンプレート・フェーズ完了報告テンプレート）も変更しない。
ユーザーは従来通りの手順でワークフローを実行でき、修正前後でCLI操作体験が変わることはないため、CLIインターフェースに関して新規設計事項は存在しない。

---

## エラーメッセージ設計

FR-R4Aはperformance_testサブフェーズのsubagentTemplateにガイダンスを追記するものであり、artifact-validatorが出力するエラーメッセージのフォーマット・エラーコード体系・表示テキストのいずれも変更しない。
FR-R4Bはbash-whitelist.jsのverificationPhases配列に`'parallel_verification'`を追加するものであり、ホワイトリスト違反時にphase-edit-guardが返すブロックメッセージの文面・構造は修正対象外である。
Bash実行がブロックされた際にユーザーへ表示されるエラーメッセージは現行のまま維持され、修正後は`parallel_verification`フェーズ中でtestingカテゴリコマンドがブロックされる頻度自体が減少することになる。
バリデーション失敗時のエラーメッセージ（セクション密度不足・実質行数不足・禁止語検出等）も変更しない設計方針であり、Orchestratorがリトライ時に解釈するエラーメッセージ形式も維持される。
以上から、エラーメッセージに関して新規設計・変更すべき事項は本タスクのスコープ内に存在しない。

---

## APIレスポンス設計

本タスクが修正対象とするのはMCPサーバーの`definitions.ts`内プロンプト文字列と`bash-whitelist.js`内の配列定義のみであり、MCPサーバーが外部に公開するAPIエンドポイント・レスポンスのJSONスキーマ・HTTPステータスコードは一切変更しない。
workflow_start・workflow_next・workflow_status・workflow_complete_sub・workflow_approve等のMCPツール関数が返すレスポンス構造は現行のまま維持される。
FR-R4Aの修正はsubagentに渡される文字列を拡張するものであり、MCPツールがOrchestratorへ返す成功・失敗レスポンスのフィールド構成を変更しない設計方針である。
FR-R4Bの修正は内部的なホワイトリスト配列の変更であり、MCPツールの呼び出しシグネチャ（入力パラメータ名・型・必須/省略可能の区分）も変更しない設計方針を採用している。
これらの理由から、APIレスポンス設計として新規に策定・変更すべき事項は本タスクには存在しない。

---

## 設定ファイル設計

本タスクにおいて修正するファイルは`definitions.ts`（TypeScriptソースコード）と`bash-whitelist.js`（JavaScriptフックファイル）の2ファイルのみであり、ユーザーが参照・編集する設定ファイルのスキーマや書式は変更対象に含まれない。
`CLAUDE.md`・`.claude/settings.json`・`workflow-state.json`・`.mcp.json`等のプロジェクト設定ファイルは読み取り参照のみを行い、FR-R4A・FR-R4Bの実装においてこれらのファイルへの書き込みは行わない。
環境変数（DOCS_BASE、DOCS_DIR、STATE_DIR、VALIDATE_DESIGN_STRICT等）の定義や解釈方法も変更せず、ユーザーが既存の設定ファイルを修正する必要もない。
settings.jsonのフック定義（phase-edit-guard、enforce-workflow等）に登録されているコマンドパスや引数も変更せず、フック設定ファイル自体の書き換えも本タスクのスコープ外として確定している。
以上から、設定ファイルに関して新規設計・変更すべきインターフェース要素は本タスクには存在しない。

## 変更なしの根拠整理

FR-R4Aはsubagentに渡すプロンプト文字列のみを修正する変更であるため、UIおよびインターフェースレイヤーへの影響は論理的に発生しない。
FR-R4BはJavaScript配列への要素追加であり、外部APIのコントラクト・CLIのコマンド仕様・エラーメッセージのフォーマット・設定ファイルのスキーマのいずれにも影響しない。
両修正はシステムの内部品質を改善する性質のバグ修正であり、外部から観測可能なインターフェースの変化を意図的に発生させないよう後方互換性を維持して設計されている。
本タスクの影響範囲はspec.mdに明示された通り、`workflow-plugin/mcp-server/src/phases/definitions.ts`と`workflow-plugin/hooks/bash-whitelist.js`の2ファイルに厳密に限定されている。
この設計方針に従い、UI設計フェーズの成果物として「ユーザー向けインターフェースへの変更なし」という判断を正式に記録する。

## 運用手順への影響

FR-R4A・FR-R4Bの実装後、運用担当者が実施すべき追加手順は`npm run build`によるTypeScriptトランスパイルとMCPサーバープロセスの再起動の2ステップのみである。
`npm run build`はワークフロープラグインの`mcp-server`ディレクトリで実行し、`definitions.ts`と`bash-whitelist.js`の変更がそれぞれ`dist/`ディレクトリに反映されることを確認する。
MCPサーバーの再起動はClaude Desktopの設定画面からサーバー再起動ボタンを押すか、プロセスを直接終了して再起動することで実施する。
ワークフローの標準運用手順（`/workflow start`→各フェーズの実行→`/workflow next`）そのものへの変更は発生しない理由は、FR-R4A・FR-R4Bがシステム内部ロジックの修正のみを対象としているためである。
再起動後は`workflow_status`を実行して現在のフェーズを確認し、同フェーズから作業を再開することでデータの整合性が保たれる。

---

## 互換性と移行計画

FR-R4A・FR-R4Bは後方互換性を完全に維持して設計されており、既存のワークフロータスク状態やドキュメントを引き継いだまま新バージョンを適用できる。
移行手順はビルド（`npm run build`）とMCPサーバー再起動の2ステップのみであり、データベースマイグレーションや設定ファイルの書き換えは不要である。
ロールバックが必要になった場合はGitの`git revert`または`git checkout`を使用して修正前のコミットに戻し、再度`npm run build`とMCPサーバー再起動を実施することで元の状態に復帰できる。
`workflow-state.json`のHMAC整合性はMCPサーバーが管理するため、サーバーバージョンを切り替えても既存タスクのHMACが再計算されることはなく、タスク状態の破損は発生しない。
本修正の適用前後においてOrchestratorが参照するフェーズ定義・subagentテンプレート形式・Bashホワイトリスト判定ロジックは段階的に移行する必要がなく、ビルドと再起動のみで即時に切り替わる設計である。
