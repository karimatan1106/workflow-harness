# 脅威モデル: FR修正プロセス第4ラウンド問題の根本原因解決

## サマリー

- 目的: FR-R4AおよびFR-R4Bの変更（definitions.tsテンプレート拡張とbash-whitelist.jsフェーズ追加）に伴う脅威を分析し、リスク評価と対策を明示する
- 主要な決定事項: 両修正はいずれも低リスクに分類され、追加のセキュリティ対策なしで実施可能と判断する。プロンプトインジェクションリスクは構造上発生しない。Bashホワイトリスト拡張は意図した範囲のみに影響する
- リスク評価の結論: 全6脅威（プロンプトインジェクション、Bash権限拡張、HMAC整合性、サーバー再起動フロー、subagent出力品質、フック動作変化）を分析した結果、全件がLow以下と判定された
- 影響範囲: FR-R4Aはdefinitions.tsの静的文字列追加のみ、FR-R4Bはbash-whitelist.jsの配列要素追加のみであり、いずれも他モジュールへの波及影響は発生しない
- 次フェーズで必要な情報: planning（spec.md作成）フェーズでは本脅威モデルの結論（両修正は後方互換かつ低リスク）を前提として実装計画を策定する

---

## 変更対象の概要

### FR-R4A対象: definitions.ts（performance_testサブフェーズテンプレート）

`workflow-plugin/mcp-server/src/phases/definitions.ts`の`performance_test`エントリ内の`subagentTemplate`文字列フィールドに、必須セクション向けガイダンス文字列を追記する変更である。
このファイルはMCPサーバーのコアモジュールとして起動時にロードされ、Node.jsのモジュールキャッシュに保持される。
変更内容は静的な文字列リテラルの追加であり、実行時の動的評価や外部データの注入は発生しない。
テンプレートはsubagent起動時のプロンプト生成に使用されるが、ユーザー入力や外部入力がテンプレート文字列に反映される仕組みは存在しない。

### FR-R4B対象: bash-whitelist.js（verificationPhases配列）

`workflow-plugin/hooks/bash-whitelist.js`の`getWhitelistForPhase`関数内にある`verificationPhases`配列に、文字列`'parallel_verification'`を1件追加する変更である。
このフックファイルはBashコマンド実行リクエストのたびに新しいNodeプロセスで読み込まれるため、ディスク上のファイル変更が即座に反映される。
変更により`parallel_verification`フェーズで`BASH_WHITELIST.testing`カテゴリが追加許可される。
testingカテゴリに含まれるコマンドは`npm test`、`npm run test`、`npx vitest`、`npx jest`、`npx playwright test`、`pytest`であり、いずれもテスト実行専用コマンドである。

## 脅威分析

### 脅威1: プロンプトインジェクション（FR-R4A）

#### 脅威の説明

`performance_test.subagentTemplate`にガイダンス文字列を追加することで、悪意ある文字列がsubagentのプロンプトに注入されるリスクを検討する。

#### 評価

テンプレート文字列はTypeScriptソースコードのリテラルとして定義されており、実行時に外部入力が挿入される箇所は存在しない。
プロンプト生成時に外部から変数展開される部分は`taskName`と`docsDir`などの内部管理値のみであり、これらはMCPサーバーが管理するタスク状態から取得され、ユーザーが直接操作できる値ではない。
追加するガイダンス文字列は固定のMarkdown形式のガイダンステキストであり、スクリプト実行や命令上書きが可能な形式ではない。
したがって、FR-R4Aの変更によりプロンプトインジェクションリスクが増大する可能性はないと判断する。

- リスクレベル: Low（ソースコードリテラルの追加のみ、外部入力経路なし）
- 対策: 追加テンプレート文字列が固定テキストであることをコードレビューで確認する

---

### 脅威2: Bashコマンド実行権限の過剰拡張（FR-R4B）

#### 脅威の説明

`verificationPhases`配列への`parallel_verification`追加により、意図しないフェーズや意図しないコマンドカテゴリが許可されるリスクを検討する。

#### 評価

変更は`verificationPhases`配列に1文字列を追加するのみであり、既存のホワイトリスト判定ロジックには手を加えない。
`parallel_verification`フェーズで許可されるカテゴリは既存の`verificationPhases`グループと同一の`readonly`・`testing`・`gh`であり、新たなカテゴリが追加されるわけではない。
`implementation`カテゴリ（`npm install`、`npm run build`、`mkdir`、`rm`、`git add`、`git commit`）は引き続き許可されないため、`parallel_verification`フェーズでファイル削除やパッケージインストールが行われるリスクは生じない。
他のフェーズのホワイトリスト判定には影響しない。フェーズ名の文字列比較はそれぞれ独立した配列によって管理されており、配列間の相互干渉はない。

- リスクレベル: Low（既存の許可カテゴリを共有するのみ、新規カテゴリの追加なし）
- 対策: 変更後の動作確認として`parallel_verification`フェーズで`git commit`などの禁止コマンドが依然としてブロックされることを確認する

---

### 脅威3: HMAC整合性への影響

#### 脅威の説明

`definitions.ts`の変更後にMCPサーバーを再起動する際、`workflow-state.json`のHMAC整合性が損なわれるリスクを検討する。

#### 評価

`definitions.ts`はフェーズ定義・テンプレート・モデル設定を保持するコアモジュールであるが、HMAC計算の対象はタスクの状態データ（`workflow-state.json`の内容）である。
`definitions.ts`の内容自体はHMAC計算に含まれない。そのため`definitions.ts`を変更しても既存の`workflow-state.json`のHMAC値は影響を受けない。
MCPサーバーの再起動後も`stateManager`は同じHMACキーを使用して既存の状態ファイルを読み込むことができる。
HMAC整合性が問題となるのは`workflow-state.json`を直接編集した場合であり、今回の変更はそれに該当しない。

- リスクレベル: Low（HMAC対象はstate内容であり、definitions.tsの変更は対象外）
- 対策: MCPサーバー再起動後に`workflow_status`でフェーズが正常に読み込まれることを確認する

---

### 脅威4: MCPサーバー再起動フローのリスク

#### 脅威の説明

`definitions.ts`変更後のビルド・再起動フローで、サーバークラッシュやデッドロック状態が発生するリスクを検討する。

#### 評価

MEMORY.mdに記録された既知パターンとして「MCPサーバーダウン時にhooksが全編集をブロックするデッドロック」が存在する。`definitions.ts`のコンパイルエラーや起動失敗が生じた場合、このデッドロック状態に陥る可能性がある。
ただし今回の変更は文字列リテラルの追加のみであり、型エラーや構文エラーが発生するリスクは極めて低い。`npm run build`でコンパイルエラーがないことを確認してからサーバー再起動を実施する手順を踏むことで、リスクを実質的に排除できる。
万一デッドロック状態が発生した場合の回避手順は既存のMEMORY.mdに記載されており（古いタスクのフェーズを`build_check`等に変更する）、対応方法が確立されている。

- リスクレベル: Low（文字列追加のみ、コンパイルエラーリスク極小。デッドロック回避手順も確立済み）
- 対策: ビルドコマンド`npm run build`を再起動前に必ず実行し、コンパイル成功を確認すること

---

### 脅威5: テンプレートガイダンスによるsubagent出力の品質低下

#### 脅威の説明

ガイダンス文字列の追加により、subagentが指示に従いすぎて成果物内で同一フレーズを繰り返し、重複行バリデーションエラーを引き起こすリスクを検討する。

#### 評価

artifact-validatorの重複行検出は、トリム後に完全一致する行が3回以上出現した場合にエラーとなる。
追加するガイダンスは「各セクションにOK例とNG例を示し、5項目を列挙するよう指示する」形式であり、成果物の実際のコンテンツ行をsubagentが生成する際の指針として機能する。
ガイダンス自体が成果物にそのままコピーされることはない（subagentは指示に従って独自コンテンツを生成する）。
既存の`## サマリー`セクション向けガイダンスが機能していることから、同様の形式で必須セクション向けガイダンスを追加しても成果物品質の向上が期待できる。

- リスクレベル: Negligible（ガイダンスは成果物に直接転写されない、既存ガイダンスの効果が実証済み）
- 対策: 実装後の初回ラウンドでperformance-test.mdのバリデーション通過を確認する

---

### 脅威6: bash-whitelist.js変更によるフック動作の予期しない変化

#### 脅威の説明

`verificationPhases`配列への追加が、配列の順序依存や変数参照の問題により他のフェーズ判定に影響するリスクを検討する。

#### 評価

`getWhitelistForPhase`関数の判定ロジックはif-else ifチェーンではなく、各フェーズグループ配列に対して`includes(phase)`を呼び出す独立した構造をとっている。
`parallel_verification`という文字列は他のフェーズグループ配列（`qualityPhases`、`implementationPhases`など）には存在しないため、追加後も重複マッチが発生しない。
JavaScriptの配列`includes`メソッドは順序非依存であるため、配列末尾への追加で問題は生じない。
既存の`security_scan`、`performance_test`、`e2e_test`、`ci_verification`の各サブフェーズ名が`verificationPhases`に含まれる状況は変更されないため、これらのサブフェーズのホワイトリスト判定は引き続き正常に動作する。

- リスクレベル: Low（`includes`による独立判定、配列間の干渉なし）
- 対策: `bash-whitelist.test.js`の既存テストが引き続きパスすることを確認する

## 採用しないアプローチとその理由

### テンプレート文字列のサニタイズ処理追加

`subagentTemplate`を動的に組み立てる前にサニタイズ処理を追加するアプローチは採用しない。
テンプレートは静的リテラルであり外部入力が注入される経路が存在しないため、サニタイズは過剰な対応となる。
実装コストに見合ったリスク低減効果が得られないため、採用しない。

### Bashホワイトリストの段階的拡張（サブフェーズ単位の個別許可）

`parallel_verification`フェーズ全体ではなく、サブフェーズの種類によって動的に許可カテゴリを変える実装は採用しない。
`workflow-state.json`の`phase`フィールドは親フェーズ名`parallel_verification`を保持するため、サブフェーズ単位の動的判定には状態ファイルの追加フィールド参照が必要となり変更規模が大きくなる。
また、全4サブフェーズ（manual_test、security_scan、performance_test、e2e_test）でtestingカテゴリが必要であり、サブフェーズ単位の制御による安全性向上効果は得られないため採用しない。
