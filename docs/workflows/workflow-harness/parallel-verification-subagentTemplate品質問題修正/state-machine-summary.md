# State Machine Diagram - Summary

## サマリー

- **目的**: definitions.tsのparallel_verificationフェーズ配下4サブフェーズテンプレートへガイダンスを追記する実装の状態遷移を表現する。4つの機能要件（FR-1～FR-4）の追記順序、各段階でのテンプレート文字列操作、そしてビルド・テスト・検証フローを図式化する。
- **主要な決定事項**: 状態遷移は4つのフェーズ追記処理（FR-1からFR-4へのシーケンシャル進行）と、その後のビルド実行・テスト実行・変更検証の3段階で構成。各FR追記時に「テンプレート挿入位置の確認」と「ガイダンス文字列の追記」を状態で明示化。変更が4サブフェーズのテンプレート文字列のみに限定されることを検証フェーズで確認。
- **対象システム**: MCP サーバーのコアモジュール（definitions.ts）。変更は既存テンプレート文字列への追記のみで、削除や変更は行わない。影響範囲は parallel_verification 配下の manual_test、security_scan、performance_test、e2e_test の4つのサブフェーズに限定される。
- **検証状況**: ビルド成功（npm run build）、テスト合格（npm test）、git diff確認、ガイダンス内容検証の4段階で実装の正確性を確保。各段階で定義された確認項目を実施することにより変更品質を保証する。
- **次フェーズで必要な情報**: PR-1～PR-4の追記テキストの完全な文字列リテラル形式。追記後の各テンプレート文字列が artifact-validator の必須セクション要件（`## 出力` セクションの維持）を満たすことを確認。MCPサーバーの再起動手順（npm run build 後にプロセス再起動）を実施して、変更がメモリ上のモジュールキャッシュに反映されることが必須。

## 状態遷移の概要

本図は定義された実装タスクを以下の階層で表現する。

### フェーズ1: 初期状態から第1機能要件（FR-1）の完了まで

タスク開始時の初期状態（Start）から PreImplementation 状態へ遷移。
ここでは definitions.ts の manual_test テンプレート内における挿入位置（906行目付近の「## 出力」の直前）を確認する。
確認が完了すると FR1_Ready 状態に遷移し、FR-1 ガイダンス文字列を実装テンプレート内に追記する。
追記対象文字列は「複数シナリオの合否行は必ずシナリオ番号または操作対象名を含めて一意にすること。」の直後、「## 出力\n${docsDir}/manual-test.md」の直前。

FR-1 の追記内容は実行日時行の一意化に関するガイダンス。複数テストシナリオで同一の日付文字列行が3件以上出現する場合、artifact-validator の重複行検出がエラーを返す問題に対処。各シナリオ番号や操作対象名を含めることで行を一意化し、重複エラーを防止する推奨パターンを記述。

### フェーズ2: FR-2の追記処理

FR1_Completed 状態から FR2_Ready へ遷移。
security_scan テンプレートの追記位置（918行目付近）を確認。

### フェーズ3: FR-3の追記処理

FR2_Completed 状態から FR3_Ready へ遷移。
e2e_test テンプレートの追記位置（942行目付近）を確認。

散文形式またはインラインコード表記を使用する推奨パターンを記述。

### フェーズ4: FR-4の追記処理

FR3_Completed 状態から FR4_Ready へ遷移。
performance_test テンプレートの追記位置（930行目付近）を確認。

FR-4 の追記内容は計測日時行の一意化と角括弧禁止パターンの横断的対策に関するガイダンス。

複数計測シナリオ・複数計測対象にわたる計測日時行について、各行に計測対象名またはシナリオ番号を含めて一意性を確保。

FR4_Completed 状態から BuildExecution へ遷移。

## 設計上の特徴

本状態遷移図の設計上の特徴は以下の通り。

### 順序性の重視
4 つの FR（FR-1～FR-4）は指定された順序で追記される。
各 FR は異なるサブフェーズテンプレートに対応し、追記位置も定められている。
シーケンシャルな処理順序により、全体の一貫性と追跡可能性を確保。

### 明示的な検証ステップ
ビルド実行、テスト実行、変更内容検証、ガイダンス内容検証の 4 段階で、実装の正確性を多層的に確認。
各検証ステップは独立した確認項目を持つため、一つのステップが失敗した場合は該当する箇所のみ修正し再実行可能。

### テンプレート構造の保護
既存テンプレート文字列の削除や変更は行わず、末尾への追記のみに限定。
「## 出力」セクションの位置を変更しない設計により、既存テンプレートロジックへの影響を最小化。

### 変更スコープの明確化
4 つのサブフェーズ（manual_test、security_scan、performance_test、e2e_test）のテンプレート文字列のみが変更対象。
threat_modeling、planning、state_machine、flowchart、ui_design等の他フェーズテンプレートは変更対象外であることを図式化で明示。

## 実装後の期待効果

本実装により parallel_verification フェーズのサブエージェントが以下の動作を実施すると期待される。

### manual_test での期待動作
複数テストシナリオを実施する際、実行日時や実行環境の行にシナリオ識別子を含めて各行を一意化。
同一日付行 3 件以上の重複によるバリデーションエラーを回避。

### security_scan での期待動作
脅弱性スキャン結果セクション内で ### サブヘッダーを使用する際、各サブヘッダー直下に十分な実質行を配置。
親 ## セクション全体で 5 行以上の実質行を確保し、セクション密度要件を満たす。

### e2e_test での期待動作
テストシナリオ結果報告でファイル名やディレクトリ名を列挙する際、散文形式またはインラインコード表記を選択。
コードフェンス外での配列記法使用を回避し、角括弧プレースホルダー検出エラーを防止。

### performance_test での期待動作
複数計測シナリオ・複数計測対象の計測結果を報告する際、計測対象名またはシナリオ番号を含めて各行を一意化。
ファイル名列挙時にも散文形式またはインラインコード表記を使用。
FR-1 と FR-3 と同等の品質配慮をパフォーマンステスト文書で実施。
