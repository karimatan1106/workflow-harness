## サマリー

- 目的: parallel_verificationフェーズの3サブフェーズ（manual_test・security_scan・e2e_test）で繰り返し発生したバリデーション失敗の根本原因を修正するための要件を定義する
- 主要な決定事項: definitions.ts のsubagentTemplateに不足している3種類のガイダンスをそれぞれ追記することで、同種の問題の再発を防止する
- 次フェーズで必要な情報: 各問題の修正対象は `C:\ツール\Workflow\workflow-plugin\mcp-server\src\phases\definitions.ts` の3カ所（manual_testテンプレート・security_scanテンプレート・e2e_testテンプレート）であり、変更はテンプレート文字列の追記のみでコアロジックへの変更は不要である
- 影響範囲の判断: performance_testテンプレートについても横断的対策として問題1・問題3と同等の注意事項を追記することを本フェーズで決定する

## 機能要件

### FR-1: manual_test テンプレートへの実行日時行一意化ガイダンス追記

問題1の根本原因は、manual_testテンプレートの「テスト結果セクションの行数ガイダンス」に、実行日時・環境情報等のメタ情報行についての一意化指示が欠落していることである。
複数のテストシナリオが存在する場合、実行日時を `2026-02-23` のように共通フォーマットで記述すると3件以上でバリデーターが重複行エラーを検出する。

修正前の状態として、現行テンプレート（definitions.ts 906行）の「テスト結果セクションの行数ガイダンス」には「実行日時の記録」という項目が存在するが、複数シナリオで同一文字列になった場合の一意化方法は記載がない。
修正後の期待状態として、「実行日時行もシナリオ識別子または操作対象名を含めて一意にすること」というガイダンスを、「テスト結果セクションの行数ガイダンス」セクションに追記する。
具体的なNGとOKパターンを示して、実行日時・環境情報行の一意化方法をsubagentが理解できる形式で記述すること。

検証方法として、修正後のテンプレートでsubagentが複数シナリオの実行日時行を異なる記述にすることを確認する。また artifact-validator.ts に対して重複行チェックを手動実行し、同一の実行日時行が3件以上存在するとエラーが発生することを確認する。

### FR-2: security_scan テンプレートへの ### サブヘッダー実質行非カウント警告追記

問題2の根本原因は、security_scanテンプレートの「セクション密度に関するガイダンス」に、`###` サブヘッダーが実質行としてカウントされないという明示的な説明が欠落していることである。
artifact-validator の isStructuralLine はルール1（`^#+\s`）により `###` ヘッダー行を構造要素と判定し、実質行にカウントしない。そのため、 `## セクション` 内に `###` サブヘッダーを多数配置すると、各サブヘッダー直下の記述が少ない場合に親セクション全体の実質行数が5行を下回る。

修正前の状態として、現行テンプレート（definitions.ts 918行）には「5行以上の実質行」という要件は記載されているが、`###` サブヘッダーが実質行にカウントされない点と、サブヘッダーを多用する場合の対処法は記載がない。
修正後の期待状態として、「セクション密度ガイダンス」に対して「`###` サブヘッダー行自体は実質行としてカウントされないため、サブヘッダーを複数配置する場合は各サブヘッダー直下に3行以上の実質行を記述して親セクション合計5行以上を確保すること」という警告を追記する。
具体的な例示として、`###` を3つ配置したセクションが実質行4行以下になるNGパターンと、各サブヘッダー直下に実質行を補充したOKパターンを記述すること。

検証方法として、修正後のテンプレートでsubagentが `###` サブヘッダー多用時も実質行5行以上を維持した成果物を作成できることをartifact-validatorでの検証により確認する。

### FR-3: e2e_test テンプレートへのファイル名列挙時の角括弧禁止パターン追記

問題3の根本原因は、e2e_testテンプレートの「角括弧プレースホルダー禁止」のガイダンスが一般的な説明にとどまり、テスト文書で実際に発生しやすい「ファイル名列挙での配列記法使用」というパターンへの具体的な言及が欠落していることである。
テスト結果の記述においてsubagentが `['code-review.md']` のような配列アクセス記法をコードフェンス外の散文テキストに書いてしまうことが再発パターンとして確認されている。

修正前の状態として、現行テンプレート（definitions.ts 942行）には角括弧プレースホルダー禁止の一般説明は存在するが、ファイル名列挙・辞書キー参照・配列要素の参照といったテスト文書に特有のユースケースへの明示的な警告がない。
修正後の期待状態として、「角括弧プレースホルダー禁止・コードフェンス外の記述ルール（e2e_test特化ガイダンス）」セクションとして以下の内容を追記する。散文テキストやリスト項目にファイル名を列挙する場合は配列記法を使わないこと、辞書のキー名を参照する場合は「キー名 `foo` の値」のようにインラインコードを使うか散文形式で記述すること、コードフェンス内であれば配列記法・正規表現文字クラスを安全に記述できることを明示すること。
具体的なNGパターンとして `['code-review.md', 'security-scan.md']` のようなコードフェンス外の配列記法を、OKパターンとして散文形式での記述例やコードフェンス内での記述例を示すこと。

検証方法として、修正後のテンプレートでsubagentが複数ファイル名の列挙をコードフェンス外で配列記法を使わずに記述できることをartifact-validatorでの検証により確認する。

### FR-4: performance_test テンプレートへの横断的対策追記（FR-1・FR-3と同等）

research.mdの調査において、performance_testテンプレートは今回の問題に直接関与していないが他3サブフェーズと同様の品質問題が発生するリスクがあると記録されている。
横断的対策として、performance_testテンプレートにも問題1で定義した「実行日時・計測日時行の一意化ガイダンス」と問題3で定義した「ファイル名列挙時の角括弧禁止パターン」の両方を追記する。
追記する内容は FR-1・FR-3 で定義したガイダンスと同等であり、performance_testの文脈に合わせて「計測日時行もシナリオ識別子を含めること」「パフォーマンスレポートで参照するファイル名を散文形式で列挙すること」という形に書き直して記述すること。

検証方法として、FR-1・FR-3の検証と同様の手順を performance_test 成果物に対して適用することで確認する。

## 非機能要件

### NFR-1: 後退テスト合格（既存テスト非破壊）

今回の変更は definitions.ts のテンプレート文字列の追記のみであり、コアロジック（artifact-validator.ts・state-manager.ts・next.ts等）への変更を行ってはならない。
既存の npm test によるテストスイートが全て合格し続けることを確認することが必須条件である。
テスト実行は `C:\ツール\Workflow\workflow-plugin\mcp-server` ディレクトリで `npm test` を実行して確認する。
特に artifact-validator.test.ts・next.test.ts・state-manager.test.ts が合格していることを重点的に確認すること。

### NFR-2: MCP サーバー再起動不要の変更範囲

今回の変更はテンプレート文字列の追記のみであるため、MCP サーバーの再起動は不要である。ただし、npm run build によるトランスパイルは実施し、dist 配下のコンパイル済みファイルが更新されることを確認する必要がある。
コアロジックのファイルを変更した場合は CLAUDE.md のルール22に従い MCP サーバーの再起動が必須となるが、本タスクではその状況を回避することが前提条件である。

### NFR-3: 追記内容の品質要件適合

各テンプレートに追記するガイダンス文字列自体が artifact-validator の要件（禁止語不使用・角括弧プレースホルダー不使用・同一行3回以上繰り返し不使用）を満たしていること。
追記内容に禁止語（英語4語・日本語8語の合計12語）が含まれていないことを実装前に確認すること。
追記するNGパターンの記述にも角括弧を使用する場合は必ずコードフェンス内に配置し、コードフェンス外には散文形式で記述すること。

### NFR-4: 変更の最小化原則

4つの修正対象テンプレートに対して、問題解決に必要な最小限のテキストを追記する形式とする。既存のガイダンス内容を削除・変更してはならない。既存ガイダンスの末尾、または関連する既存セクションの末尾への追記という形をとることで、既存のsubagentの動作に悪影響を及ぼさないことを確認すること。

### NFR-5: 影響範囲の確認

変更を行う `definitions.ts` はMCPサーバーのコアファイルであり、全フェーズのsubagentTemplateが格納されている。今回の変更範囲は parallel_verification サブフェーズの4テンプレートのみであることを git diff で確認し、他のフェーズ定義（commit・push・ci_verification等）が変更されていないことを確認すること。
