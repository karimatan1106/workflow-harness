## サマリー

- 目的: parallel_verificationフェーズ（manual_test・security_scan・performance_test・e2e_test）の各subagentTemplateに、過去のバリデーション失敗パターンを防止する具体的なガイダンスを追記する実装仕様を定める
- 主要な決定事項: 4サブフェーズそれぞれに問題固有の追記を行う。manual_testへはFR-1（実行日時行の一意化）、security_scanへはFR-2（サブヘッダー多用時の密度確保）、e2e_testへはFR-3（ファイル名列挙時の角括弧禁止パターン特化）、performance_testへはFR-4（FR-1とFR-3の横断適用）を追記する
- 変更対象ファイル: `C:\ツール\Workflow\workflow-plugin\mcp-server\src\phases\definitions.ts` の1ファイルのみ
- 変更方式: 各サブフェーズの`subagentTemplate`文字列の末尾（`## 出力\n${docsDir}/...` の直前）に追記する
- 次フェーズで必要な情報: 各追記箇所の既存テキストと追記するテキストの完全な文字列。追記後は`npm run build`でトランスパイルし、既存テストの合格を確認すること

## 概要

本仕様書は、`definitions.ts`の`parallel_verification`配下4サブフェーズのテンプレート文字列に対して行う追記の内容と場所を定義する。
変更の影響範囲は`parallel_verification`直下の4サブフェーズ（manual_test・security_scan・performance_test・e2e_test）のみであり、コアロジックや他フェーズのテンプレートは変更しない。

追記の背景として、過去の実行履歴から3種類のバリデーション失敗パターンが繰り返し発生していることが確認されている。
第1のパターンは複数シナリオにまたがる実行日時行の重複で、同一の日付文字列が3件以上並んだ際にartifact-validatorの重複行検出がエラーを返す。
第2のパターンはsecurity_scanの`###`サブヘッダー多用時の実質行不足で、`isStructuralLine()`がヘッダー行を構造要素として除外するため親セクションの実質行数が5行を下回る。
第3のパターンはe2e_testにおけるコードフェンス外の配列記法使用で、ファイル名列挙でsubagentが散文テキスト内に角括弧を書いてしまう事例が確認されている。

各追記テキストは、artifact-validatorの禁止語（英語4語・日本語8語）を含まず、コードフェンス外に角括弧を使用せず、同一行が3回以上繰り返されない内容であることを事前に確認済みである。
実装後の検証は`npm run build`によるトランスパイル成功と`npm test`による全テストスイートの合格によって行う。

## 実装計画

### 変更方針

4つの追記はすべてテンプレート文字列（TypeScriptのシングルクォート囲みの文字列リテラル）の末尾への挿入として実施する。挿入位置はそれぞれ「`## 出力\n${docsDir}/...`」の文字列の直前とする。これにより既存のガイダンス内容を一切変更することなく、新規のセクションを末尾に追加できる。

### FR-1の実装（manual_testテンプレートへの追記）

対象箇所は`definitions.ts`の906行目付近、`manual_test`の`subagentTemplate`文字列内における`## 出力\n${docsDir}/manual-test.md`の直前である。

挿入するテキスト（テンプレート文字列内での`\n`区切り形式）:

```
\n\n## 実行日時・環境情報行の一意化（FR-1: 重複行防止）\n複数のテストシナリオにわたって実行日時や実行環境を記述する場合、それらの行にもシナリオ番号または操作対象名を含めて各行を一意にすること。同一の日付文字列のみで構成された行が3件以上出現すると、artifact-validatorが重複行エラーを検出する。\n- 問題のある記述パターン: 「- 実行日時: 2026-02-23」をシナリオ1・2・3でそれぞれ記述する（3件以上の同一行でエラー）\n- 推奨パターン1: 「- TC-1の実行日時: 2026-02-23、対象ファイルはdefinitions.tsの906行目付近」のようにシナリオ識別子と対象箇所を含める\n- 推奨パターン2: 「- TC-2の実行環境: Windows 11、Node.js 20.x、テスト対象はmanual_testテンプレートの出力確認」のように環境情報にシナリオ番号を付加する\n同じ実行日に複数シナリオを実施した場合でも、各行にシナリオ番号または具体的な検証内容を含めることで一意性を確保できる。
```

### FR-2の実装（security_scanテンプレートへの追記）

対象箇所は`definitions.ts`の918行目付近、`security_scan`の`subagentTemplate`文字列内における`## 出力\n${docsDir}/security-scan.md`の直前である。

挿入するテキスト（テンプレート文字列内での`\n`区切り形式）:

```
\n\n## サブヘッダー多用時のセクション密度確保（FR-2: 実質行数不足防止）\nartifact-validatorの`isStructuralLine()`関数はルール1（`^#+\s`パターン）により`###`で始まるサブヘッダー行を構造要素と判定し、実質行としてカウントしない。このため、`##`セクション内に`###`サブヘッダーを複数配置する場合は、親セクション全体で5行以上の実質行が確保されるように各サブヘッダー直下に十分なコンテンツを記述すること。\n- 問題のある構造パターン: `## 脆弱性スキャン結果`セクション内に`### npm audit結果`、`### 静的解析結果`、`### 設定ファイル確認`の3つのサブヘッダーを配置し、各直下に実質行1行のみ記述する（合計3行で実質行不足になる）\n- 推奨構造パターン: 各`###`サブヘッダーの直下に最低3行の実質行（コンテンツを含む行）を記述し、`##`セクション全体での実質行数が5行以上になるよう構成する\n- 具体的な確保方法: `###`を3つ使用する場合は各サブヘッダー直下に3行以上の実質行を書くか、サブヘッダーを使わずに`##`セクション直下に5行以上の箇条書きを記述する\n実質行の判断基準はコロンの後にコンテンツが存在する行と通常のテキスト・箇条書き行であり、空白行・水平線・コードフェンス内の行・`###`等のヘッダー行は実質行としてカウントされない。
```

挿入するテキスト（テンプレート文字列内での`\n`区切り形式）:

```
\n\n## テスト文書固有の角括弧禁止パターン（FR-3: e2e_test特化ガイダンス）\nartifact-validatorはコードフェンス外の角括弧をプレースホルダーとして検出する。E2Eテストのシナリオ記述や結果報告でファイル名・ディレクトリ名を列挙する際は、コードフェンス外での配列記法の使用が禁止されている。\n- 禁止パターン（コードフェンス外での配列記法）: 散文テキストや箇条書きで複数ファイルを「code-review.mdとsecurity-scan.md」のように列挙する際に配列記法で記述すること\n- 推奨パターン1（散文形式）: 「code-review.mdファイルおよびsecurity-scan.mdファイルを対象とした」のように散文形式で列挙する\n- 推奨パターン2（インラインコード）: バッククォートで囲んだインラインコード表記を使用することでコードフェンス外の角括弧を回避できる\n- 安全な代替手段（コードフェンス内）: コードフェンス内であれば配列記法・正規表現の文字クラス表記を安全に記述できる\n辞書のキー名を参照する場合も同様に、コードフェンス外では散文形式またはインラインコードを使用し、コードフェンス外の角括弧を含む記述を避けること。
```

### 変更後の動作期待値

FR-1の追記により、manual_testサブエージェントが複数シナリオの実行日時行を記述する際、各行にシナリオ識別子を含めて一意にするという動作が期待される。これにより同一日付行の3件以上重複によるバリデーションエラーが回避される。

FR-2の追記により、security_scanサブエージェントが`###`サブヘッダーを多用する構造を選択した際も、各サブヘッダー直下に十分な実質行を配置するという動作が期待される。これにより親`##`セクションの実質行不足によるバリデーションエラーが回避される。

FR-3の追記により、e2e_testサブエージェントがファイル名列挙の際に散文形式またはインラインコードを選択し、コードフェンス外の配列記法を使用しないという動作が期待される。これにより角括弧プレースホルダー検出によるバリデーションエラーが回避される。

FR-4の追記により、performance_testサブエージェントがFR-1とFR-3と同等の品質配慮を計測文書でも実施するという動作が期待される。
