- 次フェーズで必要な情報: 各問題に対する具体的な追記箇所（行レベル）と追記内容の案を本ドキュメントの問題別調査セクションに記録している

## 調査対象ファイル

## 問題1: manual_test 重複行エラー（実行日時行）

### 現象と発生箇所

前タスクの parallel_verification では、複数のテストシナリオの「テスト結果」セクションに
`- **実行日時**: 2026-02-23` という行がそのまま複数回記述された。
artifact-validator の重複行検出（isStructuralLine 判定を経た後、3回以上一致でエラー）が発動した。

### isStructuralLine によるカウント対象の確認

artifact-validator.ts の isStructuralLine 関数（92〜111行）を調査した結果、以下が判明した。
`- **実行日時**: 2026-02-23` のような「太字ラベル + コロン + 日付値」の形式は、
isStructuralLine のルール6（`^\*\*[^*]+\*\*[:：]?\s*$`）には該当しない（コロン後に日付値があるため）。
ルール7（`- **ラベル**:` 形式でコロン後にコンテンツがない行）にも該当しない。
したがってこの行は実質行としてカウントされ、かつ重複行検出の対象となる。

### テンプレートの現状確認

manual_test の subagentTemplate（906行）には「評価結論フレーズの重複回避（特化ガイダンス）」セクションが存在する。
そこには「- 判定: 合格」のような短い判定行の重複を避けるための指示が記述されている。
しかし「実行日時行についてもシナリオ固有の日付または識別子を付加すること」という明示的な指示は存在しない。
テスト結果セクションの行数ガイダンスには「実行日時の記録（テストを実施した日時を具体的に記述する）」とあるが、
複数シナリオで実行日時行が同一文字列になる場合の一意化については言及がない。

### 結論

問題1の根本原因は「実行日時行もシナリオ識別子を含めて一意化する必要があること」の指示が欠落していることである。
重複行ガイダンスは判定系フレーズのみを対象としており、日時・環境行などのメタ情報行については記載がなかった。

## 問題2: security_scan セクション密度不足（追加評価セクション）

### 現象と発生箇所

前タスクでは security_scan の成果物に `## 追加評価: 変更コードの固有リスク評価` セクションが作成されたが、
そのセクション内の実質行数が4行となりバリデーションエラーが発生した。

### セクション密度計算の仕様確認

artifact-validator.ts の checkSectionDensity 関数（732〜816行）を調査した結果、以下が判明した。
セクション分割は `## ` を区切り文字として行われる（738行: `content.split(/^##\s+/m)`）。
したがって `### ` サブヘッダーはセクションを分割せず、親セクションに含まれたままになる。
ただし `###` ヘッダー行は isStructuralLine のルール1（`^#+\s`）により構造要素と判定される。
つまり `###` サブヘッダー行自体は実質行にカウントされない（structuralCount に加算される）。

### 実際の計算例

仮に `## 追加評価: 変更コードの固有リスク評価` セクションが以下の構成だったとする。
- `### セキュリティリスク評価` （構造行）
- `- 変更1のリスク評価: リスクなし` （実質行）
- `### パフォーマンス影響評価` （構造行）
- `- 変更2の影響評価: 影響なし` （実質行）
上記の構成では、実質行数は2行であり最低要件の5行を下回ってしまう。
さらに `###` ヘッダーを3つ配置した場合、各サブセクションに記述が1〜2行しかないと
親セクション全体の実質行数が5行を下回る可能性がある。

### テンプレートの現状確認

security_scan の subagentTemplate（918行）を確認した結果、セクション密度に関するガイダンスは存在する。
ただし `###` サブヘッダーは実質行にカウントされないという明示的な説明は含まれていない。
現テンプレートのガイダンスは「5行以上の実質行」を要求しているが、
`###` を使って小見出しを設けた場合に実質行の不足が発生しやすいことへの警告がない。

### 結論

問題2の根本原因は「`###` サブヘッダーは実質行としてカウントされないため、
サブヘッダーを多用するとセクション全体の実質行数が不足する可能性がある」ことの説明が
テンプレートに欠落していることである。この点を明示的に警告する記述が必要である。

## 問題3: e2e_test 角括弧プレースホルダー違反

### 現象と発生箇所

前タスクの e2e_test 成果物において、コードフェンス外の散文テキストに
`['code-review.md']` のような配列アクセス記法（角括弧を含む記述）が書かれていた。
artifact-validator の角括弧プレースホルダー検出が発動し、成果物全体がバリデーション失敗となった。

### 角括弧プレースホルダー検出の仕様確認

definitions.ts の buildQualityRequirements 関数（1150〜1159行）を調査した結果、以下が判明した。
コードフェンス外の行（Markdown本文の散文テキスト・箇条書き等）が検出対象となる。
コードフェンス内の行は extractNonCodeLines により検出から除外されるため安全である。
正規表現の文字クラスや配列アクセス記法をコードフェンス外に直接書くことが禁止対象と明記されている。

### e2e_test テンプレートの現状確認

e2e_test の subagentTemplate（942行）を確認した結果、以下のガイダンスが存在する。
「重複行回避の注意事項」「評価結論フレーズの重複回避（特化ガイダンス）」は存在する。
角括弧プレースホルダー禁止については一般的な説明として記述されているが、
「ファイル名を列挙する際に配列表記（角括弧）を使わないこと」や
「辞書のキー参照に角括弧を使わないこと」という具体的な禁止ユースケースへの言及がない。

### テンプレートとの対比

manual_test と security_scan のテンプレートにも角括弧プレースホルダーに関する言及がない。
全4サブフェーズのテンプレートに共通して「テスト対象ファイル名の列挙時に配列記法を使わないこと」
という具体的な注意事項が追記されていない状態である。

### 結論

問題3の根本原因は「テスト結果でファイル名やキー名を参照する際に配列アクセス記法を使わないよう」
という具体的な指示が e2e_test テンプレートに欠落していることである。
一般的な「角括弧プレースホルダー禁止」の説明は存在するが、
テスト文書でよく発生する「ファイル名列挙での角括弧使用」というパターンが明示されていない。

## 現状の問題箇所まとめ

各テンプレートの現状と不足ガイダンスを以下に整理する。

- manual_test テンプレート（906行）: 判定系フレーズの重複回避ガイダンスは存在するが、実行日時・環境情報等のメタ情報行についての一意化指示が欠落している
- security_scan テンプレート（918行）: セクション密度ガイダンスは存在するが、`###` サブヘッダーが実質行にカウントされない点の明示的な警告が欠落している
- e2e_test テンプレート（942行）: 角括弧プレースホルダー禁止の一般説明は存在するが、ファイル名列挙での配列記法という具体的なパターンへの言及が欠落している
- performance_test テンプレート（930行）: 今回の問題には直接関与していないが、他3サブフェーズと同様の品質問題が発生するリスクは存在する

## 次フェーズに向けた情報整理

requirements フェーズでは本調査結果を基に以下を定義する必要がある。

- 問題1への対策: manual_test テンプレートの「テスト結果セクションの行数ガイダンス」に「実行日時行もシナリオ識別子を含めて一意化すること」を追記する
- 問題2への対策: security_scan テンプレートのセクション密度ガイダンスに「`###` サブヘッダーは実質行にカウントされないため、サブヘッダーを多用する場合は各サブヘッダー直下に実質行を十分記述すること」を追記する
- 問題3への対策: e2e_test テンプレートに「ファイル名・辞書キー等を本文中に列挙する場合は配列記法（角括弧）を使わず散文形式またはコードフェンス内に記述すること」を追記する
- 影響範囲: 3問題への対応は definitions.ts のテンプレート文字列の追記のみであり、コアロジックへの変更は不要である
- 横断的対策の検討: 同様の問題が全4サブフェーズで共通して発生しうるため、performance_test テンプレートにも同等の注意事項を追記することを requirements フェーズで検討する
