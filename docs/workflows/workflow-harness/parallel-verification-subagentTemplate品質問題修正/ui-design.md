## サマリー

- 目的: parallel_verificationの4サブフェーズ（manual_test・security_scan・performance_test・e2e_test）のsubagentTemplateに追記するガイダンス文字列の構造・形式・配置位置を定義する
- 評価スコープ: `definitions.ts`内のテンプレート文字列構造と、MCPサーバーがworkflow_get_subphase_templateツール呼び出しに返すレスポンス形式、およびバリデーション失敗時のエラーメッセージ形式
- 主要な決定事項: 追記はすべて既存テンプレート末尾への挿入で実施し、既存内容の変更は行わない。追記テキストは`## 見出し`で始まる独立セクション形式とし、禁止語を含まない文体で記述する
- 検証状況: 設計フェーズのため実装前。追記内容の禁止語チェックはspec.md作成時点で目視確認済み
- 次フェーズで必要な情報: 各サブフェーズの追記直前テキストの完全文字列（spec.mdの変更対象ファイルセクションに記載済み）、および`npm run build`と`npm test`による事後検証手順

## CLIインターフェース設計

このタスクの変更対象はMCPサーバーの内部テンプレート文字列であり、外部から呼び出し可能なCLIコマンドは新設しない。
MCPサーバーとしての呼び出しインターフェースは既存のMCPプロトコルをそのまま使用する。

### workflow_get_subphase_template ツールの呼び出し形式

サブフェーズのテンプレートを取得する際に使用するMCPツールの呼び出し形式は以下のとおりである。
引数として`subPhaseName`（必須）と`taskId`（省略可能）を受け取り、テンプレート文字列を含むレスポンスを返す。
呼び出し例: `workflow_get_subphase_template({ subPhaseName: "manual_test", taskId: "task-001" })`
追記後のテンプレートは、このツール経由でOrchestrator（メイン会話）が取得してsubagentに渡す。
サブフェーズ名として指定できる値は`manual_test`、`security_scan`、`performance_test`、`e2e_test`（今回変更対象）の他に`threat_modeling`、`planning`、`state_machine`、`flowchart`、`ui_design`、`build_check`、`code_review`がある。
引数`taskId`を省略した場合はアクティブなタスクが自動選択され、`docsDir`等のプレースホルダーが当該タスクの値で置換される。

### npm run build コマンドのインターフェース

TypeScriptのトランスパイルは以下のコマンドで実行する。
実行ディレクトリ: `C:\ツール\Workflow\workflow-plugin\mcp-server`
コマンド形式: `npm run build`
成功時の出力: トランスパイル完了メッセージが標準出力に表示され、終了コードが0になる。
失敗時の出力: TypeScriptコンパイルエラーが標準エラー出力に表示され、終了コードが非0になる。
`dist/phases/definitions.js`に変換済みファイルが生成されることを確認する手段として`git diff`が利用可能である。

### npm test コマンドのインターフェース

テスト実行は以下のコマンドで実施する。
実行ディレクトリ: `C:\ツール\Workflow\workflow-plugin\mcp-server`
コマンド形式: `npm test`
成功時の出力: 全テストスイートの合格数が表示され、終了コードが0になる。
失敗時の出力: 失敗したテスト名とエラーメッセージが表示され、終了コードが非0になる。

## エラーメッセージ設計

バリデーション失敗時に返されるエラーメッセージは、artifact-validatorが生成する形式に従う。
各エラー種別とメッセージ形式を以下に示す。

### 重複行エラーのメッセージ形式

重複行エラーは、トリム後に完全一致する非構造行が3件以上出現した場合に発生する。
エラーメッセージ形式: `重複行が検出されました: "{重複している行のテキスト}" が {件数} 回出現しています`
FR-1（manual_test）およびFR-4（performance_test）への追記は、このエラーを防ぐためのガイダンスを提供する。
Orchestratorがこのエラーを受け取った場合、subagentを再起動して「実行日時行にシナリオ識別子を付加するよう修正」を指示する。
再起動時のリトライプロンプトには、エラーメッセージ全文をコードブロック内に引用し、具体的な修正指示を「改善要求」セクションとして追加する。

### セクション密度不足エラーのメッセージ形式

実質行数が5行に満たないセクションが存在する場合に発生するエラーである。
エラーメッセージ形式: `セクション "{セクション見出し}" の実質行数が不足しています ({実際の行数}/{必要行数}行)`
FR-2（security_scan）への追記は、`###`サブヘッダー多用時でもセクション密度が確保されるよう誘導する。
このエラーを受け取ったOrchestratorは、該当セクションに実質行を追加するよう指示する。

### 角括弧プレースホルダーエラーのメッセージ形式

コードフェンス外の行に角括弧が検出された場合に発生するエラーである。
エラーメッセージ形式: `角括弧プレースホルダーが検出されました: "{角括弧を含む行のテキスト}"`
FR-3（e2e_test）およびFR-4（performance_test）への追記は、このエラーを防ぐためのガイダンスを提供する。
Orchestratorがこのエラーを受け取った場合、散文形式またはインラインコードへの書き換えを指示してsubagentを再起動する。

### TypeScriptコンパイルエラーのメッセージ形式

`npm run build`失敗時に出力されるエラーは以下の形式である。
エラーの主要情報: ファイルパス、行番号、エラーコード（例: TS1005）、エラー説明文が含まれる。
追記操作はテンプレート文字列リテラル内への挿入であり、構文エラーが発生した場合はシングルクォートやバックスラッシュのエスケープ漏れが原因として疑う。

## APIレスポンス設計

workflow_get_subphase_templateツールのレスポンス形式を定義する。

### 追記前のレスポンス構造

このツールが返すレスポンスは以下のフィールドを含む。
`subagentTemplate`フィールドにはOrchestrator経由でsubagentに渡すべきテンプレート全文が格納される。
`minLines`フィールドには成果物に必要な最小行数が整数値で格納される。
`requiredSections`フィールドには必須セクション見出しの配列が格納される。
`outputFile`フィールドには成果物の出力先ファイルパスが文字列で格納される。
`docsDir`フィールドにはワークフロー成果物ディレクトリのパスが文字列で格納される。

### 追記後のレスポンス変化

FR-1からFR-4の追記完了後、`subagentTemplate`フィールドの内容が増加する。
`manual_test`の`subagentTemplate`には「実行日時・環境情報行の一意化」セクションが末尾（出力セクションの直前）に追加される。
`security_scan`の`subagentTemplate`には「サブヘッダー多用時のセクション密度確保」セクションが末尾（出力セクションの直前）に追加される。
`performance_test`の`subagentTemplate`には「計測日時行の一意化と角括弧禁止パターン」セクションが末尾（出力セクションの直前）に追加される。
`e2e_test`の`subagentTemplate`には「テスト文書固有の角括弧禁止パターン」セクションが末尾（出力セクションの直前）に追加される。
`requiredSections`と`minLines`は変更されない（既存の必須セクション定義はそのまま維持される）。

### git diff によるレスポンス変化の確認方法

追記後の実際のテンプレート内容は`git diff`コマンドで確認できる。
確認コマンド: `git diff workflow-plugin/mcp-server/src/phases/definitions.ts`
差分表示で追加行（先頭に`+`が付く行）が各サブフェーズのテンプレート末尾に集中していることを確認する。
削除行（先頭に`-`が付く行）は存在しないことを確認することで、既存テンプレートの破壊がないことを担保する。

## 設定ファイル設計

変更対象ファイルである`definitions.ts`のテンプレート文字列構造と追記箇所を定義する。

### definitions.ts のテンプレート文字列構造

`definitions.ts`内のサブフェーズ定義はオブジェクトリテラル形式で記述されており、各サブフェーズは`subagentTemplate`プロパティにシングルクォートで囲んだ文字列リテラルを持つ。
文字列リテラル内での改行はバックスラッシュnのエスケープシーケンスで表現される。
テンプレート末尾は`## 出力\n${docsDir}/{成果物ファイル名}`という固定パターンで終わる。
追記位置はこの`## 出力`セクションの直前であり、追記後も末尾は`## 出力`セクションで終わる。

### manual_test テンプレートの追記箇所

ファイル内の位置は`definitions.ts`の906行目付近である。
追記直前のテキストは`複数シナリオの合否行は必ずシナリオ番号または操作対象名を含めて一意にすること。\n\n## 出力\n${docsDir}/manual-test.md`という文字列パターンを含む。
追記するセクション見出しは`## 実行日時・環境情報行の一意化（FR-1: 重複行防止）`である。
追記後の構造: 既存テキスト → FR-1ガイダンスセクション → `## 出力\n${docsDir}/manual-test.md`の順になる。

### security_scan テンプレートの追記箇所

ファイル内の位置は`definitions.ts`の918行目付近である。
追記直前のテキストは`OK: 5観点それぞれについて「検出なし・根拠○○」の形式で記述し、5行以上を確保する\n\n## 出力\n${docsDir}/security-scan.md`という文字列パターンを含む。
追記するセクション見出しは`## サブヘッダー多用時のセクション密度確保（FR-2: 実質行数不足防止）`である。
追記後の構造: 既存テキスト → FR-2ガイダンスセクション → `## 出力\n${docsDir}/security-scan.md`の順になる。

### performance_test テンプレートの追記箇所

ファイル内の位置は`definitions.ts`の930行目付近である。
追記直前のテキストは`複数計測対象の評価行は必ず計測対象名または修正箇所の識別子を含めて一意にすること。\n\n## 出力\n${docsDir}/performance-test.md`という文字列パターンを含む。
追記するセクション見出しは`## 計測日時行の一意化と角括弧禁止パターン（FR-4: performance_test横断対策）`である。
追記後の構造: 既存テキスト → FR-4ガイダンスセクション → `## 出力\n${docsDir}/performance-test.md`の順になる。

### e2e_test テンプレートの追記箇所

ファイル内の位置は`definitions.ts`の942行目付近である。
追記直前のテキストは`複数シナリオの検証結論行は必ずシナリオ名または操作名を含めて一意にすること。\n\n## 出力\n${docsDir}/e2e-test.md`という文字列パターンを含む。
追記するセクション見出しは`## テスト文書固有の角括弧禁止パターン（FR-3: e2e_test特化ガイダンス）`である。
追記後の構造: 既存テキスト → FR-3ガイダンスセクション → `## 出力\n${docsDir}/e2e-test.md`の順になる。

### テンプレート文字列内でのエスケープ規則

テンプレート文字列はシングルクォートで囲まれているため、文字列内にシングルクォートが含まれる場合はバックスラッシュでエスケープする必要がある。
追記するガイダンス文字列にシングルクォートは含まれていないため、エスケープは不要である。
バックスラッシュnのエスケープシーケンスは改行として機能するため、追記テキスト内の改行はすべてバックスラッシュnで表現する。
テンプレート文字列内にはTypeScriptのテンプレートリテラル構文（`${docsDir}`等）が含まれており、シングルクォート文字列内で`${}`は通常文字列として扱われるため、変数展開は発生しない（`docsDir`変数は別途文字列連結で参照される設計である）。
