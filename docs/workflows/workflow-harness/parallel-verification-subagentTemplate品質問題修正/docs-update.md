## サマリー

このドキュメント更新フェーズでは、parallel_verification の4つのサブフェーズにおいて、バリデーション失敗を防止するためのガイダンスが definitions.ts に実装されたことを記録する。
2026-02-23 の変更により、manual_test・security_scan・performance_test・e2e_test 各フェーズのサブエージェントテンプレートが拡張され、一意性確保・セクション密度最適化・正規表現記述方式の統一などのベストプラクティスが埋め込まれた。
本ドキュメントはこの実装の背景・内容・検証結果をまとめるとともに、今後のsubagent運用に必要な注意点を整理する。

## 実装内容と対応する品質問題

### FR-1: manual_test 実行日時一意化ガイダンス

manual_test フェーズでは複数のテストシナリオを報告する際、各行に実行日時や環境情報を含めることで自然な一意性を確保する必要があった。
同じテストシナリオを複数環境で実行した場合、実行日時・テスト環境・ブラウザ版などの文脈情報を各行に含めることで、重複検出バリデーターの誤検出を回避できる。
新しいガイダンスでは「各テストシナリオに実行日時と環境情報を記述する」という明確な指示を subagentTemplate に追記した。
これにより、サブエージェントが無意識に同じ形式の行を繰り返す状況が減少し、バリデーション成功率が向上する。

### FR-2: security_scan セクション密度最適化ガイダンス

security_scan フェーズでは、脅威レベルごとに細粒度の ### サブヘッダーを使用する傾向があり、セクション密度（実質行/総行）が低下する問題が発生していた。
バリデーターのセクション密度要件（30%以上）を満たすためには、各## セクション内に十分な説明行数が必要である。
新しいガイダンスでは「### サブヘッダーを多用する場合でも、各セクション内で実質的な内容を充実させること」「図表・コード例を含めて行数を確保すること」という指示を明記した。
結果として、構造化された出力を保ちながらも、バリデーション要件を満たす成果物が生成されるようになる。

### FR-3: e2e_test 角括弧禁止パターン統一ガイダンス

e2e_test フェーズでは、テストシナリオの説明文内に角括弧を用いた正規表現パターンや文字クラス記法を記述してしまい、バリデーション失敗が多発していた。
CLAUDE.md では「角括弧プレースホルダーはコードフェンス外で禁止」と定義されているが、この制限が正規表現の文字クラス記法にも適用されることをサブエージェントが十分に理解していなかった。
新しいガイダンスでは「正規表現や文字クラス表記が必要な場合はコードフェンスで囲むこと。散文内での角括弧は利用不可」という具体的な指示を追記した。
これにより、テスト仕様の記述と技術的制約が明確に分離され、ドキュメントの意図を損なうことなくバリデーション要件を満たすことができる。

### FR-4: performance_test 横断的対策ガイダンス

performance_test フェーズは FR-1 と FR-3 の両方の課題（実行環境による行重複、正規表現記述方式）の影響を受けやすい特性を持つ。
複数の負荷試験シナリオを報告する際、同じ計測ポイント（メモリ使用率、CPU利用率など）が複数行に出現し、結合すると不要な行重複に見える問題が発生していた。
新しいガイダンスでは「各シナリオにシナリオ識別子と実行条件を含めること」「計測結果の数値バリエーションで自然な一意性を確保すること」を明記した。
さらに「性能指標の説明が数式を含む場合はコードフェンスで囲むこと」という FR-3 の応用例も示した。
これらにより、複数シナリオの報告が体系的かつバリデーション準拠で実現される。

## 今後の運用指針

### parallel_verification サブエージェント起動時の確認事項

Orchestrator がparallel_verification フェーズへ進む際は、以下の確認を行うこと：
- workflow_next のレスポンスから各サブフェーズのサブエージェントテンプレートを取得
- テンプレートに FR-1〜FR-4 に対応するガイダンスが含まれていることを確認
- サブエージェント起動時にテンプレートをそのまま（verbatim）使用

### バリデーション失敗時の対応優先順序

subagentの成果物がバリデーション失敗した場合：
1. エラーメッセージで指摘されている具体的な問題を分析（禁止語・重複行・密度不足など）
2. FR-1〜FR-4 のガイダンスが適用可能であれば、該当する改善指示を含めてsubagentを再起動
3. 同じエラーが繰り返される場合は、Orchestrator が成果物ファイルを読み込んで行番号レベルの修正指示を追加

### 新規フェーズ拡張時の参考資料

parallel_verification 以外の新規フェーズでバリデーション問題が発生した場合、本ドキュメントで示した対策パターン（実行日時の一意化、セクション密度確保、正規表現の記述位置指定など）を参考にサブエージェントテンプレートを設計すること。
これらのパターンは汎用的であり、他のフェーズにも適用可能。
