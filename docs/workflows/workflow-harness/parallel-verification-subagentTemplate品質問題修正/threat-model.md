## サマリー

- 目的: definitions.ts の parallel_verification サブフェーズ4テンプレートへの品質ガイダンス追記変更に対し、セキュリティ・品質・運用の観点から脅威を分析する
- 主要な決定事項: 今回の変更はプロンプトテキストの追記のみであり、コアロジックへの変更は行わないことを前提として脅威を限定的に評価する
- 評価スコープ: manual_test, security_scan, performance_test, e2e_test の4サブフェーズテンプレートを対象とし、追記内容そのものが新たな品質問題を引き起こさないことを検証する
- 特定された脅威: T-1（バリデーターが検出する語句の混入）、T-2（既存ガイダンスとの矛盾）、T-3（Node.jsキャッシュ問題）、T-4（テンプレート肥大化）の4件
- 次フェーズで必要な情報: 特定された4つの脅威（T-1〜T-4）に対して計画フェーズで緩和策を仕様に組み込むこと。NFR-3（追記内容自体の品質要件適合）がT-1の直接的な緩和策となる

## 脅威シナリオ

### T-1: 追記テキスト内へのバリデーター検出語句混入によるバリデーションループ

発生経路として、definitions.ts のテンプレート文字列にNGパターンの例示を記述する際に、バリデーターが検出する12の語句をコードフェンス外に直接記載してしまうケースが想定される。
テンプレート文字列はsubagentに渡され、subagentが受け取ったプロンプト内容を成果物に転記することがある。転記が発生すると成果物が artifact-validator に弾かれ、リトライループが発生する。
具体的なシナリオとして、FR-3（e2e_test テンプレート）のNGパターン例示で配列記法の説明に「架空のファイル名」という表現を使う場合でも、その表現の中に検出対象語句が紛れ込まないよう注意が必要である。
このリスクは禁止語混入防止ルール（NFR-3）で直接対処されており、追記内容は実装前に語句チェックを実施することが必須要件となっている。
深刻度はMediumと評価する。問題が発生しても本番データへの影響はなく、リトライが増えるだけだが、開発効率と信頼性に影響する。

### T-2: 既存ガイダンスと追記内容の矛盾による混乱

発生経路として、追記したガイダンスが既存のガイダンスと表現上矛盾または重複し、subagentが両方の指示を読んで解釈に迷うケースが想定される。
たとえば、既存ガイダンスに「同一行3回以上繰り返し禁止」という一般説明があり、追記ガイダンスに「実行日時行はシナリオ識別子を含めること」という具体化説明を加える場合、両者が別方針を示すように読めると混乱が生じる。
また、角括弧プレースホルダー禁止の一般説明と、e2e_test特化の配列記法禁止説明が互いに補完関係にあることを明確に示さないと、subagentが「どちらか一方のルールで十分」と誤解する可能性がある。
緩和策として、追記ガイダンスは既存説明の延長（具体化・補足）であることを文頭で明示し、既存ガイダンスを否定する記述を避けることが有効である。
深刻度はLowと評価する。矛盾が生じた場合もバリデーションが検出するため、本番障害には至らない。

### T-3: テンプレート変更がNode.jsキャッシュ問題により反映されないリスク

発生経路として、definitions.ts を変更してトランスパイル（npm run build）を実施しても、実行中のMCPサーバープロセスがメモリ内に古いキャッシュを保持し、変更後のテンプレートがsubagentに渡されないケースが想定される。
MEMORY.md に記録されている通り、MCPサーバーはNode.jsのrequire()でモジュールをキャッシュするため、ディスク上のdist/*.js更新だけでは動作中のサーバーに変更が反映されない。
ただし、CLAUDE.md のNFR-2（MCP サーバー再起動不要の変更範囲）では、今回の変更はテンプレート文字列の追記のみであるためMCPサーバーの再起動は不要とされている。この判断の根拠は definitions.ts がコアロジックでないことだが、実際には definitions.ts もキャッシュ対象ファイルの一つである。
したがって、バリデーション失敗が継続する場合の診断ステップとして「MCPサーバー再起動の要否を確認する」ことを手順に含めるべきである。
深刻度はMediumと評価する。キャッシュ問題が発生した場合、変更が無効化されたように見え、追記したガイダンスが機能しない状態が継続する。

### T-4: 過度に詳細なガイダンスがテンプレート肥大化を引き起こすリスク

発生経路として、4サブフェーズそれぞれに詳細なガイダンスを追記するとテンプレートの全体サイズが増大し、subagentのコンテキストウィンドウを圧迫するケースが想定される。
現状の definitions.ts ファイルは既に大量のテンプレート文字列を含んでおり、FR-1〜FR-4の追記により各テンプレートに10〜30行程度が加わる見込みである。
コンテキスト肥大化が進むと、subagentが後半の指示を読み飛ばすリスクが上昇し、本来の目的（品質問題の再発防止）と逆行する可能性がある。
緩和策として、NFR-4（変更の最小化原則）に従い追記を問題解決に必要な最小限に限定すること、かつNGパターンとOKパターンの例示はコードフェンス内に集約して散文テキストの冗長化を避けることが有効である。
深刻度はLowと評価する。影響が出た場合も即座に検知は困難だが、テンプレートサイズ増加の定期的な監視で予防できる。

## リスク評価

変更全体のリスクプロファイルとして、今回の変更はプロンプトテキストの追記のみであるためコアロジックへの影響は存在せず、全体的なリスクレベルはLow〜Mediumの範囲に収まる。

T-1（バリデーター検出語句の混入）の評価として、発生確率はMedium（実装時の不注意で発生しうる）、影響度はLow（開発効率への影響のみ）、総合リスクはMediumとする。直接的な緩和策はNFR-3の実装前チェックであり、実装担当者が追記内容を artifact-validator の語句リストと照合することで回避できる。

T-2（既存ガイダンスとの矛盾）の評価として、発生確率はLow（注意深く追記すれば回避容易）、影響度はLow（バリデーションで早期に検出可能）、総合リスクはLowとする。緩和策として「既存説明の延長である」旨を明示することで混乱を防止できる。

T-3（Node.jsキャッシュ問題）の評価として、発生確率はMedium（MCPサーバーが長時間稼働中の場合に発生しうる）、影響度はMedium（変更が無効化されたように見える診断の難しさ）、総合リスクはMediumとする。緩和策として、変更後に意図通り動作しない場合の診断ステップにMCPサーバー再起動を含めることで対処できる。

T-4（テンプレート肥大化）の評価として、発生確率はLow（今回の追記量は限定的）、影響度はLow（即座の影響は小さい）、総合リスクはLowとする。NFR-4の最小化原則を遵守することで抑制できる。

## セキュリティ要件

以下のセキュリティ要件を計画フェーズの仕様（spec.md）に反映すること。

SR-1として、追記するガイダンス文字列は実装前にコードフェンス外にバリデーターが検出する12の語句が存在しないことを確認すること。確認はartifact-validator.tsのFORBIDDEN_PATTERNSリストとの手動照合により実施する。これはT-1の直接的な緩和策である。

SR-2として、追記するガイダンス内でNGパターンを例示する場合は、必ずコードフェンス内に配置し、コードフェンス外には散文形式で同等の説明を行うこと。コードフェンス外での角括弧使用は禁止であり、例示用のファイル名列挙も散文形式で記述すること。これはT-1とT-2の両方に対応する。

SR-3として、追記内容は既存ガイダンスの末尾または関連セクション末尾への純粋な追記とし、既存テキストの削除・変更を行わないこと。git diff で変更箇所が追記のみであることを確認してから実装を完了すること。これはT-2とNFR-4に対応する。

SR-4として、変更後に parallel_verification サブフェーズのsubagentが正常に動作しない場合の診断手順として、MCPサーバーの再起動（ビルド後のサーバー再起動）を診断ステップに含めること。テンプレート変更が即座に反映されることを事前確認できる場合は再起動は不要だが、確認できない場合はキャッシュ問題を疑うこと。これはT-3への対応である。

SR-5として、各テンプレートへの追記行数が30行以内に収まることを目安とし、それを超える場合はNFR-4の最小化原則に照らして内容を見直すこと。コードフェンスを用いたNG/OKパターン例示は1ブロックにまとめて冗長化を避けること。これはT-4への対応である。
