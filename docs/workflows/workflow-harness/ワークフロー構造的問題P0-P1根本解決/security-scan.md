## サマリー

- **目的**: ワークフロープラグインに新規追加された5つのツールファイルのセキュリティレビューを実施
- **対象**: record-feedback.ts、pre-validate.ts、create-subtask.ts、link-tasks.ts、claude-md-parser.ts
- **主要な発見1**: 全体的に適切な入力バリデーションとセッショントークン検証が実装されており、重大なセキュリティリスクは検出されない状態
- **主要な発見2**: ファイルパストラバーサル対策が pre-validate.ts と claude-md-parser.ts で不十分であり、相対パス記号による攻撃が理論的に可能
- **主要な発見3**: claude-md-parser.ts のメモリキャッシュに TTL や最大サイズ制限がなく、DoS リスクが限定的に存在
- **推奨事項**: セッショントークン検証のデフォルト有効化を推進し、本番環境での厳格化を強制する設計への移行が必要

## 脆弱性スキャン結果

### record-feedback.ts のセキュリティ評価
このファイルはユーザーフィードバックをタスク状態に記録するツールです。入力値の検証として必須性チェック（validateRequiredString）、長さチェック（MAX_FEEDBACK_LENGTH=10000文字）、セッショントークン検証（verifySessionToken）が適切に実装されています。フィードバック内容が超過する場合は末尾切り詰めで対処する設計となっており、メモリ枯渇を防いでいます。不正なタスクID指定時は getTaskByIdOrError で適切に検出・拒否されるため、タスク状態の不正アクセスはブロックされます。

### pre-validate.ts のセキュリティ評価
このファイルは検証対象ファイルのパス指定を受け付けるため、パストラバーサル対策が重要です。現在の実装では targetPhase（フェーズ名）と filePath（ファイルパス）を validateRequiredString でトリム検証していますが、path.normalize() や path.resolve() による正規化処理は実装されていません。理論的には相対パス記号（../../等）を含むパスが指定可能な状態です。ただし fs.existsSync と fs.readFileSync の実行権限がプロセスユーザーの権限に制限されているため、実質的な被害は環境依存です。

### create-subtask.ts のセキュリティ評価
このファイルは新規タスク作成を行う機能であり、parentTaskId と subtaskName の入力検証が重要です。parentTaskId は validateRequiredString で非空チェック、subtaskName も同様に検証されています。taskSize パラメータは列挙値（small/medium/large）に対して厳密な検証が行われ、不正な値は 'large' にフォールバックされるため安全です。セッショントークン検証も親タスクに対して実施されており、タスク作成権限が守られています。新規タスク ID の生成は stateManager.createTask 内部で実装されており、外部からの ID 指定は不可能です。

### link-tasks.ts のセキュリティ評価
このファイルはタスク間の親子関係を設定し、循環参照検出ロジックが実装されています。MAX_TASK_DEPTH=5 の制限により、無限ループを防いでいます。detectCircularReference 関数では visited セット和 queue を使用した BFS アルゴリズムで効率的に検出し、極端に大規模なタスク階層でも 5 段階までの走査制限により計算量が制御されています。自己参照チェック（同一タスクへの親子関係設定禁止）も実装されており、基本的なロジックエラーは回避されています。

### claude-md-parser.ts のセキュリティ評価
このファイルはファイルシステムから CLAUDE.md を読み込みパースするため、パストラバーサル脆弱性が重要です。claudeMdPath パラメータに対して string 型チェックのみであり、path.normalize() 処理が未実装です。fs.existsSync と fs.readFileSync 呼び出し前にパス正規化がないため、../../ パターンを含むパスが指定可能です。ただし実行権限がプロセスユーザーに限定されているため、実質的被害は読み取り権限のあるファイルシステム領域に制限されます。モジュールレベルのメモリキャッシュ（parseCache）は鍵が `${claudeMdPath}::${phaseName}` の形式であり、悪意のある大量パスによるメモリ枯渇が理論的に可能です。

## 検出された問題

### 問題1: パストラバーサル対策の不足（中程度リスク）
pre-validate.ts と claude-md-parser.ts で入力パスの正規化が行われていません。validateRequiredString は空文字列チェックとトリムのみで、相対パス記号の除去処理を含んでいません。改善策として、各ファイル読み込み前に path.normalize() または path.resolve() を適用し、許可されたディレクトリ配下であることを確認するホワイトリスト照合が推奨されます。

### 問題2: 記録フィードバックの長さ制限切り詰めによる情報損失
record-feedback.ts では userIntent が MAX_FEEDBACK_LENGTH を超える場合、末尾を slice で切り詰めています。この設計により長いフィードバックが意図と異なる内容に変換される可能性があります。セキュリティ上の直接的な脆弱性ではありませんが、ユーザーの意図が改変される設計は避けるべきです。代替案として超過時にはエラーを返し、ユーザーに入力内容の分割を促すことが望ましいです。

### 問題3: 循環参照検出の深度制限が浅い
link-tasks.ts の MAX_TASK_DEPTH=5 は、大規模なタスク階層を持つシステムでは不十分な可能性があります。ただし検出エラーが返されるため DoS には至りませんが、正当な親子関係が設定不可になるリスクがあります。タスク階層の実装要件に応じて深度制限値の見直しが必要です。

### 問題4: メモリキャッシュの無限増殖対策が未実装
claude-md-parser.ts のモジュールレベルキャッシュ（parseCache）に TTL や最大サイズ制限がありません。理論的には異なるパス・フェーズの組み合わせを大量入力することでメモリを枯渇させる攻撃が可能です。改善策として LRU キャッシュライブラリの導入、または定期的なクリア機構（clearParseCache が存在）の自動実行タイマー実装が推奨されます。

### 問題5: セッショントークン検証の環境変数無効化が本番環境向け不適切
helpers.ts の verifySessionToken は SESSION_TOKEN_REQUIRED=false で検証をバイパスする設計です。開発環境では便利ですが、本番環境では必ず true に設定する必須事項をドキュメント化してください。無効化を検出した際のログ出力（auditLogger）が実装されており、運用監視の手掛かりはありますが、デフォルト値が無効化対応である点は改善が望まれます。
