# docs_updateフェーズ成果物 - 今回のワークフロー実行で発生した問題の根本原因調査と修正

## サマリー

本フェーズでは、research・requirements・specフェーズで特定した3件の構造的問題（FR-6・FR-7・FR-8）の根本原因に基づいて、永続ドキュメントの更新対象を評価し、更新の実施可否を判定した。

評価結果として、本タスクで特定した3件の問題に対する修正内容は、いずれも`workflow-plugin/mcp-server/src/phases/definitions.ts`ファイルの変更のみが必要であり、`docs/spec/`・`docs/architecture/`・`docs/operations/`配下の永続仕様書への反映は不要であると判定した。

理由は以下の通りである。本タスクが対象とする3件の修正（FR-6・FR-7・FR-8）は「MCPサーバーの内部テンプレート文字列の改善」であり、ユーザー向けの仕様書・設計書・運用手順書ではない。これらの修正はCLAUDE.md（ワークフロー強制ルールドキュメント）で既に定義されている内容をssubagentTemplateに正確に反映させるためのコード修正である。したがって、CLAUDE.md以外の永続ドキュメント（docs/spec配下等）に更新する実質的な新情報が存在しない。

結論として、本フェーズでは「更新対象となる永続ドキュメントが存在しない」と判定し、ドキュメント更新を行わずにフェーズを完了する。

## 更新対象ドキュメントの評価

### 対象外評価: docs/spec/ 配下（プロダクト仕様書）

research・requirements・specで特定した3件の問題に対応する修正は、プロダクト機能の変更ではなくMCPサーバーの内部ロジック改善である。

FR-6はtestingフェーズのsubagentTemplateに`workflow_capture_baseline`の呼び出し指示を追記する修正であり、プロダクト仕様書（docs/spec/features/、docs/spec/api/等）に影響を与えない。

FR-7はtest_implフェーズのsubagentTemplateにテストファイル出力先を明示する修正であり、テスト実装方針の改善であるがプロダクト機能の変更ではない。

FR-8はdocs_updateフェーズのsubagentTemplateに更新禁止ファイルの明示を追記する修正であり、プロダクト設計の変更ではない。

したがって、docs/spec/配下の機能仕様書・画面仕様書・API仕様書・コンポーネント仕様書等の永続ドキュメントの更新は不要である。

### 対象外評価: docs/architecture/ 配下（システム設計書）

本タスクで対象とするdefinitions.tsの修正は、ワークフロープラグインの内部テンプレート処理の改善であり、システム全体のアーキテクチャ・モジュール設計・認証設計等には影響を与えない。

FR-6・FR-7・FR-8のいずれもが、docs/architecture/overview.md・docs/architecture/modules/等のシステム設計書に記載すべき実装変更を含まない。

したがって、システムアーキテクチャドキュメントの更新は不要である。

### 対象外評価: docs/operations/ 配下（運用ドキュメント）

本修正はMCPサーバーの内部テンプレート改善であり、デプロイ手順・環境定義・監視設計・運用手順書等に影響を与えない。

デプロイフロー・環境構成・運用監視項目等の変更が存在しないため、docs/operations/配下の永続ドキュメントの更新は不要である。

### 対象外評価: CHANGELOG.md（変更履歴）

本タスクの修正は「MCPサーバーの内部テンプレート改善」であり、プロダクト向けの変更ログに記載する内容ではない。

CHANGELOG.mdは一般的にユーザー向けの機能変更・バグ修正・破壊的変更を記録するドキュメントであるが、本修正はsubagentへの指示内容の明確化であり、ユーザーに直接影響を与えるプロダクト変更ではない。

したがって、CHANGELOG.mdの更新は不要である。

### 対象外評価: README.md（プロジェクト概要）

本修正の内容がREADMEに記載すべき新しい情報を含まないため、プロジェクト概要ドキュメントの更新は不要である。

## 実装内容と永続ドキュメントの対応関係

### 実装対象の確認

spec.mdで定義された3件の修正（FR-6・FR-7・FR-8）の実装内容は、すべて`definitions.ts`の文字列修正に限定される。

FR-6: `testing`フェーズの`subagentTemplate`に`workflow_capture_baseline`の呼び出し手順を追記（行878付近）。
FR-7: `test_impl`フェーズの`subagentTemplate`に出力先・登録手順・禁止ツールを追記（行782付近）。
FR-8: `docs_update`フェーズの`subagentTemplate`に更新対象・禁止ファイル・取り扱いを追記（行953付近）。

### 永続ドキュメント対応の不要理由

上記3件の修正がいずれも「MCPサーバーのサブエージェント向けテンプレート文字列の改善」であり、プロダクト機能・システム設計・運用手順の変更を含まない。

CLAUDE.mdには既に「testingフェーズはworkflow_capture_baselineを呼ぶこと」「test_implフェーズはテストファイルを作成すること」「docs_updateフェーズは指定ドキュメントのみを更新すること」が記載されている。

本修正はこれらのCLAUDE.mdに記載されたルールをsubagentTemplateに反映させるためのコード修正であり、新しいビジネスルールや設計判断の追加ではない。

## 更新対象が存在しない場合の根拠

### 根拠1: 修正の性質がコード内部改善である

本タスクで対象とする3件の修正（FR-6・FR-7・FR-8）は「MCPサーバーの内部実装の改善」であり、プロダクト向けの仕様変更ではない。

永続ドキュメント（docs/spec/、docs/architecture/、docs/operations/等）は一般的に「プロダクト機能の仕様」「システムの外部インターフェース」「運用手順」等を記録するものであり、MCPサーバーのテンプレート改善のような「開発ツールの内部ロジック変更」は記録対象外である。

### 根拠2: CLAUDE.mdが既に設計を記載している

specで定義した修正内容は、すべてCLAUDE.mdの既存ルール（ルール20、AIへの厳命の項目等）から導出されたものである。

新しい設計判断が存在しないため、docs/architecture/decisions/（ADR）に記録すべき新規の決定事項が存在しない。

### 根拠3: プロダクト変更がない

FR-6・FR-7・FR-8のいずれもがプロダクトのAPI・画面・ビジネスロジックを変更していない。

test_implの修正は「テストファイルの作成方法を明確化すること」であり、プロダクトの動作変更ではなく、テスト開発プロセスの改善である。

therefore本修正はプロダクト仕様書（docs/spec/features/等）への反映対象とならない。

## 修正内容の適用タイミング

本タスクで研究・設計・仕様定義した3件の修正は、implementation・refactoring・testing等の次フェーズで以下のタイミングで適用される。

修正の適用時期はimplementationフェーズであり、definitions.tsの行782・行878・行953への具体的なコード変更はこのフェーズで実施される。

修正適用後は`npm run build`とMCPサーバー再起動が必須である。Node.jsのモジュールキャッシュにより、dist/*.jsを修正してもプロセス再起動なしに変更が反映されない点に注意する。

修正後の検証はtesting・regression_test・parallel_verificationの各フェーズで実施され、修正がサブエージェントの動作に正の影響を与えることを確認する。

## 結論

本フェーズの評価結果、永続ドキュメント（docs/spec/、docs/architecture/、docs/operations/、CHANGELOG.md、README.md等）に反映すべき実装変更は存在しない。

理由は以下の通りである。

最初の理由として、本タスクの修正がMCPサーバーの内部テンプレート改善であり、プロダクト向けの仕様変更ではないことである。

次の理由として、修正の内容すべてがCLAUDE.mdで既に定義されているルールの「コード レベルでの正確な反映」であり、新しい設計判断や仕様追加を含まないことである。

最後の理由として、FR-6・FR-7・FR-8のいずれもが「開発プロセスの改善」（ベースライン記録の明示化、テストファイル出力先の明確化、更新禁止ファイルの明示化）であり、プロダクト機能の変更ではないことである。

したがって、本フェーズではドキュメント更新を行わずにフェーズを完了する。

修正の実装はimplementationフェーズで実施され、修正後のテストと検証はtesting・regression_test・parallel_verificationフェーズで実施される予定である。
