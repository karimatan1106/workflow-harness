# UI設計 - 今回のワークフロー実行で発生した問題の根本原因調査と修正

## サマリー

本ドキュメントは、FR-6・FR-7・FR-8の3件の修正に対応する`definitions.ts`のsubagentTemplate修正後のインターフェース設計を定義する。

このタスクの対象はMCPサーバーのCLI/APIインターフェースであり、視覚的なUIは存在しない。設計対象は以下の3つのサブエージェントが使用するMCPツール呼び出しパターンである。

第1の対象はtestingフェーズのサブエージェントが使用する`workflow_capture_baseline`ツールのインターフェースであり、テスト実行結果をベースラインとして記録するためのパラメータ仕様を定義する。

第2の対象はtest_implフェーズのサブエージェントが使用する`workflow_record_test`ツールのインターフェースであり、テストファイルパスをワークフロー管理システムに登録するためのパラメータ仕様を定義する。

第3の対象はdocs_updateフェーズのサブエージェントが更新対象とするドキュメント構造の設計であり、更新許可リストと禁止リストを定義することでサブエージェントの誤操作を防止する。

次フェーズ（test_design）では本ドキュメントに基づき、各MCPツール呼び出しの正常系・異常系テストケースを設計する。

## CLIインターフェース設計

### workflow_capture_baseline 呼び出し形式

testingフェーズのサブエージェントがテスト実行後に呼び出すMCPツール。呼び出しはワークフロー制御ツール（workflow_next）より前に行う必要がある。

パラメータ仕様は以下のとおりである。

- **taskId**: 文字列型。Orchestratorがサブエージェント起動プロンプトで渡すタスクID文字列を使用する。
- **totalTests**: 整数型。テストコマンド（npm test, vitest等）が出力する集計行から読み取った総テスト数。
- **passedTests**: 整数型。集計行から読み取った成功テスト数。失敗が0件の場合は totalTests と同値になる。
- **failedTests**: 文字列配列型。失敗したテスト名の配列。全テスト成功時は空配列 `[]` を渡す。

vitest出力の集計行からのパラメータ抽出方法は次のとおりである。出力例 `Tests: 12 passed (12)` の場合、totalTests=12、passedTests=12、failedTests=空配列となる。出力例 `Tests: 2 failed | 10 passed (12)` の場合、totalTests=12、passedTests=10、failedTests=2件の失敗テスト名配列となる。

### workflow_record_test 呼び出し形式

test_implフェーズのサブエージェントがテストファイル作成後に呼び出すMCPツール。テストファイルを1つ作成するごとに1回呼び出す必要がある。

パラメータ仕様は以下のとおりである。

- **taskId**: 文字列型。Orchestratorがサブエージェント起動プロンプトで渡すタスクID文字列を使用する。
- **testFile**: 文字列型。作成したテストファイルのパス。リポジトリルートからの相対パスまたは絶対パスを指定する。

このプロジェクトのテストファイル配置規則に従い、testFileパラメータに渡すパスは `workflow-plugin/mcp-server/src/` 配下のいずれかの `__tests__` ディレクトリを指すものに限定される。ルートディレクトリ直下や `docs/workflows/` 配下のパスは渡してはならない。

## エラーメッセージ設計

### workflow_capture_baseline 呼び出し時のエラーパターン

エラーパターン1: パラメータ型不一致（totalTestsに文字列を渡した場合）

MCPサーバーは `totalTests must be a number` というエラーメッセージを返す。対処方法は、テスト出力文字列からparseInt()等の変換処理を用いて整数値を抽出し、再度呼び出すことである。

エラーパターン2: フェーズ不一致（testing以外のフェーズで呼び出した場合）

MCPサーバーは `workflow_capture_baseline can only be called in testing phase` というエラーメッセージを返す。対処方法は、現在のフェーズを workflow_status で確認し、testingフェーズで再度呼び出すことである。

エラーパターン3: taskIdが見つからない場合

MCPサーバーは `Task not found: {taskId}` というエラーメッセージを返す。対処方法は、Orchestratorから受け取ったタスクIDの文字列が正確であることを確認し、再度呼び出すことである。

### regression_testフェーズでのベースライン未設定エラー

ベースラインが未設定の状態でregression_testフェーズへ遷移しようとした場合、MCPサーバーは以下の形式のエラーメッセージを返す。

エラー内容: `ベースラインが記録されていません。testingフェーズでworkflow_capture_baselineを呼び出してベースラインを記録してください。` というメッセージが workflow_next のレスポンスに含まれる。

このエラーへの対処は2通りある。推奨対処はtestingフェーズに差し戻し（workflow_back）、workflow_capture_baselineを呼び出した後に再度遷移することである。代替対処はforceTransition: trueを指定してフェーズ遷移を強制することだが、この方法ではリグレッション比較が正確に行えないため通常は使用しない。

## APIレスポンス設計

### workflow_capture_baseline の成功レスポンス形式

正常に呼び出しが完了した場合、MCPサーバーは以下の構造を含むレスポンスを返す。

成功レスポンスには次の情報が含まれる。

- **baseline.totalTests**: 記録した総テスト数（整数値）
- **baseline.passedTests**: 記録した成功テスト数（整数値）
- **baseline.failedTests**: 記録した失敗テスト名の配列（文字列配列）
- **baseline.capturedAt**: ベースライン記録時刻（ISO 8601形式の文字列）

サブエージェントはこのレスポンス内容をテスト結果レポート（`docs/testing/reports/`配下）に記録することが推奨される。

失敗レスポンスの形式は以下のとおりである。

- **error**: エラーメッセージの文字列
- **code**: エラーコードの文字列（例: "INVALID_PARAMETER", "PHASE_MISMATCH", "TASK_NOT_FOUND"）

### workflow_record_test の成功レスポンス形式

正常に呼び出しが完了した場合、MCPサーバーは以下の構造を含むレスポンスを返す。

成功レスポンスには次の情報が含まれる。

- **testFiles**: 現在のタスクに登録されているテストファイルパスの配列（今回追加したファイルを含む）
- **addedFile**: 今回追加されたテストファイルのパス（文字列）
- **taskId**: 登録先のタスクID（文字列）

複数のテストファイルを作成した場合は、作成のたびにこのツールを呼び出す。レスポンスの testFiles 配列が増加していることを確認することで、正常に登録されたことを確認できる。

失敗レスポンスの形式は workflow_capture_baseline と同様の構造（error フィールドと code フィールドを持つオブジェクト）であり、エラー内容と対処方法を確認することができる。

## 設定ファイル設計

### docs_updateフェーズの更新対象ドキュメント構造

docs_updateフェーズのサブエージェントが更新を許可されるドキュメントの構造を以下のとおり定義する。

**更新許可パスリスト（永続ドキュメント）**

第1カテゴリは `docs/spec/` 配下の全ファイルであり、機能仕様書・画面仕様書・API仕様書・コンポーネント仕様書・ステートマシン図・フローチャート図が対象である。このフェーズの主要な更新対象であり、実装内容の変更が仕様書に反映されていない場合は積極的に更新する。

第2カテゴリは `docs/architecture/` 配下の全ファイルであり、システム概要設計書・モジュール詳細設計書・ADR（アーキテクチャ決定記録）が対象である。実装によってアーキテクチャに影響を与える変更があった場合に更新する。

第3カテゴリは `docs/operations/` 配下の全ファイルであり、デプロイ手順書・環境定義・監視設計書・運用手順書が対象である。

第4カテゴリはプロジェクトルートの `CHANGELOG.md` であり、今回のタスクで実施した変更内容を変更履歴として記録するために更新する。

第5カテゴリはプロジェクトルートの `README.md` であり、今回の変更によってユーザー向けの情報に変化がある場合のみ更新する。

**更新禁止パスリスト**

禁止カテゴリ1は `docs/workflows/{taskName}/` 配下の全ファイルであり、これらは一時的な作業フォルダとして `.gitignore` に登録されているため、docs_updateフェーズで編集する意味を持たない。

禁止カテゴリ2は `MEMORY.md`（プロジェクトメモリファイル）であり、Claude Desktopのプロジェクトメモリ機能が管理するシステムファイルであるため直接編集を禁止する。このファイルを誤って更新するとシステム全体のメモリ設定に影響する可能性がある。

禁止カテゴリ3は `.claude/state/` 配下の全ファイルであり、ワークフロー状態管理ファイル（workflow-state.json等）はHMAC整合性チェックが設定されているため直接編集するとワークフロー全体が動作不能になる。

### 更新対象が存在しない場合の取り扱い

今回のタスク（FR-6・FR-7・FR-8: subagentTemplateの文字列追記）は、APIのエンドポイント・画面レイアウト・コンポーネント構造に変化を与えない内部的な修正である。このため、docs_updateフェーズで `docs/spec/` や `docs/architecture/` を更新する必要が生じる可能性は低い。

サブエージェントが永続ドキュメントへの反映対象を見つけられない場合は、「今回の変更はsubagentTemplateのみの修正であり、永続ドキュメントに反映すべき変更は存在しない」という内容の報告をフェーズ完了の記録として残し、ドキュメント更新なしでフェーズを完了することが正しい動作である。

CHANGELOG.mdに関しては、定義ファイルの修正という変更内容を記録するため、バージョン更新有無に関わらず変更履歴エントリを追記することが推奨される。
