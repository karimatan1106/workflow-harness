# 脅威モデル - 今回のワークフロー実行で発生した問題の根本原因調査と修正

## サマリー

本脅威モデルは、`definitions.ts`の3フェーズ（testing・test_impl・docs_update）のsubagentTemplateに追記するFR-6・FR-7・FR-8の変更に伴うリスクを分析したものである。

変更対象はテンプレート文字列のみであり、フェーズロジック・HMAC整合性・状態管理に関わるコードは変更しない。この制約によって脅威の範囲は主にサブエージェントの動作変化と既存テストへの影響に限定される。

主要な決定事項として、3件の変更はいずれもリスクレベルが低から中程度であり、既存テストのリグレッションリスクは軽微と判断した。追記内容が指示の明確化に留まるため、フレームワーク全体への波及リスクは最小限に抑えられている。

次フェーズ（planning）では本分析で特定したリスク項目を踏まえて、各テンプレートの具体的な修正内容と後方互換性確保の方針をspec.mdに記述すること。

識別された脅威は合計11件であり、高リスク2件（脅威7・8）・中リスク5件（脅威1・4・5・9・10）・低リスク4件（脅威2・3・6・11）に分類されている。


## 脅威分析

### FR-6: testingテンプレートへのworkflow_capture_baseline追加

FR-6は、testingフェーズのsubagentTemplateに`workflow_capture_baseline`の呼び出し指示を追加する変更である。現在のtestingテンプレートには`workflow_record_test_result`の呼び出し手順が詳細に記載されているが、`workflow_capture_baseline`への言及が全く存在しない。この欠落が根本原因となり、regression_testフェーズへの遷移時にベースライン未設定エラーが毎回発生している。

**脅威1: パラメータ誤りによるベースライン記録失敗**

`workflow_capture_baseline`は4つの数値パラメータ（taskId・totalTests・passedTests・failedTests）を要求する。テスト実行コマンドの出力から数値を正確に抽出できない場合、サブエージェントが誤った値を渡す可能性がある。特にカスタムテストランナーを使用している場合、vitest/jestの標準出力形式と異なる出力から数値を誤解析するリスクがある。誤ったベースラインが記録された場合、regression_testフェーズの比較結果が不正確になる。

リスクレベルは中程度と判定する。誤ったベースラインが記録されてもワークフロー全体は継続できるが、regression_testでの差分比較が不正確になることで本来検出すべきリグレッションを見逃す可能性がある。

対策として、テンプレートに「テストコマンドの集計行を正確に読み取り、数値型（文字列ではなく整数値）でパラメータを渡すこと」という明示的な指示を含めることが有効である。

**脅威2: ベースライン記録タイミングの混乱**

`workflow_capture_baseline`と`workflow_record_test_result`の両方を呼び出す順序について、テンプレートに明示的な順序指定がない場合、サブエージェントが逆順で呼び出したり、片方のみ呼び出す可能性がある。

リスクレベルは低程度と判定する。呼び出し順序はワークフロー機能の正確性に影響するが、どちらを先に呼んでも技術的にはエラーが発生しないため、サービスの継続稼働には影響しない。

対策として、テンプレートに「テスト実行後、まずworkflow_capture_baselineでベースラインを記録し、次にworkflow_record_test_resultで結果を記録する」という順序を明記することが有効である。

**脅威3: 既存ベースライン済みタスクへの影響**

すでにベースラインが記録されているタスクに対して、testingフェーズのサブエージェントが再度`workflow_capture_baseline`を呼び出す場合、既存のベースラインデータが上書きされるリスクがある。上書きが発生した場合、以前のテスト実行状態との比較ができなくなる。

リスクレベルは低程度と判定する。同一タスクのtestingフェーズが複数回実行されるケースは通常の運用ではまれであり、テンプレートの追記内容はこのシナリオを想定したものではない。

### FR-7: test_implテンプレートへの出力先追加

FR-7は、test_implフェーズのsubagentTemplateに出力ファイルパスの指定と`workflow_record_test`登録手順を追加する変更である。現在のテンプレートは「テストコードを実装してください（TDD Red）。」の1行のみであり、他フェーズと比較して極端に簡素な内容となっている。

**脅威4: テストファイルパスの不正な配置**

出力先ディレクトリが明示されていない場合、サブエージェントがプロジェクト標準のディレクトリ構造に従わないパスにテストファイルを生成するリスクがある。例えばルートディレクトリ直下やdocs/workflows配下にテストファイルを配置してしまう可能性がある。

リスクレベルは中程度と判定する。テストファイルの誤配置は後続フェーズ（implementation・testing）でのファイル参照失敗を引き起こし、ワークフロー全体の継続に影響する可能性がある。

対策として、テンプレートに「workflow-plugin/mcp-server/src/tests/が標準テストディレクトリであり、テストファイルはこのディレクトリ配下に配置すること」という具体的なパス指示を含めることが有効である。

**脅威5: workflow_record_test未呼び出しによるtestingフェーズのスキップリスク**

`workflow_record_test`によるテストファイルの登録が省略された場合、testingフェーズでどのテストファイルを実行するかが不明確になる。特にtesting・regression_testフェーズでのテスト実行対象が適切に特定できない状況が発生しうる。

リスクレベルは中程度と判定する。テスト登録の欠落はTDDサイクル全体の品質担保に影響し、テストフェーズでの実行漏れを引き起こす可能性がある。

対策として、テンプレートに「作成したテストファイルのパスをworkflow_record_testで登録することが必須手順である」という明示的な指示を追加する。

**脅威6: 手動確認手順のみの場合のフェーズスタック**

test-design.mdが手動確認手順のみを記述していてコードレベルのテストケースが存在しない場合、サブエージェントが「何も作成すべきものがない」と判断してフェーズを完了できずにスタックするリスクがある。

リスクレベルは低程度と判定する。この状況はフェーズを中断させる可能性があるが、プロジェクトの大半はコードレベルのテストを含むため発生頻度は低い。

---

### FR-8: docs_updateテンプレートへの更新対象範囲明示

FR-8は、docs_updateフェーズのsubagentTemplateに更新対象ドキュメントの範囲と更新対象外ファイルを明示する変更である。現在のテンプレートは「ドキュメントを更新してください。」の1行のみであり、曖昧な指示がMEMORY.mdへの無許可追記等の問題を引き起こしている。

**脅威7: MEMORY.mdへの意図しない情報書き込み**

明示的な禁止指示がない場合、docs_updateサブエージェントがMEMORY.mdをドキュメント更新対象として誤認識し、タスク固有の知見をMEMORY.mdに書き込む可能性がある。MEMORY.mdへの誤った知見の追記はシステム全体の動作に影響し、将来のタスク実行に悪影響を与える。

リスクレベルは高程度と判定する。MEMORY.mdへの誤書き込みはシステムの動作変更を引き起こし、一度書き込まれた内容は後続の全タスクに影響するため、影響範囲が広い。

対策として、テンプレートにMEMORY.mdが更新対象外であることを明示的に記載し、「このファイルはシステムが管理するファイルであり、docs_updateフェーズでの編集対象外である」という警告を含める。

**脅威8: .claude/state/配下のワークフロー状態ファイルの誤編集**

docs_updateサブエージェントが`.claude/state/`配下のファイルをドキュメントとして誤認識し、編集した場合、HMAC整合性チェックが失敗してワークフロー全体が動作不能になるリスクがある。

リスクレベルは高程度と判定する。HMAC整合性の破壊は即時にワークフローのすべての操作をブロックし、回復に手動介入が必要になる深刻な障害を引き起こす。

対策として、テンプレートに`.claude/state/`配下のファイルが編集禁止であることを明示し、「これらのファイルはHMAC整合性チェックが設定されており、直接編集するとワークフローの動作が不正になる」という警告を含める。

**脅威9: docs/workflows/配下の一時成果物の誤更新**

docs_updateサブエージェントが`docs/workflows/{taskName}/`配下のファイルをドキュメント更新対象として扱い、作業成果物（research.md・requirements.md等）を上書きするリスクがある。これらのファイルはGitignore対象であり、後続フェーズでの参照が壊れる可能性がある。

リスクレベルは中程度と判定する。作業成果物の上書きは後続フェーズのコンテキスト引き継ぎを破壊するが、影響は現在のタスクに限定される。

### 変更全体に共通する脅威

**脅威10: definitions.tsの変更によるMCPサーバーキャッシュとの不整合**

`definitions.ts`はMCPサーバー起動時にNode.jsのモジュールキャッシュに読み込まれる。変更後のビルド（`npm run build`）を実施してもMCPサーバーを再起動しない場合、変更前のテンプレートが引き続き使用される。この状況では修正の効果が全く現れないため、問題が解決されていないと誤判断するリスクがある。

リスクレベルは中程度と判定する。動作への直接的な害はないが、修正の検証が不可能になりデバッグに無駄な時間を消費する。対策として、実装完了後にMCPサーバーの再起動を必須手順としてspec.mdの「実装後作業」セクションに明記すること。

**脅威11: テンプレート追記によるartifact-validatorの誤検出リスク**

追記するテンプレート文字列に禁止語（バリデーター検出パターン）が含まれる場合、subagentTemplateを基にした成果物の生成時にバリデーション失敗が発生するリスクがある。特にテンプレート内の説明文や警告文に日本語の禁止語グループの語句が混入することを防ぐ必要がある。

リスクレベルは低程度と判定する。プロンプトの執筆時に注意を払えば回避可能であり、発生してもテンプレートの修正で対処できる。


## リスク対策

### FR-6（testingフェーズ）への対策

FR-6の実装では、`workflow_capture_baseline`の呼び出しパラメータ（taskId・totalTests・passedTests・failedTests）の説明を具体的な数値型指定とともにテンプレートに追記すること。呼び出し順序については「workflow_capture_baseline → workflow_record_test_result」という明確な順序指定を記載し、順序の逆転を防止する。

既存のベースライン記録済みタスクへの影響については、テンプレートに「ベースラインが未記録の場合にのみ呼び出すこと」という条件分岐の指示を追加するか、または`workflow_capture_baseline`のAPI側で冪等性が確保されているかを確認してから追記内容を決定する。

既存の`workflow_record_test_result`の呼び出し手順（100文字以上の生標準出力の貼り付け要件・重複送信のブロック等）は変更しないことで、NFR-3の後方互換性要件を満たす。

### FR-7（test_implフェーズ）への対策

FR-7の実装では、このプロジェクトの標準テストディレクトリ（`workflow-plugin/mcp-server/src/tests/`）を出力先として明記し、テストファイルがルートディレクトリや不適切なパスに配置されることを防ぐ。

`workflow_record_test`の呼び出し指示は「テストファイル作成後に必ず呼び出すこと」という強制的な表現で追記し、省略が選択肢にならないよう明示する。手動確認手順のみの場合の取り扱いについては、「コードレベルのテストが不要と判断した根拠を記録すること」という代替手順を明記し、フェーズスタックを防止する。

既存のTDD Red フェーズの概念説明（失敗状態のテストを作成することがTDD原則である旨）は維持し、追記内容が既存の指示と矛盾しないよう整合性を確認する。

### FR-8（docs_updateフェーズ）への対策

FR-8の実装では、更新対象ドキュメントの範囲（docs/spec/・docs/architecture/・docs/operations/・CHANGELOG.md・README.md）と更新対象外ファイル（MEMORY.md・.claude/state/配下・docs/workflows/{taskName}/配下）の両方を明示する。

MEMORY.mdと`.claude/state/`配下についてはそれぞれ異なる理由で禁止であることを説明し、前者は「システムが管理するファイルであり直接編集禁止」、後者は「HMAC整合性に関わるため直接編集禁止」という具体的な根拠とともに記載する。

更新対象となる永続ドキュメントが存在しない場合の行動指針として、「更新対象が存在しない場合はその旨を報告してフェーズを完了すること」という明示的な指示を含め、サブエージェントが不必要なファイルを更新しようとする動機を除去する。

### 変更全体への横断対策

`definitions.ts`変更後のMCPサーバー再起動手順をspec.mdの「実装後作業」セクションに必須手順として記載する。具体的には「npm run buildでトランスパイルを実行し、MCPサーバーを再起動して変更を反映させること」という4ステップの手順を含める。

テンプレート追記内容にバリデーター検出パターンの語句が含まれていないことをplanning・implementation両フェーズでレビューし、追記テキストが成果物バリデーションエラーの原因とならないことを確認する。既存のテストスイート（definitions.test.ts・buildPrompt.test.ts）が全件パスすることを実装後のビルドチェックで確認し、リグレッションが発生していないことを検証する。
