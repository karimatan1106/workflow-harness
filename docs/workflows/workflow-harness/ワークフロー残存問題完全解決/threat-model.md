# 脅威モデル - ワークフロー残存問題完全解決

## サマリー

4つの残存問題（監査ログ記録、スコープ検証厳格化、設計検証キャッシュ、状態ファイルHMAC署名）に対するセキュリティリスク分析。

- REQ-1: バイパス環境変数を監査ログで追跡可能にし、イベント検出を実現
- REQ-2: スコープ検証をデフォルト厳格化し、管理外変更を防止
- REQ-3: ファイルキャッシュによるI/O削減（セキュリティより性能最適化）
- REQ-4: HMAC署名によるファイル改ざん検出とfail-closed設計

## 脅威

以下ではSTRIDEモデルを用いて各REQの脅威を統合的に分析する。

### REQ-1: 環境変数バイパスの監査ログ記録（脅威軽減度: HIGH）

HMAC_STRICT=false、SESSION_TOKEN_REQUIRED=false、SCOPE_STRICT=falseの環境変数バイパスが使用された際に監査ログに記録されないことが主な脅威である。

| STRIDE | REQ-1の脅威内容 | REQ-1の対策 |
|--------|----------------|-------------|
| S:詐欺 | 意図的なバイパス使用を隠蔽し、HMAC_STRICT=false で署名検証をスキップ | auditLogger.log()で全バイパス使用をJSONL形式で記録 |
| T:改ざん | バイパス有効中に状態ファイルを改ざんし、レビュー対象外の変更が混入 | audit-log.jsonlに時系列記録を残して事後監査を可能にする |
| R:否認 | バイパスを使った覚えがないと主張される可能性 | 本人認証はgitコミット情報でカバー、audit-logで行為を立証 |
| I:情報開示 | 監査ログに環境変数値そのものが記録される | ログに記録する内容はevent名とvariable名に限定し、値は記録しない |
| D:否定 | audit-log.jsonlの削除により証跡が消える | REQ-4で署名付きファイル管理を導入して改ざん検出可能にする |
| E:昇格 | 通常ユーザーが環境変数を変更して検証をスキップ | 環境変数オーバーライドはCI/CD環境のみに限定すべき（運用で対応） |

監査ログにより事後監査が可能になる。ただし改ざん検出はREQ-4に委譲する。

### REQ-2: スコープ事後検証のデフォルト厳格化（脅威軽減度: MEDIUM-HIGH）

スコープ事後検証がデフォルトで警告モードであることにより、スコープ外変更が検出されても素通りする脅威がある。

| STRIDE | REQ-2の脅威内容 | REQ-2の対策 |
|--------|-----------------|-------------|
| S:詐欺 | スコープ外変更を意図的に実行し、設計書にない機能を実装 | SCOPE_STRICT=falseで明示的に環境変数を設定しない限り厳格モードが有効 |
| T:改ざん | 設計対象外のファイルを編集してテスト段階でビジネスロジックを変更 | スコープ検証が厳格化されフェーズ外編集がfail-closedでブロック |
| R:否認 | スコープ内の変更だと主張される | 設計書との差分をreportして監査可能にする |
| I:情報開示 | スコープ変更理由が明示されない | audit-logにSCOPE_STRICT=false使用時の記録を残す |
| D:否定 | スコープ設定を事後に変更できず仕様変更に対応不可 | workflow resetで仕様フェーズに戻る仕組みが存在する |
| E:昇格 | 一般ユーザーが簡単に警告モードに変更可能 | SCOPE_STRICT=false使用時も監査ログに記録して使用痕跡を残す |

デフォルト厳格化は防御的設計だが環境変数でバイパス可能なため人的運用に依存する部分がある。

### REQ-3: 設計検証のファイルキャッシュ実装（脅威軽減度: LOW）

ファイルキャッシュは主に性能最適化が目的であり、セキュリティ脅威は限定的である。

| STRIDE | REQ-3の脅威内容 | REQ-3の対策 |
|---------|----------------|-------------|
| S:詐欺 | キャッシュされた古い設計ファイルを使用して検証をすり抜ける | バリデーション完了後キャッシュをクリアしてライフサイクルを管理 |
| T:改ざん | キャッシュ内容をメモリで改ざんして検証を回避 | cleanedMarkdownはキャッシュではなく毎回生成し変更検出を保証 |
| R:否認 | 古いファイルを読んでいなかったと主張 | キャッシュは検証ロジック内部の最適化であり外部動作に影響なし |
| I:情報開示 | ファイル内容がメモリに蓄積される | スコープは性能最適化であり、セキュリティ側面は既存のIAMに依存 |
| D:否定 | キャッシュの削除により何のファイルが検証されたか不明 | キャッシュはメモリ上のみ、ファイル検証ログはREQ-4に委譲 |
| E:昇格 | メモリキャッシュにアクセスして改ざん | Node.jsプロセス分離による保護で対応 |

性能最適化が主目的のためキャッシュライフサイクルが限定的で脅威は軽微である。

### REQ-4: spec-first-guardの状態管理強化（脅威軽減度: HIGH）

状態ファイルが平文JSONで保存されており、改ざんや削除によるバイパスが可能な脅威がある。

| STRIDE | REQ-4の脅威内容 | REQ-4の対策 |
|----------|----------------|-------------|
| S:詐欺 | 状態ファイルを削除してデフォルト状態を装う | 署名検証に失敗したらspecUpdated: falseのfail-closed設計 |
| T:改ざん | 状態ファイルの内容を直接編集しspecUpdated=trueを偽造 | HMAC-SHA256署名で改ざん検出、署名不一致で即座に検出 |
| R:否認 | ファイルが自動で削除されたと主張 | 署名検証ログで改ざんや削除の事実を記録して否認を困難にする |
| I:情報開示 | HMACキーが漏洩して任意の署名を鍛造可能 | 既存キーファイルを共用しキー管理はプロジェクト組織の責任とする |
| D:否定 | 改ざん検出ログが削除される | audit-log.jsonlに署名検証結果を記録して複数レイヤーの証跡を確保 |
| E:昇格 | CLIフックでの特権使用を悪用 | ローカルdevでのみ実行し組織内ユーザーに限定、fail-closed設計でデフォルトが安全 |

HMAC署名+fail-closed+複数層の監査ログで強固な防御を実現する。

## リスク

以下は全体的なセキュリティリスク評価をまとめたものである。

| リスク項目 | 重要度 | 現状 | 改善後 |
|----------|--------|------|---------|
| 監査ログの削除や改ざん | MEDIUM | 検出不可 | REQ-4のHMAC署名で検出可能 |
| スコープ外変更 | HIGH | 警告のみで素通り | REQ-2でデフォルト厳格化しREQ-1でログ記録 |
| 環境変数バイパスの隠蔽 | HIGH | 痕跡が残らない | REQ-1で全バイパス記録 |
| 設計と実装の不整合 | CRITICAL | 検証に失敗する場合がある | REQ-2とREQ-4で相互補完 |

総括として、REQ-1からREQ-4によりワークフロープラグインの監査可能性と厳格性が大幅に向上する。特にREQ-4のHMAC署名導入によりfail-closed設計が強化される。
