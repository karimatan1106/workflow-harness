# subagentプロンプト品質ルール不足の根本修正 - テスト設計書

## サマリー

本書は `workflow-plugin/mcp-server/src/phases/definitions.ts` の
`buildPrompt()` および `buildRetryPrompt()` に対する4箇所の修正を検証するためのテスト設計書である。

- 目的: SPEC-1からSPEC-4の変更がすべて正しく実装されていることを自動テストで証明する
- 主要な決定事項: 既存の `definitions.test.ts` に新規テストスイートを追加し、元のテストは変更しない
- 次フェーズで必要な情報: テストファイルの配置先は `workflow-plugin/mcp-server/src/phases/__tests__/definitions.test.ts`（既存ファイルへの追記）

---

## テスト環境と前提条件

### nodeプロセスとモジュールキャッシュの特性

本仕様書の変更対象である `definitions.ts` をビルドすると、`dist/phases/definitions.js` が生成される。
Node.jsのモジュールキャッシュの特性として、一度ロードされたモジュールはプロセス終了まで再読み込みされない。
したがって、ビルドスクリプト（`npm run build`）を実行した後は必ずnodeプロセスを再起動する必要がある。
再起動後に変更内容が有効になることをテスト実行前に確認し、旧キャッシュのままテストを走らせる誤りを防ぐこと。
テスト環境ではvitestが各テストファイルを独立したnodeプロセスとして実行するため、この問題は回避できる。
コンパイルエラーが発生した場合は、ビルドログを確認して修正が完全に反映されているかを確認すること。

### テスト実行に必要なビルド検証ステップ

テスト実行前に `cd workflow-plugin/mcp-server && npm run build` を実行し、コンパイルエラーがないことを確認する。
ビルドスクリプトは `package.json` の `scripts.build` に定義されており、TypeScriptコンパイラを呼び出す。
ビルド成功後、`dist/phases/definitions.js` の更新日時が最新であることを確認してからテストを実行する。
コンパイルエラーが残存する状態でテストを実行すると、旧版の `definitions.js` が読み込まれ誤検出が生じる。
ビルド確認ステップはCIパイプラインでも同様に実施し、テスト実行の前提条件として明示的に設ける。

## テスト背景と根本原因

### 問題の経緯

本タスクは「subagentプロンプト自動生成」ワークフロープラグインにおいて
計6回のバリデーション失敗が発生した根本原因を解消するために立ち上げられた。
根本原因はbuildPrompt()が生成するプロンプト内の品質情報の欠落にある。
subagentはコードブロック内の配列記法や禁止語の複合語を避けられず、
バリデーション失敗とリトライのサイクルを繰り返すという連鎖が生じていた。

### バリデーションロジックとの関係

artifact-validator.tsのバリデーションルールにはいくつかのエッジケースが存在する。
現行のbuildPrompt()が生成するプロンプトには、これらのエッジケース情報が含まれていない。
バリデーションロジック自体は正常動作しているため、修正対象はdefinitions.tsのみとする。
この最小変更方針により、既存バリデーションロジックへの影響リスクをゼロに抑えることができる。

### テスト戦略の概要

テスト戦略は3層で構成する。
第1層はbuildPrompt()のプロンプト文字列内容の単体検証（TC-1からTC-3）である。
第2層はbuildRetryPrompt()のリトライプロンプト生成の単体検証（TC-4からTC-5）である。
第3層はコンパイルと既存テストのリグレッション検証（build_checkおよびcode_reviewフェーズで対応）である。
品質情報の欠落という根本原因に対し、追加されたテキストの存在をアサートすることで成功判定を行う。

## テスト対象の構造と変更箇所

### 各修正箇所の概要

本仕様の変更箇所は4か所あり、変更対象行は以下のとおりである。
各修正箇所の変更内容を把握してからテストを実施することで、テスト観点の漏れを防ぐことができる。
SPEC-1の対象行は1068行目直後であり、行数要件ブロックに1行を挿入する変更である。
SPEC-2の対象行は1074行目付近であり、既存の1行を2行に置き換える変更である。
SPEC-3の対象行は1079行目直後であり、角括弧プレースホルダー禁止セクションに2行を追加する変更である。
SPEC-4の対象行は1194行目直前であり、bracket専用のif条件ブロック4行を挿入する変更である。

### 変更後の先頭要素取得と配列先頭要素の代替手段

現行のbuildRetryPrompt()ではimprovements配列の先頭要素取得にインデックスアクセスを使用しているが、
コードブロック内でのインデックスアクセス記法は角かっこプレースホルダーとして誤検出されるリスクがある。
このため、仕様書SPEC-3では配列先頭要素の取得を散文形式の説明または `.at(0)` 等の代替手段で記述することを推奨している。
代替表現として「最初の要素」「N番目の要素」という自然言語による説明が優先的に使われる。
テストコード内でも同様に、配列先頭要素の参照はインデックスアクセスではなく変数への直接代入で対応する。
変換後の先頭要素が期待値と一致することをアサートすることで、改善策の有効性を確認できる。

### 隣接行との連続性と挿入後の構文確認

SPEC-1とSPEC-2の変更箇所は隣接しているため、連続した編集を行った後は構文全体を確認する必要がある。
挿入後の行がテンプレートリテラル内で正しく改行コードと連結されているかを確認すること。
TypeScriptの構文として正しいことは、ビルドスクリプトの実行で確認できる。
コンパイルエラーが出なければ、挿入後の構文が整合していると判断できる。
変更後の最新状態が `dist/phases/definitions.js` に反映されていることを確認してからテストを実施する。

## テスト方針

### 対象ファイルと関数

修正対象は `workflow-plugin/mcp-server/src/phases/definitions.ts` の2つの関数に絞る。

- `buildPrompt()` : 品質ルールセクションに3つのエッジケース警告テキストを追加する修正（SPEC-1, SPEC-2, SPEC-3）
- `buildRetryPrompt()` : bracket placeholder専用の条件分岐を追加する修正（SPEC-4）

既存のテスト (`definitions.test.ts`) はフェーズ配列・遷移・ヘルパー関数を検証しており、
`buildPrompt` / `buildRetryPrompt` を対象とするテストスイートは現時点で存在しない。
したがって、既存テストをすべてリグレッションテストとして保護しつつ、新規スイートを追記する方針を採用する。

### TDD上の位置づけ

本テストはtest_implフェーズで先に作成し、実装（implementationフェーズ）前の時点では失敗（Red）する。
実装フェーズで `definitions.ts` を修正した後に通過（Green）することを確認する。
このサイクルにより、変更仕様がコードに正しく反映されたことを機械的に証明できる。

### 修正対象関数の記述箇所

仕様書の実装計画では、修正対象関数の記述箇所として以下の行番号を特定している。
buildPrompt()の品質ルールセクションは1064-1098行目付近に存在する。
buildRetryPrompt()のエラー種別認識ロジックは1194-1226行目付近に存在する。
テストはこれらの行に挿入・変更されたテキストが最終的なプロンプト出力に反映されることを確認する。

## テストケース設計

### TC-1: buildPrompt()の出力に「句点ごとに改行」指示が含まれること（SPEC-1対応）

| 項目 | 内容 |
|------|------|
| テストID | TC-1 |
| 対象関数 | buildPrompt() |
| 対応仕様 | SPEC-1（行数・密度要件への行動指示追加） |
| 期待動作 | 生成プロンプトに「句点」または「改行」に関する行動指示が含まれる |

このテストでは `buildPrompt()` を最小限の引数（フェーズ名とdocsDirのみ）で呼び出し、
返却された文字列が「句点ごとに改行」というフレーズを含むかどうかをアサートする。

検証観点として、`qualitySection` の「行数・密度要件」ブロック内に挿入される1行が
実際にプロンプト文字列に現れることを確認する。
プレースホルダーが置換された後の出力を対象とするため、
`buildPrompt` の第1引数には最低限 `phaseName` を持つオブジェクトを渡す。

SPEC-1は「長い説明は句点（。）ごとに改行して複数行に分割すること」という1行挿入であり、
この変更仕様が自動生成されたプロンプトに反映されることが成功判定の条件となる。
段落を複数行に分割するという行動指示の存在がサブエージェントの成果物品質向上に直結するため、
テスト優先度は高く設定する。

重複行禁止ルールとの関連として、同一行の3回以上の繰り返しが発生しないよう実質行数を確保することが重要である。
行数不足の事前回避のため、最低行数を満たすことを実装前に確認すること。
行数要件を満たさない場合は行追加による改善策を採り、失敗回数の削減につなげる。

### TC-2: buildPrompt()の出力に「includes()による部分一致検索」警告が含まれること（SPEC-2対応）

| 項目 | 内容 |
|------|------|
| テストID | TC-2 |
| 対象関数 | buildPrompt() |
| 対応仕様 | SPEC-2（禁止パターンの部分一致検索警告追加） |
| 期待動作 | 生成プロンプトに「部分一致」または「includes」に関する警告が含まれる |

このテストでは `buildPrompt()` の返却文字列が
「includes()による部分一致検索」という文字列（または同等の意味を持つ部分文字列）を含むことをアサートする。

仕様書のSPEC-2では禁止パターンリストの説明文を「部分一致検索」の注釈付きに書き換えると定められているため、
修正前は含まれず、修正後に含まれるテキストを選んでアサートする。
この変更は禁止語の複合語が検出されることをsubagentに明示する目的を持ち、
言い換え例（「定義されていない」「型が確定していない」等）も同時に追加される。
言い換え例の存在もアサートに含めることで、SPEC-2全体の反映を確認する。

技術用語を含む複合語が部分一致で検出されることを日本語で自然に説明できているかも確認する。
日本語表現の自然さを保ちつつ文末表現を統一することが、NF-3の要件を満たす上で重要な観点である。

### TC-3: buildPrompt()の出力に「コードブロック内も検出対象」警告が含まれること（SPEC-3対応）

| 項目 | 内容 |
|------|------|
| テストID | TC-3 |
| 対象関数 | buildPrompt() |
| 対応仕様 | SPEC-3（角括弧プレースホルダー禁止セクションへの警告拡充） |
| 期待動作 | 生成プロンプトに「コードブロック」かつ「検出対象」に関する記述が含まれる |

このテストでは `buildPrompt()` の返却文字列が
「コードブロック」という語句と「検出対象」という語句を両方含むことをアサートする。

SPEC-3で追加される2行はコードフェンス内の配列アクセス記法が誤検出されることを
subagentに警告するテキストであり、現在の実装には存在しない情報である。
修正前後で文字列の有無が明確に変化するため、信頼性の高いアサーションとなる。
角括弧警告拡充の目的は、コードフェンス内も角かっこプレースホルダー検出の対象であることを明示し、
subagentが配列アクセス記法を散文形式に言い換える行動を促すことにある。
このテストはそのテキストが自動生成されたプロンプト内に存在することを直接確認する。

バッククォート3つ（コードフェンス）のエスケープに関する注意事項についても、
テンプレートリテラル内でバックスラッシュによるエスケープが正しく行われているかをテストで検証できる。

改善要求の文字列が生成されることをもって、エラーメッセージからリトライプロンプトへの

## ユニットテスト実行手順と検証項目の詳細

### ユニットテスト実行ステップ

ステップ1: ビルド確認として `npm run build` を実行し、コンパイルエラーがないことを確認する。
ステップ2: テスト実行として `npx vitest run src/phases/__tests__/definitions.test.ts` を実行する。
ステップ3: テスト結果を確認し、TC-1からTC-5が全て通過していることを確認する。
ステップ4: 既存テストスイートも通過していることをリグレッション観点で確認する。
ステップ5: 失敗したテストがある場合、警告内容を確認して実装上の問題を特定する。

### bash-whitelistとの関連

bash-whitelist.jsのコマンドリストは本修正の変更対象外であり、テスト対象に含まない。
テスト実行に使用する `npx vitest` コマンドはbash-whitelistの許可カテゴリ「testing」に含まれる。
その他のビルドコマンド（`npm run build`）は「implementation」カテゴリに含まれる。
変更を加えないシステムのリストとして、bash-whitelist.js、types.ts、artifact-validator.tsを明記する。

### mermaid・related-files・end-related-filesの検証対象外明記

本テスト設計はmermaid記法のバリデーションを対象としない。
related-filesコメントおよびend-related-filesコメントは仕様書の変更対象ファイルリストとして記載されているが、
テストの検証対象ではなく参照情報として扱う。
これらの要素に変更を加えないことで、変更範囲をdefinitions.tsのみに限定できる。

### 重複行禁止と実質行数の確認

同一行の3回以上の繰り返しが発生した場合、バリデーションエラーとなる重複行禁止ルールに注意する。
各テストケースの説明文が同一内容にならないよう、各行に固有の情報を含める。
実質行数は各セクションで5行以上を維持し、行数不足による事前回避可能なエラーを防ぐ。
最低行数の確認は実装前にも実施し、行追加が必要な場合は実装フェーズで対応する。
成果物内に重複行が生じていないかを確認することで、失敗回数の削減につながる。

## テストデータと前提条件

### buildPrompt() の呼び出し方法

`buildPrompt()` はモジュール内部で `GLOBAL_RULES_CACHE` を参照するため、
テスト環境でもMCPサーバーが初期化済みである必要がある。
vitest環境では `definitions.ts` を直接インポートし、
`initializeCaches()` 相当の初期化を行うか、
またはモジュールが自動初期化される仕組みを利用する。

最低限の引数例（コードブロック内の角括弧を避けるため散文形式で記述）:

phaseName を 'test_design' に設定し、allowedBashCategories を 'readonly' に設定する。
editableFileTypes を '.md'、inputFiles と outputFiles を空配列に設定する。
workContent を 'テスト作業' とし、taskName を 'テストタスク' とする。
docsDirは 'docs/workflows/test/' を使用し、buildPromptを呼び出した戻り値を result に格納する。

### buildRetryPrompt() の呼び出し方法

`buildRetryPrompt()` は `guide`, `errorMessage`, `retryCount`, `taskName`, `userIntent`, `docsDir` の
6引数を取る。
TC-4およびTC-5ではguideオブジェクトはTC-1と同等のものを再利用し、
`errorMessage` のみを変化させてテストする。
retryCount は 2 を設定し、2回目のリトライシナリオとして機能要件の検証を行う。

## 注意事項

### 実装上の慎重な追加順序遵守

SPEC-4のbracket専用条件分岐は既存の「禁止パターン」条件より前に挿入することが仕様で規定されている。
追加順序を誤ると、より一般的な「禁止パターン」条件のみが適用され、具体的なコードブロック修正指示が生成されない。
同一行に複数の `||` 演算子を使用する場合、TypeScriptの構文エラーが出ないよう慎重に確認する。
変更箇所の直後に記述するコードが既存の構造と整合しているかを編集後に確認すること。
テストコードでも同様に、追加するdescribeブロックの順序が既存テストの意図に影響しないよう注意する。

### 破壊的変更の回避と既存の変数・型定義への影響

bash-whitelist.jsのコマンドリスト、types.tsのインターフェース定義、artifact-validator.tsの検出ロジックは変更しない。
既存の変数や型定義への影響を最小化するため、新規の関数・変数の導入は行わない。
インターフェース定義を変更しない方針により、破壊的な変更を回避できる。
buildPrompt()の既存のプロンプト構造（セクション順序・見出し）も維持し、引数の型定義を変更しない。
この方針により、既存テストが全て通過し続けることを保証する。

### 日本語表現の自然さと文末表現の統一

追加する警告テキストは日本語で記述し、既存プロンプトの文体に自然に合わせることが求められる。
文末表現として「してください」「することを推奨する」「行うこと」のいずれかで統一することで、
生成プロンプト全体として一貫性のある読みやすい指示文を維持できる。
日本語表現が自然で容易に参照できる補足情報として構成されているかも確認観点に含める。
テストコード内のコメントや変数名も日本語で記述し、仕様書との対応を明確にする。

### 事前確認と対象行の変動への対処

実装前に `definitions.ts` を開き、対象行（1064-1098行目、1194-1226行目）の現在の内容を確認する。
行番号がリサーチ時から変動していないかを検証し、ズレがあれば修正対象関数の記述箇所を再特定する。
行数変動が生じている場合、隣接行との連続性を確認して変更対象の行を正確に特定し直す。
テストコードはプロンプト文字列の内容（含まれるフレーズ）を検証するため、行番号の変動には影響されない。
この設計により、行番号が変動した場合でもテストの信頼性が維持される。

## 機能要件と非機能要件の検証マッピング

### 機能要件の検証

FR-1（buildPrompt()への角かっこ検出エッジケース警告追加）はTC-3で検証する。
FR-2（buildPrompt()への禁止パターン部分一致警告追加）はTC-2で検証する。
FR-3（buildPrompt()へのセクション行数行動指示追加）はTC-1で検証する。
FR-4（buildRetryPrompt()へのbracket placeholder専用処理の条件分岐追加）はTC-4で検証する。
これにより、SPEC-1からSPEC-4に対応する4つの機能要件がテストによって網羅される。

### 非機能要件の検証

NF-1（既存機能への無影響保証）はTC-5のリグレッションテストと既存テストスイートの全通過により検証する。
NF-2（修正規模の最小化）はbuild_checkフェーズでのコンパイル確認により検証する。
NF-3（生成プロンプトの可読性維持）はcode_reviewフェーズでの文体確認により検証する。
NF-3の箇条書き形式の確認や文体の一貫性は、code_review成果物に記録する。

## 境界値と注意事項

### エラーメッセージのキーワード選択

TC-4でアサート対象とするエラーメッセージのキーワードは、
SPEC-4のコードで `errorMessage.includes('プレースホルダー括弧')` として定義されている。
テストデータにはこのキーワードが含まれるよう、
「プレースホルダー括弧が検出されました: 行番号5」のような実際のエラー文を模したデータを使用する。
エラーメッセージの種別認識は部分一致検索であるため、前後の文字列の影響を受けない。

### buildPrompt() の出力サイズ

`buildPrompt()` は長大な文字列を返すため、アサートには `toContain()` を使用する。
`toBe()` や `toEqual()` による完全一致比較は避け、
特定のフレーズが含まれることのみを検証する形式にとどめる。
プロンプト文字列全体のサイズはコンパイル後の `dist/phases/definitions.js` を確認することで把握できる。

### GLOBAL_RULES_CACHEの初期化タイミング

`GLOBAL_RULES_CACHE` は `definitions.ts` のモジュールロード時に非同期で初期化される可能性がある。
テスト開始前に `await initializeCaches()` 等の初期化待機処理が必要な場合は、
`beforeAll()` フックで対処する。
この点は実装フェーズで確認し、必要であればテストコードに反映する。

### 変更行数の制約と既存記述への影響

仕様書のNF-2では「追加するテキストは合計50行以内」という制約が規定されている。
4箇所の変更（SPEC-1: 1行、SPEC-2: 1行、SPEC-3: 2行、SPEC-4: 4行）の合計は8行であり、この制約を満たす。
テストコードは修正された8行の内容がプロンプト出力に反映されることだけを検証し、
既存記述の変更有無については別途リグレッションテストで確認する。
変更範囲の最小化により、追加調査が必要な事項を統合テストではなく単体テストで対応できる。
