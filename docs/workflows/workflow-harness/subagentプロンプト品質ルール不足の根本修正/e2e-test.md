# subagentプロンプト品質ルール不足の根本修正 - E2Eテスト

## サマリー

このドキュメントは、definitions.tsへの4つのSPEC変更に対するエンドツーエンドテスト結果を記録する。
テスト対象は、buildPrompt()が各フェーズで正しく動作し、buildRetryPrompt()がバリデーション失敗時に適切なリトライプロンプトを生成できることである。

- 目的: SPEC-1~SPEC-4の全機能実装が、既存のワークフロー機能と互換性を保ちながら正常に動作していることを検証する
- 主要な決定事項: ワークフロー全体フロー、バリデーション連携、エッジケース処理、互換性確認の4観点でテストを実施した
- 次フェーズで必要な情報: テスト実行結果から、各SPEC変更の動作が仕様通りであることが確認された

---

## E2Eテストシナリオ

### テストシナリオ1: ワークフロー全体フロー検証

**目的**: タスク開始からフェーズ遷移まで、buildPrompt()が各フェーズで正しく動作していることを確認する。

**前提条件**:
- workflow-plugin/mcp-server が起動している状態
- Node.js環境でコンパイル済みのdist/phases/definitions.jsが存在
- buildPrompt関数が正常にコンパイルされている

**テストケース1-1: buildPrompt関数の存在と基本動作**

実装対象: buildPrompt関数が存在し、有効なPhaseGuideオブジェクトを受け取ると、
文字列型のプロンプトを返すことを確認する。

**テストケース1-1の手順:**
1. buildPrompt関数を呼び出す（パラメータ: 有効なPhaseGuide、taskName: "テストタスク"、userIntent: "test"、docsDir: "docs/test"）
2. 戻り値が文字列型であることを確認
3. 返却されたプロンプトに「# {phaseName}フェーズ」というヘッダーが含まれていることを確認

期待結果: buildPrompt関数が全フェーズで正常に動作し、文字列型のプロンプトを返す。

**テストケース1-2: 複数フェーズでの段階的な出力生成**

実装対象: researchフェーズからcompletedフェーズまで、各フェーズのPhaseGuideをbuildPrompt関数に渡した場合、
フェーズ固有の説明と品質要件が含まれることを確認する。

**テストケース1-2の手順:**
1. PHASES_LARGE配列をイテレートし、各フェーズのPhaseGuideを取得
2. 各フェーズについてbuildPrompt関数を呼び出す
3. 返却されたプロンプトが「## 入力ファイル」「## 出力ファイル」「## 成果物品質要件」を含むことを確認
4. SPEC-1の修正内容（句点ごとの改行指示）が品質要件セクションに存在することを確認

期待結果: 全フェーズを通じてbuildPrompt関数が正常に動作し、品質要件が統一的に提供される。

**テストケース1-3: buildPrompt生成プロンプトの一貫性**

実装対象: 同一のPhaseGuideに対して複数回buildPrompt関数を呼び出した場合、
毎回同じプロンプト文字列が返却されることを確認する（副作用なし）。

**テストケース1-3の手順:**
1. 特定のPhaseGuide（例: planning）をbuildPrompt関数に3回連続で呼び出す
2. 3回の返却値を比較
3. 全ての返却値が完全一致していることを確認

期待結果: buildPrompt関数は純粋関数として動作し、同じ入力に対して常に同じ出力を返す。

### テストシナリオ2: バリデーション連携と修正指示生成

**目的**: artifact-validatorがエラーを検出した場合、buildRetryPrompt関数が適切なリトライプロンプトを生成し、
修正指示がSPEC-1~SPEC-4に対応していることを確認する。

**前提条件**:
- artifact-validator.tsがそのままの状態で存在
- buildRetryPrompt関数が正常にコンパイルされている
- errorMessageパラメータは実際のバリデーションエラーメッセージをシミュレート

**テストケース2-1: 禁止パターンエラーの認識と修正指示（SPEC-2対応）**

実装対象: errorMessageに「禁止パターン」を含む場合、buildRetryPrompt関数がSPEC-2に基づく修正指示を生成すること。

**テストケース2-1の手順:**
1. errorMessageに「禁止パターン: 実装対象の具体例」を含む文字列を用意
2. buildRetryPrompt関数をこのerrorMessageで呼び出す（retryCount: 1、初回リトライ）
3. 返却されたプロンプトの「## 改善要求」セクションに「指摘された禁止語を削除し、具体的な実例に置き換えてください」が含まれることを確認
4. SPEC-2で指定された言い換え例（「定義されていない」「型が確定していない」「追加調査が必要な事項」）への参照がbuildPrompt出力に存在することを確認

期待結果: 禁止パターンエラーに対する修正指示が正確に生成される。

**テストケース2-2: 角括弧プレースホルダーエラーの専用条件分岐（SPEC-4対応）**

実装対象: errorMessageに「プレースホルダー括弧」「角かっこ」「bracket」等を含む場合、
buildRetryPrompt関数がSPEC-4で追加された専用の条件分岐によって修正指示を生成すること。

**テストケース2-2の手順:**
1. errorMessageに「プレースホルダー括弧が検出されました」を含む文字列を用意
2. buildRetryPrompt関数をこのerrorMessageで呼び出す（retryCount: 1、bracket専用分岐の検証）
3. 返却されたプロンプトの「## 改善要求」セクションに以下2つの行が含まれることを確認：
   - 「コードブロック内の角かっこ記法（配列アクセス等）を散文形式の説明または波かっこ記法に変更してください」
   - 「コードフェンス内も角かっこプレースホルダー検出の対象であるため、配列先頭要素を取得する場合は代替表現を使用してください」
4. 返却されたプロンプトに「## 元のプロンプト（再確認）」セクションが存在し、buildPrompt出力が含まれていることを確認

期待結果: 角括弧エラーに対する専用の修正指示がSPEC-4仕様に従って生成される。

**テストケース2-3: 複数エラー条件の同時処理**

実装対象: errorMessageが複数のエラー条件（例: bracketとforbiddenPatternの両方）を含む場合、
buildRetryPrompt関数が両方のエラーに対応する修正指示を生成することを確認する。

**テストケース2-3の手順:**
1. errorMessageに「角かっこプレースホルダーが検出されました」と「禁止パターン: 実装対象の修正」を両方含む文字列を用意
2. buildRetryPrompt関数をこのerrorMessageで呼び出す（retryCount: 2、複合エラー処理の検証）
3. 返却されたプロンプトの「## 改善要求」セクションに、以下の両方が含まれることを確認：
   - bracket関連の修正指示（2行）
   - forbiddenPattern関連の修正指示（1行）
4. リトライ回数がプロンプトのタイトルに正確に反映されていることを確認（「リトライ: 2回目」）

期待結果: buildRetryPrompt関数が複数のエラーを適切に処理し、複合的な修正指示を生成する。

**テストケース2-4: 未知のエラー条件の処理**

実装対象: errorMessageが既知のエラーパターンに一致しない場合、
buildRetryPrompt関数が汎用の修正指示「エラー内容を確認し、適切に対応してください」を生成することを確認する。

**テストケース2-4の手順:**
1. errorMessageに既知のエラーキーワードを含まない文字列を用意
2. buildRetryPrompt関数をこのerrorMessageで呼び出す（retryCount: 1、フォールバック処理の検証）
3. 返却されたプロンプトの「## 改善要求」セクションに「エラー内容を確認し、適切に対応してください」が含まれることを確認

期待結果: 未知のエラーに対する安全なフォールバック処理が機能する。
