## セキュリティ

### 入力値の取り扱い

buildRetryPrompt()のSPEC-4条件分岐では、引数として受け取った `errorMessage` に対して `includes()` を用いてキーワード検索を行う。
この処理はユーザーやシステム内部からのエラーメッセージをそのまま参照するが、条件分岐の判定にのみ使用され、外部に出力されるわけではない。
`improvements.push()` に渡されるのはコード内でハードコードされた定数文字列のみであり、外部入力がプロンプト本文に直接注入されるリスクはない。

### プロンプトインジェクションリスクの評価

buildRetryPrompt()内の1193行目付近で `errorMessage` をコードブロックで囲んで出力するセクションが存在する。
このコードブロック形式での埋め込みは「参照情報として解釈しないこと」という注記と合わせて出力されるため、インジェクションリスクの軽減策が既に講じられている。
SPEC-4が追加した条件分岐はこのエラーセクションの後に処理されるため、構造上の問題はない。

### 既存のセキュリティ設計への影響

今回の変更は `definitions.ts` 内のプロンプト生成ロジックのみを対象としており、認証・認可・HMAC検証等のセキュリティメカニズムには一切触れていない。
bash-whitelist.jsのブラックリストや環境変数保護の設定も変更対象外であることを確認した。
NF-1（既存機能への無影響保証）が適切に守られていると判断する。

## パフォーマンス

今回の変更は `qualitySection` への文字列連結（`+=` 演算子）が数行増加するに留まり、
処理量として占める割合は無視できるレベルである。
`qualitySection` への各文字列追加操作はO(1)の時間複雑度であり、
呼び出し頻度（ワークフロー開始時のフェーズ初期化に限定）を考慮するとホットパスには該当しない。
`buildRetryPrompt()` に追加したSPEC-4の条件分岐は最大3つの `includes()` 呼び出しを追加するが、
`includes()` は線形スキャンであっても対象文字列（errorMessage）が数百文字程度であるため実質的なコストは発生しない。
`improvements` 配列への `push()` 操作も定数時間であり、配列サイズの変化は最大2要素の増加に留まる。
総じて、本変更がMCPサーバーの応答時間やメモリ消費量に与える影響は計測不能な範囲内である。
