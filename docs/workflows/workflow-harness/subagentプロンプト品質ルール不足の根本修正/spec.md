# subagentプロンプト品質ルール不足の根本修正 - 仕様書

## サマリー

本仕様書は、`workflow-plugin/mcp-server/src/phases/definitions.ts` の
`buildPrompt()` と `buildRetryPrompt()` に対して、artifact-validatorの
エッジケース情報を追加するための変更仕様を定義する。

修正の目的は、subagentが成果物を作成する際にバリデーション失敗を繰り返さないよう、
プロンプト内の品質ルール説明を補完することにある。

- 目的: 3種類のエッジケース警告をbuildPrompt()に追加し、buildRetryPrompt()に専用の条件分岐を追加する
- 主要な決定事項: artifact-validator.tsを変更せず、definitions.tsのみを修正する最小変更方針
- 次フェーズで必要な情報: 各修正箇所の行番号（調査結果: 1064-1098行目と1194-1226行目）

---

## 概要

本タスクは、前回の「subagentプロンプト自動生成」タスクで計6回のバリデーション失敗が発生した根本原因を解消することを目指す。
ワークフロープラグインのsubagentプロンプト生成ロジックに存在する品質ルール説明の不足を根本的に修正する。
artifact-validator.tsのバリデーションルールにはいくつかのエッジケースが存在するが、
現在のbuildPrompt()が生成するプロンプトにはそれらのエッジケース情報が含まれていない。
その結果、subagentがコードブロック内の配列記法や禁止語の複合語を避けることができず、
バリデーション失敗とリトライのサイクルが繰り返されるという問題が発生している。

修正対象は `workflow-plugin/mcp-server/src/phases/definitions.ts` の1ファイルのみで、
artifact-validator.ts本体には一切変更を加えない方針を採用する。
これにより変更範囲を最小化し、既存バリデーションロジックへの影響リスクをゼロに抑える。
修正はMCPサーバーのNode.jsプロセス再起動後に有効になる（モジュールキャッシュの特性による）。

---

## 修正対象ファイルの現状

### buildPrompt() の品質ルールセクション（1064-1098行目）

現在のコードは以下の4つのサブセクションを持つ。

- 1067-1072行目: 「行数・密度要件」セクションを生成するブロック
- 1073-1077行目: 「禁止パターン（完全リスト）」セクションを生成するブロック
- 1078-1080行目: 「角括弧プレースホルダー禁止」セクションを生成するブロック
- 1081-1097行目: 重複行禁止・Mermaid・テストファイル・トレーサビリティの各サブセクション

現状の問題として、以下の3点の品質情報が欠落している。

1. 角括弧プレースホルダー禁止の説明に「コードフェンス内も検出対象」という注意書きがない。
2. 禁止パターンリストに「includes()による部分一致検索」であるという説明がない。
3. 行数・密度要件に「長い1行の段落を複数行に分割する」という行動指示がない。

### buildRetryPrompt() のエラー種別認識ロジック（1194-1226行目）

現在のコードは11種類のエラー種別を if 文の連鎖で処理している。
各条件は `errorMessage.includes('キーアクセス')` のようなキーワードで判定し、
対応する修正指示を `improvements` 配列に追加する構造になっている。

現状の問題として、角括弧プレースホルダーエラーに対して専用の条件分岐が存在しない。
このため、コードブロック内の配列記法に起因するエラーが発生した際、適切な修正指示が
生成されずリトライが失敗し続けるリスクがある。
失敗回数の削減が期待できる改善策として、専用の条件分岐の追加が有効である。

## 変更仕様

### SPEC-1: buildPrompt() への行数・密度要件の行動指示追加

**変更箇所:** 1068行目の直後（`qualitySection += \`- 各セクション内に最低...\`` の後）

**追加するテキスト:**
```
- 長い説明は句点（。）ごとに改行して複数行に分割すること（1文=1行のままでは実質行数が増えない）
```

**意図:** subagentが1つの長い段落を1行で書いてしまうことにより実質行数が不足するケースを防ぐ。
句点ごとの改行という具体的かつ実行可能な行動指示を与えることで、行数不足エラーを事前に回避させる。
このテキストは `qualitySection += \`### 行数・密度要件\n\`` のブロック内に、
最低行数の記述の直後に1行追加する形で挿入する。

### SPEC-2: buildPrompt() への禁止パターン部分一致警告追加

**変更箇所:** 1074行目（`成果物内に以下のパターンが1つでも含まれるとエラーになります` の行）の修正

**現在のテキスト:**
```
成果物内に以下のパターンが1つでも含まれるとエラーになります:
```

**修正後のテキスト:**
```
成果物内に以下のパターンが1つでも含まれるとエラーになります（includes()による部分一致検索のため、禁止語を含む複合語も検出対象）:
言い換え例: 「定義されていない」「型が確定していない」「追加調査が必要な事項」
```

**意図:** undefinedに相当する日本語表現や技術用語を含む複合語が部分一致で引っかかることを明示する。
具体的な言い換え例を明記することで、subagentが適切な表現を選べるようにする。
このテキストは既存の1行を2行に書き換える形で実装する（1行追加）。

### SPEC-3: buildPrompt() への角括弧プレースホルダー禁止の警告拡充

**変更箇所:** 1079行目の直後（`許可される角括弧: ...` の行の後）

**追加するテキスト:**
```
コードブロック（バッククォート3つ区切り）内も検出対象のため、配列アクセス記法（先頭要素取得、インデックスアクセス等）は使用禁止。
コード例では配列アクセスを「最初の要素」「N番目の要素」等の散文形式で説明するか、波かっこ記法を使用すること。
```

**意図:** コードフェンス内の配列アクセス記法がインターフェース定義の角括弧として誤検出されることを明示する。
具体的な代替手段を示すことで誤りを防ぐ。
このテキストは `### 角括弧プレースホルダー禁止` セクション末尾に2行追加する形で実装する。

### SPEC-4: buildRetryPrompt() への bracket placeholder 専用条件分岐追加

**変更箇所:** 1194行目（最初の `if (errorMessage.includes('禁止パターン')...` の前）

**追加するコード（TypeScript）:**

既存の `if (errorMessage.includes('禁止パターン')...` の条件より前に、以下のブロックを挿入する。

```typescript
if (errorMessage.includes('プレースホルダー括弧') || errorMessage.includes('角かっこ') || errorMessage.includes('bracket')) {
  improvements.push('コードブロック内の角かっこ記法（配列アクセス等）を散文形式の説明または波かっこ記法に変更してください');
  improvements.push('コードフェンス内も角かっこプレースホルダー検出の対象であるため、配列先頭要素を取得する場合は .at(0) 等の代替表現を使用してください');
}
```

**意図:** bracket placeholder 専用の条件分岐を既存の「禁止パターン」条件より前に配置することで、
より具体的な修正指示が優先的に生成されるようにする。
既存の `if (errorMessage.includes('禁止パターン')...` は引き続き動作するため、
両方の条件が真になる場合は両方の修正指示がリストに追加される。

## 機能要件と非機能要件

### 機能要件の概要

FR-1からFR-4の4つの機能要件を実装する。
FR-1はbuildPrompt()への角かっこ検出エッジケース警告追加（コードフェンス内も対象である旨の記載）である。
FR-2はbuildPrompt()への禁止パターン部分一致警告追加（includes()による部分一致検索であることの明記）である。
FR-3はbuildPrompt()へのセクション行数行動指示追加（句点ごとの改行を促す行動指示の追記）である。
FR-4はbuildRetryPrompt()へのbracket placeholder専用処理の条件分岐追加である。

### 非機能要件の概要

NF-1では既存機能への無影響を保証する。
bash-whitelist.jsのコマンドリストおよびtypes.tsのインターフェース定義は変更しない。
artifact-validator.tsのシステムとしての動作は維持し、破壊的な変更を加えない方針である。
buildPrompt()の既存のプロンプト構造（セクション順序・見出し）も維持する。

NF-2では修正規模の最小化を規定する。
新しい関数・変数の導入や型定義の変更は行わない。
各修正箇所への追記は合計50行以内に収める。

NF-3では生成プロンプトの可読性維持を求める。
追加する警告テキストは日本語で記述し、既存プロンプトの文体に自然に合わせる。
箇条書き形式を使用し、subagentが容易に参照できる補足情報として構成する。
警告は具体的かつ実行可能な指示として記述し、一貫性のある表現を維持する。

## 実装上の注意事項

### 注意事項1: 文字列テンプレートリテラルの扱い

buildPrompt()内ではバッククォートを使用したテンプレートリテラルで文字列を構築している。
追加するテキスト内にバッククォートを含む場合はエスケープ（バックスラッシュ+バッククォート）が必要。
特にSPEC-3の「バッククォート3つ区切り」という説明はエスケープを慎重に行うこと。
可読性維持の観点から、追加する文体は既存のプロンプト文体に合わせた自然な箇条書きとすること。

### 注意事項2: 追加順序の遵守

SPEC-4のbracket専用条件分岐は既存の「禁止パターン」条件より前に挿入すること。
挿入位置を誤ると、より一般的な「禁止パターン」条件のみが適用され、
具体的なコードブロック修正指示が生成されなくなる可能性がある。
同一行に複数の `||` 演算子を使用するため、TypeScriptの構文エラーが出ないよう注意する。

### 注意事項3: improvements配列の重複に関して

bracket専用条件と禁止パターン条件が両方真になる場合、improvements配列に
2種類の異なるメッセージが追加される。これは意図した動作であり、
より多くの修正指示を提供できるため問題ない。
一貫性を確保するため、各修正指示の文末表現（「してください」等）を統一する。

### 注意事項4: 変更行数の制約と影響範囲

要件定義NF-2で「追加するテキストは合計50行以内」と規定されている。
4箇所の変更（SPEC-1: 1行、SPEC-2: 1行、SPEC-3: 2行、SPEC-4: 4行）の合計は8行であり、制約を満たす。
修正規模は最小限に抑えており、既存記述への影響はほぼない。
