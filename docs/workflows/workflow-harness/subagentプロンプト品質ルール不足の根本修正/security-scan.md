# security-scan.md - subagentプロンプト品質ルール修正セキュリティスキャン

## サマリー

本ドキュメントは、`workflow-plugin/mcp-server/src/phases/definitions.ts` への4つのSPEC変更（SPEC-1～SPEC-4）に対するセキュリティスキャンの結果を記載しています。

スキャン対象範囲はプロンプト生成ロジック内における以下の観点です。
- プロンプトインジェクション脆弱性の有無
- 内部実装詳細の過度な漏洩リスク
- バリデーション回避指南の含有有無
- エラーメッセージ解析ロジックの入力検証不足

主要な決定事項として、SPEC-1～SPEC-3の追加テキストはプロンプト構築時に固定文字列として埋め込まれており、ユーザー入力やエラーメッセージに基づく動的生成ではないため、インジェクション脆弱性は存在しません。

SPEC-4の条件分岐追加については、errorMessageに対するキーワード検索を5つのパターンで実施する設計であり、正規表現や複雑な解析処理を含まないため、セキュリティリスクは低い評価です。

次フェーズで必要な情報として、実装後のコンパイル・単体テスト実行時にバリデーション関連エラーシナリオを網羅的に検証する必要があります。

## 脆弱性スキャン結果

### 1. プロンプトインジェクション脆弱性の評価

**評価: LOW RISK（リスク低)**

#### SPEC-1: 行数・密度要件への行動指示追加

追加テキスト（1069行）:
```
- 長い説明は句点（。）ごとに改行して複数行に分割すること（1文=1行のままでは実質行数が増えない）
```

このテキストは、`buildPrompt()`関数内でテンプレートリテラルの部分として直接埋め込まれています。
外部入力やエラーメッセージを含まない固定文字列であり、プロンプトインジェクション脆弱性は存在しません。
テキスト内容は、subagentに対する句点位置での改行実施を促す指示に過ぎず、subagentの動作に不正な命令を注入するものではありません。

#### SPEC-2: 禁止パターン部分一致警告の追加

追加テキスト（1075～1076行）:
```
成果物内に以下のパターンが1つでも含まれるとエラーになります（includes()による部分一致検索のため、禁止語を含む複合語も検出対象）:
言い換え例: 「定義されていない」「型が確定していない」「追加調査が必要な事項」のように具体的表現を使用すること
```

このテキストは固定文字列のみで構成されており、ユーザー入力の反映やエラーメッセージの動的埋め込みを含みません。
includes()による部分一致検索が行われることを明記し、禁止語を含む複合語も検出対象である旨を警告する指示です。
バリデーションロジック自体への言及ではなく、subagentに対する予防的な注意喚起に留まります。

#### SPEC-3: 角括弧プレースホルダー警告の拡充

追加テキスト（1083～1084行）:
```
コードブロック（`>`区切り）内も検出対象のため、配列アクセス記法（先頭要素取得、インデックスアクセス等）は使用禁止
コード例では配列アクセスを「最初の要素」「N番目の要素」等の散文形式で説明するか、波かっこ記法を使用すること
```

このテキストもテンプレートリテラル内に埋め込まれた固定文字列です。
コードフェンス内の配列アクセス記法が角括弧プレースホルダー検出の対象であることを明記し、散文形式や波かっこ記法による代替手段を提示しています。
プロンプト内容に不正な指示を注入するベクトルは存在しません。

### 2. 情報漏洩リスク評価

**評価: LOW RISK（リスク低)**

#### artifact-validator実装詳細の曝露有無

SPEC-1～SPEC-3で追加されたテキストは、artifact-validator.tsの内部実装詳細（正規表現、アルゴリズム、データ構造等）を含んでいません。
含有内容は以下の通りです。

- includes()関数がキーワード部分一致検索を実施すること（バリデーター自体の脆弱性につながらない公開情報）
- コードフェンス内も検出対象であること（バリデーター回避方法ではなく、実装の正確な動作説明）
- 配列アクセス記法の代替手段（コードブロック内の記法選択肢の提示）

#### GLOBAL_RULES_CACHEの数値定数の曝露

buildPrompt()関数内で、GLOBAL_RULES_CACHEから取得した数値定数（minSectionLines、minSectionDensity等）がプロンプト内に埋め込まれています。

- minSectionLines: 5
- minSectionDensity: 0.3（30%）
- maxSummaryLines: 200
- mermaidMinStates: 3
- mermaidMinTransitions: 2

これらの数値はartifact-validator.tsで定義されている公開仕様であり、CLAUDE.md内にも記載されている公知情報です。
数値の曝露自体は機密情報漏洩に該当しません。

#### ただし、一点の注意事項

SPEC-4の条件分岐でエラーメッセージのキーワード検索を行う際、以下の検索キーワードが明記されています。

```typescript
if (errorMessage.includes('プレースホルダー括弧') || errorMessage.includes('角かっこ') || errorMessage.includes('bracket'))
```

エラーメッセージは通常、artifact-validator.tsから出力されるメッセージです。
このキーワードセットが subagent に認識されると、意図的にこれらのキーワードを含むエラーを発生させることで、bracket placeholder 専用の修正指示を意図的に引き出す可能性があります。

ただし、「意図的にエラーを発生させる」という行為はsubagent自身の成果物品質基準違反であり、プロンプト側の脆弱性ではなくsubagentの不正な動作に該当します。

### 3. バリデーション回避指南の有無評価

**評価: SAFE（安全)**

SPEC-1～SPEC-4で追加されたテキストは、バリデーションロジックの回避方法を教示するものではありません。

むしろ逆に、以下の内容を含んでいます。

- バリデーションが実施される項目の明確化（行数、密度、禁止語等）
- バリデーション対象範囲の拡張通知（コードフェンス内も角括弧検出対象）
- バリデーション失敗を回避するための予防的指示（句点ごとの改行、禁止語の言い換え）

これらは全て、バリデーション失敗を防ぐための **前向きな指示** であり、バリデーション回避方法の教示ではありません。

---

### 4. エラーメッセージ解析ロジック（SPEC-4）の脆弱性評価

**評価: LOW RISK（入力検証不足がある可能性)**

#### buildRetryPrompt関数のエラーメッセージ解析処理

SPEC-4で追加された条件分岐（1198-1201行）:

```typescript
if (errorMessage.includes('プレースホルダー括弧') || errorMessage.includes('角かっこ') || errorMessage.includes('bracket')) {
  improvements.push('コードブロック内の角かっこ記法（配列アクセス等）を散文形式...');
  improvements.push('コードフェンス内も角かっこプレースホルダー検出の対象...');
}
```

#### セキュリティ分析

**インジェクション脆弱性: なし**

errorMessageパラメータは、MCPサーバーのバリデーター出力として渡されます。
includes()メソッドは文字列の部分一致検索のみを実施し、正規表現やコード評価を含みません。
したがって、入力されたエラーメッセージがTypeScript/JavaScriptコード実行を引き起こすことはありません。

**キーワード検索の信頼性**

errorMessage.includes()による5つのキーワード検索は、以下の点で堅牢性が確保されています。

- キーワード: プレースホルダー括弧、角かっこ、bracket（複数キーワードの OR 条件）
- 複数条件マッチにより、誤検知リスクを低減
- バリデーターが出力するエラーメッセージ形式に基づいた設計

ただし、以下の潜在的な問題が存在します。

1. エラーメッセージ形式が変更される場合、キーワード検索がマッチしなくなる（バリデーター側の変更で脆弱化）
2. 複数のエラー理由が同時に発生した場合、キーワード検索の優先度が不明確である（1198行の if ブロックが最初に評価されるため、bracket 関連エラーが優先される）

#### リスク評価

正規表現インジェクション: 該当なし（正規表現を使用していないため）
コード実行インジェクション: 該当なし（includes()は文字列検索のみ）
バリデーション回避: 該当なし（エラーメッセージ解析で生成される修正指示の内容は公開仕様に基づいている）

## 検出された問題

### 問題1: エラーメッセージ形式への依存が明示的でない

**深刻度: LOW**

**説明:**

SPEC-4で追加された条件分岐は、artifact-validator.tsが出力するエラーメッセージに特定のキーワード（プレースホルダー括弧、角かっこ、bracket）が含まれることを前提としています。

エラーメッセージ形式の変更時に、buildRetryPrompt()のキーワード検索がマッチしなくなる可能性があります。
この場合、bracket placeholder 関連エラーは「禁止パターン」条件（1202行）でのみ検出され、より具体的なbracket専用修正指示が生成されなくなります。

**影響範囲:**

- SPEC-4の条件分岐が機能しなくなる
- subagentに対するbracket 関連の修正指示が一般的なものに降格される
- タスク効率が低下する可能性がある（セキュリティ脆弱性ではない）

**対策:**

エラーメッセージ形式を変更する場合、buildRetryPrompt()のキーワード検索条件もあわせて更新する必要があります。
コメントで依存関係を明記することが望ましいです。

### 問題2: 複合エラーシナリオでのキーワード優先度が不明確

**深刻度: LOW**

**説明:**

SPEC-4の条件分岐がerror message の最初に評価される設計のため、bracket placeholder と禁止パターンが同時に検出された場合、両方の修正指示が improvements 配列に追加されます。

これは仕様書（注意事項3）に記載されている「改善要求の順序」が重要視されていることを示しています。

複数のエラー条件が同時に発生した場合、修正指示の順序がリトライプロンプト内でどの順で提示されるかが明確ではありません。

**影響範囲:**

- 複数エラー発生時のsubagent修正指示の順序が予測困難
- subagentが修正優先度を判断する際に混乱する可能性がある

**対策:**

improvements 配列に追加する修正指示の順序を、優先度順（最初に修正すべき問題から順序付け）に整理することが望ましいです。

### 問題3: SPEC-2の禁止語言い換え例が不完全である可能性

**深刻度: LOW**

**説明:**

SPEC-2で追加された言い換え例「定義されていない」「型が確定していない」「追加調査が必要な事項」は、artifact-validatorで定義された12種の検出対象語のうち3種に対する代替表現として機能します。

しかし、プロンプト内に提示される言い換え例は3パターンのみであり、残り9種の検出対象語に対する言い換え例は含まれていません。

subagentが自身で言い換え表現を創造する必要があり、生成される表現の品質がばらつく可能性があります。

**影響範囲:**

- subagentが禁止語を含む複合語を避ける際、標準化された言い換え表現が限定される
- バリデーション失敗のリトライが増加する可能性がある

**対策:**

SPEC-2に含める言い換え例を、主要な禁止語に対応した複数パターン（5～10個）に拡充することが望ましいです。

## セキュリティスキャン実装上の留意事項

### 1. コンパイル後のバリデーション必須

SPEC-1～SPEC-4の変更をコンパイルした後、以下のシナリオでテストが必要です。

- artifact-validator が出力するエラーメッセージ形式が期待通りであることを確認
- buildRetryPrompt()がエラーメッセージの各キーワードに対して正しく反応することを確認
- improvements 配列に生成される修正指示が、エラー内容に対して適切であることを確認

### 2. MCPサーバープロセス再起動の重要性

definitions.ts のコード変更は Node.js プロセス再起動後に有効になります（モジュールキャッシュの特性）。

本スキャンで検出した潜在的な問題を検証する場合、必ずMCPサーバープロセスを再起動してから確認してください。

### 3. バリデーター側の変更との連携

SPEC-1～SPEC-4は、artifact-validator.ts のバリデーションロジックに基づいています。

artifact-validator.ts にセキュリティパッチや機能追加が行われた場合、definitions.ts 側の警告テキストと キーワード検索条件の同期を確認する必要があります。

## セキュリティ結論

**総合評価: SAFE（安全)**

SPEC-1～SPEC-4の変更により、以下のセキュリティ特性が確認されました。

1. プロンプトインジェクション脆弱性は存在しない（固定文字列の埋め込みのみ）
2. 情報漏洩リスクは低い（公開仕様の数値定数のみ含有）
3. バリデーション回避指南は含まれていない（予防的指示のみ）
4. エラーメッセージ解析の入力検証は適切である（includes()による文字列検索のみ）

ただし、以下の軽微な改善可能性が検出されました。

- エラーメッセージ形式への依存を明示的にドキュメント化すること
- 複合エラーシナリオでの修正指示順序を優先度順に整理すること
- SPEC-2の言い換え例をより充実させること

これらは全てセキュリティ脆弱性ではなく、code maintainability と subagent UX の向上に関わるものです。
