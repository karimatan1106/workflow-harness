## サマリー

本ドキュメントは `workflow-plugin/mcp-server/src/phases/definitions.ts` の
`buildPrompt()` および `buildRetryPrompt()` が生成するプロンプトテキストを
「subagentに対するインターフェース」として設計した結果をまとめる。

- 目的: 4つの仕様変更（SPEC-1〜SPEC-4）の出力フォーマットを設計し、実装フェーズへ引き継ぐ
- 主要な決定事項: buildPrompt()の品質ルールセクションに3箇所の警告を追記し、buildRetryPrompt()にbracket専用条件分岐を追加する
- 次フェーズで必要な情報: 各セクションの挿入位置（1068行目直後、1074行目書き換え、1079行目直後、1194行目直前）

---

## CLIインターフェース設計

`buildPrompt()` はsubagentへ渡す指示文字列を生成する関数であり、その出力は
標準的なMarkdown形式のプロンプトテキストとして構成される。
関数の呼び出し元はOrchestrator（メインのClaude）であり、Taskツールのpromptパラメータに
その文字列を埋め込んで渡す。
subagentはこの文字列を「仕様書」として受け取り、成果物を作成する。

### プロンプト全体の9セクション構成

buildPrompt()が生成するプロンプトは以下のセクション順序で構成される。

1. フェーズ名ヘッダー（`# {フェーズ名}フェーズ`）
2. タスク情報（タスク名・出力先ディレクトリ）
3. 入力ファイルリスト
4. 作業内容の説明
5. 出力ファイルリスト
6. サマリーセクション必須化ルール（REQ-4準拠）
7. Bashコマンド制限セクション（許可カテゴリと各コマンド一覧）
8. 成果物品質要件セクション（artifact-validatorの検出ルール説明）
9. フェーズ固有の追加指示（phaseGuideのsubagentTemplateから展開）

### 成果物品質要件セクションの修正後フォーマット

セクション8「成果物品質要件」には以下のサブセクションが含まれる。
各サブセクションの修正点を示す。

#### 行数・密度要件サブセクション（SPEC-1追加対象）

```
### 行数・密度要件
- 各セクション（## 見出し）内に最低5行の実質行（空行・水平線・コードフェンスを除く）を含めること
- 長い説明は句点（。）ごとに改行して複数行に分割すること（1文=1行のままでは実質行数が増えない）  ← SPEC-1で追加
- 長い段落は複数行に分割すること（1段落=1行にならないように）
- セクション密度（実質行/総行）は30%以上を維持すること
```

SPEC-1追加行の配置位置は、最低行数要件の行の直後、段落分割指示の行の直前とする。
この位置に配置することで、行数不足の原因となる「1文=1行」パターンの警告を
行数要件の説明と自然に接続できる。

#### 禁止パターンサブセクション（SPEC-2修正対象）

```
### 禁止パターン（完全リスト。成果物内に1つでも含まれるとエラー）
成果物内にCLAUDE.mdで定義された英語4種・日本語8種の禁止語が1つでも含まれるとエラーになります（includes()による部分一致検索のため、禁止語を含む複合語も検出対象）:  ← SPEC-2で修正
言い換え例: 「定義されていない」「型が確定していない」「追加調査が必要な事項」  ← SPEC-2で追加
- 禁止語一覧はCLAUDE.mdの「禁止パターン（完全リスト）」セクションを参照すること
- 禁止語を含む複合語も検出対象: 例として「指定されていない」「追加調査が必要な事項」などへの言い換えが有効
```

既存の1行（「成果物内に以下のパターンが〜」）を修正し、その直後に言い換え例の行を追加する。
2行の変更でsubagentへの伝達情報量を大幅に増やすことができる。

#### 角括弧プレースホルダー禁止サブセクション（SPEC-3追加対象）

```
### 角括弧プレースホルダー禁止
- 角かっこで囲まれた変数名等はプレースホルダーとして検出される
- 許可される角括弧表現: 「関連」「参考」「注」「例」「出典」の語を角かっこで囲んだもの
- コードブロック（バッククォート3つ区切り）内も検出対象のため、配列アクセス記法（先頭要素取得、インデックスアクセス等）は使用禁止。  ← SPEC-3で追加
- コード例では配列アクセスを「最初の要素」「N番目の要素」等の散文形式で説明するか、波かっこ記法を使用すること。  ← SPEC-3で追加
```

2行の追加は既存の許可リスト行の直後に配置する。
コードフェンス内も検出対象であるという事実と、具体的な代替手段の両方を
同一サブセクション内で提示することで、subagentが即座に対処できる。

## エラーメッセージ設計

`buildRetryPrompt()` はバリデーション失敗時にOrchestratorが呼び出す関数で、
エラーメッセージを解析して具体的な修正指示を生成する。
生成されたリトライプロンプトは、Taskツールの再起動時に新しいsubagentへ渡される。

### 既存のエラー種別認識パターン（修正前の11種類）

現在の条件分岐では以下のキーワードでエラーを判定している。

- 「禁止パターン」: 禁止語が検出された場合の汎用修正指示
- 「セクション密度」: 特定セクションの実質行数不足
- 「同一行」または「繰り返し」: 3回以上同一行が出現した場合
- 「必須セクション」: 必須見出しが欠落している場合
- 「行数不足」: 成果物全体の最低行数を下回った場合

これら以外にも多数の条件分岐が存在するが、bracketプレースホルダーエラーに対応する
専用の条件分岐が存在しないことが現状の問題点である。

### SPEC-4で追加するbracket専用条件分岐のフォーマット

```typescript
if (errorMessage.includes('プレースホルダー括弧') || errorMessage.includes('角かっこ') || errorMessage.includes('bracket')) {
  improvements.push('コードブロック内の角かっこ記法（配列アクセス等）を散文形式の説明または波かっこ記法に変更してください');
  improvements.push('コードフェンス内も角かっこプレースホルダー検出の対象であるため、配列先頭要素を取得する場合は .at(0) 等の代替表現を使用してください');
}
```

この条件分岐は既存の「禁止パターン」条件より前に配置する。
配置順序を優先することで、bracket関連エラーが発生した際に
より具体的な修正指示が先に生成されるようになる。

### リトライプロンプト全体のセクション構成

buildRetryPrompt()が生成するプロンプトは以下の構造を持つ。

1. リトライ回数を示すヘッダー（`# {フェーズ名}フェーズ（リトライ: {N}回目）`）
2. タスク情報（タスク名・出力先）
3. 前回のバリデーション失敗理由（コードブロックで引用）
4. 改善要求リスト（improvements配列の内容を箇条書きで展開）
5. 入力ファイルリスト
6. 作業内容の説明
7. 出力ファイルリスト
8. 成果物品質要件の再確認（buildPrompt()と同じ内容）
9. Bashコマンド制限の再確認

SPEC-4で追加される修正指示は第4セクション（改善要求リスト）に展開される。
bracket専用の改善要求が2行追加されることで、subagentはコードフェンス内の
角かっこ記法の問題を正確に把握できるようになる。

## APIレスポンス設計

`resolvePhaseGuide()` はフェーズ名を受け取り、そのフェーズの定義オブジェクトを返す関数である。
返り値の型は `PhaseGuide` インターフェースで定義されており、
フィールドの1つとして `subagentTemplate` を持つ。

### subagentTemplateフィールドの構造

`subagentTemplate` フィールドはstring型であり、フェーズ固有のプロンプト追加指示を格納する。
buildPrompt()の第9セクション（フェーズ固有の追加指示）の内容として使用される。

現在の実装では以下の3つのフィールドがPhaseGuideに定義されている。

- `minLines`: 成果物の最低行数（number型）
- `requiredSections`: 必須セクションのリスト（文字列配列型）
- `subagentTemplate`: フェーズ固有の追加指示テキスト（string型）

### 本タスクでのPhaseGuide変更有無

SPEC-1〜SPEC-4は全て `buildPrompt()` および `buildRetryPrompt()` の関数本体内の
テキストを直接修正する変更であり、`resolvePhaseGuide()` の返り値や
`PhaseGuide` インターフェースの構造には変更を加えない。

`subagentTemplate` フィールドの値は既存のフェーズ定義オブジェクトに保持されているが、
本タスクで追加する警告テキストは全フェーズ共通の品質ルールセクションに含まれるため、
フェーズ固有の `subagentTemplate` を修正する必要はない。

### resolvePhaseGuide()の返り値例

research フェーズを例として示す。

```typescript
{
  minLines: 50,
  requiredSections: 文字列配列でサマリーヘッダーを含むセクション名を指定,
  subagentTemplate: "調査結果を体系的にまとめ、次フェーズのrequirementsフェーズで\n使用できる情報を提供してください。"
}
```

本タスクの修正後も、この返り値の構造は変化しない。
buildPrompt()がこの返り値を受け取り、セクション8（品質要件）の生成時に
`minLines` と `requiredSections` を参照する構造は維持される。

## 設定ファイル設計

本タスクでは `GlobalRules` 型および関連する設定ファイルへの変更は不要である。
この判断の根拠を以下に示す。

### GlobalRules型への追加フィールドが不要な理由

`GlobalRules` 型は全フェーズで共通するルールを静的に定義するためのインターフェースである。
SPEC-1〜SPEC-3で追加する警告テキストは、既存の品質ルールセクションの補足説明であり、
新しいルールの追加ではなく、既存ルールの説明精度を高める変更として位置付けられる。
このため、`GlobalRules` 型に新しいフィールドを追加する必要はなく、
既存の文字列テンプレートリテラルの修正のみで実装できる。

### 設定ファイルの変更対象外リスト

以下のファイルは本タスクの影響を受けない設定ファイルである。

- `workflow-plugin/mcp-server/src/validators/bash-whitelist.js`: コマンド許可リストは変更なし
- `workflow-plugin/mcp-server/src/types.ts`: インターフェース定義は変更なし
- `workflow-plugin/mcp-server/package.json`: ビルドスクリプトは変更なし（参照のみ）
- `.claude/settings.json`: フック設定は変更なし

### ビルド設定に関する考慮事項

`npm run build` 実行後に生成される `dist/phases/definitions.js` は
TypeScriptのコンパイル成果物であり、設定ファイルではなく自動生成ファイルである。
このファイルはソース管理対象外であり、本UI設計の範囲外として扱う。
実装フェーズではビルド成功の確認をAC-3の受け入れ基準として検証する。

## インターフェース変更のまとめ

本タスクで変更されるインターフェースを一覧として整理する。

| 変更対象 | 変更種別 | SPEC番号 | 変更行数 |
|---------|---------|---------|---------|
| buildPrompt()の行数・密度要件テキスト | 1行追加 | SPEC-1 | +1行 |
| buildPrompt()の禁止パターンヘッダー行 | 1行修正 | SPEC-2 | ±0行 |
| buildPrompt()の禁止パターン言い換え例 | 1行追加 | SPEC-2 | +1行 |
| buildPrompt()の角括弧警告テキスト | 2行追加 | SPEC-3 | +2行 |
| buildRetryPrompt()のbracket条件分岐 | 4行追加 | SPEC-4 | +4行 |

合計8行の変更はNF-2（追加50行以内）の制約を十分に満たしている。
全ての変更は `definitions.ts` の1ファイルに集約されており、影響範囲が明確である。
