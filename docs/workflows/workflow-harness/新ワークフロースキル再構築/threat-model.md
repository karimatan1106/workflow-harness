## サマリー

- 目的: ワークフロースキル再構築タスク（SKILL.md および phases/ テンプレートファイル群の刷新）に伴う脅威を識別し、リスク評価と対策を策定する。
- 主要な決定事項: スキルファイルの変更はコードではなく Markdown ファイルの編集であるが、AIが参照するガイドラインとして機能するため、内容の不整合や情報漏洩がワークフロー誤動作に直結するリスクがある。脅威は5カテゴリに分類し、それぞれに緩和策を定義する。
- 次フェーズで必要な情報: 各脅威シナリオの優先度評価（Critical/High/Medium/Low）、セキュリティ要件の受入条件（AC番号との対応関係）、実装フェーズで考慮すべき禁止事項の一覧。

---

## 背景と対象範囲

本脅威モデルは「新ワークフロースキル再構築」タスクを対象とし、以下のファイル群の変更に伴うリスクを分析する。

- `workflow-plugin/skills/workflow/SKILL.md`: ユーザー向けコマンドガイド兼Orchestratorガイドの2層構造文書
- `workflow-plugin/skills/workflow/phases/` 配下の13種のフェーズテンプレートファイル（F-010〜F-022）
- `workflow-plugin/skills/project/SKILL.md`: `/project resume` 手順を含むプロジェクトスキル定義

分析対象外のファイルとして、`definitions.ts`（subagentTemplate の実質的プロンプト源泉）および MCP サーバーのコアロジックは本タスクのスコープ外である。

---

## 脅威シナリオ

### シナリオ T-001: フェーズ定義の不整合によるワークフロー誤動作

**分類**: 完全性の脅威（情報の欠落・矛盾）

SKILL.md に記載するフェーズ数や順序が CLAUDE.md の定義と乖離した場合、AIが誤ったフェーズ順序でワークフローを進行させる可能性がある。具体的には、regression_test フェーズの記載漏れにより、testing フェーズ完了後に直接 parallel_verification へ遷移してしまい、リグレッション確認が省略される。また、フェーズ間依存関係（planning は threat_modeling 完了を要する等）の記述が SKILL.md と CLAUDE.md とで矛盾した場合、subagent が誤った実行順序でタスクを進める可能性がある。フェーズ数の誤記（18フェーズと記載している既存問題）は、ユーザーが存在するフェーズを認識できない状態を生み出す直接的な原因となっている。

**影響範囲**: 全ワークフロータスクのフェーズ進行・承認プロセス

### シナリオ T-002: approve コマンド記述の不完全性による承認漏れ

**分類**: プロセス完全性の脅威（承認フローの迂回）

SKILL.md に `/workflow approve design` のみが記載されていた場合、requirements/test_design/code_review フェーズにおける承認ステップがユーザーに通知されない。その結果、AIが承認待ちを省略してフェーズを自動進行させる、またはユーザーが承認操作を知らずに誤ったタイミングで操作を行う可能性がある。特に `code_review` フェーズの承認省略は、設計と実装の整合性確認を経ずに testing フェーズへ進む重大な品質ゲートのバイパスとなる。また、`test_design` フェーズの承認省略はテスト方針の合意なしに実装を開始することに直結し、後の手戻りコストを増大させる。

**影響範囲**: requirements/design_review/test_design/code_review の4承認フェーズ

### シナリオ T-003: 存在しないコマンドへの誤誘導（workflow_switch 問題）

**分類**: 操作誤誘導の脅威（存在しない機能への参照）

SKILL.md に `/workflow switch <task-id>` が記載されているが、該当する MCP ツールは実装されていない。ユーザーまたは AI がこのコマンドを実行しようとした場合、MCP サーバーからエラーが返されるか、AIが別の類似操作（workflow_list + workflow_status 組み合わせ）を独自判断で試みる。後者の場合、想定外の操作シーケンスが発生し、タスク状態の整合性が損なわれる可能性がある。また、存在しないコマンドへの参照はスキルファイルの信頼性全体を低下させ、ユーザーが他のコマンド記述も正確でないと疑う状態を生む。

**影響範囲**: 複数タスク管理操作・タスク状態確認フロー

### シナリオ T-004: sessionToken 情報の不適切な取り扱い

**分類**: セキュリティ情報漏洩の脅威（認証トークンの誤公開）

SKILL.md および phases/ ファイルに sessionToken の二層構造ルールが記述されていない場合、AIが sessionToken を subagent に広く引き渡す誤動作を引き起こす可能性がある。具体的なリスクとして、sessionToken が subagent プロンプトとして記録されることで、ログや成果物ファイルに認証情報が混入するケースがある。さらに、SKILL.md の記述例やサンプルコードに実際の sessionToken 値が例示として記載されることを防ぐため、テンプレートには具体的なトークン値を一切記載してはならない。phases/ ファイルの記述でも、subagent が利用できるMCPツールの範囲を明示し、sessionToken を誤って workflow_next 等に渡さないよう注意事項を記述する必要がある。

**影響範囲**: testing フェーズおよび regression_test フェーズのsubagent実行

### シナリオ T-005: 古い情報の残存による Orchestrator の混乱

**分類**: 情報陳腐化の脅威（旧仕様との不整合蓄積）

phases/ ディレクトリの既存ファイル（requirements.md, implementation.md 等）が更新されず、旧仕様の記述が残存した場合、subagent が古いルール（承認フロー省略、設計チェックリスト未確認等）に従って動作する可能性がある。requirements.md に `workflow_approve type="requirements"` の呼び出し手順がない場合、requirementsフェーズの subagent がユーザー承認を省略して成果物を直接完了状態にする。implementation.md に設計チェックリスト5項目が記述されていない場合、spec.md/state-machine.mmd 等の読み込みを省略した不完全な実装が行われるリスクがある。また、F-006 で追加する Orchestrator パターンの説明が phases/ ファイルと矛盾した場合、subagent がどちらを優先すべきか判断できなくなる。

**影響範囲**: requirements, implementation フェーズ（既存ファイルの更新対象）

## リスク評価

| 脅威ID | 脅威名 | 発生可能性 | 影響度 | リスクレベル |
|--------|--------|-----------|--------|-------------|
| T-001 | フェーズ定義の不整合によるワークフロー誤動作 | 高（既に18/19フェーズ乖離が発生中） | 高（全タスクに影響する基盤的誤動作） | Critical |
| T-002 | approve コマンド不完全性による承認漏れ | 高（3種別の記載が既に欠落中） | 高（品質ゲートのバイパスに直結） | Critical |
| T-003 | 存在しないコマンドへの誤誘導 | 中（ユーザーが明示的に switch を使わない限り発現しない） | 中（エラー回復は可能だが信頼性が損なわれる） | High |
| T-004 | sessionToken 情報の不適切な取り扱い | 低（現行は発現していないが記述不在が潜在的リスク） | 高（認証情報の漏洩は機密性に関わる） | High |
| T-005 | 古い情報の残存による Orchestrator 混乱 | 中（既存ファイルの更新漏れが原因） | 中（特定フェーズの動作に限定される） | Medium |

### リスクレベルの定義

Critical（最優先対応）: 現在進行中の全ワークフロータスクに直接的な誤動作をもたらし、品質ゲートが機能しなくなるリスクを指す。T-001 と T-002 は既に問題が顕在化しており、本タスクの主要な修正対象である。

High（優先対応）: 発現した場合の影響は大きいが、発生条件が限定的であるか、または現時点で潜在的リスクにとどまっているケースを指す。T-003 は既存の誤記であり、T-004 は記述の欠落による潜在リスクである。

Medium（通常対応）: 特定のフェーズや操作に限定され、他の防御機構（CLAUDE.md のルール参照等）によって緩和可能なリスクを指す。T-005 は phases/ ファイルの参照頻度が低い場合は影響が限定される。

## セキュリティ要件

### SR-001: フェーズ定義の単一真実源（SSOT）の確立

SKILL.md のフェーズ記述は CLAUDE.md の定義を権威ある情報源（Single Source of Truth）として扱い、差異が生じた場合は SKILL.md 側を更新することを原則とする。具体的には、SKILL.md のフェーズ順序・フェーズ数・並列フェーズ構成が CLAUDE.md の「フェーズ順序」セクションと字句レベルで一致していることを確認してから実装フェーズを完了とする。フェーズ構成の変更が生じた場合は SKILL.md と CLAUDE.md を同一トランザクションで更新することを義務づける。これにより T-001 の継続的な発生を予防する。

### SR-002: 承認フロー記述の完全性保証

4種別の approve コマンド（requirements/design/test_design/code_review）が SKILL.md のコマンド一覧に全て記載されていることをレビュー時の必須確認事項とする。各 approve コマンドには対応するフェーズ名と「ユーザー承認を待機する」旨の説明が付記されていることを成果物の検収基準とする。AI に対しては「4フェーズでは必ず workflow_approve を呼び出す」ルールを厳命セクションに明示し、記述の省略を構造的に防ぐ。これにより T-002 の再発を防止する。

### SR-003: 存在しない操作の完全削除と代替説明の追加

SKILL.md から `/workflow switch` の記述を削除した後、その箇所に複数タスク管理の正しい代替手順（workflow_list で taskId を確認し、workflow_status で詳細を確認する）を補足説明として記述する。削除だけでなく代替手順を提供することで、ユーザーが目的の操作を達成できる状態を維持する。これにより T-003 で生じるユーザーの混乱を防止する。

### SR-004: sessionToken の取り扱い制限の明示

SKILL.md の sessionToken 記述において、実際のトークン値の記載は一切禁止とし、全ての記述例はプレースホルダー形式（例示目的の想定値を使用した説明文）で記述する。phases/ ファイルには、そのフェーズで subagent が呼び出しを許可される MCP ツールの一覧を明示し、sessionToken を渡してよいツールと渡してはいけないツールを区分する。セッション再開後の workflow_status 呼び出し義務と taskId 指定の必要性を SKILL.md の冒頭付近に記述し、AIが見落としにくい構造とする。これにより T-004 の潜在リスクを低減する。

### SR-005: 既存ファイルの更新と矛盾排除

phases/ ディレクトリの既存6ファイル（requirements.md, implementation.md 等）の内容は、新規作成する13ファイルと記述スタイル・ルール参照方式を統一する。requirements.md への workflow_approve 手順追記と implementation.md への設計チェックリスト追記は、SKILL.md への Orchestratorパターン追加と同一タスク内で完了させ、記述の矛盾が生じない状態で納品する。矛盾の検出は、implementation フェーズ完了後の code_review サブフェーズで phases/ 全ファイルを横断的に確認することで行う。これにより T-005 のリスクを管理する。

## 緩和策の優先度と実施タイミング

実施優先度 1（implementation フェーズ開始前に確認）として、SR-001 のフェーズ定義 SSOT 原則の適用と SR-002 の承認フロー完全記述の方針確認を実施する。これらは設計段階で確定しておく必要がある。

実施優先度 2（implementation フェーズ中の実装順序として）、まず T-001/T-002 に対応する SKILL.md の修正（F-001, F-002）を先行し、その後 T-003 の対応（F-003 削除 + 代替説明追加）を行う。sessionToken に関する SR-004 の記述は F-007 として実装するが、phases/ の新規ファイル作成（F-010〜F-022）の前に SessionToken ルールの文言を確定させることを推奨する。

実施優先度 3（code_review フェーズ中の検証として）、phases/ 全ファイルの横断レビューで矛盾が生じていないことを確認し、SR-005 の緩和策が適切に機能していることを確認する。
