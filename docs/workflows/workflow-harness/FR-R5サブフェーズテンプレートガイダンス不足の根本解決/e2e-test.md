## サマリー

このE2Eテストは、FR-R5A〜FR-R5Dとして定義された4サブフェーズのsubagentTemplateへのガイダンス文字列追加が、仕様書（spec.md）の要件を満たす形で実装されていることをエンドツーエンドで検証することを目的としている。
テスト対象は`workflow-plugin/mcp-server/src/phases/definitions.ts`の1ファイルのみで、code_review・manual_test・security_scan・e2e_testの4サブフェーズのsubagentTemplateプロパティを静的に読み取り、ガイダンスブロックの存在・構造・内容・禁止語非存在を確認した。
テスト実施は静的コンテンツ検証（Grepツールによるパターンマッチ）およびファイル読み取り（Readツール）の手法で実施し、Node.js実行環境またはブラウザは使用していない。
全5シナリオ（FR-R5Aセキュリティスキャン・FR-R5Bコードレビュー・FR-R5CE2Eテスト・FR-R5D手動テスト・統合一貫性確認）において合格を確認した。検出された問題点は存在しない。
次のdocs_updateフェーズおよびcommitフェーズに引き継ぐべき情報として、4サブフェーズのガイダンスが全て正しく実装されており、ビルドは別途MCPサーバーの再起動で確認済みであることを共有する。

## E2Eテストシナリオ

### シナリオ1: security_scanサブフェーズのガイダンス構造検証

このシナリオの目的は、FR-R5A仕様書が要求する3ブロック構成（サマリーガイダンス・脆弱性スキャン結果ガイダンス・検出された問題ガイダンス）がsecurity_scanのsubagentTemplateに追加されていることを確認することである。
検証手順として、definitions.tsの行902付近のsecurity_scan定義を読み込み、subagentTemplateに「## サマリーセクションの行数ガイダンス」「## 脆弱性スキャン結果セクションの行数ガイダンス」「## 検出された問題セクションの行数ガイダンス」の3見出しが含まれることをGrepで確認する。
期待される結果として、各ガイダンスブロックには5項目以上の必須記載事項リストが含まれており、NG例とOK例の対比が存在し、特に「検出された問題」ガイダンスには問題なし時でも5行確保するよう明示されていることが求められる。
実際の確認結果として、definitions.tsの行902のsubagentTemplateには3ブロック全てが含まれており、サマリーガイダンスには「スキャン対象範囲・使用ツール・検出件数・深刻度分布・総合評価」の5項目、スキャン結果ガイダンスには「実行コマンド・スキャン対象パス・実行日時・データベースバージョン・スキャン完了状態」の5項目が列挙されていることを確認した。
判定として合格とする。3ブロックが挿入位置（`## 出力`の直前）に正確に配置されており、仕様書FR-R5Aの受け入れ基準を満たしている。

### シナリオ2: code_reviewサブフェーズのガイダンス構造検証

このシナリオの目的は、FR-R5B仕様書が要求する5ブロック構成（サマリー・設計-実装整合性・コード品質・セキュリティ・パフォーマンスの各セクションガイダンス）がcode_reviewのsubagentTemplateに追加されていることを確認することである。
検証手順として、definitions.tsの行846付近のcode_review定義を読み込み、実際に発生した「サマリーセクションの実質行数が4行」という失敗を解消するガイダンスが存在することを重点的に確認する。具体的には「## サマリーセクションの行数ガイダンス」見出しと、「## 設計-実装整合性セクションの行数ガイダンス」「## コード品質セクションの行数ガイダンス」「## セキュリティセクションの行数ガイダンス」「## パフォーマンスセクションの行数ガイダンス」の4見出しを確認する。
期待される結果として、サマリーガイダンスにはレビュー対象ファイル数・設計整合性判定・主要指摘事項数・セキュリティリスク有無・パフォーマンス課題有無の5項目が含まれることが求められる。
実際の確認結果として、行846のsubagentTemplateに5ブロック全てが確認できた。サマリーガイダンスの5項目は全て含まれており、コード品質ガイダンスには「命名規則・型安全性・エラーハンドリング・DRY原則・テストカバレッジ」の5観点が、セキュリティガイダンスには「入力バリデーション・認証認可・機密情報・依存パッケージ・XSS/SQLインジェクション」の5観点が列挙されていることを確認した。
判定として合格とする。5ブロック全てが`## 出力`の直前に配置されており、仕様書FR-R5Bの全受け入れ基準を満たしている。

### シナリオ3: e2e_testサブフェーズのガイダンス構造検証

このシナリオの目的は、FR-R5C仕様書が要求する4ブロック構成（サマリーガイダンス・E2Eテストシナリオセクションガイダンス・テスト実行結果セクションガイダンス・重複行回避ガイダンス）がe2e_testのsubagentTemplateに追加されており、変更前の「E2Eテストを実施してください。」1文のみという状態から大幅に拡充されていることを確認することである。
検証手順として、definitions.tsの行926付近のe2e_test定義を読み込み、4ブロックの見出しである「## サマリーセクションの行数ガイダンス」「## E2Eテストシナリオセクションの行数ガイダンス」「## テスト実行結果セクションの行数ガイダンス」「## 重複行回避の注意事項」が全て存在することを確認する。
期待される結果として、テスト実行結果ガイダンスにはシナリオ番号または名称を行に含めて一意性を確保する指示が含まれており、重複行回避ガイダンスにはNG例（「合格」「パス」「成功」の繰り返し）とOK例（シナリオ名を含む形式）の対比が示されていることが求められる。
実際の確認結果として、行926のsubagentTemplateには4ブロック全てが含まれており、テスト実行結果ガイダンスには「シナリオ番号または名称を行に含めて一意性を確保する（一意化方法の基本原則として全行に適用する）」という一意化指示、および「シナリオ1（ログイン処理）の実行結果: 合格、認証トークンの発行を確認した」というOK例が明示されていることを確認した。
判定として合格とする。変更前の1文のみという状態から4ブロックに大幅拡充されており、仕様書FR-R5Cの全受け入れ基準を満たしている。

### シナリオ4: manual_testサブフェーズのガイダンス構造検証

このシナリオの目的は、FR-R5D仕様書が要求する2ブロック追加（テストシナリオセクションガイダンス・テスト結果セクションガイダンス）がmanual_testのsubagentTemplateに追加されており、かつ既存の「サマリーセクションの行数ガイダンス」と「重複行回避の注意事項」が維持されていることを確認することである。
検証手順として、definitions.tsの行890付近のmanual_test定義を読み込み、新規追加の「## テストシナリオセクションの行数ガイダンス」と「## テスト結果セクションの行数ガイダンス」の2見出しが存在すること、および既存の2ブロックが保持されていることを確認する。
期待される結果として、テストシナリオセクションガイダンスには「シナリオID・テスト目的・前提条件・操作手順・期待結果」の5要素が含まれ、1シナリオで最低5行の実質行となるよう明示する指示が存在することが求められる。また、テスト結果セクションガイダンスには「実行日時・実行環境・実際の結果・合否判定・発見された不具合（なければ「不具合なし」と明記）」の5項目が列挙されていることが求められる。
実際の確認結果として、行890のsubagentTemplateには2ブロックの新規追加が確認でき、テストシナリオガイダンスには5要素と「minSectionLines（5行）を確保する」という行数指示が、テスト結果ガイダンスには5項目全てが含まれていることを確認した。また、既存のサマリーガイダンスおよび重複行回避ガイダンスが変更されずに維持されていることも確認した。
判定として合格とする。2ブロックの追加と既存2ブロックの維持が確認でき、仕様書FR-R5Dの全受け入れ基準を満たしている。

### シナリオ5: 統合検証 - 全4サブフェーズのガイダンスパターン一貫性確認

このシナリオの目的は、全4サブフェーズのガイダンスがperformance_testで確立されたパターン（ガイダンスブロックのヘッダー形式・必須記載事項5項目以上のリスト・NG例とOK例の対比）に従っており、一貫した構造になっていることを確認することである。
検証手順として、definitions.tsのperformance_testのsubagentTemplate（行914）を参照モデルとして、4サブフェーズの各ガイダンスブロックが同一の構造パターン（「## {セクション名}セクションの行数ガイダンス」形式の見出し・「必ず5行以上の実質行」を含む指示文・5項目以上のリスト）に従っていることを確認する。また、禁止語（英語4語・日本語8語）が追加テキストに含まれていないこと、および角括弧プレースホルダーが使用されていないことを確認する。
期待される結果として、全4サブフェーズの追加ガイダンスブロックがperformance_testパターンと同一の構造を持ち、各ブロック内に5項目以上の具体的な記述指示が含まれており、禁止語・角括弧プレースホルダーが一切含まれていないことが求められる。
実際の確認結果として、4サブフェーズ全ての追加ガイダンスブロックに「必ず5行以上の実質行（コンテンツを含む行）を記述すること」という指示が含まれており、各セクションガイダンスには5項目以上のリストが記述されていることを確認した。performance_testで確立されたNG例・OK例の対比パターンも全サブフェーズに展開されていることを確認した。バリデーターが検出する12語の禁止語句（確定していない状態・検討継続中・例示専用データ・架空の値等を意味する語句）が追加テキストに含まれないこと、角括弧プレースホルダーが使用されていないことも確認した。
判定として合格とする。全4サブフェーズのガイダンスがperformance_testパターンと構造的に一貫しており、仕様書の「共通実装パターン」要件を満たしている。

## テスト実行結果

テスト実施日時は2026年2月19日であり、実施環境はWindows (MSYS_NT-10.0-26100) 上のClaude Code Sonnet環境で、Readツール・Grepツールによる静的コンテンツ検証を実施した。

シナリオ1（security_scanサブフェーズ検証）の実行結果: 合格。definitions.tsの行902のsubagentTemplateに3ブロック（サマリーガイダンス・脆弱性スキャン結果ガイダンス・検出された問題ガイダンス）が全て確認でき、各ブロックに5項目の記述指示とNG/OK例の対比が存在した。`## 出力`の直前への正確な挿入が確認できた。

シナリオ2（code_reviewサブフェーズ検証）の実行結果: 合格。definitions.tsの行846のsubagentTemplateに5ブロック（サマリー・設計-実装整合性・コード品質・セキュリティ・パフォーマンスの各セクションガイダンス）が全て確認でき、実際に発生していた「サマリーセクションの実質行数が4行」という失敗を防ぐ5項目ガイダンスが存在することを確認した。

シナリオ3（e2e_testサブフェーズ検証）の実行結果: 合格。definitions.tsの行926のsubagentTemplateが変更前の1文のみから4ブロックに拡充されており、シナリオ識別子を行に含める一意化指示とシナリオ名を含むOK例（ログイン処理・検索機能の具体例）が存在することを確認した。

シナリオ4（manual_testサブフェーズ検証）の実行結果: 合格。definitions.tsの行890のsubagentTemplateに新規2ブロック（テストシナリオガイダンス・テスト結果ガイダンス）が追加されており、既存のサマリーガイダンスと重複行回避ガイダンスが内容変更なく維持されていることを確認した。4ブロックの順序が「重複行回避 → サマリー → テストシナリオ → テスト結果 → 出力」という正しい構造になっていることも確認した。

シナリオ5（統合一貫性検証）の実行結果: 合格。全4サブフェーズの追加ガイダンスブロックがperformance_testのパターンと構造的に一致していることを確認し、禁止語・角括弧プレースホルダーも含まれていないことを検証した。requiredSections・minLines・minSectionLinesの設定値が変更されていないことも確認し、後方互換性が完全に維持されていることを確認した。

検出された問題点として、全5シナリオを通じて仕様書の要件から逸脱した箇所は発見されなかった。後方互換性（requiredSections・minLines・minSectionLinesの変更なし）も確認済みである。

総合判定として全シナリオが合格であり、FR-R5A〜FR-R5Dの4サブフェーズへのガイダンス追加はすべての受け入れ基準を満たしていると判断する。
改善提案として、今後のワークフロー改善においてperformance_testガイダンスパターンを新たなサブフェーズのsubagentTemplate作成時のテンプレートとして活用することを推奨する。
