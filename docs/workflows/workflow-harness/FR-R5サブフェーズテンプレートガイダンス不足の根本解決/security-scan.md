- スキャン対象の変更概要: `workflow-plugin/mcp-server/src/phases/definitions.ts` の4サブフェーズ（security_scan・code_review・e2e_test・manual_test）のsubagentTemplateプロパティに、セクション別行数ガイダンス文字列を追記した。変更はテンプレートの静的文字列拡充のみであり、バリデーションロジック・フェーズ遷移ロジック・HMAC計算対象には手を加えていない。
- スキャン手法の説明: definitions.tsの実装コードを直接読み込んで静的解析を実施した。resolvePlaceholders関数の挙動・computeStateIntegrityの署名対象・requiredSections配列との整合性・禁止語混入有無・波括弧プレースホルダー表記の適切性を各項目ごとに個別検証した。
- 検出された脆弱性の有無と件数: 深刻な脆弱性は0件であった。軽微な運用上の注意事項として「MCPサーバーキャッシュ未反映リスク」と「禁止語混入防止の継続的確認」の2点が特定された。これらはいずれもCLAUDE.mdのルール22と禁止語ガイダンスで対処方法が既定されており、新たなセキュリティホールには該当しない。
- リスク評価の総合判定: 今回の変更は動的な外部入力経路を追加しない静的文字列の拡充であるため、テンプレートインジェクション・HMAC整合性破壊・後方互換性破壊のリスクは低く抑えられている。総合判定は「セキュリティ上の重大な問題なし、条件付き合格」とする。条件とはビルド後の再起動手順の遵守である。
- 次フェーズへの引き継ぎ事項: implementationフェーズ完了後にnpm run buildによるコンパイルを実行し、MCPサーバーを再起動してから次フェーズに進むこと。再起動確認にはworkflow_statusの実行を必須とする。追加したガイダンス文字列にはNG/OK例が含まれており、subagentが例文を成果物に直接転記しないよう留意すること。

## 脆弱性スキャン結果

スキャン対象ファイルとスキャン範囲は `workflow-plugin/mcp-server/src/phases/definitions.ts` の1ファイルのみである。本ファイルのうち、今回変更された4か所のsubagentTemplate文字列プロパティ（security_scan・code_review・e2e_test・manual_test、行846・902・926・890付近）を対象として、静的コードレビュー手法によるセキュリティ検査を実施した。

テンプレートインジェクション検査の結果詳細として、resolvePlaceholders関数の正規表現パターン `\$\{${key}\}` はdocsDir変数とuserIntent変数のみを対象として置換する設計となっていることを確認した。今回追加したガイダンス文字列には `${docsDir}` と `${userIntent}` のプレースホルダーは含まれておらず、ガイダンスブロック内のテキストは置換対象にならないことを確認した。ガイダンス文字列はJavaScript文字列リテラルとして静的に記述されており、実行時に外部入力値が動的に注入される経路はコードベース上に存在しない。

HMAC整合性への影響分析結果として、state-manager.tsの整合性計算はワークフロー状態データ（phase・taskId・sessionToken・timestamp）を署名対象としており、definitions.tsのsubagentTemplate文字列はHMAC-SHA256の計算対象に含まれないことをコードベースの調査で確認した。今回の変更はworkflow-state.jsonのstateIntegrityフィールドに影響を与えず、既存のすべてのワークフロー状態ファイルはそのまま有効な署名を維持する。

禁止語・プレースホルダー検査の結果として、追加したガイダンス文字列に対して英語4語および日本語8語からなる計12語の禁止パターンの混入がないことを文字列検索で確認した。対象の禁止語句はCLAUDE.mdの成果物品質要件に列挙されている語句群である。角括弧プレースホルダーについても、追加したガイダンス文字列全体にわたって波括弧表記が採用されていることを確認した。NG/OK例の中にも角括弧プレースホルダーは使用されておらず、すべて日本語テキストの括弧（「」）で記述されている。

後方互換性検査の結果として、security_scanのrequiredSectionsは `['## 脆弱性スキャン結果', '## 検出された問題']` であることをコードの行895で確認した。追加したガイダンスブロックの見出しは「## 脆弱性スキャン結果セクションの行数ガイダンス」「## 検出された問題セクションの行数ガイダンス」であり、requiredSections配列の値とは異なる見出し名を使用している。これは意図的な設計であり、ガイダンスセクション自体をバリデーション対象の必須セクションとして誤認させることはない。subagentへの指示文中には `「## 脆弱性スキャン結果」` および `「## 検出された問題」` の正確なセクション名が明記されており、subagentが正しいセクション名で成果物を生成することを誘導する。

## 検出された問題

テンプレートインジェクション脆弱性の検査では問題は検出されなかった。ガイダンス文字列はサーバー起動時にメモリ上に固定されるJavaScript文字列定数であり、実行時に外部クライアントやユーザー入力が文字列を操作できる経路は存在しない。resolvePlaceholdersの置換パターンは2変数（docsDir・userIntent）に厳密に限定されており、ガイダンスブロック内の波括弧表記は置換パターンと一致しないため誤展開は発生しない。

HMAC整合性への影響について問題は検出されなかった。state-manager.tsの署名計算対象はワークフロー状態データのみに限定されており、definitions.tsの文字列変更はHMAC値に影響しないことが確認された。既存ワークフローの状態ファイルはすべて引き続き有効であり、変更の前後で署名の検証結果に差異は生じない。

禁止語・プレースホルダー違反については問題は検出されなかった。追加したガイダンス文字列の全文にわたって禁止12語の混入はなく、角括弧プレースホルダー表記も使用されていないことを確認した。NG/OK例の記述も禁止語を含まない自然な日本語表現を採用しており、subagentが例文を成果物に転記した場合でもバリデーション失敗を引き起こさない設計となっている。

後方互換性の検査では、軽微なリスク指摘が1件特定された。e2e_testサブフェーズのガイダンスに追加した「## 重複行回避の注意事項」という見出しは、requiredSectionsには含まれないため、subagentが成果物にこのセクションを出力した場合でもバリデーション処理上は無視される。一方、subagentがガイダンスセクションの見出しを成果物の本文セクションとして誤認し、E2Eテストの必須セクション（「## E2Eテストシナリオ」「## テスト実行結果」）を省略するリスクは排除できない。ただし、ガイダンス見出しは「## XXXの行数ガイダンス」「## 重複行回避の注意事項」という補助的な表現であり、必須セクション名と明確に区別できるため、このリスクは軽微と判定する。

残存リスクの評価として、MCPサーバーキャッシュ問題（中リスク）が1件存在する。実装後にnpm run buildでコンパイルを実施し、MCPサーバーを再起動しなければ変更が有効にならない。この制約はCLAUDE.mdのルール22で対処方法が既定されているが、手順を省略すると古いテンプレートがsubagentに渡り続けるため、本タスクの改善効果が確認できない状態が継続する。このリスクは手順の遵守によって完全に回避可能である。

総合的なセキュリティ判定として、今回の変更はセキュリティ上の重大な問題なく安全に実施できる変更と判定する。テンプレートインジェクション・HMAC整合性破壊・後方互換性破壊の3観点はすべて問題なしと評価された。MCPサーバー再起動の手順遵守を条件として、変更の適用を推奨する。
変更前後のdefinitions.tsの行数比較により、追加されたガイダンス文字列は既存の4サブフェーズテンプレートにのみ追記されており、他のフェーズ定義（research・requirements・planning等）には影響が及んでいないことを最終確認した。
