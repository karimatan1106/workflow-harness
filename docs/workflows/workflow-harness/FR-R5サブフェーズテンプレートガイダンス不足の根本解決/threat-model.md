## サマリー

- 対象変更: definitions.tsの4サブフェーズ（security_scan、code_review、e2e_test、manual_test）のsubagentTemplateにセクション別ガイダンス文字列を追記する
- 変更の性質: サーバー起動時に静的に読み込まれる定数文字列への追記であり、外部からの動的入力を含まない
- セキュリティ評価: 今回の変更はテンプレートインジェクション・HMAC整合性・後方互換性に対して新たなリスクを導入しない
- 主要な運用リスク: MCPサーバーキャッシュによる変更未反映（中リスク）と禁止語混入によるバリデーション失敗（中リスク）の2点が実施すべき対策の対象となる
- 次フェーズへの引き継ぎ: planningフェーズのspec.mdにはMCPサーバー再ビルド・再起動手順を必須事項として明記し、implementationフェーズでの手順漏れを防ぐこと

## 変更概要

今回の変更は`workflow-plugin/mcp-server/src/phases/definitions.ts`の1ファイルのみに対して行われ、4サブフェーズのsubagentTemplateプロパティにガイダンス文字列を追加する。
変更対象のサブフェーズはparallel_verificationフェーズを構成するsecurity_scan・code_review・e2e_test・manual_testの4つであり、artifact-validatorが要求するセクション別最低行数（minSectionLines: 5行）を満たすよう誘導する文章を各テンプレートに付加する。
変更するプロパティはsubagentTemplateのみであり、requiredSections配列・minLines・minSectionLines・フェーズ遷移ロジックには一切手を加えない。
変更後のファイルはTypeScriptのコンパイル手順（npm run build）が必要であり、dist/phases/definitions.jsを更新後にMCPサーバーを再起動することで有効になる。
既存のパフォーマンスや動作中のワークフロー状態には影響せず、新規ワークフロー起動時から新しいガイダンスが適用される設計となっている。

## 脅威

### リスク1: テンプレートインジェクション（低リスク）

subagentTemplateに追加されるガイダンス文字列は、開発者が静的に記述した固定テキストであり、実行時に外部から動的に変更される経路が存在しない。
resolvePlaceholders関数はuserIntentを特定のプレースホルダーに置換するが、追加ガイダンス内にそのプレースホルダーを含める設計はなく、ガイダンス文字列とuserIntentは独立した箇所に配置される。
subagentはLLMモデルであるため元来プロンプトインジェクションの潜在的対象ではあるが、今回の変更は固定のガイダンス文字列を増やすだけであり、新たな動的入力経路を開くものではない。
対策として、ガイダンス文字列に波括弧構文のプレースホルダーを記述する場合はresolvePlaceholdersが誤展開しない固定値として記述することを徹底する。

### リスク2: HMAC整合性への影響（影響なし）

state-manager.tsのcomputeStateIntegrity関数はワークフロー状態データ（phase・taskId・sessionToken・timestamp）をHMAC-SHA256の計算対象としており、definitions.tsのテンプレート文字列は署名対象に含まれていない。
コードベース調査によりこの事実が確認されており、今回の変更はworkflow-state.jsonのstateIntegrityフィールドに影響を与えず、既存のワークフロー状態ファイルはそのまま有効である。

### リスク3: 後方互換性の破壊（低リスク）

追加するガイダンス文字列は、artifact-validatorが参照するrequiredSections配列とは独立して管理されており、テンプレート文字列の追加がバリデーション要件そのものを変更することはない。
ただし、ガイダンス文字列に誤ったセクション名（例: 「## 検出された脆弱性」と記述すべき箇所を「## 発見された問題」と記述する）を含めた場合、subagentが誤ったセクション構造の成果物を生成しリトライが増加するリスクがある。
対策として、ガイダンス文字列に記載するセクション名はrequiredSections配列の値と完全一致させること。security_scanのrequiredSectionsは「## 脆弱性スキャン結果」「## 検出された問題」であり、ガイダンスではこれらの名称をそのまま使用する必要がある。

### リスク4: MCPサーバーキャッシュによる変更未反映（中リスク）

Node.jsのrequire()はモジュールをグローバルキャッシュに保存するため、ディスク上のdist/phases/definitions.jsを変更しても実行中のMCPサーバープロセスには変更が反映されない。
definitions.tsを修正してnpm run buildを実行した後にMCPサーバーを再起動しない場合、古いテンプレートでsubagentが起動し続け、本タスクの改善効果が確認できない状態が継続する。
対策として、implementationフェーズ終了後に必ずnpm run buildでTypeScriptをコンパイルしdist/phases/definitions.jsを更新し、その後Claude Desktopのサーバー再起動ボタンまたはプロセス終了でMCPサーバーを再起動する。
再起動後はworkflow_statusを実行して現在のフェーズを確認し、同フェーズから作業を再開することでデータ損失を防ぐ。
このリスクはCLAUDE.mdのルール22で対処方法が規定されており、本タスクでも同規則を適用する。

### リスク5: 禁止語・プレースホルダー違反のガイダンス文字列混入（中リスク）

artifact-validatorは成果物の本文に対して英語4語・日本語8語の合計12語の禁止語チェックを実施する。
ガイダンス文字列に禁止語や角括弧プレースホルダーが含まれていた場合、subagentがガイダンスの記述例をそのまま成果物に転記し、バリデーション失敗を引き起こすリスクがある。
performance_testのsubagentTemplateを参照すると、NG/OK例のような記述形式が既に採用されており、subagentが指示形式例を成果物に出力する現象は実際に観察されている。
対策として、追加するガイダンス文字列自体に禁止語を含めないこと、角括弧プレースホルダーを使用せず波括弧構文か散文による説明を採用すること、NG/OK例の記述も禁止語を含まない表現で行うことを徹底する。

## 対策サマリー

本変更に対して大規模なセキュリティ対策は不要であるが、以下4点の運用上の注意事項を実施すること。

- ガイダンス文字列に記載するセクション名はrequiredSections配列の値と完全一致させ、後方互換性を維持する
- ガイダンス文字列内に禁止語（英語4語・日本語8語の合計12語）や角括弧プレースホルダーを含めない
- ガイダンス文字列に波括弧構文を使う場合はresolvePlaceholdersが誤展開しないよう固定値として記述する
- 変更後はnpm run buildによるコンパイルとMCPサーバー再起動を確実に実施する（CLAUDE.mdルール22準拠）

## 結論

本変更はサーバー起動時に静的に読み込まれる定数文字列への追記のみであり、動的な入力経路を新たに追加するものではないため、テンプレートインジェクションおよびHMAC整合性への影響はない。
後方互換性はrequiredSections配列と矛盾しないセクション名を使用することで設計上維持される。
MCPサーバーキャッシュと禁止語混入の2点は中リスクとして識別されたが、いずれも実施すべき手順が明確であり、手順遵守によって確実に回避可能である。
planningフェーズでは、MCPサーバー再ビルドおよび再起動の手順をspec.mdのデプロイ手順セクションに必須事項として明記することで、implementationフェーズでの手順漏れリスクをさらに低減できる。
本脅威モデリングの結論として、今回の変更はセキュリティ上の重大な懸念なく安全に実施できると判断する。
