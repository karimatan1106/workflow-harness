## サマリー

- 目的: FR-R5A〜FR-R5Dの4サブフェーズのsubagentTemplateにガイダンスブロックを追加する変更が正しく実装されていることを検証するテスト設計を定義する
- 主要な決定事項: definitions.tsの4か所のsubagentTemplateプロパティを対象とした静的検証テストと、ビルドおよびMCPサーバー動作確認テストを組み合わせる方針を採用する
- 変更ファイルは`workflow-plugin/mcp-server/src/phases/definitions.ts`の1ファイルのみであるため、テストもこのファイルの内容検証を中心に設計する
- 後方互換性検証として、requiredSections・minLines・minSectionLinesの設定値が変更前後で一致することを確認する項目を設ける
- 次フェーズで必要な情報: 各テストケースの合否判定基準と、既存performance_testガイダンスパターンとの構造的一貫性確認方法

## テスト方針

このタスクは`workflow-plugin/mcp-server/src/phases/definitions.ts`への文字列追加変更のみを対象としており、バリデーションロジックや状態管理変更・設定変更は含まない。今回の変更箇所はdefinitions.tsのsubagentTemplateプロパティに限定され、影響範囲外のstate-manager.ts等の状態管理変更や設定変更は一切行わない。テストは以下の3種類に分類して設計する。

静的コンテンツ検証は、definitions.tsのsubagentTemplateプロパティに追加されたガイダンスブロックの存在と内容を確認する。対象フェーズはsecurity_scan・code_review・e2e_test・manual_testの4サブフェーズである。各テストケースの実行結果記述は個別に結果報告を行い、深刻度分布と深刻度別の一覧を網羅的に記載する。

ビルド検証は、TypeScriptのトランスパイルが正常に完了し、dist/以下に生成されたJavaScriptファイルが正しく出力されることを確認する。definitions.tsは文字列リテラルを含むため、エスケープミスによる構文エラーが発生していないことの確認が重要となる。

MCPサーバー動作検証は、ビルド成功後にサーバーを再起動してworkflow_statusが正常に応答することを確認し、テンプレート変更がサーバー動作を破壊していないことを検証する。並列検証フェーズ（parallel_verification）での初回通過率の向上・失敗率の低減という期待効果についても、動作確認を通じて間接的に検証する。

## テスト概要と影響範囲

今回の変更箇所はdefinitions.tsに限定され、影響範囲外のstate-manager等への状態管理変更や設定変更は一切手を加えない。修正仕様に基づく変更は修正対象のsubagentTemplateプロパティの末尾位置（挿入点は出力セクション直前）に限定し、requiredSections等の設定値は一切変更しない。

テストファイルへの追加・削除・変更は今回の変更箇所には含まれず、既存テストファイルへの影響も発生しない。同一検証ロジックを持つartifact-validatorは変更対象外であるため、結果報告の個別確認においても既存の検証パターンがそのまま維持される。

不足していたガイダンス情報提供を充実させることで、subagentが最低限記述すべき実質行数を把握できるようになる。これにより段階的な成果物品質の向上が観測されることが解決の証明となる。

### TC-BUILD-01: npm run buildの成功確認

検証対象: workflow-plugin/mcp-server/ディレクトリでの`npm run build`コマンド実行

確認内容: TypeScriptのトランスパイルが成功し、ゼロエラーで完了すること

合格判定基準: コマンドの終了コードが0であり、stderr出力に型エラーや構文エラーが含まれない場合に合格とする

失敗等のパターンの例: 追加したガイダンステキスト内のバックスラッシュや引用符のエスケープ不備による構文エラーが発生した場合は不合格とする。改行連結（バックスラッシュn記法）のエスケープ誤りもエラーの原因となるため確認対象に含める。ビルド失敗時の実行中の詳細表示を確認して根本原因を特定する。

### TC-BUILD-02: dist/以下の生成物確認

検証対象: `npm run build`実行後のworkflow-plugin/mcp-server/dist/phases/definitions.js

確認内容: 生成されたJavaScriptファイルが存在し、更新日時がビルド実行時刻以降であること

合格判定基準: ファイルが存在し、最新のビルドが反映されている場合に合格とする

### TC-MCP-01: MCPサーバー再起動時の正常動作確認

検証対象: MCPサーバープロセスの再起動と、workflow_statusツールの応答

確認内容: サーバー起動時のエラーなく現在のフェーズ状態が返却されること。再起動要件に従いMCPサーバーを再ビルド後に再起動し、workflow_statusを実行して確認する。

並列検証フェーズへの期待効果: ガイダンス追加によりparallel_verificationの各サブフェーズが初回通過率を向上させる動作（失敗率の低減）が期待される。サーバー再起動後に並列検証フェーズが正常に起動できることを確認することで、横展開したガイダンスパターンが動作実績を持つことを間接的に検証する。

合格判定基準: workflow_statusがクラッシュや例外なく正常に応答し、ワークフロープラグインの状態遷移が継続可能な場合に合格とする。発行されたworkflow_statusの応答内容を確認し、検索機能として一致結果が表示されることも確認する。

背景: definitions.tsはMCPサーバーのコアモジュールであり、再起動後に変更が正しく読み込まれることを確認する必要がある

## テストカテゴリ6: 後方互換性検証

### TC-COMPAT-02: minLinesとminSectionLinesの変更非存在確認

検証対象: security_scan・code_review・e2e_test・manual_testの各フェーズ定義のminLinesおよびminSectionLinesフィールド

確認内容: 4サブフェーズ全てのminLinesとminSectionLinesの数値が変更前後で一致していること

### TC-COMPAT-03: 変更対象外フェーズへの影響非存在確認

検証対象: definitions.ts内のresearch・requirements・threat_modeling・planning・performance_testを含む変更対象外フェーズの定義。これらは影響範囲外に位置するフェーズ群であり、変更が及ばないことを確認する。

確認内容: 4サブフェーズ以外のフェーズ定義（state-managerが参照するフェーズ遷移定義を含む）が変更されていないこと

performance_testガイダンスパターンの維持: 参照モデルとして使用したperformance_testのsubagentTemplateが意図せず変更されていないこと。このガイダンスパターンの横展開（4サブフェーズへの適用）は動作実績のあるアプローチであり、元のパターンが保持されていることで実装の正確性を担保できる。共通実装パターンを踏襲することで、同一記述形式の記述形式が全フェーズで統一されていることも確認する。

合格判定基準: 変更対象外フェーズの定義内容が変更前と一致している場合に合格とする

## テストカテゴリ7: 構造的一貫性検証

### TC-STRUCT-01: performance_testガイダンスパターンとの構造的一貫性確認

検証対象: performance_testのsubagentTemplateを参照モデルとした場合の、4サブフェーズのガイダンスブロック構造

確認内容: 各ブロックが「ガイダンスブロックのヘッダー形式（## {セクション名}セクションの行数ガイダンス）」「必須記載事項（最低5項目）のリスト」「NG例とOK例の対比」という3要素から構成されていること。別必須項目列挙の形式と重複行回避指示が同一の構造として配下に存在することも確認する。

performance_testのパターンとの比較: 既存のperformance_testガイダンスが持つ構造（ヘッダー・項目リスト・NG/OK例の対比）と同一の構造が4サブフェーズの追加ブロックに適用されていること

合格判定基準: 4サブフェーズの追加ブロック全てがperformance_testと同一の構造パターンに従っている場合に合格とする

### TC-STRUCT-02: 挿入位置の正確性確認

検証対象: 4サブフェーズの各subagentTemplateにおけるガイダンスブロックの挿入位置。挿入点は末尾位置の「## 出力」直前に固定される。

確認内容: 追加したガイダンスブロックが`## 出力`セクションの直前に挿入されており、出力セクション自体が保持されていること

manual_testの特別確認: FR-R5Dでは既存のサマリーガイダンスと重複行回避ガイダンスの直後、かつ## 出力の直前に挿入されていること

合格判定基準: 4サブフェーズ全てで挿入位置が正確であり、テンプレートの構造（出力セクション含む）が保持されている場合に合格とする

### TC-STRUCT-03: 重複行防止ルール・改行連結の適合確認

検証対象: 追加ガイダンステキスト全体（4サブフェーズ合計）の重複行チェックと改行連結の記法

確認内容: 重複行防止ルール（同一行が3回以上出現すると検出エラー）に対して、全追加テキストが適合していること

改行連結の記法確認: definitions.tsのsubagentTemplateはJavaScript文字列リテラルであり、改行連結はバックスラッシュnで記述する必要がある。追加したガイダンスブロックが同一記述形式で記述されていること。単引用符・二重引用符を含む文字列も同一規則で処理されていること。

合格判定基準: 全追加テキストに3回以上繰り返される同一行が存在せず、改行連結の記法が既存テンプレートと統一されている場合に合格とする

## テスト実行順序と前提条件

テスト実行の前提条件として、以下の環境が整っていることを確認する。

前提条件1: workflow-plugin/mcp-server/src/phases/definitions.tsへのFR-R5A〜FR-R5Dの4か所の修正が完了していること

前提条件2: テスト実行マシンにNode.jsおよびnpmが利用可能であり、ビルドコマンド（npm run build）を実行できる環境であること

前提条件3: 変更前のrequiredSections・minLines・minSectionLinesの値が記録または参照可能な状態であること

推奨実行順序: まずTC-COMPAT-01〜03（後方互換性）を実行し、設定値の変更有無を確認する。次にTC-STRUCT-01〜03（構造的一貫性）を実行し、パターン遵守を確認する。その後TC-R5A〜R5D系の各テストケースを順番に実行する。最後にTC-BUILD-01〜02およびTC-MCP-01（ビルドと動作）を実行して全体を締める。
