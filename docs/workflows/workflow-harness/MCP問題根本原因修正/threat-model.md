# MCP問題根本原因修正 - 脅威モデル

## サマリー

workflow-pluginの6件の修正に対する脅威モデルを作成した。
BUG-3のファイル操作許可追加が最も高リスクであり、パストラバーサル攻撃の可能性がある。
SEC-ENV-1の正規表現バイパスはエスケープ文字や引用符による回避が考えられる。
SEC-LOG-1のマスキング漏れは未知のパターンの機密情報に対するリスクがある。
BUG-2の密度閾値変更は品質低下リスクがあるが影響は限定的である。
SEC-TIME-1のタイミング攻撃対策はベストプラクティスの適用であり新たなリスクは低い。
各脅威に対する対策を定義し、実装時の注意事項を記載した。

## 脅威分析

### T-1: パストラバーサル攻撃 (BUG-3関連)

BUG-3でmkdirとdocs/workflows/パスのバイパスを追加する際にパストラバーサルのリスクがある。
docs/workflows/../../../etc/ のような相対パスで任意ディレクトリにアクセスされる可能性がある。
対策としてpath.normalize()による正規化とプレフィックスの厳密な照合を実装する。
シンボリックリンクによるバイパスも考慮し、fs.realpathSync()でリンク解決後のパスを検証する。
影響度: 高（任意ディレクトリへの書き込みが可能になる可能性）。

### T-2: 正規表現バイパス (SEC-ENV-1関連)

SEC-ENV-1のインライン環境変数検出は正規表現で実装するため、バイパスの可能性がある。
引用符付き（HMAC_STRICT="false" node app.js）やバックスラッシュエスケープが考えられる。
対策として引用符付き形式も検出する拡張正規表現を使用する。
環境変数名の大文字小文字のバリエーション（Hmac_Strict等）にも対応する。
影響度: 中（セキュリティ設定の無効化が可能になる）。

### T-3: ログマスキング漏れ (SEC-LOG-1関連)

SEC-LOG-1のマスキングパターンに含まれない形式の機密情報が漏洩する可能性がある。
例えばBase64エンコードされたトークンやJSON形式の認証情報が対象外になりうる。
対策としてマスキングパターンを拡張可能な設計にし、後から追加できるようにする。
ログファイル自体のアクセス権限も0600に設定することを推奨する。
影響度: 中（機密情報の部分的な漏洩リスク）。

### T-4: 密度閾値悪用 (BUG-2関連)

密度計算式の変更により、低品質な成果物がバリデーションを通過するリスクがある。
構造行を分母から除外することで実質的な閾値が緩和される。
対策としてminSubstantiveLinesチェック（最小5行）を維持して最低限の品質を保証する。
影響度: 低（成果物品質の軽微な低下の可能性）。

### T-5: タイミング攻撃の残存 (SEC-TIME-1関連)

timingSafeEqual()導入前のlengthチェックでトークン長が漏洩する可能性がある。
ただしトークンは固定長64文字であり、長さ情報の漏洩は実質的なリスクにならない。
対策として固定長チェック（token.length !== 64）を先に行い、長さ情報を既知にする。
影響度: 極低（実践的な攻撃シナリオが存在しない）。

## リスク評価

T-1パストラバーサルは最も高リスクであり、validateMkdirTarget()関数のパス正規化が必須である。
T-2正規表現バイパスは中リスクであり、複数の引用符形式に対応する必要がある。
T-3ログマスキング漏れは中リスクであり、拡張可能な設計が重要である。
T-4密度閾値とT-5タイミング攻撃残存は低リスクであり、既存の補助チェックで十分対策される。
全体として修正による新たなリスクは管理可能な範囲であり、対策の実装で十分に軽減できる。

## 対策

M-1: validateMkdirTarget()でpath.normalize()とプレフィックス完全一致照合を実装する。
M-2: 引用符付き形式（VAR="value" cmd、VAR='value' cmd）も検出する正規表現を使用する。
M-3: maskSensitiveInfo()のパターンを配列で管理し、後から追加可能な設計にする。
M-4: minSubstantiveLinesチェックを維持し、密度閾値のデフォルト値0.3を変更しない。
M-5: トークン長チェックを先に行い、timingSafeEqual()はバッファ長一致時のみ実行する。
全ての対策について実装時のテストケースを定義し、脅威シナリオの検証を行う。
