# ワークフロー並列タスク対応 - 脅威モデル

## 概要

ワークフローシステムの並列タスク対応における潜在的な脅威と対策を分析する。

## 資産

| 資産 | 説明 | 重要度 |
|-----|------|-------|
| タスク状態 | 各タスクの進行状態（フェーズ、履歴等） | 高 |
| 成果物 | docs/workflows/ 配下のドキュメント | 高 |
| ソースコード | 実装対象のコード | 高 |
| 設定ファイル | workflow-state.json等 | 中 |

## 脅威分析（STRIDE）

### T1: 状態の競合（Tampering）

**脅威**: 複数タスクが同時に状態ファイルを更新し、データ競合が発生する

**影響**: 低
- 現実的なシナリオでは、ユーザーは一度に一つのツールを呼び出す
- MCPサーバーは同期的に処理される

**対策**:
- ファイル書き込みは同期処理（現状維持）
- 将来的にロック機構を検討（現時点では不要）

### T2: ファイルパス推論の誤り（Information Disclosure）

**脅威**: hookがファイルパスを誤ったタスクに関連付け、意図しないフェーズルールが適用される

**影響**: 中
- 誤ったタスクのフェーズでチェックされ、本来許可される操作がブロックされる

**対策**:
- docsDir/workflowDir のプレフィックスマッチを厳密に行う
- 複数タスクにマッチする場合は最長一致を選択
- デバッグログで推論結果を確認可能にする

### T3: タスクID偽装（Spoofing）

**脅威**: 存在しないtaskIdや他人のtaskIdを指定して操作を行う

**影響**: 低
- 単一ユーザー環境を想定
- taskIdはディレクトリから検証される

**対策**:
- taskIdの存在検証を必ず行う
- 存在しないtaskIdはエラーとして拒否

### T4: サービス拒否（Denial of Service）

**脅威**: 大量のタスクを作成してディレクトリスキャンを遅延させる

**影響**: 低
- 現実的なユースケースでは数十タスクが上限
- 自分の環境での自己攻撃に限定

**対策**:
- タスク数の上限警告（100タスク以上で警告）
- 完了タスクの定期的なクリーンアップを推奨

### T5: 権限昇格（Elevation of Privilege）

**脅威**: フェーズ制限をバイパスして不正な操作を行う

**影響**: 中
- 環境変数 `SKIP_PHASE_GUARD=true` で制限を無効化可能

**対策**:
- スキップ時は監査ログに記録（現状維持）
- スキップはユーザーの明示的な意思として許容

## リスクマトリクス

| 脅威 | 発生確率 | 影響度 | リスクレベル | 対応 |
|-----|---------|-------|------------|------|
| T1: 状態競合 | 低 | 中 | 低 | 許容 |
| T2: パス推論誤り | 中 | 中 | 中 | 軽減 |
| T3: ID偽装 | 低 | 低 | 低 | 許容 |
| T4: DoS | 低 | 低 | 低 | 許容 |
| T5: 権限昇格 | 低 | 中 | 低 | 許容 |

## セキュリティ要件

### SR-1: ファイルパス推論の厳密化

```
Given: 複数タスクが存在する
When: ファイルパスがタスクに関連付けられる
Then: 最長一致するdocsDir/workflowDirを持つタスクが選択される
```

### SR-2: taskId検証

```
Given: ツールにtaskIdが指定される
When: 操作を実行する前
Then: taskIdがディレクトリに存在することを検証する
And: 存在しない場合はエラーを返す
```

### SR-3: 監査ログ

```
Given: スキップフラグ（SKIP_PHASE_GUARD等）が使用される
When: 操作が実行される
Then: 監査ログに記録される（現状維持）
```

## 結論

並列タスク対応の変更による新たなセキュリティリスクは限定的。主な対策:

1. ファイルパス推論の厳密化（最長一致）
2. taskIdの存在検証
3. 既存の監査ログ機能の維持

追加のセキュリティ実装は不要。
