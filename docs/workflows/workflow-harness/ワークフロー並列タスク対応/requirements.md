# ワークフロー並列タスク対応 - 要件定義

## 背景

現在のワークフローシステムは「単一アクティブタスク」設計であり、`GlobalState.activeTasks[0]` をカレントタスクとして扱う。これにより複数タスクの並列処理時に以下の問題が発生する:

- タスクAの作業中にタスクBのstatusを確認すると、タスクBがアクティブになる
- hookがタスクBのフェーズでチェックするため、タスクAの作業がブロックされる

## 目的

グローバルでのアクティブタスク管理を廃止し、各タスクを独立・自己完結させることで、真の並列タスク処理を実現する。

## 機能要件

### FR-1: グローバル状態の簡素化

| ID | 要件 | 優先度 |
|----|-----|-------|
| FR-1.1 | `GlobalState.activeTasks` 配列を廃止する | 必須 |
| FR-1.2 | `GlobalState.phase` を廃止する | 必須 |
| FR-1.3 | `GlobalState.history` は全体履歴として残す（オプション） | 任意 |

### FR-2: タスク発見メカニズム

| ID | 要件 | 優先度 |
|----|-----|-------|
| FR-2.1 | タスク一覧はディレクトリスキャンで取得する | 必須 |
| FR-2.2 | `.claude/state/workflows/` 配下のディレクトリをタスクとして認識する | 必須 |
| FR-2.3 | 各ディレクトリ内の `workflow-state.json` からタスク状態を読み込む | 必須 |
| FR-2.4 | `completed` フェーズのタスクはアクティブ一覧から除外する | 必須 |

### FR-3: ツールAPI変更

| ID | 要件 | 優先度 |
|----|-----|-------|
| FR-3.1 | `workflow_status` に `taskId` オプションパラメータを追加する | 必須 |
| FR-3.2 | `taskId` 省略時は全アクティブタスクの一覧を返す | 必須 |
| FR-3.3 | `workflow_next` に `taskId` 必須パラメータを追加する | 必須 |
| FR-3.4 | `workflow_approve` に `taskId` 必須パラメータを追加する | 必須 |
| FR-3.5 | `workflow_reset` に `taskId` 必須パラメータを追加する | 必須 |
| FR-3.6 | `workflow_complete_sub` に `taskId` 必須パラメータを追加する | 必須 |
| FR-3.7 | `workflow_list` はディレクトリスキャンで一覧を取得する | 必須 |
| FR-3.8 | `workflow_switch` を廃止する | 必須 |

### FR-4: Hooks変更

| ID | 要件 | 優先度 |
|----|-----|-------|
| FR-4.1 | hookはファイルパスからタスクを推論する | 必須 |
| FR-4.2 | `docsDir` 配下のファイルは該当タスクとして扱う | 必須 |
| FR-4.3 | `workflowDir` 配下のファイルは該当タスクとして扱う | 必須 |
| FR-4.4 | どのタスクにも属さないファイルは許可する（緩和方針） | 必須 |

### FR-5: マイグレーション

| ID | 要件 | 優先度 |
|----|-----|-------|
| FR-5.1 | 既存の `GlobalState` から `activeTasks` を削除する | 必須 |
| FR-5.2 | 既存の `TaskState` は変更しない | 必須 |
| FR-5.3 | マイグレーションスクリプトは不要（自動で対応） | 任意 |

## 非機能要件

### NFR-1: パフォーマンス

| ID | 要件 |
|----|-----|
| NFR-1.1 | ディレクトリスキャンは1秒以内に完了すること（100タスク以下想定） |
| NFR-1.2 | 各ツール呼び出しは100ms以内にレスポンスを返すこと |

### NFR-2: 後方互換性

| ID | 要件 |
|----|-----|
| NFR-2.1 | 既存の `TaskState` フォーマットを維持すること |
| NFR-2.2 | 既存タスクのワークフロー継続が可能であること |

### NFR-3: 保守性

| ID | 要件 |
|----|-----|
| NFR-3.1 | 新しいタスクは従来通り `workflow_start` で作成可能であること |
| NFR-3.2 | ツールのエラーメッセージに `taskId` を含めること |

## 受け入れ基準

### AC-1: 並列タスク処理

```
Given: タスクAとタスクBが並行して存在する
When: タスクAで作業中にタスクBのstatusを確認する
Then: タスクAの作業がブロックされない
```

### AC-2: タスク指定操作

```
Given: タスクAとタスクBが存在する
When: workflow_next(taskId: "A")を実行する
Then: タスクAのみがフェーズ遷移する
And: タスクBは影響を受けない
```

### AC-3: タスク一覧取得

```
Given: 3つのアクティブタスクと1つの完了タスクが存在する
When: workflow_listを実行する
Then: 3つのアクティブタスクのみが返される
```

### AC-4: hookのファイルパス推論

```
Given: タスクAのdocsDirが "docs/workflows/タスクA/"
When: "docs/workflows/タスクA/research.md" を編集しようとする
Then: タスクAのフェーズルールが適用される
```

### AC-5: 属さないファイルの許可

```
Given: タスクAとタスクBが存在する
When: "src/index.ts" を編集しようとする（どのdocsDirにも属さない）
Then: 編集が許可される（アクティブタスクが存在する場合）
```

## スコープ外

- グローバル履歴（`GlobalState.history`）の廃止検討
- 完了タスクの自動アーカイブ
- タスク間の依存関係管理
- UIダッシュボード
