## サマリー

本セキュリティスキャンは、アーキテクチャ改善タスク（subagentTemplate取得API追加とrequiredSections統一）で変更・追加された4ファイルを対象として実施した。

- 目的: 新規ツール `workflow_get_subphase_template` および関連変更箇所のセキュリティリスクを評価する
- 評価スコープ: get-subphase-template.ts, artifact-validator.ts, next.ts, server.ts の4ファイルを対象とし、入力検証・情報漏洩・認可制御・依存関係・最小権限の5観点で確認した
- 検出件数: 重大な脆弱性0件、低リスクの軽微な懸念事項2件を確認した
- 主要な決定事項: 入力検証はホワイトリスト方式で実装されており、インジェクションリスクは低い。エラーメッセージによる内部情報漏洩は限定的である。セッショントークンは新ツールで使用されず、最小権限の観点から適切な設計となっている。
- 次フェーズで必要な情報: 検出された軽微な問題（userIntent情報の出力）については、運用上の影響を考慮した上で改善を分析すること。

## 脆弱性スキャン結果

### 対象ファイルと確認内容の概要

スキャン対象は以下の4ファイルである。

- `workflow-plugin/mcp-server/src/tools/get-subphase-template.ts`（FR-1: 新規ツール）
- `workflow-plugin/mcp-server/src/validation/artifact-validator.ts`（FR-2: 変更箇所）
- `workflow-plugin/mcp-server/src/tools/next.ts`（FR-3: 変更箇所）
- `workflow-plugin/mcp-server/src/server.ts`（ツール登録変更）

確認した観点は5項目である: 入力検証、情報漏洩、認可制御、依存関係リスク、最小権限原則。

### 観点1: 入力検証（subPhaseName）

`get-subphase-template.ts` の `workflowGetSubphaseTemplate` 関数において、`subPhaseName` パラメータはホワイトリスト配列 `VALID_SUB_PHASE_NAMES` と照合される。
この配列には11種類の固定文字列のみが含まれており、`Array.prototype.includes` による厳密な一致チェックを行う実装となっている。
ホワイトリスト外の値が渡された場合は即座に `success: false` のエラーレスポンスが返され、後続処理には進まない設計である。
インジェクション攻撃（パスインジェクション、コマンドインジェクション等）のベクタになるユーザー入力は、このバリデーションにより遮断される。
`taskId` パラメータは任意項目であり、`stateManager.getTaskById` に渡されるが、stateManagerはファイルシステムの状態ファイルを読み取る設計であり、パスのサニタイズは stateManager 内部の責務となる。

**判定**: 問題なし。ホワイトリスト方式は適切な実装である。

### 観点2: エラーメッセージによる情報漏洩

エラーレスポンスのメッセージ文字列には、ユーザーが指定した `subPhaseName` の値がそのまま含まれる。
ただしこの値はホワイトリスト検証を通過した後の処理ではなく、バリデーション拒否時のフィードバックとして設計されている。
有効なサブフェーズ名の一覧も `VALID_SUB_PHASE_NAMES` をそのままエラーメッセージに含めており、これはMCPサーバーの用途から見て適切な情報開示である。
また、タスクが見つからない場合のエラーメッセージ（`タスクが見つかりません: {resolvedTaskId}`）はタスクIDを含むが、MCPサーバーはローカルプロセスとして動作するため、外部ネットワークへの漏洩リスクは存在しない。
成功時のレスポンスでは `docsDir` と `userIntent` が含まれる。`userIntent` はユーザーが入力したテキストであり、機密情報を含む可能性がある点に注意が必要である。

**判定**: 軽微な懸念事項あり。`userIntent` の出力は運用環境により情報漏洩となりうる。

### 観点3: セッショントークンの取り扱い

新規ツール `workflowGetSubphaseTemplate` はセッショントークンパラメータを持たない設計である。
これはテンプレートデータが読み取り専用の操作であり、状態変更を伴わないという設計判断に基づく。
`server.ts` の `validateToolArgs` 関数に `workflow_get_subphase_template` のエントリが正しく追加されており、必須パラメータ `subPhaseName` のチェックが定義されている。
`next.ts` の既存のセッショントークン検証（`verifySessionToken`）は変更されておらず、フェーズ遷移の認可制御は維持されている。

**判定**: 問題なし。読み取り専用操作へのセッショントークン不使用は適切な設計である。

### 観点4: 依存関係リスク

今回の変更で新規外部依存関係は追加されていない。使用しているモジュールはいずれも既存の内部モジュール（`definitions.js`, `manager.js`, `types.js`）である。
`artifact-validator.ts` の変更は `exportGlobalRules` 関数の追加であり、既存の `fs` および `path` モジュールへの依存は変更なしである。
`next.ts` の変更は `getMinLinesFromPhaseGuide` 関数の追加と `PHASE_GUIDES` のインポート追加であり、いずれも既存内部モジュールへの参照である。

**判定**: 問題なし。新規外部依存関係なし。

### 観点5: 最小権限原則

新ツール `workflowGetSubphaseTemplate` は読み取り操作のみを行い、状態の書き込みや変更は一切行わない。
ファイルシステムへのアクセスは `stateManager.getTaskById` および `stateManager.discoverTasks` 経由で既存の状態ファイルを読むのみであり、新規ファイルの作成や変更は行わない。
テンプレートデータは `resolvePhaseGuide` 関数から取得されるが、これはメモリ上の定義データを参照するものであり、外部リソースへのアクセスを伴わない。

**判定**: 問題なし。最小権限の原則に沿った実装である。

## 検出された問題

### 問題1: userIntentのレスポンス出力（低リスク）

**対象ファイル**: `get-subphase-template.ts` の 138行目

`workflowGetSubphaseTemplate` の成功時レスポンスに `userIntent` フィールドが含まれる。
この値はタスク開始時にユーザーが入力した文字列であり、業務上の機密情報が含まれる可能性がある。
MCPサーバーはローカルプロセスとして動作し、外部ネットワークには公開されないため、現時点での実際のリスクは限定的である。
ただし、ログや監査証跡への記録を通じた情報漏洩の可能性がある点に留意が必要である。

リスクレベル: Low（ローカル環境での運用のため悪用可能性は低い）

推奨対応: `userIntent` の出力はOrchestratorのコンテキスト引き継ぎ目的で使用されており、機能上の必要性がある。機密情報をuserIntentに含めないようにユーザーに案内することで対応可能である。

### 問題2: taskIdによるタスク探索の副作用（低リスク）

**対象ファイル**: `get-subphase-template.ts` の 94行目

`taskId` が省略された場合、`stateManager.discoverTasks()` が呼ばれ、状態ディレクトリ全体をスキャンして全タスクを探索する。
大量のタスク履歴が蓄積された環境では、この操作がI/Oコストを引き上げる可能性がある。
セキュリティ上の問題というよりは可用性（DoS）の観点に近いが、悪意を持った大量タスク作成によりサービス低下が生じうる。

リスクレベル: Low（ローカル環境での運用かつ外部アクセス経路なし）

推奨対応: 将来的にタスク数が増大した場合はキャッシュ機構や上限設定を検討する。現時点での対応は任意である。

### 総合評価

今回の実装変更において、致命的または高リスクの脆弱性は検出されなかった。
識別された2件はいずれも低リスクであり、MCPサーバーのローカル実行という運用形態において実際の悪用可能性は極めて低い。
入力検証・認可制御・最小権限の各観点において、既存のセキュリティ設計と一貫した実装となっている。
