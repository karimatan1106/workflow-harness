## サマリー

- 目的: ワークフロープラグインのアーキテクチャ改善タスク（FR-1: 新規MCPツール追加、FR-2: requiredSections 形式統一、FR-3: minLines 単一ソース化）における変更リスクを分析し、実装上の注意点を明確にする
- 分析対象: FR-1 は新規ツール追加による ESM インポートエラーリスク、FR-2 はバリデーション互換性リスク、FR-3 は minLines 値変更による既存テスト破壊リスクをそれぞれ内包する
- 主要な決定事項: セキュリティ上の脅威は外部攻撃者よりも内部的な設計ミスやリグレッションが主要リスクであり、MCPサーバーキャッシュ問題が最も影響度の高いリスクとして特定された
- 最重要リスク: `definitions.ts` または `artifact-validator.ts` 変更後にサーバー再起動を省略した場合、古いキャッシュが動作し続けバリデーション結果が修正前の状態のまま継続する（可能性: 高、影響: 高）
- 次フェーズで必要な情報: 各リスク項目に対する具体的な緩和策（テスト戦略・型定義方針・再起動手順）を planning フェーズの設計書に反映すること。特に FR-3 の minLines 値変更が既存テストスイートに与える影響を事前に洗い出すこと

---

## 脅威分析

### 対象システムの概要

本タスクで変更するシステムはローカル動作の MCP（Model Context Protocol）サーバーである。外部ネットワークからの攻撃経路は存在せず、脅威の大部分は内部の設計変更に起因するリグレッションと整合性の破壊である。変更対象ファイルは `next.ts`、`status.ts`、`definitions.ts`、`artifact-validator.ts` の 4 ファイルであり、これらは全て MCP サーバーの起動時に一度だけロードされ、Node.js モジュールキャッシュに保持される。

### 脅威カテゴリ 1: MCPサーバーキャッシュによる変更未反映

**説明**: `definitions.ts` と `artifact-validator.ts` はコアモジュールであり、変更後にビルド（`npm run build`）を実行してもサーバープロセスを再起動しない限り古いキャッシュが動作し続ける。FR-2 で `requiredSections` の形式を変更した場合、バリデーターが古いパターンで検証を実行するため、修正済みのはずのエラーが継続して報告される症状が発生する。CLAUDE.md のルール 22 はこのリスクを明示しており、プロジェクトの既知の落とし穴として記録されている。

**影響範囲**: FR-2 と FR-3 の両方に直接関係する。FR-2 で `## ` プレフィックスを追加した後にサーバー再起動を省略した場合、バリデーションは古いパターン（プレフィックスなし）で検証し続け、新規成果物が誤って失敗と判定される可能性がある。

**可能性**: 高。過去のプロジェクト記録（MEMORY.md）に同様の事例が複数記録されている。

### 脅威カテゴリ 2: `requiredSections` 形式変更によるバリデーション誤動作

**説明**: `artifact-validator.ts` の `PHASE_ARTIFACT_REQUIREMENTS` における `requiredSections` の値に `## ` プレフィックスを付与する変更（FR-2）は、既存の `content.includes(section)` による部分一致ロジックと組み合わせると後方互換性を保つ。ただし変更対象の `threat-model.md` エントリは現在 `'## 脅威'` と `'## リスク'` を定義しているのに対し、`definitions.ts` 側の `threat_modeling` フェーズガイドは `'## 脅威シナリオ'` と `'## リスク評価'` を定義しており、セクション名が不一致である。

**影響範囲**: 現時点でのバリデーターは `'## 脅威'` を部分一致で検索するため `'## 脅威シナリオ'` にもマッチする。しかし名称を完全一致に変更する方向の要件変更が将来発生した場合、既存成果物との互換性が失われる。FR-2 での修正対象として明示的に記載されているが、修正後の値が `definitions.ts` 側と一致するかどうかの検証テストが必要である。

**可能性**: 中。現時点では `includes()` の部分一致が救済しているが、修正作業中に誤った値を設定するヒューマンエラーが発生しうる。

### 脅威カテゴリ 3: `minLines` 単一ソース化による値の不一致

**説明**: FR-3 は `artifact-validator.ts` の `PHASE_ARTIFACT_REQUIREMENTS` の `minLines` を `definitions.ts` の `PHASE_GUIDES.minLines` の値で上書きする構造に変更する。最も具体的な不一致として `research.md` の場合、`PHASE_GUIDES` は 50 行を要求するのに対し `PHASE_ARTIFACT_REQUIREMENTS` は 20 行を定義している。単一ソース化後は 50 行が正として採用される。

**影響範囲**: 既存の成果物が 20 行以上 50 行未満の場合、単一ソース化後にバリデーション失敗が発生する可能性がある。新しい実装方式（import 参照 vs 実行時上書き）の選択によって影響の広がり方が変わる。import 参照方式では全フェーズの `minLines` が一括して `PHASE_GUIDES` の値に置き換わるため、値確認が必要なフェーズが増加する。

**可能性**: 中から高。`research.md` の 50 行要件は現在の研究フェーズの標準的な出力量と比較して高い水準であり、不合格が多発しうる。

### 脅威カテゴリ 4: 新規MCP ツールのロード失敗

**説明**: `workflow_get_subphase_template` ツールを `tools/` ディレクトリ配下に新規ファイルとして追加し、`index.ts` と `server.ts` でエクスポートおよびハンドラー登録を行う必要がある。ESM 環境（`package.json` の `"type": "module"`）での import パスには `.js` 拡張子が必要であり、この拡張子が省略された場合にサーバー起動時に `ERR_MODULE_NOT_FOUND` が発生する。MEMORY.md に ESM/CommonJS 互換性の注意事項として記録されている既知の落とし穴である。

**影響範囲**: 新規ツールファイルの追加だけでなく、`index.ts` への export 追加および `server.ts` への tool handler 登録の 3 箇所に同じリスクが存在する。いずれか 1 箇所でも `.js` 拡張子が欠落するとサーバー全体が起動不可になる。

**可能性**: 低から中。既存コードのパターンを踏襲すれば防止可能だが、新規ファイル追加時の人的ミスとして発生しうる。

### 脅威カテゴリ 5: `resolvePhaseGuide()` によるサブフェーズテンプレート取得の失敗

**説明**: `workflow_get_subphase_template` の内部実装では `resolvePhaseGuide()` を呼び出し、返却された `phaseGuide.subPhases[subPhaseName].subagentTemplate` を取得する設計となっている。`resolvePhaseGuide()` がどのような入力でどのような出力を返すかについて、並列フェーズ外のフェーズ名が渡された場合のエラー処理が定義されていない場合、ランタイムエラーが発生する可能性がある。

**影響範囲**: 不正な `subPhaseName`（例: `'research'` や `'implementation'`）が指定された場合に、サーバーがクラッシュするか、あるいは不明確なエラーメッセージを返す可能性がある。NFR-3 では型安全性と実行時バリデーションを要求しているが、その具体的な実装方法が設計フェーズで確定する必要がある。

**可能性**: 中。Orchestrator による誤用または開発者のデバッグ中の誤入力で発生しうる。

## リスク評価

### リスク 1: MCPサーバーキャッシュ問題（最高優先度）

影響度は高く、発生可能性も高いため、このプロジェクトで最も優先度の高いリスクである。`definitions.ts` または `artifact-validator.ts` を変更するたびに MCP サーバーを再起動しなければ、バリデーション動作は変更前の状態のまま継続する。この問題は過去のタスクで繰り返し発生しており、MEMORY.md の「強制再起動条件」として記録済みである。

緩和策として、実装フェーズのチェックリストに「コアモジュール変更後は必ず `npm run build` を実行しサーバーを再起動する」という必須ステップを含めること。また、FR-3 実施後に `research.md` のバリデーションを手動で確認するスモークテストを実行することを推奨する。

### リスク 2: `minLines` 値の上昇による既存テスト破壊（高優先度）

`research.md` の `minLines` が 20 から 50 に引き上げられることで、既存のテストケースが不合格になる可能性がある。`artifact-file-size.test.ts` や `artifact-quality-check.test.ts` などのテストファイルは現行の 20 行基準で記述されている可能性が高く、単一ソース化後に全テストを実行して全て通過することを確認する必要がある。

緩和策として、test_impl フェーズで `minLines` 変更の影響を受けるテストケースを事前に特定し、正しい期待値に更新すること。`PHASE_GUIDES.minLines` が正の値として採用されるため、テストも同じ値を参照するか、または直接 `PHASE_GUIDES` から取得するよう修正することが望ましい。

### リスク 3: `requiredSections` の名称不一致によるサイレントエラー（中優先度）

`threat-model.md` の `requiredSections` を `definitions.ts` の `threat_modeling` フェーズガイドと一致させる修正は、`'## 脅威シナリオ'` と `'## リスク評価'` への変更を意味する。現在の `'## 脅威'` と `'## リスク'` はより短いため `includes()` でより多くのパターンにマッチする。修正後はより長いセクション名となり、マッチしなかった成果物が初めてバリデーション失敗を示す可能性がある。

緩和策として、修正後の値でバリデーター単体テストを実行し、既存のテスト用 `threat-model.md` フィクスチャが引き続き合格することを確認する。また、`manual-test.md` と `security-scan.md` の `requiredSections` 変更（`## ` プレフィックス追加）についても同様に既存テストとの整合性を確認する。

### リスク 4: 新規ツールの登録漏れ（低優先度）

`workflow_get_subphase_template` ツールは `tools/index.ts` への export 追加と `server.ts` へのハンドラー登録が必要であり、これらの登録が漏れた場合にツールは Claude に公開されない。登録漏れはサーバーのクラッシュは引き起こさないが、ツールが MCP クライアントから利用不可能な状態になる。

緩和策として、既存ツール（例: `workflow_pre_validate` や `workflow_link_tasks`）の追加パターンを参照し、同じ手順で登録を行うこと。テストフェーズで MCP クライアントからツールの呼び出しを確認する integration test を追加することが望ましい。

### リスク 5: `subPhaseName` の不正入力によるランタイムエラー（低優先度）

有効なサブフェーズ名（`threat_modeling`、`planning` 等）以外の値が `workflow_get_subphase_template` に渡された場合、内部の `resolvePhaseGuide()` または `phaseGuide.subPhases[subPhaseName]` のアクセスでエラーが発生する可能性がある。

緩和策として、ツール実装の冒頭で `subPhaseName` の値を有効なサブフェーズ名の列挙と比較するバリデーション処理を追加する。有効でない場合は明確なエラーメッセージを返し、サーバークラッシュを防ぐ。TypeScript の型定義でリテラルユニオン型を使用することで、コンパイル時にも型チェックが機能するよう整備する。

### 総合リスク評価

本タスクは外部攻撃面を持たないため、機密情報漏洩や権限昇格のリスクはほぼゼロである。主要リスクは開発プロセス上の内部リスクであり、適切なテストカバレッジと再起動手順の徹底により全て緩和可能な水準である。実装の複雑さは中程度であり、変更ファイル数が 4 ファイル（プラス新規ツールファイル 1 ファイル）と限定的であるため、リグレッションの追跡は比較的容易である。
