# ワークフロー実行時バグ修正 - 脅威モデル

## サマリー

本ドキュメントは、ワークフロー実行時に発見された4つのバグ修正に対する脅威分析を記述します。

- 目的: 各バグ修正がセキュリティホールや品質劣化を引き起こさないことを確認する
- 主要な決定事項: バグ1のハッシュチェックスキップはtestingフェーズ限定・regression_testは既存ロジック維持、バグ2の緩和は比較演算子のみに絞り正当リダイレクト検出は維持、バグ3のカテゴリ修正はドキュメント整合のみで権限昇格なし、バグ4のブランチ確認は動的取得により誤push防止
- 次フェーズで必要な情報: 各修正の影響範囲はリスク評価「低」と判定されており、追加の緩和策なしで実装フェーズへ進行可能

各修正のリスク評価結果は全て「低」であり、意図しないセキュリティホールは発生しない見込みです。

---

## 脅威分析対象の概要

4つの修正対象ファイルとその役割を整理します。

- `next.ts`: ワークフローのフェーズ遷移ロジックを実装するMCPサーバーコアモジュール
- `bash-whitelist.js`: フック側でBashコマンドの実行許可判定を行うガードロジック
- `definitions.ts`: フェーズ定義・モデル設定・subagentプロンプトテンプレートのコアファイル

修正はそれぞれ独立しており、相互依存による複合的な脅威は発生しない構造です。
各修正が影響するコンポーネントの境界を明確化することで、脅威分析の範囲を限定します。

---

## 脅威1: testingフェーズ ハッシュチェックスキップによる品質低下リスク（バグ1）

### 修正内容の概要

`next.ts` の testingフェーズブロックに、ハッシュ重複チェックをスキップする条件
(`currentPhase !== 'testing'`) を追加します。regression_testフェーズに既存のスキップロジックが
存在するため、同じパターンをtestingフェーズに適用します。

### 脅威シナリオ1-A: コピペ検出回避の悪用

スキップ条件が広範に適用された場合、サブagentが同じテスト結果を繰り返しコピーして
フェーズ遷移を完了させるという回避行動が可能になります。ただし、この修正はtestingフェーズ限定の
スキップであり、regression_testフェーズの既存チェックは影響を受けません。testingフェーズでは
`record_test_result` を1回呼ぶのが正常な使用パターンであり、複数回呼ぶ必要がある場合は
regression_testフェーズを使用する設計になっています。

リスク評価: 低（スキップ範囲がtestingフェーズの1回記録パターンに限定）

### 脅威シナリオ1-B: 他フェーズへのスキップ条件の誤波及

条件式の誤記により、testingフェーズ以外のフェーズでもチェックがスキップされる可能性があります。
`currentPhase !== 'testing'` という否定条件を使用するため、コードレビューで
条件の向きを確認する必要があります。regression_testフェーズのスキップは
`currentPhase !== 'regression_test'` という独立した条件で制御されており、
両条件は別々のブロックに記述されるため干渉しません。

リスク評価: 低（条件が独立して記述される設計であるため）

### 緩和策

testingフェーズでのスキップ条件を追加する際に、`phase === 'testing'` という
肯定条件（ホワイトリスト方式）で記述することで、意図しない範囲への適用を防ぎます。
コードレビューフェーズで条件式の向きを明示的に確認します。

---

## 脅威2: bash-whitelist緩和による不正コマンド実行リスク（バグ2）

### 修正内容の概要

`bash-whitelist.js` の `hasRedirection` 関数において、`part.includes('>')` による
粗い検索を、`>=` や `<=` の比較演算子を除外する精緻な判定に変更します。
具体的には `>` の直後が `=` の場合はリダイレクトと見なさない条件を追加します。

### 脅威シナリオ2-A: リダイレクト検出の抜け穴

`>=` の除外ロジックが不完全な場合、`> file` の形式が `>=` 除外の副作用で
検出漏れになるリスクがあります。修正の実装方針として、`>` の直後が `=` のときのみ
比較演算子と判定し、`>` の直後がスペース・ファイル名・`>` そのものの場合は
リダイレクトと正しく判定するロジックを維持します。
`>>` のような二重リダイレクトも引き続き検出できることを受入基準として明示しています。

リスク評価: 低（修正範囲が比較演算子の誤検出除外のみであり、正当リダイレクト検出は維持）

### 脅威シナリオ2-B: 文字列結合による検出回避

悪意あるコマンドが `a>=b` のような形式で記述された場合、改修後の判定で
リダイレクトと判定されなくなります。しかし `>=` はBashの比較演算子として
実際に使用される正当な構文であり、これをリダイレクトと誤判定していた現状のほうが
正規のコマンドをブロックするという問題を引き起こしていました。
また、bash-whitelistはコマンドカテゴリの許可判定を主目的としており、
リダイレクト検出はその補助機能に過ぎません。

リスク評価: 低（修正は誤検出の解消であり、セキュリティ境界の緩和ではない）

### 緩和策

修正後のロジックで `echo hello > output.txt` が依然としてリダイレクトとして
検出されることを、ユニットテストで明示的に検証します。
`cat file >> log.txt` の二重リダイレクトも同様に検証対象とします。

## 脅威3: allowedBashCategories変更による権限昇格リスク（バグ3）

### 修正内容の概要

`definitions.ts` のcommitフェーズとpushフェーズの `allowedBashCategories` を
`['readonly', 'implementation']` から `['readonly', 'git']` に変更します。
この変更はsubagentへのプロンプト表示に使用される情報であり、
実際のBashコマンド実行許可を制御する `bash-whitelist.js` の `getWhitelistForPhase`
とは独立して動作します。

### 脅威シナリオ3-A: implementationカテゴリ削除による機能喪失

現在 `['readonly', 'implementation']` と設定されているため、subagentは
`npm install`, `npm run build`, `mkdir`, `rm` などのimplementationカテゴリのコマンドを
使用可能と誤認識しています。修正後は `git` カテゴリのみが追加され、
commitフェーズでこれらのコマンドが使えなくなります。CLAUDE.mdにも
「commitフェーズではrmコマンドを含むimplementationカテゴリは許可されていません」と
明記されており、修正後の状態が設計の正しい状態です。

リスク評価: 低（修正後が設計仕様と一致しており、影響はsubagentへの情報提供のみ）

### 脅威シナリオ3-B: gitカテゴリの過剰な権限付与

`git` カテゴリに含まれるコマンドがcommit/pushフェーズで不当に実行されるリスクを
検討します。`git` カテゴリには `git add`, `git commit`, `git tag` などが含まれます。
これらはcommit/pushフェーズで必要なコマンドであり、過剰な権限付与には該当しません。
`git push --force` などの破壊的コマンドはCLAUDE.mdのGit Safety Protocolで別途禁止されています。

リスク評価: 低（gitカテゴリはcommit/pushフェーズで必要な権限の範囲内）

### 脅威シナリオ3-C: bash-whitelistとの不整合による実行可否の乖離

`definitions.ts` の情報変更が `bash-whitelist.js` の実際の判定ロジックと
乖離する場合、subagentへの誤情報提供が継続するリスクがあります。
修正後の `['readonly', 'git']` はCLAUDE.mdの「フェーズ別Bashコマンド許可カテゴリ」表および
`bash-whitelist.js` の `gitPhases` 配列（commit, push, release 等）と整合します。
三者が同じ情報を持つことでNF-2の要件を満たします。

リスク評価: 低（三者整合を受入基準として明示しているため）

## 脅威4: ブランチ名動的取得の信頼性リスク（バグ4）

### 修正内容の概要

`definitions.ts` のpushフェーズの `subagentTemplate` に、
`git branch --show-current` または `git rev-parse --abbrev-ref HEAD` を実行して
カレントブランチ名を確認してからpushする手順を追加します。

### 脅威シナリオ4-A: detached HEAD状態での誤動作

`git checkout <commit-hash>` などによりdetached HEAD状態になっている場合、
`git branch --show-current` は空文字列を返します。
`git rev-parse --abbrev-ref HEAD` は `HEAD` という文字列を返します。
どちらの場合も `git push origin HEAD` や空のブランチ名でpushすることで
意図しない動作が発生する可能性があります。subagentTemplateで、
取得したブランチ名が空または `HEAD` の場合はpushを中止し、
ユーザーに確認を求めるよう指示する記述を追加することで対処します。

リスク評価: 低（subagentTemplateへの追記で対処可能なエッジケース）

### 脅威シナリオ4-B: サブモジュールのブランチ名乖離

親リポジトリとサブモジュール（workflow-plugin）が異なるブランチを持つ場合、
サブモジュールのブランチを確認せずpushすると誤ったブランチにコミットされます。
FQ-4の要件に「サブモジュールのブランチ名も個別に確認してからpushするよう指示する」と
明記されているため、subagentTemplateにサブモジュールごとのブランチ確認手順を追記します。

リスク評価: 低（既に要件として明示されており、テンプレートへの追記で対処可能）

### 脅威シナリオ4-C: リモートリポジトリへの誤ブランチpush

ブランチ名確認を動的に行う修正後も、subagentが確認結果を誤って解釈して
異なるブランチ名を指定するリスクがあります。subagentTemplateで確認コマンドの出力を
変数として保持し、その変数をpushコマンドに直接使用する形式を明示することで、
ブランチ名の取り違えを防ぎます。

リスク評価: 低（テンプレートの記述精度で制御可能）

## 複合脅威の評価

4つの修正を同時に適用する場合の複合的なリスクを評価します。

バグ1とバグ3は共にMCPサーバーのコアモジュールを修正するため、
MCPサーバーの再起動が必要になります。NF-4の要件に従い、
次のtestingフェーズ遷移前にサーバーを再起動することで
古いキャッシュによる誤動作を防ぎます。

バグ2はhooks側のbash-whitelist.jsを修正するため、MCPサーバー再起動とは
独立したフックキャッシュのリフレッシュが必要です。hooks側の修正は
セッション開始時に再読み込みされるため、Claude Desktopの再起動により反映されます。

4つの修正が相互に干渉するシナリオは存在せず、各修正の影響範囲は独立しています。

## STRIDEフレームワークによる総合評価

| 脅威カテゴリ | 評価 | 根拠 |
|-------------|------|------|
| Spoofing（なりすまし）| 影響なし | ブランチ確認の追加はむしろ誤認防止に寄与する |
| Tampering（改ざん）| 低リスク | ハッシュスキップはtesting限定であり改ざん検出機能は維持 |
| Repudiation（否認）| 影響なし | テスト結果の記録自体は変更しない |
| Information Disclosure（情報漏洩）| 影響なし | ブランチ名取得はローカル情報であり外部送信しない |
| Denial of Service（サービス拒否）| 低リスク | MCPサーバー再起動手順の漏れが一時的な機能停止を招く可能性あり |
| Elevation of Privilege（権限昇格）| 低リスク | allowedBashCategoriesの変更は情報提供のみであり実際の権限制御はbash-whitelistが担う |

全カテゴリで許容可能なリスクレベルであると判断します。追加の緩和策なしで実装フェーズへの移行が可能です。

## リスク

各バグ修正に伴うリスクを体系的に評価し、全ての残存リスクとその対処方針を記述します。

### バグ1（testingフェーズ ハッシュチェックスキップ）の残存リスク

スキップ条件を `phase === 'testing'` のホワイトリスト方式で記述すれば、
意図しないフェーズへの誤適用リスクは排除できます。
残存リスクとして、testingフェーズで複数回 `record_test_result` を呼ぶ構成の場合、
チェックをスキップすることで本来検出すべきコピペが検出されない状態になります。
しかしtestingフェーズは1回の記録が正常パターンであるため、実運用上の影響は軽微です。
対処方針: コードレビューフェーズで条件の向きを確認し、regression_testとの独立性を検証します。

### バグ2（bash-whitelist リダイレクト検出修正）の残存リスク

`>=` のみを除外する修正は、`a>=b` のような比較演算子が実際のリダイレクトと混在する
コマンドに対して検出精度が下がる可能性を残します。
ただしbash-whitelistはカテゴリ許可判定を主目的としており、
リダイレクト検出はその補助機能に過ぎないため、影響は限定的です。
対処方針: `echo hello > output.txt` および `cat file >> log.txt` をユニットテストで検証します。

### バグ3（allowedBashCategories 変更）の残存リスク

`definitions.ts` の変更はsubagentへの情報提供のみであり、
`bash-whitelist.js` の実際の権限制御とは独立して動作します。
そのため、两者が整合していない状態が発生した場合、subagentが誤った情報に基づいて判断するリスクが残ります。
対処方針: CLAUDE.md・definitions.ts・bash-whitelist.jsの三者整合を受入基準として明示的に確認します。

### バグ4（pushフェーズ ブランチ確認追加）の残存リスク

subagentTemplateにブランチ確認手順を追加しても、subagentが確認コマンドの出力を
誤って解釈するリスクは完全には排除できません。
detached HEAD状態では確認コマンドが空文字列や `HEAD` を返すため、
subagentがそのままpushを試みるリスクが残ります。
対処方針: テンプレートに「空文字またはHEADの場合はpushを中止してユーザーに確認」という
明示的な指示を含めることで、このリスクを実運用レベルで許容可能にします。

### 総合リスク評価サマリー

全4件のバグ修正に共通するリスクは、MCPサーバーおよびClaudeセッションの
再起動タイミングの誤りによる一時的な機能不整合です。
この点を除けば、各修正のリスクレベルはすべて「低」に収まります。
既存のセキュリティ境界（HMACチェック、bash-whitelist判定、CLAUDE.mdのGit Safety Protocol）は
いずれの修正においても変更されないため、全体として許容可能なリスクプロファイルと判断します。

---

## 総合リスク判定

4つのバグ修正はいずれも既存のセキュリティ境界を維持しつつ、
動作上の問題のみを解消する最小限の変更です。

脅威分析の結果、全ての脅威シナリオが「低リスク」に分類されました。
実装フェーズへの移行にあたり追加の緩和策を要する脅威は存在しません。

MCPサーバーの再起動手順（バグ1・バグ3の修正後）と
Claudeセッションの再起動（バグ2の修正後）を確実に実施することが
唯一の注意事項です。
