## サマリー

CLAUDE.mdに追加されたBashコマンドホワイトリスト伝達メカニズムのセキュリティ分析を実施した。165-180行のマッピング表、236-244行のテンプレート、312-314行のリトライテンプレートを対象に、プロンプトインジェクション、環境変数埋め込み、ユーザー入力の汚染リスク、bash-whitelist.jsとの記述乖離を検証した。全体的には十分に防御されているが、テンプレートのプレースホルダー置換時に固有の弱点が存在する。

**主要な決定事項:**
- プレースホルダー置換は信頼できるMCPサーバーのみが実行
- ユーザー入力（taskName, userIntent等）は定型テンプレート内に埋め込まれない
- Bashコマンド一覧の複製リスクを特定
- フックによる最終検証が引き続き有効

**次フェーズで必要な情報:**
- CLAUDE.mdの記述乖離個所の修正方針
- テンプレートプレースホルダーの埋め込みロジック仕様書
- コマンドホワイトリスト同期メカニズムの実装確認

## 脆弱性スキャン結果

セキュリティスキャンは以下の5観点から実施された: プロンプトインジェクションリスク、コマンドホワイトリスト定義の正確性、環境変数の過度な公開、テンプレートプレースホルダーの置換安全性、CLAUDE.mdとbash-whitelist.jsの記述一貫性。
現在の実装はMCPサーバーの信頼できるプレースホルダー置換を前提としており、ユーザー生成コンテンツは定型テンプレート内に埋め込まれない設計となっている。
subagent側はプロンプトを参考情報として受け取り、実行可能な指示として解釈しない指示が256-279行に明記されている点は好適である。
ただし提供されるBashコマンドホワイトリストがCLAUDE.md内で複数定義されており、bash-whitelist.js実装との一貫性維持がリスクとなる。
スキャン対象のテンプレートにはユーザー入力の直接埋め込みがなく、全プレースホルダーはMCPサーバーの内部データまたは定義から提供される設計である。

## 検出された問題

### 問題1: CLAUDE.mdとbash-whitelist.jsのコマンド定義の同期リスク

**重要度: 中程度**

CLAUDE.md内に複数箇所でコマンド一覧が定義されている:

1. **CLAUDE.md 240-244行**: テンプレート内のコマンド一覧表示
   - `readonly: ls, pwd, cat, head, tail, grep, find, wc, git status, git log, git diff, git show, npm list, node --version, npm --version`
   - `testing: npm test, npm run test, npx vitest, npx jest, npx playwright test, pytest`
   - `implementation: npm install, pnpm add, npm run build, mkdir, rm, git add, git commit`
   - `git: git add, git commit, git push, git pull, git fetch`
   - `deploy: docker, kubectl, ssh, helm, gh`

2. **bash-whitelist.js 37-92行**: 実際のホワイトリスト実装
   - readonly配列に39行: `ls, cat, head, tail, less, more, wc, file, find, grep, rg, ag, git status, git log, git diff, git show, git branch, ...` などが定義
   - testing配列に57行: `npm test, npm run test, npx vitest, npx jest, npx mocha, npx ava, npx tsc --noEmit, npx eslint, npx prettier --check, ...`

**リスク**: CLAUDE.mdが簡潔版を示しており、bash-whitelist.jsが完全版を実装している。両者が乖離した場合、subagentへの指示がbash-whitelist.jsの実装と矛盾する。特にnpx wrapping（npx mocha, npx ava等）やオプション指定（npx tsc --noEmit）がCLAUDE.mdに記載されていない。

**対応**: CLAUDE.mdのコマンド一覧とbash-whitelist.jsを定期的に同期するプロセスが必要。またはCLAUDE.mdからbash-whitelist.jsへの自動生成メカニズムの検討。

### 問題2: less, more, file コマンドの記載ぶれ

**重要度: 低程度**

bash-whitelist.js内のreadonly配列には`less, more, file`が含まれているが、CLAUDE.md 240行の簡潔版には記載されていない。subagentは「ls, pwd, cat, head, tail...」の短い一覧を受け取るため、lessやmoreを使用不可と判断する可能性がある。

**リスク**: subagentが許可されていると判断したコマンドがforkによってブロックされることはないが、許可されていないと判断して使用を避ける逆の問題が生じる。

### 問題3: CLAUDE.mdに記載されないコマンドオプション

**重要度: 低程度**

bash-whitelist.js内では以下のようにオプション付きで定義されているが、CLAUDE.mdには記載されない:

- `git checkout --` （61-62行）
- `git restore` （61-62行）
- `rm -f` （90行、-fオプション限定）
- `npx tsc --noEmit` （60行）
- `npm run lint, npm run type-check` （64-65行）

subagentは「git checkoutは許可」「rmは許可」と単純に理解し、オプション検証はフックに委譲される設計である。これは正しい責任分離だが、テンプレート内に「各種オプションは--フラグによる検証が行われます」という注釈があると明確性が向上する。

**改善提案**: CLAUDE.md 246-247行に「フックによる最終検証: ホワイトリストに記載されていないオプション・パターンはブロックされます。」という一文を追加。

### 問題4: プレースホルダー置換の順序依存性

**重要度: 中程度**

CLAUDE.md内のテンプレートで複数のプレースホルダーが使用されている:

- `{フェーズ名}` - 複数箇所で使用
- `{taskName}` - 複数箇所で使用
- `{許可カテゴリ一覧}` - 237行に定型テンプレート形式で挿入

**分析結果**: プレースホルダーは全て波括弧形式で統一されており、角括弧（禁止パターン）ではない。MCPサーバーが正規表現で置換する際、置換順序を適切に管理する必要がある。例えば`{フェーズ名}`が複数回出現する場合、全て同一値で置換されるべき。

**リスク**: 低。MCPサーバー側の実装が順序を正しく処理していれば問題ない。ただし、テンプレート内にネストされたプレースホルダー（例: `{内部{サブ}構造}`）はないため、正規表現で波括弧内の文字列をグローバルマッチングすることで安全に処理可能。

### 問題5: エラーメッセージ埋め込みの信頼性

**重要度: 低程度**

CLAUDE.md 280-281行「{MCPサーバーから返されたエラーメッセージ全文}」をテンプレートに埋め込む際、エラーメッセージがsubagentの出力の「参考情報」に過ぎない旨が256-279行で警告されている。

**分析**: アーティファクトバリデーターから返されるエラーメッセージが改ざんされる可能性は低い（MCPサーバーのログ出力）。ただし、エラーメッセージ内に特殊文字やMarkdownシンタックスが含まれる場合、Markdown内での不適切な解釈が生じる可能性がある。

**具体例**: エラーメッセージが「同一行: '禁止パターン: `rm -rf`' が3回検出」という形式の場合、バックティックがコードフェンスとして解釈されるリスク。ただし、256-279行の警告文が「参考情報として解釈」するよう指示しているため、subagentが盲目的に実行する可能性は低い。

## セキュリティ改善提案

### 改善案1: CLAUDE.mdのコマンド一覧の拡張と同期

変更個所: CLAUDE.md 240-244行

現在の形式を以下のように拡張:

```
カテゴリ別コマンド:
- readonly: ls, pwd, cat, head, tail, less, more, wc, file, find, grep, rg, ag,
  git status, git log, git diff, git show, git branch, git ls-files, git ls-tree, git rev-parse, git remote
- testing: npm test, npm run test, npx vitest, npx vitest run, npx jest, npx mocha, npx ava,
  npx tsc --noEmit, npx eslint, npx prettier --check, npm run lint, npm run type-check
```

**効果**: 実装とドキュメントの同期を強化し、subagentが許可されたコマンドを正確に認識。

### 改善案2: テンプレートに同期警告を追加

変更個所: CLAUDE.md 246-247行に追加

```
注意: このコマンド一覧はbash-whitelist.jsの簡潔版です。実際の許可判定はフックで実施されます。
特にオプション（--noEmit, -f等）や複合コマンド（npm run lint等）の詳細な検証が行われます。
```

**効果**: subagent側が「ホワイトリストに記載されたコマンドは全て許可される」という誤解を防止。

### 改善案3: エラーメッセージの安全な埋め込み

変更個所: CLAUDE.md 280-281行

現在: `{MCPサーバーから返されたエラーメッセージ全文}`

改善後:
```
{エラーメッセージはMarkdownエスケープして埋め込む}
エラーメッセージ内の特殊文字は参考情報として扱い、リテラルに処理しないこと。
```

または、コードフェンスを使用:

```
\`\`\`
{MCPサーバーから返されたエラーメッセージ全文}
\`\`\`
```

**効果**: エラーメッセージのMarkdown解釈リスクを低減。

## フックによる最終検証の有効性

bash-whitelist.jsの実装を分析した結果、以下の保護層が確認された:

1. **ホワイトリストベースの検証** (779-829行): コマンドチェーンの各部分がホワイトリストに明示的に含まれることを確認。

2. **ブラックリスト多層防御** (756-764行): rm -rf, chmod +x等の危険パターンは全フェーズで禁止。

3. **ゼロ幅文字のサニタイズ** (17-27行): 隠蔽されたシェル投入の防止。

4. **複合コマンド分割とクォート追跡** (308-371行): && || ; による連結を正しく処理。

5. **環境変数保護** (646-698行): セキュリティ関連変数（HMAC_STRICT等）の設定・変更をブロック。

CLAUDE.mdのテンプレートはこれらの層の上で動作するため、phaseGuideへの信頼できるプレースホルダー埋め込みのみに依存する設計は適切である。

## 結論

**セキュリティ判定: 合格（軽微な改善点あり）**

Bashコマンドホワイトリスト伝達メカニズムは、以下の理由から十分に防御されている:

1. プレースホルダー置換はMCPサーバーの信頼できるロジックが担当
2. ユーザー入力は定型テンプレート内に埋め込まれない（taskNameはテンプレート構造を変化させない）
3. フックの多層防御により、subagentの誤った使用も検出
4. エラーメッセージは参考情報として明示的に警告されている

ただし、CLAUDE.mdとbash-whitelist.jsのコマンド定義の同期リスクが中程度であり、以下の対応を推奨:

- 定期的なCLAUDE.mdの更新プロセス確立
- コマンド一覧の自動生成メカニズム導入の検討
- テンプレート内に同期警告の追記

これらの改善により、セキュリティリスクはほぼ消滅し、保守性が向上する。
