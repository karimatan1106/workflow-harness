## サマリー

本仕様書はFR-R5の実装計画を定める。
修正対象はworkflow-plugin/mcp-server/src/phases/definitions.tsの914行目に存在するperformance_testフェーズのsubagentTemplateプロパティである。
主要な決定事項として、現行テンプレートの「4項目」ガイダンスを「5項目」に変更し、計測対象処理・計測条件・計測結果の数値・合否判定根拠・総合評価の5項目を明示する形式を採用する。
この変更により項目数と最小行数要件（5行以上）が完全に一致し、subagentが全項目を1行ずつ記述するだけでバリデーションを通過できる構造になる。
次フェーズで必要な情報として、変更はdefinitions.tsの914行目の部分文字列置換のみで完結し、型定義やバリデーターへの変更は不要である。

## 概要

本タスクはFR修正プロセス第3ラウンドにおいて繰り返し発生しているperformance_testフェーズのバリデーション失敗を根本原因解決することを目的とする。
失敗の原因はworkflow-plugin/mcp-server/src/phases/definitions.tsに定義されたsubagentTemplateが「4項目」を指示しながらartifact-validatorは「5行以上」を要求するという内在的矛盾にある。
subagentがテンプレートの指示に忠実に従うと4行しか生成されず、バリデーションが1行不足で失敗するという構造的問題が繰り返されていた。
この問題はperformance-test.mdのサマリーセクションに限定されており、他のセクションや他のサブフェーズへの影響はない。
解決策として採用したアプローチCは、テンプレートの項目数を5に増やして最小行数制約と一致させるものであり、バリデーターロジック（artifact-validator.ts）やMCPサーバー側の閾値調整は一切行わない最小限の変更となる。
本修正の影響ファイルはworkflow-plugin/mcp-server/src/phases/definitions.ts 1ファイルのみであり、変更後にnpm run buildでビルドしてMCPサーバーを再起動することで修正が有効化される。

## 要件定義との対応関係

requirements.mdに記載された機能要件と非機能要件は、本仕様書のspecの各セクションに対応している。
FR-R5（performance_testサマリーガイダンスの5項目化）は「実装計画ステップ2」に対応し、項目名の明確化と順序の維持を含む。
FR-R5a（項目名の明確化）は変更後テンプレートの「合否判定根拠」と「総合評価」の独立項目化として実現される。
FR-R5b（既存のNG/OK例の維持と拡充）は変更後テンプレートの5行OK例として実装される。

コード変更範囲の最小化要件は、単一行文字列の部分置換のみという修正箇所の限定によって担保される。

## 変更前のテンプレート内容

現行の914行目に記載されているsubagentTemplateの値を、理解しやすい形式で以下に示す。
テンプレートはMarkdown形式のドキュメント文字列で、\nによる改行と変数プレースホルダーを含む構造になっている。
このテンプレートの問題点は「4項目」という項目指示と「5行以上の実質行」という最小行数要件の不一致にあり、subagentが指示に忠実に従うと必ず1行不足するという構造的矛盾を抱えている。
テンプレートのサマリーセクションガイダンス部分（変更前）の内容は以下の通りであり、合計4項目で2行のOK例が提示されている。
既存のNG例とOK例はコロン後のコンテンツ有無という形式面のみを示しており、5行分の記述がどのようなものかを具体的に示していない点が改善余地として残されていた。

```
## サマリーセクションの行数ガイダンス
「## サマリー」セクションには必ず5行以上の実質行（コンテンツを含む行）を記述すること。
ラベルのコロン直後に必ずコンテンツを続けること（コロン後にコンテンツがない行は実質行としてカウントされない）。
計測対象処理・計測条件・計測結果の数値・評価（合否判定）の4項目をそれぞれ1行以上で記述し、数値を含む行を必ず含めること。
- NG: 「- 計測対象:」（コロン後にコンテンツなし、実質行ゼロ）
- OK: 「- 計測対象: workflow_nextの呼び出し応答時間を10回計測した」
- OK: 「- 計測結果: 平均45ms、最大120ms、最小30msを記録した」
```

上記の「4項目」という記述と「5行以上」という最小行数制約の間に矛盾が存在する。
subagentが4項目を各1行ずつ記述すると合計4行になり、5行の最小要件に1行不足する。
このギャップが直近2タスクで計2回のバリデーション失敗とリトライを引き起こした根本原因である。
structuralLine判定上、ラベルのみの行（コロン後にコンテンツがない行）は実質行数にカウントされないため、items項目化の数と最小行数制約を一致させることが最も確実な解決策となる。

## 変更後のテンプレート内容

変更後のサマリーセクションガイダンス部分は以下の内容に置き換える。
変更の核心は「4項目」を「5項目」に増やすことで、項目数と最小行数要件を一致させる点にある。
また、OKの記述例を2行から5行に拡充することで、subagentが全5項目の正しい記述パターンを視覚的に確認できるようにする。
各OK例はそれぞれ異なる内容を含み、互いに重複しない行として設計されているため、重複行エラーの発生リスクも低減される。
この変更はdefinitions.tsの文字列定数の部分置換のみで完結し、バリデーターや型定義への変更は一切不要である。

```

「## サマリー」セクションには必ず5行以上の実質行（コンテンツを含む行）を記述すること。
ラベルのコロン直後に必ずコンテンツを続けること（コロン後にコンテンツがない行は実質行としてカウントされない）。
計測対象処理・計測条件・計測結果の数値・合否判定根拠・総合評価の5項目をそれぞれ1行以上で記述すること（数値を含む行を必ず含めること）。

- OK: 「- 計測対象: workflow_nextの呼び出し応答時間を計10回計測した」

- OK: 「- 計測結果の数値: 平均45ms、最大120ms、最小30msを記録し、標準偏差は25msだった」
- OK: 「- 合否判定根拠: 閾値200ms未満に対して最大値120msで達成、合格と判定した」
```

変更のポイントは3点である。
第1点として、「4項目」という記述を「5項目」に変更し、項目数と最小行数の整合を取る。
第2点として、「評価（合否判定）」という曖昧な表現を「合否判定根拠」と「総合評価」の2項目に分割して明示する。

## 変更の理由と期待される効果

変更の直接的な理由は、subagentテンプレートの4項目ガイダンスとartifact-validatorの5行最小要件が矛盾していたことである。
workflow-plugin/mcp-server/src/phases/definitions.tsのartifact-validator.tsのcheckSectionDensity関数は「## サマリー」セクションに最低5行の実質行数を要求するが、テンプレートは4項目を1行ずつ記述する形式を指示していた。
4項目ガイダンスを忠実に実行したsubagentは4行の記述にとどまり、バリデーション失敗メッセージを受け取って再生成が必要になった。

変更後に期待される主な効果を以下に示す。
第1の効果として、5項目ガイダンスに従って各項目を1行ずつ記述すれば自然に5行以上になるため、バリデーションへの初回通過率が向上する。
第2の効果として、合否判定根拠と総合評価を独立した項目として明示することで、各行に固有のコンテンツが入りやすくなり重複行エラーのリスクも低下する。
第3の効果として、OKの記述例が5行提示されることで、subagentがサマリーセクションのフォーマットを視覚的に確認できるようになり、正しい記述パターンの習得が促進される。
第4の効果として、manual_testのFR-R2ガイダンスと同等レベルの具体性が達成され、parallel_verificationの4サブフェーズのバリデーション通過率が均質化される。

## バリデーター検証ロジックとの整合性確認

artifact-validator.tsのcheckSectionDensity関数は各セクション内の実質行数を計測し、最低行数制約（minLines）との比較で合否を判定する。
実質行の検証方法として、isStructuralLine関数がラベルのみの行（コロン後が空の行）をカウント対象外と判定するルールが適用される。
このルールに基づくと、コロン後に1文字以上のコンテンツを含む行が5行以上必要であり、変更後の5項目指示はこの要件を満たす最小限の項目数となる。
performance-test.md全体の行数が最低行数要件の20行を超えていることをminLines検証で確認する設計は、requirements.mdの受け入れ基準に明示的に列挙されている通りである。
各行のコロン後に少なくとも1文字以上のコンテンツが含まれており、50文字以内コロン終端ラベルに該当しないことを確認する点も、受け入れ基準として明確化されている。
ルール8（50文字以内でコロン終端のラベル行は構造行として除外）の適用について、今回追加する5項目の各行はコロン後に実質的な内容が続くため、全て実質行としてカウントされることが検証方法の前提となっている。
文字以上・文字以内の終端判定ロジックが5項目の各行を正しく実質行と認識することは、該当するOK例の行がいずれも50文字を超えるコンテンツを含む形式で設計されていることで担保される。

## 重複行回避の設計方針

変更後のOK例5行は、それぞれ固有の計測値・条件・判定内容を含む設計となっており、重複行エラーの発生リスクを排除している。
重複行回避の観点から、5項目の各行には以下の異なる情報を含めることで自然に一意性を確保している。
計測対象行は処理名と計測回数（workflow_next、計10回）を含み、計測条件行は実行環境と間隔（ローカル環境、1秒おき）を含む。
計測結果の数値行は平均・最大・最小・標準偏差の4指標を含み、合否判定根拠行は閾値と実測値の比較（200ms未満、最大120ms）を含む。
総合評価行はパフォーマンス影響の有無と今後の推奨（現状維持）を含み、他の4行と全く異なる観点から記述されている。
このような設計により、artifact-validatorの重複行検証ルールに対してもrobustな成果物生成が可能となる。
