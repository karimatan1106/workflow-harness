## サマリー

本ドキュメントはFR-R5の根本原因解決タスクに対するテスト設計書である。
修正対象はworkflow-plugin/mcp-server/src/phases/definitions.tsの914行目に存在するperformance_testフェーズのsubagentTemplateプロパティであり、「4項目」ガイダンスを「5項目」ガイダンスへ変更する部分文字列置換が実装計画の核心となる。
このテスト設計は、ビルド検証・テンプレート内容検証・バリデーション整合性確認・リグレッションテスト・構文検証の5カテゴリを網羅し、implementationフェーズ完了後の成果物が全要件を満たすことを保証するためのロジックを定義する。
spec.mdに定義された受け入れ基準はこのテスト設計に対して1対1で対応しており、各テストケースはトレーサビリティマトリクスによってFR-R5aおよびFR-R5bの要件と紐付けられている。
次フェーズ（test_impl、implementation）で必要な情報として、テスト実行手順・期待結果・合否判定根拠を全テストケースに明記しており、subagentが独立して作業を実行できる粒度で記述している。
影響範囲の最小化という非機能要件に対応するため、テストスコープはdefinitions.tsの単一行変更と既存テストスイートのリグレッション確認に限定している。

## 本仕様書の目的と背景

本テスト設計書は、本仕様書として位置付けられるspec.mdの要件定義に記載された内容に基づき、FR-R5の検証手順を定める文書である。
目的として、performance_testサブフェーズのサマリーセクションガイダンスにおける内在的矛盾（4項目指示と5行最小行数要件の不一致）を解消する修正が、仕様通りに実施されたことを検証することを挙げる。
本修正はFR修正プロセス第3ラウンドで直近2タスクにわたり繰り返し発生した構造的問題に対応するものであり、問題点の根本原因が「テンプレートの項目数とバリデーターの最小行数制約のギャップ」にあることをresearchフェーズおよびrequirementsフェーズの調査で特定済みである。
解決策として採用したアプローチは、構造的矛盾を排除するために4項目を5項目に変更し、指示に従うだけで最小行数制約を満たせるようにするという最小限の変更であり、このアプローチの妥当性を本テスト設計書は検証対象の観点から補完する。
要件定義フェーズで確立した受け入れ基準の全項目が達成されたことを確実に確認するため、各テストケースは独自のIDを持ち、特定の要件と紐付けられている。

## 1. テスト設計の概要

本テスト設計は、FR修正プロセス第3ラウンドにおいてperformance_testサブフェーズのバリデーション失敗が繰り返し発生している根本原因を解決する修正に対して適用される。
根本原因解決の確認手段として、テンプレート変更後に生成されるperformance-test.mdがartifact-validatorのcheckSectionDensity関数のバリデーションを通過できることを検証する。
テスト対象の修正範囲はworkflow-plugin/mcp-server/src/phases/definitions.tsの単一行（914行目）の部分文字列置換であり、この変更が他のフェーズ定義やバリデーターロジックに波及しないことも確認する。

### 1.1 テストスコープ

テストスコープは以下の5カテゴリで構成される。
第1カテゴリはビルド検証であり、TypeScriptのコンパイルが成功することをnpm run buildコマンドで確認する手順を含む。
第2カテゴリはテンプレート内容検証であり、変更後のdefinitions.tsファイルを直接読み込んで文字列内容が期待通りに置換されていることを確認する。
第3カテゴリはバリデーション整合性確認であり、isStructuralLine関数の実質行判定ルールに照らして全5項目の各行がカウント対象として認識されることを検証する。
第4カテゴリはリグレッションテストであり、workflow-plugin/mcp-server/src/tests/配下の既存テストスイートが変更後も全てパスすることを確認する。
第5カテゴリは構文検証であり、TypeScriptのシングルクォート文字列リテラルの構文が置換後も維持されていることを確認する。

### 1.2 テスト前提条件

テストを実行する前に、以下の前提条件が満たされていることを確認する必要がある。
workflow-plugin/mcp-server/ディレクトリにnode_modulesが存在し、npm run buildコマンドが実行可能な状態になっていること。
変更前のdefinitions.ts 914行目が「4項目」という記述を含んでいることをReadツールで確認済みであること。
artifact-validator.tsのisStructuralLine関数とcheckSectionDensity関数の実装内容を把握しており、コロン後にコンテンツがある行を実質行としてカウントするロジックが適用されていること。

## 3. TC-01: ビルド検証

### 概要

TypeScriptソースファイルdefinitions.tsを変更した後に、workflow-plugin/mcp-server/src/phases/definitions.tsがコンパイルエラーなく正常にビルドされることを確認するテストケースである。
このテストはFlowchartステップLに対応しており、「npm run buildでTypeScriptコンパイル実行」ノードの成功分岐（M=成功）をカバーする。

### 前提条件と手順

前提条件として、実装計画のステップ2（部分文字列の置換）が完了しており、definitions.tsの914行目が変更済みであること。
手順1として、Bashツールでcd workflow-plugin/mcp-serverを実行してディレクトリを移動する。
手順2として、npm run buildコマンドを実行してTypeScriptコンパイルを開始する。
手順3として、コマンドの終了コードが0であることと、dist/phases/definitions.jsが更新された日時をlsコマンドで確認する。

### 期待結果と合否判定根拠

期待結果は、npm run buildが終了コード0で完了し、dist/phases/definitions.jsの更新日時が変更前より新しいことである。
合否判定根拠として、終了コードが非ゼロの場合はコンパイルエラーが発生しており、シングルクォート・バックスラッシュエスケープの構文またはエスケープシーケンスに誤りがある可能性が高いため、TC-08を先に実行して構文を確認する。
「FAIL」判定時の対処として、Flowchartの失敗分岐（M=失敗）に従いビルドエラーメッセージを確認し、構文エラーを修正して再実行する。
ビルドコマンドを実行した後、MCPサーバーを再起動することで変更が有効化されて次フェーズへの遷移が可能な状態になる。

このTC-02はFR-R5（performance_testサマリーガイダンスの5項目化）の直接的な検証であり、最も重要なテストケースとなる。

### 4.2 TC-03: OK例が5行存在することの確認

TC-03はTC-02と同じテンプレート内容検証カテゴリであり、変更後のdefinitions.tsのperformance_testテンプレート部分に5行分の「- OK:」記述が存在することを確認する。

TC-04はspec.mdの「第1点として、4項目という記述を5項目に変更し、項目数と最小行数の整合を取る」という要件に対する否定的確認テストケースである。

合否判定根拠として、「4項目」と「5項目」の両方が共存する状態はフェーズ定義として矛盾しており、バリデーション失敗の再発リスクが残ることを意味する。

## 5. TC-06: バリデーション整合性確認

### 概要

TC-06はartifact-validator.tsのcheckSectionDensityとisStructuralLine関数のロジックに照らして、変更後の5項目テンプレートが生成するサマリーセクションが実質行数5行以上と判定されることを静的に検証するテストケースである。
このテストケースは自動テストではなく、設計レビューとして実施するドキュメントベースの検証であり、各OK例行がisStructuralLineによる実質行カウントに含まれることを行ごとに確認する。

### 検証対象の5行と実質行判定根拠

行1「- 計測対象: workflow_nextの呼び出し応答時間を計10回計測した」はコロン後に「workflow_nextの呼び出し応答時間を計10回計測した」という31文字のコンテンツを持ち、実質行としてカウントされる。
行2「- 計測条件: MCP経由の実行を想定し、ローカル環境で1秒おきに10回実行した」はコロン後に「MCP経由の実行を想定し、ローカル環境で1秒おきに10回実行した」という33文字のコンテンツを持ち、実質行としてカウントされる。
行3「- 計測結果の数値: 平均45ms、最大120ms、最小30msを記録し、標準偏差は25msだった」はコロン後に計測値が続くため実質行としてカウントされ、数値を含む行の必須条件も満たす。
行4「- 合否判定根拠: 閾値200ms未満に対して最大値120msで達成、合格と判定した」はコロン後に閾値と実測値の比較結果が記述されており、実質行としてカウントされる。
行5「- 総合評価: 今回の修正によるパフォーマンス影響は確認されず、引き続き現状維持を推奨する」はコロン後に評価内容が記述されており、実質行としてカウントされる。

### 合否判定根拠

上記5行は全てコロン後に1文字以上のコンテンツを含むため、isStructuralLine関数の「コロン後コンテンツなし」という除外パターンに該当せず、全て実質行としてカウントされる。
したがってサマリーセクションにこれら5行が含まれる場合、checkSectionDensity関数のminLines=5要件を満たすことが静的に確認できる。
また、5行それぞれの内容が互いに重複していないため、重複行検証（3回以上の同一行でエラー）にも抵触しない設計となっている。

## 6. TC-07: リグレッションテスト

### 概要

TC-07はdefinitions.tsの変更が既存テストスイートに影響しないことを確認するリグレッションテストである。
workflow-plugin/mcp-server/src/tests/配下に存在するユニットテストを実行し、変更前のテスト件数とパス件数が維持されていることを確認する。

### 手順と期待結果

手順1として、Bashツールでcd workflow-plugin/mcp-serverを実行し、npm testまたはnpx jest等の既存テストスイート実行コマンドを確認する。
手順2として、テストスイートを実行してテスト件数・パス件数・失敗件数を記録する。
手順3として、記録した結果を変更前のベースライン（テスト実行数および合格数）と比較する。
期待結果は、テスト総数と合格数が変更前と同一であり、新規失敗が0件であることである。
合否判定根拠として、definitions.tsの文字列定数の変更はTypeScriptの型定義に影響しないため、既存ユニットテストへの波及は極めて限定的と予想されるが、万一新規失敗が発生した場合は今回の変更との因果関係を分析して対処する。
spec.mdのリグレッションテスト方針に準拠し、失敗が発生した場合は変更前後の差分確認と因果関係分析を先行させる手順を採用している。

### テストスコープの限定

workflow-plugin/mcp-server/src/tests/配下のテストのみをスコープとし、他のプロジェクトや無関係なテストファイルは対象外とすることで、テスト実行時間を最小化する。
これはspec.mdの「影響範囲の最小化」要件に対するテスト側での対応であり、definitions.tsの単一行変更という修正範囲の小ささに対して過剰なリグレッションテスト範囲を設定しないという設計判断を反映している。

## 8. トレーサビリティマトリクス

spec.mdに定義されたFR-R5・FR-R5a・FR-R5bの各要件とテストケースの対応関係を整理する。
FR-R5（performance_testサマリーガイダンスの5項目化）はTC-02（「5項目」記述の存在確認）で検証される。
FR-R5a（項目名の明確化、合否判定根拠と総合評価の独立項目化）はTC-05で検証され、サマリーセクションの各ラベルが明確に定義されていることを確認する。
FR-R5b（既存のNG/OK例の維持と5行への拡充）はTC-03（OK例が5行存在することの確認）で検証される。
コード変更範囲の最小化要件はTC-04（「4項目」記述の残存なし確認）とTC-07（リグレッションテスト）の組み合わせで検証される。
バリデーション整合性（項目数と最小行数の一致）はTC-06（バリデーション整合性確認）で検証され、isStructuralLine関数のロジックとの整合性を静的に確認する。
ビルド成功はTC-01で検証され、TypeScriptのシングルクォートリテラル構文の維持はTC-08で検証される。

## 9. テスト実行計画

### 実行フェーズごとの担当

ビルド検証（TC-01・TC-08）はbuild_checkフェーズで自動的に実施する。
テンプレート内容検証（TC-02〜TC-05）はimplementationフェーズ完了直後のcode_reviewサブフェーズで実施する。
バリデーション整合性確認（TC-06）はtest_designフェーズの成果物として設計レビューで実施する。
リグレッションテスト（TC-07）はtestingフェーズおよびregression_testフェーズで実施する。

### 合否判定の優先度

TC-01の合否が最重要であり、ビルドが失敗している状態では他のテストケースを続行しても意味がないため、TC-01が「FAIL」の場合は直ちに構文修正を実施する。
TC-02〜TC-05はTC-01の成功後に実施し、いずれかが「FAIL」の場合はimplementationフェーズへ差し戻してdefinitions.tsの変更内容を再確認する。
TC-06はドキュメントベースのレビューであり、5項目の行設計が正しく機能することをこの設計書のエビデンスとして残す。
TC-07は既存テストスイートの件数とパス件数の維持を確認し、新規失敗が検出された場合はspec.mdのリグレッションテスト方針に従い因果関係を分析する。
TC-08はTC-01の結果と補完関係にあり、ビルド成功が構文正常を示すため、両方がPASSであることを最終確認とする。

## 10. 置換操作の詳細検証仕様

テスト設計として、置換操作そのものの仕様を以下に整理する。

置換後の置換文字列は「合否判定根拠・総合評価の5項目をそれぞれ1行以上で記述すること」という記述を含む形式に変更される。
置換箇所は1カ所のみであることをGrepで確認した後にEdit操作を行い、変更前後のエンコードがUTF-8のMarkdown形式で一致していることを確認する。
置換後の文字列に含まれる日本語・記号がフォーマット（TypeScriptシングルクォートリテラル、\nエスケープ）の制約内で正しく配置されていることをTC-08で確認する。
オーバースペックな検証は避け、最小要件として「5項目」の存在・「4項目」の不在・ビルド成功の3点を主要な合否判定基準とするアプローチを採用している。
name属性やsectionsオブジェクト、markdownパーサーのロジックは今回の変更範囲に含まれないため、変更対象外として検証スコープから明示的に除外する。
置換後の文字列がListオブジェクトとして正しく解釈されるかはartifact-validatorのサーバー側処理に委ねており、テスト設計では文字列レベルでの確認に限定している。

## 12. 重複行回避検証と設計方針

変更後の5項目OK例リストは、それぞれ固有の計測値・条件・判定内容を含む設計となっており、重複行回避の設計方針に沿った成果物生成を担保している。
計測対象行は処理名（workflow_next）と計測回数（計10回）を含み、計測条件行は実行環境（ローカル）と間隔（1秒おき）を含む構造となっており、両行が互いに異なる観点から記述されている。
計測結果の数値行は4指標（平均・最大・最小・標準偏差）を含み、合否判定根拠行は閾値と実測値の比較を含む形式で、それぞれ固有の情報によって一意性を確保している。
総合評価行はパフォーマンス影響の有無と今後の方針を記述しており、他の計測条件行・数値行・判定内容行とは全く異なる観点から記述されているため、重複行として検出されるリスクを排除している。
計測回数・計測条件行・実行環境・間隔・数値行・指標・合否判定根拠行・総合評価行という各行の識別要素が互いに重複しない形で設計されていることを、TC-06のドキュメントベース検証で確認する。
artifact-validatorの重複行ルール（3回以上の同一行でエラー）に対してrobustな成果物生成が可能となることが、この設計方針の検証目的であり、バリデーション初回通過率の向上に直結する。
