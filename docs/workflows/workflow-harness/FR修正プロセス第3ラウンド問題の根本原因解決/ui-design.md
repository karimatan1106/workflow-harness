## サマリー

本タスクの変更対象はworkflow-plugin/mcp-server/src/phases/definitions.tsの914行目に存在するperformance_testフェーズのsubagentTemplateプロパティの文字列定数のみである。
変更の内容は「4項目」という記述を「5項目」に置き換え、OKの記述例を2行から5行に拡充する文字列の部分置換である。
この修正はMCPサーバーの内部実装に閉じた変更であり、ユーザーが日常的に操作するCLIコマンド体系には一切影響しない。
エラーメッセージの文言、APIレスポンスの構造、設定ファイルのスキーマのいずれも変更を受けず、ユーザーが認識できるインターフェースの変化は存在しない。
次フェーズのimplementationで実施する作業はEditツールによる1箇所の文字列置換とnpm run buildによるビルド確認のみであり、UIやAPIの設計変更は不要と判断した。

## CLIインターフェース設計

本タスクで実施する変更はCLIインターフェースに影響を与えない。
workflow_start、workflow_next、workflow_status、workflow_complete_subなどのMCPツール関数のシグネチャは一切変更されない。
ユーザーが日常的に実行するコマンド群（workflow start、workflow next、workflow approve等）はすべて変更前と同一の引数・オプション体系を維持する。
CLIコマンドの応答メッセージ形式、コマンド一覧の出力、ヘルプテキストのいずれも修正対象に含まれていない。
今回の修正は、MCPサーバーが内部的にsubagentへ渡すテンプレート文字列の内容を調整するものであり、CLI層には伝播しない変更範囲に限定されている。
CLI設計として新規に追加・削除・変更される要素は存在せず、現行のCLI仕様をそのまま維持する。

## エラーメッセージ設計

本タスクの変更対象ファイルであるdefinitions.tsは、subagentTemplateというプロパティを通じてsubagentへのガイダンス文字列を保持するファイルである。
同ファイルに定義されたエラーメッセージ定数、例外クラス、バリデーションエラーの文言は今回の変更箇所ではなく、修正の影響を受けない。
artifact-validator.tsが出力するバリデーション失敗メッセージの文言も変更されない。
今回の修正によって変わるのはsubagentがテンプレート通りに記述した場合に生成される成果物の行数であり、エラーメッセージのフォーマット自体は現行通りである。
修正前と修正後で、ユーザーが受け取るworkflow_nextのエラーレスポンスの構造・言語・フォーマットは同一のまま維持される。
エラーメッセージ設計として新規に追加・変更・削除されるメッセージ定義は存在しない。

## APIレスポンス設計

workflow_next、workflow_status、workflow_complete_subなどのMCPサーバーAPIが返すレスポンスオブジェクトの構造は変更されない。
レスポンスのJSONスキーマ（フィールド名・型・必須項目・オプション項目）はすべて現行の定義を維持する。
今回修正するperformance_testフェーズのsubagentTemplateはsubagentへのプロンプト生成に使用される内部文字列であり、APIレスポンスのデータモデルとは分離された実装である。
成功レスポンスのフォーマット（phase、status、message等のフィールド）も、エラーレスポンスのフォーマット（errorCode、errorMessage等のフィールド）も、いずれも変更されない。
APIバージョン、エンドポイント名、認証方式についても本タスクの変更範囲外であり、既存の設計を引き継ぐ。
APIレスポンス設計として新規に定義されるフィールドや削除されるフィールドは存在しない。

## 設定ファイル設計

本タスクの実装ステップで修正するファイルはworkflow-plugin/mcp-server/src/phases/definitions.tsの1ファイルに限定されている。
mcp.json、settings.json、workflow-state.json、task-index.jsonなどの設定ファイルはすべて変更の対象外である。
definitions.tsはTypeScriptのソースファイルであり、エンドユーザーが直接編集する設定ファイルではなく、MCPサーバーの内部定数ファイルとして分類される。
同ファイル内のフェーズ定義オブジェクトのうち、変更されるのはperformance_testキーのsubagentTemplateプロパティのみであり、他の全フェーズ定義はそのまま維持される。
設定ファイルのスキーマ変更、デフォルト値の追加・削除、環境変数の新規導入はいずれも本タスクのスコープに含まれない。
ビルド後に生成されるdist/phases/definitions.jsはnpm run buildによって自動的に更新されるが、これはソース変更の副産物であり設定ファイルの設計変更には該当しない。

## 運用手順への影響

本タスクの修正はMCPサーバーの内部テンプレート文字列の変更であるため、運用担当者が参照する手順書の更新は不要である。
ワークフローの起動手順（workflow start コマンド）、フェーズ移行手順（workflow next コマンド）、承認手順（workflow approve コマンド）はすべて変更前と同一の操作手順を維持する。
障害対応手順においてperformance_testフェーズのsubagentTemplateを参照している記述がある場合、修正後のテンプレート内容が5項目に拡充された点のみ認識しておけば十分である。
監視・アラートの設定についても本タスクでは変更が発生せず、既存の監視ルールをそのまま継続適用できる。
定期メンテナンス手順に本タスクの変更内容を反映するアクションは発生しない。

## 互換性と移行計画

本タスクで行う変更は後方互換性を完全に維持しており、既存のワークフロータスクの再実行や移行作業は不要である。
変更前に作成済みのworkflow-state.jsonファイルは変更後もそのまま読み取り可能であり、フェーズ状態やHMAC整合性に影響しない。
performance_testフェーズを含む既存の進行中タスクが存在する場合、修正後のMCPサーバーで続行した際に生成されるsubagentプロンプトが新しいテンプレートに基づくものに切り替わるが、動作の正確性は維持される。
definitions.tsの変更は後述のビルドステップを経てdist/phases/definitions.jsに反映されるため、ソース変更後にnpm run buildを実行するまでは実行中のMCPサーバーに変更が反映されない点を認識しておく必要がある。
移行計画として特別なデータ変換やスキーマ移行スクリプトの実行は一切不要であり、ビルドとMCPサーバー再起動のみで変更が有効化される。
