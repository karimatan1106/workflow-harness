## サマリー

- 目的: definitions.tsのperformance_testフェーズsubagentTemplateに加えられた変更内容を手動で検証し、変更が設計意図通りに正確に適用されていることを確認する
- 主要な決定事項: 変更対象ファイルはC:\ツール\Workflow\workflow-plugin\mcp-server\src\phases\definitions.tsの914行目付近であり、performance_testエントリのsubagentTemplateプロパティが対象である
- 検証観点: 「4項目」から「5項目」への数値変更、「評価（合否判定）」を「合否判定根拠」と「総合評価」の2項目への分割、OK例の行数増加、NG例の維持、テンプレート文字列全体の整合性の5点を確認した
- 検証方法: ReadツールによるTypeScriptソースファイルの直接読み取りおよびGrepツールによるキーワード検索を組み合わせて実施した
- 次フェーズで必要な情報: 全シナリオが合格であり、security_scanおよびperformance_testフェーズの並列実行に進むことができる

## テストシナリオ

### シナリオ1: 「5項目」の文字列存在確認

変更後のsubagentTemplate内に「5項目」という表現が存在し、旧来の「4項目」が残存していないことを確認する。
- 確認対象ファイル: C:\ツール\Workflow\workflow-plugin\mcp-server\src\phases\definitions.ts
- 確認対象行: performance_testエントリのsubagentTemplateプロパティ（914行目）
- 合格条件として「5項目」の存在と「4項目」の不在を要求した
- Grepツールでのパターンマッチングによりperformance_testエントリ全体を対象に検索を実施した

### シナリオ2: 「合否判定根拠」と「総合評価」への分割確認

旧来の「評価（合否判定）」という単一項目が削除され、代わりに「合否判定根拠」と「総合評価」の2つの独立した項目として記述されていることを確認する。
- 旧表現「評価（合否判定）」は設計変更前のテンプレートに存在していた単一複合項目である
- 新設計では合否判定の根拠と総合的な評価を分離することで成果物の具体性を高める意図がある
- 両方の新表現がテンプレート内に存在し、かつ旧表現が残存しないことを合格条件とした

### シナリオ3: OK例の行数確認（5行化）

変更前のテンプレートにはOK例が2行のみ存在していた箇所が、変更後は5行のOK例に拡充されていることを確認する。
- 計測対象、計測条件、計測結果の数値、合否判定根拠、総合評価の各項目に対して1行ずつOK例が定義されているかを検証した
- 各OK例が「- OK:」形式のプレフィックスで始まる具体的な記述例を含んでいることを確認対象とした
- 5つの項目ラベルがそれぞれの対応するOK例とともに存在していることを合格条件とした

### シナリオ4: NG例の維持確認

変更によってNG例が削除されていないことを確認する。NG例は「コロン後にコンテンツがない行は実質行ゼロとしてカウントされない」という学習ポイントを示す重要なガイダンスである。
- 「- 計測対象:」のような空ラベル行のNG例が1つ存在し続けていることを確認対象とした
- NG例の削除は成果物品質要件の理解を妨げるため、必ず維持されている必要がある
- 「コロン後にコンテンツなし、実質行ゼロ」という説明文が隣接して存在することも確認した

### シナリオ5: テンプレート文字列全体の構文整合性確認

performance_testエントリのsubagentTemplateプロパティ全体が正しいTypeScript文字列リテラルとして構文的に整合していることを確認する。
- テンプレートの開始引用符と終了引用符が正しく対応していること
- performance_testエントリの前後（security_scanエントリとe2e_testエントリ）のコードブロックが変更前後で損なわれていないこと
- performance_testとe2e_testの間の区切り文字やネスト構造が維持されていることを確認対象とした
