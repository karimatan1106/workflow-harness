# UI設計: ワークフロープラグイン大規模対応根本改修

## 概要

本ドキュメントは、ワークフロープラグインの6つのセキュリティ・品質機能（REQ-1〜REQ-6）における**CLI インターフェース設計**を定義する。

視覚的なUIは存在せず、全ての操作は**コマンドラインの標準出力・エラー出力**を通じて行われる。エラーメッセージ、警告、確認プロンプトのフォーマットを統一し、ユーザーが問題を素早く理解し対応できるようにする。

---

## UI設計

### 設計原則

1. **明確性**: エラーの原因を一目瞭然にする
2. **行動性**: ユーザーが次に何をすべきかを明確にする
3. **一貫性**: 全フックで統一されたフォーマットを使用
4. **簡潔性**: 不要な情報を排除し、必須情報のみを表示

### フォーマット規定

#### エラーメッセージテンプレート
```
[フック名] エラー内容 - {詳細}
詳細説明（複数行可）
```

#### 警告メッセージテンプレート
```
[フック名] 警告: メッセージ
追加情報
```

#### 成功メッセージテンプレート
```
[フック名] OK: メッセージ
```

### カラーコード（推奨）

- **エラー**: 赤（`\x1b[31m`）
- **警告**: 黄（`\x1b[33m`）
- **成功**: 緑（`\x1b[32m`）
- **情報**: 青（`\x1b[34m`）
- **リセット**: `\x1b[0m`

## コンポーネント仕様

### REQ-1: FAIL_OPEN除去

#### 1.1 エラーメッセージ（統一フォーマット）

**件名**: エラー発生時の表示（フック失敗）

**表示タイミング**: 例外が発生した場合、常に表示

**メッセージ形式**:
```
[{フック名}] {エラー種別} - ブロック

{詳細説明}
```

**具体例**:

**例1: uncaughtException**
```
[enforce-workflow] 未捕捉エラー - ブロック
エラーの詳細: ReferenceError: undefined variable
スタックトレース: (省略可)

このエラーはセキュリティフックの障害です。
管理者に連絡してください。
```

**例2: JSONパースエラー**
```
[enforce-workflow] JSONパースエラー - ブロック
状態ファイルの読み込みに失敗しました。
ファイル: .claude/state/workflows/task123/workflow-state.json
エラー: Unexpected token } at position 45

ファイルが破損している可能性があります。
ファイルを削除して、タスクを再度開始してください。
```

**例3: stdinエラー**
```
[enforce-workflow] stdinエラー - ブロック
標準入力の読み込みに失敗しました。

このエラーはシステムレベルの問題です。
コマンドの実行環境を確認してください。
```

#### 1.2 環境変数警告（非推奨）

**件名**: FAIL_OPEN環境変数の無視

**表示タイミング**: FAIL_OPEN環境変数が設定されている場合（ログレベル）

**メッセージ形式**:
```
[{フック名}] 警告: FAIL_OPEN環境変数は無視されます
セキュリティフックは常にfail-closed動作を行います。
```

#### 1.3 ブロック確認メッセージ

**件名**: フックがコマンド実行をブロックした場合

**表示タイミング**: フェーズルール違反など、正常な拒否時

**メッセージ形式**:
```
[{フック名}] エラー - ブロック
理由: {具体的な理由}

次のアクション: {推奨される対応方法}
```

**具体例**:
```
[enforce-workflow] エラー - ブロック
理由: タスク初期化前のコマンド実行

次のアクション: 以下のコマンドでワークフローを開始してください:
  /workflow start "タスク名"
```

### REQ-2: HMAC署名

#### 2.1 署名検証失敗メッセージ

**件名**: 状態ファイル改竄の検出

**表示タイミング**: `readTaskState()` で署名検証に失敗した場合

**メッセージ形式**:
```
[WorkflowStateManager] 署名検証失敗: {ファイルパス}

エラー: タスク状態ファイルが改竄されている可能性があります。

原因と対応:
  1. 手動でファイルを編集した場合
     → ファイルを削除して、タスクを再度開始してください
  2. ファイルが破損した場合
     → .claude/state/workflows/{taskId}/ ディレクトリを削除し、
        タスクを再度開始してください
  3. マシンが異なる場合
     → 異なるマシンでは状態ファイルの互換性がありません
        同じマシンで作業を続行してください

詳細: HMAC署名 = {期待値}
     実際 = {実際の値}
```

#### 2.2 署名マイグレーション警告

**件名**: 署名なし状態ファイルの検出と自動修正

**表示タイミング**: `readTaskState()` で署名フィールドがない場合

**メッセージ形式**:
```
[WorkflowStateManager] 警告: 署名なしファイルを検出 - 署名を追加します
ファイル: {ファイルパス}

自動マイグレーション実施:
  - HMAC-SHA256署名を生成
  - stateIntegrity フィールドを追加
  - ファイルを保存

今後のアクセス時は署名検証が実行されます。
```

#### 2.3 署名付き保存の確認メッセージ

**件名**: タスク状態の保存成功

**表示タイミング**: `writeTaskState()` 実行後（ログレベル）

**メッセージ形式**:
```
[WorkflowStateManager] 状態保存: OK
ファイル: {ファイルパス}
署名: {署名値の先頭16文字}...
タイムスタンプ: {ISO 8601形式}
```

#### 2.4 エラーハンドリング

**件名**: タスク取得時の署名検証失敗

**表示タイミング**: `getTaskById()` で署名検証に失敗した場合

**メッセージ形式**:
```
タスクが見つかりません

理由: タスク状態ファイルが改竄またはロックされています

対応方法:
  /workflow list              # 他のアクティブなタスクを確認
  /workflow reset {理由}      # 調査フェーズにリセット
  rm -rf .claude/state/...    # 状態ファイルの削除と再初期化
```

### REQ-3: スコープサイズ制限

#### 3.1 スコープ超過エラーメッセージ

**件名**: ファイル数がMAX_SCOPE_FILES（200）を超過

**表示タイミング**: `workflowSetScope()` 実行時

**メッセージ形式**:
```
[set-scope] エラー - スコープが大きすぎます

ファイル数: {実際の件数}件
上限: 200件
超過: {超過数}件

原因:
  タスクの範囲が大きすぎるため、スコープ制限に達しました。

対応方法:
  タスクを機能単位に分割してください。

分割例:
  ✓ ユーザー認証機能
  ✓ 決済処理機能
  ✓ 通知機能
  ✓ レポート生成機能

各機能ごとに別タスクを作成することで、スコープを小さく保てます。
```

**例（具体的な数値）**:
```
[set-scope] エラー - スコープが大きすぎます

ファイル数: 250件
上限: 200件
超過: 50件

原因:
  タスクの範囲が大きすぎるため、スコープ制限に達しました。

対応方法:
  タスクを機能単位に分割してください。

分割例:
  ✓ ユーザー認証機能（50ファイル）
  ✓ 決済処理機能（60ファイル）
  ✓ 通知機能（40ファイル）
  ✓ レポート生成機能（100ファイル）

各機能ごとに別タスクを作成することで、スコープを小さく保てます。
```

#### 3.2 ディレクトリ数超過エラーメッセージ

**件名**: ディレクトリ数がMAX_SCOPE_DIRS（20）を超過

**表示タイミング**: `workflowSetScope()` 実行時

**メッセージ形式**:
```
[set-scope] エラー - スコープが大きすぎます

ディレクトリ数: {実際の件数}件
上限: 20件
超過: {超過数}件

原因:
  タスクが影響する機能モジュールが多すぎます。

対応方法:
  関連する機能を1つのタスクでまとめるか、
  タスクを分割してください。

分割例（推奨）:
  ✓ src/features/auth/
  ✓ src/features/payment/
  ✓ src/features/notification/
  ✓ src/features/reporting/

各ディレクトリごとに別タスクを作成することで、
スコープを小さく保てます。
```

#### 3.3 next実行時のスコープ検証エラー

**件名**: parallel_analysis → parallel_designへの遷移時にスコープが設定されていない

**表示タイミング**: `workflowNext()` がparallel_design に遷移する際

**メッセージ形式**:
```
[workflow_next] エラー - スコープが未設定です

エラー内容: 次のフェーズへの遷移にはスコープ設定が必須です

対応方法:
  以下のコマンドで影響範囲を設定してください:

  /workflow set-scope \\
    --files src/features/auth/index.ts src/features/auth/service.ts ... \\
    --dirs src/features/auth/ src/shared/

  または spec.md で `## 影響範囲` セクションを作成してください。
```

### REQ-4: Bash解析強化

#### 4.1 複合コマンド検出エラーメッセージ

**件名**: 危険な複合コマンドが検出された

**表示タイミング**: `analyzeBashCommand()` で危険なパターンが検出された場合

**メッセージ形式**:
```
[phase-edit-guard] エラー - ブロック

理由: ファイル変更コマンドが含まれています

検出されたコマンド:
  {危険なパート}

フルコマンド:
  {元のコマンド}

制限事項:
  このフェーズでは、以下のコマンドは実行できません:
  - ファイル・ディレクトリの作成・修正・削除
  - データベースの更新
  - システム設定の変更
  - 外部ツールの実行

許可されるコマンド（read-only）:
  ✓ ls, cat, grep, find, tree
  ✓ git status, git log, git diff
  ✓ npm list, npm outdated
  ✓ node --version, python --version
```

#### 4.2 パイプコマンド検出エラー

**件名**: パイプで危険なコマンドが連結されている

**表示タイミング**: パイプ（|）で危険なコマンドが検出された場合

**メッセージ形式**:
```
[phase-edit-guard] エラー - ブロック

理由: パイプで危険なコマンドが連結されています

検出: cat file.txt | bash
      └─ bash コマンドは実行できません

問題:
  パイプの右側に bash が指定されていますが、
  bash コマンドの実行はセキュリティ上許可されていません。

対応方法:
  1. ファイルの内容を確認するだけの場合:
     $ cat file.txt | head -20

  2. テキスト処理の場合:
     $ cat file.txt | grep "pattern"
```

#### 4.3 awk単一リダイレクト検出エラー

**件名**: awkでファイルに直接出力されている

**表示タイミング**: awk ... > file パターンが検出された場合

**メッセージ形式**:
```
[phase-edit-guard] エラー - ブロック

理由: ファイル変更コマンドが含まれています

検出されたコマンド:
  awk 'BEGIN{print "x"}' > file.ts

問題:
  リダイレクト（> または >>）を使用して、
  新規ファイルまたは既存ファイルにデータを書き込もうとしました。

このフェーズでは、ファイルの作成・修正はできません。

対応方法:
  - リダイレクトなしで出力確認:
    $ awk 'BEGIN{print "x"}'

  - 標準出力で結果を確認してから、
    implementation フェーズでファイル作成を行ってください。
```

#### 4.4 セミコロン連結のエラー

**件名**: セミコロンで複数コマンドが連結されている

**表示タイムング**: ; で複合コマンドが実行される場合

**メッセージ形式**:
```
[phase-edit-guard] エラー - ブロック

理由: ファイル変更コマンドが含まれています

検出: pwd; rm -rf /
      └─ rm -rf / はファイル変更コマンド

問題:
  複数のコマンドをセミコロン（;）で連結した場合、
  最初のコマンドが成功しても次のコマンドが実行されます。

このような複合コマンドは、セキュリティリスクがあるため
このフェーズでは実行できません。

対応方法:
  各コマンドを個別に実行してください:
  1. 安全なコマンドのみが必要な場合:
     $ pwd
     $ git status
```

### REQ-5: 成果物検証

#### 5.1 未作成ファイルエラー

**件名**: 必須成果物ファイルが作成されていない

**表示タイミング**: `checkRequiredArtifacts()` でファイルが見つからない場合

**メッセージ形式**:
```
[check-workflow-artifact] エラー - ブロック

未作成の必須成果物があります:

  - requirements.md
  - spec.md

対応方法:
  上記のファイルを docs/workflows/{タスク名}/ に作成してください。

ファイル内容:
  - requirements.md: 要件定義（## 機能要件 セクション必須）
  - spec.md: 実装仕様（## 実装計画 セクション必須）
```

#### 5.2 サイズ不足エラー

**件名**: 成果物ファイルがサイズ下限（200バイト）を下回っている

**表示タイミング**: `validateArtifactContent()` でサイズチェックに失敗した場合

**メッセージ形式**:
```
[check-workflow-artifact] エラー - ブロック

内容が不正な成果物があります:

  spec.md:
    - サイズ不足: 150バイト（最小: 200バイト）
    - 必須セクションが見つかりません: ## 実装計画 または ## アーキテクチャ

対応方法:
  spec.md にファイル説明や実装方針を追加してください。

最小限の例:
  # 仕様書

  ## 実装計画

  このプロジェクトは以下の要件に基づいています:
  - 認証機能の追加
  - パフォーマンス改善
  - セキュリティ対応

  ## アーキテクチャ

  Clean Architecture に基づいて実装します。
```

#### 5.3 必須セクション欠落エラー

**件名**: Markdownファイルに必須セクション（ヘッダー）がない

**表示タイミング**: `validateArtifactContent()` でセクション検証に失敗した場合

**メッセージ形式**:
```
[check-workflow-artifact] エラー - ブロック

内容が不正な成果物があります:

  requirements.md:
    - 必須セクションが見つかりません: ## 機能要件 または ## 背景

対応方法:
  requirements.md に以下のセクションのいずれかを追加してください:

  ## 機能要件

  [具体的な要件を記述]

  または

  ## 背景

  [背景説明を記述]

成果物を修正してから次フェーズに進んでください。
```

#### 5.4 禁止パターン検出エラー

**件名**: スタブまたは空のコンテンツが検出された

**表示タイミング**: `validateArtifactContent()` で禁止パターン（TODO, WIPなど）が検出された場合

**メッセージ形式**:
```
[check-workflow-artifact] エラー - ブロック

内容が不正な成果物があります:

  threat-model.md:
    - 禁止パターンが検出されました: ファイルがスタブまたは空です

対応方法:
  threat-model.md の内容を実際の脅威分析に置き換えてください。

許可されない内容:
  ✗ TODO（のみ）
  ✗ WIP（のみ）
  ✗ ヘッダーのみで本文がない

要件:
  - 実質的な内容を200バイト以上含むこと
  - 必須セクション（## 脅威 または ## リスク）を含むこと
  - スタブコメントを含まないこと
```

#### 5.5 複数ファイルの複合エラー

**件名**: 複数のファイルで検証エラーが発生

**表示タイミング**: 複数のファイルが検証に失敗した場合

**メッセージ形式**:
```
[check-workflow-artifact] エラー - ブロック

内容が不正な成果物があります:

  requirements.md:
    - サイズ不足: 120バイト（最小: 200バイト）
    - 必須セクションが見つかりません: ## 機能要件 または ## 背景

  spec.md:
    - 禁止パターンが検出されました: ファイルがスタブまたは空です

  test-design.md:
    - ヘッダーのみで本文がありません

成果物を修正してから次フェーズに進んでください。

修正優先度:
  1. requirements.md - セクション追加と内容拡充
  2. spec.md - TODO, WIP コメント削除と実装記述
  3. test-design.md - テストケース記述
```

### REQ-6: 設計検証

#### 6.1 空実装検出エラー

**件名**: ソースコード内に空メソッドやスタブが検出された

**表示タイミング**: `detectStubImplementations()` で空実装が検出された場合

**メッセージ形式**:
```
[設計検証] エラー - ブロック

空実装・スタブが残っています:

  src/services/user-service.ts:
    - メソッドボディが空です: getUser()
    - スタブ実装が残っています: createUser() - TODO
    - スタブ実装が残っています: updateUser() - FIXME

  src/models/order.ts:
    - NotImplementedError を投げるスタブ実装: calculateTotal()
    - クラスボディが空です: class OrderProcessor

  src/controllers/payment-controller.ts:
    - メソッドボディが空です: processPayment()

対応方法:
  上記のスタブを完全に実装してください。

実装例（計算ロジック）:
  // 前: スタブ
  calculateTotal() {}

  // 後: 実装
  calculateTotal(): number {
    return this.items.reduce((sum, item) =>
      sum + (item.price * item.quantity), 0);
  }

実装例（NotImplementedError を削除）:
  // 前: スタブ
  processPayment() {
    throw new NotImplementedError();
  }

  // 後: 実装
  processPayment(amount: number): boolean {
    return this.gateway.charge(amount);
  }

空実装を完了させてから次フェーズに進んでください。
```

#### 6.2 環境変数無視警告

**件名**: SKIP_DESIGN_VALIDATION 環境変数が設定されている場合の警告

**表示タイミング**: 設計検証が実行される場合（ログレベル）

**メッセージ形式**:
```
[設計検証] 警告: SKIP_DESIGN_VALIDATION 環境変数は無視されます

セキュリティポリシーにより、設計検証は常に実行されます。
この環境変数を使用してスキップすることはできません。

理由:
  空実装やスタブコードを検出し、品質を確保するため。
```

#### 6.3 設計検証成功メッセージ

**件名**: 全ての空実装検出に合格した場合

**表示タイミング**: 設計検証に成功した場合（ログレベル）

**メッセージ形式**:
```
[設計検証] OK: 全て実装済み

検証結果:
  - 空メソッド: 0個
  - スタブ: 0個
  - 空クラス: 0個
  - NotImplementedError: 0個

次のフェーズへの遷移が許可されました。
```

#### 6.4 部分的な空実装エラー

**件名**: 一部のファイルのみ空実装が検出された

**表示タイムング**: 複数ファイルのうち一部に空実装がある場合

**メッセージ形式**:
```
[設計検証] エラー - ブロック

空実装・スタブが残っています:

  ✓ src/services/user-service.ts - OK（8メソッド実装済み）
  ✗ src/services/payment-service.ts - NG（3個のスタブ検出）
  ✗ src/models/transaction.ts - NG（2個の空メソッド検出）
  ✓ src/controllers/api.ts - OK（12メソッド実装済み）

詳細:

  src/services/payment-service.ts:
    - メソッドボディが空です: refund()
    - スタブ実装が残っています: validateAmount() - TODO
```

## エラーコード体系

| コード | HTTPステータス | 説明 | 対応 |
|--------|---------------|------|------|
| `BLOCK` | N/A | フック実行エラー（exit code 2） | コマンド中止、ユーザーアクション待機 |
| `ALLOW` | N/A | フック成功（exit code 0） | コマンド続行 |
| `ALLOW_WITH_WARN` | N/A | フック成功＋警告（exit code 0） | コマンド続行、警告表示 |
