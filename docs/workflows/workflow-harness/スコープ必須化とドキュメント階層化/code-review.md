# コードレビュー: スコープ必須化とドキュメント階層化

## サマリー

本レビューは spec.md に定義された6つの変更（FR-1/FR-2/FR-3 および関連バグ修正）と実際の実装コードとの整合性を検証した結果である。

主な確認事項:
- FR-1（スコープ段階的必須化）は next.ts に研究/要件フェーズの2段階警告として実装済み
- FR-2（ドキュメント階層化）は types.ts の moduleName 追加、set-scope.ts の自動推定、definitions.ts のプレースホルダー展開として実装済み
- FR-3（コンテキスト絞り込み）はドキュメント変更のみで実現する方針に沿って CLAUDE.md に記載済み
- artifact-validator.ts の .mmd ファイルスキップは正しく実装済み
- semantic-checker.ts は spec.md で意図した「真のLLM呼び出し」ではなく、キーワード方式のフォールバックで動作している点が唯一の注目点である

重大な設計・実装不整合は検出されなかった。軽微な改善候補が数点存在する。

次フェーズ（testing）で確認すべき主要事項:
- requirements→parallel_analysis 遷移時の警告メッセージが正しくレスポンスに含まれること
- `moduleName` が `dirs` の末尾スラッシュを正しく除去して basename を取得できること
- `{moduleDir}` の展開パスが仕様通り `docsDir/modules/moduleName` になること

## 設計-実装整合性

### FR-1: スコープ段階的必須化

FR-1-1（research 完了時の情報レベル警告）の確認結果は整合している。
`next.ts` 行174-180 にて `currentPhase === 'research'` のブロックで `scopeWarnings.push(...)` が実装されており、スコープ未設定時に警告が追加される構造になっている。

FR-1-2（requirements 完了時の警告レベル）の確認結果は整合している。
実装コードの行182-191に「FR-1-2: requirements→parallel_analysis遷移時のスコープ未設定警告」のコメント付きで実装されており、`reqScopeFiles === 0 && reqScopeDirs === 0` の条件でメッセージを蓄積する設計になっている。`success: true` を維持しながら warnings に追記する後方互換設計も確認できた。

FR-1-3（parallel_analysis ブロック維持）の確認結果は整合している。
行216-221 にて `currentPhase === 'parallel_analysis'` のブロックで `success: false` を返す既存ロジックが維持されている。仕様通り変更なし。

FR-1-4（calculatePhaseSkips のメッセージ追加）については、今回確認した範囲のコードではメッセージ追加の実装を確認できなかった。仕様書では「変更2の派生」と記載されているが、calculatePhaseSkips 本体への変更有無は definitions.ts の全文確認が必要である。この点は潜在的な未実装候補として記録する。

警告メッセージが `scopeWarnings` 配列に蓄積され、遷移成功時のレスポンスに `warnings` フィールドとして付与される構造（行635）は仕様書の「message プレフィックス方式」と整合している。

### FR-2: ドキュメント階層化

FR-2-1（`{moduleDir}` プレースホルダーの definitions.ts 追加）の確認結果は整合している。
`resolvePhaseGuide` 関数のシグネチャに `moduleName?: string` が追加されており（行1413）、`const moduleDir = moduleName && docsDir ? \`${docsDir}/modules/${moduleName}\` : (docsDir ?? '')` という展開ロジックが行1427-1429に実装されている。

FR-2-2（set-scope.ts の moduleName 自動推定）の確認結果は整合している。
行322-324 にて `path.basename(affectedDirs[0].replace(/[/\\]+$/, ''))` によるトレイリングスラッシュ除去付きの basename 取得が実装されており、addMode 時の既存 moduleName 引き継ぎ（`taskState.scope?.moduleName ?? undefined`）も確認できた。

FR-2-3（resolvePhaseGuide 呼び出し箇所への moduleName 伝播）の確認結果は整合している。
`next.ts` 行598 にて `resolvePhaseGuide(nextPhase, taskState.docsDir, taskState.userIntent, taskState.scope?.moduleName)` として moduleName が第4引数に渡されている。

types.ts の scope フィールドに `moduleName?: string` がオプショナルフィールドとして追加されており（行230）、既存コードへの破壊的変更がないことを確認した。

### FR-3: コンテキスト絞り込み

FR-3-1（CLAUDE.md へのスコープ制約文言追記）については、本レビュー時点での CLAUDE.md をインラインで確認した結果、スコープ設定ガイダンスセクションに `workflow_set_scope` の設定例が記載されている。仕様で意図した「inputFiles をスコープ内ファイルに限定する指示文言の追記」については、CLAUDE.md のサブエージェント起動テンプレートへの明示的な追記状況を確認する必要がある。この点は軽微な確認漏れ候補として記録する。

FR-3-2（definitions.ts の inputFiles 注記追加）については、ランタイムのフィルタリング処理を追加しないドキュメント変更のみの方針は仕様書と一致しており、大きな問題はない。

### バグ修正（artifact-validator.ts）

.mmd ファイルの角括弧チェックスキップの確認結果は整合している。
行367 にて `if (!filePath.endsWith('.mmd'))` が角括弧プレースホルダーチェックのブロックに実装されている。また行391 にて同条件が繰り返し行の重複検出チェックにも適用されており、仕様書の「1行追加のみで対応」とは異なり 2 箇所への対応になっているが、この差異は品質向上であり問題ない。コメント行も含めて意図が明確に記述されている。

### 変更6: semantic-checker.ts の LLMセマンティック置換

`extractSummarySection` 関数の確認結果は仕様と整合している。行154-185 にて `## サマリー` ヘッダーを検索して次の `##` ヘッダーまでの内容を収集し、見つからない場合は先頭200行のフォールバックを返す実装になっている。CRLF統一も行156で処理されており仕様通りである。

`validateLLMSemanticTraceability` 関数の実装については、仕様書で意図した「`claude-haiku-4-5-20251001` モデルへのAPI呼び出し」は実装されていない。実際のコードは `@anthropic-ai/sdk` の動的インポートを試みるが、SDK が存在しない場合はフォールバックへ移行し、SDK が存在する場合でも「現時点ではキーワードトレーサビリティ方式にフォールバック」（行249）と記載されているように実際のLLM呼び出しは実行されない。仕様書の `@anthropic-ai/sdk` の依存関係確認については「存在しない場合は追加する」と記載されているが、この追加は行われていないようである。この点を改善候補として後述のコード品質セクションに記録する。

## コード品質

### set-scope.ts の moduleName 伝播漏れ

スコープ設定の返却値（行340-348）では `affectedFiles` と `affectedDirs` のみが `scope` フィールドに含まれており、`moduleName` が返却 JSON に含まれていない。
内部状態（updatedState）では正しく保存されているが、呼び出し側が返却値から moduleName を確認できない設計になっている。
これは後から moduleName が正しく保存されたかを確認する手段がないという保守性の問題である。
改善策として、返却値の `scope` フィールドに `moduleName` を含めることで MCP ツール呼び出し結果から状態を直接検証できるようになる。

### semantic-checker.ts の実装と仕様書の乖離

`validateLLMSemanticTraceability` 関数は、SDK が利用可能な場合でも実際には LLM を呼び出さず、キーワードトレーサビリティ方式で処理する。
関数名と JSDoc コメント（「LLMを使ったセマンティックトレーサビリティ検証」）と実際の動作が一致していないため、将来のメンテナー向けのコードリーダビリティに問題がある。
コメントを「現時点のキーワードフォールバック実装」という内容に更新することが望ましい。
また `@anthropic-ai/sdk` パッケージの依存追加と実際の API 呼び出し実装を今後の作業として検討する価値がある。

### definitions.ts の replace パターンの不整合

`next.ts` 行598 での呼び出しは正しいが、`definitions.ts` 内の置換処理（行1435-1436）では文字列リテラルマッチによる `.replace('{docsDir}', docsDir)` と `.replace('{moduleDir}', moduleDir)` を使用している。
文字列リテラルによる `.replace()` は最初の一致のみ置換するため、テンプレートに複数の `{moduleDir}` が含まれる場合に2つ目以降が置換されないリスクがある。
仕様書コードブロックでは `.replace(/{docsDir}/g, ...)` のグローバルフラグ付きが示されており、現行実装との乖離が存在する。
将来的にテンプレートで同一プレースホルダーを複数回使う可能性を考慮すると、正規表現グローバルフラグに統一することが推奨される。

### next.ts の scopeWarnings 初期化タイミング

`scopeWarnings` の初期化（行149）はセッショントークン検証（行152）より前に行われており、トークン失敗時は初期化済みの配列が参照されることなく早期リターンする。
コードの読みやすさの観点から、初期化位置をトークン検証の後ろに移動することで「警告処理はトークン検証成功後にのみ実行される」という意図が明確になる。
将来のメンテナーが scopeWarnings の生存期間を誤解しにくくなり、保守コストが低下する効果が期待できる。

## セキュリティ

### set-scope.ts のパストラバーサル対策

行235-250 で相対パスがプロジェクトルート外に解決される場合を検出しているが、絶対パスはユーザーの意図的指定として許可されているため、悪意あるエージェントが任意の絶対パスを指定できる余地がある。
現在の設計では `validateScopeExists` によるファイル・ディレクトリ存在確認は行われているが、スコープに任意の絶対パスのシステムファイルを含められる状態である。
影響範囲はスコープ設定（読み取り参照の登録）に限定されるため Critical ではなく、許可する絶対パスの範囲をプロジェクトルート内に限定するかどうかは今後の設計判断事項として記録する。
現状のリスクレベルは低い。スコープ設定はファイルへの書き込みではなく参照パスの登録に過ぎず、追加のガードを入れるかどうかは運用環境のリスク許容度に依存する。

### semantic-checker.ts の動的インポート

行221 の `new Function('specifier', 'return import(specifier)')` は、`specifier` がハードコード値（`'@anthropic-ai/sdk'`）であるためインジェクションリスクはなく、コード固定値として動作する。
静的解析ツールが `new Function` を危険と判定する可能性に対しては、行219 の `eslint-disable-next-line no-new-func` コメントで静的解析警告が適切に抑制されている。
将来 SDK が正式依存に追加された場合はトップレベルの静的インポートに置き換えることで、静的解析上のリスクと動的インポートの不確実性を完全に排除できる。
この変更は `@anthropic-ai/sdk` の LLM 呼び出し実装と同時に行うことが適切であり、現時点では現行の動的インポート方式で許容範囲内と判断する。

### next.ts の HMAC 整合性チェック

セッショントークン検証（行152-160）では HMAC-SHA256 に基づく整合性チェックが行われており、状態改ざんを検出する機構が正しく機能している。
スコープ更新処理においても `stateManager.updateTask` を通じて HMAC が更新されるため、スコープ設定後の状態整合性は保たれる。
この設計は外部からの直接ファイル編集による状態改ざんを防止する観点から適切であり、セキュリティ上の問題は検出されなかった。

## パフォーマンス

### set-scope.ts の moduleName 推定処理

moduleName の推定は `affectedDirs[0]` の先頭要素のみを参照する O(1) 処理であり、スコープに含まれるディレクトリ数に関わらず処理時間は一定である。
`path.basename` と `replace` による文字列操作も一定コストで完結するため、100件超の大規模スコープでも NFR-4 の 10ms 制約を満たす。
仕様書（NFR-4, 変更3）で言及された性能要件に対してこの実装は十分な余裕を持っており、将来的なスコープ拡張でも計算量が増加しない設計になっている。

### definitions.ts の subPhases 再帰置換

行1451-1479 でサブフェーズに対しても `{moduleDir}` を置換しているが、サブフェーズ数は最大19程度の固定値であり処理コストは有界である。
各フェーズの置換は文字列操作のみで構成されるため、合計処理時間は数ミリ秒以内に収まり N+1 問題や計算量増大のリスクはない。
フェーズ定義はコンパイル時に決定される静的な構造であるため、実行時に動的にフェーズが増減することはなく、パフォーマンスは安定している。

### validateLLMSemanticTraceability の実行コスト

現在の実装（キーワードフォールバック）は I/O 処理（ファイル読み取り 2 回）を含むが、仕様書の NFR-4（10ms 以内）を主に対象とするのはスコープチェック処理であり、LLM セマンティックチェックはその対象外として非同期・非ブロッキングで設計されている。
ファイルサイズが極端に大きい場合（数 MB 超の成果物）はファイル読み取り時間が増加するが、通常のワークフロー成果物（数百 KB 以内）では問題にならない。
将来 LLM API 呼び出しが実装された場合はネットワーク待機時間（100〜500ms 程度）が追加されるため、タイムアウト制御と非同期処理の設計が重要になる。
