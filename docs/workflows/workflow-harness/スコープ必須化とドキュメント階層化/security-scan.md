# セキュリティスキャン: スコープ必須化とドキュメント階層化

## サマリー

本スキャンは「スコープ必須化とドキュメント階層化」タスクで変更された以下4ファイルを対象とする。

対象ファイル:
- `workflow-plugin/mcp-server/src/tools/set-scope.ts` — パス操作・ファイルシステムアクセスを含む主要変更ファイル
- `workflow-plugin/mcp-server/src/phases/definitions.ts` — プレースホルダー展開ロジックを含む変更ファイル
- `workflow-plugin/mcp-server/src/tools/next.ts` — フェーズ遷移ロジックの警告追加部分
- `workflow-plugin/mcp-server/src/state/types.ts` — moduleName フィールド追加部分

主要な決定事項:
- パストラバーサル対策コード（行237-250）は適切に機能しており、プロジェクトルート外への相対パスを拒否する設計になっている
- glob展開において用いている自作の正規表現変換には軽微な特殊文字エスケープの漏れが存在するが、実際の影響は限定的
- `{moduleDir}` プレースホルダー展開は文字列結合のみであり、コマンド実行やシェル解釈には関与していない
- HMAC整合性は stateManager.writeTaskState によって維持されており、直接編集を迂回する手段はない
- エラーメッセージにパス情報が含まれるが、ローカルMCPサーバー環境で機密性への影響は軽微

次フェーズで必要な情報:
- glob展開の正規表現特殊文字エスケープ問題は性能劣化よりも機能誤動作リスクが主である点を把握すること
- `MAX_SCOPE_FILES` と `MAX_SCOPE_FILES_LOCAL` の二重定義は混乱源となっており、将来的な整理が推奨される

---

## 脆弱性スキャン結果

### 1. パストラバーサル（set-scope.ts 行235-250）

実装コードは相対パスに対してプロジェクトルートを基準とした正規化チェックを行っている。
`path.normalize(path.resolve(projectRoot, p)).startsWith(normalizedRoot.slice(0, -1))` という比較により、`../` を含む相対パスがプロジェクトルート外に解決される場合は拒否される。
絶対パスは「ユーザーが意図的に指定している」という設計上の判断から許可されており、コメントにも明示されている。
この判断はローカル開発ツールとして適切であり、絶対パスの検証を追加しない設計上の選択は明確に文書化されている。
検出リスク評価: 低（設計上の制約として意図的に許可されている）

### 2. glob展開の正規表現特殊文字エスケープ（set-scope.ts 行152-153）

glob パターンを正規表現に変換する際、`**` と `*` のみを置換しており、`.`、`+`、`(`、`)`、`[`、`]` といった正規表現特殊文字のエスケープ処理が行われていない。
例として、パターン `src/index.ts` を指定した場合、`src/indexXts` のような無関係なパスも誤マッチする可能性がある。
この問題はセキュリティ侵害よりも誤ったファイルのスコープ登録という機能的誤動作として現れる可能性がある。
ただし、glob展開の結果は後続の `validateScopeFiles` および `validateScopeExists` によってファイル存在確認が行われるため、実在しないパスが登録されることはなく、リスクは機能的な誤りに留まる。
検出リスク評価: 低（機能的誤動作のみ。セキュリティ境界への影響なし）

### 3. インジェクション（definitions.ts 行1427-1429の moduleName 展開）

`moduleName` は `affectedDirs` の先頭要素のバス名から `path.basename()` で取得されており（set-scope.ts 行322-323）、ディレクトリ区切り文字を除いた最後のコンポーネントに限定される。
この値は `${docsDir}/modules/${moduleName}` という文字列結合で展開されるが、展開結果はドキュメントパスの計算に使用されるのみであり、シェルコマンド実行やSQL文字列生成には使用されない。
Mermaid図やMarkdownファイルのパスに展開される用途であるため、インジェクション攻撃の経路として機能する可能性は実質ゼロである。
検出リスク評価: 問題なし

### 4. HMAC整合性（stateManager.writeTaskState 経由のスコープ保存）

set-scope.ts 行337 で `stateManager.writeTaskState` を呼び出しており、スコープ情報の保存は常にstateManagerを経由している。
stateManagerはHMAC-SHA256を更新して状態ファイルを書き出すため、スコープ設定後の整合性は確保される。
moduleName フィールドを追加しても `writeTaskState` のHMAC計算ロジックはstate全体を対象としているため、新フィールドも整合性保護の対象に自動的に含まれる。
検出リスク評価: 問題なし

### 5. 情報漏洩（エラーメッセージ中のパス情報）

`validateScopeFiles`、`validateScopeExists` の返却するエラーメッセージには存在しないパスが含まれ、ユーザーに提示される（行261-275）。
MCP サーバーはローカルプロセスとして動作しており、AIエージェントに対してのみ結果を返す設計であることから、外部ネットワークへの情報漏洩の経路は存在しない。
ローカル開発環境においてはパス情報の開示はデバッグに有用であり、意図的な設計と判断できる。
検出リスク評価: 問題なし（ローカルツールとして適切）

### 6. 入力バリデーション（環境変数からのスコープ上限値取得）

`MAX_SCOPE_FILES_LOCAL` は `parseInt(process.env.SCOPE_MAX_FILES || '10000')` で取得され、`Math.min(Math.max(..., 10), 100000)` でクランプされている。
モジュールスコープで定義された `MAX_SCOPE_FILES` と、関数内で再定義される `MAX_SCOPE_FILES_LOCAL` の二重定義が存在し、関数内では後者が優先される。
二重定義は意図せぬ上限値の差異を引き起こす可能性があるが、実際の処理では関数内の値（`MAX_SCOPE_FILES_LOCAL`）のみが使用されているため動作上の問題は発生しない。
検出リスク評価: 低（コードの可読性・保守性の問題であり、動作上のセキュリティリスクは発生しない）

## 検出された問題

### 問題1: glob展開における正規表現特殊文字の未エスケープ

- 場所: `set-scope.ts` 行152-153
- 深刻度: 低（機能誤動作であり、セキュリティ侵害ではない）
- 内容: `*` と `**` 以外の正規表現特殊文字がエスケープされないため、ピリオドなどを含むパターンで誤マッチが発生し得る
- 後段の存在確認バリデーションがリスクを緩和しているため、悪用可能な境界違反には至らない
- 推奨対応: 正規表現変換前にパターン内のメタ文字をエスケープする処理を追加することが望ましい

### 問題2: スコープ上限値の二重定義

- 場所: `set-scope.ts` 行38-44（モジュールスコープ）および行110-117（関数内スコープ）
- 深刻度: 低（動作上の問題はないが、保守性に影響）
- 内容: `MAX_SCOPE_FILES` と `MAX_SCOPE_FILES_LOCAL` の2種類の上限値定数が異なるデフォルト値（1000 vs 10000）と異なる環境変数名（`MAX_SCOPE_FILES` vs `SCOPE_MAX_FILES`）で定義されており、将来的な誤解・混乱の原因になる可能性がある
- 推奨対応: モジュールスコープの定数を除去し、関数内の定義に統一することが望ましい

### 問題3: 絶対パスのプロジェクトルート外への許可（設計上の意思決定事項）

- 場所: `set-scope.ts` 行236のコメントおよびロジック
- 深刻度: 情報提供のみ（設計上の意図的な選択）
- 内容: 絶対パスによる指定はプロジェクトルート外のパスも受け入れる設計になっており、コメントにもその旨が記述されている
- 現在のユースケース（ローカル開発MCP）では問題とならないが、将来的にマルチユーザー環境での利用が想定される場合は追加の制限が必要になる
- 推奨対応: 現時点での対応は不要だが、用途が拡張される場合に再評価が必要である

---

## 総合評価

全体として、今回の変更によって新たに重大なセキュリティリスクが導入されたとは判断されない。
パストラバーサル対策は相対パスに限定されており設計上の制約として明文化されている。
HMAC整合性は stateManager 経由での書き込みにより維持されている。
moduleName の展開はシェル実行やSQLに至らない文字列結合のみであり、インジェクション経路は存在しない。
検出された2件の軽微な問題（正規表現エスケープ欠落・定数二重定義）は機能品質と保守性の改善候補として記録するが、即時対応を要するリスクではない。
