## サマリー

本要件定義書は、`workflow-plugin/mcp-server/src/phases/definitions.ts` の `manual_test` および `security_scan` フェーズのsubagentTemplateに存在するガイダンス不足を修正するための要件を定義する。

- 目的: FR-11（manual_testの総合評価セクションガイダンス追加）およびFR-12（security_scanの行数計算ロジック明示）を実装し、subagentがバリデーション要件を満たす成果物を安定して生成できるようにする。
- 主要な決定事項: performance_testのFR-9で採用した総合評価ガイダンスパターンをmanual_testへ横展開し、security_scanの行数ガイダンスを定量的な計算例を含む表現に強化する。
- テスト追加: `definitions-subagent-template.test.ts` にFR-11（2テスト）とFR-12（1テスト）を追加し、実装後のテストスイートが全件パスすることを確認する。
- 次フェーズで必要な情報: 修正対象ファイルのパス（definitions.ts と test.ts）、既存FR-9の実装パターン、minLines/minSectionLines設定値。
- 影響範囲: 修正は2ファイルのみに限定。MCPサーバー再起動が修正後に必須である。

## 背景と目的

### 問題の発生背景

`parallel_verification` フェーズの4つのサブフェーズ（manual_test・security_scan・performance_test・e2e_test）は、過去に個別の問題が発生したタイミングで修正が積み重ねられてきた経緯がある。
過去にはFR-1からFR-10までの修正が `definitions.ts` に実装済みであるが、これらは問題が報告されたフェーズにのみ適用されており、類似フェーズへの予防的な横展開がなされていない。
今回のFR-11とFR-12は、このような非対称性から生じるバリデーション失敗を予防的に解消するものである。

### FR-11の問題状況

`manual_test` フェーズのsubagentTemplateには `## テストシナリオ` と `## テスト結果` セクションのガイダンスが記述されているが、`## 総合評価` セクションへのガイダンスが存在しない。
subagentが自己判断でこのセクションを追加した際、ガイダンスなしでは1段落（実質1行）程度の記述に留まることが多く、minSectionLines（5行）要件を満たせない。
performance_testにはFR-9として同等の総合評価ガイダンスが追加済みであるが、manual_testには横展開がなされていなかった。

### FR-12の問題状況

`security_scan` フェーズには `minLines: 20` が設定されているが、行数達成のための具体的な計算例がsubagentTemplateに存在しない。
3つの必須セクション（`## 脆弱性スキャン結果`・`## 検出された問題`・`## サマリー`）それぞれに5行ずつ書いた場合、合計15行となるが、これだけでは目標の20行に届かない。
既存のガイダンスは「内容を充実させて非空行を確保すること」という抽象的な表現のみで、5行のギャップを埋める計算ロジックがsubagentに伝達されていない。

### 修正の目的

FR-11の目的は、`## 総合評価` セクションで5行以上の実質行を確保するための5観点ガイダンスをmanual_testのsubagentTemplateに追加することである。
FR-12の目的は、`minLines: 20` を達成するための計算ロジックを定量的に明示し、subagentが行数目標を俯瞰できるようにすることである。
テストケース追加の目的は、今後のガイダンス変更がFR-11・FR-12の要件を維持するかを自動検証することである。

## 機能要件

### FR-11: manual_testフェーズのsubagentTemplateへの総合評価ガイダンス追加

**修正対象ファイル**: `workflow-plugin/mcp-server/src/phases/definitions.ts`

**修正対象箇所**: `manual_test` サブフェーズの `subagentTemplate` 文字列内、`## テスト結果セクション` ガイダンスの後かつ `## 出力` セクションの前の位置

**追加するガイダンスの内容**:
`## 総合評価` セクションへの記述指針（FR-11）として、以下の5観点それぞれについて1行以上の実質行を記述し、合計5行以上の実質行を確保するよう指示するガイダンスブロックを追加する。

- 観点1: 全テストシナリオの合否サマリーとして、実施件数・合格件数・不合格件数を数値とともに評価した1行以上を記述すること
- 観点2: 検出された問題の有無・件数・深刻度概要として、発見した不具合がある場合はその深刻度を含む評価を記述すること
- 観点3: 未実施シナリオがある場合の理由と代替措置として、全シナリオを実施できた場合は「全シナリオ実施済み」を明記すること
- 観点4: 次フェーズ（security_scan等）への引き継ぎ事項として、後続フェーズで対処が必要な懸念点や確認事項を記述すること
- 観点5: 全体的な品質評価として、合格・条件付き合格・不合格の判定と根拠を具体的に記述すること

**ガイダンスの形式**: `performance_test` の `## ★ 総合評価セクションの記述指針（FR-9）` ブロックと同様のフォーマット（`## ★ 総合評価セクションの記述指針（FR-11）` という見出し）で記述する。

**FR-9の実装パターンの参照**: 既存の performance_test テンプレートの FR-9 ブロックには「5観点それぞれについて1行以上の実質行を記述し、合計5行以上の実質行を確保すること」という構造が採用されており、FR-11でも同じ構造を採用する。

### FR-12: security_scanフェーズのsubagentTemplateへの行数計算ロジック追加

**修正対象ファイル**: `workflow-plugin/mcp-server/src/phases/definitions.ts`

**修正対象箇所**: `security_scan` サブフェーズの `subagentTemplate` 文字列内、既存の「行数カウント仕様と転記防止」セクションに計算ロジックを追記する形で修正する。

**追加する計算ロジックの内容**:
次の計算式を定量的に明示したガイダンスを追加する。

「3つの必須セクション（脆弱性スキャン結果・検出された問題・サマリー）に各セクション最低5行 = 合計15行 + 追加のセクション（総合評価等）で最低5行 = 合計20行以上」

具体的には以下の情報を含めること。
- 必須セクション3つ（`## 脆弱性スキャン結果`・`## 検出された問題`・`## サマリー`）それぞれに最低5行が必要で合計15行となること
- 全体のminLines（20行）に達するには残り5行以上を別のセクションで確保する必要があること
- 追加セクション（`## 総合評価` 等）に5行以上記述することで合計20行以上に達することができること
- この計算根拠を念頭に置いて各セクションの内容量を決定することを指示すること

### FR-11テストケース追加

**修正対象ファイル**: `workflow-plugin/mcp-server/src/phases/__tests__/definitions-subagent-template.test.ts`

**追加するテストスイート名**: `FR-11: manual_testフェーズのsubagentTemplateに総合評価セクションのガイダンスを追加`

**テストケース構成**:
- TC-11-1: `manual_test` フェーズのsubagentTemplateに「総合評価」の文字列が含まれること（`toContain('総合評価')` で検証）
- TC-11-2: `manual_test` フェーズのsubagentTemplateに5観点ガイダンスの代表キーワードが含まれること（`toContain('全テストシナリオ')` 等の形式で検証）

**テストデータの取得方法**:
既存のFR-9・FR-10テストと同様に `resolvePhaseGuide('parallel_verification', ...)` で `parentPhaseGuide` を取得し、`parentPhaseGuide?.subPhases?.manual_test` から `subagentTemplate` を取得する。

### FR-12テストケース追加

**追加するテストスイート名**: `FR-12: security_scanフェーズのsubagentTemplateに行数確保ガイダンスを追加`

**テストケース構成**:
- TC-12-1: `security_scan` フェーズのsubagentTemplateに「20行」の文字列が含まれること（`toContain('20行')` で検証）

**テストデータの取得方法**:
`resolvePhaseGuide('parallel_verification', ...)` で `parentPhaseGuide` を取得し、`parentPhaseGuide?.subPhases?.security_scan` から `subagentTemplate` を取得する。

## 非機能要件

### 後方互換性の維持

既存のガイダンス（行数カウント仕様・禁止語転記防止・重複行回避等）は変更・削除しないこと。
追加するガイダンスは既存のガイダンスブロックの後ろ、出力指定の直前の位置に配置すること。
既存のrequiredSections（manual_testは `['## テストシナリオ', '## テスト結果']`）は変更しないこと。

### テストスイートとの整合性維持

FR-11とFR-12のテストケース追加後も、既存のFR-6〜FR-10のテストケースがすべて合格すること。
追加するテストケースはFR-9・FR-10の実装パターンに準拠した形式で記述すること。
テスト実行時にエラーが発生しないよう、TypeScriptの型安全性を維持した記述を行うこと。

### MCPサーバー再起動の必須要件

definitions.tsを修正した後は、必ずMCPサーバーをビルド（`npm run build`）してプロセスを再起動すること。
再起動を行わない場合、Node.jsのモジュールキャッシュにより変更が反映されず、バリデーション失敗が継続する。
再起動後は `workflow_status` で現在のフェーズを確認してから次の作業を開始すること。

### ガイダンスパターンの一貫性

FR-11のガイダンスは既存のFR-9（performance_test総合評価ガイダンス）と同一のパターン構造を採用すること。
FR-12の計算ロジックは、具体的な数値（3セクション・各5行・合計15行・残り5行・合計20行以上）を含むこと。
追加するガイダンスの文体・記述スタイルは既存のガイダンスブロックと統一すること。

## テスト要件

### テスト実装の方針

テストは `TDD Red` フェーズの原則に従い、実装前に失敗するテストケースとして記述する。
テスト追加後、実装（definitions.tsへのガイダンス追加）完了前の段階でTCが失敗することを確認する。
実装完了後にテストスイート全体（実装前912件 + 追加3件 = 915件以上）がパスすることを確認する。

### FR-11テストケースの検証内容

TC-11-1は `manual_test` テンプレートに「総合評価」という語が含まれるかを検証する（現状不在であり追加後に合格する）。
TC-11-2は `manual_test` テンプレートに5観点のうち代表的な観点（全テストシナリオの合否サマリーに関する語）が含まれるかを検証する（追加後に合格する）。

### FR-12テストケースの検証内容

TC-12-1は `security_scan` テンプレートに「20行」という文字列が含まれるかを検証する（現状不在であり追加後に合格する）。

### リグレッション防止

FR-11・FR-12のテスト追加後も、既存のFR-6〜FR-10の各テストケースが引き続きパスすることを確認すること。
テスト追加時に既存のテストケースのコードを改変しないこと。
テスト実行コマンドは `cd workflow-plugin/mcp-server && npm test` で実施する。
