## サマリー

本ドキュメントは「subagentプロンプト情報不足とOrchestrator直接編集の根本解決」タスクのテスト設計書（リトライ第2版）である。実装仕様書（spec.md）に記載された各変更ファイルの修正前後の記述を基に、各FRの受入基準を静的検証で確認するテストケースを定義する。

目的:
- FR-1〜FR-7の各機能要件に対し、静的検証（ファイル内容の文字列照合）によるテストケースを定義する
- 変更対象ファイルはCLAUDE.mdとworkflow-plugin/mcp-server/src/phases/definitions.tsの2ファイルのみで、ロジック変更を含まない
- 各FRの実装完了を確認するための受け入れ基準を明確にし、変更漏れや誤記を検出できるテストケースを提供する
- 副作用検証として変更対象外ファイル（artifact-validator.ts・state-manager.ts・phase-edit-guard.js）が未変更であることを確認するテストケースも含む
- 非機能要件（NFR-1〜NFR-4）に対するテストケースを追加し、追加行数・プロンプトサイズ・MCPサーバー再起動・三者間整合性を検証する

主要な決定事項:
- テスト手法は全件静的検証（ReadツールでファイルをロードしてGrep照合）とし、実行環境依存の動的テストは別途testingフェーズで行う
- 各テストケースは正常系（期待する文字列の存在確認）と異常系（期待する文字列が欠落した場合の失敗条件）の2つを定義する
- spec.mdの「実装チェックリスト」セクションの全確認項目をテストケースの検証手順に直接反映する
- buildRetryPromptの11エラーパターン（FORBIDDEN_PATTERN・DUPLICATE_LINE・SECTION_MISSING・SECTION_DENSITY・MIN_LINES・PLACEHOLDER・STRUCTURAL_VIOLATION・CODE_FENCE・TABLE_FORMAT・SUMMARY_MISSING・DESIGN_IMPL_MISMATCH）の対応関係も検証対象とする

次フェーズ（実装フェーズ）で必要な情報:
- 各FRの検証に使用するGrep検索パターン（本書のテストケース詳細セクションに記載）
- 定数に定義されたFORBIDDEN_PATTERNSの値とCLAUDE.md記述の一致確認手順
- definitions.ts変更後のMCPサーバー再起動手順（FR-3〜FR-7完了後に実施）— Node.jsモジュールキャッシュによりdist/*.jsのディスク上の更新確認が不可欠
- 副作用検証の対象ファイルリスト（artifact-validator.ts・state-manager.ts・phase-edit-guard.js）

## テスト方針

### 静的検証アプローチ

本タスクの変更内容は文字列リテラルの追記・修正のみであるため、テスト手法は静的検証を採用する。ReadツールまたはGrepツールを使用して対象ファイルを読み込み、追加・修正された文字列パターンの存在を確認する。各テストケースには「対象ファイル」「検索する文字列または正規表現パターン」「期待結果」を明示する。確認方法は静的検証で確認可能な文字列照合であるため、実行環境を問わず再現可能である。

動的テスト（MCPサーバー起動・API呼び出し）はtestingフェーズで実施する対象であり、本test-designでは静的検証の設計を行う。実施者は実装フェーズ完了後に本書のチェックリストを順に確認することで、FR全件の受け入れ基準を満たしているか判定できる。

正常系テスト（実装完了時の期待動作）では、実装されるべき文字列が対象ファイルに存在することを確認する。異常系テスト（修正が欠落した場合の失敗条件）では、文字列が存在しないケースを失敗パターンとして定義する。情報不整合を解消し、正確な仕組みを確立するためには、各FRの成果物が本仕様書で特定済みの対象行番号に沿って修正されていることが前提となる。

### 禁止語検出の検証方針

artifact-validator.tsのFORBIDDEN_PATTERNS定数に定義された禁止語は、コードフェンス外での使用全般が禁止対象となる。以下に禁止語の完全リストをコードフェンス内に示す（テスト検証の参照情報として掲載）。

```
禁止語完全リスト（英語4語・日本語8語）:
英語グループ: TODO, TBD, WIP, FIXME
日本語グループ: 未定, 未確定, 要検討, 検討中, 対応予定, サンプル, ダミー, 仮置き
合計12語。英語部分と日本語部分を合わせた12語が部分一致（includes）で検出される。
複合語に含まれる場合も検出対象。成果物全体でブロックの対象となる。
コードブロック外での使用全般が禁止対象となる。
```

FR-1のテスト検証では、上記12語がCLAUDE.mdの「禁止パターン（完全リスト）」セクションにコードフェンス形式で列挙されているかを照合する。

### 重複行検出ルールの検証方針

isStructuralLine()による除外ルール（8ルール）の正規表現パターン付記を検証する。FR-2では、ルール6の「行頭から2連アスタリスク・任意の文字・行末がコロンのみ」という構造と、ルール8の「50文字以内・コロン終端・省略可能なリスト記号」という構造を表す正規表現が付記されているかを確認する。

ルール6に関する重要な境界条件: 太字ラベルの後にコンテンツが続く行（例: `**深刻度**` の後に `Medium` が続く行）は除外されない。この境界条件を明示的に説明することで、サブエージェントが同一行を繰り返さず各行に固有情報を含めた設計ができるようになる。現状は省略されているこの情報を付記することが修正後の記述方針であり、不明確さにより重複行検出エラーの原因となっている問題を解消する。

ルール8に関する境界条件: 「50文字以内」という文字制限が核心であり、行頭のリスト記号（任意・省略可能）と空白の後にラベルが続き、行末がコロンで終わる形式のみが除外対象となる。50文字を超えるラベル行は除外されず、3回以上の繰り返しで検出エラーとなる。英語表記の正規表現と境界条件を付記形式で明示することが重要である。

### バリデーターとの整合性検証方針

artifact-validator.tsが定義するstructuralLineの除外ルール（DUPLICATE_LINE検出の基盤）と、SECTION_MISSING・SECTION_DENSITY・MIN_LINES・PLACEHOLDER・STRUCTURAL_VIOLATION・CODE_FENCE・TABLE_FORMAT・SUMMARY_MISSING・DESIGN_IMPL_MISMATCHの各エラーパターンは、buildRetryPromptが生成するリトライプロンプトの内容と密接に関連する。FR-6のルール21エスカレーション手順は、これらのエラーパターン別にOrchestratorが取るべき行動を定義しているため、CLAUDE.mdの記述とdefinitions.tsのbuildRetryPrompt実装との整合性を検証する。

phaseGuideはフェーズごとの最低行数・必須セクション・禁止語の完全リストを保持する構造体であり、subagentTemplateと整合していなければならない。FR-3のテストでは、phaseGuideのPHASE_ARTIFACT_REQUIREMENTSに設定されている必須セクション名がsubagentTemplateに明示されているかを照合する。最低行数指示がプロンプト内に含まれているかも確認対象とする。

### buildPrompt生成部の検証方針

定義された保存先以外への出力禁止を強調するため、buildPrompt()のセクション1（フェーズ情報ヘッダー生成部）とセクション9（importantSection）の両方にdocsDirパスへの言及が含まれているかを確認する。生成部分に冒頭と末尾の2箇所への明示的な配置がFR-4の核心である。

bashSectionの生成部分に関しては、実装系コマンドや各カテゴリコマンドと代替ツールの対応表が追加されているかを検証する。代替手段案内のカテゴリには「ファイルコピーの移動」に相当するcp/mvブロック時の対応も含まれることを確認する。制限されたBashコマンドに対して適切な代替手段を案内することで、サブエージェントが実装系コマンドブロック後に正しく代替ツールに切り替えられることを目指す。

### ルール21の再起動義務・直接編集問題の検証方針

失敗回数が増えると無視して直接編集する行動パターンが観察されているため、FR-6ではエスカレーション手順を手順書形式で記述することが求められる。検証対象は以下の要素である。

- 再起動義務として「上限はない」という表現（リトライ回数に制限なし）
- 厳命ルールの補強として禁止指示が全リトライ回数に共通することの明記
- リトライプロンプトへの「前回のバリデーション失敗理由」の引用と「改善指示」の両セクション
- 直接修正（Edit/Writeによる成果物への直接編集）が禁止であることの再確認
- 3回目以降での行番号レベルの注釈を含む修正指示の生成

これらが全て揃って初めてFR-6の実装が完了とみなされる。

### code_review整合性検証方針

FR-7の核心は、code_reviewフェーズのsubagentが設計書4種（spec.md・state-machine.mmd・flowchart.mmd・ui-design.md）を全文読み込みすることにある。現状はspec.mdのみであるため、state-machine.mmd・flowchart.mmd・ui-design.mdを追加することで、全機能・全項目の対応関係を検証できるようになる。

2段階手順として整合性チェックを行う設計であり、第1段階で設計書から「機能一覧」「状態遷移リスト」「処理フロー一覧」「UI要素一覧」を抽出してリストアップし、第2段階で実装コードを走査して各項目が実装されているかを照合して記録することが求められる。これにより、整合性問題・実装整合性の検証が形式的なものから実質的なものになる。差分検証手順が2段階で明記されているかを確認する。

並行して行う作業として、inputFileMetadataへの3ファイルエントリ追加がある。未実装項目の解決確認までcode-review.mdに記録することが要求される。

### 副作用検証方針

副作用防止のため、artifact-validator.ts・state-manager.ts・phase-edit-guard.jsの3ファイルについて、実装前後でファイル内容が変化していないことをgit diffで確認する。spec.mdの「変更対象外のファイル」セクションに明示されたこれら3ファイルへの意図しない変更は、バリデーション動作・HMAC整合性・フック動作に深刻な副作用をもたらす。副作用最小化の原則に基づき、変更対象はCLAUDE.mdとdefinitions.tsの2ファイルに限定する。

### MCPサーバービルド・再起動テスト方針

definitions.tsを変更した後、Node.jsのモジュールキャッシュの仕組みにより、ディスク上のdist/*.jsファイルを更新しても実行中のMCPサーバープロセスには変更が反映されない。強制再起動条件の遵守として、以下の4ステップが必須となる。

1. `workflow-plugin/mcp-server`ディレクトリでnpm run buildを実行してTypeScriptをトランスパイルする
2. dist/*.jsの更新日時を確認して変更がディスクに反映されたことを検証する
3. Claude DesktopのMCPサーバー再起動ボタンでサーバープロセスを再起動してキャッシュを破棄する
4. 再起動後にworkflow_statusを実行して現在フェーズが正常応答を返すことを確認する

このビルド・再起動テスト（TC-BUILD系）はFR-3〜FR-7の変更後に必ず実施する。

### 三者間整合性テスト方針

artifact-validator.ts・definitions.ts・CLAUDE.mdの三者間の整合性維持は本タスクの核心的な目的である。FORBIDDEN_PATTERNSの12語リスト・PHASE_ARTIFACT_REQUIREMENTSの必須セクション名・buildRetryPromptのエラーパターン分類・CLAUDE.mdの各説明セクションが互いに一致しているかを検証する。将来変更時に保守担当者が参照できるよう、整合性維持ルールが文書化されているかも確認する。

各設計書ファイルの指定において、直接読み込まれるファイルのパスが正確であることと、プロンプト本体への重複埋め込みを避けるためにCLAUDE.md参照への誘導が適切に設定されていることを確認する。FORBIDDEN_PATTERNSとCLAUDE.mdの禁止リストの一致確認、および二重掲載の回避を検証項目に含める。
