## サマリー

本ドキュメントは「subagentプロンプト情報不足とOrchestrator直接編集の根本解決」タスク（FR-1〜FR-7）のUI設計書である。本タスクはGUIを持たないCLI/MCPサーバー内部修正（文字列リテラルの変更）であるため、「UI設計」はCLIインターフェース・エラーメッセージ・APIレスポンス・設定ファイル構造の観点で記述する。

目的:
- FR-1〜FR-7の各修正がOrchestrator・subagentとのインタラクションにどう影響するかを明確化する
- 変更後のプロンプト構造・エラーメッセージ形式・設定ファイルセクション構造を確定する
- 実装フェーズのsubagentが参照すべき出力形式の仕様を定める

主要な決定事項:
- MCPコマンドのレスポンス形式は変更しない（FR-4はプロンプト内テキストの拡充のみ）
- バリデーション失敗時のエラーメッセージ形式はartifact-validatorが生成するため変更対象外
- buildRetryPrompt()がsubagentに渡す改善指示の構造は、definitions.tsのテンプレート改訂で対応する
- CLAUDE.mdの新設セクションは「禁止パターン（完全リスト）」「ルール21エスカレーション手順」の2箇所

次フェーズで必要な情報:
- definitions.tsのbuildPrompt()が生成するプロンプト各セクションの現行行番号（test_implフェーズで参照）
- CLAUDE.mdの「### 禁止パターン」見出しの現在位置（実装フェーズでのEdit挿入箇所の確定に必要）
- code_reviewのsubagentTemplateにおけるinputFileMetadata配列の現在の要素数と挿入位置

## CLIインターフェース設計

本セクションでは、FR-1〜FR-7の修正がMCPコマンドの出力形式およびOrchestratorのインタラクション手順に与える変更を記述する。

### workflow_statusコマンドの出力形式（FR-4影響範囲）

FR-4の修正後、Orchestratorはsubagentを起動する前にworkflow_statusを実行してdocsDirの実際の値を取得することが義務づけられる。workflow_statusのレスポンス形式自体は変更しないが、OrchestratorがdocsDirフィールドを参照する手順が明示化される。

workflow_statusコマンドの出力（変更なし・参照形式の例）:
```
{
  "taskId": "{タスクID}",
  "taskName": "{タスク名}",
  "phase": "ui_design",
  "docsDir": "docs/workflows/{タスク名}/",
  "workflowDir": ".claude/state/workflows/{タスクID}_{タスク名}/"
}
```

FR-4修正後のOrchestrator手順の変化: subagentを起動する前にworkflow_statusを実行してdocsDirの値を確認し、その値をsubagentプロンプトの「タスク情報」セクションに展開してから起動することが明示される。従来はプロンプト末尾のみにパス警告があったが、修正後はプロンプト冒頭の「タスク情報」セクションにも同一のdocsDirパスが表示されるため、subagentが誤ったパスに成果物を保存するリスクが低減される。

### workflow_nextコマンドのバリデーションエラー出力（FR-6影響範囲）

FR-6の修正後、workflow_nextがバリデーション失敗を返した場合、Orchestratorがリトライ回数に応じた手順を実行することが義務づけられる。workflow_nextのレスポンス形式自体は変更しないが、Orchestratorの対応手順が細分化される。

workflow_nextの失敗レスポンス形式（変更なし・参照形式）:
```
{
  "success": false,
  "error": "ARTIFACT_VALIDATION_FAILED",
  "message": "成果物のバリデーションに失敗しました: {エラー詳細}",
  "phase": "requirements",
  "retryable": true
}
```

FR-6修正後のOrchestrator対応フロー:
- 1回目失敗時: エラーメッセージをそのままsubagentのリトライプロンプトに引用してTask toolで再起動する
- 2回目失敗時: Orchestratorが成果物ファイルをReadで読み込み、問題箇所の行番号を特定し、具体的な修正指示をリトライプロンプトに追加してTask toolで再起動する。haikuモデル使用時はsonnetへのモデルエスカレーションを検討する
- 3回目以降の失敗時: 必ずsonnetモデルを使用し、成果物の問題セクションを引用した詳細な改善指示をプロンプトに含めてTask toolで再起動する

### workflow_complete_subコマンドの出力形式（FR-6影響範囲）

workflow_complete_subがバリデーション失敗を返した場合も、workflow_nextと同様のリトライ手順が適用される。並列フェーズ（parallel_verification等）でサブフェーズの成果物がバリデーション失敗した際のOrchestratorの対応手順は、FR-6修正後のルール21に基づく。

## エラーメッセージ設計

本セクションでは、FR修正後のバリデーションエラーメッセージの形式と、buildRetryPrompt()がsubagentに渡すリトライプロンプトの構造を記述する。

### artifact-validatorが生成するエラーメッセージ形式（参照情報）

artifact-validatorのエラーメッセージ形式は本タスクでは変更しない。以下は参照情報として記載する。エラー種別は主に4種類あり、それぞれ異なるメッセージ形式で出力される。

FORBIDDEN_PATTERNエラーの形式（禁止語検出時）:
```
禁止語が検出されました: "{検出された語句}" (行{行番号})
成果物内の指摘された語句を削除し、具体的な表現に置き換えてください。
バリデーターが検出する語句は英語グループ4語と日本語グループ8語の計12語である。
```

DUPLICATE_LINEエラーの形式（同一行の繰り返し検出時）:
```
同一行が{回数}回以上検出されました: "{重複行の内容}"
各行に固有の情報を含めて一意性を確保してください。
```

SECTION_MISSINGエラーの形式（必須セクション欠落時）:
```
必須セクションが欠落しています: "{欠落セクション名}"
成果物に以下のセクションを追加してください: {必須セクション一覧}
```

MIN_LINESエラーの形式（行数不足時）:
```
成果物の行数が不足しています: {実際の行数}行 / 最低{必要行数}行
成果物の内容を充実させて{必要行数}行以上にしてください。
```

各エラーはバリデーション通過まで繰り返し発生するため、Orchestratorは失敗時に必ずTask toolでsubagentを再起動する必要がある。

### buildRetryPrompt()がsubagentに渡すリトライプロンプトの構造（FR-6影響範囲）

FR-6の修正により、CLAUDE.mdのルール21にbuildRetryPrompt()の構造が参照情報として追記される。リトライプロンプトは、フェーズ名・リトライ回数・エラーメッセージ引用・改善要求・入力と出力の5つのパートで構成される。

リトライプロンプトの必須セクション構造（コードフェンス内のヘッダーは記号表記を使用）:
```
★ フェーズ名フェーズ（リトライ: 1回目）

＊ タスク情報
- タスク名: {taskName}
- 出力先: {docsDir}/

＊ 前回のバリデーション失敗理由
（artifact-validatorから返されたエラーメッセージ全文をコードブロックで引用）

＊ 改善要求
（エラー内容に基づく具体的修正指示を箇条書きで列挙）

＊ 入力・作業内容・出力
（元のプロンプトと同一の内容）
```

FR-6修正後の改善要求セクションへのリトライ回数別補強内容:
- 1回目リトライでは改善要求セクションにエラーメッセージからの直接的な修正指示を記載する
- 2回目リトライでは改善要求セクションに成果物の問題箇所の行番号と具体的な書き換え案を追記する
- 3回目以降のリトライでは問題セクション全体の書き換え例を改善要求セクションに含める

### FR-1修正後のsubagentへの禁止パターン案内形式

FR-1の修正により、subagentがCLAUDE.mdを参照したときに「禁止パターン（完全リスト）」セクションで12語全てを確認できるようになる。buildPrompt()のプロンプト内では既存のリスト展開が維持され、CLAUDE.mdの新設セクションとの整合性が確保される。

CLAUDE.mdに新設する「禁止パターン（完全リスト）」セクションの形式（コードフェンス内に記載）:
```
### 禁止パターン（完全リスト）

artifact-validator.tsのFORBIDDEN_PATTERNS配列に定義された12語の完全リストを以下に示す。
英語グループ（4語）: TODO, TBD, WIP, FIXME
日本語グループ（8語）: 未定, 未確定, 要検討, 検討中, 対応予定, サンプル, ダミー, 仮置き
上記12語を含む複合語も部分一致（includes）で検出対象となる。
FORBIDDEN_PATTERNSが変更された場合、本リストも合わせて更新すること。
```

## APIレスポンス設計

本セクションでは、MCPサーバーのphaseGuide.subagentTemplate等のAPIレスポンス形式の変更点を記述する。

### phaseGuide.subagentTemplateの変更形式（FR-3・FR-7影響範囲）

FR-3の修正後、requirementsフェーズとplanningフェーズのsubagentTemplateに必須セクション名が追加される。Orchestratorがこれらのテンプレートを使用してsubagentを起動する際に、subagentが成果物の必須セクション構造を事前に認識できるようになる。

requirementsフェーズのsubagentTemplate末尾に必須セクション指示を追加することで、subagentが成果物作成前に要求されるセクション構造を把握できる。この追加により、バリデーターがSECTION_MISSINGエラーを返す頻度が低減すると期待される。

requirementsフェーズのsubagentTemplate末尾に追加する必須セクション指示の形式（コードフェンス内）:
```
★ 必須セクション（成果物に含めること）
成果物には以下のセクションを必ず含めてください（最低行数: 50行以上）。
- 背景（このタスクの背景と目的を記述）
- 機能要件（機能要件を箇条書きで列挙）
- 非機能要件（パフォーマンス・セキュリティ等の要件を記述）
- 受入条件（各機能要件の受入基準を記述）
- サマリー（先頭に配置、200行以内）
```

planningフェーズの必須セクション指示についても同様の形式を採用する。具体的には「概要」「実装計画」「変更対象ファイル」「サマリー」の4セクションを明示し、各セクションに記述すべき内容の説明を括弧書きで補足する。この形式により、subagentが成果物の構造を事前に把握してから作成を開始できる。planningフェーズでsubagentが仕様書を作成する際に必要なセクション構造を事前に認識することで、SECTION_MISSINGエラーによるリトライ回数を削減できる。

planningフェーズのsubagentTemplate末尾に追加する必須セクション指示の形式（コードフェンス内）:
```
★ 必須セクション（成果物に含めること）
成果物には以下のセクションを必ず含めてください（最低行数: 50行以上）。
- 概要（実装計画の全体像を記述）
- 実装計画（フェーズ別・ファイル別の実装方針を記述）
- 変更対象ファイル（変更するファイルの完全リストを記述）
- サマリー（先頭に配置、200行以内）
```

FR-7の修正後、code_reviewフェーズのsubagentTemplateの入力セクションに4ファイルが列挙される。code_reviewのsubagentが設計書4種を読み込むことで、設計-実装整合性チェックの網羅性が向上する。
上記の変更後の入力セクション形式を以下に示す。
```
★ 入力（設計書4種を全文読み込みすること）
以下のファイルを全文読み込んでください:
- {docsDir}/spec.md（全文読み込み必須）
- {docsDir}/state-machine.mmd（全文読み込み必須）
- {docsDir}/flowchart.mmd（全文読み込み必須）
- {docsDir}/ui-design.md（全文読み込み必須）
また、実装対象のソースコードファイルを走査してください。
```

### buildPrompt()が生成するプロンプトのセクション構造（FR-4・FR-5影響範囲）

FR-4の修正後、buildPrompt()が生成するプロンプトのセクション1（タスク情報ヘッダー）にdocsDirパスの警告が追加される。この変更によりsubagentはプロンプト冒頭でdocsDirの実際のパス値を確認でき、誤ったパスへの成果物保存というミスが防止される。修正後のセクション1末尾形式（コードフェンス内）:
```
★ タスク情報
- タスク名: {taskName}
- フェーズ: {phaseName}
- 出力先: {docsDir}/
- ★★出力パス確認★★: 成果物は必ず {docsDir}/ に保存すること。上記パス以外への保存は禁止。
```

FR-5の修正後、buildPrompt()のbashSection末尾にカテゴリ別代替ツール対応表が追加される。subagentがBashコマンドのブロック後に適切な代替ツールへ迅速に切り替えられるよう、具体的なツール名と使用場面を列挙した対応表をbashSectionの末尾に配置する。修正後のbashSection末尾形式（コードフェンス内）:
```
上記カテゴリ外のBashコマンドはフックによりブロックされます。
ブロックされた場合は以下の代替手段を使用してください:
- cat/head/tailがブロックされた場合 → Readツールを使用する
- echo/tee/sed/awkがブロックされた場合 → Writeツール（新規作成）またはEditツール（部分修正）を使用する
- find/grepがブロックされた場合 → Globツール（パターン検索）またはGrepツール（内容検索）を使用する
- cp/mvがブロックされた場合 → Read+Write（コピー）またはRead+Write+rm（移動）のツール組み合わせを使用する
- readonlyフェーズでnpm testを試みてブロックされた場合 → このフェーズではtestingカテゴリは許可されていないため、テスト実行はtestingまたはimplementationフェーズで行うこと
```

## 設定ファイル設計

本セクションでは、FR-1・FR-2・FR-6の修正によるCLAUDE.mdの構造変更を記述する。

### CLAUDE.mdの新設セクション構造（FR-1影響範囲）

FR-1の修正で「### 禁止パターン（完全リスト）」セクションを「### 禁止パターン」見出しの直後に追加する。追加後のセクション構造は以下のとおりである。

「禁止パターン」セクションの変更前後の構造:
- 変更前の構造: 「禁止パターン」見出し → 言い換えパターンの説明のみ
- 変更後の構造: 「禁止パターン」見出し → 言い換えパターンの説明 → 「禁止パターン（完全リスト）」見出し → 12語の完全リスト → 保守ルールの記載

新設セクションの必須記載要素:
- artifact-validatorのFORBIDDEN_PATTERNSからの転記であることを明示する行を含めること
- バリデーターが検出する英語グループ（4語）をコードフェンス内で列挙すること
- バリデーターが検出する日本語グループ（8語）をコードフェンス内で列挙すること
- 「複合語も部分一致（includes）で検出対象」という説明を1行で記載すること
- 「FORBIDDEN_PATTERNSが変更された場合、本リストも更新すること」という保守ルールを1行で記載すること

### CLAUDE.mdの重複行ルール説明への正規表現付記（FR-2影響範囲）

FR-2の修正で、「重複行禁止（3回以上の同一行でエラー）」セクションのルール6とルール8に正規表現パターンが付記される。

ルール6への付記内容（変更後の記述形式）:
ルール6の説明の末尾に「判定正規表現（概念）: 行頭から `^\s*\*\*[^*]+\*\*\s*:\s*$` の形式にマッチする行が除外される。太字ラベルの後にコンテンツが続く場合はこの正規表現にマッチしないため、重複検出の対象となる」という説明を付記する。

ルール8への付記内容（変更後の記述形式）:
ルール8の説明の末尾に「判定正規表現（概念）: 行の文字数が50文字以内かつコロンで終わる行が除外される。50文字を超えるラベル行は除外されないため、3回以上の繰り返しでエラーとなる。リスト記号（ハイフン、アスタリスク）を含む場合も対象となる」という説明を付記する。

### CLAUDE.mdのルール21エスカレーション手順追加（FR-6影響範囲）

FR-6の修正で、「AIへの厳命」セクションのルール21末尾に「リトライ回数別エスカレーション手順」段落を追記する。

追記する段落の構造:
- 段落タイトル: 「リトライ回数別エスカレーション手順」
- 1回目リトライの手順: エラーメッセージをコードブロックで引用してTask toolでsubagentを再起動すること
- 2回目リトライの手順: OrchestratorがReadツールで成果物を読み込み、問題箇所の行番号を特定し、具体的な修正指示（書き換え例を含む）をプロンプトに追加してTask toolでsubagentを再起動すること。haikuモデル使用時はsonnetへのモデルエスカレーションを検討すること
- 3回目以降リトライの手順: 必ずsonnetモデルを使用すること。OrchestratorがReadツールで成果物の問題セクションを特定し、具体的な書き換え例を含む詳細改善指示をプロンプトに含めてTask toolでsubagentを再起動すること
- 直接編集禁止の再確認: どのリトライ回数においてもOrchestratorがEdit/Writeツールで成果物を直接修正することは禁止である
- 参照先: buildRetryPromptの11エラーパターン対応の詳細はdefinitions.tsのbuildRetryPrompt()を参照すること

追記する段落の概算行数は20行であり、NFR-2の50行以内制約を満たす。

## FR別インタラクション変化のまとめ

FR-1からFR-7の各修正が、Orchestratorおよびsubagentのインタラクションに与える具体的な変化を整理する。

FR-1（禁止パターン完全リスト追加）の変化: subagentがCLAUDE.mdを参照した際に12語全てを確認できるため、成果物作成前の自己チェック精度が向上する。

FR-2（重複行ルール正規表現付記）の変化: subagentがルール6・ルール8の境界条件を正確に理解できるため、「太字ラベル後にコンテンツが続く行は重複検出の対象」という見落としが減少する。

FR-3（必須セクション明示）の変化: requirementsおよびplanningのsubagentが成果物作成前に必須セクション名を認識できるため、SECTION_MISSINGエラーによるリトライが減少する。

FR-4（パス警告強化）の変化: subagentがプロンプト冒頭でdocsDirの実際のパス値を確認できるため、誤ったパスへの成果物保存が防止される。

FR-5（代替ツール案内拡充）の変化: subagentがBashコマンドのブロック後に適切な代替ツールへ迅速に切り替えられるため、フックによるブロックが作業中断につながるケースが減少する。

FR-6（リトライエスカレーション強化）の変化: OrchestratorがルールMd21の具体的な手順を参照することで、バリデーション失敗時にEdit/Writeツールでの直接修正という誤った行動を取る頻度が低下する。

FR-7（code_review設計書読み込み追加）の変化: code_reviewのsubagentが設計書4ファイルを全文読み込んだうえで実装コードと照合するため、設計-実装整合性チェックの品質が向上する。
