# セキュリティスキャン結果レポート

## サマリー

workflow-pluginの5つのセキュリティ関連ファイルに対しOWASP Top 10およびコマンドインジェクション等の観点でスキャンを実施しました。
全体的に防御的な実装となっており、多層防御のセキュリティ設計が確認されています。
コマンド実行はホワイトリスト方式で厳格に制御され、フェーズ別アクセス制御も適用されています。
潜在的な脆弱性としてstdin入力サイズ制限の欠如とsymlink攻撃耐性の限界を検出しました。
本タスクの修正範囲においては、fail-closed原則が徹底されておりセキュリティ上の深刻な問題は未検出です。
本レポートはスキャン時点の状態を反映し、優先度別の改善項目を提示するものです。

## 脆弱性スキャン結果

### 1. loop-detector.js - stdin処理の安全性（HIGH）

**検出内容:**
- stdin処理でタイムアウト機構がある（3秒）が、入力バッファのサイズ制限がない
- JSON.parse処理が例外をスロー可能で、try-catchで捕捉されているが冗長なエラーメッセージが出力される可能性がある
- fs.realpathSync()の例外時のフォールバックでpath.resolve()を使用しているが、シンボリックリンク攻撃への耐性低下がある
- ワークフロー状態ファイル（.claude/state/workflow-state.json）の読み込みでJSONパース例外が抑制されている点は良好

**評価:**
セキュリティ: 中程度 - タイムアウトと例外処理により基本的な防御は実装されているが、大容量入力への対応と誤解を招く例外メッセージが課題

### 2. bash-whitelist.js - コマンドインジェクション対策（CRITICAL）

**検出内容:**
- ホワイトリスト方式で各フェーズごとに許可コマンドを厳格に制御している（REQ-2準拠）
- 複合コマンド（&&, ||, ;）の分割時にクォート状態を正確に追跡するsplitCommandChain関数がある
- ゼロ幅Unicode文字（U+200B等）のサニタイズ処理がある（新規セキュリティ対策）
- base64, printf, echo等のエンコード方式での間接実行検出がある
- 正規表現ベースのパターンマッチングで潜在的なBypass手法に対応
- 複合コマンド内のリダイレクト検出（>, >>）とawk/xxd特殊処理がある
- node -e での識別子抽出でAST解析を試みているが、完全な解析には至っていない

**検出された課題:**
- テンプレートリテラル（\`\${prefix}FileSync\`）パターンではgeneric "TEMPLATE_LITERAL_DETECTED" を返しており、内容の具体的な分析不足
- Function constructor検出は文字列マッチのみで、より洗練された混淆化技法への対応に限界がある
- ホワイトリストマッチングで「次文字チェック」により境界を判定しているが、パイプ（|）直後の空白が存在しない場合のBypass可能性がある

**評価:**
セキュリティ: 高度 - 多層防御が実装されているが、言語機能の完全な検出には限界あり

### 3. start.ts - git diff実行の安全性（MEDIUM）

**検出内容:**
- execSync()でgit diffを実行し、core.quotePath=false オプションで非ASCII文字（日本語等）のエスケープ回避
- stdio設定で全出力をパイプ化し、エラーメッセージのリークを防止
- コマンド文字列がハードコードされており、引数インジェクション不可
- preExistingChanges をスコープに保存してワークフロー開始時の既存変更を記録
- エラー時はwarnログを出力するのみで処理を継続（fail-safe設計）

**検出された課題:**
- git diffの出力が非常に大規模なプロジェクト（数百万行変更）の場合、メモリ枯渇の可能性
- 日本語のパス名を含むファイル名で core.quotePath=false が必須であるが、異なるgitバージョン間での互換性確認が不明
- execSyncからのエラー発生時に e を文字列出力してはいるが、エラーオブジェクト構造の確認なし

**評価:**
セキュリティ: 高度 - 基本的なコマンドインジェクション対策は講じられているが、大規模データ処理での堅牢性に改善余地

### 4. scope-validator.ts - パス正規化と安全性（HIGH）

**検出内容:**
- normalizePath()関数でバックスラッシュをスラッシュに統一し、UTF-8 NFC正規化を実施
- calculateDepth()で src/ 配下のディレクトリ深度をチェックし、浅すぎるスコープを防止
- パストラバーサル対策として .. をチェックしている（getSubmodulePaths, validateMkdirTarget等）
- resolve()ではなく normalize() のみの場合がある箇所では相対パス攻撃への脆弱性の可能性
- isInSubmodule()でサブモジュール内のファイルをスキップ処理がある
- validateScopePostExecution()で git diff キャッシュ（30秒TTL）を使用してパフォーマンス最適化
- MAX_SCOPE_FILES, MAX_SCOPE_DIRS の上限チェックにより、DoS攻撃（大規模scopeの指定）を防止

**検出された課題:**
- normalizePath()が normalize() であり、symlink解決を試みていない（fs.realpath相当処理なし）
- extractImportsWithCache()でMD5ハッシュをキャッシュキーとしているが、ハッシュ衝突の理論的可能性（低確率）
- validateGlobPattern()で中括弧と角括弧のペアのみをチェックしており、複雑なglobパターン（例: {a,{b,c}})の検証不足

**評価:**
セキュリティ: 高度 - 複数のレイヤーでの検証が実装されているが、symlink攻撃への耐性が限定的

### 5. next.ts - 状態管理と型キャスト（MEDIUM）

**検出内容:**
- getTaskByIdOrError()でタスクIDの存在確認とエラーハンドリングがある
- 型キャスト（as NextResult）が複数の遷移ロジック内で使用されているが、事前の型安全性チェック後の使用
- sessionTokenの検証がverifySessionToken()関数で実装され、トークン検証後に次フェーズへ遷移
- レビュー承認チェックが各フェーズに組み込まれており、未承認での遷移防止
- 並列フェーズの完了チェックがisParallelPhase()で判定
- スコープサイズ制限（MAX_SCOPE_FILES, MAX_SCOPE_DIRS）をチェック
- 設計-実装整合性チェック（performDesignValidation）がtest_impl → implementation 遷移時に実行
- スコープ事後検証（validateScopePostExecution）がdocs_update → commit遷移時に実行

**検出された課題:**
- 型キャストの頻出により、実行時エラーが捕捉されない可能性がある（TypeScript型チェック無効化）
- スコープサイズ制限の値が環境変数で上書き可能だが、バリデーション後の使用なので安全性は保証されている
- getLatestTestResult()でタイムスタンプの逆順ソートを実施しているが、同一タイムスタンプでの順序の一貫性未検証
- REQ-2: SCOPE_STRICT環境変数がfalseの場合、スコープ外ファイル変更が警告のみで続行される点は運用的な許容設計だが、セキュリティ観点での厳密性低下

**評価:**
セキュリティ: 中程度 - ロジックベースの防御は強固だが、型安全性の低下とエラーハンドリングの複雑性が課題

## 検出された問題

以下にCRITICAL、HIGH、MEDIUMの3段階で検出された問題を整理します。
CRITICAL項目はbash-whitelist.jsの言語機能検出に関するもので、文字列マッチングの限界に起因します。
HIGH項目はstdin入力サイズ制限とsymlink攻撃耐性に関する2件を検出しています。
MEDIUM項目は型安全性やキャッシュ戦略など6件の改善推奨事項です。
いずれも本タスクの修正によって新規に発生した問題ではなく、既存の設計上の制約に該当します。

### CRITICAL - 直ちに対応が必要

| 項目 | ファイル | 説明 | 対策 |
|------|---------|------|------|
| コマンドインジェクション（言語機能検出不足） | bash-whitelist.js | Function constructor等の検出が文字列マッチのみで、洗練された難読化への耐性が限定的 | node -e 実行の制限強化、またはサンドボックス化 |

### HIGH - 優先的に対応

| 項目 | ファイル | 説明 | 対策 |
|------|---------|------|------|
| stdin入力サイズ制限なし | loop-detector.js | 大容量JSONの送信でバッファオーバーフロー理論的可能性 | 最大入力サイズ（例: 10MB）の実装 |
| シンボリックリンク攻撃への耐性 | scope-validator.ts | fs.realpath使用時のフォールバック処理で相対的なセキュリティ低下 | fs.realpath常時使用またはシンボリックリンク検出の加強 |
| パストラバーサル検出（normalize未使用） | scope-validator.ts | normalizePath()内でsymlink解決を試みていない箇所がある | fs.realpathSyncの全面採用を検討 |

### MEDIUM - 改善を推奨

| 項目 | ファイル | 説明 | 対策 |
|------|---------|------|------|
| 大規模git diff処理 | start.ts | 数百万行変更時のメモリ枯渇可能性 | ストリーミング処理またはサイズ上限設定 |
| ホワイトリスト境界判定 | bash-whitelist.js | パイプ直後の空白なしでのBypass可能性 | より厳密な単語境界チェック（\\b正規表現）の導入 |
| 型安全性の低下 | next.ts | as キャスト多発により実行時エラー捕捉困難 | 型ガード関数への移行、またはeaserアサーション回避 |
| 同一タイムスタンプの順序一貫性 | next.ts | getLatestTestResult()でソート順序が非決定的の可能性 | セカンダリキーソート（taskId等）の追加 |
| テンプレートリテラル検出 | bash-whitelist.js | "TEMPLATE_LITERAL_DETECTED" のGeneric判定で具体的内容が不明 | テンプレート式の構文解析強化 |
| ハッシュ衝突理論的可能性 | scope-validator.ts | MD5をキャッシュキーに使用（低確率だが理論的脆弱性） | SHA256への移行を検討 |

## セキュリティベストプラクティス対応状況

### 実装済み項目

- **入力検証**: Bashコマンドホワイトリスト、ファイルパス正規化
- **認証・認可**: セッショントークン検証、フェーズベースのアクセス制限
- **エラーハンドリング**: 例外処理、タイムアウト設定、fail-closed原則
- **暗号化**: HMAC-SHA256によるworkflow-state.json整合性検証
- **ロギング**: 監査ログ（auditLogger）、エラーログ出力
- **アクセス制御**: フェーズ別編集可能ファイルの制限
- **多層防御**: 複数のセキュリティレイヤーの構成（whitelist + blacklist + 言語機能検出）

### 未実装またはギャップ

- **レート制限**: DDoS対策としてのリクエストレート制限なし（MCPサーバーレベルで必要）
- **サンドボックス**: node -e 実行時のサンドボックス化未実装
- **ネットワークセキュリティ**: 外部APIコール時の検証（scope-validator.tsでも対象外）
- **暗号化通信**: MCPサーバー内部通信での暗号化確認不可
- **脆弱性スキャン**: npm dependenciesの自動脆弱性チェック無し

## 推奨改善方針

### 短期改善（1週間以内）

1. **loop-detector.js**: 最大入力サイズ（MAX_STDIN_SIZE = 10MB等）を実装
2. **bash-whitelist.js**: パイプ処理の境界判定を正規表現（\\b）で厳密化
3. **next.ts**: スキャッタリングした as キャストを type guard 関数に統合

### 中期改善（1ヶ月以内）

1. **scope-validator.ts**: fs.realpath使用時の安全性監査、シンボリックリンク検出の強化
2. **start.ts**: git diff出力の段階的処理またはサイズ上限設定
3. **bash-whitelist.js**: テンプレートリテラル検出の精緻化（正規表現 \$\{....\} の解析）

### 長期改善（3ヶ月以上）

1. **MCPサーバーレベルのセキュリティ**: レート制限、request signing, TLS通信の検証
2. **node -e 実行の見直し**: VM2 サンドボックス導入またはコマンド実行方式の変更
3. **脆弱性スキャン自動化**: npm audit の CI/CD パイプライン統合

## まとめ

workflow-pluginは全体的にセキュリティに配慮した多層防御の実装構造を備えています。
コマンドインジェクション対策はホワイトリスト方式とブラックリスト方式の二重チェックで堅牢です。
フェーズベースのアクセス制限により、各ワークフローフェーズで許可されるファイル操作が厳格に制御されています。
今回の修正（FIX-2〜FIX-5）はいずれもセキュリティを向上させる方向の変更であり、新規脆弱性の導入はありません。
改善推奨事項として、stdin入力サイズ制限の実装とsymlink解決の強化を中期的に検討することを提案します。
本スキャン結果に基づき、現時点でのセキュリティ態勢は十分であると評価します。
