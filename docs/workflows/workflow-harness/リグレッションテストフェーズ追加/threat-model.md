# リグレッションテストフェーズ追加 - 脅威モデル

## 概要

ワークフロープラグインへのリグレッションテストフェーズ追加に伴うセキュリティ脅威の分析。

## 脅威分析

### T-1: テストのバイパス

| 項目 | 内容 |
|------|------|
| 脅威 | リグレッションテストフェーズをスキップして実装をデプロイ |
| 影響 | 品質低下、バグの本番流出 |
| 発生可能性 | 中（ワークフロー無視の試み） |
| 対策 | フックによるフェーズスキップ検出・ブロック |
| 緩和策 | CI/CDでもリグレッションテストを実行 |

### T-2: テスト結果の改ざん

| 項目 | 内容 |
|------|------|
| 脅威 | テスト結果ファイルを改ざんして成功に見せかける |
| 影響 | 失敗したテストが見過ごされる |
| 発生可能性 | 低（意図的な改ざんが必要） |
| 対策 | テスト実行はCI/CDで行い、結果を自動取得 |
| 緩和策 | テスト結果のハッシュ検証 |

### T-3: 不正なテストの混入

| 項目 | 内容 |
|------|------|
| 脅威 | 悪意のあるコードを含むテストファイルの追加 |
| 影響 | テスト実行時に任意コード実行 |
| 発生可能性 | 低（コードレビューで検出可能） |
| 対策 | コードレビューフェーズでテストも確認 |
| 緩和策 | テスト実行環境の分離（コンテナ） |

### T-4: リグレッションテストの削除

| 項目 | 内容 |
|------|------|
| 脅威 | 不都合なテストを削除して「成功」させる |
| 影響 | リグレッション検出能力の低下 |
| 発生可能性 | 中（短期的利益のための誘惑） |
| 対策 | テスト削除時は理由を記録、レビュー必須 |
| 緩和策 | テスト数の監視、減少時にアラート |

## 信頼境界

```
┌─────────────────────────────────────────────────────────────┐
│                     開発環境                                 │
├─────────────────────────────────────────────────────────────┤
│  ┌──────────────────────────────────────────────────────┐  │
│  │  ワークフロープラグイン（信頼済み）                   │  │
│  │  - フェーズ管理                                      │  │
│  │  - テスト実行指示                                    │  │
│  └──────────────────────────────────────────────────────┘  │
│                          ↓                                  │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  テスト実行環境（信頼済み、分離推奨）                │  │
│  │  - pytest / vitest                                   │  │
│  │  - テストコード実行                                  │  │
│  └──────────────────────────────────────────────────────┘  │
│                          ↓                                  │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  リグレッションテストコード（半信頼）                │  │
│  │  - 開発者が作成                                      │  │
│  │  - コードレビュー済み                                │  │
│  └──────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

## リスク評価サマリー

| 脅威 | 影響度 | 発生可能性 | リスクレベル | 対策状況 |
|------|--------|-----------|-------------|---------|
| T-1 | 高 | 中 | 高 | 対策済み（フック） |
| T-2 | 高 | 低 | 中 | CI/CDで緩和 |
| T-3 | 高 | 低 | 中 | コードレビューで緩和 |
| T-4 | 中 | 中 | 中 | 監視で緩和 |

## 推奨事項

1. **フェーズスキップの厳格なブロック**: 既存のフック機構を活用
2. **CI/CDとの統合**: リグレッションテストはCI/CDでも実行
3. **テスト削除の監視**: regression ディレクトリのファイル数を監視
4. **コードレビューの強化**: テストファイルもレビュー対象に明記
