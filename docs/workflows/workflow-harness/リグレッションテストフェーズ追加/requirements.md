# リグレッションテストフェーズ追加 - 要件定義

## 概要

ワークフロープラグインにリグレッションテストフェーズとリグレッションテスト蓄積ディレクトリを追加する。

## 背景と課題

### 現状の問題

1. **誤判断の発生**: 既存テストの失敗が「今回のタスクとは関係ない」として見過ごされる
2. **テストの散逸**: 各タスクで作成したテストが蓄積されず、再利用されない

### 根本原因

- `testing` フェーズは今回作成したテストのみを対象としている
- 過去のタスクで作成したテストを管理する仕組みがない
- リグレッションテストを強制実行するフェーズがない

## 機能要件

### FR-1: リグレッションテストフェーズの追加

| 要件ID | 要件 | 優先度 |
|--------|------|--------|
| FR-1.1 | `testing` フェーズの後に `regression_test` フェーズを追加する | 必須 |
| FR-1.2 | `regression_test` フェーズでは全リグレッションテストを実行する | 必須 |
| FR-1.3 | リグレッションテストが1つでも失敗したら次フェーズに進めない | 必須 |
| FR-1.4 | フェーズドキュメント `workflow-phases/regression_test.md` を作成する | 必須 |

### FR-2: リグレッションテストディレクトリの追加

| 要件ID | 要件 | 優先度 |
|--------|------|--------|
| FR-2.1 | バックエンド用: `src/backend/tests/regression/` を定義する | 必須 |
| FR-2.2 | フロントエンド用: `src/frontend/test/regression/` を定義する | 必須 |
| FR-2.3 | タスクごとにサブディレクトリを作成できる構造にする | 推奨 |
| FR-2.4 | リグレッションテストの README.md を作成する | 推奨 |

### FR-3: フェーズ定義の更新

| 要件ID | 要件 | 優先度 |
|--------|------|--------|
| FR-3.1 | `PhaseName` 型に `regression_test` を追加する | 必須 |
| FR-3.2 | `PHASES_LARGE` 配列に `regression_test` を追加する | 必須 |
| FR-3.3 | `PHASE_DESCRIPTIONS` に説明を追加する | 必須 |
| FR-3.4 | `PHASE_EXTENSIONS` に許可拡張子を追加する | 必須 |

### FR-4: CLAUDE.md の更新

| 要件ID | 要件 | 優先度 |
|--------|------|--------|
| FR-4.1 | フェーズ順序の説明を更新する | 必須 |
| FR-4.2 | リグレッションテストディレクトリのルールを追加する | 必須 |
| FR-4.3 | AIへの厳命にリグレッションテストルールを追加する | 推奨 |

## 非機能要件

### NFR-1: 互換性

| 要件ID | 要件 |
|--------|------|
| NFR-1.1 | 既存のワークフローに影響を与えない |
| NFR-1.2 | 進行中のタスクは従来のフェーズ構成で継続可能 |

### NFR-2: パフォーマンス

| 要件ID | 要件 |
|--------|------|
| NFR-2.1 | リグレッションテストは並列実行可能な設計にする |
| NFR-2.2 | テスト実行時間の上限は設けない（CI/CDで管理） |

### NFR-3: 保守性

| 要件ID | 要件 |
|--------|------|
| NFR-3.1 | テストファイルの命名規則を統一する |
| NFR-3.2 | ディレクトリ構造を文書化する |

## フェーズ構成（変更後）

### 変更前（18フェーズ）

```
... → parallel_quality → testing → parallel_verification → ...
```

### 変更後（19フェーズ）

```
... → parallel_quality → testing → regression_test → parallel_verification → ...
```

## ディレクトリ構成

### バックエンド

```
src/backend/tests/
├── unit/                    # ユニットテスト（既存）
├── integration/             # 統合テスト（既存）
├── fixtures/                # テストフィクスチャ（既存）
└── regression/              # リグレッションテスト（新規）
    ├── README.md            # 説明
    ├── conftest.py          # 共通設定
    └── {task-name}/         # タスクごとのテスト
        ├── __init__.py
        └── test_*.py
```

### フロントエンド

```
src/frontend/test/
├── fixtures/                # フィクスチャ（既存）
├── integration/             # 統合テスト（既存）
└── regression/              # リグレッションテスト（新規）
    ├── README.md            # 説明
    └── {task-name}/         # タスクごとのテスト
        └── *.test.tsx
```

## 受け入れ基準

### AC-1: フェーズ追加

- [ ] `workflow_status` で `regression_test` フェーズが表示される
- [ ] `testing` → `regression_test` → `parallel_verification` の順で遷移する
- [ ] フェーズ数が 19 になる

### AC-2: テスト実行

- [ ] `regression_test` フェーズでリグレッションテストが実行される
- [ ] テスト失敗時は次フェーズに進めない
- [ ] テスト成功時は次フェーズに進める

### AC-3: ドキュメント

- [ ] `workflow-phases/regression_test.md` が存在する
- [ ] CLAUDE.md にリグレッションテストの説明がある
- [ ] リグレッションディレクトリに README.md がある

## 変更影響範囲

| ファイル | 変更内容 |
|---------|---------|
| `mcp-server/src/state/types.ts` | PhaseName 型に追加 |
| `mcp-server/src/phases/definitions.ts` | フェーズ定義に追加 |
| `workflow-phases/regression_test.md` | 新規作成 |
| `CLAUDE.md` | ルール追加 |
| ルートCLAUDE.md | ルール追加 |

## リスクと対策

| リスク | 対策 |
|--------|------|
| フェーズ数増加による複雑化 | ドキュメントで明確に説明 |
| 既存タスクへの影響 | 新規タスクからのみ適用 |
| テスト実行時間の増加 | 並列実行、キャッシュ活用 |
