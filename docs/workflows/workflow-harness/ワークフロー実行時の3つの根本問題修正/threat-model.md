# threat-model: ワークフロー実行時の3つの根本問題修正

## サマリー

本脅威モデルでは、ワークフロー実行時に発生した3つの根本問題の修正に伴うセキュリティリスクと信頼性リスクを分析する。
FIX-1のloop-detectorファイルエントリpruning機能では、削除判定条件の誤実装により現在アクティブな編集記録が誤削除されるリスクがあり、これによりループ検出機構が機能不全に陥る可能性がある。
FIX-2のCLAUDE.md改善では、バリデーションルールの具体例が不十分な場合にsubagentが依然としてバリデーション失敗を繰り返すリスクがあり、自動化レベルの向上が達成されない可能性がある。
FIX-3のpreExistingChanges改善では、EXCLUDE_PATTERNSの過度な拡張によりスコープチェック機構が無効化されるリスクがあり、意図しないファイルの変更が検出されなくなる可能性がある。
最も重大な脅威は、FIX-1でlastWarningフィールドが設定されているエントリを誤削除することによる警告抑止期間の管理破綻であり、これによりユーザーが同一ファイルの繰り返し編集について警告を受けられなくなる危険性がある。
各修正について、脅威シナリオの詳細分析、リスク評価、セキュリティ要件、緩和策を定義し、修正実装の安全性を保証する。

## 脅威シナリオ

### T-1: FIX-1 loop-detectorのpruning処理に関する脅威

#### T-1.1: アクティブな編集記録の誤削除

削除判定条件（timestampsが空かつlastWarningが未設定）の実装ミスにより、現在アクティブなファイルエントリが誤って削除されるシナリオを分析する。
具体的には、filterOldTimestamps関数でタイムスタンプを削除した直後にpruning処理を実行する際、タイムスタンプの削除と空配列判定のタイミングで競合が発生する可能性がある。
もし5分以内に編集されたファイルのタイムスタンプが何らかの理由で削除された場合、timestampsが空になりpruning対象として誤認される危険性がある。
このシナリオが発生すると、loop-detectorはそのファイルの編集履歴を完全に喪失し、短時間での繰り返し編集を検出できなくなる。
結果として、ユーザーが同一ファイルを5分以内に5回以上編集してもループ警告が発生せず、無限ループに陥るリスクが増大する。

#### T-1.2: lastWarningフィールドを持つエントリの誤削除

lastWarningが設定されているエントリを削除判定から除外する条件が正しく実装されない場合、警告抑止期間の管理が破綻するシナリオを分析する。
loop-detectorは、ユーザーが警告を受けた後に同じファイルを再度編集しても一定期間（warningSuppressed期間）は再警告しない仕組みを持つ。
この期間管理はlastWarningフィールドに保存された警告発生時刻を基準に行われており、このフィールドが失われると警告抑止が機能しなくなる。
もしlastWarningを持つエントリがtimestampsが空という理由だけで削除された場合、ユーザーは同一ファイルの編集で毎回警告を受けることになり、警告疲れが発生する。
逆に、警告抑止期間が永続化してしまい、本来検出すべきループが検出されなくなる可能性もある。

#### T-1.3: 大量エントリ削除時のパフォーマンス劣化

既存の肥大化した状態ファイル（3000個以上のエントリ）を初回実行時にクリーンアップする際、処理時間がフックのタイムアウト3秒を超過するシナリオを分析する。
Object.entriesによるエントリ走査は、エントリ数に比例して処理時間が増加するため、3000個のエントリを全て走査すると数秒かかる可能性がある。
もしpruning処理がタイムアウトした場合、フック実行が異常終了し、Edit/Writeツールが使用不可能になる（現在の問題が再発する）。
タイムアウト発生時には、状態ファイルの保存も中断される可能性があり、部分的にクリーンアップされた状態が書き込まれずに失われるリスクがある。
この場合、次回のフック実行でも同じ肥大化した状態ファイルを読み込むことになり、問題が永続化する。

#### T-1.4: pruning処理のバグによる状態ファイル破壊

削除処理の実装ミス（delete演算子の誤用、Object.fromEntriesの誤実装等）により、state.filesオブジェクトが破壊されるシナリオを分析する。
JavaScriptのdelete演算子は、オブジェクトのプロパティを削除するが、配列に対して使用すると想定外の動作（undefined要素の残存）が発生する可能性がある。
もしstate.filesが配列として誤って扱われた場合、delete演算子による削除でundefined要素が残り、次回のcheckLoop呼び出し時にTypeErrorが発生する。
Object.fromEntriesによる再構築時にフィルタリングロジックにバグがあると、削除対象でないエントリまで削除される可能性がある。
状態ファイルが破壊された場合、loop-detectorは起動時にJSONパースエラーまたはランタイムエラーを発生し、全てのフック実行が失敗する。

### T-2: FIX-2 CLAUDE.md改善に関する脅威

#### T-2.1: バリデーションルール具体例の不整合

CLAUDE.mdに追加するバリデーションルールの具体例が、artifact-validator.tsの実装と不一致の場合、subagentが誤った理解をするシナリオを分析する。
具体的には、CLAUDE.mdで「テーブルデータ行は重複カウントから除外される」と記述しても、実際のisStructuralLine関数の正規表現が2列以上のテーブル行のみを除外する場合、1列のテーブル行は除外されない。
subagentがこの差異を理解できず、1列のテーブルを使用した成果物を作成すると、バリデーションエラーが発生する。
また、「Markdownラベル（太字終端）は除外される」という記述が、実際には行末が太字で終わる行のみを対象とする実装である場合、行中に太字が含まれるラベルは除外されない。
ドキュメントとコードの不一致が発生すると、subagentは正しい記述方法を学習できず、バリデーション失敗が継続する。

#### T-2.2: 具体例の過度な追加によるコンテキスト圧迫

CLAUDE.mdに大量の具体例を追加することで、subagent起動時のコンテキストウィンドウが圧迫され、本来のタスク内容の理解が阻害されるシナリオを分析する。
Claude CodeがsubagentにCLAUDE.mdを渡す際、ドキュメント全体がコンテキストに含まれるため、CLAUDE.mdのサイズが増加するとタスクの入力ファイル（requirements.md等）の読み込み容量が減少する。
もしCLAUDE.mdが50KB以上に肥大化した場合、subagentは長大なガイドライン読解に計算リソースを消費し、実際のタスク内容の理解が浅くなる可能性がある。
結果として、バリデーションルールは理解しても、タスクの本質的な要件（機能仕様、設計意図等）の理解が不十分になり、成果物の品質が低下する。
コンテキスト圧迫により、subagentの応答時間も増加し、ワークフロー全体の実行時間が延長される。

#### T-2.3: subagentのコンテキスト理解能力の限界

CLAUDE.mdに具体例を追加しても、subagentのモデル（haiku/sonnet）のコンテキスト理解能力が不足している場合、改善効果が得られないシナリオを分析する。
特にhaikuモデルを使用するsubagent（refactoring、build_check等）は、sonnetモデルよりも推論能力が低く、複雑なバリデーションルールの理解が困難である。
具体例として悪い例と良い例を提示しても、haikuモデルはパターンの一般化が苦手であり、提示された例と完全に同じケースでしか正しく動作しない可能性がある。
例えば、「結果: パス」の代わりに「動作確認: パス」と記述するという具体例を提示しても、subagentは「検証結果: パス」のような類似パターンを生成し、バリデーションエラーが発生する。
モデルの能力限界により、CLAUDE.md改善の効果が限定的になり、依然としてOrchestratorによる手動修正が必要になる。

#### T-2.4: 必須セクション名の誤記憶

subagent起動テンプレートに必須セクション名を明示しても、subagentがセクション名を誤記憶し、バリデーションエラーが発生するシナリオを分析する。
例えば、manual_testフェーズの必須セクションが「## テストシナリオ」「## テスト結果」である場合、subagentが「## シナリオ」「## 結果」のように短縮形で記述する可能性がある。
artifact-validatorは完全一致で必須セクションの存在を確認するため、短縮形のセクション名はエラーとして検出される。
また、複数フェーズの必須セクション名が混在した場合（security_scanフェーズでmanual_testの必須セクションを使用する等）、subagentが自分のフェーズの要件を誤認する。
セクション名の誤記憶により、成果物の構造が不適切になり、バリデーション失敗が継続する。

### T-3: FIX-3 preExistingChanges改善に関する脅威

#### T-3.1: EXCLUDE_PATTERNSの過度な拡張によるスコープチェック無効化

EXCLUDE_PATTERNSに設定ファイル（.mcp.json、tsconfig.json等）を追加することで、これらのファイルの変更が常にスコープチェックから除外され、意図しない変更が検出されなくなるシナリオを分析する。
具体的には、設定ファイルのみを変更するタスク（MCP設定変更タスク等）を実行した場合、スコープに設定ファイルが含まれていても、EXCLUDE_PATTERNSにより常に除外される。
この場合、スコープチェックは「変更されたファイルがスコープ外」と判定せず、実際には設定ファイルの変更が検出されないまま次フェーズに進む。
結果として、設定ファイルの変更がコミット対象に含まれず、タスクが完了しても期待された設定変更が反映されない。
また、設定ファイルに機密情報（APIキー等）が誤って記述された場合でも、スコープチェックで検出されず、セキュリティリスクが増大する。

#### T-3.2: git diff失敗時のpreExistingChanges空配列化

workflow_start時のgit diff実行が失敗した場合、preExistingChangesが空配列として設定され、タスク開始前の変更がスコープチェックでブロックされるシナリオを分析する。
git diffは、HEADが存在しない状態（初回コミット前）、gitコマンドが利用不可の環境、リポジトリ破損等の理由で失敗する可能性がある。
失敗時にpreExistingChangesが空配列になると、タスク開始前から存在した全ての変更がスコープ外として扱われ、commitフェーズへの遷移がブロックされる。
ユーザーは、EXCLUDE_PATTERNSに含まれないファイル（ソースコード等）を変更していた場合、手動でworkflow_set_scopeを実行してスコープに追加する必要がある。
git diffの失敗原因がログに記録されない場合、ユーザーは問題の根本原因を特定できず、手動回避が困難になる。

#### T-3.3: preExistingChangesのundefinedフォールバック時のログ未出力

next.tsのスコープ検証呼び出しで、taskState.scope?.preExistingChangesがundefinedの場合に空配列にフォールバックする際、ログが出力されないシナリオを分析する。
現在の実装では、undefinedフォールバックが発生してもユーザーに通知されず、preExistingChangesが記録されなかった原因が不明なままになる。
このシナリオが発生すると、ユーザーは「なぜタスク開始前の変更がスコープチェックでブロックされるのか」を理解できず、問題解決が困難になる。
特に、workflow_start時にgit diffがサイレント失敗（エラーログが出力されない失敗）した場合、undefinedフォールバックが唯一の手がかりとなるが、ログがなければ原因追跡が不可能である。
ログ未出力により、preExistingChanges処理の改善効果が検証できず、問題が修正されたかどうかの判断も困難になる。

#### T-3.4: EXCLUDE_PATTERNSとpreExistingChangesの優先順位競合

EXCLUDE_PATTERNSとpreExistingChangesの両方が適用される場合、どちらが優先されるかが不明確で、意図しない動作が発生するシナリオを分析する。
scope-validator.tsの実装では、EXCLUDE_PATTERNSのチェックとpreExistingChangesのチェックが独立して実行されるため、優先順位が明確でない。
もしEXCLUDE_PATTERNSが優先される場合、preExistingChangesに記録されたファイルでもEXCLUDE_PATTERNSに一致すれば常に除外される（設定ファイルは常に除外）。
逆にpreExistingChangesが優先される場合、EXCLUDE_PATTERNSに含まれないファイルでもpreExistingChangesに記録されていれば除外される（タスク開始前の変更は除外）。
優先順位が不明確な場合、特定のファイル（例：package.jsonがEXCLUDE_PATTERNSとpreExistingChangesの両方に該当）の扱いが不安定になり、スコープチェックの動作が予測不能になる。

## リスク評価

### R-1: FIX-1 loop-detectorのpruning処理のリスク評価

#### R-1.1: アクティブな編集記録の誤削除リスク

発生確率は中程度（実装ミスが発生する可能性は低いが、タイミング競合の可能性がある）。
影響度は高い（ループ検出機構が機能不全に陥り、無限ループによるシステム障害が発生する可能性がある）。
リスクレベルは高（発生確率中×影響度高）であり、優先的な緩和策が必要である。
緩和策として、削除前にタイムスタンプの存在を再確認する二重チェックを実装し、誤削除を防止する。
テストケースとして、5分以内に編集されたファイルのエントリが削除されないことを確認する単体テストを作成する。

#### R-1.2: lastWarningフィールドを持つエントリの誤削除リスク

発生確率は低い（削除判定条件で明示的にlastWarningをチェックするため、実装ミスがなければ発生しない）。
影響度は中程度（警告抑止期間の管理が破綻するが、システム全体の機能停止には至らない）。
リスクレベルは中（発生確率低×影響度中）であり、通常のコードレビューで対処可能である。
緩和策として、削除判定条件をコメントで明記し、lastWarning !== undefinedの条件が削除防止のために必須であることを文書化する。
テストケースとして、lastWarningが設定されているエントリがtimestampsが空でも削除されないことを確認する単体テストを作成する。

#### R-1.3: 大量エントリ削除時のパフォーマンス劣化リスク

発生確率は低い（3000個程度のエントリ走査は、現代のJavaScriptエンジンでは数ミリ秒で完了する）。
影響度は高い（タイムアウトが発生すると、フック実行が異常終了し、Edit/Writeツールが使用不可能になる）。
リスクレベルは中（発生確率低×影響度高）であり、実装時のパフォーマンステストで検証する。
緩和策として、初回実行時のクリーンアップ処理時間を計測し、3秒を超える場合はバッチサイズを制限する（1回のcheckLoopで最大100エントリを削除する等）。
テストケースとして、3000個のエントリを含む状態ファイルで初回クリーンアップが3秒以内に完了することを確認する統合テストを作成する。

#### R-1.4: pruning処理のバグによる状態ファイル破壊リスク

発生確率は低い（JavaScriptのdelete演算子とObject.fromEntriesは一般的な操作であり、実装ミスの可能性は低い）。
影響度は極めて高い（状態ファイル破壊により、全てのフック実行が失敗し、ワークフロー実行が完全停止する）。
リスクレベルは高（発生確率低×影響度極高）であり、堅牢なエラーハンドリングが必要である。
緩和策として、saveState関数でJSON.stringifyが失敗した場合にバックアップファイルを保持し、破壊された状態ファイルを元に戻せるようにする。
テストケースとして、状態ファイルの構造が正しく維持されることを確認する単体テスト（JSON.parseで読み込み可能か、filesオブジェクトの構造が正しいか等）を作成する。

### R-2: FIX-2 CLAUDE.md改善のリスク評価

#### R-2.1: バリデーションルール具体例の不整合リスク

発生確率は中程度（CLAUDE.mdとartifact-validator.tsのメンテナンスは独立して行われるため、不一致が発生しやすい）。
影響度は中程度（subagentが誤った理解をするが、Orchestratorの手動修正で回避可能）。
リスクレベルは中（発生確率中×影響度中）であり、ドキュメントとコードの一致性を保つプロセスが必要である。
緩和策として、CLAUDE.mdの具体例をartifact-validator.tsの実装から自動生成するスクリプトを作成し、不一致を防止する（将来の改善）。
当面の対策として、CLAUDE.md改善時にartifact-validator.tsのisStructuralLine関数の正規表現を確認し、具体例が実装と一致することを手動で検証する。

#### R-2.2: 具体例の過度な追加によるコンテキスト圧迫リスク

発生確率は低い（具体例の追加量を制限し、最頻出の失敗パターンのみに絞れば、数KB程度の増加に抑えられる）。
影響度は中程度（コンテキスト圧迫により、subagentのタスク理解が浅くなり、成果物品質が低下する可能性がある）。
リスクレベルは低（発生確率低×影響度中）であり、過度な具体例追加を避ければ問題ない。
緩和策として、CLAUDE.mdの各セクションのサイズを定期的に計測し、subagent起動テンプレートが10KB以下に収まることを確認する。
具体例は、悪い例と良い例を1-2セットのみ記述し、詳細な説明は避ける（簡潔な指針のみを提示）。

#### R-2.3: subagentのコンテキスト理解能力の限界リスク

発生確率は高い（haikuモデルの推論能力は限定的であり、複雑なルールの一般化が困難である）。
影響度は中程度（CLAUDE.md改善の効果が限定的になるが、現状と比較して改善がないわけではない）。
リスクレベルは高（発生確率高×影響度中）であり、根本的な解決には別のアプローチが必要である。
緩和策として、haikuモデルを使用するフェーズ（refactoring、build_check等）では、バリデーション要件を簡素化する（セクション行数要件を3行以上に緩和する等）。
長期的な対策として、subagentのモデルをsonnetに統一し、コンテキスト理解能力を向上させる（コスト増加とのトレードオフ）。

#### R-2.4: 必須セクション名の誤記憶リスク

発生確率は中程度（subagentが完全一致のセクション名を生成する保証はなく、短縮形や類似名を使用する可能性がある）。
影響度は低い（セクション名の誤記憶はバリデーションエラーとして即座に検出され、Orchestratorが修正できる）。
リスクレベルは低（発生確率中×影響度低）であり、現状の手動修正で対処可能である。
緩和策として、subagent起動テンプレートで必須セクション名を「完全にこの通りに記述すること」と強調し、短縮形を禁止する指示を追加する。
artifact-validatorで、セクション名の部分一致（前方一致や部分文字列一致）を許容する柔軟なチェックを実装する（将来の改善）。

### R-3: FIX-3 preExistingChanges改善のリスク評価

#### R-3.1: EXCLUDE_PATTERNSの過度な拡張によるスコープチェック無効化リスク

発生確率は低い（設定ファイルのみを変更するタスクは稀であり、大半のタスクはソースコードも変更する）。
影響度は中程度（設定ファイルの変更が検出されないが、手動でworkflow_set_scopeを実行すれば回避可能）。
リスクレベルは低（発生確率低×影響度中）であり、運用ルールで対処可能である。
緩和策として、設定ファイルのみを変更するタスクでは、タスク開始時にworkflow_set_scopeで設定ファイルをスコープに明示的に含める運用とする。
EXCLUDE_PATTERNSの追加パターンをCLAUDE.mdに文書化し、ユーザーがどのファイルが自動除外されるかを把握できるようにする。

#### R-3.2: git diff失敗時のpreExistingChanges空配列化リスク

## 緩和策

### M-1: FIX-1 loop-detectorのpruning処理の緩和策

#### M-1.1: 二重チェックによる誤削除防止

削除前にタイムスタンプの存在を再確認する二重チェックを実装し、5分以内に編集されたファイルのエントリが誤削除されないようにする。
具体的には、filterOldTimestamps呼び出し後にtimestampsが空になったエントリでも、現在時刻との差分を確認し、5分以内のエントリは削除しない。
この二重チェックにより、タイミング競合（フィルタリング直後に新しいタイムスタンプが追加される等）による誤削除を防止できる。
実装は、checkLoop関数内の削除判定条件に、timestampsが空かつlastWarningが未設定かつ最終編集時刻が5分以上前という条件を追加する形で行う。

#### M-1.2: バックアップファイルによる復元機構

saveState関数で状態ファイルを保存する前に、前回の正常な状態をバックアップファイル（loop-detector-state.json.backup）に保存する。
JSON.stringifyまたはfs.writeFileSyncが失敗した場合、バックアップファイルから前回の状態を復元し、エラーログを出力する。
バックアップファイルは、毎回のsaveState呼び出し時に上書きされ、常に最新の正常な状態を保持する。
状態ファイルが破壊された場合（JSONパースエラー等）、フック起動時にバックアップファイルから自動復元を試み、ユーザーの手動介入を不要にする。

#### M-1.3: パフォーマンス劣化の監視とバッチサイズ制限

pruning処理の実行時間を計測し、3秒を超える場合は警告ログを出力して、次回以降の処理でバッチサイズを制限する。
バッチサイズ制限は、1回のcheckLoop呼び出しで最大100エントリまでを削除する形で実装し、大量エントリの一括削除による長時間処理を防止する。
バッチサイズ制限により、初回クリーンアップが複数回のcheckLoop呼び出しに分散されるが、最終的には全てのエントリがクリーンアップされる。
実装は、削除対象エントリのカウンターを導入し、100エントリに達したら残りのエントリは次回のcheckLoop呼び出しで削除する形で行う。

フォールバック処理により、undefinedがエラーではなく空配列として扱われ、システムがクラッシュしないことを確認する。
テストは、単体テストで行い、エラーハンドリングの堅牢性を検証する。

## 次フェーズへの引き継ぎ事項

planningフェーズでは、3つの修正（FIX-1、FIX-2、FIX-3）の実装仕様を詳細に記述する必要がある。
FIX-1では、削除判定ロジックの詳細（条件式、処理タイミング、バッチサイズ制限の実装方法）を仕様書に含める。
FIX-2では、CLAUDE.mdに追加する具体例の実際のMarkdown記述例（悪い例と良い例）を仕様書に記述する。
FIX-3では、EXCLUDE_PATTERNSに追加するパターンのリストと、エラーハンドリングの詳細（ログ出力フォーマット、フォールバック処理）を明記する。
test_designフェーズでは、各修正の検証計画（単体テスト、統合テスト、リグレッションテスト）を具体的なテストケースとして設計する。
implementationフェーズでは、優先度の高い順（FIX-1 → FIX-2 → FIX-3）に実装し、各修正完了後に個別の動作確認を行う。
