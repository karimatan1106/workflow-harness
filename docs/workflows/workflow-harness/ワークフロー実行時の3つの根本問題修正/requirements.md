# requirements: ワークフロー実行時の3つの根本問題修正

## サマリー

本タスクでは、ワークフロー実行時に発生した3つの根本的な問題を修正する要件を定義する。
第一の問題は、loop-detector-state.jsonが107KBに肥大化しフッククラッシュを引き起こしており、ファイルエントリのpruningメカニズムの欠如が根本原因である。
第二の問題は、subagent成果物がartifact-validatorの検証に繰り返し失敗しており、CLAUDE.mdのバリデーションルールが抽象的すぎて具体例が不足していることが原因である。
第三の問題は、スコープチェックがタスク開始前から存在した変更をブロックしており、preExistingChangesの記録とエラーハンドリングに不備があることが原因である。
優先度は問題1が最高（緊急度高、フック完全停止）で、問題2が中（手動回避可能だが非効率）、問題3が低（手動回避が容易）である。
全3問題とも修正難易度は低く、1-2ファイルの変更で対応可能であるため、本タスクでまとめて修正を行う。

### 背景と目的

前タスク（phaseGuide追加）のワークフロー実行中に、システムの根本的な設計上の問題が3つ顕在化した。
loop-detectorは5分以上古いタイムスタンプを削除する機能を持つが、エントリ自体を削除する機能がなく、過去のワークフローで編集された3000個以上のファイルエントリが永久に蓄積される設計となっている。
subagent成果物の品質維持のためにartifact-validatorが導入されているが、CLAUDE.mdのsubagent起動テンプレートがバリデーションルールを抽象的にしか記述しておらず、subagentが具体的な書き方を理解できない状態である。
タスク開始前から存在した変更を自動的にスコープチェックから除外する機構（preExistingChanges）が実装されているが、エラーハンドリングの不備により機能していない可能性がある。
これらの問題は全て、システムの長期運用において重大な障害を引き起こす可能性があるため、早急な修正が必要である。

### 期待される成果

問題1の修正により、loop-detector-state.jsonのサイズが継続的に管理され、フッククラッシュが解消される。
問題2の修正により、subagent成果物のバリデーション成功率が向上し、Orchestratorによる手動修正の頻度が大幅に減少する。
問題3の修正により、タスク開始前の変更が自動的にスコープチェックから除外され、不要な遷移ブロックが発生しなくなる。
修正後のシステムは、長期的な運用において安定性と自動化レベルが向上し、ワークフローの実行成功率が改善される。

## 機能要件

### REQ-1: loop-detectorのファイルエントリpruning機能（問題1対応）

#### REQ-1.1: checkLoop関数内でのエントリ削除

checkLoop関数（workflow-plugin/hooks/loop-detector.js）において、filterOldTimestamps呼び出し後に空になったエントリを削除する処理を追加する。
削除判定条件は、timestampsの長さが0かつlastWarningが未設定（undefined）のエントリとする。
lastWarningが設定されているエントリは、警告抑止期間の管理に使用されるため、timestampsが空でも削除してはいけない。
削除処理は、Object.entriesでstate.filesを走査し、削除対象エントリのキーをdelete演算子で削除する方式とする。
処理のタイミングは、filterOldTimestampsで古いタイムスタンプを削除した直後とし、毎回のcheckLoop呼び出し時に自動的にクリーンアップが進むようにする。

#### REQ-1.2: saveState関数内での一括クリーンアップ

saveState関数（workflow-plugin/hooks/loop-detector.js）において、保存前に空エントリを一括削除する処理を追加する。
削除判定条件は、REQ-1.1と同一（timestampsが空かつlastWarningが未設定）とし、コードの一貫性を保つ。
削除処理は、Object.fromEntriesとObject.entriesを組み合わせ、フィルタリングした結果を新しいfilesオブジェクトとして再構築する方式とする。
処理のタイミングは、JSONシリアライズ直前とし、確実にクリーンアップ済みの状態をファイルに書き込むようにする。
この処理は、REQ-1.1の補完的な役割であり、万が一checkLoop時の削除が漏れても最終的にファイル保存時にクリーンアップされる仕組みとする。

#### REQ-1.3: 既存肥大化ファイルの自動クリーンアップ

修正実装後の初回実行時に、既存の肥大化したloop-detector-state.json（3000個以上のエントリ）が自動的にクリーンアップされることを保証する。
バッチ削除の実装は不要であり、通常のcheckLoop呼び出しで逐次削除が行われる設計とする（タイムアウト3秒以内に処理が完了する必要がある）。
クリーンアップ後のファイルサイズは、実際に最近編集されたファイルの数（通常は数十個程度）に応じて数KBから数十KBに収まることを期待する。
クリーンアップの進行状況は、ログ出力等で確認できるようにする必要はなく、フック処理がタイムアウトせずに正常完了することが成功の指標となる。

### REQ-2: CLAUDE.mdのバリデーションルール具体例追加（問題2対応）

#### REQ-2.1: subagent起動テンプレートの改善

CLAUDE.mdのsubagent起動テンプレート（行234-277）に、artifact-validatorのバリデーションルールの具体例を追加する。
現在の記述「同じ行が3回以上出現しないこと」は抽象的すぎるため、具体的な禁止パターン例を追加する（例：「結果: パス」が3回出現すると失敗、「実装確認位置：」が繰り返されると失敗）。
各行をコンテキストで差別化する方法の具体例を追加する（例：「結果: パス」の代わりに「ユーザー作成APIの動作確認: パス」のように文脈を含める方法）。
悪い例と良い例を並べて記述し、subagentが視覚的に理解できるようにする（コードブロックではなく、実際のMarkdownテキストとして提示）。
構造要素除外の仕組み（テーブルデータ行やMarkdownラベルが重複カウントから除外される仕組み）を明示し、subagentが表形式やラベル形式を適切に使用できるようにする。

#### REQ-2.2: セクション密度要件の明確化

各セクション（## 見出し）内に最低5行の実質行を含める要件を明示する（現在は「最低5行」のみ記述されており実質行の定義が不明瞭）。
セクション密度30%以上を維持する方法の具体例を追加する（例：箇条書きだけでなく説明文を含める、表形式で詳細情報を含めるなど）。
短い行（20文字未満）の比率が30%を超えないようにする要件を明示する（現在は記載がない）。
セクション行数不足の具体例を追加する（例：サマリーセクションが1-3行しかない場合は失敗）。
密度維持の悪い例と良い例を並べて記述し、subagentがボリュームのある成果物を作成する方法を理解できるようにする。

#### REQ-2.3: 必須セクション名の明示

各フェーズの必須セクション名を、subagent起動テンプレート内に明示的にリスト化する。
現在はCLAUDE.mdのフェーズ詳細説明セクション（manual_testフェーズ行1295-1311等）に記載されているが、subagent起動テンプレート内に再掲することで見落としを防ぐ。
必須セクションの例として、manual_testフェーズでは「## テストシナリオ」「## テスト結果」、security_scanフェーズでは「## 脆弱性スキャン結果」「## 検出された問題」を明示する。
全フェーズに共通の必須セクション「## サマリー」を最優先で記述し、サマリーセクションがない場合は必ず失敗することを強調する。
フェーズごとに固有の必須セクションが異なる理由（フェーズの目的に応じた成果物の構造が異なるため）を簡潔に説明し、subagentが自分のフェーズの成果物要件を理解できるようにする。

#### REQ-2.4: 禁止パターンの具体例追加

コードブロック内でのブラケット記法（`{変数名}`, `{パラメータ}`等）の禁止を明示し、具体的な代替記述方法を提示する。
コードブロック内では実際のコード例を記述するか、散文で説明する方法を推奨する（例：「パラメータは文字列型で、ユーザー名を指定します」のように記述）。
「未定」「該当なし」「N/A」「TBD」の使用禁止を明示し、代替として具体的な調査結果や判断根拠を記述する方法を提示する（例：「現時点で該当する脅威は特定されていない」の代わりに「認証フローの分析を実施したが追加の脅威は検出されなかった」と記述）。
テンプレート的な記述の禁止を明示し、各行を実際のタスク内容に応じて具体化する方法を提示する。

### REQ-3: preExistingChanges記録のエラーハンドリング改善（問題3対応）

#### REQ-3.1: workflow_start時のgit diff失敗時の処理改善

start.ts（workflow-plugin/mcp-server/src/tools/start.ts）のpreExistingChanges記録処理（行100-119）において、git diff実行失敗時のエラーハンドリングを改善する。
現在の実装では、エラー発生時にconsole.warnで警告を出力するのみであり、preExistingChangesが明示的に空配列として設定されない可能性がある。
エラーハンドリング内で確実にpreExistingChangesを空配列として設定し、undefinedになることを防ぐ。
git diffコマンド実行前にundefinedチェックを追加し、HEADが存在しない場合（初回コミット前）も空配列を設定する。
エラーログには、git diffが失敗した理由（コマンド実行エラー、HEADが存在しない等）を含め、デバッグを容易にする。

#### REQ-3.2: preExistingChanges記録のログ出力強化

workflow_start時に、preExistingChangesの記録件数と先頭5件のファイルパスをconsole.logで出力する。
ログ出力は、git diff実行成功時のみ行い、失敗時は「preExistingChanges: 0 files (git diff failed)」のようなメッセージを出力する。
ログフォーマットは、「preExistingChanges: N files recorded」のように件数を明示し、続けて先頭5件のファイルパスを改行区切りで出力する。
commitフェーズでのスコープ検証時にも、preExistingChangesの内容をデバッグログとして出力し、スコープチェックから除外されたファイルを確認できるようにする。
ログ出力はMCPサーバーの標準出力経由でClaude Codeのログに記録され、ユーザーがワークフロー実行時に確認できる。

#### REQ-3.3: EXCLUDE_PATTERNSの拡張

scope-validator.ts（workflow-plugin/mcp-server/src/scope-validator.ts）のEXCLUDE_PATTERNS配列に、設定ファイル系のパターンを追加する。
追加対象は、.mcp.json、.gitignore、.env.example、tsconfig.json、vitest.config.ts、vite.config.tsとする（ソースコードは除外対象外）。
パターンマッチングは、既存の実装に倣い、文字列の末尾マッチまたは完全一致で行う（例：path.endsWith('.mcp.json')またはpath === '.mcp.json'）。
EXCLUDE_PATTERNSの拡張により、設定ファイルの変更が常にスコープチェックから除外され、preExistingChangesの記録が不完全でも意図しないブロックを防ぐことができる。
追加したパターンは、既存のパターン（Markdownファイル、package.json等）と同様の方式で処理され、パターンマッチングのロジック変更は不要である。

#### REQ-3.4: preExistingChangesのundefinedフォールバック改善

next.tsのスコープ検証呼び出し（行366）において、taskState.scope?.preExistingChangesがundefinedの場合のフォールバック処理を改善する。
現在の実装では、undefinedの場合に空配列にフォールバックしているが、これはstart.tsでの記録が正しく行われなかった場合のフォールバックである。
フォールバック時にワーニングログを出力し、preExistingChangesが記録されていないことをユーザーに通知する。
ワーニングログには、「preExistingChanges not recorded, falling back to empty array」のようなメッセージを含め、手動でworkflow_set_scopeが必要になる可能性を示唆する。
フォールバック処理は、システムの堅牢性のために維持するが、正常なワークフロー実行では発生しないことが期待される。

## 非機能要件

### NFR-1: パフォーマンス要件

loop-detectorのpruning処理は、フック実行のタイムアウト3秒以内に完了する必要がある。
既存の肥大化した状態ファイル（3000個以上のエントリ）に対しても、初回実行時にタイムアウトせずにクリーンアップが完了することを保証する。
pruning処理のアルゴリズム複雑度は、O(n)（エントリ数に比例）とし、ファイル数が増加しても線形に処理時間が増加する程度に抑える。
git diff実行のタイムアウトは、現在の実装で設定されている値を維持し、大規模リポジトリでも適切に動作することを保証する。
EXCLUDE_PATTERNSの拡張により、パターンマッチングの処理時間が有意に増加しないことを確認する（パターン数が少数であるため影響は軽微）。

### NFR-2: 信頼性要件

loop-detectorのpruning処理は、lastWarningが設定されているエントリを誤って削除しないことを保証する（警告抑止期間の管理が正しく機能するため）。
preExistingChangesの記録処理は、git diffが失敗した場合でも確実に空配列を設定し、undefinedによるランタイムエラーを防ぐ。
EXCLUDE_PATTERNSのパターンマッチングは、ソースコードファイルを誤って除外しないことを保証する（設定ファイルのみを対象とする）。
subagent起動テンプレートの改善により、バリデーション成功率が向上することを期待するが、既存の正しく記述された成果物に対するバリデーション結果は変わらないことを保証する。

### NFR-3: 保守性要件

pruning処理の削除判定条件は、checkLoop関数とsaveState関数で統一し、コードの一貫性を保つ。
CLAUDE.mdのバリデーションルール具体例は、artifact-validator.tsの実装と一致させ、ドキュメントとコードの乖離を防ぐ。
EXCLUDE_PATTERNSの追加は、既存のパターンと同様の方式で実装し、パターンマッチングのロジック変更を最小限に抑える。
ログ出力のフォーマットは、統一的なスタイルとし、デバッグ時に情報を見つけやすくする。
コメントには、削除判定条件の理由（lastWarningが設定されている場合は削除しない理由等）を記述し、将来の保守担当者が意図を理解できるようにする。

### NFR-4: 互換性要件

loop-detectorの状態ファイル形式は変更せず、既存のフィールド（files, counts, timestamps, lastWarning, warningSuppressed）をそのまま使用する。
pruning処理により古いエントリが削除されても、過去のワークフローの実行履歴には影響を与えない（loop-detection-log.jsonは別管理）。
preExistingChangesのフィールド追加は既に実装済みであり、既存のworkflow_start, workflow_nextの動作に影響を与えない。
EXCLUDE_PATTERNSの拡張は、既存の除外パターンの動作に影響を与えず、追加パターンのみが新たに除外される。
CLAUDE.mdの変更は、既存のワークフロー定義に影響を与えず、次回以降起動されるsubagentから新しいテンプレートが適用される。

### NFR-5: テスト容易性要件

loop-detectorのpruning処理は、ユニットテストで削除判定条件を検証できるように、削除判定ロジックを独立した関数として抽出することが望ましい（既存のfilterOldTimestamps関数と同様）。
preExistingChanges記録のログ出力により、ワークフロー実行時に記録内容を確認でき、手動テストが容易になる。
EXCLUDE_PATTERNSの拡張は、統合テストで新しいパターンが正しく除外されることを検証できる。
subagent起動テンプレートの改善効果は、実際のワークフロー実行でsubagent成果物のバリデーション成功率を計測することで確認できる。

## 成功基準

### SC-1: loop-detectorの肥大化解消

loop-detector-state.jsonのファイルサイズが、修正実装後のワークフロー実行により100KB未満に削減される。
フック実行時のタイムアウトエラー（exit code 2）が発生しなくなり、Edit/Writeツールが安定して使用できる。
新しいワークフロー実行時に、古いエントリが自動的に削除され、ファイルサイズが肥大化しない。
状態ファイルに含まれるエントリ数が、実際に最近編集されたファイルの数（数十個程度）に収束する。

### SC-2: subagent成果物のバリデーション成功率向上

parallel_verificationフェーズのsubagent成果物が、初回生成時にバリデーションに成功する確率が向上する（現状は複数回の修正が必要）。
Orchestratorによる手動修正の頻度が大幅に減少し、ワークフローの自動化レベルが向上する。
subagent成果物に含まれる同一行の繰り返し（3回以上）が減少し、各行がコンテキストで差別化される。
セクション行数不足（5行未満）のエラーが減少し、各セクションに十分なボリュームの内容が含まれる。

### SC-3: preExistingChanges処理の自動化

タスク開始前から存在した変更が、自動的にスコープチェックから除外され、commitフェーズへの遷移がブロックされない。
workflow_start時のログ出力で、preExistingChangesの記録件数が確認でき、記録が正しく行われていることが検証できる。
設定ファイル（.mcp.json、tsconfig.json等）の変更が、常にスコープチェックから除外され、意図しないブロックが発生しない。
手動でworkflow_set_scopeを実行する必要がなくなり、ワークフローの自動化レベルが向上する。

## 制約事項

### CONST-1: 後方互換性の維持

loop-detectorの状態ファイル形式を変更せず、既存のフィールド構造を維持する。
pruning処理により古いエントリが削除されても、現在実行中のワークフローには影響を与えない（状態の一貫性を保つ）。
CLAUDE.mdの変更は、既に起動済みのsubagentには適用されず、次回以降起動されるsubagentから新しいテンプレートが適用される。
EXCLUDE_PATTERNSの拡張は、既存の除外パターンの動作を変更せず、追加パターンのみが新たに除外される。

### CONST-2: 修正範囲の限定

本タスクでは、loop-detector.js、CLAUDE.md、start.ts、scope-validator.tsの4ファイルのみを修正対象とする。
artifact-validator.tsのバリデーションロジック自体は変更せず、CLAUDE.mdのドキュメント改善のみで対応する。
definitions.tsのPHASE_GUIDESへのバリデーション要件追加は、本タスクの範囲外とする（将来の改善タスクで対応）。
loop-detection-log.jsonの管理方法（100件制限）は変更せず、loop-detector-state.jsonのpruningのみを実装する。

### CONST-3: パフォーマンス制約

loop-detectorのpruning処理は、フック実行のタイムアウト3秒以内に完了する必要がある。
大量のエントリ（3000個以上）を一度にクリーンアップする初回実行時でも、タイムアウトしないアルゴリズムを使用する。
git diff実行のタイムアウトは、現在の実装で設定されている値を維持し、変更しない。
EXCLUDE_PATTERNSの拡張によるパターンマッチング処理の増加は、無視できる程度（数ミリ秒以下）に抑える。

### CONST-4: テスト環境の制約

修正実装後のテストは、実際のワークフロー実行で行い、手動で成功基準を確認する。
loop-detectorのpruning処理は、既存の肥大化した状態ファイルを使用してテストを行い、クリーンアップが正しく動作することを確認する。
subagent起動テンプレートの改善効果は、parallel_verificationフェーズの実際のsubagent実行で確認する。
preExistingChanges処理の改善は、タスク開始前に意図的に設定ファイルを変更した状態でワークフローを実行してテストする。

## 依存関係

### DEP-1: 既存実装への依存

loop-detectorのpruning処理は、既存のfilterOldTimestamps関数の実装に依存し、同様のパターンで実装する。

CLAUDE.mdのバリデーションルール具体例は、artifact-validator.tsの実装（isStructuralLine関数、形式的テキスト検出ロジック等）に依存し、実装と一致した内容を記述する。

### DEP-2: MCPサーバーの動作環境

修正実装後、MCPサーバーの再起動が必要である（Node.jsのモジュールキャッシュにより、dist/*.jsファイルの変更が即座に反映されないため）。
loop-detectorのpruning処理は、Node.jsのファイルシステムAPI（fs.readFileSync, fs.writeFileSync）に依存する。
git diff実行は、システムにgitコマンドがインストールされていることに依存する。
CLAUDE.mdの変更は、Claude CodeがCLAUDE.mdを読み込んでsubagentに渡す仕組みに依存する。

### DEP-3: ワークフローフェーズへの依存

問題3の修正（preExistingChanges処理）は、docs_updateフェーズからcommitフェーズへの遷移時にのみ動作確認できる。
問題2の修正（subagent起動テンプレート改善）は、parallel_verificationフェーズのsubagent実行時にのみ効果が確認できる。
問題1の修正（loop-detectorのpruning）は、Edit/Writeツール使用時に動作するため、implementationフェーズ以降で確認できる。

## リスク

### RISK-1: pruning処理のバグによるデータ損失

削除判定条件（timestampsが空かつlastWarningが未設定）が誤っている場合、必要なエントリが削除される可能性がある。
lastWarningが設定されているエントリを誤って削除すると、警告抑止期間の管理が正しく機能しなくなる。
回避策として、削除前にエントリの内容をログ出力し、削除されるエントリを確認できるようにする（開発時のみ）。
削除判定ロジックを独立した関数として抽出し、ユニットテストで削除条件を検証する。

### RISK-2: preExistingChanges記録の不完全性

git diff実行が常に失敗する環境では、preExistingChangesが常に空配列となり、タスク開始前の変更がスコープチェックでブロックされる。
HEADが存在しない状態（初回コミット前）では、git diffが実行できず、preExistingChangesが記録されない。
回避策として、EXCLUDE_PATTERNSに設定ファイルを追加し、preExistingChangesが不完全でも最低限の除外が機能するようにする。
エラーハンドリングで詳細なログを出力し、git diffが失敗した理由をユーザーが把握できるようにする。

### RISK-3: CLAUDE.mdの具体例が不十分

バリデーションルールの具体例を追加しても、subagentが理解できない可能性がある（subagentのコンテキスト理解能力に依存）。
具体例が多すぎると、CLAUDE.mdのサイズが肥大化し、subagentのコンテキストウィンドウを圧迫する可能性がある。
回避策として、最も頻出する失敗パターン（同一行の繰り返し、セクション行数不足）に絞って具体例を記述する。
実際のワークフロー実行でsubagent成果物のバリデーション成功率を計測し、効果が不十分な場合は追加の改善を検討する。

### RISK-4: EXCLUDE_PATTERNSの拡張による意図しない除外

EXCLUDE_PATTERNSに設定ファイルを追加することで、設定ファイルの変更が常に除外され、意図的に設定ファイルのみを変更するタスクでスコープチェックが機能しなくなる。
回避策として、設定ファイルの変更が主目的のタスクでは、workflow_set_scopeで明示的にスコープに含める運用とする。
EXCLUDE_PATTERNSの拡張は、設定ファイルの変更が副次的な変更として発生する場合（依存パッケージ追加等）を想定した対応である。

### planning フェーズで作成する仕様書に含めるべき内容

問題1（loop-detectorのpruning）の実装仕様として、削除判定ロジックの詳細（条件式、処理タイミング、アルゴリズム）を記述する。

loop-detectorのpruning処理のテストケースとして、削除対象エントリと非削除対象エントリの判定が正しく行われることを確認するテストを設計する。
preExistingChanges記録のテストケースとして、git diff成功時と失敗時の両方でpreExistingChangesが正しく設定されることを確認するテストを設計する。
EXCLUDE_PATTERNSのテストケースとして、追加したパターンが正しく除外されることを確認するテストを設計する。

### implementation フェーズでの実装順序

優先度の高い問題1（loop-detector）から実装し、フッククラッシュを最優先で解消する。

各問題の実装完了後、個別に動作確認を行い、全問題の修正が完了してから統合テストを実施する。
