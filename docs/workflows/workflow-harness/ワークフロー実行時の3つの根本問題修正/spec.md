# 仕様書: ワークフロー実行時の3つの根本問題修正

## サマリー

本仕様書では、ワークフロー実行時に発生した3つの根本的な問題の修正仕様を定義する。
第一の問題であるloop-detector-state.jsonの肥大化は、ファイルエントリのpruningメカニズム欠如が根本原因であり、checkLoop関数とsaveState関数に削除処理を追加することで解決する。
第二の問題であるsubagent成果物のバリデーション失敗は、CLAUDE.mdのバリデーションルールが抽象的すぎることが原因であり、具体例と禁止パターンを追加することで解決する。
第三の問題であるスコープチェックのpreExistingChanges処理は、git diff失敗時のエラーハンドリング不備が原因であり、確実に空配列を設定する処理とログ出力強化で解決する。
全3問題とも修正箇所は明確で、実装難易度は低く、1-2ファイルの変更で対応可能である。
各問題の修正仕様を詳細に記述し、implementationフェーズで迷いなく実装できるようにする。

- 目的: 3つの根本問題の修正仕様を詳細化し、実装フェーズで確実に修正を完了させる
- 主要な決定事項: FIX-1はpruning処理の追加、FIX-2はCLAUDE.mdの具体例追加、FIX-3はエラーハンドリング改善
- 次フェーズで必要な情報: 各修正の具体的な実装箇所と変更内容（後述）

---

## 概要

前タスク（phaseGuide追加）のワークフロー実行中に、システムの根本的な設計上の問題が3つ顕在化した。
問題1はloop-detectorの状態ファイルが107KBに肥大化しフッククラッシュを引き起こしている。
問題2はsubagent成果物がartifact-validatorの検証に繰り返し失敗しOrchestratorが手動修正を強いられている。
問題3はスコープチェックがタスク開始前から存在した変更をブロックしワークフロー遷移を妨げている。
本仕様書では、各問題の根本原因分析結果に基づき、修正箇所と変更内容を具体的に定義する。
修正後のシステムは、長期的な運用において安定性と自動化レベルが向上し、ワークフローの実行成功率が改善される。

## 実装計画

### FIX-1: loop-detector.js修正仕様

#### 修正箇所1: checkLoop関数内でのエントリ削除

対象ファイルはworkflow-plugin/hooks/loop-detector.jsであり、修正位置はcheckLoop関数の行342付近（filterOldTimestamps呼び出し直後）である。

追加する処理の内容は以下のとおりである。filterOldTimestampsで古いタイムスタンプを削除した直後に、fileEntryのtimestampsの長さが0かつlastWarningが未設定（falsyな値）であるかを判定する。両条件を満たす場合、delete演算子でstate.filesからnormalizedPathキーのエントリを削除し、saveStateを呼び出してreturnする。

削除判定条件はtimestamps配列の長さが0であること、かつlastWarningが設定されていない（undefinedまたはnull）ことの2つである。lastWarningが設定されているエントリは、警告抑止期間の管理に使用されるため、timestampsが空であっても削除してはいけない。

処理タイミングはfilterOldTimestamps呼び出し直後であり、毎回のcheckLoop実行時に行われる。

#### 修正箇所2: saveState関数内での一括クリーンアップ

対象ファイルは同じくworkflow-plugin/hooks/loop-detector.jsであり、修正位置はsaveState関数の行183-191（JSONシリアライズ直前）である。

変更内容は、ensureStateDir呼び出し後、fs.writeFileSync呼び出し前に、Object.fromEntriesとObject.entries(state.files).filterを使用して空エントリを一括削除する処理を追加する。フィルタリング条件はcheckLoop関数と同一で、entry.timestamps.lengthが0より大きいか、entry.lastWarningがtruthyである場合にエントリを保持する。フィルタリング結果をstate.filesに代入してからJSONシリアライズを行う。

この処理はcheckLoop内での削除漏れを補完するためのセーフティネットである。

#### 期待される効果

修正後の初回実行時に、既存の肥大化したloop-detector-state.json（3000個以上のエントリ）が自動的にクリーンアップされる。
クリーンアップはObject.fromEntriesによる一括フィルタリング方式で行われ、フック処理のタイムアウト3秒以内に完了する。
クリーンアップ後のファイルサイズは、実際に最近編集されたファイルの数（通常は数十個程度）に応じて数KBから数十KBに収まる。
新しいワークフロー実行時に、古いエントリが自動的に削除され、ファイルサイズが肥大化しない状態が維持される。

### FIX-2: CLAUDE.md改善仕様

#### 修正箇所: subagent起動テンプレートの成果物品質要件セクション

対象ファイルはCLAUDE.mdであり、修正位置はsubagent起動テンプレートの「成果物品質要件（artifact-validator準拠）」セクションである。

#### 追加内容1: 重複行禁止の具体例

「重複行禁止」セクションの後に具体例を追加する。悪い例として、テスト結果を列挙する際に全ての行末が同じ結語で終わるパターンを示す（例: 3行とも「正常に動作する」で終わる）。良い例として、各行にテスト対象の固有情報（応答時間、ステータスコード、検証内容）を含めて差別化するパターンを示す。

構造要素除外の仕組みも説明する。テーブルデータ行（パイプ区切りの2列以上の行）とMarkdownラベル（太字テキストにコロンが続く形式）は重複カウントから除外される旨を記載する。

#### 追加内容2: セクション密度要件の具体例

「行数・密度要件」セクションの後に具体例を追加する。悪い例として、セクション内に実質行が2行しかなく密度が低いパターンを散文で説明する。良い例として、6行以上の実質行を含み各行が具体的な情報を持つパターンを散文で説明する。

#### 追加内容3: フェーズ別必須セクション一覧

「必須セクション」セクションの後に一覧を追加する。全フェーズ共通でサマリーセクションが必須である旨と、manual_test（テストシナリオ、テスト結果）、security_scan（脆弱性スキャン結果、検出された問題）、performance_test（パフォーマンス計測結果、ボトルネック分析）、e2e_test（E2Eテストシナリオ、テスト実行結果）の各フェーズ固有の必須セクション名を明示する。

#### 追加内容4: 禁止パターンの具体例

新規「禁止パターン」セクションを追加する。コードブロック内のブラケット記法（角括弧で変数名やプレースホルダーを囲む記法）が禁止されている旨を説明し、代わりに散文での説明または実際の値を使ったコード例を推奨する。また、「該当なし」や決定していないことを示す略語等の禁止ワードの代わりに具体的な調査結果と判断根拠を記述すべきことを説明する。

#### 期待される効果

subagentがバリデーションルールの具体例を参照することで、バリデーション成功率が向上する。
Orchestratorによる手動修正の頻度が大幅に減少し、ワークフローの自動化レベルが向上する。
subagent成果物に含まれる同一行の繰り返し（3回以上）が減少し、各行がコンテキストで差別化される。
セクション行数不足（5行未満）のエラーが減少し、各セクションに十分なボリュームの内容が含まれる。

### FIX-3: start.ts / scope-validator.ts修正仕様

#### 修正箇所1: start.tsのエラーハンドリング改善

対象ファイルはworkflow-plugin/mcp-server/src/tools/start.tsであり、修正位置はpreExistingChanges記録処理（行100-119）である。

変更内容は3点ある。第一に、git diff成功時にpreExistingChangesの件数と先頭5件をconsole.logで出力する。0件の場合もその旨を出力する。第二に、git diff失敗時のcatchブロック内で、明示的にpreExistingChangesに空配列を代入する（letで宣言済みの変数に対する再代入）。第三に、エラーがErrorインスタンスの場合、e.messageにHEADが含まれるかを判定し、含まれる場合は「初回コミット前のためpreExistingChangesは空」という旨の情報ログを出力する。

#### 修正箇所2: scope-validator.tsのEXCLUDE_PATTERNS拡張

対象ファイルはworkflow-plugin/mcp-server/src/validators/scope-validator.tsであり、修正位置はEXCLUDE_PATTERNS配列（行646-656付近）である。

追加するパターンは6個である。ドット始まりのmcp.json（MCPサーバー設定）、ドット始まりのgitignore（Git除外設定）、ドット始まりのenv.example（環境変数テンプレート）、tsconfig.json（TypeScript設定）、vitest.config.ts（Vitestビルド設定）、vite.config.ts（Viteビルド設定）をそれぞれ正規表現パターンとして追加する。

パターンマッチング方式は既存の実装に倣い、正規表現の末尾一致（$）を使用する。

#### 期待される効果

タスク開始前から存在した変更が自動的にスコープチェックから除外されcommitフェーズへの遷移がブロックされない。
workflow_start時のログ出力でpreExistingChangesの記録件数が確認でき記録が正しく行われていることが検証できる。
設定ファイルの変更が常にスコープチェックから除外され意図しないブロックが発生しない。
手動でworkflow_set_scopeを実行する必要がなくなりワークフローの自動化レベルが向上する。

## 非機能要件

### パフォーマンス要件
loop-detectorのpruning処理はフック実行のタイムアウト3秒以内に完了する必要がある。
既存の肥大化した状態ファイル（3000個以上のエントリ）に対しても初回実行時にタイムアウトせずにクリーンアップが完了することを保証する。
pruning処理のアルゴリズム複雑度はO(n)（エントリ数に比例）とする。
git diff実行のタイムアウトは現在の実装の設定値を維持する。
EXCLUDE_PATTERNSの追加による正規表現マッチングの性能への影響は6パターン追加で無視できる程度である。

### 信頼性要件
loop-detectorのpruning処理はlastWarningが設定されているエントリを誤って削除しないことを保証する。
preExistingChangesの記録処理はgit diffが失敗した場合でも確実に空配列を設定しundefinedによるランタイムエラーを防ぐ。
EXCLUDE_PATTERNSのパターンマッチングはソースコードファイルを誤って除外しないことを保証する。
CLAUDE.mdの改善により既存の正しく記述された成果物に対するバリデーション結果は変わらないことを保証する。
saveState関数内のクリーンアップ処理はJSON書き込みエラー時にも状態の一貫性を維持する。

### 保守性要件
pruning処理の削除判定条件はcheckLoop関数とsaveState関数で統一しコードの一貫性を保つ。
CLAUDE.mdのバリデーションルール具体例はartifact-validator.tsの実装と一致させドキュメントとコードの乖離を防ぐ。
EXCLUDE_PATTERNSの追加は既存のパターンと同様の方式で実装しパターンマッチングのロジック変更を最小限に抑える。
ログ出力のフォーマットは統一的なスタイルとしデバッグ時に情報を見つけやすくする。
コメントには削除判定条件の理由を記述し将来の保守担当者が意図を理解できるようにする。

### 互換性要件
loop-detectorの状態ファイル形式は変更せず既存のフィールド構造をそのまま使用する。
pruning処理により古いエントリが削除されても過去のワークフロー実行履歴には影響を与えない。
preExistingChangesのフィールドは既に実装済みであり既存のワークフローツールの動作に影響を与えない。
EXCLUDE_PATTERNSの拡張は既存の除外パターンの動作に影響を与えず追加パターンのみが新たに除外される。
CLAUDE.mdの変更は既存のワークフロー定義に影響を与えず次回以降起動されるsubagentから新しいテンプレートが適用される。

## 実装順序

Phase 1としてFIX-1（loop-detector.js修正）を最優先で実装する。緊急度が最も高くフック完全停止を解消する効果がある。checkLoop関数内の削除処理追加、saveState関数内のクリーンアップ処理追加、削除判定条件のコメント追加を行い、MCPサーバー再起動後に既存の肥大化した状態ファイルがクリーンアップされることを確認する。

Phase 2としてFIX-2（CLAUDE.md改善）を実装する。手動回避は可能だが非効率であるため優先度は中である。重複行禁止の具体例、セクション密度要件の具体例、フェーズ別必須セクション一覧、禁止パターンの具体例の4項目を追加し、artifact-validator.tsの実装と一致していることを確認する。

Phase 3としてFIX-3（start.ts / scope-validator.ts修正）を実装する。手動回避が容易であるため優先度は低い。start.tsのログ出力追加とエラーハンドリング改善、scope-validator.tsのEXCLUDE_PATTERNSへの設定ファイルパターン追加を行い、MCPサーバー再起動後にログ出力を確認する。

統合テストとして全3問題の修正が完了した状態で実際のワークフロー実行を行い、loop-detectorの状態ファイル削減、subagent成果物のバリデーション合格、commitフェーズ遷移の成功を確認する。
各Phaseの修正は独立しているため、Phase 1の修正完了後にMCPサーバーを再起動してloop-detector動作を確認し、問題なければPhase 2とPhase 3の修正に進む。
