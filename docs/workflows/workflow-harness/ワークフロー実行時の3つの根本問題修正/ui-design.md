# UI設計: ワークフロー実行時の3つの根本問題修正

## サマリー

本タスクはバックエンド（hooks/MCPサーバー/CLAUDE.md）の内部処理改善であり、ユーザーインターフェース画面の変更は伴わない。
本ドキュメントでは、修正がシステムの運用面に与える影響を「運用者向けインターフェース変更」として記述する。
主な影響は、CLIログ出力の追加強化、エラーメッセージの改善、設定ファイル除外による自動化の向上である。
FIX-1はloop-detector状態ファイルの削減、FIX-2はCLAUDE.mdの具体例追加、FIX-3はpreExistingChangesのログ出力とEXCLUDE_PATTERNS拡張である。
次のimplementationフェーズでは、これらの仕様に基づいて各ファイルの修正を進める。

- 目的: バックエンド修正がもたらす運用面への影響を明確化し、ログ形式・除外パターンの仕様を固める
- 主要な決定事項: FIX-1は状態ファイルサイズ削減、FIX-2はバリデーション具体例の追加、FIX-3は設定ファイル自動除外
- 次フェーズで必要な情報: 各修正の具体的なログ形式、除外パターンの実装詳細、状態ファイルのクリーンアップ仕様

---

## CLIインターフェース設計

### workflow_startコマンド実行時の出力変更

FIX-3の修正により、workflow_start実行時にpreExistingChangesの記録状況がCLI出力に追加される。
成功時にはpreExistingChangesの件数と先頭5件のファイルパスが表示される。
0件の場合にはタスク開始前の変更が存在しない旨が表示される。
git diffが失敗した場合（初回コミット前等）には、失敗理由とともに空配列が設定された旨が表示される。
出力フォーマットはworkflowプレフィックスで統一し、既存のログ形式との一貫性を維持する。

### workflow_nextコマンド実行時の出力変更

FIX-1の修正により、loop-detectorのタイムアウトエラーが発生しなくなるため、workflow_next実行前のEdit/Write操作が安定する。
FIX-3の修正により、commitフェーズへの遷移時にスコープ外エラーが発生しなくなる。
CLIコマンドの入力形式（引数、オプション）には一切変更がない。
既存のworkflow_status、workflow_list等のコマンド出力にも影響はない。
ユーザーが実行するコマンド形式は全て現行のまま維持される。

## エラーメッセージ設計

### FIX-1: loop-detectorタイムアウトエラーの消滅

修正前は、loop-detector-state.jsonが107KBに肥大化した状態でEdit/Writeツールを使用すると「No stderr output」エラーが発生していた。
このエラーはフック処理が3秒のタイムアウトを超過した際にexit code 2で発生するものである。
修正後はpruningメカニズムにより状態ファイルサイズが数KB〜数十KBに維持され、タイムアウトが発生しなくなる。
既存のループ検出警告メッセージ（同一ファイルの繰り返し編集検出）は変更されない。
新たなエラーメッセージの追加は行わず、既存のエラー発生率を低減する方向の改善である。

### FIX-3: スコープ外エラーの消滅

修正前は、タスク開始前に変更された設定ファイル（.mcp.json等）がcommitフェーズでスコープ外として検出されていた。
エラーメッセージ形式は「commitフェーズ内: ファイル .mcp.json はスコープ外です」であった。
修正後はEXCLUDE_PATTERNSの拡張により設定ファイルが自動的に除外されるため、このエラーが発生しなくなる。
preExistingChangesの記録改善により、事前変更ファイルも正確にスコープ検証から除外される。
既存のスコープ外検出ロジック自体は変更されず、除外パターンの追加のみで対応する。

## APIレスポンス設計

### workflow_startレスポンスの変更

workflow_startのMCPレスポンス構造自体には変更はない。
内部処理としてpreExistingChangesの記録がより確実に行われるようになるが、レスポンスフィールドの追加や削除はない。
エラーレスポンスの形式も変更されず、既存のクライアント実装との互換性は完全に維持される。
console.logによるログ出力はMCPサーバーの標準出力に表示されるものであり、MCPプロトコルのレスポンスには含まれない。
taskState.scope.preExistingChangesフィールドは既に定義済みであり、型定義の変更は不要である。

### workflow_nextレスポンスの変更

workflow_nextのレスポンス構造にも変更はない。
スコープ検証ロジックの内部処理が改善されるが、成功時・失敗時のレスポンスフォーマットは現行のまま維持される。
EXCLUDE_PATTERNSの追加はscope-validator内部の処理であり、MCPレスポンスには影響しない。
phaseGuide情報の返却（前タスクで実装済み）は本タスクの修正範囲外であり変更されない。
全てのMCPツール（workflow_status、workflow_complete_sub等）のレスポンス形式も変更されない。

## 設定ファイル設計

### EXCLUDE_PATTERNSの拡張

scope-validator.tsのEXCLUDE_PATTERNS配列に6個の設定ファイルパターンを追加する。
追加パターンは.mcp.json（MCPサーバー設定）、.gitignore（Git除外設定）、.env.example（環境変数テンプレート）、tsconfig.json（TypeScript設定）、vitest.config.ts（Vitest設定）、vite.config.ts（Vite設定）の6つである。
各パターンは正規表現の末尾一致を使用し、既存の実装方式を踏襲する。
ソースコードファイル（.ts、.tsx、.js等）は除外パターンに含まれず、意図的な除外を防止する。
既存の除外パターン（Markdownファイル、package.json、ロックファイル、.claude/state/配下等）は全て維持される。

### loop-detector-state.jsonの構造維持

loop-detector-state.jsonのJSON構造（filesオブジェクト内にファイルパスをキー、count・timestamps・lastWarning・warningSuppressを値として持つ形式）は変更されない。
pruning処理はエントリの削除のみを行い、新しいフィールドの追加や既存フィールドの変更は行わない。
削除判定条件（timestamps配列の長さが0かつlastWarningが非設定）はcheckLoop関数とsaveState関数の両方で統一される。
状態ファイルの読み込み・パース処理に変更はなく、既存のloadState関数はそのまま動作する。
ファイルの書き込み処理はsaveState関数内でフィルタリングが追加されるのみで、書き込み方式自体は変更されない。

## 運用者向けインターフェース変更

### FIX-1: loop-detectorの動作変化

修正前はloop-detector-state.jsonが107KBの大きさで、複数回のワークフロー実行エントリを全て保持していた。
修正後はpruningメカニズムが動作し、古いタイムスタンプエントリの自動削除により、ファイルサイズが初回実行時に数KB〜数十KBに削減される。
MCPサーバーのファイル読み書き性能が向上し、フック実行時のタイムアウトエラーが解消される。
新しいワークフロー実行時に古いエントリが自動的に削除されるため、手動クリーンアップが不要になる。
pruning処理はフック実行のたびに自動的に行われ、運用者の介入は不要である。

### FIX-2: CLAUDE.mdの具体例追加による品質向上

修正前のCLAUDE.mdでは「同じ行が3回以上繰り返されないこと」という抽象的な要件のみが記載されていた。
修正後は重複行禁止の具体例、セクション密度要件の具体例、フェーズ別必須セクション一覧、禁止パターンの具体例の4項目が追加される。
subagent（Sonnet、Haikuモデル）がこれらの具体例を参照することで、成果物生成時のバリデーション成功率が向上する。
Orchestratorによる手動修正の回数が大幅に減少し、ワークフロー実行時間が短縮される。
成果物の品質が安定し、ドキュメント内容の信頼性が向上する。

### FIX-3: preExistingChanges処理のログ出力と設定ファイル除外

workflow_start時のpreExistingChanges記録にログ出力が追加され、記録状況が可視化される。
成功時には件数と先頭5件のファイルパスが出力される。
git diff失敗時には失敗理由とともに空配列が設定された旨が出力される。
EXCLUDE_PATTERNSに6パターンが追加され、設定ファイルの変更が自動的にスコープチェックから除外される。
手動でworkflow_set_scopeを実行する必要がなくなり、ワークフロー自動化レベルが向上する。
