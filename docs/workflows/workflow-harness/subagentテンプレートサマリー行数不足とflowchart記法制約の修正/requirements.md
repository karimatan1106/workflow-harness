## サマリー

本ドキュメントは、ワークフロー実行中に発生した3つの問題に対する要件定義をまとめたものである。
調査フェーズで根本原因が確定しているため、各問題に対して具体的な修正要件を記述する。
修正対象ファイルは `definitions.ts`（問題1・問題2）および `enforce-workflow.js`（問題3）の2ファイルに限定される。
問題1はサマリーテンプレートが例示する項目数（4行）とバリデーター要件（5行）の乖離であり、テンプレートへの項目追加で解消できる。
問題2はflowchartノード記法の使い分けに関するガイダンスが欠如しているため、definitions.ts のMermaid記法セクションに制約を追記する。
問題3はenforce-workflow.js の許可パターンにmemory/ディレクトリが含まれていないため、パターンを1件追加する。
これら3つの修正はいずれも既存の動作を破壊することなく、バリデーション安定性とhook動作の信頼性を向上させるものである。

## 機能要件

### 問題1: サマリーテンプレートの行数修正（REQ-1）

- definitions.ts 行1251付近のサマリーテンプレートに、現在の3項目に加えて最低2項目を追加し、例示行数を5行以上とすること
- 追加する項目は、subagentが自然に記述できる内容とする。具体的には「評価スコープ: 対象となるシステムや機能の範囲」と「検証状況: テスト実施の有無と結果の概要」の2項目が適切である
- 修正後のテンプレートを subagent に渡したとき、忠実に再現するだけで最低5行の実質行が生成されることを保証すること
- バリデーターの `minSectionLines: 5` の値自体は変更しないこと（テンプレート側を要件に合わせる）
- 修正はテンプレート文字列の追記のみで完結させ、バリデーションロジックの変更は不要とする

### 問題2: flowchartノード記法制約の追加（REQ-2）

- definitions.ts の Mermaid 図に関するガイダンス生成部分（行1179〜1182付近）に、flowchart ノード記法の制約を追記すること
- 追記する制約の内容: 「flowchartノードは `NodeID(text)` 形式を使うこと。`.mmd` ファイルは全行がバリデーター検出対象となるため `NodeID[text]` や `NodeID["text"]` の角括弧形式は使用禁止とする」
- 丸括弧形式 `NodeID(text)` が角丸四角形として正しく描画され、かつ角括弧検出を回避できる安全な代替であることを明記すること
- CLAUDE.md の「図式設計」セクションにある flowchart 例示も、角括弧形式から丸括弧形式に変更すること（例: `A[開始]` → `A(開始)`）
- この制約は `.mmd` ファイルを生成するフェーズ（state_machine、flowchart）のすべての subagent に伝達されること

### 問題3: MEMORY.mdのhookブロック回避（REQ-3）

- `enforce-workflow.js` の `WORKFLOW_CONFIG_PATTERNS` 配列に、memory/ ディレクトリへのアクセスを許可するパターンを追加すること
- 追加するパターンの対象パス: `C:\Users\owner\.claude\projects\` 以下の `memory\` ディレクトリ内のファイル全般
- パターンは Windows と Unix の両方のパス区切り文字（`\` と `/`）に対応した正規表現とすること
- タスク未開始（ゼロタスク）状態でも MEMORY.md の Write/Edit 操作が通過できるよう、チェック順序を考慮した実装とすること
- phase-edit-guard.js については、memory/ ディレクトリは `.md` 編集許可フェーズ（research, requirements 等）でアクセスされることが多いため、現状の許可ルールで問題は発生しにくい。ただし確実性を高めるため ALWAYS_ALLOWED_PATTERNS へのパターン追加も検討対象とする

## 非機能要件

### 後方互換性

- 既存のバリデーションルール（minSectionLines, minSectionDensity 等）の数値は変更しないこと
- 既存の flowchart バリデーションロジック（ノード・エッジ数の最低要件）は変更しないこと
- enforce-workflow.js の既存の WORKFLOW_CONFIG_PATTERNS パターンは削除しないこと
- 修正後のコードは既存のテストスイートをすべてパスすること

### 一貫性

- definitions.ts と CLAUDE.md の記述が矛盾しないよう、両ファイルを同一の修正内容で更新すること
- サマリーテンプレートの追加項目は、全フェーズのテンプレートに一律に反映されるよう、テンプレート生成ロジックの共通部分を修正すること
- enforce-workflow.js と phase-edit-guard.js の許可パターンは、memory/ ディレクトリへのアクセスに関して整合性を持たせること

### 修正の最小性

- 修正範囲は特定された根本原因に限定し、関連のない機能への波及を避けること
- definitions.ts の変更行数は最小限とし、テンプレート文字列の追記に留めること
- enforce-workflow.js の変更は WORKFLOW_CONFIG_PATTERNS への1パターン追加のみとすること
- TypeScriptの型定義や既存インターフェースの変更は不要とすること

### 検証可能性

- 問題1の修正後、サマリーセクションのみを持つ最小サイズのテスト成果物をバリデーターに渡したとき、minSectionLines エラーが発生しないことを確認できること
- 問題2の修正後、subagentが丸括弧形式を優先して使用するようになることをコードレビューで確認できること
- 問題3の修正後、タスク未開始状態で MEMORY.md への Write 操作が hook にブロックされないことを手動テストで確認できること
