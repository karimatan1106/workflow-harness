## サマリー

本ドキュメントは、ワークフロープラグインの3件の小規模修正（definitions.ts へのテンプレート追記、Mermaid記法ガイダンスの追記、enforce-workflow.js への許可パターン追加）に対する脅威モデリングの結果をまとめたものである。
修正対象はすべてプロンプトテンプレートまたはhookの設定値に相当するテキスト変更であり、新たなコードロジックや外部通信を伴わない低リスクな変更である。
各修正について、想定されるセキュリティリスク・想定外動作・後方互換性への影響を順に評価した。
評価の結果、いずれの修正も重大な脅威は検出されなかったが、問題3（MEMORY.md許可パターン追加）については過剰許可のリスクが軽微に存在するため、パターン範囲を最小限に絞ることを推奨する。
このドキュメントは planningフェーズおよびimplementationフェーズの実施判断に活用される。

## 脅威モデル

### 修正1: サマリーテンプレート行数追加（REQ-1）

修正内容はdefinitions.ts のテンプレート文字列に2項目（評価スコープ・検証状況）を追記するだけであり、バリデーションロジック自体に変更を加えない。
セキュリティ上のリスクとして考えられるのは、追加した項目が subagent に誤ったコンテキストを注入するケースであるが、今回の追記内容は一般的な文書構成項目であり注入攻撃の経路にはなり得ない。
想定外の動作として、追加項目が特定フェーズの成果物で意味をなさない（例: テスト実施の有無がresearch成果物に不要）ケースが考えられる。
ただし、サマリーセクションはすべてのフェーズで記述可能な汎用情報を列挙する設計であるため、過剰な内容にはならない。
後方互換性への影響はなく、既存成果物の再検証も不要である。

### 修正2: flowchart ノード記法制約の追加（REQ-2）

definitions.ts の Mermaid ガイダンス文字列に「NodeID(text) 形式を使うこと、角括弧形式は禁止」の1行を追記する変更である。
セキュリティ上のリスクは実質ゼロであり、プロンプトテキストの変更がシステムの認証・認可や外部通信に影響を与える経路は存在しない。
想定外の動作として、ガイダンスを受け取った subagent が丸括弧形式のみを使用するようになり、一部のMermaidダイアグラム（例: 菱形の条件分岐ノード `{}`）の記述に迷う可能性がある。
この場合は条件分岐にはflowchart 標準の `{text}` 形式を使い、通常ノードのみ丸括弧を適用すると明記すれば回避できる。
CLAUDE.md の図式例示も同時に更新することで記述の一貫性が保たれ、矛盾による誤解を防止できる。

### 修正3: WORKFLOW_CONFIG_PATTERNS へのパターン追加（REQ-3）

enforce-workflow.js の WORKFLOW_CONFIG_PATTERNS 配列に、memory/ ディレクトリへのアクセスを許可する正規表現を1件追加する変更である。
この変更の最大のリスクは、パターンが意図より広くなり、memory/ ディレクトリ以外のファイル（例: 任意のmemory文字列を含むパス）までホワイトリストに入ることである。
正規表現を `[/\\\\]memory[/\\\\]` のように区切り文字で囲む形式とすることで、ディレクトリ名の前後に必ずパス区切りを要求し、誤検知を抑制できる。
また、WORKFLOW_CONFIG_PATTERNS は「タスク未開始でも編集可能なワークフロー設定ファイル」を定義する配列であり、memory/ ディレクトリは自動記憶ファイル（MEMORY.md）の格納場所として適切なカテゴリに属する。
想定外の動作として、memory/ ディレクトリに悪意ある設定ファイルを配置してhookをバイパスする攻撃経路が生じる可能性がある点については、次のリスク評価セクションで詳述する。

## リスク評価

### REQ-1 のリスク評価

深刻度は低（Low）であると判定する。テンプレート追記はプロンプト生成への影響のみであり、システムの権限・通信・データ整合性に変化をもたらさない。
後退リスクとして、サマリーセクションが肥大化して subagent のコンテキスト使用量が増加する懸念があるが、追加は2項目のみであり影響は軽微である。
対策として追記後に既存テストスイートを実行し、全フェーズのサマリー生成が正常であることを確認する。

### REQ-2 のリスク評価

深刻度は極めて低（Very Low）であると判定する。Mermaid記法ガイダンスの追記はsubagentの出力品質向上を目的とした変更であり、セキュリティ影響は皆無である。
唯一のリスクは記法の誤解によって生成されるMermaid図が不正確になることであるが、これはバリデーターではなくレビュープロセスで検出・修正可能である。
CLAUDE.md との同期更新を実施することで、情報の一貫性が保たれ誤解を防止できる。

### REQ-3 のリスク評価

深刻度は低〜中（Low-Medium）であると判定する。許可パターンの追加はセキュリティ上の許可範囲を拡大するため、他の2件より慎重な評価が必要である。
具体的なリスクシナリオとして、悪意あるユーザーが `memory\malicious-config.json` のようなファイルをmemory/ ディレクトリに配置し、hookをバイパスした状態でワークフロー設定を改ざんする可能性がある。
ただし、このシステムはローカル開発環境での個人利用を前提としており、外部からのファイル配置を許可する経路（Webアップロード・ネットワークAPI等）が存在しないため、リスクは許容範囲内である。
推奨対策として、パターンは対象パスを `projects` ディレクトリ以下の `memory` ディレクトリに限定し、末尾に `.md` 拡張子の条件を加えることで許可範囲を最小限に絞ることを推奨する。
総合的な評価として、3件の修正はいずれもリスクが低く、提案された対策を実施することで安全に導入可能であると結論づける。
