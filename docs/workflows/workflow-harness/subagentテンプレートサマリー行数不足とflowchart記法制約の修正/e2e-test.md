## E2Eテストシナリオ

### シナリオ1: サマリーテンプレート修正の効果確認

このシナリオでは、`definitions.ts` の `importantSection` ブロック（行1251〜1252付近）で生成されるサマリーテンプレート文字列が5項目構成に拡張されていることを検証する。

検証対象: `C:\ツール\Workflow\workflow-plugin\mcp-server\src\phases\definitions.ts` の行1252の文字列リテラル。

検証の観点は2点ある。第1の観点として、5項目（目的・評価スコープ・主要な決定事項・検証状況・次フェーズで必要な情報）がすべてテンプレート文字列に含まれているかを確認する。第2の観点として、このテンプレートを忠実に再現したときの実質行数が5行以上となることを確認する。

artifact-validatorの `minSectionLines` は5に設定されており、サマリーセクションにはこの要件が適用される。修正前のテンプレートは3項目（実質4行）であったため、バリデーション失敗リスクが常に存在していた。

### シナリオ2: flowchart記法制約の効果確認

このシナリオでは、Mermaidガイダンスセクション（行1182直後）にflowchartノード記法制約が追加されていること、および両CLAUDE.mdの図式設計セクションのflowchart例示が丸括弧形式に変更されていることを検証する。

検証対象ファイルは3つある。`definitions.ts` の `qualitySection` への1行追加、ルートの `CLAUDE.md` の flowchart 例示コード、および `workflow-plugin/CLAUDE.md` の flowchart 例示コードである。

丸括弧形式（NodeID(text)）は角丸四角形としてMermaidでレンダリングされ、角括弧形式との視覚的区別が保たれる。この記法制約が伝達されることで、state_machineおよびflowchartフェーズのsubagentが安全な記法を選択できるようになる。

### シナリオ3: memory/ディレクトリアクセスの効果確認

このシナリオでは、`enforce-workflow.js` の `WORKFLOW_CONFIG_PATTERNS` 配列に memory/ ディレクトリ用パターンが追加されていること、および `isWorkflowConfigFile` チェックがゼロタスクブロックより前に実行されることを検証する。

検証対象: `C:\ツール\Workflow\workflow-plugin\hooks\enforce-workflow.js` の行219〜225（パターン配列）および行329・行365（実行順序）。

パターンは Windows と Unix 両方のパス区切り文字に対応した正規表現を使用することが要件となっている。ゼロタスクブロック（行365）より先に `isWorkflowConfigFile` チェック（行330）が実行されるため、memory/ ディレクトリへのアクセスはタスク未開始状態でも通過できる設計となっている。
