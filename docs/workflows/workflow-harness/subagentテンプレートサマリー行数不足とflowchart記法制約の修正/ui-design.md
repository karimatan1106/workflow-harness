## CLIインターフェース設計

このシステムは GUI を持たず、MCP サーバーが subagent へ渡すプロンプト文字列がユーザーインターフェースに相当する。
プロンプトの品質がそのまま subagent の成果物品質に直結するため、テンプレートの設計は高い優先度を持つ。

### サマリーテンプレートの形式設計（修正1対応）

subagent が受け取るプロンプト内のサマリー指示セクションは、以下の5項目構成に統一する。

- 項目1: 目的 — このドキュメントの目的を1〜2文で記述する
- 項目2: 評価スコープ — 対象となるシステム、ファイル、機能の範囲を列挙する
- 項目3: 主要な決定事項 — 重要な設計決定や技術選定の要点を記述する
- 項目4: 検証状況 — テスト実施の有無と結果の概要を記述する
- 項目5: 次フェーズで必要な情報 — 後続フェーズで必須となる情報を列挙する

この5項目は説明文行1行と合わせて合計6実質行をサマリーセクションに確保し、バリデーターの minSectionLines=5 を安定的に満たす。
各項目はコロンに続いてコンテンツが記述されるため、artifact-validator の実質行カウント対象として正しく認識される。

### flowchart プロンプトガイダンスの文言設計（修正2対応）

definitions.ts の Mermaid ガイダンスセクションに追記するフレーズは、subagent が読んだ瞬間に判断できる具体性を持たせる。

禁止形式の明示が必要な理由は、subagent が Mermaid の公式ドキュメントを参照すると角括弧形式が標準の矩形ノードとして紹介されているためである。
標準との差異を明示しない場合、Mermaid 仕様に従った subagent ほど角括弧形式を選択してしまうリスクがある。
したがって、「丸括弧形式を使うこと」という指示と「角括弧形式は禁止」という禁止の両方を同一行に含める設計とする。

禁止理由の説明文に「.mmd ファイルは全行がバリデーター検出対象となるため」という技術的根拠を付記することで、
subagent が代替案を誤用した場合に自己診断できるようにする。

## エラーメッセージ設計

artifact-validator が返すエラーメッセージの形式は既存の実装に従い、変更対象外とする。
本セクションでは各修正に関連するエラーメッセージの発生条件と期待される表示形式を設計する。

### サマリー行数不足エラー（修正1が解消する問題のエラー形式）

修正1実施前に発生していたエラーの発生条件は、サマリーセクションの実質行数が4行以下の場合である。
エラーメッセージの形式は以下のパターンに従う。

- 発生フェーズ: workflow_next または workflow_complete_sub 実行時のバリデーション
- エラー種別: セクション密度不足（sectionDensityError）またはセクション実質行数不足（sectionLinesError）
- メッセージ例: 「## サマリー セクションの実質行数が不足しています（検出: 4行、必要: 5行以上）」
- Orchestrator の対応: buildRetryPrompt を使用して「該当セクションに実質的な内容を追加してください」と指示するリトライプロンプトを生成する

修正1により、テンプレートを忠実に実装した subagent でも6実質行を確保できるため、このエラーの発生頻度が低下する。
ただしテンプレートを省略した subagent に対しては引き続きエラーが発生する可能性があるため、エラーメッセージ自体は変更しない。

### 角括弧プレースホルダー検出エラー（修正2が解消する問題のエラー形式）

flowchart ノードに角括弧形式を使用した場合に発生するエラーの発生条件と表示形式を以下に示す。

- 発生フェーズ: .mmd ファイルの生成後、workflow_complete_sub 実行時
- エラー種別: 角括弧プレースホルダー検出（squareBracketError）
- メッセージ例: 「コードフェンス外に角括弧プレースホルダーが検出されました（行番号: XX）」
- Orchestrator の対応: 成果物を直接修正せず、ノード記法を丸括弧形式に変更するよう指示するリトライプロンプトを生成する

修正2により、subagent がプロンプト内のガイダンスを参照した時点で丸括弧形式を選択するため、このエラーの発生頻度が低下する。

### memory/MEMORY.md アクセスブロックエラー（修正3が解消する問題のエラー形式）

enforce-workflow.js がタスク未開始状態で MEMORY.md への Write/Edit を拒否する場合のエラー形式を以下に示す。

- 発生条件: アクティブタスクが0件の状態で memory/ 配下のファイルへの書き込みが要求された場合
- フック応答: hook が exit code 2 を返し操作がブロックされる
- メッセージ例: 「ワークフローが開始されていません。コード変更の前に /workflow start を実行してください」
- 修正3実施後: isWorkflowConfigFile 判定でパターン一致し、ゼロタスクブロックを経由せずに許可される

## APIレスポンス設計

enforce-workflow.js の isWorkflowConfigFile 関数は、hook の中核となる許可/拒否分岐ロジックを担う。
本セクションでは修正3実施後の関数の動作設計を定義する。

### isWorkflowConfigFile 関数の返り値設計

この関数はファイルパス文字列を入力として受け取り、boolean を返すシンプルな判定関数である。
WORKFLOW_CONFIG_PATTERNS 配列の各パターンに対して Array.prototype.some を使用した順次照合を行い、
いずれかのパターンが一致した場合に true を返す設計になっている。

修正3実施後の返り値のパターンは以下のとおりである。

- memory/ パスが入力の場合: 新追加パターンが一致し、true を返す
- .claude/state/ 配下の JSON ファイルの場合: 既存パターンが一致し、true を返す
- ソースコードや設定ファイルの場合: いずれのパターンも一致せず、false を返す

isWorkflowConfigFile が true を返した場合、フックは後続のゼロタスクブロック判定をスキップし、操作を許可する。
この処理順序は現在の enforce-workflow.js の行329（isWorkflowConfigFile チェック）と行364（ゼロタスクブロック）の配置によって保証されている。

### パターンマッチの入力パス例と期待結果

修正3追加後のパターンに対する主要な入力パスと期待される返り値を以下に示す。

- 入力パス: C:\Users\owner\.claude\projects\C------Workflow\memory\MEMORY.md → 期待結果: true（新パターン一致）
- 入力パス: /home/user/.claude/projects/myproject/memory/MEMORY.md → 期待結果: true（Unix パス対応）
- 入力パス: .claude/state/workflows/task123/workflow-state.json → 期待結果: true（既存パターン一致）

- 入力パス: workflow-plugin/mcp-server/src/phases/definitions.ts → 期待結果: false（パターン不一致）

## 設定ファイル設計

WORKFLOW_CONFIG_PATTERNS 配列は enforce-workflow.js の設定定数として定義されており、
ワークフロー管理ファイルとして扱うべきパスパターンを RegExp オブジェクトの配列で管理している。

### 修正3実施後の WORKFLOW_CONFIG_PATTERNS 配列の設計

修正3実施後の配列は5要素で構成される。既存の4パターンは変更せず、末尾に新パターンを追加する。

要素1から4は既存パターンであり、workflow-state.json、task-index.json、settings.json、.mmd ファイル等をカバーしている。
要素5として追加する新パターンの設計根拠と詳細を以下に示す。

新パターンの要件は3点である。
第1の要件として、プロジェクト固有のパスをハードコードしない汎用性を持つこと。
第2の要件として、Windows のバックスラッシュと Unix のスラッシュの両方のパス区切り文字に対応すること。
第3の要件として、memory/ ディレクトリ直下の任意のファイルに対応し、ファイル名を限定しないこと。

新パターンの正規表現は以下の構造を持つ。

```javascript
/\.claude[\/\\]projects[\/\\][^\/\\]+[\/\\]memory[\/\\]/i
```

この正規表現の各部分の設計意図は以下のとおりである。
`.claude` の部分は .claude ディレクトリ名に固定し、ドットをエスケープして文字クラスと区別する。
`[\/\\]` の部分はスラッシュとバックスラッシュの両方を受け入れるパス区切りパターンであり、Windows/Unix 両対応を実現する。
`projects` の部分は Claude の projects サブディレクトリ名に固定する。
`[^\/\\]+` の部分はプロジェクト名に相当する任意の文字列を受け入れ、ディレクトリ階層の1段分を表す。
`memory[\/\\]` の末尾部分は memory ディレクトリの直下であることを示し、これ以降のパス（ファイル名）は限定しない。
末尾の `i` フラグは大文字小文字を区別しないマッチを有効にし、MEMORY.md および memory.md 等を同等に扱う。

### CLAUDE.md の flowchart 例示コードの変更設計

修正2b として、プロジェクトルートの CLAUDE.md に含まれる flowchart 例示コードを丸括弧形式に変更する。
変更対象は「図式設計」セクションの flowchart のコードブロック内の角括弧ノードである。

変更前の形式（角括弧を使用した矩形ノード）から、変更後の形式（丸括弧を使用した角丸ノード）への移行を示す。

変更後の設計意図として、CLAUDE.md の例示コードが実際の subagent プロンプトのガイダンスと整合することが重要である。
definitions.ts に追記した制約と CLAUDE.md の例示コードが一致することで、subagent がどちらを参照しても一貫した指示を受け取れる。
