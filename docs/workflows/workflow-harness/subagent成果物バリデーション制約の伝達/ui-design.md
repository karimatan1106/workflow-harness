## サマリー

本UI設計書は、subagent成果物バリデーション制約の伝達タスクにおけるインターフェース設計を定義します。
本タスクはCLI/フック処理の修正が中心であり、Webベースのユーザーインターフェースは含まれません。
設計対象は、bash-whitelist.jsのvalidateMkdirTarget関数が返すレスポンス形式、エラーメッセージのフォーマット、CLAUDE.md設定ファイルの構造です。
FIX-A（CLAUDE.md修正）では、subagent起動テンプレートに成果物品質要件セクションを追加し、既存のREQ-4セクション直後に配置します。
FIX-B（bash-whitelist.js修正）では、validateMkdirTarget関数の戻り値インターフェースを維持しつつ、内部処理でパス正規化を実施します。
エラーメッセージは既存の文言を変更せず、拒否理由として「許可されたディレクトリのみ作成可能です」という表現を維持します。
次フェーズであるtest_designでは、これらのインターフェース設計に基づいたテストケースを作成します。

## CLIインターフェース設計

### validateMkdirTarget関数の戻り値型

validateMkdirTarget関数は、mkdirコマンドの引数検証結果をJavaScriptオブジェクトとして返します。
戻り値の型は、allowed（真偽値）とreason（文字列、オプショナル）の2つのプロパティを持ちます。
許可する場合は「{ allowed: true }」という形式で、reasonプロパティは含まれません。
拒否する場合は「{ allowed: false, reason: "拒否理由のメッセージ" }」という形式で、必ずreasonプロパティを含みます。
この戻り値型は、bash-whitelist.jsの他の検証関数と統一されており、フック処理全体で一貫性を保っています。

### 関数シグネチャ

関数名はvalidateMkdirTargetであり、引数は1つのみです。
第1引数commandは文字列型であり、実行されるmkdirコマンド全体が渡されます。
コマンド例として「mkdir -p docs/workflows/タスク名」や「mkdir "/c/ツール/Workflow/docs/workflows/タスク名"」が含まれます。
関数は同期的に動作し、非同期処理を含みません。
返却値は即座に決定され、外部APIやファイルシステムアクセスは行われません。

### 入力パターンのバリエーション

相対パス形式の入力として「mkdir -p docs/workflows/タスク名」が渡される場合があります。
絶対パス形式（Windows標準）の入力として「mkdir "C:/ツール/Workflow/docs/workflows/タスク名"」が渡される場合があります。
絶対パス形式（MSYS環境）の入力として「mkdir "/c/ツール/Workflow/docs/workflows/タスク名"」が渡される場合があります。
オプション付きの入力として「mkdir -p -v docs/workflows/タスク名」が渡される場合があります。
複数ディレクトリ指定の入力として「mkdir docs/workflows/タスク1 docs/workflows/タスク2」が渡される場合があり、この場合は最初の引数のみを検証します。

### 処理フローの概要

関数はまずコマンド文字列から「mkdir」以降の引数部分を抽出します。
次に、オプションフラグ（ハイフン記号で始まる引数）をスキップし、最初のディレクトリパス引数を取得します。
取得したパスのトリム処理とクォート除去を行い、バックスラッシュをスラッシュに統一します。
FIX-Bで追加される正規化処理により、絶対パスからプロジェクトルート相対の相対パスを抽出します。
最後に、相対パスがallowedPrefixesリスト（「docs/workflows/」「docs/security/」「.claude/state/」）のいずれかで始まるかをチェックします。
パストラバーサル（「..」を含むパス）は、プレフィックスチェックの前に検出して拒否します。

## エラーメッセージ設計

### mkdir拒否時の標準メッセージ

許可されたプレフィックスで始まらないパスに対しては、以下のメッセージを返します。
「mkdir は許可されたディレクトリのみ作成可能です: docs/workflows/, docs/security/, .claude/state/」
このメッセージは、ユーザーに対して許可されたディレクトリの一覧を明示的に提示します。
メッセージ中のディレクトリリストは、allowedPrefixes配列の内容をカンマ区切りで連結したものです。
FIX-Bの実装後も、このメッセージ文言は変更されません。

### パストラバーサル検出時のメッセージ

パス中に「..」が含まれる場合、以下のメッセージを返します。
「mkdir での親ディレクトリ参照（..）は禁止されています」
このメッセージは、セキュリティ上の理由からパストラバーサルを明示的に拒否することを示します。
パストラバーサル検出は、正規化処理の後、プレフィックスチェックの前に実行されます。
相対パス形式でも絶対パス形式でも、「..」が含まれれば一律に拒否されます。

### エラーメッセージの出力先

validateMkdirTarget関数の戻り値であるreasonプロパティにエラーメッセージが格納されます。
このreasonプロパティは、呼び出し元のフック処理によってClaude Codeのターミナル出力に表示されます。
メッセージは日本語で記述され、英語訳は含まれません。
ユーザーはターミナルでエラーメッセージを確認し、許可されたディレクトリにパスを修正する必要があります。
エラー発生時、Bashコマンドの実行は中断され、ワークフローは現在のフェーズに留まります。

### メッセージの一貫性

bash-whitelist.jsの他の検証関数も同様のメッセージ形式を使用しています。
例えば、cpコマンド拒否時には「cp コマンドは禁止されています。Read tool + Write tool で代替してください。」というメッセージが返されます。
rmコマンド拒否時には「rm コマンドは現在のフェーズでは許可されていません: フェーズ名」というメッセージが返されます。
すべてのメッセージは「コマンド名 + 拒否理由 + 代替手段または許可条件」という構造を持ちます。
この構造により、ユーザーは拒否の理由と解決方法を迅速に理解できます。

### エラーメッセージのローカライゼーション

現在のエラーメッセージはすべて日本語で記述されています。
将来的に国際化（i18n）対応が必要になった場合、reasonプロパティの代わりにエラーコードを返す方式への移行を検討します。
エラーコード方式では、「MKDIR_NOT_ALLOWED_PREFIX」や「MKDIR_PATH_TRAVERSAL」のような識別子を返し、フロントエンド側で言語に応じたメッセージを表示します。
ただし、本タスクの範囲では国際化対応は実施せず、既存の日本語メッセージを維持します。
国際化対応は、別タスクとして計画フェーズから実施する必要があります。

## APIレスポンス設計

### レスポンス形式の定義

validateMkdirTarget関数は、JavaScriptの素のオブジェクト（Plain Object）を返します。
型定義として、「{ allowed: boolean, reason?: string }」というTypeScript風の表記で表現できます。
allowedプロパティは常に存在し、真偽値（trueまたはfalse）のいずれかを持ちます。
reasonプロパティは、allowedがfalseの場合のみ存在し、文字列型のエラーメッセージを格納します。
reasonプロパティは、allowedがtrueの場合には含まれず、undefinedでもnullでもなく、プロパティ自体が存在しません。

### 成功レスポンスのパターン

許可されたディレクトリに対するmkdirコマンドの場合、関数は「{ allowed: true }」を返します。
このレスポンスには、reasonプロパティは含まれません。
呼び出し元のフック処理は、allowedプロパティがtrueであることを確認し、Bashコマンドの実行を許可します。
成功レスポンスには、追加の情報（作成されるディレクトリパスの正規化後の値など）は含まれません。
検証成功の事実のみがレスポンスとして返されます。

### 失敗レスポンスのパターン（プレフィックス不一致）

許可されていないディレクトリに対するmkdirコマンドの場合、関数は以下のレスポンスを返します。
「{ allowed: false, reason: "mkdir は許可されたディレクトリのみ作成可能です: docs/workflows/, docs/security/, .claude/state/" }」
reasonプロパティには、エラーメッセージ設計セクションで定義した標準メッセージが格納されます。
呼び出し元のフック処理は、allowedプロパティがfalseであることを確認し、reasonプロパティの内容をユーザーに提示します。
Bashコマンドの実行は中断され、ワークフローの進行は停止します。

### 失敗レスポンスのパターン（パストラバーサル検出）

パストラバーサルを含むパスに対するmkdirコマンドの場合、関数は以下のレスポンスを返します。
「{ allowed: false, reason: "mkdir での親ディレクトリ参照（..）は禁止されています" }」
パストラバーサル検出時のメッセージは、プレフィックス不一致時のメッセージとは異なります。
この違いにより、ユーザーはセキュリティ上の制約によって拒否されたことを明確に理解できます。
パストラバーサルチェックは、プレフィックスチェックよりも優先され、先に評価されます。

### 既存インターフェースの維持

FIX-Bの実装により、validateMkdirTarget関数の内部処理は変更されますが、外部インターフェース（戻り値の型）は変更されません。
呼び出し元のコードは一切変更せずに、新しい実装を利用できます。
この後方互換性により、フック処理全体の安定性が保たれます。
テストコードも、既存のアサーションをそのまま使用できます。
関数の責務（mkdir引数の検証）は変わらず、処理方法のみが改善されます。

### レスポンスの拡張可能性

現在のレスポンス形式は、将来的に追加情報を含める拡張が可能です。
例えば、normalizedPathプロパティを追加して、正規化後のパスを返すことができます。
suggestedPathプロパティを追加して、拒否時に代替パスを提案することも可能です。
ただし、本タスクの範囲では、既存のインターフェースを変更せず、新しいプロパティも追加しません。
拡張が必要になった場合は、別タスクとして要件定義フェーズから実施します。

## 設定ファイル設計

### CLAUDE.mdの構造的配置

CLAUDE.mdは、プロジェクトルートに配置される単一のMarkdownファイルです。
ファイルサイズは約2000行であり、ワークフローのルール、フェーズ定義、subagent起動テンプレートが含まれます。
subagent起動テンプレートのセクションは、ファイル中盤（約1000行目付近）に配置されています。
テンプレート内には、既存の「★重要★ サマリーセクション必須化（REQ-4）」セクションが存在します。
FIX-Aでは、このREQ-4セクションの直後に、新しい「★重要★ 成果物品質要件（artifact-validator準拠）」セクションを挿入します。

### 追加セクションの階層構造

新しいセクションは、最上位見出し「## ★重要★ 成果物品質要件（artifact-validator準拠）」で始まります。
その下に、4つの中見出しレベルのセクションが配置されます。
第1中見出しは「### 1. 行数・密度要件」であり、各セクション内の最低行数とセクション密度の要件を説明します。
第2中見出しは「### 2. 重複行禁止（3回以上の同一行でエラー）」であり、同一行の繰り返し検出ルールを詳述します。
第3中見出しは「### 3. 必須セクション」であり、フェーズごとに要求されるMarkdownヘッダーをリスト化します。
第4中見出しは「### 4. 禁止パターン」であり、英語プレースホルダ4語、日本語暫定系8語、角括弧プレースホルダーの禁止ルールを列挙します。

### セクション内の箇条書き構造

各中見出しの下には、箇条書き（ハイフン記号で始まる行）でルールを記述します。
箇条書きは、ネストレベル1（トップレベル）とネストレベル2（インデント2スペース）の2階層を使用します。
トップレベルの箇条書きには、ルールの主題（例: 「実質行数の定義」）を記述します。
ネストレベル2の箇条書きには、具体的な詳細（例: 「空行、水平線、コードフェンスを除く」）を記述します。
箇条書きの後に、具体例として「悪い例」と「良い例」のペアを記述することで、理解を促進します。

### コードブロックとの統合

subagent起動テンプレート全体は、バッククォート3つで囲まれたコードブロック内に記述されています。
コードブロックの言語指定は行われていません（バッククォート3つのみ）。
新しいセクションも、このコードブロック内の一部として挿入されます。
コードブロックの開始記号と終了記号は変更せず、既存の構造を維持します。
Editツールを使用する際は、コードブロック全体を置換するのではなく、挿入対象の部分のみをold_stringとnew_stringに指定します。

### 参照元ファイルとの表現統一

workflow-plugin/CLAUDE.mdの259行目から293行目に、同等の成果物品質要件セクションが既に存在します。
FIX-Aでは、このセクションの内容を参考にして、表現、用語、構造を統一します。
統一項目として、中見出しの番号付け（「### 1.」「### 2.」形式）、箇条書きの記号（ハイフン記号）、悪い例・良い例の表記方法があります。
具体例の記述方法も統一し、「悪い例: ...」「良い例: ...」という形式で記述します。
両ファイルで同一の制約情報を持つことで、subagentが受け取る指示の一貫性を保ち、バリデーションエラーの発生率を低減します。

### セクション挿入位置の詳細

REQ-4セクションの最終行は、「これにより、次フェーズのsubagentがサマリーのみを読み込むことで効率的にコンテキストを引き継ぐことができます。」という文で終わります。
この文の後に空行が1行あり、その後に次のコードブロック要素（prompt文字列の終了やバッククォート3つ）が続きます。
新しいセクションは、REQ-4セクション最終行と空行の後、次のコードブロック要素の前に挿入します。
挿入時には、セクション間の空行を適切に配置し、視覚的な区切りを明確にします。
具体的には、REQ-4セクション最終行の後に空行2行、新しいセクション全体、空行1行、次のコードブロック要素という配置になります。

### 設定ファイルのバージョン管理

CLAUDE.mdはGit管理下にあり、変更履歴が追跡されます。
FIX-Aの実装により、CLAUDE.mdのコミット履歴に「subagent起動テンプレートに成果物品質要件セクションを追加」というコミットメッセージが記録されます。
変更差分（diff）には、挿入された新しいセクションの内容が表示されます。
この変更により、以降のワークフロー実行では、全てのsubagentが成果物品質要件を認識した状態で起動します。
バリデーションエラーの発生頻度が低減し、タスク完了までのフェーズ数が減少することが期待されます。

### 設定ファイルの読み込みタイミング

CLAUDE.mdは、Claude Codeのセッション開始時に一度だけ読み込まれます。
ワークフロー実行中にCLAUDE.mdが更新された場合、その変更は現在のセッションには反映されません。
変更を有効にするには、Claude Codeを再起動する必要があります。
FIX-Aの実装後、テストフェーズでClaude Codeを再起動してから、subagentの動作確認を行います。
再起動後、新しいワークフロータスクを開始し、成果物バリデーションが正しく機能することを検証します。
