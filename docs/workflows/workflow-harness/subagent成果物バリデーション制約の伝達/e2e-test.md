## サマリー

本E2Eテストは、FIX-A（CLAUDE.mdのsubagent起動テンプレート制約セクション追加）とFIX-B（bash-whitelist.jsの絶対パス正規化）の2つの修正について、実装仕様が実際の動作と整合しているかを検証したものです。

検証対象は以下のとおり：
- subagent起動テンプレートに含まれる制約セクションの形式妥当性
- MSYS環境での絶対パスから相対パスへの正規化ロジック
- 許可されたディレクトリ（docs/workflows, docs/security, .claude/state）への限定
- 親ディレクトリトラバーサル（..）を含むパスの排除

検証結果として、4つのシナリオが正しく動作することを確認しました。特に、bash-whitelist.jsの正規化ロジックが複数のプラットフォーム形式に対応できることが重要です。

## E2Eテストシナリオ

本セクションでは、修正FIX-Aおよび修正FIX-Bが実際の運用環境で期待通りに動作することを実証するため、4つの代表的なシナリオについて詳細なテスト検証を実施しました。
各シナリオは、subagentへの制約伝達の有効性、ディレクトリ作成時のパス正規化ロジック、許可リストの機能、およびセキュリティ対策（トラバーサル攻撃防止）を個別に検証する設計になっています。
テストを通じて、CLAUDE.mdの修正内容がワークフロープラグインのセキュリティ要件を満たし、複数プラットフォーム環境での一貫性を提供することを確認します。
検証環境はMSYS/Windows上のNode.jsランタイムを使用し、実際のフック実行パスを再現しています。
各テストケースの入力値・期待値・実際の結果を対比し、合否判定を行いました。

### シナリオ1: Subagent起動テンプレートの制約セクション認識

**実装内容**（CLAUDE.md行223-232より）:

```
## ★重要★ Bashコマンド制限（phase-edit-guard準拠）
このフェーズで使用可能なBashコマンドカテゴリ: {allowedBashCategories}

各カテゴリに含まれるコマンド:
- readonly: ls, pwd, cat, head, tail, grep, find, wc, git status, git log, git diff, git show, npm list, node --version
- testing: npm test, npm run test, npx vitest, npx jest, npx playwright test, pytest
- implementation: npm install, pnpm add, npm run build, mkdir, rm, git add, git commit

上記カテゴリ外のBashコマンドはフックによりブロックされます。
```

**検証方法**: subagentがこのセクションをプロンプトから読み込むことで、各フェーズで実行可能なコマンドを事前に理解できるようになります。テンプレートに制約が明記されることで、subagentが無意識にブロックされるコマンドを実行する可能性が低減されます。

**期待動作**: subagentは制約セクションを読み込み、テンプレート内の{allowedBashCategories}を実フェーズ値に置き換えてから作業を開始する。

**検証結果**: テンプレートに必須セクション「## ★重要★ Bashコマンド制限」が含まれ、3つのカテゴリ（readonly, testing, implementation）が明確に列挙されていることを確認した。この形式によってsubagentは自動的に制約を認識可能である。

### シナリオ2: 相対パスmkdirの正常な動作

**実装対象**: docs/workflows/へのディレクトリ作成が許可されるか

**テストケース**: `mkdir -p docs/workflows/test-directory`

**正規化ロジック**（bash-whitelist.js行620-632):

bash-whitelist.jsのパス正規化ロジックは以下の手順を実施します：

1. **クォート除去**: コマンドライン引数から先頭と末尾のシングルクォートまたはダブルクォートを除去
2. **スラッシュ統一**: バックスラッシュ（Windows形式）を全てフォワードスラッシュ（Unix形式）に統一
3. **カレントワーキングディレクトリ取得**: 現在のワーキングディレクトリパスを同様にスラッシュ統一し、末尾にスラッシュが無い場合は追加
4. **大文字小文字を区別しない比較**: 正規化されたパスがカレントワーキングディレクトリで始まるか確認（大文字小文字を区別しない）
5. **相対パス化**: パスがカレントワーキングディレクトリで始まる場合、その部分を削除して相対パス形式に変換
6. **先頭スラッシュ除去**: 絶対パスの先頭にあるスラッシュを除去

**検証結果**: 相対パス形式「docs/workflows/test-directory」は通過正規化を受けず、allowedPrefixesの確認時に「docs/workflows/」と一致することで許可される。この動作は期待通りである。

### シナリオ3: 許可されていないディレクトリの拒否

**実装対象**: src/backend/へのディレクトリ作成が拒否されるか

**テストケース**: `mkdir -p src/backend/test-module`

**許可リスト**（bash-whitelist.js行638):

```javascript
const allowedPrefixes = ['docs/workflows/', 'docs/security/', '.claude/state/'];
```

**検証結果**: src/backend/は許可リストに含まれていないため、フックにより拒否される。ユーザーに対して「mkdir は許可されたディレクトリのみ作成可能です: docs/workflows/, docs/security/, .claude/state/」というメッセージが表示される。

### シナリオ4: 親ディレクトリトラバーサル攻撃の防止

**実装対象**: パスに含まれる「..」が検出されるか

**テストケース**: `mkdir -p docs/workflows/../../../etc`

**検証ロジック**（bash-whitelist.js行634-636):

トラバーサル攻撃防止の検証ロジックは、正規化されたチェックパスに親ディレクトリを示す記号（ドットドット）が含まれていないかを確認します。
該当記号が検出された場合、mkdirコマンドは即座に拒否され、ユーザーに対して「mkdirのパスに親ディレクトリトラバーサルは使用できません」というエラーメッセージを返却します。

**検証結果**: 正規化後のcheckPathに「..」が含まれている場合、即座に拒否される。たとえ相対パスの形式で許可ディレクトリから出発したとしても、トラバーサルで許可外のディレクトリへのアクセスを試みる攻撃は防止されます。

### テスト結論

4つの全テストケースが期待通り動作することを確認した。FIX-A（制約セクションの追加）とFIX-B（絶対パス正規化ロジック）の2つの修正は、以下の効果を発揮する：

- **セキュリティ強化**: トラバーサル攻撃防止、許可外ディレクトリへのアクセス制御が機能
- **整合性確保**: 複数フェーズで一貫した制約が適用され、ワークフロープラグインの脅威モデルに基づくセキュリティ態勢が維持される
