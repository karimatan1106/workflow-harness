# 脅威モデル: ワークフロープロセス阻害要因完全解消

## サマリー

本脅威モデルは、ワークフロープロセス阻害要因の5件の修正（N-1〜N-5）に対するセキュリティリスクを評価し、対策を提案するものである。
最も深刻な脅威はN-1のgitコマンドオプション追加に伴うコマンドインジェクションリスクであり、静的ハードコード化により緩和可能である。
N-2のstderr出力追加は情報漏洩リスクを伴うが、内部パス構造の露出は限定的であり重大な影響は想定されない。
N-3のバリデーション緩和はテスト偽装による品質保証の劣化が懸念されるため、多段階バリデーションの維持が重要となる。
N-4のJavaScript拡張子追加は不正コード実行リスクを理論上増加させるが、既存のスコープ検証機構により実害は抑制される。
N-5のフェーズ制限緩和はスコープ拡大攻撃の新たな経路を提供するが、スコープサイズ制限とパストラバーサル対策の維持によりリスクは許容範囲内である。
全体として既存のセキュリティメカニズムを維持しつつ新たな防御層を追加することで、5件の修正を安全に実装可能と判断する。

## 脅威分析

### N-1: scope-validator日本語パス対応

#### 脅威T-1-1: コマンドインジェクション

**脅威の説明**:
`-c core.quotePath=false`オプションがgit diffコマンドに動的に追加される場合、攻撃者が悪意あるタスク名や環境変数を通じて追加のgitオプションやコマンドを注入する可能性がある。

**攻撃シナリオ**:
1. 攻撃者がタスク名に`; rm -rf /`のようなコマンドを含める
2. scope-validator.tsがこのタスク名を使用してgit diffのパス引数を構築
3. 不適切なサニタイゼーションにより、悪意あるコマンドが実行される

**影響度**: 高（システム全体の破壊が可能）
**発生確率**: 低（現行実装ではパス引数はgit diff出力のパース結果から取得され、タスク名は直接コマンドに渡されない）

**対策**:
- `-c core.quotePath=false`をハードコード化し、ユーザー入力から一切生成しない
- git diffのパス引数には既に正規化されたパスのみを使用（scope-validator.tsの既存ロジック維持）
- シェルエスケープ処理を行わず、Node.jsのchild_process.spawn配列形式でコマンド実行

#### 脅威T-1-2: パストラバーサル

**脅威の説明**:
`core.quotePath=false`により、gitが`../`を含む相対パスを生のUTF-8文字列として返す場合、スコープ検証ロジックが迂回される可能性がある。

**攻撃シナリオ**:
1. 攻撃者がスコープ外のファイル（例: `/etc/passwd`）へのシンボリックリンクを作成
2. git diffがリンクを追跡し、相対パス`../../../etc/passwd`を返す
3. スコープ検証がこのパスを正しく拒否できず、スコープ外変更を許可してしまう

**影響度**: 中（機密ファイルへのアクセス）
**発生確率**: 低（git diffは通常リポジトリ内のファイルのみを報告し、シンボリックリンクはリンク自体の変更として扱われる）

**対策**:
- 既存のパス正規化ロジック（バックスラッシュ→スラッシュ変換）を維持
- `.normalize()`を使用してパスを正規化し、`../`を解決
- スコープ検証前に、全パスがリポジトリルート配下であることを確認
- シンボリックリンクはリンク自体の変更として扱い、リンク先の検証は行わない

#### 脅威T-1-3: Gitバージョン依存の脆弱性

**脅威の説明**:
古いgitバージョンで`-c`オプションの処理に脆弱性が存在する場合、予期しない動作が発生する可能性がある。

**攻撃シナリオ**:
1. git 1.x等の古いバージョンを使用している環境で実行
2. `-c`オプションが不正に解釈され、設定ファイルの破損やクラッシュが発生
3. ワークフローが停止し、DoS状態に陥る

**影響度**: 低（可用性の低下のみ）
**発生確率**: 極低（git 2.0は2014年リリースで広く普及済み）

**対策**:
- 要件でgit 2.0以降を明示
- エラーハンドリングでgitコマンドの失敗を捕捉し、明確なエラーメッセージを表示
- 代替手段として`git config --global`による手動設定を案内

### N-2: phase-edit-guardのstderr出力完全化

#### 脅威T-2-1: 情報漏洩（内部パス構造）

**脅威の説明**:
stderr出力にエラーメッセージと共に内部ファイルパスやディレクトリ構造が含まれる場合、攻撃者がシステム構成を推測できる可能性がある。

**攻撃シナリオ**:
1. 攻撃者が意図的にフックをトリガーし、stderrメッセージを観察
2. メッセージに含まれる`.claude/state/`等の内部パス情報を収集
3. この情報を基に、システムの脆弱な箇所や重要ファイルの位置を特定

**影響度**: 低（パス情報のみでは直接的な攻撃は困難）
**発生確率**: 中（stderrは通常ユーザーに表示される）

**対策**:
- エラーメッセージには相対パスのみを含め、絶対パスを避ける
- 内部実装の詳細（例: `workflow-state.json`のHMAC検証失敗）は汎用メッセージに置き換える
- 既存のdisplayBlockMessage系関数のメッセージフォーマットを踏襲

#### 脅威T-2-2: ログ注入攻撃

**脅威の説明**:
ユーザー入力（ファイル名やコマンド）がstderrメッセージに含まれる場合、攻撃者が改行文字や制御文字を注入し、ログを偽装できる可能性がある。

**攻撃シナリオ**:
1. 攻撃者がファイル名に`\nERROR: System compromised\n`を含める
2. phase-edit-guardがこのファイル名をstderrメッセージに含める
3. ログ閲覧者が偽のエラーメッセージを信じ、誤った対応を取る

**影響度**: 低（混乱を招く程度）
**発生確率**: 低（ファイルシステムは通常改行文字を許可しない）

**対策**:
- stderrメッセージに含めるユーザー入力を全てサニタイズ
- 改行文字（`\n`, `\r`）をエスケープまたは削除
- 既存のdebugLogユーティリティを使用（既にサニタイズ機能を持つ）

#### 脅威T-2-3: DoS（大量エラーメッセージ）

**脅威の説明**:
攻撃者が大量のフックトリガーを発生させ、stderrに膨大なメッセージを出力させることで、ログシステムやディスク容量を圧迫する可能性がある。

**攻撃シナリオ**:
1. 攻撃者がスクリプトで数千個のファイル編集を連続実行
2. 各編集がphase-edit-guardをトリガーし、stderrメッセージを生成
3. ログファイルが肥大化し、ディスク容量を消費

**影響度**: 低（ローカル環境では影響限定的）
**発生確率**: 極低（git hookは各コミットで1回のみ実行され、大量トリガーは困難）

**対策**:
- メッセージサイズを制限（1メッセージ最大1KB程度）
- ログローテーション機能の利用を推奨（Claude Code側の責任）
- 特別な対策は不要（既存のconsole.error実装で十分）

### N-3: テスト出力真正性バリデーションの緩和

#### 脅威T-3-1: テスト偽装（False Positive）

**脅威の説明**:
MIN_OUTPUT_LENGTHの削減（200→100文字）とパターン追加により、非テスト出力がテスト出力として誤認識される可能性が増加する。

**攻撃シナリオ**:
1. 攻撃者が通常のログ出力を100文字以上作成し、"passed: 1, failed: 0, total: 1"を含める
2. workflow_record_test_resultがこれをテスト出力として受理
3. 実際のテストを実行せずにtestingフェーズを通過
4. 品質保証が機能せず、バグを含むコードがデプロイされる

**影響度**: 高（品質保証の劣化）
**発生確率**: 中（意図的な偽装は容易）

**対策**:
- TEST_OUTPUT_INDICATORSの複数パターンマッチングを維持（単一パターンでは不十分）
- タイムスタンプ整合性チェック（第4段階バリデーション）を維持
- 出力に`node`プロセス起動の証拠（`node xxx.test.js`）を要求
- exitCodeとテスト結果の整合性を検証（exitCode=0なのにfailed>0の場合は拒否）

#### 脅威T-3-2: 非テスト出力の誤認識

**脅威の説明**:
カスタムテストランナー向けパターン（"passed", "failed", "total"）が汎用的すぎるため、アプリケーションログやビルド出力を誤認識する可能性がある。

**攻撃シナリオ**:
1. ビルドスクリプトが"Build passed: 2 files, failed: 0"を出力
2. TEST_OUTPUT_INDICATORSの"passed"と"failed"にマッチ
3. TEST_FRAMEWORK_PATTERNSで数値を抽出
4. ビルド出力がテスト結果として記録される

**影響度**: 中（誤った記録によるレポート混乱）
**発生確率**: 中（ビルドツールは類似の出力形式を使用することが多い）

**対策**:
- TEST_OUTPUT_INDICATORSの複数マッチングを要求（例: "passed"と"total"の両方が必要）
- 出力の文脈を分析し、テストフレームワーク特有の語彙（"test", "spec", "suite"）の存在を確認
- vitest, jest等の既知フレームワークを優先し、カスタムパターンは補助的に使用
- 統合テストで非テスト出力の拒否を検証

#### 脅威T-3-3: バリデーション緩和による品質低下

**脅威の説明**:
バリデーション基準の緩和が意図せず過度に適用され、低品質なテスト出力が受理される可能性がある。

**攻撃シナリオ**:
1. 開発者が最小限のテスト出力（100文字ぎりぎり）を作成
2. テストケースの詳細やカバレッジ情報が欠落
3. バリデーションは通過するが、実質的な品質検証は行われていない
4. バグが本番環境に到達

**影響度**: 中（品質プロセスの形骸化）
**発生確率**: 中（開発者が意図せず簡素な出力を使用する可能性）

**対策**:
- MIN_OUTPUT_LENGTHを100文字に設定するが、推奨値は200文字以上とドキュメント化
- テストケース数の最小値を要求（total >= 1）
- タイムスタンプの範囲検証を強化（実行時刻から±5分以内）
- code_reviewフェーズでテスト出力の充実性を人間が確認

### N-4: enforce-workflowのJavaScript拡張子追加

#### 脅威T-4-1: 不正コード実行リスクの拡大

**脅威の説明**:
`.test.js`ファイルの作成許可により、攻撃者がトランスパイル不要な悪意あるJavaScriptコードを直接実行可能にする経路が増加する。

**攻撃シナリオ**:
1. 攻撃者がtest_implフェーズで`.test.js`ファイルを作成
2. ファイル内に`require('child_process').exec('malicious command')`を記述
3. testingフェーズでこのファイルが実行され、悪意あるコマンドが実行される
4. システムが侵害される

**影響度**: 高（任意コード実行）
**発生確率**: 低（攻撃者はすでに`.test.ts`で同等のコードを記述可能）

**対策**:
- スコープ検証を維持し、テストファイルが指定ディレクトリ（`src/backend/tests/`, `src/frontend/test/`）配下にのみ作成可能とする
- testingフェーズでのテスト実行は、サンドボックス化されたNode.js環境で行う（既存の動作）
- code_reviewフェーズで`.test.js`ファイルの内容を人間が確認
- `.test.`プレフィックスは業界標準であり、誤用は少ないと想定

#### 脅威T-4-2: 拡張子偽装による制限回避

**脅威の説明**:
攻撃者が非テストコードを`.test.js`として偽装し、テスト関連フェーズ以外でコード変更を試みる可能性がある。

**攻撃シナリオ**:
1. 攻撃者がimplementationフェーズで`.test.js`拡張子のファイルを編集しようとする
2. phase-edit-guardが拡張子のみを確認し、許可してしまう
3. テストではない本番コードがimplementationフェーズ外で変更される

**影響度**: 中（フェーズ制限の意味が薄れる）
**発生確率**: 低（PHASE_EXTENSIONSは各フェーズで許可する拡張子を制限しており、implementationでは`.test.js`は許可されない）

**対策**:
- PHASE_EXTENSIONSの定義を維持し、`.test.js`はtest_impl, testing, regression_test, e2e_testのみで許可
- implementationフェーズでは`.ts`, `.tsx`等のソースコード拡張子のみ許可（既存動作）
- スコープ検証により、ファイル配置ディレクトリも検証
- 現行設計で十分に対策済み

#### 脅威T-4-3: JSX拡張子の悪用

**脅威の説明**:
`.test.jsx`と`.spec.jsx`の追加により、Reactコンポーネントを含む複雑なコードがテストファイルとして実行される可能性がある。

**攻撃シナリオ**:
1. 攻撃者が`.test.jsx`ファイルにReactコンポーネントと悪意あるロジックを記述
2. testingフェーズでこのファイルがレンダリングされ、副作用が発生
3. ブラウザ環境でXSS等の攻撃が実行される

**影響度**: 中（フロントエンド環境への影響）
**発生確率**: 極低（テスト実行はNode.js環境で行われ、ブラウザレンダリングは発生しない）

**対策**:
- テスト実行環境をNode.js+JSDOMに限定（既存の動作）
- ブラウザ環境でのE2Eテストは別途管理（`e2e/`ディレクトリ）
- code_reviewフェーズでJSXファイルの内容を人間が確認
- 特別な追加対策は不要

### N-5: workflow_set_scopeのフェーズ制限緩和

#### 脅威T-5-1: スコープ拡大攻撃

**脅威の説明**:
docs_updateとregression_testフェーズでのスコープ変更許可により、攻撃者が後続フェーズで不正にスコープを拡大し、本来変更不可能なファイルを編集する可能性がある。

**攻撃シナリオ**:
1. 攻撃者がregression_testフェーズに到達
2. workflow_set_scopeを使用して`/etc/passwd`や`.git/config`をスコープに追加
3. 次のcommitフェーズでこれらのファイルを編集
4. システム設定が改ざんされる

**影響度**: 高（任意ファイル編集）
**発生確率**: 低（既存のパストラバーサル対策とスコープサイズ制限が機能）

**対策**:
- scope-validator.tsのパストラバーサル対策を維持（`../`を含むパスを拒否）
- スコープサイズ制限（100ファイル）を維持
- リポジトリルート外のファイルをスコープに追加できないよう検証
- `.git/`配下のファイルを明示的に禁止（既存の動作を確認）

#### 脅威T-5-2: ドキュメント改竄リスク

**脅威の説明**:
docs_updateフェーズでのスコープ追加により、攻撃者が重要なドキュメント（要件定義、設計書等）を事後的に改竄する可能性がある。

**攻撃シナリオ**:
1. implementationフェーズで攻撃者が意図的に仕様と異なるコードを実装
2. docs_updateフェーズに到達後、workflow_set_scopeで`docs/spec/features/`をスコープに追加
3. 既存の仕様書を改竄し、実装と整合させる
4. レビュー時に矛盾が検出されず、不正な実装が承認される

**影響度**: 中（品質プロセスの劣化）
**発生確率**: 低（git履歴で改竄が追跡可能、code_reviewで検出される）

**対策**:
- docs_updateフェーズの目的を明確化（新規ドキュメント作成のみ、既存ドキュメント更新は最小限）
- git履歴を活用し、ドキュメント変更のレビューを義務化
- 重要ドキュメント（requirements.md, spec.md等）の更新は設計フェーズに差し戻し
- スコープに追加可能なパスパターンを制限（例: `docs/operations/`, `README.md`のみ許可）

#### 脅威T-5-3: リグレッションテスト偽装

**脅威の説明**:
regression_testフェーズでのスコープ追加により、攻撃者が既存テストを削除・改竄し、リグレッション検出を回避する可能性がある。

**攻撃シナリオ**:
1. 攻撃者がimplementationで既存機能を破壊する変更を実施
2. regression_testフェーズでworkflow_set_scopeを使用し、既存テストファイルをスコープに追加
3. 破壊を検出するテストケースを削除または無効化
4. リグレッション検出が機能せず、破壊的変更がデプロイされる

**影響度**: 高（品質保証の破壊）
**発生確率**: 低（regression_testは既存テスト実行が主目的で、編集は想定外）

**対策**:
- regression_testフェーズの目的を明確化（新規リグレッションテスト作成のみ、既存テスト編集は禁止）
- スコープに追加可能なパスを`src/*/tests/regression/`配下の新規ファイルのみに制限
- 既存テストファイルの変更をphase-edit-guardで検出し、ブロック
- code_reviewフェーズで既存テストの削除・無効化を重点的に確認

### 高リスク脅威の詳細評価

#### T-1-1: コマンドインジェクション（N-1）

**現行実装での露出状況**:
- git diffコマンドは`child_process.execSync`で実行されており、シェル展開の可能性がある
- ただし、`-c core.quotePath=false`はハードコード化され、ユーザー入力から生成されない
- パス引数もgit diff出力のパース結果から取得され、タスク名は直接含まれない

**残存リスク**:
- 極めて低い。ハードコード化により、コマンドインジェクションのベクトルは実質的に存在しない

**推奨対策**:
- `child_process.spawn`配列形式への移行を検討（シェル経由を完全に回避）
- または現行の`execSync`実装を詳細レビューし、ハードコード化を確認

#### T-3-1: テスト偽装（N-3）

**現行実装での露出状況**:
- MIN_OUTPUT_LENGTH削減により、偽装が容易になる
- 新規パターン（"passed", "failed", "total"）は汎用的で、誤マッチの可能性が高い
- タイムスタンプ整合性チェックは維持されるが、偽装者は容易に適切なタイムスタンプを生成可能

**残存リスク**:
- 中程度。意図的な偽装は容易だが、検出メカニズム（複数パターンマッチング、exitCode整合性）が一定の防御を提供

**推奨対策**:
- TEST_OUTPUT_INDICATORSの複数マッチングを必須化（単一パターンでは不十分）
- exitCodeとテスト結果の整合性検証を強化
- 出力にテストフレームワーク名や`node`プロセス起動の証拠を要求
- code_reviewフェーズでテスト出力の妥当性を人間が確認

#### T-5-1: スコープ拡大攻撃（N-5）

**現行実装での露出状況**:
- docs_updateとregression_testフェーズでworkflow_set_scopeが許可される
- 既存のスコープサイズ制限（100ファイル）とパストラバーサル対策は維持される
- ただし、これらのフェーズは後続フェーズ（commit, push）に近く、影響範囲が大きい

**残存リスク**:
- 低〜中程度。既存の防御層が機能するが、新たな攻撃経路が開かれる

**推奨対策**:
- スコープ追加可能なパスパターンを制限（例: `docs/operations/`, `src/*/tests/regression/`のみ）
- `.git/`配下やシステムディレクトリを明示的に禁止
- code_reviewフェーズでスコープ変更履歴を確認
- 追加された2フェーズでのスコープ変更を監査ログに記録

#### T-5-3: リグレッションテスト偽装（N-5）

**現行実装での露出状況**:
- regression_testフェーズでworkflow_set_scopeが許可される
- 既存テストファイルのスコープ追加が可能となり、編集・削除のリスクが生じる
- phase-edit-guardは新規作成のみを想定しており、既存ファイル編集の検出が不十分

**残存リスク**:
- 中程度。既存テストの改竄により、品質保証が機能しなくなる可能性

**推奨対策**:
- regression_testフェーズでのスコープ追加を`src/*/tests/regression/`配下の新規ファイルのみに制限
- 既存テストファイル（`regression/`以外）の編集をphase-edit-guardで検出し、ブロック
- git diffでregression_test前後の既存テスト変更を検出し、警告
- code_reviewフェーズで既存テストの削除・無効化を重点的に確認

## 残存リスク

実装後も以下のリスクが残存する:

### 残存リスク1: ソーシャルエンジニアリング

**内容**: 攻撃者が開発者を騙し、意図的に低品質なテストやドキュメントを作成させる可能性
**対策**: code_reviewフェーズでの人間によるレビューを徹底
**受容理由**: 技術的対策の範囲外であり、運用プロセスで対応

### 残存リスク2: タイミング攻撃

**内容**: 攻撃者がテスト実行タイミングを操作し、タイムスタンプ検証を回避する可能性
**対策**: タイムスタンプ範囲を±5分に厳格化（M-3-4）
**受容理由**: 完全な防止は困難であり、exitCode整合性検証で補完

### 残存リスク3: gitフック無効化

**内容**: 攻撃者が`--no-verify`オプションでgitフックをバイパスする可能性
**対策**: Claude CodeのBashコマンド実行監視で`--no-verify`を検出（既存のCLAUDE.mdルール）
**受容理由**: gitの仕様上、完全な防止は不可能

### 残存リスク4: MCPサーバー改竄

**内容**: 攻撃者がMCPサーバーのコード（dist/*.js）を直接改竄する可能性
**対策**: ファイルシステムレベルのアクセス制御、改竄検知ツールの使用
**受容理由**: ローカル環境への物理アクセスが必要であり、脅威モデルのスコープ外
