# 要件定義書: ワークフロープロセス阻害要因完全解消

## サマリー

本要件定義では、ワークフロー実行中に発見された5件のプロセス阻害要因（N-1〜N-5）に対する修正仕様を規定する。
最優先はN-1のscope-validator日本語パス問題で、日本語タスク名を使う全ワークフローのcommitフェーズ遷移失敗を引き起こしている。
N-2はphase-edit-guardのstderr未出力問題で、4箇所のブロックパスにおいてユーザーへのエラー通知が欠落している。
N-3はtest-authenticity.tsのバリデーション基準緩和で、カスタムテストランナーの出力を受理可能にする必要がある。
N-4とN-5は軽微な改善事項で、それぞれJavaScriptテスト拡張子の追加とset-scopeフェーズ制限の緩和を目的としている。
次フェーズのplanningでは5ファイルの修正設計とテスト戦略を立案し、design_reviewで妥当性を検証する。

## 機能要件

### 要件N-1: scope-validator日本語パス対応

**要件ID**: N-1
**優先度**: 高（Critical）
**問題の説明**:

`workflow-plugin/mcp-server/src/validation/scope-validator.ts`の行453で実行されるgit diffコマンドが、gitデフォルト設定のcore.quotePath=trueにより非ASCIIパスをoctalエスケープする。
具体的には`docs/workflows/ワ-クフロ-/spec.md`が`"docs/workflows/\\343\\203\\257-\\343\\202\\257\\343\\203\\225\\343\\203\\255-/spec.md"`のように変換され、行463-484のスコープ比較ロジックでUnicode文字列との照合に失敗する。
結果として日本語タスク名を含むワークフローでは、commitフェーズへの遷移時にスコープ外変更エラーが発生し、ワークフローが停止する。
前回タスクでは`git config --global core.quotePath false`により手動で回避したが、新規環境では再発する構造的欠陥である。

**修正要件**:

- 行453のgit diff実行コマンドに`-c core.quotePath=false`オプションを追加すること
- オプション追加によりgit diffの出力が常にUTF-8の生のパス文字列となること
- 既存のパス正規化ロジック（バックスラッシュ→スラッシュ変換等）との互換性を維持すること
- ASCII専用パスを含む既存環境での動作に影響を与えないこと

**修正対象ファイル**:

- `workflow-plugin/mcp-server/src/validation/scope-validator.ts` (行453)

**受入基準**:

1. 日本語文字を含むタスク名（例: `ワ-クフロ-プロセス阻害要因完全解消`）を使用したワークフローで、commitフェーズへの遷移が成功すること
2. git diffの出力にoctalエスケープが含まれず、生のUTF-8文字列が返されること
3. スコープ比較ロジックが日本語パスを正しく認識し、スコープ内変更として判定すること
4. ASCII専用パスの環境でも既存の動作が維持され、レグレッションが発生しないこと
5. ユニットテストで日本語パスとASCIIパスの両方をカバーすること

### 要件N-2: phase-edit-guardのstderr出力完全化

**要件ID**: N-2
**優先度**: 中（High）
**問題の説明**:

`workflow-plugin/hooks/phase-edit-guard.js`の4箇所のブロックパスで、stderrへのエラーメッセージ出力が欠落している。
Bashホワイトリスト違反ブロック（行1595-1612）では`console.log`でstdoutに出力しているため、Claude Codeがstderrを参照する際にメッセージが見えない。
Fail Closedのcatchブロック（行1859-1862）では`process.exit(2)`を呼ぶのみで、ブロック原因の説明がユーザーに伝わらない。
stdinエラーハンドラ（行1880-1884）とJSONパースエラー（行1891-1894）も同様に、メッセージなしでブロックするため、ユーザーは「No stderr output」という汎用エラーしか見ることができない。
D-7修正でdisplayBlockMessage系関数をconsole.errorに変更したが、これらの独立した出力箇所は修正対象外だった。

**修正要件**:

- 行1595-1612のBashホワイトリスト違反出力を`console.log`から`console.error`に変更すること
- 行1859-1862のFail Closedキャッチブロックに`console.error`メッセージを追加すること（例: "Hook validation failed unexpectedly. Please check hook configuration."）
- 行1880-1884のstdinエラーハンドラに`console.error`メッセージを追加すること（例: "Failed to read input from stdin."）
- 行1891-1894のJSONパースエラーハンドラに`console.error`メッセージを追加すること（例: "Invalid JSON input from stdin."）

**修正対象ファイル**:

- `workflow-plugin/hooks/phase-edit-guard.js` (行1595-1612, 1859-1862, 1880-1884, 1891-1894)

**受入基準**:

1. Bashホワイトリスト違反でブロックされた際に、stderrに違反内容を示すメッセージが出力されること
2. Fail Closedパスでブロックされた際に、stderrに汎用エラーメッセージが出力されること
3. stdinエラーでブロックされた際に、stderrに入力エラーメッセージが出力されること
4. JSONパースエラーでブロックされた際に、stderrに解析エラーメッセージが出力されること
5. 全てのブロックパスで、Claude Codeがstderrからエラーメッセージを読み取れること

### 要件N-3: テスト出力真正性バリデーションの緩和

**要件ID**: N-3
**優先度**: 中（Medium）
**問題の説明**:

`workflow-plugin/mcp-server/src/validation/test-authenticity.ts`のvalidateTestAuthenticity関数が、テスト出力に対して200文字以上の最小文字数制限を課している。
さらにTEST_OUTPUT_INDICATORSの4パターン（"Tests:", "✓", "✗", "PASS"/"FAIL"/"ERROR"）にマッチしない出力はテスト出力として認識されない。
TEST_FRAMEWORK_PATTERNSの6パターンでテスト数の抽出も必須であり、Node.jsのassertモジュールを使用したカスタムテストランナーは対応が困難である。
具体的には`console.log('passed: 2, failed: 0, total: 2')`のような簡素な出力形式が、既存パターンとマッチせず拒否される。
前回タスクでは150文字以下のテスト出力が真正性バリデーション失敗により記録できず、回避策として冗長な出力を追加する必要があった。

**修正要件**:

- `MIN_OUTPUT_LENGTH`を200文字から100文字に削減すること
- `TEST_OUTPUT_INDICATORS`に以下のカスタムランナー向けパターンを追加すること:
  - `"passed"` (カスタムランナーの成功表記)
  - `"failed"` (カスタムランナーの失敗表記)
  - `"total"` (カスタムランナーの総数表記)
- `TEST_FRAMEWORK_PATTERNS`に以下の形式を追加すること:
  - `passed\s*:\s*(\d+)` (カスタムランナーのpassed表記)
  - `failed\s*:\s*(\d+)` (カスタムランナーのfailed表記)
  - `total\s*:\s*(\d+)` (カスタムランナーのtotal表記)

**修正対象ファイル**:

- `workflow-plugin/mcp-server/src/validation/test-authenticity.ts` (行17, 40-49, 51-60)

**受入基準**:

1. 100文字以上のテスト出力が真正性バリデーションを通過すること
2. `passed: N, failed: M, total: T`形式の出力がテスト出力として認識されること
3. Node.jsのassertモジュールを使用したカスタムテストランナーの出力が受理されること
4. 既存のvitest, jest, mocha等の標準フレームワーク出力も引き続き認識されること
5. バリデーションロジックが適切にテスト数を抽出できること

### 要件N-4: enforce-workflowのJavaScriptテスト拡張子追加

**要件ID**: N-4
**優先度**: 低（Low）
**問題の説明**:

`workflow-plugin/hooks/enforce-workflow.js`のPHASE_EXTENSIONS定義（行60-73）で、testing関連フェーズの許可拡張子リストに`.test.js`と`.spec.js`が含まれていない。
現在はTypeScript系テストファイル（`.test.ts`, `.test.tsx`, `.spec.ts`, `.spec.tsx`）のみが許可されており、Node.jsで直接実行可能なプレーンJavaScriptテストファイルの作成がブロックされる。
前回タスクでは`.test.ts`拡張子でTypeScript構文を含まないJavaScriptコードを記述して回避したが、ファイル拡張子と実際の内容が不一致となり好ましくない。
JavaScriptテストファイルはトランスパイル不要で実行でき、シンプルなテストケースや統合テストで有用性が高い。

**修正要件**:

- `PHASE_EXTENSIONS`のtest_design, test_impl, testing, regression_test, e2e_testの5フェーズに以下の拡張子を追加すること:
  - `.test.js`
  - `.spec.js`
  - `.test.jsx` (React等のJSXテストケース用)
  - `.spec.jsx` (React等のJSXテストケース用)
- 既存のTypeScript系拡張子との共存を維持すること
- isAllowedExtension関数の複合拡張子判定ロジックとの互換性を確保すること

**修正対象ファイル**:

- `workflow-plugin/hooks/enforce-workflow.js` (行60-73のPHASE_EXTENSIONS定義)

**受入基準**:

1. test_implフェーズで`.test.js`拡張子のファイル作成がブロックされないこと
2. testingフェーズで`.spec.js`拡張子のファイル編集がブロックされないこと
3. regression_testフェーズで`.test.jsx`拡張子のファイル作成がブロックされないこと
4. 既存のTypeScript系テストファイルも引き続き許可されること
5. 5つ全てのテスト関連フェーズで4つの新規拡張子が使用可能であること

### 要件N-5: workflow_set_scopeのフェーズ制限緩和

**要件ID**: N-5
**優先度**: 低（Low）
**問題の説明**:

`workflow-plugin/mcp-server/src/tools/set-scope.ts`の行25-32でALLOWED_PHASESがresearch, requirements, planning, implementation, refactoring, testingの6フェーズに限定されている。
docs_updateフェーズでスコープ不足に気づいた場合、`workflow_back`でtestingまで戻る必要がありフロー効率が悪い。
具体的には`docs/spec/features/`配下の新規仕様書作成や、README.mdの大幅更新など、事前にスコープ設定されていないファイルへの書き込みが発生しうる。
regression_testフェーズでもリグレッションテスト用の新規テストファイル作成が必要になるケースがあり、同様の制限に直面する。
ALLOWED_PHASESの制限自体はFR-6仕様に基づく安全性確保のための設計だが、実運用で不便が生じている。

**修正要件**:

- `ALLOWED_PHASES`配列に以下の2フェーズを追加すること:
  - `'docs_update'` (ドキュメント更新時のスコープ追加用)
  - `'regression_test'` (リグレッションテスト作成時のスコープ追加用)
- validatePhasePermission関数のエラーメッセージが新規フェーズに対応すること
- 既存のバリデーション（スコープサイズ制限、パストラバーサル対策等）は維持すること
- 他のフェーズでのスコープ変更を許可しないこと（セキュリティ維持）

**修正対象ファイル**:

- `workflow-plugin/mcp-server/src/tools/set-scope.ts` (行25-32のALLOWED_PHASES定義)

**受入基準**:

1. docs_updateフェーズで`workflow_set_scope`が正常に実行できること
2. regression_testフェーズで`workflow_set_scope`が正常に実行できること
3. 追加された2フェーズでもスコープサイズ制限が適用されること
4. 追加された2フェーズでもパストラバーサル対策が機能すること
5. 他のフェーズ（例: completed, commit等）ではスコープ変更が引き続き拒否されること

## 技術的制約

### 制約1: MCPサーバーモジュールキャッシング

MCPサーバー（Node.js）はロード済みモジュールをメモリにキャッシュするため、dist/*.jsファイルの変更は再起動なしでは反映されない。
修正完了後は必ずMCPサーバーを再起動（Claude Code再起動、または手動プロセスkill）する必要がある。
テスト実行時もこのキャッシュ動作を考慮し、各テストケース前にサーバーを再起動する手順を含める。

### 制約2: git hookの静的解析依存

phase-edit-guard.jsとenforce-workflow.jsはgit hookとして動作するため、実行時エラーが発生してもユーザーには「hook failed」としか表示されない。
N-2の修正ではstderr出力を確実に行うことで、この制約を緩和するが、根本的にはdebugLogを活用したトラブルシューティング手段も維持する。

### 制約3: TypeScriptコンパイル必須

N-1, N-3, N-5のソースコードはTypeScriptで記述されており、修正後はtscコンパイルを実行してdist/ディレクトリに出力する必要がある。
コンパイルエラーが発生した場合、MCPサーバーは起動できず、ワークフロー全体が機能停止する。
修正時は型定義の整合性を維持し、tsc --noEmitで事前検証を行う。

### 制約4: Hook実行環境の制限

git hookはNode.js環境で実行されるが、Claude Code実行環境のNode.jsバージョンに依存する。
N-2の修正ではES2020以降の機能（??演算子等）を使用しており、Node.js 14以降が必要である。
既存コードもES2020を使用しているため、この制約は新規に追加されるものではない。

## リスクと対策

### リスク1: git diffオプションの互換性問題（N-1）

**発生確率**: 低
**影響度**: 高
**内容**: 古いgitバージョンで`-c core.quotePath=false`が認識されない可能性
**対策**:
- git 2.0以降でのサポート確認（2014年リリース、広く普及済み）
- エラーハンドリングでgitバージョンチェックを追加
- 最悪の場合は従来の`git config`による手動設定を案内

### リスク2: stderr出力の副作用（N-2）

**発生確率**: 低
**影響度**: 低
**内容**: console.error出力がClaude Code以外のツール（CI等）でノイズとなる可能性
**対策**:
- 出力メッセージを簡潔かつ明確にする
- 既存のdisplayBlockMessage系と形式を統一
- CI環境でのstderr処理を確認（既存のエラー出力と同様に扱われる）

### リスク3: バリデーション緩和による偽陽性（N-3）

**発生確率**: 中
**影響度**: 中
**内容**: MIN_OUTPUT_LENGTH削減とパターン追加により、テスト以外の出力を誤認識する可能性
**対策**:
- TEST_OUTPUT_INDICATORSの複数パターンマッチングを維持（1パターンだけでは不十分）
- タイムスタンプ整合性チェック（第4段階）を維持
- 統合テストで非テスト出力（通常のログ等）が拒否されることを確認

### リスク4: JavaScript拡張子の過剰許可（N-4）

**発生確率**: 低
**影響度**: 低
**内容**: `.test.js`が非テストファイルにも使用され、フェーズ制限が機能しない可能性
**対策**:
- `.test.`と`.spec.`のプレフィックス規則は業界標準であり、誤用は少ない
- enforce-workflow.jsの他のバリデーション（スコープチェック等）は引き続き機能
- ドキュメントで`.test.js`の用途を明記

### リスク5: set-scopeフェーズ拡張による悪用（N-5）

**発生確率**: 低
**影響度**: 中
**内容**: docs_updateやregression_testフェーズで不必要なスコープ拡大が行われる可能性
**対策**:
- スコープサイズ制限（100ファイル）を維持
- パストラバーサル対策を維持
- 追加された2フェーズは元々ファイル作成が想定される性質のフェーズであり、悪用リスクは低い
