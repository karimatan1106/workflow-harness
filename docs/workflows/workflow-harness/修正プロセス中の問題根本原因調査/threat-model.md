## サマリー

- 目的: 修正プロセス中の問題根本原因調査タスクで実施される4種類の変更（テストコード修正・コードロジック修正・ドキュメント変更）に対してSTRIDE脅威分析を実施し、セキュリティリスクを評価する
- 主要な決定事項:
  - テストコード変更（FR-REQ-1）は機能的影響が限定的であり、脅威レベルは低い
  - フォールバック値修正（FR-REQ-4）はセキュリティバリデーションロジックに関連するため、最も慎重なレビューが必要である
  - ドキュメント変更（FR-REQ-2・FR-REQ-3・FR-REQ-5・FR-REQ-6）は技術的な攻撃面を持たないが、誤った情報の記録は将来の脆弱性につながる
  - 全変更において「情報の正確性」が最重要のセキュリティ特性である
- 次フェーズで必要な情報: 各修正対象ファイルのパス、修正後パターンの検証手順、ビルド確認コマンド

---

## 脅威シナリオ

### シナリオ1: テストモック不備によるテスト警告の誤認（FR-REQ-1関連）

**対象変更**: `tests/validation/design-validator.test.ts` への `mkdirSync: vi.fn()` 追加

**STRIDE分類**: Tampering（改ざん）、Information Disclosure（情報漏洩）

**脅威の説明**: `vi.mock('fs')` ブロックにモック設定が存在しない関数があると、テスト実行時にstderrへ警告が出力される。この警告が継続すると、将来の実際のエラーメッセージが警告ノイズに埋もれてしまい、本物の障害を見逃すリスクが発生する。また、モック対象の関数定義が不完全な状態では、テストスイートが意図しない副作用（実際のファイルシステム操作）を間接的に引き起こす可能性がある。

**攻撃経路**: 悪意ある実装者がモックの不備を利用して、`persistCache()` 経由でテスト環境のファイルシステムに意図しない書き込みを行う可能性がある。ただし、現行の `catch` ブロックがエラーを吸収するため、この経路は実際には機能しない。

**修正後のリスク**: `vi.fn()` の追加により `mkdirSync` が明示的にモックされ、実際のファイルシステムアクセスは完全に遮断される。警告ノイズが除去されることでテスト出力の信頼性が向上する。

---

### シナリオ2: フォールバック値のパターン不一致による誤バリデーション（FR-REQ-4関連）

**対象変更**: `definitions.ts` 34〜35行目の `GLOBAL_RULES_CACHE` フォールバック値更新

**STRIDE分類**: Tampering（改ざん）、Denial of Service（サービス拒否）

**脅威の説明**: フォールバック値の `bracketPlaceholderRegex` が旧形式（`/\[(?!関連|参考|注|例|出典)[^\]]{1,50}\]/g`）のまま残存していると、MCP サーバーの `artifact-validator.ts` インポートが失敗した起動時に誤ったパターンでバリデーションが実行される。その結果、正当なドキュメントがバリデーション失敗と誤判定される（False Positive）か、禁止パターンを含む不正なドキュメントが通過する（False Negative）かのどちらかが発生する。

**攻撃経路**: 攻撃者がMCPサーバーの起動シーケンスを操作して `artifact-validator.ts` のインポートを意図的に失敗させ、緩やかなフォールバックパターンを適用させることで、禁止パターン（`[#xxx#]`形式以外の括弧プレースホルダー）を含む成果物を不正に通過させる可能性がある。

**修正後のリスク**: フォールバック値を現行パターン（`/\[#[^\]]{0,50}#\]/g`）と完全一致させることで、正常系と異常系（フォールバック時）のバリデーション動作が統一される。`allowedKeywords` を空配列化することで旧許可キーワードによる意図しないバイパスを防止できる。

---

### シナリオ3: MEMORY.md 記録漏洩による知識ギャップ再発（FR-REQ-2・FR-REQ-3・FR-REQ-6関連）

**対象変更**: MEMORY.md への FR-22・FR-A〜FR-D 実装記録追記、および task-index.json 記述の実装済み更新

**STRIDE分類**: Information Disclosure（情報漏洩）、Repudiation（否認）

**脅威の説明**: 実装記録が MEMORY.md に記載されていない場合、将来のセッションで同じ問題が再発したときに根本原因の特定が困難になる。特に FR-22（スコープ情報プレースホルダー）と FR-A〜FR-D（角括弧プレースホルダーパターン修正）は複数ファイルにまたがる変更であり、記録がないと同様の誤りが繰り返される可能性が高い。また、task-index.json の既知バグ（FIX-1）が「対応が完了していない状態」として記録されたままだと、実装済みの問題を再修正しようとして、既存の実装を意図せず破壊するリスクがある。

**攻撃経路**: 技術的な攻撃面はないが、不正確な知識ベースが運用上の意思決定を誤らせる。特に「Root fix: MCP server should update task-index.json」という記述は、実装済みにもかかわらず対応が完了していないと誤認させ、不要な再実装を誘発する可能性がある。

**修正後のリスク**: 実装記録が適切に追記されることで、将来の開発者が正確な状態把握のもとで作業できる。ただし、記録内容の正確性（コミットハッシュ・変更ファイル名・変更内容）が誤っている場合、かえって誤情報が永続化するリスクがある。

---

### シナリオ4: 情報源の権威性が明示されていないことによる更新漏れの再発（FR-REQ-5関連）

**対象変更**: CLAUDE.md に `artifact-validator.ts` を唯一の権威情報源として明示する保守ルールの追記

**STRIDE分類**: Repudiation（否認）、Information Disclosure（情報漏洩）

**脅威の説明**: 角括弧プレースホルダーパターンが `artifact-validator.ts`・`definitions.ts`・CLAUDE.md の3箇所に重複して存在し、権威情報源が明示されていない状態では、一箇所を変更した際に他箇所の更新が漏れる。過去のFR-A〜FR-Dで実際にこの問題が発生しており、`definitions.ts` のフォールバック値が古いパターンのまま残存した。保守ルールが文書化されていないと、修正者が更新対象の全ファイルを把握できず、同様の不整合が繰り返される。

**攻撃経路**: 攻撃者がCLAUDE.mdの古いパターン記述を「正しい定義」として参照させることで、新規実装者が誤ったパターン（旧形式の角括弧プレースホルダー）でバリデーションを実装するよう誘導できる可能性がある。権威情報源が明示されていない状況ではソーシャルエンジニアリング的な手法で情報の混乱を引き起こしやすい。

**修正後のリスク**: 保守ルールが明文化されることで、将来の変更時に更新すべきファイル（`definitions.ts` フォールバック値・テンプレート文字列・CLAUDE.md 本文）が明確になる。ただし保守ルール自体が古くなった場合（`artifact-validator.ts` のリファクタリング等）に更新されないリスクが残る。

## リスク評価

### FR-REQ-1: テストモック不備（mkdirSync 追加）

- **可能性（Likelihood）**: 低い — 現行の `catch` ブロックがエラーを吸収しているため実際の被害は発生しない
- **影響度（Impact）**: 低い — テスト警告ノイズの除去が主な効果であり、機能的変化はない
- **総合リスクレベル**: Low（レビューで検出可能）
- **対策状況**: FR-REQ-1 の修正後、`vi.fn()` によって完全にモック化されるため残存リスクは実質ゼロになる
- **残存リスク**: `vi.fn()` が呼び出されたことを検証するアサーションがないため、`persistCache()` の呼び出し有無をテストが検知できない点は依然として限定的な不透明性として残る

### FR-REQ-4: フォールバック値のパターン不整合

- **可能性（Likelihood）**: 低い — `artifact-validator.ts` のインポート失敗は通常発生しない
- **影響度（Impact）**: 高い — バリデーション動作の不整合はセキュリティバリデーション全体の信頼性を損なう
- **総合リスクレベル**: Medium（HMAC保護ありの実装に影響）
- **対策状況**: フォールバック値を現行パターンと同期させることで、起動失敗時のセキュリティデグレードを防止する
- **残存リスク**: フォールバック値の同期は手動作業であり、将来の `artifact-validator.ts` 変更時に再び不整合が生じる可能性がある。FR-REQ-5 の保守ルール追記と組み合わせることで軽減される

### FR-REQ-2・FR-REQ-3: MEMORY.md 記録漏洩

- **可能性（Likelihood）**: 中程度 — 記録漏洩は人的ミスとして繰り返し発生している
- **影響度（Impact）**: 中程度 — 将来の調査コストが増大し、同様の問題が再発するリスクがある
- **総合リスクレベル**: Low（削除検出可能）
- **対策状況**: 記録の追記により知識ギャップが解消される
- **残存リスク**: ドキュメント内容の正確性は執筆者の注意力に依存するため、誤った記録が永続化するリスクがある

### FR-REQ-5: 情報源権威性の明示が欠如している状態

- **可能性（Likelihood）**: 高い — 保守ルールがないと更新漏れは高確率で繰り返される
- **影響度（Impact）**: 中程度 — 複数ファイルの不整合が蓄積し、バリデーション動作の信頼性が低下する
- **総合リスクレベル**: Medium（定期的なレビューで低減可能）
- **対策状況**: 保守ルールの明文化により更新義務を周知し、将来の変更時の対応漏れを防止する
- **残存リスク**: ルール自体の陳腐化を防ぐための定期レビュープロセスが存在しない

### FR-REQ-6: task-index.json 誤記録の継続

- **可能性（Likelihood）**: 中程度 — 誤記録がそのまま参照され続ける可能性がある
- **影響度（Impact）**: 低い — 実装済みの問題を再調査する無駄コストが発生するが、システムへの直接的な影響はない
- **総合リスクレベル**: Low（参照時に実コードで確認可能）
- **対策状況**: セクション見出しと記述内容の両方を「実装済み」に更新することで誤解を防止する
- **残存リスク**: 実装の詳細（ファイルパスや行番号）が将来のリファクタリングで変化した場合、記録が再び不正確になるリスクがある

## セキュリティ要件

### SEC-1: バリデーションパターンの一貫性確保

フォールバック値（`definitions.ts`）・テンプレート文字列（各フェーズのプロンプト）・CLAUDE.md の説明文が全て `artifact-validator.ts` の現行パターン（`/\[#[^\]]{0,50}#\]/g`）と整合していること。整合性の確認は `artifact-validator.ts` の `bracketPlaceholderPattern` を参照元として行うこと。変更作業後に `npm run build` を実行してTypeScript型チェックを通過することを確認すること。

### SEC-2: テスト環境のファイルシステム分離

テストスイートが実際のファイルシステムに書き込みを行わないこと。`vi.mock('fs')` ブロックが全ての `fs` モジュール関数を明示的にモック定義していること。特に `mkdirSync`・`writeFileSync`・`unlinkSync` のような書き込み系関数は `vi.fn()` によって無操作にモック化されていること。テスト実行後にテスト環境外のファイルが変更されていないことを確認できること。

### SEC-3: 知識ベースの正確性維持

MEMORY.md への記録追記は、コミットハッシュ・変更対象ファイル・変更内容の3点を含む形式で行うこと。記録内容は実際のコード変更（`git log` や `git show` で確認可能な事実）に基づくこと。推測や曖昧な表現は避け、検証可能な情報のみを記載すること。誤った実装状態の記録（「実装済み」と「対応が完了していない状態」の取り違え）は後続の開発判断に直接影響するため、修正前に実コードで現状を確認してから記録すること。

### SEC-4: 権威情報源の明示と更新義務の文書化

角括弧プレースホルダーパターンの変更は必ず `artifact-validator.ts` への変更を起点として行うこと。下流ファイル（`definitions.ts`・テンプレート文字列・CLAUDE.md）は `artifact-validator.ts` の変更後に整合性を確認して更新すること。CLAUDE.md の保守ルールセクションには、更新が必要なファイル一覧を列挙し、省略なく記述すること。保守ルールへの参照を容易にするため、該当セクションに固定ヘッダー名を付与して参照しやすい形式で記述すること。

### SEC-5: 変更の最小性と副作用の回避

各修正は要件で定義された対象箇所のみを変更すること。FR-REQ-1 では `vi.mock` ブロック内の1行追加のみを行い、テストロジックの変更は行わないこと。FR-REQ-4 では `GLOBAL_RULES_CACHE` の `catch` ブロック内の特定プロパティ（`bracketPlaceholderRegex`・`bracketPlaceholderInfo.pattern`・`bracketPlaceholderInfo.allowedKeywords`）のみを変更し、`duplicateLineThreshold`・`minSectionDensity` 等の他プロパティは変更しないこと。修正後は全テスト（912件以上）が引き続き合格することを確認すること。
