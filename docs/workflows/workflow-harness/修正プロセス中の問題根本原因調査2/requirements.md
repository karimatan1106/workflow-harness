## サマリー

- 目的: 前回ワークフロー（FR-REQ-1〜FR-REQ-6）の修正過程で発生した5件の問題（バリデーション失敗・重複行エラー・禁止語検出・sessionToken未返却）について要件を定義し、受け入れ基準を確定する。
- 対象問題: 問題A（manual_test重複行）・問題B（performance_test行数不足）・問題C（e2e_test禁止語+行数不足）・問題D（sessionToken未返却）・問題E（FR-REQ-1/FR-REQ-4修正済み確認）
- 修正方針の概要: 問題Dはstatus.tsのコードバグとしてCLAUDE.md説明補強で対処。問題A/B/Cはdefinitions.tsのテンプレートガイダンス改善で対処。問題EはFR-REQ-1/FR-REQ-4が153587aで既修正済みであることを確認し、要件内で記録する。
- 主要な決定事項: テンプレート改善はLLMの確率的失敗を「低減」するものであり完全排除を目標としない。コードバグ（問題D）は設計上の問題として確実に修正し、受け入れ基準を明確化する。
- 次フェーズで必要な情報: 修正対象ファイルはdefinitions.ts（問題A/B/C）とCLAUDE.md（問題D）。問題E（FR-REQ-1/FR-REQ-4）は既に修正済みのため実装フェーズの修正対象外。

## 機能要件

### FR-A: manual_testテンプレートの実行環境行一意化ガイダンス強化

**目的:** 複数テストシナリオにわたって実行環境情報を記述する際に、同一テキストの実行環境行が3回以上出現して重複行エラーが発生する問題を防止する。

**修正対象ファイル:** `workflow-plugin/mcp-server/src/phases/definitions.ts` の manual_test subagentTemplate（行900付近）

**要件内容:**
- 現行のFR-1ガイダンス（推奨パターン1/2）に「MT-1」「MT-2」形式の具体例を追加すること。
- 推奨パターンの表記を manual_test のシナリオ命名規約（MT-N形式）に合わせて更新すること。
- 「全シナリオで実行環境が同一である場合もシナリオ番号または確認対象名を行末に付加すること」という補完的ガイダンスを追記すること。
- ガイダンスの文言が英語例示に偏らず、日本語のシナリオ記述においても自然に適用できる形式とすること。

**受け入れ基準（Acceptance Criteria）:**
- 修正後のテンプレートが、MT-1〜MT-4の4シナリオそれぞれで異なる実行環境行を生成するよう誘導できること。
- 既存のFR-1ガイダンスの意図（シナリオ番号付加による一意化）を損なわず、例示を拡充する形で変更すること。
- definitions.tsに対するユニットテスト（有効なテンプレートフォーマット確認）がパスすること。
- 変更後にnpm run buildでビルドエラーが発生しないこと。

### FR-B: performance_testテンプレートへの任意セクション行数要件注意書き追加

**目的:** performance_testのsubagentが任意セクション（requiredSections未定義のセクション）を追加した際に、そのセクションにもminSectionLines（5行）が適用されることを意識させ、セクション行数不足エラーを防止する。

**修正対象ファイル:** `workflow-plugin/mcp-server/src/phases/definitions.ts` の performance_test subagentTemplate（行930付近）

**要件内容:**
- 「任意で追加した ## セクション（requiredSectionsに含まれないもの）にも、artifact-validatorのminSectionLines（5行）が適用される」という警告を追記すること。
- 任意セクションを追加する際のガイドとして、最低5行の実質行を確保するための具体的な記述例を含めること。
- 既存の必須セクション（パフォーマンス計測結果・ボトルネック分析）向けのガイダンスと矛盾しない位置・形式で追加すること。

**受け入れ基準（Acceptance Criteria）:**
- 修正後のテンプレートで任意セクションを追加した場合に5行以上の記述が促進されること。
- テンプレートの警告文がコードフェンス外に記述され、subagentが参照できる形式であること。
- 変更後にnpm run buildでビルドエラーが発生しないこと。
- 既存の912件（修正前時点）以上のユニットテストがパスすること。

### FR-C1: e2e_testテンプレートへの禁止語部分一致ガイダンス追加

**目的:** 「未定義」「未定形」等、禁止語「未定」を部分一致で内包する技術用語をe2e_testの成果物に記述した際にバリデーション失敗が発生する問題を防止する。

**修正対象ファイル:** `workflow-plugin/mcp-server/src/phases/definitions.ts` の e2e_test subagentTemplate（行942付近）

**要件内容:**
- 現行の「禁止語転記防止（重要）」セクションに、「未定義」という技術用語が禁止語「未定」の部分一致で検出されることへの具体的な警告を追加すること。
- 「未定義」の安全な言い換えパターン（例：「型が確定していない状態」「参照先が設定されていない変数」「モックが登録されていない状態」）を少なくとも2例示すること。
- 警告文はコードフェンス外に記述し、subagentが参照しやすいセクションに配置すること。

**受け入れ基準（Acceptance Criteria）:**
- 修正後のテンプレートに「未定義」という語句を成果物本文に書いてはならない旨の警告が含まれること。
- 「未定」を内包する語の言い換え例が少なくとも2つ提示されていること。
- 変更後にnpm run buildでビルドエラーが発生しないこと。

### FR-C2: e2e_testテンプレートへの総合評価セクション記述指針追加

**目的:** e2e_testの「## 総合評価」セクションが4行しかなくminSectionLines（5行）に達しなかった問題（問題Cの行数不足）を防止する。manual_testのFR-11に相当するガイダンスをe2e_testにも適用する。

**修正対象ファイル:** `workflow-plugin/mcp-server/src/phases/definitions.ts` の e2e_test subagentTemplate（行942付近）

**要件内容:**
- 「## 総合評価」セクションに関する記述指針を追加すること。
- 指針には全シナリオの合否サマリー・検出された問題の一覧・未実施シナリオがある場合の理由・次フェーズへの引き継ぎ事項・全体的な品質評価の5観点を含めること。
- 各観点を個別の行として記述することで5行以上の実質行を自然に確保できるよう誘導すること。
- manual_testのFR-11と同等の構造でガイダンスを記述し、フェーズ間の一貫性を保つこと。

**受け入れ基準（Acceptance Criteria）:**
- 修正後のテンプレートに「## 総合評価」セクション専用の記述指針が含まれること。
- 指針が5観点以上の記述内容を示しており、5行以上の実質行確保を促進できること。
- manual_testのFR-11との構造的な整合性が保たれていること。
- 変更後にnpm run buildでビルドエラーが発生しないこと。

### FR-D: CLAUDE.mdへのworkflow_status taskId指定必須の明記

**目的:** セッション再開後にOrchestratrorがworkflow_statusをtaskIdなしで呼び出した場合にsessionTokenが返されないという設計上の問題を、ドキュメントの補強によって解消する。

**背景:** status.tsの実装では、taskIdを指定してタスク詳細を取得するモードでのみsessionTokenを返す。taskIdなしの全タスク一覧モードではsessionTokenフィールドが存在しない。CLAUDE.mdのAIへの厳命23番目に「workflow_statusでsessionTokenを再取得すること」とあるが、taskId指定が必須であることが明記されていない。

**修正対象ファイル:** `C:\ツール\Workflow\CLAUDE.md`

**要件内容:**
- AIへの厳命23番目の記述を更新し、「workflow_status({ taskId: '<タスクID>' })のようにtaskIdを明示指定して呼び出すこと」という説明を追加すること。
- taskIdなしの呼び出し（全タスク一覧モード）ではsessionTokenが返されないことを明記すること。
- taskIdの取得方法（workflow_listを先に呼び出してtaskIdを確認する）を付記すること。

**受け入れ基準（Acceptance Criteria）:**
- 修正後のCLAUDE.mdのAIへの厳命23番目に、taskId明示指定の必要性が明記されていること。
- 「taskIdなし呼び出しではsessionTokenが返されない」という警告が含まれること。
- taskId取得のための補助手順（workflow_list参照）が記述されていること。
- 修正後の文言が既存の厳命リストと整合性を保っていること。

## 非機能要件

### NFR-1: テストカバレッジの維持

全ての修正（FR-A〜FR-E）適用後に既存のユニットテスト数が修正前と同数以上であること。
テスト実行コマンド `npm run test` がエラーなく完了すること。
リグレッションテストで今回の修正起因のテスト失敗が発生しないこと。
新たに追加するガイダンスやコード変更に対応するテストケースがある場合は追加すること。

### NFR-2: ビルドの成功

修正後の `npm run build` がエラーなく完了すること。
TypeScriptの型エラーが発生しないこと。
definitions.tsのテンプレート文字列変更によるバックスラッシュエスケープ等の構文エラーがないこと。
修正内容がdist/フォルダへ正しくトランスパイルされること。

### NFR-3: 変更の最小性

各問題（FR-A〜FR-D）の修正は該当するテンプレートまたはドキュメントのセクションに限定すること。
無関係なフェーズのテンプレートや設定ファイルを変更しないこと。
テンプレート改善はガイダンスの追記に限定し、既存のガイダンス構造を破壊しないこと。
CLAUDE.mdの変更はAIへの厳命23番目の記述更新に限定し、他のセクションを変更しないこと。

### NFR-4: 横展開漏れ防止

同種の問題が他フェーズに存在しないかを実装前に確認すること。
問題Cの「e2e_testへのFR-11相当ガイダンス追加」のように、あるフェーズへの改善が他フェーズにも適用可能な場合は横断的に適用すること。
ただし過剰な横展開（効果が低い改善の全フェーズ適用）は避け、同種の問題が実際に発生したフェーズに限定して適用すること。

### NFR-5: 修正プロセスの文書化

各修正に対してFRナンバー（FR-A〜FR-D）を付与して追跡可能にすること。
修正完了後のコミットメッセージにはFRナンバーを含めること。
同種の乖離（2ファイル間の定義を独立して変更するリスク）を防ぐためのガイドラインをCLAUDE.mdまたは開発ドキュメントに記録すること。
artifact-validator.tsのFORBIDDEN_PATTERNSが変更された場合はCLAUDE.mdの禁止パターンリストも同時更新するという運用ルールが既存の保守ルールとして記載されており、今後も遵守すること。

## 優先度と実施順序

本タスクで対処する問題は以下の優先度で実施する。

1. 最優先（コードバグ・ドキュメント補強）: FR-D（CLAUDE.mdへのtaskId必須明記）
2. 高優先（テンプレート改善・複合問題）: FR-C1 + FR-C2（e2e_testテンプレートの禁止語と行数不足）
3. 中優先（テンプレート改善・単一問題）: FR-A（manual_testテンプレート例示更新）
4. 中優先（テンプレート改善・単一問題）: FR-B（performance_testテンプレート任意セクション警告）
5. 確認のみ（修正済み）: FR-E（FR-REQ-1・FR-REQ-4の修正済み確認）

## 制約事項

修正作業はimplementationフェーズで行われるものとし、requirementsフェーズでは要件の定義のみを行う。
テンプレート改善はLLMの確率的失敗を低減するものであり、失敗率をゼロにすることを目標としない。
status.tsのコード修正は今回の対象外とする。問題Dの対処はCLAUDE.mdのドキュメント補強のみとする。
MCP サーバーの再起動を必要とする変更（コアモジュール変更）は最小限に抑え、テンプレート文字列変更のみの場合はビルド後に再起動が必要であることを認識すること。
