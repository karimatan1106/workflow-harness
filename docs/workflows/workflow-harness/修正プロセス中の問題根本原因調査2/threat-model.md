## サマリー

- 目的: FR-A〜FR-D の修正（definitions.ts テンプレート改善と CLAUDE.md ドキュメント補強）に対するセキュリティ脅威モデルを作成し、実装フェーズへのリスク情報を提供する。
- 評価スコープ: workflow-plugin/mcp-server/src/phases/definitions.ts の4箇所のテンプレート文字列変更と CLAUDE.md の AIへの厳命23番更新が評価対象であり、コアロジック（artifact-validator.ts・state-manager.ts）は変更対象外である。
- 主要な決定事項: テンプレート文字列変更はロジック変更を伴わないため、インジェクション・認証バイパス等の高リスク脅威は該当しない。主要リスクはバックスラッシュエスケープ構文エラーと HMAC 整合性への間接的影響の 2 点に限定される。
- 検証状況: 脅威分析はドキュメントベースの静的分析であり、4件の脅威シナリオ（TS-1〜TS-4）を識別した。実際のコード変更後のビルド検証は implementation フェーズで実施する。
- 次フェーズで必要な情報: implementation フェーズでは Node.js テンプレートリテラル内のバックスラッシュエスケープが正しいか必ず検証すること。CLAUDE.md 変更後は Orchestrator の動作確認（taskId 指定あり・なし双方）を実施すること。

---

## 脅威シナリオ

### TS-1: definitions.ts テンプレート文字列のバックスラッシュエスケープ構文エラー

**脅威の概要:** FR-A・FR-B・FR-C1・FR-C2 の修正はいずれも definitions.ts の subagentTemplate 文字列にガイダンスを追記する変更である。TypeScript の通常文字列リテラルおよびテンプレートリテラルでは、バックスラッシュ（`\n`、`\t` 等）が特殊文字として解釈されるため、誤ったエスケープが構文エラーまたは実行時の予期しない挙動を引き起こす。

**発生条件の詳細:**
- ガイダンス追記時に正規表現パターン（`\s+`、`\S` 等）を文字列リテラル内に記述した場合、バックスラッシュが二重にエスケープ（`\\s+`）されるべきところを単一（`\s+`）のままにする誤り。
- テンプレートリテラルの変数展開記法（`${variableName}`）を誤ってガイダンス本文に含めた場合、実行時にエラーになる。

**影響範囲:** 影響が発生すると `npm run build` のトランスパイルが失敗し、MCPサーバーが起動不能になる。全フェーズのワークフロー機能が停止するため、severity は High と評価する。

**対策:**
- 修正後に `cd workflow-plugin/mcp-server && npm run build` を実行してビルド成功を確認すること。
- テンプレート文字列に正規表現パターンを記述する場合はコードフェンス内に限定し、散文テキストでは自然言語で説明すること。

### TS-2: HMAC 整合性の間接的破損リスク

**脅威の概要:** state-manager.ts が管理する workflow-state.json には HMAC-SHA256 による整合性チェックが設定されている。definitions.ts は直接 workflow-state.json を操作しないが、definitions.ts に構文エラーが含まれた場合、MCPサーバーが起動時にクラッシュし、workflow-state.json の HMAC が更新されないまま状態が中断される可能性がある。

**発生条件の詳細:**
- TS-1 の構文エラーが発生してサーバークラッシュが起きた状態で、強制的に workflow-state.json を手動編集した場合、HMAC 不一致が発生する。
- HMAC 不一致の状態では phase-edit-guard フックが全 Bash コマンドをブロックし、修正作業そのものができなくなるデッドロック状態になる。

**影響範囲:** デッドロックが発生した場合の復旧は task-index.json の手動修正や phase を build_check 等に設定する作業が必要となり、開発者の工数を大幅に消費する。severity は Medium と評価する。

**対策:**
- TS-1 の対策（ビルド確認）を先行実施することでこのリスクも連動して低減される。
- 万一サーバークラッシュが発生した場合は workflow-state.json を直接編集せず、MCP サーバーを修正してから再起動すること。

### TS-3: Orchestrator の sessionToken 取得失敗による後段フェーズのブロック

**脅威の概要:** FR-D は CLAUDE.md の厳命 23 番目に taskId 明示指定を追記する変更である。修正前の記述では taskId なしの全タスク一覧モードを使用した Orchestrator が sessionToken を受け取れず、workflow_record_test_result の呼び出しが失敗する。これにより testing フェーズ以降の regression_test への遷移がブロックされる。

**発生条件の詳細:**
- 新規セッション再開時に Orchestrator が `workflow_status()` を taskId なしで呼び出すと、返却データに sessionToken フィールドが含まれない（status.ts の実装が taskId 指定時のみ sessionToken を返す設計）。
- sessionToken なしで workflow_record_test_result を呼び出しても testing フェーズが正常完了できないため、regression_test への遷移条件が満たされない。

**影響範囲:** testing フェーズ以降が全て進行不能になる。影響対象は全タスクに及ぶため severity は High と評価する。

**対策:**
- CLAUDE.md の修正によって Orchestrator に taskId 指定の必要性を明記し、誤った呼び出しパターンを防止する。
- workflow_list を先に呼び出して taskId を確認する補助手順を CLAUDE.md に追記することで、再現性のある正しい手順を確立する。

### TS-4: バリデーターと CLAUDE.md ドキュメントの乖離リスク

**脅威の概要:** artifact-validator.ts の FORBIDDEN_PATTERNS リストが変更された場合に CLAUDE.md の禁止パターンリストを同時更新しない場合、Orchestrator が誤った禁止語情報に基づいてサブエージェントに指示を出し、バリデーション失敗が繰り返される。FR-D の修正はこのような乖離を防ぐ運用ルールの参照先を明確化する意味でも重要である。

**発生条件の詳細:**
- 将来の修正で FORBIDDEN_PATTERNS に語句が追加された場合に、CLAUDE.md の「禁止パターン（完全リスト）」を更新し忘れる。
- サブエージェントが CLAUDE.md を参照して禁止語を判定するため、CLAUDE.md が古いリストを示している場合、新たに追加された禁止語を使用した成果物が生成される。

**影響範囲:** バリデーション失敗によるリトライが増加し、タスク完了に要する時間が増大する。severity は Low と評価する（バリデーターが機能しているため実害は軽微）。

**対策:**
- 既存の保守ルール（artifact-validator.ts 変更時は CLAUDE.md も同時更新）を引き続き遵守する。
- 今回の FR-D 修正はこのルールの対象外（status.ts の動作変更であり FORBIDDEN_PATTERNS は変更しない）。

## リスク評価

### リスクマトリクス

| ID | 脅威名 | 発生可能性 | 影響度 | 総合リスク | 優先度 |
|----|--------|------------|--------|-----------|--------|
| TS-1 | テンプレート文字列エスケープ構文エラー | 中（修正箇所が多いため） | 高（サーバー停止） | High | 1位 |
| TS-2 | HMAC 整合性の間接的破損 | 低（TS-1 が前提条件） | 中（デッドロック） | Medium | 2位 |
| TS-3 | sessionToken 取得失敗によるフェーズブロック | 高（現状の既知バグ） | 高（testing 以降停止） | High | 1位（並列） |
| TS-4 | バリデーターと CLAUDE.md の乖離 | 低（運用ルールで防止） | 低（リトライ増加のみ） | Low | 3位 |

### TS-1 の発生可能性評価

FR-A・FR-B・FR-C1・FR-C2 の計 4 箇所でテンプレート文字列を編集するため、各編集機会にバックスラッシュエスケープ誤りが発生する可能性がある。1 箇所あたりの誤り率は低くとも 4 箇所合計では中程度のリスクとなる。特に FR-C1 では正規表現パターンの言及を含む可能性があるため注意が必要である。

### TS-3 の発生可能性評価

TS-3 は現在進行形の既知バグであり、CLAUDE.md の修正がなければ新規セッション再開のたびに再発する。FR-D の修正（CLAUDE.md への taskId 必須明記）が実施されることで発生可能性が大幅に低下する。修正後は確認テストを実施することを推奨する。

## セキュリティ要件

### SR-1: ビルド検証の必須化（TS-1・TS-2 への対策）

definitions.ts のテンプレート文字列を変更した後は、変更ごとに `npm run build` を実行してトランスパイルエラーがないことを確認すること。全 4 箇所（FR-A・FR-B・FR-C1・FR-C2）の修正完了後に再度ビルドを実行して最終確認すること。ビルドエラーが発生した場合はテンプレート文字列を修正してからビルドを再実行し、成功を確認してから次の修正箇所に進むこと。

この要件は NFR-2（ビルドの成功）と整合しており、実装フェーズの完了条件として位置付ける。

### SR-2: テンプレート文字列への技術記号記述の制限（TS-1 への対策）

subagentTemplate 文字列に正規表現パターン（バックスラッシュを含む文字列）を記述する場合は、必ずコードフェンス（バッククォート 3 つ）で囲んだ中に記述すること。ガイダンスの本文（散文テキスト・箇条書き）では自然言語による説明に置き換えること。この制限により TS-1 の発生可能性を大幅に低減できる。

FR-C1 でe2e_test テンプレートに禁止語の技術的説明を追記する際は特にこの制限を意識し、「正規表現パターン」という表現を使う場合はその正規表現自体をコードフェンス外に記述しないこと。

### SR-3: CLAUDE.md 変更後の動作確認（TS-3 への対策）

CLAUDE.md の厳命 23 番目を修正した後は、以下の 2 パターンで動作確認すること。
確認手順 1: `workflow_status({ taskId: '<実際のID>' })` を呼び出し、レスポンスに sessionToken フィールドが含まれることを確認する。
確認手順 2: `workflow_status()` を taskId なしで呼び出し、レスポンスに sessionToken フィールドが含まれないことを確認する。
これにより SR-3 の修正が正しく機能していることを検証できる。

### SR-4: 変更最小性の維持（TS-4・NFR-3 への対策）

今回の修正対象外のテンプレートや設定値を変更しないこと。特に artifact-validator.ts の FORBIDDEN_PATTERNS および state-manager.ts の HMAC 計算ロジックは変更対象外である。これらのコアモジュールを変更する場合は MCP サーバーの再起動が必要となり、追加リスクが発生するため、テンプレート文字列の追記のみで目的を達成できる本タスクではコアモジュール変更を避けること。

FORBIDDEN_PATTERNS を変更しない限り、CLAUDE.md の禁止パターンリストの更新は不要であり、TS-4 のリスクは発生しない。
