## テスト方針

今回の変更対象はdefinitions.tsのテンプレート文字列3箇所（manual_test・performance_test・e2e_test）とCLAUDE.mdの1箇所のテキスト追記である。バリデーターのロジック変更は含まれず、TypeScriptの型定義やフェーズ遷移ロジックにも影響しない。このため以下の方針でテストを実施する。

テスト優先順位の考え方として、最も重要な確認事項はビルドが通ること（TypeScriptコンパイルエラーがないこと）と、既存テストが全てパスすること（リグレッションがないこと）の2点である。テンプレート文字列の変更は文字列の連結・追記のみであるため、TypeScriptのコンパイルエラーが発生するリスクは低いが、シングルクォートのエスケープや改行記号の誤りが混入した場合にビルドが失敗する可能性がある。

手動検証の位置付けとして、テンプレート文字列の内容（FR-A〜FR-Cの追記内容）が仕様書（spec.md）の記述と一致しているかどうかは、自動テストではなく目視確認で行う。CLAUDE.md の厳命23番の更新内容（FR-D）も同様に手動検証である。これらは内容の正確性を確認するためのものであり、バリデーターの動作確認ではない。

FR-Eの扱いとして、commit 153587aの修正はFR-REQ-1（design-validator.test.tsのfsモック追加）とFR-REQ-4（definitions.tsのbracketPlaceholderRegexフォールバック修正）であり、既存テストスイートがこれらを検証している。vitestで全テストが通過することをもってFR-Eの受け入れ基準とする。

テスト実行環境は`workflow-plugin/mcp-server`ディレクトリを作業基点とし、`npm run build`と`npm test`（`vitest run`）の2コマンドのみで検証する。

## テストケース

### TC-1: TypeScriptビルド検証

- 目的: definitions.tsへのテンプレート文字列追記変更が、TypeScriptコンパイルエラーを引き起こしていないことを確認する。特にシングルクォート文字列内のエスケープ誤りや、テンプレートリテラルの混入がないことを検証する。
- 手順: `cd workflow-plugin/mcp-server && npm run build` を実行する。ビルドが完了するまで待機し、標準出力とエラー出力を確認する。
- 期待結果: ビルドコマンドの終了コードが0であり、標準エラー出力にTypeScriptのエラーメッセージ（error TS接頭辞）が含まれていないこと。`dist/`ディレクトリ内の対応するJavaScriptファイルが更新日時として実行時刻と一致していることを確認する。
- 異常系: ビルドが失敗した場合は、エラーメッセージで指定された行番号周辺のdefinitions.tsを確認し、シングルクォートのエスケープ、改行の`\n`記述、テンプレートリテラル記号の混入を確認して修正する。

### TC-2: 既存ユニットテスト全通過確認（リグレッション検証）

- 目的: FR-A〜FR-Dの変更が既存テストスイートに影響を与えていないことを確認する。これにより、FR-Eの修正済み確認（commit 153587aの検証）も同時に達成する。
- 手順: TC-1のビルドが成功した後に`cd workflow-plugin/mcp-server && npm test`を実行する。`vitest run`が全テストスイートを順次実行するため、完了まで待機する。
- 期待結果: テスト件数が変更前と同等（950件前後）であり、失敗件数が0件であること。テスト出力の最終行に`Tests X passed`と表示されること。design-validator.test.tsのfsモック関連テストが通過していること、bracketPlaceholder関連のテストが通過していることを個別に確認する。
- 異常系: テストが失敗した場合は、失敗テストのファイル名と行番号を確認し、今回の変更（definitions.tsのテンプレート文字列追記）との因果関係を分析する。definitions.tsの変更はテンプレート文字列の追記のみであるため、テスト失敗の原因としてありうるのは、getPhaseDefinitionやbuildRetryPromptなどのテンプレート生成関数の内部ロジックに関連するテストが、テンプレート内容の変化に敏感に反応している場合に限られる。

### TC-3: manual_testテンプレートのFR-1ガイダンス拡充確認（手動）

- 目的: FR-Aの変更として、manual_testのsubagentTemplateにMT-N形式の具体例（2行）と、全シナリオ同一環境時の補完ガイダンス（1文）が追記されていることを確認する。
- 手順: definitions.tsのmanual_test定義箇所（FR-1ガイダンスセクション周辺）をReadツールで読み込み、以下の3点が存在するかを目視確認する。
  - 推奨パターン（MT-1形式）として「MT-1の実行日時:」を含む例示行が存在すること。
  - 推奨パターン（MT-2形式）として「MT-2の実行環境:」を含む例示行が存在すること。
  - 「全シナリオで実行環境が同一である場合」から始まる補完ガイダンス文が存在すること。
- 期待結果: 上記3点が全てテンプレート文字列内に存在する。既存のFR-1ガイダンス（「## 実行日時・環境情報行の一意化」セクション）の記述内容が削除・変更されていないこと。
- 異常系: 追記内容が見当たらない場合は、spec.mdの「FR-A: manual_testテンプレートのFR-1ガイダンス更新」セクションに従い再実装する。

### TC-4: performance_testテンプレートの任意セクション警告確認（手動）

## テスト実行順序と前提条件

テスト実行前の前提条件として、implementationフェーズでFR-A〜FR-Dの変更がdefinitions.tsとCLAUDE.mdに適用済みであることが必要である。未適用の状態でテストを実行しても全テストケースが失敗するため、必ず実装完了後にテストを開始すること。

推奨実行順序として、まずTC-1（ビルド確認）を実行してTypeScriptのコンパイルエラーがないことを確認する。次にTC-2（vitestユニットテスト）を実行して既存テストが全件通過することを確認する。これら2件の自動確認が通過した後に、TC-3からTC-7の手動確認を順に実施する。

テスト実行ディレクトリの明示として、TC-1とTC-2のBashコマンドはいずれも`workflow-plugin/mcp-server`を作業ディレクトリとして実行する。プロジェクトルートから実行する場合は`cd workflow-plugin/mcp-server && npm run build`と`cd workflow-plugin/mcp-server && npm test`の形式を使用すること。

## 合否判定基準

全7件のテストケースについて以下の基準で合否を判定する。TC-1とTC-2は自動実行であり、終了コードが0であることをもって合格とする。TC-3からTC-7は手動確認であり、確認対象の記述が存在していることを確認者が判断した場合に合格とする。

テスト全件合格を条件として、implementationフェーズからrefactoringフェーズへの遷移を許可する。TC-1またはTC-2が失敗した場合はimplementationフェーズに戻り、対応する問題を修正した後に再度テストを実行する。TC-3からTC-7のいずれかが不合格の場合は、該当するFR番号の実装内容を修正してから手動確認を再度実施する。
