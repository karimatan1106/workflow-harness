## 概要

本仕様書はrequirements.mdで定義されたFR-A〜FR-Eの5件の修正に対する実装詳細を記述する。

### 問題の背景

前回ワークフローの parallel_verification フェーズでsubagentが生成した成果物において、以下の問題が発生した。

問題A（manual_test重複行）: 複数のテストシナリオで実行環境行が同一内容で3回以上出現したため、artifact-validatorの重複行検出でエラーとなった。現行の FR-1 ガイダンスにはMT-N形式の具体例が含まれておらず、全シナリオで実行環境が同一の場合の対処指針が不足していた。

問題B（performance_test行数不足）: subagentが任意セクションを追加したが、そのセクションの実質行数が5行未満だったため minSectionLines 違反が発生した。任意セクションにもminSectionLines（5行）が適用されるという説明がテンプレートに欠落していた。

問題C（e2e_test禁止語・行数不足）: 「型が確定していない状態」などの意味で「参照先が設定されていない変数」相当の表現を使うべき場面で禁止語の部分一致が発生し、また「## 総合評価」セクションの実質行が4行にとどまり minSectionLines（5行）違反も同時に発生した。

問題D（sessionToken未返却）: セッション再開後にOrchestratrorがworkflow_statusをtaskIdなしで呼び出した場合、status.tsの実装上sessionTokenが返されないが、CLAUDE.mdの厳命23番にはtaskId指定が必須である旨の記載がなかった。

問題E（FR-REQ-1/FR-REQ-4修正済み確認）: commit 153587aで両者が修正済みであることをgit showで確認した。実装フェーズの追加変更は不要。

### FR-B: performance_testテンプレートへの任意セクション警告追加

**修正対象**: `workflow-plugin/mcp-server/src/phases/definitions.ts` の performance_test subagentTemplate内

**修正箇所の特定**: 既存テンプレートの「## 評価結論フレーズの重複回避（特化ガイダンス）」セクションの後

**変更内容**（新規追加）:

「## 任意セクション追加時の行数要件（FR-B: minSectionLines注意事項）」という新しいセクションを追加する。内容は以下の通り。

artifact-validatorは requiredSections で定義された必須セクション以外の任意セクション（## 見出し）にもminSectionLines（5行）を適用する。「## テスト実行証拠」「## 計測環境詳細」「## 比較分析」等の任意セクションを追加する場合は、そのセクション内に実質行（コンテンツを含む行）を5行以上記述すること。

任意セクションを追加して行数不足が発生した場合の原因として、サブヘッダー行や空白行が実質行に含まれないことが挙げられる。任意セクション追加時は、実際にコンテンツを含む5行以上の箇条書きまたは複数行の段落を記述すること。

**実装手順**:
performance_testテンプレートの末尾付近（「## 計測日時行の一意化...」セクションの後）に新セクションを追記する。

### FR-C1: e2e_testテンプレートへの禁止語複合語警告追加

**修正対象**: `workflow-plugin/mcp-server/src/phases/definitions.ts` の e2e_test subagentTemplate内

**修正箇所の特定**: 既存の「## テスト文書固有の角括弧禁止パターン（FR-3: e2e_test特化ガイダンス）」セクションの前か直後

**変更内容**（新規追加）:

「## 禁止語の部分一致検出に注意（FR-C1: 技術用語の言い換えガイダンス）」という新しいセクションを追加する。内容は以下の通り。

artifact-validatorの禁止語は部分一致（includes）で検出されるため、技術文書に頻出する複合語も検出対象になる場合がある。E2Eテストシナリオの記述で型や変数の状態を説明する場合は以下の言い換え表現を使用すること。

言い換え例1: 「型が確定していない状態」という表現を使うこと。宣言されたが初期化されていない変数や、戻り値の型が推論段階の変数の状態を指す場合に適切な表現である。

言い換え例2: 「参照先が設定されていない変数」という表現を使うこと。nullやundefinedが代入されている変数状態を説明する場面でバリデーターに検出されない安全な代替表現となる。

言い換え例3: 「モックが登録されていない状態」という表現を使うこと。テストセットアップが完了する前のモックオブジェクトの状態を説明する場合に使用できる代替表現である。

これらの言い換え表現を活用することで、技術的な正確さを保ちながらバリデーションエラーを回避できる。

**実装手順**:
e2e_testテンプレートの「## テスト文書固有の角括弧禁止パターン」セクションの直前に新セクションを挿入する。

### FR-C2: e2e_testテンプレートへの総合評価セクション指針追加

**修正対象**: `workflow-plugin/mcp-server/src/phases/definitions.ts` の e2e_test subagentTemplate内

**修正箇所の特定**: 「## テスト文書固有の角括弧禁止パターン（FR-3）」セクションの後、「## 出力」セクションの前

**変更内容**（新規追加）:

「## 総合評価セクションの記述指針（FR-C2）」という新しいセクションを追加する。manual_testのFR-11と同等の構造で記述する。内容は以下の通り。

`## 総合評価` セクションには、以下の5観点それぞれについて1行以上の実質行を記述し、合計5行以上の実質行を確保すること。

全E2Eシナリオの合否サマリーとして、実施件数・成功件数・失敗件数を数値とともに評価した行を記述すること。失敗がゼロの場合も「全X件成功」のように具体的な数値を含めること。

検出された問題の有無・件数・深刻度概要として、実行中に発見した不具合がある場合はその深刻度を含む評価を記述すること。不具合が存在しない場合は「不具合なし」と判断した根拠を記述すること。

未実施シナリオがある場合の理由と代替措置として、全シナリオを実施できた場合は「全シナリオ実施済み」と明記し、未実施のシナリオがある場合はその理由と後続フェーズでの対応方針を記述すること。

次フェーズ（docs_update等）への引き継ぎ事項として、後続フェーズで対処が必要な懸念点や確認事項を具体的に記述すること。引き継ぎ事項がない場合は「引き継ぎ事項なし」と判断した根拠を記述すること。

全体的な品質評価として、合格・条件付き合格・不合格のいずれかの判定を行い、その根拠を具体的に記述すること。各観点について1行以上の実質行を記述し、合計5行以上の実質行を確保すること。

**実装手順**:
「## テスト文書固有の角括弧禁止パターン（FR-3）」セクションの末尾に追記するか、「## 出力」の直前に新規セクションを挿入する。

## 実装上の注意事項

### テンプレート文字列変更時の構文注意

definitions.ts のテンプレートはシングルクォートで囲まれた文字列であり、改行は `\n` でエスケープされている。テンプレートへの追記時は以下の点に注意する。

シングルクォートの文字列内に改行を挿入する場合は `\n` を使用する。バックティック（テンプレートリテラル）は使用しないこと。テンプレート内の変数参照（`${docsDir}` 等）はそのまま維持する。TypeScriptの型チェックが通ることをビルドで確認すること。

### ビルドと再起動の手順

実装完了後に以下の手順を実行する。

ビルドコマンド: `cd workflow-plugin/mcp-server && npm run build` を実行してTypeScriptをトランスパイルする。ビルドエラーがないことを確認してからMCPサーバーを再起動すること。再起動後は `workflow_status` で現在のフェーズを確認し、同フェーズから作業を再開すること。
