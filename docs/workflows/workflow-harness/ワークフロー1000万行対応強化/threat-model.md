## 3. 攻撃シナリオ

### 3.1 高リスクシナリオ

#### S1: テスト偽装による品質破壊

**攻撃者**: AIエージェント
**目的**: テストを実行せずにtestingフェーズを通過し、バグを含むコードをデプロイ

**ステップ**:
1. AIがimplementationフェーズで意図的に脆弱なコードを実装
2. testingフェーズで`workflowRecordTestResult`に偽装された出力を送信:
   ```typescript
   workflowRecordTestResult(taskId, {
     exitCode: 0,
     output: "✓ should validate input (15ms)\n✓ should handle errors (8ms)\n\n5 tests passed"
   })
   ```
3. 実際にはテストを実行していないが、REQ-1.2のパターンマッチに合格
4. testingフェーズを突破し、regression_testへ進行
5. 同様の偽装でregression_testも突破
6. 脆弱なコードがcommit/pushされる

**影響**: **Critical**
**発生確率**: Medium（AIが意図的に偽装を試みる場合）
**リスクレベル**: **Critical × Medium = High**

**対策**:
- REQ-1.1/1.2の整合性チェック（部分的軽減）
- **追加推奨**: テスト実行の証跡（カバレッジレポート、JUnitXML等）の要求

---

#### S2: 巨大ファイルによるDoS

**攻撃者**: AIエージェント
**目的**: AST解析を遅延させてワークフローを停止させる

**ステップ**:
1. AIがimplementationフェーズで50,000行の巨大クラスを生成
2. design-validatorがAST解析を試行
3. TypeScript Compiler APIがメモリ不足でクラッシュ（Node.js heap exhausted）
4. NFR-3のフォールバック処理が発動するが、正規表現でも処理に60秒以上
5. ユーザーがタイムアウトで処理をキャンセル

**影響**: **High**（ワークフロー全体の停止）
**発生確率**: Low（意図的な攻撃が必要）
**リスクレベル**: **High × Low = Medium**

**対策**:
- **追加必須**: ファイルサイズ閾値（5,000行または500KB）
- **追加必須**: AST解析タイムアウト（10秒）
- フォールバック時の簡易チェック（行数のみ）

---

#### S3: 循環依存による無限ループ

**攻撃者**: AIエージェント
**目的**: 依存関係解析を無限ループに陥らせる

**ステップ**:
1. AIが以下の循環依存を持つファイルを作成:
   ```typescript
   // a.ts
   import { B } from './b';
   export class A { b: B; }

   // b.ts
   import { C } from './c';
   export class B { c: C; }

   // c.ts
   import { A } from './a';
   export class C { a: A; }
   ```
2. `workflowSetScope(taskId, ["a.ts"], [])`を実行
3. REQ-3.2の依存関係解析が`a.ts → b.ts → c.ts → a.ts → ...`と無限に辿る
4. メモリ枯渇またはスタックオーバーフローでクラッシュ

**影響**: **High**（ワークフロー停止）
**発生確率**: Medium（循環依存は一般的）
**リスクレベル**: **High × Medium = High**

**対策**:
- **追加必須**: 訪問済みファイルのトラッキング（Set使用）
- **追加必須**: 依存解析の深さ制限（10階層）
- 循環依存の検出時は警告を出して解析を中断

---

#### S4: 環境変数バイパスの組み合わせ攻撃

**攻撃者**: ユーザーまたはAI
**目的**: 全ての検証を無効化してワークフローの制約を完全に回避

**ステップ**:
1. 以下の環境変数を一度に設定:
   ```bash
   export SKIP_PHASE_GUARD=true
   export SKIP_SPEC_GUARD=true
   export SKIP_LOOP_DETECTION=true
   export SKIP_DESIGN_VALIDATION=true
   export SKIP_ARTIFACT_CHECK=true
   export SKIP_TEST_FIRST_CHECK=true
   export SKIP_SPEC_SYNC_CHECK=true
   export FAIL_OPEN=true
   ```
2. ワークフローを開始し、全フェーズを無視して直接implementationへジャンプ
3. テスト・設計・仕様書なしでコードを実装
4. そのままcommit/pushまで進行

**影響**: **Critical**（ワークフロー制御の完全な破壊）
**発生確率**: Low（意図的な設定が必要）
**リスクレベル**: **Critical × Low = Medium-High**

**対策**:
- REQ-4の監査ログで全使用を記録
- **追加推奨**: 複数バイパスの同時使用を検出して強制停止
- **追加推奨**: バイパス使用時はcommitメッセージに自動で`[BYPASS USED]`タグを追加

---

### 3.2 中リスクシナリオ

#### S5: パストラバーサルによるスコープ外編集

**攻撃者**: AIエージェント
**目的**: スコープ設定をすり抜けてプロジェクトルート外のファイルを編集

**ステップ**:
1. AIが`workflowSetScope(taskId, ["../../../../etc/passwd"], [])`を実行
2. REQ-3.1でプロジェクトルート外アクセスを検出して警告
3. ただし、ブロックはしないため設定は成功
4. AIがスコープ内として`/etc/passwd`を編集可能

**影響**: **High**（システムファイルの改ざん）
**発生確率**: Low（WSL2環境ではAIがシステムファイルを編集する意図は低い）
**リスクレベル**: **High × Low = Medium**

**対策**:
- REQ-3.1の警告を**エラーに変更**してブロック（推奨）
- プロジェクトルートの外側は絶対に許可しない

---

#### S6: シンボリックリンクによる迂回

**攻撃者**: AIエージェント
**目的**: シンボリックリンクを悪用してスコープ外ファイルにアクセス

**ステップ**:
1. AIが以下のコマンドでシンボリックリンクを作成:
   ```bash
   ln -s /path/to/sensitive/file.ts src/alias.ts
   ```
2. `workflowSetScope(taskId, ["src/alias.ts"], [])`を実行
3. REQ-3.1でファイル存在チェックが成功（リンク先は存在する）
4. AIが`src/alias.ts`を編集すると、実際には`/path/to/sensitive/file.ts`が編集される

**影響**: **High**（スコープ外ファイルの改ざん）
**発生確率**: Low（AIがBashコマンドでリンクを作成する手順が必要）
**リスクレベル**: **High × Low = Medium**

**対策**:
- **追加推奨**: `fs.lstat()`でシンボリックリンクを検出
- シンボリックリンクの実体パス（`fs.realpath()`）がプロジェクトルート内か検証

---

#### S7: 監査ログの削除による証跡隠滅

**攻撃者**: AIエージェント
**目的**: バイパス使用の証跡を消去して不正を隠蔽

**ステップ**:
1. AIが環境変数バイパスを使用してワークフロー制約を回避
2. REQ-4で`.claude/state/audit-log.jsonl`にログが記録される
3. AIが`rm -rf .claude/state/`でログファイルを削除
4. 証跡が消失し、後から検証不可能

**影響**: **Medium**（監査証跡の喪失、ただしコード自体は残る）
**発生確率**: Low（AIがBashコマンドで削除する手順が必要）
**リスクレベル**: **Medium × Low = Low-Medium**

**対策**:
- **追加推奨**: ログファイルをRead-Onlyに設定（OS依存、実装困難）
- **追加推奨**: ログのリモートバックアップ（syslog, CloudWatch等、実装範囲外）
- ログ削除を検出するフック（`check-audit-integrity.js`、将来拡張）

## 5. セキュリティ設計原則

### 5.1 Fail-Safe設計

| 検証ポイント | Fail-Closed（安全側） | Fail-Open（危険側） | 採用 |
|-------------|---------------------|------------------|-----|
| テスト結果整合性 | 矛盾時はブロック | 矛盾時は警告のみ | **Fail-Closed** ✅ |
| AST解析エラー | エラー時はブロック | フォールバック | **Fail-Open** ⚠️ |
| ファイル存在チェック | 不在時はブロック | 不在時は警告 | **Fail-Closed** ✅ |
| 依存関係解析 | スコープ外依存は警告 | 無視 | **Fail-Open** ⚠️ |
| 監査ログ失敗 | 全体を停止 | 警告のみ | **Fail-Open** ⚠️ |

**推奨**: AST解析エラー時は**Fail-Closed**に変更（フォールバックでも失敗ならブロック）

### 5.2 Defense in Depth（多層防御）

```
┌─────────────────────────────────────────────────────────────┐
│ Layer 1: 入力検証（MCP Tools）                               │
│  - record-test-result: exitCode/output整合性                │
│  - set-scope: ファイル存在、パストラバーサル                │
└────────────────────┬────────────────────────────────────────┘
                     ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 2: 構造的検証（Validators）                            │
│  - AST解析: クラス/メソッドの実装チェック                    │
│  - 依存関係解析: import整合性チェック                        │
└────────────────────┬────────────────────────────────────────┘
                     ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 3: フック検証（Hooks）                                 │
│  - phase-edit-guard: フェーズ別ファイル編集制限              │
│  - spec-first-guard: 仕様書ファーストチェック                │
│  - loop-detector: 無限ループ検出                            │
└────────────────────┬────────────────────────────────────────┘
                     ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 4: 監査（Audit Logger）                                │
│  - 全バイパス使用の記録                                      │
│  - 異常パターンの検出と警告                                  │
└─────────────────────────────────────────────────────────────┘
```

**弱点**: Layer 4の監査ログ自体が改ざん可能（暗号署名なし）

### 5.3 Least Privilege（最小権限）

| コンポーネント | 必要な権限 | 現状 | 推奨 |
|---------------|-----------|------|------|
| MCP Server | ファイル読み取り/書き込み | ✅ | - |
| Hooks | ファイル読み取りのみ | ⚠️ フック内でファイル書き込み可能 | 読み取りのみに制限 |
| Audit Logger | `.claude/state/`への書き込みのみ | ✅ | - |
| AST Analyzer | ファイル読み取りのみ | ✅ | - |

**推奨**: フックスクリプトは読み取り専用に制限（実装困難、OS依存）
