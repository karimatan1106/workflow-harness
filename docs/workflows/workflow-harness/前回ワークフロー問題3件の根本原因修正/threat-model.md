## サマリー

このドキュメントはCLAUDE.mdへの3つの修正項目（テスト品質ルール、subagentテンプレート品質要件、commitフェーズ確認手順）に対する脅威モデルを記述している。修正対象はドキュメント自体であり、技術的強制メカニズムがないため、AIの自律的遵守が重要である。主要な脅威は各ルールが無視される可能性、品質要件カバー不足、形骸化である。緩和策として具体性の向上、validatorルールとの整合化、チェックリスト形式の明確化を推奨している。

---

## 修正項目の概要

CLAUDE.mdには現在、ワークフロー実行に関する19個のフェーズと複数の禁止事項が定義されている。今回の修正は以下の3つの既存ドキュメントの不備を補うものである。

まず第1の修正項目は「テスト品質ルール」の追加である。現在のドキュメントではtest_implフェーズでテストコードを作成するよう指示されているが、テストコード自体の品質要件が明確でない。複雑な条件分岐を持つテストケースが形式的に追加されるだけで、実質的な検証が不足する可能性がある。

第2の修正項目は「subagentテンプレート内の品質要件」の追加である。テンプレートには現在「## サマリーセクション必須化（REQ-4）」という記載があるが、サマリーの具体的な内容（どのような項目を50行以内に記述するべきか）のみで、他の成果物セクション全体の品質基準が不明確である。

第3の修正項目は「commitフェーズの確認手順強化」である。commitフェーズ実行前に、実装内容が設計書と整合しているか、全テストがパスしているか、ドキュメントが完全に更新されているかを確認するチェックリストが不足している。確認なく進行するリスクがある。

---

## 脅威1: ルール追加後もAIが無視する可能性

### 脅威の説明

CLAUDE.mdに記載されたルールは「指示文書」という形式に過ぎず、プログラミング言語の文法制約やコンパイラによる強制がない。各フェーズをsubagentが実行する際、subagentは指示テキストを読み込むが、複数の競合する指示や曖昧な指示がある場合、AI自体の推論で優先順位を判断する。

新たに追加されたテスト品質ルール（「複雑な条件分岐を持つテストは複数のテストケースに分割する」など）は、指示としては理解されるが、実装段階で「時間制約」「コンテキスト制限」「優先度判断」により省略される可能性がある。

さらに、既存の「AIへの厳命」セクションには18個もの細則が列挙されており、新規ルールとの競合や優先度の曖昧性が生じやすい環境にある。subagentが新規ルールを軽視する可能性は低くない。

### 脅威の影響度

**高**。テスト品質ルールが無視されると、テストカバレッジが形式的になり、実装のバグ検出能力が低下する。QAフェーズ以降で問題が多く発見されることになり、開発サイクル全体が延長される。

### 緩和策

**A1-1: ルールの具体化と判断基準の明記**

各新規ルールに対し、判断基準と具体例を含める。例：

```
テスト品質ルール:
  - 単一の条件判定（if-else）のみを持つテストケース → 1つのテスト関数で可
  - 複数の条件分岐（if-else if-else if-else ...）を持つテスト → 各分岐ごとに分割
  - 具体例: ステータス遷移テストで「成功」「失敗」「タイムアウト」の3分岐がある場合、
    3つの独立したテスト関数として実装する
```

**A1-2: ルールの優先度体系の確立**

「AIへの厳命」を複数のレベルに分類し、新規ルールの位置付けを明確にする：

- レベル1（最優先）: フェーズスキップ禁止、仕様書なし実装禁止
- レベル2（次優先）: テスト品質基準、subagent品質要件
- レベル3（確認項目）: 完了宣言ルール、通知テンプレート

**A1-3: Orchestratorパターンでの検証強化**

メイン（Orchestrator）側でsubagent完了時に、成果物が品質ルールを満たしているか簡易チェックするプロセスを追加する。チェック項目は「サマリーセクション存在」「見出し数」「テスト関数数」など定量的に確認可能な項目に限定する。

## 脅威2: subagentテンプレートの品質要件が不十分

### 脅威の説明

subagent起動テンプレート（CLAUDE.mdの「subagent起動テンプレート」セクション）には、「## サマリーセクション必須化（REQ-4）」として50行以内での記述が指示されている。しかし、「目的」「主要な決定事項」「次フェーズで必要な情報」という項目は記載されているものの、それ以外のセクション（成果物本体）の構成要件が不明確である。

例えば、threatModelingフェーズの脅威モデルは何を記述すべきか、どの程度の詳細度が必要か、artifact-validatorではどのような検証基準があるか、が明確に結びついていない。このため、subagentが「文字数が足りている」「テーブルが存在する」といった形式的な要件のみで品質を判断する可能性がある。

加えて、テンプレートには「成果物品質ルール」というセクションがないため、各フェーズで「文字行の最小数」「重複行の許容度」「図解の必須性」といった基準がsubagentに伝わらない。

### 脅威の影響度

**中**。成果物が形式的に完成しても、実質的な内容が不足しており、次フェーズでの効率が低下する。特に、design_reviewやcode_reviewなど、前フェーズの成果物に依存する後続フェーズでの判断精度が落ちる。

### 緩和策

**A2-1: 各フェーズの品質ルール標準化**

subagent起動テンプレートに「成果物品質ルール」セクションを必須化し、各フェーズ共通の基準を定義する：

```
## 成果物品質ルール
成果物は以下の要件を満たすこと:
- 各セクション（## 見出し）内に最低5行の実質行を含める
- 長い段落は複雑さを避けるため複数行に分割する
- 同じ表現・行が3回以上出現しないようにする
- テーブル・図表は各セクションに最低1つ含める
```

**A2-2: フェーズ別の詳細品質基準**

各フェーズの具体的な成果物形式を明記する。例：

```
threat_modeling フェーズ:
  出力: threat-model.md
  必須セクション:
    - ## サマリー（50行以内）
    - ## 脅威ID-1: xxx（説明、影響度、緩和策各5行以上）
    - ## 脅威ID-2: xxx（以下同じ）
    - 脅威の総数は最低3個以上
  品質ルール:
    - 各脅威セクションは独立した内容とする（重複禁止）
    - 「影響度」は定量的な説明（高/中/低＋具体的なビジネス影響）
    - 「緩和策」は複数の選肢を列挙する（A-1, A-2, A-3, ...）
```

**A2-3: artifact-validatorルールとの整合化**

artifact-validator（ドキュメント検証ツール）の実際の検証ルール（行の重複排除ロジック、最小行数チェックなど）と、CLAUDE.mdの品質ルールを同じ基準で定義する。validator側で検査する内容をCLAUDE.mdテンプレートに反映させ、subagentがvalidatorで合格するレベルでの成果物作成を促す。

## 脅威3: commitフェーズの確認手順が形骸化する可能性

### 脅威の説明

commitフェーズはdocs_update、parallel_verification（manual_test、security_scan、performance_test、e2e_test）の後に実行される。このタイミングで、実装内容が設計書と整合しているか、全テストがパスしているか、ドキュメントが完全に更新されているか、といった重要な確認が行われるべきである。

しかし、CLAUDE.mdの「AIへの厳命」セクションではcommitに関する具体的な確認手順が明記されていない。「commit」コマンドの説明は「タスクを終了するcommitを実行」という1行のみで、実行前のチェックリストがない。このため、subagentが形式的にgit commitを実行するだけで、実質的な確認が省略される可能性がある。

特に、以下のようなケースで形骸化が懸念される：

- 設計書で定義された機能が一部未実装だが、「今回は省略」として強行
- テストが全パスしていないが、「既知バグとして記録」として強行
- ドキュメント更新が不完全だが、「運用マニュアルは後で」として強行

このような状況でコミットが実行されると、後続の`push → ci_verification → deploy`フェーズで問題が発覚し、修正サイクルが増加する。

### 脅威の影響度

**中**。commitフェーズは前フェーズ（特にtesting、parallel_verification、docs_update）の成果物の品質を確認する最後の門番役である。ここで確認が行われないと、不完全な実装がリポジトリにコミットされ、CI/CDパイプラインで検出されるまで問題が判明しない。

### 緩和策

**A3-1: commitフェーズのチェックリスト標準化**

「AIへの厳命」セクションに新たに以下の項目を追加する：

```
19. **commitフェーズでの最終確認チェックリスト**
    □ parallel_verification 全フェーズが完了したか？
      - manual_test: テスト結果レポートが存在するか？
      - security_scan: セキュリティスキャン結果が存在するか？
      - performance_test: パフォーマンステスト結果が存在するか？
      - e2e_test: E2Eテスト結果が存在するか？
    □ 設計書と実装の整合性が確認されたか？
      - spec.md の全機能が実装されているか？
      - state-machine.mmd の全状態遷移が実装されているか？
      - flowchart.mmd の全処理フローが実装されているか？
      - ui-design.md の全UI要素が実装されているか？
      - 設計書にない追加実装はないか？
    □ テストが全パスしているか？
      - testing フェーズでのテスト合格率は100%か？
      - regression_test で新たな既知バグが記録されていないか？
    □ ドキュメントが完全に更新されているか？
      - docs/spec/ ファイルが最新の実装に基づいて更新されているか？
      - docs/architecture/ の関連ドキュメントが更新されているか？
      - CHANGELOG.md / README.md が更新されているか？
    □ コミットメッセージが適切に記述されているか？
      - タスク名が含まれているか？
      - 主な変更点が列挙されているか？
```

**A3-2: 確認失敗時のエスカレーション**

チェックリストのいずれかに「NG」がある場合、commit実行を中断し、対象フェーズに戻すプロセスを明記する：

```
チェックリスト確認結果:
- [ ] 完全 → commit実行
- [!] 不完全 → 対象フェーズに差し戻し

具体的なマッピング:
- 設計-実装不整合 → implementation フェーズに差し戻し
- テスト不合格 → testing or regression_test フェーズに差し戻し
- ドキュメント不完全 → docs_update フェーズに差し戻し
```

**A3-3: Orchestratorによる自動確認**

commitフェーズに移行する直前に、メイン（Orchestrator）が以下の自動チェックを実施する：

```
自動チェック項目:
- parallel_verification の全サブフェーズのマーク確認
- {docsDir} ディレクトリに必須ドキュメントが存在するか確認
  (manual-test.md, security-scan.md, performance-test.md, e2e-test.md)
- docs/spec/ の主要ファイル（features/*.md, api/*.md等）が更新されているか確認
- git status で未ステージ状態のファイルがないか確認
```

## 緩和策の実装順序

緩和策は以下の優先順で実装することを推奨する：

1. **優先1（即時実施）**: A3-1 commitフェーズのチェックリスト
   - 最も効果が高く、実装が簡単
   - 次のワークフロー実行から効果が見込める

2. **優先2（次回の修正）**: A1-1 ルールの具体化と判断基準明記
   - subagent起動テンプレートの複数フェーズに影響
   - 一括修正の効率が良い

3. **優先3（計画段階）**: A2-1〜A2-3 品質ルール標準化
   - artifact-validator との整合化が必要
   - コード側の修正も伴う可能性がある

## 結論

CLAUDE.md への3つの修正は、AIの自律的遵守に依存したドキュメント更新である。技術的な強制メカニズム（フック、バリデータ）がないため、ルールの「具体性」「判断基準の明確化」「優先度体系の確立」が重要である。

特に、commitフェーズのチェックリスト追加（脅威3の緩和策A3-1）は即座に実装可能で、効果が高い。一方、テスト品質ルールと品質要件の標準化（脅威1、2の緩和策）は、artifact-validator との整合性確認が必要であり、段階的な実装が適切である。
