# FR-R6 researchフェーズ: バリデーション失敗の根本原因分析

## サマリー

- 目的: FR-R5タスクのparallel_verificationフェーズで発生した複数のバリデーション失敗を根本原因レベルで分析し、再発防止のための修正方針を明確にする
- 主要な決定事項: 4つの根本原因が特定された。CLAUDE.mdとdefinitions.tsのbuildPromptにおける禁止語ガイダンスの記述方式の差異、行数カウントの仕様（空白行除外）と成果物構造の相性問題、subagentが禁止語を言い換えるつもりで使用した複合語が部分一致で検出される問題、そしてリトライプロンプト内で禁止語を間接参照すべきところを直接転記してしまう問題がある
- 次フェーズで必要な情報: artifact-validator.tsの禁止語リスト（12語）の正確な内容、definitions.tsのbuildPrompt関数が生成する禁止パターンセクションの記述方式、CLAUDE.mdのsubagentテンプレートと実際のbuildPrompt出力の差異

---

## 調査結果

### 調査1: CLAUDE.mdとバリデーター間の禁止語リスト照合

CLAUDE.mdの「禁止パターン（完全リスト）」セクション（行306-309）には以下が記載されている。

英語グループ（4語）: 作業管理で使われる4つの英語略語
日本語グループ（8語）: 検討・暫定状態を示す8つの日本語表現
合計12語が部分一致検索（includes）で検出される

artifact-validator.tsの`forbiddenPatterns`配列（行343-356）に定義されている語句は、CLAUDE.mdの記述と完全に一致している。12語の内容はどちらも同一であり、リスト自体の不一致は存在しない。

ただし重要な発見として、CLAUDE.mdのサブエージェントテンプレートセクション（行291-292）では「禁止語の完全リストは下記「禁止パターン（完全リスト）」を参照すること」と案内しているが、definitions.tsのbuildPrompt関数が生成するqualitySectionでは（行1093-1098）語句そのものを列挙せず「英語系略語グループ（4語）」「検討系グループ（4語）」「予定・仮値系グループ（4語）」というグループ名のみを記述している。つまりbuildPromptが生成するプロンプトには禁止語の実体が含まれず、subagentは具体的に何が禁止されているかを把握しにくい状態になっている。

### 調査2: definitions.tsのsubagentTemplateにおける禁止語ガイダンス

manual_test・security_scan・e2e_testの各subagentTemplateは固定テキストとして定義されており、buildPromptが生成するqualitySectionとは独立している。各subagentTemplateの内容を確認した結果、以下の問題が判明した。

manual_testのsubagentTemplate（行890）には重複行回避の注意事項が記述されているが、禁止語グループの具体的な内容は記載されていない。テンプレートテキストに「作業内容」として手動テストの実施を指示するのみで、バリデーターが検出する語句の詳細な回避方法が不足している。

security_scanのsubagentTemplate（行902）は重複行回避の注意事項と各セクションの行数ガイダンスが含まれており、他のサブフェーズより充実している。しかし禁止語そのものを列挙せずグループ分類のみで説明する構成はbuildPromptと同様であり、subagentが「具体的に何を避けるべきか」を判断しにくい状態は変わらない。

e2e_testのsubagentTemplate（行926）も同様に禁止語の実体が記述されていない。buildPromptが後から追加するqualitySectionに委ねる設計になっているが、subagentTemplateが先に読まれる構造であるためsubagentが適切な対策を事前に立てられない可能性がある。

### 調査3: 行数カウント仕様の確認

artifact-validator.tsの行数チェック（行317）の実装は以下の通りである。

```typescript
const lines = content.split('\n').filter(line => line.trim().length > 0);
if (lines.length < requirements.minLines) {
```

この実装によって、行数カウントは「空白行（トリム後の長さが0の行）を除外した非空行数」となる。これはコードフェンス内の行も含まれる。つまりMarkdownの区切り線（`---`）・見出し行・コードフェンス境界行・テーブル行・実質的な本文行がすべて1行としてカウントされる。

FR-R5で発生したsecurity_scanの失敗では「34行あるが非空行が19行」と報告されたが、この報告は行数カウントの仕様と一致している。コードフェンス内の行はextractNonCodeLinesでは除外されるが行数カウントでは含まれる。ファイル内に多数の空白行がある場合、コンテンツとして見えるより実際の非空行数は少なくなる。

CLAUDE.mdのsubagentテンプレートでは「最低20行以上」という表現が使われているが、これが非空行数（コードフェンス内の行を含む）を指すことの明示が不足している。特にMarkdownの書式として空行を多用するとカウントが期待より下回る。

### 調査4: subagentTemplateの禁止語回避ガイダンスの十分性評価

FR-R5の失敗事例を詳細に分析すると以下の4パターンに分類できる。

**パターンA（manual_test初回失敗）**: subagentがバリデーション失敗の説明のために禁止語をそのまま成果物に記述した。これはリトライプロンプトのテンプレートにある「前回のバリデーション失敗理由」コードブロックの内容を成果物本文に転記してしまったケースと推定される。definitions.tsの入力ファイル転記禁止ガイダンス（行1109-1113）はリトライ時の転記も明示的に禁止すべきところ、初回プロンプトの文脈でしか記述されていない。

**パターンB（manual_testリトライ1回目失敗）**: subagentが「状態が確定していない」という意味を表現しようとして類似表現を使用したが、その表現に禁止語が部分一致で含まれていた。buildPromptの禁止語グループ説明（行1095-1097）では「検討・未確定の状態を表す日本語表現」と説明しているが、「未確定」という語自体も検出対象であることが明示されていない。

**パターンC（security_scan初回・リトライ失敗）**: 行数が継続して最低行数を下回った。subagentTemplateに最低行数の確認方法や、行が不足しがちな原因（空白行の多用、コードフェンスの多用）についての具体的な対策が記述されていない。

**パターンD（e2e_test・code_review失敗）**: 複数のサブフェーズで同様の禁止語検出が発生しており、buildPromptのqualitySectionが実際のフェーズ実行時に正しく機能しているか、あるいはsubagentTemplateの内容がbuildPromptの内容と重複・矛盾していないかの確認が必要である。

## 既存実装の分析

### artifact-validator.tsの構造

artifact-validator.ts（1332行）は以下の検証を順序通りに実行する。

1番目の検証: ファイル存在チェック（空ファイル含む）
2番目の検証: 非空行数チェック（空白行除外後の行数 >= minLines）
3番目の検証: 必須セクションヘッダーの存在確認（includes検索）
4番目の検証: 禁止パターン検索（extractNonCodeLinesで取り出したコードフェンス外テキストに対してincludes検索）
5番目の検証: 重複行検出（3回以上の同一行でエラー、構造行は除外）
6番目の検証: ヘッダーのみ成果物の排除（5行以上の非ヘッダー行が必要）
7番目以降: セクション密度・短行比率・コードパス参照・サマリー行数の追加検証

このうちバリデーション失敗の主因となるのは2番目（行数）と4番目（禁止パターン）であり、FR-R5の各失敗もこの2つに集中していた。

### definitions.tsのbuildPrompt構造

buildPrompt関数（行994以降）は9つのセクションを動的生成して結合する。セクション5にqualitySection（禁止パターン・重複行・密度ガイダンスを含む）が組み込まれ、各subagentTemplateの後に付加される。

subagentTemplateはフェーズ固有の作業指示を含む固定テキストであり、buildPromptが生成するqualitySectionは汎用的な品質要件の説明である。この2層構造において、subagentTemplateが先に読まれた後でqualitySectionが後付けされる形になる。subagentが成果物の最初のドラフトを作成する際にはsubagentTemplateの指示が優先的に処理され、qualitySectionの詳細な回避策が見落とされる可能性がある。

### CLAUDE.mdのsubagentテンプレートとdefinitions.tsの乖離

CLAUDE.mdのサブエージェント起動テンプレート（メインプロジェクトの指示書）には「禁止パターン（完全リスト）」セクションが存在し、12語の禁止語が具体的にコードフェンス内に列挙されている。このCLAUDE.mdを読んだ場合、subagentは禁止語の実体を把握できる。

一方、definitions.tsのbuildPromptが生成するqualitySectionはグループ分類のみで語句の実体を含まない。definitions.tsのqualitySectionはCLAUDE.mdのテンプレートセクションと同様の情報を提供するが、語句の実体をコードフェンス内に列挙するという点でCLAUDE.mdの「禁止パターン（完全リスト）」形式には対応していない。

## 修正方針（次フェーズ要件定義に向けて）

以下の4点が修正候補として挙げられる。

**修正候補1**: definitions.tsのbuildPromptのqualitySectionに禁止語の実体リストをコードフェンス内で列挙する。現在のグループ分類説明に加えて「コードフェンス内に安全に記述可能な形式で実際の禁止語一覧」を付加することで、subagentが具体的な回避対象を把握できるようになる。

**修正候補2**: 各subagentTemplateの冒頭に「成果物作成前に必ずqualitySectionの禁止パターンセクションを確認すること」という注意書きを追加する。現在のsubagentTemplateには品質要件への参照が明示されていない。

**修正候補3**: セクション行数ガイダンスに「行数カウントは空白行を除外した非空行数であり、コードフェンス内の行は行数に含まれるがextractNonCodeLinesによる禁止語検索からは除外される」という仕様説明を追加する。現在のガイダンスはこの点を明示していない。

**修正候補4**: リトライプロンプトテンプレート（CLAUDE.md行341-386）において「前回バリデーション失敗理由のコードブロック内にある禁止語を成果物に転記してはならない」という注意を、初回プロンプトにも展開する。現在この警告はリトライ時のみ記載されている。
