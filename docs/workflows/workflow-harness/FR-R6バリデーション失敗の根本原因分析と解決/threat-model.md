# FR-R6 脅威モデル: バリデーション失敗根本原因解決のための変更安全性分析

## サマリー

- 目的: definitions.tsのbuildPrompt関数とCLAUDE.mdのsubagentテンプレートに対して実施する修正が、既存の安全機構（HMAC整合性・フック制御・バリデーター）を破壊しないことを確認し、修正が意図せず新たな問題を引き起こすリスクを分析する
- 分析範囲: 変更対象はbuildPromptのqualitySection生成ロジックとCLAUDE.mdのsubagentテンプレートセクションに限定されており、HMAC計算や状態管理コードは変更しない
- 主要な決定事項: buildPrompt/qualitySectionはHMAC対象外であることを確認済みであり、修正の影響範囲はsubagentへのプロンプト生成ロジックに限定される。禁止語の実体をqualitySectionに追加する場合のフィードバックループリスクが最高深刻度の脅威として特定された。resolvePlaceholders関数が波括弧を変数展開として解釈する可能性が中程度のリスクとして特定された。
- 特定された脅威数: 計6件（最高深刻度1件・中深刻度2件・低深刻度2件・最低深刻度1件）。T1の禁止語転記フィードバックループが最優先対処事項である。
- 次フェーズで必要な情報: 脅威T1（禁止語転記フィードバックループ）に対する具体的な緩和策として採用するアプローチの決定（コードフェンス内配置 vs 間接参照方式 vs 専用変数展開の無効化）、脅威T3（resolvePlaceholders誤展開）への対処として波括弧エスケープ方針の確認
- 推奨アクション: planningフェーズでT1緩和策の実装方針を確定し、implementationフェーズ開始前にresolvePlaceholders関数の挙動確認を必須手順として追加すること

---

## 分析対象の変更概要

今回のタスクで変更する対象ファイルと変更内容を以下に整理する。変更の主な目的は、parallel_verificationフェーズのsubagentがバリデーション失敗を繰り返す根本原因を除去することにある。

- 変更対象1: `workflow-plugin/mcp-server/src/phases/definitions.ts`のbuildPrompt関数内のqualitySection生成ロジック。禁止語の実体リスト・行数カウント仕様の明示・安全な代替表現パターン・転記防止指示を追加する変更である。
- 変更対象2: `CLAUDE.md`のsubagentテンプレートセクション・禁止パターンセクション・リトライプロンプトテンプレートセクション。manual_test・security_scan・e2e_testの各subagentTemplateに行数カウント仕様と転記防止指示を追加する変更である。
- 変更対象外として確認: `artifact-validator.ts`のバリデーションロジックは変更しない。`state-manager.ts`のHMAC計算処理は変更しない。フックスクリプトは変更しない。
- 変更の影響範囲: qualitySection変更はbuildPromptを通じて全フェーズのsubagentプロンプトに反映されるが、CLAUDE.md変更はオーケストレーター（メインClaude）が参照するテンプレートにのみ影響する。
- 変更後の検証方針: npm run buildによるビルド確認、MCPサーバー再起動、そしてworkflow_statusによるHMAC整合性確認を必須手順として実施する。

## 脅威T1: 禁止語転記によるフィードバックループ（最高深刻度）

### 脅威の説明

qualitySectionに禁止語の実体リストをそのまま記述した場合、subagentはその文字列を成果物本文に転記してしまうリスクがある。この現象は過去のparallel_verificationフェーズで実際に確認されており、リトライプロンプトに含まれるエラーメッセージの禁止語が成果物に混入してバリデーションが連続失敗するという循環を生じさせた。

### リスク評価

- 深刻度: 最高（修正が逆効果となり、改善の目的が達成されないどころか悪化する可能性がある）
- 発生確率: 高（subagentはプロンプト内のテキストを忠実に再現しようとする傾向があり、例示として提示した禁止語を成果物中に引用するパターンが繰り返し観察されている）
- 影響範囲: 全フェーズのsubagent（qualitySectionはbuildPromptを通じて全フェーズのプロンプトに挿入される）

### 緩和策

- 緩和策A（推奨）: 禁止語の実体をコードフェンス内にのみ記述し、「コードフェンス内の記述は成果物本文に転記しないこと」という明示的な指示を冒頭に配置する。artifact-validatorはコードフェンス内の行を禁止語検索から除外するため、この配置は技術的に安全である。
- 緩和策B（補完）: 各禁止語グループの説明文には実体を記述せず、「バリデーターが検出するパターンの実体はコードフェンス内の参照表を確認すること」という誘導を行い、本文中での直接記述を避けるよう促す。
- 緩和策C（転記防止）: qualitySectionの冒頭に「このセクションに記載されている禁止語の実体は参照情報であり、成果物本文・改善要求セクション・リトライプロンプトの本文には直接記述してはならない」という強調ブロックを配置する。

## 脅威T2: HMAC整合性への意図しない影響（低深刻度）

### 脅威の説明

buildPrompt関数はMCPサーバーのプロンプト生成ユーティリティであり、ワークフロー状態（workflow-state.json）のHMAC計算には含まれない。しかし、definitions.tsを変更した際に誤ってstate-manager.tsやhmac.tsに依存する関数を変更してしまった場合、HMACの検証が失敗し、フックがタスク状態を読み取れなくなるリスクがある。

### リスク評価

- 深刻度: 中（HMAC破損はワークフロー操作を完全にブロックする可能性があるが、発生確率は低い）
- 発生確率: 低（今回の変更はbuildPrompt関数のqualitySection生成ロジックに限定されており、HMAC計算に関与するコードパスから十分に分離されている）
- 影響範囲: HMAC破損が発生した場合は全タスクの状態読み取りが失敗し、ワークフロー操作が完全に不能になる

### 緩和策

- definitions.tsの変更後にnpm run buildでトランスパイルを実行し、型エラーがないことを確認してからMCPサーバーを再起動する手順を必須とする（NF-R6-2に既定）。
- 変更箇所をqualitySection生成の文字列テンプレートに限定し、エクスポートされる関数シグネチャや型定義を変更しないことをimplementationフェーズの制約とする。
- 変更後にworkflow_statusでタスク状態が正常に読み取れることを確認し、HMAC整合性に問題がないことを手動で検証する手順を定める。

## 脅威T3: resolvePlaceholders関数による波括弧の誤展開（中深刻度）

### 脅威の説明

buildPrompt関数が生成するqualitySectionには、波括弧で囲まれた変数プレースホルダーが含まれる場合がある。definitions.tsの中にresolvePlaceholders関数またはそれに相当するロジックが存在した場合、qualitySectionに記述した波括弧で囲まれた文字列が変数展開の対象として誤って処理され、意図しない文字列に置換されるリスクがある。

### リスク評価

- 深刻度: 中（誤展開が発生した場合、subagentに渡るプロンプトが誤った内容になり、ガイダンスが意図通りに機能しない）
- 発生確率: 中（CLAUDE.mdのsubagentテンプレートにはすでに波括弧プレースホルダーが多数使用されており、qualitySectionを追加することでプレースホルダーとして誤認識される文字列が増加する可能性がある）
- 影響範囲: qualitySectionを受け取る全フェーズのsubagentプロンプト

### 緩和策

- 安全な代替表現リストに含まれる文字列に波括弧を使用しない設計とし、コードフェンス外の例示文には角括弧（バリデーターは成果物本文の角括弧を検出するが、definitions.tsのソースコード文字列は成果物ではないため検出対象外）またはバッククォートを使用する。
- resolvePlaceholders関数の動作をimplementationフェーズで事前確認し、qualitySectionに含まれる文字列がどのように処理されるかをテスト用の小規模なスクリプトで検証する（NF-R6-3の検証方法1と組み合わせる）。
- CLAUDE.mdの変更については、プレースホルダー展開が発生しないため波括弧の制約は不要であるが、一貫性のため同様のエスケープ方針を適用することが望ましい。

## 脅威T4: MCPサーバーキャッシュによる変更未反映（高発生確率・低深刻度）

### 脅威の説明

Node.jsのrequire()によるモジュールキャッシュの性質上、definitions.tsを変更してnpm run buildでトランスパイルした後も、MCPサーバープロセスを再起動しない限り変更は反映されない。この問題はMEMORY.mdでも記録されており、実際に過去に何度か発生している既知の動作である。

### リスク評価

- 深刻度: 低（再起動を忘れた場合の影響は「変更が反映されない」という状態に留まり、既存機能の破壊は発生しない）
- 発生確率: 高（手動の再起動手順を要求する運用では人的ミスが発生しやすい）
- 影響範囲: definitions.tsの変更が必要な全フェーズのsubagentプロンプト生成。CLAUDE.mdの変更はMCPサーバーとは無関係に即座にセッション再起動後から反映されるため影響を受けない。

### 緩和策

- implementationフェーズの完了基準として「npm run build成功後にMCPサーバー再起動を完了し、workflow_statusでフェーズ確認済みであること」を必須とする（NF-R6-2に既定）。
- 変更が反映されたことの確認として、buildPromptの出力をworkflow_statusのレスポンスまたはログで観察できる仕組みが理想的だが、今回のスコープ外とする。
- testing/regression_testフェーズでqualitySectionの実際の出力内容を手動確認することで、キャッシュ問題の有無を間接的に検証できる。

---

## 脅威T5: qualitySection追記による既存フェーズの品質低下（低深刻度）

### 脅威の説明

qualitySectionへの追記によってプロンプト全体の長さが増加し、subagentのコンテキスト処理に影響する可能性がある。具体的には、禁止語リスト・代替表現パターン・転記防止指示を全て追加した場合、qualitySectionが200トークンを超過し、subagentが重要な指示を見落とすリスクが高まる。また、NF-R6-1で「追記内容は200トークン以内」と定められているが、この制約を守ることでガイダンスが不十分になる可能性もある。

### リスク評価

- 深刻度: 低（既存の正常動作フェーズへの影響は軽微であり、バリデーション失敗が増加したとしても修正可能である）
- 発生確率: 低（200トークンの制約を遵守する設計を行えば、既存フェーズへの影響は最小化できる）
- 影響範囲: qualitySectionを受け取る全フェーズのsubagent。特にコンテキスト量が多いimplementation・code_reviewフェーズで影響が大きくなる可能性がある。

### 緩和策

- qualitySectionの追記内容を「禁止語参照表（コードフェンス内）」「転記防止警告（1〜2行）」「代替表現の要点（箇条書き3行以内）」の3項目に絞り込み、冗長な説明を排除する。
- 全subagentTemplateへの追記を一括で行うのではなく、最も頻繁に失敗するmanual_test・security_scan・e2e_testから優先的に変更し、既存の正常フェーズへの影響を最小化した漸進的なアプローチを採用する。

---

## 脅威T6: performance_testガイダンスとの不整合（最低深刻度）

### 脅威の説明

requirements.mdのNF-R6-3には、FR-R5で失敗したmanual_test・security_scan・e2e_testのフェーズを変更後の環境で再実行することが検証方法として定められているが、performance_testフェーズへのsubagentTemplateの変更については言及されていない。performance_testフェーズが既存のqualitySection記述と新たな追記内容で不整合を起こした場合、このフェーズも失敗する可能性がある。

### リスク評価

- 深刻度: 最低（performance_testフェーズは過去の失敗ログに含まれておらず、現状は正常動作している）
- 発生確率: 最低（今回の変更はperformance_testのsubagentTemplateを変更しない方針であるため、影響は発生しない）
- 影響範囲: performance_testサブフェーズのみ

### 緩和策

- performance_testのsubagentTemplateには今回の変更を適用しないことをplanningフェーズのspec.mdに明示し、変更スコープを正確に管理する。
- 変更後のregression_testフェーズで全フェーズの動作を確認する際に、performance_testも確認対象に含める。
