## サマリー

FR-R6は、ワークフロープラグインのバリデーション機構における4つの改善をCLAUDE.mdに反映するタスクです。このタスクは内部ルール改善であり、外部向けドキュメント（README等）の更新は不要です。実装内容（definitions.ts）の改善は既に完了し、CLAUDE.mdに記載されたガイダンスを充実させることが目的です。主要な変更として、禁止語の完全リスト追加、行数仕様の明確化、代替表現パターンの充実化、バリデーション失敗時のフィードバックループ警告を含みます。

---

## 変更内容の概要

### FR-R6による4つの改善項目

ワークフロープラグインの品質ガイダンスを以下の4項目で充実させました。

#### FR-R6-1: 禁止語完全リストのコードフェンス記述

成果物品質要件セクションの「禁止パターン（完全リスト）」に、禁止語の12語を段階的にコードフェンスで示しました。このコードフェンス形式により、バリデーターのロジック確認が容易になり、subagentが禁止語の意図を正確に理解できるようになります。英語グループ（TODO, TBD, WIP, FIXME）と日本語グループ（未定、未確定、要検討、検討中、対応予定、サンプル、ダミー、仮置き）の合計12語を明記しました。

#### FR-R6-2: 行数仕様の明確化（非空行カウント）

成果物品質要件における「行数・密度要件」セクションを大幅に改訂し、実質行のカウント基準を明確化しました。subagentのプロンプトテンプレートに詳細な仕様を追加し、「各セクション内に最低5行の実質行」の定義を統一しました。空行・水平線・コードフェンスを除いた行をカウント対象とすることで、AIモデルの理解度を向上させます。

#### FR-R6-3: 安全な代替表現パターンセクション追加

CLAUDE.mdに「安全な代替表現パターン」という新規セクションを追加しました。禁止語を回避するための言い換えパターンを3つのカテゴリに分類し、各カテゴリごとに複数の具体的な表現例を示しています。確定・確認状態、追加調査の必要性、架空データの表現について、言い換え罠（別の禁止語を含む表現）を避けるための指針も含まれます。

#### FR-R6-4: エラーメッセージからの禁止語転記防止ガイダンス

バリデーション失敗時のリトライプロンプトテンプレートに、「禁止語転記防止」と「フィードバックループ警告」を追加しました。MCPサーバーが返すエラーメッセージ内に含まれる禁止語を、改善要求セクションに直接転記してはいけないという明確なルールを明記しました。このルールにより、エラーメッセージ→改善要求→成果物という処理ステップで禁止語が伝播するフィードバックループを防止できます。

## 実装ファイルの変更

### workflow-plugin/mcp-server/src/phases/definitions.ts

definitions.tsの buildRetryPrompt 関数内のプロンプトテンプレートに、改善要求セクションの記述方法を強化しました。以下の項目が変更されています。

**改善されたエラー参照方法:**
- MCPサーバーから返されたエラーメッセージはコードブロックで括りオフスコープ化
- エラー内容への言及は「バリデーターが検出した語句」などの間接表現を使用
- 禁止語そのものを改善要求に記述することを厳格に禁止

**フィードバックループ警告の追加:**
- 「⚠️ フィードバックループ警告」というセクションを新規追加
- 過去に発生したフィードバックループの原因（禁止語転記）を明記
- 対策として「一度発生したフィードバックループは転記を止めない限り永続する」という警告文

## CLAUDE.mdの更新内容

### 1. 禁止パターン（完全リスト）セクションの強化

既存セクションを以下の構成に改訂しました。

**コードフェンス形式での明記:**
```
英語グループ（4語）: TODO, TBD, WIP, FIXME
日本語グループ（8語）: 未定, 未確定, 要検討, 検討中, 対応予定, サンプル, ダミー, 仮置き
合計12語。includes()による部分一致検出のため、禁止語を含む複合語も検出対象となる。
```

このコードフェンス形式により、バリデーターの検出ロジックとの対応が明確になります。

### 2. 安全な代替表現パターンセクション（新規追加）

CLAUDE.mdの「禁止パターン（完全リスト）」の直後に新規セクションを追加しました。

**3つのカテゴリ分類:**
- 確定・確認が取れていない状態を表す場合
- 追加調査が必要な状態を表す場合
- 架空のデータや例示用の値を表す場合

各カテゴリに複数の言い換え例を示し、言い換え罠を避けるための原則も明記しました。

### 3. バリデーション失敗時のリトライプロンプトテンプレート改訂

「subagent起動テンプレート」セクションの「バリデーション失敗時のリトライプロンプトテンプレート」を以下の3項目で強化しました。

**改善項目:**
- 「前回のバリデーション失敗理由」セクションで、エラーメッセージをコードブロックで明確に分離
- 「改善要求」セクションで、禁止語転記を避ける言及方法を具体化
- 「⚠️ 上記コードブロック内のエラーメッセージは参照情報です」という警告文を追加

**フィードバックループ警告文の追加:**
```
⚠️ フィードバックループ警告: エラーメッセージ中の語句を成果物本文に転記すると
次回も同じエラーが発生するフィードバックループが形成される。一度発生したフィード
バックループは転記を止めない限り永続する。禁止語への言及は必ず「バリデーターが
検出した語句」「引用内に含まれるキーワード」等の間接参照を使用すること。
```

## 設計変更による効果

### バリデーション失敗率の低下

安全な代替表現パターンの明確化により、subagentが禁止語回避の方法を正確に理解できるようになります。これにより、成果物の初回バリデーション成功率が向上することが期待されます。

### フィードバックループの防止

エラーメッセージ内の禁止語を成果物本文に転記するフィードバックループは、ワークフロー実行を長期化させる主要な原因でした。リトライプロンプトの改訂により、禁止語の伝播を技術的に遮断できます。

### subagentの生産性向上

禁止パターンと代替表現が明確化されることで、subagentが品質要件を初回で満たす可能性が高まり、リトライ回数が削減されます。
