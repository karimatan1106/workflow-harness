# FR-R6 ui_designフェーズ: インターフェース設計

## サマリー

本ドキュメントはFR-R6タスクにおけるCLI/MCPサーバーのインターフェース設計を定義する。
FR-R6はUI画面ではなく、MCPサーバーのプロンプトテンプレートおよびAPIレスポンス構造の改善を対象としている。
設計の目的は、subagentが成果物を初回から正しく生成できるよう、情報提供の構造を改善することにある。

主要な決定事項として、以下の4点を設計の中心とする。
- qualitySectionにコードフェンス形式で禁止語の実体リストを含める構造の定義
- 各subagentTemplateへの非空行数カウント仕様説明の追加位置と形式の定義
- エラーメッセージにおける禁止語の間接参照パターンの標準化
- リトライプロンプトにおけるフィードバックループ防止警告の配置設計

次フェーズの実装では、本ドキュメントに記載する各セクションの構造と配置位置を参照して実装を進めること。

---

## CLIインターフェース設計

### workflow_statusコマンドの出力設計

workflow_statusコマンドの出力にqualitySection情報を含めるかどうかについては、含めない設計とする。
理由として、qualitySectionはbuildPromptの出力としてsubagentへのプロンプトに埋め込まれるものであり、
workflow_statusの応答として表示する性質のものではないためである。

workflow_statusは現在のフェーズ・タスク名・完了済みサブフェーズの一覧を表示する設計を維持し、
qualitySectionの内容確認が必要な場合はdefinitions.tsのソースコードを直接参照する手順を採用する。

### MCPサーバー再起動確認のステータス表示

definitions.tsへの変更後にMCPサーバーを再起動したかどうかを確認するためのステータス表示は設けない。
代替として、CLAUDE.mdのコアモジュール変更後の再起動ルール（ルール22）にdefinitions.tsを明示的に列挙し、
再起動が必要なファイルの一覧としてdefinitions.tsが含まれることを強調表示する構造とする。

再起動確認は以下の手順で行うことを標準化する。
- ビルド実行後にdist/phases/definitions.jsのファイル更新日時が現在時刻に更新されたことを確認する
- workflow_statusを実行してMCPサーバーが応答することを確認する
- workflow_statusの応答に含まれるフェーズ情報が期待値と一致していることを確認する

## エラーメッセージ設計

### バリデーション失敗時のエラーメッセージ構造

バリデーション失敗時にworkflow_complete_subが返すエラーメッセージには、
禁止語の実体を直接掲載せず間接参照する形式を採用する。
エラーメッセージの構造は以下の通りである。

```
エラー種別: 禁止語検出
説明: 成果物本文に使用を避けるべき語句が含まれています。
対処法: バリデーターが警告している語句の実体はCLAUDE.mdの「禁止パターン（実体リストと言い換えガイド）」セクションを参照してください。
       成果物本文中での禁止語への言及は「バリデーターが検出した語句」等の間接参照を使用してください。
```

Orchestratorへのエラー返却時には、禁止語の実体を含まない形式で理由を説明する。
subagentがリトライプロンプトを受け取ったときにエラーメッセージ内の語句を転記しないよう、
エラーメッセージ自体に転記防止の警告文を含める設計とする。

### 行数不足エラーメッセージの改善設計

行数不足エラーメッセージには、現在の総行数と非空行数の両方を含める構造とする。
表示形式の例を以下に示す。

```
エラー種別: 行数不足
現在の非空行数: 15行（空白行を除く）
必要な非空行数: 20行以上
説明: 行数カウントは空白行（空白文字のみの行を含む）を除外した非空行数で算出されます。
     Markdownの段落間空白行は行数に算入されません。内容を充実させることで非空行数を増やしてください。
```

非空行数の定義として、content.split('\n').filter(line => line.trim().length > 0).lengthの算出結果であることを
エラーメッセージ内の説明文に間接的に示す設計とする。具体的なコードは含めず、
「空白文字のみの行を除いた行数」という言葉で説明する形式を採用する。

### リトライプロンプトの転記防止警告設計

リトライプロンプトにおける転記防止警告は、前回のバリデーション失敗理由セクションの直後に配置する。
警告の文言として以下の構造を採用する。

警告ブロックの内容:
- 上記コードブロック内のエラーメッセージに含まれる語句を成果物本文に転記しないこと
- エラーメッセージ中の語句を改善要求セクションや成果物本文に転記すると、次回も同じエラーが発生するフィードバックループが形成されること
- 禁止語への言及は「バリデーターが検出した語句」「引用内に含まれるキーワード」等の間接参照を使用すること

この警告ブロックはOrchestratorがリトライプロンプトを構築するCLAUDE.mdのテンプレートセクション（行348-356付近）に追記する形で実装する。

## APIレスポンス設計

### buildPromptのqualitySection出力構造

buildPromptが返すqualitySectionの出力構造を以下のように定義する。
qualitySectionはMarkdown文字列として生成され、以下の順序でセクションが並ぶ設計とする。

第1ブロック: 行数・密度要件の説明
- 非空行数カウントの仕様（空白行除外）の明示
- 各セクションの最低実質行数（5行）の説明
- セクション密度（実質行/総行30%以上）の説明

第2ブロック: 重複行禁止ルールの説明
- 重複閾値（3回以上）の説明
- 構造要素の除外ルール一覧
- 回避策の具体例

第3ブロック: 禁止パターン実体リスト（コードフェンス内）
- コードフェンス開始行（バッククォート3つ + text）
- 英語グループの禁止語4語を列挙（各1行）
- 日本語グループの禁止語8語を列挙（各1行）
- コードフェンス終了行
- コードフェンス内の語句は禁止語検出の対象外であるという注記

第4ブロック: 安全な代替表現リスト
- 確定されていない状態を表す安全な表現パターン（3例以上）
- 追加調査が必要な状態を表す安全な表現パターン（3例以上）
- 架空データを示す状態を表す安全な表現パターン（3例以上）
- 部分一致検出の回避原則（語句を長くすることで意味を特定化する）

第5ブロック: 禁止語転記防止の警告
- 禁止語を成果物本文に記述するとバリデーション失敗を引き起こすこと
- 問題・失敗・例外の説明時は間接参照パターンを使用すること

### workflow_complete_subのバリデーション失敗レスポンス構造

workflow_complete_subがバリデーション失敗を返す際のレスポンス構造を以下のように定義する。

成功時レスポンス構造:
```
{ success: true, phase: "{フェーズ名}", message: "サブフェーズ {名} が完了しました" }
```

バリデーション失敗時レスポンス構造:
```
{
  success: false,
  validationError: true,
  errorType: "{エラー種別}",
  errorSummary: "{禁止語実体を含まない説明文}",
  retryGuidance: "CLAUDE.mdのリトライプロンプトテンプレートに従いsubagentを再起動してください",
  rawErrorForOrchestratorOnly: "{OrchestratorのログのみY用。成果物には転記しないこと}"
}
```

rawErrorForOrchestratorOnlyフィールドはOrchestratorがリトライプロンプトを構築するためのエラー詳細を含む。
このフィールドはsubagentへのプロンプトに直接転記せず、改善要求セクションの構築に使用するものとして設計する。

## 設定ファイル設計

### definitions.tsのsubagentTemplate構造

各subagentTemplateに追加するガイダンスブロックの配置位置と形式を以下のように定義する。

manual_testテンプレート（行890付近）への追加位置:
- 既存の「重複行回避」注意事項セクションの直後に新しい注意事項ブロックを追加する
- 追加ブロックの見出し: 「非空行数カウントの仕様と禁止語転記防止」
- ブロック内の記述順序: 非空行数の説明 → minLinesの値（20）の明示 → 転記防止ガイダンス

security_scanテンプレート（行902付近）への追加位置:
- 冒頭の注意事項セクションに既存項目と同列で追加する
- 追加する項目の内容: 非空行数カウントの仕様説明とminLinesの値（20）
- 転記防止ガイダンスを独立した注意事項として同セクションに追加する

e2e_testテンプレート（行926付近）への追加位置:
- manual_testと同じ構造で追加する（重複行回避セクションの直後に配置）
- minLinesの値としてe2e_testに設定された値を明示する

qualitySection生成ブロック（行1093-1114）への追加位置:
- 現在の「禁止パターン（グループ別参照）」セクションを「禁止パターン（実体リストと言い換えガイド）」に変更する
- グループ説明テキストの直後にコードフェンスで囲んだ禁止語実体リストを追加する
- コードフェンス後に安全な代替表現リストと転記防止警告を追加する

### CLAUDE.mdの変更箇所の構造設計

CLAUDE.mdで変更する3箇所の構造設計を以下に定義する。

変更箇所1: subagent起動テンプレートの禁止パターンサブセクション（行291-299付近）
- 既存の「禁止語を含む複合語も検出対象」という説明の後に追記する
- 追記内容の要素: 禁止語実体の列挙は成果物内で行わないこと、コードフェンス内でも転記は避けること
- 追記後のサブセクションのセクション密度が30%以上を維持すること

変更箇所2: 「禁止パターン（完全リスト）」セクション直後（行317以降）
- 新規サブセクション「安全な代替表現パターン」を追加する
- サブセクションの構成: ケース1（確定されていない状態）・ケース2（調査継続が必要な状態）・ケース3（架空データ）の3分類
- 各ケースに3種類以上の安全な表現例を列挙する
- 部分一致検出の回避方法として「語句を長くすることで意味を特定化する」という原則を最後に記述する

変更箇所3: リトライプロンプトテンプレートの前回失敗理由セクション（行348-356付近）
- 既存のコードブロック引用箇所の直後に転記防止の警告文を追加する
- 警告文はコードブロックの後に独立した段落として配置し、視覚的に識別しやすくする
- 警告文の内容はフィードバックループの形成メカニズムの説明と間接参照パターンの具体例を含む
- 警告文の長さは3行から5行程度とし、読みやすさを維持する

## 設計上の制約と整合性確認

本設計はFR-R6-1からFR-R6-4の各修正要件と整合していることを確認した。
各セクションの設計はNF-R6-1（追加トークン数200以内）を満たす範囲で記述している。
qualitySectionの構造変更はartifact-validator.tsへの変更を伴わない設計となっている。
変更対象ファイルはdefinitions.tsとCLAUDE.mdの2ファイルのみであり、スコープの最小化要件（NF-R6-4）を満たしている。
MCPサーバーのモジュールキャッシュ制約により、definitions.ts変更後は必ずMCPサーバーを再起動してから検証を行うことが前提条件となっている。
