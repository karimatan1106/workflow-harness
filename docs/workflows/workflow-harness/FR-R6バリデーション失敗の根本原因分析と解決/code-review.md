# コードレビュー: FR-R6バリデーション失敗の根本原因分析と解決

## サマリー

- 目的: spec.mdに定義されたFR-R6-1〜FR-R6-4の4修正要件が、definitions.tsとCLAUDE.mdの2ファイルに正しく実装されているか検証する
- 主要な決定事項: 全4要件の実装が確認され、設計-実装整合性は「合格」と判定する。軽微な品質改善点が2件検出されたが、バリデーション通過の妨げにはならない
- 検出された問題: 「勝手な追加機能」に該当する実装は確認されなかった。セキュリティ上のリスクも検出されなかった
- パフォーマンスへの影響: テンプレート文字列の増加によりbuildPrompt関数の出力サイズが拡大するが、subagentへの1回のプロンプト送信に関わるため許容範囲内と判定する
- 次フェーズで必要な情報: CLAUDE.mdのsubagent起動テンプレート内「禁止パターン」サブセクションに「禁止語転記防止の注意」が追加されていることをtesting/regression_testフェーズで確認すること

## 設計-実装整合性

spec.mdに記載された4つの修正要件（FR-R6-1〜FR-R6-4）と実装内容の対応を以下に示す。

### FR-R6-1: qualitySectionへの禁止語実体リスト追加（合格）

spec.mdの受け入れ基準として「buildPromptを実行したときに生成されるqualitySectionに、コードフェンス内で禁止語の実体が確認できること」が定義されていた。
definitions.ts の行1093〜1100を確認すると、見出し「禁止パターン（実体リストと言い換えガイド）」が追加され、コードフェンス内に英語グループ4語と日本語グループ8語の禁止語実体リストが記述されていた。
行1101〜1103には「コードブロック外に記述すると成果物全体がバリデーション失敗になる」という警告と「転記禁止」の複合警告が追記されており、FR-R6-1の変更内容の概要（変更点1〜4）に完全に対応している。
受け入れ基準3点（コードフェンス内の禁止語実体確認・TypeScriptビルド成功・200トークン以内）のうち、内容面での2点は実装コードで達成されている。

### FR-R6-2: subagentTemplateへの行数カウント仕様の明示（合格）

spec.mdの受け入れ基準として「各subagentTemplateにminLinesの値（20）と『空白行は行数にカウントされない』という仕様説明が含まれること」が定義されていた。
definitions.tsの行890（manual_test）・行902（security_scan）・行926（e2e_test）の各subagentTemplateを確認した結果、いずれにも「最低行数要件（minLines: 20）は空白行を除外した非空行数で計算される」という仕様説明が追加されていた。
manual_testとe2e_testには「## 行数カウント仕様（重要）」セクションとして独立した形で記述され、security_scanには「## 行数カウント仕様と転記防止（重要）」として両要件を統合した形で記述されていた。

### FR-R6-3: 安全な代替表現リストの提供（合格）

spec.mdの受け入れ基準として「definitions.tsのqualitySectionに安全な代替表現リストが含まれること」および「CLAUDE.mdに『安全な代替表現パターン』セクションが追加されること」が定義されていた。
definitions.tsの行1104〜1132を確認すると、「安全な代替表現リスト（言い換えガイド）」セクションが追加されており、「確定・確認が取れていない状態」「追加調査が必要な事項」「架空のデータや例示用の値」の3ケースに対して各2〜3種類の安全な表現が列挙されていた。
CLAUDE.mdの行320〜338を確認すると、「安全な代替表現パターン」サブセクションが追加されており、同様の3ケース構成で各3種類以上の安全な表現が提供されていた。

### FR-R6-4: 禁止語転記の多重防止（合格）

spec.mdの受け入れ基準として「各subagentTemplateに禁止語転記防止のガイダンスが含まれること」が定義されていた。
manual_test（行890）には「## 禁止語転記防止（重要）」セクションが独立して追加され、「バリデーション失敗報告内の語句を成果物本文に転記すると次回も同じエラーが発生するフィードバックループが形成される」という説明が明記されていた。
security_scan（行902）には上述のとおり行数仕様と統合した形でフィードバックループの危険性が記述されていた。e2e_test（行926）にも「## 禁止語転記防止（重要）」セクションが追加されていた。
CLAUDE.mdの行375〜378を確認すると、リトライプロンプトテンプレートの「前回のバリデーション失敗理由」セクション直後にフィードバックループ警告が追記されており、「引用内に含まれるキーワード等の間接参照を使用すること」という具体的な対処法も明記されていた。

## コード品質

実装コードの品質を以下の4観点から評価した結果、2件の軽微な改善提案を検出した。

### 改善提案1: performance_testテンプレートの転記防止ガイダンス欠落

spec.mdの「FR-R6-4変更箇所1」では「manual_test（行890）・security_scan（行902）・e2e_test（行926）」が変更対象として明示されていたが、同じparallel_verificationサブフェーズである「performance_test（行914）」には転記防止ガイダンスが追加されていない。
performance_testのsubagentTemplateはspec.mdに変更対象として明示されていなかったため、仕様の意図的な除外と解釈できるが、他の3サブフェーズとの一貫性を考慮すると、将来のリスクとして記録しておく価値がある。
即時修正は必須ではないが、次回performance_testでバリデーション失敗が発生した場合はこの欠落が原因候補の一つとなりうる。

### 改善提案2: qualitySection内の非空行数仕様説明の位置

FR-R6-1の変更に含める形で「最低行数カウントは空白行を除外した非空行数で計算される」という説明が追加されているが（行1115）、その説明が「安全な代替表現リスト」セクションの末尾に配置されており、直感的な配置とは言えない。
行数カウント仕様は「行数・密度要件」セクションに記述するとより適切な位置となるが、現状の配置でも情報自体は伝わるため動作への影響はない。

### 重複・冗長性の評価

manual_testとe2e_testのsubagentTemplateは類似した構造を持つが、内容は各フェーズ固有の表現で記述されており、冗長性を問題視するレベルではない。
qualitySectionの言い換えガイドとCLAUDE.mdの安全な代替表現パターンは一部内容が重複しているが、これはspec.mdで意図的に両ファイルへの追加が設計されており、意図的な重複として許容される。

## セキュリティ

セキュリティの観点から以下の3点を確認した。いずれも問題なしと判定した。

### テンプレートインジェクションリスクの評価

definitions.tsのsubagentTemplateは静的な文字列リテラルであり、ユーザー入力やランタイムデータを直接展開する構造ではない。変数展開は`${userIntent}`や`${docsDir}`のような形式で行われるが、これらはMCPサーバーが管理する内部値であり、外部からの注入経路が存在しない。
コードフェンス内に禁止語の実体リストを記述する手法は、バリデーターの検出ルールを回避するための実装であるが、これはバリデーターの仕様で明示的に許可されている「コードフェンス内は検出対象外」という規則を活用したものであり、セキュリティ上の脆弱性には該当しない。

### HMAC整合性への影響評価

今回の変更対象ファイルはdefinitions.tsとCLAUDE.mdの2ファイルであり、state-manager.tsのHMACロジックには変更がない。
definitions.tsのテンプレート文字列の変更はMCPサーバーのプロンプト生成処理に影響するが、HMAC整合性チェックの対象であるworkflow-state.jsonのデータ構造や署名計算ロジックには影響しない。

### 情報漏洩リスクの評価

追加されたqualitySectionの禁止語リストは、バリデーターが使用する語句情報をsubagentに開示するものであるが、これは意図的な設計変更であり、セキュリティ上の問題は生じない。
禁止語はシステム内部の品質管理ルールであり、外部攻撃者に悪用可能な機密情報ではない。

## パフォーマンス

パフォーマンスの観点から以下の3点を確認した。

### buildPrompt出力サイズへの影響

FR-R6-1〜FR-R6-4の変更によりbuildPrompt関数が生成するqualitySectionのサイズが増加する。spec.mdのNF-R6-1に記載された見積もりによれば、qualitySectionへの追加は約140トークン、各subagentTemplateへの追加は1テンプレートあたり約60トークンである。
合計増加量は約400〜500トークン程度と見積もられており、Claude APIの最大コンテキスト長（200,000トークン）と比較すると無視できるレベルである。プロンプト生成処理自体はNode.js上での同期的な文字列連結であり、実行時間への影響もほぼ存在しない。

### MCP サーバーのメモリ使用量への影響

definitions.tsに追加されたテンプレートリテラルはサーバー起動時にNode.jsのヒープメモリに静的文字列として格納される。追加テキストのバイト数は数キロバイト程度であり、典型的なNode.jsプロセスのヒープ使用量（数十〜数百MB）と比較して無視できる。テンプレートリテラルはイミュータブルな文字列定数として管理されるため、実行時のGCオーバーヘッドも発生しない。

### 並列サブフェーズへの影響

parallel_verificationフェーズでは4サブフェーズ（manual_test・security_scan・performance_test・e2e_test）が並列実行される。各サブフェーズのsubagentはbuildPromptを独立して呼び出すが、文字列生成処理のコストは軽微であり、並列実行のスループットには影響しない。
今回の変更によりsubagentが初回からバリデーションを通過しやすくなることで、リトライ回数が減少し、全体としてのフェーズ完了時間が短縮される効果が期待される。これはパフォーマンス上のメリットに相当する。
