## 検出された問題

### 問題1: プロンプトインジェクション耐性（深刻度: Medium）

- 問題の分類: buildPrompt関数が生成するプロンプト文字列は、taskName・userIntent・docsDirの3変数をresolvePlaceholders関数で単純な文字列置換によって埋め込んでいる。
- 影響を受けるコンポーネント: buildPrompt関数（definitions.ts の resolvePlaceholders呼び出し箇所）全体が対象であり、今回の変更ではなく既存実装に起因する。
- 評価の根拠: 攻撃者がtaskNameまたはuserIntentに特殊文字列（例: Markdownヘッダー記法や「##」から始まる誘導テキスト）を挿入した場合、subagentへのプロンプトの意図が変化する可能性がある。ただし、taskNameとuserIntentはClaude Desktop経由でユーザーが入力する値であり、外部の攻撃者が直接操作できる攻撃経路は現時点で存在しない。
- 今回のFR-R6変更との関係: FR-R6で追加された禁止語リスト（qualitySection内のコードブロック）や転記防止ガイダンスの文字列は定数であり、外部入力を埋め込む箇所ではない。今回の変更によってプロンプトインジェクション耐性が悪化した事実は認められない。
- 推奨対応: 今後のバージョンでuserIntentのサニタイズ処理（特殊文字のエスケープまたはホワイトリスト検証）を実装することを将来的な改善事項として記録する。今回の変更に対する緊急対応は不要と判定する。

### 問題2: 禁止語リストの平文露出（深刻度: Low）

- 問題の分類: buildPromptが生成するプロンプトには禁止語12語がコードフェンス内に列挙される。このプロンプトはsubagentに送信されるため、subagentのログや会話履歴に禁止語が記録されることになる。
- 影響を受けるコンポーネント: qualitySectionの禁止語コードブロック部分（definitions.ts 行1096〜1100）と、それを受け取るsubagentの動作環境。
- 評価の根拠: 禁止語リストはセキュリティ上の機密情報ではなく、バリデーション品質管理のためのオープンな仕様である。リストが露出しても攻撃者が利用できる悪用シナリオは極めて限定的であり、実害はほぼ存在しない。
- 今回のFR-R6変更との関係: FR-R6でコードフェンス内への格納（検出対象外）という安全な設計が明示的に採用されており、むしろセキュリティ上の改善である。
- 推奨対応: 現在の実装を維持してよい。禁止語リストを暗号化または難読化する必要はなく、対策不要と判定する。

### 問題3: HMAC整合性への影響確認（深刻度: Low）

- 問題の分類: FR-R6の変更対象ファイルがHMAC整合性検証の対象ファイルに該当するかどうかを確認した。
- 影響を受けるコンポーネント: state-manager.tsが管理するworkflow-state.jsonのHMAC保護は、state-manager.ts・definitions.ts・artifact-validator.tsのコードが変更された場合にMCPサーバーの再起動を要求する運用ルールに関連する。
- 評価の根拠: definitions.tsの変更はbuildPromptおよびbuildRetryPromptのqualitySectionへの文字列追加であり、HMAC署名の対象であるワークフロー状態データ（workflow-state.json）の構造には影響しない。state-manager.tsのHMACキー生成ロジックは変更されていないため、既存タスクのHMAC整合性は維持される。
- 今回のFR-R6変更との関係: definitions.tsを変更した場合はMCPサーバーの再起動が必要（CLAUDE.md「コアモジュール変更後はMCPサーバーを再起動」ルール）であるが、これはHMAC破壊を防ぐためではなくモジュールキャッシュのリフレッシュのためである。再起動後は正常にHMAC検証が継続される。
- 推奨対応: MCPサーバーを再起動して変更を反映させること。それ以外の追加対応は不要と判定する。

### 補足: テンプレート文字列のインジェクション安全性

definitions.tsのbuildPrompt関数では、qualitySection内の禁止語リストをバッククォート3個で囲まれたコードブロックとして文字列連結している。
この実装では禁止語がプロンプト本文として解釈されるのではなく、コードブロック内の参照情報として扱われる意図が明確に設計されている。
subagentがコードブロック内の禁止語を成果物本文に転記しないよう、行1101以降に三重の警告文が付加されており、フィードバックループの発生を防ぐ多層防御が実装されている点はセキュリティ上の改善として評価できる。
CLAUDE.md への「禁止語転記防止」および「フィードバックループ警告」の追加も同様に、subagentの誤動作を抑制するドキュメント上の防御策として機能する。
