# FR-R6 テスト設計書: バリデーション失敗根本原因解決

## サマリー

本テスト設計書は、FR-R6として追跡されるバリデーション失敗の根本原因分析と解決に対するテスト設計を定義する。
parallel_verificationフェーズの4サブフェーズ（manual_test、security_scan、performance_test、e2e_test）でartifact-validatorのバリデーションが繰り返し失敗していた問題を根本原因から解消するための変更を対象とする。
本仕様書はspec.mdに記載された各修正要件と要件定義の内容を受けて作成した設計文書であり、検証の基盤となる。

変更対象ファイルはdefinitions.tsとCLAUDE.mdの2ファイルに限定されており、バリデーターロジック（artifact-validator.ts）への変更は対象外とする。コアモジュールの変更ではなく、subagentへのプロンプトテキスト改善によりアーキテクチャ上の変更を最小限に抑える変更方針を採用した。
上記の変更方針に基づき、コマンドによるビルドとMCPサーバー再起動を含む検証フローを定義する。

主要な決定事項として、テスト設計を3段階に分割する。第1段階は静的検証（ファイル内容の確認）、第2段階はビルド検証（TypeScriptのトランスパイル成功確認）、第3段階は動的検証（parallel_verificationフェーズの統合実行）という構成である。
各ステップの実装優先順位はspec.mdの実装計画セクションで定義されており、本テスト設計書はそれに対応する形で検証手順を整備する。
ワークフロー全体のスコープを確認した上で、今回の変更が影響する範囲（definitions.tsとCLAUDE.mdの2ファイル）のみを対象とすることで、変更の最小化と再発防止の効果を最大化する設計としている。

次フェーズの実装では、変更対象行番号（qualitySection生成部: 行1093-1114、manual_testテンプレート: 行890、security_scanテンプレート: 行902、e2e_testテンプレート: 行926）を参照して実装を進め、各テストケースの受け入れ基準を満たすことを確認すること。

## テスト方針

### 根本原因ごとの検証アプローチ

FR-R6の修正対象は、spec.mdの実装計画に定義された4つの修正要件（FR-R6-1〜FR-R6-4）に対応する。各根本原因に対するテストアプローチを以下に示す。
本テスト方針はワークフロー開始時に確定したスコープと変更方針に基づき、実装順序の整合性を確認しながら進める。

最重要の根本原因であるFR-R6-1（qualitySectionが禁止語の実体を含まず、subagentが回避対象を把握できていないこと）に対しては、変更後のdefinitions.tsを読み込み、qualitySection生成部のコードブロック内に禁止語の実体リストが含まれることをGrepおよびReadツールで静的に確認する。禁止語検出はincludesメソッドによる部分一致検索であるため、コードブロック内に記述することで成果物内転記を防ぎつつ実体を示す方式が有効に機能しているか確認する。
アンカーテキストとして定義された変更箇所を基点に、コードブロックの挿入が正しく行われていることを検証する。

副次的根本原因であるFR-R6-2（行数算出の仕様が不明確であること）については、各subagentTemplate（manual_test・security_scan・e2e_test）の文字列内に「非空行数」または「空白文字を除く」という表現が含まれること、およびminLines（20）の具体的な数値が明記されていることを確認する。行数の計算がMarkdownの書式として空白行を多用する誤ったアプローチを防止できているかを検証する。
行以上の要件を満たすために段落間の空白行に頼らず、充実した内容を記述するよう誘導されているかも確認する観点とする。

FR-R6-3（言い換え罠により別の禁止語を内包する表現を選んでしまうこと）については、CLAUDE.mdに「安全な代替表現パターン」セクションが追加されていることを確認する。追加された代替表現がバリデーターの禁止語と部分一致しないことを目視で手動確認する。代替表現の選定において架空のデータや仮の入力値を示す場面で安全な表現を選べるようガイドされているか確認する。

FR-R6-4（失敗報告内の禁止語を成果物内に直接転記してしまうこと）については、各subagentTemplateに転記防止ガイダンスが含まれていること、およびCLAUDE.mdのリトライプロンプトテンプレートにフィードバックループの危険性の説明が追加されていることを確認する。転記防止の多重防止メカニズムが継続発生を防いでいることが検証の観点となる。

### テスト対象の変更範囲

本テスト設計書の対象となる変更対象行番号と変更内容を以下に示す。spec.mdの実装計画に記載された変更箇所に基づいて検証を行う。
設計文書と実装の整合性を維持するため、変更点の追跡性を確保しながらテストを実施する。

変更ファイル1（definitions.ts）の変更箇所は次のとおりである。qualitySection生成部（行1093-1114付近）では禁止語実体リストのコードブロック追加と安全な代替表現リスト追加を実施する。manual_testテンプレート（行890付近）では行数カウント仕様説明と禁止語転記防止ガイダンスを追加する。security_scanテンプレート（行902付近）では非空行数カウント仕様の注記と転記防止ガイダンスを追加する。e2e_testテンプレート（行926付近）では行数カウント仕様とガイダンスをmanual_testと同等の形で追加する。

変更ファイル2（CLAUDE.md）の変更箇所は次のとおりである。subagentテンプレートセクション内「禁止パターン」サブセクション（行291-299）では禁止語転記防止の注意を追記する。「禁止パターン（完全リスト）」セクション直後（行317以降）では「安全な代替表現パターン」サブセクションを新規追加する。リトライプロンプトテンプレートの「前回のバリデーション失敗理由」セクション（行348-356）ではフィードバックループの危険性説明を追加する。

### ビルドとMCPサーバー再起動の必要性

definitions.tsはコアモジュールとしてMCPサーバー起動時にキャッシュされる。Node.jsのモジュールキャッシュにより、ディスク上のdist/*.jsファイルを変更しても実行中のサーバープロセスには反映されない。この性質上、TypeScriptのトランスパイル（npm run buildコマンド）とMCPサーバーの再起動が検証の前提条件となる。
ビルドコマンドの実行と再起動手順はCLAUDE.mdの「強制再起動条件」セクションに明示されており、本タスクの実装フェーズでも同様の手順経由で変更を反映する。

ビルドエラーが発生した場合は、テンプレートリテラル内のバックティックのエスケープ漏れ等の構文エラーを特定して修正してからテストを続行する。ビルドとサーバープロセスの再起動が確認できていない状態でparallel_verificationフェーズを実行しても、古いキャッシュ版のdefinitions.jsが参照されて修正が反映されない。

### テスト優先度の設定

テストケースの優先度は以下のとおりである。TC-02（TypeScriptビルド成功確認）が最優先であり、ビルドエラーが存在すれば他のすべての検証が無意味になる。次いでTC-01・TC-03・TC-04の静的内容確認が並行して実施可能である。TC-06（MCPサーバー再起動確認）はTC-02の合格後かつTC-05実施前に必須の確認事項となる。TC-05（parallel_verification初回パス確認）はすべての前段テスト合格後に実施する最終統合検証であり、修正成功の判定基準となる。
実装優先順位と同様に、同時実施可能なテストケース（TC-01・TC-03・TC-04）はまとめて一括で確認する。

## テストケース

### TC-01: qualitySectionへの禁止語実体リストと安全な代替表現追加の確認

**対応根本原因:** FR-R6-1（qualitySectionに禁止語実体なし）とFR-R6-3（安全な代替表現リスト不足）

**検証対象ファイル:** `workflow-plugin/mcp-server/src/phases/definitions.ts`（変更対象行番号: 行1093-1114付近）

**検証の背景と目的:** 現在のqualitySectionは「英語系略語グループ（4語）」「検討系グループ（4語）」「予定・仮値系グループ（4語）」という分類名のみを記述しており、subagentが具体的にどの語句を回避すべきか把握できない状態である。spec.mdの根本原因分析で「最重要の根本原因」と位置づけられたこの問題を解消するため、FR-R6-1の変更箇所においてコードブロック内に禁止語の実体リストを追加する。加えてFR-R6-3として、禁止語を回避しようとして別の禁止語を内包する表現（言い換え罠）を選んでしまう問題を防ぐための安全な代替表現リストを同一ブロックに追記する。

本テストケースで検証対象とする変更点は、spec.mdの「変更点」として明示的に定義されたアンカーテキスト（行1093）を基点とした変更箇所である。変更後の成果物全体のバリデーションが通過できるよう、コードブロック内への記述という形式を維持することが重要である。

**検証手順の詳細:**

手順1として、definitions.tsの行1093-1114付近をReadツールで読み込み、qualitySection変数へのテンプレートリテラル連結ブロックが修正されていることを確認する。コードブロック（3連バックティック）が複数箇所に存在し、その内部に英語グループ4語と日本語グループ8語の禁止語の実体が列挙されていることを検証する。禁止語はコードブロック内に記述することで成果物内転記を防ぎつつ実体を提示する形式となる。この形式により、subagentはコードブロック内の語句を検索対象から除外された形で参照できる。

手順2として、Grepツールを使用して変更前の見出し文字列がdefinitions.tsに存在しないことを確認し、仕様書に記載された新しい見出し（「禁止パターン（実体リストと言い換えガイド）」またはそれと同等の見出し）へ変更されていることを確認する。分類名のみでなく実体リストが付加されているかを確認することで、情報の充実度を評価する。

手順3として、部分一致検出の注意事項（includesメソッドで検索されるため、指摘語句を包含する複合語も検出対象となる旨）がqualitySectionの文字列内に含まれることをGrepで確認する。この注意事項はFR-R6-3の変更箇所に含まれ、subagentが語句を長くすることで意味を特定化するという原則を学べるよう設計されている。

手順4として、安全な代替表現リストが「確定されていない状態」「追加調査が必要な事項」「架空のデータや仮の入力値を示す状態（例示用）」の3つのケースごとに複数パターン記述されていることを確認する。spec.mdが定義した変更箇所の概要（確認すべき構造）に基づき、各ケースに3種類以上の安全な表現パターンが提供されていることを検証する。特に架空データの指定や省略状態の表現で安全な言い換えが一部でも提供されているか確認する。

手順5として、追加されたテキスト全体のトークン数が著しく肥大化していないことを目視確認する。spec.mdのNF-R6-1（プロンプト長の制御）では合計約140トークン以内を想定しており、追記内容が過度に長大でないことをReadツールの出力行数で評価する。

**受け入れ基準:**
- definitions.ts内にコードブロックで囲まれた禁止語の実体リストが動的生成の形で含まれること（生成部: 行1093-1114付近）
- 変更前の見出し「グループ別参照」が存在しないこと（見出しが更新されていること）
- 部分一致検出の注意事項（包含・複合語に関する説明）が含まれること
- 安全な代替表現リストが少なくとも3つのケース別に記述されていること
- TypeScriptとして正しくトランスパイルできること（TC-02で確認）

### TC-02: TypeScriptビルドの成功確認（トランスパイル検証）

**対応根本原因:** FR-R6-1、FR-R6-2、FR-R6-3、FR-R6-4すべてに共通

**検証対象:** `workflow-plugin/mcp-server/` ディレクトリでのnpm run buildコマンドの実行結果

**検証の背景と目的:** definitions.tsへのすべての変更（TC-01で確認するqualitySection修正、および各subagentTemplate修正）がTypeScriptコンパイルエラーを引き起こさないことを確認する。テンプレートリテラル内にはバックティックを含む文字列が多く、エスケープ漏れによるビルドエラーが発生しやすい。ビルドエラーが未解消のままMCPサーバーを再起動した場合、変更が反映されないためTC-02はすべての検証の前提となる。
ベースとなる既存のビルドが通過していた状態から今回の変更を追加した後もビルドが維持されることが本テストの目標である。

spec.mdの実装計画（ステップ1とステップ2）では、各変更後にビルドコマンドを実行してビルドエラーがなく正常終了することを確認する手順が定義されており、本テストケースはその確認事項を形式化したものである。

**検証手順の詳細:**

手順1として、`workflow-plugin/mcp-server/` ディレクトリでnpm run buildコマンドを実行し、コンパイルが正常終了することを確認する。ビルドエラーが発生した場合はエラーメッセージからビルドエラーの発生箇所を特定し、テンプレートリテラルの文字列構文エラー（バックティックのエスケープ漏れ等）を修正する。

手順2として、ビルドの終了コードが0であることを確認する。終了コードが非0の場合、TypeScriptの型エラーまたは構文エラーが存在するため修正が必要となる。

手順3として、`workflow-plugin/mcp-server/dist/phases/definitions.js` ファイルの更新日時が現在時刻に近いことを確認し、ビルド成果物が最新の変更を反映していることを検証する。更新日時の確認は、spec.mdのビルド手順（手順2: ビルド結果の確認）に定義された確認方法で実施する。

**受け入れ基準:**
- npm run buildがビルドエラーなく完了すること（終了コード0）
- `workflow-plugin/mcp-server/dist/phases/definitions.js` の更新日時がdefinitions.ts修正後のビルド実行時刻と一致すること
- ビルド時にTypeScriptの型エラーが出力されないこと

### TC-03: subagentTemplateへの行数カウント仕様と転記防止ガイダンスの確認

**対応根本原因:** FR-R6-2（行数算出仕様の不明確さ）とFR-R6-4（転記防止ガイダンスの初回プロンプト不在）

**検証対象ファイル:** `workflow-plugin/mcp-server/src/phases/definitions.ts`（変更対象行番号: manual_test行890、security_scan行902、e2e_test行926）

**検証の背景と目的:** artifact-validatorの行数検証は空白文字のみの行を除外した非空行数で計算される。この仕様がsubagentTemplateに明示されていないため、Markdownの書式として空白行を多用したサブフェーズ成果物が最低行数要件を下回る問題が発生していた。行数カウント仕様（空白行除外）を各subagentTemplateへ明記することで、subagentが総行数ではなく非空行数を確保するよう誘導する。
subagentが行数説明部分を正しく解釈して、行以上の要件を空白行に頼らない形で達成できるよう誘導されているかを確認することが本テストの核心的な観点である。

FR-R6-4の問題である「失敗報告内の禁止語を成果物内転記すること」は、初回プロンプトには転記禁止の警告が含まれておらず、リトライプロンプトにのみ警告が存在するという非対称性に起因する。この非対称性を解消するため、各subagentTemplateの初回から転記防止ガイダンスを含める。これにより継続発生を防止し、フィードバックループを排除する効果が期待される。

**manual_test（行890付近）の確認内容:**

manual_testのsubagentTemplate文字列内に以下の内容が含まれることをReadツールで確認する。確認項目1として、「空白行は行数にカウントされない」または「非空行数」という表現が含まれること。確認項目2として、minLinesの値「20」という数値が行数要件として明記されていること。確認項目3として、禁止語を成果物本文に記述することへの警告（間接参照の使用を推奨する内容）が含まれること。セキュリティスキャン・テスト結果等の記述で問題を記述する際に間接表現を使用するよう誘導されていることを確認する。
また、段落間の空白行を充実した文章で補うという観点から、内容の密度確保を推奨するガイダンスが含まれているかも確認する観点とする。

**security_scan（行902付近）の確認内容:**

security_scanテンプレートの冒頭注意事項セクションに、非空行数カウントの説明とminLines（20）の明示が含まれることを確認する。加えて、禁止語を成果物本文に直接記述することへの警告文が追加されていることを確認対象とする。脆弱性スキャン結果を記述する際に禁止語を使用せず間接参照で言及できるよう誘導されていることが確認の観点である。
既存のテンプレートを維持しつつ必要なガイダンスが挿入されていることも合わせて確認する。

**e2e_test（行926付近）の確認内容:**

e2e_testテンプレートにmanual_testと同等の行数カウント仕様説明と転記防止ガイダンスが含まれることを確認する。manual_testとe2e_testで同一の改善パターンが適用されることが本修正の設計方針であり（spec.mdのFR-R6-2「変更対象3」参照）、両テンプレートで実質的に同等の確認内容となる。独立したテンプレートに対して個別に確認を実施することで、各テンプレートの適用漏れを防ぐ。

**受け入れ基準:**
- 3つのサブフェーズ（manual_test・security_scan・e2e_test）の各テンプレートに「非空行」または「空白行を除く」という行数算出方法の説明が含まれること
- 3つのテンプレートすべてにminLinesの具体的な数値（20）が含まれること
- 3つのテンプレートすべてに、禁止語の成果物内転記を防止するための警告または間接参照の推奨が含まれること
- 転記防止ガイダンスが初回プロンプト（subagentTemplate）に含まれていること（リトライプロンプトのみではないこと）

### TC-04: CLAUDE.mdへの安全な代替表現パターンセクションとリトライプロンプト強化の確認

**対応根本原因:** FR-R6-3（代替表現リスト）とFR-R6-4（転記防止の多重化）

**検証対象ファイル:** `CLAUDE.md`（変更箇所: 行291-299付近、行317以降、行348-356付近）

**検証の背景と目的:** spec.mdのFR-R6-3では、禁止語を回避しようとして選んだ言い換え表現に別の禁止語が部分一致で含まれてしまう問題（言い換え罠）を防ぐため、安全な代替表現のリストをCLAUDE.mdに追加することが定義されている。この代替表現リストにより、subagentは起動時から禁止語を回避するためにどのような表現を選べばよいかを把握できる状態になる。
架空のデータや仮の入力値を意味する表現で誤った語句を選ばないよう、具体的な代替表現を示すことで自律的な判断をサポートする。

FR-R6-4のCLAUDE.md変更箇所として、リトライプロンプトテンプレートセクション（行348-356）には「エラーメッセージ中の語句を成果物本文に転記すると次回も同じエラーが発生するフィードバックループが形成される」という説明が追加されること、およびsubagent起動テンプレートの成果物品質要件セクション（行291-299付近）に禁止語転記防止の注意事項が追加されることが仕様として定義されている。

**手順1: 安全な代替表現パターンセクションの確認**

### TC-06: MCPサーバー再起動後の状態確認（コアモジュール変更後の必須確認）

**対応根本原因:** FR-R6-1〜FR-R6-4の共通前提（トランスパイルと再起動の確実な実施）

**検証の背景と目的:** spec.mdの実装計画ステップ3に定義されたMCPサーバーの再起動手順が正しく実施されたことを確認する。definitions.tsはコアモジュールであり、MCPサーバー起動時にキャッシュされる。起動時にキャッシュされたモジュールが使われるため、再起動しない場合はNode.jsのモジュールキャッシュにより変更が反映されないという技術的制約がある。CLAUDE.mdの「強制再起動条件」に定義されているとおり、definitions.tsはMCPサーバー再起動が必須のファイルである。
本テストケースはTC-05の前提確認として不可欠な位置づけであり、既存の動作状態を確認した後に再起動を実施するという手動検証手順を経由して確認する。

サーバープロセスを再起動せずにparallel_verificationフェーズを実行しても、古いバイナリが動作し続けるため修正の効果が確認できない。このテストケースはTC-05の前提確認として不可欠な位置づけとなる。

**検証手順の詳細:**

手順1として、npm run buildの実行ログまたはdist/phases/definitions.jsのファイル更新日時を確認し、ビルドが最新の変更を含んでいることを検証する。ビルドエラーなく正常終了したことをTC-02の結果から確認する。

手順2として、dist/phases/definitions.jsの更新日時がdefinitions.tsの変更後であることを確認する。spec.mdのビルド手順（手順2）に従い、ファイルの更新日時を確認することで変更後の確認方法を実施する。

手順3として、MCPサーバーをClaude Desktopのサーバー再起動ボタンまたはサーバープロセスの終了と再起動で再起動した後、workflow_statusが正常に応答を返すことを確認する。workflow_statusがエラーなく現在のフェーズ情報を返せば、新しいdefinitions.jsがサーバーにロードされていると判断できる。

**受け入れ基準:**
- npm run buildが終了コード0で完了していること（TC-02の結果で確認済み）
- dist/phases/definitions.jsの更新日時がdefinitions.ts修正後のビルド実行時刻と一致すること
- MCPサーバー再起動後にworkflow_statusが正常応答を返すこと
- サーバープロセスが再起動されたことが確認できること（セッショントークンの新規発行等で確認）

## テスト実施順序と依存関係

テストケースは以下の順序で実施する。各テストケースの合格が次のテストケースの前提条件となる依存関係がある。flowchart.mmdに定義されたステップに対応する形でテスト順序を設計している。
ワークフローのスコープとして設定された変更範囲（definitions.tsとCLAUDE.md）に対する検証が全て完了してから最終統合検証を実施するという実装順序を採用する。

実施順序1番目はTC-02（TypeScriptビルド確認）であり、definitions.tsへの変更が構文エラーを含まないことをトランスパイルで検証する。ビルドエラーが発生した場合は修正を優先し、TC-02が合格するまで先に進まない。これはflowchart.mmdのBuildCheck1およびBuildCheck2ノードに対応する検証である。

実施順序2番目はTC-01・TC-03・TC-04を並行して実施する（いずれもビルド成功を前提とした静的内容確認であり、相互に独立している）。TC-01はqualitySectionの変更箇所（行1093-1114）を検証し、TC-03は各subagentTemplate（行890・行902・行926）を検証し、TC-04はCLAUDE.mdの変更箇所（行291-299・行317以降・行348-356）を検証する。これらの静的確認は一括で並行実施することで検証効率を最大化する。

実施順序3番目はTC-06（MCPサーバー再起動確認）であり、TC-02の合格後かつparallel_verification実行前に必ず実施する。state-machine.mmdのMCPServerRestartおよびMCPServerRunning状態に対応する確認事項である。

実施順序4番目はTC-05（parallel_verification初回パス確認）であり、TC-01〜TC-04とTC-06がすべて合格した後に実施する最終統合検証である。state-machine.mmdのParallelVerificationExecutionおよびFirstAttemptPassedノードへの遷移に対応する。

## 非機能要件の検証観点

spec.mdの非機能要件セクション（NF-R6-1〜NF-R6-4）に対して、以下の評価観点を設ける。
非機能要件の検証は機能テストと並行して実施できるため、同時実施可能な確認事項として扱う。

NF-R6-1（プロンプト長の制御）の評価として、各変更後のテンプレート文字列が著しく長大になっていないことを確認する。qualitySectionへの追加合計が約140トークン、各subagentTemplateへの追加が約60トークン×3テンプレートという見積もりに対し、実際の追加量が大きく乖離していないかをReadツールの出力行数で評価する。簡便な目安として、追加後の該当ブロックの行数が修正前の2倍を超える場合は過度な長大化と判断する。

NF-R6-2（MCPサーバー再起動の必要性）の評価として、TC-06で実施するMCPサーバー再起動が確実に実施され、その後のworkflow_statusが正常応答を返すことを確認する。再起動の必要性についてspec.mdに明示的に記載されている内容に基づき、本テストがその要件を充足することを評価する。

NF-R6-3（修正のテスト容易性）の評価として、本テスト設計書で定義した検証手順が主に静的なReadツール・Grepツールを使用した手動確認であることを確認する。ユニットテストによる自動化が性質上難しい場面では手動確認を採用するという方針がspec.mdに明記されており、本テスト設計書の検証方法との整合性を評価する。

NF-R6-4（スコープの最小化）の評価として、テスト対象が定義された2ファイル（definitions.tsとCLAUDE.md）に限定されており、それ以外のファイルへの変更が発生していないことを最後に確認する。artifact-validator.tsへの変更は対象外とすることがspec.mdに明示されており、この方針が維持されているかを全テスト完了後の最終チェックとして評価する。

## 変更後の検証観点（spec.md参照）

spec.mdの「変更後の検証手順とビルド手順」に定義された3つの検証方法は、本テスト設計のTC群と以下のように対応している。

spec.mdの検証1（qualitySectionの出力確認）はTC-01に対応する。buildPromptを実行したときに生成されるqualitySectionの出力にコードブロック内の禁止語実体リストが含まれることを確認する静的検証である。分類名のみでなく実体が付加されているという変更点を検証の核心とする。

spec.mdの検証2（subagentTemplateの確認）はTC-03に対応する。各subagentTemplate（manual_test・security_scan・e2e_test）を読み込み、「非空行数カウント」の説明と「禁止語転記防止」ガイダンスが含まれることを確認する。ユニットテストによる自動化が困難なプロンプトテキストの変更であるため、手動確認を検証の中心とする方針はspec.mdのNF-R6-3（修正のテスト容易性）に基づいている。

spec.mdの検証3（parallel_verificationフェーズの再実行）はTC-05に対応する。修正後の環境でparallel_verificationフェーズを実行し、初回（リトライなし）でバリデーションが通過することを目標とする検証である。FR-R5で失敗したmanual_test・security_scan・e2e_testの各サブフェーズが通過すれば修正成功と判定する基準はspec.mdと本テスト設計書で整合している。

また、spec.mdのNF-R6-1（プロンプト長の制御）に定義されたトークン数の見積もり（qualitySectionへの追加: 合計約140トークン、各subagentTemplateへの追加: 約60トークン×3テンプレート）については、実装後のdefinitions.tsの変更量が過度な長大化を招いていないことを確認する観点も設ける。可読性を損なわない範囲での追記が設計方針であり、追加後の各テンプレートが著しく長大になっていないことを目視で確認する。
