# FR-R6 requirementsフェーズ: バリデーション失敗の根本原因分析と解決

## サマリー

- 目的: FR-R5タスクのparallel_verificationフェーズで繰り返し発生したバリデーション失敗を根本原因レベルで解消するため、4つの修正要件を定義する
- 主要な決定事項: definitions.tsのbuildPromptが生成するqualitySectionには禁止語の実体が含まれず、subagentが具体的な回避対象を把握できていないことが最重要の根本原因と判定した。加えて行数カウント仕様の不明確さ、言い換え罠（複合語による部分一致検出）、禁止語の直接転記という3つの副次的な根本原因も合わせて解消する。
- 次フェーズで必要な情報: definitions.tsのbuildPrompt関数の変更対象行番号（qualitySection生成部）、各subagentTemplateの変更対象行番号（manual_test・security_scan・e2e_test）、CLAUDE.mdのリトライプロンプトテンプレートの変更対象行番号

---

## 機能要件

### FR-R6-1: buildPromptのqualitySectionに禁止語の実体を列挙する

definitions.tsのbuildPrompt関数が生成するqualitySectionは、現在「英語系略語グループ（4語）」「検討系グループ（4語）」「予定・仮値系グループ（4語）」というグループ分類名のみを記述しており、subagentは具体的にどの語句を使用してはならないかを把握できない状態になっている。

この問題を解消するため、qualitySectionの禁止パターンセクションに以下の変更を行う。

- 変更点1: グループ分類名の説明に加えて、コードフェンスで囲まれた語句の実体リストを付加する
- 変更点2: 「これらの語句は部分一致（includesメソッド）で検出されるため、語句を含む複合語（例: 指摘語句を包含する長い表現）も検出対象となる」という注意を明記する
- 変更点3: 言い換え時に使用しても安全な代替表現パターン（例: 「確定していない状態」の代わりに「検証対象の入力値」「型が確定していない」等）を列挙する
- 変更点4: 禁止語に言及する際は必ず間接参照（「指摘された語句」「バリデーターが検出した語句」等）を用いることを明記し、成果物本文への直接転記を禁止する旨を強調する

この変更はdefinitions.tsのbuildPrompt関数内のqualitySection生成ロジックに対して行う。CLAUDE.mdの既存の「禁止パターン（完全リスト）」セクションとの整合性を保つこと。

### FR-R6-2: 行数カウント仕様を全subagentTemplateで明示する

artifact-validator.tsの行数検証は「content.split('\n').filter(line => line.trim().length > 0).length」で非空行数を算出する。この仕様により、Markdownの書式として挿入した空白行はカウントに含まれない。

security_scanのsubagentTemplateには行数ガイダンスが存在するが、「空白行を除外した非空行数」であることが明示されていない。他のsubagentTemplate（manual_test・e2e_test等）にはそもそも行数ガイダンス自体が不足している。

この問題を解消するため、各subagentTemplateに以下の変更を行う。

- 変更点1: 「行数カウントは空白行（空白文字のみの行を含む）を除いた非空行数で計算される」という仕様説明を追加する
- 変更点2: 「コードフェンス（\`\`\`で囲まれた範囲）内の行は行数カウントに含まれるが、禁止語の検索対象からは除外される」という点も併記する
- 変更点3: 「Markdownの書式として空白行を多用すると非空行数が不足しやすい。段落間の空白行を最小限にするか、内容を充実させて総行数を増やすこと」という実践的な指針を追加する
- 変更点4: 各フェーズの最低行数要件（minLinesの値）をsubagentTemplateに明示する

変更対象はmanual_test・security_scan・e2e_test・code_reviewの各subagentTemplateとする。buildPromptのqualitySectionにも同様の説明を追記することで、すべてのフェーズで一貫して伝わるようにする。

### FR-R6-3: 言い換え罠を回避する「安全な代替表現リスト」を提供する

禁止語の部分一致検出の性質上、禁止語を回避しようとして使用した言い換え表現が別の禁止語を内包することがある。例として、「確定していない状態」を表現しようとした語句が禁止語リスト中の語句と部分一致するケースが報告されている。

この問題を解消するため、buildPromptのqualitySectionに以下を追加する。

- 変更点1: 「確定・確認が取れていない状態」を安全に表現するための代替パターンを列挙する（例: 「検証対象の入力値」「型が確定していない」「指定が省略されている」等）
- 変更点2: 「追加調査が必要な事項」を安全に表現するための代替パターンを列挙する（例: 「分析を継続する必要がある項目」「追加の検証が求められる箇所」等）
- 変更点3: 「架空のデータや仮の入力値」を安全に表現するための代替パターンを列挙する（例: 「例示用のデータ」「テストに使用する値」等）
- 変更点4: 禁止語の複合語については「語句の一部として含まれる場合も検出対象となる」という警告を明記し、短い語ほど多くの複合語を汚染することを説明する

この変更はCLAUDE.mdの「禁止パターン（完全リスト）」セクション直後に「安全な代替表現パターン」サブセクションとして追記することと、definitions.tsのqualitySectionに同じ情報を動的生成で含めることの両方を実施する。

### FR-R6-4: 禁止語の成果物内転記を多重防止する

subagentがバリデーション失敗報告を参照する際、禁止語を成果物本文に転記してしまう問題が継続して発生している。現在のリトライプロンプトテンプレートには「⚠️ 上記コードブロック内のエラーメッセージは参照情報です。エラーメッセージに含まれる禁止語を成果物本文に直接転記しないでください。」という警告が存在するが、初回プロンプトには同等の警告が含まれていない。

この問題を解消するため、以下の変更を行う。

- 変更点1: 各subagentTemplateの冒頭または成果物品質要件セクションに「禁止語に言及する場合は必ず間接参照（『指摘された語句』『バリデーターが検出した語句』等）を用いること。禁止語を成果物本文に直接記述すると、その記述自体がバリデーション失敗を引き起こす」という警告を追加する
- 変更点2: CLAUDE.mdのsubagent起動テンプレートの成果物品質要件セクションに「禁止語の列挙は成果物内では行わず、コードフェンス内であっても禁止語の実体を書かないこと」という注意を追記する
- 変更点3: definitions.tsのqualitySectionに「セキュリティスキャン・テスト結果等のフェーズで検出された問題を記述する場合、問題の種類は『バリデーターが警告するパターン』等の間接表現で記述すること」という実践的な指針を追加する
- 変更点4: CLAUDE.mdのリトライプロンプトテンプレート（「前回のバリデーション失敗理由」セクション）の警告文を強化し、「エラーメッセージ中の語句を改善要求セクションや成果物本文に転記すると次回も同じエラーが発生する」というフィードバックループの危険性を明示する

## 非機能要件

### NF-R6-1: 既存の正常動作フェーズへの影響を排除する

今回の修正はdefinitions.tsのbuildPrompt関数とCLAUDE.mdのsubagentテンプレートセクションを変更する。これらの変更は既存の全フェーズのsubagentに影響するため、修正後に正常動作していたフェーズが意図せず破壊されないことを要件とする。

具体的には、qualitySectionへの追記によって生成されるプロンプトが過度に長大化しない設計とすること。追記内容は200トークン以内に収め、既存の成果物品質要件セクションの可読性を損なわない形式にすること。

### NF-R6-2: 修正後のMCPサーバー再起動の必要性を明確化する

definitions.tsはMCPサーバー起動時にキャッシュされる。修正内容がdefinitions.tsに含まれる場合、変更を反映するにはMCPサーバーを再起動する必要がある。実装フェーズでの手順として「npm run buildでトランスパイル後にMCPサーバープロセスを再起動し、workflow_statusで現在フェーズを確認する」ことを非機能要件として定義する。

CLAUDE.mdの修正はMCPサーバー再起動を必要としないが、次回のClaude Desktopセッション開始から有効になる点に留意すること。

### NF-R6-3: 修正のテスト容易性を担保する

今回の修正はプロンプトテキストの変更が中心であり、ユニットテストでの検証が難しい。そのため修正の検証方法として以下を採用する。

- 検証方法1: 修正後のdefinitions.tsでbuildPromptを実行し、qualitySectionの出力に禁止語の実体リストが含まれることを手動確認する
- 検証方法2: 修正後のsubagentTemplateを読み込み、行数カウント仕様の説明が冒頭または品質要件セクションに存在することを手動確認する
- 検証方法3: FR-R5で失敗したmanual_test・security_scan・e2e_testの各フェーズを修正後の環境で再実行し、初回（リトライなし）でバリデーションを通過することを目標とする

### NF-R6-4: 変更範囲のスコープを最小化する

本タスクの目的はバリデーション失敗の再発防止であり、ワークフロー全体のアーキテクチャ変更は対象外とする。変更対象ファイルは以下の2ファイルに限定する。

- `workflow-plugin/mcp-server/src/phases/definitions.ts`: buildPrompt関数のqualitySection生成ロジック
- `CLAUDE.md`: subagentテンプレートセクション、禁止パターンセクション、リトライプロンプトテンプレートセクション

artifact-validator.tsの変更は原則として対象外とする。バリデーターのロジック自体に問題があることは確認されておらず、subagentへのガイダンスを改善することで対処できるためである。
