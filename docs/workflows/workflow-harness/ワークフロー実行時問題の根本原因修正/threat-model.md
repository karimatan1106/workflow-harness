# ワークフロー実行時問題の根本原因修正 - 脅威モデル

## サマリー

本脅威モデルは、ワークフロー実行時問題の根本原因修正（FR-1〜FR-5）に対する
セキュリティリスクを包括的に分析する。修正対象は `definitions.ts`（MCP サーバーの
フェーズ定義）および `CLAUDE.md`（AI 指示書）の2ファイルである。

主要な変更内容として、testing テンプレートへの sessionToken 取得方法・生出力要件の明記と、
testing および regression_test 両テンプレートへのワークフロー制御ツール禁止指示の追加、
そして CLAUDE.md の承認フェーズ記述の修正（4フェーズへの更新）が含まれる。

セキュリティ観点での重要な変化点を3点に絞ると、まず sessionToken の言及箇所が
testing・regression_test テンプレートに追加されることで、トークン管理の注意点が増加する。
次に、ワークフロー制御ツール禁止指示はむしろ攻撃面を縮小する方向の変更であり、
全体としてセキュリティ態勢が改善する。最後に、CLAUDE.md の記述修正は Orchestrator の
認識精度を向上させるが、MCP サーバー側の実装を変えないため技術的な影響はない。

次フェーズ（planning）では、各テンプレートへの追記内容が本脅威モデルの
セキュリティ要件を満たしていることを確認しながら実装仕様を策定すること。

---

## 脅威シナリオ

### TS-1: sessionToken の不正利用による権限昇格

**対象修正**: FR-2（sessionToken 取得方法の明示）

FR-2 の実装により、testing テンプレートに sessionToken の取得方法と使用方法が
明示される。これにより、テンプレートを参照する subagent が sessionToken の存在を
認識しやすくなる一方、誤った使用（ワークフロー制御ツールへの転用）が発生するリスクも生まれる。

具体的な攻撃経路として、subagent がテスト実行結果の標準出力にプロンプトインジェクション
文字列を埋め込み、次回の subagent 起動時のプロンプトに sessionToken 値が流れ込む
シナリオが挙げられる。このシナリオが成立するには、外部のテスト実行結果を
subagent が無加工でプロンプトに展開するという条件が必要であり、現実的な可能性は低い。

sessionToken の有効期間はセッション単位に制限されているため、漏洩しても
別セッションで再利用することは設計上困難である。Orchestrator が引数として渡す設計を
維持することで、テンプレート本文への値の直接埋め込みを回避できる。

---

### TS-2: プロンプトインジェクションによるワークフロー制御禁止指示の回避

**対象修正**: FR-4（testing 制御禁止）・FR-5（regression_test 制御禁止）

FR-4・FR-5 で追加される禁止指示はプレーンテキストの指示であり、
subagent が処理するテスト出力や外部ファイルに細工されたプロンプトが含まれる場合、
禁止指示を無効化したうえで `workflow_next` 等を呼び出す誘導が試みられる可能性がある。

前回の実障害では regression_test subagent が `workflow_next` を連続して呼び出し、
parallel_verification の全4サブフェーズを実行したことが確認されている。
この事象の根本原因はテンプレートに禁止指示が存在しなかった点にあり、
FR-4・FR-5 の追加はこの問題への直接的な対策である。

単一の禁止指示テキストに依存するのは防御層として不完全であるため、
MCP サーバー側での sessionToken 認可チェックと組み合わせた多層防御が重要である。

---

### TS-3: workflow_record_test_result への改ざん出力の注入

**対象修正**: FR-3（生出力要件の明記）

`workflow_record_test_result` の `output` パラメータに実際のテスト実行結果ではなく
細工された文字列を渡すことで、テスト通過状態を偽装しようとする行為が考えられる。
具体的には全件パスを示す偽の集計行を手作りして `output` として渡すシナリオや、
subagent が意図せず出力を要約・短縮することで `validateTestAuthenticity` 検証を
誤って通過させるシナリオがある。

前者は悪意ある行為であり、後者は善意ある誤動作である。FR-3 は主に後者（誤動作の防止）
を目的とした要件であり、「100文字以上の完全な生出力」という明示的な基準を設けることで
短縮・要約が誤検証通過につながるリスクを低減する。

MCP サーバー側に `validateTestAuthenticity` 検証が実装されているため、
前者（悪意ある偽装）についても技術的な保護が存在する。

---

### TS-4: CLAUDE.md 承認フェーズ記述更新後の移行期間中の誤動作

**対象修正**: FR-1（承認フェーズ記述の更新）

CLAUDE.md を更新して requirements・design_review・test_design・code_review の
4フェーズが承認必要であることを明示すると、更新後に起動した Orchestrator は
test_design フェーズで `workflow_approve type:test_design` を呼び出すようになる。

更新前のセッションが継続している状態（CLAUDE.md を参照せずにキャッシュした状態）で
test_design フェーズに到達した場合、旧認識のまま承認をスキップしようとするリスクがある。
ただし MCP サーバー側の `REVIEW_PHASES` が実装の正として機能しているため、
承認なしに次フェーズへ進もうとしてもサーバー側でブロックされる。

この変更はセキュリティ上の脆弱性ではなく、ドキュメントと実装の整合性改善であり、
技術的リスクは低い。

---

### TS-5: definitions.ts テンプレート変更による既存テストのリグレッション

**対象修正**: FR-2〜FR-5（definitions.ts への追記全般）

`definitions.ts` のテンプレート文字列に追記を行う際、誤って禁止パターンワードを
含む文字列を記述した場合、artifact-validator のテストが失敗するリスクがある。
また、正規表現やバリデーションロジックに誤った変更を加えた場合には、
既存の912件のテストスイートが壊れる可能性がある。

本修正はテンプレート文字列の追記のみであり、実行ロジックは変更しないため、
リグレッションリスクは低いと評価される。ただし `definitions.ts` は MCP サーバーの
コアモジュールであり、変更後は必ず `npm run build` でトランスパイルを実施し、
MCP サーバーを再起動してから動作確認することが必須である。

---

## リスク評価

### 評価基準

可能性（Likelihood）と影響度（Impact）をそれぞれ3段階（Low・Medium・High）で評価し、
総合リスクスコアを算出する。可能性の評価軸は、攻撃または誤動作が発生する現実的な
可能性とする。影響度の評価軸は、事象が成功した場合にワークフロー整合性・セキュリティ
状態に与える損害の大きさとする。

---

### TS-1 のリスク評価（sessionToken 不正利用）

- **可能性**: Low（テスト出力を意図的に操作できる状況は限定的、セッション間での転用も困難）
- **影響度**: High（成功した場合、認可外の MCP ツール呼び出しが可能になるため影響は大きい）
- **総合リスク**: Medium（HMAC によるセッション保護と設計上の制約により現実的リスクは低減されている）
- **対策方針**: sessionToken はプロンプト本文への直接埋め込みを避け、安全な引数渡しの設計を維持する

---

### TS-2 のリスク評価（プロンプトインジェクション）

- **可能性**: Medium（外部ファイル・標準出力を subagent が処理する際に発生しうる経路が存在する）
- **影響度**: High（`workflow_next` 等の不正呼び出しが成功すればフェーズ飛ばしが発生し品質チェックが無効化される）
- **総合リスク**: High（前回の実障害として確認済みであり、最優先で対策が必要な脅威として評価する）
- **対策方針**: FR-4・FR-5 によるテンプレートレベルの禁止指示と、MCP サーバー側の認可チェックの多層防御を維持する

---

### TS-3 のリスク評価（テスト出力改ざん）

- **可能性**: Low（悪意ある改ざんの動機は薄く、主な発生経路は subagent の善意ある誤動作である）
- **影響度**: Medium（偽のテスト結果がワークフロー状態に記録されても、CI/CD パイプラインで検出可能）
- **総合リスク**: Low（validateTestAuthenticity が有効に機能している限り、改ざん試行は検出される）
- **対策方針**: FR-3 にて完全な生出力要件を明記し、subagent が意図せず出力を短縮する誤動作を予防する

---

### TS-4 のリスク評価（承認フェーズ記述更新）

- **可能性**: Low（CLAUDE.md の更新はリスクを低減する方向の変更であり、新規の脅威としての発現可能性は低い）
- **影響度**: Low（MCP サーバー側の REVIEW_PHASES が実装の正として機能しており、ドキュメントの記述が変わっても技術的な影響はない）
- **総合リスク**: Low（コードとドキュメントの整合性維持によって継続的に低減可能であり、現時点でのリスクは受容水準以内）
- **対策方針**: FR-1 の更新を確実に実施し、CLAUDE.md とコード実装の乖離をゼロに近づける

---

### TS-5 のリスク評価（テストリグレッション）

- **可能性**: Low（definitions.ts への追記はテンプレート文字列のみであり、実行ロジックへの影響経路は限定的）
- **影響度**: Low（テスト失敗が発生しても CI/CD で検出可能であり、デプロイ前に修正できる）
- **総合リスク**: Low（テストスイート912件による継続的な回帰テストが保護層として機能する設計になっている）
- **対策方針**: NFR-1 に従い、全修正完了後に `npm run build` および `npm test` を必ず実行して確認する

## セキュリティ要件

### SR-1: sessionToken の取り扱い基準

subagent テンプレートに sessionToken の取り扱いに関して以下の制約を明記することを必須とする。
sessionToken は Orchestrator から引数として受け取るものであり、subagent が
独自に生成・取得してはならない設計を保持する。
sessionToken の使用は `workflow_record_test_result` の呼び出し時に限定し、
ワークフロー制御ツール（workflow_next・workflow_approve 等）への転用を禁止する。
sessionToken が渡されなかった場合でも subagent は正常に動作しなければならず、
引数を省略して `workflow_record_test_result` を呼び出す手順をテンプレートに明示する。
テンプレート本文への sessionToken 値の直接埋め込みはログ漏洩のリスクがあるため禁止する。
この要件は FR-2 の実装において遵守すること。

---

### SR-2: ワークフロー制御ツールへのアクセス最小化

testing および regression_test subagent が呼び出せるワークフロー制御 MCP ツールは
`workflow_record_test_result` のみに限定されることを FR-4・FR-5 で明示する。
この設計方針は「最小権限の原則（Principle of Least Privilege）」に基づくものであり、
subagent が必要以上のワークフロー制御権限を持たないように設計する。
MCP サーバー側で sessionToken による認可チェックが実施されていたとしても、
テンプレートレベルの明示的な禁止指示は多層防御の重要な一層として機能する。
FR-4 と FR-5 の禁止指示は同一の文言・禁止ツールリストとし（NFR-2 に準拠）、
将来の変更時には両テンプレートを同時に更新することを設計制約として記録する。

---

### SR-3: テスト結果の真正性確保

`workflow_record_test_result` に渡す `output` パラメータは、テストコマンドの
生の標準出力をそのまま使用することを必須要件として definitions.ts に明記する。
出力の要約・編集・短縮は `validateTestAuthenticity` 検証に失敗する原因となるため禁止する。
テストフレームワーク（vitest・jest・pytest 等）が出力する集計行・パス結果・失敗詳細を
含む完全な出力をそのまま貼り付けることを FR-3 の要件として実装する。
出力の最低文字数（100文字以上）の基準を明記することで、空文字や極端に短い出力による
真正性検証の回避を防止する。
subagent が出力を処理する際に誤って短縮するケースへの防御として、
この要件を FR-3 のテンプレートに目立つ形式で記述することを推奨する。

---

### SR-4: 変更管理と再起動手順の遵守

`definitions.ts` を変更した場合は必ず `npm run build` でトランスパイルを実施し、
MCP サーバーを再起動してから変更の有効性を確認すること（NFR-4 に準拠）。
再起動しないままワークフローを継続すると古いキャッシュが動作し続け、
変更内容が一切反映されないまま次フェーズへ進む危険がある。
`CLAUDE.md` の変更は再起動不要であるが、definitions.ts と CLAUDE.md の変更を
同一セッションで実施する場合は、definitions.ts の変更後に必ず再起動を実施する。
再起動後は `workflow_status` で現在のフェーズを確認し、
同フェーズから作業を再開してフェーズの二重実行を防止すること。
この手順を省略すると、変更後のテンプレートが機能しているかどうかの確認が
取れない状態で次フェーズへ進む結果になり、品質確保が担保されなくなる。

---

### SR-5: FR-4 と FR-5 の禁止指示の一貫性維持

FR-4（testing テンプレート）と FR-5（regression_test テンプレート）の禁止指示は
同一の文言・禁止ツールリストを使用し、メンテナンス時の一貫性を保証すること。
どちらか一方の禁止ツールリストを変更する場合は、もう一方も必ず同時に更新する。
この一貫性要件は NFR-2 として要件定義に明記されており、コードレビューの
チェック項目として確認することを推奨する。
将来的に禁止ツールを追加・削除する際には、両テンプレートの文言を同期した上で、
変更差分を PR レビューで明示的に確認できる状態を維持すること。
文言の乖離が生じた場合、一方のテンプレートを参照する subagent と
もう一方を参照する subagent の間で挙動が異なる可能性があり、
トラブルシューティングを困難にするリスクがある。
