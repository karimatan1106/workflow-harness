## 2. テストデータ妥当性検証

テストデータの妥当性をゼロ幅文字、エンコードペイロード、ファイルパスの3カテゴリで検証しました。
全テストデータが実装の検証に適切であり、セキュリティ上の問題も含まれていないことを確認しています。
各カテゴリの詳細な検証結果を以下に記載します。
ゼロ幅文字は対象正規表現と完全に一致しており、エンコードペイロードは安全な"hello"文字列のみを使用しています。
ファイルパスは存在しないパスと実在パスの両方を使用し、正常系と異常系を網羅しています。

### 2.1 ゼロ幅文字の正確性

テストで使用されるゼロ幅文字は、bash-whitelist.jsのsanitizeZeroWidthChars関数の正規表現と完全に一致しています。
U+200B（ゼロ幅スペース）はWebコンテンツで頻出するため最優先でテストされています。
U+200C（ゼロ幅非接合子）はアラビア語やペルシャ語で使われる文字で、コマンド偽装に悪用される可能性があります。
U+200D（ゼロ幅接合子）は絵文字結合子として知られますが、コマンド文字列に混入するとセキュリティリスクとなります。
U+FEFF（バイトオーダーマーク）はファイル先頭に付与されることがあり、コマンドパースを妨害する可能性があります。
SEC1-05では4種類全てを同時に含む複合テストケースを実行し、正規表現の網羅性を検証しています。

### 2.2 エンコードコマンドペイロード

全テストケースで安全な文字列"hello"のエンコードを使用しています。
base64テスト（SEC2-01）ではaGVsbG8=を使用し、これはhelloのbase64エンコード出力と一致します。
printf hexテスト（SEC2-02）では\\x68\\x65\\x6c\\x6c\\x6fを使用し、各バイトがhelloのASCIIコードに対応しています。
echo octalテスト（SEC2-03）では\\150\\145\\154\\154\\157を使用し、各値がhelloの8進ASCIIコードです。
この設計により実際の攻撃ペイロードを含まずにエンコード検出機能を検証できます。

### 2.3 ファイルパス検証

SEC3テストスイートでは2種類のパスを使用しています。
存在しないパス/nonexistent/path/file.txtはfs.realpathSyncを確実に失敗させるために選択されました。
このパスはファイルシステムに影響を与えず、テストの再現性も高いです。
存在するパスとしてはworkflow-plugin/hooks/loop-detector.jsを使用しています。
プロジェクトルートからの動的パス解決を経由してフルパスを構築し、fs.realpathSyncの成功パスをテストします。

    warnMessage = args.join(' ');

### 7.1 軽微な改善点

#### 項目1: Spy パターンのエラー時復元（SEC3-01, SEC3-02）

**現在の実装**:
```javascript
try {
  const originalWarn = console.warn;
  console.warn = function(...args) { ... };
  // ... テスト実行 ...
  console.warn = originalWarn;  // try 内でのみ復元
  assert.ok(...);
} catch (e) {
  // エラー時、console.warn は復元されない可能性
  failed++;
}
```

**推奨**:
```javascript
try {
  const originalWarn = console.warn;
  console.warn = function(...args) { ... };
  try {
    // ... テスト実行 ...
    assert.ok(...);
  } finally {
    console.warn = originalWarn;  // 必ず復元
  }
} catch (e) { ... }
```

**重要度**: 低（テストスイートの規模が小さいため実害なし）
