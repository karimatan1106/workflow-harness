## サマリー

NEW-SEC-1/2/3専用テスト追加は、ゼロ幅Unicode文字サニタイズ、Fail-Closedデコード失敗時ブロック、ログ出力強化の3つのセキュリティ強化機能をテストするタスクである。
本脅威モデル分析は、テストコード追加に伴う潜在的なセキュリティリスクを特定し、対策を提案するものである。
テストデータの悪用、セキュリティ機構の無効化、情報漏洩の3つの観点からリスクを分析している。
テストは既存のセキュリティ機構を検証するためのものであり、本来の防御機制を強化する方向で設計されている。
テストコード自体がリスク導入媒体とならないことを確認し、安全なテスト実装の指針を示す。

---

## 脅威分析

### A. テストデータ（攻撃パターンのハードコード化）

**脅威**: テストコードに含まれる悪意あるコマンドパターン（base64エンコード、16進エスケープ、8進エスケープ等）が、ソースコード管理システムに保存され、将来的に本番環境構築時に誤ってコピー・ペーストされるリスク

**具体例**:
- base64エンコード化されたpython実行コマンド
- printf/echo 16進・8進エスケープシーケンス
- eval/exec/bash -c による間接実行パターン
- ゼロ幅Unicode文字を含む偽装コマンド

**発生シナリオ**:
1. テストファイルが外部ユーザー（コントラクタ、インターン）に共有される
2. テストコードから攻撃パターンが抽出され、実際の攻撃に転用される
3. マージ後、リポジトリ全体が公開リポジトリになり、gitコミット履歴から復元可能になる

**リスク等級**: 中程度（Medium）- 攻撃パターンが文書化されるリスク

---

### B. テストがセキュリティ検証機構の無効化を促さないか

**脅威**: テストコードが`checkBashWhitelist`関数を直接呼び出し、ブラックリストやホワイトリスト検証をテストで迂回できるようになるリスク

**具体例**:
- `checkBashWhitelist(command, phase)`の直接呼び出しにより、ホワイトリストを無視する可能性
- `detectEncodedCommand`をテストで実行し、デコード機構の動作を外部に露出
- `detectIndirectExecution`のテストで間接実行検出のロジックが明かされる

**発生シナリオ**:
1. テスト実行時に各関数が直接呼び出される
2. テストが本番コードの内部動作を詳細に検証する
3. 攻撃者がテストコードを参考に、検証機構のバイパス手段を企画する
4. テストカバレッジが100%に近くなり、隠蔽すべき内部ロジックが全て明かされる

**リスク等級**: 低程度（Low）- テストはホワイトボックステストで、セキュリティ検証機構の詳細な検証が必要なため、完全に隠蔽することは現実的でない

**対策**:
- テスト関数名を曖昧にせず、明確にセキュリティテストであることを示す
- テストコードにコメントを付けて、検証対象を明示し、攻撃パターンそのものではなく「検証方法」に焦点を当てる

---

### C. テストログの情報漏洩（NEW-SEC-3: ログ出力強化）

**脅威**: ログ出力強化テストで、機密情報（パスワード、秘密キー、内部ファイルパス）がログに記録され、ログファイルが外部に漏洩するリスク

**具体例**:
- `console.error`で攻撃パターンを出力するテスト
- `console.warn`で失敗原因を詳細に出力するテスト
- ファイルパスが完全パスで出力される場合、ディレクトリ構造が露出

**loop-detector.js の例**:
```javascript
console.warn(`[loop-detector] Warning: fs.realpathSync failed for path: ${filePath}, falling back to path.resolve. Reason: ${e.message}`);
console.error(`[loop-detector] Error: Both fs.realpathSync and path.resolve failed for path: ${filePath}, using fallback normalization. Reason: ${e2.message}`);
```

ここで`${filePath}`に秘密ファイルパス（`/Users/admin/.ssh/id_rsa`等）が含まれる場合、ログに記録される。

**発生シナリオ**:
1. テストで機密パスを含むファイルパスでテストを実行
2. ログファイルが`/tmp`や`.tmp/`に保存される
3. ログファイルが外部に漏洩（誤削除、バックアップ流出等）
4. 内部ディレクトリ構造や秘密ファイル場所が露出

**リスク等級**: 中程度（Medium）- ログ出力強化による情報漏洩リスク

**対策**:
- ログにファイルパスを出力する際は、相対パスまたはハッシュ化したパスを使用
- 機密ファイル名を含むパスはマスキングして出力
- テストで使用するファイルパスは、テンポラリディレクトリ配下のダミーパスに限定

---

### D. テスト実行による副作用（ファイルシステム操作）

**脅威**: テストが`fs.realpathSync`や`path.resolve`を呼び出す際に、実際のファイルシステム操作が発生し、本番データを誤削除するリスク

**具体例**:
- NEW-SEC-3テストで、`normalizeFilePath`関数が実際のファイルシステムにアクセス
- テスト中に`loop-detector-state.json`ファイルが実際に読み書きされる
- テストの状態が本番環境に影響を及ぼす可能性

**発生シナリオ**:
1. テストが`loadState()`を呼び出す
2. `STATE_FILE`（`.claude/state/loop-detector-state.json`）が実際に読み込まれる
3. テストが`saveState()`を呼び出す
4. 本番の状態ファイルが上書きされる
5. 無限ループ検出の状態がリセットされ、セキュリティ機構が一時的に機能しなくなる

**リスク等級**: 高程度（High）- テスト実行により本番セキュリティ状態が変更される可能性

**対策**:
- テスト用の独立した状態ファイルパスを使用
- テスト環境と本番環境のファイルシステムを分離
- テスト終了後に必ずクリーンアップを実施

---

### E. テスト環境でのセキュリティ機構の無効化（フラグ依存）

**脅威**: `SKIP_LOOP_DETECTOR`環境変数がテスト中に設定されたまま、テスト終了後もリセットされないため、本番でセキュリティ機構が無効のままになるリスク

**具体例**:
```javascript
// loop-detector.js に存在する環境変数チェック
if (process.env.SKIP_LOOP_DETECTOR === 'true') {
  // スキップ
}
```

テストで`SKIP_LOOP_DETECTOR=true`を設定して実行した後、明示的にリセットされない場合、本番環境でこの設定が残存する可能性。

**発生シナリオ**:
1. テスト実行スクリプトで`SKIP_LOOP_DETECTOR=true`を設定
2. テスト完了後に環境変数がリセットされない
3. 以降のテスト実行で無限ループ検出が無効のままになる
4. セキュリティ機構が機能せず、無限ループ問題が検出されない

**リスク等級**: 中程度（Medium）- テスト環境に限定される可能性が高いが、CI/CDパイプラインに影響を及ぼす場合がある

**対策**:
- テスト実行スクリプトで環境変数を明示的にリセット
- テストを`before/after`フック内で実行し、状態をクリーンアップ

---

### F. テスト用ダミーデータの永続化

**脅威**: テスト中に作成されたダミーデータ（ゼロ幅Unicode文字を含むコマンド等）がテストファイル内に永続化され、ソースコード上で攻撃パターンのカタログになるリスク

**具体例**:
```javascript
it('ゼロ幅Unicode文字のサニタイズ', () => {
  const maliciousCommand = 'echo\u200B\u200C\u200D"code"';  // ゼロ幅文字埋め込み
  const result = sanitizeZeroWidthChars(maliciousCommand);
  assert.strictEqual(result, 'echo"code"');
});
```

このテストコードは、ゼロ幅文字の埋め込み方法を文書化してしまい、攻撃者の参考資料になりうる。

**発生シナリオ**:
1. テストコードがソースコード管理に登録される
2. リポジトリが外部に公開される
3. 攻撃パターンがカタログ化され、他の攻撃に転用される

**リスク等級**: 低〜中程度（Low to Medium）- テストコードは既に公開の可能性が高いが、明示的に攻撃パターンを記述することは避けるべき

**対策**:
- テストデータを外部ファイルに分離
- テストコード内にコメントを付けて「検証対象のセキュリティ脅威」であることを明示
- ダミーコマンドは実際の攻撃パターンを直接参照するのではなく、「パターン名」で表現

## リスク評価行列

以下の表は各脅威の重大度、発生可能性、リスクレベル、対策優先度を総合的に評価したものである。
テスト実行による副作用（脅威D）が最もリスクが高く、最優先で対策を実施する必要がある。
ログの情報漏洩（脅威C）は中程度のリスクだが、本番環境への影響を考慮して高優先で対策する。
テストデータの悪用（脅威A）とセキュリティ機構の無効化（脅威E）は中程度の優先度で対応する。
セキュリティ検証機構の無効化（脅威B）は低リスクであり、テストコード内のコメント整備で十分である。

| 脅威 | 重大度 | 発生可能性 | リスク | 対策優先度 |
|------|--------|---------|--------|----------|
| A. テストデータの悪用 | 高 | 低 | 中 | 中 |
| B. セキュリティ検証機構の無効化 | 中 | 低 | 低 | 低 |
| C. ログの情報漏洩 | 中 | 中 | 中 | 高 |
| D. テスト実行による副作用 | 高 | 高 | 高 | 最高 |
| E. セキュリティ機構の無効化（フラグ） | 中 | 中 | 中 | 中 |
| F. テスト用ダミーデータの永続化 | 中 | 高 | 中 | 中 |

## 推奨対策

### 最優先対策（D: テスト実行による副作用）

1. **テスト用ファイルシステムの隔離**
   - 本番状態ファイル：`.claude/state/loop-detector-state.json`
   - テスト状態ファイル：`.tmp/loop-detector-test-state.json`
   - テストで`STATE_FILE`をオーバーライド可能にする

2. **テスト前後のクリーンアップ**
   - テスト実行前に`beforeEach`で状態をリセット
   - テスト実行後に`afterEach`で一時ファイルを削除

3. **ファイルシステムのモック化**
   - Node.jsの`fs`モジュールをモック
   - 実ファイルアクセスをテスト環境から排除

### 高優先対策（C: ログの情報漏洩）

1. **ログ出力のサニタイズ**
   - ファイルパスをマスキングして出力
   - 相対パスまたはハッシュ化したパスを使用

2. **機密情報の保護**
   - ユーザーディレクトリ名を含むパスは出力しない
   - テスト中は明確にダミーパスを使用

3. **ログレベルの厳格化**
   - `console.error`で出力する前に機密情報チェック
   - 本番環境ではログレベルを`info`以上に設定

### 中優先対策（A, E, F）

1. **テストデータの分離**
   - 攻撃パターンを外部テスト入力ファイルに移動
   - テストコード内は「パターン名」で参照

2. **環境変数の厳格管理**
   - テスト実行スクリプトで`dotenv`を使用
   - テスト終了時に自動リセット

3. **ドキュメンテーション**
   - テストコメントに「セキュリティ検証」タグを付与
   - 攻撃パターンの解説ではなく、「検証の目的」を記述

## 結論

NEW-SEC-1/2/3専用テスト追加は、セキュリティ機構の強化を検証するための正当なテストである。
テスト実行による副作用（特にファイルシステムの状態変更）とログの情報漏洩リスクが最も懸念される脅威として特定された。
テスト用ファイルシステムの隔離、ログ出力のサニタイズ、環境変数の厳格管理により、これらのリスクは十分に軽減可能である。
テストコード自体がセキュリティリスク導入媒体とならないよう、推奨対策の実装を強く推奨する。
全体として6件の脅威を特定し、うち1件が高リスク、3件が中リスク、2件が低リスクであり、適切な対策により安全にテスト実装が可能である。
