# NEW-SEC-1/2/3専用テスト追加 - テスト出力フォーマット設計書

## サマリー

本テスト出力フォーマット設計書ではワークフロープラグインのセキュリティ強化機能（NEW-SEC-1のゼロ幅文字サニタイズ、NEW-SEC-2のFail-Closed化、NEW-SEC-3のエラーログ出力）の専用テストであるtest-n6-security-new.test.tsの出力フォーマットを定義しています。

本タスクはUIコンポーネントを持たないCLIベースのセキュリティテストであるためターミナル上に表示されるテスト実行結果の見やすさと情報の完全性を確保するために出力フォーマットを細部まで設計することが重要です。

既存のtest-n1からtest-n5までのテストスイートとの一貫性を保ちながらもセキュリティ専用テストとしての特性を反映したフォーマットを採用することでメンテナンス時の可視化と問題発生時のデバッグ効率を向上させます。

ユニコード文字（✓と✗）の使用によりパスと失敗を直感的に識別できるようにしテスト結果の読み込み時間を短縮します。

各テストスイートの見出しには実装するセキュリティチェック内容を明確に記述し次フェーズの実装担当者が何をテストするのかを即座に理解できる設計とします。

## CLIインターフェース設計

test-n6-security-new.test.tsはNode.jsのコマンドラインインターフェースとして直接実行される設計であり起動コマンドは「node src/backend/tests/unit/hooks/test-n6-security-new.test.ts」の形式で実行されます。

テスト実行時のプロセス終了コードは全テスト成功時にexit code 0を返し1件以上のテスト失敗時にexit code 1を返すことでCI環境でのテスト結果判定に対応します。

モジュール読み込み失敗（workflow-pluginディレクトリ未検出やrequireエラー）の場合にはexit code 2を返して環境設定の問題とテスト失敗を明確に区別します。

テスト実行時に引数は不要であり環境変数による動作変更も行わないためユーザーはファイルパスの指定のみでテストを実行可能です。

標準出力にはテスト結果の全項目が出力され標準エラー出力はNode.jsランタイムエラーやアサーション失敗のスタックトレースに限定されます。

## エラーメッセージ設計

テスト項目が失敗した場合のエラーメッセージは「✗ N6-SEC1-01: U+200Bゼロ幅スペースのサニタイズ: Expected sanitized output but got raw input」のようにテストID、テスト説明、失敗理由の3要素で構成されます。

アサーション失敗時にはNode.jsのassert.strictEqualやassert.deepStrictEqualが生成するメッセージを自然言語的に変換してテスト失敗の原因が一読で把握できるようにします。

モジュール読み込みエラーの場合には「ERROR: Cannot load module bash-whitelist.js from workflow-plugin/hooks/」のようにエラー種別とファイルパスを含むメッセージを標準エラー出力に表示します。

プロジェクトルート探索に失敗した場合のメッセージは「ERROR: Project root not found. Please run from within the Workflow project directory.」として実行環境の問題を明確に示します。

予期しない例外がテスト実行中に発生した場合にはtry-catchブロックで捕捉してテストスイート名とスタックトレースを含むエラーメッセージを出力した上でexit code 1で終了します。

## APIレスポンス設計

本タスクはCLIベースのテストファイル追加であるためHTTP APIエンドポイントやREST APIレスポンスの設計は対象外です。

テスト対象であるbash-whitelist.jsのcheckBashWhitelist関数は内部APIとして「allowed: boolean, reason: string」のオブジェクトを返却しこの戻り値の形式がテストで検証されます。

detectEncodedCommand関数の返却値は「detected: boolean, type: string, decoded: string」の形式であり不正なエンコードパターンの検出時にdetected:trueとtype文字列が正しく設定されることをテストします。

loop-detector.jsのnormalizeFilePath関数は文字列型のパスを返却しfs.realpathSync失敗時のフォールバック動作はpath.resolveへの切り替えとconsole.warn出力を含むことがテスト対象のAPIの振る舞いです。

テスト内で使用するassertモジュールのAPIはNode.js標準ライブラリのassert.strictEqualとassert.deepStrictEqualに限定し外部テストフレームワークへの依存を排除しています。

## 設定ファイル設計

test-n6-security-new.test.tsはpackage.jsonやtsconfig.jsonなどの設定ファイルを参照せず直接Node.jsで実行可能なスタンドアロン形式で設計されています。

テスト対象モジュールのパス解決にはprocess.cwdからの相対パス探索を使用しworkflow-pluginディレクトリを再帰的に検索する方式を採用しています。

テスト実行に必要な環境設定は一切なくNode.jsランタイム（v18以上）がインストールされていれば任意のディレクトリからテストを実行可能です。

既存のtest-n1からtest-n5と同じディレクトリ（src/backend/tests/unit/hooks/）に配置することでテスト実行スクリプトによる一括実行にも自動的に対応します。

テストファイル内部でのモジュールパス解決はprocess.cwdとpath.joinを使用した動的パス生成により実行環境に依存しないポータブルな設計となっています。

## テスト出力構造設計

ワークフロープラグインの新規セキュリティテストはNODE環境で直接実行されるJavaScriptテストファイルであり標準出力（stdout）と標準エラー出力（stderr）の両方を利用してテスト結果をClaude Codeのターミナルに表示します。

テスト実行時の最初の出力は空行を1行出力してターミナルの視覚的なセパレーションを確保し以降のテスト結果との境界を明確にします。

その直後にテストスイート全体のタイトル（Test Suite: NEW-SEC-1: ゼロ幅文字サニタイズ）を表示してどのセキュリティ機能をテストするのかを明確に示します。

タイトルの下に等号記号（=）を28文字分並べて視覚的な区切り線を形成しCLIツール標準の出力フォーマットに合致させます。

テスト項目の前には空行を1行挿入してテスト項目群とタイトルの間に視覚的な空間を作ることでスクリーンリーダーでの読み上げ時も含めて構造的な区分を明確にします。

## テスト項目フォーマット設計

各テスト項目の行頭には2つのスペース文字を配置してインデントレベルを統一し複数のテストスイートが出力される場合の可読性を確保します。

テスト結果の記号として✓（U+2713チェックマーク）でパスを表示しユニコード互換の全てのターミナルで正確に表示されることを確認しています。

失敗時には✗（U+2717バツ記号）を使用してパスケースとの明確な視覚的区別を実現しテスト結果を一目で把握できます。

チェックマークまたはバツ記号の直後に半角スペース1文字を挿入してテストIDとの適切な間隔を確保します。

テストIDはN6-SEC1-01のような形式で「N6」はフェーズを示す識別子、「SEC1」はセキュリティ機能1を意味し、末尾の「01」はテスト項目番号として機能します。

テストIDの直後にコロン文字（:）を配置してテスト説明文との区切りを形成しメッセージパーサーにとってもテストID終端が明確になります。

コロンの後に半角スペース1文字を挿入してテスト説明の開始位置を統一し複数行のテスト出力でも位置ずれが生じないようにします。

テスト説明文は日本語で記述してプロジェクト参加者の理解を促進し技術的な正確性と可読性のバランスを取ります。

パスケースではテスト説明の末尾に追加の詳細情報を付加せずシンプルに終了しターミナル出力の行幅を効率的に使用します。

失敗ケースではテスト説明の後に「: 」とコロンとスペースを配置して詳細エラーメッセージを開始し失敗理由を明確に記述します。

## NEW-SEC-1テストスイート設計

NEW-SEC-1のゼロ幅文字サニタイズテストは6つのテスト項目で構成されておりゼロ幅スペース、ゼロ幅非接合子、ゼロ幅接合子、ゼロ幅フォーマッタといった異なるUnicode文字に加え複数ゼロ幅文字混合テストと正常入力非破壊性テストも含めて包括的に検証します。

N6-SEC1-01ではU+200Bゼロ幅スペース（ZERO WIDTH SPACE）の検出と除去が正しく機能することをテストしコマンド文字列内に隠された制御文字が攻撃に利用されることを防止しています。

N6-SEC1-02ではU+200Cゼロ幅非接合子（ZERO WIDTH NON-JOINER）のサニタイズが機能することを確認しアラビア語やデーヴァナーガリーなどの複雑な文字体系に対応した攻撃ベクトルを防ぎます。

N6-SEC1-03ではU+200Dゼロ幅接合子（ZERO WIDTH JOINER）の除去が動作することを検証して文字の合成を悪用した攻撃から保護します。

N6-SEC1-04ではU+FEFFゼロ幅フォーマッタ（BYTE ORDER MARK、BOM）の処理が正しく行われることをテストしてファイルエンコーディングの混乱を利用した攻撃にも対応しています。

N6-SEC1-05では複数種類のゼロ幅文字（U+200B、U+200C、U+200D、U+FEFF）が同一コマンド内に混在する場合のサニタイズが全文字に対して正しく機能することを確認します。

N6-SEC1-06では通常のASCII文字列やマルチバイト日本語文字列がサニタイズ処理を通過した後も元の文字列と完全に一致することを検証し正当な入力への非破壊性を保証します。

## NEW-SEC-2テストスイート設計

NEW-SEC-2のFail-Closedロジックテストは4つのテスト項目でエンコードされたコマンドのデコード失敗時に確実にコマンドブロックが実施されることを検証します。

N6-SEC2-01ではbase64エンコード形式（echo "base64_encoded_string" | base64 -d）のデコード失敗時にdetectEncodedCommand関数がdetected:trueを返すことを確認しデコード失敗は自動的にコマンド実行を拒否することを保証します。

N6-SEC2-02ではprintf hexエスケープパターン（printf '\\x...'など）のデコード失敗がコマンドブロックをトリガーすることをテストして数値形式のエンコーディングから保護されていることを検証します。

N6-SEC2-03ではecho octalエスケープパターン（echo $'\\...'など）のデコード失敗時にもdetected:trueが返されることを確認してoctalエンコーディング形式の攻撃をも防止していることを検証します。

N6-SEC2-04では正常なコマンド（lsやgit statusなど一般的なコマンド）がdetectEncodedCommand関数を通過してdetected:falseが返されることを確認して誤検知が発生しないことを保証します。

Fail-Closed化の実装により不正な形式のエンコードコマンド検出時には許可ではなく確実に拒否する動作を検証することがこのテストスイートの核心的な目的です。

## NEW-SEC-3テストスイート設計

NEW-SEC-3のエラーログ出力テストは3つのテスト項目でファイルパス正規化失敗時の警告ログ出力が正しく機能することを検証しconsole.warnとconsole.errorのスパイ（モック）を使用して出力の有無と内容を確認します。

N6-SEC3-01ではfs.realpathSync失敗時にnormalizeFilePath関数がフォールバック処理を実行し警告メッセージがconsole.warnに出力されることをスパイ関数で捕捉してテストします。

N6-SEC3-02ではfs.realpathSyncとpath.resolveの両方が失敗する状況をシミュレートしconsole.errorによる重大エラーのログ出力が行われることを検証して障害原因の特定に必要な情報が記録されていることを確認します。

N6-SEC3-03では正常なファイルパスに対してnormalizeFilePath関数を呼び出した場合にconsole.warnもconsole.errorも呼び出されないことを確認して正常系での余分なログ出力が発生しないことを保証します。

テスト完了後には必ずconsole.warnとconsole.errorのスパイを元のメソッドに復元して他のテストスイートや後続の処理に影響を与えないクリーンな状態を維持します。

## テスト結果サマリー出力設計

全テストスイート（NEW-SEC-1から3まで）が完了した後は空行を1行出力してテスト結果サマリーセクションとの視覚的なセパレーションを確保します。

サマリーラインの形式は「Tests: 13 passed, 0 failed, 13 total」という統一されたフォーマットで表示されpassed、failed、totalの順序は既存のtest-n3.test.tsと同じ規則に従います。

数値と単語の組み合わせ方式により複数のテスト結果を1行に集約してターミナル出力の効率を高めます。

Time行ではテスト全体の実行時間を「Time:  0.01s」というフォーマットで表示し小数点以下2桁の精度で秒単位の経過時間を記録します。

実行時間の表示は処理パフォーマンス監視の観点から重要であり特にセキュリティチェックのオーバーヘッド測定に役立ちます。

## テスト失敗時の出力形式

テスト項目が失敗した場合は✗記号を使用して視覚的に区別されます。

例えば「✗ N6-SEC1-01: U+200Bゼロ幅スペースのサニタイズ: Error message details」というように失敗理由がコロン後に記述されます。

失敗理由の部分はアサーション失敗時のエラーメッセージをそのまま引き継ぎユーザーが問題箇所を特定しやすくします。

複数のテスト失敗がある場合でもフォーマットは統一されておりterminaldiffツールによる自動検証にも対応できます。

サマリー行では失敗テスト数がfailed項目に正確にカウントされ「Tests: 10 passed, 3 failed, 13 total」のように表示されることで全体の成否が一目で把握可能です。

## テスト実装での出力記述パターン

test-n6-security-new.test.ts内での実装ではconsole.log関数を使用してstdoutに出力し各テスト結果を逐次的に表示します。

ハードコーディングされた出力テンプレートではなくテストロジック内で動的に出力内容を構成することで拡張性と保守性を確保します。

例えば「console.log('  ✓ ' + testId + ': ' + testDescription)」というように変数を組み合わせて出力を生成する実装パターンを採用します。

テスト実行時の進行状況をリアルタイムで可視化できるようにテスト項目の処理とその直後のconsole.log呼び出しを対にして実装します。

ヘルパー関数としてrunTest(testId, description, testFn)を定義してtry-catch内でテスト関数を実行しpass/failの出力を自動化することで各テストケースの記述量を最小限にします。

## テスト出力フォーマットと自動化ツール互換性

test-authenticity.ts（既存バリデーションコード）ではテスト出力の「passed」「failed」「total」パターンを検出して出力の真正性を検証しています。

新規のtest-n6-security-new.test.ts出力も同じパターンマッチング対象となるため「Tests: 13 passed, 0 failed, 13 total」というフォーマットを厳密に守る必要があります。

testStatusIndicatorsの検出ルールと一致する形式でサマリー行を生成することで自動バリデーションツール（loop-detectorやtest-authenticity）との互換性を確保します。

テスト出力のパフォーマンス指標（Time行）は自動解析対象外ですが人間による読み込み時の参照情報として機能します。

既存のtest-n1からtest-n5までが全て同一の出力フォーマットを採用しているためtest-n6もこの規約に完全準拠することでテスト実行スクリプトでの一括処理が可能です。

## テストスイートと既存テストの一貫性

既存のtest-n1.test.ts（scope-validator）からtest-n5.test.ts（set-scope）までのテスト出力フォーマットと完全に一致する形式で統一されることが重要です。

テストスイートのタイトル行、等号記号による区切り線、スペースによるインデント、ユニコード記号の使用方法がすべて既存テストと同じパターンに従うことで複数テストの一括実行時にも混乱が生じません。

新規のセキュリティテストスイートを既存のテスト実行スクリプトに統合する際にもフォーマット一貫性により特別な処理が不要になり保守工数を削減できます。

テストファイルの命名規則もtest-n6-security-new.test.tsとして既存のtest-n{番号}パターンに従い自動検出スクリプトでの認識が確実に行われます。

テスト実行の順序はファイル名のアルファベット順で決定されるためtest-n6はtest-n5の後に自動的に実行されます。

## テスト出力の言語設定

テスト項目の説明文をすべて日本語で記述することにより非英語話者のプロジェクト参加者にも理解しやすいドキュメントを実現しています。

テストIDの形式（N6-SEC1-01など）は言語に依存しない記号体系であるため国際化対応が容易です。

パスと失敗の記号（✓と✗）もユニコード文字として定義されているため全言語環境で同じ外観で表示されます。

サマリー行の「passed」「failed」「total」は英語表記のままですがこれはtest-authenticity.tsのパターンマッチング対象であるため変更不可の制約があります。

テストスイート名のTest Suiteプレフィックスも英語表記を維持して既存テストとの一貫性を確保しコロンの後のサブタイトル部分で日本語による説明を補完します。

## テスト出力とCI環境での活用

GitHub ActionsやClaude Codeの統合テスト実行環境ではテスト出力がログファイルに記録される際にもユニコード文字が正確に保存されることが確認されています。

テスト実行時の出力ログをCSV形式またはJSON形式に変換する場合でもテストID、テスト説明、テスト結果の3要素が構造的に抽出可能な出力フォーマットを採用しています。

継続的インテグレーションパイプラインではテスト出力ログを自動解析してテスト失敗を検出しコミットメッセージやGitHub Issueの自動作成につなげることが想定されています。

そのためテスト出力フォーマットの安定性と予測可能性がCI環境での運用に直結する重要な要件です。

テスト実行時の標準出力はCI環境のビルドログに自動的に保存されるためテスト失敗時のデバッグに必要な全情報がログから参照可能です。

## テスト出力アーティファクトの配置

test-n6-security-new.test.ts実行時の出力は標準出力として直接Claude Codeのターミナルに表示されるだけでなくワークフローのテスト結果ログにも記録される設計です。

テスト結果の永続保存によりテスト履歴の追跡とリグレッション検証が可能になります。

テスト出力ログはworkflow_record_test_result MCPツールを通じてワークフロー状態に記録されフェーズ遷移の判断材料として使用されます。

テストファイル自体はsrc/backend/tests/unit/hooks/ディレクトリに配置され既存のtest-n1からtest-n5と同じ場所で管理されます。

テスト実行時に一時ファイルやキャッシュファイルを生成しないため実行後のクリーンアップ処理は不要です。

## テスト検証時の手動確認事項
