# workflow_nextレスポンスにphaseGuide追加 - UI設計

## サマリー

本改修はCLIおよびMCP serverの内部改修であり、エンドユーザー向けのGUI画面は存在しない。
本ドキュメントではAPIレスポンスのインターフェース設計、エラーメッセージ設計、CLAUDE.mdのOrchestrator手順設計を定義する。
workflow_nextとworkflow_statusのレスポンスにphaseGuideフィールドを追加し、Orchestratorがsubagent promptを確定的に構築できるようにする。
並列フェーズではsubPhasesフィールドにサブフェーズごとのphaseGuideを含め、Orchestratorが複数subagentを同時起動する際の情報源とする。
エラー発生時はphaseGuideを含めずに既存のエラーレスポンスを返すことで、後方互換性を完全に維持する。
phaseGuideはフェーズごとの成果物要件、入力ファイル、実行可能コマンド、編集可能ファイル種別、最低行数を含む総合的なメタデータである。
Orchestratorはこのメタデータを基に確定的なpromptテンプレートを構築し、subagentの実行品質を担保する。
Layer2事後検証とLayer3自動修正を組み合わせた多層防御により、成果物品質を段階的に保証する設計とする。

## CLIインターフェース設計

本機能はMCP serverのツールレスポンスとして動作するため、CLI上での直接的なユーザーインターフェースは持たない。
Orchestratorがworkflow_nextまたはworkflow_statusを呼び出した際に、レスポンスJSONの中にphaseGuideフィールドが含まれる形式である。
Orchestratorはこのphaseガイド情報を読み取り、subagentに渡すpromptの構築に使用する。
ユーザーから見える変化はなく、subagentの成果物品質が向上するという間接的な効果のみである。
workflow_statusコマンドを手動で実行した場合にも、phaseGuide情報がレスポンスに含まれるが、ユーザーが直接参照する必要はない。

Orchestratorから見たインターフェースとしては、workflow_nextレスポンスに含まれるphaseGuideオブジェクトのプロパティを読み取ることになる。
phaseGuide.phaseNameにはフェーズの識別名、phaseGuide.descriptionにはフェーズの目的説明が含まれる。
phaseGuide.outputFileには成果物の出力先パスが絶対パスで示され、Orchestratorはこれをsubagent promptに含める。
phaseGuide.inputFilesには入力として読み込むべきファイルパスのリストが含まれ、OrchestratorはこれをRead指示としてpromptに含める。
phaseGuide.requiredSectionsには成果物に必須のMarkdownヘッダーセクションが含まれ、Orchestratorはこれを検証基準として利用する。
phaseGuide.minLinesには成果物の最低行数が示され、Orchestratorはこれを品質基準として利用する。
phaseGuide.subagentTypeにはsubagentの種別が示され、Orchestratorはこれを基にTask toolのsubagent_typeパラメータを決定する。
phaseGuide.modelには推奨モデル名が示され、Orchestratorはこれを基にTask toolのmodelパラメータを決定する。
phaseGuide.allowedBashCategoriesには実行可能なBashコマンドカテゴリが示され、Orchestratorはsubagentに対してこの範囲内でのBash実行を許可する。
phaseGuide.editableFileTypesには編集可能なファイル拡張子パターンが示され、Orchestratorはsubagentに対してこの範囲内での編集を許可する。

並列フェーズの場合は、phaseGuide.subPhasesがレコード形式でサブフェーズごとのPhaseGuideオブジェクトを含む。
Orchestratorは各サブフェーズに対して独立したTask tool呼び出しを構築し、並列実行を開始する。
各サブフェーズのphaseGuideは独立した成果物要件を持ち、Orchestratorはこれを基に各subagentのpromptを確定的に構築する。

## エラーメッセージ設計

phaseGuide生成時のエラーは内部エラーとして処理し、ユーザーには既存のエラーメッセージ形式で返す。
PHASE_GUIDESに対象フェーズの定義が見つからない場合は、phaseGuideフィールドをundefinedとしてレスポンスを返す。エラーとはしない。
docsDirが未設定の場合は、パス解決をスキップしてプレースホルダーのまま返す。Orchestratorが手動でパスを構築する際の参考情報として機能する。
レスポンスサイズが上限（1フェーズ1KB、並列フェーズ5KB）を超過した場合は、ログに警告を出力するがレスポンスはそのまま返す。
HMAC整合性検証が失敗した場合は、既存のエラーハンドリングに従い、phaseGuideを含めずにエラーレスポンスを返す。

入力ファイルが存在しない場合は、エラーとせずinputFilesにパスを含めたままレスポンスを返す。subagentが読み込みを試行して失敗する形となる。
出力ファイルのディレクトリが存在しない場合も同様であり、subagentがWrite実行時に親ディレクトリ作成を含めて処理する。
requiredSectionsやminLinesはガイドラインであり、これらを満たさない成果物が作成された場合はLayer2事後検証で検出される。
Layer2検証で不足が検出された場合は、Orchestratorが不足内容をsubagentに通知し、Layer3自動修正フローを開始する。
Layer3自動修正でも解決できない場合は、最終的にユーザーに通知してworkflow_backによるフェーズ差し戻しを提案する。

phaseGuideの構造的な検証エラー（必須フィールド欠落など）はMCP serverの実装エラーとみなし、詳細なエラーメッセージをログに出力する。
Orchestrator側でのphaseGuide解釈エラー（未知のsubagentType値など）は、デフォルト値にフォールバックしてログに警告を出力する形とする。

## APIレスポンス設計

workflow_nextの成功レスポンスには既存フィールドに加えてphaseGuideフィールドが追加される。
phaseGuideにはphaseName、description、requiredSections、outputFile、allowedBashCategories、inputFiles、editableFileTypes、minLines、subagentType、modelの各フィールドが含まれる。
各フィールドはoptionalであり、対象フェーズに該当しない情報は含まれない。例えばcommitフェーズにはrequiredSectionsやoutputFileは含まれない。
並列フェーズの場合はsubPhasesフィールドが追加され、キーがサブフェーズ名、値がPhaseGuideオブジェクトとなるレコード形式である。
workflow_statusの成功レスポンスにも同様のphaseGuideフィールドが追加されるが、現在フェーズの情報を返す点がworkflow_nextと異なる。

resolvePhaseGuide関数はPHASE_GUIDESからフェーズ定義を取得し、outputFileとinputFilesのdocsDirプレースホルダーを実際のパスに置換する。
並列フェーズの場合はsubPhases内の各PhaseGuideに対しても再帰的にパス解決を実行する。
返却されるPhaseGuideオブジェクトは元の定義のシャローコピーであり、PHASE_GUIDESマスター定義を直接変更しない。

phaseGuideの各フィールドの型定義は既存のPhaseGuide型に従い、TypeScriptの型安全性を維持する。
phaseNameは文字列型であり、PHASE_NAMES定数に定義された正規のフェーズ名を使用する。
descriptionは文字列型であり、フェーズの目的を1行で説明する内容を含む。
requiredSectionsは文字列配列型であり、Markdownヘッダー形式（シャープ記号を含む）で表記される。
outputFileは文字列型であり、絶対パス形式で解決された成果物の出力先を示す。
inputFilesは文字列配列型であり、各要素が絶対パス形式で解決された入力ファイルを示す。
allowedBashCategoriesは文字列配列型であり、readonly、testing、implementationのいずれかの値を含む。
editableFileTypesは文字列配列型であり、.mdや.tsなどの拡張子パターンを含む。
minLinesは数値型であり、成果物の最低行数を正の整数で示す。
subagentTypeは文字列型であり、Explore、Plan、general-purpose、Bashのいずれかの値を取る。
modelは文字列型であり、haiku、sonnetのいずれかの値を取る。
subPhasesはレコード型であり、キーが文字列型のサブフェーズ名、値がPhaseGuideオブジェクトとなる。

レスポンスサイズの制限として、1フェーズのphaseGuideオブジェクトは1KB以内、並列フェーズ全体で5KB以内を目安とする。
超過した場合でもエラーとはせず、ログに警告を出力して完全なレスポンスを返す実装とする。
将来的にフェーズ定義が複雑化した場合は、inputFilesを最大10件まで、requiredSectionsを最大20件までといった制限を導入する可能性がある。

## 設定ファイル設計

CLAUDE.mdのsubagentによるフェーズ実行セクションを更新し、phaseGuideベースのprompt構築手順を追記する。
Orchestratorはworkflow_nextまたはworkflow_statusのレスポンスからphaseGuide情報を取得し、subagent promptのテンプレートに埋め込む手順に従う。
Layer2事後検証として、subagent完了後にphaseGuide.outputFileの存在確認、phaseGuide.requiredSectionsの包含確認、phaseGuide.minLines以上の実質行数確認を順に実行する。
Layer3自動修正として、Layer2検証で不足が検出された場合にOrchestratorが不足セクションの追記、コンテンツの拡充、再検証を実行する手順を定義する。
phaseGuide情報はガイドラインであり最終的な制御はhooks側が行うという多層防御の原則をCLAUDE.mdに明記する。

CLAUDE.mdのsubagent起動テンプレートセクションには、phaseGuide.outputFileをpromptに埋め込む例を示す。
phaseGuide.inputFilesをRead指示としてpromptに含める例も示す。
phaseGuide.requiredSectionsを成果物の必須セクションとしてpromptに明示する例を示す。
phaseGuide.minLinesを品質基準としてpromptに含める例を示す。

並列フェーズでの複数subagent起動の例として、phaseGuide.subPhasesから各サブフェーズ情報を取得して独立したTask tool呼び出しを構築する手順を示す。
各subagentのpromptは対応するサブフェーズのphaseGuideに基づいて確定的に構築され、並列実行される。

CLAUDE.mdには、phaseGuideが存在しない場合（対象フェーズの定義が存在しない場合）のフォールバック手順も記載する。
Orchestratorはデフォルトのpromptテンプレートを使用し、成果物の品質検証はhooks側に完全に委ねる形とする。
この場合でもworkflowの進行は妨げられず、既存の動作と同等の品質を維持する。
