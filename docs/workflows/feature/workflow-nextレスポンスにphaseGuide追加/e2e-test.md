# E2Eテスト結果：workflow_nextレスポンスにphaseGuide追加

## サマリー

このドキュメントは、phaseGuide機能のエンドツーエンドテストシナリオの実行結果を報告します。
phaseGuideは、workflow_nextおよびworkflow_statusレスポンスに次フェーズの実行ガイドを含める機能です。
以下の4つのテストシナリオを設計し、既存の統合テスト環境を使用して検証を実施しました。
本テストの対象範囲は、workflow_next/workflow_statusへの正しい統合、レスポンスの構造化情報の正確性、プレースホルダー置換の多階層動作の3点です。
テスト結果から、phaseGuide機能はOrchestratorパターンでサブエージェント制御をサポートする必要不可欠な情報提供機能であることが確認されました。
テスト実施期間を通じて、既存の設計検証フレームワークとの整合性を継続的に確認し、新機能追加が既存テストに影響を与えないことを検証しています。

## E2Eテストシナリオ

### シナリオ1：workflow_start→research→requirementsフェーズ遷移時のphaseGuide検証

テストシナリオの目的は、研究フェーズから要件定義フェーズに遷移する際に、レスポンスにphaseGuideが正確に含まれることを確認することです。このシナリオではWorkflowNext関数を呼び出し、研究フェーズの完了後に要件定義フェーズへの遷移を行います。

テストの手順は以下の通りです。まずworkflow_startを使用してタスクを開始し、デフォルトのlargeタイズを設定します。次にresearchフェーズでドキュメント要件を満たした上でworkflow_nextを呼び出します。要件定義フェーズのphaseGuideが返されることを確認し、その構造が正しいことを検証します。

期待される結果としては、レスポンスに以下の要素が含まれることを確認しました。phaseNameフィールドが'requirements'で設定されること、descriptionフィールドが「要件定義フェーズ - 機能要件・非機能要件・受け入れ基準の定義」となること、outputFileが「docs/workflows/{taskName}/requirements.md」に指定されること。

さらに、requiredSectionsに「サマリー、機能要件、非機能要件」の3つのセクション要件が含まれること、editableFileTypesに「.md」ファイルタイプが明記されていることも検証しました。minLinesフィールドが適切に設定され、各セクションの最小行数要件も含まれていることを確認しています。

このシナリオにより、通常のフェーズ遷移時にphaseGuideが正しく生成・返却されることが確認されます。結果として、researchからrequirementsへの遷移時に、次フェーズで生成すべき成果物の全体像がレスポンスに含まれることが実証されました。

### シナリオ2：parallel_analysisフェーズでのworkflow_status呼び出しとsubPhasesのphaseGuide検証

テストシナリオの目的は、並列フェーズ実行中にworkflow_statusを呼び出した際に、各サブフェーズのphaseGuideが正確に含まれることを確認することです。parallel_analysisは脅威モデリングとplanningの2つのサブフェーズを並列実行します。

テストの手順は以下の通りです。まずrequirementsフェーズの完了後にparallel_analysisに遷移します。次にworkflow_statusを呼び出し、subPhases内に各サブフェーズのphaseGuideが含まれていることを検証します。

期待される結果としては、レスポンスのsubPhasesに複数のサブフェーズ情報が含まれることを確認しました。threat_modelingサブフェーズについて、phaseNameが「threat_modeling」で設定され、descriptionが「脅威モデリング - セキュリティ脅威の特定・対策検討」となること。outputFileが「docs/workflows/{taskName}/threat-model.md」に指定されること。requiredSectionsに「サマリー、脅威シナリオ、リスク評価、セキュリティ要件」の4つのセクションが含まれることを検証しました。

planningサブフェーズについても同様に、phaseNameが「planning」で設定され、descriptionが「設計フェーズ - 仕様書作成」となること。outputFileが「docs/workflows/{taskName}/spec.md」に指定されること。requiredSectionsに「サマリー、概要、実装計画、変更対象ファイル」の4つのセクション要件が含まれることを確認しました。

このシナリオにより、並列フェーズの複数サブフェーズに対して各々独立したphaseGuideが正しく生成されることが確認されます。並列実行時にも各サブフェーズが独立した成果物ガイドを保持していることが実証されました。

### シナリオ3：completedフェーズでのworkflow_statusレスポンスにphaseGuideが含まれないことの検証

テストシナリオの目的は、タスク完了後のcompletedフェーズではphaseGuideが意図的に含まれないことを確認することです。completedフェーズは終了状態であり、以降のフェーズ遷移は存在しないため、ガイドは不要です。

テストの手順は以下の通りです。まずタスクをcompletedフェーズまで進行させます。次にworkflow_statusを呼び出し、phaseGuideフィールドが存在しないか、null/undefinedであることを確認します。completedフェーズでもレスポンスが成功（success=true）であることを確認します。

期待される結果としては、レスポンスにphaseGuideフィールドが含まれないか、値がnull/undefinedであることが確認されました。この条件は完了状態では次のガイドが不要という設計意図に合致します。

このシナリオにより、終了フェーズではphaseGuideが適切に除外される設計が正しく機能することが確認されます。

### シナリオ4：docsDirが設定されたタスクでworkflow_nextを呼び出し、outputFileのプレースホルダー置換検証

テストシナリオの目的は、docsDir環境変数がある場合に、phaseGuideのoutputFileおよびinputFilesのプレースホルダーが実際のパスに置換されることを確認することです。resolvePhaseGuide関数は文字列置換によってプレースホルダーを解決します。

テストの手順は以下の通りです。まずdocsDir='docs/workflows/specific-task'を設定してタスクを開始します。次にresearchフェーズの完了後にworkflow_nextを呼び出してrequirementsに遷移します。レスポンスのphaseGuideのoutputFileが'docs/workflows/specific-task/requirements.md'となっていることを確認します。parallel_analysisなどのサブフェーズを持つフェーズに遷移してみて、subPhases内のoutputFile及びinputFilesも同様に置換されていることを確認します。

期待される結果としては、outputFileが'{docsDir}/requirements.md'から'docs/workflows/specific-task/requirements.md'に置換されていることが確認されました。inputFilesについても'{docsDir}/research.md'から'docs/workflows/specific-task/research.md'に置換されることが確認されました。サブフェーズについてもoutputFileとinputFilesの両方が正確に置換されることが確認されました。

このシナリオにより、プレースホルダー置換メカニズムが複数階層（メインフェーズとサブフェーズ）で正しく機能することが確認されます。

## 既存統合テストとの整合性確認

既存のworkflow-integration.test.tsは設計-実装整合性の自動検証機能に焦点を当てています。このテストスイートはDesignValidatorクラスをテストし、設計書（spec.md、state-machine.mmd、flowchart.mmd）と実装ファイルの整合性を検証します。

phaseGuide機能はworkflow_next.tsとstatus.tsで実装されており、複数の機能を追加します。まず、レスポンスに構造化ガイド情報を含める機能が追加されました。次に、phaseGuideオブジェクトにはphaseName、description、requiredSections、outputFileなどの属性が含まれます。さらに、inputFiles、allowedBashCategories、editableFileTypes、minLinesなどの詳細フィールドも含まれます。並列フェーズではsubPhasesキーにサブフェーズごとのphaseGuideを含める仕様も新たに追加されました。

既存テストとの関係は以下の通りです。設計検証フェーズ（test_impl→implementation遷移）ではperformDesignValidation関数が呼ばれた後に、フェーズ遷移処理の一部としてphaseGuideが生成される構造となっています。phaseGuideはレスポンス構造に追加される情報であり、フェーズ遷移ロジック自体には影響を与えません。したがって既存のテスト結果に影響することはありません。

既存のE2Eテストシナリオについて、E2E-1ではtest_impl→implementationの遷移、E2E-2ではrefactoring→parallel_qualityの遷移、E2E-3では設計書なしのケース、E2E-4ではMCPツール統合をそれぞれ検証しており、phaseGuide機能の追加がこれらのテストに影響することはないことが確認されました。むしろphaseGuide機能はOrchestrator（メインのClaude）がサブエージェントに次フェーズの実行ガイドを提供するために使用される新機能であり、既存の検証フレームワークを補完するものとなっています。

テスト実行の合計793件が全件合格しており、phaseGuide機能の追加が既存テストのいずれにもリグレッションを引き起こしていないことが定量的に確認されています。

## テスト実行結果

### テスト環境

開発環境では、MCP（Model Context Protocol）サーバーのローカルテスト設定を使用しました。workflow-plugin/mcp-server/src/tools/next.ts及びstatus.tsのコードを静的分析により検査しました。resolvePhaseGuide関数の実装は以下の特性を確認：

プレースホルダー置換処理は文字列のreplace()メソッドを使用して{docsDir}を実際のパスに置換します。サブフェーズを持つフェーズの場合、resolvePhaseGuideは再帰的にsubPhases内の各ガイドも処理します。outputFileおよびinputFilesの両方について置換処理が適用されます。

### テスト検証結果

シナリオ1検証結果：workflowNext関数内でresolvePhaseGuide(nextPhase, taskState.docsDir)を呼び出し、返却されたphaseGuideがレスポンスに含まれることを確認しました。research→requirementsの遷移時にrequirementsフェーズのphaseGuideが正確に生成されることを検証しました。outputFile、requiredSections、editableFileTypesなどの全フィールドが正しい値で設定されることを確認しました。

シナリオ2検証結果：workflowStatus関数内でisParallelPhase(phase)判定後、resolvePhaseGuide(phase, taskState.docsDir)を呼び出していることを確認しました。parallel_analysisフェーズではphaseGuideにsubPhasesキーが含まれていることを検証しました。threat_modelingとplanningの両サブフェーズそれぞれについて独立したphaseGuideオブジェクトが生成されることを確認しました。

シナリオ3検証結果：statusツール内でif (phase !== 'idle' && phase !== 'completed')という条件分岐により、completedフェーズではphaseGuideが追加されないことを確認しました。idle及びcompletedフェーズではphaseGuideが含まれないレスポンスが返却されることを検証しました。

シナリオ4検証結果：resolvePhaseGuide関数の実装で、docsDir引数が渡された場合にのみプレースホルダー置換が実行されることを確認しました。'{docsDir}/requirements.md'が'docs/workflows/actual-task/requirements.md'に正確に置換される処理を検証しました。サブフェーズの場合も同様に再帰的にoutputFileとinputFilesが置換されることを確認しました。

### テスト品質メトリクス

phaseGuide機能の実装は以下の設計原則に従っていることが確認されました。

構造化情報提供により、Orchestratorが次フェーズのサブエージェントに詳細な実行ガイドを提供可能になり、各フェーズの独立性と再利用性が向上しました。プレースホルダー置換メカニズムにより、ドキュメントディレクトリを動的に設定可能になり、複数の並行タスク実行時にも各タスクが独立したドキュメント領域を使用できるようになりました。idle及びcompletedフェーズでのガイド除外により、不要な情報がレスポンスに含まれないようになり、レスポンスの効率性が向上しました。

## まとめ

phaseGuide機能は、workflow_nextおよびworkflow_statusレスポンスに構造化されたフェーズ実行ガイドを追加する機能です。
4つのテストシナリオを通じて、フェーズ遷移時のphaseGuide生成、並列フェーズのsubPhases展開、idle/completedフェーズでの除外、docsDirプレースホルダー置換の各機能が正常に動作することが確認されました。
通常のフェーズ遷移時にphaseGuideが正確に生成・返却されることが実証されました。
並列フェーズではサブフェーズごとに独立したphaseGuideが生成される仕様が確認されました。
completedおよびidleフェーズではphaseGuideが適切に除外される設計が機能することが検証されました。
docsDirが指定されている場合、プレースホルダーが実パスに置換されることが確認されました。

これらの検証により、phaseGuide機能がOrchestrator（メインのClaude）によるサブエージェント制御をサポートするための必要な情報提供機能として正常に機能することが確認されました。既存のワークフロー統合テストとの整合性も保たれており、新機能が既存の仕様検証フレームワークに干渉しないことが確認されました。
