# Threat Model: agent-delegation-prompt-templates

## STRIDE分析

| ID | 脅威カテゴリ | 脅威 | リスクレベル | 軽減策 |
|----|------------|------|-------------|--------|
| T-1 | Spoofing | テンプレートファイルの偽装（不正なテンプレートが正規品として読み込まれる） | 低 | gitリポジトリ管理 + PRレビュー必須フロー |
| T-2 | Tampering | テンプレート内容の改竄（Why/What/How/Constraintsの構造破壊） | 低 | PRレビュー + DoDゲートによるフォーマット検証 |
| T-3 | Repudiation | テンプレート変更履歴の否認 | 低 | gitコミット履歴で全変更を追跡可能 |
| T-4 | Information Disclosure | テンプレート内にAPIキー・シークレットのプレースホルダが混入 | 中 | テンプレートに秘密情報のプレースホルダを含めない設計方針。レビューで検出 |
| T-5 | Denial of Service | 全テンプレートの同時読み込みによるコンテキストウィンドウ枯渇 | 中 | 200行制限 + フェーズグループ単位のテンプレート分割 + 必要時のみ読み込み |
| T-6 | Elevation of Privilege | テンプレートプロンプトによるAgent権限昇格（Worker権限でOrchestrator操作を指示） | 低 | テンプレートはプロンプト構造であり実行権限を持たない。hookによるツール制限が独立して機能 |

## リスクサマリー

最大リスクはT-5（コンテキストウィンドウ枯渇）。本タスクはMarkdownスキルファイルの追加・編集のみであり、実行コードの変更を含まない。そのため攻撃面は限定的である。

T-4（情報漏洩）は設計方針で排除する。テンプレート内に環境変数名・パス・認証情報のプレースホルダを一切含めない。

T-5はテンプレート数の増加に伴い顕在化する可能性がある。200行制限の遵守とフェーズグループ単位での分割により、単一テンプレートのトークン消費を抑制する。

## 軽減策一覧

| リスクID | 軽減策 | 実施タイミング | 検証方法 |
|---------|--------|--------------|---------|
| T-1 | gitリポジトリでのファイル管理、PRレビュー必須 | 常時 | git log / PR履歴で確認 |
| T-2 | DoDゲートでテンプレート構造（4層: Why/What/How/Constraints）を検証 | DoD実行時 | ゲート通過の成否 |
| T-3 | 全変更をgitコミットで記録 | コミット時 | git blame で追跡 |
| T-4 | テンプレートに秘密情報プレースホルダを含めない設計方針を策定 | planning時 | PRレビューで秘密情報パターンを目視確認 |
| T-5 | 200行制限 + フェーズグループ単位分割 + 遅延読み込み（必要時のみ） | 設計・実装時 | 各ファイルの行数検証（200行以下） |
| T-6 | hookによるツール使用制限が独立動作。テンプレートからの権限昇格は構造的に不可能 | 常時 | hook実行ログで確認 |

## decisions

- T-4対策: テンプレート内に環境変数名・APIキー・シークレットのプレースホルダを一切含めない -- プロンプトテンプレートは構造ガイドであり、秘密情報の注入ポイントとなるべきでない
- T-5対策: 各テンプレートファイルは200行以下とし、フェーズグループ単位で分割する -- 全テンプレート同時読み込み時のトークン消費を制限するため
- T-5対策: テンプレートは必要なフェーズ実行時にのみ読み込む遅延読み込み方式とする -- 不要なコンテキスト消費を防止するため
- T-2対策: テンプレートの4層構造（Why/What/How/Constraints）をDoDゲートで検証する -- 構造破壊を自動検出するため
- T-6対策: テンプレートにツール使用指示を含める場合、既存のtool-delegation.mdの権限表と整合させる -- hookが独立して権限制御するが、テンプレート自体も権限境界を尊重すべきであるため
- コード変更なし: 本タスクはMarkdownファイルのみの変更であり、実行コード変更に伴うリスク（インジェクション等）は存在しない -- 変更スコープの限定がリスク軽減の前提
