## サマリー

- 目的: `task-index.json` へのフック（hook）と MCPサーバーの競合書き込みを防止するため、ロックファイル存在チェックを `writeTaskIndexCache` に追加する
- 主要な決定事項:
  - 変更対象は `workflow-plugin/hooks/lib/discover-tasks.js` の `writeTaskIndexCache` 関数1箇所のみとする
  - ロックファイル（`task-index.json.lock`）の存在チェックをフック側の書き込み処理先頭に挿入することで競合を防止する
  - MCP サーバーが起動していない環境ではロックファイルが生成されないため、既存動作に影響しない
- 次フェーズで必要な情報:
  - 変更対象ファイル: `workflow-plugin/hooks/lib/discover-tasks.js`（`writeTaskIndexCache` 関数、L106-146 付近）
  - テスト対象: `workflow-plugin/mcp-server/src/__tests__/` 配下の既存テスト群、および新規テスト
  - 検証シナリオ: ロックファイル存在時のスキップ動作、ロックファイル不在時の通常動作、TTL 期間経過後の再スキャン動作

---

## 背景・課題

### 問題の本質

フェーズ遷移時に MCP サーバーが `task-index.json.lock` を排他取得してアトミック書き込みを行う間、フック（discover-tasks.js）が並行して `task-index.json` を `renameSync` で上書きする可能性がある。フック側のコードはロックファイルのセマンティクスを認識していないため、MCP サーバーの保護機構をバイパスして古いフェーズ情報を書き込む競合が発生しうる。

### 既存対策の限界

- `updatedAt` 1秒チェックはロック保持中（書き込み前）の割り込みを防げない
- `mtime チェック`は競合書き込み後の読み取り時に機能するが、書き込み自体は抑制できない
- `task-index.json` には HMAC 整合性チェックが適用されていないため、不整合データの書き込みを事後検出できない

---

## 機能要件

### F-001: ロックファイル存在チェックによる書き込みスキップ

- ユーザーストーリー: ワークフローを利用するユーザーとして、フェーズ遷移中にフックがタスクインデックスを上書きしても最新フェーズ情報が失われないことを保証したい
- 詳細: `writeTaskIndexCache` 関数の処理開始直後に `fs.existsSync(TASK_INDEX_FILE + '.lock')` を呼び出し、ロックファイルが存在する場合は後続の書き込み処理全体（一時ファイル書き込み・`renameSync`）をスキップして関数から早期リターンする
- 実装箇所: `workflow-plugin/hooks/lib/discover-tasks.js` の `writeTaskIndexCache` 関数先頭（現行 L106 付近）
- 追加コード量: 約5行以内

successCriteria:
- L3: `task-index.json.lock` が存在する状態で `writeTaskIndexCache` を呼び出した場合、`task-index.json` の内容が変更されないこと（書き込みが行われないこと）をユニットテストで確認できる
- L4: 実際にMCPサーバーとフックを並行動作させ、フェーズ遷移後の `task-index.json` のフェーズフィールドが正しい値を保持していることを結合テストで確認できる

### F-002: ロックファイル不在時の通常動作維持

- ユーザーストーリー: ワークフローを利用するユーザーとして、MCP サーバーが起動していない環境や通常のロック未保持期間中にフックのタスク検出が遅延なく正常動作することを保証したい
- 詳細: ロックファイルが存在しない場合は既存の `updatedAt` 1秒チェックおよびファイル書き込み処理を従来どおり実行する。ロックチェック追加による通常パスへの副作用がないこと
- 実装箇所: F-001 の早期リターン分岐以外のコードパスは変更しない

successCriteria:
- L3: `task-index.json.lock` が存在しない状態で `writeTaskIndexCache` を呼び出した場合、従来と同じ書き込みが行われることをユニットテストで確認できる
- L4: MCP サーバーが停止した環境でフックを動作させ、`task-index.json` が正常に更新されることを手動検証で確認できる

### F-003: スキップ時のログ出力

- ユーザーストーリー: プラグイン開発者として、フックがロックによりスキップした事実をデバッグ時に追跡できるようにしたい
- 詳細: ロックファイル存在によりスキップした場合、`console.warn` または既存ログ関数で「ロックファイル検出によりスキップ」に相当するメッセージを1行出力する。ログ出力は既存の実装で使用されているパターンに合わせた形式とする
- 実装箇所: F-001 の早期リターン直前にログ出力を追加する

successCriteria:
- L3: ロックファイル存在によりスキップした際、標準エラーまたはログバッファに対象メッセージが1件記録されることをユニットテストで確認できる
- L4: デバッグモードで実行したフックの出力にスキップメッセージが含まれることを手動確認できる

### F-004: ゾンビロックファイルの自動除去考慮

- ユーザーストーリー: ワークフローを利用するユーザーとして、MCP サーバークラッシュ後にロックファイルが残存して hook の書き込みが永続的にブロックされる状況を防ぎたい
- 詳細: ロックファイルを検出した場合、ロックファイルの `mtime` が設定した閾値（60秒）を超えていればゾンビロックと判断して削除し、書き込みを続行する。閾値は MCP サーバー側の自動削除閾値（10秒）と整合性を確認したうえで決定する（10秒未満では MCP の自動削除前にフックが削除してしまうリスクがあるため、余裕をもって60秒とする）
- 実装箇所: F-001 のロックファイル存在チェック後に `mtime` チェックを追加する

successCriteria:
- L3: `mtime` が60秒を超えるロックファイルが存在する状態で `writeTaskIndexCache` を呼び出した場合、ロックファイルが削除されて書き込みが実行されることをユニットテストで確認できる
- L4: MCP サーバーをクラッシュさせた後に60秒超経過させると、次のフック実行でロックファイルが自動削除されて正常動作に復帰することを手動検証で確認できる

## 非機能要件

### NF-001: 処理性能

- ロックチェック（`fs.existsSync` 1回）による追加レイテンシは1ms 未満とする
- 通常ケース（ロックなし）において、変更前後の `writeTaskIndexCache` 実行時間の差分が5%以内であること

### NF-002: 信頼性

- ロックファイルの存在チェックと書き込みスキップ判断の間に発生するロック解放（TOCTOU）については、既存の MCP サーバー側 `updatedAt` 1秒チェックが二重防衛として機能するため許容する
- ゾンビロック閾値（60秒）は MCP サーバー側の自動削除閾値（10秒）の6倍以上を維持して、正常稼働中の MCP ロックを誤って削除しないことを保証する

### NF-003: 後方互換性

- MCP サーバーが起動していない環境（ロックファイルが生成されない環境）では動作が変わらないこと
- 既存のユニットテスト（`workflow-plugin/mcp-server/src/__tests__/` 配下）が全件パスすること

### NF-004: コード量

- 変更対象は `workflow-plugin/hooks/lib/discover-tasks.js` の `writeTaskIndexCache` 関数のみとし、追加行数は10行以内に収める

## 受け入れ基準

1. `discover-tasks.js` の `writeTaskIndexCache` 関数に F-001〜F-004 の処理が実装されていること
2. 新規ユニットテストが `__tests__/` 配下に追加されており、以下4ケースが網羅されていること:
   - ロックあり → 書き込みスキップ（F-001）
   - ロックなし → 通常書き込み（F-002）
   - ロックあり → ログ出力確認（F-003）
   - ゾンビロック（60秒超） → ロック削除後に書き込み（F-004）
3. 既存テスト（`__tests__/` 配下全件）が変更後も全件パスすること
4. `writeTaskIndexCache` の通常パスにおける処理時間が変更前と比較して5%以内の増加であること（NF-001）
5. 変更ファイルが `discover-tasks.js` 1ファイルのみであること（NF-004）

## スコープ外事項

以下の項目は本タスク（CC-1）の対象外とする:

- `task-index.json` への HMAC 整合性チェックの追加（将来の別タスクで検討）
- MCP サーバー側（`manager.ts`）の変更（既存の `acquireLockSync` の仕様は変更しない）
- 複数 MCP サーバーインスタンスが同時起動するケースへの対応（現行アーキテクチャでは単一インスタンスを前提とする）
- フック以外の書き込み経路（CI スクリプト等）への対応
- `readTaskIndexCache` 側の変更（読み取り競合は現行の mtime チェックで対処済みとする）
