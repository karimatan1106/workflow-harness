## サマリー

- 目的: FR-19 で実装された `lock-utils.ts` の `atomicWriteJson` 関数および `sleepSync` 関数、ならびに既存の `acquireLock` 関数に対してセキュリティ観点のレビューを実施する。
- 評価スコープ: `workflow-plugin/mcp-server/src/state/lock-utils.ts` の全3エクスポート関数（`acquireLock`・`atomicWriteJson`・`logLockEvent`）および内部関数 `sleepSync`。
- 主要な決定事項: リトライロジックは最大3回の固定上限があり無限ループのリスクが排除されている。一時ファイル名に `process.pid` を含める設計により同一ホスト上の名前衝突リスクが低減されている。`Atomics.wait` によるスレッドブロックはメインスレッド使用時に留意が必要だが、MCPサーバーの実行モデルでは許容範囲と判断される。
- 検証状況: コードスタティック解析による評価を実施。実際のファイルシステム操作を伴うスキャンツール実行は未実施だが、コードレビューによる評価は完了。
- 次フェーズで必要な情報: このセキュリティスキャン結果で指摘された低リスク項目（パストラバーサル入力検証・ロック競合ウィンドウ）は次フェーズで既知課題として引き継ぐ必要がある。

## 脆弱性スキャン結果

FR-19 で変更されたファイル `lock-utils.ts` に対して静的コードレビューによるセキュリティスキャンを実施した。スキャン対象は3つのエクスポート関数（`acquireLock`・`atomicWriteJson`・`logLockEvent`）および内部実装の `sleepSync` 関数である。

### atomicWriteJson のリトライロジック評価

`atomicWriteJson` 関数における `EPERM`/`EBUSY` エラー時のリトライロジックは、`maxRetries = 3` という固定上限により無限ループのリスクが存在しないことを確認した。ループ構造は `for (let attempt = 0; attempt <= maxRetries; attempt++)` で最大4回の試行（初回 + 3回リトライ）に制限されており、ループ終了後に必ず `lastError` をスローする設計になっている。恒久的なエラー（`EPERM`/`EBUSY` 以外）については即時 `throw` で再スローされ、ループが継続しない設計も適切に実装されている。

### sleepSync の Atomics.wait 評価

`sleepSync` 関数は `SharedArrayBuffer` と `Atomics.wait` を使用して同期的にスレッドをブロックする実装を採用している。`SharedArrayBuffer` の使用に際してはブラウザ環境での Cross-Origin Isolation 要件が存在するが、本コードは Node.js サーバーサイドで動作するため当該制限は適用されない。`Atomics.wait` はワーカースレッドでの使用を前提とした API であり、メインスレッドでの呼び出しは Node.js の実行環境によっては例外となる場合がある。MCP サーバーは非同期イベントループを基盤とするため、メインスレッドで `sleepSync` を呼び出すとイベントループがブロックされ、他のリクエスト処理が遅延するリスクがある。

### 一時ファイル名の衝突リスク評価

`atomicWriteJson` が生成する一時ファイル名は `${filePath}.tmp.${process.pid}` の形式を採用している。プロセス ID を含む命名により同一ホスト上の複数プロセスが同一ファイルを同時書き込みする際の名前衝突リスクは低減されている。ただし、同一プロセス内で `atomicWriteJson` を並列に呼び出した場合（同一 `filePath` に対して）、一時ファイルが同名になるため後続の書き込みが先行書き込みの一時ファイルを上書きする競合が発生しうる。MCP サーバーの実行モデルにおいて同一パスへの並列書き込みが発生するかどうかはコンテキストに依存するが、`acquireLock` と組み合わせて使用することでこのリスクは管理される。

### acquireLock のステールロック処理評価

`acquireLock` 関数はロックファイルの存在確認と内容解析の間に TOCTOU（Time-Of-Check-to-Time-Of-Use）競合ウィンドウが存在する。具体的には、ロックファイルを読み込んで「ステール」と判定した後、`fs.unlinkSync` で削除する前に別のプロセスが同じロックを取得する可能性がある。この競合ウィンドウは同期 I/O を使用しているため極めて短いが、理論的なリスクとして記録する。実用的なリスクレベルは低い。

## 検出された問題

### 問題1: メインスレッドでの Atomics.wait 使用（低リスク）

`sleepSync` 関数が MCP サーバーのメインスレッドから呼び出される経路において、Node.js の環境設定によっては `Atomics.wait` が例外をスローする可能性がある。テスト環境では `vi.spyOn(Atomics, 'wait')` でモック化することで即時リターンが可能だが、本番環境での動作はメインスレッド呼び出しを想定した設計ではない点に注意が必要である。

- 影響範囲: `atomicWriteJson` を呼び出す全ての実行パス
- 深刻度の評価: 軽微（機能は動作するが、イベントループのブロック時間は最大 300ms、3回 × 100ms）
- 緩和状況: `acquireLock` により同時書き込みは制御されており、`sleepSync` が呼び出されるのはリネーム失敗時のみ

### 問題2: パストラバーサル入力検証の欠如（低リスク）

`atomicWriteJson` および `acquireLock` は `filePath` 引数に対してパストラバーサル検証を実施していない。呼び出し元が信頼されたコンテキスト（MCP サーバー内部の状態管理モジュール）であれば問題はないが、将来的に外部入力が `filePath` に到達する経路が生じた場合に潜在的なリスクとなりうる。

- 影響範囲: `atomicWriteJson` 関数、`acquireLock` 関数の `filePath` 引数
- 深刻度の評価: 情報レベル（現在の呼び出し元は全て内部モジュールであり外部入力は到達しない）
- 推奨対応: 呼び出し元での入力検証を維持し続けること

### 問題3: ロック解放失敗時のサイレントエラー（情報レベル）

`acquireLock` が返す解放関数において、`fs.unlinkSync` が失敗した場合に `console.warn` のみで処理を継続する実装になっている。ロックファイルが削除されない場合、ステールロックとして次のリトライ処理で自動除去されるため機能的な問題は限定的であるが、エラーが自動的に伝播しないため監視上の死角となる可能性がある。

- 影響範囲: `acquireLock` の解放関数内
- 深刻度の評価: 情報レベル（ステールロック検出機構が代替として機能する）
- 現在の緩和策: `staleTimeout`（デフォルト 10 秒）を超えたロックは自動的に削除される

### スキャン総合評価

今回のコードレビューで検出された問題は全て低リスクまたは情報レベルに分類される。クリティカルな脆弱性や高リスクの問題は検出されなかった。`atomicWriteJson` のリトライ上限固定・恒久エラーの即時再スロー・一時ファイルのクリーンアップ処理はいずれも適切に実装されており、セキュリティ観点での設計は概ね妥当である。既存の `acquireLock` が提供する排他制御機構と組み合わせて使用することで、FR-19 実装後のファイル書き込み安全性は向上していると評価する。
