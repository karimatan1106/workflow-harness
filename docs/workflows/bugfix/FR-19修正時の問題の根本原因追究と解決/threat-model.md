# FR-19修正時の問題の根本原因追究と解決 - 脅威モデル

## サマリー

本タスクは前回のFR-19実装後に発生した3つの副次問題を解決するものである。主な脅威はリポジトリ整合性の損失、ファイル削除失敗による残存、ワークフロー設計の不完全さに起因する再発可能性である。

**主要な脅威キャテゴリー**:
1. ファイル管理脅威: 削除コマンドブロック・代替手段の失敗によるファイル残存
2. Git追跡脅威: 永続配置ファイルの漏れ・除外設定による自動削除
3. テンプレート脅威: 改善されたテンプレートの解釈違い・不完全な実装
4. プロセス脅威: subagentが判断材料不足で再び同じ失敗を繰り返す可能性

**タスク完了時の目標状態**:
- ルート直下の一時ファイル完全削除
- 永続設計ドキュメントの正規ディレクトリへの配置確認
- ワークフロー全体の一時ファイル・アーティファクト管理ルール明確化

---

## 脅威シナリオ

### TS-1: ファイル削除失敗による未削除ファイル残存（MEDIUM リスク）

**脅威説明**:
ルート直下の3つの検証スクリプト（verify-templates.js, full-template-verify.js, detailed-verify.js）が完全に削除されず、git status に表示されたまま残る状況。

**発生パターン**:
1. Bash の `rm` コマンドが phase-edit-guard フックでブロックされる（コマンドチェーン誤判定、実装カテゴリ不許可）
2. subagent が代替削除方法（Read + Write ツール、git rm 等）の実施を試みるが、ファイルパスが複雑で失敗
3. エラー発生時に処理を中断せず続行し、削除未完了のまま次フェーズに進む
4. 最終的に implement フェーズ完了時に未追跡ファイルが残存

**技術的背景**:
- 前回タスク（ID: 20260224_215845）の refactoring フェーズで `node << 'EOF'` というヒアドキュメント実行がフックでブロックされた
- その代替案として検証スクリプトをルート直下に書き込む判断がなされた
- 削除時に複数ファイルの `rm` コマンド（例: `rm file1 file2 file3`）が「コマンドチェーン」と誤認識されてブロックされた
- その後、代替削除手段の実施が完了せず、ファイルが残存

**影響度**:
- git status がクリーンでない状態で残る
- CI/CD パイプライン実行時にチェック通過をブロックする可能性
- リポジトリ履歴が「未削除ファイルあり」として記録される

**根本原因要因**:
1. subagent テンプレートにルート直下への一時ファイル配置禁止を明示していなかった
2. 削除コマンドがブロックされた際の代替手段（.tmp/ への配置変更、個別削除、ツール使用）が提示されていなかった
3. phase-edit-guard のコマンドチェーン判定が、複数の独立した引数と真のコマンドチェーン（`;`、`&&`、`||`）を区別できなかった

---

### TS-2: 永続設計ドキュメントの未追跡状態（HIGH リスク）

**脅威説明**:
ui_design フェーズで作成した修正プロセス.flowchart.mmd が docs/workflows/ に一時配置されたまま、.gitignore による除外対象となり、git 追跡対象外のままになる状況。

**発生パターン**:
1. ui_design フェーズで Mermaid フローチャートを作成し、デフォルトの docsDir（docs/workflows/{taskName}/）に配置
2. CLAUDE.md のルール「ワークフロー成果物の配置先」に従い、一時的なワークフロー成果物として配置
3. docs_update フェーズでこのファイルを docs/spec/diagrams/ に移動する指示がテンプレートに明示されていなかった
4. そのまま docs/workflows/ に残存し、.gitignore の `**/docs/workflows/` パターンにより自動除外
5. git status でも表示されない（未追跡でなく、除外ファイル）ため、問題が発見されない

**技術的背景**:
- docs/workflows/ は一時的なワークフロー作業フォルダであり、デフォルトで .gitignore に含まれている
- 設計図（.mmd）や永続仕様書（.md）は、一時的にこのフォルダに配置されるが、docs_update フェーズでプロダクト仕様フォルダ（docs/spec/diagrams/, docs/spec/screens/ 等）に移動する必要がある
- この「フェーズ間引き継ぎ」の責任が明示されていなかった

**影響度**:
- 永続的な設計ドキュメントが git リポジトリに記録されない
- チーム内で修正プロセス図の情報が共有されない
- コードレビューや将来の仕様確認時に設計図が参照できない

**根本原因要因**:
1. ui_design テンプレートに「一時的に docsDir に配置される」という説明はあるが、「docs_update で移動する責任」が明示されていなかった
2. docs_update テンプレートに、前回のワークフロー成果物フォルダから永続ドキュメントを移動する具体的指示がなかった
3. CLAUDE.md の「プロダクト仕様への反映」セクションが原則的な記述のみで、フェーズテンプレート内での実装指示に反映されていなかった

### TS-3: テンプレート改善の解釈違い（MEDIUM リスク）

**脅威説明**:
refactoring, ui_design, docs_update フェーズのテンプレートに新規追加する一時ファイル配置ルール・アーティファクト移動ガイダンスが、subagent に正しく理解されず、不完全に実装される状況。

**発生パターン**:
1. テンプレートに「一時ファイルは .tmp/ に配置」という指示を追加するが、相対パスが複雑で混乱
2. subagent が「.tmp/ への配置」と「ファイル削除」の責任範囲を誤解（別フェーズが担当すべき処理と混同）
3. ガイダンス内の代替手段（「ヒアドキュメント がブロックされた場合は…」）が列挙されるが、優先順序が不明確
4. subagent がどの手段を選ぶべきか判断できず、結果として処理が不完全に終わる

**技術的背景**:
- テンプレートは自然言語テキストであり、subagent による解釈の揺らぎがある
- 複数の代替手段を提示する場合、優先順序や条件分岐を明確にしないと、subagent が任意の手段を選択してしまう
- 「フェーズ間の責任分担」という概念は、単一フェーズのテンプレート内では明示しにくい

**影響度**:
- テンプレート改善の効果が期待値以下になる
- 次回以降のタスクでも同じ問題が繰り返される可能性
- Orchestrator がテンプレート実装を検証しなければ、不完全な改善が固定化される

**根本原因要因**:
1. 代替手段の優先順序が不明確（個別削除 vs .tmp/ 配置変更 vs ツール使用）
2. フェーズ間の引き継ぎ情報（「ui_design で配置したファイルを docs_update で移動する」）が各フェーズのテンプレート内に分散している
3. テンプレート改善の「完了基準」が明示されていない（どのレベルまで詳細に書けば十分か）

---

### TS-4: ワークフロー設計の根本的不完全さ（HIGH リスク）

**脅威説明**:
root直下への一時ファイル配置やアーティファクト配置ミスは、個別タスクの問題ではなく、ワークフロー全体の設計思想の不十分さに起因する可能性。本タスクで部分的にテンプレートを改善しても、類似パターンが他フェーズで再現される危険がある。

**発生パターン**:
1. テンプレート改善が refactoring, ui_design, docs_update に限定されている
2. 他フェーズ（test_impl, implementation, etc.）でも同様の一時ファイル配置の判断が必要になる可能性がある
3. 改善ルールが各フェーズで「断片的」に記述され、ワークフロー全体で一貫していない
4. 次回以降のタスクで「CLAUDE.md では〜と書いてあるが、テンプレートでは書かれていない」という矛盾が生まれる

**技術的背景**:
- CLAUDE.md には「テスト出力・一時ファイルの配置ルール」として統一的な原則がある
- しかし、個別フェーズのテンプレートにはこのルールが完全には反映されていない
- ワークフロー全体の「設計思想」（どのような判断基準でファイルを配置するか）が言語化されていない

**影響度**:
- 本タスクの改善が短期的な「バンドエイド」に終わる
- 類似問題が他フェーズ・他タスクで再発する
- ワークフロー全体の信頼性が低下する

**根本原因要因**:
1. ワークフロー設計の「原則」と「実装」のギャップ（CLAUDE.md vs テンプレート）
2. フェーズ別テンプレートが独立して管理され、整合性確保の仕組みがない
3. subagent が迷わないよう「判断基準」をテンプレートレベルで言語化するプロセスが不確立

---

### TS-5: MCP サーバーの状態不整合（LOW リスク）

**脅威説明**:
definitions.ts のテンプレート文字列を修正してコミットしても、実行中の MCP サーバーがメモリ内キャッシュを保有したままになり、古いテンプレートでサブエージェントが起動される状況。

**発生パターン**:
1. definitions.ts を編集・コミットしてプッシュ
2. MCP サーバープロセスが再起動されない（または再起動タイミングが不明確）
3. 実行中のサーバーが古いテンプレート（メモリキャッシュ版）を返す
4. サブエージェントが改善されたテンプレートを受け取れず、相変わらず古い問題が発生

**技術的背景**:
- Node.js の `require()` はモジュールをグローバルキャッシュに保存し、プロセス終了までクリアされない
- MCP サーバーを再起動しない限り、ディスク上のテンプレート変更が反映されない
- CLAUDE.md ルール 22（「コアモジュール変更後はMCPサーバーを再起動してから次フェーズに進むこと」）が存在するが、運用上の周知が十分でない可能性

**影響度**:
- テンプレート改善が実際には機能しない
- 次フェーズ以降の subagent が古いテンプレートで起動され、同じ問題が再発する
- 問題の原因がテンプレート改善にあるのか MCP サーバーキャッシュにあるのか特定が困難

**根本原因要因**:
1. MCP サーバー再起動の責任が明示されていない（Orchestrator の責務か、ユーザーの手動操作か）
2. テンプレート改善の「効果確認」プロセスが確立されていない
3. MCP モジュールキャッシュの仕組みが、ワークフロー実装者に十分周知されていない

---

### TS-6: テンプレート検証の欠落（MEDIUM リスク）

**脅威説明**:
refactoring, ui_design, docs_update フェーズのテンプレート改善後、改善内容が実際に正しく機能しているかを検証するメカニズムが存在しない状況。

**発生パターン**:
1. definitions.ts を修正し、テンプレートに新規セクションを追加
2. MCP サーバーを再起動
3. 次タスク実行時に改善されたテンプレートが subagent に配信されるが、実装が期待通りかは確認されない
4. subagent が新規セクションを読み込まない、または誤解して不完全な実装をする
5. 改善の効果がない（または副作用がある）ことが後続フェーズで判明

**技術的背景**:
- テンプレート改善の検証は、実際にサブエージェントを起動して成果物を確認する必要がある
- 単純な「ソースコード検査」では不十分（テンプレート文字列が正しくても、subagent が誤解する可能性）
- テンプレート改善の「受け入れ基準」を明示する必要がある

**影響度**:
- テンプレート改善の意図が実現されない
- 次回以降のタスクで「テンプレートを改善したはずだが、問題が再発している」という混乱が生まれる
- 改善プロセス自体の信頼性が低下

**根本原因要因**:
1. テンプレート改善の「受け入れ基準」が明示されていない
2. 改善後の検証タイミングが不明確（同フェーズ内か、別途テストタスクか）
3. サブエージェントの「テンプレート実装度」を測定する仕組みがない

## セキュリティ要件

### SR-1: ファイル操作の完全性（TS-1 対応）

**要件**: ルート直下のファイル削除が失敗しても、代替手段により完全に削除されること。削除完了を git status で確認できること。

**実装ガイダンス**:
- rm コマンドブロック時の代替手段を明示的に実装（個別削除、ツール使用等）
- 各削除試行の結果をログに記録し、最終的に全ファイルが削除されたことを確認
- 削除失敗時は Orchestrator 或いは subagent が例外処理を実施

---

### SR-2: Git 追跡対象の確保（TS-2 対応）

**要件**: 永続設計ドキュメントが確実に docs/spec/ 以下に配置され、git 追跡対象に含まれること。.gitignore による除外がないこと。

**実装ガイダンス**:
- docs/spec/diagrams/修正プロセス.flowchart.mmd が存在することを確認
- git ls-files で追跡対象に含まれていることを確認
- docs/workflows/ に残存ファイルがないことを確認

---

### SR-3: テンプレート一貫性（TS-3, TS-4 対応）

**要件**: ワークフロー全体のテンプレートにおいて、一時ファイル配置・アーティファクト移動ルールが統一されていること。CLAUDE.md の原則と乖離がないこと。

**実装ガイダンス**:
- refactoring, ui_design, docs_update のテンプレートに「一時ファイル配置ルール」「アーティファクト移動責任」を追加
- 各テンプレート内の説明が CLAUDE.md の「テスト出力・一時ファイルの配置ルール」と一貫していることを確認
- 他フェーズ（test_impl, implementation 等）でも必要に応じて同様の改善を検討

---

### SR-4: サーバー状態の確認（TS-5 対応）

**要件**: テンプレート修正後、MCP サーバーが再起動され、改善されたテンプレートが有効であることを確認すること。

**実装ガイダンス**:
- definitions.ts 修正後、npm run build でコンパイル
- MCP サーバープロセスを再起動
- workflow_status で現在のフェーズと対応するテンプレートが改善版であることを確認
- 次サブエージェント起動時に改善内容が配信されていることを検証

---

### SR-5: テンプレート検証の実施（TS-6 対応）

**要件**: テンプレート改善の効果を検証するプロセスが実装されること。改善されたテンプレートでサブエージェントが正しく実装することを確認すること。

**実装ガイダンス**:
- テンプレート改善の「受け入れ基準」を明示（新規セクションが存在する、説明が明確である、等）
- 次タスク実行時にサブエージェント成果物でテンプレート改善の反映を確認
- 改善内容が実装されていない場合は、Orchestrator がサブエージェントを再起動してリトライ

---

## セキュリティ対策チェックリスト

本タスク完了時に以下を確認すること：

- **TS-1 対策**: git status で verify-templates.js, full-template-verify.js, detailed-verify.js が表示されないこと
- **TS-2 対策**: git ls-files で docs/spec/diagrams/修正プロセス.flowchart.mmd が含まれていることを確認
- **TS-3 対策**: definitions.ts の refactoring, ui_design, docs_update テンプレートに新規セクション（一時ファイル配置ルール、アーティファクト移動責任）が正しく記載されていることを確認
- **TS-4 対策**: ワークフロー全体で一時ファイル配置ルールが一貫していることを確認（CLAUDE.md の「テスト出力・一時ファイルの配置ルール」との整合性確認）
- **TS-5 対策**: MCP サーバーが再起動されていることを確認（npm run build → プロセス再起動）

## 実装上の注意点

### 対策の相互依存関係

- **TS-1 と TS-3**: ファイル削除の代替手段がテンプレートに明示されることで、subagent の判断基準が明確になり、TS-1 の再発を防止
- **TS-2 と TS-4**: アーティファクト移動責任がテンプレートで明示されることで、永続ドキュメントの漏れを防止。同時にワークフロー全体の設計思想を確立
- **TS-5 と TS-6**: MCP サーバー再起動の確認と改善テンプレートの検証が連携。サーバー再起動後に改善内容が実際に有効か確認できる

### 段階的な対策実施

1. **即座の問題解決**（FR-1, FR-2）: ファイル削除、アーティファクト永続化
2. **テンプレート改善**（FR-3, FR-4, FR-5）: definitions.ts の修正
3. **検証・確認**: git status, git ls-files, workflow_status, 次フェーズ成果物の確認
