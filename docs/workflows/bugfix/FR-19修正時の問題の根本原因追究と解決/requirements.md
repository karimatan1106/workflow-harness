# FR-19修正時の問題の根本原因追究と解決 - requirements フェーズ

## サマリー

前回のFR-19タスク（ID: 20260224_215845）の実装後、3つの副次問題が発生した。本フェーズでは、これら問題の根本原因を分析し、以下のアクションアイテムを定義する：

1. ルート直下に残存した3つの検証スクリプト（verify-templates.js, full-template-verify.js, detailed-verify.js）の削除
2. 未追跡ファイル docs/spec/diagrams/修正プロセス.flowchart.mmd の永続配置への移動確認
3. 将来のサブエージェントが同様の問題を回避できるよう、ワークフローテンプレートに一時ファイル配置ルールを明示化

これらの対応により、リポジトリの整合性を回復し、開発プロセスの安定性を向上させる。

---

## 機能要件

### FR-1: ルート直下の一時検証スクリプト削除

**目的**: ルートディレクトリに残存した不要なスクリプトファイルを完全削除し、git status をクリーンな状態に戻す

**削除対象ファイル**:
- /c/ツール/Workflow/verify-templates.js
- /c/ツール/Workflow/full-template-verify.js
- /c/ツール/Workflow/detailed-verify.js

**削除方法**:
- 推奨手段: Bash の rm コマンドを使用して個別に削除
- 代替手段: コマンドがブロックされた場合は Read / Edit / Write ツールを使用した代替削除方法を実施
- 削除後に git status で未追跡ファイルが消去されていることを確認

**根本原因の解析**:
前回タスクの refactoring フェーズで、ヒアドキュメント実行（`node << 'EOF'`）がフックでブロックされたため、代替案としてスクリプトをルートに直接書き込む判断が行われた。その後、削除時に「複数ファイルを単一の rm コマンドで削除する」際に「コマンドチェーン違反」と誤判定されてブロックされ、ファイルが残存した。この状況は以下の設計不足によるものである：

- subagent がルート直下への一時ファイル配置を避けるべき旨の明示的な指示がテンプレートに含まれていなかった
- 削除コマンドがブロックされた時点で、代替手段（個別削除、または .tmp/ への配置変更）を講じるべき判断材料が不足していた

### FR-2: 未追跡ファイル docs/spec/diagrams/修正プロセス.flowchart.mmd の永続配置確認

**目的**: ワークフロー成果物フォルダ（docs/workflows/...）に一時配置されていたフローチャート図を、永続的なドキュメントとして docs/spec/diagrams/ に移動し、git 追跡対象にする

**実施内容**:
1. docs/spec/diagrams/修正プロセス.flowchart.mmd が既に docs/spec/diagrams/ に存在するか確認
2. ファイルが存在しない場合、docs/workflows/FR-19実装で発生した問題の根本原因調査と残課題解決/ フォルダから同名ファイルをコピー
3. ファイルが存在する場合、内容の完全性を確認（Mermaid 文法の妥当性、UI設計フェーズの要件を全て満たしているか）
4. git add で追跡対象に含めることを確認

**受け入れ基準**:
- docs/spec/diagrams/修正プロセス.flowchart.mmd がリポジトリルートから確認できる
- ファイルが git 追跡対象（.gitignore による除外なし）
- 内容が有効な Mermaid flowchart-TD 形式で記述されている

**根本原因の解析**:
修正プロセス図は前回タスク（ID: 20260224_215845）の ui_design フェーズで作成された。CLAUDE.md の「ワークフロー成果物の配置先」ルールに従い、デフォルトの docsDir（docs/workflows/{taskName}/）に配置されたため、.gitignore による除外対象となり git 追跡対象外になった。本来これは「永続的な設計ドキュメント」であり、docs_update フェーズで手動で docs/spec/diagrams/ に移動する必要があったが、その指示が明示化されていなかった。

### FR-3: テンプレート改善 - refactoring フェーズへの一時ファイル配置ガイダンス追加

**目的**: refactoring フェーズの subagent テンプレートに、一時ファイル配置ルールを明示化し、ルート直下への配置を明確に禁止する

**テンプレート改善内容**:
- refactoring フェーズのテンプレートに以下のセクションを追加
  - 「## 一時ファイル配置ルール」
  - ヒアドキュメント実行や複雑なスクリプト実行がブロックされた場合の対応ガイダンス
  - 一時スクリプトは `.tmp/` ディレクトリに配置し、フェーズ完了前に削除すること
  - ルート直下にはいかなる .js, .py ファイルも配置しないこと

**CLAUDE.md からの参照**:
- 「テスト出力・一時ファイルの配置ルール」セクションの内容を subagent テンプレートで要約・引用する
- ルート直下への配置が禁止であることを明示（「禁止事項: ルートディレクトリへの以下の配置は禁止」）

**展開対象フェーズ**:
- 直接対象: refactoring
- 関連対象: test_impl（テストファイル配置）

### FR-4: テンプレート改善 - ui_design フェーズへのアーティファクト移動指示追加

**目的**: ui_design フェーズの subagent テンプレートに、作成した Mermaid ファイルの永続配置ルールを明示化する

**テンプレート改善内容**:
- ui_design フェーズのテンプレートに以下のセクションを追加
  - 「## 成果物の永続化（docs_update フェーズでの処理）」
  - ステートマシン図・フローチャート等の .mmd ファイルは、一時的に docsDir に配置されることを説明
  - docs_update フェーズでこれらのファイルを docs/spec/diagrams/ に移動する責任が生じることを明記
  - デザインドキュメント（.md）についても同様に docs/spec/screens/, docs/spec/components/ への移動が必要であることを説明

**関連セクション**:
- CLAUDE.md の「ドキュメント構成」→「プロダクト仕様への反映」ルールを要約・引用する

### FR-5: テンプレート改善 - docs_update フェーズへのアーティファクト移動責任明示

**目的**: docs_update フェーズの subagent テンプレートに、ワークフロー成果物からプロダクト仕様フォルダへの移動責任を明確に示す

**テンプレート改善内容**:
- docs_update フェーズのテンプレートの「## 作業内容」セクションに以下を追加
  - 前回のワークフロー（docs/workflows/{previousTaskName}/）フォルダの全成果物（.mmd, .md）を確認
  - 設計図（.state-machine.mmd, .flowchart.mmd）を docs/spec/diagrams/ に移動
  - 永続的なドキュメント（機能仕様、API仕様、画面設計等）を docs/spec/features/, docs/spec/api/, docs/spec/screens/ に移動
  - 一度移動したら、元のワークフロー フォルダ内ファイルを確認してクリーンアップ（ただし .gitignore により自動除外される）

**チェックリスト形式での提示**:
- [ ] .state-machine.mmd ファイルが docs/spec/diagrams/ に存在するか確認
- [ ] .flowchart.mmd ファイルが docs/spec/diagrams/ に存在するか確認
- [ ] .md ファイル（仕様書）が対応する docs/spec/* フォルダに存在するか確認
- [ ] git add で追跡対象になっていることを確認

## 非機能要件

### NF-1: 削除プロセスの堅牢性

**要件**: ルート直下のファイル削除が失敗した場合、Bash コマンドブロックに依存せず代替手段で完了させる

**実装ガイダンス**:
- rm コマンドがブロックされた場合は Read ツールでファイル存在確認後、Edit ツール（ファイル内容編集）または直接 Read しないアプローチで代替削除を試みる
- またはファイル管理ツール（git rm）使用の検討
- 削除完了後に git status でファイルが除去されていることを確認

### NF-2: ドキュメント追跡可能性

**要件**: 全ての永続的なドキュメント（設計図、仕様書）が git 追跡対象になり、.gitignore により除外されないようにする

**実装ガイダンス**:
- docs/spec/ 以下のファイルは全て git add 対象
- docs/workflows/ 以下のファイルは全て .gitignore により除外（一時的な作業フォルダ）
- 移動完了後に git ls-files でファイルが追跡対象になっていることを確認

### NF-3: テンプレート一貫性

**要件**: ワークフロー全体のテンプレートにおいて、一時ファイル配置ルール・アーティファクト移動ルールが統一されていること

**実装ガイダンス**:
- CLAUDE.md の既存ルール（「テスト出力・一時ファイルの配置ルール」等）に一貫したテンプレート文言を追加
- 複数フェーズにまたがるルール（ui_design → docs_update）の場合、各フェーズのテンプレートで役割分担を明記
- subagent が迷わないよう、フェーズ遷移時の引き継ぎ情報を明示化

### NF-4: エラーハンドリング

**要件**: subagent がコマンドブロックや予期しない失敗に遭遇した場合、単に処理を続行するのではなく、原因分析と代替手段を講じるプロセスが組み込まれていること

**実装ガイダンス**:
- テンプレートに「ヒアドキュメント実行や複雑なスクリプト実行がブロックされた場合」「削除コマンドが失敗した場合」などの具体的なシナリオを列挙
- 各シナリオに対する代替手段（.tmp/ 配置、分割削除、ツール使用等）を提示
- 代替手段での完了確認（git status, git ls-files 等）を指示

## ユーザー意図との整合性

ユーザーの意図は「前回のFR-19実装プロセスで発生した副次問題の根本原因を追究し、将来の同様問題を回避できるワークフロー設計に改善する」である。

本要件定義では以下の観点でその意図を反映している：

1. **即座の問題解決**（FR-1, FR-2）: 残存ファイルの削除と未追跡ファイルの永続化
2. **根本原因分析**: 各問題について「なぜ起きたのか」を技術的に解析
3. **設計改善**（FR-3, FR-4, FR-5）: テンプレートに明示的なルール・ガイダンスを追加し、subagent が判断しやすい環境を構築
4. **再発防止**: 同じ失敗パターンが将来繰り返されないよう、判断基準をプロンプトレベルで強化

これにより、ワークフロー全体の堅牢性が向上し、開発プロセスがより安定する。

| フェーズ | 作業内容 | 成果物 | 担当 |
|---------|---------|--------|------|

本要件定義を foundation として、次フェーズ（planning）で詳細な実装計画を策定する。
