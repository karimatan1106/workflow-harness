# parallel_verification大量リトライの根本原因解決 - UI設計

## サマリー

本タスクはCLI/バックエンドの変更であり、GUIの画面設計は対象外です。
本ドキュメントではartifact-validatorが出力するエラーメッセージの形式、
MCPサーバーのAPIレスポンス構造、CLIインターフェース設計、
および設定ファイルの外部化方針を定義します。

- 目的: artifact-validatorの出力をsubagentが解析しやすい形式に統一し、
  FR-1（CLAUDE.md更新）・FR-2（isStructuralLine拡張）・FR-3（重複ファイル削除）の
  各変更がユーザーおよびOrchestratorに対してどのように見えるかを明確化する
- 主要な決定事項:
  - エラーメッセージは種別コードを先頭に付与する形式とし、Orchestratorが自動解析できる構造を維持する
  - MCPサーバーのレスポンスはJSON構造として、validationResultフィールドを必ず含める
  - 設定の外部化は本タスクのスコープ外とし、将来課題の方針のみを記述する
- 次フェーズで必要な情報:
  - isStructuralLine関数のルール8変更後の正規表現パターンが確定すれば、
    エラーメッセージの実際の出力形式をtesting・manual_testフェーズで検証できる
  - artifact-validator.tsのビルド後に、実際のCLI出力を確認してこのドキュメントと照合すること（例示出力）

## CLIインターフェース設計

本タスクで変更されるartifact-validatorは、MCPサーバー内部から呼び出されるモジュールです。
直接的なCLI実行機能は持ちませんが、MCPサーバーを通じてOrchestratorへ返すメッセージが
実質的なCLIインターフェースとして機能します。

### 成功時の出力形式

バリデーションが全ての品質要件を通過した場合、MCPサーバーは以下の形式で結果を返します。
出力は単行JSON形式を基本とし、人間が読みやすいフォーマットは採用しません。

成功時レスポンスは以下の構造を持ちます：statusフィールドで"ok"を返し、phaseフィールドに
対象フェーズ名、artifactPathフィールドに成果物ファイルパスを含めます。
checksフィールドは辞書形式で各チェック項目の結果を格納し、duplicateLines、sectionDensity、
requiredSections、forbiddenPatternsの各キーに対して"pass"の値を返します。

成功時の出力にはエラー詳細フィールドは含まれません。
checksフィールド内の各項目がすべて"pass"であることが合格条件です。
このシンプルな構造により、Orchestratorは成功判定を最小限のJSON解析で実施できます。
workflow_nextおよびworkflow_complete_subの応答に埋め込まれ、フェーズ移行の可否を伝えます。

### バリデーション失敗時の出力形式

失敗時は詳細なエラー情報をOrchestratorに返します。
以下はmanual_testフェーズのプレーンラベル行重複エラーの出力例の仕様です。

statusフィールドで"error"を返し、phaseフィールドに対象フェーズ名、
artifactPathフィールドに成果物ファイルパスを含めます。
errorsフィールドは複数エラー情報を順序付きリスト形式で返し、
各エラーはcodeフィールド（エラー種別）、severityフィールド（重大度），
lineフィールド（行番号）、contentフィールド（エラー行の内容）、
occurrencesフィールド（重複出現回数）、messageフィールド（エラー説明）を持ちます。
hintフィールドには修正のための具体的なアドバイスが記述されます。

例えば、DUPLICATE_LINEエラーが発生した場合、codeフィールドに"DUPLICATE_LINE"が格納され、
severity値は"error"となり、line値は42（エラーが検出された行番号）となります。
contentフィールドには重複が検出された行の実際の内容「実行結果: 成功」が格納され、
occurrences値は3（3回出現）となります。
Orchestratorはこのcodeフィールドを参照してリトライプロンプトを生成する際の分岐条件として利用します。

### リトライ時のOrchestratorへの案内出力

Orchestratorがsubagentを再起動する際、MCPサーバーは以下のメタ情報を追加出力します。
この出力はOrchestratorがリトライプロンプトを組み立てる際の素材として使用されます。

```
バリデーション失敗（1回目）: manual_test
- エラー種別: DUPLICATE_LINE
- 影響行数: 4行
- 推奨修正: 各行にシナリオ識別子を付加して一意にしてください
- リトライプロンプト例: 「シナリオ2（ファイル書き込みテスト）の実行結果: 成功」
```

このテキスト形式の案内はJSON埋め込みの補足として提供され、
開発者がMCPサーバーのログから問題を追跡する際にも役立ちます。
リトライカウントは1から始まりバリデーション成功まで増加し続けます。

## エラーメッセージ設計

artifact-validatorが検出・報告するエラーの種別とメッセージフォーマットを定義します。
FR-2の変更後、DUPLICATE_LINEエラーの発生頻度は減少しますが、
他のエラー種別は引き続き同じフォーマットで出力されます。

### エラー種別コード体系

各エラーには種別コードが割り当てられており、Orchestratorが自動分岐できる設計です。

| 種別コード | 深刻度 | 説明 |
|-----------|--------|------|
| DUPLICATE_LINE | error | トリム後に同一内容の行が3回以上出現している |
| SECTION_DENSITY_LOW | error | あるセクション内の実質行の密度が30%を下回っている |
| SECTION_MIN_LINES | error | あるセクション内の実質行数が5行未満である |
| REQUIRED_SECTION_MISSING | error | 必須セクション見出しが成果物に存在しない |
| FORBIDDEN_PATTERN | error | 禁止語または角括弧プレースホルダーが検出された |
| MIN_TOTAL_LINES | error | 成果物全体の行数が最低行数基準を下回っている |

深刻度はerrorとwarningの2種類を定義しますが、現行実装では全てerrorとして扱われます。

### DUPLICATE_LINEエラーのメッセージフォーマット

FR-2の変更前後でのDUPLICATE_LINEエラーの発生条件と出力例を以下に示します。

変更前の動作（FR-2適用前）:
リスト記号なしの行「実行結果: 成功」が3回出現した場合にエラーが発生していました。
エラーメッセージの例を以下に示します。

DUPLICATE_LINE エラーコード：行42に検出された「実行結果: 成功」が3回出現しました（重複検出対象）。
修正方法は各行にシナリオ番号または固有の識別情報を追加して一意にすることです。
修正例として「シナリオ1（初期化処理）の実行結果: 成功」「シナリオ2（基本動作）の実行結果: 成功」
のように各行の内容に文脈固有の情報を付加することが有効です。

変更後の動作（FR-2適用後）:
ルール8が拡張され、リスト記号なし・50文字以内・コロン終端の行は
構造的行として除外されるため、同じパターンでのDUPLICATE_LINEエラーは発生しません。
この変更により、manual_testやsecurity_scanのような検証フェーズでの
誤検出が解消されます。

### FORBIDDEN_PATTERNエラーのメッセージフォーマット

禁止語・禁止パターンが検出された場合のメッセージ形式です。
部分文字列マッチで検出されるため、複合語への注意が必要です。

FORBIDDEN_PATTERN エラーコード：行18に禁止語が検出されました。
検出内容は禁止語を含む箇所であり、行内容の一部は伏字で表示されます。
推奨代替表現についてはspec.mdの禁止語テーブルを参照することで正確な言い換えが可能です。
重要な点として複合語も検出対象となるため、例えば「確認が必要」は「確認が求められる」に
言い換える等の対応が必要となります。

禁止語の具体的な種類はOrchestratorには返さず、
リトライプロンプトにはhintフィールドの内容のみが含まれます。
これは誤ったコンテキスト引き継ぎによる再発を防ぐための設計上の判断です。

### SECTION_DENSITY_LOWエラーのメッセージフォーマット

セクション密度が不足している場合のメッセージ形式です。
密度は「実質行数を総行数で除した値」として計算され、30%以上が合格基準です。

SECTION_DENSITY_LOW エラーコード：セクション「テスト結果」の密度が18%で、
計算結果は実質行3行を総行17行で除した値となります。
合格基準は密度30%以上であり、目標を達成するには実質行数を5行以上追加する必要があります。
参考として実質行とは空行・水平線・コードフェンス境界を除いた行を指しており、
これらの構造要素はセクション密度計算の母数に含まれません。

このメッセージにより、Orchestratorは不足セクション名と必要な追加行数を
リトライプロンプトに埋め込んで具体的な修正を指示できます。
追加すべき行数を明示することで、subagentが1回のリトライで解決できる情報量を提供します。

## APIレスポンス設計

MCPサーバーのworkflow_complete_subとworkflow_nextが返すレスポンス構造を定義します。
これらのAPIはOrchestratorが各フェーズ完了時に呼び出すエンドポイントです。

### workflow_complete_subのレスポンス構造

parallel_verificationのサブフェーズ（manual_test等）完了時のレスポンスを以下に示します。
バリデーションが成功した場合の応答の構造例です。

workflow_complete_subの成功応答はJSON形式で、以下のフィールドを含みます。
taskIdフィールドにはタスクの一意識別子が格納され、
subPhaseフィールドにはサブフェーズ名が格納され、
statusフィールドには状態を表す文字列「completed」が格納されます。
validationResultは辞書形式で、passedキーに真偽値、checksキーに辞書形式のチェック結果が格納されます。
checksの内部にはduplicateLines、sectionDensity、requiredSections、forbiddenPatterns、minTotalLinesの
各キーに対して「pass」の文字列値が格納されます。
messageフィールドにはサブフェーズの完了を表するメッセージが格納されます。

バリデーションが失敗した場合は、validationResultのpassedが偽となり、
errorsフィールドが追加されます。Orchestratorはこのpassedフィールドを確認して
subagentの再起動要否を判断します。

### workflow_nextのレスポンス構造

フェーズ移行時のレスポンス構造例を以下に示します。
並列フェーズ全サブフェーズが完了した後に呼び出されます。

workflow_nextの応答はJSON形式で、以下のフィールドを含みます。
taskIdフィールドにはタスク識別子が格納され、
previousPhaseフィールドには遷移前フェーズ名が格納され、
nextPhaseフィールドには遷移先フェーズ名が格納されます。
statusフィールドには「transitioned」という文字列が格納されます。
validationResultは辞書形式で、passedキーに真偽値、
subPhaseResultsキーに辞書形式のサブフェーズ別結果が格納されます。
subPhaseResultsの内部にはmanual_test、security_scan、performance_test、e2e_testの
各キーに対して「pass」または「fail」の文字列値が格納されます。
messageフィールドにはフェーズ遷移の完了と遷移先を表するメッセージが格納されます。

subPhaseResultsフィールドにより、Orchestratorはどのサブフェーズで問題が生じたかを
確認できます。全サブフェーズの結果が「pass」の場合のみ移行が実行されます。
いずれかのサブフェーズが「fail」の場合、移行はブロックされ、エラー詳細が返されます。

### バリデーション失敗時のworkflow_complete_subレスポンス

FR-2変更前の状況（参考）として、DUPLICATE_LINEエラーが発生した場合のレスポンス例です。
この形式のレスポンスが返されたとき、Orchestratorはsubagentを再起動する義務があります。

バリデーション失敗時の応答はJSON形式で、以下のフィールドを含みます。
taskIdフィールドにはタスク識別子が格納され、subPhaseフィールドにはサブフェーズ名が格納され、
statusフィールドには「validation_failed」という文字列が格納されます。
validationResultは辞書形式で、passedキーに偽の真偽値が格納され、
errorsキーに複数のエラーオブジェクトを含むリスト形式のデータが格納されます。

各エラーオブジェクトはcodeキー（エラー種別）、severityキー（重大度）、
lineキー（行番号）、contentキー（エラー行の内容）、occurrencesキー（重複出現回数）、
messageキー（エラー説明）を持ちます。例えば、DUPLICATE_LINEエラーでは
codeキーに「DUPLICATE_LINE」が格納され、severityキーに「error」が格納され、
lineキーに42が格納され、contentキーに「実行結果: 成功」が格納され、
occurrencesキーに4が格納され、messageキーに重複の詳細が格納されます。

hintフィールドには修正のための具体的なアドバイスが記述されます。
retryCountフィールドにより、Orchestratorは現在何回目のリトライかを把握できます。
このカウンターはMCPサーバー内部で管理されており、成功時にリセットされます。
Orchestratorはこのカウンターを参照して、リトライプロンプト内にリトライ回数を明示します。

## 設定ファイル設計

isStructuralLineのルール設定を将来的に外部化する場合の設計方針を記述します。
本タスクのスコープは設定外部化を含みませんが、方針を文書化して将来の変更を支援します。

### 現行の設定管理方式

現行実装では、isStructuralLine関数の8つのルールはTypeScriptの正規表現リテラルとして
artifact-validator.ts内にハードコードされています。
ルールの変更はTypeScriptコードの修正・ビルド・MCPサーバー再起動を必要とします。
この方式はシンプルですが、ルール調整のたびにエンジニアの介入が必要という制約があります。

### 外部化の将来方針

将来的にルールを外部設定ファイルで管理する場合、以下の設計アプローチを推奨します。

設定ファイルの形式はJSONまたはYAMLとし、各ルールを配列要素として定義します。
各要素にはルール名・正規表現パターン文字列・説明文・有効/無効フラグを持たせます。
このような構造により、MCPサーバー再起動なしで設定のホットリロードが実現可能となります。

設定ファイルの配置先は `workflow-plugin/mcp-server/config/` ディレクトリとし、
バージョン管理対象に含めることでルール変更の履歴を追跡できます。
設定ファイルが存在しない場合はコード内のデフォルト値にフォールバックする
フェイルセーフ設計とします。

### 現行ルールの定義一覧（将来外部化時の参考）

外部化した場合のルール定義のイメージを以下に記述します。
各ルールは番号・種別・マッチ条件・除外対象の説明で構成されます。

- ルール1: 見出し行 — シャープ記号（1文字以上）で始まる行を除外する
- ルール2: 水平線行 — ハイフン・アスタリスク・アンダースコアの3文字以上の繰り返しを除外する
- ルール3: コードフェンス境界行 — バックティック3文字以上で始まる行と、フェンス内の全行を除外する
- ルール4: テーブルセパレータ行 — パイプ区切りでコロンとハイフンのみを含む行を除外する
- ルール5: テーブルデータ行 — パイプ区切りで2列以上を持つ行を除外する
- ルール6: 太字ラベル行 — 2連アスタリスクで囲まれたラベルのみの行（行末コロン）を除外する
- ルール7: リスト先頭の太字ラベル行 — リスト記号に続く太字ラベルのみの行を除外する
- ルール8（FR-2変更後）: 短いラベル行 — リスト記号の有無を問わず、50文字以内でコロン終端の行を除外する

この一覧はCLAUDE.mdの「重複検出から除外される行」セクションの記述と一致させることで、
ドキュメントと実装の整合性を保証します。設定外部化の際にはFR-1の更新内容を基に
設定ファイルのスキーマ定義を策定してください。
