## サマリー

- 目的: `workflow-plugin/mcp-server/src/phases/definitions.ts` の `buildPrompt()` 関数内 qualitySection（行 1155〜1162 付近）に追加された正規表現パターンおよび配列アクセス記述に関する角括弧警告ガイダンスのセキュリティスキャンを実施する。
- スキャン範囲: 行 1130〜1188 の qualitySection 文字列構築ロジック、および `artifact-validator.ts` の `extractNonCodeLines`・`bracketPlaceholderPattern` との整合性検証。
- 主要な決定事項: 修正箇所はすべて TypeScript ソースコードに埋め込まれた固定の文字列リテラルであり、実行時に外部入力を受け取る処理を持たない。プロンプトインジェクションの経路は存在しないと判断した。
- 検出結果概要: 深刻度が高い脆弱性は検出されなかった。ただし、軽微なリスクとして「サブエージェントへの説明文が実際のバリデーター動作と微妙に異なる記述を含む可能性」を 1 件記録した（詳細は「検出された問題」セクション参照）。
- 次フェーズで必要な情報: performance_test および e2e_test フェーズは本修正箇所と直接関係がないが、MCP サーバーの再ビルドが完了していることを前提として検証を進めること。

## 脆弱性スキャン結果

### 観点 1: プロンプトインジェクション

修正箇所（行 1155〜1162）はすべて TypeScript の文字列リテラルとして記述されており、ユーザー入力・ファイル入力・環境変数等の外部データを動的に展開する処理は含まれていない。
`qualitySection += \`...\`` の形式で追記される各行は、コンパイル時に確定した固定テキストである。
外部から注入された文字列がサブエージェントへのプロンプトに混入する経路は、本修正箇所には存在しない。
スキャン結果: 問題なし（プロンプトインジェクションの経路が確認されなかった）。

### 観点 2: 情報漏洩

行 1150〜1162 のプロンプト追記内容を全文確認した。APIキー、パスワード、シークレットトークン、内部ホスト名、データベース接続文字列等の機密情報はいずれも含まれていない。
`rules.bracketPlaceholderInfo.allowedKeywords.join('、')` という動的展開（行 1152）は allowedKeywords 配列の内容をサブエージェントに開示するが、この配列は公開設計情報であり機密性はない。
スキャン結果: 問題なし（機密情報の漏洩経路が確認されなかった）。

### 観点 3: サブエージェント誤誘導リスク

追加されたガイダンスは「コードフェンス外に正規表現の文字クラス表記を直接記述してはならない」「コードフェンス内であれば安全」という 2 点の行動指示をサブエージェントに与える。
この記述がサブエージェントを意図しない動作（たとえばコードフェンスの改ざん・無限ループ・不正ファイルアクセス等）に誘導するリスクは認められなかった。
行 1157 の「OK（推奨）: コードフェンス内に正規表現パターンを記述すること」という指示は、バリデーターの仕様と合致した正当な代替手段の提示である。
行 1158 の「OK（代替）: コードフェンス外では散文形式で説明すること」という指示も、バリデーターが期待する安全な記述方式を正確に示している。
スキャン結果: 問題なし（意図しない動作への誘導リスクは確認されなかった）。

### 観点 4: バリデーターとの整合性

`artifact-validator.ts` の `extractNonCodeLines()` 関数（行 155〜）は、コードフェンス内の行を検出対象から除外することを確認した。
`bracketPlaceholderPattern`（`/\[(?!関連|参考|注|例|出典)[^\]]{1,50}\]/g`）は非コード行のみに適用されるため、コードフェンス内の記述は角括弧検出から除外される。
行 1153〜1154 のプロンプト記述「コードフェンス内であれば配列アクセス記法や正規表現の文字クラス表記を安全に記述できる」は、バリデーターの実際の動作（`extractNonCodeLines` による除外）と整合している。
行 1156 の NG 例示「コードフェンス外の散文や箇条書きに正規表現の文字クラス表記を直接記述すること」は、バリデーターが検出対象とする状況と正確に一致している。
スキャン結果: 軽微な懸念事項 1 件（後述）。バリデーター動作と説明文の主要部分は整合していると判断した。

## 検出された問題

### 問題 1: 行 1153 の説明文とインラインコード除去の挙動に関する表記の精度

深刻度: 低（実害なし、ドキュメント精度の問題）

行 1153 のプロンプト記述には「コードフェンス内であれば配列アクセス記法や正規表現の文字クラス表記を安全に記述できる」と記載されており、これ自体は正確である。
しかし同じ行の後半で「シングルバックティック内のインラインコードも除去されるため安全である」という旨の説明が省略されていることを確認した。
`artifact-validator.ts` の `removeInlineCode()` 関数（行 128〜）はシングルバックティックで囲まれたインラインコード部分を空文字列に置換する処理を実装しており、コードフェンス外のインラインコード内に記述された角括弧も実質的に検出対象から除外される。
この挙動がプロンプト内で明示されていないため、サブエージェントが「コードフェンス以外の安全な代替手段」としてインラインコードを活用できない可能性がある。
ただしこの問題は機能的な誤動作を引き起こすものではなく、プロンプトの有用性に関する軽微な不足である。バリデーターが想定外の方向に動作するリスクはない。

対処の必要性: 低優先度で対処を推奨する。将来の改善として、行 1153〜1154 付近に「シングルバックティックで囲まれたインラインコード内も検出対象から除外される（removeInlineCode による）」という補足を追記することを提案する。

### 問題 2 以降

その他の観点（プロンプトインジェクション・情報漏洩・サブエージェント誤誘導）においては、深刻度が中以上の問題は検出されなかった。
修正箇所が固定文字列のみで構成されているため、攻撃対象となる動的処理が存在せず、スキャンの結論は全体として「合格」と判定する。
本スキャンの根拠として、修正対象ファイルのソースコード（行 1155〜1162）と `artifact-validator.ts` の `extractNonCodeLines`・`bracketPlaceholderPattern`・`removeInlineCode` の各実装を直接照合した。
バリデーターの実際の動作と、サブエージェントへのプロンプトが伝える行動指示の間に矛盾は認められなかった。
