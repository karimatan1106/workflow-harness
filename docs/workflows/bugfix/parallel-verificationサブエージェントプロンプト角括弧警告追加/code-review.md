## サマリー

本ドキュメントは `definitions.ts` の `buildPrompt()` 関数内 NG/OK 例修正（行 1155〜1162 の8行）に対するコードレビュー結果をまとめたものである。

- 目的: spec.md に定義された受け入れ基準 AC-1〜AC-4 に対して、実装コードが正しく適合しているかを検証する
- 変更の背景: 修正前の `buildPrompt()` は「コードブロックへの記述が NG」と誤ったプロンプトを全フェーズのサブエージェントに送信しており、parallel_verification の4サブエージェントがバリデーションエラーを連続発生させた根本原因となっていた
- 主要な決定事項: 全4つの受け入れ基準がいずれも「合格」と判定された。設計と実装の整合性は確認済みであり、コード品質・セキュリティ・パフォーマンスの各観点でも問題は検出されなかった
- 修正の影響範囲: 変更は `qualitySection` 生成部の6行から8行への書き換え1箇所のみであり、`buildPrompt()` を通じて全フェーズおよび全サブフェーズのサブエージェントプロンプトに効果が波及する
- 次フェーズで必要な情報: testing および regression_test フェーズでは `workflow-plugin/mcp-server` ディレクトリで `npx vitest run` を実行し、既存テストスイートの継続的通過を確認すること。また MCP サーバー再起動後にサブエージェントプロンプトの実際の出力を確認することが推奨される

## 設計-実装整合性

spec.md に記載された受け入れ基準との照合を実施した。以下に4項目の検証結果を示す。

AC-1 の検証では「コードフェンス外の散文や箇条書きへの直接記述が NG」と明示されているかを確認した。実装コードの行 1156 は「コードフェンス外の散文や箇条書きに正規表現の文字クラス表記を直接記述すること」を NG として定義しており、行 1160 は「コードフェンス外の散文や箇条書きに配列のインデックスアクセス記法を直接記述すること」を NG として定義している。修正前の誤記（「コードブロックに書く」「コードブロック内に直接記述する」という逆方向の NG 記述）は完全に除去されており、AC-1 は合格と判定する。

AC-2 の検証では「コードフェンス内への記述が OK（推奨）として明示されているか」を確認した。行 1157 が「コードフェンス内に正規表現パターンを記述すること（バリデーターはコードフェンス内の行を検出対象から除外するため安全）」という OK（推奨）行として追加されており、行 1161 が「コードフェンス内に配列アクセス記法を記述すること（バリデーターはコードフェンス内の行を検出対象から除外するため安全）」という OK（推奨）行として追加されている。両セクションにバリデーターの動作原理（extractNonCodeLines による除外）への言及が含まれており、AC-2 は合格と判定する。

AC-3 の検証はビルド成功確認であり、このフェーズでは静的なコード読み取りのみ実施する。対象の8行はいずれも既存の `qualitySection += \`...\n\`` という構文パターンを踏襲しており、コードフェンス区切り文字（バッククォート3個）はいずれの行にも含まれていない。したがって TypeScript テンプレートリテラルのエスケープ問題は発生しないと判断する。build_check フェーズでのビルド実行結果が AC-3 の最終確認となる。

AC-4 の検証では「既存テストに影響しないか」を確認した。変更対象は `qualitySection` という文字列変数への追記のみであり、`buildPrompt` 関数のシグネチャ・戻り値の型・`PHASES` 配列・`resolvePhaseGuide` 関数の動作は一切変更されていない。`definitions.test.ts` が検証する項目（フェーズ数、遷移ロジック、プレースホルダー展開）は本変更の影響を受けないため、AC-4 は合格と判定する。

## コード品質

実装された8行のコードを構文・一貫性・可読性の観点から分析した。

構文の正確性については、8行全てが既存コードと同一の `qualitySection += \`...\n\`` 構文を使用しており、テンプレートリテラルの開始・終了バッククォートが正しく対応している。各行末の改行文字 `\n` も一貫して付与されており、生成されるプロンプト文字列の改行構造に問題はない。

ラベル表記の一貫性については、両セクション（正規表現パターン・配列アクセス）とも「NG」「OK（推奨）」「OK（代替）」という3段階の対比構造で統一されており、CLAUDE.md でも同様のパターンが採用されていることと整合している。

行数の変化については、修正前の6行（各セクション3行: ヘッダー・NG・OK各1行）から修正後の8行（各セクション4行: ヘッダー・NG・OK推奨・OK代替各1行）への増加であり、spec.md の記載（「6行から8行に増加」）と一致している。

変更の局所性については、行 1155〜1162 の8行のみが変更対象であり、その前後の行（行 1153〜1154 の説明文および行 1163 以降の重複行禁止セクション）は変更されていない。spec.md の「変更しない行の確認」セクションに記載された内容が正しく保持されている。

OK（代替）行の内容については、正規表現パターンセクションの OK（代替）行「英小文字の1文字以上の繰り返しを表す正規表現」という散文説明と、配列アクセスセクションの OK（代替）行「先頭要素を取得する」「インデックス番号によるアクセス」という散文説明は、いずれもコードフェンス外で安全に使用できる表現形式であり適切である。

## セキュリティ

修正内容がサブエージェントに誤誘導を与えないか、バリデーターとの整合性を検証した。

バリデーターとの整合性については、修正後のプロンプトが「コードフェンス外が NG」と明示することは、`artifact-validator.ts` の `extractNonCodeLines` 関数の実際の動作と完全に一致している。サブエージェントに提供する指示内容が技術的事実と一致しているため、誤誘導の問題は存在しない。

コードフェンス安全性の根拠提示については、OK（推奨）行の括弧内に「バリデーターはコードフェンス内の行を検出対象から除外するため安全」という理由が明記されている。この根拠は実際のバリデーター動作（`extractNonCodeLines` による除外処理）と一致しており、サブエージェントが理由を理解した上で判断できる形式になっている。

プロンプトインジェクションのリスクについては、修正箇所は純粋な文字列リテラルであり、外部入力を受け取るものではない。`qualitySection` の内容はコンパイル時に固定されるため、プロンプトインジェクション攻撃の攻撃面は存在しない。

過去の誤記の完全除去については、修正前に問題の根本原因となっていた「コードブロックに書く」および「コードブロック内に直接記述する」というフレーズが行 1156 と行 1160 から完全に削除されており、同一の誤解を引き起こすフレーズは行 1153〜1162 の全範囲で検出されなかった。

## パフォーマンス

文字列追加の行数・長さおよびプロンプト生成への影響を評価した。

文字列長の増加については、8行の `qualitySection +=` 代入によって追加される文字数は合計でおよそ400〜450文字程度であり、これはプロンプト全体（数千文字規模）の数パーセントに相当する。この増加量はサブエージェントのコンテキスト消費量に対して軽微であり、パフォーマンス上の問題を引き起こすものではない。

行数の適切性については、spec.md は「各セクションに OK 代替行を1行追加するため行数が増加する」と明記しており、6行から8行への変化は仕様書の記載通りの増分である。サブエージェントへの情報提供密度（NG の理由・OK 推奨の理由・代替手段）が向上することによる品質向上効果は、わずかな行数増加コストを上回ると評価する。

関数呼び出しへの影響については、`buildPrompt()` 関数は各フェーズのプロンプト生成時に一度呼び出されるものであり、ホットパスに位置しない。文字列連結操作の追加が実行時パフォーマンスに与える影響は無視できるレベルである。

スコープの適切性については、修正が `buildPrompt()` 内の `qualitySection` 生成部分に集中しているため、全フェーズおよび全サブフェーズのサブエージェントプロンプトに効果が波及する設計になっている。この一箇所の変更で parallel_verification の4サブフェーズを含む全フェーズが恩恵を受ける点は、コード変更の費用対効果として適切な設計判断である。
