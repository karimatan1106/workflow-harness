## サマリー

このドキュメントは、`buildPrompt()` 関数内のNG/OK例逆転バグを修正する変更に対して行った脅威分析の結果をまとめたものである。
分析の目的は、安全な実装方針を確立し、修正作業中および修正後に発生しうるリスクを事前に特定して対策を示すことである。
変更範囲は `definitions.ts` の行 1156〜1160 の文字列定数6行に限定されており、コアロジック（バリデーター本体、状態管理、フェーズ遷移）への副作用は存在しない。
影響範囲は全フェーズへのプロンプト生成テキストに限られるが、全サブエージェントの動作指針に直接関わるため、修正の正確性は非常に重要である。
MCPサーバーのモジュールキャッシュ制約を考慮し、修正後はトランスパイル（`npm run build`）とサーバー再起動が必須の手順として計画フェーズで明示すること。
修正後の文字列内容を既存の単体テストが検証しているかを確認し、テスト更新の必要性を計画フェーズで判断すること。
修正前後のdiff確認と再起動後の動作確認手順についても計画フェーズで策定する必要がある。

## 変更概要と対象ファイル

本タスクの変更対象は `workflow-plugin/mcp-server/src/phases/definitions.ts` の `buildPrompt()` 関数内にある文字列定数の6行である。
具体的には、角括弧プレースホルダー禁止セクションの「正規表現パターンの記述」と「配列アクセスの記述」に関するNG/OK例が実際のバリデーター動作と逆になっている。
バリデーターの `extractNonCodeLines` 関数はコードフェンス内の行を検出対象から除外するため、コードフェンス内への記述は完全に安全である。
現行のNG/OK例はコードフェンス内への記述をNG、散文形式での説明をOKとしているが、正しくはコードフェンス外の散文記述がNGでコードフェンス内への記述がOKである。
この誤りが全フェーズのサブエージェントに誤った理解をもたらし、parallel_verification の4サブエージェントが連続してバリデーションエラーを発生させた根本原因となっている。

## 脅威カテゴリ 1: 修正内容のリグレッション

### 脅威 T1-1: NG/OK例の再逆転

修正作業中に誤って例の方向をさらに反転させてしまうリスクがある。
修正前の状態が「コードフェンス内 = NG、散文 = OK」であり、修正後の正しい状態は「コードフェンス外の散文 = NG、コードフェンス内 = OK」である。
修正者がバリデーターの動作原理を誤解したまま編集を行うと、三度目の逆転が発生する可能性がある。
この場合、サブエージェントの誤動作が継続し、全フェーズのバリデーションエラーが解消されない。

対策として、修正前に `artifact-validator.ts` の `extractNonCodeLines` 関数の実装を確認し、コードフェンス除外ロジックを明文化してから編集を行うこと。

### 脅威 T1-2: 隣接行への意図しない変更

行 1156〜1160 を編集する際に、隣接する行 1153〜1155 および行 1161 以降に誤って変更が及ぶリスクがある。
行 1153〜1154 は現在正しい記述（「コードフェンス外が検出対象、コードフェンス内は安全」）であり、変更してはならない。
テキストエディタでの編集ミスや、文字列結合パターンの誤認識によって隣接行を上書きする事故が起こりうる。

対策として、変更後に `git diff` で差分を厳密に確認し、変更対象の6行のみが修正されていることを検証すること。

### 脅威 T1-3: 文字列エスケープの破損

TypeScript の文字列リテラル内にバックスラッシュ、バッククォート、ドル記号等が含まれる場合、編集時にエスケープが崩れてコンパイルエラーが発生するリスクがある。
現行の行 1157 には `npm run` のような安全な文字列しか含まれないが、新しいOK例にコードフェンス記法（バッククォート3連）を散文で説明する場合はエスケープが必要になる。
ビルドエラーが発生した場合、MCPサーバーの再起動ができなくなり、ワークフロー全体が停止するリスクがある。

対策として、修正後に `npm run build` を実行し、コンパイルエラーがないことを確認してから再起動に進むこと。

## 脅威カテゴリ 2: MCPサーバーキャッシュに関するリスク

### 脅威 T2-1: ビルドなし再起動による変更未反映

`definitions.ts` を修正しても、TypeScript をトランスパイルせずに MCPサーバーを再起動した場合、旧バージョンの `dist/phases/definitions.js` が引き続き使用される。
この場合、修正が反映されていないにもかかわらず「再起動済み」と認識され、根本原因の修正が完了していないまま次フェーズに進んでしまう。
サブエージェントは依然として誤ったNG/OK例のプロンプトを受け取り続け、バリデーションエラーが再発する。

対策として、修正後は必ず `cd workflow-plugin/mcp-server && npm run build` を実行し、`dist/phases/definitions.js` の更新日時を確認してから MCPサーバーを再起動すること。

### 脅威 T2-2: 再起動後のフェーズ状態の不整合

MCPサーバーを再起動すると、タスクの内部状態（フェーズ情報、HMAC整合性）が正常に維持されているかを確認する必要がある。
再起動中にタスクの状態ファイルが不整合になった場合、フェーズ遷移ができなくなり、ワークフローが停止するリスクがある。
特に parallel_analysis フェーズで複数のサブフェーズが並列実行中に再起動を行うと、サブフェーズの完了状態が失われる可能性がある。

対策として、再起動後に `workflow_status` コマンドでタスクの現在フェーズとサブフェーズの完了状態を確認すること。

## 脅威カテゴリ 3: 全フェーズへの波及リスク

### 脅威 T3-1: 修正内容が新たな誤解を招くリスク

修正後のNG/OK例が正確でも、サブエージェントが受け取る文脈によっては依然として誤解が生じる可能性がある。
例えば「コードフェンス内に正規表現を書くことが推奨」と説明しても、どの種類の角括弧（正規表現の文字クラス vs プレースホルダー）が対象なのかが不明確な場合、混乱が生じる。
行 1153〜1154 の既存説明と修正後の行 1156〜1160 が矛盾なく一貫した内容になっているかを検証すること。

対策として、修正後の `qualitySection` 全体（行 1150〜1160 相当）を通読し、脈絡が一貫していることを確認してから実装を承認すること。

### 脅威 T3-2: 既存テストが修正後の文字列を検証している場合の破損

`definitions.ts` の `buildPrompt()` 出力文字列を検証するテストが存在する場合、修正によってそのテストが失敗する可能性がある。
`workflow-plugin/mcp-server/src/` 配下のテストファイルを確認し、`buildPrompt` の出力内容をスナップショットやパターンで検証するテストが存在するかを事前に調査すること。
既存テストが壊れた場合、CI が失敗し、コミット・プッシュフェーズで問題が発覚する。

対策として、実装フェーズでテストファイルを確認し、必要に応じてテストの期待値を更新すること。

## 脅威カテゴリ 4: 変更範囲外への副作用

### 脅威 T4-1: `subagentTemplate` との二重管理問題

要件定義書によると、`buildPrompt()` によって上書きされるため `subagentTemplate` のハードコード部分は修正主体ではないとされている。
しかし、将来的に `subagentTemplate` と `buildPrompt()` の間で同一セクションの内容が乖離した場合、どちらが優先されるかが不明確になるリスクがある。
この乖離はコードメンテナンスの難易度を高め、将来の修正作業で同様のバグが再発する原因となりうる。

対策として、`planning` フェーズでこの二重管理の解消方針についても言及し、将来の技術的負債を記録すること。

### 脅威 T4-2: CLAUDE.md と definitions.ts のドキュメント乖離

CLAUDE.md にも角括弧プレースホルダー禁止に関する説明（コードフェンス内は安全という正しい記載）が含まれており、今回の修正により `definitions.ts` の内容が CLAUDE.md と一致する方向に修正される。
ただし、CLAUDE.md 側に今後誤った情報が混入した場合、再び二つのドキュメントが矛盾するリスクがある。
この矛盾が発生しても、`definitions.ts` の内容が実際のサブエージェントへのプロンプトを生成するため、そちらが優先されることを認識しておくこと。

## 対策の優先順位と実施手順

最優先で対処すべき脅威は T2-1（ビルドなし再起動）である。
修正フロー全体を以下の手順で実施すること：
手順1として `definitions.ts` の行 1156〜1160 を修正し、隣接行（1153〜1154）が変更されていないことを `git diff` で確認する。
手順2として `cd workflow-plugin/mcp-server && npm run build` を実行し、コンパイルエラーがないことを確認する。
手順3として `dist/phases/definitions.js` の更新日時が修正後の日時になっていることをファイル属性で確認する。
手順4として Claude Desktop の MCPサーバー再起動機能を使用してサーバーを再起動し、`workflow_status` で現在フェーズを確認する。
手順5として planning フェーズで既存テストの有無を調査し、必要に応じてテスト更新の計画を立てる。

## まとめ

本変更は `buildPrompt()` 内の6行の文字列定数修正という極めて限定的なスコープであり、コアロジックへの副作用はない。
最大のリスクはビルド・再起動手順の省略による変更未反映であり、`npm run build` と MCPサーバーの再起動という2ステップを必ず実施することが最重要の対策となる。
リグレッション防止のためには、修正前に `artifact-validator.ts` の `extractNonCodeLines` 実装を確認してコードフェンス除外ロジックを明文化し、修正後に `git diff` で変更範囲が意図した6行のみであることを検証すること。
既存テストが `buildPrompt()` の出力文字列をスナップショット等で検証している場合は、テストの期待値も合わせて更新しなければ CI が失敗するリスクがあるため、実装フェーズでテストファイルの調査を行うこと。
`buildPrompt()` と `subagentTemplate` の二重管理問題は今回のスコープ外だが、将来の技術的負債として計画フェーズでも記録しておくことが望ましい。
修正が完了した後は、全フェーズのサブエージェントが正しい理解（コードフェンス内は安全・コードフェンス外の散文は禁止）でプロンプトを受け取るようになる。
この結果として、parallel_verification での4サブエージェント連続バリデーションエラーが解消され、今回の根本原因に起因する再発を防止できる見込みである。
