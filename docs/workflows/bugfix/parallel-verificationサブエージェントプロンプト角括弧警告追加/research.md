## サマリー

- 目的: parallel_verificationの4サブフェーズ（manual_test, security_scan, performance_test, e2e_test）のサブエージェントプロンプトに角括弧プレースホルダー禁止の明示的な警告を追加するための調査
- 主要な決定事項: プロンプトは`buildPrompt`関数で動的生成されており、角括弧警告は`qualitySection`に含まれているが、NG/OKの例示が誤解を招く記述になっている。サブフェーズ固有の`subagentTemplate`文字列には角括弧警告が含まれていない
- 次フェーズで必要な情報: 修正対象は`definitions.ts`の2か所（`buildPrompt`関数内の`qualitySection`行1155〜1160の誤解招くNG/OK例示と、4つの`subagentTemplate`文字列への追記箇所）

## 調査結果

### ファイルの場所と構成

調査対象ファイルはプロジェクト配下の単一ファイルである。

- ファイルパス: `workflow-plugin/mcp-server/src/phases/definitions.ts`
- ファイルの行数は1582行を超える大規模なファイルである
- parallel_verificationサブフェーズの定義は行896〜943に集中している

### parallel_verificationサブフェーズの定義位置

4つのサブフェーズはそれぞれ以下の行範囲に定義されている。

- `manual_test`: 行896〜907（`subagentTemplate`は行906の長い文字列）
- `security_scan`: 行908〜919（`subagentTemplate`は行918の長い文字列）
- `performance_test`: 行920〜931（`subagentTemplate`は行930の長い文字列）
- `e2e_test`: 行932〜943（`subagentTemplate`は行942の長い文字列）

各サブフェーズは`phaseName`, `description`, `requiredSections`, `outputFile`, `allowedBashCategories`, `editableFileTypes`, `minLines`, `subagentType`, `model`, `subagentTemplate`フィールドを持つ。

### プロンプト生成の仕組み

`resolvePhaseGuide`関数（行1529〜1578）が`buildPrompt`を呼び出してサブエージェントプロンプトを最終決定する。

```
// 行1544〜1558の処理（サブフェーズ処理部分）
if (resolved.subPhases) {
  for (const [subPhaseName, subPhase] of Object.entries(resolved.subPhases)) {
    try {
      subPhase.subagentTemplate = buildPrompt(subPhase, subPhaseName, userIntent, docsDir);
    } catch (e) {
      // フォールバック: resolvePlaceholders使用
    }
  }
}
```

このため、各サブフェーズの`subagentTemplate`文字列は実際には`buildPrompt`の戻り値で上書きされる。サブエージェントが受け取るプロンプトは`buildPrompt`の生成物であり、ハードコードされた`subagentTemplate`文字列ではない。

### 現在のbuildPromptにおける角括弧警告の記述

`buildPrompt`関数は行1092〜1256に定義されており、セクション5として「成果物品質要件」を生成する。

角括弧警告は行1150〜1160に記述されている。

```typescript
qualitySection += `\n### 角括弧プレースホルダー禁止\n`;
qualitySection += `[変数名]、[パス]等の角括弧プレースホルダーは使用禁止です。\n`;
qualitySection += `許可される角括弧: ${rules.bracketPlaceholderInfo.allowedKeywords.join('、')}\n`;

qualitySection += `\n配列アクセスの記述:\n`;
```

### 誤解を招くNG/OKの例示（BUG-4の直接的原因）

上記のNG/OKの例示には重大な誤りが含まれている。

行1156の記述: 「NG: 正規表現で「英小文字1文字以上」を表すパターンをそのままコードブロックに書く」

この例示は**逆**である。コードブロック（コードフェンス）内に角括弧を書くことは安全であり、バリデーターはコードフェンス内の行を検出対象から除外している。

行1159の記述: 「NG: 配列のインデックスアクセス記法をコードブロック内に直接記述する」

同様に、コードフェンス内への記述は安全であるため、この記述も誤っている。

この誤解招く記述により、サブエージェントが「コードフェンス内も危険」と誤解し、正規表現パターンをコードフェンス外の散文（箇条書き等）に直接記述してしまった可能性が高い。これがBUG-4の直接的な原因と推定される。

### バリデーターの実際の動作

`artifact-validator.ts`の`extractNonCodeLines`関数（行155〜185）はコードフェンスを検出して内部行をスキップする。

バリデーター行368の角括弧検出パターン:
```
/\[(?!関連|参考|注|例|出典)[^\]]{1,50}\]/g
```

この正規表現は`extractNonCodeLines`でフィルタリングされた後のコードフェンス外のテキストにのみ適用される。コードフェンス内の行はスキップされるため、正規表現や配列アクセス記法を含む技術的な記述はコードフェンス内であれば安全である。

## 既存実装の分析

### サブフェーズのsubagentTemplateに角括弧警告が存在しない理由

4サブフェーズのハードコードされた`subagentTemplate`文字列にはそれぞれ固有のガイダンスが記述されているが、角括弧警告は含まれていない。

`manual_test`の`subagentTemplate`に含まれる注意事項:
- 行数カウント仕様の説明
- 禁止語転記防止の説明
- 重複行回避の注意事項
- サマリーセクションの行数ガイダンス
- テストシナリオセクションの行数ガイダンス
- テスト結果セクションの行数ガイダンス

`security_scan`の`subagentTemplate`に含まれる注意事項:
- 行数カウント仕様と転記防止
- 重複行回避の注意事項
- サマリーセクションの行数ガイダンス
- 脆弱性スキャン結果セクションの行数ガイダンス
- 検出された問題セクションの行数ガイダンス

`performance_test`の`subagentTemplate`に含まれる注意事項:
- サマリーセクションの行数ガイダンス
- パフォーマンス計測結果セクションの行数ガイダンス
- ボトルネック分析セクションの行数ガイダンス

`e2e_test`の`subagentTemplate`に含まれる注意事項:
- 行数カウント仕様の説明
- 禁止語転記防止の説明
- サマリーセクションの行数ガイダンス
- E2Eテストシナリオセクションの行数ガイダンス
- テスト実行結果セクションの行数ガイダンス
- 重複行回避の注意事項

いずれのサブフェーズにも、正規表現の文字クラス表記やコードフェンスの安全な活用について言及がない。

### ハードコードされたsubagentTemplateとbuildPromptの関係

`resolvePhaseGuide`が`buildPrompt`を呼ぶと、`buildPrompt`は`guide`オブジェクトのプロパティ（`phaseName`, `description`, `requiredSections`, `inputFileMetadata`, `outputFile`, `allowedBashCategories`等）を参照して9セクションのプロンプトを生成する。`guide.subagentTemplate`フィールドは`buildPrompt`内では参照されない。したがって、ハードコードされた`subagentTemplate`の内容はそのまま引き継がれるのではなく、`buildPrompt`が`guide`のメタデータから新しいプロンプトを生成してsubagentTemplateフィールドに上書きする。

つまり、4サブフェーズのハードコードされた`subagentTemplate`文字列の内容は最終的なサブエージェントプロンプトには直接反映されない。最終プロンプトは`buildPrompt`が生成する。

### buildPromptに追加されるサブフェーズ固有コンテンツ

`buildPrompt`にはサブフェーズ固有のコンテンツを追加するフック箇所が現在存在しない。9つのセクション（フェーズ情報、入力ファイル、出力ファイル、必須セクション、品質要件、Bashコマンド制限、ファイル編集制限、チェックリスト、重要事項）はすべてフェーズ共通のロジックで生成される。

サブフェーズ固有の注意事項（行数ガイダンス等）は`subagentTemplate`のハードコード文字列に記述されているが、`buildPrompt`がこれを上書きするため、これらの内容はサブエージェントに渡らない状態になっている可能性がある。

この点については、`buildPrompt`がhardcoded`subagentTemplate`の内容を保持する仕組みがないため、サブフェーズ固有ガイダンスを`PhaseGuide`の`checklist`フィールドに移すか、`buildPrompt`のセクション8「フェーズ固有チェックリスト」を通じて追加するかの検討が必要である。

### 修正すべき箇所の特定

調査の結果、以下の2か所に修正が必要と判断する。

修正箇所1: `buildPrompt`の品質要件セクション内のNG/OK例示（行1155〜1160）

現在の記述はコードフェンス内への記述を「NG」としており、これが誤解の根本原因である。正しくは「コードフェンス外の散文・箇条書きへの記述がNG」であり「コードフェンス内は安全」という対比を明確に示すべきである。

修正箇所2: 4サブフェーズの`subagentTemplate`文字列（行906, 918, 930, 942）

`buildPrompt`によって上書きされるため、ここへの追記はサブエージェントに届かない可能性がある。しかしbuildPromptへのフォールバック時（`docsDir`が空の場合等）に`subagentTemplate`が使用されるケースを考慮し、フォールバック時の安全網として角括弧警告を追記する価値がある。

本質的な修正は修正箇所1（`buildPrompt`のNG/OK例示の修正）であり、修正箇所2は追加的な安全網として位置づける。

### 正しいNG/OK例示の形式

BUG-4の再発防止のため、以下の形式に修正すべきである。

正規表現の文字クラスを含む記述について:
- NG（バリデーションエラーになる）: コードフェンスの外の箇条書きや散文テキストに正規表現パターンを直接記述すること
- OK（安全）: コードフェンス内に正規表現パターンを配置すること（バリデーターはコードフェンス内の行を検出対象から除外するため）
- OK（代替）: コードフェンス外では「英小文字の1文字以上の繰り返しを表す正規表現」のように散文形式で説明すること

配列アクセス記法を含む記述について:
- NG（バリデーションエラーになる）: コードフェンスの外の箇条書きや散文テキストに配列のインデックスアクセス記法を直接記述すること
- OK（安全）: コードフェンス内に配列アクセス記法を配置すること（バリデーターはコードフェンス内の行を検出対象から除外するため）
- OK（代替）: コードフェンス外では「先頭要素を取得する」「インデックス番号によるアクセス」のように散文形式で説明すること

### parallel_verificationサブフェーズが繰り返し違反した背景

BUG-4で4サブフェーズが全て「角括弧プレースホルダー禁止」に違反した原因は複合的である。

第一の要因: `buildPrompt`のNG/OK例示が逆であった（コードフェンス内が危険と誤解させる記述）。
第二の要因: `subagentTemplate`のハードコード文字列に角括弧に関する正しいガイダンスが存在しなかった。
第三の要因: セキュリティスキャンやE2Eテストの成果物では正規表現パターン（文字クラス表記を含む）を記述する機会が多く、他のフェーズより誤用リスクが高い。

これらの要因が重なり、4サブエージェント全員がコードフェンス外に正規表現を記述してしまった。
