## サマリー

本手動テストは、`workflow-plugin/mcp-server/src/phases/definitions.ts` の `buildPrompt()` 関数内 qualitySection に対して加えられた変更を検証することを目的としている。
変更の背景として、parallel_verification フェーズのサブエージェントが成果物のコードフェンス外に正規表現の文字クラス表記や配列インデックスアクセス記法を直接記述し、バリデーター（artifact-validator.ts）が角括弧プレースホルダー禁止ルールの違反として検出するという障害が繰り返し発生していた。

主要な決定事項として、definitions.ts の行 1155〜1162 において、正規表現パターンおよび配列アクセスの記述ガイダンスを「NG（コードフェンス外への直接記述）」と「OK（推奨: コードフェンス内）」と「OK（代替: 散文形式）」の3種類に明示分類する形式に改訂した。
変更前は6行の記述が変更後は8行に拡充され、各方向性が個別の行として明確化された。

テスト環境は Windows 11 上の MSYS_NT-10.0-26200 であり、ファイルの読み取りは Read ツール、コード検索は Grep ツールを使用して実施した。
次フェーズ（security_scan, performance_test, e2e_test）で必要な情報として、今回の変更がコードの動作（ランタイム挙動）には影響せず、プロンプトテキスト生成にのみ影響する点を共有する。

## テストシナリオ

### シナリオ1: NG記述の方向性確認

検証対象の行は `definitions.ts` の 1156 行目（正規表現）および 1160 行目（配列アクセス）である。
これら2行が「コードフェンス外の散文や箇条書きへの直接記述」をNGとして示しているかどうかを確認した。

確認する観点として、行の先頭が「- NG:」というラベルから始まり、対象の違反行為が「コードフェンス外」の文脈において記述されていることを求めた。
正規表現シナリオでは行 1156 の内容が「コードフェンス外の散文や箇条書きに正規表現の文字クラス表記を直接記述すること」であり、NG の意味が正確に伝わる構成になっているかを評価した。
配列アクセスシナリオでは行 1160 の内容が「コードフェンス外の散文や箇条書きに配列のインデックスアクセス記法を直接記述すること」であり、同様の観点で評価した。

### シナリオ2: OK（推奨）行の確認

行 1157（正規表現）および行 1161（配列アクセス）が、コードフェンス内を安全な記述手段として推奨しているかを確認した。
推奨の根拠として、artifact-validator.ts の `extractNonCodeLines()` 関数がコードフェンス内の行を検出対象から除外するという技術的事実が正確に記述されているかを評価の軸とした。

行 1157 の内容は「コードフェンス内に正規表現パターンを記述すること（バリデーターはコードフェンス内の行を検出対象から除外するため安全）」であった。
行 1161 の内容は「コードフェンス内に配列アクセス記法を記述すること（バリデーターはコードフェンス内の行を検出対象から除外するため安全）」であった。
両行ともカッコ内に技術的根拠を含めており、サブエージェントが理解しやすい形式になっていると判定した。

### シナリオ3: OK（代替）行の確認

行 1158（正規表現）および行 1162（配列アクセス）が、コードフェンス外での記述における散文形式の代替表現を示しているかを確認した。
代替表現の具体例が含まれているかどうかを判断基準とした。

行 1158 の内容は「コードフェンス外では「英小文字の1文字以上の繰り返しを表す正規表現」のように散文形式で説明すること」であった。
行 1162 の内容は「コードフェンス外では「先頭要素を取得する」「インデックス番号によるアクセス」のように散文形式で説明すること」であった。
両行ともに具体例がカッコ内に埋め込まれており、サブエージェントが模倣しやすい書き方になっている。

### シナリオ4: バリデーターとの整合性確認

artifact-validator.ts の `extractNonCodeLines()` 関数の実際の動作と、definitions.ts に記述されたガイダンスの内容が一致するかを確認した。
確認対象は `extractNonCodeLines` の実装（行 155〜）および角括弧プレースホルダーの検出ロジック（行 367〜373）である。

## テスト結果

### シナリオ1の検証結果（NG記述の方向性確認）

行 1156 は「- NG: コードフェンス外の散文や箇条書きに正規表現の文字クラス表記を直接記述すること」と記述されており、NGの対象が「コードフェンス外」かつ「散文や箇条書きへの直接記述」であることが明確に示されている。
行 1160 は「- NG: コードフェンス外の散文や箇条書きに配列のインデックスアクセス記法を直接記述すること」と記述されており、同様にNGの対象が適切に特定されている。
両行ともに「直接記述すること」という行為を対象として禁止しており、禁止の意図が正確に伝わる文章構成になっている。
この形式であれば、サブエージェントが「なぜ禁止か」ではなく「何が禁止か」を即座に把握できる。
シナリオ1の検証結果: 合格（正規表現・配列アクセスの両ケースで正しく記述されている）。

### シナリオ2の検証結果（OK推奨行の確認）

行 1157 および行 1161 の両行は、「コードフェンス内」という場所の指定と「バリデーターはコードフェンス内の行を検出対象から除外するため安全」という技術的根拠を明示している。
実際に artifact-validator.ts を確認したところ、行 155 の `extractNonCodeLines()` 関数がコードフェンスを追跡してその内部行をスキップする実装になっており、ガイダンスの記述と実装が一致している。
つまりサブエージェントがコードフェンス内に正規表現を記述しても、バリデーターは角括弧検出の対象から除外するため、バリデーション違反は発生しない。
シナリオ2の検証結果: 合格（推奨理由の技術的根拠がバリデーター実装と整合している）。

### シナリオ3の検証結果（OK代替行の確認）

行 1158 の「英小文字の1文字以上の繰り返しを表す正規表現」という散文例は、角括弧を一切使わずに正規表現パターンの意味を説明できることを示している。
行 1162 の「先頭要素を取得する」「インデックス番号によるアクセス」という散文例は、配列インデックスアクセス記法を角括弧なしで表現できることを示している。
どちらの例も既存の CLAUDE.md 推奨パターン（「先頭要素」「2番目の要素」という表記）と整合しており、成果物品質要件との矛盾はない。
シナリオ3の検証結果: 合格（代替表現の具体例が実用的かつ角括弧を含まない形式で提供されている）。

### シナリオ4の検証結果（バリデーターとの整合性確認）

artifact-validator.ts の 368 行目に、角括弧プレースホルダーの検出正規表現が定義されている。この正規表現は `extractNonCodeLines()` の返り値（コードフェンス外の行のみ）に対して適用される。
`extractNonCodeLines()` 関数の実装では、コードフェンス開始記号（3連バックティック）を検出してフラグを立て、フェンス内部の行をスキップする設計になっている。
definitions.ts の行 1153〜1154 には「コードフェンス内の行は extractNonCodeLines により検出から除外される」という説明が既に存在しており、今回追加された行 1155〜1162 のガイダンスは既存の説明を具体的な記述例として補強する形になっている。
また、definitions.ts 行 1298〜1302 の `bracketPlaceholderInfo` オブジェクトには、許可されるキーワード（「関連」「参考」「注」「例」「出典」）が定義されており、ガイダンス内でもこれらの例外パターンについて案内されている。
シナリオ4の検証結果: 合格（ガイダンス記述とバリデーター実装が技術的に整合している）。

### 総合判定

全4シナリオで合格を確認した。追加された8行（行 1155〜1162）は、subagent が parallel_verification フェーズで成果物を生成する際に、コードフェンス外への角括弧含有表記を回避するための具体的なガイダンスを提供しており、バリデーター実装との整合性も確認できた。この変更により、角括弧プレースホルダー違反による繰り返しのバリデーション失敗を未然に防ぐ効果が期待される。
