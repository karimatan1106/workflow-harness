## サマリー

このドキュメントは `definitions.ts` の `buildPrompt()` 関数内 NG/OK 例の誤記修正タスクに関する UI 設計書である。
本タスクにグラフィカルな UI は存在せず、「UI」はサブエージェントが受け取るプロンプトテキストの構造（テキストインターフェース設計）を指す。

- 目的: `buildPrompt()` が生成する `qualitySection` 内の角括弧禁止セクションにおいて、NG/OK 例示の記述方向を正しい向きに修正し、全フェーズのサブエージェントが正確な角括弧回避手順を受け取れるようにする。
- 主要な決定事項: 正規表現パターンおよび配列アクセス記述の NG/OK 例を各 4 行構成（ヘッダー行・NG 行・OK 推奨行・OK 代替行）に拡張し、コードフェンス内が安全であることを明示する。
- 次フェーズで必要な情報: 変更対象ファイルパス（`workflow-plugin/mcp-server/src/phases/definitions.ts`）、変更行番号（行 1155〜1160 を 8 行に置き換え）、ビルドコマンド（`cd workflow-plugin/mcp-server && npm run build`）。

## CLIインターフェース設計

このセクションでは `buildPrompt()` が生成するプロンプトの「角括弧禁止セクション」のテキスト構造を、修正前後の比較形式で示す。

### 修正前（バグあり）の出力構造

修正前の `qualitySection` が含む角括弧関連セクションは以下の 6 行で構成されており、NG と OK の方向が実際のバリデーター動作と逆になっている。

```
角括弧プレースホルダーは禁止（波括弧を使用）。検出の対象はコードフェンス外の
Markdown本文に記述された角括弧であり、コードフェンス内の行は検出から除外される。

正規表現パターンの記述:
- NG: 正規表現で「英小文字1文字以上」を表すパターンをそのままコードブロックに書く
- OK: 「英小文字の1文字以上の繰り返しを表す正規表現」のように散文で説明する

配列アクセスの記述:
- NG: 配列のインデックスアクセス記法をコードブロック内に直接記述する
- OK: 「配列の先頭要素を取得する」「インデックス番号によるアクセス」のように散文形式で説明する
```

この出力を受け取ったサブエージェントは「コードブロックへの記述が NG」と解釈し、正規表現パターンをコードフェンス外の箇条書きに直接記述してしまう。
その結果、`artifact-validator.ts` の角括弧検出が発動してバリデーションエラーが生じる。
この誤解が parallel_verification フェーズの 4 サブエージェント連続失敗の根本原因となった。

### 修正後（正しい）の出力構造

修正後の `qualitySection` が含む角括弧関連セクションは以下の 8 行で構成され、NG/OK の方向がバリデーターの実際の動作と一致する。

```
角括弧プレースホルダーは禁止（波括弧を使用）。検出の対象はコードフェンス外の
Markdown本文に記述された角括弧であり、コードフェンス内の行は検出から除外される。

正規表現パターンの記述:
- NG: コードフェンス外の散文や箇条書きに正規表現の文字クラス表記を直接記述すること
- OK（推奨）: コードフェンス内に正規表現パターンを記述すること（バリデーターはコードフェンス内の行を検出対象から除外するため安全）
- OK（代替）: コードフェンス外では「英小文字の1文字以上の繰り返しを表す正規表現」のように散文形式で説明すること

配列アクセスの記述:
- NG: コードフェンス外の散文や箇条書きに配列のインデックスアクセス記法を直接記述すること
- OK（推奨）: コードフェンス内に配列アクセス記法を記述すること（バリデーターはコードフェンス内の行を検出対象から除外するため安全）
- OK（代替）: コードフェンス外では「先頭要素を取得する」「インデックス番号によるアクセス」のように散文形式で説明すること
```

修正後はコードフェンス内への記述が「OK（推奨）」として明示され、サブエージェントは技術的記法を安全に使用できる場所を正確に理解できる。
「コードフェンス外の散文・箇条書きへの直接記述が NG」という表現により、禁止対象が明確になる。
各セクションが 4 行（ヘッダー・NG・OK 推奨・OK 代替）の統一構成を持つため、可読性と一貫性が向上する。

### 変更の差分サマリー

修正前の 6 行構成（正規表現セクション 3 行 + 配列セクション 3 行）が修正後の 8 行構成（正規表現セクション 4 行 + 配列セクション 4 行）に拡張される。
各セクションに「OK（推奨）」行（コードフェンス内への記述を奨励）と「OK（代替）」行（コードフェンス外での散文表現）の 2 択が提示される。
この変更により、サブエージェントは状況に応じて最適な記述方法を選択できるようになる。

## エラーメッセージ設計

このセクションでは `artifact-validator.ts` がコードフェンス外の角括弧を検出した際に返すエラーメッセージの形式と、修正前後でエラー発生状況がどのように変化するかを説明する。

### 修正前に発生した典型的なエラーパターン

修正前のプロンプトを受け取ったサブエージェントが正規表現パターンをコードフェンス外の箇条書きに記述すると、以下のようなエラーが発生していた。
エラーの種別は「角括弧プレースホルダー検出」であり、バリデーターが Markdown 本文（コードフェンス外）から角括弧を含む行を検出したことを示す。
エラーメッセージには検出された行の内容と行番号が含まれ、成果物全体がバリデーション失敗となる。
このエラーが並列フェーズの 4 サブフェーズで連続して発生したため、リトライループが繰り返された。

### 修正後にエラーが発生しなくなる理由

修正後のプロンプトでは「コードフェンス内への記述が OK（推奨）」と明示されるため、サブエージェントは正規表現パターンや配列アクセス記法をコードフェンス内に配置する。
`artifact-validator.ts` の `extractNonCodeLines` 関数（行 155〜185）はコードフェンス内の全行を検出対象から除外するため、技術的記法がコードフェンス内に記述されてもバリデーションエラーは発生しない。
コードフェンス外での記述が必要な場合も「OK（代替）」として提示された散文説明方法（「英小文字の1文字以上の繰り返しを表す正規表現」のような表現）を使用するため、角括弧が成果物に現れなくなる。
この設計変更により、バリデーション失敗の根本原因（誤った NG/OK 例示によるサブエージェントの誤解）が解消される。

### バリデーターとプロンプトの整合性確認

バリデーターが検出するのはコードフェンス外の散文・箇条書きに記述された角括弧のみである。
修正後のプロンプトが示す「NG」の定義（コードフェンス外への直接記述）はこのバリデーター動作と完全に一致する。
「OK（推奨）」として示すコードフェンス内への記述は `extractNonCodeLines` により除外されるため、バリデーションエラーが発生しない。
プロンプトとバリデーターの整合性が回復することで、サブエージェントが正しいガイダンスに従った場合にバリデーションが必ず通過するという保証が成立する。

## APIレスポンス設計

このセクションでは `buildPrompt()` 関数の入出力仕様を TypeScript 型レベルで設計し、修正箇所の文脈を明確にする。

### buildPrompt 関数のシグネチャ

`buildPrompt()` 関数は以下の入力パラメーターを受け取り、サブエージェントへの完全なプロンプト文字列を返す。

```typescript
function buildPrompt(
  phase: string,         // フェーズ識別子（例: "manual_test", "security_scan"）
  phaseName: string,     // フェーズ表示名（例: "手動テスト", "セキュリティスキャン"）
  userIntent: string,    // ユーザーの意図・タスク説明
  docsDir: string        // 成果物出力先ディレクトリのパス
): string                // 完全なサブエージェントプロンプト文字列を返す
```

戻り値の文字列は複数のセクションを含む長大なプロンプトであり、その中に `qualitySection` が組み込まれる。
`qualitySection` は Bash コマンド制限セクションや成果物品質要件セクションの一部として最終プロンプトに埋め込まれる。
修正によって変化するのは `qualitySection` の内部構造のみであり、関数シグネチャ・戻り値の型・呼び出し元の動作には影響しない。

### qualitySection の内部構造

`qualitySection` は `buildPrompt()` 内でローカル変数として宣言され、複数の `+=` 代入によって構築される文字列である。

修正対象の 6 行（行 1155〜1160）は `qualitySection` の中の「角括弧プレースホルダー禁止」サブセクションの一部を構成している。
このサブセクションは行 1150 の「角括弧プレースホルダーは禁止」宣言から始まり、行 1160 の配列アクセス OK 行で終わる。
修正後は行 1161 以降（重複行禁止セクション）の前に、正しい NG/OK 例示を含む 8 行が配置される。

### resolvePhaseGuide との連携

`resolvePhaseGuide()` 関数（行 1529〜1578）は `buildPrompt()` を呼び出してサブフェーズのプロンプトを最終決定する。
サブフェーズ（manual_test, security_scan, performance_test, e2e_test）についても `buildPrompt(subPhase, subPhaseName, userIntent, docsDir)` が呼ばれ、その戻り値が `subagentTemplate` フィールドを上書きする。
したがって `buildPrompt()` の修正は全フェーズ・全サブフェーズのサブエージェントプロンプトに自動的に波及する。
この設計により、1 箇所の修正で全サブエージェントの動作改善が実現する点が本タスクの重要な特性である。

### ビルドコマンドとデプロイ手順

ビルドコマンドは `cd workflow-plugin/mcp-server && npm run build` であり、TypeScript ソースを `dist/` ディレクトリの JavaScript ファイルにトランスパイルする。
ビルドの出力ファイルは `workflow-plugin/mcp-server/dist/phases/definitions.js` であり、このファイルの更新日時を確認することでビルド成功を検証できる。
ビルド完了後は Claude Desktop のサーバー再起動ボタンまたはプロセス終了で MCPサーバーを再起動する必要がある。
Node.js のモジュールキャッシュにより再起動しない限り修正前のコードがメモリ上で動作し続けるため、この手順は省略不可である。
再起動後に `workflow_status` を実行して現在のフェーズを確認し、同フェーズから作業を再開する。

### ビルド検証の基準

`npm run build` がエラーなく完了し、exit code 0 で終了すること。
`dist/phases/definitions.js` の更新日時がビルド実行時刻以降であること。
TypeScript コンパイルエラーが標準エラー出力に出力されていないこと。
特に文字列テンプレートリテラル内のバッククォートのエスケープが正しいかどうかを確認すること（日本語の括弧や特殊文字を含む文字列でエスケープ漏れが発生するケースがある）。
