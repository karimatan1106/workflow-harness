## サマリー

このドキュメントは、`workflow-plugin/mcp-server/src/phases/definitions.ts` の `buildPrompt()` 関数内に存在する NG/OK 例の誤記バグを修正するための仕様書である。
現在のプロンプトでは「コードフェンス内への記述が NG」と誤って記述されているため、サブエージェントがコードフェンス外の散文に技術的記法を書いてしまうという逆効果が生じている。
修正により「コードフェンス外の散文・箇条書きへの直接記述が NG」「コードフェンス内への記述が OK（推奨）」という正しい対比に修正し、全フェーズのサブエージェントプロンプトの品質向上を図る。
変更対象は `definitions.ts` の行 1155〜1160 の6行のみであり、行数は6行から8行に増加する（各セクションに OK 代替行を1行追加）。
次フェーズでは `definitions.ts` の行 1155〜1160 を修正後の8行に置き換え、ビルドおよび MCPサーバー再起動を実施する。

## 概要

本タスクが対処する問題は、`buildPrompt()` 関数が生成する品質要件セクション（`qualitySection`）内の NG/OK 例示の記述方向の誤りである。
現在の記述では「コードブロックに書く」行為が NG、「散文形式で説明する」行為が OK とされているが、これは実際のバリデーターの動作と逆である。
`artifact-validator.ts` の `extractNonCodeLines` 関数はコードフェンス内の全行を角括弧検出の対象から除外するため、コードフェンス内への記述は安全である。
一方、コードフェンス外の散文・箇条書きに正規表現の文字クラス表記や配列アクセス記法を直接記述した場合にのみバリデーションエラーが発生する。
この矛盾した指示を受け取ったサブエージェントが正規表現パターンをコードフェンス外の箇条書きに記述したことが、parallel_verification フェーズの4サブエージェントが連続してバリデーションエラーを起こした根本原因である。
修正後のプロンプトでは「コードフェンス外が NG」「コードフェンス内が OK（推奨）」「コードフェンス外での散文説明が OK（代替）」という3段階の対比を提示し、サブエージェントの誤解を根本から解消する。

実装フェーズでは以下の手順でコード修正とビルドを実施する。

手順1として、`workflow-plugin/mcp-server/src/phases/definitions.ts` を開き、行 1155〜1160 の6行を特定する。対象行は `qualitySection += ...` 形式の連続した文字列代入であり、正規表現パターンと配列アクセスの NG/OK 例示を含む。

手順2として、特定した6行を修正後の8行（正規表現パターンセクション4行 + 配列アクセスセクション4行）に置き換える。各セクションはヘッダー行1行、NG 行1行、OK（推奨）行1行、OK（代替）行1行の構成とする。

手順3として、`cd workflow-plugin/mcp-server && npm run build` を実行して TypeScript をトランスパイルし、`dist/phases/definitions.js` が更新されたことを確認する。ビルドエラーが発生した場合は文字列中のエスケープ文字を確認して修正する。

手順5として、再起動後に `workflow_status` を実行して現在のフェーズを確認し、同フェーズから作業を再開する。

## 問題の背景

### バグが発生した状況

parallel_verification フェーズの4つのサブエージェント（manual_test, security_scan, performance_test, e2e_test）が連続してバリデーションエラーを発生させた。
エラーの直接原因は、各サブエージェントの成果物にコードフェンス外の散文・箇条書きに角括弧を含む技術的記法が記述されていたことである。
調査の結果、サブエージェントへの指示文（`buildPrompt()` が生成するプロンプト）において、コードフェンス内への記述を「NG」と記述するという重大な誤りが含まれていることが判明した。
この誤りによりサブエージェントが「コードフェンス内も危険」と誤解し、正規表現パターンをコードフェンス外の箇条書きに直接記述してしまった。

### バリデーターの実際の動作

`artifact-validator.ts` の `extractNonCodeLines` 関数（行 155〜185）はコードフェンスで囲まれた行を全て検出対象から除外する。
角括弧検出に用いる正規表現パターンは、この関数でフィルタリングされた後のコードフェンス外のテキストにのみ適用される。
したがって、コードフェンス内に正規表現の文字クラス表記や配列アクセス記法を記述してもバリデーションエラーは発生しない。
コードフェンス外の散文テキストや箇条書きにこれらの記法を直接記述した場合にのみエラーが発生する。
この事実と現在のプロンプト記述の間に根本的な矛盾が存在する。

### buildPrompt と subagentTemplate の関係

`resolvePhaseGuide` 関数（行 1529〜1578）は `buildPrompt` を呼び出してサブエージェントプロンプトを最終決定する。
サブフェーズ（manual_test, security_scan, performance_test, e2e_test）についても行 1544〜1558 の処理で `buildPrompt(subPhase, subPhaseName, userIntent, docsDir)` が呼ばれ、その戻り値が `subagentTemplate` フィールドを上書きする。
したがって、4サブフェーズのハードコードされた `subagentTemplate` 文字列の内容は最終的なサブエージェントプロンプトには反映されず、`buildPrompt` の生成物が使用される。
修正を `buildPrompt` 内の品質要件セクション（`qualitySection`）に集中させれば、全フェーズおよび全サブフェーズのサブエージェントプロンプトに効果が波及する。

## 修正前の現状コード

### 現在の問題箇所（definitions.ts 行 1155〜1160）

現在の6行は以下の内容であり、NG/OK の方向が逆になっている。コードフェンス内への記述を誤って「NG」と記述している点が問題の核心である。

```
qualitySection += `\n正規表現パターンの記述:\n`;
qualitySection += `- NG: 正規表現で「英小文字1文字以上」を表すパターンをそのままコードブロックに書く\n`;
qualitySection += `- OK: 「英小文字の1文字以上の繰り返しを表す正規表現」のように散文で説明する\n`;
qualitySection += `\n配列アクセスの記述:\n`;
qualitySection += `- NG: 配列のインデックスアクセス記法をコードブロック内に直接記述する\n`;
qualitySection += `- OK: 「配列の先頭要素を取得する」「インデックス番号によるアクセス」のように散文形式で説明する\n`;
```

行 1156 の記述ではコードブロックへの記述が NG とされているが、実際にはコードフェンス内への記述は安全であるためこの記述は誤りである。
行 1159 の記述もコードブロック内が NG とされており同様の誤りである。コードフェンス内は安全な記述場所であり、サブエージェントが積極的に活用すべき場所である。
サブエージェントがこのプロンプトを読むと「コードフェンス内も危険」と誤解し、正規表現パターンをコードフェンス外の散文に記述するという逆効果が生じる。
これらの誤った NG 記述は `definitions.ts` の行 1156 と行 1159 の2箇所に存在し、いずれもコードフェンス内への記述を誤って制限していた。

## 修正後の正しいコード

### 修正後の内容（行 1155〜1160 の置き換え）

行 1155〜1160 の6行を以下の8行の内容に置き換える。各セクションはヘッダー行、NG 行、OK（推奨）行、OK（代替）行の4行構成とする。

```
qualitySection += `\n正規表現パターンの記述:\n`;
qualitySection += `- NG: コードフェンス外の散文や箇条書きに正規表現の文字クラス表記を直接記述すること\n`;
qualitySection += `- OK（推奨）: コードフェンス内に正規表現パターンを記述すること（バリデーターはコードフェンス内の行を検出対象から除外するため安全）\n`;
qualitySection += `- OK（代替）: コードフェンス外では「英小文字の1文字以上の繰り返しを表す正規表現」のように散文形式で説明すること\n`;
qualitySection += `\n配列アクセスの記述:\n`;
qualitySection += `- NG: コードフェンス外の散文や箇条書きに配列のインデックスアクセス記法を直接記述すること\n`;
qualitySection += `- OK（推奨）: コードフェンス内に配列アクセス記法を記述すること（バリデーターはコードフェンス内の行を検出対象から除外するため安全）\n`;
qualitySection += `- OK（代替）: コードフェンス外では「先頭要素を取得する」「インデックス番号によるアクセス」のように散文形式で説明すること\n`;
```

修正後は各セクション（正規表現パターン、配列アクセス）がそれぞれ4行（ヘッダー1行 + NG1行 + OK推奨1行 + OK代替1行）で構成され、全体で8行となる。
「コードフェンス内は安全」という情報が「OK（推奨）」として明示的に提示されるため、受け入れ基準 AC-2 を満たす。
「コードフェンス外の散文や箇条書き」への直接記述が「NG」として明示されるため、受け入れ基準 AC-1 を満たす。
修正により6行から8行に増加するが、TypeScript 文字列テンプレートの構文パターンは既存行と同一であるため、コンパイルエラーのリスクは極めて低い。

## コードフェンス内が安全である理由

### extractNonCodeLines による除外の仕組み

`artifact-validator.ts` における角括弧検出処理は、必ず `extractNonCodeLines` でフィルタリングされたテキストに対して実行される。
`extractNonCodeLines` 関数はコードフェンス境界行（バッククォート3個で始まる行）を検出し、対応する開始〜終了フェンス間の全行をスキップする。
この処理により、コードフェンス内の行は角括弧検出パターンの適用対象外となり、どのような記法を記述してもバリデーションエラーが発生しない。
インラインコードのバッククォートで囲まれた部分も同様に除去されるため、シングルバッククォート内への記述も安全である。
この仕組みを理解したサブエージェントは、技術的な記法（正規表現パターン、配列アクセス記法、コマンドラインの引数など）をコードフェンス内に適切に配置できるようになる。

## 既存テストへの影響分析

### definitions.test.ts の内容と影響範囲

`workflow-plugin/mcp-server/src/phases/__tests__/definitions.test.ts` は PHASES 配列のフェーズ数・順序・構成を検証するテストであり、`buildPrompt` 関数の内部出力（`qualitySection` の文字列内容）を直接検証するテストは含まれていない。
したがって、今回の修正（`qualitySection` の文字列内容の変更）は `definitions.test.ts` のテストケースに直接影響を与えない。
`resolvePhaseGuide` 関数の呼び出し結果も検証されていないため、プロンプト生成の出力変化による既存テスト失敗は発生しない。

### 他のテストファイルへの影響

`buildPrompt` 関数を直接呼び出すテストは `definitions.test.ts` 以外にも存在しない（Grep 確認済み）。
`artifact-validator.ts` のテスト群（`artifact-quality-enhanced.test.ts` 等）はバリデーター側の挙動を検証しており、プロンプト生成の変更による影響を受けない。
今回の修正はプロンプト文字列の内容変更のみであり、`buildPrompt` の関数シグネチャ・戻り値の型・呼び出し元の動作には影響しない。

### 回帰リスクの評価

修正による型エラー・構文エラーのリスクは最小限である。変更対象は TypeScript の文字列テンプレートリテラル（バッククォートで囲まれた文字列）への `+=` 代入であり、既存の構文パターンを踏襲する。
唯一の注意点は、OK（推奨）行と OK（代替）行が追加されることで行数が6行から8行に増加することである。これは機能拡張であり、既存の動作を壊すものではない。
