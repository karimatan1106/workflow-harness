## サマリー

- 目的: `buildPrompt()` 関数内の「角括弧プレースホルダー禁止」セクションにおいて、NG/OK 例が逆に記述されているバグを修正するための要件定義。この誤りにより全フェーズのサブエージェントがコードフェンスに対して誤った理解を持ち、正規表現パターンをコードフェンス外に記述してしまうという問題が発生している。
- 主要な決定事項: 修正対象は `buildPrompt()` 関数のみとする。ハードコードされた `subagentTemplate` 文字列は `buildPrompt()` によって上書きされるため、修正の主体を `buildPrompt()` に集中させる。正しい NG/OK 例は「コードフェンス外への直接記述が NG」「コードフェンス内への記述または散文形式での説明が OK」というルールに基づいて再構成する。
- 次フェーズで必要な情報: 修正対象ファイルは `workflow-plugin/mcp-server/src/phases/definitions.ts` の行 1155〜1160 の6行（`qualitySection` に追加される正規表現パターンおよび配列アクセスに関する NG/OK 例の文字列）。修正後にはビルド（`npm run build`）とMCPサーバーの再起動が必要。

## 背景と問題の概要

### 問題が発生した経緯

parallel_verification フェーズの4サブエージェント（manual_test, security_scan, performance_test, e2e_test）が連続してバリデーションエラーを発生させた。
調査の結果、エラーの根本原因はサブエージェントへの指示が誤解を招く記述になっていたことであり、コードフェンス内を危険と誤解させる説明が含まれていた。
この誤解招く記述は `buildPrompt()` 関数で全フェーズ共通に生成されるため、parallel_verification の4フェーズだけでなく全フェーズのサブエージェントが影響を受けている。

### 現在のコードの問題箇所

`workflow-plugin/mcp-server/src/phases/definitions.ts` の行 1155〜1160 が問題箇所である。
現在の記述では、正規表現パターンをコードブロックに書くことを「NG」と記述し、散文で説明することを「OK」と記述している。
また、配列のインデックスアクセス記法をコードブロック内に記述することを「NG」と記述し、散文形式での説明を「OK」としている。
しかし、バリデーターの実際の動作は `extractNonCodeLines` によってコードフェンス内の行を検出対象から除外するため、コードフェンス内への記述は完全に安全である。
つまり、現在の NG/OK の対応が完全に逆になっており、サブエージェントを誤った方向へ誘導している。

### バリデーターの実際の動作

`artifact-validator.ts` の `extractNonCodeLines` 関数はコードフェンスで囲まれた行を全てスキップする。
角括弧検出パターン（`/\[(?!関連|参考|注|例|出典)[^\]]{1,50}\]/g`）はこの関数でフィルタリングされた後のテキストにのみ適用される。
この事実から、コードフェンス内に角括弧を含む記述（正規表現の文字クラス、配列アクセス記法など）を書いてもバリデーションエラーは発生しない。
コードフェンス外の散文テキストや箇条書きにこれらの記法を直接書いた場合にのみエラーが発生する。

## 機能要件

### 要件 FR-1: 正規表現パターンに関する NG/OK 例の修正

`buildPrompt()` 関数内の正規表現パターンに関する NG/OK 例を以下の内容に修正すること。

修正前の誤った記述（現在の行 1156〜1157 相当）:
行 1156: `- NG: 正規表現で「英小文字1文字以上」を表すパターンをそのままコードブロックに書く`
行 1157: `- OK: 「英小文字の1文字以上の繰り返しを表す正規表現」のように散文で説明する`

修正後の正しい記述:
行 1156 相当: `- NG: コードフェンス外の散文や箇条書きに正規表現の文字クラス表記を直接記述すること`
行 1157 相当: `- OK（推奨）: コードフェンス内に正規表現パターンを記述すること（バリデーターはコードフェンス内の行を検出対象から除外するため安全）`
行 1157b 相当: `- OK（代替）: コードフェンス外では「英小文字の1文字以上の繰り返しを表す正規表現」のように散文形式で説明すること`

### 要件 FR-2: 配列アクセス記法に関する NG/OK 例の修正

`buildPrompt()` 関数内の配列アクセス記法に関する NG/OK 例を以下の内容に修正すること。

修正前の誤った記述（現在の行 1159〜1160 相当）:
行 1159: `- NG: 配列のインデックスアクセス記法をコードブロック内に直接記述する`
行 1160: `- OK: 「配列の先頭要素を取得する」「インデックス番号によるアクセス」のように散文形式で説明する`

修正後の正しい記述:
行 1159 相当: `- NG: コードフェンス外の散文や箇条書きに配列のインデックスアクセス記法を直接記述すること`
行 1160 相当: `- OK（推奨）: コードフェンス内に配列アクセス記法を記述すること（バリデーターはコードフェンス内の行を検出対象から除外するため安全）`
行 1160b 相当: `- OK（代替）: コードフェンス外では「先頭要素を取得する」「インデックス番号によるアクセス」のように散文形式で説明すること`

### 要件 FR-3: 修正範囲の限定

修正は `buildPrompt()` 関数内の行 1155〜1160 の範囲（「正規表現パターンの記述」セクションおよび「配列アクセスの記述」セクション）に限定する。
行 1153〜1154（コードフェンス外が禁止対象であることの説明）は現在正しい記述であるため変更しないこと。
行 1150〜1152（角括弧プレースホルダー禁止の宣言と許可キーワード列挙）も変更対象外である。

### 要件 FR-4: MCPサーバービルドと再起動

`definitions.ts` を修正した後は、以下の手順を順番に実行すること。
手順1: `cd workflow-plugin/mcp-server && npm run build` を実行してTypeScriptをトランスパイルする。
手順2: `dist/phases/definitions.js` の更新日時を確認して変更が反映されていることを検証する。
手順3: MCPサーバープロセスをClaude Desktopのサーバー再起動機能で再起動する。
手順4: 再起動後に `workflow_status` を実行して現在のフェーズを確認してから作業を再開する。

## 非機能要件

### 要件 NFR-1: 後方互換性の維持

修正後も行 1153〜1154 の「コードフェンス外が禁止対象」という情報は保持されること。
修正は NG/OK 例の方向を正すだけであり、角括弧プレースホルダー禁止ルールそのものを廃止・変更してはならない。
コードフェンス内が安全であるという情報は明示的に追加することが求められるが、コードフェンス外が禁止であるという既存の記述を削除してはならない。

### 要件 NFR-2: プロンプト長の節度

修正によって `buildPrompt()` が生成するプロンプト文字列が過度に長くなることを避けること。
各記述は1〜2行以内に収めるよう努め、冗長な説明は避けること。
正規表現セクションと配列アクセスセクションはそれぞれ NG 1行、OK（推奨）1行、OK（代替）1行の計3行以内で完結させること。

### 要件 NFR-3: サブエージェントの誤解防止

修正後のプロンプトを読んだサブエージェントが「コードフェンス内は安全」という正しい理解を得られること。
サブエージェントが正規表現の文字クラス表記を含む記述をコードフェンス内に配置する動機が生まれるよう、コードフェンスの活用を「OK（推奨）」として明示的に案内すること。
「コードフェンス外への直接記述が NG」「コードフェンス内への配置または散文形式が OK」という対比が一読で理解できる構成にすること。

### 要件 NFR-4: 全フェーズへの影響

`buildPrompt()` は全フェーズで呼び出されるため、修正の効果は全フェーズのサブエージェントプロンプトに即座に反映される。
特に正規表現パターンや配列アクセス記法を記述する機会が多い security_scan, e2e_test フェーズでの誤用防止効果が最も重要である。
修正後に security_scan および e2e_test フェーズのサブエージェントがバリデーションエラーを起こさないことを受け入れ基準の一部とする。

## 受け入れ基準

### AC-1: NG/OK 例の方向が正しいこと

修正後の `buildPrompt()` が生成するプロンプトにおいて、正規表現パターンをコードフェンス内に書くことが「NG」と記述されていないこと。
修正後のプロンプトにおいて、配列のインデックスアクセス記法をコードフェンス内に書くことが「NG」と記述されていないこと。
修正後のプロンプトにおいて、コードフェンス外の散文や箇条書きへの直接記述が「NG」と記述されていること。

### AC-2: コードフェンスの安全性が明示されていること

修正後のプロンプトにおいて、コードフェンス内への記述が安全（バリデーター除外対象）であることが明記されていること。
「OK（推奨）」として少なくとも一つのコードフェンス活用例が提示されていること。

### AC-3: ビルドが成功すること

`npm run build` がエラーなく完了し、`dist/phases/definitions.js` が生成されること。
TypeScriptコンパイルエラーが発生していないこと。

### AC-4: 既存テストが通過すること

`buildPrompt()` に関連する既存のユニットテストが全て通過すること。
`definitions.ts` の変更がテストスイート全体のパス率に影響を与えていないこと。

### AC-5: 修正後フェーズのサブエージェントがバリデーションを通過すること

修正後に実行される security_scan または e2e_test フェーズのサブエージェントが、角括弧に関するバリデーションエラーを発生させないこと。
これはバリデーター動作の回帰テストとして機能する。
