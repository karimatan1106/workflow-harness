## サマリー

本ドキュメントは `workflow-plugin/mcp-server/src/phases/definitions.ts` の `buildPrompt()` 関数内に存在する NG/OK 例の誤記（行 1155〜1160）を修正するタスクのテスト設計書である。

- 目的: 修正後の `buildPrompt()` が正しい NG/OK 対比を含んでいることを検証し、ビルドと既存テストスイートの継続的な通過を確認する。
- 主要な決定事項: テスト対象は `buildPrompt()` の戻り値文字列の内容検査であり、6行から8行への行数増加が正しく実施されることを確認する。既存の `definitions.test.ts` は `buildPrompt()` の文字列内容を直接検証していないため、新規テストケースを追加する設計とする。
- 次フェーズで必要な情報: test_impl フェーズでは `workflow-plugin/mcp-server/src/phases/__tests__/definitions.test.ts` に新規テストケースを追加する。ビルド後の `dist/phases/definitions.js` ファイルが最新であることを確認してからテストを実行すること。

---

## テスト方針

本タスクの変更は `definitions.ts` の `buildPrompt()` 関数内の文字列リテラルのみであり、関数シグネチャや戻り値の型は変更されない。そのため、テストは以下の3層で構成する。

テストの第1層は「出力文字列の内容検査」である。`buildPrompt()` を呼び出し、生成されたプロンプト文字列に正しい NG/OK 対比が含まれているかを確認する。この層が本タスクの主要テスト対象であり、AC-1 および AC-2 に対応する。

テストの第2層は「ビルド成功確認」である。TypeScript ソースを `npm run build` でトランスパイルし、`dist/phases/definitions.js` が生成されることを確認する。この層は AC-3 に対応し、文字列テンプレートのエスケープ問題等をコンパイル時に検出するために実施する。

テストの第3層は「回帰テスト」である。既存の全テストスイートを実行し、修正前後でテスト結果が変化しないことを確認する。この層は AC-4 に対応し、今回の修正が既存の PHASES 配列テストや `resolvePhaseGuide` テストに影響を与えないことを保証する。

テスト実行は `workflow-plugin/mcp-server` ディレクトリ内で `npx vitest run` を使用する。テスト種別ごとの実施優先度は、第1層（出力内容検査）が最高優先、第2層（ビルド確認）が中優先、第3層（回帰テスト）が標準優先である。

## テストケース

### TC-1: NG記述の除去確認

- テスト種別: 単体テスト（`buildPrompt()` 出力の文字列検査）
- 対応受け入れ基準: AC-1
- テスト対象ファイル: `workflow-plugin/mcp-server/src/phases/__tests__/definitions.test.ts`
- 前提条件: `definitions.ts` の行 1155〜1160 が正しい8行に修正されていること

実装方針: `resolvePhaseGuide` 関数を呼び出すか、あるいは `buildPrompt` 関数を直接インポートして呼び出し、戻り値の文字列に `コードブロックに書く` および `コードブロック内に直接記述する` というフレーズが含まれていないことを `not.toContain()` で検査する。

期待結果: 両方のフレーズが戻り値に含まれない（`not.toContain()` がパスする）。

## リスクと注意事項

修正箇所に含まれる文字列テンプレートリテラルは、バッククォートで区切られた TypeScript テンプレートリテラル（テンプレートリテラル）の中にさらにバッククォートを含む可能性がある。spec.md に記載された修正後の8行を確認すると、「バリデーターはコードフェンス内の行を検出対象から除外するため安全」というコメントはバッククォートを含まないが、OK（代替）行の散文説明文も同様にバッククォートを含まない形式になっている。そのため TypeScript のコンパイルエラーリスクは低いが、ビルド確認（TC-5）で必ず検証すること。

既存テスト `definitions.test.ts` は `buildPrompt` 関数を直接インポートしていない（`resolvePhaseGuide` のみをインポートしている）ため、TC-1 から TC-4 のテストでは `resolvePhaseGuide` の戻り値を経由するか、`buildPrompt` をエクスポートして直接呼び出す形式を採用する必要がある。`buildPrompt` が現在 `definitions.ts` からエクスポートされているかどうかを実装前に確認すること。`buildPrompt` がエクスポートされていない場合は `resolvePhaseGuide` の戻り値の `subagentTemplate` フィールドを使用して文字列内容を検査する方針に変更する。

MCPサーバーのモジュールキャッシュにより、ビルド後に MCP サーバープロセスを再起動しないと修正が反映されない。テストの実行自体は Vitest が直接ソースファイルをインポートするため MCP サーバー再起動は不要である。ただし、修正の効果を実際のワークフロー実行で確認する場合は MCP サーバー再起動が必須である。
