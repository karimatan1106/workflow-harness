# 脅威モデル: task-index.jsonキャッシュ同期の根本原因修正

## サマリー

本脅威モデルはSTRIDEモデルに基づき、task-index.jsonの即時更新メカニズムおよびphase-edit-guardのチェック順序変更に関連するセキュリティリスクを分析する。

主要な脅威は以下の通り:
- task-index.jsonの改ざんによるフェーズ偽装とセキュリティコントロールのバイパス
- レースコンディションによるファイル破損と不整合状態の発生
- B-2チェック順序変更による意図しないコマンド許可
- キャッシュTTL短縮によるDoS攻撃ベクトルの拡大

対策として、ファイルロック機構の実装、B-2チェックの厳格な検証、エラーハンドリングの強化、監査ログの充実を提案する。
全ての対策を実装することで、セキュリティリスクを許容範囲内に抑えつつ、commitフェーズでのブロック問題を根本解決する。

次フェーズ(planning)では、各対策の詳細設計と実装計画を作成する。

## STRIDE分析

### S - Spoofing (なりすまし)

#### 脅威1: task-index.jsonの手動改ざんによるフェーズ偽装

**説明:**
悪意のあるユーザーまたは攻撃者が、task-index.jsonを直接編集してフェーズ情報を偽装する。
例えば、現在のフェーズがresearchであるにもかかわらず、task-index.json内のphaseフィールドを"implementation"に書き換える。
hookはこの偽装されたフェーズ情報を信頼し、本来禁止されているはずのコード編集を許可してしまう。

**影響:**
- ワークフロープロセスのバイパス(TDDサイクルの破壊)
- 仕様書なしでの実装開始
- テスト設計なしでのコード変更
- 品質ガードレールの無効化

**重大度:** 高（ワークフロープロセス全体のバイパスにつながるため）

**緩和策:**
- task-index.jsonにHMAC署名を追加し、改ざん検出を実装する
- hook実行時にworkflow-state.jsonとtask-index.jsonの整合性をクロスチェックする
- 不整合検出時はworkflow-state.jsonを信頼源としてtask-index.jsonを再構築する
- 改ざん試行を.claude/state/audit-log.jsonlに記録し、異常検知アラートを出す

**実装優先度:** 中(現行のHMACはworkflow-state.jsonのみだが、task-index.jsonへの拡張は将来課題)

#### 脅威2: 古いキャッシュを悪用したタイムアウト攻撃

**説明:**
攻撃者が意図的にMCP serverプロセスを停止または遅延させ、hook側のキャッシュTTL(1時間)を悪用する。
フェーズ遷移後に1時間以内にhookが実行されると、古いフェーズ情報が返されることを利用し、本来許可されないコマンドを実行する。

**影響:**
- 一時的なセキュリティコントロールの無効化
- フェーズ制約の回避

**重大度:** 低(MCP serverの停止は管理者権限が必要)

**緩和策:**
- REQ-4によりキャッシュTTLを30秒に短縮し、攻撃ウィンドウを狭める
- MCP server停止時はhook側で検出し、フォールバック動作として厳格なデフォルト拒否ポリシーを適用する

**実装優先度:** 低(REQ-4採用時に自動的に緩和)

### T - Tampering (改ざん)

#### 脅威3: レースコンディションによるtask-index.jsonの破損

**説明:**
複数のMCP serverプロセスまたは並行ワークフローが同時にtask-index.jsonを更新しようとする。
ファイルロック機構が実装されていない場合、部分的な書き込みやJSON構造の破壊が発生する。
破損したtask-index.jsonをhookがパースしようとするとエラーが発生し、全てのhookが機能停止する。

**影響:**
- hook全体の機能停止(git操作が全てブロックされる)
- ワークフロープラグインの完全な機能不全
- データ復旧が必要になるまでの開発停止

**重大度:** 高（hook全体の機能停止に直結するため）

**緩和策:**
- fs.promises.openでO_EXCLフラグを使用した排他制御を実装する
- 書き込み前に一時ファイルに書き出し、完了後にatomic renameで上書きする
- 更新失敗時は非致命的エラーとしてログ出力し、次回のhook実行時にworkflow-state.jsonから再構築する
- テストケースで並行更新のシナリオを検証し、ロック機構の正しい動作を確認する

**実装優先度:** 高(REQ-2の実装で必須)

#### 脅威4: updateTaskIndexForSingleTask()のロジックバグによる不整合

**説明:**
新規実装するupdateTaskIndexForSingleTask()メソッドにバグがあり、特定タスクのphaseは更新されるが他のフィールドが失われる。
例えば、taskNameやdocsDirが消失し、hookが必要な情報を取得できなくなる。

**影響:**
- hookの誤動作や機能低下
- タスク情報の部分的な喪失

**重大度:** 中（hookの部分的な誤動作を引き起こすため）

**緩和策:**
- updateTaskIndexForSingleTask()実装時、既存フィールドを全て保持し、phaseのみを更新するロジックを厳密に実装する
- ユニットテストで、更新前後のフィールド整合性を検証する
- 更新後にJSON schemaバリデーションを実行し、必須フィールドの欠損を検出する

**実装優先度:** 高(REQ-2の実装で必須)

### R - Repudiation (否認)

#### 脅威5: フェーズ遷移操作の監査ログ欠落

**説明:**
updateTaskIndexForSingleTask()が実行されても、.claude/state/audit-log.jsonlへの記録が行われない。
後からフェーズ遷移の履歴を追跡しようとしても、誰がいつフェーズを変更したかが不明になる。

**影響:**
- セキュリティインシデント調査の困難化
- コンプライアンス違反(監査証跡の欠如)

**重大度:** 中（セキュリティインシデント調査の困難化を招くため）

**緩和策:**
- updateTaskIndexForSingleTask()実行時に、audit-log.jsonlにフェーズ遷移イベントを記録する
- ログエントリに{timestamp, taskId, fromPhase, toPhase, actor: "mcp-server"}を含める
- ログ書き込み失敗はエラーログに出力するが、フェーズ遷移自体は継続する(非致命的エラー)

**実装優先度:** 中(セキュリティベストプラクティス)

#### 脅威6: チェック順序変更による許可ログの記録漏れ

**説明:**
B-2チェックをbash-whitelistより前に移動した結果、許可されたコマンドのログが従来と異なる箇所に記録される。
既存の監査ログ解析スクリプトがこの変更に対応しておらず、commit/pushフェーズでのgit操作が監査から漏れる。

**影響:**
- 監査ログの不完全性
- セキュリティ分析の精度低下

**重大度:** 低（既存のログ解析への影響は限定的）

**緩和策:**
- B-2チェック実行時に明示的なログエントリを追加し、"B-2 commit/push check: ALLOW"のような形式で記録する
- ログフォーマットの変更をdocs/security/audit-log-format.mdに文書化する
- 既存の監査ログ解析スクリプトを更新し、新しいログフォーマットに対応させる

**実装優先度:** 低(監査機能の充実は将来課題)

### I - Information Disclosure (情報漏洩)

#### 脅威7: task-index.jsonに含まれる機密情報の漏洩

**説明:**
task-index.jsonはファイルシステム上に平文で保存される。
taskNameやdocsDirに機密プロジェクト名や内部パス情報が含まれている場合、不正アクセス時に情報が漏洩する。

**影響:**
- プロジェクト構造の漏洩
- 内部パス情報の暴露

**重大度:** 低(ローカルファイルシステムへのアクセスが前提)

**緩和策:**
- task-index.jsonのファイルパーミッションを600(所有者のみ読み書き可能)に設定する
- taskNameやdocsDirには機密情報を含めないガイドラインを策定する
- 機密プロジェクトではtask-index.jsonの暗号化を検討する(オプション)

**実装優先度:** 低(現状のアクセス制御で十分)

#### 脅威8: エラーログに含まれるファイルパス情報の過剰出力

**説明:**
updateTaskIndexForSingleTask()のエラーハンドリングで、エラーログにtask-index.jsonの完全なファイルパスが出力される。
ログがネットワーク越しに送信される環境では、内部ディレクトリ構造が外部に漏洩するリスクがある。

**影響:**
- ディレクトリ構造の漏洩
- 攻撃者への偵察情報の提供

**重大度:** 低（ログの外部送信がない環境では影響が最小）

**緩和策:**
- エラーログでは絶対パスではなく相対パスを使用する
- 機密性の高い環境では、ログ出力前にパス情報をサニタイズする
- ログの外部送信時はTLS暗号化を必須とする

**実装優先度:** 低(内部開発環境での使用が主)

### D - Denial of Service (サービス拒否)

#### 脅威9: task-index.json更新の頻発によるディスクI/O過負荷

**説明:**
攻撃者が意図的に高頻度でフェーズ遷移を実行し、updateTaskIndexForSingleTask()を連続的に呼び出す。
毎回のファイルI/O(読み込み+書き込み)が蓄積し、ディスク負荷が上昇してシステム全体が遅延する。

**影響:**
- システム全体のパフォーマンス劣化
- MCP serverのレスポンス遅延
- 他のプロセスへの影響

**重大度:** 中（システム全体のパフォーマンスに波及するため）

**緩和策:**
- フェーズ遷移APIにレートリミットを実装し、短時間での連続実行を制限する(例: 1秒に最大10回)
- updateTaskIndexForSingleTask()内でインメモリキャッシュを使用し、連続する更新でファイルI/O回数を削減する
- ディスクI/O監視を追加し、異常な負荷を検出したらアラートを出す

**実装優先度:** 中(パフォーマンステストで許容範囲を確認)

#### 脅威10: キャッシュTTL短縮によるファイルI/O増加

**説明:**
REQ-4でキャッシュTTLを1時間から30秒に短縮すると、hook実行のたびにtask-index.jsonの再読み込みが発生する頻度が増加する。
高頻度のgit操作が行われる環境では、ディスクI/Oが過剰になりパフォーマンスが劣化する。

**影響:**
- hookの実行時間増加(50ms以上)
- git操作の体感速度低下
- バッテリー消費増加(ラップトップ環境)

**重大度:** 低（REQ-4は任意実装であり見送り可能）

**緩和策:**
- パフォーマンステストで、TTL 30秒での実行時間を測定し、50ms以内に収まることを確認する
- 許容範囲外であればREQ-4を見送り、REQ-1/REQ-2/REQ-3のみで問題解決する
- task-cache.jsのメモリキャッシュを活用し、同一プロセス内での再読み込みを避ける

**実装優先度:** 低(REQ-4は任意実装)

#### 脅威11: ファイルロック競合によるタイムアウト

**説明:**
排他制御のためのファイルロックが長時間保持され、後続の更新リクエストがタイムアウトする。
複数のワークフローが同時実行される環境で、ロック待ちによりフェーズ遷移APIがハングする。

**影響:**
- フェーズ遷移の失敗
- ユーザー操作のブロック
- ワークフロープラグインの一時的な機能停止

**重大度:** 中（ユーザー操作が一時的にブロックされるため）

**緩和策:**
- ファイルロックのタイムアウトを設定し、一定時間(例: 5秒)経過後は強制解除する
- ロック取得失敗時は非致命的エラーとして扱い、次回のhook実行時に再試行する
- ロック競合が頻発する場合はアラートを出し、システム管理者に通知する

**実装優先度:** 中(REQ-2の実装で検討)

### E - Elevation of Privilege (権限昇格)

#### 脅威12: B-2チェック順序変更による意図しないコマンド許可

**説明:**
B-2チェックをbash-whitelistより前に移動した結果、B-2のロジックバグにより本来許可されないコマンドが通過する。
例えば、"git add; rm -rf /"のような悪意のあるコマンドが、git addで始まるという理由だけで許可される。

**影響:**
- システム破壊
- データ喪失
- セキュリティポリシーの完全なバイパス

**重大度:** 極めて高

**緩和策:**
- B-2チェックのロジックを厳密に実装し、コマンド全体を検証する(セミコロン区切りの複数コマンドを検出してブロック)
- git add/git commitで始まるコマンドであっても、正規表現で^(git add|git commit)で始まり、かつ許可された引数のみを含むことを確認する
- B-2でマッチしなかった場合は必ずbash-whitelistチェックにフォールバックさせる
- 既存のphase-edit-guard.test.jsに回帰テストを追加し、悪意のあるコマンドが確実にブロックされることを確認する

**実装優先度:** 最高(REQ-3の実装で必須)

#### 脅威13: task-index.json偽装によるフェーズ制約のバイパス

**説明:**
脅威1(フェーズ偽装)が成功した場合、攻撃者は本来実行できないフェーズのコマンドを実行できる。
例えば、researchフェーズでcode_reviewフェーズに偽装し、コード編集を行う。

**影響:**
- ワークフロープロセスの完全なバイパス
- 品質ガードレールの無効化

**重大度:** 高（フェーズ制約の完全なバイパスを許すため）

**緩和策:**
- 脅威1の緩和策(HMAC署名、整合性クロスチェック)を実装する
- hookでworkflow-state.jsonの直接読み取りをフォールバックとして実装し、task-index.jsonの信頼性が低い場合は正規のソースから取得する

**実装優先度:** 中(脅威1の緩和策と同一)

#### 脅威14: MCP serverの脆弱性を利用したフェーズ遷移の強制

**説明:**
攻撃者がMCP serverのAPIエンドポイントに不正なリクエストを送信し、認証や承認をバイパスしてフェーズを強制遷移させる。
例えば、design_reviewフェーズでの承認を待たずにtest_implフェーズに進む。

**影響:**
- ワークフローのセキュリティガードレールのバイパス
- レビュー承認プロセスの無効化

**重大度:** 高（レビュー承認プロセスが無意味になるため）

**緩和策:**
- MCP serverのフェーズ遷移APIに認証・認可機構を実装する(現行は未実装と思われる)
- workflow_nextやworkflow_approveの実行時に、セッショントークンやHMAC検証を追加する
- 不正なフェーズ遷移試行をaudit-log.jsonlに記録し、異常検知アラートを出す

**実装優先度:** 低(本タスクのスコープ外だが、将来の重要課題)

## 攻撃シナリオ

### シナリオ1: 悪意のある開発者による品質ガードレールのバイパス

**攻撃者:** 悪意のある開発者(内部脅威)

**動機:** 短納期のプレッシャーからテストやレビューをスキップして実装を進めたい

**攻撃手順:**
1. researchフェーズでワークフローを開始する
2. task-index.jsonを直接編集し、phaseフィールドを"implementation"に書き換える
3. hookはこの偽装されたフェーズを信頼し、コード編集を許可する
4. 仕様書やテスト設計なしで実装を完了し、commitする
5. 後からrequirements.mdやtest-design.mdを形式的に作成してアリバイを作る

**影響:**
- TDDサイクルの破壊
- 品質低下
- 技術的負債の蓄積

**対策:**
- T1の緩和策(HMAC署名、整合性クロスチェック)を実装する
- hookでworkflow-state.jsonとのクロスチェックを行い、不整合時はブロックする

### シナリオ2: レースコンディションを利用したDoS攻撃

**攻撃者:** システムに詳しい悪意のある開発者

**動機:** 開発プロセスを混乱させ、プロジェクト遅延を引き起こす

**攻撃手順:**
1. スクリプトで複数のワークフローを並行起動する
2. 各ワークフローで高頻度にフェーズ遷移を実行する
3. updateTaskIndexForSingleTask()が同時に実行され、ファイルロック競合が発生する
4. task-index.jsonが部分的に書き込まれ、JSON構造が破壊される
5. 以降のhook実行が全てエラーになり、git操作が一切できなくなる

**影響:**
- ワークフロープラグインの完全な機能停止
- 開発作業の停止

**対策:**
- T3の緩和策(ファイルロック、atomic rename)を実装する
- T11の緩和策(ロックタイムアウト、非致命的エラー処理)を実装する

### シナリオ3: B-2チェック順序変更の脆弱性を悪用した権限昇格

**攻撃者:** システムに詳しい悪意のある開発者

**動機:** 本来実行できないコマンドを実行して、システムを破壊する

**攻撃手順:**
1. commitフェーズでgit addコマンドを実行するふりをする
2. 実際のコマンドは"git add .; rm -rf /important/data"のような破壊的コマンド
3. B-2チェックがコマンド全体を検証せず、git addで始まるという理由だけで許可する
4. セミコロン以降の破壊的コマンドが実行され、データが喪失する

**影響:**
- データ喪失

## セキュリティ要件

### SEC-1: ファイル整合性保護

task-index.jsonの更新は、以下の条件を満たすこと:
- 排他制御により並行更新を防止する
- atomic renameでファイル破損を防止する
- 更新失敗時は非致命的エラーとして扱い、次回のhook実行時に再構築する

### SEC-2: コマンド検証の厳格化

B-2チェックは、以下の条件を満たすこと:
- コマンド全体を検証し、セミコロン区切りの複数コマンドを検出してブロックする
- git add/git commitで始まるコマンドであっても、許可された引数パターンのみを許可する
- 不正なコマンドパターンを検出した場合は、audit-log.jsonlに記録してブロックする

### SEC-3: 監査ログの完全性

フェーズ遷移および権限昇格に関連する操作は、全てaudit-log.jsonlに記録すること:
- updateTaskIndexForSingleTask()の実行(taskId, fromPhase, toPhase)
- B-2チェックでの許可/拒否判定(command, phase, decision)
- task-index.jsonの整合性エラー検出(taskId, expected, actual)

### SEC-4: エラーハンドリングの安全性

エラー発生時は、以下の原則に従うこと:
- デフォルト拒否(Fail Secure): 不明な状態では操作を許可しない
- 非致命的エラー処理: task-index.json更新失敗でフェーズ遷移を止めない
- 詳細なエラーログ: デバッグに必要な情報を記録するが、機密情報は含めない
