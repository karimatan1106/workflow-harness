## サマリー

- 目的: FR-20（security_scan テンプレートへの総合評価ガイダンス追加）と FR-21（CLAUDE.md へのセッション再開時 sessionToken 再取得手順追記）に関するセキュリティ脅威を評価する。
- 主要な決定事項: 両変更はドキュメント・テンプレート文字列の追記に留まり、実行コードの追加は伴わない。そのため脅威の中心は「ガイダンス文言の不正な誘導」「sessionToken の不適切な取り扱い」「テストスイートのリグレッション」の3領域となる。
- 評価対象コンポーネント: `workflow-plugin/mcp-server/src/phases/definitions.ts`（FR-20）および `CLAUDE.md`（FR-21）の2ファイル。
- 対象脅威数: 4件（シナリオ1〜4）。全て Low〜Medium-High の範囲内であり、致命的な脆弱性は確認されていない。
- 次フェーズで必要な情報: 実装フェーズでは禁止語チェックと既存テスト合格確認を最重要の検証項目として扱うこと。sessionToken の漏洩リスクは低いが、CLAUDE.md の記述が subagent に誤解を与えないよう文言の精度を優先すること。

---

## 脅威シナリオ

### シナリオ1: FR-20 ガイダンスへの禁止語混入

- **対象コンポーネント**: `definitions.ts` の security_scan subagentTemplate
- **脅威の内容**: 5観点ガイダンスの追加テキストにバリデーターの禁止リストに収録されている語句（英語4語・日本語8語の計12語）が含まれた場合、そのテンプレートを受け取った subagent が成果物に同じ語句を転記してバリデーション失敗を引き起こす。
- **影響範囲**: security_scan フェーズを実行する全タスクに影響し、parallel_verification が永続的に失敗し続ける状態（フィードバックループ）を形成する。
- **攻撃ベクター**: テンプレート文字列の変更時に、バリデーターの禁止リストに含まれる語句（分析が継続中の観点・後続フェーズで対処する項目などを指す語句）が安易に使用されること。
- **発生条件**: 実装担当者が禁止語リストを参照せずに文言を作成した場合に発生する。

### シナリオ2: FR-21 の記述が subagent に sessionToken の誤使用を促す

- **対象コンポーネント**: `CLAUDE.md` の「AIへの厳命」セクション
- **脅威の内容**: ルール23の記述が「sessionToken を取得後に workflow_next 等のワークフロー制御ツールへ渡してよい」と誤解される文言になった場合、subagent がワークフロー制御を奪取して自律実行する既知問題（FR-4/FR-5/FR-19 で過去に発生）が再発する。
- **影響範囲**: testing・regression_test・docs_update 以外のフェーズでも subagent がワークフロー制御ツールを呼び出す動作が誘発される可能性がある。
- **攻撃ベクター**: ルール23の文章が「sessionToken を取得して活用せよ」という積極的な指示として読まれること。
- **発生条件**: 文章が「sessionToken の再取得先と用途の限定」ではなく「sessionToken の活用奨励」として記述された場合に発生する。

### シナリオ3: FR-20 テンプレート変更によるスナップショットテストのリグレッション

- **対象コンポーネント**: `workflow-plugin/mcp-server` のテストスイート
- **脅威の内容**: definitions.ts のテンプレート文字列を変更すると、既存のスナップショットテスト（存在する場合）が不一致を検出してテスト失敗となる。CI/CD がブロックされ、次のリリースが停止する。
- **影響範囲**: workflow-plugin の全テスト（現時点で 945 テスト程度）に影響する可能性がある。
- **攻撃ベクター**: テンプレート変更後に `npm test` を実行せず、そのまま commit・push した場合に発現する。
- **発生条件**: スナップショットテストが存在し、更新されないまま CI に到達した場合に発生する。

### シナリオ4: FR-21 記述が sessionToken の保存方法を明示しすぎることによる情報漏洩リスク

- **対象コンポーネント**: `CLAUDE.md`（公開リポジトリに含まれる可能性のあるファイル）
- **脅威の内容**: sessionToken の生成アルゴリズム・保存パス・再取得方法を詳細に記述しすぎることで、悪意ある第三者がworkflow-state.jsonを偽造してセッションを乗っ取る攻撃の参考情報になる。
- **影響範囲**: workflow-state.json に対する HMAC 整合性チェックの回避につながる可能性がある。
- **攻撃ベクター**: CLAUDE.md に具体的な sessionToken の構造（HMAC-SHA256 等）が記述されること。
- **発生条件**: ルール23の記述が「取得方法の概要」を超えて「セキュリティ実装の詳細」まで踏み込んだ場合に発生する。

## リスク評価

### 脅威1: FR-20 ガイダンスへの禁止語混入
- **深刻度**: Medium（security_scan フェーズのバリデーション失敗が継続するが、他フェーズには影響しない）
- **発生確率**: Low（実装者が CLAUDE.md の禁止語リストを参照すれば防止可能）
- **総合リスク**: Low-Medium（防止策が明確であり、発生しても影響範囲は限定的）
- **対策**: 実装時に禁止語リスト全 12 語を参照し、5観点ガイダンスの文言を事前に検証すること。

### 脅威2: FR-21 の記述が subagent に sessionToken の誤使用を促す
- **深刻度**: High（ワークフロー制御奪取は過去に実際に発生した障害であり、再発時の影響が大きい）
- **発生確率**: Medium（FR-19 で全フェーズに禁止指示が追加済みだが、CLAUDE.md の記述次第で誤解が生じうる）
- **総合リスク**: Medium-High（過去に同種の障害が発生しており、再発防止の記述精度が重要）
- **対策**: ルール23の文言は「sessionToken の再取得先は workflow_status のみ、使用先は workflow_record_test_result のみ」と明確に限定すること。

### 脅威3: FR-20 テンプレート変更によるスナップショットテストのリグレッション
- **深刻度**: Low（テストの再実行とスナップショット更新で解消可能）
- **発生確率**: Medium（テンプレート文字列の変更はスナップショットテストに直接影響する）
- **総合リスク**: Low（通常の開発プロセスで検出・修正が可能）
- **対策**: 変更後に `npm test` を実行し、スナップショット不一致が発生した場合はスナップショットを更新してから commit すること。

### 脅威4: FR-21 記述が sessionToken の保存方法を明示しすぎることによる情報漏洩リスク
- **深刻度**: Low（CLAUDE.md は開発者向けガイドであり、公開リポジトリに含まれていても実用的な攻撃には workflow-state.json への物理アクセスが別途必要）
- **発生確率**: Very Low（CLAUDE.md が攻撃者に参照される機会は限定的）
- **総合リスク**: Very Low（実装上の防御はすでに HMAC 整合性チェックで担保されている）
- **対策**: ルール23の記述は概要レベルに留め、セキュリティ実装の詳細（HMAC アルゴリズム等）には言及しないこと。

## セキュリティ要件

### 要件1: FR-20 テンプレートの禁止語ゼロ保証

- FR-20 で追加する5観点ガイダンスのテキスト全体を禁止語リスト（英語4語・日本語8語・計12語）と照合し、含有がないことを実装者が事前に確認すること。
- 特にバリデーターの禁止リストに収録されている語句（次フェーズに積み残された脆弱性・分析途中のリスク・後続フェーズへの引き継ぎが確定していない事項などを示す語句）を含む表現は使用しないこと。
- 代替表現として「次フェーズに引き継ぐ問題」「スコープ外として記録する脆弱性」などの禁止語を含まない形式を採用すること。

### 要件2: FR-21 の sessionToken 使用範囲の明示的限定

- ルール23には「取得先: workflow_status のみ」「使用先: workflow_record_test_result のみ」の両方を1文ずつ記述すること。
- 「sessionToken を活用せよ」と読める積極的な表現は避け、「コンテキスト再開時の一時的な同期手段」として位置づける記述にすること。
- ルール23の追記後も FR-4/FR-5/FR-19 で定めたワークフロー制御ツール禁止の優先順位が変わらないことを明確にすること。

### 要件3: 変更後のテスト合格確認義務

- FR-20 の実装後、`workflow-plugin/mcp-server` で `npm test` を実行し、既存の全テスト（945 件相当）が合格することを確認すること。
- スナップショットテストが不一致になった場合は、ガイダンス追記の内容が正しいことを確認した上でスナップショットを更新し、更新後に再度全テストがパスすることを確認すること。
- CLAUDE.md の変更（FR-21）はテストには直接影響しないが、変更後にワークフローの動作確認を実施することが望ましい。

### 要件4: 変更の最小化によるリスク低減

- FR-20 は FR-12 セクションへの追記（または直後への新セクション追加）のみとし、既存テンプレートの削除・全面書き換えは行わないこと。
- FR-21 は CLAUDE.md の「AIへの厳命」末尾への1ルール追記のみとし、他のセクションへの変更は行わないこと。
- 変更ファイルを2ファイル（definitions.ts と CLAUDE.md）に限定することで、予期しない副作用の発生範囲を最小化すること。
