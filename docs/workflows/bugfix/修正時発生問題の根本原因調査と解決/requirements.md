## サマリー

- 目的: 「parallel_verificationバリデーション失敗の根本原因修正」タスク実行中に発生した2つの問題を修正するための要件を定義する。
- 主要な決定事項: 問題1（FR-20相当）はsecurity_scanテンプレートのFR-12セクションに `## 総合評価` 向けの5観点ガイダンスを追加する。問題2（FR-21相当）はCLAUDE.mdの「AIへの厳命」にセッション再開時のsessionToken再取得手順を明記する。
- 次フェーズで必要な情報: 変更対象ファイルは `workflow-plugin/mcp-server/src/phases/definitions.ts`（FR-20）と `CLAUDE.md`（FR-21）の2ファイル。既存FR-11（manual_test総合評価ガイダンス）の5観点リストを参照してsecurity_scan向けに内容を変更することが実装の出発点となる。

## 機能要件

### FR-20: security_scanテンプレートへの総合評価セクションガイダンス追加

`definitions.ts` の security_scan subagentTemplate 内には `## ★ 行数確保の記述指針（FR-12）` セクションが存在する。このセクションは合計20行に達するための計算ロジックを提示しているが、`## 総合評価` セクションに対して5観点を個別に列挙するガイダンスがない。

この不足により、subagentが `## 総合評価` セクションに4行程度の記述を行い、バリデーション失敗（4行 < 5行）が発生するリスクが継続している。

要件の詳細は以下の通りである。

- 変更対象: `workflow-plugin/mcp-server/src/phases/definitions.ts` の security_scan テンプレート
- 追加箇所: FR-12セクション内、または直後に `## ★ 総合評価セクションの記述指針（FR-20）` を新たに追加する
- 追加内容: FR-11（manual_test）と同等の5観点ガイダンスをsecurity_scanに特化した内容で記述する
- 5観点の内容（security_scan向け）:
  - 観点1: 検出された全脆弱性の深刻度別サマリー（Critical/High/Medium/Low件数を数値とともに記述）
  - 観点2: インジェクション・認証バイパス等の高リスク脅威に対する対応状況の評価
  - 観点3: threat-model.mdに記載された脅威と今回のスキャン結果との整合性確認
  - 観点4: 検出問題のうち即時対応が必要なものと後続フェーズへ引き継ぐものの区分
  - 観点5: 全体的なセキュリティ評価（合格・条件付き合格・不合格の判定と根拠）
- 指示文言: 「各観点について1行以上の実質行を記述し、合計5行以上の実質行を `## 総合評価` セクションで確保すること」を明記する
- 参考実装: manual_testの `## ★ 総合評価セクションの記述指針（FR-11）` セクションの構造を踏襲する

### FR-21: CLAUDE.mdへのセッション再開時sessionToken再取得手順の追記

CLAUDE.mdの「AIへの厳命」セクションには現在、セッション（コンテキスト）再開後のsessionToken再取得手順が記載されていない。これによりOrchestratorがコンテキスト再開後にsessionTokenなしで `workflow_complete_sub` 等を呼び出してエラーになるリスクが存在する。

sessionTokenはworkflow-state.jsonに永続保存されており、`workflow_status` のレスポンスに含まれる（FR-18実装済み）。OrchestratorがコンテキストのリセットなしにsessionTokenを再取得できる仕組みはすでに動作している。

要件の詳細は以下の通りである。

- 変更対象: `CLAUDE.md` の「AIへの厳命」セクション
- 追加箇所: 既存のルール一覧の末尾（現在のルール22の後）に新しいルールとして追加する
- 追加内容: 以下の3点を明記する
  - ルール本文: 「セッション再開後は必ず `workflow_status` を呼び出してsessionTokenを再取得すること」
  - 理由の補足: コンテキストウィンドウのリセットによりOrchestratorメモリ上のsessionTokenは消失するが、サーバー側は永続保存しているため `workflow_status` で再取得可能である
  - 適用タイミング: タスクが進行中の状態でコンテキストが中断・再開された場合に実施する
- 記載形式: 他のルールと同じ番号付き箇条書き形式を踏襲し、ルール23として追記する

## 非機能要件

### テスト互換性の維持

`definitions.ts` の変更は既存のテストスイートに影響を与えないこと。具体的には以下の条件を満たす必要がある。

- 既存のスナップショットテストが存在する場合、テンプレート文字列の変更によりスナップショットが更新される点について確認が必要である
- security_scan テンプレートの変更後、`npm test` を `workflow-plugin/mcp-server` ディレクトリで実行し全テストが合格することを確認する
- 新規追加のFR-20セクションは禁止語（英語4語・日本語8語）を含まないこと

### CLAUDE.mdの可読性維持

追加するルール23の記述は以下の品質を維持すること。

- 既存のルール22までと同じ文体・番号付き形式で記述し、一貫性を保つこと
- sessionTokenに関する新規追記は100文字以内の簡潔な文で表現し、AIへの厳命セクションの重厚さを保つこと
- 追記後のAIへの厳命セクションが視覚的に読みやすい構造を維持すること

### 変更の最小化原則

今回の修正は調査で特定された2つの問題のみを対象とすること。

- FR-12セクションの削除や全面書き換えは行わず、5観点ガイダンスの追加（または既存FR-12への内容追記）のみを行う
- CLAUDE.mdのOrchestratorパターンセクションへの追記は任意であり、AIへの厳命セクションへの追記を主要変更として位置づける
- 他のフェーズテンプレート（manual_test等）への変更は本タスクのスコープ外である

### 実装後の検証方法

FR-20とFR-21の実装後、以下の確認を行うことを想定している。

- FR-20: security_scanテンプレートの `## 総合評価` 向けガイダンスが `## ★ 総合評価セクションの記述指針（FR-20）` として存在することをファイル読み込みで確認する
- FR-21: CLAUDE.mdの「AIへの厳命」末尾にルール23が追記されており、`workflow_status` による再取得手順が記載されていることをファイル読み込みで確認する
- リグレッション: `workflow-plugin/mcp-server` で `npm test` を実行し、変更前と同等のテスト合格数を達成していることを確認する
