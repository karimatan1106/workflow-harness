## サマリー

- 目的: 「parallel_verificationバリデーション失敗の根本原因修正」タスク実行中に発生した2つの問題（security_scan総合評価4行不足・sessionToken消失）の根本原因を調査し、再発防止策の方向性を特定する。
- 主要な決定事項: security_scanの `## 総合評価` セクション失敗はFR-12の記述指針が「合計20行の計算ロジック」に留まり各セクションへの具体的な5行要件の割り当てが不明確であることが根本原因。sessionToken消失はコンテキスト再開時にOrchestratorがworkflow_statusを呼び出してsessionTokenを再取得する必要があることを明示したドキュメントが存在しないことが原因。
- 次フェーズで必要な情報: security_scanテンプレートのFR-12セクションに「総合評価」セクションへの5行以上の実質行を明示する指示が必要。Orchestratorガイドとして「セッション再開時にworkflow_statusでsessionTokenを取得すること」の明記が必要。

## 調査結果

### 問題1: security_scan「総合評価」セクション失敗の根本原因

#### バリデーターの動作確認

`artifact-validator.ts` の `checkSectionDensity` 関数は `/^##\s+/m` 正規表現でセクションを分割し、各セクションの実質行数を検証する。`## 総合評価` セクションを含む全ての `##` レベルセクションが個別に評価対象となる。セクション内の `###` サブヘッダー行は `trimmed.startsWith('#')` の条件で構造要素としてカウントされ、実質行数には含まれない。実質行として認められるのはテキスト行・箇条書き行（コロン後にコンテンツを持つ行）・テーブルデータ行であり、空白行・水平線・コードフェンス内行・ヘッダー行は全て構造要素として除外される。最低実質行数はデフォルト5行（`minSubstantiveLines: number = 5`）であり、全フェーズに一律適用される。

#### FR-12の実装内容確認

`definitions.ts` の `security_scan` テンプレート中のFR-12セクション（「## ★ 行数確保の記述指針（FR-12）」）の内容は以下の通りである。

- 「3つの必須セクション（脆弱性スキャン結果・検出された問題・サマリー）にそれぞれ最低5行の実質行が必要であり、合計15行となる」
- 「全体のminLines（20行）に達するには残り5行以上を別のセクションで確保する必要がある」
- 「追加セクション（総合評価等）に5行以上の実質行を記述することで合計20行以上に達することができる」

この記述は「追加セクション（総合評価等）に5行以上」と述べているが、**`## 総合評価` セクション固有の要件として明示的に5行以上必要であることを指示していない**。subagentは「合計20行」という全体目標を意識しながら、総合評価セクションには4行程度の段落を書いた可能性が高い。

#### 比較: FR-11（manual_test）との差異

`manual_test` テンプレートには `## ★ 総合評価セクションの記述指針（FR-11）` として、以下の5観点を各1行以上で記述し合計5行以上を確保することを明示している：
- 全テストシナリオの合否サマリー（実施件数・合格件数・不合格件数を数値とともに）
- 検出された問題の有無・件数・深刻度概要
- 未実施シナリオがある場合の理由と代替措置
- 次フェーズ（security_scan等）への引き継ぎ事項
- 全体的な品質評価（合格・条件付き合格・不合格の判定）

一方、`security_scan` テンプレートには `## 総合評価` セクションに関する同等の指示が存在しない。FR-12は全体行数の計算ロジックとして「追加セクション（総合評価等）に5行以上」と述べているが、5観点を列挙してsubagentを誘導する構造になっていない。

#### 失敗パターンの再現性

現在読んでいるsecurity-scan.mdは**リトライ後の修正済みファイル**であり、`## 総合評価` セクションには5行の実質行が存在する（44, 46, 48, 50, 52行目がいずれも実質行）。失敗した初回ファイルは上書きされているが、バリデーターが報告した「4行 < 5行」というエラーは、初回subagentが `## 総合評価` セクションに4段落（各1行）ではなく、段落の1行がさらに空白で区切られていた等の理由により実質行数が4行になったと考えられる。FR-12の指示が「追加セクションには5行以上」という計算ロジック提示に留まっているため、subagentが `## 総合評価` セクションの実質行数を4行にしてしまうリスクが残っている。

### 問題2: sessionToken消失の根本原因

#### sessionTokenの保存・取得機構

`state/types.ts` の `TaskState` インターフェースに `sessionToken?: string` フィールドが定義されており、`workflow_start` 時に生成されたsessionTokenはworkflow-state.jsonに永続保存される。`tools/status.ts` の `workflow_status` ハンドラーは `...(taskState.sessionToken ? { sessionToken: taskState.sessionToken } : {})` という条件付きスプレッド（FR-18で実装）によりsessionTokenをレスポンスに含める実装になっている。`tools/helpers.ts` の `verifySessionToken` 関数はsessionTokenをOrchestratorのAPIアクセス認証に使用する。

#### 消失のメカニズム

sessionTokenはworkflow-state.jsonに永続保存されているため、**サーバー側では消失しない**。しかしOrchestratorのメモリ（Claude Code本体のコンテキスト）に保持されていたsessionTokenは、**コンテキストウィンドウが更新・再開された際にリセットされる**。これがsessionToken「消失」と認識される原因である。正しい解決方法は、「コンテキスト再開後に `workflow_status` を呼び出してsessionTokenを再取得する」ことであり、これは調査前のタスクで実際に採用された解決策と一致する。

#### Orchestratorガイダンスの欠如

CLAUDE.mdのAIへの厳命一覧や「Orchestratorパターン」セクションに「セッション再開時のsessionToken再取得義務」が明示されていない。これは既知の問題ではあるが、ドキュメント化されていない。工程上のリスクとして、Orchestratorがコンテキスト再開後にsessionTokenなしで `workflow_complete_sub` などを呼び出すとエラーになる可能性がある。

### 問題3: FR-A〜FR-D修正の有効性検証

#### 角括弧検出パターンの実際の動作

`artifact-validator.ts` の `bracketPlaceholderPattern`（正規表現 `/\[#[^\]]{0,50}#\]/g`）はハッシュ記号付きプレースホルダー（`[#xxx#]` 形式）のみを対象としていることを確認した。通常の角括弧（Markdownリンク `[text](url)`・配列アクセス記法 `array[0]`・正規表現の文字クラス表記 `[a-z]`）はこのパターンでは検出されない。この実際の動作と、FR-A〜FR-Dで修正されたテンプレートの説明内容が一致していることを確認した。

## 既存実装の分析

### artifact-validator.tsのセクション密度チェック

`checkSectionDensity` 関数は `## サマリー` を含む全ての `##` セクションに対して均一に最低5行の実質行数要件を適用する。`requiredSections` に含まれるセクションかどうかに関わらず、ファイル中の全ての `##` セクションが評価対象となる。このため、`## 総合評価` のように必須セクションではないが存在する場合も5行の実質行が必要である。

### FR-12と関連するFR-11の比較分析

`manual_test` の FR-11 では「## 総合評価」に対して5観点を箇条書きで列挙し、「各観点について1行以上の実質行を記述し、合計5行以上の実質行を確保すること」と明示している。`security_scan` の FR-12 は全体行数の達成計算ロジックとして「追加セクション（総合評価等）に5行以上」と記述しているが、5観点の箇条書きによるガイダンスが存在しない。このガイダンスの粒度の差が今回のバリデーション失敗の直接的な原因となった。

### セッション再開時のsessionToken取得フロー

`workflow_status` のレスポンスには sessionToken が含まれる（FR-18実装）。Orchestratorはコンテキスト再開時に `workflow_status` を呼び出すことでsessionTokenを再取得できる。この仕組みは正常に機能しているが、CLAUDE.mdの「AIへの厳命」または「Orchestratorパターン」セクションにこの手順が明記されていない点が改善余地として存在する。

## 修正方針

### 修正1: security_scanテンプレートのFR-12改善

`definitions.ts` の security_scan subagentTemplate 内の `## ★ 行数確保の記述指針（FR-12）` セクションに、FR-11（manual_test）と同様の5観点ガイダンスを追加する必要がある。具体的には `## 総合評価` セクションに記述すべき5観点（全脆弱性の深刻度別サマリー・インジェクションリスク評価・設計-実装整合性・推奨対応・総合判定）を列挙し、各観点1行以上・合計5行以上の実質行を確保するよう指示する。

### 修正2: OrchestratorガイドへのsessionToken再取得手順の追記

CLAUDE.mdの「AIへの厳命」または「Orchestratorパターン」セクションに「コンテキスト再開後は必ずworkflow_statusを呼び出してsessionTokenを再取得すること」を追記することで、同様の問題の再発を防止できる。

### 修正3: 検証済み（対応不要）

FR-A〜FR-Dの修正内容は artifact-validator.ts の実際の検出ロジックと一致しており、追加修正は不要である。角括弧プレースホルダーの説明が正確化されており、subagentへの誤指示リスクが低減されている。
