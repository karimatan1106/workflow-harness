- 目的: FR-20（security_scan総合評価ガイダンス追加）とFR-21（sessionToken再取得手順追記）のインターフェース設計を定義し、後続の実装フェーズで参照できる設計書を提供する。
- 評価スコープ: MCPツール（workflow_status等）のコマンドインターフェース、バリデーション失敗時のエラーメッセージ、MCP APIレスポンス形式、definitions.tsのテンプレート文字列設計、CLAUDE.mdのルール設計の5領域を対象とする。

- 検証状況: 本フェーズはUI設計段階であり、実装前のインターフェース定義のみを行う。テスト実施は後続フェーズで行う。

## CLIインターフェース設計

本タスクで変更されるインターフェースはMCPツールとして提供されるコマンド群である。ここでは各ツールの入出力仕様と変更後の期待動作を定義する。

### workflow_status ツール（FR-21で変更されるレスポンス活用方法）

FR-21の実装によりOrchestratorがセッション再開後に呼び出すべきコマンドとして`workflow_status`が明示的に指定される。以下に期待する呼び出し手順と返却情報を定義する。

- **呼び出しタイミング**: コンテキストウィンドウのリセット後または会話の中断・再開後に最初に実行する。
- **呼び出し引数**: なし（引数不要）
- **期待する返却フィールド**: `sessionToken`（FR-18の実装により存在する場合に返却される文字列）、`phase`（現在のフェーズ名）、`taskId`（アクティブなタスクID）
- **sessionToken取得後の処理**: 取得した`sessionToken`をOrchestrator内部変数として保持し、`workflow_complete_sub`・`workflow_record_test_result`等の呼び出し時に引数として使用する。
- **sessionTokenが返却されない場合の処理**: sessionTokenなしのままフェーズ遷移を試みる（エラーが発生する場合は後続の対処手順に従う）。
- **禁止事項**: セッション再開後に`workflow_status`を省略してフェーズ遷移を実行することは禁止とする（sessionToken消失によるエラーが発生するリスクがある）。

### workflow_complete_sub ツール（sessionToken必須フィールド）

- **呼び出し形式**: `workflow_complete_sub(taskId, subPhase, sessionToken)`
- **sessionToken引数の必須性**: FR-21のルール23追記後、sessionTokenは`workflow_status`で再取得した値を必ず渡すこととする。
- **sessionToken省略時の挙動**: エラーが返却される可能性がある。省略する場合は意図的な設計判断であることを確認すること。
- **適用対象フェーズ**: parallel_analysis・parallel_design・parallel_quality・parallel_verificationのいずれかのサブフェーズ完了宣言時に呼び出す。

### npm test コマンド（実装後の検証コマンド）

実装フェーズ完了後のリグレッション確認として、以下のコマンドを実行する設計とする。

- **実行ディレクトリ**: `C:\ツール\Workflow\workflow-plugin\mcp-server`
- **コマンド**: `npm test`
- **期待する出力**: 全テスト合格（前回のベースライン945件以上）
- **失敗時の対処**: エラーメッセージを確認し、FR-20追加テンプレート文字列に禁止語が含まれていないかを検証する。

## エラーメッセージ設計

本タスクが修正するエラー条件と、修正後に期待されるエラーメッセージの変化を定義する。

### FR-20修正前後のバリデーションエラー比較

修正前は`## 総合評価`セクションが4行以下で記述された場合に以下のエラーが発生する。

- **エラー種別**: セクション最低行数未満
- **エラー発生条件**: `## 総合評価`セクションの実質行数が4行以下（5行未満）の場合
- **エラーメッセージ形式**: `セクション '## 総合評価' の実質行数が不足しています（検出: X行、最低: 5行）`
- **発生するフェーズ**: parallel_verificationのsecurity_scanサブフェーズ完了時（workflow_complete_sub呼び出し時）

修正後はFR-20ガイダンスの5観点が示されるため、subagentが各観点につき1行以上を記述することで5行以上が確保される。

### FR-21修正前後のエラー比較

修正前はセッション再開後にsessionTokenなしでフェーズ遷移を試みた場合に以下のエラーが発生する。

- **エラー種別**: 認証エラーまたはセッション不整合
- **エラー発生条件**: sessionTokenが存在するタスク状態でsessionTokenなしの`workflow_complete_sub`を呼び出した場合
- **エラーメッセージ形式**: MCPサーバーが返すエラーレスポンス（詳細はstate-manager.tsの実装による）
- **修正後の対処**: ルール23に従い`workflow_status`を先に呼び出すことでsessionTokenを再取得してから遷移を実施する。

### バリデーター禁止語エラーの防止設計

FR-20で追加するテンプレート文字列には禁止語（英語4語・日本語8語）を含めないよう設計する。

- **チェック対象**: 追加するテンプレート文字列全体
- **検証方法**: npm test実行時のartifact-validator関連テストで確認する。
- **禁止語への言及方法**: テンプレート内で禁止語を直接記述せず、「深刻度別サマリー」「高リスク脅威」「整合性確認」等の具体的なセキュリティ観点の用語を使用する。

## APIレスポンス設計

本タスクで変更・改善されるMCPサーバーのAPIレスポンス仕様を定義する。

### workflow_status レスポンス（FR-18実装済み・FR-21で利用方法を明確化）

FR-18の実装により、以下のフィールドが条件付きで返却されるようになっている。FR-21はこのレスポンスを活用する手順をCLAUDE.mdに追記することで利用方法を明確化する。

- **レスポンスフィールド `sessionToken`**: タスク状態にsessionTokenが存在する場合のみ含まれる文字列型フィールド。値はHMAC-SHA256形式のセッション識別子。
- **含まれる条件**: `taskState.sessionToken` が存在する場合（`...(taskState.sessionToken ? { sessionToken: taskState.sessionToken } : {})`の条件式による）。
- **含まれない条件**: タスクが開始されていない場合またはsessionTokenが発行されていない場合。
- **フィールドの用途**: Orchestratorがコンテキスト再開後にsessionTokenを復元するために使用する。
- **セキュリティ考慮**: sessionTokenはサーバー側でHMAC検証が行われるため、クライアント側での改ざんは検出される。

### workflow_complete_sub レスポンス（sessionToken検証後）

- **成功レスポンス**: サブフェーズが正常に完了した場合のレスポンス。次のサブフェーズへの遷移状態が返却される。
- **失敗レスポンス（sessionToken不一致）**: sessionTokenが一致しない場合はエラーレスポンスが返却される。エラーメッセージには不一致の旨が含まれる。
- **失敗レスポンス（バリデーション失敗）**: 成果物のバリデーション失敗時はエラーメッセージにバリデーション失敗の詳細（不足セクション・行数不足等）が含まれる。

### security_scan サブエージェント起動テンプレートの変更点（FR-20）

FR-20の実装後、security_scan subagentTemplateは以下の構造を持つ。テンプレートを受け取ったsubagentはFR-20のガイダンスに従い`## 総合評価`セクションを記述する。

- **既存セクション（変更なし）**: `## 脆弱性スキャン結果`・`## 検出された問題`・`## ★ 行数確保の記述指針（FR-12）`の各セクション定義。
- **新規追加セクション（FR-20）**: `## ★ 総合評価セクションの記述指針（FR-20）`セクションがFR-12末尾と`## 出力`の間に挿入される。
- **5観点の記述順序**: 1. 深刻度別サマリー、2. 高リスク脅威対応状況、3. threat-model.md整合性確認、4. 即時対応区分、5. 全体評価の順で記述することをガイドする。
- **各観点の最低行数**: それぞれ1行以上の実質行を記述し、合計5行以上を確保する。

## 設定ファイル設計

本タスクで変更される設定ファイル（definitions.ts・CLAUDE.md）のスキーマと変更内容の設計を定義する。

### definitions.ts テンプレート文字列設計（FR-20）

変更対象は`workflow-plugin/mcp-server/src/phases/definitions.ts`内のsecurity_scan subagentTemplateの文字列リテラルである。

- **挿入位置の特定方法**: FR-12セクションの末尾行（`この計算根拠を念頭に置いて各セクションの内容量を決定すること。具体的な行数配分の例として、脆弱性スキャン結果5行＋検出された問題5行＋サマリー5行＋追加コンテンツ5行の合計20行以上が推奨される。`）の直後に改行2つを挟んで挿入する。
- **挿入後の直後**: `## 出力`セクションが続く構造を維持する。
- **文字列内の改行**: TypeScript文字列リテラル内では改行を`\n`で表現する。実際のサブエージェントへのプロンプト表示では改行として展開される。
- **セクション見出しの形式**: `## ★ 総合評価セクションの記述指針（FR-20）`の形式を使用し、他のFR番号付きセクション（FR-12等）と整合性を保つ。
- **5観点の記述内容の設計原則**: 各観点は「何を記述すべきか」と「該当なし時の記述方法」の両方を明示する。subagentが条件分岐なく5行を記述できるように設計する。

### CLAUDE.md ルール設計（FR-21）

変更対象は`C:\ツール\Workflow\CLAUDE.md`の「AIへの厳命」セクション末尾である。

- **追記位置**: 現在のルール22（コアモジュール変更後はMCPサーバーを再起動してから次フェーズに進むこと）の直後に追記する。
- **ルール番号**: 23番として追記する。他のルールと番号が重複しないことを事前に確認する。
- **ルールタイトルの形式**: `23. **セッション再開後は必ず \`workflow_status\` を呼び出してsessionTokenを再取得すること**` の形式を使用し、他のルールの記述スタイルと整合性を保つ。
- **本文の構成**: ルールタイトル1行＋4行のインデント付き補足説明の計5行構成とする。補足説明はsessionToken消失の原因・再取得の手順・省略した場合のリスクの3点を明示する。
- **インデントの形式**: ルール22以前の記述スタイルに合わせて、補足説明行には4スペースのインデントを使用する。
- **バリデーション要件との整合**: 追記内容に禁止語（英語4語・日本語8語）が含まれないことを事前に確認する。「sessionToken再取得」「コンテキストウィンドウのリセット」「フェーズ遷移」等の用語は禁止語に該当しないことを確認済みである。
