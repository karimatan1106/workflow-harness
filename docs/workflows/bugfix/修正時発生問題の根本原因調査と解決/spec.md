## 概要

本タスクは、前回のタスク「parallel_verificationバリデーション失敗の根本原因修正」の実行中に新たに発見された2つの問題を修正する。発見の経緯として、前タスクでsecurity_scanサブエージェントが実行された際、`## 総合評価`セクションが4行しか記述されず、バリデーターの最低5行要件を満たさなかった。本タスクが対処する2問題はいずれも再現性があり、定期的に発生するリスクが確認されていたため、テンプレートとドキュメント両面からの根本対策を実施する。

### 問題1（FR-20相当）: security_scan `## 総合評価` ガイダンス不足

security_scan subagentTemplate にはFR-12（行数確保ガイダンス）が存在し、minLines 20行の達成方法として「追加セクション（総合評価等）に5行以上」と記述している。しかし具体的に何を書けば5行になるかを示す観点別ガイダンスが欠落していた。その結果、subagentが`## 総合評価`セクションを4行程度で終わらせ、バリデーション失敗（minSectionLines 5行未満）が発生するリスクが継続している。

### 問題2（FR-21相当）: コンテキスト再開後のsessionToken再取得手順の欠落

Orchestratorがコンテキストウィンドウのリセットや会話の中断後に作業を再開する場合、メモリ上のsessionTokenが失われる。一方、serverside（workflow-state.json）にはsessionTokenが永続保存されており、FR-18の実装により`workflow_status`のレスポンスで返されるようになっている。にもかかわらず、CLAUDE.mdの「AIへの厳命」にこの再取得手順が記載されていなかったため、Orchestratorが再開後にsessionTokenなしでフェーズ遷移を試みてエラーになるリスクがあった。

両問題に共通する特徴として、バリデーター要件とガイダンスの乖離が根本原因であり、テンプレート文字列を修正することで即座に再発防止が可能である点が挙げられる。
本タスクの変更範囲はdefinitions.tsとCLAUDE.mdの2ファイルに限定されており、artifact-validator.tsのロジックや既存のFR-1〜FR-19の実装には一切影響を与えない。

### FR-20実装手順

変更対象ファイルは `C:\ツール\Workflow\workflow-plugin\mcp-server\src\phases\definitions.ts`。
変更対象箇所はline 918付近のsecurity_scan subagentTemplateの文字列内である。

追加する文字列の構造（TypeScript文字列リテラル内の改行は`\n`で表記）:

```
\n## ★ 総合評価セクションの記述指針（FR-20）\n\n`## 総合評価` セクションには、以下の5観点それぞれについて1行以上の実質行を記述し、合計5行以上の実質行を確保すること。\n検出された全脆弱性の深刻度別サマリーとして、Critical・High・Medium・Lowの各深刻度について検出件数を数値とともに明記した行を記述すること。深刻度の内訳が全てゼロの場合は「全深刻度で検出なし（確認済み依存パッケージ数：xx件）」のように根拠を示す行を記述すること。\nインジェクション・認証バイパス・機密情報露出等の高リスク脅威に対する対応状況の評価として、各脅威カテゴリが対処済みか未対処かを評価した行を記述すること。高リスク脅威が存在しない場合は「インジェクション・認証バイパス等の高リスク脅威は該当なしと判定した理由：（根拠）」と記述すること。\ntheat-model.mdに記載された脅威と今回のスキャン結果との整合性確認として、脅威モデリングで特定された脅威に対してスキャンが網羅的に対応しているかを評価した行を記述すること。脅威モデルがない場合は「脅威モデル未作成のためスキャン範囲を（範囲）に設定した」と記述すること。\n検出された問題のうち即時対応が必要なものと後続フェーズへ引き継ぐものの区分として、緊急度別の仕分け結果を記述した行を含めること。問題が検出されない場合は「即時対応が必要な問題なし・後続フェーズへの引き継ぎ事項なし」の根拠を記述すること。\n全体的なセキュリティ評価として、合格・条件付き合格・不合格のいずれかの判定を行い、その根拠を具体的に記述した行を含めること。条件付き合格の場合は条件内容を明記し、不合格の場合は必須対処事項をリストアップすること。\n各観点について1行以上の実質行を記述し、合計5行以上の実質行を確保すること。\n
```
