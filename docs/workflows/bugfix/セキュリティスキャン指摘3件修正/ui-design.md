# UI設計書

## サマリー

本タスクはワークフローシステムの内部セキュリティ強化を目的としたバックエンド修正であり、ユーザー向けUIの変更は一切含まれない。
修正対象はworkflow-plugin/hooks/配下のbash-whitelist.jsとloop-detector.jsの2ファイルのみである。
SEC-1は環境変数保護リストへの追加、SEC-2はUnicode空白文字判定の強化、SEC-3はシンボリックリンク解決の改善を行う。
これらはCLIツールの内部バリデーションロジックの修正であり、コマンド実行時の挙動変化はエラーメッセージの追加のみである。
ユーザーインターフェース、画面レイアウト、視覚的デザイン、インタラクションパターンへの影響は一切存在しない。

## UI設計

### タスクの性質

本タスクはMCP (Model Context Protocol) サーバーのhookシステムに関するセキュリティ修正である。
workflow-pluginはClaude Codeエディタが実行するコマンドを検証する内部コンポーネントであり、エンドユーザーが直接操作する画面を持たない。
修正内容は環境変数保護、文字判定、パス正規化の3つのロジック改善であり、いずれも内部処理の強化である。
これらの変更はコマンドライン実行時のセキュリティチェックを厳格化するものであり、視覚的要素やユーザーインタラクションは一切関与しない。
したがって、従来の意味でのUI設計（画面、コンポーネント、レイアウト、スタイル）は本タスクの範囲外である。

### 変更対象ファイル

`workflow-plugin/hooks/bash-whitelist.js`はBashコマンドのホワイトリスト検証を実装したNode.jsモジュールである。
このファイルはセキュリティ上保護すべき環境変数のリスト管理と、コマンド文字列の単語境界判定ロジックを含む。
SEC-1による変更はSECURITY_ENV_VARS配列へのSPEC_FIRST_TTL_MS追加であり、既存の7要素に1要素を追加する。
SEC-2による変更はnextChar条件式を等価比較から正規表現判定に置換し、ASCII空白だけでなくUnicode空白も検出可能にする。

`workflow-plugin/hooks/loop-detector.js`は同一ファイルの繰り返し編集を検出するNode.jsモジュールである。
このファイルはファイルパスの正規化処理を通じて、異なる表記で同一ファイルを参照する試行を検出する。
SEC-3による変更はpath.resolveからfs.realpathSyncへの置換であり、シンボリックリンクの実体解決を追加する。
エラー時のフォールバック処理も含めて実装することで、パス解決失敗時の安全性を保つ。

### ユーザーへの影響

通常の開発作業において、本修正はユーザーに一切知覚されない内部改善である。
環境変数SPEC_FIRST_TTL_MSを意図的にexport/unsetしようとした場合のみ、新規にエラーメッセージが表示される。
Unicode空白文字を使用したコマンド回避を試みた場合も、従来通りホワイトリスト違反として検出される。
シンボリックリンクを介して同一ファイルを編集しようとした場合、ループ検出が正しく機能するようになる。

これらはいずれも不正な操作の試行時にのみ影響する動作であり、正常な開発フローでは検知されない。
画面表示、入力フォーム、ボタン、メニュー、ダイアログ等のGUI要素は一切存在しない。
本タスクはCLIツールのバックエンドロジック修正であり、フロントエンド技術（HTML/CSS/JavaScript等）は関与しない。
ユーザー体験 (UX) の変化は「セキュリティが強化された」という定性的な改善のみである。

### UI設計書が不要である理由

CLAUDE.mdのワークフロー定義では全てのタスクがui_designフェーズを通過することが要求される。
しかし本タスクはバックエンドのセキュリティパッチであり、視覚的UI要素の設計対象が存在しない。
parallel_designフェーズではstate_machine、flowchart、ui_designの3サブフェーズが並列実行されるが、本タスクではui_designが空となる。
それでもワークフロープロセスの完全性を保つため、本ドキュメントにて「UI変更なし」を明示的に宣言する。

従来のUI設計書に含まれるべき要素（画面仕様、コンポーネント仕様、インタラクション仕様、レスポンシブ設計等）は全て該当しない。
Storybookストーリー定義、ワイヤーフレーム、モックアップ、アクセシビリティ要件等も本タスクには無関係である。
docs/spec/screens/、docs/spec/components/、docs/spec/interactions/等のディレクトリへの成果物配置も不要である。
本ドキュメントはフェーズ完全性確保のための形式的な記録であり、実質的なUI設計内容は含まれない。

## CLIインターフェース設計

本タスクのCLIインターフェースへの影響は環境変数操作コマンドのブロック追加のみである。
SEC-1修正後はexport SPEC_FIRST_TTL_MSおよびunset SPEC_FIRST_TTL_MSがセキュリティ違反として拒否される。
SEC-2修正後はUnicode空白文字を含むコマンドが正しくホワイトリスト境界判定の対象となる。
SEC-3修正後はシンボリックリンクを経由したファイルパスが実体パスに正規化されてループ検出に使用される。
いずれの変更もCLIの基本的な操作フロー（コマンド入力→検証→実行）には影響しない。

## エラーメッセージ設計

SEC-1ではSPEC_FIRST_TTL_MSへのexport/unset操作時に既存のセキュリティ環境変数保護メッセージが表示される。
このメッセージはbash-whitelist.jsのisCommandAllowed関数が返す既存フォーマットに従い新規メッセージの追加は不要である。
SEC-2では空白文字判定の変更のみでありエラーメッセージへの影響は発生しない。
SEC-3ではfs.realpathSyncのエラーがcatchブロックで捕捉されpath.resolveにフォールバックするためユーザー向けエラーは発生しない。
全3件の修正において新規エラーメッセージの定義や既存エラーメッセージの変更は行わない。

## APIレスポンス設計

本タスクはMCPサーバーのAPI応答には影響しないhookレベルの修正である。
bash-whitelist.jsはPreToolUseイベントに対するhookとして動作しAPIエンドポイントではない。
loop-detector.jsも同様にhookとして動作しMCPプロトコルのリクエスト・レスポンスには関与しない。
hookの実行結果はClaudeエディタ内部で処理されHTTP APIレスポンスとしてユーザーに返されることはない。
したがってAPIレスポンス形式やステータスコードやペイロード構造への変更は本タスクの範囲外である。

## 設定ファイル設計

本タスクでは設定ファイルの変更や新規作成は一切行わない。
SECURITY_ENV_VARS配列はbash-whitelist.js内にハードコードされており外部設定ファイルでの管理は行われていない。
nextChar判定ロジックも同様にコード内に直接実装されており設定による動作変更の仕組みは存在しない。
fs.realpathSync導入もloop-detector.js内の関数変更であり設定パラメータの追加は不要である。
将来的に環境変数保護リストの外部化が検討される可能性があるが本タスクのスコープ外である。

### 次フェーズへの引き継ぎ

test_designフェーズではSEC-1、SEC-2、SEC-3の各修正に対応する検証項目を定義する。
test_implフェーズでは既存テストファイル（src/backend/tests/unit/hooks/配下）への追加テストケース実装を行う。
implementationフェーズではbash-whitelist.jsとloop-detector.jsに対して仕様書通りのコード変更を適用する。
いずれのフェーズもバックエンドのTypeScript/JavaScriptコード修正であり、UI関連作業は発生しない。

本ui-design.mdの主要メッセージは「本タスクはUI変更を伴わないバックエンド修正である」という事実確認である。
この情報により、後続フェーズで誤ってフロントエンドコードを編集する事故を防止できる。
workflow-plugin/hooks/配下の2ファイルのみが編集対象であることを全フェーズで共有する。
セキュリティ修正の性質上、内部ロジック変更に集中し、外部インターフェースへの影響を最小化する方針を維持する。
