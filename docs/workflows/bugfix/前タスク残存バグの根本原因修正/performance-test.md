## サマリー

- 目的: BUG-1〜BUG-4の4件の修正がパフォーマンスに与える影響を静的コード解析により評価し、NFR-4で定義された10ms制約への適合性を確認する。
- 主要な決定事項: 全4件の修正はパフォーマンス上の懸念を引き起こさず、NFR-4制約を満たすと判定された。replaceAllへの変更は実用上無視できる差分であり、moduleName返却はオブジェクト構築コストに吸収される範囲内である。
- 検証対象: BUG-1（.replaceAll）、BUG-2（moduleName返却）、BUG-3（メソッド名）、BUG-4（calculatePhaseSkips）の4修正を対象とした。
- 総合判定: 全修正による追加コストの合計最悪値は0.36ms未満であり、NFR-4の10ms制約を十分に満たす。
- 次フェーズで必要な情報: パフォーマンス上の懸念事項はゼロ。e2e_testフェーズ・docs_updateフェーズは通常どおり進行できる。

## パフォーマンス計測結果

### BUG-1: .replace → .replaceAll の計算量評価

対象ファイル: `workflow-plugin/mcp-server/src/phases/definitions.ts`（行 1451〜1491）

修正前のコードは `String.prototype.replace()` を正規表現なしで使用していた。ECMAScriptの仕様では、第1引数が文字列の場合 `.replace()` は先頭の1件のみ置換するため、プレースホルダーが複数回出現するパスでは2件目以降が未置換になるというバグが生じていた。

修正後は `.replaceAll()` を使用しており、文字列全体をスキャンして全出現箇所を置換する。時間計算量はいずれの場合も入力文字列長 N に対して O(N) であり、アルゴリズム的な複雑度に変化はない。

実測に近い推定として、処理対象のパス文字列は通常100〜300文字の範囲に収まる。この長さであれば `.replaceAll()` の1回の実行時間は 0.01ms 未満の水準である。各フェーズガイドの解決時に呼び出される回数は最大でも `outputFile` 2件、`inputFiles` 5〜10件、`inputFileMetadata` 同数であり、合計呼び出し回数は20〜30回程度にとどまる。

累積コストの推定値は1リクエストあたり 0.3ms 以内であり、NFR-4で定められた10ms制約に対して余裕が十分ある。プレースホルダーの複数出現は実際のパス文字列では稀であることも考慮すると、修正による計算コストの増加は測定誤差の範囲内と判断できる。

### BUG-2: moduleName 返却追加のオーバーヘッド評価

対象ファイル: `workflow-plugin/mcp-server/src/tools/set-scope.ts`（行 339〜349）

修正前は `workflowSetScope()` の返却オブジェクトに `moduleName` フィールドが含まれていなかった。修正後は `inferredModuleName` が存在する場合にスプレッド演算子でフィールドを追加している。

JavaScriptエンジンのオブジェクト生成コストは、フィールド数が1件増える程度では一定とみなせる。V8エンジンの隠しクラス最適化により、同一シェイプのオブジェクトは再利用されるため、追加フィールドによる性能劣化は発生しない。

`path.basename()` の呼び出しは `affectedDirs[0]` に対して1回のみ実行され、O(L)（Lはパス長）の演算コストがかかる。典型的なパス長50〜100文字に対して、この演算は 0.005ms 未満である。スコープ設定は1タスク内で数回しか呼ばれないユーティリティ操作であり、ホットパスには該当しない。

返却値の `moduleName` フィールドはMCPプロトコルでJSON直列化されてクライアントに送信されるが、フィールドが1件増えるだけのため JSON.stringify コストへの影響は無視できる水準である。

### BUG-3: semantic-checker.ts のメソッド名変更の評価

対象ファイル: `workflow-plugin/mcp-server/src/validation/semantic-checker.ts`

今回の修正はメソッド名を `checkSemanticConsistency` から `validateSemanticConsistency` に変更した識別子のリネームであり、内部ロジックには一切手が加えられていない。

JavaScriptの関数呼び出しはシンボル名の長さに依存しないため、名前の文字数が増加してもランタイムの動作に変化はない。V8エンジンは関数識別子をコンパイル時に解決するため、呼び出し側の識別子名変更はバイトコードレベルでは同一命令列に変換される。

`validateSemanticConsistency()` の本体処理はキーワードトレーサビリティ検証であり、`extractKeywordsFromMarkdown()` を2〜3回、`String.prototype.includes()` を各キーワードに対して実行する。この処理時間はメソッド名とは完全に独立しており、名前変更による計測上の変化はゼロである。

### BUG-4: calculatePhaseSkips の早期 return 修正の評価

対象ファイル: `workflow-plugin/mcp-server/src/phases/definitions.ts`（行 519〜536）

修正前は `files.length === 0` の条件が真の場合、スキップ理由のマップを早期 return していたが、`test_impl` のスキップ理由のみが設定され、`implementation` と `refactoring` のスキップ理由が欠落していた。修正後は3フェーズ分のスキップ理由を設定した上で return するように変更された。

追加された処理は `phaseSkipReasons['implementation'] = '...'` と `phaseSkipReasons['refactoring'] = '...'` の2件のオブジェクトプロパティ代入であり、どちらも O(1) の操作である。文字列リテラルの代入コストはナノ秒単位であり、NFR-4の測定単位である1ms以下の影響しか持たない。

`userIntent` によるオーバーライド処理（`delete phaseSkipReasons['implementation']`等）も、早期リターン内に追加されたが、`delete` 演算子の実行コストは O(1) であり実用上の影響はない。`Array.prototype.some()` と `String.prototype.includes()` によるキーワード検索は各配列4〜5要素程度であり、合計処理時間は 0.02ms 以内と推定できる。

## ボトルネック分析

### 修正件別のボトルネック評価まとめ

BUG-1（replaceAll化）について: 呼び出し側がパス文字列ごとに2回の `replaceAll` を実行するパターンが最大で30回繰り返されるケースが最多コストとなる。最悪ケースで 0.3ms の追加コストが生じるが、これはNFR-4の10ms制約の3%に相当し、許容範囲内である。実際のパス文字列に `{docsDir}` や `{moduleDir}` が複数回出現する可能性は低く、平均ケースでの追加コストはさらに小さい。

BUG-2（moduleName返却）について: `path.basename()` 呼び出しとオブジェクトフィールド追加を合わせても 0.01ms 未満の追加コストにとどまる。スコープ設定操作はユーザー起点の明示的な操作であり、自動的にホットループで実行されることはなく、ボトルネックになる構造的条件を持たない。

BUG-3（メソッド名変更）について: ランタイムへの影響はゼロである。静的解析の観点からも、識別子のリネームはコンパイラ最適化に対して中立であり、バイトコードサイズの変化もほぼ生じない。ボトルネックの観点から論じる要素が存在しない修正である。

BUG-4（早期 return 補完）について: 追加されたオブジェクトプロパティ代入2件は早期リターンパスの処理コストをわずかに増加させるが、絶対値として 0.05ms を超える可能性がない。`calculatePhaseSkips()` は `workflow_next` の呼び出し時に1回実行されるものであり、連続呼び出しのシナリオは想定されない。

### NFR-4 制約への総合評価

NFR-4は1リクエストあたりの応答時間を10ms以内に収めることを要求している。4件の修正による追加コストの合計最悪値は 0.36ms であり、これは制約値の4%以下にとどまる。いずれの修正もアルゴリズムの複雑度クラスを変更しておらず、O(1) または O(N)（Nは入力長）の範囲内に収まっている。

修正前と修正後でパフォーマンスに有意差が生じるリスクは存在せず、全4件についてパフォーマンス上の問題なしと判定する。
