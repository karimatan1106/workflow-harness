## サマリー

- 目的: 前タスク「スコープ必須化とドキュメント階層化」のコードレビューで指摘された4件の修正（BUG-1〜BUG-4）について、各変更がシステムの安全性・後方互換性・保守性に与えるリスクを分析し、対策を明確化する。
- 主要な決定事項(1): BUG-2の返却値変更は後方互換性を維持した安全な拡張であり破壊的変更ではない。BUG-3の関数名変更は内部関数リネームであり、呼び出し元の同時変更を完了条件とすることでリスクを排除できる。
- 主要な決定事項(2): BUG-4のスキップメッセージ追加は既存フェーズスキップロジックを変更せずメッセージ付与のみを行うため動作的副作用はない。BUG-1の `replaceAll` 化は実害のない予防的変更だが、TypeScript の `lib` 設定確認が前提条件となる。
- 分析対象ファイル: 修正対象は `definitions.ts`（BUG-1, BUG-4）、`set-scope.ts`（BUG-2）、`semantic-checker.ts`（BUG-3）の3ファイルであり、それぞれに独立したリスクが存在する。
- リスク優先度: BUG-3（関数名変更）とBUG-4（スキップ実行順序）が中リスク、BUG-1（ビルド環境前提）とBUG-2（型定義整合性）が低〜中リスクと評価した。
- 次フェーズで必要な情報: planning フェーズでは各BUGの影響範囲・実装方針・テスト戦略を仕様化する。特に BUG-4 の `userIntent` オーバーライド処理の実行順序（早期リターン前後）と BUG-3 の関数名候補の確定が重要な決定事項となる。

## 脅威分析

### BUG-1: replaceAll 化によるプレースホルダー未置換リスク

現状の `resolvePhaseGuide` 関数では `String.prototype.replace` を文字列リテラル引数で使用している。
ECMAScript 仕様において文字列リテラルを第一引数とした `replace` は、最初にマッチした箇所のみを置換する。
将来的に `PHASE_GUIDES` の定義が拡張され、同一パス文字列内に `{docsDir}` または `{moduleDir}` が2回以上登場した場合、2回目以降のプレースホルダーが未置換のまま残留する。

脅威の発現条件は「PHASE_GUIDES 定義の拡張」という外部要因であり、現時点では実害が生じていない。
しかし未置換プレースホルダーがファイルパスとして使用された場合、ファイルが見つからないエラーや誤ったパスへの書き込みが発生しうる。
このクラスのバグは静的解析では検出できず、テストも既存の定義を前提に書かれるため見落としやすい。

対象コードは `resolvePhaseGuide` 関数内の `outputFile`・`inputFiles`・`inputFileMetadata.path`・サブフェーズの同名フィールドへの計12箇所の `.replace` 呼び出しである（行1432〜1476付近）。

**リスクレベル**: 予防的修正（現在は潜在リスク、定義拡張時に顕在化）

### BUG-2: workflow_set_scope 返却値の moduleName 欠落リスク

`set-scope.ts` の処理では `inferredModuleName` を計算し、TaskState の `scope.moduleName` に正しく書き込んでいる（行333）。
しかし返却値オブジェクト（行339〜348）の `scope` フィールドに `moduleName` が含まれていない。
この欠落により、Orchestrator が `workflow_set_scope` の返却値を参照して確認メッセージを生成する際、推定されたモジュール名を参照できない。

返却値の `moduleName` が存在しない場合でも TaskState は正しく保存されているため、後続フェーズのフェーズガイド解決（`{moduleDir}` 展開）は正常に機能する。
したがって、この問題の影響はOrchestratorの確認ロジックのみに限定される。
ただし、Orchestratorが `scope.moduleName` を使って条件分岐を行うコードが将来追加された場合、誤情報を元にした判定が生じうる。

**リスクレベル**: 低（現時点での機能的影響は確認メッセージの誤表示のみ）

### BUG-3: validateLLMSemanticTraceability 関数名・JSDoc 乖離リスク

`semantic-checker.ts` の `validateLLMSemanticTraceability` 関数は、名称および JSDoc（行199〜211）が「LLMを用いたAPI呼び出し」を示している。
しかし実装の実態はキーワードマッチングによるトレーサビリティ検証であり、Anthropic API は呼び出されない。
SDK の可用性チェック構造（行216〜234）が残存しているが、SDK が利用可能と判定された場合でも実際の動作はキーワード方式である。

この乖離が引き起こす具体的なリスクは2点ある。
第1に、将来の開発者が関数名を見て「LLM呼び出しが実装済み」と誤解し、実際には行われていない重複実装や、LLM検証を前提にしたコードを書く可能性がある。
第2に、実際には呼ばれていない API のコスト計算やレート制限の見積もりが誤った基準で行われうる。

関数名の変更自体は新たなバグを生む変更ではなく、呼び出し元の参照も同時に変更することでコンパイルエラーを防止できる。
重要な実施条件は「`validateLLMSemanticTraceability` を呼び出している全箇所を検索・更新すること」である。

**リスクレベル**: 中（保守性劣化が主要リスク、現時点での動作への影響なし）

### BUG-4: calculatePhaseSkips スキップメッセージ欠落リスク

`definitions.ts` の `calculatePhaseSkips` 関数（行511〜569）では、`files.length === 0` の場合に空の `phaseSkipReasons` をそのまま返している（行518〜521）。
この結果、スコープを設定せずにタスクを進めた場合でも `test_impl`・`implementation`・`refactoring` フェーズが実行されてしまう。

この問題が引き起こす具体的なリスクは、スコープ未設定のまま実装フェーズに進んだ際に、意図しないファイルへの変更が行われることである。
特に `test_impl` フェーズでスコープなしにテストファイルを生成した場合、それらのファイルが誤ったディレクトリに配置されるおそれがある。

修正方針は「スキップ判定ロジック（拡張子チェック・テストパターンマッチング）を変更せず、早期リターン前にメッセージを設定して返す」形式である。
重要な実施条件は、`userIntent` によるスキップオーバーライド処理（行553〜566）が空配列の場合にも適用されるよう、実行順序を維持することである。
早期リターンにスキップメッセージを付与した後、`userIntent` チェックを経由してから返すことで、ユーザーの明示指示が優先される設計が維持される。

**リスクレベル**: 中（フェーズスキップ制御の誤動作が主要リスク）

## リスク

### リスク1: BUG-1 の TypeScript 環境前提条件

`replaceAll` メソッドは ES2021 以降の仕様であり、TypeScript プロジェクトの `tsconfig.json` の `lib` 設定が ES2021 以上でない場合、コンパイルエラーが発生する。
要件定義（行172）では「`lib` 設定が `ES2021` 以上であることを前提とする」と明記されているが、実際の `tsconfig.json` の設定を事前に確認する必要がある。

対策として、`test_impl` フェーズのテスト作成前に `tsconfig.json` の `lib` 値を確認し、ES2021 未満の場合は正規表現を用いた代替実装（`replace(/\{docsDir\}/g, docsDir)`）を選択肢として用意することを推奨する。

### リスク2: BUG-3 の関数名変更における参照箇所の網羅的確認

`validateLLMSemanticTraceability` を呼び出している箇所は `semantic-checker.ts` 内部の `validateSemanticConsistency` 関数のみと見られるが、テストファイルや他モジュールにも参照が存在する可能性がある。
リネームを実施する前に、プロジェクト全体に対してグレップ検索を実施し、全参照箇所を特定することが前提条件である。

対策として、`implementation` フェーズでは関数定義の変更とともに `grep -r validateLLMSemanticTraceability` の結果に基づいた全参照箇所の更新を同一コミットで完了させること。

### リスク3: BUG-4 の userIntent オーバーライド処理の実行順序

現行コードでは `files.length === 0` の場合に早期リターンしており、`userIntent` チェック（行553〜566）が実行されない。
修正後は「空配列の場合のスキップメッセージ付与」と「`userIntent` オーバーライド」を両方経由した上で返す必要がある。
実装上の誤りとして、スキップメッセージ付与後に `userIntent` チェックをバイパスして即座に `return` してしまうパターンが考えられる。

対策として、`test_impl` フェーズでは「空配列 かつ `userIntent` に実装関連キーワード含む」場合に `implementation` スキップが除去されることを検証するテストケースを必須とする。

### リスク4: BUG-2 の返却値型定義の整合性

`set-scope.ts` の返却値型が `ToolResult` または同等の型定義として宣言されている場合、`moduleName` フィールドを追加する際に型定義の更新が必要になりうる。
現行の型定義で `scope` フィールドの型が `{ affectedFiles: string[]; affectedDirs: string[] }` と固定されている場合、`moduleName?: string` の追加で TypeScript の型エラーが発生する。

対策として、返却値の型定義を確認し、`scope` フィールドに `moduleName?: string` を追加するか、型アサーションを使用して互換性を保つことを `implementation` フェーズで確認する。

## 対策

### BUG-1 への対策

`resolvePhaseGuide` 関数内の全 `.replace('{docsDir}', docsDir)` を `.replaceAll('{docsDir}', docsDir)` に、全 `.replace('{moduleDir}', moduleDir)` を `.replaceAll('{moduleDir}', moduleDir)` に変更する。
変更前に `tsconfig.json` の `lib` 設定を確認し、ES2021 未満の場合は `replace(/\{docsDir\}/g, docsDir)` 形式を使用する。
テストでは単一パス文字列内に `{docsDir}` が2回登場するケースで両方が置換されることを確認する。

### BUG-2 への対策

`set-scope.ts` 行339〜348の返却値オブジェクトの `scope` フィールドに `moduleName: inferredModuleName` を追加する。
TaskState の書き込みロジックには変更を加えない。
型定義に `scope.moduleName` が含まれていない場合は型定義を拡張する。
テストでは `dirs` を指定した場合の返却値に `scope.moduleName` が存在することを確認する。

### BUG-3 への対策

関数名を `validateKeywordSemanticTraceability` に変更し、JSDoc を実際の動作（キーワードマッチング・フォールバック方式）に沿った内容に書き直す。
変更前にプロジェクト全体で旧関数名の参照箇所を検索し、全件を同一フェーズで更新する。
テストでは関数の呼び出しが正常に機能することと、旧名称の参照が残っていないことを確認する。

### BUG-4 への対策

`calculatePhaseSkips` 関数の早期リターン部分（行518〜521）を修正し、空配列の場合に3件のスキップ理由を `phaseSkipReasons` に設定してから `userIntent` オーバーライド処理を経由して返す。
`userIntent` チェックのコード（行553〜566）を早期リターン前に実行されるよう順序を調整する。
テストでは「空配列 かつ `userIntent` なし」「空配列 かつ実装キーワードあり」「空配列 かつテストキーワードあり」「1件以上のファイルあり」の4ケースで返却値を検証する。
