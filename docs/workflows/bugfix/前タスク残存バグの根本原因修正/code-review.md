## サマリー

本コードレビューは、前タスク「スコープ必須化とドキュメント階層化」で指摘された4件の残存バグ（BUG-1〜BUG-4）の修正が仕様書（spec.md）の受け入れ基準を満たしているか検証した結果をまとめる。
レビュー対象のソースコードファイルは `definitions.ts`（BUG-1・BUG-4）、`set-scope.ts`（BUG-2）、`semantic-checker.ts`（BUG-3）の3ファイルである。
全4件のバグ修正について、仕様書の受け入れ基準との整合性を確認した結果、全て正常に実装されていることを確認した。
設計書に記載のない余分な変更は存在せず、後方互換性も維持されている。
コード品質・セキュリティ・パフォーマンスいずれの観点からも問題となる実装は検出されなかった。

## 設計-実装整合性

### BUG-1: resolvePhaseGuide 内の replace をグローバル置換に変更

仕様書の受け入れ基準は「関数内の全ての文字列 `.replace('{docsDir}'` が `.replaceAll('{docsDir}'` に変更されていること」および「`.replace('{moduleDir}'` が `.replaceAll('{moduleDir}'` に変更されていること」と定義されている。
実装ファイル `definitions.ts` の行1451〜1492を精査した結果、メインフィールド処理エリア（行1450〜1465）とサブフェーズ処理エリア（行1477〜1492）の全ての置換箇所で `replaceAll` が使用されていることを確認した。
具体的には `resolved.outputFile`、`resolved.inputFiles`、`resolved.inputFileMetadata`、`subResolved.outputFile`、`subResolved.inputFiles`、`subResolved.inputFileMetadata` の全フィールドで `replaceAll('{docsDir}', docsDir).replaceAll('{moduleDir}', moduleDir)` の連鎖置換が適用されている。
設計書が「8箇所」と述べた変更箇所はリファクタリングにより統合されており、実際のコードでは連鎖メソッドとして各フィールドで2回のreplaceAllが呼ばれる形に整理されている。
この整理は機能的に同等であり、仕様の受け入れ基準を満たしている。

### BUG-2: workflow_set_scope 返却値への moduleName 追加

仕様書の受け入れ基準は「`workflow_set_scope` の返却値オブジェクトの `scope` フィールドに `moduleName` キーが存在すること」および「`inferredModuleName` が `undefined` の場合はスプレッド省略されること」と定義されている。
`set-scope.ts` の行342〜346を精査した結果、`scope` オブジェクトリテラルにスプレッド演算子を使用した条件付き追加が実装されていることを確認した。
具体的には `...(inferredModuleName !== undefined && { moduleName: inferredModuleName })` の形式で `moduleName` が追加されており、`undefined` の場合はスプレッドが空オブジェクトになるため `moduleName` フィールド自体が含まれない設計になっている。
加えて行347の `message` フィールドにも `inferredModuleName` が存在する場合にモジュール名が末尾に追記される条件式が実装されており、確認メッセージの完全性も向上している。
`TaskState` への書き込みロジック（行327〜335）は変更されておらず、受け入れ基準の「`TaskState` への書き込みロジックは一切変更しないこと」も満たされている。

### BUG-3: validateLLMSemanticTraceability の関数名・JSDoc リネーム

仕様書の受け入れ基準は「関数名から `LLM` という表現が除去されていること」「変更後の JSDoc に『LLM を使う』または『API を呼び出す』という記述が含まれないこと」と定義されている。
`semantic-checker.ts` の行190〜272を精査した結果、インターフェース名が `KeywordSemanticTraceabilityResult`（行190）、関数名が `validateKeywordSemanticTraceability`（行214）に変更されていることを確認した。
JSDoc（行199〜213）の内容は「キーワードマッチング方式による検証であること」「LLM APIの呼び出しは行わないこと」「SDK可用性チェックは将来のLLM統合への拡張ポイントであること」を明示する内容に書き直されており、受け入れ基準を満たしている。
ファイル全体の参照箇所をGrepツールで検索した結果、旧名称 `validateLLMSemanticTraceability` および `LLMSemanticTraceabilityResult` は `semantic-checker.ts` 内に一切残存せず、新名称のみが使用されていることを確認した。
外部ファイルへの参照も存在しないため、変更による影響範囲は `semantic-checker.ts` 単体に限定されている。

### BUG-4: calculatePhaseSkips でスコープ空配列時のスキップ理由メッセージを追加

仕様書の受け入れ基準は「`scope.affectedFiles` と `scope.files` がどちらも空配列の場合、返却オブジェクトに3キーが含まれること」および「`userIntent` によるオーバーライドが早期リターン内でも適用されること」と定義されている。
`definitions.ts` の行519〜537を精査した結果、早期リターンブロック内でスキップ理由が3件設定されており、その後 `userIntent` によるオーバーライド処理が実行されることを確認した。
`phaseSkipReasons['test_impl']` には「スコープが未設定のためテスト実装フェーズをスキップ」、`phaseSkipReasons['implementation']` には「スコープが未設定のため実装フェーズをスキップ」、`phaseSkipReasons['refactoring']` には「スコープが未設定のためリファクタリングフェーズをスキップ」が設定されている。
`userIntent` によるオーバーライド処理は `TEST_KEYWORDS` と `IMPL_KEYWORDS` を使ったキーワード判定として実装され、テスト関連キーワードがある場合は `test_impl` のスキップが削除され、実装関連キーワードがある場合は `implementation` と `refactoring` のスキップが削除される。
設計書が「前タスクの FR-1-4 要件の実装漏れ」と指摘した部分が正確に補完されており、受け入れ基準を全て満たしている。

## コード品質

### 設計方針との整合性

4件全ての修正は「既存の外部インターフェース（関数シグネチャ・TaskState 形式）を破壊しない後方互換の形で実施する」という設計方針に従っている。
BUG-2の返却値追加はスプレッド演算子による条件付き追加のため、`moduleName` が推定できない呼び出しでは従来と同じ2フィールド構造が返却され、既存の呼び出し元を破壊しない。
BUG-3の関数名変更は `semantic-checker.ts` 単体の変更であり、Grep検索で外部参照がないことを確認済みのため影響範囲が制限されている。
BUG-4のスキップ理由メッセージ追加は、以前は空オブジェクトを返していた早期リターンに情報を付与するものであり、呼び出し元が返却値を利用していなければ挙動の変化はなく、後方互換性を維持している。

### コードスタイル

BUG-1の修正ではメソッドの連鎖（`.replaceAll(...).replaceAll(...)`）が全フィールドで統一されており、元の `replace` との一貫したパターンを維持している。
BUG-2の `...(condition && { field: value })` パターンはTypeScriptにおける条件付きフィールド追加の標準的な慣用表現であり、可読性が高い。
BUG-4の実装では `delete phaseSkipReasons['test_impl']` によるオーバーライドが採用されており、既存の行523以降のロジックと一貫したアプローチを使用している点は保守性の観点で適切である。

### 設計書にない余分な変更

設計書の変更対象として定義された3ファイル以外のソースコードへの変更は確認されなかった。
また、設計書に記載のない新機能の追加や副作用をもたらす変更も検出されなかった。

## セキュリティ

### 入力値の扱い

`resolvePhaseGuide` の `replaceAll` 変更（BUG-1）は文字列リテラル引数を使用しているため、正規表現の特殊文字問題は発生せず、セキュリティリスクはない。
`set-scope.ts` の `moduleName` 追加（BUG-2）では推定済みの変数を返却するだけであり、外部入力を直接露出させる処理ではなく、インジェクションリスクは存在しない。

### 認証・認可への影響

BUG-1〜BUG-4の修正は認証・認可フローに関与するコードを変更しておらず、既存のセキュリティ境界に影響しない。
`semantic-checker.ts` の関数名変更（BUG-3）は署名や実行ロジックを変更しないため、セキュリティ特性に変化はない。

### 情報漏洩リスク

`set-scope.ts` の `message` フィールドに追加されたモジュール名情報は、ツール呼び出し側がスコープ設定内容を確認するための情報であり、外部公開されるAPIではないため情報漏洩リスクは低い。
`phaseSkipReasons` に追加されたスキップ理由メッセージ（BUG-4）は内部ツールのレスポンスとして返却されるオーケストレーター向け情報であり、外部に公開される形式ではない。

## パフォーマンス

### replaceAll の性能特性

BUG-1の `replace` から `replaceAll` への変更は文字列リテラル引数を使用しているため、正規表現オブジェクトの生成オーバーヘッドは発生しない。
対象文字列（ファイルパス）は通常数十〜数百文字程度であり、複数回の置換処理であっても実行時間への影響は無視できるレベルである。
`resolvePhaseGuide` はフェーズ遷移時に一度呼び出される関数であり、ホットパスではないため、パフォーマンスへの実質的な影響はない。

### 計算量への影響

BUG-4の早期リターンブロックに追加されたオーバーライド処理（行525〜534）は、`TEST_KEYWORDS` と `IMPL_KEYWORDS` の配列を1回ずつ走査するため線形時間の処理が追加されている。
ただし両配列の要素数は固定の少数（最大10件程度）であり、`userIntent` 文字列の長さも実用的な範囲内に収まるため、パフォーマンス上の問題はない。
BUG-2・BUG-3の変更はオブジェクトフィールドの追加と関数名変更のみであり、計算量に影響を与えない。
