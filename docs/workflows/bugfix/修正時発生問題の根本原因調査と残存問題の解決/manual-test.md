## テストシナリオ

### シナリオ1: security_scan テンプレートへの NG/OK ガイダンス追記の確認

確認対象ファイルは `workflow-plugin/mcp-server/src/phases/definitions.ts`（行 908〜918）の `security_scan.subagentTemplate` フィールドである。
今回の修正意図は、サブエージェントが評価結論フレーズ（「評価結果: リスクなし」等）を 3 回以上同じ文言で繰り返してバリデーション失敗する問題を防ぐため、具体的な NG 例と OK 例をテンプレートに追記することだった。

検証観点は以下の 3 点である。
まず、NG パターンを示す具体例（3 回以上の同一行になるケース）がテンプレート内に記述されているかを確認する。
次に、OK パターンを示す具体例（各評価行に固有識別子を付けて行を一意にするケース）がテンプレート内に記述されているかを確認する。
最後に、追記内容がコードフェンス外であることを確認する（禁止語の角括弧プレースホルダーに触れていないか）。

前提条件として、ファイルが正常に読み取り可能であること、および `security_scan.subagentTemplate` 内にガイダンス文が存在することを確認した。

### シナリオ2: workflow_status レスポンスの 3 フィールド除外実装の確認

確認対象ファイルは `workflow-plugin/mcp-server/src/tools/status.ts`（行 121〜143）のフェーズガイド処理部分である。
今回の修正意図は `workflow_status` の返すレスポンスが大容量になりすぎる問題を解決するため、`subagentTemplate`・`content`・`claudeMdSections` の 3 フィールドをレスポンスから除外することだった。

検証観点は以下の 4 点である。
最初に、シャロウコピーされた `slimGuide` オブジェクトから 3 フィールドが `delete` されているかを確認する。
次に、`subPhases` を持つ並列フェーズの場合、各サブフェーズオブジェクトからも同じ 3 フィールドが除外されているかを確認する。
さらに、`workflow_next`（next.ts）では `resolvePhaseGuide` の返り値がそのまま全フィールドを含む形でレスポンスに渡されていることを確認して、後方互換性が維持されているかを確認する。
最後に、コメント文が修正意図（status はスリム化、next は全フィールド維持）を説明していることを確認する。

## テスト結果

### シナリオ1（security_scan テンプレート NG/OK ガイダンス追記）の手動検証結果

definitions.ts の `security_scan.subagentTemplate`（行 918）を読み取り、以下の内容が正しく追記されていることを確認した。

テンプレート本文には「重複行回避の注意事項」セクションが追記されており、複数の評価対象を同一フォーマットで評価する場合に 3 回以上の同一行になることへの警告文が含まれている。この記述はサブエージェントがバリデーション失敗を回避するための実質的なガイダンスとして機能する。

NG 例として「`- セキュリティリスク: 問題なし`」を FR-A・FR-B・FR-C で繰り返すパターンが明示されており、同一行が 3 回以上出現するとエラーになる旨が説明されている。これはアーティファクトバリデーターの検出ロジックと整合した記述である。

OK 例として FR-A・FR-B をそれぞれ固有識別子（「FR-A（state_machine 定義）」「FR-B（flowchart 定義）」等）を含む形式で記述する例が明示されており、各行を一意にする方法が具体的に示されている。

さらに評価結論フレーズに特化した注意事項として、「`- 評価結果: リスクなし`」を BUG-1・BUG-2・BUG-3 で繰り返す NG 例と、BUG 番号・ファイル名・変更内容の概要を行に含める OK 例の両方が記述されている。これにより今回の修正のような複数バグ修正を扱う際に有効なガイダンスとなっている。

シナリオ1の検証結果: 仕様どおりのガイダンスが追記されており、意図した問題（評価結論フレーズの 3 回以上の同一行繰り返し）を防止するための具体的な NG/OK 例が存在することを確認した。静的検証として合格と判定する。

### シナリオ2（workflow_status 3 フィールド除外）の手動検証結果

status.ts の行 125〜141 を読み取り、以下の実装が正しく行われていることを確認した。

行 127 で `phaseGuide` のシャロウコピー `slimGuide` が作成されており、元のオブジェクトを破壊しないように設計されていることを確認した。これは `workflow_next` での再利用を可能にするための重要な設計判断である。

行 128〜130 で `slimGuide['subagentTemplate']`・`slimGuide['content']`・`slimGuide['claudeMdSections']` の 3 フィールドが `delete` されていることを確認した。フィールド名がそれぞれ正確に記述されており、修正仕様と一致している。

行 131〜139 の分岐で、`subPhases` キーが存在して値がオブジェクト型である場合に `Object.values()` でサブフェーズオブジェクトを列挙し、各サブフェーズについても同一の 3 フィールドを `delete` する処理が実装されていることを確認した。これにより parallel_verification 等の並列フェーズでもサブフェーズの大容量フィールドが除外される。

行 125 のコメント文「workflow_status ではサイズの大きなフィールドを除外してレスポンスを削減する / workflow_next には引き続き全フィールドを含めることで後方互換性を維持する」が修正意図を正確に説明していることを確認した。

シナリオ2の検証結果: 3 フィールド除外の実装が仕様どおりに行われており、並列フェーズのサブフェーズについても再帰的に除外処理が適用されていること、および workflow_next 側の後方互換性が維持されていることを確認した。静的検証として合格と判定する。
