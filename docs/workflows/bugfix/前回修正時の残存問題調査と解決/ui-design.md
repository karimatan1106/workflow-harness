## サマリー

本ドキュメントは「前回修正時の残存問題調査と解決」タスクにおけるUI設計書である。
本タスクはWebUIを持たないCLI操作ベースの整理作業であり、設計対象はコマンドラインインターフェース、エラーメッセージ、MCPワークフローAPIレスポンス、および設定ファイルの仕様に限定される。
対象の中核は `workflow-plugin/mcp-server/src/verify-sync.test.ts`（git未追跡の誤配置ファイル826行）の削除操作である。

### 目的

本設計書の目的は、実装フェーズのsubagentがCLI操作を迷いなく実行できるよう、コマンドの引数・出力形式・エラー対処・APIレスポンス・設定ファイルの仕様を明文化することである。
設計書の提供により、実行者がアドホックにコマンドを組み立てることなく、事前定義済みの手順に沿って安全かつ再現可能な作業を行える環境を整備する。

### 主要な決定事項

削除コマンドとして `rm` を採用し `git rm` は使用しない。理由は対象ファイルがgit未追跡であるため、git rmを使用すると不要な履歴エントリが生成されるリスクがあるためである。
削除後の確認手段として `ls` によるディレクトリ一覧確認と `git -C workflow-plugin status --short` によるgit状態確認の2段階を採用する。
テスト実行は `npm test` を使用し、失敗件数ゼロをもって作業完了の判断基準とする。

### 次フェーズで必要な情報

削除コマンドの絶対パス: `C:\ツール\Workflow\workflow-plugin/mcp-server/src/verify-sync.test.ts`
git状態確認コマンド: `git -C "C:\ツール\Workflow\workflow-plugin" status --short`
テスト実行コマンド: `cd "C:\ツール\Workflow\workflow-plugin/mcp-server" && npm test`
テスト基準値: 失敗件数ゼロ、総数は897件前後（変動があっても失敗件数ゼロなら合格）

## CLIインターフェース設計

本タスクで使用するコマンドは4種類である。それぞれの役割、引数、期待出力、エラー時の対処を以下に定義する。

### コマンド1: ls（事前確認）

コマンド: `ls "C:\ツール\Workflow\workflow-plugin/mcp-server/src/"`

目的は削除対象ファイルが存在することを確認し、誤ったパスへの操作を防ぐことである。
期待される出力には `verify-sync.test.ts` が含まれること、および `__tests__/` ディレクトリが存在することを確認する。
`verify-sync.test.ts` が出力に現れない場合は、既に削除済みであるか、パスの指定に誤りがある可能性を示す。
その場合はパスをダブルチェックし、正しいパスを再度確認してから次のステップに進む。

### コマンド2: rm（削除実行）

コマンド: `rm "C:\ツール\Workflow\workflow-plugin/mcp-server/src/verify-sync.test.ts"`

目的は誤配置ファイルをディスクから完全に削除することである。
このコマンドは正常終了時に何も出力しない（UNIX慣例に従い、エラーがなければ無出力）。
権限エラーが発生した場合は「エラーメッセージ設計」セクションの対処手順に従う。
削除は元に戻せないため、コマンド1の確認完了後にのみ実行すること。

### コマンド3: ls（事後確認）

コマンド: `ls "C:\ツール\Workflow\workflow-plugin/mcp-server/src/"`

目的は削除後にファイルが存在しないことを確認し、`__tests__/` ディレクトリと正規ファイルへの影響がないことを検証することである。
期待される出力に `verify-sync.test.ts` が含まれないこと、`__tests__/` ディレクトリが引き続き存在することを確認する。
事後確認で `verify-sync.test.ts` が表示された場合は、削除コマンドが正しいパスを指定していなかった可能性を調査する。

### コマンド4: git status（git状態確認）

コマンド: `git -C "C:\ツール\Workflow\workflow-plugin" status --short`

目的はサブモジュール内のgit追跡状態が削除前後で変化していないことを確認することである。
誤配置ファイルはgit未追跡であったため、削除後の出力は空、またはそのファイルに関するエントリを含まないことが期待される。
未追跡の削除操作であれば `??` や `D` のエントリは現れない。
予期しないエントリが表示された場合は、本タスクのスコープ外の変更が混入していないかを確認し、必要に応じてOrchestratorへ報告する。

### コマンド5: npm test（テスト実行）

コマンド: `cd "C:\ツール\Workflow\workflow-plugin/mcp-server" && npm test`

目的はテストスイート全体を実行し、削除による回帰が生じていないことを確認することである。
テスト実行はtestingフェーズで行われ、implementationフェーズでは実行しない。
期待される出力はテスト総数897件前後、失敗件数ゼロである。
テスト総数が変動した場合は変動前後の件数を記録し、変動が誤配置ファイルのグロブ除外によるものかを確認する。

## エラーメッセージ設計

本セクションでは各コマンド実行時に発生し得るエラーのメッセージパターンと、それぞれの対処手順を定義する。

### ファイルが見つからない場合

発生タイミング: `rm` コマンドまたは事前 `ls` 実行時
エラー出力の例: `rm: cannot remove '...verify-sync.test.ts': No such file or directory`
意味: 対象ファイルが指定パスに存在しない。すでに削除済みか、パスの記述に誤りがある可能性を示す。
対処手順: `ls` コマンドでディレクトリ内容を確認し、ファイルが削除済みであれば次のステップ（git status確認）に進む。パスの誤りが疑われる場合は、スペース・スラッシュの方向・大文字小文字を確認してから再実行する。

### 削除権限エラーが発生した場合

発生タイミング: `rm` コマンド実行時
エラー出力の例: `rm: cannot remove '...verify-sync.test.ts': Permission denied`
意味: 現在のプロセスがそのファイルを削除する権限を持っていない。Windows環境では読み取り専用属性が設定されている場合がある。
対処手順: `ls -la` で当該ファイルのパーミッションを確認する。Windows環境では読み取り専用属性の解除が必要な場合がある。また、別のプロセスがファイルをロックしている可能性もあるため、他のプロセスを終了してから再試行する。
本対処で解決しない場合はOrchestratorに報告し、手動介入を依頼する。

### テスト失敗が検出された場合

発生タイミング: `npm test` の実行完了時
エラー出力の形式: `FAIL {テストファイル名}` に続き失敗したテストケースの詳細が表示される。
意味: 削除操作が既存テストの依存関係に影響した可能性を示す。または削除操作とは無関係な既存バグが顕在化した可能性もある。
対処手順: 失敗したテストファイルを確認し、削除した `verify-sync.test.ts` との依存関係が存在するかを調査する。依存関係がない場合は本タスクのスコープ外の問題として `workflow_record_known_bug` に記録する。依存関係が確認された場合はOrchestratorに報告し、対応方針を再検討する。

### git status で予期しないエントリが表示された場合

発生タイミング: `git -C workflow-plugin status --short` 実行時
エラー出力の形式: `D mcp-server/src/verify-sync.test.ts` や `?? mcp-server/src/verify-sync.test.ts` などのエントリ
意味: gitがファイルの状態を追跡している場合、削除操作がgit追跡状態に影響した可能性を示す。ただし、このファイルはgit未追跡として調査済みであるため、このエントリが表示された場合は事前調査の誤りを疑う必要がある。
対処手順: `git -C workflow-plugin ls-files --others workflow-plugin/mcp-server/src/verify-sync.test.ts` コマンドでgit追跡状態を再確認し、Orchestratorに状況を報告する。

## APIレスポンス設計

本セクションでは、ワークフロー進行に使用するMCP workflow APIのレスポンス形式を定義する。
本タスクで関連するAPIはフェーズ遷移を管理する `workflow_next` と成果物バリデーションを行う各フェーズ完了トリガーである。

### workflow_next 正常応答

フェーズ遷移が成功した場合、MCPサーバーは以下の形式でレスポンスを返す。

レスポンスのフィールドとして `success: true` が含まれ、`nextPhase` フィールドに遷移先フェーズ名（例: `implementation`, `testing`）が含まれる。
`taskId` フィールドにはタスクの識別子が含まれ、`currentPhase` フィールドには遷移後のフェーズ名が格納される。
このレスポンスを受けた後、Orchestratorは次フェーズのsubagentを起動する。

### workflow_next 失敗応答（バリデーションエラー）

成果物バリデーションに失敗した場合、MCPサーバーは以下の形式でエラーレスポンスを返す。

レスポンスのフィールドとして `success: false` が含まれ、`error` フィールドにエラーメッセージが格納される。
`validationErrors` フィールドには具体的なバリデーション失敗理由（セクション密度不足、禁止語検出、同一行重複など）が配列形式で格納される。
このレスポンスを受けたOrchestratorは、エラー内容をsubagentに伝えてリトライを実施する。

### workflow_complete_sub 正常応答

並列フェーズのサブフェーズ完了時には `workflow_complete_sub` APIを呼び出す。
正常応答として `success: true` と `completedSubPhase` フィールドが返される。
全サブフェーズが完了するまで次の `workflow_next` への呼び出しはブロックされる。

### workflow_next 失敗応答（フェーズ順序エラー）

前提条件を満たさないフェーズへの遷移を試みた場合、`phaseOrderError` フィールドが含まれるエラーレスポンスが返される。
このエラーはサブフェーズの完了を待たずに `workflow_next` を呼び出した場合などに発生する。
対処としてはサブフェーズを全て完了させてから `workflow_next` を再度呼び出す。

## 設定ファイル設計

本セクションでは、本タスクの実行に関係する設定ファイルの仕様と注意事項を定義する。

### vitest.config.ts のグロブパターン設計

ファイルパス: `workflow-plugin/mcp-server/vitest.config.ts`
本タスクで特に注意が必要な点として、vitest.config.ts のテストファイルのグロブパターンが `src/**/*.test.ts` のような広範なパターンを使用していた場合、誤配置ファイル（`src/verify-sync.test.ts`）がテスト実行対象に含まれていた可能性がある。
削除後にテスト総数が減少した場合は、このグロブパターンに起因するものとして解釈する。
理想的なグロブパターンは `src/__tests__/**/*.test.ts` のように `__tests__/` ディレクトリに限定した指定であるが、パターンの変更は本タスクのスコープ外であり、変更は行わない。
現在のグロブパターンの確認は `ls` コマンドやRead toolで vitest.config.ts を参照することで実施可能である。

### workflow-state.json の状態遷移設定

ファイルパス: `.claude/state/workflows/{taskId}_{taskName}/workflow-state.json`
本ファイルはHMAC-SHA256による整合性保護が施されており、直接編集するとフックがエラーを返す構造になっている。
状態遷移はMCPサーバーの `workflow_next`・`workflow_complete_sub` APIを通じてのみ行うべきであり、ファイルを直接編集することは禁止されている。
現在のフェーズ確認には `workflow_status` APIを使用する。フェーズが期待と異なる場合はMCPサーバーのログを確認し、HMAC整合性エラーが発生していないかを調べる。
HMAC整合性エラーが発生した場合の根本対処は、task-index.json のキャッシュをクリアするか、MCPサーバーを再起動して状態を再読み込みさせることである。

### task-index.json のキャッシュ設計

ファイルパス: `.claude/state/task-index.json`
このファイルはタスクのフェーズ情報を1時間のTTLでキャッシュしており、MCPサーバーがフェーズを遷移させてもキャッシュが更新されない既知の問題が存在する。
フックが古いフェーズを参照してBashコマンドをブロックする症状が発生した場合は、task-index.json の対象タスクの `phase` フィールドを現在のフェーズに合わせて手動更新することで回避できる。
このファイルにはHMAC保護がないため直接編集が可能である。ただし編集後はフックが次回キャッシュ期限切れ時に上書きするため、永続的な変更とはならない点に注意する。

### .gitignore の設定確認

ファイルパス: `.gitignore`（プロジェクトルート）
`docs/workflows/` が `.gitignore` に登録されていることを前提としており、本タスクで作成するワークフロー成果物（research.md, spec.md, ui-design.md 等）はGitで追跡されない。
本タスクの作業はサブモジュール `workflow-plugin` 内のファイル削除のみであるため、メインリポジトリへのコミットは発生しない設計となっている。
削除後に `git status` でコミット対象が生じた場合は、スコープ外の変更が混入していないかを確認する。
