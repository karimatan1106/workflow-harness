## サマリー

本仕様書は、RCA-1タスクの実行時に生じた誤配置テストファイル（`workflow-plugin/mcp-server/src/verify-sync.test.ts`）を削除するための実装計画を記述するドキュメントである。本ドキュメントは research.md（調査結果）および requirements.md（要件定義）の内容を引き継ぎ、残存問題の解決手順をまとめる。

### 目的

誤配置ファイルはgit未追跡のまま826行・約38,657バイトのディスク容量を占有しており、将来の開発者が正規ファイルの配置場所を誤認するリスクを持つ。本タスクでは `rm` コマンドを用いてこのファイルを削除し、ディレクトリ構造を正規の状態に戻す。

### 主要な決定事項

削除対象は `workflow-plugin/mcp-server/src/verify-sync.test.ts` の単一ファイルのみである。正規ファイル（`workflow-plugin/mcp-server/src/__tests__/verify-sync.test.ts`）は `git ls-files` で追跡済みであることが確認されており、変更を加える必要はない。削除後はテストスイートを実行して回帰がないことを確認する。

### 次フェーズで必要な情報

- 削除コマンド: `rm "C:\ツール\Workflow\workflow-plugin/mcp-server/src/verify-sync.test.ts"`
- 削除後の確認コマンド: `git -C workflow-plugin status --short` および vitest 実行
- 期待される結果: サブモジュール内 git status にこのファイルが表示されないこと（削除前同様にクリーン）、テストが 897 件前後でパスすること

## 概要

本タスクは、RCA-1タスクの副産物として生じた誤配置テストファイルの削除を目的とする単一ステップの整理作業である。

RCA-1タスクにおいてsubagentがコードを自動生成した際、正規のテスト配置ディレクトリ（`workflow-plugin/mcp-server/src/__tests__/`）ではなく、親ディレクトリ（`workflow-plugin/mcp-server/src/`）に誤ってテストファイルを作成した。このファイルはgitに追跡されていないため `git status` には表示されないが、ディスク上には残存しており、ディレクトリ構造の明確性を損なっている。

対応は `rm` コマンドによるファイル削除のみであり、他のソースコードへの変更は行わない。
削除操作はgitの追跡状態に何ら影響を与えないため、コミット不要で完了できる単純な整理作業である。
本仕様書では削除手順、確認方法、スコープ制約、品質確認基準をまとめている。
実装フェーズのsubagentが迷いなく作業を進められるようにすることを目的としている。

## 実装計画

本タスクの実装は以下の3ステップで構成される。

ステップ1として、implementationフェーズにおいて `rm` コマンドで誤配置ファイルを削除する。削除前後に `ls` コマンドでファイルの存在有無を確認し、意図した対象のみを削除したことを記録する。

ステップ2として、削除後にサブモジュールの `git status` を確認し、git追跡状態に変化がないことを検証する。誤配置ファイルはgit未追跡であったため、削除後もステータスはクリーンのままであることが期待される。

ステップ3として、testingフェーズでテストスイート全体を実行し、削除によって既存テストケースの実行に影響が生じていないことを回帰確認する。失敗件数がゼロであることをもって作業完了の判断基準とする。

実装上の留意点として、テストランナー（vitest）が誤配置ファイルをグロブパターンで拾っていた場合、削除後にテスト総数が減少することがある。その場合は変動前後の件数を記録し、失敗件数がゼロであれば回帰なしとして扱う方針を採用する。

## 実装手順

### ステップ1: 誤配置ファイルの削除

implementationフェーズで以下の操作を実施する。削除対象が正しいパスであることを事前に `ls` で確認してから実行することが推奨される。

削除コマンドは以下の通りである:
```bash
rm "C:\ツール\Workflow\workflow-plugin/mcp-server/src/verify-sync.test.ts"
```

削除後、ファイルが存在しないことを確認する:
```bash
ls "C:\ツール\Workflow\workflow-plugin/mcp-server/src/"
```

期待される出力には `verify-sync.test.ts` が含まれてはならない。`__tests__/` ディレクトリ、`verify-sync.ts`（本体ファイル）、その他のソースファイルは引き続き存在すべきである。削除前後のディレクトリ一覧を記録しておくことで、意図しない削除が発生していないことを事後に確認できる。

### ステップ2: サブモジュール git ステータスの確認

削除操作がサブモジュールの git 追跡状態に影響していないことを確認する。

```bash
git -C "C:\ツール\Workflow\workflow-plugin" status --short
```

誤配置ファイルはgit未追跡であったため、削除後もサブモジュールの `git status` には何も表示されないことが期待される。出力が空またはクリーンであれば確認完了である。もし予期しないエントリが表示された場合は、本タスクのスコープ外の変更が混入していないかを確認する必要がある。

### ステップ3: テストスイートの実行（testingフェーズ）

削除後のリグレッション確認はtestingフェーズで実施する。実行コマンドは以下の通りである:

```bash
cd "C:\ツール\Workflow\workflow-plugin/mcp-server" && npm test
```

期待される結果として、テスト総数が897件前後であること、失敗件数がゼロであることを確認する。誤配置ファイルがテストランナーによって実行されていた場合、削除によりテスト数が若干減少する可能性があるが、その場合は新しい総数を正しい状態として記録する。

## 変更スコープの制約

本タスクで行う変更は誤配置ファイルの削除操作のみであり、他のファイルへの変更は一切行わない。

変更スコープを明示的に限定する理由は以下の通りである。
RCA-1タスクで修正された `next.ts` のデッドコード削除（コミット5d0450f）は正しく適用済みであり、追加変更が必要な箇所は存在しない。
正規テストファイルの内容も現行仕様に完全に適合しており、修正の必要がない。
その他のソースファイルへの変更は本タスクの目的を超えており、subagentは実装中に関連する問題を発見した場合でも本タスクでの修正は行わず、別タスクとして起票する方針を採用する。

スコープ外の作業として、メインリポジトリ側の未コミット変更（削除済みワークフロー状態ファイル群、動画ファイル群）の整理は、本タスクとは独立した課題であり対応しない。

### 削除完了の判断基準

以下の条件が全て満たされた場合に削除作業が完了したと判断する。

削除対象ファイルが `ls` コマンドの出力に現れないこと、サブモジュール内の `git status` がクリーンであること（または誤配置ファイルに起因するエントリが存在しないこと）、正規ファイル `workflow-plugin/mcp-server/src/__tests__/verify-sync.test.ts` が引き続き存在し変更されていないこと、の3条件を全て確認する。

### テスト回帰確認の判断基準

テストスイート実行後に以下の状態であれば回帰なしと判断する。

vitest の出力において失敗件数がゼロであること。テスト総数の変動がある場合は、変動前後の件数を記録し、変動の原因が誤配置ファイルの削除によるものであることを確認する。897件から変動があった場合でも、失敗件数がゼロであれば回帰なしとして扱う。
3条件（削除確認、git状態確認、テスト実行確認）が全て揃った時点で品質基準を満たしたと判定する。
いずれかの条件が未達の場合は、条件を満たすまで追加対応を継続することを原則とする。

## 調査結果の引き継ぎ

research.mdの調査で明らかになった事項を仕様書に引き継ぐ。本セクションは設計・実装判断の根拠を一元化するためのものである。

テストカウントの変動について: ベースライン852件から現在897件への増加は45件の増加を意味する。この増加の主な原因は、コミット4ead400で新規追加された `bug-fix-regression-transition.test.ts`（628行、12テストケース）および、コミットb642cc2で追加された `verify-sync.test.ts`（30テストケース相当）によるものである。これらの追加はバグ修正に伴う意図的なテストカバレッジ拡充であり、リグレッション問題ではないことが調査で確認されている。

タイムスタンプ調査の結果: 誤配置ファイルの作成日時は2026年2月19日 20:49であり、正規ファイルの更新日時（20:53）より4分早い。RCA-1タスクのコミット5d0450fが21:03に記録されていることから、これらのファイルはいずれもRCA-1タスクの実行中に操作されていた。この時系列から、誤配置ファイルは正規ファイルを追跡済みとして確定する前の中間産物と推定される。

ワーキングツリーの状態: サブモジュール `workflow-plugin` はコミット5d0450fが未プッシュであるほかはワーキングツリーがクリーンな状態である。メインリポジトリには管理外の削除済みワークフロー状態ファイル群が残存しているが、本タスクの対象外と位置付けられている。

コミットメッセージ「fix: remove dead code in regression_test hash validation (RCA-1)」は、削除されたデッドコードがハッシュ検証処理（`testOutputHash` 関連）の条件分岐であったことを示している。この `recordTestOutputHash` のimportと非並列の条件分岐コードは `record-test-result.ts` 側での処理が実行中に担うとされ、next.tsではサブフェーズの通過処理のみが残った形となっている。

## 要件の引き継ぎ

requirements.mdで定義された機能要件（FR）と非機能要件（NFR）の要点を本仕様書で参照できるよう整理する。

機能要件の要点として、FR-1では誤配置テストファイルの削除、FR-2では正規テストファイルへの変更禁止、FR-3ではテストスイートの正常動作確認の3点が求められる。
削除の識別情報として、対象ファイルのパスはサブモジュール内相対表記で `workflow-plugin/mcp-server/src/verify-sync.test.ts`、作成日時は2026年2月19日 20:49、gitの追跡状態は未追跡（git ls-filesに現れない）である。
FR-3では削除前後のテスト総数の変動を記録することも要件として含まれており、増加や減少があった場合でも失敗件数がゼロであることが正常動作確認の合否判断基準となる。

非機能要件の要点として、NFR-1（変更の最小性）では本タスクで実施する変更を誤配置ファイルの削除のみに限定することを求めている。
余分な変更は副作用のリスクを増加させるため、最小化の観点から厳しくスコープを管理する。
NFR-2（Gitリポジトリの整合性維持）では削除操作を `git rm` ではなく通常の `rm` コマンドで実施し、コミットメッセージを付与する場合も識別情報が明確になるよう規定している。
NFR-3（テスト実行環境の安定性）では削除操作の前後においてテストスイートが安定した状態を維持することを評価基準として設けており、誤配置ファイルがテストランナーによって実行されていたケースも想定してテスト総数の変動を記録する方針を採用している。
NFR-4（作業の可逆性）では削除対象が正規ファイルのサブセット（古い版）であるため、正規ファイルが存在する限り理論上復元可能であることを根拠として可逆性を確保している。

コードレビューの観点から、本タスクは単一ファイルの削除という最小単位の変更であるため、レビュー負荷が低く承認フローを簡略化できる利点がある。
サブセットとしての関係性（誤配置ファイルは正規ファイルの古い版）が確認されており、削除によって仕様情報の損失が生じないことも要件として明記されている。

## 根本原因と再発防止

### 根本原因の記録

本問題の根本原因は、RCA-1タスクにおいてsubagentがテストファイルを作成する際、正しいパス（`workflow-plugin/mcp-server/src/__tests__/verify-sync.test.ts`）を指定せず、親ディレクトリの `workflow-plugin/mcp-server/src/verify-sync.test.ts` にファイルを生成したことである。

subagentがファイルを作成する際に既存のディレクトリ構造を事前確認せずにWriteツールを実行したことが直接的な原因である。テストファイルはすでに `workflow-plugin/mcp-server/src/__tests__/` に存在していたが、subagentはその確認を行わなかった。

この問題はsubagentのコンテキスト管理と既存ファイル確認手順の不足に起因しており、単発の人的ミスではなくプロセス上の改善点として認識する必要がある。

### 再発防止の考え方

テストファイルを作成・編集するフェーズ（test_impl、implementation）では、Orchestratorがsubagentに対して、既存のテストファイルの配置場所をGlobまたは `ls` コマンドで事前確認するよう明示的に指示することが有効である。
本タスクで採用した実装計画においても、削除前に `ls` でディレクトリ内容を確認するステップを設けることで、誤った対象への操作を防ぐ設計としている。
このような事前確認ステップを標準化することにより、同種の誤配置が将来のタスクで再発することを構造的に防止できる。
