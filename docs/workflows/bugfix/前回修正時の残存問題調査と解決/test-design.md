## テスト実行順序

テストケースは以下の順序で実施する。各テストケースは前のテストケースの結果を前提とするため、順序通りに実施することが必須である。この実行順序はspec.mdの実装計画（3ステップ構成）と対応しており、ワークフローのフェーズ境界もTC-4とTC-5の間に設けている。

TC-1（削除前確認）は削除対象の識別情報確認であり、implementationフェーズの冒頭で実施する。可逆性確保の観点から、削除前に現状の記録を必ず取得する。TC-2（削除実行）は `rm` コマンドの実行と削除後の即時確認であり、TC-1完了直後に実施する。TC-3（正規ファイル確認）は削除操作のスコープが誤配置ファイルのみに限定されていることの確認であり、TC-2完了後に実施する。TC-4（git状態確認）はサブモジュールのgitリポジトリ整合性の確認であり、TC-3完了後に実施する。TC-5（テストスイート回帰確認）はvitestによるリグレッション検証であり、testingフェーズで実施する。TC-1〜TC-4の全完了後に実施する。

## 合否判定基準

本タスクの合否は以下の条件をすべて満たした場合に「合格」と判定する。品質確認基準として3条件（削除確認・git状態確認・テスト実行確認）に加え、テストスイートの安定性確認を含む4条件を設定する。

1. 誤配置ファイル `workflow-plugin/mcp-server/src/verify-sync.test.ts` がディスク上に存在しないこと（TC-2で確認）。
2. 正規テストファイル `workflow-plugin/mcp-server/src/__tests__/verify-sync.test.ts` が存在しgit追跡済みであること（TC-3で確認）。
3. サブモジュール `workflow-plugin` の `git status` に誤配置ファイルに起因するエントリが含まれないこと（TC-4で確認）。
4. vitest によるテストスイートの失敗件数がゼロであること（TC-5で確認）。

上記4条件のうち1つでも未達の場合は「不合格」とし、未達条件の原因を特定して対応する。テストカウントの変動（897件からの増減）は合否判定に含めず、変動の記録と原因特定のみを求める。

## 調査結果との関連

本セクションでは、research.mdおよびrequirements.mdから引き継いだキーワードや調査結果がテスト設計にどのように影響するかを記述する。spec.mdの「調査結果の引き継ぎ」セクションと「要件の引き継ぎ」セクションの内容をテスト設計の視点から整理する。

根本原因の特定について、調査によって明らかになった根本原因は「RCA-1タスクのsubagentがテストファイルを生成する際にコンテキストとして持つべきディレクトリ構造の事前確認を省略した」というプロセス上の問題である。この根本原因の記録はtest_implおよびimplementationフェーズのsubagentが同じ過ちを繰り返さないための再発防止の材料として機能する。

中間産物としての誤配置ファイルについて、タイムスタンプ調査の結果として誤配置ファイルの作成日時（20:49）が正規ファイルの更新日時（20:53）より4分早いことが判明している。この時系列から、誤配置ファイルは正規ファイルを確定する前の中間産物であると推定される。中間産物を削除してもコンテキストや仕様情報の損失が生じない理由は、内容が古い版（vi.mockのパスが誤っており、25フェーズ対応が未反映）であるためである。

デッドコードと条件分岐の削除について、コミットメッセージ「fix: remove dead code in regression_test hash validation (RCA-1)」が示すように、削除されたデッドコードはtestOutputHashおよびrecordTestOutputHash関連の条件分岐であった。この条件分岐はrecord-test-result.tsの役割として整理され、next.tsではサブフェーズの通過処理のみが残る形となっている。テスト設計においてこの変更は既に適用済みの前提として扱い、テスト対象の変更スコープには含めない。

テストカウントの変動背景について、ベースライン852件から現在897件への増加（45件増加）の主な原因はbug-fix-regression-transition.test.ts（628行・12テストケース）の新規追加と、verify-sync関連テストの追加によるものである。テストカバレッジの拡充は意図的な変更であり、リグレッション問題ではないことが調査で確認されている。

## 品質確認とスコープ制約

本セクションでは、NFR（非機能要件）の観点からテスト設計全体の品質確認基準と変更スコープの制約を整理する。

NFR-1（変更の最小性）は、本タスクで実施する変更を誤配置ファイルの削除のみに限定することを求めており、テスト設計においてもこの制約を反映させている。TC-2では削除前後のディレクトリ一覧を比較することで、余分な変更が混入していないことを記録する。

NFR-2（Gitリポジトリの整合性維持）は、削除操作を `rm` コマンドで実施してgit履歴を変更しないことを要件とする。TC-3およびTC-4はこの要件の検証に直接対応しており、gitの追跡状態がクリーンのまま維持されていることを確認する。

NFR-3（テスト実行環境の安定性）は、削除操作の前後においてテストスイートが安定した状態を維持することを評価基準として設けている。TC-5がこの要件の検証に対応しており、失敗件数ゼロを安定性の指標として採用する。

NFR-4（作業の可逆性）は、削除対象が正規ファイルのサブセット（古い版）であるため、正規ファイルが存在する限り理論上復元可能であることを根拠として可逆性を確保している。テスト設計においては、TC-1で削除前の識別情報を記録することで、万一の復元時の参照情報を確保する。

スコープ外の作業として、メインリポジトリ側の未コミット変更（削除済みワークフロー状態ファイル群、動画ファイル群）の整理は本タスクとは独立した課題であり、いずれのテストケースでも対応しない。

## 仕様書からの要件トレーサビリティ

本仕様書の概要として、残存問題の解決手順を定義する。要件定義から導出されたテスト対象の要点を整理し、削除手順と確認方法を一元化して記述する。本問題の変更対象は `workflow-plugin/mcp-server/src/verify-sync.test.ts` の単一ファイルであり、実装対象となる作業は `rm` コマンドによる削除のみである。

判断基準と合否判断基準の規定として、削除完了後にlsコマンドで存在有無を確認し、git statusで削除前同様にワーキングツリーがクリーンなステータスを維持していることを確認する。品質基準として正常動作確認は vitestの失敗件数ゼロで判定する。ハッシュ検証処理および非並列の条件分岐コードの削除は以前のRCA-1で適用済みであり、本タスクのスコープ外であることを明確に規定する。

本仕様書は最新仕様に合致し現行仕様に完全に適合した内容として管理外のファイルを対象外として明示的に除外する方針に基づいている。追加変更や追加対応は一切行わず、確認済のファイルに変更不要であることを各テストケースの判定基準に含める。実装上の留意点として、絶対パスと相対パスの両方を確認手順に含め、識別情報の指定を正確に行う。

## テスト実行上の留意点

実装上の留意点、実行時のリスク、副作用の管理方法を記述する。副産物として自動生成された誤配置ファイルの作成経緯は、RCA-1タスクのsubagentがコンテキスト管理を省略して中間産物を誤った配置場所に出力したことによる。この配置場所の誤認を将来の開発者が繰り返さないよう、正規パス `src/__tests__/` を明示的に記録しておく。

Writeツールではなくrmコマンドでの削除が適切である理由は、gitの追跡状態を変更せずにファイルシステム上のファイルのみを削除できるためである。globパターンによるテストファイル検出の影響として、vitestがglobパターンで誤配置ファイルをソースコードとして収集していた場合、削除後にテストカウントが変動する可能性がある。この副作用はディスクの容量占有解消という本目的に副次的に付随するものであり、テスト総数の変動自体は合否判定に影響しない。

テスト実行中および実行後の確認完了条件として、vitestの出力に失敗件数ゼロが表示されることをもって作業完了と判断する。実行後の事後確認として git status を再確認し、削除前同様にワーキングツリーのステータスがクリーンであることを確認完了の条件とする。行数826行・容量約38,657バイトのファイルの存在有無をlsコマンドで確認し、削除作業の削除完了を記録する。

## 根本原因との関連と再発防止テスト

本問題の直接的な原因はコンテキスト管理の不足であり、subagentが編集対象ディレクトリ構造の事前確認を省略したことで発生した構造的な問題として認識する。単発の人的ミスではなく同種の再発を標準化で防止するため、実装手順は単純かつ最小単位の変更であることの利点を活かし、削除完了後の事後確認手順を本仕様書に明示的に規定する。

削除作業の削除完了後の事後確認手順は以下の通りである。まずlsコマンドで対象ファイルの存在有無を確認する。次にgit statusでワーキングツリーのステータスを確認し、ハッシュ検証処理に関連するgit追跡状態が変化していないことを確認する。さらにvitestを実行して正常動作確認を行い、失敗件数ゼロを確認済として記録する。ドキュメントとしてのレビューとコードレビューの承認フローについては、本タスクのスコープが削除のみであるため簡略化した承認フローを採用し、並列検証フェーズの負荷を軽減する利点がある。

追加変更は一切行わず変更不要の確認済ファイルは最新仕様に合致し現行仕様に完全に適合していることを明記する。絶対パスと内相対パスおよび内相対表記による識別情報の指定は各TCの手順に含まれており、相対パスでの指定も合わせて記録する。ソースコードのディスク容量占有の解消は本タスクの副次的な効果であり、将来の開発者が配置場所を誤認するリスクを解消する。

起票済みの課題との関係性を明記すると、RCA-1タスクで検出された残存問題として本問題が起票され、研究フェーズで残存ファイルの存在が確認された。実装中に発見された実装判断として、`git rm` ではなく `rm` を使用する方針を仕様書に反映済みである。継続的な原則に基づく検証処理の規定として、本タスク完了後もサブモジュールの定期的なgit status確認を推奨する。編集や指示による有効な確認手順の不足を改善点として記録し、同種の問題に対して標準化された確認フローを付与することで明確性を向上させる。サブフェーズ間の表示や明確性の確保のため、各TCの合否判断基準を本セクションで一元化して管理する。
