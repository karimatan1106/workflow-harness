# セキュリティスキャン新規指摘3件修正 - 要件定義

## サマリー

本タスクでは前回のセキュリティスキャンで検出された3件の新規脆弱性に対する修正要件を定義する。指摘NEW-SEC-1はbash-whitelist.jsのsplitCommandParts関数におけるゼロ幅Unicode文字の処理欠陥であり、U+200B ZERO WIDTH SPACEなどの不可視文字が正規表現の\sにマッチしないためコマンド分割に使用された場合にホワイトリスト検証を回避される可能性がある。指摘NEW-SEC-2は同じくbash-whitelist.jsのdetectEncodedCommand関数におけるFail-Open設計の問題であり、base64やprintf形式のエンコードコマンドのデコード処理が失敗した場合に検証をスキップしてallowed:trueを返すため攻撃者が意図的に不正なエンコードを使用することで検証を回避できる。指摘NEW-SEC-3はloop-detector.jsのnormalizeFilePath関数におけるシンボリックリンク解決のフォールバック処理に理論的なTOCTOU脆弱性が存在する問題であるが、ローカル開発環境での使用に限定されるため実務的なリスクは極めて低い。これら3件の修正は既存テスト42件との互換性を維持しながら最小限の変更でセキュリティ強度を向上させることを目的とする。優先度はFail-Closed化が最も高く次にUnicode対応、ログ強化は低優先度として段階的に修正を実施する。

## 機能要件

splitCommandParts関数とsplitCompoundCommand関数はコマンド文字列を分割する前にゼロ幅Unicode文字を全て除去するサニタイズ処理を実装する必要がある。対象となるゼロ幅文字はU+200B ZERO WIDTH SPACE、U+200C ZERO WIDTH NON-JOINER、U+200D ZERO WIDTH JOINER、U+FEFF ZERO WIDTH NO-BREAK SPACE（BOM）の4文字であり、正規表現/[\u200B\u200C\u200D\uFEFF]/gを使用して全てのゼロ幅文字を空文字列に置換する。このサニタイズ処理はコマンド文字列を複合コマンド演算子で分割する直前に実行し、元のコマンド文字列に含まれていたゼロ幅文字は全て削除された状態で後続の処理を行う。detectEncodedCommand関数はbase64デコード、printf形式のhexデコード、echo形式のoctalデコードのいずれかが失敗した場合に検証をスキップせずに明示的にallowed:falseとreasonを返すFail-Closed設計に変更する。具体的にはdecodeBase64Safe関数がnullを返した場合、decodePrintfHex関数がnullを返した場合、decodeEchoOctal関数がnullを返した場合のそれぞれで、デコード失敗を示す適切なreasonメッセージを含むレスポンスオブジェクトを返す。normalizeFilePath関数はfs.realpathSyncが失敗してpath.resolveにフォールバックした場合に警告レベルのログメッセージを出力してシンボリックリンクが解決されなかったことを記録する。

## 非機能要件

全ての修正は既存のテストスイート42件の合格を維持することが必須であり、テストが失敗する変更は受け入れられない。ゼロ幅文字のサニタイズ処理は正規表現のコンパイルが1回だけ実行されるように関数スコープ外で定数として定義し、実行時のパフォーマンスへの影響を最小限に抑える。detectEncodedCommand関数のFail-Closed化はデコード失敗時のレスポンスオブジェクト構築において既存のallowed:falseのレスポンスフォーマットと完全に一致させ、呼び出し側のコードに影響を与えない。normalizeFilePath関数のログ出力はconsole.warnを使用し、既存のログ出力機構と整合性を保ち、ログメッセージには元のファイルパスとフォールバック理由を含める。修正後のコードはworkflow-pluginのMCPサーバー再起動により即座に反映され、既存のワークフローセッション中のユーザーに影響を与えない。全ての修正はTypeScriptの型定義に違反せず、ESLintの静的解析ルールに準拠し、コードスタイルは既存のコーディング規約に従う。

## 受け入れ基準

splitCommandParts関数とsplitCompoundCommand関数はU+200BやU+FEFFを含むコマンド文字列を受け取った場合にゼロ幅文字を全て除去してから分割処理を実行し、ホワイトリスト検証が正しく機能することを確認できる。detectEncodedCommand関数は不正なbase64エンコード文字列を受け取った場合にallowed:falseとreason:"Base64 decoding failed"を返すことを確認でき、同様にprintf形式のhexエンコード失敗時とecho形式のoctalエンコード失敗時にもallowed:falseと適切なreasonメッセージを返すことを確認できる。normalizeFilePath関数は存在しないファイルパスを受け取ってfs.realpathSyncが失敗した場合にconsole.warnが呼び出され、ログメッセージに元のファイルパスとフォールバック理由が含まれることを確認できる。既存のテストスイート42件を全て実行した結果、全てのテストが合格し、修正前と同じ動作を維持していることを確認できる。修正後のコードでnpm run testを実行した場合にエラーや警告が発生せず、ビルドプロセスが正常に完了することを確認できる。各修正に対応するユニットテストケースが追加され、ゼロ幅文字のサニタイズ、デコード失敗時のFail-Closed動作、フォールバック時のログ出力がそれぞれ検証されることを確認できる。

## 制約事項

本修正はworkflow-pluginのhooksディレクトリに配置されたbash-whitelist.jsとloop-detector.jsの2ファイルのみを対象とし、他のフックファイルやMCPサーバーのコアロジックには変更を加えない。ゼロ幅文字のサニタイズ処理はUnicode文字クラスの範囲でJavaScriptの正規表現が標準的にサポートする4文字のみを対象とし、将来追加される可能性のある他のゼロ幅文字には対応しない。detectEncodedCommand関数のFail-Closed化はデコード失敗時のエラーメッセージをハードコードされた英語文字列として実装し、国際化対応は行わない。normalizeFilePath関数のログ出力はconsole.warnを使用するシンプルな実装とし、構造化ログやログレベルの動的制御などの高度なログ機構は導入しない。本修正はローカル開発環境での使用を前提としたワークフロープラグインのセキュリティ強化であり、本番環境での使用やマルチテナント環境での使用は想定していない。修正後のコードはNode.js v18以降とTypeScript 5.x環境での動作を保証するが、それ以前のバージョンでの互換性は保証しない。本修正は2026年2月時点のworkflow-pluginのコードベースに対するものであり、将来のリファクタリングやアーキテクチャ変更により修正箇所が無効化される可能性がある。

- `workflow-plugin/hooks/loop-detector.js` - normalizeFilePath関数の修正対象

## 次フェーズへの情報

planningフェーズでは3件の指摘に対する具体的な実装仕様を定義する必要がある。指摘1のゼロ幅文字サニタイズはsanitizeZeroWidthChars関数として切り出してsplitCommandPartsとsplitCompoundCommandの両方から呼び出す設計を検討する。指摘2のFail-Closed化はデコード失敗時のレスポンスオブジェクトの構造を統一し、各デコード関数のnull返却後に適切なエラーレスポンスを生成するヘルパー関数の導入を検討する。指摘3のログ強化はnormalizeFilePathのフォールバック箇所を特定し、適切なログメッセージフォーマットを定義する。test_designフェーズではゼロ幅文字を含むコマンドのテストケース、不正なエンコード文字列のテストケース、シンボリックリンク解決失敗時のログ検証テストケースをそれぞれ設計する。
