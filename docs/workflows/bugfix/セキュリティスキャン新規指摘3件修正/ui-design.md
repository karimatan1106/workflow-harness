# セキュリティスキャン新規指摘3件修正 - UIインターフェース設計書

## サマリー

本UIインターフェース設計書ではワークフロープラグインのフックシステムにおけるセキュリティ強化のためのCLI出力とエラーメッセージの設計を定義しています。
今回の修正対象はCLIツールの内部セキュリティフックであるためグラフィカルなユーザーインターフェースは存在せず主にコマンドブロック時のエラーメッセージとログ出力が主要なユーザー接点となります。
NEW-SEC-1のゼロ幅文字サニタイズについては通常は透過的に動作しますが不正なコマンドを検出した場合には明確な理由を含むエラーメッセージを表示する必要があります。
NEW-SEC-2のFail-Closed化では不正なエンコード文字列を検出した際にコマンド実行を拒否する旨を明確に通知しユーザーが問題の原因を理解できるようにします。
NEW-SEC-3のログ強化ではファイルパス正規化の失敗時に十分なコンテキスト情報を含む警告ログを出力し運用時のトラブルシューティングを支援します。

## CLIインターフェース設計

ワークフロープラグインのフックシステムはBashコマンド実行前に自動的に介入するためユーザーが明示的にコマンドを呼び出すわけではありませんがコマンドブロック時にはClaude Codeのターミナル出力として拒否理由が表示されます。
ゼロ幅文字が含まれるコマンドが検出された場合はコマンド文字列の視覚的な表示では気づけない攻撃を防ぐためサニタイズが自動適用されますがログには警告として記録されません。
エンコードされたコマンドのデコード失敗時にはClaude Codeの実行ログに赤色のエラーメッセージが表示されユーザーはコマンド実行が拒否されたことを即座に認識できます。
normalizeFilePath関数のフォールバック時には黄色の警告メッセージがコンソールに出力されファイルシステム操作に潜在的な問題があることを通知します。
エラーメッセージの設計ではセキュリティ上の理由を明示しつつ攻撃者に有用な情報を与えないバランスを考慮した文言を採用します。

## エラーメッセージ設計

NEW-SEC-1のゼロ幅文字サニタイズ処理は透過的に動作するためユーザーに対するエラーメッセージは基本的に表示されませんが将来的にログ記録が必要な場合にはコマンド文字列にゼロ幅Unicode文字が含まれていたことを示すメッセージを用意します。
NEW-SEC-2のbase64デコード失敗時には"Command blocked: Base64 encoded command could not be decoded"という明確なエラーメッセージをreasonフィールドに設定しコマンド実行を拒否します。
printf hexパターンのデコード失敗時には"Command blocked: Printf hex encoded command could not be decoded"というメッセージを返しhexエスケープシーケンスの検証失敗を通知します。
echo octalパターンのデコード失敗時には"Command blocked: Echo octal encoded command could not be decoded"というメッセージを返しoctalエスケープシーケンスの問題を明示します。
NEW-SEC-3のファイルパス正規化失敗時には"Warning: Failed to normalize file path [元のパス], falling back to path.resolve: [エラー詳細]"という警告メッセージをconsole.warnで出力します。

## APIレスポンス設計

bash-whitelist.jsのsanitizeZeroWidthChars関数は入力文字列からゼロ幅文字を除去したクリーンな文字列を返すシンプルな文字列変換関数として設計されます。
detectEncodedCommand関数の戻り値はFail-Closed化により変更されデコード成功時には従来通りallowed:trueまたはfalse、デコード失敗時には必ずallowed:falseとreasonフィールドを含むオブジェクトを返します。
reasonフィールドの値は前述のエラーメッセージ設計に基づいた文字列でありコマンドブロック理由の詳細をホワイトリスト検証ロジックに伝達します。
normalizeFilePath関数の戻り値自体に変更はなく引き続き正規化されたファイルパス文字列を返しますがフォールバック発生時には副作用としてログ出力が追加されます。
テスト用のモックAPIでは各関数の入力に対する期待される戻り値を厳密に定義しFail-Closedの動作が正しく実装されていることを検証可能にします。

## 設定ファイル設計

今回の修正は既存の設定ファイルへの変更を必要としませんがworkflow-plugin/hooks/bash-whitelist.jsとloop-detector.jsのコード内定数として新しい設定値が追加されます。
ZERO_WIDTH_CHARS_PATTERNは正規表現オブジェクトとしてbash-whitelist.jsのモジュールトップレベルで定義されゼロ幅文字の検出パターンを一元管理します。
この正規表現定数は/[\u200B\u200C\u200D\uFEFF]/gという形式でありU+200B、U+200C、U+200D、U+FEFFの4種類のゼロ幅文字をグローバルマッチで検出します。
ログメッセージのフォーマットもloop-detector.js内でハードコーディングされており外部設定ファイルからの読み込みは行いませんが将来的な拡張性を考慮して明確な命名規則を採用します。
既存のworkflow-plugin/settings.jsonやCLAUDE.mdとの整合性は維持されフックシステムの全体的な動作仕様に影響を与えることなくセキュリティ強化が適用されます。

## ユーザー通知設計

ゼロ幅文字サニタイズが実行された場合通常のユーザーには通知されませんが開発者モードやデバッグログが有効な環境では"Sanitized zero-width characters from command"という情報メッセージが記録される可能性を考慮します。
エンコードコマンドのデコード失敗によるブロックが発生した場合にはClaude Codeのターミナル出力に赤色で"Error: Command execution blocked due to security check failure"という見出しとともに詳細なreasonメッセージが表示されます。
ファイルパス正規化の警告については黄色のテキストでconsole.warnの出力が表示されユーザーはファイルシステムの問題を認識できますが処理自体は継続されるため致命的なエラーではないことが色で区別されます。
新規セキュリティチェックの導入により既存のワークフローが意図せずブロックされる場合にはエラーメッセージ内に"If you believe this is a false positive, please report to the project maintainers"という一文を追加してフィードバックチャネルを提供します。
全てのエラーメッセージは英語で統一されていますがCLAUDE.mdの多言語対応方針に従って将来的には日本語メッセージのサポートも検討する余地があります。

## ログ出力設計

NEW-SEC-3のログ出力では元のファイルパスとフォールバック理由とエラーの詳細メッセージを含む構造化された警告ログをconsole.warnで出力します。
fs.realpathSync失敗時のログフォーマットは"Warning: Failed to normalize file path with fs.realpathSync: [originalPath], falling back to path.resolve. Error: [error.message]"という形式です。
path.resolve失敗時のログフォーマットは"Error: Failed to normalize file path with path.resolve: [originalPath]. Returning original path. Error: [error.message]"という形式でconsole.errorを使用します。
ログメッセージにはファイルパスが含まれるため機密情報の漏洩を防ぐために絶対パスの一部をマスキングする処理は現時点では実装しませんが将来的なセキュリティ監査で検討事項となります。
本番環境でのログレベル設定によってはconsole.warnやconsole.errorが外部ログ収集システムに送信される可能性がありログ量の増加に注意する必要があります。

## 既存UIとの整合性

ワークフロープラグインの既存エラーメッセージスタイルは"Command blocked: [理由]"という形式を採用しており今回のエンコードコマンドデコード失敗メッセージもこの形式に従います。
既存のbash-whitelist.jsでは不正なコマンドパターン検出時に詳細な理由をreasonフィールドに設定しており新規のFail-Closedレスポンスも同じオブジェクト構造を使用します。
loop-detector.jsの既存ログ出力は編集頻度超過検出時に警告メッセージを表示しており新規のnormalizeFilePath警告ログも同じconsole.warn APIを使用して一貫性を保ちます。
既存のテストスイートではエラーメッセージの文言を厳密に検証しているケースがあるため新規メッセージが既存テストの期待値と衝突しないことを確認する必要があります。
全体として既存のフックシステムのエラー処理パターンとログ出力パターンを踏襲することで運用チームが新しいメッセージに戸惑うことなく対応できるようにします。

## アクセシビリティ考慮事項

CLIツールのエラーメッセージはスクリーンリーダーで読み上げられる可能性があるため冗長な記号や装飾文字の使用を避けシンプルな英文で構成します。
ターミナルの色分けに依存しない情報設計を行いエラーと警告の違いはテキストの内容自体で判別できるようにします。
ゼロ幅文字のような視覚的に見えない攻撃ベクトルに対してはエラーメッセージで明示的に"zero-width characters"と言及することでユーザーが問題の性質を理解できるようにします。
ログメッセージの構造は機械可読性も考慮しており自動化されたログ解析ツールが警告とエラーを正しく分類できるフォーマットを採用します。
視覚障害を持つ開発者がコマンドラインツールを使用する際にもセキュリティフックの動作を理解できるよう十分な説明を含むドキュメントをCLAUDE.mdに追記することを推奨します。

## パフォーマンス考慮事項

ゼロ幅文字サニタイズはコマンド文字列に対して正規表現マッチを1回実行するだけでありパフォーマンスへの影響はナノ秒単位に留まります。
ZERO_WIDTH_CHARS_PATTERN定数をモジュールトップレベルで定義することで正規表現オブジェクトのコンパイルが起動時に1回だけ実行されランタイムでの正規表現生成コストを回避します。
detectEncodedCommand関数のFail-Closed化はデコード失敗時に早期リターンを行うためむしろパフォーマンスが向上する可能性があります。
normalizeFilePath関数のログ出力はファイルパス正規化が失敗した例外的なケースでのみ実行されるため通常の処理パスには影響しません。
全体として今回のセキュリティ修正によるパフォーマンス劣化は測定不可能なレベルであり既存の高速なコマンド検証処理を維持します。

## 国際化対応

現在のワークフロープラグインは英語のエラーメッセージのみをサポートしていますが将来的な多言語対応の可能性を考慮してメッセージ文字列をハードコーディングせずに定数として定義することを推奨します。
エラーメッセージ定数はERROR_MESSAGES.BASE64_DECODE_FAILEDのような形式で定義しメッセージカタログへの移行を容易にします。
ゼロ幅文字やエンコードパターンのような技術的な用語は翻訳が困難なため英語での説明を維持しつつ日本語の補足説明をドキュメントに追加します。
ログメッセージのタイムスタンプやファイルパスは言語に依存しない情報であるためそのまま出力しますがエラー説明文のみを翻訳可能にします。
国際化対応はNEW-SEC-1からNEW-SEC-3の修正完了後の将来的な改善項目として位置づけられており今回の実装では英語メッセージのみをサポートします。

## テスト時のUI検証

新規テストファイルtest-n6-security-new.test.tsではエラーメッセージの文言が期待通りに返されることをassert文で検証します。
ゼロ幅文字サニタイズのテストではサニタイズ前後のコマンド文字列を比較し目に見えない文字が正しく除去されていることを確認します。
Fail-ClosedレスポンスのテストではdetectEncodedCommand関数の戻り値オブジェクトがallowed:falseとreasonフィールドを含むことを検証します。
normalizeFilePathのログ出力テストではconsole.warnとconsole.errorをモックに置き換えて期待されるログメッセージが出力されることを確認します。
手動テストではClaude Codeのターミナルで実際にブロックされたコマンドのエラーメッセージを目視確認しユーザビリティが適切であることを検証します。

## ドキュメント更新

CLAUDE.mdのセキュリティセクションに今回追加された3つの新規セキュリティチェックの説明を追加します。
ゼロ幅文字サニタイズについては攻撃ベクトルの説明とサニタイズが自動適用されることをユーザーに通知するドキュメントを記述します。
エンコードコマンドのFail-Closed化についてはデコード失敗時にコマンドがブロックされる動作仕様を明記します。
ファイルパス正規化の警告ログについては運用時にログが出力される条件と対処方法をトラブルシューティングガイドとして追加します。
全てのドキュメント更新はdocs_updateフェーズで実施されますが本ui-designフェーズではドキュメントに記載すべき内容の要件を定義します。
