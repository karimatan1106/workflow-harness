# 脅威モデル: regression_test サブエージェントの workflow_capture_baseline 誤用防止ガイダンス追加

## サマリー

本ドキュメントは、regression_test フェーズの subagentTemplate に `workflow_capture_baseline` の誤用防止ガイダンスを追加する変更（FR-13・FR-14）に対する脅威モデルを記述する。

変更対象はローカルの MCP サーバープラグイン内にある definitions.ts の文字列フィールドであり、ネットワーク通信や認証機構には直接影響しない。しかし、サブエージェントが誤ったフェーズでワークフロー制御ツールを呼び出すという「行動レベルの脅威」が存在するため、それを体系的に整理する。

- 主要な懸念領域: ガイダンス欠落による subagent の誤動作、ワークフロー制御の意図しない呼び出し、ベースライン記録フェーズの不整合
- 変更範囲は definitions.ts の regression_test エントリの subagentTemplate フィールドのみに限定されており、影響範囲は局所的である
- 脅威の大半はガイダンス追加によって緩和されるが、実装ミスや Orchestrator 側の対応不備が二次リスクとなり得る

## 脅威シナリオ

### シナリオ1: workflow_capture_baseline の誤フェーズ呼び出し

regression_test サブエージェントが、ベースラインの存在を確認しようとして `workflow_capture_baseline` を呼び出すケース。このツールは testing フェーズでのみ MCP サーバーが受け付ける設計であり、regression_test フェーズからの呼び出しはアーキテクチャ上エラーとなる。ガイダンスが欠落している現状では、サブエージェントがこのエラーを予測できず、無駄なリトライや処理停止が発生する可能性がある。

この誤用は悪意によるものではなく、テンプレートの情報不足に起因する。サブエージェントは「ベースラインを確認したい」という正当な動機から誤ったツールを選択する。FR-13 および FR-14 のガイダンス追加によって禁止対象が明示されることで、この誤用は根本的に防止できる。

### シナリオ2: ワークフロー制御ツールの連鎖呼び出し

過去の障害事例（MEMORY.md に記録）として、regression_test サブエージェントが `workflow_next` を連続呼び出しして parallel_verification の全サブフェーズを自律実行した事例がある。FR-13 で禁止対象リストに `workflow_capture_baseline` を追加する際、既存の禁止リスト（workflow_next・workflow_approve・workflow_complete_sub・workflow_start・workflow_reset）との整合性が保たれなければ、サブエージェントが「このツールは言及されていないから使えるはず」と誤解するリスクがある。

リストへの追記漏れや記述の不明確さが二次的な混乱を生む可能性があるため、追記内容の正確性と文体の一貫性が重要である。

### シナリオ3: ベースライン未設定状態での regression_test 実行

testing フェーズでベースライン記録（`workflow_capture_baseline` の呼び出し）が実施されないまま regression_test フェーズに遷移した場合、サブエージェントは比較基準を持たない状態でテスト結果の評価を試みる。この状態でサブエージェントが `workflow_capture_baseline` を誤用しようとすることが、シナリオ1に繋がる根本原因でもある。

FR-14 の「ベースライン前提条件」セクションに Orchestrator が testing フェーズへ差し戻すべきことを明記することで、フェーズ遷移の設計上の前提条件を subagent に伝達できる。

### シナリオ4: MCP サーバー再起動未実施による変更の非反映

definitions.ts を変更した後に MCP サーバーを再起動しない場合、Node.js のモジュールキャッシュにより古いテンプレートが引き続き動作する。この場合、FR-13・FR-14 のガイダンスが追加されたにもかかわらず、subagent には古いテンプレートが渡されるため、変更の効果がゼロになる。NF-3 の要件として再起動を義務付けているが、実施確認の手順が明確でない場合にリスクが顕在化する。

### シナリオ5: 追記によるテンプレート文字列の構文破損

definitions.ts の subagentTemplate はテンプレートリテラル文字列として定義されており、不適切な挿入位置や誤ったエスケープ処理によって TypeScript のビルドエラーが発生する可能性がある。ビルドエラーが発生した場合、MCP サーバーが起動できなくなり、全ワークフローが停止する。NF-2 の後方互換性要件（追記のみ、既存内容の書き換え禁止）と、ビルド確認の実施がこのシナリオを防ぐための主要な手段である。

## リスク評価

### リスク1: workflow_capture_baseline 誤用（シナリオ1）

- **深刻度**: 低（MCP サーバーがエラーを返すだけで、ワークフロー状態は変更されない）
- リスクレベル: ガイダンス追加前は中程度、追加後は低（サブエージェントが自律的に回避できるようになる）
- 発生可能性: ガイダンス欠落のまま放置した場合は中程度（サブエージェントがベースライン確認のためにツールを探索する場面で発生しうる）
- 緩和策: FR-13 で禁止対象リストへの明示的な追加、FR-14 でベースライン前提条件の説明を行う

### リスク2: ワークフロー制御の意図しない連鎖（シナリオ2）

- **深刻度**: 高（過去に parallel_verification 全サブフェーズの自律実行という障害を引き起こした実績がある）
- リスクレベル: 既存の FR-4・FR-5 の対策後は中程度まで低下しているが、禁止リストに抜け漏れがあると残存する
- 発生可能性: `workflow_capture_baseline` がリストから欠落している現状では低〜中程度
- 緩和策: FR-13 でリストを完全化し、既存の禁止セクションと一貫した文体で記述する

### リスク3: ベースライン未設定状態での誤動作（シナリオ3）

- **深刻度**: 中（テスト結果の評価精度が低下するが、ワークフロー自体の破壊には至らない）
- リスクレベル: Orchestrator が適切なフェーズ遷移管理を行えば低い
- 発生可能性: testing フェーズでベースライン記録が省略された場合に発生（CLAUDE.md ルール20が義務付けているが、強制力は限定的）
- 緩和策: FR-14 でサブエージェントへの明確な指示と、Orchestrator への差し戻し指示を追記する

### リスク4: MCP サーバー再起動未実施（シナリオ4）

- **深刻度**: 中（変更が無効化されるが、既存の動作を破壊するわけではない）
- リスクレベル: NF-3 の手順が明記されていれば低い
- 発生可能性: 実装フェーズ完了後に再起動手順を省略した場合に発生
- 緩和策: CLAUDE.md ルール22および実装フェーズのチェックリストで再起動を義務付ける

### リスク5: テンプレート文字列の構文破損（シナリオ5）

- **深刻度**: 高（MCP サーバーが起動できなくなり、全ワークフローが停止する）
- リスクレベル: 追記のみ・ビルド確認実施の前提で低い
- 発生可能性: 実装ミスが発生した場合は低確率だが影響が大きい
- 緩和策: NF-2 の後方互換性要件（追記のみ）を厳守し、build_check フェーズでビルド確認を必須とする

## セキュリティ要件

### SR-1: 禁止ツールリストの完全性確保

regression_test テンプレートの「ワークフロー制御ツール禁止」セクションには、subagent がアクセスできる全ての MCP ツールのうち、regression_test フェーズで使用してはならないものを網羅的に列挙すること。現在の禁止リスト（workflow_next・workflow_approve・workflow_complete_sub・workflow_start・workflow_reset）に加え、`workflow_capture_baseline` を明示的に追加する。リストへの追加を行う際は、他の禁止ツールと同等の文体・強度で記述し、禁止の根拠（testing フェーズでのみ有効という設計上の制約）を必ず明示すること。

### SR-2: ベースライン前提条件の明文化

FR-14 で追加する「ベースライン前提条件」セクションには、testing フェーズでのベースライン記録が完了していることを前提として regression_test が実行されるという設計意図を明記すること。前提条件が満たされていない場合のエスケープパス（Orchestrator が testing フェーズへ差し戻す）も具体的に示すことで、subagent が不確定な状態で自律的な判断を行うことを防止する。ベースライン情報の確認手段として `workflow_get_test_info` が利用可能であることも案内し、サブエージェントが適切な方法でベースラインにアクセスできるようにする。

### SR-3: ビルド整合性の確保

definitions.ts への変更後は必ず `npm run build` を実行してビルドエラーがないことを確認すること。ビルドエラーが発生した場合は MCP サーバーが起動できなくなるため、変更前の状態に戻して原因を特定・修正してから再度ビルドを行うこと。ビルド成功後には MCP サーバーを再起動し、`workflow_status` で正常動作を確認することを必須とする。この手順を省略した場合、変更内容がキャッシュにより無効化されるか、またはビルドエラーにより全ワークフローが停止するリスクがある。

### SR-4: テンプレート変更の影響範囲の限定

本変更は definitions.ts の regression_test エントリの subagentTemplate フィールドへの追記のみに限定すること。testing テンプレートや他のフェーズテンプレートへの変更は本タスクの範囲外であり、意図しない副作用を防ぐために手を加えてはならない。変更後は既存テストスイートの全テストが通過することを確認し、リグレッションがないことを保証すること。

### SR-5: Orchestrator の責任範囲の明確化

サブエージェントへのガイダンス追加は、Orchestrator（メインの Claude）がフェーズ制御の責任を保持することを前提として設計されている。subagent は自律的にワークフロー制御ツールを呼び出してはならず、テスト実行・結果記録・結果の Orchestrator への報告のみを責任範囲とすること。この責任分担をテンプレートに明記することで、将来の新しいサブエージェントが誤った設計に基づいて動作するリスクを低減する。
