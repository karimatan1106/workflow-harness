# テスト設計書: VDB-1 retryCount reset修正

## サマリー

本テスト設計書は、`harness_reset` および `harness_back` 呼び出し時に `retryCount` がリセットされないバグ（VDB-1）の修正に対するテスト方針を定義する。
StateManager の `resetTask` メソッドと `goBack` メソッドを対象に、単体テストによる検証を実施する。
修正の核心は、フェーズ移行時に `retryCount` オブジェクトを空オブジェクト `{}` に初期化する処理の追加である。
ビルドの健全性確認として `npm run build` の終了コード検証を含め、既存テストスイートとの整合性確認として `npm test` の全件通過を検証する。

Delta Entry 一覧:
- TD-001: ユニットテスト主体の検証方針を採用する
- TD-002: AC-1 と AC-2 各2件のテストを設計する（retryCount あり・なし両パターン）
- TD-003: AC-3 と AC-4 は npm コマンドの終了コード確認で検証する
- TD-004: 境界値としてカウントが 0 の状態でのリセット・ロールバックを検証する
- TD-005: 既存テストファイル `manager-lifecycle.test.ts` に新規テストを追加する

## テスト方針

本修正は StateManager の内部ロジックに対する局所的な変更であるため、ユニットテストを主体とした検証方針を採用する。
`resetTask` および `goBack` はいずれも TaskState オブジェクトを直接変更する同期関数であり、モックや非同期処理を必要とせず、単体で完結したテストが可能である。
テストケースは「retryCount が設定されている状態」と「retryCount が初期化されていない状態（空オブジェクト）」の2パターンで各メソッドを検証し、型安全性と動作の正確性を両方確認する。
既存のテストファイル `manager-lifecycle.test.ts` に追加する形式を採用することで、テストセットアップの重複を避け、既存のテストインフラを最大限に活用する。
ビルド健全性とリグレッション防止については、npm コマンドの終了コードを定量的な合否基準として採用し、CI パイプラインでの自動検証を可能にする。

## テストケース

テストケースは4つの受入基準（AC-1 から AC-4）に対応して定義する。
AC-1 と AC-2 はそれぞれ2件のユニットテストで構成し、retryCount に値が設定されている場合と初期化されていない場合の両方を網羅する。
AC-3 は TypeScript コンパイルの成功を確認するビルドテストであり、型エラーや構文エラーが残存していないことを保証する。
AC-4 は既存の全テスト（290件）に加え、本修正で追加する新規テスト2件を含めた合計292件の通過を確認する。
各テストケースの「期待値」欄の `{}` は空オブジェクト（初期化済み retryCount）を意味する。

| AC-N | テストケースID | テスト内容 |
|------|--------------|-----------|
| AC-1 | TC-AC1-01 | `incrementRetryCount` で planning フェーズのカウントを 3 に設定してから `resetTask` を実行すると `retryCount` が `{}` になる |
| AC-1 | TC-AC1-02 | `retryCount` が初期化されていない状態で `resetTask` を実行すると `retryCount` が `{}` になる（型安全性） |
| AC-2 | TC-AC2-01 | `incrementRetryCount` で research フェーズのカウントを 3 に設定してから `goBack` を実行すると `retryCount` が `{}` になる |
| AC-2 | TC-AC2-02 | `retryCount` が初期化されていない状態で `goBack` を実行すると `retryCount` が `{}` になる（型安全性） |
| AC-3 | TC-AC3-01 | `workflow-harness/mcp-server` で `npm run build` が終了コード 0 で完了する |
| AC-4 | TC-AC4-01 | `npm test` で全テストが通過し終了コード 0 が返される（既存290件 + 新規2件） |

## エッジケース

境界値テストとして、`retryCount` オブジェクトが存在しない（プロパティ自体が未設定）の状態からのリセットと、値が 0 のカウントが存在する状態からのリセットを検証する。
特に TypeScript のオブジェクトスプレッドや代入時に `undefined` プロパティが残存しないことを `toEqual({})` による厳格な等値チェックで確認する。
`goBack` は `completedPhases` の末尾スライスも行うため、`retryCount` リセットが `completedPhases` 操作と独立して正しく機能することを確認する。
複数フェーズにわたって `retryCount` が設定されている場合（例: research=2, planning=3）にリセット後すべてのカウントが消去されることも検証対象とする。
`resetTask` と `goBack` の両方において、リセット後に再度 `incrementRetryCount` を呼び出したときに正常にカウントが 1 から再開されることを副次的に確認する。
フェーズを越えた `retryCount` の累積は意図しない挙動の原因となるため、各フェーズ移行の入り口で必ず初期化済み状態になっていることを不変条件として扱う。
