# テスト設計書

## サマリー

- 目的: FR-15（CLAUDE.md ルール20への補足説明追記）の完了を検証するためのテスト設計を行う。変更対象はドキュメント1ファイルのみであり、コードへの変更は伴わないため、目視確認（Read ツールによる内容検証）がテスト手段となる
- 主要な決定事項: 新規テストコードの作成は不要。Orchestrator が Read ツールで CLAUDE.md の変更後内容を確認することで受入条件を検証する方針とする。FR-13/FR-14 で追加済みの TC-FIX テストスイートが今回の変更後も合格することを回帰確認として定義する
- 次フェーズで必要な情報: implementation フェーズでは CLAUDE.md の695〜699行目付近を Edit ツールで変更し、その直後に Read ツールで4つの受入条件をすべて確認する。テスト実装フェーズは今回スキップされるため、test_impl フェーズでは本テスト設計書の内容を参照しながら Orchestrator 自身が確認作業を実施する

## テスト方針

今回の変更対象は CLAUDE.md という Markdown ドキュメント1ファイルのみである。コードファイル（TypeScript・JavaScript）への変更は一切伴わない。したがって自動テストスイート（vitest・jest 等）による機械的な検証は対象外となる。

テストの主体は Orchestrator 自身であり、実装フェーズで Edit ツールを用いた追記が完了した直後に Read ツールで CLAUDE.md の該当箇所を読み込み、各受入条件を目視確認することで合否を判定する。

確認は逐次的に行い、1つの条件が満たされない場合は実装フェーズへの差し戻しを判断する。確認作業は以下の4条件を順番に実施し、全条件がパスした場合に FR-15 の完了とみなす。

Read ツールを用いる際の具体的な方法として、CLAUDE.md 全体を読み込んでルール20のセクションを探し、695行目付近から始まるリスト項目を確認することを標準手順とする。実際の行番号は追記前後で変動する可能性があるため、「20. \*\*既存テストのベースライン記録義務」という見出し文字列を目印として該当セクションを特定すること。

## テストケース

### TC-FR15-1: calculatePhaseSkips によるセットスキップ記述の存在確認

本テストケースは spec.md の受入条件1に対応する。

確認手順として、Orchestrator は implementation フェーズ完了後に Read ツールで CLAUDE.md を読み込む。読み込んだ内容の中から「20. \*\*既存テストのベースライン記録義務」セクションを見つけ、そのセクション内に「calculatePhaseSkipsによって常にセット」という文字列が存在することを確認する。

文字列が存在する場合は TC-FR15-1 合格とする。文字列が存在しない場合は「追記内容の挿入が失敗または誤った位置に挿入された」と判断し、implementation フェーズへ差し戻す。

合格の判定基準は以下の通りである。
- ルール20の箇条書きリスト内に「calculatePhaseSkipsによって常にセット（両方スキップまたは両方実行）」という表現またはその同等表現が含まれること
- 当該箇条書き項目が「testing がスキップされる場合は regression_test も同時にスキップされる」という趣旨の説明を含むこと
- 当該項目が「既存テストが存在しない新規プロジェクトでは記録不要」という行の直後または末尾に位置していること

### TC-FR15-2: forceTransition 使用時の警告記述の存在確認

本テストケースは spec.md の受入条件2に対応する。

確認手順として、Orchestrator は TC-FR15-1 の確認に続いて、ルール20のセクション内に「forceTransition: true」という文字列および parallel_verification への遷移失敗リスクに関する警告が存在することを確認する。

文字列が存在する場合は TC-FR15-2 合格とする。文字列が存在しない場合は「2項目目の追記が欠落している」と判断し、implementation フェーズへ差し戻す。

合格の判定基準は以下の通りである。
- ルール20の箇条書きリスト内に「forceTransition: true」という表記が存在すること
- 当該箇条書き項目が「parallel_verification への遷移時にベースラインが必須となり進行不能になる可能性がある」という趣旨の警告を含むこと
- 当該項目が「Orchestrator が workflow_capture_baseline を呼び出すか、workflow_back で testing フェーズへ差し戻してベースラインを記録すること」という趣旨の対処法を示していること

### TC-FR15-3: 既存4行の保全確認

本テストケースは spec.md の受入条件3に対応する。変更前に存在していたルール20の4行が削除・変更されていないことを確認する。

確認手順として、Orchestrator は Read ツールで読み込んだ CLAUDE.md のルール20セクションに以下の4つの文字列が全て存在することを順番に確認する。

確認する文字列1は「testingフェーズまでに既存テストスイートを実行し、workflow_capture_baselineで結果を記録すること」であり、これがルール20の最初の箇条書き項目として存在することを確認する。

確認する文字列2は「researchフェーズはphase-edit-guardによりtestingカテゴリ（npm test等）がブロックされるため、テスト実行はtestingフェーズで行うことを標準とする」であり、2番目の箇条書き項目として存在することを確認する。

確認する文字列3は「ベースラインが未設定の場合、regression_testフェーズで変更前後の比較ができなくなる」であり、3番目の箇条書き項目として存在することを確認する。

確認する文字列4は「既存テストが存在しない新規プロジェクトでは記録不要」であり、4番目の箇条書き項目として存在することを確認する。

4つ全ての文字列が存在する場合は TC-FR15-3 合格とする。1つでも欠落または変更が見られた場合は「既存行が意図せず変更された」と判断し、implementation フェーズへ差し戻す。

### TC-FR15-4: 角括弧なし・禁止語なしの確認

本テストケースは spec.md の受入条件4に対応する。追記内容にバリデーター禁止パターンが含まれていないことを確認する。

確認手順として、Orchestrator は Read ツールで読み込んだルール20の追記箇所（5行目・6行目）を対象に以下の2点を確認する。

確認点1は角括弧の不在である。追記した箇条書きテキスト内にコードフェンス外の角括弧文字（開き括弧・閉じ括弧）が含まれていないことを確認する。CLAUDE.md の本文はコードフェンスで囲まれていないため、全ての括弧表現が検出対象となる。

確認点2は禁止語の不在である。追記した箇条書きテキスト内に以下の禁止語が含まれていないことを確認する。英語グループ（4語）として TODO・TBD・WIP・FIXME が含まれていないことを確認する。日本語グループ（8語）として「未定」「未確定」「要検討」「検討中」「対応予定」「サンプル」「ダミー」「仮置き」が含まれていないことを確認する。

両確認点をクリアした場合は TC-FR15-4 合格とする。いずれかが検出された場合は「追記内容に禁止パターンが含まれている」と判断し、implementation フェーズへ差し戻す。

## テスト手順: 全テストケースの実施順序

Orchestrator が test_impl フェーズ完了後に実施する確認作業の全体手順を以下に記載する。

ステップ1として、implementation フェーズで Edit ツールを使用した追記が完了した直後に Read ツールで CLAUDE.md を読み込む。読み込み範囲は少なくともルール20のセクション全体（695行目付近から追記後の末尾まで）を含むこと。

ステップ2として、TC-FR15-3（既存4行の保全確認）を実施する。既存行は追記の前提条件であるため、最初に確認することが望ましい。

ステップ3として、TC-FR15-1（calculatePhaseSkips 記述の存在確認）を実施する。

ステップ4として、TC-FR15-2（forceTransition 警告の存在確認）を実施する。

ステップ5として、TC-FR15-4（角括弧なし・禁止語なしの確認）を実施する。

全5ステップが合格した場合に FR-15 の実装完了と判定し、次フェーズへ進む。いずれかのステップで不合格が発生した場合は implementation フェーズへ差し戻し、再追記を依頼する。

## 既存テストの回帰確認

FR-13/FR-14 の実装で追加済みの TC-FIX テストスイートが今回の変更後も合格することを回帰確認として定義する。

今回の変更対象は CLAUDE.md という Markdown ドキュメントのみであり、TypeScript のソースコードは一切変更しない。そのため TC-FIX テストスイートを含む既存テストが今回の変更によって失敗するリスクは理論上ゼロである。

ただし、保守的な観点から regression_test フェーズにおいてテストスイート全体が合格することを確認する。具体的な回帰確認の手順は次の通りである。

regression_test フェーズ開始時に `npm test` を実行し（testing カテゴリ許可フェーズであることを前提とする）、テスト結果の中に今回の変更に起因する新規失敗が含まれていないことを確認する。前タスクで実装された TC-FIX-1 および TC-FIX-2 のテストケースが引き続き合格していることを個別に確認する。

変更前のベースラインと比較して新たに失敗したテストケースが存在する場合は、変更との因果関係を分析し、CLAUDE.md 変更との直接的な関係がない場合は `workflow_record_known_bug` で記録して継続する判断をすることができる。

## テスト実施上の注意事項

本テスト設計書は Orchestrator 自身が確認作業を実施することを前提として作成されている。サブエージェントに確認を委任する場合は、Read ツールのみを使用すること（実装変更ツールである Edit/Write ツールの使用は確認作業では禁止）。

CLAUDE.md は行数が多いドキュメントであるため、全体読み込みを行う際は必要な範囲を特定してから確認することが効率的である。行番号を使った部分読み込みよりも見出し文字列を検索して特定するアプローチが安定している。
