## CLIインターフェース設計

本タスクは CLAUDE.md へのドキュメント追記のみを行う変更であり、グラフィカルなUIは存在しない。
Orchestrator（AI）が操作する「インターフェース」とは、Read ツールおよび Edit ツールを用いた一連の操作手順である。
以下に、実装フェーズで Orchestrator が実行すべき操作の全体フローを設計する。

### ステップ1: 現状確認操作（Read ツール）

Read ツールで `C:\ツール\Workflow\CLAUDE.md` を読み込み、ルール20の現在の記述を確認する。
読み込む行範囲として、695行目から710行目程度を指定することで追記対象箇所を特定できる。
Read ツールの呼び出し時には `offset: 694` と `limit: 20` を組み合わせて使用する。
確認すべき4行の識別文字列として、「testingフェーズまでに既存テストスイートを実行し」が存在することを検証する。
この確認ステップは、後続の Edit ツール呼び出しで使用する `old_string` の正確なパターンを特定するために必須である。

### ステップ2: 追記操作（Edit ツール）

Edit ツールを使用して `C:\ツール\Workflow\CLAUDE.md` のルール20末尾に2行を追記する。
`old_string` パラメータには「    - 既存テストが存在しない新規プロジェクトでは記録不要」を指定し、追記後の完全な文字列を `new_string` に設定する。
`new_string` の末尾には追記する1項目目（calculatePhaseSkips の設計説明）と2項目目（forceTransition 警告）を含める。
インデント形式はスペース4つとハイフンを使用し、既存の箇条書きスタイルと一致させる。
Replace_all パラメータは false に設定し、該当行が1箇所だけ変更されることを保証する。

### ステップ3: 変更後の確認操作（Read ツール）

追記完了後に同じファイルを再度 Read ツールで読み込み、変更が正しく反映されていることを確認する。
確認する行数は追記後の701行目付近まで拡張し、6行すべてが存在することを検証する。
既存の4行が削除・変更されていないことを行単位で比較する。
追記された2行がルール20の末尾に正しい位置で挿入されていることを確認する。
コードフェンス外の角括弧が存在しないことを目視確認する。

### 既存4行の保全確認方法

保全確認では、Read ツールの出力結果に以下の4文字列が存在することを確認する。
保全確認対象の1行目は「testingフェーズまでに既存テストスイートを実行し」という文字列である。
保全確認対象の2行目は「researchフェーズはphase-edit-guardにより」という文字列である。
保全確認対象の3行目は「ベースラインが未設定の場合」という文字列である。
保全確認対象の4行目は「既存テストが存在しない新規プロジェクトでは記録不要」という文字列である。
いずれかの文字列が見つからない場合は Edit ツールの `old_string` が誤っていた可能性があるため、処理を停止してファイルの状態を確認すること。

## エラーメッセージ設計

追記作業中に発生しうるエラーのパターンと対処法を以下に示す。

### エラーパターン1: ファイルが見つからない場合

Read ツールまたは Edit ツールが「file not found」等のエラーを返した場合、作業ディレクトリの確認が必要である。
絶対パス `C:\ツール\Workflow\CLAUDE.md` を使用しているにもかかわらずエラーが発生する場合は、ファイルシステムのアクセス権限または Windows パスのエスケープが問題の可能性がある。
対処法として、Glob ツールを使用してパターン `**/CLAUDE.md` でファイルを検索し、正確なパスを再確認する。
パスに日本語ディレクトリが含まれているため、パスの文字コードに起因する問題が発生した場合はバックスラッシュをフォワードスラッシュに変換して再試行する。

### エラーパターン2: 追記箇所が既に存在する場合

Edit ツールで `old_string` が一致しない場合は、対象箇所が既に変更済みである可能性がある。
具体的には、「calculatePhaseSkipsによって常にセット」という文字列が既に存在するかどうかを Read ツールで確認する。
この文字列が存在する場合は前回の追記が成功しているため、変更は既に完了している。
既存のテキストに一部変形が加わっている場合は、Read ツールで実際のテキストを読み取り、`old_string` を正確なテキストに修正してから Edit ツールを再実行する。

### エラーパターン3: 禁止語が含まれていた場合の対処

バリデーターが禁止語（英語グループの4語・日本語グループの8語）を検出した場合、成果物全体がバリデーション失敗となる。
追記する文言にはこれらの禁止語が意図的に含まれていないことを spec.md の確定文言で確認済みである。
万一バリデーション失敗が発生した場合は、エラーメッセージに含まれる「検出された語句」の間接参照を手がかりに、追記したテキストを Read ツールで確認する。
禁止語と判定されたテキストは「該当する語句」という間接表現を使って CLAUDE.md 上での記述を修正し、同等の意味を持つ別の表現に置き換えること。

### エラーパターン4: Edit ツールの一意性エラー

Edit ツールは `old_string` がファイル内に複数存在する場合にエラーを返す。
ルール20の箇条書きパターンは他のルールでも同様のインデント形式を使用している可能性があるため、より長い `old_string`（前後の行を含む複数行パターン）を使用して一意性を確保する。
一意性を確保するために、`old_string` に「既存テストが存在しない新規プロジェクトでは記録不要」の前行（ベースラインが未設定の場合）を含めることを推奨する。

## APIレスポンス設計

本タスクは CLAUDE.md の変更のみであり、ワークフロープラグインのコードには変更を加えない。
しかし、このタスクの完了後にワークフロープラグインの各 MCP ツールが返すレスポンスの期待値を以下に設計する。

### workflow_next のレスポンス（バリデーション成功時）

ui_design フェーズから test_design フェーズへの遷移時に成功する場合、workflow_next は次フェーズへの遷移確認メッセージを返す。
成功レスポンスには `phase: "test_design"` または同等の遷移確認フィールドが含まれる。
ui-design.md のバリデーションが成功した場合、artifact-validator が「検証合格」相当の内部状態を返し、workflow_next は正常遷移を実行する。
成功時のレスポンスには次フェーズのサブエージェントテンプレート（subagentTemplate）が含まれ、Orchestrator はそれを次のサブエージェント起動に使用する。

### workflow_next のレスポンス（バリデーション失敗時）

ui_design フェーズからの遷移試行時に ui-design.md のバリデーション失敗が検出された場合、workflow_next はエラーメッセージを返す。
失敗レスポンスには失敗したバリデーションルールの詳細（どのセクションが不足しているか、どの禁止語が検出されたか等）が含まれる。
失敗を受け取った Orchestrator は CLAUDE.md のルール21に従い、Edit ツールで直接修正するのではなく、Task ツールでサブエージェントを再起動してリトライプロンプトを渡すこと。
リトライの際には `workflow_next` の失敗レスポンスを `前回のバリデーション失敗理由` としてコードブロックに引用する。

### workflow_capture_baseline の制限（regression_test フェーズでの呼び出し禁止）

workflow_capture_baseline は testing フェーズまでに呼び出す必要があり、regression_test フェーズ開始後の呼び出しは本来の設計意図から外れる。
FR-15 の追記によって、Orchestrator は「calculatePhaseSkips によるセットスキップ設計」を認識するため、testing フェーズのスキップ時に regression_test での workflow_capture_baseline 呼び出しを試みる誤った動作が防止される。
forceTransition を使用して regression_test に直接遷移した場合は例外として workflow_capture_baseline の呼び出しが必要になるが、このケースは CLAUDE.md ルール20の警告（追記する6行目）に対処手順が記載される。
ベースラインが存在しない状態で regression_test から parallel_verification への遷移を試みると、workflow_next はベースライン必須のエラーを返し進行不能状態が発生する。

## 設定ファイル設計

CLAUDE.md のルール20は Orchestrator の動作指針を記述した設定ファイルとしての役割を担う。
以下に、追記後のルール20の構造設計を示す。

### ルール20の構造設計（6行の箇条書き形式）

追記後のルール20は見出し行を含めて7行で構成される。
1行目は見出し行として「20. **既存テストのベースライン記録義務（regression_test対応）**」という形式を使用する。
2行目から7行目は箇条書き形式で、それぞれ固有の役割を持つ内容を記載する。
2行目（既存）は testing フェーズでの標準的な記録手順を説明する行である。
3行目（既存）は research フェーズでのブロック理由と testing フェーズ推奨の根拠を説明する行である。
4行目（既存）はベースライン未設定時の影響を説明する行である。
5行目（既存）は新規プロジェクトでの例外ケースを説明する行である。
6行目（追記）は calculatePhaseSkips によるセットスキップ設計の事実を説明する行である。
7行目（追記）は forceTransition 使用時の潜在的リスクと対処法を警告形式で説明する行である。

### インデント形式の設計

CLAUDE.md のルール20では箇条書きのインデントにスペース4つとハイフンを組み合わせた形式を使用する。
この形式は「    - 」という7文字のプレフィックスであり、4つのスペースが見出し番号との視覚的な階層を表現する。
追記する6行目と7行目もこの同一形式を使用し、既存の5行との視覚的一貫性を保つ。
Markdown レンダリング時には、見出し番号（20.）に対してネストされたリストとして表示される。
インデント形式の統一は、Orchestrator が CLAUDE.md を読み取る際の構文的な整合性を確保するために重要である。

### 禁止語・角括弧を含まないことの確認ポイント

追記する2行の文言に禁止語が含まれないことを事前確認する。
英語グループの4語（成果物本文には記載しないが、コードフェンス外での出現を確認対象とする）が追記文言に存在しないこと。
日本語グループの8語（同様にコードフェンス外での出現を確認対象とする）が追記文言に存在しないこと。
角括弧文字（ASCII の左右角括弧）が追記する箇条書きテキスト内に存在しないこと。
確認は Read ツールで追記後のファイルを読み込んで目視確認するか、Grep ツールを使用してパターン検索することで実施できる。
spec.md の確定文言に基づいて追記を行う限り、これらの禁止要素は含まれない設計になっている。
