# 脅威モデル: CLAUDE.md ルール20補足追記（FR-15）

## サマリー

- 目的: CLAUDE.md ルール20への補足追記（FR-15）が Orchestrator の動作に悪影響を与えないことを脅威シナリオとして整理し、リスク対応方針を確立する
- 評価スコープ: 変更対象は CLAUDE.md のルール20（1箇所）のみであり、コード変更は一切行わない。next.ts・definitions.ts・state-manager.ts 等のコアモジュールは変更対象外であるため、実行時エラーの発生リスクはドキュメント変更の範囲に限定される
- 主要な決定事項: 変更対象は CLAUDE.md のみ（コード変更なし）であるため、実行時エラーのリスクは存在しない。主要リスクは不正確な記述による Orchestrator の誤解釈と、禁止パターン違反による成果物バリデーション失敗の2種類に集約される
- 検証状況: 脅威シナリオを4件識別した。脅威1（Orchestrator 誤動作）は Low-Medium、脅威2（後方互換性破壊）は Low、脅威3（禁止パターン違反）は Low、脅威4（角括弧混入）は Low と評価されており、全件が対処可能な低リスクの範囲内に収まっている
- 次フェーズで必要な情報: 脅威はいずれも低リスクと評価された。planning フェーズでは CLAUDE.md ルール20の正確な追記内容を設計し、後方互換性を維持した形で補足セクションを挿入すること

## 脅威シナリオ

### 脅威1: 不正確な記述による Orchestrator の誤動作

CLAUDE.md ルール20への追記内容が不正確または誤解を招く表現であった場合、Orchestrator がその記述を誤読して誤った行動をとるリスクがある。

具体的な誤動作パターンとして以下が考えられる。

パターンA として、セットスキップ説明の記述が不明確であった場合、Orchestrator が「testing フェーズをスキップしたら regression_test は個別にスキップせず実行するべき」と誤解し、不要な regression_test フェーズを実行しようとする可能性がある。これは実際には `calculatePhaseSkips` が両フェーズをセットで処理するため、誤解に基づく行動が実際の動作と矛盾してシステムエラーを引き起こす可能性がある。

パターンB として、forceTransition 警告の記述が不明確であった場合、Orchestrator が「forceTransition=true を使えばいつでも安全に regression_test に入れる」と誤解し、並行して parallel_verification に遷移できない状態に陥る可能性がある。この状態では `workflow_back` での差し戻しまたは `workflow_capture_baseline` の呼び出しが必要となるが、その対処法が追記されていない場合に進行不能状態が継続するリスクがある。

### 脅威2: 後方互換性の破壊による既存動作の変化

既存のルール20の4行の記述（testingフェーズでの記録標準・ベースライン未設定時の問題・新規プロジェクトの例外を含む）を誤って削除または変更した場合、既存の正常な Orchestrator 動作が破壊されるリスクがある。

既存記述が持つ役割として、「testingフェーズでの記録を標準とする」という指示が削除されると、Orchestrator がどのタイミングでベースラインを記録するべきか判断できなくなる。また「ベースライン未設定時の regression_test フェーズで比較できない」という警告が削除されると、Orchestrator がベースライン記録の重要性を認識できなくなる。さらに「既存テストが存在しない新規プロジェクトでは記録不要」という例外が削除されると、新規プロジェクトで不要な `workflow_capture_baseline` 呼び出しが発生する可能性がある。

これらの既存記述は現在の Orchestrator の正常動作を支える情報源であり、変更はリグレッションリスクを持つ。

### 脅威3: 禁止パターン違反による成果物バリデーション失敗

CLAUDE.md への追記内容が artifact-validator の禁止パターンを含む場合、文書ファイルとしての整合性は維持されるが、ワークフローフェーズの成果物（threat-model.md 等）でこれらの語句を誤って転記した場合にバリデーション失敗が発生するリスクがある。

英語グループの禁止語（4語）は通常の技術文書では頻繁には出現しないが、「現時点では確定されていない設計事項」を表現する際に意図せず含まれるケースがある。日本語グループの禁止語（8語）もドキュメント記述中に自然に現れやすいパターンを含む。

CLAUDE.md 本体は artifact-validator の検査対象外であるが、今回の threat-model.md を含む各フェーズの成果物は検査対象である。脅威3は本成果物（threat-model.md）自体の品質リスクとして識別する。

### 脅威4: 角括弧プレースホルダー混入による成果物拒否

追記内容の起草過程または成果物作成過程で、角括弧を用いたプレースホルダー表記（例えばコードフェンス外での文字クラス表記や変数参照表記）が混入するリスクがある。artifact-validator はコードフェンス外の角括弧をプレースホルダーとして検出し、成果物全体を拒否する。

CLAUDE.md への追記文中にコードブロックとして forceTransition の使用例を示す場合、コードフェンス内に記述すれば安全だが、散文テキスト内に直接記述すると違反となる可能性がある。

## リスク評価

### 脅威1のリスク評価

深刻度は中程度と評価する。誤解釈が発生した場合の影響範囲は testing/regression_test フェーズの実行判断に限定されるが、進行不能状態が発生する可能性があるため中程度と判断した。発生可能性は低い。追記内容を「通常は問題なし、forceTransition 使用時のみ注意」という明確な2段階メッセージ構造で記述することで、誤解釈のリスクを低減できる。

総合リスクレベルは低から中程度（Low-Medium）とする。適切な文章設計によってコントロール可能なリスクであり、追記のドラフトを planning フェーズで慎重に設計することで対処する。

### 脅威2のリスク評価

深刻度は中程度と評価する。既存動作が変化した場合の影響は、現在正常に動作している全ての Orchestrator インスタンスに及ぶ可能性がある。発生可能性は低い。追記を「補足サブセクションの挿入」形式とし、既存記述を変更しない方針が requirements.md で明確化されているため、このリスクは制御されている。

総合リスクレベルは低（Low）とする。既存テキストを削除または変更しないという明示的な制約がすでに requirements.md に記載されているため、実装時にこの制約を順守することで十分に対処可能である。

### 脅威3のリスク評価

深刻度は低程度と評価する。成果物バリデーション失敗が発生した場合は subagent のリトライで対処可能であり、データの損失や進行不能状態には至らない。発生可能性は低い。本成果物（threat-model.md）の作成時に禁止語リストを参照して語句を選定しているため、直接的なリスクは回避済みである。

総合リスクレベルは低（Low）とする。CLAUDE.md の禁止パターンリストを常に参照した語句選定で対処する。

### 脅威4のリスク評価

深刻度は低程度と評価する。角括弧混入が発生しても、subagent のリトライ機構で修正可能である。発生可能性は低い。コードフェンス内への記述ルールを順守し、散文テキスト内では角括弧を一切使用しない方針を徹底する。

総合リスクレベルは低（Low）とする。

## セキュリティ要件

### SR-1: 追記内容の正確性保証

CLAUDE.md ルール20への追記内容は、`calculatePhaseSkips` の実際の動作（testing と regression_test のセットスキップ）を正確に反映した記述とすること。推測や「おそらく」といった曖昧な表現は使用しない。research フェーズで確認した実装の事実に基づいた記述のみを使用する。

追記内容は AI が読んで即座に行動方針を判断できる明確な2段階構造（通常動作の説明と forceTransition 使用時の警告）とすること。これにより脅威1のリスクを低減する。

### SR-2: 既存記述の保全義務

CLAUDE.md ルール20の既存4行の記述は変更・削除しないこと。追記は「補足」として独立したサブセクションまたは箇条書きで挿入すること。変更前後の diff を planning フェーズで明示し、削除がないことを確認してから実装フェーズに進むこと。これにより脅威2のリスクを低減する。

### SR-3: 禁止パターン準拠

本フェーズの成果物（threat-model.md）および CLAUDE.md への追記内容において、artifact-validator が検出する禁止語（英語グループ4語・日本語グループ8語、合計12語）をコードフェンス外に含めないこと。追記内容の起草時には禁止パターンのリストを事前確認し、代替表現を使用すること。これにより脅威3のリスクを低減する。

### SR-4: 角括弧使用禁止の遵守

成果物および CLAUDE.md への追記において、コードフェンス外での角括弧使用を禁止する。コードフェンス内であれば `forceTransition: true` のようなコード表記を安全に記述できる。散文テキスト内では角括弧を使用せず、「波括弧のみ許可」のルールを徹底する。これにより脅威4のリスクを低減する。

### SR-5: 変更範囲の最小化による攻撃面の抑制

今回の変更を CLAUDE.md のルール20の補足追記のみに限定する。コアモジュール（next.ts、definitions.ts、state-manager.ts）への変更は今回スコープ外とする。変更範囲を最小化することで、意図しない副作用の発生リスクを抑制し、レビューコストも最小化する。この制約は requirements.md の NFR-1 として明文化されており、本セキュリティ要件でもその方針を引き継ぐ。
