## サマリー

BUG-4変更箇所のパフォーマンス観点における静的解析を実施した。
対象は `spec-parser.ts` の正規表現処理、`definitions.ts` の `calculatePhaseSkips` 関数、
および新規追加された `calculate-phase-skips.test.ts` のテスト実行時間の理論値評価である。
主要な決定事項として、各変更箇所に重大なパフォーマンス上の問題は検出されなかった。
`spec-parser.ts` の正規表現は線形時間で動作し、`calculatePhaseSkips` は小規模な配列走査のみを行う設計である。
次フェーズで必要な情報として、ボトルネックは存在せず、追加の最適化は現時点では不要と判断する。

## パフォーマンス計測結果

### 計測項目1: spec-parser.ts の正規表現パフォーマンス

`spec-parser.ts` には BUG-4修正として `removeCodeBlocks` 関数と `cleanedForMethods` の分離が導入された。

`removeCodeBlocks` 関数が使用する正規表現はトリプルバッククォートで囲まれたコードブロック全体にマッチするパターンである。
この正規表現は遅延量指定子を使用しており、最短一致で動作する。
通常の Markdown 文書（数千文字以下）に対しては、マッチング処理がほぼ線形時間で完了する。
コードブロックが存在しない場合もスキャンは一度の線形走査で終了するため、無駄な再走査は発生しない。

具体的な正規表現パターンは以下のコードフェンスで示す:

```
/\`\`\`[\s\S]*?\`\`\`/g
```

`cleanedForMethods` で使用される正規表現はインラインコードスパン（バッククォート囲み）にマッチするパターンである。
この正規表現は否定文字クラスによる貪欲マッチで、バックトラッキングが最小化される設計となっている。

インラインコード検出の具体的なパターンは以下の通り:

```
/\`[^\`]+\`/g
```
インラインコードスパンのネストは Markdown 仕様上許容されないため、最悪ケースでも線形時間の動作が保証される。

メソッド抽出に使用される正規表現は、関数名・引数リスト・戻り値型を抽出するパターンである。
この正規表現は否定文字クラスによりカッコ内のバックトラッキングを抑制している。
具体的なパターンは以下の通り:

```
/(?:def\s+)?(\w+)\s*\([^)]*\)\s*(?::\s*\w+)?\s*[{=]/g
```

関数引数リストが非常に長い場合でも一度の線形走査で処理が完了する。

### 計測項目2: calculatePhaseSkips の計算量

`calculatePhaseSkips` 関数は `files` 配列を最大2回走査する設計である。
一度目の走査は `hasCodeFiles` の判定（`Array.some`）で、マッチが見つかった時点で打ち切られる。
二度目の走査は `hasTestFiles` の判定（`Array.some`）で、同様に早期終了が保証される。

各ファイルパスに対して実行される処理は文字列分割（`split('.')`）と正規表現マッチのみである。
ファイル拡張子判定の正規表現は以下の通り:

```
/\.(test|spec)\.(ts|tsx|js|jsx)$/
```
正規表現は文字列末尾に固定されたパターン（`$` アンカー）のため、マッチング時間は事実上定数時間に近い。

`TEST_KEYWORDS` と `IMPL_KEYWORDS` の走査は4要素と4要素の固定長配列であり、計算量は定数（O(1)）とみなせる。
`userIntent.toLowerCase()` を一度計算してから配列走査に使用するため、無駄な文字列変換は発生しない設計となっている。

### 計測項目3: テスト実行速度の理論値

`calculate-phase-skips.test.ts` は純粋関数テストで構成された7テストケースを含む。
外部依存（ファイルシステム、ネットワーク、非同期処理）を一切持たないため、テスト実行時間は関数の計算時間のみに依存する。
各テストケースで `calculatePhaseSkips` が呼び出す処理は数回の配列走査と文字列操作のみで、1回あたりの実行時間は1ミリ秒未満と推定される。
7テストケースすべての合計実行時間は理論上10ミリ秒以内に収まる見込みである。

`skip-env-removal.test.ts` は `vi.mock` によるモック設定を含むが、これは Vitest のコンパイル時変換で処理されるため、
実行時のオーバーヘッドは最小限に抑えられる。ファイルシステムモックは `vi.mocked(fs.existsSync)` 等の軽量な代替関数として動作し、
ディスクI/Oが一切発生しない設計となっている。

## ボトルネック分析

### 正規表現の理論的最悪ケース

`removeCodeBlocks` で使用される正規表現は、コードブロックが閉じられていない（開きバッククォートのみ）Markdown 入力に対して、
文書末尾まで走査を続ける可能性がある。
具体的には、以下のパターンで:

```
/\`\`\`[\s\S]*?\`\`\`/g
```

ただし MCP サーバーの実際の使用文脈では、入力は `spec.md` 等の構造化されたワークフロー成果物に限定される。
これらのファイルは数百行から数千行の Markdown であり、最悪ケースでも数ミリ秒以内で処理が完了すると推定される。

### calculatePhaseSkips のファイル数スケーリング

`files` 配列が非常に大きい場合（数千件以上）、`Array.some` の線形走査が相対的に長くなる可能性がある。
ただし、ワークフロースコープ設定の実際の使用パターンでは、`files` に含まれるエントリ数は通常10件から100件程度に収まる。
この規模では `Array.some` の実行時間は無視できるレベル（マイクロ秒オーダー）である。

### 全体評価: ボトルネックなし

BUG-4変更箇所のすべてにおいてパフォーマンス上の問題は検出されなかった。
`removeCodeBlocks` と `cleanedForMethods` の分離（旧実装では `cleanedMarkdown` に対してインラインコード除去を省略していた）は、
メソッド抽出精度を向上させる目的で導入されたものであり、追加の正規表現走査は1回のみで計算量への影響は軽微である。
`calculatePhaseSkips` の `userIntent` 対応は定数長の配列走査の追加のみであり、性能への影響はない。
新規テストファイル `calculate-phase-skips.test.ts` の追加は、テストスイート全体の実行時間を数十ミリ秒以内増加させる程度にとどまる。
追加の最適化は現時点では不要と結論づける。
