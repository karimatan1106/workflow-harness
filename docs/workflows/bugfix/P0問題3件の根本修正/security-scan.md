## サマリー

本セキュリティスキャンは、workflow-pluginのP0修正3件に対する脆弱性検証を実施しました。対象は以下の3ファイルです：

- workflow-plugin/mcp-server/src/tools/next.ts: スコープ警告ロジック追加（P0-1）
- workflow-plugin/hooks/lib/discover-tasks.js: アトミック書き込みパターン（P0-3）
- workflow-plugin/mcp-server/src/state/types.ts: NextResult型拡張（P0-1）

主要な安全性確認項目は、入力検証、ファイル操作の安全性、HMAC整合性、クライアント処理の安全性、リソース管理です。検証の結果、各修正は適切なセキュリティ対策を含んでいることが確認されました。

次フェーズでは、本スキャンで指摘した警告項目（環境変数検証の強化、ファイル権限の確認）について、運用面でのリスク軽減を検討することが推奨されます。

## 脆弱性スキャン結果

### 検査対象ファイル分析

#### 1. next.ts（スコープ警告ロジック）

**検査項目**: scopeWarningsフィールド（行148-180）の入力検証と出力安全性

検査内容としては、scopeWarningsは taskState.scope から直接取得した値を格納する配列です。ここで検証された重要なポイントは、以下の通りです：

- taskState.scope は既に stateManager.readTaskState() で読み込まれた検証済みオブジェクト
- JSON スキーマ validation が workflow-state.json に対して実施済み（タスク開始時）
- scopeWarnings 配列要素は文字列リテラル（コンストラクタで静的生成）のみ追加
- ユーザー入力から直接的に値が追加される経路は存在しない

**セキュリティ評価**: 入力値は既に複数段階の検証を通過しており、インジェクション攻撃のリスクは低い

#### 2. discover-tasks.js（アトミック書き込みパターン）

**検査項目**: tmpFile名生成（行131）と rename 失敗時の処理（行135-141）

ファイル操作の安全性を確認した結果：

- tmpFile名は process.pid を含む形式（`TASK_INDEX_FILE + '.' + process.pid + '.tmp'`）を採用
- PID予測攻撃について、Unix環境では process.pid は全プロセスに一意であり、同一プロセス内での同時書き込みは技術的に不可能（Node.js のシングルスレッド特性）
- tmpFile の write および rename は fs モジュールのシステムコール経由で実行され、ファイルディスクリプタレベルでの原子性が保証される
- rename 失敗時の catch ブロック（行135）で unlinkSync による削除を試行するため、ゾンビファイル発生のリスクが軽減されている

**セキュリティ評価**: アトミック書き込みパターンは適切に実装されている

#### 3. types.ts（NextResult型拡張）

**検査項目**: warnings フィールド（line 493）の型安全性とクライアント処理

拡張された NextResult 型を検証：

- warnings は optional な string[] 配列として定義（行493）
- API レスポンス生成時に sprintf や string concatenation ではなく配列の直接構築（next.ts 行639）
- クライアント側では JSON deserialize 後に配列の各要素として単純な文字列アクセスのみ実行される

**セキュリティ評価**: 型定義は安全であり、クライアント処理も予測可能

## 検出された問題

### 警告レベル（運用改善推奨）

**警告 #1: 環境変数検証の強化**

- 問題箇所: next.ts 行33-39（MAX_SCOPE_FILES, MAX_SCOPE_DIRS の parseInt）
- 詳細: parseInt() の第3引数（基数）が指定されているため基数解析の脆弱性はないが、TASK_INDEX_TTL_MS（discover-tasks.js 行65）の parseInt では基数が未指定
- リスク程度: 低（デフォルトは10進数と判断される環境がほとんど）
- 推奨対応: parseInt(value, 10) を明示的に指定して防御的プログラミングを徹底

詳細な追加情報として、環境変数が意図しない基数（例：0x で始まる16進数表記）として解釈される可能性を排除するため、基数を明示することが業界標準です。

**警告 #2: tmpFile削除時のエラーハンドリング**

- 問題箇所: discover-tasks.js 行137-141（fs.unlinkSync の catch ブロック）
- 詳細: unlink 失敗時の catch ブロックで処理が無視される。システムディスク満杯時等にゾンビファイルが残存する可能性
- リスク程度: 低（ディスク満杯は稀な状況。ワークフロー機能に直接的な影響なし）
- 推奨対応: ログ記録を追加（console.warn）して、ディスク容量監視との組み合わせで事前警告

実運用では、定期的なクリーンアップスクリプト（古い .tmp ファイル削除）の導入で十分です。

**警告 #3: HMAC検証の再確認**

- 問題箇所: next.ts 全体（state integrity の依存関係）
- 詳細: scopeWarnings は stateIntegrity フィールドの計算対象に含まれるか不明
- 確認結果: warnings フィールドは直接的には stateIntegrity に含まれず（state.warnings ではなく response.warnings）、レスポンスのみの出力であるため整合性破損のリスクはなし
- 推奨対応: 追加アクション不要。現状設計で HMAC 保護は維持される

## セキュリティ対策の確認

### インジェクション攻撃防止

scopeWarnings で使用される全メッセージはコード内で静的に定義されており、ユーザー入力や外部データは含まれません。各警告メッセージを確認した結果：

- 行178のメッセージ「スコープが設定されていません...」
- 行210のメッセージ「スコープが設定されていません（parallel_analysis時）...」

いずれも文字列リテラルであり、動的生成される部分がないため SQL インジェクション、コマンドインジェクション、XSS 等の攻撃経路が存在しません。

### ファイルシステムセキュリティ

アトミック書き込みパターンの実装により、以下の脅威が軽減されています：

- TOCTOU（Time-of-Check-Time-of-Use）: 一時ファイル作成と rename が不可分に実行される
- ファイル置き換え攻撃: 既存 task-index.json は rename 直前まで保護される
- 部分的な書き込み: ディスク満杯時も task-index.json は不完全な状態に遷移しない

권장사항として、task-index.json の ファイル権限は読み取り専用（0644以下）に設定し、workflow-plugin ディレクトリ全体の所有者を特定の実行ユーザーに限定することで、追加の保護層が実現できます。

### クライアント処理の安全性

warnings フィールドの JSON シリアライズ・デシリアライズプロセスを確認：

1. サーバー側（next.ts）: string[] 配列を直接レスポンスに含める
2. JSON エンコード: 標準の JSON.stringify で処理（改行やダブルクォートは自動エスケープ）
3. クライアント側: JSON パース後、配列の各要素を順番にログ出力

一連のプロセスで外部エンティティの挿入経路がないため、XSS リスクは最小化されています。

## 結論

**セキュリティ評価の総括**: P0修正3件は全体的にセキュリティに配慮した設計となっており、一般的な脆弱性（インジェクション、TOCTOU、権限昇格）に対する防御が適切に実装されています。特にスコープ警告ロジックではユーザー入力から直接的に値が追加される経路が存在せず、全メッセージがコード内で静的に定義されているため、インジェクション攻撃のリスクが最小化されています。

**検出された脆弱性レベル**: 実装上の重大な欠陥（Critical/High レベル）は検出されませんでした。本スキャンで指摘した項目は全て警告レベル（Low/Medium）であり、運用面での改善推奨として位置づけられます。次フェーズでの対応により、セキュリティポスチャーがさらに強化されます。

**修正適用の効果**: 本修正の適用により、workflow-plugin のスコープ管理とファイル I/O の信頼性が向上することが期待されます。特にアトミック書き込みパターンの導入により、並行処理時のデータ競合リスクが大幅に軽減されます。tmpFile の write および rename がシステムコール経由で原子性を持つことで、TOCTOU 攻撃やファイル置き換え攻撃が効果的に防止されます。

**総合リスク評価**: 本修正に伴うセキュリティリスクは全項目において許容範囲内（Low/Medium）と判定されました。Critical/Highレベルの脆弱性は検出されておらず、既存のHMAC保護やJSON スキーマ検証との組み合わせにより、実運用環境での安全性が十分に確保されています。

**推奨フォローアップアクション**: セキュリティの継続的改善のため、以下のアクションを次フェーズで実施することを推奨します。環境変数検証では parseInt(value, 10) の基数を明示的に指定し、基数解析脆弱性を排除してください。ファイル権限では task-index.json を読み取り専用（0644以下）に設定し、workflow-plugin ディレクトリ全体の所有者を特定の実行ユーザーに限定してください。ディスク容量監視では定期的なクリーンアップスクリプト（古い .tmp ファイル削除）を導入し、ゾンビファイル発生を防止してください。これらにより、本修正で実現されたセキュリティ向上を維持・強化できます。
