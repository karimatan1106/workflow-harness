# P0問題3件の根本修正 - 脅威モデル

## サマリー

本ドキュメントはworkflow-pluginのP0問題3件（P0-1: スコープ設定促進、P0-2: バリデーション全フェーズ拡張、P0-3: task-index.jsonアトミック書き込み）の修正実装に対する脅威モデルを定義する。

### 目的

各P0修正が新たなリスクを生じさせないことを確認し、実装時の注意点を特定する。

### 主要な決定事項

- P0-1の脅威: スコープ設定促進は完全なブロックではなく警告のため、偽陰性（スコープ未設定のままフェーズ進行）は意図的な仕様
- P0-2の脅威: バリデーション追加は偽陽性（正当な成果物のブロック）リスクを持ち、既存成果物フォーマットの互換性検証が必須
- P0-3の脅威: アトミック書き込みのフォールバック動作が不完全な場合、一時ファイルがディスクに蓄積するリスクがある

### 次フェーズで必要な情報

- P0-2の偽陽性を避けるため、state-machine.mmdとflowchart.mmdに対してartifact-validatorが何を検証するかを事前に確認する
- P0-3の孤立一時ファイルクリーンアップロジックをtask-index.json読み込み時ではなく書き込み時に実行するか検討する

---

## 脅威の全体像

各P0修正は独立したコンポーネントを変更するが、修正の副作用が他のコンポーネントに伝播するリスクがある。以下のコンポーネント間の依存関係を考慮して脅威を評価する。

修正対象となる3件の問題はそれぞれ異なるレイヤーに存在するが、workflow-pluginというシステム全体を通じて相互に影響し合うため、個別の脅威だけでなくシステム全体としての整合性リスクも評価する必要がある。

P0-1の修正（スコープ設定促進）はresearch フェーズのサブエージェントテンプレートを変更し、スコープ設定呼び出し指示を追加することで、手動呼び出しの煩雑性を軽減する。P0-2の修正（バリデーション全フェーズ拡張）は PHASE_TO_ARTIFACT と SUBPHASE_TO_ARTIFACT 定義を追加し、既存フェーズのバリデーション対象をレイアウト化する。P0-3の修正（アトミック書き込み）は task-index.json の write-then-rename パターンを実装し、concurrent write による破損リスクを低減する。これら3つの修正は独立しているが、next.ts と discover-tasks.js を共有ファイルとして参照するため、修正の依存関係が生じている。

```
definitions.ts (P0-1)
    ↓ phaseGuide.checklist, subagentTemplate
next.ts (P0-1, P0-2, P0-3)
    ↓ PHASE_TO_ARTIFACT, syncTaskIndex
complete-sub.ts (P0-2)
    ↓ SUBPHASE_TO_ARTIFACT
discover-tasks.js (P0-3)
    ↓ writeTaskIndexCache
artifact-validator.ts (P0-2で呼び出し)
```

特に P0-2 と P0-3 の組み合わせが重要である。P0-2 はバリデーション拡張により成果物の品質チェックを厳密化し、P0-3 はそのバリデーション実行結果を task-index.json に記録するため、両者の機能が緊密に連動する。バリデーション失敗時の再試行ロジックも両者に影響し、一方の変更が他方の動作を変える可能性がある。

また、P0-1のスコープ設定促進は、上記の依存グラフにおいて入力側（research フェーズ）に位置するため、スコープが正しく設定されない場合にはP0-2のバリデーション対象ファイルが存在しないというエラーを引き起こす二次的影響がある。実装順序はP0-3（アトミック書き込み）→P0-2（バリデーション拡張）→P0-1（スコープ促進）の順が安全であり、基盤となるファイル書き込みの安全性を確保してから品質ゲートを強化し、最後にサブエージェントの誘導を改善するという順序で影響範囲を最小化できる。

## P0-1の脅威分析（スコープ設定促進）

### 脅威T1-1: researchサブエージェントがスコープ設定を無視する

**発生条件:**
subagentTemplateの末尾にworkflow_set_scope呼び出し指示を追加しても、サブエージェントがLLM的な判断で「このタスクにはスコープが不要」と判断し、呼び出しをスキップする場合がある。

**影響:**
スコープが空のままparallel_analysisフェーズに到達し、既存のブロックガードが発動してフェーズ遷移が妨げられる。ユーザーは手動でworkflow_set_scopeを呼び出す必要が生じる。

**深刻度:** 中程度。既存の動作（parallel_analysisでブロック）と同じため、修正前と比べて悪化はしない。

**緩和策:** checklistにスコープ確認項目を追加することで、サブエージェントがチェックリスト消化を重視する場合に呼び出し率が向上する。

### 脅威T1-2: スコープの過大設定によるパフォーマンス劣化

**発生条件:**
sonnetモデルは推論精度が高いが、userIntentのキーワードが曖昧な場合（「ワークフロー全体の修正」等）、プロジェクト全体を影響範囲として設定する可能性がある。

**影響:**
affectedDirsにプロジェクト全ディレクトリが登録されると、test_implフェーズの選択でテスト対象が過剰に広がり、テスト実行時間が増大する。ループ検出フックが誤作動するリスクも生じる。

**深刻度:** 低。スコープはtest_implのファイル選択に影響するが、テスト実行自体をブロックするわけではない。

**緩和策:** subagentTemplateに「変更対象ファイルに限定してスコープを設定すること」と「ディレクトリ指定よりファイル指定を優先すること」を明示する。

### 脅威T1-3: スコープの過小設定によるtest_implスキップ継続

**発生条件:**
sonnetへのモデル引き上げにより精度は向上するが、ファイル名の命名規則が非標準的なプロジェクトでは関連ファイルを特定できず、影響範囲が狭すぎるスコープが設定される。

**影響:**
テストファイルが影響範囲に含まれない場合、test_implフェーズがスキップされるという元のP0-1問題が再現する。

**深刻度:** 中程度。P0-1の根本原因が解消されず、修正効果が不完全になる。

**緩和策:** subagentTemplateにGlobパターン指定の例（glob: "workflow-plugin/mcp-server/src/phases/*.ts"）を具体的な書式で示し、パターンマッチを優先させる。

### 脅威T1-4: sonnetモデル引き上げによるコスト増加

**発生条件:**
researchフェーズのモデルをhaikuからsonnetに変更することで、API呼び出しコストがトークン単価比で5～10倍程度増加する。大量のタスクを実行するユーザーや長大なuserIntentを持つタスクでは顕著になる。

**影響:**
ユーザーのAPI費用が増加し、ワークフローの経済的利用を妨げる可能性がある。

**深刻度:** 低。researchフェーズは通常1回のみ実行され、費用対効果（スコープ精度向上）はおおむね正当化できる。

**緩和策:** 脅威としては認識するが、要件上の意思決定（コスト増加を許容してsonnetを使う）は既に確定しており、追加緩和策は実装しない。

## P0-2の脅威分析（バリデーション全フェーズ拡張）

### 脅威T2-1: PHASE_TO_ARTIFACT拡張による偽陽性（正当な成果物のブロック）

**発生条件:**
planning→threat_modeling フェーズに新しくバリデーションが追加されると、既存の成果物フォーマット（たとえば行数が少ない短い脅威モデルや、必須セクションの見出し文字列が微妙に異なる）がバリデーションに失敗する。

**影響:**
既存タスクを持つユーザーが、修正前に作成済みの脅威モデルに対してworkflow_nextを呼び出した際にフェーズ遷移がブロックされる。ユーザーは手動でファイルを修正するか、バリデーションを回避する手段を探す必要が生じる。

**深刻度:** 高。既存ユーザーの作業継続を阻害し、プラグインへの信頼を損なう。

**緩和策:**
- PHASE_TO_ARTIFACTに追加したフェーズが参照するartifact-validatorのルールを事前に確認し、既存の成果物フォーマットとの互換性を検証する。
- planning/threat_modelingの検証ルールに必須セクション要件を設定する場合は、過去の成果物フォーマットとの整合性を確認してから設定する。

### 脅威T2-2: SUBPHASE_TO_ARTIFACTのcomplete-sub.ts統合における不整合

**発生条件:**
complete-sub.tsに新設するSUBPHASE_TO_ARTIFACTのファイルパス構築ロジックが、実際のdocsDirの解決方法と異なる場合、バリデーション対象ファイルが見つからずエラーになる。

**影響:**
workflow_complete_subが「ファイルが存在しない」エラーを返し、サブフェーズが完了できなくなる。サブエージェントは成果物を正しく作成したにもかかわらず、完了できないデッドロック状態に陥る。

**深刻度:** 高。デッドロックが発生するとworkflow_resetで最初から実行し直す必要がある。

**緩和策:**
- docsDirの解決ロジックはnext.tsとcomplete-sub.tsで同一の関数またはユーティリティを使用すること。
- checkSubPhaseArtifacts実装時にファイルが存在しない場合のハンドリングを明示的に定義する（「ファイルなし=検証スキップ」か「ファイルなし=エラー」かを事前に決定する）。

### 脅威T2-3: .mmdファイルへのartifact-validatorの誤適用

**発生条件:**
state-machine.mmdやflowchart.mmdは専用のMermaid記法を持つファイルである。artifact-validatorがMarkdown専用のルール（必須セクション、セクション密度等）をこれらのファイルに適用した場合、正当なMermaidファイルが「不正なMarkdown」と判定される。

**影響:**
state_machineやflowchartサブフェーズのcomplete-sub呼び出しが常にブロックされ、これらのサブフェーズが完了不可能になる。

**深刻度:** 致命的。設計フェーズ全体が永続的にブロックされる。

**緩和策:**
- .mmdファイルはartifact-validatorの検証スコープ外とすることを明示的に定義する。
- SUBPHASE_TO_ARTIFACTにstate_machineとflowchartを含める場合は、これらのエントリに対してバリデーション実行をスキップする条件を追加する。または、.mmdファイル専用の軽量バリデーターを用意する（最低行数のみを確認する等）。

### 脅威T2-4: semantic-checkerのCRLF誤検出がブロッキングモードで拡大

**発生条件:**
validateSemanticConsistencyの呼び出し対象フェーズをコメントで明確化した後、将来の拡張でCRLF環境（Windows環境）での改行n-gramが誤検出される問題が新フェーズに波及する。

**影響:**
CRLF環境で実行しているユーザーのフェーズ遷移が、意味的整合性チェックの誤検出によりブロックされる。

**深刻度:** 低。このタスクでは追加の意味的チェック実装はスコープ外であるため、既存の誤検出範囲（requirementsとspec）以外への拡大は発生しない。

**緩和策:** 現タスクのスコープ（コメントによる明確化のみ）に留め、意味的チェックの実行フェーズ拡大は別タスクとして扱う。

## P0-3の脅威分析（アトミック書き込みとMCP同期）

### 脅威T3-1: 一時ファイル残留によるディスク消費の蓄積

**発生条件:**
writeTaskIndexCacheがwrite-then-renameパターンを実装した後、MCP serverプロセスがwriteFileSyncとrenameSync の間でクラッシュした場合、`task-index.json.{pid}.tmp`という一時ファイルがディスクに残留する。長期間稼働するシステムでタスクが頻繁に実行されると、複数の孤立一時ファイルが蓄積する。

**影響:**
ディスク消費量は通常軽微（1ファイル=数KB程度）だが、クリーンアップが行われない場合、多数のタスク実行後に数百KBのゴミファイルが残る。

**深刻度:** 低。データ整合性への影響はなく、ディスク消費量も軽微。

**緩和策:**
- 要件書に記載通り、discoverTasksの次回呼び出し時に「作成から60秒以上経過したtmpファイル」を検出・削除するクリーンアップを実装する。
- クリーンアップはtask-index.jsonと同ディレクトリのglob(`*.tmp`)で孤立ファイルを探索する実装が適切。

### 脅威T3-2: rename失敗時のフォールバック動作の欠如

**発生条件:**
WindowsとLinux/macOSでfs.renameSync/fs.renameの動作が異なる。Windowsでは書き込み先ファイルが別プロセスによって読み取り中の場合、renameが`EBUSY`エラーで失敗する場合がある（現環境はWindows環境）。

**影響:**
renameが失敗した場合、要件書の仕様では「一時ファイルを削除してエラーをスロー」とされているが、呼び出し元のtry-catchがこのエラーを握りつぶすと、古いtask-index.jsonがそのまま残り更新が無視される（サイレントフェイルアー）。

**深刻度:** 中程度。Windowsでのみ発生し得る問題で、フェーズ遷移は成功するがtask-index.jsonが更新されないため、フックが古いフェーズを参照し続けるFIX-1問題が再現する。

**緩和策:**
- Windows環境での動作確認を実施する。
- renameが`EBUSY`で失敗した場合のみ、フォールバックとしてfs.writeFileSyncによる直接書き込みを試みる実装を検討する（完全なアトミック性は失われるが、データ消失よりはサイレントフェイルアーが望ましい）。
- エラーログにrename失敗を記録し、デバッグを容易にする。

### 脅威T3-3: syncTaskIndex関数のMCP→task-index.json同期タイミング問題

**発生条件:**
next.tsのsaveState呼び出し後にsyncTaskIndex関数を追加しても、syncTaskIndexがtask-index.jsonを読み込む時点でdiscover-tasks.jsが同時に書き込みを実行していた場合、読み取ったJSONが中途半端な状態になる。

**影響:**
syncTaskIndexがJSON.parseに失敗してクラッシュし、フェーズ遷移のレスポンスが返されないか、タスクIDの更新が失敗してtask-index.jsonが壊れる。

**深刻度:** 中程度。P0-3のアトミック書き込み修正後は、読み取り時に中途半端なJSONが見える確率は大幅に減少する。しかしゼロにはならない。

**緩和策:**
- syncTaskIndex内のJSON.parseをtry-catchで囲み、パース失敗時はtask-index.json更新をスキップして正常終了する（MCPサーバーのフェーズ遷移自体は成功させる）。
- syncTaskIndex内でもREQ-P03-1のアトミック書き込みを使用し、書き込み側の安全性を確保する。

### 脅威T3-4: start.tsへのsyncTaskIndex追加による新規タスク登録競合

**発生条件:**
タスク開始時にstart.tsがsyncTaskIndexを呼び出してtask-index.jsonに新規タスクを追加するが、discover-tasks.jsが同じ時刻にタスク一覧をスキャンして全体を書き直す場合、start.tsが追加したエントリが上書きされて消える。

**影響:**
新規タスクがtask-index.jsonに存在しない状態で次のhook呼び出しが発生すると、enforce-workflowフックがアクティブなタスクを認識できずコマンドをブロックする。

**深刻度:** 中程度。P0-3のアトミック書き込みで書き込みの競合は低減されるが、discover-tasks.jsはスキャン結果で全体を書き直すため、start.tsの追記とdiscover-tasks.jsの全書き換えは本質的に競合する。

**緩和策:**
- start.tsへのsyncTaskIndex追加は「discover-tasks.jsが次回スキャン時に新規タスクを拾うまでのギャップを埋める」目的に限定する。
- discover-tasks.jsのスキャン周期（現在はキャッシュTTL=1時間）の開始直後にタスクが作成された場合でも、次回のhook呼び出しまでに最大1時間のギャップが生じることをドキュメント化する。

## 後方互換性の脅威分析

### 脅威T4-1: PHASE_TO_ARTIFACT拡張による既存進行中タスクへの影響

**発生条件:**
修正リリース前から進行中のタスクが、planningまたはthreat_modelingフェーズを完了した状態でworkflow_nextを呼び出した場合、新しくバリデーションが適用されるフェーズに対して過去に作成した成果物の品質チェックが走る。

**影響:**
過去のspre.mdやthreat-model.mdが新しいバリデーション基準を満たさない場合、進行中タスクが突然ブロックされる。

**深刻度:** 中程度。進行中タスクを持つユーザーへの直接的な影響であり、アップグレード後に即座に問題が発覚する。

**緩和策:**
- PHASE_TO_ARTIFACT拡張による追加バリデーションは、必須セクションの最低限の要件のみに留め、過去のドキュメントスタイルを尊重した緩やかなルールから始める。
- バリデーション追加後の最初のメジャーリリースでは変更点をリリースノートに明記する。

### 脅威T4-2: complete-sub.tsの変更によるカスタムサブフェーズへの影響

**発生条件:**
SUBPHASE_TO_ARTIFACTに登録されていないカスタムサブフェーズ名（例: プラグインを拡張したユーザーが追加したサブフェーズ）をworkflow_complete_subで完了させようとした場合、checkSubPhaseArtifactsが予期しない動作をする。

**影響:**
カスタムサブフェーズがバリデーションエラーで完了不可能になる、またはSUBPHASE_TO_ARTIFACTにエントリがないためスキップされるかによって動作が変わる。

**深刻度:** 低。標準のワークフローには影響しない。プラグインをカスタマイズしているユーザーのみが影響を受ける。

**緩和策:** checkSubPhaseArtifactsはSUBPHASE_TO_ARTIFACTにエントリが存在しないサブフェーズは検証をスキップする実装とし、未知のサブフェーズを許容する。

## 実装への推奨事項

上記の脅威分析を踏まえ、実装フェーズで特に注意すべき事項を以下に列挙する。

**P0-2実装において最優先の対処が必要な事項:**
SUBPHASE_TO_ARTIFACTにstate_machineとflowchartを含める場合、artifact-validatorが.mmdファイルに対してMarkdownルールを適用しないよう拡張子チェックを追加すること（T2-3対応）。具体的には checkSubPhaseArtifacts 関数内で PHASE_TO_ARTIFACT のphaseキー値からファイルパスを取得した際、その拡張子が .mmd である場合はバリデーションをスキップし、Markdown検証ルールは .md ファイルのみに適用する条件分岐を追加する必要がある。この対処を怠ると設計フェーズ全体が永続的にブロックされる最も深刻なリスクとなる。

**P0-3実装においてWindows環境での動作確認が必須である事項:**
fs.renameSync のEBUSYエラーはWindowsでのみ発生し得るため、現在のWindows開発環境で実際にwrite-then-renameが正常動作するかをtest_implフェーズで検証する必要がある（T3-2対応）。writeTaskIndexCache 関数の実装時に、rename失敗時のフォールバック処理として fs.writeFileSync による直接上書きを追加し、エラーログに記録する実装を含めること。Windows環境での EBUSY エラーが実際に発生する条件を test_impl で再現テストすることが重要である。

**後方互換性を確保するための実装方針:**
新規追加するバリデーションルールは最低限の要件（ファイルの存在確認、最低行数、必須セクション）にとどめ、過去の成果物が新しいルールで無効化されないことを確認してからリリースすること（T4-1対応）。PHASE_TO_ARTIFACT に planning と threat_modeling を追加する際は、既存タスクの spec.md と threat-model.md がそれぞれのバリデーション基準を満たしているかを事前に検査し、基準を満たさない場合は緩和策（セクション密度チェック除外等）を実装する。バリデーション追加後の最初のリリースではリリースノートに互換性に関する注記を含める。

**実装優先度のまとめ:**
脅威マトリクスのP0優先度（T2-3）は実装着手前に必ず解決すること。P1優先度（T2-1、T2-2）は実装フェーズ中に対処し、P2優先度（T3-2、T3-3等）は実装フェーズとテストフェーズにまたがって検証する。P3優先度は今回のスコープ内で対処可能な範囲で対処し、残るものは別タスクとして記録する方針を採る。実装チームは上記の緩和策を実装計画に盛り込み、test_designフェーズでそれぞれの脅威に対応するテストケースを設計することが求められる。
