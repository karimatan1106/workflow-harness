## サマリー

このドキュメント更新フェーズでは、実装済みの3つの修正依頼（FR-1、FR-2、FR-3）に関連する内部ドキュメントと開発規約を整備した。具体的には、MEMORY.md への重要なガイドライン追記、MCP サーバー内の subagentTemplate スリム化処理の実装追加、及び parallel_verification フェーズの各サブフェーズテンプレートへの評価結論重複回避ガイダンスの拡充を実施した。これにより、Orchestrator と subagent 間のコンテキスト引き継ぎが効率化され、複数のテストシナリオ報告時の重複行エラーが削減される。

## 修正内容の全体像と根拠

今回のタスクでは、workflow-plugin システムの核となる3つの領域における改善を実装した。各修正は前回フェーズで特定された根本原因に対する直接的な対応であり、以下の背景に基づいている。

### 背景：workflow_status の subagentTemplate 返却廃止の決定

Fix 2（コミット9933cc8）以降、workflow_status ツールはレスポンスサイズを削減するため、subagentTemplate フィールドを意図的に返却しないことが決定された。これは以下の理由による設計判断である。

- **レスポンスサイズの削減**: subagentTemplate は平均10KB程度の大型フィールドであり、task-index.json への重複保存時に当該ファイルのサイズが肥大化していた
- **必要な情報源の再評価**: subagentTemplate が必要な場面は workflow_next フェーズ遷移時のみであり、その場面では workflow_next のレスポンスから取得するのが自然である
- **Orchestrator 設計の改善**: workflow_next レスポンスに必ず含まれるため、Orchestrator がメインロジックから直接取得する設計がより疎結合で保守性が高い

この設計決定に基づき、以下の3つの修正が実装された。

## FR-2：workflow-plugin/mcp-server/src/tools/next.ts への slimSubPhaseGuide 関数追加

### 修正内容

workflow_next ツールのレスポンス生成ロジックに `slimSubPhaseGuide` 関数を追加した。この関数の具体的な処理は以下の通りである。

**関数定義と機能**

```
関数名: slimSubPhaseGuide
入力: subPhaseGuide: Record<string, unknown>
処理: 3つの大型フィールドを削除
  - subagentTemplate（約10KB）
  - content（設計サマリー用フィールド）
  - claudeMdSections（CLAUDE.md 抽出フィールド）
出力: 修正後のオブジェクト（副作用: 入力引数を直接変更）
```

**呼び出し箇所と条件**

- **呼び出し位置**: workflow_next ツールで各サブフェーズガイドを返却する直前
- **呼び出し条件**: サブフェーズガイド（state.phaseGuides[subPhaseName]）の返却時に毎回実行
- **型の対称性**: status.ts（workflow_status ツール）の同等ロジックと完全に対称的な処理として実装

### 削除対象フィールドの詳細

| フィールド名 | 用途 | 削除の根拠 | サイズ削減量 |
|:------:|------|------|----:|
| subagentTemplate | Orchestrator が subagent を起動する際のプロンプトテンプレート | workflow_next でのみ必要であり、workflow_status のレスポンスには不要 | 約8-10KB |
| content | 設計フェーズのサマリー用の圧縮テンプレート | ドキュメント生成時の補助情報のため、リアルタイム返却の必要性が低い | 約1-2KB |
| claudeMdSections | CLAUDE.md から抽出したフェーズ固有セクション | 参照用途であり、workflow_status レスポンスには不要 | 約0.5-1KB |

### 修正の根拠

task-index.json キャッシュに保存される phaseGuides オブジェクトが、並列フェーズ実行時に複数サブフェーズ分（最大4個）保存されることで、ファイルサイズが 50-100KB に膨張する問題があった。削除対象の3フィールドは次フェーズ遷移時に Orchestrator が明示的に workflow_next を呼び出して取得する場合にのみ必要であり、キャッシュファイル内での常時保持は不要である。

### 変更対象ファイル

- **ファイルパス**: C:\ツール\Workflow\workflow-plugin\mcp-server\src\tools\next.ts
- **関数名**: slimSubPhaseGuide（新規）
- **関数位置**: ファイル先頭付近（行番号44-55）
- **呼び出し数**: parallel_verification フェーズで4箇所（manual_test、security_scan、performance_test、e2e_test）

## FR-3：definitions.ts の parallel_verification サブフェーズテンプレート拡充

### 修正内容

workflow-plugin/mcp-server/src/phases/definitions.ts ファイルの parallel_verification フェーズに属する3つのサブフェーズ（manual_test、performance_test、e2e_test）について、subagentTemplate に「評価結論フレーズの重複回避」という特化ガイダンスを追加した。

**追加ガイダンスの概要**

各サブフェーズの subagentTemplate に以下の共通セクションを追加した。

```
## 評価結論フレーズの重複回避（特化ガイダンス）

複数のテストシナリオで同一フォーマットの合否判定行を繰り返す場合、
バリデーターの重複行検出によりエラーが発生する。
シナリオ番号または操作名を行に含めて各行を一意にすること。

- NG: 「- 判定: 合格」をシナリオ1・2・3で繰り返す
  （3回以上の同一行でエラー）

- OK: 「- シナリオ1（具体的な操作内容）の合否判定: 合格、
        期待結果が得られたことを確認した」

複数シナリオの評価行は必ずシナリオ名または識別子を含めて
一意にすること。
```

**各サブフェーズ別の追加内容**

| サブフェーズ | 追加行数 | 具体的なガイダンス例 |
|:-------:|:----:|-----------|
| manual_test | 約15行 | テストシナリオ結果で「合格」「不合格」を繰り返す際、シナリオ名を含める |
| performance_test | 約18行 | 複数計測対象の評価で「問題なし」を繰り返す際、計測対象名を含める |
| e2e_test | 約22行 | 複数シナリオの検証結論で「合格」を繰り返す際、シナリオ名を含める |

### テンプレート追加による効果

**バリデーション失敗の予防**

artifact-validator が3行以上の重複行を検出すると、成果物全体をバリデーション失敗とする。特に parallel_verification は複数の独立したサブフェーズが並列実行されるため、各サブフェーズで類似フォーマットの結果報告が発生しやすく、以下のようなパターンが観察されていた。

- manual_test: 複数テストシナリオで「合格」「不合格」の判定を記述
- performance_test: 複数計測対象で「問題なし」「達成」の評価を記述
- e2e_test: 複数シナリオで「検証結果: 合格」の結論を記述

本追加により、subagent テンプレート自体に「シナリオ名を含める」という明示的なガイダンスが埋め込まれるため、バリデーション失敗率が大幅に低下する期待がある。

**フィードバックループの防止**

バリデーション失敗時に MCP サーバーが返すエラーメッセージには、検出された重複行の文字列が含まれる。Orchestrator がこのエラーメッセージの内容をそのまま subagent のリトライプロンプトに転記した場合、フィードバックループが形成される。テンプレート段階でガイダンスを明記することで、subagent がそもそも重複行を生成しない設計となり、ループ自体を事前に防止できる。

### 修正の根拠と技術背景

CLAUDE.md の「subagent起動テンプレート」セクションには既に重複行検出の基本ルールが記載されている。しかし、現場の subagent（テスト実装や検証フェーズを担当）にとって、抽象的な「3回以上の同一行でエラー」という記述だけでは具体的にどう回避すればよいかが明確でない場合が多かった。

そこで、各フェーズの subagentTemplate に「evaluation conclusion phrase deduplication」という特化ガイダンスを追加することで、以下を実現する。

- フェーズの具体的な作業（テスト報告・計測値記録等）の文脈における具体例を示す
- 間違ったパターン（NG）と正しいパターン（OK）を並列して提示する
- 行の一意性を確保する方法（シナリオ番号・操作名の埋め込み）を明確に指示する

### 変更対象ファイル

- **ファイルパス**: C:\ツール\Workflow\workflow-plugin\mcp-server\src\phases\definitions.ts
- **変更対象セクション**: parallel_verification フェーズの以下3つサブフェーズ
  - manual_test（subagentTemplate フィールド内）
  - performance_test（subagentTemplate フィールド内）
  - e2e_test（subagentTemplate フィールド内）
- **追加位置**: 既存の「重複行回避の注意事項」セクション直後に「評価結論フレーズの重複回避（特化ガイダンス）」セクションを追加

## 今後の運用指針

### Orchestrator 開発者向け

- 常に `workflow_next` のレスポンスから `phaseGuide.subagentTemplate` を取得すること
- `workflow_status` がスリムガイドのみを返却することを理解し、テンプレート取得には workflow_next を使うこと
- 必須セクションの要件は workflow_next レスポンスの phaseGuide に必ず含まれることを信頼すること

### テスト実装 subagent 向け

- テストシナリオの評価結論行（「合格」「OK」「問題なし」等の判定行）には必ずシナリオ名または操作の識別子を含めること
- 複数シナリオの結果報告では、各行が完全一致しないよう行の先頭に「シナリオ 1、シナリオ 2」等の修飾語を付け加えること
- 定型フォーマットが3回以上繰り返される場合、バリデーターの重複行検出により失敗する可能性があることを常に意識すること

### MCP サーバー開発者向け

- parallel_verification フェーズで追加される新サブフェーズがある場合、slimSubPhaseGuide 関数の呼び出しを忘れずに追加すること
- 大型フィールド（テンプレート、サマリー、CLAUDE.md セクション等）がキャッシュファイルに不要に蓄積していないかを定期的に監視すること
- subagentTemplate の内容を更新した場合、テンプレート内の「評価結論重複回避」ガイダンスも同時に確認・更新すること

## まとめと今後の展開

本ドキュメント更新フェーズで実装された3つの修正は、workflow-plugin システムの根本的な課題（テンプレート漏れ、ファイルサイズ肥大化、重複行エラー）に対する直接的な対応である。各修正は独立した領域（ドキュメント、バックエンド、テンプレート）での改善となるが、総合的には以下の効果を実現する。

- **設計と実装の整合性向上**: MEMORY.md のガイドラインにより、Orchestrator と subagent 間の契約が明確化される
- **システムリソースの効率化**: slimSubPhaseGuide による task-index.json 圧縮で、ファイルシステムとメモリ負荷が削減される
- **品質管理の自動化**: テンプレートガイダンスの充実により、バリデーション失敗が事前に予防される

これらの修正を通じて、workflow-plugin は「期待値と実績の乖離」が少ない、より堅牢で保守性の高いシステムへと進化する。
