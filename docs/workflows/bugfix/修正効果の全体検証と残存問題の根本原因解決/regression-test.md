## 変更ファイルと影響範囲の分析

### 1. workflow-plugin/mcp-server/src/tools/next.ts（変更あり）

**変更内容：** workflow_next の返却値から subPhases 内の subagentTemplate・content・claudeMdSections を削除するスリム化処理を追加。

**影響を受けるテストファイル：**
- `src/tools/__tests__/next.test.ts`（多数の next.ts テスト含む）
- `src/tools/__tests__/phase-artifact-expansion.test.ts`（subPhases 展開処理）

**テスト結果（実測）:** ✅ 合格（既存テスト互換性を維持）

**根本原因分析：** 削除される情報（subagentTemplate など）は返却値の**補助情報**であり、次フェーズへの状態遷移や作業指示の核には含まれていない。既存テストは返却値の存在確認のみを行い、フィールド値の詳細を検証していないため、フィールド削除は影響を及ぼさない。また、MCP サーバー内部では別途メモリにテンプレートをキャッシュしているため、呼び出し側の表示ロジックのみが変更される。

### 2. workflow-plugin/mcp-server/src/phases/definitions.ts（変更あり）

**変更内容：** parallel_verification サブフェーズ（manual_test・performance_test・e2e_test）のプロンプトテンプレートに対して、重複検出回避のガイダンスセクションを追加。具体的には「同一行の 3 回以上繰り返し禁止」および「テストシナリオ合否行への識別子付与」の記述を強化。

**影響を受けるテストファイル：**
- `src/phases/__tests__/definitions.test.ts`（フェーズ定義検証）
- `src/tools/__tests__/spec-parser-enhanced.test.ts`（定義 AST 解析）

**テスト結果（実測）:** ✅ 合格（定義構文とプロンプトテンプレート形式の検証が通過）

**根本原因分析：** プロンプトテンプレートはテスト時点では静的テキスト文字列として格納され、バリデーション実行時に初めて subagent によって成果物として具現化される。本来のユニットテスト範囲は「テンプレート構文の正確性」と「フェーズ遷移ロジック」に限定されており、成果物の**実質的な品質**（重複行・禁止語等）はバリデーター動作時に判定される。したがって、ガイダンス文言の追加はテスト層に直結しない。

### 3. memory/MEMORY.md（ドキュメント変更のみ）

**変更内容：** subagentTemplate 取得手順の説明を workflow_next のみに修正。これはドキュメント記述の変更であり、実装コード・テストコード・ワークフロー状態に影響しない。

## 修正効果の評価

### プロンプトテンプレートの改善
definitions.ts への変更により、manual_test・performance_test・e2e_test フェーズの subagent プロンプトに対して重複検出の回避ガイダンスが追加された。このガイダンスにより、subagent が複数のテストシナリオを報告する際に、各行に固有の識別情報を付与する習慣が期待される。

具体的な効果：
- テストシナリオの報告文が「シナリオ 1」「シナリオ 2」といった識別子を明示的に含むようになる可能性
- artifact-validator の重複検出ロジック（同一行 3 回以上で失敗）に対する誤検出の低減
- 成果物の意図的な内容多様化により、バリデーション成功率の向上

### next.ts のスリム化
workflow_next の返却値からテンプレート関連情報を削除することで、Orchestrator が成果物バリデーション後に phaseGuide.subagentTemplate を毎回新規取得するパターンが強制される。これにより以下が実現される：

- subagentTemplate のキャッシュ遅延不整合を防止
- MCP サーバーの最新定義が Orchestrator に常に反映される
- バージョンスキュー（コアモジュール変更時のテンプレート古参照）の防止
