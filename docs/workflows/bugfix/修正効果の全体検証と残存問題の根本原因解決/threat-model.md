## サマリー

本文書は「修正効果の全体検証と残存問題の根本原因解決」タスクに対する脅威モデルである。
変更対象は MEMORY.md（FR-1）、next.ts（FR-2）、definitions.ts（FR-3）の 3 ファイルである。
最も深刻な脅威は「Orchestrator の誤動作による不完全なサブエージェントプロンプト生成」と
「テンプレート変更によるバリデーション連鎖失敗」の 2 点であると判定した。
各脅威の優先度と対策方針を本文書で整理し、次フェーズの設計・実装計画に引き継ぐ。
特に FR-2 の後方互換性リスクと FR-3 のテンプレート品質確認手順を実装フェーズで継続して考慮すること。

---

## 脅威モデル

### 対象システムの概要

本タスクの変更対象は以下の 3 ファイルである。
MEMORY.md は Orchestrator が参照する手順書であり、会話をまたいで行動規範として機能する。
workflow-plugin/mcp-server/src/phases/next.ts はフェーズ遷移レスポンスを構築するコアモジュールであり、
workflow-plugin/mcp-server/src/phases/definitions.ts はサブエージェントテンプレートを格納するコアモジュールである。
これら 3 ファイルは Claude の Orchestrator が各フェーズでサブエージェントを起動する際のコアコンポーネントであり、
誤った変更は複数のワークフローセッションにわたって継続的な影響を与えうる。

### 脅威1: MEMORY.md 誤記による Orchestrator の誤動作（FR-1）

**脅威の種類**: 情報の不整合による誤動作（Integrity）

**攻撃経路**: Orchestrator がセッション開始時に MEMORY.md を参照し、誤った手順に従って
workflow_status から subagentTemplate を取得しようとする。Fix 2 以降、workflow_status の
レスポンスには subagentTemplate が含まれないため、取得に失敗する。失敗後、Orchestrator は
CLAUDE.md の「subagent起動テンプレート」セクションを独自解釈してプロンプトを構築しようとするが、
テンプレートの必須セクションや禁止パターンが欠落した不完全なプロンプトになる可能性がある。

**影響範囲**: 全フェーズのサブエージェントに波及する可能性がある。不完全なプロンプトから生成された
成果物はバリデーションで失敗し、リトライが多発する。最悪の場合、Orchestrator は修正指示を含まない
リトライプロンプトを送り続け、無限ループ状態に陥る。

**深刻度**: High（複数フェーズ・複数セッションにわたって影響が持続する）

**発生確率**: 中程度（MEMORY.md は参照優先度が高く、誤記があれば高頻度で誤動作が発生する）

**STRIDE 分類**: Tampering（情報の完全性の喪失による誤動作）

---

### 脅威2: next.ts スリム化による後方互換性の破壊（FR-2）

**脅威の種類**: 設計変更による非互換動作（Reliability）

**攻撃経路**: next.ts のスリム化処理が誤ってトップレベルの phaseGuide.subagentTemplate まで
除外した場合、Orchestrator は workflow_next のレスポンスから subagentTemplate を取得できなくなる。
フェーズ遷移直後に必要なテンプレートが欠落するため、サブエージェントプロンプトが不完全になる。
この問題は「サブフェーズ（subPhases 配下）」と「現フェーズ自身（トップレベル）」の区別が
不明確なコードになった場合に発生しうる。

**影響範囲**: parallel_* フェーズだけでなく、全フェーズの遷移後のサブエージェント起動に影響する。
テンプレートが欠落しても即座にエラーにはならず、Orchestrator が別の手段でプロンプトを構築しようとするため、
問題の発見が遅れる可能性がある。

**深刻度**: Critical（全フェーズに影響し、問題発見が遅れると多数のバリデーション失敗が発生する）

**発生確率**: 低（スリム化の対象範囲が明確に定義されており、レビューで検出可能）

**STRIDE 分類**: Denial of Service（サブエージェントが正しいプロンプトを受け取れず機能不全になる）

---

### 脅威3: definitions.ts テンプレートへの禁止語混入（FR-3）

**脅威の種類**: バリデーション失敗の連鎖（Integrity）

**攻撃経路**: manual_test・performance_test・e2e_test のテンプレートに追加するガイダンス文章が、
artifact-validator が検出する 12 語の禁止語グループ（英語略語 4 語・日本語 8 語）のいずれかを
含んでしまう場合がある。テンプレートはサブエージェントのプロンプトに埋め込まれるが、
テンプレート自体に禁止語があっても起動時はブロックされない。しかし、サブエージェントがガイダンスの
例示文を成果物にそのまま転記してしまった場合、成果物が禁止語を含む形になりバリデーションで失敗する。

**影響範囲**: manual_test・performance_test・e2e_test フェーズのバリデーション失敗が増加する。
リトライプロンプトに前回エラーメッセージのコードブロックが含まれるため、エラーメッセージ内の
禁止語がフィードバックループを形成し、リトライが繰り返されるリスクがある（過去に実際に発生したパターン）。

**深刻度**: Medium（特定フェーズのみに影響するが、フィードバックループ形成時は深刻になりうる）

**発生確率**: 中程度（ガイダンスには禁止語の「言い換え例」を記述するため、禁止語そのものを含むリスクがある）

**STRIDE 分類**: Tampering（成果物の整合性を損なう誤ったガイダンスによる副作用）

---

### 脅威4: security_scan テンプレートにない問題が manual_test 等に再発（FR-3 の不完全な適用）

**脅威の種類**: 不完全な修正による将来の問題再発（Availability）

**攻撃経路**: security_scan に追加されたガイダンスの構成要素（NG 例・OK 例・評価結論フレーズの列挙）の
一部のみを manual_test 等に追加した場合、追加が不十分な状態になる。特に「評価結論フレーズの具体例リスト」が
欠けると、サブエージェントが重複行を生成するパターンを避けられない。ガイダンスの内容が security_scan の
ものと構造的に異なる場合、Orchestrator が参照する際に混乱が生じる可能性もある。

**影響範囲**: parallel_verification フェーズの manual_test・performance_test・e2e_test サブフェーズで、
バリデーション失敗が再発する。Fix 1 で解決したはずの問題が別のサブフェーズで継続する形になる。

**深刻度**: Medium（Fix 1 の効果が限定的になる）

**発生確率**: 中程度（移植するガイダンスの内容の網羅性に依存する）

**STRIDE 分類**: Denial of Service（バリデーション失敗の繰り返しによる作業継続の阻害）

---

### 脅威5: MCP サーバー再起動忘れによる変更の非適用（NFR-2 への違反）

**脅威の種類**: 運用手順の省略による変更未適用（Reliability）

**攻撃経路**: definitions.ts および next.ts の修正後に npm run build を実行したとしても、
MCP サーバーのプロセスを再起動しない場合、Node.js のモジュールキャッシュにより古いバイナリが動作し続ける。
変更が適用されたと思って次フェーズに進んだ場合、実際には修正前のテンプレートがサブエージェントに
提供され続ける。この問題はエラーメッセージが出ないため発見が困難であり、「修正したのに問題が再発する」
という誤認識につながる。

**影響範囲**: コアモジュール（definitions.ts、next.ts）の変更が一切効かなくなる。デバッグに多くの時間を要する。

**深刻度**: Medium（気付きにくい問題で、調査コストが高くなる）

**発生確率**: 中程度（CLAUDE.md に再起動義務が明記されているが、手動作業のため省略されやすい）

**STRIDE 分類**: Tampering（古いコードが動作することによる整合性の喪失）

## リスク

### リスク評価マトリクス

本タスクで識別した 5 つの脅威に対して深刻度と発生確率を組み合わせたリスク評価を行う。
リスク値が高い順に優先度を決定し、対策の実施順序を決定する。

### R-1: Orchestrator の誤動作リスク（脅威1 由来）

深刻度 High・発生確率 中程度の組み合わせにより、総合リスクを「高」と評価する。
MEMORY.md の誤記は会話セッションをまたいで影響を与え続けるため、早期修正が不可欠である。
修正の遅延は全フェーズのサブエージェント品質に影響するため、本タスクで最初に対処すべきリスクの一つである。

### R-2: 後方互換性破壊リスク（脅威2 由来）

深刻度 Critical・発生確率 低の組み合わせにより、総合リスクを「高」と評価する。
発生確率が低くても深刻度が Critical であるため、実装後に必ずレスポンス構造の検証を行う必要がある。
並列フェーズ以外のシングルフェーズでの動作確認も実施することで、見落としを防ぐ。

### R-3: 禁止語混入によるバリデーション連鎖失敗リスク（脅威3 由来）

深刻度 Medium・発生確率 中程度の組み合わせにより、総合リスクを「中」と評価する（フィードバックループ形成時は深刻化する）。
過去に同一パターン（フィードバックループ）で問題が発生した実績があるため、
追記内容の事前スクリーニングを実施することで予防する。

### R-4: ガイダンス移植の不完全性リスク（脅威4 由来）

深刻度 Medium・発生確率 中程度の組み合わせにより、総合リスクを「中」と評価する（移植ガイダンスの欠落により Fix 1 の効果が限定的になる点が懸念）。
移植元となる security_scan のガイダンスと構成要素を明示的に対比しながら追記することで、
欠落を防ぐことができる。

### R-5: MCP サーバー再起動忘れリスク（脅威5 由来）

深刻度 Medium・発生確率 中程度の組み合わせにより、総合リスクを「中」と評価する（再起動省略によるモジュールキャッシュ問題が主因）。
このリスクは手順の明文化と実施後の確認ステップ（workflow_status 呼び出しによるフェーズ確認）
を義務化することで大幅に低減できる。

## 対策

### 対策1: MEMORY.md 更新内容のダブルチェック（脅威1 への対応）

MEMORY.md の更新箇所は修正前後で意味が変わらないことを確認するため、変更差分を明示して実施する。
具体的には「workflow_next または workflow_status」という記述を「workflow_next」のみに限定する書き換えを行い、
workflow_status から取得できない情報（subagentTemplate）を明確にリストアップした注釈を追記する。
修正後は MEMORY.md 全体を通読し、手順の論理的な整合性（フェーズ遷移 → subagentTemplate 取得 → Task 起動の流れ）を確認する。

### 対策2: next.ts のスリム化対象を型レベルで明確化（脅威2 への対応）

スリム化処理の実装では、subPhases の各要素（配列要素）に対してのみ除外処理を行うコードを書く。
トップレベルの phaseGuide オブジェクトに対して除外処理が適用されないことを、コードの構造（ループのスコープ）で保証する。
実装後のテスト手順として、parallel_verification への遷移を模擬した手動確認を行い、
レスポンスのトップレベルに subagentTemplate が存在し、各 subPhase 配下に subagentTemplate が存在しないことを検証する。
レスポンスサイズの計測を行い、NFR-3 の 15K 文字以下目標を達成していることを確認する。

### 対策3: テンプレート追記内容の禁止語スキャン（脅威3 への対応）

definitions.ts に追記するガイダンス文章は、追記前に artifact-validator の 12 語の禁止語グループとの照合を行う。
特に「NG 例」として記述するフレーズが禁止語そのものを含む可能性があるため、NG 例は禁止語を直接書くのではなく
「バリデーターが検出する評価結論フレーズの例」という間接表現で記述する方針とする。
OK 例は禁止語を含まない具体的な文章を用い、コードフェンス内に記述して検出対象外とすることを検討する。

### 対策4: security_scan のガイダンス構成をテンプレートとして活用（脅威4 への対応）

manual_test・performance_test・e2e_test への追記は、security_scan に追加されたガイダンスの構成要素を
そのままの形式で移植することで、内容の一貫性と網羅性を担保する。
security_scan のガイダンスに含まれる「フレーズ重複回避の説明」「NG 例」「OK 例」「一意化の指針」の 4 要素を
全て含めることを必須とし、テスト対象ごとに適切な言葉（シナリオ名、計測対象名）に差し替える。

### 対策5: ビルドと再起動の手順を固定化（脅威5 への対応）

definitions.ts および next.ts を修正した後は、以下の手順を省略なく実行することを本タスクの実施条件とする。
まず、workflow-plugin/mcp-server ディレクトリで npm run build を実行して全ファイルのトランスパイルを完了させる。
次に、dist/phases/next.js および dist/phases/definitions.js の更新日時を確認してビルド成功を確かめる。
その後、MCP サーバープロセスを再起動し、workflow_status を呼び出して現在のフェーズが正しく返ることを確認する。
この手順は CLAUDE.md の「強制再起動条件」セクションに既に明記されており、遵守を徹底する。
