## サマリー

workflow-pluginテスト基盤の修正（vitest.config.ts作成、tsconfig.json修正、DesignValidatorモック修正）に対する脅威分析を実施した。

修正の範囲はテスト設定とテストコード内のモック実装に限定されており、プロダクションコードの機能的変更は最小限である。

セキュリティ脅威としてはテスト環境特有の構成上のリスクが主となり、本番環境への直接的な影響は低い。

修正に伴う予期しない副作用を防ぐため、5つの主要な脅威ポイントを特定し対策を実装する必要がある。

全修正適用後の検証により、テスト信頼性の向上とビルドアーティファクトの正常化が期待される。

## 脅威分析

### 脅威T-001: vitest.config.tsのglobals設定による意図しないグローバル汚染

workflow-pluginルートのvitest.config.tsでglobals: trueを設定するとテスト環境でdescribe、test、expectなどのグローバル関数が提供される。

この設定がmcp-server以外のディレクトリ内のファイルに対しても適用される可能性がある。

もしテストファイル以外のソースコードがincludeパターンの範囲外にあれば影響は限定されるが、パターン設定の誤りがある場合はテスト専用関数が本番コードで使用される恐れがある。

具体的には、includeパターンを「src/**」のように広く設定した場合、ソースコードディレクトリ内のテスト以外のファイルでもグローバル関数にアクセス可能になる。

対策としてincludeパターンを「mcp-server/src/**/*.test.ts」と「hooks/__tests__/**/*.test.ts」に厳密に限定し、テストファイルの拡張子パターンを明示する。

テストディレクトリと本番ディレクトリを明確に分離することでグローバル汚染リスクを最小化する。

### 脅威T-002: tsconfig.jsonのexclude追加による必要なファイルの除外

tsconfig.jsonのexclude配列に「**/__tests__/**」と「**/*.test.ts」を追加する修正では、パターン指定の誤りが本番ファイルの誤削除を招く可能性がある。

特に「__tests__」という命名が他の目的で使用されているディレクトリが存在する場合、意図しないファイルがビルド対象から除外される危険性がある。

プロジェクト内を調査した結果、__tests__ディレクトリはmcp-server/src配下のテストディレクトリのみであり、他の用途での使用は確認されない。

本件では修正範囲がmcp-server内に限定されており、他プロジェクトへの波及是影響は発生しない。

対策として修正前に全体のディレクトリ構造を確認し、__tests__や.test.tsのパターンがテストファイル以外の用途で使用されていないことを事前確認する。

### 脅威T-003: dist/ディレクトリの既存テストファイル削除による依存関係破損

BUG-002修正後にnpm run buildを実行するとdist/内の既存テストファイル（dist/**/__tests__/**/*.js、dist/**/*.test.js）が削除される。

もしCI/CDパイプラインやドキュメント生成プロセスが明示的にdist/内のテストファイルに依存している場合、パイプラインの失敗が発生する可能性がある。

プロジェクト調査の結果、package.jsonのスクリプト、CI設定ファイル（.github/workflows/*.yml）、ドキュメント（README.md等）でdist/内のテストファイルへの参照は存在しない。

依存関係破損のリスクは実質的に存在しないと判断される。

対策として修正実行前にCI設定、デプロイスクリプト、外部ドキュメントを全文検索してdist/テストファイルへの依存関係を最終確認する。

### 脅威T-004: DesignValidatorモック変更による他のテストユースケースへの影響

session-token.test.ts、skip-env-removal.test.ts、next.test.tsの3ファイルでDesignValidatorのモックをvi.fn(() => ({}))からクラス構文またはfunction式に変更する修正では、モック動作の微妙な変更により他のテストが影響を受ける可能性がある。

具体的には、クラス構文を使用した場合、新しいコンストラクタのprototypeプロパティがvi.fn式とは異なるオブジェクト構造を持つため、instanceof チェックやプロトタイプチェーンの動作が変わる可能性がある。

3つのテストファイルは相互に独立した個別のテストスコープを持つモジュールであり、共有されるグローバルな状態は存在しない。

各テストファイルは独自のモック定義を持つため、修正の影響は当該ファイル内に局所化される。

対策として修正後に「npx vitest run --reporter=verbose」で各テストの詳細出力を確認し、モック動作の期待値との乖離がないことを検証する。

### 脅威T-005: vitest.config.ts作成によるmcp-server設定との競合

workflow-pluginルートにvitest.config.tsを新規作成する場合、既存のmcp-server/vitest.config.tsとの設定優先度関係が明確でない場合、ネストされた設定の優先度に基づく予期しない動作が発生する可能性がある。

vitestはネストされたディレクトリに複数のvitestファイルが存在する場合、実行ディレクトリに最も近いファイルの設定を優先する仕様である。

workflow-pluginルートから実行した場合はルートレベルのvitestが優先され、mcp-server/ディレクトリから実行した場合はmcp-server/vitest.config.tsが優先される設計である。

実装調査により、既存mcp-server/vitest.config.tsの設定（globals: true、environment: node、includeパターン等）がワークフロープラグイン全体で共有される構造になっていることを確認した。

対策として新規作成するワークフロープラグイン/vitest.config.tsをmcp-serverの設定を上位互換として継承し、hooks/__tests__のディレクトリを追加includeパターンとして拡張する形で実装する。

## リスク対策

### 対策A-001: vitest設定の厳密化とドキュメント化

workflow-plugin/vitest.config.tsでは、includeパターンを明示的に「mcp-server/src/**/*.test.ts」と「hooks/__tests__/**/*.test.ts」に制限する。

設定ファイル内のコメントでincludeパターンの意図を説明し、将来の誤変更を防ぐ。

「// テストファイルのみを対象とする」などのコメントを添付して保守性を向上させる。

### 対策A-002: tsconfig.json修正の前提条件確認

修正実行前に以下を実施する。
- プロジェクト全体をgrep検索して__tests__ディレクトリの用途を全て確認する。
- .test.tsファイルが全てテストファイルであることを確認する。
- 修正対象外の他プロジェクト（frontend等）への影響がないことを確認する。

### 対策A-003: CI/CD依存関係の事前確認

dist/ディレクトリへの依存関係を全て確認する。
- GitHub Actionsワークフロー内の参照をすべて検索する。
- デプロイスクリプト、ドキュメント生成スクリプトでの参照を確認する。
- 修正実行後の初回ビルド時に全CI パイプラインが成功することを確認する。

### 対策A-004: DesignValidatorモック検証の強化

修正後に以下の項目でテストを実行する。
- 当該3ファイルの単体テスト実行（npx vitest run session-token.test.ts等）。
- DesignValidatorを使用する全テストのコンストラクタ動作確認（new構文の成功）。
- 各テストの期待値との一致確認（console.logでのモック呼び出し検証）。

### 対策A-005: vitest設定の統合テスト

修正後に以下の実行パターンで全テスト成功を確認する。
- workflow-pluginルートから「npx vitest run」実行（全テスト、hooks含む）。
- mcp-server/ディレクトリから「npx vitest run」実行（既存テスト、変更なし）。
- hooks/__tests__ディレクトリから「npx vitest run」実行（hooks のみ、新規）。

## 結論と推奨事項

本修正はテスト基盤に限定された変更であり、セキュリティリスクは低い。
各脅威に対して具体的な対策が実装可能であり、事前・事後の検証により全てのリスクを許容可能なレベルに軽減できる。

修正順序（BUG-002 → BUG-001 → BUG-003）に従い、段階的に実装することで、各段階での問題発生時に早期発見が可能である。

全修正適用後の包括的なテスト実行と検証により、修正に伴うリスクはゼロレベルまで軽減される。

本脅威モデルの分析に基づき、修正実装を進めることを推奨する。
