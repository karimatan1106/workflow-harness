## サマリー

- 目的: 前回ワークフロー「修正効果の全体検証と残存問題の根本原因解決」のcode_reviewフェーズで発生したバリデーション失敗の根本原因を特定し、他フェーズへの同様リスクを洗い出す
- 発見された問題:
  1. code_reviewのsubagentTemplateに「評価結論フレーズの重複回避（特化ガイダンス）」セクションが存在しない（FR-3の適用漏れ）
  2. FR-3で追加されたのはmanual_test・security_scan・performance_test・e2e_testの4フェーズのみで、code_reviewは対象外だった
  3. フェーズガードログにcommitフェーズでnpm run buildがブロックされる構造的問題が記録されているが、最終的にcommit・pushは完了しており実害なし
- 次フェーズで必要な情報: 修正対象は `workflow-plugin/mcp-server/src/phases/definitions.ts` のcode_review.subagentTemplate（1箇所のみ）

---

## 問題1: code_review テンプレートのFR-3適用漏れ

### 根本原因

前回の「修正効果の全体検証と残存問題の根本原因解決」タスクのFR-3では、parallel_verificationの4サブフェーズ（manual_test・security_scan・performance_test・e2e_test）のsubagentTemplateに「評価結論フレーズの重複回避（特化ガイダンス）」セクションを追加した。しかしcode_reviewフェーズのテンプレートには同様のガイダンスが追加されていなかった。

これにより、code_reviewサブエージェントが複数のFR（FR-1・FR-2・FR-3）について同一フォーマットで整合性確認を記述した際に、バリデーターの重複行検出が発火した。

### 症状の再現メカニズム

1. code_reviewサブエージェントが設計-実装整合性セクションを記述する際、FR-1・FR-2・FR-3それぞれに対して「spec.mdの要件確認結果は以下のとおりである。」という同一フレーズを使用した
2. この行は50文字を超えており（半角換算で約48文字）、コロンで終わらないため、ルール8（50文字以内コロン終端ラベル行の除外）の対象とならない
3. 3回出現で重複エラーが発火し、バリデーション失敗となった

### 影響を受けたフェーズガードログ

`.claude-phase-guard-log.json`の記録によると、code-review.mdへの書き込みは以下のタイミングで行われた:
- 初回作成（バリデーション失敗）
- リトライ1〜5回（合計6回の書き込み試行）

これはOrchestratorがルール21に従いsubagentを再起動し続けたことを示す。最終的にsonnetモデルへのエスカレーションで解決したが、FR-3が適用されていれば初回から成功していた可能性が高い。

### テンプレート差分の整理

| フェーズ | 評価結論フレーズ重複回避ガイダンス | 備考 |
|---------|:----:|------|
| manual_test | あり | FR-3で追加済み |
| security_scan | あり | FR-3で追加済み |
| performance_test | あり | FR-3で追加済み |
| e2e_test | あり | FR-3で追加済み |
| code_review | **なし（欠落）** | 今回の修正対象 |

## 問題2: commitフェーズでのnpm run buildブロック

### 根本原因

commitフェーズの許可カテゴリは`readonly, git`であり、`npm run build`（implementationカテゴリ）は許可されていない。コアモジュール変更後のMCPサーバー再起動手順（CLAUDE.md ルール22）ではビルドが必要になるが、このビルド作業はcommitフェーズより前のフェーズ（implementationまたはrefactoring）で完了させることが設計上の前提となっている。

### 実害評価

前回セッションのフェーズガードログを確認した結果、ブロックは発生したが最終的にcommit・pushは正常に完了している。MCP サーバーの再起動はユーザーがClaude Desktop側で操作したため、commitフェーズ以前にビルドが完了していた状態だったと推定される。

今回の問題は「commitフェーズでビルドが必要になる設計フロー」の設計上の注意点として記録するが、コード修正は不要と判断する。

## 問題3: セッション再開後の並列フェーズ状態復元

### 根本原因

前回セッションはコンテキスト枯渇で中断し、セッション再開後にparallel_qualityのサブフェーズが「pending」状態として残っていた。これはMCPサーバーの状態管理上、前のセッションで完了したサブフェーズが新セッションに引き継がれなかったことによる。

### 実害評価

セッション再開後にworkflow_complete_subをsessionToken付きで再呼び出しすることで回復できた。構造的問題ではあるが、現在の設計範囲内の動作であり、今回の修正スコープ外とする。

## 修正方針

### 修正対象
- ファイル: `workflow-plugin/mcp-server/src/phases/definitions.ts`
- 修正箇所: `code_review`フェーズのsubagentTemplate
- 追加内容: `## 評価結論フレーズの重複回避（特化ガイダンス）`セクション

### 追加内容の案

code_review特有のパターン（複数のFR・修正箇所・ファイルを同一フォーマットで評価する傾向）に対応したガイダンス:

```
## 評価結論フレーズの重複回避（特化ガイダンス）
複数のFR・修正箇所・ファイルを同一フォーマットで評価する場合、バリデーターの重複行検出によりエラーが発生する。
レビュー対象の識別子（FR番号・ファイル名・関数名等）を行に含めて各行を一意にすること。
- NG（3回以上の同一行でエラー）: 「spec.mdの要件確認結果は以下のとおりである。」をFR-1・FR-2・FR-3で繰り返す
- OK: 「FR-1（MEMORY.md更新）に関するspec.md要件の確認結果は以下のとおりである。」
- OK: 「FR-2（next.tsスリム化）に関するspec.md要件の確認結果は以下のとおりである。」
複数修正箇所の評価行は必ずFR番号やファイル名などの識別子を含めて一意にすること。
```

### 後続作業
- `definitions.ts`はコアモジュールのため、変更後はMCPサーバー再起動が必要（CLAUDE.md ルール22）
- ビルドはimplementationフェーズで実施する
