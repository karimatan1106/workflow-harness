## サマリー

- 目的: code_reviewフェーズのsubagentTemplateに「評価結論フレーズの重複回避（特化ガイダンス）」セクションを追加するための設計文書であり、テンプレートプロンプトのインターフェース設計を確定する。
- 主要な決定事項: 追加するガイダンステキストの最終文言を本ドキュメントで確定し、挿入位置（`## パフォーマンスセクションの行数ガイダンス` 末尾と `## 出力` の間）および挿入形式（TypeScript文字列リテラル内の `\n\n` 区切り）をimplementationフェーズへ引き継ぐ。
- APIレスポンス設計では、workflow_complete_subの成功・失敗パターンとバリデーションエラーのメッセージ形式を整理し、subagentがエラーを受け取った場合の対処パターンを設計する。
- CLIインターフェース設計では、workflow_next / workflow_complete_subの入力パラメータと出力フィールドを整理し、subagentTemplate取得の正しい経路を確定する。
- 次フェーズで必要な情報: 設定ファイル設計セクションに記載した最終ガイダンステキスト（Markdown本文形式）とTypeScript文字列リテラル形式の両方をimplementationフェーズのsubagentへ渡すこと。

## CLIインターフェース設計

本タスクで中心的に使用するMCPサーバーAPIは `workflow_next` と `workflow_complete_sub` の2種類である。それぞれのインターフェース設計を以下に示す。

### workflow_next インターフェース

入力パラメータとしてセッショントークンを受け付け、現在のフェーズを次フェーズへ遷移させる。レスポンスには次フェーズの `phaseGuide` オブジェクトが含まれ、その中に `subagentTemplate` フィールドが格納される。Orchestratorはこのフィールドからsubagentへ渡すプロンプトを取得する設計となっている。

workflow_nextのレスポンスに含まれる主要フィールドの一覧:

- `phase`: 遷移後の現在のフェーズ名（例: `parallel_quality`）を示す文字列フィールド。
- `phaseGuide`: フェーズ固有のガイダンスオブジェクトであり、`subagentTemplate` を含む。
- `phaseGuide.subagentTemplate`: subagentへ渡すプロンプトテンプレートの完全文字列。このフィールドがsubagentTemplate取得の唯一の正規経路である。
- `artifactValidationError`: 成果物バリデーション失敗時のエラーメッセージを格納するフィールド。値が存在する場合はリトライが必要であることを示す。
- `message`: 遷移結果のメッセージ文字列。成功・失敗の概要を含む。

workflow_statusはFix 2以降 `subagentTemplate` を返さないスリムガイド設計のため、subagentTemplate取得源として使用してはならない。

### workflow_complete_sub インターフェース

入力パラメータとしてサブフェーズ名（例: `code_review`）を受け取り、並列フェーズ内のサブフェーズを完了マークする。レスポンスには成功・失敗の状態と、成果物バリデーション結果が含まれる。

- `subPhase`: 完了対象のサブフェーズ名（例: `code_review`、`manual_test` 等）を示す入力パラメータ。
- `success`: サブフェーズが正常に完了マークされたかどうかを示すboolean値。
- `validationErrors`: 成果物バリデーション失敗時のエラー詳細リスト。空配列の場合はバリデーション通過を意味する。
- `message`: 完了処理の結果メッセージ。成果物パスや検出された問題の概要を含む。

## エラーメッセージ設計

code_reviewフェーズのsubagentがcode-review.mdを出力した際に、バリデーターが重複行を検出する場合のエラーメッセージ形式を以下に示す。

### 現状のエラーメッセージ形式（修正前）

ガイダンスセクションが欠落している現状では、subagentが複数のFRを同一フォーマットで評価した際に以下のようなエラーが発生する。

エラーの典型パターンとして、バリデーターは「同一行が3回以上出現」というルールを適用し、エラーメッセージに重複した行の内容と出現回数を含めて返す。エラーメッセージに記載される行は成果物本文に記述された評価結論フレーズであり、例えば「spec.mdの要件確認結果は以下のとおりである。」という行がFR-1・FR-2・FR-3の各評価で3回繰り返された場合にエラーが発生する。

バリデーターが返すエラーオブジェクトの構造:

- `type`: エラー種別を示す文字列フィールド（重複行エラーの場合は `duplicate_line` に相当する値）。
- `line`: 重複が検出された行の内容を示すフィールド。この行のテキストが成果物本文から抽出される。
- `count`: 同一行が出現した回数を示すフィールド。閾値は3であり、count が3以上の場合にエラーとなる。
- `message`: Orchestratorに渡されるエラー概要テキスト。リトライプロンプトに含めるべき内容が記述される。

### 改善後のエラー削減効果

ガイダンスセクション追加後は、subagentがFR番号と対象ファイル名を各行に含める形式を採用するため、評価結論行が一意になり重複行エラーの発生を防止できる。エラーメッセージの発生頻度はリトライ率で計測され、ガイダンス追加によりcode_reviewフェーズのリトライ回数の削減が期待される。

バリデーション成功時のレスポンスに含まれるフィールド:

- `validated`: 成果物バリデーション通過を示すboolean値（true）。
- `artifactPath`: 検証した成果物ファイルのパス（例: `docs/workflows/{taskName}/code-review.md`）。
- `lineCount`: 成果物の総行数を示す数値。minLines要件と照合して合否が判定される。
- `sectionCount`: 検出された `## ` セクション数を示す数値。requiredSections との照合に使用される。

## APIレスポンス設計

workflow_complete_subの成功レスポンスと失敗レスポンスの形式を設計する。code_reviewサブフェーズの完了処理において想定される2パターンを示す。

### 成功レスポンス設計

code-review.mdが品質要件を満たした場合、workflow_complete_subは以下の形式でレスポンスを返す。

- `success`: 値 `true` を返し、サブフェーズの完了マークが正常に行われたことを示す。
- `message`: 「code_review サブフェーズが完了しました」に相当するメッセージを含む文字列フィールド。
- `nextAvailableActions`: 次に実行可能なアクション（例: build_checkの完了後にworkflow_nextを呼び出す等）のリストを含むフィールド。
- `subPhaseStatus`: 各サブフェーズの完了状態マップ。`{code_review: 'completed', build_check: 'pending'}` の形式で現在の並列フェーズの進捗を示す。
- `allSubPhasesCompleted`: すべてのサブフェーズが完了した場合に `true` となるboolean値。このフラグが `true` になったタイミングでworkflow_nextを呼び出すことができる。

### 失敗レスポンス設計（バリデーションエラー）

成果物バリデーション失敗時は以下の形式でレスポンスが返り、Orchestratorはsubagentを再起動してリトライを実施する。

- `success`: 値 `false` を返し、サブフェーズの完了マークが行われていないことを示す。
- `artifactValidationError`: バリデーターが検出した問題の詳細を含むオブジェクトまたは文字列フィールド。
- `retryPrompt`: リトライ用プロンプトの雛形テキストを含むフィールド（buildRetryPromptの返り値に相当）。
- `suggestModelEscalation`: haikuモデルで2回以上リトライが失敗した場合に `true` となるフラグ。Orchestratorはこのフラグを確認してsonnetへのモデルエスカレーションを判断する。

Orchestratorはこのレスポンスを受け取った場合、CLAUDE.mdルール21に従いTaskツールでsubagentを再起動してバリデーション成功まで繰り返す。

## 設定ファイル設計

追加するガイダンスセクション名は「評価結論フレーズの重複回避（特化ガイダンス）」であり、manual_test・security_scan・performance_testの各テンプレートで使用されているセクション名と完全一致させる必要がある。
ガイダンスの本文は「複数のFRや修正ファイルを同一フォーマットで評価する場合、バリデーターの重複行検出によりエラーが発生する」という問題説明を起点とし、FR番号や対象ファイル名などの識別子を行に含める指示を続ける構成とする。
NG例はFR番号なしの評価フレーズを3回繰り返すパターンを示し、バリデーターの重複行検出が発火する具体的ケースをsubagentに提示する。
OK例はFR番号と対象ファイル名を含む識別子付きフレーズを示し、各評価行が一意になる記述方法をsubagentが学習できる形式で提供する。
追加するガイダンステキストの合計行数は6行であり、spec.mdで定義された20行以内の文字列長制約を大幅に下回り、保守性の観点でも問題がない。

`workflow-plugin/mcp-server/src/phases/definitions.ts` の `code_review.subagentTemplate` 文字列に追加するガイダンステキストの最終設計を以下に示す。

追加対象のセクション名は「評価結論フレーズの重複回避（特化ガイダンス）」であり、manual_test・security_scan・performance_testの各テンプレートに存在する同名セクションと完全一致させる。挿入位置はパフォーマンスセクション行数ガイダンスの末尾（`\n- レスポンスタイムへの影響...`の行）と出力セクション（`## 出力`）の間である。追加するガイダンスは6行構成（見出し1行・本文1行・NG例1行・OK例2行・締め文1行）であり、20行以内のFR-3要件を満たす。NG例としてFR番号なしの同一フレーズを3回繰り返すケースを示し、OK例としてFR-1・FR-2それぞれのFR番号と修正内容を含む一意な記述を示すことで、subagentが各評価行に識別子を付与する習慣を形成する。

### 追加するセクションの最終文言（Markdown本文形式）

以下が追加する「評価結論フレーズの重複回避（特化ガイダンス）」セクションの最終文言である。

```
## 評価結論フレーズの重複回避（特化ガイダンス）
複数のFRや修正ファイルを同一フォーマットで評価する場合、バリデーターの重複行検出によりエラーが発生する。FR番号や対象ファイル・関数名などの固有識別子を行に含めて各行を一意にすること。
- NG: 「spec.mdの要件確認結果は以下のとおりである。」をFR-1・FR-2・FR-3で繰り返す（3回以上の同一行でエラー）
- OK: 「FR-1（code_review subagentTemplate末尾へのガイダンスセクション追加）に関するspec.md要件の確認結果は以下のとおりである。」
- OK: 「FR-2（manual_testテンプレートとのセクション名一貫性確保）に関するspec.md要件の確認結果は以下のとおりである。」
複数のFR・修正箇所の評価行は必ずFR番号または対象ファイル・関数名の識別子を含めて一意にすること。
```

### 挿入位置の特定

挿入位置は `code_review.subagentTemplate` 文字列の末尾付近にある `\n- レスポンスタイムへの影響（重い処理の有無と改善可能性を確認した結果を記述する）` の後であり、`\n\n## 出力\n${docsDir}/code-review.md'` の直前である。

Edit toolを使用する際の `old_string` として指定すべき末尾部分:

```
\n- レスポンスタイムへの影響（重い処理の有無と改善可能性を確認した結果を記述する）\n\n## 出力\n${docsDir}/code-review.md'
```

Edit toolを使用する際の `new_string` として指定すべき置換後の文字列:

```
\n- レスポンスタイムへの影響（重い処理の有無と改善可能性を確認した結果を記述する）\n\n## 評価結論フレーズの重複回避（特化ガイダンス）\n複数のFRや修正ファイルを同一フォーマットで評価する場合、バリデーターの重複行検出によりエラーが発生する。FR番号や対象ファイル・関数名などの固有識別子を行に含めて各行を一意にすること。\n- NG: 「spec.mdの要件確認結果は以下のとおりである。」をFR-1・FR-2・FR-3で繰り返す（3回以上の同一行でエラー）\n- OK: 「FR-1（code_review subagentTemplate末尾へのガイダンスセクション追加）に関するspec.md要件の確認結果は以下のとおりである。」\n- OK: 「FR-2（manual_testテンプレートとのセクション名一貫性確保）に関するspec.md要件の確認結果は以下のとおりである。」\n複数のFR・修正箇所の評価行は必ずFR番号または対象ファイル・関数名の識別子を含めて一意にすること。\n\n## 出力\n${docsDir}/code-review.md'
```

### 既存テンプレートとの整合性確認

manual_testフェーズのsubagentTemplateには `## 評価結論フレーズの重複回避（特化ガイダンス）` というセクションが既に存在しており、今回追加するセクション名と完全一致させる必要がある。security_scanおよびperformance_testにも同様の趣旨のガイダンスが存在するが、code_review特有の文脈（複数のFRや修正ファイルの評価）に特化した内容でNG例・OK例を記述することで、各テンプレート間での内容差異を保持する設計とする。

追加するガイダンスが10行以内であることを確認すること。spec.mdのFR-3では「20行以内」という要件が設定されており、最終文言は本セクションに記載した6行（見出し1行・本文1行・NG例1行・OK例2行・締め文1行）の構成で要件を満たしている。
