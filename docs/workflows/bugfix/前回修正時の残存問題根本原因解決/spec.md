## サマリー

- 目的: `definitions.ts` の `code_review.subagentTemplate` に「評価結論フレーズの重複回避（特化ガイダンス）」セクションを追加し、複数のFR・修正ファイルを評価する際のバリデーター重複行検出エラーを再発させないようにする。
- 主要な決定事項: 変更対象は `workflow-plugin/mcp-server/src/phases/definitions.ts` の1箇所のみ。追加するセクション名は `## 評価結論フレーズの重複回避（特化ガイダンス）` とし、manual_testフェーズのテンプレートに既に存在するセクション名と完全一致させる。
- 挿入位置: `## パフォーマンスセクションの行数ガイダンス` ブロックの末尾と `## 出力` の間に追加する。
- 追加するガイダンス内容: 問題説明・NG例・OK例の3要素を含め、10行以内に収めて保守性を維持する。
- 次フェーズで必要な情報: 追加するガイダンス文字列の正確な内容（NG例・OK例を含む）、および `## 出力` セクション直前への挿入位置の行番号レベルの特定。

## 概要

本タスクは、前回の修正作業で残存していた根本原因を解決することを目的としている。具体的には、code_reviewフェーズのsubagentTemplateに「評価結論フレーズの重複回避（特化ガイダンス）」セクションが欠落しているという問題を修正する。

この欠落により、code_reviewフェーズのサブエージェントが複数のFRや修正ファイルを同一フォーマットで評価する際に、バリデーターの重複行検出エラーが繰り返し発生していた。parallel_verificationの各サブフェーズ（manual_test、security_scan、performance_test）には既に類似のガイダンスセクションが存在しているが、code_reviewには存在していなかったことが直接の原因である。

変更のスコープは `definitions.ts` 内の文字列リテラルへのテキスト追記のみであり、ビジネスロジックやバリデーターのコアロジックには一切手を加えない。変更後はMCPサーバーの再起動が必要となり、CLAUDE.mdルール22に従ってユーザーへの案内を実施する。

対象読者: 本仕様書はimplementationフェーズのサブエージェントが参照することを前提としており、実装者が迷わず変更箇所を特定してEditツールを適用できる粒度で記述している。

変更の影響範囲: code_reviewフェーズのサブエージェントが受け取るプロンプトに新セクションが追加されるが、既存のバリデーターロジックやAPIシグネチャへの影響はなく、既存テストスイートの全件パスが維持される見込みである。

期待される効果: ガイダンス追加後、code_reviewフェーズで複数のFRを評価する際の重複行エラーが発生しなくなることが期待され、サブエージェントのリトライ回数が削減されワークフロー全体の効率が向上する。

## 機能仕様

### FR-1: code_review.subagentTemplateへのガイダンスセクション追加

変更対象ファイルは `workflow-plugin/mcp-server/src/phases/definitions.ts` であり、`code_review.subagentTemplate` 文字列の末尾（`\n\n## 出力\n${docsDir}/code-review.md` の直前）に新セクションを挿入する。

挿入するテキストの論理的位置は、既存の `## パフォーマンスセクションの行数ガイダンス` ブロックの末尾（5観点の記述例を含むブロック終端）と `\n\n## 出力` 開始部の間である。

追加するセクションのMarkdown見出しは `## 評価結論フレーズの重複回避（特化ガイダンス）` とする。このセクション名はmanual_testフェーズのsubagentTemplateに既存するセクション名と完全に一致させる必要がある（FR-2参照）。

追加するセクションの内容には以下の3要素を含めること。

- 問題の説明: 複数のFR・修正ファイル・修正箇所を同一フォーマットで評価する際にバリデーターの重複行検出によりエラーが発生するという事実を説明する文章。「バリデーターが検出するフレーズ」「成果物バリデーターのエラー対象となる表現」等の間接参照を使用し、禁止語そのものは成果物本文に記述しない。
- NG例: 設計書の要件確認結果を示す同一フレーズをFR-1・FR-2・FR-3で繰り返す形式が3回以上の同一行としてバリデーターのエラーを引き起こすことを示す例。
- OK例: FR番号と対象ファイル名・関数名などの固有識別子を行に含めて各行を一意にした記述形式を示す例。

追加するテキストのドラフト（TypeScript文字列リテラル内での改行は `\n` で表現）は以下の通りである。

---

ガイダンステキストのドラフト（Markdown本文として記述した場合の表現）:

```
## 評価結論フレーズの重複回避（特化ガイダンス）
複数のFRや修正ファイルを同一フォーマットで評価する場合、バリデーターの重複行検出によりエラーが発生する。FR番号や対象ファイル・関数名などの固有識別子を行に含めて各行を一意にすること。
- NG: 「spec.mdの要件確認結果は以下のとおりである。」をFR-1・FR-2・FR-3で繰り返す（3回以上の同一行でエラー）
- OK: 「FR-1（code_review subagentTemplate末尾へのガイダンスセクション追加）に関するspec.md要件の確認結果は以下のとおりである。」
- OK: 「FR-2（manual_testテンプレートとのセクション名一貫性確保）に関するspec.md要件の確認結果は以下のとおりである。」
複数のFR・修正箇所の評価行は必ずFR番号または対象ファイル・関数名の識別子を含めて一意にすること。
```

---

このドラフトをTypeScript文字列リテラル内の `\n\n` 区切りに変換した挿入テキストは以下の通りである。

```
\n\n## 評価結論フレーズの重複回避（特化ガイダンス）\n複数のFRや修正ファイルを同一フォーマットで評価する場合、バリデーターの重複行検出によりエラーが発生する。FR番号や対象ファイル・関数名などの固有識別子を行に含めて各行を一意にすること。\n- NG: 「spec.mdの要件確認結果は以下のとおりである。」をFR-1・FR-2・FR-3で繰り返す（3回以上の同一行でエラー）\n- OK: 「FR-1（code_review subagentTemplate末尾へのガイダンスセクション追加）に関するspec.md要件の確認結果は以下のとおりである。」\n- OK: 「FR-2（manual_testテンプレートとのセクション名一貫性確保）に関するspec.md要件の確認結果は以下のとおりである。」\n複数のFR・修正箇所の評価行は必ずFR番号または対象ファイル・関数名の識別子を含めて一意にすること。
```

このテキストを `subagentTemplate` 文字列の末尾付近で `## 出力` の直前に挿入することで、実装が完了する。

### FR-2: 既存テンプレートとの一貫性確保

追加するセクションの見出しは `## 評価結論フレーズの重複回避（特化ガイダンス）` とし、manual_testフェーズのsubagentTemplateに含まれる同名セクションと完全一致させる。

NG例・OK例の記述スタイルは、security_scanフェーズやperformance_testフェーズのsubagentTemplateに既存する重複回避ガイダンスと同様の形式（箇条書き・記号を用いたNG/OK対比）を採用する。

code_review特有の文脈として、「複数のFRや修正ファイルの評価」という状況に特化したNG例とOK例を採用する。manual_testの「テストシナリオの合否判定行」・security_scanの「FR評価結論行」・performance_testの「計測対象評価行」とは異なる文脈の記述例を使用し、テンプレート間の内容差異を保持する。

### FR-3: definitions.tsの変更内容の妥当性

挿入対象の文字列リテラルは `subagentTemplate: '...'` の形式でシングルクォートで囲まれている。挿入テキスト内に含まれる日本語の読点・括弧等の特殊文字はTypeScriptの文字列リテラルでエスケープ不要な文字であり、そのまま記述できる。

挿入後の `subagentTemplate` 文字列の末尾は `...パフォーマンスの行数ガイダンス系テキスト...` → `\n\n## 評価結論フレーズの重複回避（特化ガイダンス）\n...` → `\n\n## 出力\n${docsDir}/code-review.md` という順序になる。

追加するガイダンスセクションの行数は10行以内に収め、保守性を維持する（NFR-3の要件「20行以内」の範囲内）。

## 実装計画

### 変更対象の特定

変更対象ファイル: `workflow-plugin/mcp-server/src/phases/definitions.ts`

変更箇所は `code_review.subagentTemplate` プロパティ値の文字列内部であり、以下の位置に挿入する。

挿入前の末尾部分（概略）:
```
...\n- レスポンスタイムへの影響（重い処理の有無と改善可能性を確認した結果を記述する）\n\n## 出力\n${docsDir}/code-review.md'
```

挿入後の末尾部分（概略）:
```
...\n- レスポンスタイムへの影響（重い処理の有無と改善可能性を確認した結果を記述する）\n\n## 評価結論フレーズの重複回避（特化ガイダンス）\n...\n\n## 出力\n${docsDir}/code-review.md'
```

### 実装手順

実装フェーズでのsubagentが実施する作業ステップを以下に示す。

ステップ1: `definitions.ts` の `code_review.subagentTemplate` 文字列内の末尾部分 `\n\n## 出力\n${docsDir}/code-review.md'` を検索し、その直前に挿入テキストを配置する。

ステップ2: Edit toolを使用して `old_string` に末尾の `\n\n## 出力\n${docsDir}/code-review.md'` を指定し、`new_string` に挿入テキストを追記した形式で指定することで挿入を実現する。

ステップ3: 編集完了後、`workflow-plugin/mcp-server/src/phases/definitions.ts` を読み込んで挿入が正しく行われたことを確認する。確認観点は「`## 評価結論フレーズの重複回避（特化ガイダンス）` セクションが `## パフォーマンスセクションの行数ガイダンス` ブロックと `## 出力` の間に存在すること」である。

ステップ4: `cd workflow-plugin/mcp-server && npm run build` でTypeScriptをトランスパイルし、エラーなく完了することを確認する。このコマンドはimplementationフェーズのBashカテゴリで許可された `npm run build` に該当する。

ステップ5: ビルド成功後、CLAUDE.mdルール22に従いMCPサーバーの再起動をユーザーに案内する。subagentはMCPサーバーの再起動操作自体を実施することはできないため、ユーザーへの案内を成果物またはフェーズ完了メッセージに含める。

### 変更影響範囲

変更はTypeScript文字列リテラルへのテキスト追記のみであり、以下のコンポーネントへの影響はない。

- `artifact-validator.ts`: バリデーターのロジックには変更なし。
- `state-manager.ts`: 状態管理には変更なし。
- parallel_verificationの4サブフェーズ定義: 前回修正済みのため変更不要。
- テストファイル（`definitions.test.ts`）: テンプレート文字列のエクスポートAPIに変更はないため、既存テストは全てパスする想定。

## テスト計画

### テスト方針

今回の変更はTypeScriptファイルの文字列リテラルへの追記のみであり、ロジックの変更は一切含まれない。そのため、既存テストスイートの全テストが引き続きパスすることを確認することが主要なテスト観点となる。

新規テストケースの追加についても検討するが、test_implフェーズで具体的なテストコードを設計・実装する。以下はtest_designフェーズで設計すべきテストケースの候補を示す。

### テストケース候補

テストケース候補1: `code_review.subagentTemplate` に `## 評価結論フレーズの重複回避（特化ガイダンス）` が存在することを確認するテスト。`definitions.ts` の `PHASE_GUIDES` からcode_reviewのsubagentTemplateを取得し、当該セクション見出し文字列が含まれることをアサートする。

テストケース候補2: `code_review.subagentTemplate` が `## 出力` セクションで終わることを確認するテスト。挿入後もテンプレートの末尾構造が維持されていることを確認する。

テストケース候補3: 既存のユニットテスト一式（`workflow-plugin/mcp-server/src/phases/__tests__/definitions.test.ts` を含む全テストファイル）が `npm test` で全件パスすることを確認するリグレッションテスト。

### リグレッションテストの実行

テストコマンドは `cd workflow-plugin/mcp-server && npm test` であり、testingフェーズのBashカテゴリ（`npm run test`）で許可されている。実行結果はworkflow_record_test_resultで記録する。

既存テストが61件以上存在することはrequirements.mdで確認済みであり、追加するガイダンスセクションによって既存テストが失敗する可能性は極めて低いと判断している。根拠は、既存テストがエクスポートAPIの構造（PHASES_LARGE・PHASE_GUIDES・resolvePhaseGuide等のシグネチャ）を検証するものであり、テンプレート文字列の内容そのものを厳密に一致検証するテストは含まれていないためである。
