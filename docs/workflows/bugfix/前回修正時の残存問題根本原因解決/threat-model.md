## サマリー

本ドキュメントは、`workflow-plugin/mcp-server/src/phases/definitions.ts` のcode_reviewサブフェーズ定義内 `subagentTemplate` への1セクション追記について、STRIDE脅威モデルを用いて安全性を分析したものである。
変更の性質は静的なテンプレート文字列の追記のみであり、外部入力処理・認証・ネットワーク通信は一切含まれない。
STRIDE各カテゴリを精査した結果、脅威の大半は本変更において「非該当」と判定された。
主要なリスクは「バリデーター禁止語の転記によるフィードバックループ形成」であり、リスクレベルは「低」と評価した。
後続フェーズでは、ガイダンス文字列に禁止語が含まれていないことをplanning/test_designで確認し、MCPサーバー再起動後に動作を検証することを推奨する。

---

## 変更概要

変更対象は `definitions.ts` のTypeScript文字列リテラル（`subagentTemplate` プロパティ）への追記1件のみである。
追加するのは「評価結論フレーズの重複回避（特化ガイダンス）」セクション（20行以内）であり、実行時の動作変化はcode_reviewサブエージェントへ渡されるプロンプト文字列の末尾に新セクションが付加されることのみである。
データストアへの書き込み、外部APIへの呼び出し、認証情報の処理は変更対象外である。
本変更はGitコミットとして記録されるため、変更履歴の追跡性は既存の管理体制と同等に維持される。
ビルドツール（TypeScriptコンパイラ）によって `dist/*.js` に変換されることで、MCPサーバープロセスへ反映される。

---

## 脅威分析

### Spoofing（なりすまし）

本変更はTypeScriptソースファイルの静的テキストを編集するものであり、ランタイムにおける認証や身元確認の仕組みには関与しない。
なりすましが成立する前提となる「外部アクターがリクエストを送信する経路」が変更対象に存在しないため、Spoofingの脅威は本変更においては非該当と判定する。
リスクレベル: 非該当（経路なし）

### Tampering（改ざん）

テンプレート文字列の改ざんが生じる経路として2つを検討した。
経路1は「ソースファイルへの不正なコンテンツ混入」である。攻撃者がリポジトリへのコミット権限を取得してバリデーター禁止語をテンプレートに埋め込んだ場合、以降のcode_reviewサブエージェントが生成する成果物がバリデーションで失敗し続けるという影響が想定される。対策はコードレビュー（プルリクエスト審査）による検出であり、既存の開発フローで十分に対応可能である。
経路2は「トランスパイル後のdist/*.jsへの改ざん」である。ただしdist/はビルド生成物であり本番環境への配備前にnpm run buildで上書きされるため、永続的な改ざんには至りにくい。
リスクレベル: 低（コードレビューで検出可能、影響はMCPサーバー再起動で解消）

### Repudiation（否認）

変更はGitコミット履歴に記録されるため、「誰がどの行を変更したか」は追跡可能である。
テンプレート文字列はソースコード管理下にあり、変更の否認は困難である。
リスクレベル: 非該当（Gitが証跡を保持）

### Information Disclosure（情報漏洩）

追加するガイダンス文字列はNG例・OK例の説明であり、機密情報・認証情報・個人情報は含まれない。
このテンプレートはMCPサーバーがサブエージェントに渡すプロンプトに埋め込まれるが、プロンプトはClaude APIの通信路（TLS）を経由するため、通信経路上の漏洩リスクは既存の仕組みと変わらない。
ガイダンス文字列に秘匿すべき内容が存在しないため、情報漏洩のリスクは実質的にゼロである。
リスクレベル: 非該当（追加コンテンツに機密情報なし）

### Denial of Service（サービス妨害）

テンプレート文字列の追記によってプロンプト長が増加する。増加量は20行以内（概算1,000文字未満）であり、Claude APIのコンテキスト上限（数万トークン）に対して無視できる量である。
サーバーのメモリ消費も文字列長の増加分のみであり、実用上の問題は生じない。
リスクレベル: 非該当（増加量が無視できる水準）

### Elevation of Privilege（権限昇格）

本変更はプロンプトテキストの追記であり、MCPサーバーの実行権限・ファイルシステムアクセス権・OSプロセス権限には影響しない。
サブエージェントへ渡すテキストが変化するだけであり、そのテキストを解釈するのはClaudeモデルであって権限昇格の経路にはなり得ない。
リスクレベル: 非該当（権限変更の仕組みなし）

## バリデーターバイパスリスク（追加分析）

ガイダンスの意図を誤解したサブエージェントがバリデーター禁止語を成果物本文に直接記述するリスクを分析する。
リスクのシナリオ: ガイダンスが「バリデーターが検出するパターン」を間接参照で説明できていない場合、サブエージェントが禁止語を成果物内に転記し、フィードバックループが形成される。
対策1: ガイダンスのNG例・OK例には禁止語そのものを記述せず、「バリデーターが検出するフレーズ」等の間接参照を使用する。
対策2: ガイダンス文字列自体がバリデーターの禁止パターンに該当しないことを、実装フェーズでの変更後にnpm run buildおよびworkflow_nextの成果物バリデーションで確認する。
対策3: manual_testフェーズの既存ガイダンスセクション（前回FR-3で追加済み）と同一の表現スタイルを踏襲することで、実績のある安全な文言を再利用する。
リスクレベル: 低（実績あるスタイルの再利用と間接参照で制御可能）

## リスク

ここでは、上記の脅威分析から抽出した主要リスクを一覧化し、各リスクの深刻度・発生可能性・対応方針を示す。

リスク1: バリデーター禁止語転記によるフィードバックループ
- 深刻度: 中（code_reviewフェーズが繰り返し失敗することで開発速度が低下する）
- 発生可能性: 低（ガイダンスの記述スタイルを間接参照に統一することで防止可能）
- 対応方針: ガイダンスのNG例・OK例を間接参照表現で記述し、実装完了後のバリデーション検証で確認する

リスク2: ソースファイルへの不正コンテンツ混入（Tamperingの経路1）
- 深刻度: 低（コードレビューで検出可能であり、影響範囲はcode_reviewフェーズのサブエージェント動作に限定される）
- 発生可能性: 非常に低（リポジトリのコミット権限管理が前提）
- 対応方針: 既存のプルリクエスト審査フローで対応済み。追加の仕組みは不要と判断した

リスク3: MCPサーバー再起動忘れによるモジュールキャッシュ問題
- 深刻度: 中（旧バージョンのテンプレートが使われ続けることで変更が無効になる）
- 発生可能性: 中（CLAUDE.mdルール22を認識していないOrchestratorが再起動を省略するケースが想定される）
- 対応方針: definitions.ts変更後は必ずnpm run buildおよびMCPサーバー再起動を実施する（CLAUDE.mdルール22準拠）

リスク4: dist/*.js直接改ざんによるテンプレート汚染（Tamperingの経路2）
- 深刻度: 低（次回ビルド時に上書きされるため恒久的な影響はない）
- 発生可能性: 非常に低（ビルドプロセスが改ざんを無効化する）
- 対応方針: 既存のビルドパイプラインで対応済み。追加対策は不要と判断した

## 対策

本変更に対して実施すべき対策を優先度順に整理する。

対策A（必須）: ガイダンス文字列に禁止語を直接記述しないこと。説明が必要な場合は「バリデーターが検出するフレーズ」「成果物バリデーターのエラー対象となる表現」等の間接表現を使用すること。これにより、バリデーターバイパスリスクを低減できる。

対策B（必須）: 実装フェーズ終了後に `npm run build` でトランスパイルを実行し、MCPサーバーを再起動すること（CLAUDE.md ルール22）。モジュールキャッシュに起因する「変更前のテンプレートが使われ続ける」問題を防止する。

対策C（推奨）: 既存のmanual_testフェーズのテンプレートからガイダンスの文言パターンを参照し、一貫性を維持すること。独自の表現を発明するより既存の実績ある表現を再利用することで、表現上のリスクを低減できる。

対策D（推奨）: code_reviewフェーズの既存ユニットテスト（`definitions.test.ts`）が引き続きパスすることをtestingフェーズで確認すること。テンプレート文字列の追記がエクスポートAPIの変更を伴わないことを自動テストで担保する。

これらの対策は既存の開発フローおよびCLAUDE.mdのルールの範囲内で実施可能であり、新たな仕組みの導入は不要である。
