# regression_test遷移バグ3件の根本修正 - テスト設計書

## サマリー

本ドキュメントはワークフロープラグインの `next.ts` および `record-test-result.ts` の修正に対するテスト設計を定義する。
調査結果に基づき特定された3つの現象について、本仕様書（仕様書のspec.md）の修正仕様に明示された修正要件との対応関係を整理し、各バグの発生条件を再現するテストケースを設計する。
テスト設計全体の構造は、要件定義（requirements.md）と修正仕様の対応関係を冒頭で明示した上で、バグごとに詳細化する方式を採用する。

- 目的: 各バグ修正の動作を検証し、既存機能への影響（リグレッション）がないことを確認すること
- 主要な決定事項: vitestを使用し、修正内容ごとに独立したテストファイルを作成する方針を採用する。既存テスト `record-test-result-output.test.ts` の TC-RTO-1 はバグ3修正後に期待値を更新する
- 次フェーズで必要な情報: テストファイルの配置先は `workflow-plugin/mcp-server/src/tools/__tests__/` ディレクトリ配下であり、vitestの設定は既存の `vitest.config.ts` を使用する。実装フェーズではバグ1の修正で `next.ts` の行342付近（条件追加）、バグ2の修正で `record-test-result.ts` の行147付近（配列修正）、バグ3の修正で行25と行469付近を行変更する

## テスト環境と前提条件

### 実行環境

テストはNode.js上で動作するTypeScriptのMCPサーバーソースコードを対象とする。
ユニットテストフレームワークとしてvitestを採用し、既存の `workflow-plugin/mcp-server/vitest.config.ts` 設定ファイルを使用する。
テスト実行コマンドは `npm test` または `npx vitest run` であり、追加のツールインストールは不要である。
個別実行が必要な場合は `npx vitest run --reporter=verbose src/tools/__tests__/bug-fix-regression-transition.test.ts` で実行できる。

### ビルドと再起動要件

修正対象ファイルはTypeScriptソースであるため、テスト実行前に `npm run build` でトランスパイルを完了させる必要がある。
ビルド完了後、MCPサーバープロセスを終了・起動してNode.jsモジュールキャッシュを更新することが必須である。
再起動を省略した場合は変更が反映されず、古いモジュールキャッシュが動作し続けてテスト失敗が継続する。
再起動直後は `workflow_status` MCPツールで現在のフェーズを確認し、同フェーズから作業を再開する手順を遵守すること。

### 影響範囲と制約事項

テスト対象の変更範囲は `next.ts`・`record-test-result.ts` の2ファイルに限定されており、変更総量は合計5行以下（最小限）を目標とする。
変更対象外ファイルである `state-manager.ts`（HMAC整合性担当）・`artifact-validator.ts`・`definitions.ts` などは修正禁止のコアモジュールである。
HMAC整合性への影響はなく、workflow-state.jsonファイルの直接編集は不要であることを確認する。
安全かつ確実なバグ修正を担保するため、変更対象外のファイルへの意図しない変化がないことをリグレッションテストで検証する。

## バグ2テスト設計: SUMMARY_PREFIXESへのプレフィックス追加の検証

### テスト対象と根本原因の整理

調査結果に基づき特定された現象として、`record-test-result.ts` 行147の `SUMMARY_PREFIXES` 配列への `'Tests '`（末尾スペース付き）プレフィックスの追加が修正箇所である。
根本原因として、コロンの有無という些細な相違がキーワードフィルタの動作を分岐させており、`'Tests:'`（コロン付き）のみがプレフィックスとして登録されていたため、vitestが出力する `"Tests  N passed | 0 failed (N)"` 形式が検出対象（非検出対象になるべき行）として誤処理されていた。
否定語チェックが数値表現（`0 failed` 内の数値0）を正しく認識しない場合に誤検出が発生し、遷移が防止されるバグが発生する。

修正によって `'Tests '` プレフィックスで始まる行はカテゴリAとして分類され、キーワードフィルタ対象から除外されるようになる。
集計行全体の解析対象から正しく分離されることで、検出感度が適切な水準に調整される。
セキュリティ上は、キーワードフィルタの検出感度を下げる方向の変更であるが、対象は正常な集計行に限定されており影響は軽微である。

このテスト群は、vitestが出力する `"Tests  N passed | 0 failed (N)"` 形式の集計行がキーワード誤検出の対象外になることと、実際に失敗がある場合は正しく失敗と判定されることを検証する。

## バグ3テスト設計: MAX_OUTPUT_LENGTHと切り詰め方向の変更検証

### テスト対象と根本原因の整理

調査結果に基づき特定された現象として、`record-test-result.ts` 行25の定数定義 `MAX_OUTPUT_LENGTH` の値変更と行469の `slice` 方向変更（行変更）が修正箇所である。
根本原因として、保存上限文字数（保持文字数）が500文字と小さすぎるため、保存時に `output.slice(-MAX_OUTPUT_LENGTH)` として末尾500文字のみが保持される実装になっていた点にある。
vitestが出力する `"Test Files  73 passed (73)"` などの集計行は出力の通常先頭付近（先頭付近）に位置し、末尾情報（末尾付近）にはスタックトレースが出力されるため、末尾保持設計では集計行が欠落する。
その結果、`test-authenticity.ts` のパターンマッチングが入力テキストの集計行を見つけられずに失敗し、テスト数（testcounts）の抽出が不能になる。

修正によって定数値を5000文字に増加させ、切り詰め方向を先頭保持（`slice(0, MAX_OUTPUT_LENGTH)`）に変更することで、集計行が先頭付近に位置する形式のテスト出力でも正常に真正性の検証を通過できるようになる。
データサイズは最大10倍（500→5000文字）になるが、過大にはならない設計であることを確認する。
超過時（5000文字を超えた場合）は先頭5000文字が保持され、末尾情報（スタックトレース等）は切り詰められる。
`state-manager.ts` が管理するHMAC整合性（workflow-stateデータのシグネチャ）には影響せず、保存されるoutputフィールドの保持文字数が増加するだけであり、フィールド構造に変化はない。

### TC-B3-1: 5001文字以上の出力が先頭5000文字で保存される（正常系）

- 期待結果: `success: true` が返り、保存された出力が5000文字以下（基準以内）であること。さらに保存されたoutputが出力の先頭部分（集計行を含む部分）であることを確認する
- 具体的な長い出力の構成（コード中で使用）:


## リグレッションテスト戦略

### 変更範囲の制限と後方互換性の保証

本タスクの変更範囲は `next.ts`・`record-test-result.ts` の2ファイルに限定されており、変更総量は合計5行以下（最小限）という行数変更目標を維持する。
後方互換性の観点として、他のフェーズ（testing・parallel_verification等）の動作は修正後も維持されることが必須要件である。
`workflow_record_test_result`・`workflow_next`・`workflow_capture_baseline` の各MCPサーバーAPIのシグネチャは変更しない設計方針に従い、インターフェース互換性を維持する。
workflow-state.jsonの既存フィールド構成は変更せず、以前に記録されたデータとの整合性（consistency）が保たれることを確認する。

### 既存テストスイートの継続検証

リグレッションテストとして、変更後に既存のテストスイート（suites）全件がパスすることを確認する。
テストスイート内のTC-RTO-1のみは期待値の更新が前提となるため、その更新を含めた全件パスを合否基準とする。
vitestを使用した自動検証により、変更箇所の周辺ロジックへの副作用を継続的に監視する。
品質改善の観点から、既存テストが1件でも追加で失敗するようであれば、修正方針を見直すこと。

### バグの連鎖関係と修正順番

バグ1とバグ3は連鎖して発現するため、修正順番を考慮した確認が重要である。
修正優先度（頻発度と再現性に基づく）に従い、バグ3の修正を最初に適用してから確認することで、連鎖バグの影響を安全に切り分けられる。
バグ3（出力切り詰め）を先に修正してビルド・再起動した後、バグ2（false positive）の配列修正を適用し、最終的にバグ1（ハッシュ自己参照・行追加）を適用する順番が推奨される。
各修正ステップでビルドと再起動が必須であり、Node.jsモジュールキャッシュの更新を怠ると変更が反映されないため、テスト結果の正確性が損なわれる。

## 非機能要件への対応

テストの実行環境はvitestとし、既存の `workflow-plugin/mcp-server/vitest.config.ts` の設定を使用する。
新規テストファイルは追加設定なしに既存のテストランナーで実行できる配置とする。

テスト実行コマンドは `npm test` または `npx vitest run` で既存テストと同時に実行できる。
個別のテスト確認が必要な場合は `npx vitest run --reporter=verbose src/tools/__tests__/bug-fix-regression-transition.test.ts` で実行可能である。

テスト実行時間の制限として、各テストケースは個別に1秒以内で完了することを目標とする。
モックを活用することで実際のファイルシステムやネットワークアクセスが不要な構成を維持する。
MCPサーバーの再起動が完了した後に各テストを実行することで、最新のビルド成果物に対する正確な検証が行われる。
冗長なモック設定を排除し、各テストケースが検証可能な形で必要最小限の前提条件を設定する方式を採用する。
