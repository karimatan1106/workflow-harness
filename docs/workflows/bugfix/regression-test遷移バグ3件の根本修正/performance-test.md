# performance-test.md — regression_test遷移バグ3件の根本修正

## サマリー

本ドキュメントは `regression_test` フェーズ遷移バグ3件の修正が MCP サーバーのランタイムパフォーマンスに与える影響を静的分析および計測ベースの推定により評価した結果をまとめたものである。
評価対象の修正は next.ts のハッシュスキップ条件追加、SUMMARY_PREFIXES への要素追加、MAX_OUTPUT_LENGTH の500文字から5000文字への拡大の3件である。

- 目的: 3件の修正（ハッシュスキップ追加・SUMMARY_PREFIXES拡張・MAX_OUTPUT_LENGTH拡大）がスループットおよびメモリ使用量に対して許容範囲内であることを確認する
- 主要な決定事項: ローカルの Node.js MCP サーバーはリクエスト頻度が低い（1ユーザーが数十回/日）ため、実行時間の絶対値よりも「修正前後の変化率」を評価基準とした
- 結論として全3件の修正による追加コストの合計は0.5ms以下であり、ローカルネットワーク往復時間（数ms）の1割にも満たないため、ユーザーが体感できる性能劣化は発生しないと判定した
- 次フェーズで必要な情報: 各修正の影響は軽微であり、ボトルネックは SHA-256 ハッシュ計算の入力サイズ増加のみであるが、その絶対増加量（約4.5KB相当）はCPU負荷に影響しないと判定した

## パフォーマンス計測結果

### 計測環境と前提条件

MCP サーバーは Claude Desktop からローカルで起動される Node.js プロセスである。
ユーザーの操作1回（`workflow_record_test_result` 呼び出し）あたりの処理時間が主要な指標となる。
計測は静的コード分析と Big-O 推定に基づく。ベンチマーク実行環境は Windows 11 / Node.js v20 系を想定している。

実際のテスト実行コマンドを発行できない（readonly カテゴリのみ許可）ため、コード分析結果から算定した推定値を以下に示す。

### バグ1修正: next.ts ハッシュ重複チェックスキップ（条件分岐1行）

修正内容は `if (currentPhase !== 'regression_test')` の条件分岐を追加してハッシュ計算を回避するものである。
修正前は `regression_test` フェーズでも `recordTestOutputHash` が呼び出され、SHA-256 計算が実行されていた。
修正後は条件が偽になるため、`regression_test` フェーズの `workflow_next` 呼び出しでは SHA-256 計算が完全にスキップされる。
これは純粋なパフォーマンス改善であり、追加の CPU 時間は発生しない。条件評価自体は文字列比較1回（約1ナノ秒）で完結する。

推定処理時間変化: `regression_test` フェーズの `workflow_next` で約 0.1〜0.5ms の短縮（SHA-256 省略効果）。

### バグ2修正: SUMMARY_PREFIXES への 'Tests ' 追加（配列要素1つ追加）

`SUMMARY_PREFIXES` 配列は `record-test-result.ts` の `validateTestOutputConsistency` 内で使用される。
修正前の配列は4要素（'Tests:', 'Test Files', 'Test Suites:', 'Summary'）であったが、修正後は5要素に増える。
各テスト出力行に対して `SUMMARY_PREFIXES.some(prefix => trimmed.startsWith(prefix))` が呼び出されるため、
最悪ケースでは1行あたり5回の `startsWith` 呼び出しが発生する（修正前は4回）。

推定追加時間: 1000行の出力に対して `startsWith` が最大1000回追加実行される。1回の `startsWith` 呼び出しは約0.001マイクロ秒であるため、合計追加コストは約1マイクロ秒（0.001ms）未満となる。実際的な影響はゼロと評価できる。

### バグ3修正: MAX_OUTPUT_LENGTH 拡大（500→5000文字）と先頭保持への変更

この修正が3件の中でパフォーマンスへの影響が最も大きい。
修正前は出力が500文字を超えた場合に末尾500文字を保持していたが、修正後は先頭5000文字を保持する。
`workflow-state.json` への書き込みサイズが最大で 4500 文字（約4.4KB）増加する可能性がある。

ディスク書き込み負荷: `stateManager.writeTaskState` は JSON.stringify 後にファイルへ同期書き込みを行う。
4.4KB の追加書き込みはローカル SSD 環境で 0.1ms 以下の追加時間であり、無視できる水準である。

SHA-256 入力サイズ変化: `recordTestOutputHash` は `createHash('sha256').update(output).digest('hex')` を実行する。
入力サイズが最大5000文字（約5KB）になる。SHA-256 のスループットは通常数GB/秒であるため、5KBの処理時間は約0.001ms未満となる。修正前（最大500文字）との差異も0.001ms以下であり、計測誤差の範囲内に収まる。

## ボトルネック分析

### 処理コストの相対評価

3件の修正のうち、ランタイムコストに最も関わるのはバグ3修正（MAX_OUTPUT_LENGTH 拡大）である。
以下に各修正の影響を比較する。

バグ1修正（ハッシュスキップ）はパフォーマンスを改善するため、コスト増加はゼロである。
条件分岐の評価コスト（約1ns）は SHA-256 を呼び出すコスト（約0.1〜0.5ms）よりも4〜5桁小さく、実質的な差し引きは改善方向となる。

バグ2修正（SUMMARY_PREFIXES 要素追加）の追加コストは全体の処理時間の0.01%以下であり、ボトルネックにはなり得ない。
実際のテスト出力で集計行（カテゴリAに分類される行）の割合は低く、大半の行はカテゴリBまたはCで早期リターンするため、追加された要素のチェックに到達する頻度は限定的である。

バグ3修正（MAX_OUTPUT_LENGTH 拡大）の影響は `workflow-state.json` のファイルサイズ増加として現れる。
ファイルサイズが大きくなると、`stateManager.readTaskState` の読み込み時間も比例して増加する。
状態ファイルが10KBから15KB程度に増加したとして、JSON.parse の追加コストは約0.1ms以下であり、ユーザーが体感できる遅延は生じない。

### 累積的な影響の評価

regression_test フェーズの完了フローは `workflow_record_test_result` 1回と `workflow_next` 1回で構成される。
両ツールの呼び出しは MCP プロトコル経由で JSON メッセージとして送受信され、ネットワーク往復時間（ローカルのため数ms）が支配的なコストとなる。
今回の3件の修正による追加コストの合計は0.5ms以下であり、ネットワーク往復時間の1割にも満たない。
ユーザーが体感できる応答速度の変化は発生しないと結論付けられる。

### メモリ使用量への影響

`workflow-state.json` を読み込んでメモリ上に保持する際の追加メモリは最大4.5KB程度である。
Node.js の V8 エンジンにおけるオブジェクトのオーバーヘッドを考慮しても、追加メモリ使用量は数十KB以下に収まる。
MCP サーバーのプロセス全体のメモリ使用量（通常50〜200MB）に対して無視できる割合である。

### 長期的な蓄積リスク

`testOutputHashes` 配列は `regression_test` フェーズで空配列として渡されるため（バグ1修正後）、ハッシュが蓄積されることはない。
一方、`testResults` 配列にはテスト結果が追加されていくため、タスク完了まで `workflow-state.json` のサイズが増加し続ける。
`output` フィールドが最大5000文字になると、複数回のテスト記録で合計サイズが増える可能性があるが、1タスクのライフサイクル（通常数日）では問題になる水準には達しない。

パフォーマンス上の懸念事項がある場合の将来的な対策として、`output` フィールドを `testResults` に直接保存せず別ファイルに切り出す設計変更が考えられるが、現時点では不要である。
