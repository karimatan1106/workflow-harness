# UI設計: 既存バグ3件の根本原因追究と修正

## サマリー

本タスクは既存バグ3件（bash-whitelist.js、loop-detector.js、record-test-result.ts）のバグ修正であり、UI/UXの変更を伴いません。修正対象はワークフロープラグインのバックエンド機構とフック処理であるため、ユーザーインターフェース（画面、画面遷移、コンポーネント）には影響がありません。

代わりに本セクションでは、ユーザーに見える変化として以下を記載します。

**修正によるCLIインターフェース改善:**
- BUG-1修正により、testingフェーズとimplementationフェーズでのnodeコマンド実行がホワイトリスト判定で正常に許可されるようになります。
- BUG-2修正により、ファイルパス正規化の不要なエラーハンドリングが削除され、パス処理がより効率的になります。
- BUG-3修正により、テスト結果記録時の誤検出（失敗件数ゼロの出力やセキュリティ用語の誤判定）が削除されます。

**ユーザー体験の向上:**
- `/workflow` コマンド実行時のnodeコマンドブロックが解消され、スムーズなワークフロー進行が可能になります。
- テスト出力の真正性バリデーションの精度が向上し、無用な警告やエラーが減少します。

本セクションはUI設計フェーズとして形式的に配置されていますが、このタスクの性質上、プレゼンテーション層への変更はありません。


## 変更対象システムの構造

本タスクの修正対象は以下の3つのコンポーネントです。

### コンポーネント1: Bashコマンドホワイトリスト検証機構

**ファイル:** `workflow-plugin/hooks/bash-whitelist.js`

**役割:** ワークフロープラグインのfreezing mechnaismで、testingおよびimplementationフェーズ中のBashコマンド実行を制限し、セキュリティを維持します。

**修正対象:** testingフェーズ（行64）とimplementationフェーズ（行75）のホワイトリスト配列に登録されている'node 'エントリの末尾スペース削除。

**ユーザーへの影響:** nodeコマンド実行時に「ホワイトリスト外のコマンド」と誤判定されていたバグが修正され、正常に実行が許可されます。これによりワークフロー実行中のnodeコマンド（テスト実行、ビルドスクリプト等）がブロックされなくなります。

**CLI表示の変化:** エラーメッセージ「コマンド'node'がホワイトリスト外です」が表示されなくなり、nodeコマンドが正常に実行されます。


### コンポーネント2: ファイルパス正規化メカニズム

**ファイル:** `workflow-plugin/hooks/loop-detector.js`

**役割:** ワークフロー実行中のループ（無限再帰）を検出し、同一ファイルの過度な編集を防止します。ファイルパスを正規化して比較対象にしています。

**修正対象:** normalizeFilePath関数内の到達不能なデッドコード（内側try-catch）を除去。

**ユーザーへの影響:** パス正規化処理の冗長性が削除され、処理が簡潔かつ効率的になります。関数の入出力挙動に変化はなく、ユーザーには直接的な影響がありません。

**内部改善:** Node.jsの仕様理解に基づいてコードの品質が向上し、将来の保守性が改善されます。


### コンポーネント3: テスト出力真正性バリデーション機構

**ファイル:** `workflow-plugin/mcp-server/src/tools/record-test-result.ts`

**役割:** `/workflow record-test <テストファイル>` コマンドでテスト結果を記録する際、テスト出力の矛盾（exitCodeと出力内容の不一致）を検出します。

**修正対象:** validateTestOutputConsistency関数内の大文字キーワード検出ロジック。否定語チェック（「0 Failed」のような失敗件数ゼロパターン）とハイフン結合語チェック（「Fail-Closed」のようなセキュリティ用語）を追加します。

**ユーザーへの影響:** テスト結果記録時の誤検出が減少します。具体的には以下の2パターンでの不正な拒否が解消されます：
- 失敗件数ゼロを示す出力（「0 Failed」など）が正常に受理されるようになります。
- セキュリティ用語の「Fail-Closed」が失敗表現と誤判定されなくなります。

**CLI表示の変化:** `/workflow record-test` コマンド実行時に無用な警告「出力に矛盾が検出されました」が表示されなくなり、ユーザーの混乱が軽減されます。


## CLIインターフェースの動作改善

### 改善点1: nodeコマンド実行の正常化

**実行前の状態:**
```bash
$ /workflow next
[error] コマンド'node'がホワイトリスト外です。
        testing フェーズでは'node'の実行が許可されていません。
```

**修正後の状態:**
```bash
$ /workflow next
[success] Testing フェーズに進みました。
          nodeコマンドが正常に実行されます。
```

**根本原因:** ホワイトリスト登録時の末尾スペースによる前方一致判定の不一致。

**REQ-R6検証の維持:** 修正後も'nodejs'や'nodemon'などの派生コマンドはホワイトリストの境界チェックで引き続き拒否され、セキュリティが維持されます。


### 改善点2: テスト結果記録の精度向上

**実行前の状態:**
```bash
$ /workflow record-test "src/backend/tests/unit/test.ts"
[warning] 出力に矛盾が検出されました。
          出力: "Tests: 5 passed, 0 failed, 5 total"
          exitCode: 0 (成功)
          → 矛盾: 出力に 'failed' が含まれています
```

**修正後の状態:**
```bash
$ /workflow record-test "src/backend/tests/unit/test.ts"
[success] テスト結果を記録しました。
          出力: "Tests: 5 passed, 0 failed, 5 total"
          exitCode: 0 (成功)
          判定: 一貫性OK
```

**根本原因:** 大文字キーワード検出時に否定語チェック（「0 Failed」の「0」）が適用されていなかったこと、およびハイフン結合語の除外ロジックが不足していたこと。

**検出精度の維持:** 修正後も「1 test Failed」のような実際の失敗表現は正常に検出され、品質管理機能が維持されます。


## ワークフロープラグインのアーキテクチャ図

本修正は以下のフック処理パイプラインの3箇所に影響します。

```
ワークフロー実行
    ↓
[Phase Edit Guard] ← 現在のフェーズを検証
    ↓
[Bash Whitelist Hook] ← BUG-1修正ここ
    ├─ Bashコマンド前方一致判定
    ├─ REQ-R6境界チェック
    └─ コマンド実行許可/拒否
    ↓
[Loop Detector Hook] ← BUG-2修正ここ（ファイルパス正規化）
    ├─ ファイルパス正規化処理
    ├─ 編集回数カウント
    └─ ループ検出判定
    ↓
[Test Authenticity Validator] ← BUG-3修正ここ
    ├─ exitCodeと出力内容の一貫性検証
    ├─ キーワード検出（大文字/小文字）
    └─ テスト結果記録許可/拒否
    ↓
タスク状態を次のフェーズへ遷移
```


## ユーザージャーニー: ワークフロー実行フロー

本修正によるユーザー体験の変化を示します。

### シナリオ1: testingフェーズでのnodeコマンド実行

**修正前のジャーニー:**
1. ユーザーが `testing` フェーズ中にテストを実行するコマンドを入力
2. nodeコマンドが実行される
3. bash-whitelist hookがnodeコマンドをホワイトリスト照合 → 末尾スペースミスで不一致
4. 「ホワイトリスト外」エラーが表示される
5. ユーザーは「なぜnodeが拒否されるのか」と混乱
6. ワークフローが停止

**修正後のジャーニー:**
1. ユーザーが `testing` フェーズ中にテストを実行するコマンドを入力
2. nodeコマンドが実行される
3. bash-whitelist hookがnodeコマンドをホワイトリスト照合 → スペース削除で正常にマッチ
4. REQ-R6境界チェックで「node」の次の文字が「s」（パス開始）であることを確認 → 許可
5. nodeコマンドが正常に実行される
6. テスト実行が完了し、次のフェーズへ進む


### シナリオ2: テスト結果記録時の矛盾検出

**修正前のジャーニー:**
1. ユーザーが `/workflow record-test` で成功したテスト結果を記録
2. テスト出力に「0 failed」が含まれている
3. validateTestOutputConsistency がキーワード「failed」を検出 → 否定語チェック未適用で誤検出
4. 「exitCode=0 (成功) なのに出力に 'failed' が含まれている」警告が表示
5. ユーザーは「0 failed なのに失敗と判定されている」と混乱
6. テスト結果記録が拒否される

**修正後のジャーニー:**
1. ユーザーが `/workflow record-test` で成功したテスト結果を記録
2. テスト出力に「0 failed」が含まれている
3. validateTestOutputConsistency が大文字キーワード検出時に isKeywordNegated を呼び出し
4. 「0」が NEGATION_WORDS に含まれているため「失敗」と判定されない
5. テスト結果が正常に記録される
6. ユーザーは何の警告も見ずにワークフロー進行が続く


## CLIインターフェース設計

本タスクはGUI画面を持たないCLIツール（ワークフロープラグイン）のバグ修正であり、CLIインターフェースの設計変更はありません。
修正はホワイトリスト判定ロジック、パス正規化ロジック、テスト出力バリデーションロジックの内部改善に限定されます。
CLIコマンドの引数仕様、オプション仕様、出力フォーマット仕様は変更されません。
ユーザーが直接操作するコマンド体系（`/workflow next`、`/workflow record-test`等）のインターフェースはそのまま維持されます。
修正後のCLIインターフェースにおいて、ユーザーが新たに覚えるべき操作や手順はありません。


## エラーメッセージ設計

BUG-1修正により、nodeコマンドがホワイトリストで誤ブロックされた際のエラーメッセージが表示されなくなります。
BUG-3修正により、テスト結果記録時の誤検出に起因する矛盾警告メッセージが表示されなくなります。
いずれも既存のエラーメッセージ文言を変更するものではなく、誤って発生していたエラー条件を解消する修正です。
正当なエラー条件（派生コマンドのブロック、真の矛盾検出）に対するメッセージは従来どおり表示されます。
BUG-2のデッドコード除去ではconsole.errorの出力パスが削除されますが、到達不能コードのため実際のメッセージ出力に変化はありません。


## APIレスポンス設計

本タスクの修正対象はClaude CodeのフックシステムとMCPサーバー内部のバリデーションロジックであり、外部向けAPIエンドポイントは存在しません。
MCPサーバーのworkflow_record_test_resultツールのレスポンス形式（success/failureとメッセージ）は変更されません。
バリデーション判定ロジックの改善により、これまで誤ってfailureを返していたケースがsuccessに変わりますが、レスポンス構造自体は同一です。
ホワイトリストフックとループ検出フックはMCPプロトコルではなくClaude Codeフック機構で動作しており、API形式のレスポンスは関与しません。
将来的にMCPサーバーのバリデーションエラー詳細を拡張する場合は、別タスクとして対応します。


## 設定ファイル設計

本タスクでは設定ファイル（.claude/settings.json等）の変更は不要です。
bash-whitelist.jsのホワイトリスト定義はソースコード内にハードコードされており、外部設定ファイルからの読み込みではありません。
loop-detector.jsの閾値設定（編集回数5回/5分間隔）も同様にソースコード内定数であり変更対象外です。
record-test-result.tsのBLOCKING_FAILURE_KEYWORDSとNEGATION_WORDSもソースコード内配列であり、設定ファイル化は本タスクのスコープ外です。
将来的にホワイトリストやキーワード設定を外部化する場合は、設計フェーズから再検討が必要です。


## アクセシビリティと互換性

本修正はUIコンポーネント、画面設計、インタラクションデザインに影響しません。
エラーメッセージやコマンド実行結果の表示フォーマットは変更されないため、既存のスクリーンリーダー対応やテキスト形式のログ出力に影響はありません。
BUG-1およびBUG-2の修正は既存の入出力仕様を変えておらず、BUG-3も内部ロジックの改善でありAPIインターフェースには影響しません。
ワークフロープラグインのバージョン内での修正のため、既存のプロジェクト設定ファイルやスクリプトを修正する必要はありません。
CLIツールとしてのキーボード操作性、コマンドヘルプの表示、エラーコードの体系は全て維持されます。


## 影響を受けるコマンドとイベント

BUG-1はtestingフェーズとimplementationフェーズでのBashコマンドホワイトリスト判定に影響し、nodeコマンドの実行が正常に許可されるようになります。
具体的にはnpm test、node script.js、npx vitest等のnodeベースのコマンドがブロックされなくなります。
BUG-2はワークフロー全体のファイル編集追跡に影響しますが、normalizeFilePathの入出力は変わらないためユーザーには不可視の改善です。
BUG-3はtestingフェーズとregression_testフェーズでのworkflow_record_test_resultツール呼び出しに影響し、誤検出が減少します。
pnpm run buildやnpx vitestのように内部でnodeプロセスを起動するコマンドも、BUG-1修正の恩恵を間接的に受けます。


## デザインシステムとの整合性

本タスクはバックエンド/フック機構の修正であるため、デザインシステム、カラーパレット、タイポグラフィ、スペーシング規則に影響しません。
修正前後でエラーメッセージやコマンド出力のフォーマットは変更されず、既存のCLIスタイルガイドに準拠した出力が継続されます。
ワークフロープラグインのフック出力（console.warn、console.error）のフォーマットも維持されます。
CLIの色付きテキスト出力やプログレスバー表示などの視覚的要素には一切変更がありません。
将来的にCLIのデザインシステムを導入する場合も、本修正は内部ロジックのみの変更のため競合しません。
