## サマリー

- 目的: 開発環境整合性修正タスクの全4ステップに対して、手動検証チェックリスト形式のテスト設計を提供する。workflow-stateのHMACによる整合性保護、discover-tasksキャッシュ（task-cache）の陳腐化防止、リポジトリポインタの正確な更新を含む検証を網羅する
- 主要な決定事項:
  - テスト対象はソースコード実装ではなくGit操作（コミット・ファイル作成・設定確認）であるため、すべてのテストケースは手動検証チェックリストとして設計する
  - 各テストケースはgitコマンドの出力結果（exit code・標準出力の内容）を合否基準とする
  - ステップ間の依存関係（Step1完了後にStep3を実施）を考慮した段階的な実行順序を明示する
  - 異常系テストは各ステップで発生しうるエラー状態と回復手順を記述し、ブロック状態の解消方法を示す
  - 全体統合テストはすべてのステップ完了後に実施し、リポジトリ全体の整合性を総合評価する
  - ワークフロープラグインのアーティファクト検証フローとフックエントリの整合性も確認対象に含める
- 次フェーズで必要な情報:
  - 各テストケースはtest-design.mdに従って手動で実施し、合否を記録してtesting成果物に反映する
  - Step1のコミット後に得られるworkflow-pluginのHEADハッシュはStep3の検証で、リポジトリポインタとインデックスが一致するかの確認に使用する

## テスト方針

### 全体方針

本タスクはGitリポジトリの整合性修復のみを目的としており、ソースコードの新規実装を伴わない。
そのため、ユニットテストやインテグレーションテストを自動化する追跡対象が存在せず、テストはすべてgitコマンドを用いた手動検証として実施する。
各テストケースは「実行するコマンド」「期待される出力条件」「合否判定基準」の3要素で記述し、実施者がそのまま手順を追えるよう設計する。
前回の不整合（.gitmodules欠落・bash-whitelist.js未コミット・サブモジュールポインタ陳腐化）を根本的に解消していることを客観的なgitコマンド出力で確認する。
テスト全項目の合格条件を満たすことが、整合性回復の達成基準となる。
修正実施後に全テストが合格した最終状態が、本タスクの完了条件である。

### テストスコープとテスト対象の範囲

テスト対象の変更対象は以下の4つのスコープに分類される。
第1スコープは `workflow-plugin` リポジトリ内の `hooks/bash-whitelist.js` および `.gitignore` の修正（FR-2）であり、Step1完了後に検証する。
第2スコープは親リポジトリへの `.gitmodules` ファイルの新規作成（FR-1）であり、Step2完了後に検証する。
第3スコープは親リポジトリの `workflow-plugin` サブモジュールのリポジトリポインタ更新（FR-3）であり、インデックスに正式登録されたか確認する。
第4スコープは `.claude/settings.json` のフックエントリ確認（FR-4）であり、UserPromptSubmit欠落と check_ocr.py 参照の不在を確認する。
除外スコープとして、mcp-server/hooks/lib/ と mcp-server/src/verify-sync.test.ts は .gitignore 追加により追跡対象から除外されるため、個別の内容検証は行わない。

非機能要件については、NFR-1（Gitリポジトリ整合性）に対応する統合検証を統合テストで実施する。
NFR-2（既存ワークフロー機能への無影響）については、bash-whitelist.jsの既存フェーズ定義が変更されていないことをgit diffで確認するテストケースを設ける。
NFR-3（冪等性）については、Step3のgit addを再実行しても同一ハッシュが記録されることを確認する観点で検証する。
NFR-4（最小侵襲性）については、修正範囲をgit整合性に必要な最小限に限定し、変更対象ファイル以外のフック実装ファイルへの機能変更がないことをコミット差分で確認する。bash-whitelist.jsへの変更内容は行追加1行のみであり、実装計画の具体的な変更後の状態が保たれていることを検証する。

### 合否基準の定義

各テストケースの合否判定は以下の基準に従う。
コマンドが終了コード0で完了し、かつ期待する文字列が標準出力に含まれていれば合格とする。
コマンドが終了コード非0を返す、または期待する文字列が出力に含まれない場合は不合格とし、対応する回復手順を実施してから再テストを行う。
git statusの出力が「nothing to commit, working tree clean」または「nothing to commit (working directory clean)」であることをクリーン状態の定義とする。
workflow-stateのHMAC整合性が失われた状態は修正作業中にブロックを引き起こすリスクがあるため、HMAC関連の変更（workflow-state.jsonの直接編集等）は一切実施せず、MCPサーバー経由の操作のみを使用する。

### テスト実施場所と事前準備

Step1のテストはworkflow-pluginディレクトリ（C:/ツール/Workflow/workflow-plugin）で実施する。
Step2、Step3、Step4および統合テストは親リポジトリのルートディレクトリ（C:/ツール/Workflow）で実施する。
誤ったディレクトリでgitコマンドを実行すると意図しないリポジトリに対して操作が行われるため、各テストケース実施前にカレントブランチおよびカレントディレクトリを確認することを推奨する。
workflow-pluginのブランチはmainであり、親リポジトリのブランチはmasterであることを事前に確認する。
discover-tasksキャッシュ（task-cache）が陳腐化している場合、フックがステイルなフェーズ情報を返すリスクがあるため、テスト実施前にtask-index.jsonのキャッシュ状態を確認することが推奨される。
実行時に自動生成される mcp-server/hooks/lib/ ディレクトリは、格納されるファイルが環境固有であるため .gitignore で除外し、削除状態（git管理外）として維持されることを想定している。
この不在確認は専用操作（grep等）ではなく git status の出力で間接的に行う。

### 段階構成と実装順序の制約

テストは段階構成に従い、依存関係のある順序で実施する。
Step1のテストを完全に合格させた後でなければStep3のテストを開始することはできない。これはStep1のコミット完了前にStep3のgit addを実行すると、インデックスに古いリポジトリポインタが記録されるリスクがあるためである。
Step2とStep4は独立して準備可能であり、それぞれStep3完了前後に実施しても構わない。
各ステップのテストが不合格になった場合は、後続ステップのテストを実施せず、不合格箇所の修正を完了前に行うこと。

### TC-S1-001: bash-whitelist.jsへのparallel_verification追加の確認

**テスト対象機能:** FR-2（workflow-plugin内の変更をコミット）の事前確認

**実施タイミング:** Step1のコミット実行後

**検証コマンド:**
```
cd C:/ツール/Workflow/workflow-plugin
grep -n "parallel_verification" hooks/bash-whitelist.js
```

**期待される出力条件:** 出力に `'parallel_verification'` という文字列が含まれた行が表示されること。verificationPhases配列に追加されたことを示す1行が確認できる

**合否判定の具体例:** 出力例として `218: const verificationPhases = ['security_scan', 'performance_test', 'e2e_test', 'ci_verification', 'parallel_verification'];` が返れば合格、出力が空または対象文字列を含まない場合は不合格

**不合格時の対処:** bash-whitelist.jsをReadツールで確認し、verificationPhases配列の末尾に `'parallel_verification'` が追記されているかを再確認する。変更前の状態（4エントリのみ）のままであれば、EditツールでStep1の変更を再適用する

### TC-S1-003: Step1コミットの変更ファイル数と内容の確認

**テスト対象機能:** FR-2のコミット原子性確認（2ファイルが1コミットに含まれること）


**合否判定の具体例:** `2 files changed` が出力に含まれ、ファイル名として `hooks/bash-whitelist.js` と `.gitignore` の両方が表示されれば合格

**不合格時の対処:** 変更ファイルが1件以下の場合はいずれかのファイルのステージングが漏れている可能性があり、`git log --oneline -3` で直近コミット履歴を確認してStep1の手順を再実施する

### TC-S1-004: 既存フェーズ定義への変更なし確認（NFR-2対応）

**テスト対象機能:** NFR-2（既存ワークフロー機能への無影響）の検証。ワークフロープラグインの既存フェーズ制御が変更されていないことを確認し、追加的な変更（許可範囲拡張）のみが実施後の状態として記録されていることを検査する

**実施タイミング:** Step1のコミット実行後

**検証コマンド:**
```
cd C:/ツール/Workflow/workflow-plugin
git show HEAD -- hooks/bash-whitelist.js | grep "^+"
```

**期待される出力条件:** `+` で始まる追加行（差分の追加部分）が `parallel_verification` を含む1行のみであり、`getWhitelistForPhase` 関数の他の分岐やフェーズ定義には変更がないこと。既存のフェーズへの許可範囲拡張はこの修正作業の変更対象に含まれない

**合否判定の具体例:** `+` 行として `'parallel_verification'` の追加を示す1行のみが表示され、他のフェーズ名（`implementation`、`testing` 等）が追加・削除された行に含まれない場合に合格

**不合格時の対処:** 意図しない既存フェーズへの変更が含まれている場合は、直前のコミットへのrevertを検討し、bash-whitelist.jsを再編集してStep1の実装作業を再実施する。予期しない行目付近の変更が含まれていないかも合わせて確認する

### TC-NFR3-001: ポインタ更新の冪等性確認（NFR-3対応）

**テスト対象機能:** NFR-3（修正の冪等性）の検証

**実施タイミング:** Step3のコミット実行後

**検証手順の説明:**
この検証は「git add workflow-pluginを再実行してもステージングされる内容が変化しないこと」を確認する。複数回実行しても最終結果が変わらないことが冪等性の定義であり、Step3の手順が複数回実行された場合の影響を事前に評価する。

**検証コマンド:**
```
cd C:/ツール/Workflow
git add workflow-plugin
git status
```

**期待される出力条件:** `git add workflow-plugin` を複数回実行しても `git status` が `nothing to commit, working tree clean` を示し最終結果が変わらないこと。上書きによってポインタが変化しないことが固定形式のgitlink動作として保証されており、再度のgit addで新たな変更がステージングされないことが各操作前に想定される動作である

**合否判定の具体例:** git statusの出力にステージング済みの変更が表示されない場合に合格。新たに `workflow-plugin` が「modified」としてステージングされた場合は不合格であり、ポインタが最新を指していない可能性がある

**不合格時の対処:** ポインタが最新を指していない場合はTC-S3-002を再実施し、インデックスのハッシュとworkflow-pluginのHEADハッシュが一致しているかを確認する

## テスト実行順序

全テストケースの実行順序は以下のとおりである。依存関係により、TC-S3-*はTC-S1-*の全合格後に実施する。この段階的な実行順序は実装順序の制約（Step1 → Step3の依存関係）に対応している。

Step1完了後に実施するテスト: TC-S1-001、TC-S1-002、TC-S1-003、TC-S1-004（既存フェーズ定義への無影響を含む）
Step2完了後に実施するテスト: TC-S2-001、TC-S2-002（.gitmodulesの存在とインデント形式の確認）
Step3完了後に実施するテスト: TC-S3-001、TC-S3-002、TC-S3-003、TC-NFR3-001（ポインタ更新と冪等性確認）
Step4の検証時に実施するテスト: TC-S4-001、TC-S4-002、TC-S4-003（フックエントリとcheck_ocr.py参照の不在確認）
全ステップ完了後に実施する統合テスト: TC-INT-001、TC-INT-002、TC-INT-003、TC-INT-004

各テストケースが不合格となった場合は、対処手順に従い実装作業としての修正を行い、当該テストケースを再実施してから次のテストケースに進むこと。
Step1とStep3の間には問題間の依存関係（Step1のコミット完了 → Step3のgit add）が存在し、完了前に次のステップへ進むことは意図しない状態遷移を引き起こすリスクがある。この制約を無視すると旧ハッシュが親リポジトリに記録されるため、上記の段階的な順序を遵守することが本テスト計画の指定する制約である。
推奨する並行実施の可否: Step2の検証（TC-S2-001、TC-S2-002）はStep1の完了に依存しないため、Step1のテストと並行して準備確認のみ実施可能である。ただし、各操作前に正しいカレントブランチとディレクトリを確認することを全テストケースに共通した前提条件とする。
