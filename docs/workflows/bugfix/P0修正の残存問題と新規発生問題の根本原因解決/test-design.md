## サマリー

本ドキュメントは、CLAUDE.mdドキュメントの3件の記述誤りを修正するタスクに対するテスト設計を定義する。修正対象は文字列置換と行分割のみであり、コード変更を伴わない。テスト方針として、修正完了後のドキュメントをReadツールで読み込み、修正内容が正確に反映されており、かつ修正対象外の箇所が変更されていないことを確認する。また、definitions.tsの正規設定値との完全一致を検証することで、Orchestratorが参照した際に正確なガイダンスを提供できることを保証する。

- 修正対象: ワークフロー関連のドキュメント（workflow-plugin/CLAUDE.md、ルートCLAUDE.md）のカテゴリテーブルとコード例
- テスト対象: 3つの機能要件（FR-1、FR-2、FR-3）の実装完了後の状態
- 検証方針: 文字列の存在確認、合格条件の客観的判定、definitions.tsとの相互整合性確認

---

## テスト背景と概要

### 前回修正の背景と残存問題

前回修正（コミット91c3270）はフェーズ別subagent設定テーブルの行レベル修正に限定されたものであり、コード例内やBashカテゴリテーブルの他の行等は修正対象外であった。その結果、3件の記述誤りが残存問題として見落とされた状態となっていた。本仕様書（spec.md）の実装計画に明記されている通り、これらの残存問題はdefinitions.tsの正規設定値とCLAUDE.mdの記述の不整合であり、Orchestratorやsubagentがドキュメントを参照して誤った情報を取得するリスクを持つ。

本テスト設計は、その実装計画における各修正箇所を確実に検証し、完全整合性の回復が達成されたことを確認するために作成された。前回修正で解消されなかった残存問題を今回の修正で確実に取り除くことが主要な目的であり、修正完了後に本仕様書で定義された主要な決定事項が全て反映されていることを確かめる。

## テスト対象機能

### FR-1: workflow-plugin/CLAUDE.md の subagent_type 修正

修正対象ソースファイル: `C:\ツール\Workflow\workflow-plugin\CLAUDE.md`

修正内容: parallel_analysisの例示コードブロック内のplanning行から、存在しない値 `'Plan'` を `'general-purpose'` に置換する。この修正は、definitions.tsで全フェーズのsubagentTypeが `general-purpose` に統一されていることと整合性を確保する目的を持つ。TypeScript・JavaScript側のソースファイルであるdefinitions.tsでは `Plan` という値は定義されておらず、ドキュメント例内の記載が実装コードと矛盾していた。

修正位置: 330行目付近のコードブロック内（コードファイルへの変更は行わない）

重要度: 高レベル。Orchestratorが誤ったsubagent_typeでTask toolを起動した場合、planningフェーズの起動が失敗してワークフロー全体が停止するリスクがある。phase-edit-guardフックはBashコマンドのホワイトリスト制御（bash-whitelist）を担当するが、subagent_typeの誤設定は実行時にTask tool側で検出されるエラーである。

### FR-2: ルートCLAUDE.md の deploy 行修正

修正対象ソースファイル: `C:\ツール\Workflow\CLAUDE.md`

修正内容: フェーズ別Bashコマンド許可カテゴリテーブルのdeploy行から、過剰な権限付与を表す `readonly, implementation, deploy` を `readonly` のみに絞り込む。この修正は、definitions.tsで設定されるreadonly-onlyという実装値と記述の乖離を解決する。bash-whitelist設定でdeploy専用カテゴリは定義されておらず、架空のカテゴリを記載していた実態がある。

修正位置: 182行目（FR-3の分割後は183行目）のテーブル行（1行を1行に置換するシンプルな操作）

重要度: 中レベル。deployフェーズでの制限に対する誤解を防ぎ、ドキュメントの信頼性を確保する。フック（phase-edit-guard）がdeploy行の記述を参照してBashコマンドを制限する場合、過剰なカテゴリが記載されていると誤ったコマンドが許可されるリスクが生じる。

### FR-3: ルートCLAUDE.md の test_impl 行分割

修正対象ソースファイル: `C:\ツール\Workflow\CLAUDE.md`

修正内容: フェーズ別Bashコマンド許可カテゴリテーブルの test_impl、implementation、refactoring の3フェーズをまとめた1行を、test_impl用と implementation/refactoring用の2行に分割する。test_impl行の許可カテゴリは `readonly, testing` のみに設定し、implementation/refactoringの行は `readonly, testing, implementation` として分離する。テストコードの先行作成（TDD Redフェーズ）に特化したスコープを明確にするための修正である。

修正位置: 176行目（1行を2行に分割。フックによるブロック対象を正確に限定するための変更）

重要度: 中レベル。TDD Redフェーズであるtest_implに対して implementation カテゴリが不適切に付与されている問題を解決し、テストコードの作成段階で不要なコマンド（npm install、git add等）が使用されることを防止する。

## テスト方針の詳細

### 修正方針とテスト観点の対応

本テスト設計は、spec.mdで定義された修正方針（基本原則）に基づいてテスト観点を設定している。具体的には以下の対応関係を持つ。

修正の基本原則として「各修正は指定された修正箇所のみに最小限の変更を加え、周辺の記述・書式・空白・インデントを維持する」と定められており、TC-5（修正対象外の行の不変性確認）がこの原則の遵守を検証する。また、「変更対象外のセクションには一切変更を加えない」という方針は、TC-5の合格条件においてrequirements行・code_review行等の不変性確認として具体化されている。

フック（phase-edit-guard）によるBashコマンド制限は、ドキュメント記述が正規設定値と完全一致することを前提条件としており、TC-6〜TC-8のクロスリファレンステストがこの前提条件の成立を検証する。同一のカテゴリ名がフック設定とドキュメントで使用されることで、動作の一貫性が確保される。

### 修正対象のスコープとテストカバレッジ

修正の対象範囲は2つのファイルに限定されており、TypeScript・JavaScriptなどのコードファイルは変更対象外である。テストカバレッジは以下の通りとなる。

- FR-1（workflow-plugin/CLAUDE.md の1箇所）: TC-1, TC-6で検証
- FR-2（CLAUDE.md の1箇所）: TC-2, TC-7で検証
- FR-3（CLAUDE.md の1箇所を2行に分割）: TC-3, TC-4, TC-8で検証
- 副次的検証（修正対象外の不変性）: TC-5で検証

全8テストケースが合格することで、修正の完全整合性の回復と変更の最小性の両立を確認できる。

## テスト実行のステップと実装手順

### テスト実行準備（前提条件の確認）

テスト実行に先立ち、以下の前提条件を確認する。実装フェーズで各ステップが正しく実行されていることがテスト成功の前提となる。

- 実装手順のステップ1（FR-3: test_impl行の分割）が完了していること
- 実装手順のステップ2（FR-2: deploy行のカテゴリ修正）が完了していること
- 実装手順のステップ3（FR-1: subagent_type値の修正）が完了していること
- 全3件の修正が適用された後にテスト実行を開始すること

各ステップは独立しているため、並列に実施することも可能である。ただし、FR-3の行分割により行番号が変動するため、実装手順では最優先でFR-3を実施する論理的な順序が定められている。

### 具体的なテスト実行方法

各テストケースは以下の具体的な方法で実行する。

**Readツールを使用したファイルパス指定の注意**: Readツールで対象ファイルを読み込む際は、絶対パス（例: `C:\ツール\Workflow\CLAUDE.md`）を正確に指定すること。行番号を指定してオフセット読み込みを行う場合は、行番号変動（FR-3の分割による+1ずれ）を考慮すること。

**文字列検索による判定**: 合格条件の多くは「文字列の存在・不存在」という客観的基準で定義されており、自動化した文字列検索（grep等）または手動でのReadツール目視確認のいずれでも判定可能である。判定可能な形式で条件が定義されているため、ステップごとに合否を明確に記録できる。

**段階的な確認**: 修正ごとに直接原因となった不整合が解消されているかを個別に検証してから、クロスリファレンステスト（TC-6〜TC-8）で定義tsとの整合性を最終確認する順序で実行する。

テスト実行は以下の順序で進める。TDD Redフェーズのため、修正実装後にテストを実行する。

1. **修正実装**（implementationフェーズで実施）
   - FR-3 修正（test_impl行の分割・最優先ステップ）
   - FR-2 修正（deploy行のカテゴリ修正・第2ステップ）
   - FR-1 修正（subagent_type値の修正・第3ステップ）

2. **単体テスト実行**（TC-1〜TC-5）
   - Readツールで修正内容を確認する
   - 各合格条件が満たされているかを個別にチェックする

3. **統合テスト実行**（TC-6〜TC-8）
   - definitions.tsとの相互参照による整合性確認
   - TypeScript/JavaScriptソースコードと記述の一致を最終検証する

4. **最終検証**
   - 全テストケースの合格確認（修正件数3件・修正番号FR-1〜FR-3の全完了）
   - ドキュメント全体を読み直し、意図した範囲のみの変更であることを確認する

## テスト実施の重要ポイント

### Editツールの文字列マッチング特性とテスト観点

Editツールは old_string 全体をexact matchで検索するため、修正対象行のファイルパスと行全体を正確に指定する必要がある。old_stringに行全体を含めることは誤適用防止の観点から重要であり、テスト時にも同一の文字列が修正後に消えていることを確認することで、Editツールが正しく動作したことを間接的に検証できる。

行分割時の改行文字（`\n`）の処理が正しく行われたかどうかは、TC-3・TC-4の確認（test_impl行とimplementation/refactoring行が別行として存在すること）によって検証される。

### Readツールによる確認の客観性

修正内容確認はReadツールで対象行を読み込み、視覚的に検証する。各テストケースの合格条件は全て「文字列の存在確認・不存在確認」という客観的基準に基づいており、解釈の余地がない形式で定義されている。これにより、合否を自動化した文字列検索でも判定可能であり、テストの再現性と客観性が確保される。

### 非機能要件のテスト観点

本タスクには4つの非機能要件が定義されており、それぞれに対応するテスト観点を持つ。

**NFR-1（変更の最小性）**: TC-5（修正対象外の行の不変性確認）によって検証する。変更対象外のセクションが維持されていることを網羅的に確認することで、最小性の原則が守られていることを保証する。

**NFR-2（definitions.tsとの整合性）**: TC-6〜TC-8（クロスリファレンステスト）によって検証する。確認対象のフィールドはsubagentTypeとallowedBashCategoriesであり、修正された3箇所全ての設定値を網羅的に比較確認する。

**NFR-3（両CLAUDE.mdの相互整合性）**: TC-1（FR-1修正後の確認）とTC-6（definitions.tsとのクロスリファレンス）を組み合わせることで検証する。workflow-plugin/CLAUDE.mdのplanningフェーズと、ルートCLAUDE.mdのthreat_modeling行（既に修正済み）が同一の `general-purpose` 値を持つことを確認する。

**NFR-4（検証の客観性）**: 全テストケースの合格条件が文字列の存在・不存在で判定可能な形式で定義されており、grep等の文字列検索で合否を判定できる。判定可能な基準設計が本テスト設計の非機能要件への対応となっている。

**NFR-5（コードおよび実装ファイルの不変性）**: 本テスト設計でdefinitions.ts、bash-whitelist.js、その他のTypeScript・JavaScriptファイルは参照のみの確認対象であり、変更対象ではないことを明示する。実装フェーズで使用するツールはEditツール（ドキュメント修正用）とReadツール（確認用）のみであり、コードファイルへのWriteツール使用は禁止される。

## テスト完了判定基準

以下の全ての条件が満たされた場合、テスト設計フェーズは完了と判定される。

1. TC-1〜TC-5 の単体テスト全てが PASS であること（FR-1〜FR-3の各修正箇所が正確に修正されていること）
2. TC-6〜TC-8 の統合テスト全てが PASS であること（definitions.tsとの整合性が確認されていること）
3. 各テストケースの合格条件が漏れなく検証されており、全修正完了後の状態が確認されていること
4. 修正前後のドキュメント変更が意図した変更対象外の箇所に影響を与えていないことの確認が完了していること

実装フェーズにおいて、各テストケースの合格条件を逐一確認し、testingフェーズで最終的なドキュメント整合性検証を実施する手順となる。本テスト設計によって、前回修正で残存した問題が今回の修正で確実に解消されたことを客観的に証明できる。
