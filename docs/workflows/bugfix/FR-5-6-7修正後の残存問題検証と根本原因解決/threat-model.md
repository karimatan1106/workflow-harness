# FR-5-6-7修正後の残存問題検証と根本原因解決 - 脅威モデル

## サマリー

本ドキュメントは `verify-sync.ts` スクリプトの実装に対する脅威モデルを定義する。
スクリプトは `definitions.ts`、ルート `CLAUDE.md`、`workflow-plugin/CLAUDE.md` の3ファイル間の整合性を自動検証する。
検証対象の主要な脅威は4つ: importパス変更による破綻、Markdownパーサーの脆弱性、偽陽性（誤って全一致と報告）、並列フェーズのネスト構造展開失敗。
各脅威に対して深刻度・発生確率・緩和策を評価し、実装時に組み込むべきセキュリティ要件を導出した。
このスクリプトはCI/CD統合を前提とするため、誤検知（偽陽性・偽陰性）のいずれも深刻な問題となる。

---

## 脅威シナリオ

### 脅威1: importパス変更によるスクリプト破綻

**概要:**
`verify-sync.ts` は `definitions.ts` を ESM import で直接読み込む実装方針をとる。
`definitions.ts` のファイルパスやディレクトリ構造が将来変更された場合、import文がコンパイルエラーまたは実行時エラーとなり、スクリプトが機能しなくなる。
例えば、`workflow-plugin/mcp-server/src/` 直下にあった `definitions.ts` が `phases/` サブディレクトリに移動されたとき、スクリプトは即座に動作停止する。
また、TypeScript の `tsconfig.json` で `paths` エイリアスが変更された場合も同様の破綻が発生しうる。
このリスクは静的importであるため、実行時まで気づかない「サイレントブレーキング」になりにくいが、
モジュールリゾルバーの設定変更（ESM/CJS切り替え等）は実行時エラーとなる場合がある。

**攻撃ベクタ:**
意図的な攻撃ではなく、リファクタリング作業による副作用として発生する。

---

### 脅威2: Markdownテーブルパーサーの形式変更対応不足

**概要:**
CLAUDE.md のテーブル書式が変更されると、正規表現ベースのパーサーが正しくフィールドを抽出できなくなる。
具体的なリスクケースを以下に示す:
- ヘッダー行のテキストが変更される（例: 「フェーズ別subagent設定」→「subagent設定一覧」）と、対象セクションを検出できなくなる
- セル内の値に `|` 文字が含まれる場合、列分割が誤作動する
- インデントやスペース数の変更により、セパレータ行の識別正規表現がマッチしなくなる
- 6列テーブル（workflow-plugin側）の列数が変更された場合、列インデックスのハードコードが誤読を引き起こす
- マルチライン型のMarkdownテーブル（GFM拡張）への変更は、行単位のパーサーで完全に対応不能となる

---

### 脅威3: 偽陽性リスク（誤って「全一致」を報告する問題）

**概要:**
これが最も深刻な脅威である。パーサーが空配列や空マップを返した場合でも、スクリプトが「全フェーズ一致」と誤報する可能性がある。
具体的なシナリオを以下に示す:
1. CLAUDE.md のセクションヘッダー検出に失敗し、パーサーが空のマップ `{}` を返す
2. definitions.ts 側も何らかの理由で空になる（型定義変更等）
3. 両方空のため「差分なし」と判定され、終了コード0で終了する
4. CI/CDは「成功」と判断するが、実際には検証が何も行われていない

この偽陽性リスクは「スクリプト自身の整合性チェック不足」に起因する。
比較前に抽出件数がゼロでないことを確認する防御的チェックが実装されていない場合に発生する。

---

### 脅威4: 並列フェーズのネスト構造展開失敗

**概要:**
`PHASE_GUIDES` の一部エントリは並列フェーズのサブフェーズ（`subPhases` フィールド）をネストして持つ。
例えば `parallel_analysis` は `threat_modeling` と `planning` をサブフェーズとして持つ。
単純に `Object.entries(PHASE_GUIDES)` でイテレートしても、サブフェーズが展開されず比較対象から漏れる可能性がある。
CLAUDE.md のテーブルにはサブフェーズが独立した行として記載されているため、
definitions.ts 側でサブフェーズを展開しないと「CLAUDE.mdにのみ存在するエントリ」として誤検出される。
結果として、`threat_modeling` / `planning` / `state_machine` / `flowchart` / `ui_design` 等が比較から外れ、検証精度が低下する。

## リスク評価

| 脅威ID | 脅威名 | 深刻度 | 発生確率 | 総合リスク |
|--------|--------|--------|----------|------------|
| T-1 | importパス変更による破綻 | 中（スクリプト停止のみ、誤検知は起きない） | 低（短期）、中（長期） | 中 |
| T-2 | Markdownパーサーの形式変更対応不足 | 高（偽陰性・偽陽性の両方が起きうる） | 中（CLAUDE.mdは頻繁に編集される） | 高 |
| T-3 | 偽陽性（空配列での全一致誤報） | 致命的（CI/CDで問題が見逃される） | 中（パーサー実装の初期バグとして発生しやすい） | 致命的 |
| T-4 | 並列フェーズのネスト展開失敗 | 高（サブフェーズ全体が未検証になる） | 高（ネスト展開を明示的に実装しなければ自動的に発生） | 致命的 |

**T-3（偽陽性）とT-4（ネスト展開失敗）は最優先で対処すべき致命的リスクである。**
これらは「スクリプトが動いているように見えるが実際には何も検証していない」状態を引き起こすため、
実装フェーズで防御的チェックを組み込むことが必須となる。

## セキュリティ要件

### SR-1: 抽出件数の最小値チェック（T-3対策）

スクリプトは比較処理を開始する前に、definitions.ts から抽出されたフェーズ数が `MIN_PHASE_COUNT`（設定値: 20）以上であることを確認しなければならない。
同様に、ルートCLAUDE.mdから抽出されたエントリ数がゼロでないことを確認する。
いずれかがゼロまたは最小値未満の場合は、終了コード2（実行エラー）で終了し、パーサーエラーとして標準エラー出力に報告する。
この要件により、パーサーが空配列を返した状態で「全一致」と誤報する偽陽性を排除できる。
実装例: `if (definitionsPhases.size < MIN_PHASE_COUNT) { process.stderr.write("ERROR: 抽出フェーズ数が少なすぎます: " + definitionsPhases.size); process.exit(2); }`

### SR-2: 並列フェーズのサブフェーズ再帰展開（T-4対策）

definitions.ts から PHASE_GUIDES を読み込む際、各エントリの `subPhases` フィールドを再帰的に展開して平坦化したフェーズリストを構築しなければならない。
展開後のリストにはサブフェーズ名（`threat_modeling`, `planning`, `state_machine`, `flowchart`, `ui_design`, `build_check`, `code_review`, `manual_test`, `security_scan`, `performance_test`, `e2e_test`）が個別エントリとして含まれること。
展開処理は `subPhases` フィールドが存在しない場合もエラーとならないよう、オプショナルチェーン（`?.`）で安全にアクセスする。
展開後のフェーズ数が期待値（25件）と一致するかを検証ログに出力することで、展開漏れを早期発見できる。

### SR-3: パーサーエラー時の明示的な失敗（T-2対策）

CLAUDE.md のテーブル解析時に対象セクションが見つからない場合、パーサーは空配列を返すのではなく、例外をスローして終了コード2で終了しなければならない。
「セクションが見つからない」状態と「セクションは存在するがエントリが0件」を区別して報告することで、テーブル形式変更を早期に検知できる。
セクション識別文字列（例: 「フェーズ別subagent設定」）は定数として定義し、変更追跡を容易にする。
テーブルセパレータ行（`|---|`パターン）の検出に失敗した場合も同様にエラーとして扱う。

### SR-4: importパス変数の一元管理（T-1対策）

`definitions.ts` へのimportパスは、スクリプト内の1箇所にのみ記述し、コメントで「このパスを変更する場合はREADMEのセットアップ手順も更新せよ」と明記する。
コンパイル時エラーではなく実行時エラーとなるESM動的importの使用は避け、静的importを採用することで、パス変更を早期（コンパイル段階）に検知できる構造とする。
`tsconfig.json` の `baseUrl` や `paths` 設定に依存したエイリアスimportは使用せず、相対パスで記述することで設定変更の影響を最小化する。

## 残存リスクと許容範囲

### 許容する残存リスク

CLAUDE.md が全く異なる構造に書き換えられた場合（例: テーブルをコードブロックに変更）、現行のパーサー設計では対応できない。
このような抜本的な形式変更は発生頻度が極めて低く、発生した際は手動での対応（パーサーの更新）が現実的であるため、残存リスクとして許容する。
リスクが顕在化した場合の対処時間目標は1営業日以内とする。

### 監視指標

- CI/CDでのスクリプト実行時に終了コード2（実行エラー）が発生した場合は最優先対応とする
- 検証対象フェーズ数が24件以下になった場合はアラートを発する（SR-2の展開漏れ検知）
- 月次でスクリプトの出力行数（期待値: 25件のOK/NGライン + 1件のサマリー）を確認する
