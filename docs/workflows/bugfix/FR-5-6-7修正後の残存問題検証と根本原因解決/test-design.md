# テスト設計書: verify-sync.ts（FR-5-6-7修正後の残存問題検証と根本原因解決）

## サマリー

本仕様書は `verify-sync.ts` の設計内容に基づくテスト設計を記述する。
このスクリプトは `definitions.ts`（ソースオブトゥルース）、ルート `CLAUDE.md`、および `workflow-plugin/CLAUDE.md` 配下の3ファイル間のフェーズ設定整合性を自動検証するシステムである。

**目的:** FR-5-6-7修正後の残存問題検証として実施した調査（researchフェーズ）で確認された「手動更新に依存した構造」という根本原因を解消するため、機械的かつ恒久的な解消策として `verify-sync.ts` を検証する。

**主要な決定事項:**
- 実装言語は TypeScript であり、ts-node 互換の tsx ライブラリを用いた実行方式を採用する
- PhaseGuide・PhaseEntry の型定義と型安全性の検証を重点テスト項目とする
- 配置場所は `workflow-plugin/mcp-server/src/verify-sync.ts`（新規作成ファイル1件のみ）
- テストフレームワークはプロジェクト既存の vitest を使用し、fs モジュールを vi.mock でモックする
- 外部依存パッケージの追加インストールは禁止され、プロジェクトルート既存のパッケージのみ使用可能

**次フェーズで必要な情報:**
- テストファイルの配置先: `workflow-plugin/mcp-server/src/verify-sync.test.ts`
- 実行時間（NFR-3）として総実行時間が1秒以内で完了することを確認すること
- 終了コード（0: 完全同期状態、1: 差分あり、2: パースエラー）がCI/CDパイプライン統合に使用される

## テスト方針

### 課題背景と根本原因に基づく検証アプローチ

FR-5-6-7修正後の残存問題検証と根本原因解決タスクの課題背景として、`definitions.ts` の `PHASE_GUIDES` オブジェクト（ソースオブトゥルース）と CLAUDE.md の記述が手動更新で同期されており、フェーズ追加や設定変更の際に同期漏れが発生しうる構造があった。
現時点では3ファイルが同期済みであることが確認されているが、将来的な同期漏れを防ぐ恒久的な解消策として本スクリプトを実装することで、機械的な自動検証を実現する。
テスト設計では、本仕様書で詳述される関数設計（`extractFromDefinitions`、`parseRootCLAUDEMdSubagentTable`、`parseRootCLAUDEMdBashTable`、`parsePluginCLAUDEMdSubagentTable`、`compareAndReport`）の全パスを検証対象とする。
TypeScript のコンパイル、ts-node 互換の tsx ライブラリによる実行方式、ESM インポートの互換性確保が核心的な技術要件であることを踏まえ、実行環境依存のテストケースも設ける。
フィードバックループの短縮という目的を達成するため、3ファイル間の乖離が即座に検出されることをテストで保証する。

### 対象ファイルと実行環境

テスト対象はソースコード `workflow-plugin/mcp-server/src/verify-sync.ts` であり、プロジェクト既存の tsconfig.json の設定を継承してコンパイルする。
実行方式は `npx tsx workflow-plugin/mcp-server/src/verify-sync.ts` であり、事前ビルドは不要で、プロジェクトルートから直接実行する。
Node.js の ESM モジュールとして動作するため、`package.json` の `"type": "module"` 宣言に対応した環境で実行する。
ts-node でも実行可能だが、ESM インポートの解決には tsx の方が互換性が高い点をテスト実施時に確認する。
外部依存の追加インストールは禁止であり、使用可能なパッケージは Node.js 組み込みモジュール（`fs`、`path`、`url`）および既存プロジェクトのパッケージに限定する。

### 型安全性の検証方針

TypeScript の型定義に基づく検証として、`PhaseGuide` 型・`PhaseEntry` 型・`TableEntry` 型・`Object` 型・`Record` 型・`Partial` 型の適切な使用を確認する。
`PHASE_GUIDES` は `Partial<Record<string, PhaseGuide>>` 型として定義されているため、`Object.entries()` でイテレート後に型キャストなしでフィールドアクセスが可能であることを検証する。
`subPhases` フィールドはオプショナルフィールドとして `Record<string, PhaseGuide> | undefined` 型であり、オプショナルチェーン（`?.`）による存在確認が実装されていることを確認する。
`allowedBashCategories` は文字列配列として型定義されており、抽出処理で型を維持したまま比較できることを検証する。
`any` 型キャストを使用せず、コンパイル時エラーが実行時エラーを防ぐ構造を維持しているかを `tsc --noEmit` で確認する。

### モック戦略とカバレッジ目標

`fs` モジュールの `readFileSync` を vi.mock で差し替えてテスト用の Markdown テキストを返すよう制御する。
`definitions.ts` の ESM import は `vi.mock` を使用してモジュール全体を差し替え、`PHASE_GUIDES` オブジェクトに任意のフェーズデータを注入できるようにする。
実ファイルへのアクセスは一切発生しないため、CI/CD パイプライン環境でも安定して実行できる。
spec.md に記載された AC-1、AC-2、AC-3、NFR-3、NFR-4 の受け入れ基準に対応するテストケースを明示的に設け、個別解析と統合的な動作確認の両方を網羅する。
テスト実行カテゴリは `readonly` および `testing` であり、`implementation` カテゴリのコマンドは使用しない。

### 非機能要件（NFR）の検証方針

実行時間（NFR-3）として、`fs.readFileSync` による同期的な全文読み込みで非同期待機のオーバーヘッドが発生せず、総実行時間が1秒以内で完了することを確認する。
ファイルサイズが数百 KB 程度のテキストファイルに対して正規表現でテーブル解析を行い、処理が数ミリ秒以内に完了することを `time` コマンドで計測する。
型安全性（NFR-4）として、TypeScript コンパイル時の型チェックが通ることを `npx tsc --noEmit` で検証する。

## テストケース

### グループ1: extractFromDefinitions 関数設計の検証

#### TC-1-1: 正常系 - subPhases を持たない単純な PhaseGuide のフラット化

`PHASE_GUIDES` に `subPhases` プロパティを持たないエントリが3件含まれる場合、`extractFromDefinitions` は3件の PhaseEntry 配列を返す。
各要素の `phaseName`（識別子文字列）、`subagentType`（種別文字列）、`model`（モデル名）、`allowedBashCategories`（文字列配列）が入力の PhaseGuide フィールドアクセス結果と一致することを確認する。
TypeScript の型定義に基づく型安全なフィールドアクセスが動作していることも検証する。

期待結果: 返却配列の長さが3であり、各要素の4フィールドが PhaseGuide の対応フィールドと一致する。

#### TC-1-2: 正常系 - subPhases 再帰展開による展開対象の確認（AC-1 対応）

`parallel_analysis` エントリが `subPhases` として `threat_modeling` と `planning` を持つ場合、`extractFromDefinitions` はトップレベルの `parallel_analysis` を展開対象外とし、2件のサブフェーズ PhaseEntry を含むフラット配列を返す。
`subPhases` フィールドへのオプショナルチェーン（`?.`）による存在確認が機能していることを確認する。
他に3件の単純エントリがある場合、合計5件のフラット配列が返却される。

期待結果: 返却配列に `parallel_analysis` が含まれず、`threat_modeling` と `planning` が含まれる。

#### TC-1-3: 正常系 - 全25フェーズの展開後配列（AC-1 の単体テスト）

全フェーズ相当の `PHASE_GUIDES` モックを使用した場合、返却配列の長さが25件になることを確認する。
主要フェーズ15件とサブフェーズ10件の合計として、展開後の総エントリ数が25件と一致することを検証する。
`Partial<Record<string, PhaseGuide>>` 型の `PHASE_GUIDES` を `Object.entries()` でイテレートする実装が正しく動作することを確認する。

期待結果: 返却配列の長さが25であり、並列フェーズ本体（`parallel_analysis` 等）が含まれない。

#### TC-1-4: 境界値 - PHASE_GUIDES が空 Object の場合

`PHASE_GUIDES` が空の `Partial<Record<string, PhaseGuide>>` オブジェクトの場合、`extractFromDefinitions` は空の PhaseEntry 配列を返す。
例外が発生せず安全に空配列が返ることを確認し、後続の比較処理が「0フェーズ中0件の不一致」というサマリーを出力できることを検証する。

期待結果: 返却値が長さ0の配列であり、例外が発生しない。

#### TC-1-5: 境界値 - オプショナルフィールドが未定義の PhaseGuide

`allowedBashCategories` フィールドが未定義（オプショナルフィールドとして省略）された PhaseGuide を入力とした場合、オプショナルチェーン（`?.`）による存在確認で実行時エラーが発生しないことを確認する。
フィールドアクセス時のプロパティキャストが `any` 型を使用せず型安全に行われていることを検証する。

期待結果: 例外が発生せず、未定義フィールドが空配列として扱われるか、`undefined` を示す安全なデフォルト値が返る。

### グループ6: エラーハンドリングとプロセス終了コードの検証

#### TC-6-1: ファイル不存在時の終了コード2とデバッグ情報出力

`fs.readFileSync` がファイル不存在エラー（ENOENT）を投げるよう vi.mock で設定した場合、スクリプトが終了コード2で終了し、標準エラー出力にスタックトレースが含まれることを確認する。
CI/CD パイプラインが終了コード2を検出してビルド失敗と判定できることを検証する。

期待結果: `process.exit(2)` が呼び出され、標準エラーにスタックトレースが出力される。

#### TC-6-2: definitions.ts の ESM import 失敗時の終了コード2

`definitions.ts` の ESM import が失敗した場合（ファイル不存在や ESM 解決エラー）、終了コード2で終了することを確認する。
実行自体の失敗（終了コード2）と検証失敗（終了コード1）が区別されていることを検証する。

期待結果: 終了コード2相当の動作が確認される。

#### TC-6-3: テーブルセクション未検出時のパースエラーと終了コード2

「`## フェーズ別subagent設定`」セクションが存在しない Markdown 文字列を全文読み込み入力とした場合、パースエラーが発生して終了コード2が返ることを確認する。
パースエラー時のスタックトレースが標準エラー出力に含まれることも検証する。

期待結果: 例外がスローされ、最終的に終了コード2相当の動作となる。

#### TC-6-4: フェーズ数不足時のバリデーションエラーと終了コード2

フラット化後のフェーズ数が25未満の場合、フェーズ数チェックロジックにより終了コード2が返ることを確認する。
state-machine.mmd の `CountError --> ExitCode2` 遷移と flowchart.mmd の「フェーズ数チェック」ノードに対応するテストケースである。

期待結果: フェーズ数不足のエラーメッセージとともに終了コード2が返る。

## 制約と禁止事項

本テスト設計において遵守する制約を以下に示す。
ソースコードへの変更は `verify-sync.ts` の新規作成1件のみであり、`definitions.ts`・ルート `CLAUDE.md`・`workflow-plugin/CLAUDE.md` への変更は一切行わない。
外部ライブラリの追加インストールは禁止であり、使用可能なパッケージはプロジェクト既存の依存に限定する。
テスト実行フェーズで使用可能な Bash カテゴリは `readonly` および `testing` であり、`implementation` カテゴリのコマンドは使用しない。
テスト用一時ファイルはルートディレクトリに配置せず、`.tmp/` ディレクトリに配置する。
統合テスト（TC-7-2）でルート CLAUDE.md を一時変更した場合は、必ず検証完了後に元に戻すこと。
`package.json` への `scripts` エントリ追加は本タスクのスコープ外であり、CI/CD パイプラインへの組み込みは後続タスクで実施する。
