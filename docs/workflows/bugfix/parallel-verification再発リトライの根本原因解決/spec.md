## サマリー

- 目的: parallel_verificationフェーズで16回以上のリトライが発生した根本原因5件に対応する、プロジェクトルートCLAUDE.mdおよびworkflow-plugin/CLAUDE.mdへの具体的な文言変更仕様を定義する。
- 主要な決定事項: FR-1ではプロンプトテンプレート内の禁止語直接列挙を数値参照に置き換える。FR-3ではプレーンラベル行（50文字以内コロン終端）が実質行にカウントされないことを明示する。
- FR-2・FR-4・FR-5は補完的な説明追加として実施し、FR-1・FR-3と並行して対応する。
- 変更対象ファイル: プロジェクトルートCLAUDE.md（主要）とworkflow-plugin/CLAUDE.md（補完）の2ファイルが中心。
- 次フェーズで必要な情報: 各FRの変更前後の文言、変更対象ファイルの行番号、受け入れ検証方法。

## 概要

本仕様書は、parallel_verificationフェーズで16回以上のバリデーション失敗リトライが発生した根本原因を分析し、再発防止のためのCLAUDE.md改修内容を定義するものである。
背景と意図として、今回の問題はバリデーターのコード自体に欠陥があったのではなく、subagentへの指示内容が不十分であったことに起因する。
リトライの原因は以下の5種類に分類された。
根本原因Aはsubagentがプロンプト内の禁止語リストを成果物に転記してしまうという問題である。
根本原因Bはコードフェンス内でも角括弧プレースホルダーが検出される仕様がsubagentに伝わっていない問題であり、散文形式での記述を徹底する指針が必要である。
根本原因Cはプレーンラベル行が実質行としてカウントされない構造行除外仕様の説明不足である。
根本原因Dは禁止語を含む複合語に対する回避表現例が不足している問題であり、言い換え事例の充実が求められる。
根本原因Eはcommitフェーズでのファイル削除操作に関するガイダンス欠如である。
修正の主対象ファイルは `CLAUDE.md` および `workflow-plugin/CLAUDE.md` であり、バリデーターのコード自体（`workflow-plugin/mcp-server/src/validation/artifact-validator.ts`）は変更対象外とする。
本仕様に基づいて実装フェーズで既存のファイルに追記・修正を行う。

## 機能要件と非機能要件の構成

本仕様は機能要件（FR-1〜FR-5）と非機能要件（NFR-1〜NFR-4）で構成される。
機能要件は具体的な変更内容を定義し、非機能要件は変更後に維持すべき品質基準を定義する。
各機能要件の影響範囲・変更種別・対応する根本原因を明確化することで、実装フェーズでの作業漏れを防止する。
NFR-1はバリデーション成功率の改善目標を数値で規定する。
NFR-2は後方互換性の維持を要求し、NFR-3は両ファイルの整合性を確保する。
NFR-4は可読性と保守性の観点から追加行数の上限を設定している。

## 実装計画

FR-1からFR-5を以下の手順で実施する。

FR-1を最初に実施する。subagentプロンプトテンプレート内の禁止語の直接列挙を削除し、英語4語・日本語8語という数値参照形式に置き換える。変更対象は行289〜298（禁止パターンセクション）および行358（リトライテンプレート内）である。この変更によりバリデーションエラーの発生件数を大幅に削減できると推定される。

FR-3をFR-1と並行して実施する。実質行の説明セクションにプレーンラベル行除外の具体例を追加する。行249の直後への挿入が主な変更内容となり、「実質行5行以上を確保するための指針」を明示する。コロン終端50文字以内の行が構造行として除外される仕様を、回避表現例とともに記述する。

FR-2・FR-4・FR-5をその後に実施する。FR-2では角括弧のコードフェンス内検出に関する注記を追加し、散文形式での記述を促す指示を追記する。FR-4では禁止語複合語の言い換え表現を充実させ、3種類以上の回避表現例を項目として列挙する。FR-5ではcommitフェーズのBashカテゴリ表を「readonly, git」に修正し、ファイル削除の代替手順を案内するガイダンスを追加する。

各FRの修正後、Grepを用いて変更内容の反映を確認する。修正内容の記録として、各変更箇所の行番号とdiff相当の情報を成果物に残す方針とする。

## FR-1: プロンプトテンプレート内の禁止語直接列挙の除去

### 変更対象と現状

プロジェクトルートCLAUDE.mdの行289〜297が変更対象である。この箇所には「### 禁止パターン（完全リスト）」というセクションヘッダーの下に、英語4語と日本語8語の禁止パターンが箇条書き形式で直接列挙されている。subagentがこのプロンプトを受け取ると、禁止語リストを成果物に転記してしまうという誤動作が発生した。前回のe2e_testフェーズでは12語全てが成果物内に列挙されていたことが確認されている。同様の問題が行355〜361のリトライテンプレートの「## 成果物品質要件（再確認）」セクションにも存在する（行358に禁止語が列挙されている）。workflow-plugin/CLAUDE.mdの対応箇所（行313〜321）も同一の問題を抱えており、同期修正が必要である。

### 変更内容（プロジェクトルートCLAUDE.md 行289〜298）

変更前の「### 禁止パターン（完全リスト。成果物内に1つでも含まれるとエラー）」セクション全体を、以下の内容で置き換える。

変更後の文言（禁止語を含まない数値参照形式）:
```
    ### 禁止パターン（英語4語・日本語8語が部分一致で検出される）
    - 禁止語リストはCLAUDE.md「成果物品質要件」セクション本体を参照すること（テンプレート内には列挙しない）
    - 角括弧プレースホルダーは別途禁止（後述）
    - 禁止語を含む複合語も部分一致で検出される。言い換えのガイダンスは成果物品質要件セクション本体を参照
    - Mermaid図のstateDiagram-v2では開始・終了に名前付き状態（Start, End）を使うこと
    これらの要件を満たさない成果物はバリデーションで拒否される。
```

### 変更内容（プロジェクトルートCLAUDE.md 行358 リトライテンプレート内）

変更前の記述は英語4語と日本語8語の禁止語を直接列挙する形式であった。

変更後: 「- 禁止語: 英語4語・日本語8語が部分一致で検出される。具体的な語はCLAUDE.mdの成果物品質要件セクション本体を参照すること」

### CLAUDE.md本体への禁止語記述の維持

「禁止パターン（完全リスト）」のセクションは**プロンプトテンプレート外**のCLAUDE.md本文（行289より前の本体記述部分）には維持する。プロンプトテンプレートは本体の説明を「参照すること」と記述するだけにとどめる。この方針により、テンプレート内では禁止語を作成・転記することなく、本体のみで禁止語の一覧を管理できる構成となる。

### 受け入れ検証方法

修正後のCLAUDE.mdのテンプレート部分（行189〜299の範囲）に英語4語・日本語8語の禁止語が1語も存在しないことをGrepで確認する。リトライテンプレート（行320〜371）にも禁止語が存在しないことを確認する。workflow-plugin/CLAUDE.mdについても同様に確認する。

## FR-2: 角括弧制約の詳細説明追加

### 変更対象と現状

プロジェクトルートCLAUDE.mdの行292〜295が変更対象である。この箇所には角括弧プレースホルダーの禁止に関する説明として「コードフェンスの内外を問わず角括弧の使用は禁止される」「コードフェンス内に正規表現を記述する場合は文字クラスの代わりに散文で説明すること」という記述が存在している。workflow-plugin/CLAUDE.mdの行316〜319にも同内容が存在しており、両ファイルの記述はほぼ同等である。ただし「シングルバックティック内のインラインコードは除去されるため安全」という補足説明が一部に存在しており、これがsubagentの混乱を招く可能性がある。

### 変更内容（プロジェクトルートCLAUDE.md 行292〜295）

現在の記述に以下の明示的な注意書きを追加する。

追加する行（行295の直後に挿入）:
```
    - 重要: コードフェンスの中であっても角括弧は成果物バリデーターに検出される。コード例を示す際は角括弧を使わない表記を選ぶこと
    - インラインコード（バックティック1個で囲んだ記法）は検証前にテキストから除去されるため、変数名などは散文ではなくインラインコードとして記述すると安全
```

散文形式での説明を推奨する方針は、コードフェンス内の角括弧問題を回避するための基本的な指針として位置づける。

### 受け入れ検証方法

## FR-3: 実質行カウントの詳細説明追加

### 変更対象と現状

プロジェクトルートCLAUDE.mdの行271〜274が変更対象である。行264には「ルール8: 50文字以内の内容がコロンで終わるラベル行（リスト記号は省略可能）」という重複行検出の除外ルールが記述されている。しかし行242〜253の「実質行にカウントされない行の例」セクションには、このルール8に相当するプレーンラベル行の除外について説明が不足している。subagentは「実質行にカウントされない行」として「リスト先頭の太字ラベルのみの行」を認識しているが、太字なしの「- 結果: OK」のような短いラベル行も除外されることを知らない場合がある。

### 変更内容（プロジェクトルートCLAUDE.md 行249の直後に挿入）

変更前の行248〜249:
```
    - 空白行（何も書かれていない行または空白のみの行）
    - コードフェンス内の全ての行（開始行・終了行を含む）
```

行249の直後に以下の行を追加する（実質行5行以上の確保を促す指針として追記）:
```
    - 50文字以内でコロンまたは全角コロンで終わるプレーンラベル行（リスト記号の有無を問わない）— 例: 「- 結果: OK」「合否: 合格」はカウントに入らない
    - 太字なしの短いラベル行であっても、コロン終端かつ50文字以内であれば構造行として除外される点に注意
```

また行253の後に以下を追加する:
```
    重要な補足: コロンの後にコンテンツがある行でも、行全体が50文字以内でコロン終端であればプレーンラベルとして除外される。実質行5行を確保するには、50文字を超える説明文を各行に含めるか、コロン終端でない通常の文章を使用すること。
```

### 受け入れ検証方法

修正後のCLAUDE.mdで「50文字以内」「コロン終端」「プレーンラベル」という用語または同等の説明が「実質行にカウントされない行」セクションに含まれていることをGrepで確認する。workflow-plugin/CLAUDE.mdの対応箇所（行269〜277）と実質的に同等の説明が両ファイルに存在することを目視で確認する。

## FR-4: 禁止語複合語の回避表現拡充

### 変更対象と現状

プロジェクトルートCLAUDE.mdの行296が変更対象である。
現在の記述は禁止語を含む複合語の言い換えに関する1行の説明のみである。
manual_test・e2e_testフェーズで特に発生しやすい表現パターンが3種類ある。
「架空の入力データ」「処理が継続中の状態」「値が指定されていない状態」の3種類について具体的な回避表現例が不足している。
これらの表現パターンが過去のバリデーションエラー発生の主な要因となっていた。

### 変更内容（プロジェクトルートCLAUDE.md 行296の置き換え）

変更前（1行）:「禁止語を含む複合語も検出される。言い換えの事例は成果物品質要件の本体を参照すること」

変更後（複数行）:
```
    - 禁止語を含む複合語も部分一致で検出される。以下の言い換え例を参照すること
    - 架空・仮定の入力データを表す語が禁止語と部分一致する場合は「検証対象の入力値」「テストに使用する値」に言い換えること
    - 処理が継続中または完了していない状態を示す語が禁止語と部分一致する場合は「実行中の状態」「処理が継続している」に言い換えること
    - 設定や値が指定されていない状態を示す語が禁止語と部分一致する場合は「設定が省略されている」「値が明示されていない」に言い換えること
    - 「指定されていない変数」「追加調査が必要な事項」のような具体的な言い換えを採用すること
```

### 受け入れ検証方法

修正後のCLAUDE.mdで言い換え例が3種類以上記述されていることをGrepで確認する。workflow-plugin/CLAUDE.mdの対応箇所（行320付近）にも同等の言い換え例が追加されていることを確認する。

## FR-5: commitフェーズのファイル削除ガイダンス

### 変更対象と現状

プロジェクトルートCLAUDE.mdの行186が変更対象である。現在のフェーズ別Bashコマンド許可カテゴリ表（行173〜187）では、commitとpushのフェーズに対して「readonly, implementation」が許可カテゴリとして設定されている。implementationカテゴリにはrmコマンドが含まれているため、表の定義上はcommitフェーズでのrm実行が可能のように見える。しかしCLAUDE.md本文の他の箇所では「readonly, git」と記述されており、表の内容との矛盾が生じている。さらにphase-edit-guard.jsの実際のコマンドホワイトリストでcommitフェーズのgit操作は「git add」「git commit」「git tag」に限定されており、一般的なrmコマンドの実行可否については不明確である。

### 変更内容（フェーズ別Bashコマンド許可カテゴリ表 行186の更新）

変更前:
「| commit, push | readonly, implementation | Git操作のため |」

変更後:
「| commit, push | readonly, git | Git操作のため（ファイル削除が必要な場合はimplementationまたはrefactoringフェーズで完了すること） |」

### 追加するガイダンス（commitフェーズの説明として行186の下に注記を追加）

行187の後に以下の注記を追加する:
```
commitフェーズでのファイル削除が必要な場合は、implementationフェーズまたはrefactoringフェーズに戻して削除を完了すること。commitフェーズでrmがブロックされた場合、Orchestratorはworkflow_backでimplementationフェーズに差し戻すこと。subagentはエスカレーションとしてOrchestratorにブロック状況を報告し、フェーズ差し戻しの判断を求めること。
```

### 受け入れ検証方法

修正後のCLAUDE.mdでcommitフェーズのBashカテゴリが「readonly, git」と明記されていることをGrepで確認する。ファイル削除の代替手段として「implementationまたはrefactoringフェーズで完了すること」というガイダンスが記述されていることを確認する。

## 非機能要件

### NFR-1: バリデーション成功率の改善目標

修正後にFR-1〜FR-5を全て適用した状態で、parallel_verificationフェーズの4サブフェーズ合計リトライ回数が4回以下になることを目標とする。
1サブフェーズあたり平均1回以下のリトライに抑えることが基準である。
現状では16回以上のリトライが発生しており、修正により75%以上の削減を目指す。
計測はFR適用後の最初のタスク実行時にリトライ回数を記録することで行う。
改善目標の達成状況を評価し、成功率の低下が確認された場合は追加の修正を実施する。

### NFR-2: 後方互換性の維持

修正はCLAUDE.mdのテンプレート文言の変更が中心であり、artifact-validator.tsのバリデーションロジックには変更を加えない。
既存のタスクワークフローの挙動に影響を与えないことが必須条件である。
フェーズ遷移ロジック・HMAC整合性チェック・タスクインデックス管理については変更対象外とする。
後方互換性を確保するため、今回の変更は説明の充実と言い換え指示の追加のみに限定する。
修正前のプロンプトテンプレートで生成済みの成果物が再バリデーションされても結果に影響がないことを確認する。

### NFR-3: プロジェクトルートとworkflow-pluginの整合性維持

プロジェクトルートCLAUDE.mdとworkflow-plugin/CLAUDE.mdの両方に同等の説明が存在するよう、修正内容を両ファイルに同期すること。
どちらのファイルを参照するsubagentも同じ品質要件の理解が得られる状態を維持する。
二重定義を解消することが理想的だが、まず両ファイルの内容を一致させることを最優先の方針とする。
今後の修正時にも同期漏れが発生しないよう、docs_updateフェーズでの同期確認チェック項目を追加する。
整合性の管理方法として、両ファイルの差分をGrepで定期的に確認する運用を推奨する。

### NFR-4: 修正の可読性と保守性

追加する説明は簡潔かつ具体的な例示を伴うものとし、subagentが一読で理解できる構成にする。
過度に長い説明は却ってコンテキストを消費するため、各FRの追加行数は20行以内に収める。
将来の修正が容易なよう、変更箇所を局所化してセクション単位での管理ができるようにする。
可読性を高めるため、箇条書きよりも説明文（散文形式）を基本とし、具体例を行内に含める指針を採用する。
追加された説明と既存の説明の一貫性を保つため、同一セクション内での記述スタイルを統一する。

## テスト方法

修正完了後の検証は以下の手順で行う。

プロジェクトルートCLAUDE.mdとworkflow-plugin/CLAUDE.mdに対してGrepを実行し、テンプレート部分に禁止語が存在しないことを確認する。
両ファイルに「コードフェンス内でも角括弧は検出される」という旨の記述が存在することをGrepで確認する。
両ファイルに「50文字以内」または「プレーンラベル行」という記述が実質行の説明箇所に存在することをGrepで確認する。
フェーズ別Bashコマンド許可カテゴリ表のcommit行が「readonly, git」と記述されていることをGrepで確認する。
各確認が通過した後、新規タスクを対象にparallel_verificationフェーズを実行し、リトライ回数が4回以下になることを観測する。
成功率の改善を記録し、NFR-1の達成基準と照合する。
