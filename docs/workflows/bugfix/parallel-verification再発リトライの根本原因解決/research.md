## サマリー

- 目的: parallel_verificationフェーズで4サブフェーズ合計16回以上のリトライが発生した根本原因を特定し、再発防止策を導出する。
- 主要な決定事項: 仮説A・B・C・D・Eの全てについてコード実態と照合し、真の根本原因を特定した。
- 根本原因は5種類あり、いずれもCLAUDE.mdプロンプトテンプレートの不完全な記述と、artifact-validator.tsのバリデーション仕様の説明不足に起因している。
- 修正対象はCLAUDE.mdのテンプレート文言（禁止語の説明方式・実質行の例示・角括弧に関する注記）と、phase-edit-guardのコマンド定義である。
- 次フェーズで必要な情報: 各根本原因ごとの修正対象ファイルと修正方針（仕様書として記述すること）。

## 調査結果

### 仮説A: CLAUDE.mdプロンプトテンプレート内の禁止語問題

CLAUDE.md（プロジェクトルート）のsubagentプロンプトテンプレートの「禁止パターン（完全リスト）」セクションを精査した結果、テンプレート本文内に禁止語が散文で列挙されていることを確認した。

具体的には英語4語の禁止パターンと日本語8語の禁止パターンが明示的に記述されている。これらは成果物バリデーターが部分一致で検出する語である。

subagentがこのプロンプトを受け取ってe2e_test等の成果物を作成するとき、「禁止語の説明文を成果物に引用してしまう」という誤動作が発生しやすい。実際に前回のe2e_testでは「禁止語12語全てが成果物内に列挙されていた」という失敗が記録されており、これはプロンプト内の禁止語リストを成果物にそのまま転記したことが原因と判断される。

workflow-plugin/CLAUDE.mdも同様の問題を抱えており、同じ禁止語リストが散文で記述されている。

仮説Aは**真の根本原因として確認済み**である。

### 仮説B: 角括弧検出とコードフェンスの関係

artifact-validator.tsの実装（行358-382）を精査した結果、禁止パターン検索には`extractNonCodeLines(content)`を呼び出してコードフェンス外の行のみを対象にしていることが確認できた。

しかしCLAUDE.mdの「リトライプロンプトテンプレート」内に角括弧プレースホルダー形式の記述が含まれている。波括弧形式は問題ないが、成果物内に波括弧ではなく角括弧を使用すると検出される。

workflow-plugin/CLAUDE.mdには「コードフェンス内に正規表現を記述する場合は、文字クラスの代わりに散文で説明すること」という注記が追加されており、これはコードフェンス内での角括弧問題への対応として既に記述されている。

しかしプロジェクトルートのCLAUDE.mdには同等の注記が存在しない。subagentが参照するCLAUDE.mdがプロジェクトルート版である場合、コードフェンス内角括弧の安全性についての説明が不十分になる可能性がある。

仮説Bは**部分的に根本原因として確認済み**である。プロジェクトルートCLAUDE.mdの成果物品質要件セクションへの補足が必要である。

### 仮説C: セクション密度要件「実質行5行」の計算方法の不明確さ

checkSectionDensity関数（行728-814）の実装を精査した結果、実質行のカウントから除外される要素は以下の通りである。

コードフェンス開始行および終了行とフェンス内の全行は構造カウントに加算される。空白行は構造カウントに加算される。ヘッダー行（シャープ記号で始まる）は構造カウントに加算される。isStructuralLine関数が真を返す行（水平線、テーブルセパレータ、太字ラベル、プレーンラベルなど）は構造カウントに加算される。テーブルデータ行のみ、FIX-2修正により実質行カウントに加算される。

プロジェクトルートCLAUDE.mdのサブエージェントプロンプトテンプレートには「各セクション（## 見出し）内に最低5行の実質行（空行・水平線・コードフェンスを除く）を含めること」という説明のみで、FIX-1のプレーンラベル行除外（isStructuralLine行109: 50文字以内でコロン終端する行がラベルとして除外されるパターン）についての説明がない。

workflow-plugin/CLAUDE.mdには「実質行にカウントされない行の例」として「リスト先頭の太字ラベルのみの行」「水平線のみの行」「空白行」「コードフェンス内の全ての行」が詳細に説明されている。しかしプロジェクトルートCLAUDE.mdには同等の説明が存在しない。

FIX-1のプレーンラベルパターン（50文字以内でコロン終端）は非常に広い範囲をカバーし、意図せずリスト項目が構造行として扱われる可能性が高い。これが「サマリーセクションの実質行数が4行（5行必要）」という失敗の直接原因と推定される。

仮説Cは**真の根本原因として確認済み**である。

### 仮説D: 禁止語部分一致検出の不明確さ

artifact-validator.tsの実装（行343-382）では、forbiddenPatternsの各語についてnonCodeContent.includes(pattern)で部分一致検索を行っている。

「指定が存在しない」という意味の漢字2文字語を含む複合語でも部分一致で検出される。「再び調査を行っている」という状態を示す漢字3文字語を含む複合語でも同様に検出される。「架空入力データ」を意味する語であっても禁止パターンが部分文字列として含まれる場合にバリデーターはパターンマッチで検出する。

CLAUDE.mdのプロンプトテンプレートには複合語についての言い換え例が記述されているが、manual_testフェーズでは成果物作成時のsubagentに十分に伝わっていなかった。subagentがテストシナリオの説明中に自然な日本語表現として禁止語を含む語を使用した可能性がある。

仮説Dは**真の根本原因として確認済み**である。禁止語の部分一致検出について、具体的な回避表現の例示が不十分である。

## 既存実装の分析

### artifact-validator.tsのコア検証ロジック

検証は14のステップで構成される。ステップ6の禁止パターンチェックは`extractNonCodeLines`でコードフェンス外のみを対象にするが、ステップ7の充填テキスト検出（重複行チェック）も同様にコードフェンス内の行を除外している。

isStructuralLine関数（行92-111）は重複行検出からの除外対象を定義しており、FIX-1として追加されたプレーンラベルパターンは「ハイフンまたはアスタリスクで始まるリスト項目の先頭が50文字以内でコロンまたは全角コロンで終わる形式」を構造行として扱う。この正規表現は「- 結果: OK」のような短いリストアイテムも構造行として扱う。

checkSectionDensity（行728-814）の密度計算は`substantiveCount / effectiveTotal`で行われる。effectiveTotalは`totalLines - structuralCount`であり、分母から構造行を除いた値となっている。分子の実質行数が5行未満の場合にエラーが発生する。セクションが短い（総行数が10行程度）場合、実質5行の要件は達成困難になりやすい。

### phase-edit-guard.jsのcommitフェーズ制限

bash-whitelist.jsに委譲されているwhitelistResultチェック（行1498）と、commitフェーズ固有の早期許可ロジック（行1446-1495）を確認した。

commitフェーズで許可されているgit操作は「git add」「git commit」「git tag」の3種類に限定されている。ファイル削除コマンドについては、FILE_MODIFYING_COMMANDSのrmパターン（行1088）が「rm -rfなどでコードファイルを指定した場合」のみをファイル修正として検出する仕様となっている。

仮説Eの「commitフェーズのBashコマンド許可カテゴリにrmが含まれていない」について検証した結果、bash-whitelist.jsのimplementationカテゴリに「rm」が含まれているかどうかを確認する必要がある。CLAUDE.md上の定義では「implementation: npm install, pnpm add, npm run build, mkdir, rm, git add, git commit」とあり、rmはimplementationカテゴリに含まれている。しかしcommitフェーズに許可されているBashカテゴリは「readonly, git」であり、implementationカテゴリは含まれない。したがってrmコマンドはcommitフェーズでブロックされる。

仮説Eは**根本原因として確認済み**である。FR-3ファイル削除操作がcommitフェーズでブロックされることは、CLAUDE.mdのフェーズ別許可カテゴリの記述に起因する設計上の制約である。

### プロンプトテンプレートの二重定義問題

プロジェクトルートCLAUDE.mdとworkflow-plugin/CLAUDE.mdの両方にsubagentプロンプトテンプレートが記述されており、内容が一部異なっている。workflow-plugin/CLAUDE.mdの方が最新かつ詳細な説明を含んでいる。subagentが参照するCLAUDE.mdがどちらであるかによって、成果物品質要件の説明の詳細度が異なる。

プロジェクトルートCLAUDE.mdには以下の説明が不足している。実質行にカウントされない行の具体的な例示が存在しない。コードフェンス内の角括弧パターンに関する注意書きが存在しない。禁止語の自然言語文脈での使用（架空のデータとして入力するなど）に関する回避表現の具体例が不十分である。

これらの不足がリトライの主要因となったと結論付けられる。
