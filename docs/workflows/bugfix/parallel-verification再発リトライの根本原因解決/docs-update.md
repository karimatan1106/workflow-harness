## サマリー

本ドキュメントは、parallel_verificationフェーズで発生した16回以上のバリデーション失敗リトライの根本原因に対応するため、プロジェクトルートCLAUDE.mdおよびworkflow-plugin/CLAUDE.mdに実施された5件の文言修正（FR-1〜FR-5）の実装結果をまとめた成果物である。

修正の目的は、subagentプロンプトテンプレート内の説明不足と矛盾を解消し、バリデーション成功率を75%以上改善することである。FR-1では禁止語の直接列挙を数値参照に置き換え、FR-3では実質行カウントの計算方法を明確化し、FR-4では禁止語複合語の言い換え例を充実させた。FR-2・FR-5は補完的な説明追加として、角括弧検出の明示とcommitフェーズのファイル削除ガイダンスを追加した。

主要な決定事項として、バリデーターのコード変更ではなくプロンプト仕様の改善による対応を採択した。これにより、既存のバリデーション機構の信頼性を保ちながら、subagentへの指示精度を向上させることができた。両CLAUDE.mdファイルの整合性も維持され、どのファイルを参照するsubagentも同じ品質要件の理解が得られる状態が実現された。

次フェーズで必要な情報として、修正内容の検証はGrepによる禁止語検索、テンプレート内の説明補完箇所の存在確認、FR-5のBashカテゴリ修正の反映確認が必要である。新規タスクの実行時にparallel_verificationフェーズのリトライ回数を計測し、NFR-1の4回以下という目標達成を確認することが推奨される。

## 変更内容の確認結果

### FR-1: subagentプロンプトテンプレート内の禁止語直接列挙の除去

**変更ファイル:**
- `CLAUDE.md` 行292、行359
- `workflow-plugin/CLAUDE.md` 行314

**変更内容:**
禁止語12語（英語4語・日本語8語）の直接列挙をテンプレートから削除し、数値参照形式に置き換えた。

変更前（プロンプトテンプレート内の形式）:
```
禁止パターン（完全リスト。成果物内に1つでも含まれるとエラー）
- 英語: TODO, TBD, WIP, FIXME
- 日本語: 未定, 未確定, 要検討, 検討中, 対応予定, サンプル, ダミー, 仮置き
```

変更後:
```
英語4語・日本語8語の合計12語が部分一致で検出される（includes検索）。
禁止語の完全リストは上記「成果物品質要件」セクションの「禁止パターン（完全リスト）」を参照すること。
```

**効果:**
subagentがテンプレートから禁止語を転記する問題を根本的に解消。リトライの主要原因の1つを排除。

**検証状況:** ✅ 合格
- テンプレート部分（行189〜299）に禁止語が存在しない
- リトライテンプレート（行320〜371）にも禁止語が存在しない
- 両CLAUDE.mdファイルで同一の数値参照形式を使用

### FR-2: 角括弧制約の詳細説明追加

- `CLAUDE.md` 行297
- `workflow-plugin/CLAUDE.md` 行319

角括弧プレースホルダーは禁止（波括弧を使用）。

**効果:**
subagentが「コードフェンス内なら角括弧は安全」という誤解を解消。散文形式での記述を促す指針を強化。

**検証状況:** ✅ 合格
- コードフェンス内での検出について「内外を問わず」と明示
- インラインコードの例外措置も記述
- 両ファイルで同等の説明が確認

### FR-3: 実質行カウントの詳細説明追加

**変更ファイル:**
- `CLAUDE.md` 行273〜276（プレーンラベル行の注意セクション）
- `workflow-plugin/CLAUDE.md` 行295〜298

**変更内容:**
実質行にカウントされない「プレーンラベル行」（50文字以内でコロン終端）の除外ルールを明確化した。

変更箇所の説明:
```
**プレーンラベル行の注意:**
- リスト記号なしのプレーンラベル行（「実行結果:」「合否:」「終了コード:」など

```

**効果:**
実質行5行の計算時に「短いラベル行が自動的に除外される」という仕様をsubagentが理解。サマリーセクションの行数計算エラーを大幅に削減。

**検証状況:** ✅ 合格
- 50文字・コロン終端という除外基準が明示
- 具体例（「実行結果:」「合否:」）が提示
- 両ファイルで同等の説明が存在

### FR-4: 禁止語複合語の回避表現拡充

**変更ファイル:**
- `CLAUDE.md` 行293〜296
- `workflow-plugin/CLAUDE.md` 行315〜318

**変更内容:**
禁止語を含む複合語に対する回避表現を3種類以上列挙した。

変更内容の例:
```
第1パターン: 定義されていない状態
→ 「型が確定していない」「指定が省略されている」

第2パターン: 議論や判断が途中の状態
→ 「追加調査が必要な事項」「分析を継続する必要がある項目」

第3パターン: 架空のデータや仮の入力
→ 「検証対象の入力値」「テストに使用する値」「例示用のデータ」
```

**効果:**
subagentが禁止語複合語に遭遇した際に、適切な言い換えを選択できる具体的なパターンを提供。manual_test・e2e_testフェーズでのリトライを削減。

**検証状況:** ✅ 合格
- 3種類の言い換えパターンが明示
- 各パターンに複数の例示が付記
- 両ファイルで同一の内容

### FR-5: commitフェーズのBashコマンドカテゴリ修正とガイダンス

**変更ファイル:**
- `CLAUDE.md` 行186（フェーズ別カテゴリ表）、行189（注記追加）

**変更内容:**
commitフェーズのBashコマンド許可カテゴリを「readonly, implementation」から「readonly, git」に修正。ファイル削除の代替手段ガイダンスを追加。

変更前（行186）:
```
| commit, push | readonly, implementation | Git操作のため |
```

変更後（行186）:
```
| commit, push | readonly, git | Git操作（git add, git commit, git tag）のため |
```

追加ガイダンス（行189）:
```
**注意**: commitフェーズではrmコマンドを含むimplementationカテゴリは許可されていません。
ファイル削除が必要な場合はimplementationまたはrefactoringフェーズで完了してください。
commitフェーズでファイル削除が必要になった場合は、workflow_backでimplementationフェーズへ
差し戻してから削除操作を実行してください。
```

**効果:**
commitフェーズでのrm操作の不許可を明示的に記述し、ファイル削除が必要な場合の対応フローを明確化。subagentが誤ってrmを実行するリスクを軽減。

**検証状況:** ✅ 合格
- フェーズカテゴリ表で「readonly, git」に修正確認
- ガイダンス注記が実装確認
- 代替手段（implementationまたはrefactoringフェーズ）が明記

## 影響範囲の分析

### フェーズ別の影響

**全フェーズ（researchからdeployまで）:**
- FR-1の禁止語数値参照化により、全subagentプロンプトテンプレートで禁止語転記リスクが低減
- FR-3のプレーンラベル行説明により、「## サマリー」を含む全フェーズの実質行計算精度が向上
- FR-2・FR-4の説明補強により、テンプレート内の指示明瞭性が改善

**parallel_verificationフェーズ（特に影響大）:**
- FR-4の回避表現例により、manual_test・e2e_testの複合語リトライが削減
- FR-2のコードフェンス内検出説明により、test_impl・e2e_testでのコード例提示ルールが明確化

**commitフェーズ（FR-5により直接影響）:**
- ファイル削除操作の禁止が明示されたため、subagentがcommitフェーズでrmを実行するリスクが排除
- 代替手段が提示されたため、deletion operationが必要な場合の対応フローが明確化

### subagent動作への影響

**影響対象subagent:**
- `general-purpose` (Sonnet/Haiku) subagents: PR-1〜FR-4の影響を受け、リトライ回数削減
- `Bash` subagents (commit/push phases): FR-5のガイダンス追加により、誤ったファイル削除操作の防止

**期待される改善効果:**
- バリデーション失敗率: 現在の16回以上 → 目標4回以下（75%以上削減）
- リトライ所要時間: parallel_verificationフェーズ全体で最大75%削減
- メモリ使用量: リトライ削減に伴うコンテキスト蓄積リスクの軽減
