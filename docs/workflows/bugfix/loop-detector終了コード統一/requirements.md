# 要件定義書

## サマリー

loop-detector.jsのグローバルエラーハンドラにおいて、終了コードの不一貫性が存在します。現在42行目と47行目のuncaughtExceptionとunhandledRejectionハンドラではprocess.exit(1)が使用されていますが、同じファイル内のループ検出時（345行目）やタイムアウト時（395行目）ではprocess.exit(2)が使用されています。これは他のフック（enforce-workflow.js、phase-edit-guard.js）との不統一であり、Fail-Closed原則（REQ-3）に対する準拠の一貫性を損なっています。本タスクはこの不一貫性を解決するため、loop-detector.jsのグローバルエラーハンドラの終了コードを統一してexit(1)からexit(2)に変更することです。修正対象は2箇所のみの軽微な変更であり、ファイル内の他のexit()呼び出しには影響を与えません。

## 機能要件

### REQ-1: グローバルエラーハンドラの終了コード統一

loop-detector.jsのグローバルエラーハンドラにおいて、uncaughtException（42行目）とunhandledRejection（47行目）の2箇所でprocess.exit()の終了コードをexit(1)からexit(2)に変更する必要があります。これらのハンドラは予期しないエラーが発生した時点でプロセスを中止する責務を担っており、他のフック（enforce-workflow.js、phase-edit-guard.js）ではグローバルエラーハンドラを含む全障害系でexit(2)を使用しています。

### REQ-2: 既存exit()呼び出しとの整合性保証

loop-detector.js内には合計10箇所のprocess.exit()呼び出しが存在します。本タスクの修正はグローバルエラーハンドラの2箇所のみに限定され、ループ検出時（345行目）やタイムアウト時（395行目）の既存exit(2)呼び出しには変更を加えません。正常系の許可・スキップ処理（359行目、367行目、374行目、384行目）のexit(0)、およびstdinエラー・入力なし（404行目、413行目）のexit(0)も変更対象外です。修正後、ファイル内のexit()使用は0（正常系）と2（エラー系）に明確に分離されます。

### REQ-3: Fail-Closed原則の完全準拠

Claude Codeのフック仕様ではexit code 2がブロック（処理中止）を意味し、exit code 1は警告レベルの終了コードとして扱われます。グローバルエラーハンドラは予期しないエラーが発生した際に確実に処理を中止する必要があるため、exit(2)の使用が求められます。enforce-workflow.jsおよびphase-edit-guard.jsではグローバルエラーハンドラを含む全障害系でexit(2)を統一使用しており、loop-detector.jsがこの原則に従わないことは一貫性の欠損を意味します。本修正により3フック全てでFail-Closed原則が統一されます。

### REQ-4: 他フックとの設計パターン統一

enforce-workflow.jsのグローバルエラーハンドラ（37行目、42行目）ではexit(2)を使用し、phase-edit-guard.jsではEXIT_CODES定数（SUCCESS=0、WARNING=1、BLOCK=2）を定義してグローバルエラーハンドラ（39行目、44行目）でBLOCK（2）を使用しています。loop-detector.jsのみ終了コード定数が未定義であり、ハードコード値を直接使用しています。本タスクはまずprocess.exit(1)をprocess.exit(2)に変更することで3フック間の動作を統一し、将来的には終了コード定数の導入も検討の余地があります。

## 非機能要件

### NFR-1: 変更の最小限性

本修正はloop-detector.jsの42行目と47行目の2行のみの変更に限定します。変更対象はprocess.exit(1)からprocess.exit(2)への引数変更のみであり、ハンドラの処理ロジック、エラーメッセージ、ログ出力、その他のコード部分には一切の変更を加えません。最小限の変更により回帰リスクを排除し、修正の正当性を明確にします。

### NFR-2: コード品質の維持

修正後のコードはリント、フォーマット基準を満たし、既存のテストケースを継続して通過する必要があります。プロセスの安全性を損なわないよう、グローバルエラーハンドラの機能（エラーログ出力、プロセス終了）に変更を加えず、終了コードのみを変更します。

### NFR-3: ドキュメント整合性

修正によりloop-detector.jsのコメント（存在する場合）と実装が整合していることを確認します。Fail-Closed原則（REQ-3、FR-2）に関する既存ドキュメント（README、アーキテクチャドキュメント等）との矛盾がないことを保証します。

## 受け入れ基準

### AC-1: グローバルエラーハンドラの終了コード統一

修正後、loop-detector.jsの42行目（uncaughtException）と47行目（unhandledRejection）でprocess.exit(2)が使用されていることを確認します。修正前後でのコード差分は「1」から「2」への変更のみであることを視覚的に確認できます。

### AC-2: ファイル全体の整合性確認

修正後のloop-detector.jsにおいて、他のprocess.exit()呼び出しが期待される値のままであることを確認します。具体的には、ループ検出時（345行目）がexit(2)、タイムアウト時（395行目）がexit(2)、正常系処理（359行目、367行目、374行目、384行目、404行目、413行目）がexit(0)であることを検証します。

### AC-3: 他フックとの統一性確認

修正後、loop-detector.js、enforce-workflow.js、phase-edit-guard.jsの3フック全てにおいて、グローバルエラーハンドラがexit(2)を使用していることを確認します。これにより、Fail-Closed原則の統一的な準拠が達成されたことを示します。

### AC-4: テストの確認

既存のテストケース（特にfail-closed.test.ts内の関連テスト）が修正後も継続して通過することを確認します。必要に応じて、グローバルエラーハンドラのexit(2)動作を直接検証するテストケースが追加されていることを確認します。

## 影響範囲

### 影響を受けるファイル

修正対象はworkflow-plugin/hooks/loop-detector.jsの42行目と47行目のみです。他のフック（enforce-workflow.js、phase-edit-guard.js）、テストファイル、ドキュメント、その他のプロジェクトファイルは変更対象外です。

### 影響分析

終了コードの変更により、loop-detector.jsがグローバルエラーハンドラで返すシグナルが1から2に変更されます。Claude Codeのフック仕様ではexit code 2がブロック（強制中止）を意味するため、予期しないエラー発生時により確実に処理が中止される可能性が高まります。既存の動作フロー（エラー時のプロセス終了）は変わらないため、外部の呼び出し側への影響はありません。

### テスト対象範囲

グローバルエラーハンドラのexit(2)動作が正常に機能することを検証するため、uncaughtExceptionおよびunhandledRejectionのテストケースを対象とします。これらのハンドラが意図的にエラーを発生させた場合に、ログ出力とexit(2)が適切に実行されることを確認します。

### 後方互換性

グローバルエラーハンドラは予期しないエラーが発生した際のみ実行される例外的な処理パスです。正常系の処理フロー（ループ検出、タイムアウト、許可・スキップ処理）にはいかなる影響も与えません。既存の動作仕様（エラー時にプロセスが終了する）は変わらないため、後方互換性は保証されます。
