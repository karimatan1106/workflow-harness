# 脅威モデル

## サマリー

本タスクはloop-detector.jsのグローバルエラーハンドラにおいて、プロセス終了コードをexit(1)からexit(2)に統一する修正です。
修正対象は42行目のuncaughtExceptionハンドラと47行目のunhandledRejectionハンドラの2行のみです。
修正により、loop-detector.jsが他のフック（enforce-workflow.js、phase-edit-guard.js）と一貫してFail-Closed原則に準拠します。
修正範囲が2行に限定されているため、脅威の種類と影響範囲も非常に限定的です。
主な脅威は終了コード値の意味解釈と既存テストコードとの整合性の2点に集約されます。

---

## 脅威

本修正に関連する脅威はSTRIDE分析の結果、実質的なリスクが極めて低いことが確認されました。
修正はprocess.exit()の引数を1から2に変更するだけであり、新たな攻撃面を生成しません。
終了コード値は単一の整数であり、機密情報を含まないメタデータです。
主要な脅威は改ざん（Tampering）カテゴリにおける理論上のモンキーパッチ攻撃のみですが、発生確率は極めて低いです。
全体として、本修正はセキュリティリスクを増加させず、むしろFail-Closed原則の統一によりセキュリティ姿勢を強化します。

## STRIDE分析

### Spoofing（なりすまし）

スプーフィングの脅威は本修正には適用されません。終了コード値の変更は、エラーハンドラが予期しないエラー発生時にプロセスを中止するメカニズムには影響を与えません。グローバルエラーハンドラの実行主体は引き続きNode.jsランタイムの内部イベント（uncaughtException、unhandledRejection）であり、攻撃者が当該ハンドラになりすまして別の終了コードを返す可能性はありません。終了コードは単なる数値であり、ハンドラの真正性確認に用いられるものではないため、この観点からのリスクは存在しません。

### Tampering（改ざん）

改ざん脅威は本修正の実装時点では排除されます。修正内容は確定的な値変更（1から2への変更）であり、修正後のコードレビューとテスト実行により改ざんの有無は容易に検証可能です。ただし、本修正が本番環境にデプロイされた後、不正なプロセスやエージェントが終了コード値を動的に改ざんする可能性は存在します。Claude Codeのフック実行時、悪意あるスクリプトがprocess.exit()の引数を横取りして別の値を使用する可能性は低いものの、Node.js内部の動的パッチやモンキーパッチ攻撃は理論上可能です。ただし、グローバルエラーハンドラの実装がNode.jsのコア機能に依拠しているため、修正による追加的な改ざん脅威は発生しません。

### Repudiation（否認）

否認の脅威は本修正には該当しません。終了コード値の変更は監査ログやエラーメッセージの生成には影響を与えません。グローバルエラーハンドラは引き続きstderrにエラーログを出力し、例外の発生事実を記録します。終了コードはプロセス終了時に親プロセスに返されるメタデータであり、当該エラーハンドラの実行日時や実行者を示すものではありません。したがって、終了コード値の変更により、エラーハンドラの実行事実を否認する道筋が増すことはありません。

### Information Disclosure（情報漏洩）

情報漏洩の脅威は本修正には実質的に該当しません。終了コード値は単一の数値（0、1、2等）であり、機密情報を含まない一般的なメタデータです。修正後、グローバルエラーハンドラのexit(2)呼び出しにより、予期しないエラー発生時により確実にプロセスが中止される点は、むしろ情報漏洩を防ぐ効果を持ちます。プロセスが即座に中止されることで、エラー状態が持続し続ける時間が短縮され、その間に機密情報が外部に漏洩する窓口が縮小します。ただし、終了コード値の変更そのものが新たな情報チャネルを生成するわけではないため、このリスクは無視できる水準です。

### Denial of Service（サービス不可）

サービス不可（DoS）の脅威は本修正の実装方法により排除されます。グローバルエラーハンドラの実行条件（uncaughtException、unhandledRejection）は修正前後で変わらず、修正は終了コード値の変更のみです。したがって、修正後もグローバルエラーハンドラはエラー発生時に同様のリソース消費（ログ出力、プロセス終了）を実行し、追加的なDoS脆弱性を導入しません。むしろ、終了コード2によるより強制的な中止により、エラー状態が持続することによるリソース枯渇の可能性が低下します。

### Elevation of Privilege（権限昇格）

権限昇格の脅威は本修正には該当しません。終了コード値は、プロセスが実行される際の権限レベルや、システムリソースへのアクセス権限に影響を与えません。グローバルエラーハンドラは引き続きNode.jsプロセスが起動時に付与された権限の範囲内でのみ動作し、終了コード値の変更により新たに高い権限を取得することはありません。Claude Codeのフック実行メカニズムでは、終了コード2の返却は処理を中止する指示として解釈されるだけであり、フックプロセスの権限レベルを変更するものではありません。

## リスク評価

### リスク1: 終了コード値の意味解釈違い

**発生確率**: 低（修正内容が明確）

**影響度**: 中（フック動作の不確定性）

終了コード値の変更により、Claude Codeのフック実行エンジン側での解釈が予期と異なる可能性は低いものの、0以外（正常系）、1（警告），2（ブロック）の3値の意味が設計仕様と乖離する場合、フック全体の動作が不確定になる可能性があります。本修正は終了コード1から2への変更であり、これはより強制的なブロック動作を意味するため、単に関連仕様への準拠を強化するものです。ただし、既存のテストコードやドキュメント内で終了コード値の意味が異なる記載方法をされている場合、整合性の欠損が生じる可能性があります。

**対策**: テスト実行時にfail-closed.test.tsおよび関連テストケースが継続して通過することを確認し、終了コード2がブロック動作として正しく機能することを検証します。

### リスク2: 既存テストコードとの整合性

**発生確率**: 中（テスト仕様との依存性）

**影響度**: 中（修正の正当性確認）

既存のテストスイート（特にfail-closed.test.ts）がuncaughtExceptionおよびunhandledRejectionのハンドラに対して終了コード1を期待値として記載している可能性があります。修正後、テストコードが「exit code 2 expected, got 1」のような形式で失敗する可能性があります。ただし、テストコードが正しく更新されていれば、修正によるテスト失敗は発生しません。一方で、修正の正当性（終了コード2が正しいのか1なのか）を確認するための証拠としてテスト内容が重要です。

**対策**: テスト実行段階（testing フェーズ）で既存テストスイートを実行し、グローバルエラーハンドラのexit(2)動作に関連するテストが通過することを確認します。テスト失敗が発生した場合は、テストコードの期待値が正しいものか（exit(2)か exit(1)か）を再検証します。

### リスク3: 他フックとの動作不一致

**発生確率**: 低（修正目的が統一）

**影響度**: 中（ワークフロー全体の堅牢性）

本修正の目的はloop-detector.js、enforce-workflow.js、phase-edit-guard.jsの3フック間でFail-Closed原則を統一することです。もし他フックの動作仕様が最新ドキュメントと乖離している場合、修正後も全フックの統一性が達成されない可能性があります。例えば、enforce-workflow.jsが実際にはexit(1)を返しているのに、ドキュメント上ではexit(2)とされている場合、修正後も不整合が残存します。

**対策**: 修正後、enforce-workflow.jsおよびphase-edit-guard.jsのグローバルエラーハンドラが実際にexit(2)を使用していることをコード確認により検証します。設計ドキュメントと実装が整合していることを確認してから、テスト実行に進みます。

### リスク4: グローバルエラーハンドラの副作用

**発生確率**: 極低（修正範囲が限定的）

**影響度**: 低（ハンドラ内部処理の不変）

修正は終了コード値のみの変更であり、グローバルエラーハンドラの処理ロジック（エラーログ出力、スタックトレース表示等）には変更を加えません。したがって、修正によりハンドラの副作用が増加することはありません。ただし、終了コード2がより強制的なプロセス中止を意味する場合、修正後のプロセス中止タイミングが若干異なる可能性があります（例えば、残存するコールバック処理の実行有無）。ただし、Node.jsのprocess.exit()動作は終了コード値に関係なく即座にプロセスを中止するため、副作用の差は無視できる水準です。

**対策**: テスト実行時にグローバルエラーハンドラのログ出力が正常に記録されていることを確認します。

## 緩和策

### 対策1: 修正前後のコード差分確認

修正を実装した直後に、loop-detector.jsの42行目と47行目のコード差分を視覚的に確認し、引数値が1から2に正確に変更されていることを検証します。grep等のツールを用いて、当該ファイル内に「process.exit(1)」という文字列が残存していないことを確認することで、修正漏れを排除します。

### 対策2: ファイル全体の整合性検証

修正後、loop-detector.js全体をレビューし、ループ検出時（345行目）とタイムアウト時（395行目）の既存exit(2)呼び出し、および正常系処理のexit(0)呼び出しが変更されていないことを確認します。修正が意図した範囲に限定されていることを示す証拠とします。

### 対策3: 既存テストスイートの実行

test-n1-scope-validator.test.ts、test-n2-phase-edit-guard.test.ts、test-n3-test-authenticity.test.ts、test-n4-enforce-workflow.test.ts、test-n5-set-scope.test.tsの既存テストスイートを実行し、全テストケースが通過することを確認します。特にfail-closed.test.tsが存在する場合は、グローバルエラーハンドラのexit(2)動作に関連するテストが通過していることを確認します。

### 対策4: 他フックとの統一性確認

修正後、enforce-workflow.jsおよびphase-edit-guard.jsのグローバルエラーハンドラが実際にexit(2)を使用していることをコード検査により確認します。3フック間の終了コード値が統一されていることを示す証拠（コード引用等）を記録します。

### 対策5: コード品質基準の維持

修正後のコードがリント、フォーマット基準を満たしていることを確認します。修正内容が最小限（2行の引数変更）であるため、コード品質への負の影響は極めて低いものの、念のため既存の静的解析ツール（eslint等）を実行して基準適合を検証します。

### 対策6: ドキュメント整合性の確認

本修正に関連するドキュメント（README、アーキテクチャドキュメント、フック仕様等）がexit code 2の意味（Fail-Closed、処理ブロック）を正確に記載していることを確認します。不整合がある場合は、テスト実行段階で改めて検証してから修正を進めます。

---

## 結論

本修正は終了コード値のみの軽微な変更であり、グローバルエラーハンドラの処理ロジックには影響を与えません。
主な脅威は終了コード値の意味解釈と既存テストコードとの整合性の2点に限定されます。
テスト実行による検証とコード差分確認により、修正の正当性と安全性を確保できます。
修正後、loop-detector.jsが他フックと一貫してFail-Closed原則に準拠することで堅牢性が向上します。
STRIDE分析の全6カテゴリにおいて実質的なセキュリティリスクの増加は確認されませんでした。
